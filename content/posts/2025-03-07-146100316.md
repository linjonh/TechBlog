    <h2 id="%E5%BC%95%E8%A8%80" name="%E5%BC%95%E8%A8%80">
     引言
    </h2>
    <p>
     单例模式（Singleton Pattern）是设计模式中最简单且最常用的创建型模式之一。它的核心思想是确保一个类只有一个实例，并提供一个全局访问点来获取该实例。单例模式在许多场景中非常有用，例如配置管理、线程池、数据库连接池等。
    </p>
    <hr/>
    <h3 id="1.%20%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F%E7%9A%84%E6%A0%B8%E5%BF%83%E6%80%9D%E6%83%B3" name="1.%20%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F%E7%9A%84%E6%A0%B8%E5%BF%83%E6%80%9D%E6%83%B3">
     1. 单例模式的核心思想
    </h3>
    <p>
     单例模式的核心思想是：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        私有化构造函数
       </strong>
       ：防止外部通过
       <code>
        new
       </code>
       关键字创建实例。
      </p>
     </li>
     <li>
      <p>
       <strong>
        提供一个静态方法
       </strong>
       ：用于获取类的唯一实例。
      </p>
     </li>
     <li>
      <p>
       <strong>
        确保唯一性
       </strong>
       ：在整个应用程序生命周期中，类的实例只有一个。
      </p>
     </li>
    </ol>
    <hr/>
    <h3 id="2.%20%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F%E7%9A%84%E5%AE%9E%E7%8E%B0%E6%96%B9%E5%BC%8F" name="2.%20%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F%E7%9A%84%E5%AE%9E%E7%8E%B0%E6%96%B9%E5%BC%8F">
     2. 单例模式的实现方式
    </h3>
    <p>
     单例模式有多种实现方式，每种方式都有其优缺点。以下是几种常见的实现方式：
    </p>
    <h4 id="2.1%20%E9%A5%BF%E6%B1%89%E5%BC%8F%E5%8D%95%E4%BE%8B" name="2.1%20%E9%A5%BF%E6%B1%89%E5%BC%8F%E5%8D%95%E4%BE%8B">
     2.1 饿汉式单例
    </h4>
    <p>
     饿汉式单例在类加载时就创建实例，因此是线程安全的。
    </p>
    <pre><code class="language-java">public class Singleton {
    // 在类加载时创建实例
    private static final Singleton INSTANCE = new Singleton();

    // 私有化构造函数
    private Singleton() {}

    // 提供全局访问点
    public static Singleton getInstance() {
        return INSTANCE;
    }
}</code></pre>
    <p>
     饿汉式是最简单的单例模式的写法，保证了线程的安全，在很长的时间里，我都是饿汉模式来完成单例 的，因为够简单，后来才知道饿汉式会有一点小问题，看下面的代码：
    </p>
    <pre><code class="language-java"> public class Hungry {
    private byte[] data1 = new byte[1024];
    private byte[] data2 = new byte[1024];
    private byte[] data3 = new byte[1024];
    private byte[] data4 = new byte[1024];
    
    private Hungry() {
    }
    private final static Hungry hungry = new Hungry();
    public static Hungry getInstance() {
        return hungry;
    }
 }</code></pre>
    <p>
     在Hungry类中，我定义了四个byte数组，当代码一运行，这四个数组就被初始化，并且放入内存了，如 果长时间没有用到getInstance方法，不需要Hungry类的对象，这不是一种浪费吗？我希望的是 只有用 到了 getInstance方法，才会去初始化单例类，才会加载单例类中的数据。所以就有了 第二种单例模 式：懒汉式。
    </p>
    <p>
     <strong>
      优点
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       实现简单，线程安全。
      </p>
     </li>
    </ul>
    <p>
     <strong>
      缺点
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       如果实例未被使用，会造成资源浪费。
      </p>
     </li>
    </ul>
    <hr/>
    <h4 id="2.2%20%E6%87%92%E6%B1%89%E5%BC%8F%E5%8D%95%E4%BE%8B" name="2.2%20%E6%87%92%E6%B1%89%E5%BC%8F%E5%8D%95%E4%BE%8B">
     2.2 懒汉式单例
    </h4>
    <p>
     懒汉式单例在第一次调用
     <code>
      getInstance()
     </code>
     时才创建实例。
    </p>
    <pre><code class="language-java">public class LazyMan {
    private LazyMan() {
        System.out.println(Thread.currentThread().getName()+"Start");
    }
    private static LazyMan lazyMan;
    public static LazyMan getInstance() {
        if (lazyMan == null) {
            lazyMan = new LazyMan();
        }
        return lazyMan;
    }
    // 测试并发环境，发现单例失效
    public static void main(String[] args) {
        for (int i = 0; i &lt; 10; i++) {
            new Thread(()-&gt;{
                LazyMan.getInstance();
                  }).start();
        }
    }
 }</code></pre>
    <p>
     <strong>
      缺点
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       线程不安全，多个线程可能同时进入
       <code>
        if (instance == null)
       </code>
       条件，导致创建多个实例。
      </p>
     </li>
    </ul>
    <hr/>
    <h4 id="2.3%20%E7%BA%BF%E7%A8%8B%E5%AE%89%E5%85%A8%E7%9A%84%E6%87%92%E6%B1%89%E5%BC%8F%E5%8D%95%E4%BE%8B" name="2.3%20%E7%BA%BF%E7%A8%8B%E5%AE%89%E5%85%A8%E7%9A%84%E6%87%92%E6%B1%89%E5%BC%8F%E5%8D%95%E4%BE%8B">
     2.3 线程安全的懒汉式单例
    </h4>
    <p>
     通过在
     <code>
      getInstance()
     </code>
     方法上加锁，可以解决懒汉式单例的线程安全问题。
    </p>
    <pre><code class="language-java">public class LazyMan {
    private LazyMan() {
    }
    private static LazyMan lazyMan;
    public static LazyMan getInstance() {
        if (lazyMan == null) {
            synchronized (LazyMan.class) {
                if (lazyMan == null) {
                    lazyMan = new LazyMan();
                }
            }
        }
        return lazyMan;
    }
 }</code></pre>
    <p>
     保证了线程的安全性，又符合了懒加载，只有在用到的时候，才会去初始化，调用 效率也比较高，但是这种写法在极端情况还是可能会有一定的问题。因为 ：
    </p>
    <blockquote>
     <p>
      lazyMan = new LazyMan();
     </p>
    </blockquote>
    <p>
     不是原子性操作，至少会经过三个步骤：
    </p>
    <p>
     1. 分配对象内存空间
    </p>
    <p>
     2. 执行构造方法初始化对象
    </p>
    <p>
     3. 设置instance指向刚分配的内存地址，此时instance ！=null；
    </p>
    <p>
     <span style="color:#fe2c24">
      由于指令重排，导致A线程执行 lazyMan = new LazyMan();的时候，可能先执行了第三步（还没执行第 二步），此时线程B又进来了，发现lazyMan已经不为空了，直接返回了lazyMan，并且后面使用了返回 的lazyMan，由于线程A还没有执行第二步，导致此时lazyMan还不完整，可能会有一些意想不到的错 误，所以就有了下面一种单例模式。
     </span>
    </p>
    <hr/>
    <h4 id="2.4%20%E5%8F%8C%E9%87%8D%E6%A3%80%E6%9F%A5%E9%94%81%E5%AE%9A%EF%BC%88Double-Checked%20Locking%EF%BC%89" name="2.4%20%E5%8F%8C%E9%87%8D%E6%A3%80%E6%9F%A5%E9%94%81%E5%AE%9A%EF%BC%88Double-Checked%20Locking%EF%BC%89">
     2.4 双重检查锁定（Double-Checked Locking）
    </h4>
    <p>
     双重检查锁定是一种优化后的线程安全懒汉式单例实现方式。
    </p>
    <p>
     这种单例模式只是在上面DCL单例模式增加一个volatile关键字来避免指令重排：
    </p>
    <pre><code class="language-java"> public class LazyMan {
    private LazyMan() {
    }
    private volatile static LazyMan lazyMan;
    public static LazyMan getInstance() {
        if (lazyMan == null) {
            synchronized (LazyMan.class) {
                if (lazyMan == null) {
                    lazyMan = new LazyMan();
                }
            }
        }
        return lazyMan;
    }
}</code></pre>
    <p>
     <strong>
      优点
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       线程安全，且只有在第一次创建实例时加锁，性能较好。
      </p>
     </li>
    </ul>
    <p>
     <strong>
      注意
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       必须使用
       <code>
        volatile
       </code>
       关键字，防止指令重排序导致的问题。
      </p>
     </li>
    </ul>
    <hr/>
    <h4 id="2.5%20%E9%9D%99%E6%80%81%E5%86%85%E9%83%A8%E7%B1%BB%E5%AE%9E%E7%8E%B0%E5%8D%95%E4%BE%8B" name="2.5%20%E9%9D%99%E6%80%81%E5%86%85%E9%83%A8%E7%B1%BB%E5%AE%9E%E7%8E%B0%E5%8D%95%E4%BE%8B">
     2.5 静态内部类实现单例
    </h4>
    <p>
     静态内部类实现单例是一种优雅且线程安全的方式。
    </p>
    <pre><code class="language-java">public class Singleton {
    private Singleton() {}

    private static class SingletonHolder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return SingletonHolder.INSTANCE;
    }
}</code></pre>
    <p>
     <strong>
      优点
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       线程安全，且只有在调用
       <code>
        getInstance()
       </code>
       时才会加载
       <code>
        SingletonHolder
       </code>
       类，实现懒加载。
      </p>
     </li>
    </ul>
    <hr/>
    <h4 id="2.6%20%E6%9E%9A%E4%B8%BE%E5%AE%9E%E7%8E%B0%E5%8D%95%E4%BE%8B" name="2.6%20%E6%9E%9A%E4%B8%BE%E5%AE%9E%E7%8E%B0%E5%8D%95%E4%BE%8B">
     2.6 枚举实现单例
    </h4>
    <p>
     枚举实现单例是《Effective Java》推荐的方式，它天然支持线程安全和防止反射攻击。
    </p>
    <pre><code class="language-java">public enum Singleton {
    INSTANCE;

    public void doSomething() {
        System.out.println("Doing something...");
    }
}</code></pre>
    <p>
     <strong>
      优点
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       线程安全，代码简洁，防止反射和序列化破坏单例。
      </p>
     </li>
    </ul>
    <hr/>
    <h3 id="3.%20%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F%E7%9A%84%E4%BD%BF%E7%94%A8%E5%9C%BA%E6%99%AF" name="3.%20%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F%E7%9A%84%E4%BD%BF%E7%94%A8%E5%9C%BA%E6%99%AF">
     3. 单例模式的使用场景
    </h3>
    <p>
     单例模式适用于以下场景：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        全局配置管理
       </strong>
       ：例如读取配置文件，确保配置信息全局唯一。
      </p>
     </li>
     <li>
      <p>
       <strong>
        数据库连接池
       </strong>
       ：确保连接池只有一个实例，避免资源浪费。
      </p>
     </li>
     <li>
      <p>
       <strong>
        日志管理
       </strong>
       ：确保日志记录器全局唯一。
      </p>
     </li>
     <li>
      <p>
       <strong>
        线程池
       </strong>
       ：确保线程池的唯一性，避免重复创建线程。
      </p>
     </li>
    </ol>
    <hr/>
    <h3 id="4.%20%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F%E7%9A%84%E4%BC%98%E7%BC%BA%E7%82%B9" name="4.%20%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F%E7%9A%84%E4%BC%98%E7%BC%BA%E7%82%B9">
     4. 单例模式的优缺点
    </h3>
    <h4 id="%E4%BC%98%E7%82%B9%EF%BC%9A" name="%E4%BC%98%E7%82%B9%EF%BC%9A">
     优点：
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        节省资源
       </strong>
       ：避免重复创建对象，减少内存开销。
      </p>
     </li>
     <li>
      <p>
       <strong>
        全局访问点
       </strong>
       ：方便对唯一实例的管理和访问。
      </p>
     </li>
    </ul>
    <h4 id="%E7%BC%BA%E7%82%B9%EF%BC%9A" name="%E7%BC%BA%E7%82%B9%EF%BC%9A">
     缺点：
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        扩展性差
       </strong>
       ：单例类通常难以扩展，因为其构造函数是私有的。
      </p>
     </li>
     <li>
      <p>
       <strong>
        违背单一职责原则
       </strong>
       ：单例类既负责创建实例，又负责业务逻辑。
      </p>
     </li>
     <li>
      <p>
       <strong>
        测试困难
       </strong>
       ：单例类的全局状态可能导致测试困难。
      </p>
     </li>
    </ul>
    <hr/>
    <h3 id="5.%20%E6%80%BB%E7%BB%93" name="5.%20%E6%80%BB%E7%BB%93">
     5. 总结
    </h3>
    <p>
     单例模式是一种简单但强大的设计模式，适用于需要全局唯一实例的场景。通过不同的实现方式（如饿汉式、懒汉式、双重检查锁定、静态内部类、枚举等），可以满足不同的需求。
    </p>
    <p>
     在实际开发中，应根据具体场景选择合适的单例实现方式。如果需要懒加载且线程安全，推荐使用静态内部类或枚举实现；如果需要更高的性能，可以考虑双重检查锁定。
    </p>
