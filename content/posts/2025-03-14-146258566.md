    <p>
     分布式系统中，为了保证数据的唯一性和可扩展性，通常需要一个高效的全局唯一 ID（GUID） 生成方案。以下是几种常见的分布式全局 ID 生成方案，各有优缺点，具体选择要根据业务需求、性能要求和系统架构来决定。
    </p>
    <hr/>
    <h3>
     <a id="1_Snowflake_4">
     </a>
     <strong>
      1. 雪花算法（Snowflake）
     </strong>
    </h3>
    <h4>
     <a id="_5">
     </a>
     <strong>
      原理
     </strong>
    </h4>
    <p>
     雪花算法由 Twitter 提出，它生成 64 位的唯一 ID，格式如下：
    </p>
    <pre><code>| 1bit 符号位 | 41bit 时间戳 | 10bit 机器 ID | 12bit 计数器 |
</code></pre>
    <ul>
     <li>
      <strong>
       1bit 符号位
      </strong>
      ：始终为 0。
     </li>
     <li>
      <strong>
       41bit 时间戳
      </strong>
      ：可以支持 69 年（(2^{41}/(365
      <em>
       24
      </em>
      60
      <em>
       60
      </em>
      1000) )）。
     </li>
     <li>
      <strong>
       10bit 机器 ID
      </strong>
      ：最多支持 1024 个节点。
     </li>
     <li>
      <strong>
       12bit 计数器
      </strong>
      ：同一毫秒内最多生成 4096 个 ID。
     </li>
    </ul>
    <h4>
     <a id="_15">
     </a>
     <strong>
      优缺点
     </strong>
    </h4>
    <p>
     ✅
     <strong>
      优点
     </strong>
    </p>
    <ul>
     <li>
      按时间递增，适合数据库索引优化。
     </li>
     <li>
      本地生成，无需数据库，提高性能。
     </li>
     <li>
      可扩展性好，适用于分布式系统。
     </li>
    </ul>
    <p>
     ❌
     <strong>
      缺点
     </strong>
    </p>
    <ul>
     <li>
      <strong>
       时间回拨问题
      </strong>
      ：如果服务器时间发生回退，会导致 ID 可能重复。
     </li>
     <li>
      <strong>
       节点 ID 分配问题
      </strong>
      ：需要一个机制来确保每个节点 ID 唯一。
     </li>
    </ul>
    <h4>
     <a id="_25">
     </a>
     <strong>
      适用场景
     </strong>
    </h4>
    <p>
     适用于高并发、大规模分布式系统，如订单号、消息 ID、日志 ID。
    </p>
    <hr/>
    <h3>
     <a id="2_UUIDUniversally_Unique_Identifier_30">
     </a>
     <strong>
      2. UUID（Universally Unique Identifier）
     </strong>
    </h3>
    <h4>
     <a id="_31">
     </a>
     <strong>
      原理
     </strong>
    </h4>
    <p>
     UUID 是 128 位的唯一标识符，通常使用
     <strong>
      UUID v1/v4
     </strong>
     ：
    </p>
    <ul>
     <li>
      <strong>
       UUID v1
      </strong>
      ：基于时间戳 + MAC 地址，可能泄露设备信息。
     </li>
     <li>
      <strong>
       UUID v4
      </strong>
      ：随机数生成，碰撞概率极低。
     </li>
    </ul>
    <h4>
     <a id="_36">
     </a>
     <strong>
      优缺点
     </strong>
    </h4>
    <p>
     ✅
     <strong>
      优点
     </strong>
    </p>
    <ul>
     <li>
      全球唯一，无需中心化服务。
     </li>
     <li>
      直接由算法生成，无需网络通信。
     </li>
    </ul>
    <p>
     ❌
     <strong>
      缺点
     </strong>
    </p>
    <ul>
     <li>
      体积大（128 位），索引性能差，不适合数据库主键。
     </li>
     <li>
      UUID v4 无序，不利于数据库索引优化。
     </li>
    </ul>
    <h4>
     <a id="_45">
     </a>
     <strong>
      适用场景
     </strong>
    </h4>
    <p>
     适用于分布式系统中不需要严格递增的场景，如日志、Session ID、文件名等。
    </p>
    <hr/>
    <h3>
     <a id="3__ID_50">
     </a>
     <strong>
      3. 数据库自增 ID
     </strong>
    </h3>
    <h4>
     <a id="_51">
     </a>
     <strong>
      原理
     </strong>
    </h4>
    <p>
     利用 MySQL / PostgreSQL 的
     <code>
      AUTO_INCREMENT
     </code>
     或 Redis 的
     <code>
      INCR
     </code>
     生成递增 ID。
    </p>
    <h4>
     <a id="_54">
     </a>
     <strong>
      优缺点
     </strong>
    </h4>
    <p>
     ✅
     <strong>
      优点
     </strong>
    </p>
    <ul>
     <li>
      简单易用，易于管理。
     </li>
     <li>
      适合单机或小规模分布式系统。
     </li>
    </ul>
    <p>
     ❌
     <strong>
      缺点
     </strong>
    </p>
    <ul>
     <li>
      <strong>
       单点瓶颈
      </strong>
      ：数据库成为性能瓶颈。
     </li>
     <li>
      <strong>
       分布式扩展难
      </strong>
      ：需要主从同步或分区机制。
     </li>
     <li>
      <strong>
       不适合高并发场景
      </strong>
      ：锁竞争严重。
     </li>
    </ul>
    <h4>
     <a id="_64">
     </a>
     <strong>
      适用场景
     </strong>
    </h4>
    <p>
     适用于小规模系统，如企业内部应用、低并发业务。
    </p>
    <hr/>
    <h3>
     <a id="4_Segment_ID_69">
     </a>
     <strong>
      4. 号段模式（Segment ID）
     </strong>
    </h3>
    <h4>
     <a id="_70">
     </a>
     <strong>
      原理
     </strong>
    </h4>
    <ul>
     <li>
      采用数据库存储 ID 号段，每个应用实例预分配一批 ID，如：
      <ul>
       <li>
        <code>
         SELECT max_id, step FROM id_table WHERE biz_type = 'order'
        </code>
       </li>
       <li>
        <code>
         UPDATE id_table SET max_id = max_id + step
        </code>
       </li>
      </ul>
     </li>
     <li>
      业务服务器本地缓存一批 ID，避免频繁访问数据库。
     </li>
    </ul>
    <h4>
     <a id="_76">
     </a>
     <strong>
      优缺点
     </strong>
    </h4>
    <p>
     ✅
     <strong>
      优点
     </strong>
    </p>
    <ul>
     <li>
      解决数据库单点瓶颈，提升并发能力。
     </li>
     <li>
      支持分布式系统，不依赖特定存储方案。
     </li>
    </ul>
    <p>
     ❌
     <strong>
      缺点
     </strong>
    </p>
    <ul>
     <li>
      需要定期预分配，可能会有 ID 溢出或浪费。
     </li>
     <li>
      数据库依然是瓶颈，但比直接
      <code>
       AUTO_INCREMENT
      </code>
      性能好很多。
     </li>
    </ul>
    <h4>
     <a id="_85">
     </a>
     <strong>
      适用场景
     </strong>
    </h4>
    <p>
     适用于高并发的分布式业务，如订单系统、用户 ID 生成等。
    </p>
    <hr/>
    <h3>
     <a id="5_Redis__ID__90">
     </a>
     <strong>
      5. Redis 分布式 ID 生成
     </strong>
    </h3>
    <h4>
     <a id="_91">
     </a>
     <strong>
      原理
     </strong>
    </h4>
    <ul>
     <li>
      使用
      <code>
       INCR
      </code>
      或
      <code>
       INCRBY
      </code>
      递增 ID，如：
      <pre><code class="prism language-sh">INCR order_id
</code></pre>
     </li>
     <li>
      也可结合
      <code>
       SETEX
      </code>
      过期时间避免占用过多内存。
     </li>
    </ul>
    <h4>
     <a id="_98">
     </a>
     <strong>
      优缺点
     </strong>
    </h4>
    <p>
     ✅
     <strong>
      优点
     </strong>
    </p>
    <ul>
     <li>
      生成 ID 高性能（Redis 是内存数据库）。
     </li>
     <li>
      分布式部署，支持水平扩展。
     </li>
    </ul>
    <p>
     ❌
     <strong>
      缺点
     </strong>
    </p>
    <ul>
     <li>
      需要 Redis 高可用架构（如主从 + 哨兵）。
     </li>
     <li>
      可能有 ID 不连续的问题（节点故障）。
     </li>
    </ul>
    <h4>
     <a id="_107">
     </a>
     <strong>
      适用场景
     </strong>
    </h4>
    <p>
     适用于高并发场景，如电商订单号、短链服务等。
    </p>
    <hr/>
    <h3>
     <a id="6_Leaf_ID__112">
     </a>
     <strong>
      6. Leaf（美团分布式 ID 方案）
     </strong>
    </h3>
    <h4>
     <a id="_113">
     </a>
     <strong>
      原理
     </strong>
    </h4>
    <p>
     美团开源的
     <strong>
      Leaf
     </strong>
     提供两种模式：
    </p>
    <ol>
     <li>
      <strong>
       Leaf-Snowflake
      </strong>
      （类似 Twitter Snowflake）
     </li>
     <li>
      <strong>
       Leaf-Segment
      </strong>
      （基于号段模式）
     </li>
    </ol>
    <h4>
     <a id="_118">
     </a>
     <strong>
      优缺点
     </strong>
    </h4>
    <p>
     ✅
     <strong>
      优点
     </strong>
    </p>
    <ul>
     <li>
      高可用，支持分布式部署。
     </li>
     <li>
      号段模式减少数据库压力。
     </li>
    </ul>
    <p>
     ❌
     <strong>
      缺点
     </strong>
    </p>
    <ul>
     <li>
      依赖美团的开源库，维护成本较高。
     </li>
    </ul>
    <h4>
     <a id="_126">
     </a>
     <strong>
      适用场景
     </strong>
    </h4>
    <p>
     适用于大型互联网系统，如订单 ID、日志 ID。
    </p>

    <h3>
     <a id="_143">
     </a>
     <strong>
      推荐方案
     </strong>
    </h3>
    <ul>
     <li>
      <strong>
       高并发、分布式系统
      </strong>
      ：雪花算法（Snowflake）/ Leaf-Snowflake / Redis INCR
     </li>
     <li>
      <strong>
       数据库主键 ID 生成
      </strong>
      ：号段模式 / Leaf-Segment
     </li>
     <li>
      <strong>
       非结构化数据（如 Session、文件名）
      </strong>
      ：UUID v4
     </li>
     <li>
      <strong>
       小规模系统
      </strong>
      ：数据库自增 ID
     </li>
    </ul>
    <p>
     如果你的业务需要
     <strong>
      高并发、分布式、全局唯一 ID
     </strong>
     ，建议使用
     <strong>
      雪花算法（Snowflake）
     </strong>
     或
     <strong>
      Leaf
     </strong>
     ，同时做好时钟同步和节点 ID 规划。
    </p>
