    <h3>
     <a id="1__Zookeeper_1">
     </a>
     1. 什么是 Zookeeper？
    </h3>
    <p>
     Zookeeper 是 Apache 旗下的一个开源分布式协调框架，主要用于解决分布式系统中的一致性问题，提供高效可靠的分布式数据管理能力。
    </p>
    <h4>
     <a id="11_Zookeeper__5">
     </a>
     1.1 Zookeeper 的核心特性
    </h4>
    <ul>
     <li>
      <strong>
       顺序一致性
      </strong>
      ：客户端的更新请求按顺序执行。
     </li>
     <li>
      <strong>
       原子性
      </strong>
      ：更新操作要么成功要么失败，不存在中间状态。
     </li>
     <li>
      <strong>
       可靠性
      </strong>
      ：一旦数据写入 Zookeeper，它就不会丢失，除非主动删除。
     </li>
     <li>
      <strong>
       高可用性
      </strong>
      ：采用主从架构，保证服务的高可用性。
     </li>
     <li>
      <strong>
       观察者模式
      </strong>
      ：支持 Watch 机制，客户端可以监听数据变更。
     </li>
     <li>
      <strong>
       支持集群
      </strong>
      ：提供主备模式，保证分布式环境的稳定性。
     </li>
    </ul>
    <h4>
     <a id="12_Zookeeper__13">
     </a>
     1.2 Zookeeper 的典型应用场景
    </h4>
    <ul>
     <li>
      <strong>
       分布式配置管理
      </strong>
      ：存储和管理全局配置信息。
     </li>
     <li>
      <strong>
       分布式锁
      </strong>
      ：实现分布式环境下的互斥锁。
     </li>
     <li>
      <strong>
       服务注册与发现
      </strong>
      ：跟踪集群中的服务。
     </li>
     <li>
      <strong>
       Leader 选举
      </strong>
      ：在分布式系统中选出一个主节点。
     </li>
     <li>
      <strong>
       分布式队列
      </strong>
      ：管理任务调度，实现有序执行。
     </li>
     <li>
      <strong>
       分布式事务协调
      </strong>
      ：保证多个系统间数据一致性。
     </li>
    </ul>

    <h3>
     <a id="2__Zookeeper_23">
     </a>
     2. 安装与配置 Zookeeper
    </h3>
    <h4>
     <a id="21__25">
     </a>
     2.1 下载与安装
    </h4>
    <p>
     从
     <a href="https://zookeeper.apache.org/" rel="nofollow">
      Apache Zookeeper 官网
     </a>
     下载最新版 Zookeeper 并解压。
    </p>
    <pre><code class="prism language-bash"><span class="token function">wget</span> https://downloads.apache.org/zookeeper/stable/apache-zookeeper-3.8.0-bin.tar.gz
<span class="token function">tar</span> <span class="token parameter variable">-xzf</span> apache-zookeeper-3.8.0-bin.tar.gz
<span class="token builtin class-name">cd</span> apache-zookeeper-3.8.0-bin
</code></pre>
    <h4>
     <a id="22__Zookeeper__35">
     </a>
     2.2 配置 Zookeeper 集群
    </h4>
    <p>
     在
     <code>
      conf
     </code>
     目录下创建
     <code>
      zoo.cfg
     </code>
     配置文件，并添加以下内容：
    </p>
    <pre><code class="prism language-properties"># Zookeeper 配置文件
clientPort=2181
dataDir=/var/lib/zookeeper
tickTime=2000
initLimit=10
syncLimit=5
server.1=192.168.1.1:2888:3888
server.2=192.168.1.2:2888:3888
server.3=192.168.1.3:2888:3888
</code></pre>
    <p>
     在每个 Zookeeper 节点上创建
     <code>
      myid
     </code>
     文件，并写入唯一的服务器编号，例如：
    </p>
    <pre><code class="prism language-bash"><span class="token builtin class-name">echo</span> <span class="token string">"1"</span> <span class="token operator">&gt;</span> /var/lib/zookeeper/myid  <span class="token comment"># 服务器 1</span>
<span class="token builtin class-name">echo</span> <span class="token string">"2"</span> <span class="token operator">&gt;</span> /var/lib/zookeeper/myid  <span class="token comment"># 服务器 2</span>
<span class="token builtin class-name">echo</span> <span class="token string">"3"</span> <span class="token operator">&gt;</span> /var/lib/zookeeper/myid  <span class="token comment"># 服务器 3</span>
</code></pre>
    <h4>
     <a id="23__Zookeeper_59">
     </a>
     2.3 启动 Zookeeper
    </h4>
    <pre><code class="prism language-bash">bin/zkServer.sh start
</code></pre>
    <p>
     查看 Zookeeper 状态：
    </p>
    <pre><code class="prism language-bash">bin/zkServer.sh status
</code></pre>

    <h3>
     <a id="3_Zookeeper__73">
     </a>
     3. Zookeeper 的核心概念
    </h3>
    <h4>
     <a id="31_ZNode_75">
     </a>
     3.1 ZNode（数据节点）
    </h4>
    <p>
     Zookeeper 中的数据存储在 ZNode 中，类似于文件系统的目录结构。
    </p>
    <ul>
     <li>
      <strong>
       持久节点
      </strong>
      ：客户端断开连接后仍然存在。
     </li>
     <li>
      <strong>
       临时节点
      </strong>
      ：客户端断开连接后自动删除。
     </li>
     <li>
      <strong>
       顺序节点
      </strong>
      ：创建时自动添加序号，通常用于分布式队列。
     </li>
    </ul>
    <h4>
     <a id="32_Watcher__82">
     </a>
     3.2 Watcher 机制
    </h4>
    <p>
     Zookeeper 允许客户端对 ZNode 进行监听，一旦数据发生变化，客户端会收到通知。
    </p>
    <h5>
     <a id="_ZNode__85">
     </a>
     监听 ZNode 的变化
    </h5>
    <pre><code class="prism language-java"><span class="token class-name">ZooKeeper</span> zk <span class="token operator">=</span> <span class="token keyword">new</span> <span class="token class-name">ZooKeeper</span><span class="token punctuation">(</span><span class="token string">"localhost:2181"</span><span class="token punctuation">,</span> <span class="token number">3000</span><span class="token punctuation">,</span> event <span class="token operator">-&gt;</span> <span class="token punctuation">{<!-- --></span>
    <span class="token class-name">System</span><span class="token punctuation">.</span>out<span class="token punctuation">.</span><span class="token function">println</span><span class="token punctuation">(</span><span class="token string">"事件触发: "</span> <span class="token operator">+</span> event<span class="token punctuation">.</span><span class="token function">getType</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
<span class="token punctuation">}</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
zk<span class="token punctuation">.</span><span class="token function">exists</span><span class="token punctuation">(</span><span class="token string">"/myNode"</span><span class="token punctuation">,</span> <span class="token boolean">true</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <h4>
     <a id="33_Leader__93">
     </a>
     3.3 Leader 选举与选举机制
    </h4>
    <p>
     Zookeeper 采用
     <strong>
      ZAB 协议
     </strong>
     进行 Leader 选举，保证分布式系统的一致性。当集群启动时，所有节点会尝试竞选 Leader，投票由多数派决定，获得最多票数的节点成为 Leader。
    </p>
    <ul>
     <li>
      如果 Leader 故障，剩余 Follower 节点重新选举。
     </li>
     <li>
      选举过程需要保证超过半数的节点存活，Zookeeper 适用于奇数节点集群（例如 3、5、7 个节点）。
     </li>
    </ul>
    <hr/>
    <h3>
     <a id="4__102">
     </a>
     4. 集群可用性与故障处理
    </h3>
    <h4>
     <a id="41__104">
     </a>
     4.1 高可用性架构设计
    </h4>
    <ul>
     <li>
      部署
      <strong>
       奇数个节点
      </strong>
      （如 3、5、7）防止脑裂。
     </li>
     <li>
      采用
      <strong>
       多个数据中心部署
      </strong>
      ，保证跨机房容灾。
     </li>
     <li>
      监控 Leader 选举，防止长时间无 Leader 现象。
     </li>
    </ul>
    <h4>
     <a id="42__109">
     </a>
     4.2 处理节点故障
    </h4>
    <h5>
     <a id="_Leader__111">
     </a>
     发现 Leader 故障
    </h5>
    <p>
     当 Leader 故障时，Follower 节点会检测到无法连接 Leader，并触发重新选举过程。
    </p>
    <h5>
     <a id="_Follower__115">
     </a>
     处理 Follower 故障
    </h5>
    <p>
     如果某个 Follower 失效，不影响集群可用性，只要 Leader 存活且半数以上节点可用，Zookeeper 仍可正常运行。
    </p>
    <h5>
     <a id="_119">
     </a>
     监控与报警
    </h5>
    <p>
     使用
     <strong>
      Prometheus + Grafana
     </strong>
     或
     <strong>
      ZKMonitor
     </strong>
     进行监控：
    </p>
    <pre><code class="prism language-bash"><span class="token builtin class-name">echo</span> mntr <span class="token operator">|</span> <span class="token function">nc</span> <span class="token number">127.0</span>.0.1 <span class="token number">2181</span>
</code></pre>
    <p>
     如果返回
     <code>
      zk_server_state=leader
     </code>
     ，表示当前节点是 Leader，否则是 Follower。
    </p>

    <h3>
     <a id="5_Zookeeper__130">
     </a>
     5. Zookeeper 进阶功能
    </h3>
    <h4>
     <a id="51__132">
     </a>
     5.1 分布式锁
    </h4>
    <p>
     Zookeeper 通过创建
     <strong>
      临时顺序节点
     </strong>
     实现分布式锁。
    </p>
    <pre><code class="prism language-java"><span class="token class-name">InterProcessMutex</span> lock <span class="token operator">=</span> <span class="token keyword">new</span> <span class="token class-name">InterProcessMutex</span><span class="token punctuation">(</span>client<span class="token punctuation">,</span> <span class="token string">"/my-lock"</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
lock<span class="token punctuation">.</span><span class="token function">acquire</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
<span class="token keyword">try</span> <span class="token punctuation">{<!-- --></span>
    <span class="token comment">// 执行业务逻辑</span>
<span class="token punctuation">}</span> <span class="token keyword">finally</span> <span class="token punctuation">{<!-- --></span>
    lock<span class="token punctuation">.</span><span class="token function">release</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
    <p>
     这种方式能够避免多个客户端同时访问资源导致的冲突，确保只有一个客户端能获得锁，从而实现互斥访问。
    </p>
    <h4>
     <a id="52__Dubbo_147">
     </a>
     5.2 服务注册与发现（结合 Dubbo）
    </h4>
    <p>
     Zookeeper 在微服务架构中作为注册中心用于服务注册与发现。下面是一个简单的例子，结合
     <strong>
      Dubbo
     </strong>
     框架进行服务注册与发现：
    </p>
    <h5>
     <a id="Service_Provider_151">
     </a>
     服务提供者（Service Provider）
    </h5>
    <p>
     服务提供者通过 Dubbo 注册服务到 Zookeeper 中。
    </p>
    <pre><code class="prism language-xml"><span class="token tag"><span class="token tag"><span class="token punctuation">&lt;</span><span class="token namespace">dubbo:</span>registry</span> <span class="token attr-name">address</span><span class="token attr-value"><span class="token punctuation attr-equals">=</span><span class="token punctuation">"</span>zookeeper://localhost:2181<span class="token punctuation">"</span></span> <span class="token punctuation">/&gt;</span></span>

</code></pre>
    <h5>
     <a id="Service_Consumer_159">
     </a>
     服务消费者（Service Consumer）
    </h5>
    <p>
     服务消费者从 Zookeeper 中发现服务并调用。
    </p>
    <pre><code class="prism language-xml"><span class="token tag"><span class="token tag"><span class="token punctuation">&lt;</span><span class="token namespace">dubbo:</span>registry</span> <span class="token attr-name">address</span><span class="token attr-value"><span class="token punctuation attr-equals">=</span><span class="token punctuation">"</span>zookeeper://localhost:2181<span class="token punctuation">"</span></span> <span class="token punctuation">/&gt;</span></span>

</code></pre>
    <p>
     Zookeeper 作为 Dubbo 的注册中心，在服务启动时自动将服务信息（如地址、端口等）注册到 Zookeeper 中，消费者从 Zookeeper 中动态发现可用服务。
    </p>
    <h4>
     <a id="53__169">
     </a>
     5.3 作为服务注册中心使用
    </h4>
    <p>
     Zookeeper 作为服务注册中心，负责存储服务信息并提供高效的服务发现能力。服务实例（Provider）在启动时将服务信息注册到 Zookeeper 上，消费者（Consumer）则通过 Zookeeper 查找和调用可用的服务。
    </p>
    <p>
     Zookeeper 的 Watcher 机制能够在服务上线或下线时实时通知消费者，确保消费者获取到最新的服务状态。
    </p>
    <p>
     例如，服务注册的过程：
    </p>
    <ol>
     <li>
      服务启动时，客户端将服务信息（如
      <code>
       ip:port
      </code>
      ）写入 Zookeeper 中指定的目录。
     </li>
     <li>
      Zookeeper 会创建一个持久节点，存储服务信息。
     </li>
     <li>
      服务消费者通过 Watcher 监听 Zookeeper 上的服务信息，当服务变动时，消费者能够收到通知并更新本地服务列表。
     </li>
    </ol>

    <h3>
     <a id="6_Zookeeper__CAP_Zookeeper_vs_Eureka_182">
     </a>
     6. Zookeeper 与 CAP 理论对比：Zookeeper vs Eureka
    </h3>
    <p>
     <strong>
      CAP 理论
     </strong>
     （Consistency, Availability, Partition Tolerance）描述了在分布式系统中，如何平衡一致性（Consistency）、可用性（Availability）和分区容忍性（Partition Tolerance）。根据 CAP 理论，一个系统最多只能同时满足这三者中的两个。
    </p>
    <h4>
     <a id="ZookeeperCP__186">
     </a>
     Zookeeper：CP 类型
    </h4>
    <p>
     Zookeeper 优先保证
     <strong>
      一致性
     </strong>
     和
     <strong>
      分区容忍性
     </strong>
     。它通过 ZAB 协议保证集群中的数据一致性，并容忍网络分区的发生。在网络分区时，Zookeeper 会牺牲可用性，确保数据的一致性。
    </p>
    <ul>
     <li>
      <strong>
       一致性
      </strong>
      ：所有的节点保持一致的数据视图。
     </li>
     <li>
      <strong>
       分区容忍性
      </strong>
      ：即使发生网络分区，Zookeeper 依然可以保证数据一致性。
     </li>
    </ul>
    <p>
     适用场景：需要强一致性保证的场景，如分布式锁、配置管理、Leader 选举等。
    </p>
    <h4>
     <a id="EurekaAP__194">
     </a>
     Eureka：AP 类型
    </h4>
    <p>
     Eureka 优先保证
     <strong>
      可用性
     </strong>
     和
     <strong>
      分区容忍性
     </strong>
     。它允许系统在发生网络分区时依然能够继续工作，尽管此时可能存在短暂的数据不一致（比如服务的注册信息未能及时同步）。
    </p>
    <ul>
     <li>
      <strong>
       可用性
      </strong>
      ：即使某些 Eureka Server 节点无法连接，系统仍能正常提供服务。
     </li>
     <li>
      <strong>
       分区容忍性
      </strong>
      ：即使网络出现分区，系统依然能继续工作。
     </li>
    </ul>
    <p>
     适用场景：适用于微服务架构中的服务注册与发现，特别是在大规模分布式环境中需要高可用性的情况下。
    </p>
    <h4>
     <a id="_202">
     </a>
     总结：
    </h4>
    <ul>
     <li>
      <strong>
       Zookeeper
      </strong>
      适用于需要强一致性和高可靠性的场景，如分布式锁、Leader 选举等，遵循
      <strong>
       CP
      </strong>
      模型。
     </li>
     <li>
      <strong>
       Eureka
      </strong>
      适用于微服务架构中的服务注册与发现，强调高可用性和分区容忍性，遵循
      <strong>
       AP
      </strong>
      模型。
     </li>
    </ul>
