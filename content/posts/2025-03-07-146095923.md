    <h3>
     <a id="_1">
     </a>
     一、死锁的产生原因
    </h3>
    <p>
     死锁是多个线程（或进程）因竞争资源而陷入无限等待的状态，需同时满足以下
     <strong>
      四个必要条件
     </strong>
     ：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        互斥条件（Mutual Exclusion）
       </strong>
      </p>
      <ul>
       <li>
        资源一次只能被一个线程独占使用（如锁、文件句柄）。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        请求与保持（Hold and Wait）
       </strong>
      </p>
      <ul>
       <li>
        线程在持有至少一个资源的同时，请求其他线程占有的资源。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        不可剥夺（No Preemption）
       </strong>
      </p>
      <ul>
       <li>
        资源只能由持有者主动释放，不能被强制抢占。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        循环等待（Circular Wait）
       </strong>
      </p>
      <ul>
       <li>
        多个线程形成环形等待链，每个线程都在等待下一个线程释放资源。
       </li>
      </ul>
     </li>
    </ol>

    <h3>
     <a id="_53">
     </a>
     三、死锁的检测与诊断
    </h3>
    <h4>
     <a id="1__54">
     </a>
     1. 使用工具检测死锁
    </h4>
    <ul>
     <li>
      <strong>
       jstack
      </strong>
      （Java自带工具）：
      <pre><code class="prism language-bash">jstack <span class="token operator">&lt;</span>pid<span class="token operator">&gt;</span>  <span class="token comment"># 输出线程快照，显示死锁的线程及持有/等待的锁</span>
</code></pre>
     </li>
     <li>
      <strong>
       VisualVM
      </strong>
      或
      <strong>
       JConsole
      </strong>
      ：
      <br/>
      图形化界面查看线程状态，直接标记死锁。
     </li>
    </ul>
    <h4>
     <a id="2__62">
     </a>
     2. 日志分析
    </h4>
    <p>
     若日志中线程长时间处于
     <code>
      BLOCKED
     </code>
     状态且无进展，可能发生死锁。
    </p>
    <hr/>
    <h3>
     <a id="_67">
     </a>
     四、死锁的避免策略
    </h3>
    <h4>
     <a id="1__68">
     </a>
     1. 破坏“请求与保持”条件
    </h4>
    <ul>
     <li>
      <strong>
       一次性申请所有资源
      </strong>
      ：
      <br/>
      线程在开始执行前申请全部所需资源，否则不执行。
      <br/>
      <strong>
       缺点
      </strong>
      ：资源利用率低，可能导致饥饿。
     </li>
    </ul>
    <h4>
     <a id="2__73">
     </a>
     2. 破坏“不可剥夺”条件
    </h4>
    <ul>
     <li>
      <strong>
       允许抢占资源
      </strong>
      ：
      <br/>
      若线程请求资源失败，强制释放已持有的资源（需支持回滚操作）。
      <br/>
      <strong>
       缺点
      </strong>
      ：实现复杂，适用于特定场景（如数据库事务）。
     </li>
    </ul>
    <h4>
     <a id="3__78">
     </a>
     3. 破坏“循环等待”条件
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        资源有序分配法
       </strong>
       ：
       <br/>
       为所有资源类型定义全局顺序，线程按顺序申请资源。
       <br/>
       <strong>
        示例
       </strong>
       ：
       <br/>
       规定必须先申请
       <code>
        lockA
       </code>
       再申请
       <code>
        lockB
       </code>
       ，避免交叉申请。
      </p>

    <h4>
     <a id="4__99">
     </a>
     4. 使用超时机制
    </h4>
    <ul>
     <li>
      <strong>
       尝试获取锁时设置超时
      </strong>
      ：
      <br/>
      若在指定时间内未获得锁，放弃并释放已持有的资源，避免无限等待。
      <br/>
      <strong>
       Java实现
      </strong>
      （使用
      <code>
       ReentrantLock
      </code>

    <h4>
     <a id="5__117">
     </a>
     5. 减少锁的粒度
    </h4>
    <ul>
     <li>
      <strong>
       缩小同步范围
      </strong>
      ：
      <br/>
      仅对必要代码加锁，减少锁的持有时间。
     </li>
     <li>
      <strong>
       使用线程安全的数据结构
      </strong>
      ：
      <br/>
      如
      <code>
       ConcurrentHashMap
      </code>
      替代
      <code>
       synchronized
      </code>
      +
      <code>
       HashMap
      </code>
      。
     </li>
    </ul>

    <h3>
     <a id="_135">
     </a>
     六、总结
    </h3>
    <p>
     死锁的避免需结合业务场景选择合适的策略：
    </p>
    <ul>
     <li>
      <strong>
       关键系统
      </strong>
      （如金融交易）：优先使用资源有序分配和超时机制。
     </li>
     <li>
      <strong>
       高并发系统
      </strong>
      ：减少锁粒度，采用无锁数据结构。
     </li>
     <li>
      <strong>
       复杂事务
      </strong>
      ：结合事务管理和回滚机制。
     </li>
    </ul>
    <p>
     通过代码规范、工具检测和设计优化，可显著降低死锁发生概率。
