    <h5>
     <strong>
      一、缓存穿透（Cache Penetration）
     </strong>
    </h5>
    <h6>
     <strong>
      问题描述
     </strong>
    </h6>
    <p>
     请求
     <strong>
      数据库中不存在的数据
     </strong>
     （如非法ID），导致请求绕过缓存直接击穿到数据库。
     <br/>
     <strong>
      典型场景
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       恶意攻击：频繁请求
       <code>
        id=-1
       </code>
       或随机不存在的用户ID。
      </p>
     </li>
     <li>
      <p>
       业务逻辑缺陷：未校验参数合法性（如非数字ID查询）。
      </p>
     </li>
    </ul>
    <h6>
     <strong>
      解决方案
     </strong>
    </h6>
    <ul>
     <li>
      <p>
       <strong>
        空值缓存
       </strong>
      </p>
      将查询结果为
      <code>
       null
      </code>
      的请求也缓存，设置较短的过期时间（如 5分钟）。
     </li>
    </ul>
    <pre><code class="language-java">public Object getData(String key) {
    Object value = redis.get(key);
    if (value != null) {
        return value;  // 命中缓存
    }
    // 查数据库
    value = db.query(key);
    if (value == null) {
        redis.setex(key, 300, "NULL");  // 缓存空值
    } else {
        redis.setex(key, 3600, value);  // 缓存真实值
    }
    return value;
}</code></pre>
    <p>
     <strong>
      布隆过滤器（Bloom Filter）
     </strong>
    </p>
    <ul>
     <li>
      <p>
       在缓存层前加布隆过滤器，判断请求的 key 是否可能存在：
      </p>
      <ul>
       <li>
        <p>
         <strong>
          存在
         </strong>
         → 查缓存或数据库。
        </p>
       </li>
       <li>
        <p>
         <strong>
          不存在
         </strong>
         → 直接返回，避免查库。
        </p>
       </li>
      </ul>
     </li>
    </ul>
    <p>
     <strong>
      接口层校验
     </strong>
    </p>
    <ul>
     <li>
      对请求参数进行合法性校验（如ID范围、格式）。
     </li>
    </ul>
    <h5>
     <strong>
      二、缓存击穿（Cache Breakdown）
     </strong>
    </h5>
    <h6>
     <strong>
      问题描述
     </strong>
    </h6>
    <p>
     <strong>
      热点数据过期瞬间
     </strong>
     ，大量并发请求直接涌入数据库。
     <br/>
     <strong>
      典型场景
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       明星爆款商品详情页缓存过期。
      </p>
     </li>
     <li>
      <p>
       秒杀活动中核心商品信息缓存失效。
      </p>
     </li>
    </ul>
    <h6>
     <strong>
      解决方案
     </strong>
    </h6>
    <ol>
     <li>
      <p>
       <strong>
        互斥锁（分布式锁）
       </strong>
      </p>
      <ul>
       <li>
        <p>
         只允许一个线程重建缓存，其他线程等待。
        </p>
       </li>
      </ul>
     </li>
    </ol>
    <pre><code class="language-java">public Object getData(String key) {
    Object value = redis.get(key);
    if (value == null) {
        String lockKey = key + "_LOCK";
        if (redis.setnx(lockKey, "1", 10)) {  // 获取锁
            try {
                value = db.query(key);        // 查数据库
                redis.setex(key, 3600, value); 
            } finally {
                redis.del(lockKey);           // 释放锁
            }
        } else {
            Thread.sleep(100);                // 等待后重试
            return getData(key);              // 递归调用
        }
    }
    return value;
}</code></pre>
    <p>
     <strong>
      逻辑过期时间（永不过期 + 异步更新）
     </strong>
    </p>
    <ul>
     <li>
      <p>
       缓存不设置物理过期时间，但存储逻辑过期字段。
      </p>
     </li>
     <li>
      <p>
       后台线程定期检测并更新缓存。
      </p>
     </li>
    </ul>
    <pre><code class="language-java">// 缓存数据结构
class CacheData {
    Object data;
    long expireTime;  // 逻辑过期时间
}

public Object getData(String key) {
    CacheData cacheData = redis.get(key);
    if (cacheData == null) {
        return db.query(key);  // 首次加载
    }
    if (System.currentTimeMillis() &gt; cacheData.expireTime) {
        // 提交异步任务更新缓存
        executor.submit(() -&gt; {
            Object newData = db.query(key);
            redis.set(key, new CacheData(newData, System.currentTimeMillis() + 3600000));
        });
    }
    return cacheData.data;
}</code></pre>
    <p>
     <strong>
      热点数据永不过期
     </strong>
    </p>
    <ul>
     <li>
      <p>
       对极热点数据不设置过期时间，通过异步线程定期更新。
      </p>
     </li>
    </ul>
    <h5>
     <strong>
      三、缓存雪崩（Cache Avalanche）
     </strong>
    </h5>
    <h6>
     <strong>
      问题描述
     </strong>
    </h6>
    <p>
     <strong>
      大量缓存同时过期
     </strong>
     或
     <strong>
      缓存服务宕机
     </strong>
     ，导致请求全部涌入数据库。
     <br/>
     <strong>
      典型场景
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       缓存服务器重启后所有数据丢失。
      </p>
     </li>
     <li>
      <p>
       批量缓存设置相同过期时间（如每日零点统一过期）。
      </p>
     </li>
    </ul>
    <h6>
     <strong>
      解决方案
     </strong>
    </h6>
    <ul>
     <li>
      <p>
       <strong>
        随机过期时间
       </strong>
      </p>
     </li>
    </ul>
    <p>
     在基础过期时间上增加随机值（如
     <code>
      基础时间 + 随机0~300秒
     </code>
     ）。
    </p>
    <pre><code class="language-java">int baseExpire = 3600;
int randomExpire = baseExpire + new Random().nextInt(300);
redis.setex(key, randomExpire, value);</code></pre>
    <ul>
     <li>
      <p>
       <strong>
        集群高可用
       </strong>
      </p>
      <ul>
       <li>
        <p>
         使用 Redis 集群（Cluster）或哨兵模式（Sentinel）避免单点故障。
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        熔断降级
       </strong>
      </p>
      <ul>
       <li>
        <p>
         当数据库压力过大时，启用熔断机制（如返回默认值、限流）。
        </p>
       </li>
      </ul>
     </li>
    </ul>
    <pre><code class="language-java">// 使用 Hystrix 实现熔断
@HystrixCommand(fallbackMethod = "fallbackGetData")
public Object getData(String key) {
    return redis.get(key);
}

public Object fallbackGetData(String key) {
    return "系统繁忙，请稍后重试";  // 降级响应
}</code></pre>
    <ul>
     <li>
      <p>
       <strong>
        多级缓存
       </strong>
      </p>
      <ul>
       <li>
        <p>
         本地缓存（如 Caffeine） + Redis 缓存，降低 Redis 压力。
        </p>
       </li>
      </ul>
     </li>
    </ul>
