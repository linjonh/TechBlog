    <h3>
     <a id="1__0">
     </a>
     1. 引言
    </h3>
    <p>
     现代应用程序是为适应当前技术环境需求而设计的软件，采用现代开发工具和实践，针对云部署和可扩展性优化。它们由多个模块化小组件组成，便于集成和缩放，具有高度的敏捷性和适应性，能快速响应用户或业务需求变化。
    </p>
    <p>
     相比传统软件，现代应用程序的优势包括：
    </p>
    <ol>
     <li>
      <strong>
       灵活性与可扩展性
      </strong>
      ：模块化架构和云基础设施使其更易适应变化。
     </li>
     <li>
      <strong>
       弹性和可靠性
      </strong>
      ：设计上更健壮，能应对故障或高峰负载。
     </li>
     <li>
      <strong>
       成本效益
      </strong>
      ：构建和部署更快，开销更低。
     </li>
     <li>
      <strong>
       用户体验
      </strong>
      ：注重响应速度、可访问性和可用性，提供更优体验。
     </li>
    </ol>
    <p>
     下图是 AWS 为用户构建现代化应用所提供的主要的无服务类型的计算服务：
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/68e38bca822549a987d81bfa092a304e.png"/>

     现代应用程序通常使用敏捷开发方法论进行开发，这种方法强调迭代、协作和以用户为中心的软件开发方法。它们使用现代开发工具和框架，如 React、Angular 和 Vue.js，并经常使用
     <code>
      Docker
     </code>
     和
     <code>
      Kubernetes
     </code>
     等技术进行容器化。此外，现代应用程序设计时考虑到基于云的部署和扩展，使用基础设施即代码工具如
     <code>

     <code>
      AWS Lambda
     </code>

     <a id="2_Vercel__24">

     2. Vercel 公司和服务简介
    </h3>
    <p>
     Vercel 公司（
     <a href="http://vercel.com/" rel="nofollow">
      Vercel.com
     </a>
     ）提供为前端开发者设计的一体化平台，可以高效创建和部署 Web 应用程序。它通过自动配置构建设置和提供详细的构建日志，简化了部署管理过程。同时借助集成的 CI/CD 功能，开发者可以预览其更改，并无缝合并到主分支进行生产部署。Vercel 还支持自定义域名，可以轻松配置并为部署分配个性化的 URL。此外，Vercel 提供了检查、分析和使用情况等监控工具，以跟踪网站性能和用户参与度。借助 Vercel，开发者可以简化开发流程，实现快速、个性化的 Web 内容交付。
    </p>
    <p>
     Vercel 的整个的架构理念基于如下这幅图所示的 web 应用的演进：
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/0811a566a1be4bdb9c3f9b70f4ff86a0.png"/>
    </p>
    <p>
     传统基于三层架构的 web 应用（web 服务器+应用服务器+数据库），适合有经常性的动态内容生成和交互的场景。但如果一个网站绝大部分的内容都不经常更新，比如新闻网站、企业官网、小型电商， CMS（
     <code>
      Content Management System
     </code>
     ）网站，如果使用传统的三层架构，那无论如何都需要一个实时的在线服务，依旧需要大量后端和运维人员维护网站的安全性、稳定性、可伸缩性等。那是否有一种新的可能——通过某种架构，直接动态生成前端页面，并使用 CDN 持续分发静态页面以及数据，从而把内容的管理和内容的显示被完全分隔开来？
    </p>
    <h3>
     <a id="3__JAMStack_34">
     </a>
     3. 什么是 JAMStack
    </h3>
    <p>
     <a href="https://jamstack.org/" rel="nofollow">
      JAMStack
     </a>
     架构在这些场景下给出了新的解决方案：
    </p>
    <blockquote>
     <p>
      JAM 是
      <code>
       Javascript
      </code>
      、
      <code>
       APIs
      </code>
      、
      <code>
       Markup
      </code>
      三个术语的首字母组合。简单来说，
      <code>
       JAMStack
      </code>
      就是使用 SSG（Static Site Generators）技术，并且不依赖 Web 服务的前端技术栈。
     </p>
    </blockquote>
    <p>
     <strong>
      对于 SSR，Next.js 对每个请求在服务器上将页面预渲染为 HTML
     </strong>
     ，TTFB（首字节到达时间）较慢，但数据始终是最新的。
     <strong>
      对于 SSG，Next.js 在每个请求之前（例如在构建时）将页面预渲染为 HTML
     </strong>
     。HTML 可以由 CDN 进行全局缓存，并且可以立即提供服务。静态生成具有更高的性能，但由于预渲染是提前发生的，数据在请求时可能会变得过时。在 Next.js 中，过期数据的问题，可以通过在构建时间之后逐步添加和更新静态预渲染的页面的方式或者在不包含数据的情况下静态生成页面的部分内容，并由客户端获取更新数据的方式来解决。
    </p>
    <p>
     所以
     <code>
      JAMStack
     </code>
     并不是传统意义上的静态网站，事实上，
     <code>
      JAMStack
     </code>
     包含一个叫
     <code>
      Headless CMS
     </code>
     （无头内容管理系统）的后台系统。
     <code>
      Vercel
     </code>
     支持很多种流行的
     <code>
      Headless CMS
     </code>
     <a href="https://vercel.com/guides/using-a-headless-cms-with-vercel" rel="nofollow">
      系统集成
     </a>
     。
     <code>
      Headless CMS
     </code>
     提供与传统 CMS 相同的丰富协作和创作体验，同时将您的数据作为 API 开放出来。通过
     <code>
      Headless API
     </code>
     就可以将数据与表示层解耦。
    </p>
    <p>
     综上所述，基于
     <code>
      JAMStack
     </code>
     的网站通常还会有下面的特性：
    </p>
    <ul>
     <li>
      全站托管于 CDN 上
     </li>
     <li>
      原子化发布（每次发布都是一次全量、原子性的发布）
     </li>
     <li>
      灵活的文件缓存策略
     </li>
     <li>
      基于 Git 的全自动构建、部署流程
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/7f56c32fea784798bdf7263cfe899121.png"/>
    </p>
    <p>
     <code>
      Vercel
     </code>
     在此基础上，对
     <code>
      JAMStack
     </code>
     的理念进行了商业化的落地，结合 AWS 的
     <a href="https://aws.amazon.com/lambda/" rel="nofollow">
      AWS Lambda
     </a>
     ，
     <a href="https://aws.amazon.com/s3/" rel="nofollow">
      Amazon S3 – Cloud Object Storage
     </a>
     等服务，
     <code>
      Vercel
     </code>
     实现了对 Web 应用的云上无服务化部署。网站的开发人员，只需要专注于代码实现，每一次通过
     <code>
      Github
     </code>
     的代码变更，都会自动触发云上的构建，一个完整的流程包含了对多个云上服务的依次部署和调用。
    </p>
    <h3>
     <a id="4_Vercel__56">
     </a>
     4. Vercel 部署便利性背后的无服务架构
    </h3>
    <p>
     对于复杂的网站，在构建过程中，Vercel 会使用到 AWS S3 来存储大部分的静态内容，对于每一次的构建，Vercel 都会根据用户购买的服务配额，通过
     <a href="https://aws.amazon.com/sqs/" rel="nofollow">
      Amazon Simple Queue Service
     </a>
     服务进行构建任务的排队，同时使用
     <a href="https://aws.amazon.com/fargate/" rel="nofollow">
      AWS Fargate
     </a>
     来弹性生成基于容器的构建环境。
    </p>
    <p>
     构建的产出包含了
     <code>
      Serveless
     </code>
     函数（
     <a href="https://aws.amazon.com/lambda/" rel="nofollow">
      AWS Lambda
     </a>
     ）， Edge 函数，优化后的图像，以及静态存储等。
    </p>
    <p>
     以下就是 Vercel 从构建到发布一个现代化应用的完整过程：
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/37a98eecbc7143838e9003503d4c1e28.png"/>
    </p>
    <p>
     为了让构建出的环境能够被世界各地的访问者快速访问到，Vercel 利用
     <a href="https://aws.amazon.com/global-accelerator/" rel="nofollow">
      Amazon Global Accelerator
     </a>
     服务来进行基于域名的全球加速，使用 GA，不仅利用了 anycast 路由的优点，还可以让托管在 Vercel 上的应用程序获得自动故障转移和 DDoS 保护的好处，从而提高应用弹性和减少攻击影响。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/7fcb5c2a3c244b7daa9ef521a7f246c4.png"/>
    </p>
    <p>
     如上图所示，客户端的请求，会通过域名的加速服务，抵达
     <code>
      Vercel Edge
     </code>
     的某一个最近的部署位置。这是请求进入 Vercel 的
     <code>
      Kubernetes
     </code>
     集群（
     <a href="https://aws.amazon.com/eks/" rel="nofollow">
      Amazon EKS
     </a>
     ）的地方。该请求被检查并过滤以防恶意用户，然后路由到充当反向代理的虚拟机网关。首先，它根据数据（例如传入请求的主机名）确定应向用户提供部署的哪个版本，并获取该部署的元数据。然后，根据请求路径是否与部署元数据中指定的路由匹配，请求要么返回状态代码 404，要么继续生成响应。
    </p>
    <p>
     针对不同的请求和响应内容，网关会路由到不同的资源上进行处理。上述所有步骤的响应会根据缓存标头进行缓存，以加快将来的查找速度。
    </p>
    <p>
     借助 Vercel 这样的平台，您可以大大简化现代化应用的部署过程，无需自己设置和维护基础设施架构。Vercel 的部署方式，让你的应用天然具备全球性能和可访问性，确保无论用户位于何处都可以获得最佳的用户体验。
    </p>
    <p>
     构建在无服务基础上的 Vercel，每周会部署超过百万的
     <code>
      Lambda
     </code>
     函数，并被调用超过 50 亿次。Vercel 帮助
     <a href="https://www.washingtonpost.com/" rel="nofollow">
      华盛顿邮报（Washinton Post）
     </a>
     平稳度过总统竞选日，帮助
     <a href="https://www.netflix.com/jp-en/" rel="nofollow">
      奈飞（Netflix）
     </a>
     大幅度缩短前端的构建时间，更是帮助
     <a href="https://desenio.com/" rel="nofollow">
      Desenio
     </a>
     和
     <a href="https://www.harryrosen.com/en" rel="nofollow">
      Harry Rosen
     </a>
     这样的电商平台，在类似于国内双十一的促销日平稳地承接了突发的客户流量。
    </p>

    <p>

    <h5>

     5.2.1 传统无服务器
    </h5>
    <p>
     这些收益是这样实现的：请记住，在传统的无服务器模型中，每次调用都与单个函数实例绑定。

    <h5>
     <a id="522_Vercel__107">
     </a>
     5.2.2 Vercel 并发模式
    </h5>
    <p>
     Vercel 的新系统允许单个实例利用等待后端响应的空闲时间来处理多个调用。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/165f80d007ad40a78054066cfdc30503.png"/>
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/c83d8e8743414d15af350b1bd49d7d66.png"/>
    </p>
    <p>
     总结：
    </p>
    <p>
     例如，假设一个请求需要 100 毫秒，其中 50 毫秒用于计算，50 毫秒用于等待后端响应。
    </p>
    <ul>
     <li>
      <strong>
       旧模型
      </strong>
      ：两个请求总共需要 200ms 的计算时间
     </li>
     <li>
      <strong>
       新模型
      </strong>
      ：一次调用即可处理两个请求，将实际计算时间缩短至约 100 毫秒
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/fc859fe98fdd42698662a4b3c3717e9b.png"/>
    </p>
    <p>
     可以看到，同样是四个请求，fluid compute 模式仅需要3s 多一点，但是传统无服务model 下则需要 12s。
    </p>
    <h4>
     <a id="53_Fluid__129">
     </a>
     5.3 Fluid 的优势和特点：
    </h4>
    <p>
     <code>
      Fluid Compute
     </code>
     融合了无服务器灵活性和类似服务器的功能。与可能面临冷启动和
     <a href="https://vercel.com/docs/fundamentals/what-is-compute#serverless-disadvantages" rel="nofollow">
      功能有限
     </a>
     等问题的传统
     <a href="https://vercel.com/docs/fundamentals/what-is-compute#serverless" rel="nofollow">
      无服务器架构
     </a>
     不同，
     <code>
      Fluid Compute
     </code>
     提供了一种混合解决方案。它克服了无服务器和基于服务器的方法的局限性，发挥了两者的优势，包括：
     <a href="https://vercel.com/docs/fundamentals/what-is-compute#serverless-disadvantages" rel="nofollow">
     </a>
    </p>
    <ul>
     <li>
      <a href="https://vercel.com/docs/functions/fluid-compute#default-settings-by-plan" rel="nofollow">
       开箱即用零配置
      </a>
      ：
      <code>
       Fluid Compute
      </code>
      带有预设默认值，可自动优化您的功能，以提高性能和成本效益。
     </li>
     <li>
      <a href="https://vercel.com/docs/functions/fluid-compute#optimized-concurrency" rel="nofollow">
       优化并发性
      </a>
      ：通过处理单个函数实例中的多个调用来优化资源使用率。可与Node.js和Python运行时一起使用。
     </li>
     <li>
      动态扩展：
      <code>
       Fluid Compute
      </code>
      会自动优化现有资源，然后再进行扩展以满足流量需求。这可确保在高流量事件期间实现低延迟，并在较安静的时段实现成本效益。
     </li>
     <li>
      后台处理：在满足用户请求后，您可以继续使用执行后台任务
      <a href="https://vercel.com/docs/functions/vercel-functions-package#waituntil" rel="nofollow">
       <code>
        waitUntil
       </code>
      </a>
      。这可以在后台执行耗时的操作（如日志记录和分析）时提供响应迅速的用户体验。
     </li>
     <li>
      自动冷启动优化：通过
      <a href="https://vercel.com/docs/functions/fluid-compute#bytecode-caching" rel="nofollow">
       自动字节码优化
      </a>
      和生产部署中的功能预热来减少冷启动的影响。
     </li>
     <li>
      跨区域和可用区故障转移：如果一个可用区 (AZ) 发生故障，则首先故障转移到同一区域内的
      <a href="https://vercel.com/docs/functions/configuring-functions/region#automatic-failover" rel="nofollow">
       另一个可用区 (AZ)
      </a>
      ，从而确保高可用性。如果该区域内的所有区域都不可用，Vercel 会自动将流量重定向到下一个最近的区域。区域级故障转移也适用于非流动部署。
     </li>
    </ul>
    <h5>

    <h5>
     <a id="534__166">
     </a>
     5.3.4 优化并发性
    </h5>
    <p>
     <code>
      Fluid Compute
     </code>
     允许多个调用共享单个函数实例，这对于 AI 应用程序尤其有价值，因为其中诸如获取嵌入、查询矢量数据库或调用外部 API 之类的任务可能受I/O 限制。通过允许在同一实例内并发执行，您可以减少冷启动、最大限度地减少延迟并降低计算成本。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/7dd7f28d02184b9bafaf7d3410d7aac2.png"/>
    </p>
    <h5>
     <a id="535_httpsvercelcomdocsfunctionsfluidcomputebytecodecaching_173">
     </a>
     5.3.5
     <a href="https://vercel.com/docs/functions/fluid-compute#bytecode-caching" rel="nofollow">
      字节码缓存
     </a>
    </h5>
    <p>
     使用
     <a href="https://vercel.com/docs/functions/runtimes/node-js/node-js-versions" rel="nofollow">
      Node.js 版本 20+
     </a>
     时，Vercel 函数使用字节码缓存来减少冷启动时间。这会在 JavaScript 文件首次执行后存储其编译后的字节码，从而无需在后续冷启动期间重新编译。
    </p>
    <p>
     因此，第一个请求尚未缓存。但是，后续请求受益于缓存的字节码，从而实现更快的初始化。这种优化对于不经常调用的函数尤其有益，因为它们将实现更快的冷启动并减少最终用户的延迟。
    </p>
    <p>
     字节码缓存仅适用于生产环境，在开发或预览部署中不可用。
    </p>
    <blockquote>
     <p>
      对于输出 ESM 的
      <a href="https://vercel.com/docs/frameworks" rel="nofollow">
       框架
      </a>
      ，所有 CommonJS 依赖项（例如，、
      <code>
       react
      </code>
      ）都
      <code>
       node-fetch
      </code>
      将选择加入字节码缓存。
     </p>
    </blockquote>
    <h5>
     <a id="536_httpsvercelcomdocsfunctionsfluidcomputeisolationboundariesandglobalstate_183">
     </a>
     5.3.6
     <a href="https://vercel.com/docs/functions/fluid-compute#isolation-boundaries-and-global-state" rel="nofollow">
      隔离边界和全局状态
     </a>
    </h5>
    <p>
     在传统的无服务器计算中，隔离边界是指函数的各个实例之间的分离，以确保它们不会互相干扰。这为每个函数提供了一个安全的执行环境。
    </p>
    <p>
     但是，由于每个功能都使用 microVM 进行隔离，这可能会导致启动时间变慢，因此，当 microVM 处于不活动状态时，您会发现由于空闲时间导致资源使用量增加。
    </p>
    <p>
     流动计算使用不同的隔离方法。多个调用可以同时共享同一个物理实例（全局状态/进程），而不是为每个函数调用使用一个 microVM。这允许函数共享资源并在同一环境中执行，从而提高性能并降低成本。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/425bd3dc3ecb42189d4e3085041a1ea3.png"/>
    </p>

     <code>
      Fluid Compute
     </code>
     是
     <code>
      Vercel
     </code>
     的下一代计算模型，通过融合服务器和无服务器的优势，提供更快、更便宜、更灵活的函数执行方式，无需代码更改即可提升应用的性能和效率。
    </p>
