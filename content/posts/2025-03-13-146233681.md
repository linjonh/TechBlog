     nginx中忽略已.开头的文件
    </h1>
   </div>
  </div>
 </div>
 <article class="baidu_pl">
  <div class="article_content clearfix" id="article_content">
   <link href="../../assets/css/kdoc_html_views-1a98987dfd.css" rel="stylesheet"/>
   <link href="../../assets/css/ck_htmledit_views-704d5b9767.css" rel="stylesheet"/>
   <div class="markdown_views prism-atom-one-dark" id="content_views">
    <svg style="display: none;" xmlns="http://www.w3.org/2000/svg">
     <path d="M5,0 0,2.5 5,5z" id="raphael-marker-block" stroke-linecap="round" style="-webkit-tap-highlight-color: rgba(0, 0, 0, 0);">
     </path>
    </svg>
    <p>
     这个 Nginx 配置规则表示
     <strong>
      禁止访问以点（
      <code>
       .
      </code>
      ）开头的文件或目录
     </strong>
     ，并返回
     <code>
      404
     </code>
     错误。具体解释如下：
    </p>
    <pre><code class="prism language-nginx">location ~ /\. {
    deny all;
    return 404;
}
</code></pre>
    <h4>
     <a id="_9">
     </a>
     <strong>
      解释
     </strong>
    </h4>
    <ul>
     <li>
      <p>
       <code>
        location ~ /\. { ... }
       </code>
      </p>
      <ul>
       <li>
        <strong>
         <code>
          ~
         </code>
        </strong>
        ：表示正则表达式匹配。
       </li>
       <li>
        <strong>
         <code>
          \.
         </code>
        </strong>
        ：表示匹配文件或目录路径中的
        <strong>
         “.”
        </strong>
        （点），例如
        <code>
         .git
        </code>
        、
        <code>
         .env
        </code>
        、
        <code>
         .htaccess
        </code>
        等隐藏文件或目录。
       </li>
       <li>
        这意味着该规则会匹配任何路径中包含
        <strong>
         以点（
         <code>
          .
         </code>
         ）开头的文件或目录
        </strong>
        ，例如
        <code>
         .git/
        </code>
        、
        <code>
         .env
        </code>
        、
        <code>
         .htaccess
        </code>
        等。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <code>
        deny all;
       </code>
      </p>
      <ul>
       <li>
        <strong>
         <code>
          deny all;
         </code>
        </strong>
        ：拒绝所有用户访问匹配的文件或目录。即当访问匹配的路径时，会被拒绝，不能继续访问。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <code>
        return 404;
       </code>
      </p>
      <ul>
       <li>
        <strong>
         <code>
          return 404;
         </code>
        </strong>
        ：对于匹配到的路径，Nginx 会返回
        <strong>
         404 Not Found
        </strong>
        错误页面，而不是直接拒绝连接。
       </li>
       <li>
        这表示请求的文件或目录
        <strong>
         不存在
        </strong>
        （即返回 404 错误），而不是直接显示 403（禁止访问）。
       </li>
      </ul>
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="_24">
     </a>
     <strong>
      实际效果
     </strong>
    </h4>
    <p>
     假设你的服务器有以下隐藏文件或目录：
    </p>
    <ul>
     <li>
      <code>
       .git/
      </code>
     </li>
     <li>
      <code>
       .env
      </code>
     </li>
     <li>
      <code>
       .htaccess
      </code>
     </li>
    </ul>
    <p>
     当用户访问任何这些路径时：
    </p>
    <ul>
     <li>
      例如，访问
      <code>
       http://example.com/.git/
      </code>
      或
      <code>
       http://example.com/.env
      </code>
     </li>
     <li>
      Nginx 会拒绝访问，并返回
      <strong>
       404 错误
      </strong>
      ，表示文件或目录不存在。
     </li>
    </ul>
    <p>
     这种配置常用于
     <strong>
      安全防护
     </strong>
     ，避免泄露敏感的隐藏文件（如
     <code>
      .git
     </code>
     、
     <code>
      .env
     </code>
     、
     <code>
      .htaccess
     </code>
     等）。
    </p>
    <hr/>
    <h4>
     <a id="_38">
     </a>
     <strong>
      总结
     </strong>
    </h4>
    <p>
     这个
     <code>
      location
     </code>
     配置的作用是：
    </p>
    <ul>
     <li>
      <strong>
       拒绝访问以点（
       <code>
        .
       </code>
       ）开头的文件或目录
      </strong>
      ，例如
      <code>
       .git
      </code>
      、
      <code>
       .env
      </code>
      等。
     </li>
     <li>
      <strong>
       返回 404 错误
      </strong>
      ，而不是 403，来隐藏这些敏感资源。
     </li>
    </ul>
