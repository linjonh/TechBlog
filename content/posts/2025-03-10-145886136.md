    <h2>
     <a id="1__1">
     </a>
     1. 通信
    </h2>
    <ul>
     <li>
      <p>
       通信概念：两个或者多个进程实现
       <strong>
        数据
       </strong>
       层面的交互
       <br/>
       因为进程独立性的存在，导致进程通信的成本比较高
      </p>
     </li>
     <li>
      <p>
       <strong>
        进程间通信的本质
       </strong>
       ：必须让不同的进程看到同一份”资源”------特定形式的内存空间。
      </p>
     </li>
     <li>
      <p>
       这个”资源”谁提供？一般是
       <strong>
        操作系统
       </strong>
       ，
       <br/>
       为什么不是我们两个进程中的一个呢？假设一个进程提供，这个资源属于谁？这个进程独有，破坏进程独立性。
      </p>
     </li>
     <li>
      <p>
       “资源”从创建，使用，释放，都是使用系统调用接口！
      </p>
     </li>
     <li>
      <p>
       基于文件级别的通信方式--------管道
      </p>
     </li>
    </ul>
    <p>
     进程间通信目的：
     <br/>
     数据传输：一个进程需要将它的数据发送给另一个进程
     <br/>
     资源共享：多个进程之间共享同样的资源。
     <br/>
     通知事件：一个进程需要向另一个或一组进程发送消息，通知它（它们）发生了某种事件（如进程终止时要通知父进程）。
     <br/>
     进程控制：有些进程希望完全控制另一个进程的执行（如Debug进程），此时控制进程希望能够拦截另一个进程的所有陷入和异常，并能够及时知道它的状态改变。
    </p>

     2.2 匿名管道的有关代码编写
    </h3>
    <h4>
     <a id="221__36">
     </a>
     2.2.1 匿名管道代码编写
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/3fd9f908d2154a46b818421a05be400a.png">
      <br/>
      管道的特征：
     </img>
    </p>
    <ul>
     <li>
      具有血缘关系的进程进行进程间通信
     </li>
     <li>
      管道只能单向通信
     </li>
     <li>
      父子进程是会进程协同的，同步与互斥的——保护管道文件的数据安全
     </li>
     <li>
      管道是面向字节流的
     </li>
     <li>
      管道是基于文件的，而文件的生命周期是随
      <strong>
       进程
      </strong>
      的！
     </li>
    </ul>
    <p>
     代码gitee链接:
     <a href="https://gitee.com/stream_ocean/linux/tree/master/25-02-27" rel="nofollow">
      管道代码编写gitee链接
     </a>
    </p>
    <p>
     管道的4种情况：
     <br/>
     1．读写端正常，管道如果为空，读端就要阻塞
     <br/>
     2．读写端正常，管道如果被写满，写端就要阻塞
     <br/>
     3．读嘴正常读，写端关闭，读一段时间后，读端就会读到0，表明读到了文件（pipe)结尾，不会被阻塞
     <br/>
     4. 写端是正常写入，读端关闭了。操作系统就要杀掉正在写入的进程。如何干掉？
     <strong>
      通过信号杀掉
     </strong>
     。杀掉因为操作系统是不会做，低效，浪费等类似的工作的。
     <br/>
     通过13号信号-----
     <code>
      SIGPIPE
     </code>
     杀掉进程
    </p>

    <h3>
     <a id="32__63">
     </a>
     3.2 命名管道的原理
    </h3>
    <p>
     具有血缘关系的进程使用匿名管道，没有血缘关系的进程则使用命名管道。
     <br/>
     <mark>
      如果两个不同的进程打开同一个文件的时候，在内核中，操作系统只会打开一个文件。
     </mark>
     <br/>
     因为该管道文件不需要刷盘，所以直接创建内存级文件。
     <br/>
     如何打开同一个文件，文件的文件名+路径相同即可。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/36f901fc5dc246c8b9b85a21df40c44b.png"/>

     4.1 system V共享内存的原理
    </h3>
    <p>
     再次强调：进程间通信的原理是让进程看到同一份资源
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/5baed784dfbf4e9182625f7cd6e48337.png"/>

     如果要申请共享内存、释放共享内存、关联、去关联-----这些操作都是进程直接做的吗？不是。直接由操作系统来做。
     <br/>
     需求方—&gt; 系统调用 —&gt; 执行方
    </p>

    <h4>
     <a id="422_ftok_87">
     </a>
     4.2.2 ftok函数
    </h4>
    <p>
     作用：用pathname和proj_id确定一个key
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/0e0f8313250a4ca08db3e7c193605a64.png"/>
    </p>
    <ol>
     <li>
      key是一个数字，这个数字是几，不重要。关键在于它必须在内核中具有唯一性，能够让不同的进程进行唯一性标识
     </li>
     <li>
      第一个进程可以通过key创建共享内存，第二个之后的进程，只要拿着同一个key就可以和第一个进程看到同一个共享内存了！
     </li>
     <li>
      对于一个已经创建好的共享内存，key在哪？key在共享内存的描述对象中！
     </li>
     <li>
      该函数使
      <code>
       key
      </code>
      的矛盾冲突较小
     </li>
     <li>
      当路径不存在时，key就会返回-1；当key与系统中的共享内存相同时，就需要更改pathname或proj_id；
     </li>
    </ol>

    </h3>
