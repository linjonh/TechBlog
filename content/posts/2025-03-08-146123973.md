    <p>
     在 Spring Boot 项目中，带有
     <code>
      Query
     </code>
     后缀的查询对象通常属于
     <strong>
      请求参数
     </strong>
     或
     <strong>
      数据传输层
     </strong>
     ，推荐将其放置在以下位置之一：
    </p>
    <h3>
     <a id="_3">
     </a>
     <strong>
      推荐目录结构
     </strong>
    </h3>
    <pre><code class="prism language-plaintext">src
 └── main
     └── java
         └── com
             └── example
                 ├── controller
                 ├── dto            # ✅ 推荐，DTO/Query 对象统一管理
                 │   ├── request    # ✅ 查询对象 (UserQuery, OrderSearchRequest)
                 │   ├── response   # ✅ 响应对象 (UserDTO, OrderDetailDTO)
                 │   └── common     # 可选：通用DTO
                 ├── entity
                 ├── mapper
                 ├── service
                 └── util
</code></pre>
    <h3>
     <a id="_21">
     </a>
     <strong>
      最佳实践说明
     </strong>
    </h3>
    <p>
     ✅
     <code>
      dto/request
     </code>
     ➔ 放置查询参数对象 (
     <code>
      UserQuery
     </code>
     )，明确标识为请求数据
     <br/>
     ✅
     <code>
      dto/response
     </code>
     ➔ 放置响应数据对象 (
     <code>
      UserDTO
     </code>
     )，区分不同数据流向
     <br/>
     ✅
     <code>
      dto/common
     </code>
     ➔ 放置通用数据对象（如
     <code>
      PageInfoDTO
     </code>
     、
     <code>
      ResultDTO
     </code>
     等）
    </p>

    <hr/>
    <h3>
     <a id="__57">
     </a>
     🚨 为什么推荐这种结构？
    </h3>
    <p>
     ✅
     <strong>
      清晰的职责分离
     </strong>
     ：将查询、返回、通用数据分类，项目结构更直观。
     <br/>
     ✅
     <strong>
      易于维护
     </strong>
     ：项目扩展时，新增 DTO 类更有条理。
     <br/>
     ✅
     <strong>
      符合主流实践
     </strong>
     ：Spring Boot 项目中较为常见的目录规范。
    </p>
    <p>
     这种方法更利于项目的长期维护，尤其是当项目规模扩大时，目录清晰度会大幅提升。
    </p>
