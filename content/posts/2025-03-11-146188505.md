    <h2>
     <a id="BootloaderNAND_FlashSTM32_1">
     </a>
     设备树、Bootloader、NAND Flash与STM32启动流程详解
    </h2>
    <h3>
     <a id="_3">
     </a>
     引言
    </h3>
    <p>
     在嵌入式系统开发中，设备树(Device Tree)、Bootloader(引导加载程序)和NAND Flash是三个常见的概念，它们在不同的嵌入式平台中扮演着不同的角色。对于STM32微控制器，理解这些概念以及启动流程对于开发稳定可靠的应用至关重要。本文将详细介绍这些概念，并重点解析STM32微控制器的启动过程。
    </p>
    <h3>
     <a id="_7">
     </a>
     设备树详解
    </h3>
    <h4>
     <a id="_9">
     </a>
     基本概念与作用
    </h4>
    <p>
     设备树是一种描述硬件配置的数据结构，以树状结构表示系统中的硬件组件、它们的连接关系和属性。
    </p>
    <p>
     <strong>
      主要作用
     </strong>
     ：
    </p>
    <ul>
     <li>
      将硬件配置与软件代码分离，提高代码可移植性
     </li>
     <li>
      动态描述硬件资源，避免硬编码
     </li>
     <li>
      简化驱动开发，统一硬件描述方式
     </li>
     <li>
      支持运行时硬件配置的修改
     </li>
    </ul>
    <p>
     设备树通常以
     <code>
      .dts
     </code>
     (Device Tree Source)文件编写，经过编译后生成
     <code>
      .dtb
     </code>
     (Device Tree Blob)二进制文件，供操作系统或Bootloader读取。
    </p>
    <h4>
     <a id="STM32_21">
     </a>
     在STM32中的应用
    </h4>
    <p>
     <strong>
      对于普通的STM32项目，通常不需要使用设备树
     </strong>
     。设备树主要应用于以下场景：
    </p>
    <ul>
     <li>
      运行嵌入式Linux的复杂系统（如STM32MP1系列）
     </li>
     <li>
      使用高级RTOS且需要硬件抽象的复杂应用
     </li>
     <li>
      需要支持多种硬件配置的通用平台
     </li>
    </ul>
    <p>
     对于标准STM32应用，硬件配置通常直接在代码中定义，使用HAL库或寄存器直接操作硬件，无需设备树的抽象层。
    </p>
    <h3>
     <a id="Bootloader_31">
     </a>
     Bootloader详解
    </h3>
    <h4>
     <a id="_33">
     </a>
     基本概念与功能
    </h4>
    <p>
     Bootloader是上电后执行的第一段程序，负责初始化硬件并加载主应用程序。
    </p>
    <p>
     <strong>
      主要功能
     </strong>
     ：
    </p>
    <ul>
     <li>
      基础硬件初始化（时钟、内存等）
     </li>
     <li>
      检测启动模式和条件
     </li>
     <li>
      加载应用程序到RAM或直接从Flash执行
     </li>
     <li>
      提供程序更新机制
     </li>
     <li>
      执行安全检查和验证
     </li>
    </ul>
    <h4>
     <a id="_44">
     </a>
     类型与特点
    </h4>
    <p>
     Bootloader可分为多个层次：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        一级Bootloader
       </strong>
       ：通常固化在芯片ROM中，不可修改
      </p>
      <ul>
       <li>
        STM32的系统内存Bootloader就属于这类
       </li>
       <li>
        功能简单但高度可靠
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        二级Bootloader
       </strong>
       ：存储在Flash中，可更新
      </p>
      <ul>
       <li>
        提供更丰富的功能，如多应用选择、加密验证等
       </li>
       <li>
        可根据应用需求定制
       </li>
      </ul>
     </li>
    </ol>
    <h4>
     <a id="STM32_56">
     </a>
     在STM32中的应用
    </h4>
    <p>
     STM32微控制器内置了出厂Bootloader，存储在系统内存中，不可修改。此外，开发者还可以实现自定义Bootloader。
    </p>
    <p>
     <strong>
      STM32内置Bootloader特点
     </strong>
     ：
    </p>
    <ul>
     <li>
      支持通过UART、USB、CAN等接口更新固件
     </li>
     <li>
      通过BOOT引脚配置激活（通常BOOT0=1）
     </li>
     <li>
      支持读写Flash、SRAM和选项字节
     </li>
     <li>
      支持跳转执行应用程序
     </li>
    </ul>
    <p>
     <strong>
      自定义Bootloader应用场景
     </strong>
     ：
    </p>
    <ul>
     <li>
      实现固件加密和安全启动
     </li>
     <li>
      支持远程固件更新(OTA)
     </li>
     <li>
      实现多固件管理和回滚机制
     </li>
     <li>
      添加自定义通信协议
     </li>
    </ul>
    <h3>
     <a id="NAND_Flash_72">
     </a>
     NAND Flash详解
    </h3>
    <h4>
     <a id="_74">
     </a>
     基本概念与特点
    </h4>
    <p>
     NAND Flash是一种非易失性存储器，以块(Block)为单位擦除，以页(Page)为单位读写。
    </p>
    <p>
     <strong>
      主要特点
     </strong>
     ：
    </p>
    <ul>
     <li>
      高存储密度，适合大容量存储
     </li>
     <li>
      擦写次数有限（通常10,000-100,000次）
     </li>
     <li>
      需要坏块管理和错误检测与修正(ECC)
     </li>
     <li>
      不支持随机字节访问，需要按页读写
     </li>
    </ul>

    <h4>
     <a id="STM32_98">
     </a>
     在STM32中的应用场景
    </h4>
    <p>
     <strong>
      大多数STM32项目不需要使用NAND Flash
     </strong>
     ，通常使用：
    </p>
    <ul>
     <li>
      内部Flash：STM32自带NOR Flash，从几KB到2MB不等
     </li>
     <li>
      外部SPI NOR Flash：如W25Qxx系列，容量从512KB到16MB
     </li>
     <li>
      SD卡/eMMC：需要文件系统时的常用选择
     </li>
    </ul>
    <p>
     <strong>
      需要NAND Flash的场景
     </strong>
     ：
    </p>
    <ul>
     <li>
      需要大容量存储（数GB）的应用
     </li>
     <li>
      需要高速数据记录的场景
     </li>
     <li>
      成本敏感但需要大容量的产品
     </li>
    </ul>
    <p>
     使用NAND Flash需要额外的控制器或软件支持，处理ECC和坏块管理，增加了开发复杂度。
    </p>
    <h3>
     <a id="STM32_112">
     </a>
     STM32启动流程
    </h3>
    <p>
     STM32微控制器上电后的启动流程相对简单但灵活，下面详细介绍从上电到应用程序运行的完整过程。
    </p>
    <h4>
     <a id="1__116">
     </a>
     1. 上电复位阶段
    </h4>
    <p>
     当STM32上电或收到复位信号时：
    </p>
    <ul>
     <li>
      复位电路激活，将芯片重置到初始状态
     </li>
     <li>
      所有寄存器恢复默认值
     </li>
     <li>
      内部RC振荡器(HSI)自动启动作为系统时钟
     </li>
     <li>
      GPIO引脚设置为默认状态（通常为浮空输入）
     </li>
    </ul>
    <h4>
     <a id="2__124">
     </a>
     2. 启动模式选择
    </h4>
    <p>
     STM32根据BOOT引脚的状态选择启动模式：
    </p>
    <ul>
     <li>
      <p>
       <strong>
        主闪存启动
       </strong>
       ：BOOT0=0
      </p>
      <ul>
       <li>
        从内部Flash启动用户程序（最常用模式）
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        系统存储器启动
       </strong>
       ：BOOT0=1, BOOT1=0
      </p>
      <ul>
       <li>
        启动内置的出厂Bootloader
       </li>
       <li>
        用于通过串口、USB等接口下载程序
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        SRAM启动
       </strong>
       ：BOOT0=1, BOOT1=1
      </p>
      <ul>
       <li>
        从内部SRAM执行代码
       </li>
       <li>
        主要用于调试或特殊应用
       </li>
      </ul>
     </li>
    </ul>
    <h4>
     <a id="3__139">
     </a>
     3. 向量表加载
    </h4>
    <p>
     无论哪种启动模式，STM32都会：
    </p>
    <ul>
     <li>
      加载向量表（从0x00000000或重映射地址）
     </li>
     <li>
      获取初始堆栈指针值（向量表第一个条目）
     </li>
     <li>
      获取复位处理程序地址（向量表第二个条目）
     </li>
    </ul>
    <p>
     向量表结构：
    </p>
    <pre><code>地址 0x00000000: 初始堆栈指针值
地址 0x00000004: 复位处理程序地址
地址 0x00000008: NMI处理程序地址
地址 0x0000000C: 硬故障处理程序地址
...其他中断向量...
</code></pre>
    <h4>
     <a id="4__155">
     </a>
     4. 系统初始化
    </h4>
    <p>
     STM32执行复位处理程序，通常包括：
    </p>
    <ul>
     <li>
      调用
      <code>
       SystemInit()
      </code>
      函数配置时钟系统
     </li>
     <li>
      初始化C/C++运行环境
     </li>
     <li>
      跳转到
      <code>
       main()
      </code>
      函数开始执行用户代码
     </li>
    </ul>
    <p>
     启动文件中的典型代码（以ARM汇编为例）：
    </p>
    <pre><code class="prism language-assembly">; Reset handler
Reset_Handler   PROC
                EXPORT  Reset_Handler             [WEAK]
                IMPORT  SystemInit
                IMPORT  __main
                LDR     R0, =SystemInit   ; 加载SystemInit函数地址
                BLX     R0                ; 调用SystemInit函数
                LDR     R0, =__main       ; 加载__main函数地址
                BX      R0                ; 跳转到__main函数
                ENDP
</code></pre>
    <h4>
     <a id="5__176">
     </a>
     5. 应用程序执行
    </h4>
    <p>
     用户程序通常执行以下初始化步骤：
    </p>
    <ul>
     <li>
      配置系统时钟（如使用外部晶振、PLL倍频等）
     </li>
     <li>
      初始化所需外设（GPIO、UART、SPI等）
     </li>
     <li>
      初始化中断系统（优先级、使能等）
     </li>
     <li>
      初始化应用程序所需的其他组件
     </li>
     <li>
      进入主循环，开始执行应用逻辑
     </li>
    </ul>
    <p>
     典型的用户程序结构：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">int</span> <span class="token function">main</span><span class="token punctuation">(</span><span class="token keyword">void</span><span class="token punctuation">)</span>

    <h3>
     <a id="STM32_265">
     </a>
     不同复杂度STM32系统的启动需求
    </h3>
    <p>
     根据系统复杂度，STM32的启动需求可分为三类：
    </p>
    <h4>
     <a id="1__269">
     </a>
     1. 简单系统（如传感器节点、控制器）
    </h4>
    <p>
     <strong>
      典型配置
     </strong>
     ：
    </p>
    <ul>
     <li>
      使用内部Flash存储程序
     </li>
     <li>
      直接从主Flash启动
     </li>
     <li>
      无需自定义Bootloader
     </li>
     <li>
      无需设备树
     </li>
     <li>
      无需NAND Flash
     </li>
    </ul>
    <p>
     <strong>
      启动流程
     </strong>
     ：
    </p>
    <ol>
     <li>
      上电
     </li>
     <li>
      从内部Flash加载程序
     </li>
     <li>
      初始化系统
     </li>
     <li>
      执行应用程序
     </li>
    </ol>
    <h4>
     <a id="2__284">
     </a>
     2. 中等复杂度系统（如工业控制器、智能设备）
    </h4>
    <p>
     <strong>
      典型配置
     </strong>
     ：
    </p>
    <ul>
     <li>
      使用内部Flash + 外部SPI NOR Flash
     </li>
     <li>
      自定义简单Bootloader
     </li>
     <li>
      无需设备树
     </li>
     <li>
      可能使用外部SDRAM
     </li>
     <li>
      通常不使用NAND Flash
     </li>
    </ul>
    <p>
     <strong>
      启动流程
     </strong>
     ：
    </p>
    <ol>
     <li>
      上电
     </li>
     <li>
      执行内部Flash中的Bootloader
     </li>
     <li>
      初始化外部存储器
     </li>
     <li>
      加载并验证应用程序
     </li>
     <li>
      跳转到应用程序
     </li>
    </ol>
    <h4>
     <a id="3_STM32MP1Linux_300">
     </a>
     3. 高复杂度系统（如STM32MP1运行Linux）
    </h4>
    <p>
     <strong>
      典型配置
     </strong>
     ：
    </p>
    <ul>
     <li>
      多级Bootloader（如ROM、SPL、U-Boot）
     </li>
     <li>
      设备树描述硬件
     </li>
     <li>
      可能使用NAND Flash或eMMC
     </li>
     <li>
      外部DDR内存
     </li>
     <li>
      完整操作系统
     </li>
    </ul>
    <p>
     <strong>
      启动流程
     </strong>
     ：
    </p>
    <ol>
     <li>
      ROM Bootloader初始化基本硬件
     </li>
     <li>
      加载第一阶段Bootloader
     </li>
     <li>
      初始化内存控制器和关键外设
     </li>
     <li>
      加载主Bootloader
     </li>
     <li>
      加载设备树和内核
     </li>
     <li>
      启动操作系统
     </li>
     <li>
      运行应用程序
     </li>
    </ol>
    <h3>
     <a id="_318">
     </a>
     实际开发建议
    </h3>
    <h4>
     <a id="_320">
     </a>
     开发板上电准备
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        检查跳线配置
       </strong>
       ：
      </p>
      <ul>
       <li>
        确认BOOT0/BOOT1引脚设置正确
       </li>
       <li>
        通常开发时设置BOOT0=0（从主Flash启动）
       </li>
       <li>
        下载程序时可能需要设置BOOT0=1（进入Bootloader）
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        电源连接
       </strong>
       ：
      </p>
      <ul>
       <li>
        确保电源稳定且电压正确
       </li>
       <li>
        使用USB供电时，确保USB电缆质量良好
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        调试器连接
       </strong>
       ：
      </p>
      <ul>
       <li>
        如使用ST-Link/J-Link，确保正确连接
       </li>
       <li>
        检查驱动程序是否已安装
       </li>
      </ul>
     </li>
    </ol>

    <h3>
     <a id="_392">
     </a>
     总结
    </h3>
    <p>
     回到原问题：“如果要启动一个STM32板子，需要用到这些吗，上电之后的流程和过程是什么呢？”
    </p>
    <h4>
     <a id="_396">
     </a>
     是否需要这些技术？
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        设备树
       </strong>
       ：对于普通STM32项目，
       <strong>
        不需要
       </strong>
       。设备树主要用于运行Linux的复杂系统。
      </p>
     </li>
     <li>
      <p>
       <strong>
        Bootloader
       </strong>
       ：STM32已内置出厂Bootloader，所以基本功能已经具备，可以直接使用。但对于需要特殊功能（如OTA升级、安全启动）的项目，可能需要开发自定义Bootloader。
      </p>
     </li>
     <li>
      <p>
       <strong>
        NAND Flash
       </strong>
       ：大多数STM32项目
       <strong>
        不需要
       </strong>
       。通常使用内部Flash或外部SPI NOR Flash即可满足存储需求。
      </p>
     </li>
    </ol>
    <h4>
     <a id="STM32_404">
     </a>
     STM32上电后的流程
    </h4>
    <ol>
     <li>
      <strong>
       上电/复位
      </strong>
      ：芯片重置到初始状态
     </li>
     <li>
      <strong>
       启动模式选择
      </strong>
      ：根据BOOT0/BOOT1引脚状态选择启动模式
     </li>
     <li>
      <strong>
       向量表加载
      </strong>
      ：加载堆栈指针和复位处理程序地址
     </li>
     <li>
      <strong>
       系统初始化
      </strong>
      ：配置时钟、外设和中断系统
     </li>
     <li>
      <strong>
       应用程序执行
      </strong>
      ：跳转到main函数，执行用户代码
     </li>
    </ol>
    <p>
     对于大多数STM32应用，这个启动流程是自动完成的，开发者只需要关注应用程序的编写。只有在需要特殊功能（如安全启动、固件更新）时，才需要深入了解和定制启动流程。
    </p>
    <p>
     理解这些概念和启动流程，将有助于开发更可靠、更高效的STM32应用程序，特别是在调试启动问题或实现高级功能时。
    </p>
