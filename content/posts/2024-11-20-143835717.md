     <strong>
      防走弯路为防止走弯路，强烈建议先仔细阅读以下加粗内容：
     </strong>
     <br/>
     <font color="red">
      <b>
       如果你的应用是因为公司最近降成本调小实例物理内存才出现
       <code>
        docker oom
       </code>
       ，而之前从来没有出现过，那么大概率是堆内存太大导致，这种情况尤其在实例物理内存
       <code>
        &lt;=4G
       </code>
       的情况下多发。所以如果你的
       <code>
        case
       </code>
       满足以下场景，基本是堆内存设置不合理导致：
      </b>
     </font>
     <br/>
     <strong>
      【1】先看看你的应用
      <code>
       heap
      </code>
      实机用量，然后调小
      <code>
       xmx
      </code>
      到一个比较合理的数值，这里一定要自己定义xmx，不要用
      <code>
       ops
      </code>
      默认的计算值，因为那个计算值会把
      <code>
       xmx
      </code>
      设置为物理内存的
      <code>
       3/4
      </code>
      ，对于小内存来说，这太大了；
     </strong>
     <br/>
     <strong>
      【2】很多应用为了避免堆内存扩容导致性能下降，会将
      <code>
       xms
      </code>
      和
      <code>
       xms
      </code>
      设置成一样大，也会加剧小物理内存实例的
      <code>
       docker oom
      </code>
      。所以建议小内存实例把
      <code>
       xms
      </code>
      设置为物理内存的
      <code>
       1/2
      </code>
      或
      <code>
       1/3
      </code>
      ；
     </strong>
     <br/>
     <strong>
      【3】如果有的
      <code>
       group
      </code>
      有
      <code>
       oom
      </code>
      (比如
      <code>
       ALI
      </code>
      集群有
      <code>
       oom
      </code>
      )，有的一直没有，那么基本不是内存泄露导致，因为如果有内存泄露，那么任何集群都会出现
      <code>
       oom
      </code>
      。猜测这种情况因为实例底层参数差异导致(暂无法求证)，建议你把
      <code>
       oom
      </code>
      集群堆内存调小
      <code>
       256m
      </code>
      看看
      <code>
       oom
      </code>
      是否得到解决；
     </strong>
     <br/>
     <strong>
      【4】如果不满足上述两种情形，那就继续往下看吧；
     </strong>
    </p>
    <h3>
     <a id="_9">
     </a>
     一、前言
    </h3>
    <p>
     <code>
      Java
     </code>
     进程使用的内存分为
     <code>
      3
     </code>
     部分：堆内存、虚拟机所使用的内存(一般叫
     <code>
      Native Memory
     </code>
     )、堆外内存(
     <code>
      off-heap
     </code>
     )组成。
     <br/>
     【1】堆
     <code>
      heap
     </code>
     内存也就是你
     <code>
      jvm
     </code>
     参数里面设置的
     <code>
      xmx
     </code>
     和
     <code>
      xms
     </code>
     所指定的大小。如果你的工程里面的
     <code>
      extraenv.sh
     </code>
     没有指定
     <code>
      xms/xms
     </code>
     ，那么
     <code>
      ops
     </code>
     会默认给你指定成物理内存的
     <code>
      3/4
     </code>
     。比如物理内存
     <code>
      4G
     </code>
     ，那么堆内存会是
     <code>
      3072m
     </code>
     ，这其实有点太大了；
     <br/>
     【2】
     <code>
      Native memory
     </code>
     ：虚拟机使用的内存，分为很多细分的区域，比如
     <code>
      class
     </code>
     、
     <code>
      gc
     </code>
     、
     <code>
      thread
     </code>
     、
     <code>
      code
     </code>
     、
     <code>
      internal
     </code>
     等；
    </p>

       </td>

       </td>

       </td>

       </td>
      </tr>
      <tr>
       <td>

       </td>

    <p>
     服务器
     <code>
      RSS
     </code>
     过高会导致被拉出或者
     <code>
      OOM
     </code>
     ，虽然各个公司现有的高可用机制能自动检测并通过进程重启的方式恢复服务，但我们需要知道
     <code>
      RSS
     </code>
     过高是什么原因导致的。本文会梳理一下
     <code>
      RSS
     </code>
     过高的排查套路，避免你走弯路。
    </p>
    <p>
     首先，我们需要知道的是，
     <code>
      Java
     </code>
     进程消耗的内存绝不仅仅是你设置的
     <code>
      Xmx
     </code>
     这么简单，
     <code>
      Java
     </code>
     进程占用的内存分为
     <code>
      3
     </code>
     部分：堆内存、
     <code>
      JVM
     </code>
     自身消耗的内存、堆外内存。所以，后续的排查思路我们也是按照上面的
     <code>
      3
     </code>
     个方面来顺序展开。下图是各种
     <code>
      JVM
     </code>
     垃圾回收器消耗内存的比例，注意这部分内存是堆内存之外的：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ba01f6515d7b4c64a0c088c8464c313f.png"/>
    </p>
    <p>
     实践证明，
     <code>
      G1
     </code>
     内存开销甚至能占到堆内存的
     <code>
      20%
     </code>
     ，恐怖吧，惊喜不……
    </p>
    <p>
     1 确认堆内存是否太大
     <br/>
     首先要确认一下Java应用的堆内存是否太大，这个可以通过hickwall的appid docker linux tomcat多机模板 VM 查看。因为JVM自身也会消耗一些内存，所以你至少需要预留出部分内存存给JVM使用。如果应用涉及较多的网络通信，那还需要预留一些内存给堆外使用，所以一般来说你的堆内存最多为服务器物理内存减去1.2G，如4G服务器，那么堆内存最大为2.8G；(之前为3G，但就目前的趋势看，jvm自己用的内存也越来越多，所以这里调整成2.8G。当然这里是经验值，还是要结合你的应用所使用的堆内存决定)。
     <br/>
     如果确认你的堆内存使用过多的话，那么问题相对简单了，我们可以通过dump堆内存，使用mat分析基本能定位到问题。这方面的问题很多，大家可以自行搜索。
     <br/>
     2 确认是否DirectByteBuffer用量太大
     <br/>
     我们知道，很多有网络交互的组件底层是基于netty或者nio的，而nio为了提高性能，会申请一部分堆外内存，有的是基于DirectByteBuffer申请的，如果网络交互频繁而且报文尺寸大，这部分内存的占用还是比较可观的，比如下面这个应用，DirectByteBuffer用量都能达到700m。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/8632da4b5fdc41739cd62ccb293954db.png"/>
    </p>
    <p>
     需要注意的是：Java对于DirectByteBuffer的用量是没有做限制的，也就是它可以申请物理内存那么多的用量。所以如果你的DirectByteBuffer用量比较多，可以通过在jvm参数中设置-XX:MaxDirectMemorySize这个参数控制一下。在DirectByteBuffer用量超过你设置的大小时，系统会触发一次System.gc()(未必是FGC)。
     <br/>
     3 确认是否存在大量ARENA区
     <br/>
     如果堆内存不大，那么继续排查非堆内存。首先我们去看一下ARENA区。在高并发的应用中，往往ARENA区占用的内存会比较多。至于ARENA是什么东东，可以读一下这篇文章：https://blog.csdn.net/maokelong95/article/details/51989081
     <br/>
     执行如下命令：
    </p>
    <pre><code class="prism language-java">sudo <span class="token operator">-</span>u deploy pmap <span class="token operator">-</span>x <span class="token generics"><span class="token punctuation">&lt;</span>pid<span class="token punctuation">&gt;</span></span><span class="token operator">|</span>sort <span class="token operator">-</span>gr <span class="token operator">-</span>k2 <span class="token operator">|</span>less 
</code></pre>
    <p>
     如果存在大量大小为
     <code>
      65536
     </code>
     或
     <code>
      65404
     </code>
     的内存区域，则说明
     <code>
      ARENA
     </code>
     区域占用了太多的内存，如下图所示：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/eb558a5b525a40d1bab2ca40e5d7a7a9.png"/>
    </p>
    <p>
     这种情况下，有2个选择：内存分配器换成jemalloc，jemalloc 服务粗暴的办法是修改环境变量的值，在extraenv.sh中添加这么一行：export MALLOC_ARENA_MAX=1需要注意的是，上述的数值只能是1，其他大于1的数值经实践证明是无法控制ARENA数量的。4 确认Native Memory开销过大如果前面2个步骤看到的内存开销都不大，还有很多内存区域你不知道消耗在哪里了，那么我们开始第三步：开启Native Memory Tracking。前面我们说过，Java应用的执行，JVM自身也需要消耗一些内存的，通过开启Native Memory Tracking，我们就能知道JVM自身消耗了多少内存。请不要迷信vi里面的非堆内存消耗，那是不准确的……书归正传，通过修改/opt/tomcat/bin/setenv.sh中的JVM参数并重启Java进程开启：
    </p>
    <pre><code class="prism language-java"><span class="token operator">-</span>XX<span class="token operator">:</span><span class="token class-name">NativeMemoryTracking</span><span class="token operator">=</span>detail 
</code></pre>
    <p>
     进程重启后，可以通过以下命令查看NativeMemory的占用情况：
    </p>
    <pre><code class="prism language-java">sudo <span class="token operator">-</span><span class="token class-name">Esu</span> deploy <span class="token operator">/</span>usr<span class="token operator">/</span>java<span class="token operator">/</span><span class="token keyword">default</span><span class="token operator">/</span>bin<span class="token operator">/</span>jcmd <span class="token generics"><span class="token punctuation">&lt;</span>pid<span class="token punctuation">&gt;</span></span> VM<span class="token punctuation">.</span>native_memory summary 
</code></pre>
    <p>
     你会看到以下类似的执行结果：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/918727df7f264c02844b5459b13110b6.png"/>
    </p>
    <p>
     通过上图，你可以看到JVM各个区域所使用的内存大小，主要包含了Java Heap、Class、Thread、Code、GC、Compiler、Internal、Other、Symbol、Native Memory Tracking、Arena Chunk这几部分。
     <br/>
     需要注意的是Class、Thread、GC几个区域的大小。
     <br/>
     上面的图片是27G的堆内存，使用G1回收器，你能看到GC区居然占用了3.8G的内存。
     <br/>
     针对上面的各个区域大小做加法，看一下是否接近于RSS的大小，如果是，恭喜你可以到此结束了。后续你需要做的就是针对内存占用比较大的JVM区去做优化，这里就不详细介绍了。
     <br/>
     如果不是，很遗憾，你进入到了最难啃的环节，继续往下看吧。
    </p>
    <p>
     需要注意的是，开启NativeMemoryTracking会造成5%的性能下降，用完记得关闭。
     <br/>
     可以通过以下命令临时关闭：
    </p>
    <pre><code class="prism language-java">sudo <span class="token operator">-</span><span class="token class-name">Esu</span> deploy <span class="token operator">/</span>usr<span class="token operator">/</span>java<span class="token operator">/</span><span class="token keyword">default</span><span class="token operator">/</span>bin<span class="token operator">/</span>jcmd <span class="token generics"><span class="token punctuation">&lt;</span>pid<span class="token punctuation">&gt;</span></span> VM<span class="token punctuation">.</span>native_memory stop 
</code></pre>
    <p>
     或者修改/opt/tomcat/bin/setenv.sh中的JVM参数并重启永久关闭。
    </p>
    <p>
     如果你嫌上面的文章安装配置比较麻烦，你也可以用我的jtoolkit去排查，安装命令很简单：
    </p>
    <pre><code class="prism language-java">curl <span class="token operator">-</span><span class="token class-name">L</span> http<span class="token operator">:</span><span class="token operator">/</span><span class="token operator">/</span>devresources<span class="token punctuation">.</span>flight<span class="token punctuation">.</span>ctripcorp<span class="token punctuation">.</span>com<span class="token operator">/</span>jtoolkit<span class="token operator">/</span>shells<span class="token operator">/</span>install<span class="token punctuation">.</span>sh<span class="token operator">|</span>bash 
</code></pre>
    <p>
     安装完毕，根据提示去使用就可以了，希望通过上面的介绍，能解决你的问题。
    </p>
    <p>
     如果很不幸你的问题还没有得到解决的话，那么抱歉，这超出了我的认知了。
    </p>
