    <h4>
     <a id="Copilot_Edits_38">
     </a>
     Copilot Edits
    </h4>
    <p>
     上个月，我们在 VS Code Insiders 中引入了 Copilot Edits 的 Agent 模式。在 Agent 模式下，Copilot 可以自动在你的工作区中搜索相关上下文，编辑文件，检查错误，并在获得你的许可后运行终端命令，从而实现端到端的任务自动化。
    </p>
    <blockquote>
     <p>
      <strong>
       注意
      </strong>
      ：代理模式目前可在
      <a href="https://code.visualstudio.com/insiders/" rel="nofollow">
       VS Code Insiders
      </a>
      版中使用，并已开始逐步推广至 VS Code Stable。一旦 Agent 模式对你可用，你将在 Copilot Edits 视图中看到模式下拉菜单——只需选择 Agent 即可启用。
     </p>
    </blockquote>
    <p>
     本月，我们对 UX 进行了多项改进：
    </p>
    <ul>
     <li>
      <p>
       终端命令现在会内联显示，方便你跟踪已执行的命令。
      </p>
     </li>
     <li>
      <p>
       你可以在运行命令前，先在聊天回复中编辑建议的终端命令。
      </p>
     </li>
     <li>
      <p>
       通过快捷键 ⌘Enter 确认终端命令的执行。
      </p>
     </li>
    </ul>
    <p>
    </p>

    <p>
     代理模式会自主搜索你的代码库，以查找相关上下文。展开消息即可查看执行的搜索结果。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/08c14dd3edb047cebaae097708566ff2.png">
      <br/>

     </img>

    <p>
     现在，聊天中的 Undo 和 Redo 操作会撤销或重做聊天回复中对文件的最后一次编辑。这对于 Agent 模式非常实用，因为你可以仅撤销模型执行的特定步骤，而无需回滚整个聊天回复。
    </p>
    <p>
     Agent 模式现在可以自动运行你的构建
     <a href="https://code.visualstudio.com/docs/editor/tasks" rel="nofollow">
      任务
     </a>
     ，或者在收到指令时执行。如发现模型在不应运行任务时执行了任务，你可以通过
     <code>
      github.copilot.chat.agent.runTasks
     </code>
     设置来禁用此功能。
    </p>
    <p>
     了解更多有关
     <a href="https://chatgpt.com/c/67cd3891-6c70-800c-8f85-dfc928c7d091#" rel="nofollow">
      Copilot Edits Agent 模式

     <a href="https://chatgpt.com/c/67cd3891-6c70-800c-8f85-dfc928c7d091#" rel="nofollow">
      Agent 模式公告博客
     </a>
     。
    </p>

     <p>
      <strong>
       注意
      </strong>
      ：如果你是 Copilot Business 或 Enterprise 用户，你的组织管理员
      <a href="https://docs.github.com/en/copilot/managing-copilot/managing-github-copilot-in-your-organization/managing-policies-for-copilot-in-your-organization#enabling-copilot-features-in-your-organization" rel="nofollow">
       必须
      </a>
      启用 Copilot“编辑器预览功能”，Agent 模式才能使用。

    <h5>
     <a id="Copilot_Edits__Notebook__73">
     </a>
     Copilot Edits 的 Notebook 支持（预览版）
    </h5>
    <p>
     我们在 VS Code Insiders 中引入了 Copilot Edits 的 Notebook 支持，作为一项预览功能。现在，你可以像编辑 notebook 文件一样直观地编辑 Notebook 文件。创建新笔记本，修改多个单元格的内容，插入和删除单元格，更改单元格类型。这项预览功能可为 数据科学 或 文档编写 提供流畅的工作流程。
    </p>
    <blockquote>
     <p>
      <strong>
       注意
      </strong>
      ：此功能目前仅在
      <a href="https://code.visualstudio.com/insiders/" rel="nofollow">
       VS Code Insiders
      </a>
      中提供，并需要使用 GitHub Copilot Chat 预发布版本。我们将继续优化体验，并计划在未来的稳定版本中正式推出。
     </p>
    </blockquote>

    <h5>
     <a id="_82">
     </a>
     优化的编辑器集成
    </h5>
    <p>
     我们对 Copilot Edits 在代码编辑器和 notebook 编辑器中的集成进行了优化：
    </p>
    <p>
     应用更改时不再自动滚动，视图保持原位，让你更专注于具体的更改内容。
    </p>
    <p>
     调整编辑审核操作名称：“Accept”→ “Keep”，“Discard”→ “Undo”，这样更能准确反映实际操作。Copilot Edits 的更改是实时应用并保存的，用户可以 选择保留或撤销这些更改。
    </p>
    <p>
     在处理完一个文件后，系统会自动显示下一个文件，提高编辑效率。
    </p>
    <p>
     以下视频演示了更改是如何实时应用并保存的。预览会动态更新，用户可以选择 “Keep” 保留更改，同时仍然可以撤销或进一步调整。

    <h5>
     <a id="_UI_99">

     焕新的 UI
    </h5>
    <p>
     为了将 Copilot Edits 与 Copilot Chat 统一，我们对 Copilot Edits 进行了界面升级。待发送的附件文件现在会以常规聊天附件的形式显示。只有被 AI 修改过的文件才会出现在更改文件列表中，该列表位于聊天输入框上方。

    <p>
     此外，你可以通过 chat.renderRelatedFiles 设置启用相关文件建议。启用后，相关文件建议会显示在聊天附件的下方。

    <h5>
     <a id="_Copilot_Edits__106">
     </a>
     移除 Copilot Edits 的限制
    </h5>
    <p>
     此前，在 Copilot Edits 中，最多只能向提示中附加 10 个文件。在本次更新中，我们移除了这一限制。此外，我们还移除了客户端每 10 分钟最多 14 次交互的速率限制。
    </p>
    <blockquote>
     <p>
      请注意，服务器端的使用速率限制仍然适用。
     </p>
    </blockquote>
    <h4>
     <a id="_111">
     </a>
     自定义指令正式推出
    </h4>
    <p>
     <strong>
      设置
     </strong>
     ：
     <code>
      github.copilot.chat.codeGeneration.useInstructionFiles
     </code>
    </p>
    <p>
     自定义指令允许你调整 GitHub Copilot，使其聊天回复和代码建议更符合你和团队的工作方式。你可以在工作区内的
     <code>
      .github/copilot-instructions.md
     </code>
     文件中，使用 Markdown 格式描述你的具体需求。
    </p>
    <p>
     在本次更新中，我们正式推出了基于
     <code>
      .github/copilot-instructions.md
     </code>
     的 自定义指令功能。请确保启用了
     <code>
      github.copilot.chat.codeGeneration.useInstructionFiles
     </code>
     设置，Copilot 将根据这些指令生成响应。
    </p>
    <p>
     了解更多关于
     <a href="https://chatgpt.com/c/67cd3891-6c70-800c-8f85-dfc928c7d091#" rel="nofollow">
      Copilot 自定义指令
     </a>
     。
    </p>
    <h4>
     <a id="_126">
     </a>
     更顺畅的聊天身份验证流程
    </h4>
    <p>
     如果你的源代码托管在 GitHub 仓库，你可以使用高级代码搜索、
     <code>
      @github
     </code>
     聊天参与者等功能！
    </p>
    <p>
     不过，对于私有 GitHub 仓库，VS Code 需要获得权限才能与 GitHub 交互。此前，VS Code 会通过弹出对话框进行身份验证，例如使用
     <code>
      @workspace
     </code>
     或
     <code>
      @github
     </code>
     提问，或使用
     <code>
      #codebase
     </code>
     工具。
    </p>
    <p>
     为了提供更顺畅的体验，我们在聊天界面内引入了确认机制：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/881b439279174f6bb6c40fcda44efdd5.png">
      <br/>
      它不仅不像弹出对话框那样突兀，而且还具有新功能：
     </img>
    </p>
    <ul>
     <li>
      <p>
       <strong>
        授权（Grant）
       </strong>
       ：你将按照之前的方式通过常规身份验证流程（通过弹出对话框）进行身份验证。
      </p>
     </li>
     <li>
      <p>
       <strong>
        稍后（Not Now）
       </strong>
       ：VS Code 会记住你的选择，在下次 VS Code 窗口会话之前不会再次打扰你。唯一的例外是，如果某个功能需要额外的权限才能运行（例如
       <code>
        @github
       </code>
       ），它仍会请求权限。
      </p>
     </li>
     <li>
      <p>
       <strong>
        不再询问（Never Ask Again）
       </strong>
       ：VS Code 会记住你的选择，并通过
       <code>
        github.copilot.advanced.authPermissions
       </code>
       设置将其持久化。任何需要此额外权限的功能将会失败。
      </p>
     </li>
    </ul>
    <p>
     需要注意的是，这个确认框并不代表确认或拒绝 Copilot（服务）访问你的仓库。这仅仅是 VS Code 中 Copilot 体验的身份验证方式。要配置 Copilot 可以访问的内容，请阅读关于
     <a href="https://docs.github.com/en/copilot/managing-copilot/configuring-and-auditing-content-exclusion/excluding-content-from-github-copilot" rel="nofollow">
      内容排除
     </a>
     的文档。
    </p>
    <h4>
     <a id="Copilot_Chat__149">
     </a>
     Copilot Chat 中的更高级代码库搜索
    </h4>
    <p>
     <strong>
      设置
     </strong>
     ：
     <code>
      github.copilot.chat.codesearch.enabled
     </code>
    </p>
    <p>
     当你在 Copilot Chat 查询中添加
     <code>
      #codebase
     </code>
     时，Copilot 会帮助你查找与聊天提示相关的代码，来自你工作区中的相关内容。现在，
     <code>
      #codebase
     </code>
     可以运行工具，如文本搜索和文件搜索，以从你的工作区提取更多上下文。
    </p>
    <p>
     请设置
     <code>
      github.copilot.chat.codesearch.enabled
     </code>
     来启用此功能。可用的工具列表包括：
    </p>
    <ul>
     <li>
      <p>
       基于嵌入的语义搜索
      </p>
     </li>
     <li>
      <p>
       文本搜索
      </p>
     </li>
     <li>
      <p>
       文件搜索
      </p>
     </li>
     <li>
      <p>
       Git 修改过的文件
      </p>
     </li>
     <li>
      <p>
       项目结构
      </p>
     </li>
     <li>
      <p>
       读取文件
      </p>
     </li>
     <li>
      <p>
       读取路径
      </p>
     </li>
     <li>
      <p>
       工作区符号搜索
      </p>
     </li>
    </ul>
    <h4>
     <a id="_176">
     </a>
     将文件夹附加为上下文
    </h4>
    <p>
     以前，你可以通过从 Explorer 视图拖放来将文件夹作为上下文附加。现在，你还可以通过选择 📎 图标或在聊天提示中输入
     <code>
      #folder:
     </code>
     后跟文件夹名称来附加文件夹。
    </p>
    <h4>
     <a id="Next_Edit_Suggestions_179">
     </a>
     Next Edit Suggestions（预览版）的折叠模式
    </h4>
    <p>
     <strong>
      设置
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       <code>
        github.copilot.nextEditSuggestions.enabled
       </code>
      </p>
     </li>
     <li>
      <p>
       <code>
        editor.inlineSuggest.edits.showCollapsed
       </code>
      </p>
     </li>
    </ul>
    <p>
     我们为 NES（Next Edit Suggestions）添加了折叠模式。启用此模式后，左侧编辑器边缘仅显示 NES 建议指示器。只有当按
     <strong>
      Tab
     </strong>
     键导航到该建议时，代码建议才会被显示。连续的建议会立即显示，直到某个建议未被接受为止。
    </p>

    </p>
    <p>

    <h4>
     <a id="_432">
     </a>
     内联建议检测
    </h4>
    <p>
     <strong>
      设置
     </strong>
     ：
     <code>
      terminal.integrated.suggest.inlineSuggestion
     </code>
    </p>
    <p>
     到目前为止，内联建议检测遇到的一个问题是，由于来自不同来源的建议相互竞争，导致了混淆。特别是在终端中输入时，通常会出现内联建议：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/762a7687628f42339e14fe54d77e65e6.png"/>
     <br/>
     这些建议实际上是 shell 层级的功能（如 fish/zsh 中的自动建议、pwsh 中的预测等），对用户来说可能不太明显，尤其是当它们与 IntelliSense 一起呈现时。
    </p>
    <p>
     IntelliSense 功能要求我们检测到这些内联建议，之前我们使用的简单实现只检查文本是否使用了淡色或斜体的 SGR 样式属性。事实证明，这种方法不够，特别是当用户自定义了样式时，fish shell 默认情况下也不会使用这些样式。我们现在通过分析命令行上下文和光标位置来检测大多数情况。
    </p>
    <p>
     基于这种新的改进检测，内联建议现在作为顶部选项显示，并带有星形图标，这样可以更好地与编辑器的行为对齐，并且使用户更加清楚按下 Tab 键时将会发生什么。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/d63e47d04c0948d2b50dbe2b4bae9310.png"/>
     <br/>
     默认情况下，始终将此建议显示为顶部建议，但可以通过
     <code>
      terminal.integrated.suggest.inlineSuggestion
     </code>
     进行配置。
    </p>

     身份验证
    </h5>
    <blockquote>
     <p>
      <strong>
       重要
      </strong>
      ：我们正在将
      <code>
       AuthenticationForceNewSessionOptions
      </code>
      重命名为
      <code>
       AuthenticationGetSessionPresentationOptions
      </code>
      ，并暂时保留已弃用的
      <code>
       AuthenticationForceNewSessionOptions
      </code>
      。这两者在功能上没有区别，因此这不是一个破坏性更改，但你应该更新代码，改用
      <code>
       AuthenticationGetSessionPresentationOptions
      </code>
      ，因为它将来会被移除。
     </p>
    </blockquote>

    </p>

    <p>
     <code>
      createIfNone
     </code>
     和
     <code>
      forceNewSession
     </code>
     现在可以接受布尔值或
     <code>
      AuthenticationGetSessionPresentationOptions
     </code>

    </p>
    <pre><code class="prism language-bash">/**
 * Optional options to be used when calling <span class="token punctuation">{<!-- --></span>@link authentication.getSession<span class="token punctuation">}</span> with interactive options <span class="token variable"><span class="token variable">`</span>forceNewSession<span class="token variable">`</span></span> <span class="token operator">&amp;</span> <span class="token variable"><span class="token variable">`</span>createIfNone<span class="token variable">`</span></span><span class="token builtin class-name">.</span>

<span class="token builtin class-name">export</span> interface AuthenticationGetSessionPresentationOptions <span class="token punctuation">{<!-- --></span>
  /**
   * An optional message that will be displayed to the user when we ask to re-authenticate. Providing additional context

  detail?: string<span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>

     <a href="https://github.com/microsoft/vscode/blob/release/1.98/src/vscode-dts/vscode.d.ts#L17520-L17551">

     </a>

    </p>
    <p>
     这是
     <code>
      createIfNone
     </code>
     的新添加，但对于
     <code>
      forceNewSession
     </code>
     来说，这是一个修改，原本它接受的是
     <code>
      AuthenticationForceNewSessionOptions
     </code>

     <code>

     <code>

    <h4>
     <a id="_658">
     </a>
     开发时追踪泄漏的可释放资源
    </h4>
    <p>
     VS Code 使用可释放模式进行显式资源管理，例如关闭文件、清理 DOM 元素或移除事件监听器。如果不释放资源，将导致内存浪费，并且内存使用量会随着时间积累。
    </p>
    <p>
     我们一直在寻找此类泄漏，并新增了一个检测工具。我们使用 FinalizationRegistry API，它可以告知我们何时某个对象被垃圾回收。如果该对象是一个未被释放的可释放资源，这就意味着我们存在内存泄漏。我们将这些信息收集并展示给 VS Code 开发人员，以便在开发过程中进行清理。
    </p>
