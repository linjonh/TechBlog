     <p style="text-align:left;">
      golang官方指南
      <sup>
       [1]
      </sup>
      给了一些代码片段，层层递进演示了信道的能力:
     </p>
     <p style="text-align:left;">
      1&gt;. 信号量
      <br/>
      2&gt;. 限流能力
     </p>
     <pre class="has"><code class="language-go">var sem = make(chan int, MaxOutstanding) 
 
func Serve(queue chan *Request) {
    for req := range queue {
        req:= req
        sem &lt;- 1   
        go func() {   // 只会开启MaxOutstanding个并发协程
            process(req)
            &lt;-sem
        }()
    }
}</code></pre>
     <p style="text-align:left;">
      上面出现了两个信道：
     </p>
     <p style="text-align:left;">
      ①
      <code>
       sem
      </code>
      提供了限制服务端并发处理请求的信号量
      <br/>
      ②
      <code>
       queue
      </code>
      提供了一个客户端请求队列，起媒介/解耦的作用
     </p>
     <hr/>
     <p style="text-align:left;">
      进一步指南给出了信道的另一个用法： 3&gt;.
      <strong>
       解多路复用
      </strong>
     </p>
     <p style="text-align:left;">
      多路复用是网络编程中一个耳熟能详的概念，nginx redis等高性能web、内存kv都用到了这个技术 。
     </p>
     <p style="text-align:left;">
      这个解多路复用是怎么理解呢？
     </p>
     <p style="text-align:left;">
      离散/独立/并发的客户端请求被服务端Serve收敛之后， Serve就起到了多路复用的概念，在Request定义resultChan信道，就给每个客户端请求提供了独立获取请求结果的能力，这便是一种解多路复用。
     </p>
     <hr/>
     <p style="text-align:left;">
      从实际效果看这就是常见的互联网web服务器：
      <strong>
       一款具备请求排队功能的并发限流服务器
      </strong>
      。
     </p>
     <img alt="f09bb9cbd0a0799b91ce51dfd7c16b23.png" src="https://i-blog.csdnimg.cn/img_convert/f09bb9cbd0a0799b91ce51dfd7c16b23.png">
      <p style="text-align:left;">
       官方指南并没有完整实现客户端和服务器端工程。
      </p>
      <p style="text-align:left;">
       下面是我的工程化实现， 记录实践中遇到的问题。
      </p>
      <h3>
       并发受限服务器
      </h3>
      <ul>
       <li>
        <p style="text-align:left;">
         <code>
          信道queue
         </code>
         接收客户端请求,解耦客户端和服务器，天然具备排队能力
        </p>
       </li>
       <li>
        <p style="text-align:left;">
         <code>
          信号量信道sem
         </code>
         提供了并发受限的能力
        </p>
       </li>
       <li>
        <p style="text-align:left;">
         服务器处理完，向
         <code>
          解多路复用信道req.resultChan
         </code>
         写入响应结果。
        </p>
       </li>
      </ul>
      <pre class="has"><code class="language-go">/* 实现一个有请求队列功能的并发请求受限服务器*/

package main

import (
 "fmt"
 "sync"
 "time"
)

var sem = make(chan int, Maxoutstanding)

var wg2 sync.WaitGroup

func server(queue chan *Request) {
 fmt.Printf("Server is already, listen req \n")

 for req := range queue {
 req := req
 sem &lt;- 1

 wg2.Add(1)
 go func() {
 defer wg2.Done()
 process(req)
 &lt;-sem
 }()
 }
}

func process(req *Request) {
 s := sum(req.args)
 req.resultChan &lt;- s
}
func sum(a []int) (s int) {
 for i := 1; i &lt;= a[0]; i++ {
 s += i
 }
 time.Sleep(time.Millisecond * 20)
 return s
}</code></pre>
      <blockquote>
       <p style="text-align:left;">
        time.Sleep模拟服务器处理请求单次耗时20ms， 输出数字的累加，
        <br/>
        eg： input： 100；
        <br/>
        output: (1+100)/2*100 =5050
       </p>
      </blockquote>
      <p style="text-align:left;">
       <code>
        wg2 sync.WaitGroup
       </code>
       是一个动态活跃的Goroutine计数器，注意用法和位置，wg2的作用是：等待所有请求处理完成。
      </p>
      <h3>
       并发客户端请求
      </h3>
      <p style="text-align:left;">
       for循环开启并发客户端请求，
      </p>
      <ul>
       <li>
        <p style="text-align:left;">
         每个请求入驻一个独立的Goroutine，独立向
         <code>
          信道queue
         </code>
         投递请求和接收响应
        </p>
       </li>
      </ul>
      <pre class="has"><code class="language-go">package main

import (
 "fmt"
 "sync"
)

type Request struct {
 args       []int
 resultChan chan int
}

var wg1 sync.WaitGroup

func clients() {
 fmt.Printf("start %d concurrency client request\n ", concurrencyClients)
 for i := 1; i &lt;= concurrencyClients; i++ {
 r := &amp;Request{
 args:       []int{i},
 resultChan: make(chan int),
 }
 wg1.Add(1)
 go ClientReq(r)
 }
 wg1.Wait() 

}

func ClientReq(r *Request) {
 defer wg1.Done()
 queue &lt;- r
 go func() {
 res := &lt;-r.resultChan
 fmt.Printf("current args is %d, the result is %d \n", r.args[0], res)
 }()
}</code></pre>
      <p style="text-align:left;">
       <code>
        wg1 WaitGroup
       </code>
       的目的是确保所有的客户端请求都已经发出，之后客户端任务结束，所以此处我们新开Goroutine处理响应结果（这里又有闭包的参与）。
      </p>
      <h3>
       工程化
      </h3>
      <p style="text-align:left;">
       工程化代码的先后顺序，决定了代码是否死锁。
       <br/>
       server需要处于监听状态，故先启动。
      </p>
      <blockquote>
       <p style="text-align:left;">
        本处clients在主协程整体上是同步发送，如果放在
        <code>
         clients()
        </code>
        的后面，clients内的wg1可能会有部分请求Goroutine阻塞在信道queue, 且没法唤醒, 运行时会检测出报死锁。
       </p>
      </blockquote>
      <pre class="has"><code class="language-go">package main

import (
 "fmt"
 "time"
)

var concurrencyClients = 1000
var queueLength = 100
var queue = make(chan *Request, queueLength) // 请求队列长度
var Maxoutstanding int = 10                  // 服务器并发受限10

func main() {

 go server(queue)
 var start = time.Now()

 clients() // 确保所有的请求都已经发出去

 wg2.Wait() // 确保服务器处理完所有的请求
 fmt.Printf("客户端并发%d请求，服务器请求队列长度%d，服务器限流%d，总共耗时%d ms \n", concurrencyClients, queueLength, Maxoutstanding, time.Since(start).Milliseconds())
}</code></pre>
      <p style="text-align:left;">
       上面出现了3个配置变量
       <br/>
       1&gt;.  客户端并发请求数量
       <code>
        concurrencyClients=100
       </code>
       <br/>
       2&gt;.  服务器排队队列长度
       <code>
        queueLength=100
       </code>
       ， 会作用到
       <code>
        信道queue
       </code>
       <br/>
       3&gt;.  服务器并发受限阈值
       <code>
        Maxoutstanding=10
       </code>
      </p>
