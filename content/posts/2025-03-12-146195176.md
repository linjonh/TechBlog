     二、DHCP协议
    </h2>
    <p>
     <a href="https://www.bilibili.com/video/BV1Gd4y1n7Xz/?spm_id_from=333.337.search-card.all.click&amp;vd_source=c6d25b8c148a15b1c45b8da5d613a042" rel="nofollow">

    <h3>
     <a id="1_8">
     </a>
     1、理解
    </h3>
    <p>
     DHCP 的全称是 Dynamic Host Configuration Protocol
     <strong>
      动态主机配置协议
     </strong>
     。DHCP与IP地址紧密联系。
     <br/>
     <strong>
      作用
     </strong>
     ：自动配置IP地址、统一管理IP地址分配。
    </p>
    <blockquote>
     <p>
      注意：
      <br/>
      除了IP地址，DHCP服务器还会把
      <strong>
       子网掩码，默认路由，网关信息，DNS服务器
      </strong>
      告诉你。
     </p>
    </blockquote>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/f09a96a24a9344b8957bce3faeacdce7.png"/>
    </p>
    <h3>
     <a id="2DHCP_15">
     </a>
     2、DHCP服务器
    </h3>
    <blockquote>
     <p>
      动态IP地址最终的去处。
     </p>
    </blockquote>
    <ul>
     <li>
      由于IP地址是
      <strong>
       动态的（临时分配）
       <strong>
        而不是
       </strong>
       静态的（永久分配）
      </strong>
      ，因此
      <strong>
       不再使用的IP地址会自动返回IP地址池中进行重新分配
      </strong>
      。
     </li>
    </ul>
    <blockquote>
     <p>
      那么DHCP服务器由谁服务呢？
     </p>
    </blockquote>
    <ul>
     <li>
      <strong>
       网络管理员负责建立DHCP服务器
      </strong>
      ，并以租约的形式向启用DHCP的 客户端提供地址配置。
     </li>
     <li>
      DHCP会为每个客户端提供一个动态地址，到了截止时期，会自动收回。
     </li>
    </ul>
    <blockquote>
     <p>
      当虚拟机的静态地址是192.168.1.9与手机的IP地址一样。虚拟机接入网络时，会连不上。IP地址发生冲突。
      <br/>
      那么DHCP如何知道哪些IP地址是空闲的呢？
     </p>
    </blockquote>
    <ul>
     <li>
      这就需要数据库了。通过DHCP维护的信息，来合理分配IP地址。那么
      <strong>
       DHCP维护了哪些信息
      </strong>
      ？
      <ul>
       <li>
        <strong>
         有效的TCP/IP配置参数
        </strong>
        <ul>
         <li>
          主机名（Host name）、DHCP 客户端（DHCP client）、域名（Domain name）、IP 地址(IPaddress）、网关（Netmask）、广播地址（Broadcast address）、默认路由（default rooter）。
         </li>
        </ul>
       </li>
       <li>
        <strong>
         有效的 IP 地址和排除的 IP 地址
        </strong>
        ，保存在 IP 地址池中等待分配给客户端。
       </li>
       <li>
        为某些特定的 DHCP 客户端保留的地址，这些地址是
        <strong>
         静态 IP
        </strong>
        ，这样可以将单个 IP 地址⼀致地分配给单个DHCP 客户端。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="3DHCP_34">
     </a>
     3、DHCP工作机制
    </h3>
    <p>
     DHCP报文有哪些？
    </p>
    <ul>
     <li>
      DHCP discover ：客户端广播寻找可用DHCP服务器。
     </li>
     <li>
      DHCP offer ：服务器接收到 DHCPDISCOVER 之后
      <strong>
       做出的响应
      </strong>
      ，它
      <strong>
       包括了给予客户端的 IP 租约过期时间、服务器的识别符以及其他信息
      </strong>
      。【
      <strong>
       单播或广播
      </strong>
      】
     </li>
     <li>
      DHCP request ：客户端确认接受服务器提供的IP地址。【
      <strong>
       广播
      </strong>
      】
     </li>
     <li>
      DHCP ACK ：服务器确认分配（Ack）。【
      <strong>
       单播或广播
      </strong>
      】
     </li>
     <li>
      DHCP NAK ：服务器拒绝了客户端的请求。【
      <strong>
       单播或广播
      </strong>
      】
     </li>
     <li>
      DHCP RELEASE ：一般出现在客户端关机、下线等状况。这个报文将会使 DHCP 服务器
      <strong>
       释放发出此报文的客户端的 IP 地址
      </strong>
      。
     </li>
     <li>
      DHCP INFORM ：客户端发出的向服务器
      <strong>
       请求⼀些信息的报文
      </strong>
      。
     </li>
     <li>
      DHCP DECLINE :当客户端发现服务器
      <strong>
       分配的 IP 地址无法使用
      </strong>
      （如 IP 地址冲突时），将发出此报文，通知服务器
      <strong>
       禁止使用该 IP 地址
      </strong>
      。
     </li>
    </ul>
    <h2>
     <a id="HTTP_2030_45">
     </a>
     三、HTTP 2.0/3.0的理解
    </h2>
    <p>
     <a href="https://www.bilibili.com/video/BV1vv4y1U77y/?spm_id_from=333.337.search-card.all.click&amp;vd_source=c6d25b8c148a15b1c45b8da5d613a042" rel="nofollow">
      学习前可以看这个视频，理解整个过程，可以一截一截的看，里面是有标记内容位置的。
     </a>
    </p>
    <h3>
     <a id="1HTTP_47">
     </a>
     1、最初的HTTP
    </h3>
    <p>
     <strong>
      最初的HTTP只用于Web端的内容获取
     </strong>
     。web端那时候是很简单的界面。但是时代的发展，界面变得越来越好看，需要的技术文件越来越多。就会导致发送数据的阻碍。
    </p>
    <h4>
     <a id="11_49">
     </a>
     1.1、影响
    </h4>
    <p>
     <strong>
      影响HTTP的网络请求因素
     </strong>
     主要有：
     <strong>
      带宽和延迟
     </strong>
     。
    </p>
    <blockquote>
     <ul>
      <li>
       <h3>
        <a id="_51">
        </a>
        带宽
       </h3>
       <ul>
        <li>
         因为
         <strong>
          单位时间传输的数据很少
         </strong>
         。如果按以前的老年机来输入信息，那么是无法满足现代聊天的需求。
        </li>
       </ul>
      </li>
     </ul>
    </blockquote>
    <blockquote>
     <ul>
      <li>
       <h3>
        <a id="_55">
        </a>
        延迟
       </h3>
       <ul>
        <li>
         <h3>
          <a id="_57">
          </a>
          浏览器阻塞
         </h3>
         <ul>
          <li>
           浏览器阻塞是指浏览器在执行某些操作时被阻止，导致页面加载或功能无法正常运行。
          </li>
         </ul>
        </li>
        <li>
         <h3>
          <a id="DNS_60">
          </a>
          DNS查询
         </h3>
         <ul>
          <li>
           览器需要知道目标服务器的 IP 才能建立连接。将域名解析为 IP 的这个系统就是 DNS。但是这个通常可以利用 DNS 缓存结果来达到减少这个时间的目的。
          </li>
         </ul>
        </li>
        <li>
         <h3>
          <a id="_63">
          </a>
          建立连接
         </h3>
         <ul>
          <li>
           HTTP 是基于 TCP 协议的，浏览器最快也要在第三次握手时才能捎带 HTTP 请求报文，达到真正的建立连接，但是这些连接无法复用会导致每次请求都经历三次握手和慢启动。三次握手在高延迟的场景下影响较明显，慢启动则对文件类大请求影响较大。
          </li>
         </ul>
        </li>
       </ul>
      </li>
     </ul>
    </blockquote>
    <h5>
     <a id="12HTTP1_67">
     </a>
     1.2、HTTP的问题1
    </h5>
    <ul>
     <li>
      HTTP被抱怨最多的问题是：
      <strong>
       连接无法复用
      </strong>
      。每次请求都要进行TCP三次握手，4次挥手。建立连接的过程消耗的资源很大。
     </li>
     <li>
      所以后面
      <strong>
       HTTP1.0会设置connection：Keep-Alive（持久连接）
      </strong>
      。这样就可以减少TCP的握手和挥手，至于可以
      <strong>
       复用多久是由服务器决定
      </strong>
      。
     </li>
     <li>
      <strong>
       HTTP 1.1的话默认就是Keep-Alive
      </strong>
      了，要关闭这个功能就把connection改成
      <strong>
       Close
      </strong>
      。
     </li>
    </ul>
    <h5>
     <a id="13HTTP2_73">
     </a>
     1.3、HTTP的问题2
    </h5>
    <ul>
     <li>
      HTTP还有个大问题是
      <strong>
       队头阻塞
      </strong>
      （head of blocking）。
      <ul>
       <li>
        <strong>
         队头阻塞导致带宽无法充分利用。后续请求被阻塞
        </strong>
        。
        <br/>
        <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/f9ab30857e404aa587b498afade823ac.png"/>
       </li>
      </ul>
     </li>
    </ul>
    <blockquote>
     <ul>
      <li>
       例如：假如有五个请求被同时发出，如果第⼀个请求没有处理完成，就会导致后续的请求也无法得到处理，如下图所示：
       <ul>
        <li>
         如果发送的第一个请求被阻塞，其他的4个请求也就被阻塞在客户端，等到请求1被处理完，其他才发出。
        </li>
       </ul>
      </li>
     </ul>
     <p>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/2134d70f19a449f88ded2860f85ebc6c.png"/>
     </p>
    </blockquote>
    <blockquote>
     <p>
      HTTP1.1里面有个
      <strong>
       管线化技术(流水线)
      </strong>
      ：单个连接可以同时发多个请求，但是发送的请求顺序是怎么样，响应的顺序也是一样的。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/4e85ff985c0c4bf48fd5f4e264266f91.png"/>
     </p>
     <ul>
      <li>
       <h3>
        <a id="_84">
        </a>
        但是这种它有不少缺陷。
       </h3>
       <ul>
        <li>
         如果管道中的
         <strong>
          第一个请求的响应延迟或阻塞
         </strong>
         ，后续请求的响应也会被阻塞，即使它们已经处理完成。
        </li>
        <li>
         <strong>
          并非
         </strong>
         所有服务器和代理都正确支持流水线技术。
        </li>
        <li>
         如果管道中的某个请求失败，
         <strong>
          客户端难以确定具体是哪个请求出了问题
         </strong>
         。
        </li>
        <li>
         流水线可能导致
         <strong>
          服务器资源竞争
         </strong>
         ，尤其是在高并发场景下。
        </li>
        <li>
         <strong>
          流水线的实现对客户端和服务器都增加了复杂性
         </strong>
         。
        </li>
       </ul>
      </li>
     </ul>
    </blockquote>
    <h3>
     <a id="2SPDY_92">
     </a>
     2、SPDY
    </h3>
    <p>
     SPDY（发音为 “speedy”）是 Google 开发的一种网络协议，旨在减少网页加载时间并改善 Web 性能。
     <strong>
      它是 HTTP/2 协议的前身
     </strong>
     ，引入了许多创新特性，为现代 Web 性能优化奠定了基础。
    </p>
    <ul>
     <li>
      SPDY主要解决了HTTP的延迟和安全性。
     </li>
    </ul>
    <blockquote>
     <ul>
      <li>
       <h3>
        <a id="SPDY__HTTP11__95">
        </a>
        SPDY 的主要目标是解决 HTTP/1.1 的性能瓶颈
       </h3>
       <ul>
        <li>
         <strong>
          高延迟
         </strong>
         ：HTTP/1.1 的串行请求和响应机制导致延迟较高。
        </li>
        <li>
         <strong>
          队头阻塞
         </strong>
         （Head-of-Line Blocking）：一个请求的延迟会影响后续请求。
        </li>
        <li>
         <strong>
          低效的头部传输
         </strong>
         ：HTTP/1.1 的头部信息重复且未压缩。
        </li>
        <li>
         <strong>
          并发连接限制
         </strong>
         ：浏览器对同一域名的并发连接数有限制。
        </li>
       </ul>
      </li>
     </ul>
    </blockquote>
    <blockquote>
     <ul>
      <li>
       <h3>
        <a id="SPDY_102">
        </a>
        SPDY的功能
       </h3>
       <ul>
        <li>
         <ol>
          <li>
           多路复用（Multiplexing）
          </li>
         </ol>
         <ul>
          <li>
           功能：在单个 TCP 连接上同时传输多个请求和响应。
          </li>
          <li>
           优势：避免了 HTTP/1.1 的队头阻塞问题，提高了连接利用率。
          </li>
         </ul>
        </li>
        <li>
         <ol start="2">
          <li>
           头部压缩
          </li>
         </ol>
         <ul>
          <li>
           功能：使用专门的算法（如 HPACK）压缩 HTTP 头部。
          </li>
          <li>
           优势：减少了头部数据的传输量，降低了延迟。
          </li>
         </ul>
        </li>
        <li>
         <ol start="3">
          <li>
           请求优先级
          </li>
         </ol>
         <ul>
          <li>
           功能：允许客户端为请求设置优先级，确保关键资源优先加载。
          </li>
          <li>
           优势：优化了页面渲染性能，提升了用户体验。
          </li>
         </ul>
        </li>
        <li>
         <ol start="4">
          <li>
           服务器推送（Server Push）
          </li>
         </ol>
         <ul>
          <li>
           功能：服务器可以主动向客户端推送资源，而无需等待客户端请求。
          </li>
          <li>
           优势：减少了额外的请求往返时间，加快了页面加载速度。
          </li>
         </ul>
        </li>
        <li>
         <ol start="5">
          <li>
           加密传输
          </li>
         </ol>
         <ul>
          <li>
           功能：SPDY 默认使用 TLS 加密传输数据。
          </li>
          <li>
           优势：提高了安全性，防止数据被窃听或篡改
          </li>
         </ul>
        </li>
       </ul>
      </li>
     </ul>
    </blockquote>
    <h3>
     <a id="3HTTP_20_120">
     </a>
     3、HTTP 2.0
    </h3>
    <h4>
     <a id="31_122">
     </a>
     3.1、二进制格式
    </h4>
    <p>
     HTTP 1.x是文本格式。那么它的协议解析是基于文本，而文本的复杂度很高。而二进制格式只识别0和1，所以在
     <strong>
      HTTP2.0里开始使用二进制格式
     </strong>
     。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/dad3e67976fd415bbbf8dd35ebda8edd.png"/>
    </p>
    <h4>
     <a id="32_125">
     </a>
     3.2、连接共享
    </h4>
    <p>
     连接共享意味着客户端与服务器之间也只需要⼀个连接即可，这样即使来自
     <strong>
      很多流的数据包也能够混合在⼀起通过同样连接传输
     </strong>
     ，再根据不同帧首部的 **stream id 标识符（流标识符）**重新连接将不同的数据流进行组装。
    </p>
    <blockquote>
     <p>
      stream 是连接中的⼀个虚拟信道，可以承载
      <strong>
       双向消息传输
      </strong>
      。每个流有唯⼀整数标识符。为了
      <strong>
       防止两端 streaam id 冲突
      </strong>
      ，客户端发起的流具有奇数 id，服务器端发起的流具有偶数 id。HTTP 2.0 你可以对不同的 stream 设置不同的优先级。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/549ed416586e4385b6e7d28409e6773b.png"/>
     </p>
    </blockquote>
    <h4>
     <a id="33_130">
     </a>
     3.3、头部压缩
    </h4>
    <p>
     <strong>
      HTTP1.x 中的 header 由于 cookie 和 user agent 不存在记忆性
     </strong>
     ，导致每次都要带着这些头
     <strong>
      重新发送请求
     </strong>
     。HTTP 2.0 可以使用 HPACK 算法压缩 HTTP 头部，减少冗余数据。通信双方会各自
     <strong>
      缓存⼀份 header 字段表
     </strong>
     ，这样能够避免重复传输 header ，也能够减小传输的大小。
    </p>
    <h4>
     <a id="34_133">
     </a>
     3.4、服务端推送
    </h4>
    <p>
     服务器可以主动向客户端推送资源，而无需等待客户端请求。
    </p>
    <h4>
     <a id="35HTTP_20_136">
     </a>
     3.5、HTTP 2.0的缺陷
    </h4>
    <ul>
     <li>
      多路复用的情况下，连接（连接是基于TCP协议上的）频繁的话，容易导致TCP性能瓶颈。
     </li>
     <li>
      HTTP 2.0会增加一次TLS，就多了一次RTT。
     </li>
     <li>
      多个请求是在同一个 TCP 管道中，这样当 HTTP 2.0 出现丢包时，整个 TCP 都要开始等待重传，那么就会阻塞该 TCP。连接中的所有请求。
     </li>
    </ul>
    <h3>
     <a id="4HTTP_30_141">
     </a>
     4、HTTP 3.0
    </h3>
    <ul>
     <li>
      <strong>
       TCP 队头阻塞问题
      </strong>
      <ul>
       <li>
        在一个 TCP 连接中，
        <strong>
         如果某个数据包丢失或延迟
        </strong>
        ，后续的数据包即使已经到达接收方，也无法被处理，
        <strong>
         必须等待丢失的数据包重传并到达
        </strong>
        。
       </li>
      </ul>
     </li>
    </ul>
    <blockquote>
     <p>
      解决方法：使用了基于UDP的QUIC协议。QUIC把TLS和TCP整合在一起了，减少了时间。
