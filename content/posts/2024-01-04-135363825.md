    <h2 id="%E4%B8%80%E3%80%81%E8%BD%AF%E4%BB%B6%E5%B7%A5%E7%A8%8B">
     一、软件工程
    </h2>
    <h3 id="1.1%E8%BD%AF%E4%BB%B6%E7%9A%84%E5%AE%9A%E4%B9%89%E5%8F%8A%E7%89%B9%E7%82%B9">
     1.1软件的定义及特点
    </h3>
    <p>
     定义：
    </p>
    <p>
     软件是计算机中硬件相互依存的部分，它不仅包括
     <span style="background-color:#f9eda6;">
      程序
     </span>
     ，还包括
     <span style="background-color:#f9eda6;">
      数据
     </span>
     和
     <span style="background-color:#f9eda6;">
      相关文档
     </span>
     ，是三者的完整集合。
    </p>
    <ul>
     <li>
      程序：按事先设计好的功能和性能要求执行的指令序列。
     </li>
     <li>
      数据：程序处理、加工的对象和处理信息的数据结构。
     </li>
     <li>
      文档：与程序开发、维护和使用相关的各类资料总称。
     </li>
    </ul>
    <p>
     特点：
    </p>
    <ol>
     <li>
      软件是一种逻辑产品，具有抽象性，更多地带有个人智慧因素，这使得软件与其他的机械制造、建筑工程有许多不同。
     </li>
     <li>
      软件没有明显的制造过程，不会损坏，但会因为质量和不可维护性被废弃。因此，软件产品数量和质量在相当长的实践中还得依靠技术人员和管理人员。
     </li>
     <li>
      软件的开发和维护成本高，且涉及诸多社会因素。
     </li>
    </ol>
    <h3 id="1.2%E8%BD%AF%E4%BB%B6%E5%8D%B1%E6%9C%BA">
     1.2软件危机
    </h3>
    <p>
     软件危机是指在计算机软件开发和维护过程中遇到的一系列严重问题。
    </p>
    <h4 id="1.2.1%E4%BA%A7%E7%94%9F%E5%8E%9F%E5%9B%A0">
     1.2.1产生原因
    </h4>
    <ol>
     <li>
      与软件自身特点有关，逻辑产品，缺乏可见性，规模大且复杂；修改、维护困难。
     </li>
     <li>
      软件开发与维护的方法不当：忽视需求分析；重视现，轻设计；轻视软件维护。
     </li>
     <li>
      供求矛盾是永恒的主题：需求的增长和变更，使得开发者力不从心。
     </li>
    </ol>
    <h4 id="1.2.2%E5%85%B8%E5%9E%8B%E8%A1%A8%E7%8E%B0">
     1.2.2典型表现
    </h4>
    <ol>
     <li>
      软件开发成本和进度不可控。
     </li>
     <li>
      软件产品不能满足用户的需求。
     </li>
     <li>
      软件产品的质量难以保证。
     </li>
     <li>
      软件可维护性差。
     </li>
     <li>
      软件缺乏适当的软件文档支持。
     </li>
     <li>
      软件成本在计算机系统总成本中所占的比例逐年上升。
     </li>
     <li>
      软件生产率跟不上计算机应用普及和深入的趋势。
     </li>
    </ol>
    <h3 id="1.3%E8%BD%AF%E4%BB%B6%E5%B7%A5%E7%A8%8B%E6%A6%82%E8%BF%B0">
     1.3软件工程概述
    </h3>
    <h4 id="1.3.1%E8%BD%AF%E4%BB%B6%E5%B7%A5%E7%A8%8B%E6%96%B9%E6%B3%95%E5%AD%A6">
     1.3.1软件工程方法学
    </h4>
    <p>
     软件工程方法学包括三个要素，即方法、工具和过程。
    </p>
    <ol>
     <li>
      方法：完成软件工程项目的技术手段，回答“怎么做”的问题。
     </li>
     <li>
      工具：为运用方法提供自动或半自动的支撑环境。
     </li>
     <li>
      过程：为了获取高质量软件所需完成的一系列任务框架，包括支持软件开发各个环节的控制和管理。
     </li>
    </ol>
    <p>
     目前使用最广泛的软件工程方法学为
     <span style="background-color:#f9eda6;">
      传统方法学
     </span>
     和
     <span style="background-color:#f9eda6;">
      面向对象方法学
     </span>
     。
    </p>
    <h4 id="1.3.2%E8%BD%AF%E4%BB%B6%E5%B7%A5%E7%A8%8B%E7%9A%84%E5%9F%BA%E6%9C%AC%E5%8E%9F%E5%88%99">
     1.3.2软件工程的基本原则
    </h4>
    <ol>
     <li>
      采用适宜的开发模型
     </li>
     <li>
      采取合适的设计方法
     </li>
     <li>
      提供高质量的工程支持
     </li>
     <li>
      重视开发过程的管理
     </li>
    </ol>

     2.2软件过程的定义
    </h3>
    <p>
     软件过程（Software Produce）是为了获得高质量软件所需要完成的一系列任务的框架，它规定了完成各项任务的工作步骤。概括地说，软件过程描述为了开发出用户需要的软件，什么人（Who）在什么时候（When）做什么事（What）以及怎么做（How）这些事，以实现某一个特定的具体目标。
    </p>

    </h4>
    <p>
     1970年，温斯顿·罗伊斯（Winston Royce）提出了著名的“瀑布模型”，直到20世纪80年代早期，它一直是唯一被广泛采用的软件开发模型。
    </p>
    <ul>
     <li>
      核心思想：
     </li>
    </ul>
    <p>
     按工序将问题化简，将功能的实现与设计分开，便于分工协作，即采用结构化的分析与设计方法将逻辑实现与物理实现分开。将软件生命周期划分为
     <span style="background-color:#f9eda6;">
      可行性研究
     </span>
     ，
     <span style="background-color:#f9eda6;">
      需求分析
     </span>
     ，
     <span style="background-color:#f9eda6;">
      设计
     </span>
     ，
     <span style="background-color:#f9eda6;">
      编码
     </span>
     ，
     <span style="background-color:#f9eda6;">
      综合测试
     </span>
     ，
     <span style="background-color:#f9eda6;">
      运行维护
     </span>
     六个部分，并且规定了这些活动自上而下，相互衔接的固定次序，如同瀑布流水，逐级下落。
    </p>
    <ul>
     <li>
      过程：
     </li>
    </ul>
    <p>
     从上一项活动接受该项活动的工作对象作为输入，利用这一输入实施该项活动应完成的内容，给出该项活动的工作成果，并作为输出传给下一项活动，甚至更前面的活动。
    </p>
    <p>
     因此，对于需求经常变化的项目而言，瀑布模型毫无价值。
    </p>
    <p>
     瀑布模型如图1图2所示（实线代表开发过程，虚线表示维护过程）
    </p>
    <div class="img-center">
     <figure class="image">
      <img alt="" height="80" src="https://i-blog.csdnimg.cn/blog_migrate/66dfeb0f20c9feda19c2b2e23c7e50a4.png" width="753">
       <figcaption>
        图1 传统的瀑布模型
       </figcaption>
      </img>
     </figure>
    </div>
    <div class="img-center">
     <figure class="image">
      <img alt="" height="404" src="https://i-blog.csdnimg.cn/blog_migrate/5f334e31ff37f51c6764144449b925c3.png" width="575">
       <figcaption>
        图2 实际的瀑布模型
       </figcaption>
      </img>
     </figure>
    </div>
    <ul>
     <li>
      瀑布模型的优点：
     </li>
    </ul>
    <ol>
     <li>
      可强迫开发人员采用规范的方法。
     </li>
     <li>
      严格地规定了每个阶段必须提交的文档。
     </li>
     <li>
      要求每个阶段交出的所有产品都必须经过质量保证小组的仔细验证。
     </li>
    </ol>
    <ul>
     <li>
      瀑布模型的缺点：
     </li>
    </ul>
    <ol>
     <li>
      只能通过文档谅解产品，不能经过实践的需求是不切实际的。
     </li>
     <li>
      固定的顺序导致纠正前期错误的代价越来越高。
     </li>
     <li>
      需求变动带来很大风险。
     </li>
    </ol>
    <ul>
     <li>
      瀑布模型适用于：
     </li>
    </ul>
    <ol>
     <li>
      需求是预知的。
     </li>
     <li>
      软件实现方法是成熟的。
     </li>
     <li>
      项目周期较短。
     </li>
    </ol>
    <h4 id="2.3.2%E8%9E%BA%E6%97%8B%E6%A8%A1%E5%9E%8B">
     2.3.2螺旋模型
    </h4>
    <p>
     1988年，巴利·玻姆（Barry Boebm）正式发表了软件系统开发的“螺旋模型”，强调了其他模型忽视的风险分析，特别适合于大型复杂且昂贵的系统级应用软件。
    </p>
    <div class="img-center">
     <figure class="image">
      <img alt="" height="436" src="https://i-blog.csdnimg.cn/blog_migrate/1177e8a2d4fff52e0fcbd11f106c3bcd.png" width="498">
       <figcaption>
        图3 螺旋模型的结构
       </figcaption>
      </img>
     </figure>
    </div>
    <p>
     螺旋模型沿着螺旋线进行若干次迭代，图3中的四个象限分别代表了
     <span style="background-color:#f9eda6;">
      制定计划
     </span>
     、
     <span style="background-color:#f9eda6;">
      风险分析
     </span>
     、
     <span style="background-color:#f9eda6;">
      实施工程
     </span>
     、
     <span style="background-color:#f9eda6;">
      客户评估
     </span>
     。
    </p>
    <ul>
     <li>
      基本做法：
     </li>
    </ul>
    <p>
     在“瀑布模型”的每个开发阶段前引入一个非常严格的风险识别、风险分析和风险控制，它把软件项目分解成为一个个小项目。每个小项目都标识一个或多个主要风险，直到所有的主要风险因素都被确定。
    </p>
    <ul>
     <li>
      螺旋模型的优点：
     </li>
    </ul>
    <ol>
     <li>
      螺旋模型是一个风险驱动的模型，强调可选方案和约束条件，从而支持软件的重用，有助于将软件质量作为特殊目标融入产品开发中。
     </li>
     <li>
      设计上的灵活性，可以在项目的各个阶段进行变更。
     </li>
     <li>
      以小的分段构建大型系统，使成本计算变得简单容易。
     </li>
     <li>
      客户始终参与每个阶段的开发，保证了项目不偏离正确方向以及项目的可控性。
     </li>
     <li>
      对可选方案和约束条件的强调有利于已有软件的重用，也有助于把软件质量作为软件开发的一个重要目标。
     </li>
     <li>
      维护只是模型的另一个周期，维护和开发之间没有本质区别。
     </li>
    </ol>
    <ul>
     <li>
      螺旋模型的缺点：
     </li>
    </ul>
    <ol>
     <li>
      采用螺旋模型需要有相当丰富的风险评估经验和专业知识，在风险较大的项目开发中如果未及时标识风险，势必造成重大损失。
     </li>
     <li>
      建设周期长，而软件技术发展比较快，所以经常出现软件开发完毕后，和当前的技术水平有了较大的差距，无法满足当前用户的需求。
     </li>
     <li>
      过多的迭代次数会增加开发成本，延迟提交时间。
     </li>
    </ol>
    <ul>
     <li>
      螺旋模型的项目适用的几个方面：
     </li>
    </ul>
    <ol>
     <li>
      就近开发，需求不明确的情况，便于风险控制和需求变更。
     </li>
     <li>
      大规模软件的项目开发。
     </li>
    </ol>
    <h4 id="2.3.3%E5%96%B7%E6%B3%89%E6%A8%A1%E5%9E%8B">
     2.3.3喷泉模型
    </h4>
    <p>
     喷泉模型是一种以用户需求为动力，以对象为驱动的模型，主要用于描述面向对象的软件开发过程。
    </p>
    <p>
     <span style="background-color:#f9eda6;">
      该模型中，软件开发过程自上而下周期的各个阶段是相互迭代和无间隙的。
     </span>
    </p>
    <p>
     喷泉模型如图4所示
    </p>
    <div class="img-center">
     <figure class="image">
      <img alt="" height="402" src="https://i-blog.csdnimg.cn/blog_migrate/9619c88f90d4a2504aa7e1adc4f9150d.png" width="243">
       <figcaption>
        图4 喷泉模型
       </figcaption>
      </img>
     </figure>
    </div>
    <p>
     无间隙指在各项活动之间无明显边界，如分析和设计活动之间没有明显的界限。
    </p>
    <ul>
     <li>
      喷泉模型优点：
     </li>
    </ul>
    <ol>
     <li>
      该模型的各个阶段没有明显的界限，开发人员可以同步进行开发。
     </li>
     <li>
      多次反复地增加或明确目标系统，而不是本质性的改动，降低错误的可能性。
     </li>
     <li>
      可以提高软件项目的开发效率，节省开发时间。
     </li>
    </ol>
    <ul>
     <li>
      喷泉模型缺点：
     </li>
    </ul>
    <ol>
     <li>
      由于喷泉模型在各个开发阶段是重叠的，因此在开发过程中需要大量的开发人员，这样很不利于项目管理。
     </li>
     <li>
      要求严格管理文档，使得审核的难度加大，尤其是面对可能随时加入各种信息、需求与资料的情况。
     </li>
    </ol>
    <ul>
     <li>
      喷泉模型适用于面型对象的软件开发过程。
     </li>
    </ul>

    <h4 id="3.1.2%E5%8A%A8%E5%8A%A8%E8%84%91">
     3.1.2动动脑
    </h4>
    <p style="text-align:justify;">
     1.简述结构化方法与面向对象方法的主要特点，如何将两种方法结合在一起，有效开发软件项目？
    </p>
    <ol>
     <li style="margin-left:0px;text-align:justify;">
      结构化方法：
     </li>
     <li style="text-align:justify;">
      适合小型项目开发
      <ol>
       <li style="text-align:justify;">
        数据和功能分离
       </li>
       <li style="text-align:justify;">
        文档驱动（瀑布模型）
       </li>
       <li style="text-align:justify;">
        模块化（功能）
       </li>
       <li style="text-align:justify;">
        软件结构：自顶向下，逐层分解
       </li>
       <li style="text-align:justify;">
        便于测试和维护
       </li>
       <li style="text-align:justify;">
        SA➡SD➡SP（数据流程图）
       </li>
       <li style="text-align:justify;">
        数据模型、功能模型、行为模型（ER、DFD、SC）
       </li>
      </ol>
     </li>
    </ol>
    <p style="margin-left:0;text-align:justify;">
     面向对象方法：
    </p>
    <ol>
     <li style="text-align:justify;">
      适合中大型项目开发
      <ol>
       <li style="text-align:justify;">
        数据和功能封装à对象
       </li>
       <li style="text-align:justify;">
        UML驱动（UP）
       </li>
       <li style="text-align:justify;">
        模块化（对象）
       </li>
       <li style="text-align:justify;">
        软件结构：对象之间交互（6种关系）
       </li>
       <li style="text-align:justify;">
        便于维护和扩展
       </li>
       <li style="text-align:justify;">
        OOA➡OOD➡OOP（类图）
       </li>
       <li style="text-align:justify;">
        对象模型、功能模型、动态模型（Class、User Case、Statement）
       </li>
      </ol>
     </li>
    </ol>
    <p style="margin-left:0;text-align:justify;">
     无论使用哪种开发方法，或者是混合哪几种开发方法，都要因地制宜，依据需求分析和系统要求，做出最好的选择或组合。
    </p>
    <p style="text-align:left;">
     <span style="background-color:#ffffff;">
      <span style="color:#000000;">
       2.软件工程教材中，经常会叙述发现并改正错误的重要性，有人不同意这个观点，认为要求在错误进入软件之前就清楚它们是不现实的，并举例说：“如果一个故障是编码错误造成的，那么，怎么能在设计阶段就清楚它？”，判断这个观点是否正确，并且给出理由。
      </span>
     </span>
    </p>
    <p style="margin-left:0;text-align:justify;">
     【解答】
    </p>
    <ol>
     <li style="text-align:justify;">
      设计也能发现Bug：
      <ol>
       <li style="text-align:justify;">
        设计过程本身就应该设计校验。
       </li>
       <li style="text-align:justify;">
        在软件开发流程中，TDD是解决这个问题的方式。
       </li>
       <li style="text-align:justify;">
        减少问题不是直接不产生Bug，而是能够及时发现并消除Bug。
       </li>
       <li style="text-align:justify;">
        编码之前，先编写测试单元。
       </li>
       <li style="text-align:justify;">
        每次编码结束，都应该运行测试单元，确保代码能够通过。
       </li>
      </ol>
     </li>
    </ol>

        <span style="background-color:#ffffff;">
         <span style="color:#000000;">

         </span>
        </span>
       </li>
      </ol>
     </li>
    </ol>
    <p style="margin-left:0;text-align:justify;">
     （2）为什么说喷泉模型较好地体现了面向对象软件开发过程无缝和迭代的特点，并举例说明。
    </p>
    <p style="margin-left:0;text-align:justify;">
     【解答】
    </p>
    <ol>
     <li style="text-align:left;">
      <span style="background-color:#ffffff;">
       <span style="color:#000000;">
        喷泉模型特点
       </span>
      </span>
      <ol>
       <li style="text-align:left;">
        <span style="background-color:#ffffff;">
         <span style="color:#000000;">
          使用面向对象方法学开发软件时，各个阶段都使用统一的概念和表示符号。
         </span>
        </span>
       </li>
       <li style="text-align:left;">
        <span style="background-color:#ffffff;">
         <span style="color:#000000;">
          用建模工具描绘的模型贯穿在开发各个阶段。
         </span>
        </span>
       </li>
       <li style="text-align:left;">
        <span style="background-color:#ffffff;">
         <span style="color:#000000;">
          整个开发过程都是吻合一致的，或者说是无缝连接的。
         </span>
        </span>
       </li>
       <li style="text-align:left;">
        <span style="background-color:#ffffff;">
         <span style="color:#000000;">
          容易实现各个开发步骤反复多次迭代，达到认识的逐步深化。
         </span>
        </span>
       </li>
       <li style="text-align:left;">
        <span style="background-color:#ffffff;">
         <span style="color:#000000;">
          喷泉模型则体现了面向对象软件开发过程迭代和无缝的特性。
