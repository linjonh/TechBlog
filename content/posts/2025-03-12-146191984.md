    <h2>
     <a id="_0">
     </a>
     核心思想
    </h2>
    <p>
     责任链模式是一种行为设计模式，允许多个对象都有机会处理请求，从而避免请求的发送者与接收者之间的耦合。请求沿着处理链传递，直到某个对象处理它为止。
    </p>
    <h2>
     <a id="_2">
     </a>
     解决的问题
    </h2>
    <p>
     ​
     <strong>
      解耦请求发送者与处理者
     </strong>
     ：请求的发送者无需知道具体由哪个对象处理请求。
     <br/>
     <strong>
      ​动态分配责任
     </strong>
     ：可以在运行时动态调整处理链，灵活添加或移除处理者。
     <br/>
     <strong>
      ​避免硬编码
     </strong>
     ：避免将请求处理逻辑硬编码在某个类中，提高代码的可扩展性和可维护性。
    </p>
    <h2>
     <a id="_6">
     </a>
     使用场景
    </h2>
    <p>
     <strong>
      ​多级审批流程
     </strong>
     ：如请假审批、报销审批等，每一级领导都可以处理或传递给上级。
     <br/>
     <strong>
      ​事件处理系统
     </strong>
     ：如 GUI 事件处理，事件可以沿着组件链传递，直到被处理。
     <br/>
     <strong>
      ​日志记录系统
     </strong>
     ：不同级别的日志消息可以被不同的日志处理器处理。
     <br/>
     <strong>
      ​过滤器链
     </strong>
     ：如 HTTP 请求过滤器，每个过滤器可以处理请求或传递给下一个过滤器。
    </p>
    <h2>
     <a id="_11">
     </a>
     优点
    </h2>
    <p>
     <strong>
      ​解耦
     </strong>
     ：请求发送者与处理者解耦，发送者无需知道具体处理者。
     <br/>
     <strong>
      ​动态性
     </strong>
     ：可以在运行时动态调整处理链。
     <br/>
     <strong>
      ​单一职责
     </strong>
     ：每个处理者只关注自己的职责，符合单一职责原则。
    </p>
    <h2>
     <a id="_15">
     </a>
     缺点
    </h2>
    <p>
     <strong>
      ​性能问题
     </strong>
     ：如果链过长，可能导致请求传递效率低下。
     <br/>
     <strong>
      ​不确定性
     </strong>
     ：请求可能未被任何处理者处理，需要额外逻辑处理这种情况。
    </p>

    <h2>
     <a id="_125">
     </a>
     代码解析
    </h2>
    <p>
     ​
     <strong>
      Request 类
     </strong>
     ：封装请求内容和级别。
     <br/>
     ​
     <strong>
      Handler 基类
     </strong>
     ：定义处理请求的接口，并持有下一个处理者的指针。
     <br/>
     <strong>
      ​具体处理者 Manager、Director、CEO
     </strong>
     ​：实现自己的处理逻辑，若无法处理则传递给下一个处理者。
     <br/>
     <strong>
      ​处理链
     </strong>
     ：通过 setNext 方法动态构建处理链。
    </p>
