    <h2>
     <a id="_1">
     </a>
     上一集回顾
    </h2>
    <p>
     具体来说，我们通过隐式计算来解决问题，而不是像数字微分分析器那样逐步增加数据。我们已经涵盖了这个部分，并计划继续处理音量问题。不过，实际上我们现在不需要继续处理音量的问题，虽然我们可以修复它，但它没有给我们带来实际的麻烦，在音量混合器中也没有遇到问题。
    </p>
    <h2>
     <a id="_4">
     </a>
     关于如何高效管理编程时间
    </h2>
    <p>
     我们现在先暂时假设声音混音器已经足够好了。就像引擎的其他部分一样，如果在游戏开发的过程中发现引擎无法完成需要的功能，我们会在那个时候进行修订。这种方式通常是比较好的，因为你不想花大量时间去实现一些根本不会用到的引擎功能，或者实现一些实际上不需要的引擎功能。因此，我总是建议采取探索式的方法，只在实际需要时才去做，这样可以确保不会浪费大量时间在那些不必要的工作上，或者是花很多时间做的功能，最后却发现它并不适合任务。
    </p>
    <p>
     在开发引擎的过程中，我们始终努力采取这种方法，只有在看到确切的使用场景时，才会加入新功能。对于不确定的部分，我们尽量保持简单。我们不想花时间去做那些最终可能对游戏没有影响或者无关紧要的部分。
    </p>
    <p>
     无论做什么，当你在编程，或者在做游戏时，每花费的一小时都是在做某件事。如果最终这件事没有出现在最终的游戏里，那就是浪费了这一小时。因此，在编程时，确保自己不花太多时间做那些最后不会在游戏中体现的内容是非常重要的。总之，这是我们在这个项目中所尝试的方式。
    </p>
    <h2>
     <a id="_11">
     </a>
     混音器与游戏的接口尚未最终确定，但我们已经知道它需要从资源系统中获得什么
    </h2>
    <p>
     现在我们想要回到资产系统。我们已经写了声音混音器，目的是让大家了解资产系统可能需要的功能，因此我们可能会从另一个方向来处理声音系统。当我们真正开始将声音融入游戏时，像环境音、背景音乐等内容会加入游戏，那时我们再回过头来审视声音混音器，看看是否需要做什么调整。所以，我们目前需要从两个不同的角度来看待这个问题。
    </p>
    <p>
     第一个角度是声音混音器可能需要资产系统提供什么功能。通过这样做，我们可以完成资产系统的开发。接下来，我们会从游戏代码的需求出发来看待声音系统，预计到时候我们对声音混音器的修改不会很多，因为目前它的工作方式已经很直接，且与资产混音器已经对接得很好。不过，声音混音器的接口部分，比如播放声音的结构，可能会发生显著的变化。现在的声音混音器其实只是一个非常简单的层，它能做的事情非常基础，目的只是为了能让声音播放出来。因此，声音混音器的接口部分会在后续有所扩展。
    </p>
    <p>
     我们不打算现在就对声音混音器做太多修改，因为更合适的时机是在游戏中加入一定的声音环境时，这时我们才会清楚知道声音混音器的接口应该是什么样子的。因此，现在更专注于资产系统，直到我们有更清晰的需求时，才会进一步完善声音混音器的功能，尤其是在如何驱动混音代码这部分。
    </p>
    <h2>
     <a id="_18">
     </a>
     今天的计划：定义资源系统
    </h2>
    <p>
     我们现在计划回顾一下资源系统，并重新整理我们的思路，讨论我们希望资源系统具备哪些功能，并可能开始编写一些代码，将目前的测试代码逐步迁移到资源系统中，逐步定义其结构和功能。
    </p>
    <p>
     在正式开始之前，我们会先在黑板上进行一些讨论，介绍一下资源系统的背景和我们接下来要做的事情，和以往一样，在进入新的模块之前，先打下一些理论基础。如果我们今天能够顺利完成这些讨论，并有时间开始编程的话。
    </p>
    <p>
     接下来我们将集中讨论资源系统的内容，并定义我们期望它实现的功能。目前，我们已经对资源系统的基本工作方式进行了初步定义，但现在我们需要做的是更深入地设计它的结构。
    </p>
    <h2>
     <a id="_25">
     </a>
     资源系统的两个方面：文件管理和内存管理
    </h2>
    <p>
     目前我们已经有了一个初步的资源系统，并且我们对它的实现效果感到相当满意。目前系统中基于标签的资源匹配机制运行得非常好，这种匹配方式我们认为非常不错，效果出乎意料的理想。而接下来我们要做的，就是将资源系统推进到一个更完整的状态，使其更加完善。
    </p>
    <p>
     接下来有两个主要任务需要完成：
    </p>
    <h4>
     <a id="_30">
     </a>
     <strong>
      第一，定义资源文件的格式
     </strong>
    </h4>
    <p>
     我们需要确定资源文件的具体格式，也就是如何组织和存储游戏中的资源内容。资源文件主要包含的就是游戏中使用的所有素材，包括纹理、声音、模型、动画等等。这些文件将是资源系统的核心内容，因此我们需要确定一种合适的文件格式，使得资源系统能够方便地加载、解析和管理这些资源内容。
    </p>
    <p>
     我们的目标是尽可能让资源文件的格式清晰且易于扩展，并确保资源系统能够快速解析这些文件，同时满足不同平台或不同设备的运行需求。这也是我们接下来需要讨论和规划的重要内容之一。
    </p>
    <h4>
     <a id="_35">
     </a>
     <strong>
      第二，解决资源系统中的内存管理问题
     </strong>
    </h4>
    <p>
     内存管理是资源系统中更复杂、更关键的一个部分。我们的目标是让资源系统像操作系统的虚拟内存管理机制一样，但同时具备更高的可预测性和可控性。也就是说，我们希望资源系统能够灵活地管理资源加载和卸载的过程，避免内存占用超出设备的可用范围，同时保证游戏运行过程中需要的资源始终可用。
    </p>
    <p>
     我们设想了一种极端情况：假设游戏中存在
     <strong>
      16GB未压缩的资源
     </strong>
     ，而玩家的设备可能只有
     <strong>
      2GB或4GB内存
     </strong>
     ，在这种情况下，我们的资源系统需要具备类似虚拟内存的功能，即只加载当前游戏场景所需的资源，而不会一次性将所有资源加载进内存中。
    </p>
    <p>
     资源系统需要允许我们设定一个
     <strong>
      工作集大小（Working Set Size）
     </strong>
     ，例如我们可以指定最大只能占用
     <strong>
      2GB内存
     </strong>
     ，那么无论游戏拥有多少资源，比如
     <strong>
      64GB甚至更多
     </strong>
     ，资源系统都能够动态调整，使其只加载当前游戏状态下所需要的资源，将其他暂时用不到的资源释放或保持在硬盘中。
    </p>
    <p>
     我们目前无法确定在实际游戏中是否真的会达到如此庞大的资源规模，但我们希望提前为这种情况做好准备。即使目前的游戏资源规模远小于16GB，甚至仅有4GB，但未来如果资源量激增，或者游戏内容持续扩展，资源系统仍然具备良好的可扩展性和适应性。
    </p>
    <p>
     即使最终所有用户的设备都有足够的内存，并且游戏的资源总量也不会达到64GB，但我们仍然希望通过设计这样一个灵活的资源系统，使游戏可以轻松应对不同硬件环境下的运行需求，并保持良好的内存利用率。
    </p>
    <p>
     因此，我们将资源系统的内存管理目标定义为：
    </p>
    <ul>
     <li>
      <strong>
       允许设置固定的工作集大小
      </strong>
      ，即无论游戏资源总量多大，资源系统只占用设定的内存上限。
     </li>
     <li>
      <strong>
       动态加载和卸载资源
      </strong>
      ，确保只有当前游戏状态下需要的资源保留在内存中，其他资源暂时保存在硬盘中或者压缩状态中。
     </li>
     <li>
      <strong>
       最大化利用内存
      </strong>
      ，在确保不溢出的情况下尽可能加载更多的资源，以保证游戏流畅运行。
     </li>
    </ul>
    <p>
     这一点将是整个资源系统中最复杂、最具挑战性的部分。我们需要设计一套类似
     <strong>
      虚拟内存管理机制
     </strong>
     的资源加载方案，使其既能满足游戏资源庞大的需求，又能确保设备内存占用处于合理范围。
    </p>
    <p>
     我们也不确定在实际开发过程中，游戏是否真的会达到需要动态分页加载的程度。但我们想通过这种设计，让资源系统具有更高的可扩展性和灵活性，从而确保即使未来资源规模激增，游戏依然能够在有限内存中顺畅运行。
    </p>
    <h4>
     <a id="_55">
     </a>
     <strong>
      总结
     </strong>
    </h4>
    <p>
     接下来我们需要完成两个核心任务：
    </p>
    <ol>
     <li>
      <strong>
       定义资源文件格式
      </strong>
      ，确保资源内容易于管理、解析和加载。
     </li>
     <li>
      <strong>
       设计内存管理机制
      </strong>
      ，确保资源系统具备类似虚拟内存的能力，能够动态加载和卸载资源，同时允许我们设定固定的工作集大小，使游戏在有限内存下依然能够运行流畅。
     </li>
    </ol>
    <p>
     我们预计
     <strong>
      内存管理
     </strong>
     将是最大的挑战，而资源文件格式的设计相对简单。因此，我们会优先将内存管理方案设计好，然后再着手定义资源文件的格式。
    </p>
    <p>
     最终目标是确保：
    </p>
    <ul>
     <li>
      无论游戏资源总量多大，内存占用始终可控。
     </li>
     <li>
      动态加载和释放资源，保证游戏流畅性。
     </li>
     <li>
      资源系统具备高扩展性和可控性，为未来的资源规模扩展做好准备。
     </li>
    </ul>
    <p>
     这是我们当前需要解决的两个核心问题，接下来我们将重点推进内存管理的开发工作，使资源系统具备更强的可扩展性和控制力。
    </p>
    <h2>
     <a id="_69">
     </a>
     重点关注资源文件
    </h2>
    <p>
     目前我们需要专注处理资源文件部分的问题，虽然它与内存管理有关联，但我们现在主要聚焦于
     <strong>
      资源文件的结构设计
     </strong>
     。我们目前的资源管理方式仍然比较零散，主要表现在：
    </p>
    <h4>
     <a id="_72">
     </a>
     <strong>
      当前资源存储方式的局限性
     </strong>
    </h4>
    <p>
     目前我们在磁盘上存放资源的方式非常原始，所有资源文件（如位图、音频等）都是
     <strong>
      独立的文件
     </strong>
     ，随意分布在磁盘的不同目录中。而游戏中的一小段代码负责将这些文件与特定的ID、标签或段关联起来。这种方式存在以下几个问题：
    </p>
    <ol>
     <li>
      <strong>
       文件分散
      </strong>
      ：资源在磁盘上的分布是随机的，导致游戏加载资源时需要频繁进行文件系统的访问，增加了I/O开销。
     </li>
     <li>
      <strong>
       难以分发
      </strong>
      ：如果我们需要将游戏分发出去，必须保证每个用户的文件路径和资源布局一致，否则游戏无法正常加载资源。
     </li>
     <li>
      <strong>
       缺乏统一管理
      </strong>
      ：目前的资源并没有形成一个统一的结构，游戏无法快速定位资源，也无法通过一个集中化的管理文件来索引所有资源。
     </li>
     <li>
      <strong>
       加载效率低
      </strong>
      ：由于资源是零散的，加载时需要多次打开和关闭文件，从而导致加载效率低下。
     </li>
    </ol>
    <h4>
     <a id="_79">
     </a>
     <strong>
      新的目标：统一的资源文件格式
     </strong>
    </h4>
    <p>
     为了彻底解决这些问题，我们的目标是将
     <strong>
      所有资源
     </strong>
     打包成
     <strong>
      一个统一的文件
     </strong>
     ，即一个
     <strong>
      资源包文件（Packed Asset File）
     </strong>
     ，其中包含：
    </p>
    <ul>
     <li>
      <strong>
       所有的位图、音频、模型等资源数据
      </strong>
      。
     </li>
     <li>
      <strong>
       资源的元数据（Metadata）
      </strong>
      ，包括资源的ID、标签、类型、尺寸等信息。
     </li>
     <li>
      <strong>
       资源的分区信息
      </strong>
      ，用于快速定位文件中某个资源的位置。
     </li>
    </ul>
    <p>
     通过设计这种资源包文件，我们可以使游戏在读取资源时只需要打开
     <strong>
      一个文件
     </strong>
     ，然后通过内部的索引快速定位到需要的资源数据，从而大幅提升资源加载的速度并简化资源分发流程。
    </p>
    <h4>
     <a id="_87">
     </a>
     <strong>
      资源文件的结构设计
     </strong>
    </h4>
    <p>
     我们打算将资源包文件设计成
     <strong>
      分区结构
     </strong>
     ，整体类似一个小型的虚拟文件系统，主要由以下几个部分组成：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        文件头（Header）
       </strong>
       ：
      </p>
      <ul>
       <li>
        包含资源包的基本信息，如文件版本号、资源数量、索引表的偏移量等。
       </li>
       <li>
        文件头是固定大小的，方便我们在游戏启动时快速解析。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        资源索引表（Asset Index Table）
       </strong>
       ：
      </p>
      <ul>
       <li>
        包含所有资源的元数据（如资源名称、ID、大小、类型、偏移量等）。
       </li>
       <li>
        游戏加载时只需要读取索引表，就能快速定位资源在文件中的位置。
       </li>
       <li>
        这部分是资源包中最重要的部分，直接决定了资源加载的速度和准确性。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        资源数据区（Asset Data Block）
       </strong>
       ：
      </p>
      <ul>
       <li>
        存放实际的资源数据，包括位图、音频、模型等二进制数据。
       </li>
       <li>
        每个资源在数据区中按照固定的偏移量存储，索引表会记录这些偏移量和大小信息。
       </li>
       <li>
        通过这种方式，游戏可以直接跳转到某个资源的位置，而无需逐个扫描所有文件。
       </li>
      </ul>
     </li>
    </ol>
    <h4>
     <a id="_104">
     </a>
     <strong>
      资源文件的优点
     </strong>
    </h4>
    <p>
     通过采用统一的资源文件格式，我们可以获得以下几个核心优点：
    </p>
    <h5>
     <a id="1__107">
     </a>
     1.
     <strong>
      提升加载效率
     </strong>
    </h5>
    <p>
     将所有资源集中打包成一个文件，游戏只需要打开该文件即可进行所有资源的读取，大大减少了文件系统的访问次数，从而提升加载速度。
    </p>
    <h5>
     <a id="2__110">
     </a>
     2.
     <strong>
      便于分发
     </strong>
    </h5>
    <p>
     游戏的所有资源都打包在一个文件中，当我们需要将游戏分发给玩家时，只需要分发一个资源包文件即可，无需担心文件路径或依赖文件丢失的问题。
    </p>
    <h5>
     <a id="3__113">
     </a>
     3.
     <strong>
      便于管理
     </strong>
    </h5>
    <p>
     在开发过程中，我们可以使用一个专用工具（Asset Packager）将所有资源打包成一个文件，并生成索引表，使资源管理更加系统化、可控化。
    </p>
    <h5>
     <a id="4__116">
     </a>
     4.
     <strong>
      方便扩展
     </strong>
    </h5>
    <p>
     如果将来需要添加新的资源，只需要重新打包生成新的资源包文件即可，无需修改游戏代码。
    </p>
    <h5>
     <a id="5__119">
     </a>
     5.
     <strong>
      减少内存碎片化
     </strong>
    </h5>
    <p>
     当游戏加载资源时，可以一次性加载大量数据并存入内存中，避免频繁的内存分配和释放，从而减少内存碎片化问题，提高内存利用率。
    </p>
    <hr/>
    <h4>
     <a id="_124">
     </a>
     <strong>
      资源加载流程
     </strong>
    </h4>
    <p>
     在游戏运行时，资源加载的流程将如下：
    </p>
    <ol>
     <li>
      <strong>
       打开资源包文件
      </strong>
      ，读取文件头和索引表。
     </li>
     <li>
      <strong>
       解析索引表
      </strong>
      ，将资源名称与内存地址进行映射。
     </li>
     <li>
      当游戏需要某个资源时，通过索引表
      <strong>
       快速定位
      </strong>
      资源在文件中的偏移量，并直接读取数据加载到内存。
     </li>
     <li>
      <strong>
       关闭文件
      </strong>
      ，释放文件句柄。
     </li>
    </ol>
    <p>
     由于游戏只打开一个文件，并通过索引表快速定位资源，这样可以大幅减少I/O操作，提高加载速度。
    </p>
    <hr/>
    <h4>
     <a id="_135">
     </a>
     <strong>
      预加载和动态加载
     </strong>
    </h4>
    <p>
     为了进一步优化内存利用率，我们还可以设计
     <strong>
      预加载
     </strong>
     和
     <strong>
      动态加载
     </strong>
     两种模式：
    </p>
    <ul>
     <li>
      <strong>
       预加载（Preloading）
      </strong>
      ：在游戏启动时，将部分核心资源（如UI、菜单、背景音乐等）预加载到内存中，保证基础功能的流畅性。
     </li>
     <li>
      <strong>
       动态加载（Streaming Load）
      </strong>
      ：在游戏运行过程中，根据游戏场景的需求，动态加载当前需要的资源并卸载不需要的资源，确保内存使用处于合理范围。
     </li>
    </ul>
    <p>
     这种设计类似于操作系统的虚拟内存机制，使得游戏能够加载超出物理内存容量的资源集，而不会导致内存溢出。
    </p>
    <hr/>
    <h4>
     <a id="_145">
     </a>
     <strong>
      未来的优化方向
     </strong>
    </h4>
    <p>
     在完成资源文件的基本设计后，我们还计划对其进行以下优化：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        压缩资源
       </strong>
       ：
      </p>
      <ul>
       <li>
        将资源数据进行无损或有损压缩，减少资源文件的体积，提高加载速度。
       </li>
       <li>
        在加载资源时实时解压，保持资源质量。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        异步加载
       </strong>
       ：
      </p>
      <ul>
       <li>
        将资源加载过程放入单独的线程中，避免阻塞游戏主线程，提高流畅性。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        分层加载
       </strong>
       ：
      </p>
      <ul>
       <li>
        根据资源的优先级（如UI资源优先、背景资源次之），分层加载资源，保证关键内容优先可用。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        热更新支持
       </strong>
       ：
      </p>
      <ul>
       <li>
        将资源包文件拆分成多个小文件，使游戏可以通过网络更新某个资源包文件，而无需重新下载整个游戏。
       </li>
      </ul>
     </li>
    </ol>
    <hr/>

    <h2>
     <a id="_179">
     </a>
     类型和标签将游戏代码与资源表解耦
    </h2>
    <p>
     我们目前正在解决的核心问题是
     <strong>
      资源匹配机制
     </strong>
     ，即如何让
     <strong>
      游戏代码
     </strong>
     通过
     <strong>
      类型（Type）
      <strong>
       和
      </strong>
      标签（Tag）
      <strong>
       来请求资源，而不直接依赖于具体的资源文件，从而实现
      </strong>
      资源与游戏逻辑的解耦
     </strong>
     。
    </p>
    <hr/>
    <h4>
     <a id="_184">
     </a>
     <strong>
      游戏资源匹配机制的核心思想
     </strong>
    </h4>
    <p>
     在我们的游戏设计中，
     <strong>
      游戏代码
     </strong>
     在处理资源时只关心
     <strong>
      类型（Type）
      <strong>
       和
      </strong>
      标签（Tag）
     </strong>
     ，而不直接操作具体的资源文件（如位图、音频等）。这种方式的核心思想是：
    </p>
    <ul>
     <li>
      <strong>
       游戏代码只描述需求
      </strong>
      ：游戏代码不需要关心资源的具体文件是什么，也不需要知道该资源在哪里，它只需要描述
      <strong>
       想要的资源类型
      </strong>
      以及
      <strong>
       资源的特征
      </strong>
      。
     </li>
     <li>
      <strong>
       资源系统负责匹配
      </strong>
      ：资源系统根据游戏代码提供的
      <strong>
       类型
      </strong>
      和
      <strong>
       标签
      </strong>
      信息，从资源表（Asset Table）中找到
      <strong>
       最符合需求
      </strong>
      的资源并返回给游戏代码。
     </li>
     <li>
      <strong>
       解耦
      </strong>
      ：游戏代码和资源文件是完全解耦的，这意味着：
      <ol>
       <li>
        <strong>
         游戏代码不需要因为新增资源而修改
        </strong>
        。
       </li>
       <li>
        <strong>
         艺术家可以随时增加、调整或替换资源，而无需修改游戏代码
        </strong>
        。
       </li>
       <li>
        <strong>
         游戏内容可以更加动态
        </strong>
        ，只需要调整资源文件即可改变游戏内容，而无需重新编译代码。
       </li>
      </ol>
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="_196">
     </a>
     <strong>
      资源匹配机制的工作方式
     </strong>
    </h4>
    <h5>
     <a id="1_Type_197">
     </a>
     1.
     <strong>
      资源类型（Type）
     </strong>
    </h5>
    <p>
     <strong>
      资源类型
     </strong>
     是指游戏代码需要的
     <strong>
      资源类别
     </strong>
     ，例如：
    </p>
    <ul>
     <li>
      <strong>
       英雄角色
      </strong>
      （Hero）
     </li>
     <li>
      <strong>
       武器
      </strong>
      （Weapon）
     </li>
     <li>
      <strong>
       背景音乐
      </strong>
      （Background Music）
     </li>
     <li>
      <strong>
       UI按钮
      </strong>
      （UI Button）
     </li>
    </ul>
    <p>
     当游戏代码请求资源时，它首先会指定
     <strong>
      资源类型
     </strong>
     ，表示需要的是哪类资源。
    </p>
    <p>
     例如：
    </p>
    <pre><code class="prism language-c++">RequestAsset(Type_Hero)
</code></pre>
    <p>
     表示游戏需要一个
     <strong>
      英雄角色
     </strong>
     的资源，但不关心具体是哪个英雄。
    </p>
    <hr/>
    <h5>
     <a id="2_Tag_214">
     </a>
     2.
     <strong>
      资源标签（Tag）
     </strong>
    </h5>
    <p>
     <strong>
      资源标签
     </strong>
     是对资源的特征描述，它用来进一步限定所需资源的具体特征。
    </p>
    <p>
     比如在请求
     <strong>
      英雄角色
     </strong>
     时，可能需要指定以下标签：
    </p>
    <ul>
     <li>
      <strong>
       性别
      </strong>
      ：男(Male)、女(Female)、非二元(Non-binary)等。
     </li>
     <li>
      <strong>
       年龄
      </strong>
      ：儿童(4岁)、青年(20岁)、中年(50岁)、老年(80岁)等。
     </li>
     <li>
      <strong>
       性取向
      </strong>
      ：异性恋(Straight)、同性恋(Gay)、双性恋(Bisexual)等。
     </li>
     <li>
      <strong>
       头发颜色
      </strong>
      ：金发(Blonde)、黑发(Black Hair)、红发(Red Hair)等。
     </li>
    </ul>
    <p>
     请求示例：
    </p>
    <pre><code class="prism language-c++">RequestAsset(Type_Hero, Tag_Gender_Male, Tag_Age_20, Tag_HairColor_Blonde)
</code></pre>
    <p>
     此时游戏引擎会返回一个
     <strong>
      尽量符合这些标签的英雄资源
     </strong>
     。
    </p>
    <hr/>
    <h5>
     <a id="3__231">
     </a>
     3.
     <strong>
      模糊匹配机制
     </strong>
    </h5>
    <p>
     由于资源文件并不总是能完全匹配所有标签，因此我们采用
     <strong>
      模糊匹配机制
     </strong>
     ，即
     <strong>
      尽量返回最符合条件的资源
     </strong>
     ，而不是强制匹配所有标签。
    </p>
    <p>
     例如：
    </p>
    <ul>
     <li>
      游戏代码请求：
      <code>
       RequestAsset(Type_Hero, Tag_Age_20)
      </code>
     </li>
     <li>
      资源表中没有20岁的英雄，但有18岁和22岁的英雄。
     </li>
     <li>
      <strong>
       资源系统会优先选择最接近20岁的资源
      </strong>
      返回。
     </li>
    </ul>
    <p>
     这就是
     <strong>
      模糊匹配机制
     </strong>
     的核心，它允许我们：
    </p>
    <ul>
     <li>
      <strong>
       避免资源缺失导致游戏崩溃
      </strong>
      。
     </li>
     <li>
      <strong>
       允许艺术家不断新增资源而不影响游戏运行
      </strong>
      。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="_245">
     </a>
     <strong>
      资源表的结构
     </strong>
    </h4>
    <p>
     为了支持这种匹配机制，我们需要设计一个**资源表（Asset Table）**来存储所有资源的元数据。该资源表的核心结构如下：
    </p>
    <h5>
     <a id="1_Asset_Entry_248">
     </a>

     </strong>

     </strong>

      <strong>
       资源ID

     </li>

      <strong>
       资源类型

      <strong>
       资源路径
      </strong>
      ：该资源在磁盘中的位置。
     </li>
     <li>
      <strong>
       标签列表
      </strong>
      ：该资源所拥有的所有标签。
     </li>

    <hr/>
    <h4>
     <a id="_276">
     </a>
     <strong>
      资源匹配的具体过程
     </strong>
    </h4>
    <p>
     当游戏代码发出一个资源请求时，资源系统的匹配过程如下：
    </p>
    <ol>
     <li>
      <strong>
       读取资源表
      </strong>
      ：加载资源表中的所有资源项。
     </li>
     <li>
      <strong>
       过滤资源类型
      </strong>
      ：根据请求的
      <strong>
       资源类型
      </strong>
      ，过滤出所有符合该类型的资源。
     </li>
     <li>
      <strong>
       计算匹配分数
      </strong>
      ：遍历所有符合类型的资源，对每个资源计算与请求标签的匹配程度（匹配分数）。
     </li>
     <li>
      <strong>
       选择最佳匹配
      </strong>
      ：根据匹配分数，选择
      <strong>
       最接近
      </strong>
      请求条件的资源并返回。
     </li>
    </ol>
    <hr/>
    <h4>
     <a id="_286">
     </a>
     <strong>
      匹配分数计算
     </strong>
    </h4>
    <p>
     为了确定最符合请求的资源，我们采用**匹配分数（Matching Score）**的方式，计算公式类似：
    </p>
    <pre><code class="prism language-plaintext">匹配分数 = 满足的标签数量 / 总标签数量
</code></pre>
    <p>
     例如：
     <br/>
     请求：
     <code>
      RequestAsset(Type_Hero, Tag_Gender_Male, Tag_Age_20)
     </code>
    </p>
    <p>
     资源A：
     <code>
      Male, 20岁
     </code>
     → 匹配分数 = 2/2 = 100%
     <br/>
     资源B：
     <code>
      Male, 25岁
     </code>
     → 匹配分数 = 1/2 = 50%
     <br/>
     资源C：
     <code>
      Female, 20岁
     </code>
     → 匹配分数 = 1/2 = 50%
    </p>
    <p>
     最终资源A的匹配分数最高，因此它会被优先返回。
    </p>
    <hr/>
    <h4>
     <a id="_303">
     </a>
     <strong>
      动态更新资源表
     </strong>
    </h4>
    <p>
     最大优势是
     <strong>
      资源表可以动态更新
     </strong>
     ，而无需修改游戏代码。例如：
    </p>
    <ol>
     <li>
      <strong>
       艺术家增加新资源
      </strong>
      ：
      <ul>
       <li>
        艺术家绘制一个全新的

    <h2>
     <a id="_329">
     </a>
     复习资源系统代码
    </h2>
    <p>
     我们现在的目标是将游戏中的资源数据以一种
     <strong>
      有结构、易加载、可扩展
     </strong>
     的方式存储到磁盘上，并在游戏运行时快速加载这些数据，从而实现
     <strong>
      游戏代码与资源数据的解耦
     </strong>
     。为此，我们需要从数据结构的设计开始，确定
     <strong>
      资源类型（Asset Type）
     </strong>
     、**资源标签（Asset Tag）
     <strong>
      以及
     </strong>
     资源本体（Asset）**之间的映射关系，并将这些数据打包成一个统一的文件格式，使游戏能够高效加载和使用这些资源。
    </p>
    <hr/>
    <h4>
     <a id="_334">
     </a>
     <strong>
      数据的存储结构设计
     </strong>
    </h4>
    <p>
     我们在存储数据时，采用的是一种
     <strong>
      层级结构
     </strong>
     ，这种结构的核心思想是：
    </p>
    <ul>
     <li>
      <strong>
       最底层是资源本体（Asset）
      </strong>
      ，即具体的图像、音频、模型等数据。
     </li>
     <li>
      <strong>
       上层是资源标签（Tag）
      </strong>
      ，用来描述资源的属性，比如性别、年龄、颜色等。
     </li>
     <li>
      <strong>
       最顶层是资源类型（Asset Type）
      </strong>
      ，用来对资源进行分类，比如英雄（Hero）、阴影（Shadow）、武器（Weapon）等。
     </li>
    </ul>
    <p>
     通过这种
     <strong>
      层级结构
     </strong>
     ，我们就可以将
     <strong>
      游戏代码
     </strong>
     和
     <strong>
      资源数据
     </strong>
     解耦，让游戏代码只关心
     <strong>
      资源类型和标签
     </strong>
     ，而不关心资源本体，从而使资源可以随时更新而无需修改游戏代码。
    </p>
    <hr/>
    <h4>
     <a id="Asset_Type_344">
     </a>
     <strong>
      资源类型（Asset Type）的定义
     </strong>
    </h4>
    <p>
     首先，我们需要定义
     <strong>
      资源类型（Asset Type）
     </strong>
     ，它是最上层的概念，用来表示游戏中某一类资源。
    </p>
    <p>
     例如：
    </p>
    <ul>
     <li>
      <strong>
       Shadow
      </strong>
      ：表示阴影资源，可能用于角色、物品的投影。
     </li>
     <li>
      <strong>
       Head
      </strong>
      ：表示角色的头部资源。
     </li>
     <li>
      <strong>
       Cape
      </strong>
      ：表示角色的披风资源。
     </li>
     <li>
      <strong>
       Torso
      </strong>
      ：表示角色的身体资源。
     </li>
    </ul>
    <p>
     我们使用**枚举（Enum）**来表示这些资源类型，使得游戏代码在请求资源时，可以直接使用枚举，而无需关心具体的资源文件。
    </p>
    <p>
     示例：
    </p>
    <pre><code class="prism language-c++">enum AssetType
{
    AssetType_Shadow,
    AssetType_Head,
    AssetType_Cape,
    AssetType_Torso,
    AssetType_Count
};
</code></pre>
    <p>
     这里
     <code>
      AssetType_Count
     </code>
     只是用来记录总数，方便遍历所有类型。
    </p>
    <hr/>
    <h4>
     <a id="_371">
     </a>
     <strong>
      资源类型的索引表
     </strong>
    </h4>
    <p>
     在磁盘文件中，所有的资源都会按照
     <strong>
      资源类型
     </strong>
     进行索引。
     <br/>
     例如：
    </p>
    <table>
     <thead>
      <tr>
       <th>
        索引 (Index)
       </th>
       <th>
        资源类型 (Asset Type)
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        0
       </td>
       <td>
        Shadow
       </td>
      </tr>
      <tr>
       <td>
        1
       </td>
       <td>
        Head
       </td>
      </tr>
      <tr>
       <td>
        2
       </td>
       <td>
        Cape
       </td>
      </tr>
      <tr>
       <td>
        3
       </td>
       <td>
        Torso
       </td>
      </tr>
     </tbody>
    </table>
    <p>
     这种结构是**扁平化（Flat）**的，即所有资源都是连续存储在磁盘上的。
     <br/>
     <strong>
      游戏代码只需要知道索引
     </strong>
     ，就可以快速找到该类型的资源。
    </p>
    <hr/>
    <h4>
     <a id="Asset_Tag_387">
     </a>
     <strong>
      资源标签（Asset Tag）的定义
     </strong>
    </h4>
    <p>
     **资源标签（Asset Tag）**是描述资源特征的属性，例如：
    </p>
    <ul>
     <li>
      <strong>
       性别
      </strong>
      ：男（Male）、女（Female）、非二元（Non-binary）。
     </li>
     <li>
      <strong>
       年龄
      </strong>
      ：4岁、20岁、50岁、80岁。
     </li>
     <li>
      <strong>
       颜色
      </strong>
      ：黑发、金发、红发。
     </li>
     <li>
      <strong>
       风格
      </strong>
      ：像素风、写实风、漫画风。
     </li>
    </ul>
    <p>
     标签的作用是
     <strong>
      在同一资源类型中进一步区分资源
     </strong>
     ，确保游戏代码可以通过标签找到更符合需求的资源。
    </p>
    <h5>
     <a id="_396">
     </a>
     <strong>
      标签的结构
     </strong>
    </h5>
    <p>
     每个资源都拥有一个
     <strong>
      标签数组（Tag Array）
     </strong>
     ，其中包含：
    </p>
    <ul>
     <li>
      <strong>
       标签ID
      </strong>
      ：表示该标签的唯一标识。
     </li>
     <li>
      <strong>
       标签值
      </strong>
      ：表示该标签的具体取值。
     </li>
    </ul>
    <p>
     示例：
    </p>
    <pre><code class="prism language-c++">enum AssetTag
{
    AssetTag_Gender,
    AssetTag_Age,
    AssetTag_HairColor,
    AssetTag_Style,
    AssetTag_Count
};
</code></pre>
    <p>
     <strong>
      标签的取值
     </strong>
     可以是动态的，资源表中只记录标签ID和取值，而不存储标签名称。
    </p>
    <p>
     例如：
    </p>
    <table>
     <thead>
      <tr>
       <th>
        标签ID
       </th>
       <th>
        标签值
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        0
       </td>
       <td>
        Male
       </td>
      </tr>
      <tr>
       <td>
        1
       </td>
       <td>
        20岁
       </td>
      </tr>
      <tr>
       <td>
        2
       </td>
       <td>
        金发
       </td>
      </tr>
      <tr>
       <td>
        3
       </td>
       <td>
        像素风
       </td>
      </tr>
     </tbody>
    </table>
    <hr/>
    <h4>
     <a id="Asset_425">
     </a>
     <strong>
      资源本体（Asset）的定义
     </strong>
    </h4>
    <p>
     <strong>
      资源本体（Asset）
      <strong>
       是指
      </strong>
      实际存储的文件
     </strong>
     ，比如图像、音频、模型等。
    </p>
    <p>
     每个
     <strong>
      资源
     </strong>
     都具备以下属性：
    </p>
    <ul>
     <li>
      <strong>
       资源ID
      </strong>
      ：该资源的唯一标识符。
     </li>
     <li>
      <strong>
       资源路径
      </strong>
      ：该资源在磁盘中的文件路径。
     </li>
     <li>
      <strong>
       资源类型
      </strong>
      ：该资源对应的
      <code>
       AssetType
      </code>
      。
     </li>
     <li>
      <strong>
       资源标签
      </strong>
      ：该资源的所有标签及取值。
     </li>
    </ul>
    <p>
     示例结构：
    </p>
    <pre><code class="prism language-c++">struct Asset
{
    uint32 ID;
    char* FilePath;
    AssetType Type;
    Tag Tags[MAX_TAGS];
};
</code></pre>
    <hr/>
    <h4>
     <a id="_447">
     </a>
     <strong>
      资源在磁盘中的存储形式
     </strong>
    </h4>
    <p>
     我们需要将所有资源信息存储到一个**打包文件（Packed File）**中，便于游戏快速加载。
    </p>
    <h5>
     <a id="_450">
     </a>
     <strong>
      文件结构
     </strong>
    </h5>
    <pre><code class="prism language-plaintext">[资源类型表]
[资源索引表]
[资源标签表]
[资源文件路径]
[资源数据]
</code></pre>
    <p>
     举个例子，假设我们有三个资源：
    </p>
    <ol>
     <li>
      20岁男性金发英雄（Hero）
     </li>
     <li>
      50岁女性红发英雄（Hero）
     </li>
     <li>
      20岁男性金发阴影（Shadow）
     </li>
    </ol>
    <p>
     存储形式大致如下：
    </p>
    <pre><code class="prism language-plaintext">AssetType: Hero
AssetID: 1
Tags: {Gender: Male, Age: 20, HairColor: Blonde}
FilePath: assets/hero1.bmp

AssetType: Hero
AssetID: 2
Tags: {Gender: Female, Age: 50, HairColor: Red}
FilePath: assets/hero2.bmp

AssetType: Shadow
AssetID: 3
Tags: {Gender: Male, Age: 20}
FilePath: assets/shadow1.bmp
</code></pre>
    <p>
     游戏代码只需调用：
    </p>
    <pre><code class="prism language-c++">RequestAsset(AssetType_Hero, Tag_Gender_Male, Tag_Age_20)
</code></pre>
    <p>
     资源系统会自动找到
     <strong>
      最匹配的资源
     </strong>
     返回。
    </p>
    <hr/>
    <h4>
     <a id="_490">
     </a>
     <strong>
      加载资源时的匹配过程
     </strong>
    </h4>
    <p>
     当游戏需要加载资源时，匹配过程如下：
    </p>
    <ol>
     <li>
      <strong>
       读取资源表
      </strong>
      ：将所有资源的类型、标签、路径加载到内存。
     </li>
     <li>
      <strong>
       过滤资源类型
      </strong>
      ：根据请求的
      <code>
       AssetType
      </code>
      筛选所有该类型的资源。
     </li>
     <li>
      <strong>
       计算匹配分数
      </strong>
      ：遍历该类型下的资源，计算与标签的匹配程度。
     </li>
     <li>
      <strong>
       选择最优资源
      </strong>
      ：返回匹配分数最高的资源。
     </li>
    </ol>
    <p>
     <strong>
      匹配分数计算公式
     </strong>
     ：
    </p>
    <pre><code class="prism language-plaintext">匹配分数 = 匹配的标签数量 / 总标签数量
</code></pre>
    <p>
     例如：
     <br/>
     请求：
     <code>
      Hero + Male + 20岁 + 金发
     </code>
     <br/>
     资源A：
     <code>
      Hero + Male + 20岁 + 金发
     </code>
     → 匹配分数 = 4/4 = 100%
     <br/>
     资源B：
     <code>
      Hero + Male + 25岁 + 金发
     </code>
     → 匹配分数 = 3/4 = 75%
    </p>
    <p>
     最终返回匹配分数最高的资源。
    </p>

    <h2>
     <a id="_530">
     </a>
     资源类型引用一个连续范围的实际资源，这些资源存储在单独的表中
    </h2>
    <p>
     在设计和存储游戏资源（Asset）的过程中，我们需要考虑
     <strong>
      如何高效地组织和引用资源
     </strong>
     ，使得游戏在加载资源时能够快速找到所需的内容，同时保证存储结构简洁且易于扩展。为此，我们采用了一种类似
     <strong>
      数据库的设计思想
     </strong>
     ，通过
     <strong>
      资产类型（Asset Type）
     </strong>
     、**资产表（Asset Table）
     <strong>
      以及
     </strong>
     子地址索引（Sub Addressing）**来实现资源的高效管理和加载。
    </p>
    <hr/>
    <h3>
     <a id="Asset_Type_535">
     </a>
     <strong>
      资源类型（Asset Type）与子地址索引
     </strong>
    </h3>
    <p>
     首先，我们定义了一个
     <strong>
      资源类型表（Asset Type Table）
     </strong>
     ，该表的作用是
     <strong>
      将资源按照类型进行分类
     </strong>
     ，例如：
    </p>
    <ul>
     <li>
      <strong>
       Shadow
      </strong>
      ：表示阴影资源。
     </li>
     <li>
      <strong>
       Head
      </strong>
      ：表示角色的头部资源。
     </li>
     <li>
      <strong>
       Cape
      </strong>
      ：表示角色的披风资源。
     </li>
     <li>
      <strong>
       Torso
      </strong>
      ：表示角色的身体资源。
     </li>
    </ul>
    <p>
     这个
     <strong>
      资源类型表
     </strong>
     本质上是一个索引表，每个资源类型都有一个唯一的索引编号。例如：
    </p>
    <pre><code class="prism language-plaintext">索引    资源类型
------------------
0      Shadow
1      Head
2      Cape
3      Torso
</code></pre>
    <p>
     在游戏加载资源时，程序只需要知道资源类型编号，就可以通过该编号快速找到
     <strong>
      该类型资源在资源表中的起始地址
     </strong>
     以及
     <strong>
      资源数量
     </strong>
     。
    </p>
    <hr/>
    <h3>
     <a id="Asset_Table_555">
     </a>
     <strong>
      资产表（Asset Table）的作用
     </strong>
    </h3>
    <p>
     我们在磁盘上存储的所有资源数据被打包成一个
     <strong>
      资源表（Asset Table）
     </strong>
     ，该表存储的是
     <strong>
      所有资源的具体信息
     </strong>
     ，包括：
    </p>
    <ul>
     <li>
      <strong>
       资源编号
      </strong>
      （Asset Index）。
     </li>
     <li>
      <strong>
       资源文件地址
      </strong>
      （File Path 或 Offset）。
     </li>
     <li>
      <strong>
       资源标签
      </strong>
      （Tag）。
     </li>
     <li>
      <strong>
       资源大小
      </strong>
      （Size）。
     </li>
    </ul>
    <p>
     资源表在磁盘上的形式类似于：
    </p>
    <pre><code class="prism language-plaintext">索引    文件地址       标签信息     文件大小
-------------------------------------------
0       offset_0     [Tag1,Tag2]   2MB
1       offset_1     [Tag3,Tag4]   3MB
2       offset_2     [Tag1,Tag5]   1MB
3       offset_3     [Tag2,Tag3]   4MB
</code></pre>
    <p>
     所有资源以
     <strong>
      连续存储
     </strong>
     的方式存在磁盘中，通过
     <strong>
      索引
     </strong>
     快速定位资源的起始地址和大小，实现高效加载。
    </p>
    <hr/>
    <h3>
     <a id="Sub_Addressing_575">
     </a>
     <strong>
      子地址索引（Sub Addressing）的作用
     </strong>
    </h3>
    <p>
     由于我们采用的是
     <strong>
      资源类型分类存储
     </strong>
     的方式，即
     <strong>
      同一类型的资源集中存放
     </strong>
     ，因此我们需要一种方法来
     <strong>
      记录每种资源类型在资源表中的起始位置和数量
     </strong>
     。
    </p>
    <h4>
     <a id="_578">
     </a>
     <strong>
      为什么需要子地址索引
     </strong>
    </h4>
    <p>
     假设我们有以下资源：
    </p>
    <pre><code class="prism language-plaintext">资源类型    资源编号
-------------------
Shadow      0
Shadow      1
Shadow      2
Head        3
Head        4
Cape        5
Cape        6
Cape        7
Torso       8
</code></pre>
    <p>
     如果游戏需要加载
     <strong>
      所有 Shadow 资源
     </strong>
     ，我们希望通过
     <strong>
      资源类型表
     </strong>
     直接找到
     <strong>
      Shadow 类型的起始地址
     </strong>
     ，并知道该类型资源的数量。
    </p>
    <hr/>
    <h3>
     <a id="_597">
     </a>
     <strong>
      子地址索引的存储形式
     </strong>
    </h3>
    <p>
     我们在
     <strong>
      资源类型表
     </strong>
     中添加两个额外字段：
    </p>
    <ul>
     <li>
      <strong>
       起始索引（First Asset Index）
      </strong>
      ：表示该资源类型的
      <strong>
       第一个资源
      </strong>
      在资源表中的索引。
     </li>
     <li>
      <strong>
       资源数量（Asset Count）
      </strong>
      ：表示该资源类型的资源数量。
     </li>
    </ul>
    <p>
     最终的
     <strong>
      资源类型表
     </strong>
     如下：
    </p>
    <pre><code class="prism language-plaintext">资源类型     起始索引    资源数量
-------------------------------
Shadow       0          3
Head         3          2
Cape         5          3
Torso        8          1
</code></pre>
    <p>
     这意味着：
    </p>
    <ul>
     <li>
      <strong>
       Shadow
      </strong>
      类型资源从索引0开始，有3个资源。
     </li>
     <li>
      <strong>
       Head
      </strong>
      类型资源从索引3开始，有2个资源。
     </li>
     <li>
      <strong>
       Cape
      </strong>
      类型资源从索引5开始，有3个资源。
     </li>
     <li>
      <strong>
       Torso
      </strong>
      类型资源从索引8开始，有1个资源。
     </li>
    </ul>
    <p>
     游戏只需要根据资源类型查找
     <strong>
      起始索引
     </strong>
     和
     <strong>
      数量
     </strong>
     ，就可以快速找到所有同类型资源。
    </p>
    <hr/>
    <h3>
     <a id="_622">
     </a>
     <strong>
      资源的连续存储
     </strong>
    </h3>
    <p>
     这里的核心设计是
     <strong>
      同类型的资源在磁盘中是连续存储的
     </strong>
     ，即：
    </p>
    <pre><code class="prism language-plaintext">索引    资源类型    文件地址
--------------------------
0       Shadow     offset_0
1       Shadow     offset_1
2       Shadow     offset_2
3       Head       offset_3
4       Head       offset_4
5       Cape       offset_5
6       Cape       offset_6
7       Cape       offset_7
8       Torso      offset_8
</code></pre>
    <p>
     这带来了几个重要优势：
    </p>
    <ol>
     <li>
      <strong>
       同类型资源连续存储
      </strong>
      ，保证加载速度更快。
     </li>
     <li>
      <strong>
       通过索引+数量即可快速找到所有同类型资源
      </strong>
      。
     </li>
     <li>
      <strong>
       资源表结构紧凑，存储效率更高
      </strong>
      。
     </li>
    </ol>

     <a id="_644">

    <h4>
     <a id="_647">
     </a>
     <strong>
      多个文件的加载问题
     </strong>
    </h4>
    <p>
     如果我们直接使用
     <strong>
      连续存储
     </strong>
     的方式，多个文件将导致问题：
    </p>
    <ul>
     <li>
      <strong>
       文件1
      </strong>
      存储
      <code>
       Shadow, Head
      </code>
      。
     </li>
     <li>
      <strong>
       文件2
      </strong>
      存储
      <code>

      </code>
      。
     </li>
     <li>

      <code>

      </code>

     </li>
    </ul>
    <h4>
     <a id="File_Offset_653">
     </a>
     <strong>
      解决方法：文件偏移地址（File Offset）
     </strong>
    </h4>
    <p>
     为了解决这个问题，我们将
     <strong>
      文件偏移地址
     </strong>
     加入资源表：
    </p>
    <pre><code class="prism language-plaintext">索引    资源类型    文件索引    文件偏移
--------------------------------------
0       Shadow     file_1     offset_0
1       Shadow     file_1     offset_1
2       Shadow     file_1     offset_2
3       Head       file_1     offset_3
4       Head       file_1     offset_4
5       Cape       file_2     offset_0
6       Cape       file_2     offset_1
7       Cape       file_2     offset_2
8       Torso      file_2     offset_3
</code></pre>
    <p>
     这样，游戏在加载资源时：
    </p>
    <ul>
     <li>
      <strong>
       先通过资源类型表
      </strong>
      找到起始索引和数量。
     </li>
     <li>
      <strong>
       再通过资源表
      </strong>
      找到文件索引和文件偏移。
     </li>
     <li>
      <strong>
       最后打开文件，按偏移读取资源数据
      </strong>
      。
     </li>
    </ul>

    <h2>
     <a id="_700">
     </a>
     将资源系统通用化，允许在不同的资源文件中拥有相同资源类型的多个范围
    </h2>
    <p>
     当我们处理资产文件时，我们需要考虑在磁盘上和内存中对资产数据的组织方式。首先，我们在磁盘上的资产表实际上有点类似数据库，因为它以结构化的方式存储数据。我们在这里将资产分为不同的资产类型，并通过表格进行索引管理。在资产类型表中，每个条目代表一种资产类型，并指向一个连续的资产范围。例如，如果我们有一种资产类型，其资产索引范围为4到6，则该条目会存储第一个资产的索引以及该类型包含的资产数量。
    </p>
    <p>
     这种连续索引的方式在只有一个资产文件时是非常方便的，因为我们可以确保相同类型的所有资产在磁盘上是连续存储的，这样可以提高加载效率。然而，当我们引入多个资产文件时，例如游戏的DLC或扩展包时，这种方式就会出现问题。
    </p>
    <p>
     假设我们最初的资产文件中包含了一种名为“巧克力酒”的资产类型，并且该资产的索引范围为16到22，共有7个条目。当我们发布一个新的DLC时，其中又增加了930种新的巧克力酒资产，如果我们继续沿用连续索引的方式，就无法表示这些新的资产，因为它们不再与原文件中的资产连续。
    </p>
    <p>
     为了解决这个问题，我们需要在磁盘上的资产类型表中存储更多信息，使其支持合并加载。在原始资产文件中，每个条目仅包含索引范围信息，而在新的结构中，每个条目需要记录它所表示的具体资产类型（例如巧克力酒）的枚举值，以及该类型在文件中的索引范围。这样，当我们加载多个资产文件时，可以在内存中合并同一类型的资产，使其形成连续的内存结构，方便游戏使用。
    </p>
    <p>
     这种合并加载的过程是这样的：
    </p>
    <ul>
     <li>
      加载第一个资产文件时，将其资产类型表和资产数据加载到内存中。
     </li>
     <li>
      加载第二个资产文件时，检查资产类型表中的枚举值，如果发现与内存中已有的资产类型匹配（如“巧克力酒”），就将其资产数据追加到内存中已存在的同类型资产之后。
     </li>
     <li>
      如果枚举值是新的，则创建新的资产类型条目并添加资产数据。
     </li>
    </ul>
    <p>
     这种方式的优点是：
    </p>
    <ol>
     <li>
      <strong>
       支持扩展包
      </strong>
      ：我们可以在不修改原始资产文件的情况下直接加载新的资产文件并合并数据。
     </li>
     <li>
      <strong>
       内存结构优化
      </strong>
      ：在内存中我们仍然可以保持同类型的资产连续存储，以便高效地访问和渲染。
     </li>
     <li>
      <strong>
       增量加载
      </strong>
      ：当我们需要增量更新资产文件时（例如添加新内容或修复旧内容），只需要将新资产追加到文件末尾，无需重写整个文件。
     </li>
    </ol>
    <p>
     此外，我们还需要处理资产的具体信息。每个资产都有一个信息结构体（Info Struct），用于存储该资产的具体数据，例如：
    </p>
    <ul>
     <li>
      <strong>
       位图
      </strong>
      ：宽度、高度、像素数据等。
     </li>
     <li>
      <strong>
       声音
      </strong>
      ：通道数、采样率、数据偏移等。
     </li>
     <li>
      <strong>
       通用数据
      </strong>
      ：数据块在文件中的偏移量、数据大小等。
     </li>
    </ul>
    <p>
     这些信息需要直接存储在磁盘上的资产文件中，并在加载时解析到内存中，便于游戏直接使用。我们还需要移除一些无用的信息，例如文件名，因为我们已经将所有数据打包到资产文件中，因此不再需要外部文件路径。
    </p>
    <p>
     关于标签（Tags），我们将其设计成简单的整数和浮点数的数组，每个资产都可以拥有一组标签，用于快速定位或筛选资产。例如：
    </p>
    <ul>
     <li>
      标签1：颜色类型（0=红色，1=蓝色等）。
     </li>
     <li>
      标签2：适用场景（0=室内，1=户外等）。
     </li>
     <li>
      标签3：品质等级（0=普通，1=稀有等）。
     </li>
    </ul>
    <p>
     在磁盘上，标签也是以二进制形式直接存储的，加载时无需解析或转换，可以直接映射到内存中使用。
    </p>
    <p>
     在处理加载流程时，我们需要特别注意的是：
    </p>
    <ul>
     <li>
      在磁盘上的资产类型表只作为临时数据，加载时会根据枚举值重新组合形成新的内存资产类型表。
     </li>
     <li>
      在内存中保证相同类型的资产连续存储，以提高访问效率。
     </li>
     <li>
      当出现多个资产文件（如DLC）时，需要合并相同类型的资产，使其在内存中保持连续。
     </li>
    </ul>
    <p>
     未来我们可能还会考虑对资产文件进行压缩，以减少文件大小。通常我们可以使用LZ压缩算法在保存时对数据进行压缩，在加载时解压缩，以达到最优的加载效率和存储空间利用率。
    </p>
    <p>
     接下来，我们需要设计一个标准的资产文件格式，以便存储这些数据，并编写加载代码，使其能够正确地合并加载多个资产文件。我们还需要确保加载过程中不会影响游戏的运行状态，因此我们可以先保留当前内存模拟的加载代码，待新格式加载流程完善后再替换。
    </p>
    <h2>
     <a id="_742">
     </a>
     编写一个生成虚拟资源文件的程序
    </h2>
    <p>
     我们现在需要做的一件重要的事情是，将之前在内存中模拟资产文件的代码拆分出来，单独创建一个生成资产包文件（Asset Pack File）的程序。这个新程序的唯一任务就是生成一个虚拟的资产包文件，以便我们进行加载和测试。这样可以确保我们的游戏在加载资产文件时，流程是完整且正确的，并且也有助于我们验证后续的加载逻辑。
    </p>
    <p>
     我们要做的第一步是在我们的构建脚本（CMakeLists.txt）中添加一个新的构建目标，即专门用来生成资产包文件的工具。这个工具的名称暂定为
     <strong>
      test_asset_builder.cpp
     </strong>
     ，它将是一个独立的可执行文件，用来生成测试用的资产文件。
    </p>
    <p>
     在构建脚本中，我们可以看到目前已有的多个构建目标，例如游戏的主可执行文件等。我们的目标是增加一个新的构建项，使其编译生成
     <strong>
      test_asset_builder.exe
     </strong>
     文件。我们不打算为它生成映射文件（map file）或其他调试文件，因为它只是一个测试工具，只需要最基本的可执行文件。
    </p>
    <h4>
     <a id="_749">
     </a>
     <strong>
      添加新的构建目标
     </strong>
    </h4>
    <p>
     我们在
     <code>
      test_asset_builder.cpp
     </code>
     文件中添加一行新的构建：

add_executable(test_asset "test_asset_builder.cpp")
</code></pre>

     何时使用库的一个例子
    </h2>
    <p>
     我们通常不使用库或引擎，几乎完全从零开始编写代码。这不仅是出于教育目的，同时也是为了避免在长期维护项目中遇到的依赖问题。在实际项目中，如果使用第三方库，可能会因为库停止维护、存在兼容性问题或引入新的bug，从而增加维护成本。因此，对于希望长期运行的代码，我们倾向于避免使用第三方库。但如果是临时测试、原型开发或其他不影响核心功能的场景，我们是愿意使用标准库的。
    </p>
    <p>
     在这里，我们决定创建一个
     <strong>
      测试用的资源打包工具
     </strong>
     ，该工具的作用是生成一个资源包文件，以供游戏加载测试使用。这个工具完全独立，不影响核心游戏的构建逻辑，因此可以放心地使用C标准库进行开发。
    </p>
    <p>
     然后，我们创建
     <code>
      test_asset_builder.cpp
     </code>
     ，并为其添加标准的
     <code>
      main
     </code>
     函数。此工具的作用是模拟游戏运行时打包的过程，只是这次打包的内容是提前定义的。这样我们就可以快速测试资源加载功能，同时也为将来创建资源打包工具提供参考。
    </p>
    <p>
     接下来，我们在该工具中使用
     <code>
      fopen
     </code>
     打开一个名为
     <code>
      test.hha
     </code>
     的文件，模式为
     <code>
      wb
     </code>

     <code>
      fwrite
     </code>
     、
     <code>

     </code>

     在生成资源包之前，我们需要整理一下资源数据结构。在游戏中，资源包括位图、声音、字体等，但在这里我们仅关心位图数据的存储。我们定义一个结构

    <p>
     在资源定义部分，我们直接从原游戏代码中提取资源加载的函数

     </code>
     、
     <code>
      BeginAssetType
     </code>

     <code>

    <h2>
     <a id="SlotID_832">
     </a>
     通过去除SlotID简化资源结构
    </h2>
    <p>
     我们目前正在对资源加载系统进行一个重要的架构调整，目的是简化代码结构，减少不必要的复杂性，提高运行时的效率。在之前的实现中，我们使用了一个所谓的
     <strong>
      Slot ID（槽位ID）
     </strong>
     的概念，用来表示游戏运行时资源的加载状态及对应的内存位置。但在当前的文件结构中，
     <strong>
      Slot ID
     </strong>
     似乎变得没有实际意义，因此我们决定将其移除，并通过更直接的方式来定位资源数据的位置，从而简化代码和数据结构的设计。
    </p>
    <hr/>
    <h4>
     <a id="_837">
     </a>
     <strong>
      问题的起源
     </strong>
    </h4>
    <p>
     在之前的实现中，我们有两个重要的数组：
    </p>
    <ul>
     <li>
      <strong>
       Bitmaps数组
      </strong>
      ：存储游戏中的所有位图资源（如图片、纹理等）。
     </li>
     <li>
      <strong>
       Sounds数组
      </strong>
      ：存储游戏中的所有声音资源（如背景音乐、音效等）。
     </li>
    </ul>
    <p>
     同时我们还为每个资源分配了一个
     <strong>
      Slot ID
     </strong>
     ，该
     <strong>
      Slot ID
     </strong>
     负责在运行时跟踪资源的加载状态，以及与内存中的特定位置相对应。
    </p>
    <p>
     但是在实际查看代码时我们发现，
     <strong>
      Slot ID
     </strong>
     的存在并没有明显的必要性。
    </p>
    <hr/>
    <h4>
     <a id="_Slot_ID__849">
     </a>
     <strong>
      为什么 Slot ID 没有必要？
     </strong>
    </h4>
    <p>
     我们意识到，资源本身（比如位图、声音）和其加载状态是两回事。资源本身应该有一个固定的索引，而不需要一个额外的 Slot ID。
    </p>
    <p>
     原先的做法是：
    </p>
    <ul>
     <li>
      <strong>
       Slot ID
      </strong>
      是独立于资源索引的，它指向内存中的某个位置。
     </li>
     <li>
      <strong>
       资源数组
      </strong>
      存储的是具体资源。
     </li>
     <li>
      <strong>
       加载状态数组
      </strong>
      存储的是加载状态，并通过 Slot ID 索引对应的资源。
     </li>
    </ul>
    <p>
     但仔细想想，我们完全可以直接用
     <strong>
      资源数组的索引
     </strong>
     代替 Slot ID，因为：
    </p>
    <ul>
     <li>
      资源索引本身就唯一且确定。
     </li>
     <li>
      在加载资源时，我们本来就是通过资源索引去获取资源内容的。
     </li>
     <li>
      <strong>
       Slot ID
      </strong>
      实际上是多余的，它只是做了一层无意义的映射。
     </li>
    </ul>
    <p>
     换句话说，
     <strong>
      Slot ID 本质上是资源索引的冗余表示
     </strong>
     ，而我们完全可以直接使用资源数组的索引来替代 Slot ID。
    </p>
    <hr/>
    <h4>
     <a id="_867">
     </a>
     <strong>
      解决方案
     </strong>
    </h4>
    <p>
     我们决定移除 Slot ID，并将所有资源（包括位图和声音）统一存放在同一个数组中，并通过
     <strong>
      资源索引
     </strong>
     直接访问数据。
    </p>
    <hr/>
    <h4>
     <a id="_872">
     </a>
     <strong>
      具体修改的内容
     </strong>
    </h4>
    <h5>
     <a id="1__Slot_ID_873">
     </a>
     <strong>
      1. 移除 Slot ID
     </strong>
    </h5>
    <p>
     我们首先将
     <strong>
      Slot ID
     </strong>
     的概念彻底移除，包括：
    </p>
    <ul>
     <li>
      将原本分开的
      <strong>
       Bitmap 数组
      </strong>
      和
      <strong>
       Sound 数组
      </strong>
      ，合并成一个
      <strong>
       资源数组
      </strong>
      (
      <code>
       Assets[]
      </code>
      )。
     </li>
     <li>
      将原本的 Slot ID 替换为资源索引，直接通过索引访问资源。
     </li>
     <li>
      将原来的
      <strong>
       加载状态数组
      </strong>
      与资源数组合并，避免冗余的内存访问。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="2__882">
     </a>
     <strong>
      2. 合并资源数组
     </strong>
    </h4>
    <p>
     我们决定将所有资源（位图、声音）合并成同一个数组，并通过联合体 (
     <code>
      union
     </code>
     ) 表示资源的具体类型。
    </p>
    <p>
     新的结构如下：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">struct</span> <span class="token class-name">asset</span>
<span class="token punctuation">{<!-- --></span>
    <span class="token keyword">union</span>
    <span class="token punctuation">{<!-- --></span>
        asset_bitmap_info Bitmap<span class="token punctuation">;</span>
        asset_sound_info Sound<span class="token punctuation">;</span>
    <span class="token punctuation">}</span><span class="token punctuation">;</span>
<span class="token punctuation">}</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     这意味着：
    </p>
    <ul>
     <li>
      每个资源无论是位图还是声音，都会存在同一个
      <code>
       Assets[]
      </code>
      数组中。
     </li>
     <li>
      通过联合体 (
      <code>
       union
      </code>
      ) 的方式区分资源的类型。
     </li>
    </ul>
    <p>
     例如：
    </p>
    <pre><code class="prism language-c">Assets<span class="token punctuation">[</span>Index<span class="token punctuation">]</span><span class="token punctuation">.</span>Bitmap
Assets<span class="token punctuation">[</span>Index<span class="token punctuation">]</span><span class="token punctuation">.</span>Sound
</code></pre>
    <p>
     无需再维护单独的
     <strong>
      Bitmaps 数组
     </strong>
     或
     <strong>
      Sounds 数组
     </strong>
     ，只需要维护一个统一的
     <strong>
      Assets 数组
     </strong>
     。
    </p>
    <hr/>
    <h4>
     <a id="3__BitmapsCount__SoundsCount_912">
     </a>
     <strong>
      3. 移除 BitmapsCount 和 SoundsCount
     </strong>
    </h4>
    <p>
     在旧的实现中，我们有两个计数器：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">int</span> BitmapCount<span class="token punctuation">;</span>
<span class="token keyword">int</span> SoundCount<span class="token punctuation">;</span>
</code></pre>
    <p>
     分别记录位图和声音资源的数量。
    </p>
    <p>
     但在新的实现中，所有资源都合并成同一个数组，因此我们只需要一个总的
     <strong>
      AssetCount
     </strong>
     即可：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">int</span> AssetCount<span class="token punctuation">;</span>
</code></pre>
    <p>
     通过
     <code>
      AssetCount
     </code>
     我们就能遍历所有资源，无需区分位图或声音。
    </p>
    <hr/>
    <h4>
     <a id="4__928">
     </a>
     <strong>
      4. 移除加载状态数组
     </strong>
    </h4>
    <p>
     在旧的实现中，我们有一个单独的
     <strong>
      加载状态数组
     </strong>
     ，通过 Slot ID 去索引资源的加载状态：
    </p>
    <pre><code class="prism language-c">slots<span class="token punctuation">[</span>SlotID<span class="token punctuation">]</span><span class="token punctuation">.</span>State
</code></pre>
    <p>
     但在新的实现中，加载状态完全可以直接附加在资源结构体上：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">struct</span> <span class="token class-name">asset</span>
<span class="token punctuation">{<!-- --></span>
    asset_state State<span class="token punctuation">;</span>
<span class="token punctuation">}</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     因此，加载状态数组也就不再需要了。
    </p>
    <hr/>
    <h4>
     <a id="5__944">
     </a>
     <strong>
      5. 资源的定位
     </strong>
    </h4>
    <p>
     在新的实现中，我们直接通过资源索引来访问资源，无需 Slot ID。
    </p>
    <p>
     旧的访问方式：
    </p>
    <pre><code class="prism language-c">Bitmap <span class="token operator">=</span> Assets<span class="token punctuation">.</span>Slots<span class="token punctuation">[</span>SlotID<span class="token punctuation">]</span><span class="token punctuation">.</span>Bitmap<span class="token punctuation">;</span>
</code></pre>
    <p>
     新的访问方式：
    </p>
    <pre><code class="prism language-c">bitmap_id ID

</code></pre>
    <p>
     简单、直接、无需映射。
    </p>
    <hr/>
    <h4>
     <a id="6__AddSoundAsset__AddBitmapAsset_960">
     </a>
     <strong>
      6. 更新 AddSoundAsset 和 AddBitmapAsset
     </strong>
    </h4>
    <p>
     在旧版本中，我们的
     <code>
      AddSoundAsset()
     </code>
     和
     <code>
      AddBitmapAsset()
     </code>
     都会返回一个 Slot ID：
    </p>
    <pre><code class="prism language-c">uint32 <span class="token function">AddSoundAsset</span><span class="token punctuation">(</span><span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">)</span>
<span class="token punctuation">{<!-- --></span>
    uint32 SlotID <span class="token operator">=</span> <span class="token function">GetNextAvailableSlot</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
    Assets<span class="token punctuation">.</span>Slots<span class="token punctuation">[</span>SlotID<span class="token punctuation">]</span><span class="token punctuation">.</span>Sound <span class="token operator">=</span> <span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">.</span>
    <span class="token keyword">return</span> SlotID<span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
    <p>
     现在我们将其改成直接返回
     <strong>
      资源索引
     </strong>
     ：
    </p>
    <pre><code class="prism language-c">uint32 <span class="token function">AddSoundAsset</span><span class="token punctuation">(</span><span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">)</span>
<span class="token punctuation">{<!-- --></span>

    <span class="token keyword">return</span> AssetIndex<span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
    <p>
     这种方式更加直观，也避免了不必要的 Slot ID。
    </p>
    <hr/>
    <h4>
     <a id="7__983">
     </a>
     <strong>
      7. 减少解引用操作
     </strong>
    </h4>
    <p>
     在旧版本中，由于通过 Slot ID 找资源，存在解引用的过程：
    </p>
    <pre><code class="prism language-c">Bitmap <span class="token operator">=</span> Assets<span class="token punctuation">.</span>Slots<span class="token punctuation">[</span>SlotID<span class="token punctuation">]</span><span class="token punctuation">.</span>Bitmap<span class="token punctuation">;</span>
</code></pre>
    <p>
     现在我们直接通过索引访问资源，省去解引用的过程：
    </p>
    <pre><code class="prism language-c">Bitmap <span class="token operator">=</span> Assets<span class="token punctuation">[</span>AssetIndex<span class="token punctuation">]</span><span class="token punctuation">.</span>Bitmap<span class="token punctuation">;</span>
</code></pre>
    <p>
     这样不仅代码简化，运行效率也会更高。
    </p>
    <hr/>
    <h4>
     <a id="8__IDSoundIDBitmapID_996">
     </a>
     <strong>
      8. 保留索引 ID（SoundID、BitmapID）
     </strong>
    </h4>
    <p>
     虽然我们移除了
     <strong>
      Slot ID
     </strong>
     ，但我们仍然需要一种方式来标识资源，比如
     <strong>
      SoundID
     </strong>
     和
     <strong>
      BitmapID
     </strong>
     。
     <br/>
     我们决定直接使用索引值作为 ID：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">struct</span> <span class="token class-name">asset_id</span>
<span class="token punctuation">{<!-- --></span>
    uint32 Value<span class="token punctuation">;</span>
<span class="token punctuation">}</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     在添加资源时，直接返回索引即可：
    </p>
    <pre><code class="prism language-c">SoundID <span class="token operator">=</span> <span class="token function">AddSoundAsset</span><span class="token punctuation">(</span><span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     访问时直接通过索引访问：
    </p>
    <pre><code class="prism language-c">Assets<span class="token punctuation">[</span>SoundID<span class="token punctuation">.</span>Value<span class="token punctuation">]</span><span class="token punctuation">.</span>Sound<span class="token punctuation">;</span>
</code></pre>
    <p>
     这样做的好处是：
    </p>
    <ul>
     <li>
      <strong>
       没有 Slot ID
      </strong>
      ，避免无意义的映射。
     </li>
     <li>
      <strong>
       资源索引即 ID
      </strong>
      ，简化所有逻辑。
     </li>
    </ul>

    <h4>
     <a id="9__1019">
     </a>

      9. 资源状态和加载的处理

    </h4>
    <p>
     在原先的实现中，我们通过 Slot ID 管理加载状态。
     <br/>
     现在我们直接将加载状态挂在资源上：
    </p>
    <pre><code class="prism language-c">Assets<span class="token punctuation">[</span>Index<span class="token punctuation">]</span><span class="token punctuation">.</span>State <span class="token operator">=</span> Loaded<span class="token punctuation">;</span>
</code></pre>
    <p>

    <h2>
     <a id="union_1082">
     </a>
     在结构体中使用“union”有什么作用？
    </h2>
    <p>
     在结构体中使用
     <code>
      union
     </code>
     的意思是这两个变量在内存中是重叠存储的。具体来说，当我们声明一个
     <code>
      union
     </code>
     时，编译器知道这两个成员变量不会同时使用，因此它们在内存中共享同一块区域。换句话说，结构体中某个变量的内存空间如果用到了
     <code>
      union
     </code>
     ，那么另一个变量将会占用相同的内存区域。
    </p>
    <p>
     以实际代码为例，当我们定义一个
     <code>
      asset
     </code>
     结构体时，它有两个成员变量，分别是一个
     <code>
      sound
     </code>
     和一个
     <code>
      bitmap
     </code>
     ，我们知道在任何时刻，一个资产只能是声音或位图中的一个。因此，使用
     <code>
      union
     </code>
     关键字，编译器会将这两个成员放在内存的同一位置，只会使用其中一个。
    </p>
    <p>
     在实际操作中，可以看到，在内存中这两个变量地址是重叠的。举个例子，当我们分别查看
     <code>
      FirstTagIndex
     </code>
     和
     <code>
      OnePastLastTagIndex
     </code>
     的地址时，它们是分开的，表示这两个变量各自占用了不同的内存空间。如果我们改变其中一个变量的值，另一个不会受影响，因为它们是分别存储的。
    </p>
    <p>
     但是，当我们使用
     <code>
      union
     </code>
     时，情况就不同了。例如，在访问
     <code>
      bitmap file name
     </code>
     和
     <code>
      sound file name
     </code>
     时，我们发现它们指向的是相同的内存位置，改变其中一个会导致另一个值也发生变化。这是因为它们共享同一块内存区域。当一个值发生改变时，另一个值也会随之更新。
    </p>
    <p>
     总结来说，
     <code>
      union
     </code>
     是一种内存共享技术，它允许多个成员变量使用相同的内存区域。我们通过
     <code>
      union
     </code>
     来告诉编译器，我们知道在任何时刻这些成员中只有一个会被使用，因此它们可以重叠存储，而不需要分别占用不同的内存空间。

    <h2>
     <a id="bitmap_idbitmap_id___variable__1095">
     </a>
     我看到bitmap_id赋值时有一个越界错误吗？bitmap_id 结果 = {…-&gt;variable++} 这个递增正确吗？
    </h2>
    <p>
     在这段代码中，主要是处理位图 ID 的分配和递增问题。具体来说，代码的目标是确保每次分配一个新的 ID 时，这个 ID 是可用的，并且在分配后会自动递增，确保该 ID 不会被重复使用。也就是说，每次请求一个新的 ID 时，系统会选择当前可用的下一个 ID，然后将其标记为已使用，并递增 ID 以供下次使用。
    </p>
    <p>
     这种机制保证了 ID 的唯一性，并且每次分配的 ID 都是按顺序递增的，不会发生重复分配的问题。如果有其他的代码或场景与此相关，可能会有不同的处理方式，但就当前的代码而言，它的功能就是按照这种方式管理 ID 的分配。
