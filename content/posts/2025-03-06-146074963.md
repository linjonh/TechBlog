    <p>
     在使用 Terraform 管理 AWS EC2 实例时，更改实例类型 (
     <code>
      instance_type
     </code>
     ) 需要谨慎处理，以避免实例被意外重建或系统盘数据丢失。接下来将详细介绍如何通过 Terraform 安全地更改 EC2 实例的类型，并保证系统盘（根卷）的数据完整无损。

      <code>

      </code>
      更改不会影响系统盘。
     </li>

       实例停机与变更策略
      </strong>
      ：如何执行实例停机并应用变更。

       <code>
        terraform plan

       <code>

     </li>

      </strong>
      ：变更完成后如何确认实例恢复正常，数据未丢失。
     </li>

    <h3>
     <a id="1_Terraform__10">
     </a>
     1. Terraform 变更影响分析
    </h3>
    <p>
     在 Terraform 的配置中更新 EC2 实例的
     <code>
      instance_type
     </code>
     会对资源产生什么影响呢？根据 Terraform 的设计，新版本的 Terraform 已支持对 EC2 实例类型的原地更新（in-place update），无需销毁重建整个实例 (
     <a href="https://github.com/hashicorp/terraform-provider-aws/issues/4838#:~:text=Terraform%20has%20supported%20changing%20the,0">
      Terraform-How to change AWS Instance type without deleting it if the instances are already running. · Issue #4838 · hashicorp/terraform-provider-aws · GitHub
     </a>
     )。也就是说，当我们仅修改
     <code>
      aws_instance
     </code>
     资源的
     <code>
      instance_type
     </code>
     属性并运行
     <code>
      terraform plan
     </code>
     时，计划输出应显示实例将
     <strong>
      原地修改
     </strong>
     （在输出中以
     <code>
      ~
     </code>
     前缀标识），而不是销毁再创建 (
     <a href="https://github.com/hashicorp/terraform-provider-aws/issues/4838#:~:text=provider%20split%20in%200">
      Terraform-How to change AWS Instance type without deleting it if the instances are already running. · Issue #4838 · hashicorp/terraform-provider-aws · GitHub
     </a>
     )。例如，Plan 中可能出现类似：
     <code>
      ~ instance_type: "t2.micro" =&gt; "t3.small"
     </code>
     。只要 Terraform Plan 显示为修改（
     <code>
      ~
     </code>
     ）而
     <strong>
      非
     </strong>
     销毁/重建（
     <code>
      -/+
     </code>
     ），就意味着 Terraform 将通过停止实例、调整类型然后重新启动实例的方式来应用更改，而不会替换实例 (
     <a href="https://github.com/hashicorp/terraform-provider-aws/issues/4838#:~:text=If%20you%20just%20update%20the,rather%20than%20destroy%20and%20create">
      Terraform-How to change AWS Instance type without deleting it if the instances are already running. · Issue #4838 · hashicorp/terraform-provider-aws · GitHub
     </a>
     )。
    </p>
    <p>
     值得注意的是，在应用此修改时实例会经历一次
     <strong>
      停机
     </strong>
     。Terraform 提供商会调用 AWS API 停止实例、修改实例类型，然后再启动实例 (
     <a href="https://github.com/hashicorp/terraform-provider-aws/issues/4838#:~:text=Note%20that%20it%20will%20stop,being%20up%20all%20the%20time">
      Terraform-How to change AWS Instance type without deleting it if the instances are already running. · Issue #4838 · hashicorp/terraform-provider-aws · GitHub
     </a>
     )。这一过程相当于人工在 AWS 控制台执行“停止实例-&gt;更改类型-&gt;启动实例”，因此会有短暂的不可用时间（停机时间）。在变更前，应该评估这一停机对业务的影响，必要时规划维护窗口。
    </p>
    <p>
     <strong>
      系统盘数据安全性分析
     </strong>
     ：由于 Terraform 更改
     <code>
      instance_type
     </code>
     是原地完成，实例的 ID 不变，附加在该实例上的EBS卷（包括根卷）也不会改变或重新创建。因此，只要我们遵循正确的步骤，实例的系统盘数据将保持完整，Terraform 不会删除或替换现有的根磁盘。需要避免的是其他会强制重建实例的改动，例如更换 AMI 映像（这会导致 Terraform 计划销毁并重建实例），那样系统盘数据就可能丢失 (
     <a href="https://developer.hashicorp.com/terraform/tutorials/aws-get-started/aws-change#:~:text=changing%20the%20AMI%20for%20an,Terraform%20to%20replace%20the%20instance" rel="nofollow">
      Change infrastructure | Terraform | HashiCorp Developer
     </a>
     )。仅调整
     <code>
      instance_type
     </code>
     并不涉及AMI或磁盘配置变更，因而默认情况下不会影响系统盘的数据。
    </p>
    <p>
     当然，
     <strong>
      前提
     </strong>
     是实例是由 EBS 卷作为根存储（大多数现代实例均是如此）。如果实例使用实例存储（Instance Store）作为根盘，那么无法通过停止/启动保留数据——实例存储在停止后数据即丢失，此种情况下 Terraform 也会强制重新创建实例。但对于EBS支持的实例类型，按照正常流程更改类型是安全的，不会丢失根盘数据。
    </p>
    <h3>
     <a id="2__20">
     </a>
     2. 实例停机与变更策略
    </h3>
    <p>
     根据 AWS 官方要求，更改 EC2 实例类型之前
     <strong>
      必须先将实例停止
     </strong>
     。AWS 文档明确指出：“在您可以更改实例类型之前，必须先停止实例” (
     <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/change-instance-type-of-ebs-backed-instance.html#:~:text=,and%20start%20Amazon%20EC2%20instances" rel="nofollow">
      Change the instance type for your Amazon EC2 instance - Amazon Elastic Compute Cloud
     </a>
     )。因此，无论是通过控制台手动操作还是使用 Terraform，修改实例类型都离不开“停止实例-&gt;更改配置-&gt;启动实例”这一流程。具体来说：
    </p>
    <ul>
     <li>
      <p>
       <strong>
        AWS 官方方法（控制台/CLI）
       </strong>
       ：在 AWS 管理控制台中，选中实例，选择“实例状态 -&gt; 停止”，等待实例进入停止状态；然后在“操作 -&gt; 实例设置”中选择“更改实例类型”，选择新的类型并应用；最后再启动实例。AWS 会在实例运行时禁止直接修改类型，如果实例处于运行状态，控制台操作会先将其停止再更改类型。同理，使用 AWS CLI 执行
       <code>
        aws ec2 modify-instance-attribute --instance-id &lt;ID&gt; --instance-type &lt;newType&gt;
       </code>
       也要求实例已停止，否则会报错。因此，停机是更改实例类型的必要步骤。
      </p>
     </li>
     <li>
      <p>
       <strong>
        Terraform 管理方法
       </strong>
       ：Terraform 的 AWS 提供商在检测到
       <code>
        instance_type
       </code>
       需要更新时，会自动按照上述流程执行。也就是说，Terraform
       <strong>
        会负责停止实例、修改类型、再启动实例
       </strong>
       ，而不需要用户手动干预 (
       <a href="https://github.com/hashicorp/terraform-provider-aws/issues/4838#:~:text=If%20you%20just%20update%20the,rather%20than%20destroy%20and%20create">
        Terraform-How to change AWS Instance type without deleting it if the instances are already running. · Issue #4838 · hashicorp/terraform-provider-aws · GitHub
       </a>
       )。Terraform 将此视为资源的生命周期内的正常修改，而非销毁重建。这一点由 Terraform 状态和计划来保障：Terraform 知道该实例已经存在，会将修改约束在该资源上，不会去创建新实例。因而根卷（系统盘）仍然附着在原实例上，不会被替换或丢失。
      </p>
     </li>
    </ul>
    <p>
     在停止实例进行类型变更时，有几点需要特别注意：
    </p>
    <ul>
     <li>
      <p>
       <strong>
        停机导致的附加影响
       </strong>
       ：当实例停止后，AWS 可能会将实例迁移到不同的底层硬件。当实例重新启动时，实例ID保持不变，之前关联的Elastic IP和私有IP地址仍会保留；但如果之前使用的是自动分配的公有IP（非弹性IP），则重启后该公有IP会发生变化 (
       <a href="https://www.logicata.com/blog/aws-ec2-change-instance-type/#:~:text=,IPv4%20address%20this%20will%20change" rel="nofollow">
        How to Change or Upgrade an EC2 Instance Type | Logicata
       </a>
       )。因此，如果我们的实例没有绑定弹性IP，变更后需要留意其公网IP变动对外部系统的影响（例如DNS映射）。此外，如果实例在一个自动伸缩组中，长时间停止可能被判定为不健康而替换，所以在变更期间应暂停该Auto Scaling组的健康检查或缩容该实例 (
       <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/change-instance-type-of-ebs-backed-instance.html#:~:text=,EC2%20Auto%20Scaling%20User%20Guide" rel="nofollow">
        Change the instance type for your Amazon EC2 instance - Amazon Elastic Compute Cloud
       </a>
       )。
      </p>
     </li>
     <li>
      <p>
       <strong>
        实例存储卷 (Ephemeral)
       </strong>
       ：对于具有实例存储的实例类型（少数旧类型或带本地NVMe存储的类型），停止实例会导致这些
       <strong>
        临时存储的数据丢失
       </strong>
       。Terraform 在执行类型变更时无法避免这一点——这是AWS的特性限制。因此，在变更前应确保没有将重要数据仅存放在实例的临时硬盘上 (
       <a href="https://github.com/hashicorp/terraform/issues/12195#:~:text=However%20this%20does%20not%20need,as%20ephemeral%20disks%20not%20used">
        Changing instance type recreates the instance and doesnt need to · Issue #12195 · hashicorp/terraform · GitHub
       </a>
       )。一般来说，系统盘为EBS时数据是安全的，但实例上的临时盘（如果有）在停机后内容即消失，此属正常现象，非Terraform导致。
      </p>
     </li>
     <li>
      <p>
       <strong>
        <code>
         delete_on_termination
        </code>
        标志：
       </strong>
       <code>
        root_block_device
       </code>
       配置中的
       <code>
        delete_on_termination
       </code>
       决定了实例终止时是否删除其绑定的EBS根卷。通常该值默认为
       <code>
        true
       </code>
       （表示实例销毁时删除磁盘），保持此默认设置有助于Terraform正确管理卷的生命周期，避免状态不一致。不过，在我们确保不会销毁实例的前提下，这个值不影响类型调整过程。然而
       <strong>
        务必确保
       </strong>
       在修改
       <code>
        instance_type
       </code>
       时，没有错误地调整
       <code>
        root_block_device
       </code>
       配置或改动
       <code>
        delete_on_termination
       </code>
       值。保持其为
       <code>
        true
       </code>
       （默认值）可以防止Terraform在计划时将其视作需要更改的属性，避免出现额外的卷操作，从而确保系统盘在整个过程中保持原样，不会被意外删除或分离。
      </p>
     </li>
    </ul>
    <p>
     总之，遵循“先停机再修改再重启”的策略是安全更改实例类型的核心。Terraform 已经帮我们自动实现了这一策略，但作为用户仍需知晓其过程，以便在变更时配合检查和确认。
    </p>
    <h3>
     <a id="3__Terraform__terraform_plan__38">
     </a>
     3. 检查 Terraform 计划 (
     <code>
      terraform plan
     </code>
     ) 输出
    </h3>
    <p>
     在正式应用更改前，务必运行
     <code>
      terraform plan
     </code>
     来预览即将发生的变更，并仔细检查计划输出，确认修改范围仅限于
     <code>
      instance_type
     </code>
     属性。以下是执行和解读
     <code>
      terraform plan
     </code>
     的步骤：
    </p>
    <ol>
     <li>
      **执行 Terraform Plan：**在 Terraform 配置中更新 EC2 实例的
      <code>
       instance_type
      </code>
      （例如从
      <code>
       "t2.micro"
      </code>
      改为
      <code>
       "t3.small"
      </code>
      ）后，运行命令：
     </li>
    </ol>

    <p>
     Terraform 将根据当前状态和配置计算出执行计划，并输出资源更改列表。建议针对目标资源执行 plan（例如
     <code>
      terraform plan -target="module.ec2_instance__standard_contract.aws_instance.server-dev"
     </code>
     ）以专注于该实例的变更（如果Terraform版本较新，也可以直接查看整体计划输出中的相关部分）。
    </p>
    <ol start="2">
     <li>
      <p>
       **识别计划符号：**留意计划输出中的符号含义：
      </p>
      <ul>
       <li>
        <code>
         ~
        </code>
        表示
        <strong>
         将更新
        </strong>
        现有资源的某些属性（in-place修改）。
       </li>
       <li>
        <code>
         -
        </code>
        表示销毁资源，
        <code>
         +
        </code>
        表示创建资源；如果同时出现
        <code>
         -/+
        </code>
        则表示资源将被替换。
       </li>
       <li>
        在本次操作中，我们
        <strong>
         期望只看到
         <code>
          ~
         </code>
         修改
        </strong>
        ，而不应出现销毁/创建操作。正常情况下，Terraform 会显示类似
        <strong>
         <code>
          ~ resource "aws_instance" "server-dev"
         </code>
        </strong>
        ，表示我们的 EC2 实例将被更新而非重建 (
        <a href="https://github.com/hashicorp/terraform-provider-aws/issues/4838#:~:text=provider%20split%20in%200">
         Terraform-How to change AWS Instance type without deleting it if the instances are already running. · Issue #4838 · hashicorp/terraform-provider-aws · GitHub
        </a>
        )。
       </li>
      </ul>
     </li>
     <li>
      <p>
       **检查变更细节：**在 plan 输出中，找到对应
       <code>
        aws_instance
       </code>
       资源的详细变更列表。例如：
      </p>
      <pre><code class="prism language-hcl">   # module.ec2_instance_standard_contract.aws_instance.server-dev will be updated in-place

      <p>
       重点关注
       <code>
        instance_type
       </code>
       行，应显示旧值 -&gt; 新值的更改。另外，检查
       <code>
        root_block_device
       </code>
       等与存储相关的部分：正常情况下，这些块设备配置不会在 plan 中出现变更行，因为根卷未发生任何改变。也就是说，诸如
       <code>
        volume_id
       </code>
       、
       <code>
        delete_on_termination
       </code>
       等字段应该
       <strong>
        保持不变
       </strong>
       。如果 plan 输出未Explicit列出它们的更改，基本可以确定 Terraform 不会对系统盘执行销毁或重新创建操作。
      </p>
      <p>
       <em>
        示例:
       </em>
       若
       <code>
        terraform plan
       </code>
       输出中包含了
       <code>
        root_block_device.0.volume_id
       </code>
       或类似字段，并显示更改（例如更换了
       <code>
        vol-XXXX
       </code>
       的ID），那表示Terraform认为根卷需要更换，这是不正常的，务必停止并调查原因（例如是否错误地修改了
       <code>
        root_block_device
       </code>
       配置）。在理想情况下，plan 的变更应仅限于
       <code>
        instance_type
       </code>
       一项，其它如
       <code>
        ami
       </code>
       、
       <code>
        subnet_id
       </code>
       、
       <code>
        volume_id
       </code>
       等都不应出现变化。
      </p>
     </li>
     <li>
      <p>
       <strong>
        确认计划摘要：
        <strong>
         查看 plan 输出底部的摘要行，确保它类似于：
        </strong>
        <code>
         Plan: 0 to add, 1 to change, 0 to destroy.
        </code>
       </strong>
       这表示将有1个现有资源被更新，无新建也无销毁。这与我们的预期一致。如果出现了“将销毁 (destroy)”的提示，切勿贸然继续，而应该回滚修改或检查Terraform配置，以找出是哪个改动引起了实例重建计划。
      </p>
     </li>
    </ol>
    <p>
     通过仔细解读
     <code>
      terraform plan
     </code>
     ，我们可以有信心地确认此次操作只是更改实例类型，不会影响系统盘数据。例如，有用户报告在 Terraform 0.8.8 及以后版本中，更改实例类型时 Terraform 会显示原地更新且不会销毁实例 (
     <a href="https://github.com/hashicorp/terraform-provider-aws/issues/4838#:~:text=provider%20split%20in%200">
      Terraform-How to change AWS Instance type without deleting it if the instances are already running. · Issue #4838 · hashicorp/terraform-provider-aws · GitHub
     </a>
     )。如果我们的计划输出与此一致，那么就可以进入下一步。反之，如果计划不符预期，应先行解决再进行应用。
    </p>
    <h3>
     <a id="4__terraform_apply__109">
     </a>
     4. 应用变更 (
     <code>
      terraform apply
     </code>
     ) 过程
    </h3>
    <p>
     在确认
     <code>
      terraform plan
     </code>
     符合预期且仅修改
     <code>
      instance_type
     </code>
     之后，就可以执行
     <code>
      terraform apply
     </code>
     将更改付诸实践。以下是应用变更的过程和注意事项：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        执行 Terraform Apply
       </strong>
       ：运行命令：
      </p>
      <pre><code class="prism language-bash">PS C:<span class="token punctuation">\</span>src<span class="token punctuation">\</span>devops<span class="token punctuation">\</span>terraform<span class="token operator">&gt;</span> terraform apply <span class="token string">"ftplan"</span>
Apply complete<span class="token operator">!</span> Resources: <span class="token number">0</span> added, <span class="token number">1</span> changed, <span class="token number">0</span> destroyed.
</code></pre>
      <p>
       Terraform会再次显示执行计划并要求确认。请再次核对输出（尤其在 plan 之后有进一步修改的情况下）。输入
       <code>
        yes
       </code>
       确认后，Terraform将开始对AWS执行实际更改。
      </p>
     </li>
     <li>
      <p>
       <strong>
        Terraform 输出监控：
       </strong>
       <code>
        terraform apply
       </code>
       过程中的终端输出会显示资源的操作进展。例如：
      </p>
      <ul>
       <li>
        <code>
         aws_instance.my_instance: Modifying... [id=i-1234567890abcdef]
        </code>
        表示正在修改该实例。
       </li>
       <li>
        此时，您可以在 AWS 控制台观察到对应实例进入
        <strong>
         stopping
        </strong>
        状态，随后实例状态变为
        <strong>
         stopped
        </strong>
        ，实例类型属性在停止期间被更新，紧接着实例进入
        <strong>
         pending
        </strong>
        然后
        <strong>
         running
        </strong>
        状态重新启动。这一系列操作都是 Terraform 在幕后调用 AWS API 完成的。
       </li>
       <li>
        Terraform 完成后会输出类似
        <code>
         aws_instance.my_instance: Modifications complete after Xs
        </code>
        ，最终提示
        <code>
         Apply complete! Resources: 0 added, 1 changed, 0 destroyed.
        </code>
        表明应用成功，1个资源已更新。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        异常情况处理
       </strong>
       ：通常，更改实例类型是一个直接的过程，但仍需防范潜在异常：
      </p>
      <ul>
       <li>
        <strong>
         类型不兼容或配额问题
        </strong>
        ：如果新实例类型与当前实例配置不兼容（例如架构不同、虚拟化类型不同）或者 AWS 资源不足，AWS 可能会返回错误，导致 Terraform 应用失败。这时终端会显示错误信息。遇到此类错误，实例可能会保持在已停止状态（如果类型修改未成功应用）。应根据错误信息采取措施，例如选择兼容的实例类型或在AWS控制台手动将实例启动回原类型以恢复服务，然后重新规划变更方案。
       </li>
       <li>
        <strong>
         Terraform 中断
        </strong>
        ：在 apply 过程中，
        <strong>
         不要强制中断 Terraform
        </strong>
        ，以免状态不同步。如果不得不中断（比如网络问题），在恢复后应先使用
        <code>
         terraform refresh
        </code>
        （或
        <code>
         terraform plan
        </code>
        ）检查当前状态，再决定后续步骤。
       </li>
       <li>
        <strong>
         生命周期保护
        </strong>
        ：如果之前在 Terraform 配置中对该实例设置了
        <code>
         lifecycle { prevent_destroy = true }
        </code>
        防止销毁，那么即使计划是重建实例，Terraform 也会在 apply 时阻止删除操作。这在本场景下应不适用（因为我们不期望销毁），但值得了解其存在，以防配置中有类似保护导致 apply 失败需要解除。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        最坏情况预案
       </strong>
       ：虽然按照上述方法 Terraform 不会重建实例，但如果由于某种原因计划出现了实例重建（销毁+创建），为了保险起见可以采取额外预防措施：
      </p>
      <ul>
       <li>
        临时将根卷的
        <code>
         delete_on_termination
        </code>
        设置为
        <code>
         false
        </code>
        （如果允许修改的话），使得即便实例被销毁，其系统盘EBS卷也不会被删除，从而保留数据。此操作需谨慎，并记得在操作完成后根据需要改回原设置。
       </li>
       <li>
        或者在 Terraform 外手动创建快照备份当前根卷。在极端意外发生时（实例被删），仍可通过快照还原数据。
       </li>
      </ul>
      <p>
       当然，这些仅作为容灾备选。按照我们规划的流程，Terraform
       <strong>
        不应
       </strong>
       删除系统盘或使数据受损。
      </p>
     </li>
    </ol>
    <p>
     总的来说，
     <code>
      terraform apply
     </code>
     步骤在正确的计划下是安全的。应用过程中多关注Terraform输出及AWS控制台实例状态，可以及时发现异常并处理。正常情况下，Terraform 会顺利完成实例类型调整，实例将以新类型重新启动运行。
    </p>
    <h3>
     <a id="5__140">
     </a>
     5. 实例重启与验证
    </h3>
    <p>
     变更完成后，需要对实例进行检查和验证，以确保实例按新配置正常运行且系统盘数据完好无损。请按照以下步骤进行验证：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        检查实例状态
       </strong>
       ：在 Terraform 提示应用完成后，登录 AWS 管理控制台，找到该 EC2 实例，确认其
       <strong>
        状态为 running（运行中）
       </strong>
       ，实例类型已更新为您期望的新类型。如果实例没有自动启动（例如实例之前就是停止状态，Terraform 修改后保持停止），则需要手动启动实例。在 AWS 控制台或使用 CLI (
       <code>
        aws ec2 start-instances --instance-ids &lt;实例ID&gt;
       </code>
       ) 启动它，并等待其变为运行状态。
      </p>
     </li>
     <li>
      <p>
       <strong>
        确认实例标识
       </strong>
       ：核对实例的
       <strong>
        Instance ID
       </strong>
       ，确保仍然是原来的ID（Terraform 原地更新不会改变实例ID）。实例ID不变意味着我们依然在操作原来的实例 (
       <a href="https://www.logicata.com/blog/aws-ec2-change-instance-type/#:~:text=,IPv4%20address%20this%20will%20change" rel="nofollow">
        How to Change or Upgrade an EC2 Instance Type | Logicata
       </a>
       )。这一点可以从 Terraform 输出或 AWS 控制台直接看到。如果实例ID发生变化，那表明出现了实例替换，应立即调查原因。
      </p>
     </li>
     <li>
      <p>
       <strong>
        验证系统盘挂载
       </strong>
       ：查看该实例的存储卷配置。在 AWS 控制台的 “存储” 或 “Volumes” 部分，找到附加到该实例的根卷（通常挂载点为
       <code>
        /dev/xvda
       </code>
       或
       <code>
        /dev/sda1
       </code>
       ）。核对其
       <strong>
        卷ID (Volume ID)
       </strong>
       是否与变更前一致。如有需要，可在变更前先记录下原始卷ID，然后在变更后对比。理想情况下，卷ID应完全相同，表示实例仍使用原来的系统盘。您也可以通过 Terraform 命令验证这一点：
      </p>
      <pre><code class="prism language-bash">terraform state show  module.ec2_instance_standard_contract.aws_instance.server-dev
</code></pre>
      <p>
       在输出中寻找
       <code>
        root_block_device
       </code>
       信息，确认其中的
       <code>
        volume_id
       </code>
       未发生变化，且依然与实例关联。如果
       <code>
        terraform state show
       </code>
       显示 Terraform 正在跟踪同一个卷，那么说明Terraform状态与实际资源匹配，一切正常。
      </p>
     </li>
     <li>
      <p>
       <strong>
        数据完整性检查
       </strong>
       ：确保系统盘中的数据确实未丢失。最直接的方法是登录到该实例（通过SSH或RDP等方式），检查关键文件或数据是否完好。例如，查看之前创建的文件是否仍然存在，应用或服务是否保留了原有的配置和数据。由于我们没有更换或重新创建根卷，按理说数据不会改变。如果无法登录实例，也可将该EBS卷挂载到其他临时实例进行检查（不太常见，仅在怀疑数据完整性时采取）。
      </p>
     </li>
     <li>
      <p>
       <strong>
        其他附加验证：
       </strong>
      </p>
      <ul>
       <li>
        <strong>
         网络连通性
        </strong>
        ：如果实例有公共IP且未使用弹性IP，那么重启后IP可能变化 (
        <a href="https://www.logicata.com/blog/aws-ec2-change-instance-type/#:~:text=,IPv4%20address%20this%20will%20change" rel="nofollow">
         How to Change or Upgrade an EC2 Instance Type | Logicata
        </a>
        )。确认新的公共IP并更新相关的DNS记录或通知客户端。如果是私有实例，检查其私有IP是否保持不变（通常停止/启动会保留私有IP，但不同子网策略可能有影响）。
       </li>
       <li>
        <strong>
         应用服务
        </strong>
        ：重启后确保实例上的应用服务启动并正常运行。观察应用日志或监控指标，确保因为实例类型变更（比如CPU/内存资源变化）没有引入性能或兼容性问题。
       </li>
       <li>
        <strong>
         CloudWatch 检查
        </strong>
        ：查看CloudWatch中该实例的新指标，以验证实例规格变更生效（如CPU基线性能对应新实例类型）。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        清理与后续
       </strong>
       ：
      </p>
      <ul>
       <li>
        如果在变更过程中对 Terraform 配置做了临时调整（例如 lifecycle 规则或
        <code>
         delete_on_termination
        </code>
        设置），在确认变更成功后可以恢复原配置以保持代码一致性。
       </li>
       <li>
        将此次变更记录在案，包括变更前后的实例类型、卷ID、停机时间段等，方便日后审计或排查。
       </li>
       <li>
        未来若再次需要变更实例类型，可参考本指南的步骤重复进行。
       </li>
      </ul>
     </li>
    </ol>
    <p>
     通过以上验证步骤，我们可以确信此次 Terraform 执行的实例类型更改是成功且安全的。系统盘数据仍然保存在原有的 EBS 卷上，没有因为实例替换而丢失 (
     <a href="https://github.com/hashicorp/terraform/issues/12195#:~:text=However%20this%20does%20not%20need,as%20ephemeral%20disks%20not%20used">
      Changing instance type recreates the instance and doesnt need to · Issue #12195 · hashicorp/terraform · GitHub
     </a>
     )。Terraform 状态也与实际资源匹配，后续管理可以继续依赖 Terraform 对该实例及其卷的追踪。
    </p>
    <h3>
     <a id="_168">
     </a>
     关键点回顾
    </h3>
    <ul>
     <li>
      <p>
       <strong>
        Terraform 原地更新能力：
       </strong>
       现代版本的 Terraform 支持直接修改 EC2 实例的
       <code>
        instance_type
       </code>
       而不重建资源 (
       <a href="https://github.com/hashicorp/terraform-provider-aws/issues/4838#:~:text=provider%20split%20in%200">
        Terraform-How to change AWS Instance type without deleting it if the instances are already running. · Issue #4838 · hashicorp/terraform-provider-aws · GitHub
       </a>
       )。利用
       <code>
        terraform plan
       </code>
       预先确认变更为原地更新（
       <code>
        ~
       </code>
       标识）至关重要。
      </p>
     </li>
     <li>
      <p>
       <strong>
        停机操作遵循AWS最佳实践：
       </strong>
       更改实例类型必须先停止实例 (
       <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/change-instance-type-of-ebs-backed-instance.html#:~:text=,and%20start%20Amazon%20EC2%20instances" rel="nofollow">
        Change the instance type for your Amazon EC2 instance - Amazon Elastic Compute Cloud
       </a>
       )。Terraform 会自动执行停机-&gt;修改-&gt;启动流程，期间实例ID不变、根卷不变，但需考虑停机带来的公网IP变化和服务中断影响 (
       <a href="https://www.logicata.com/blog/aws-ec2-change-instance-type/#:~:text=,IPv4%20address%20this%20will%20change" rel="nofollow">
        How to Change or Upgrade an EC2 Instance Type | Logicata
       </a>
       )。
      </p>
     </li>
     <li>
      <p>
       <strong>
        系统盘保护：
       </strong>
       确保 Terraform 计划中未对
       <code>
        root_block_device
       </code>
       进行任何修改，
       <code>
        volume_id
       </code>
       等字段保持不变。
       <code>
        delete_on_termination
       </code>
       等参数请保持与当前状态一致（通常为 true），避免Terraform尝试修改卷的生命周期设置。
      </p>
     </li>
     <li>
      <p>
       <strong>
        充分利用 Terraform 工具：
       </strong>
       运用
       <code>
        terraform plan
       </code>
       、
       <code>
        terraform state show
       </code>
       等命令来核实变更影响和结果。Plan 输出应仅呈现
       <code>
        instance_type
       </code>
       更改，没有隐藏的销毁/创建操作。Apply 后通过 state 和控制台检查实例和卷，确保 Terraform 状态正确映射实际资源。
      </p>
     </li>
     <li>
      <p>
       <strong>
        数据验证：
       </strong>
       变更完成后，一定要验证实例上的应用和数据是否正常。由于根卷未更换，理论上数据不会丢失，但通过登录实例检查关键数据来
       <strong>
        双重确认
       </strong>
       始终是稳妥的做法。
      </p>
     </li>
    </ul>
    <p>
     按照本文指南操作，可以安全地使用 Terraform 更改 EC2 实例类型，而无需担心系统盘数据的丢失。在获得更高配置或更优成本的同时，依然保持基础设施状态的一致和受控。祝大家在Terraform的使用中一切顺利！
    </p>
