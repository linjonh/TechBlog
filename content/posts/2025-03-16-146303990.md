    <p>
     为下一个版本的 Java 提出了 21 项功能，包括旨在保护 Java 应用程序免受未来量子计算攻击的抗量子加密密钥。

    <div style="margin-left:0; margin-right:0; text-align:start">
     <div style="margin-left:0; margin-right:0">
      <div style="margin-left:0; margin-right:0">
       <p style="margin-left:0; margin-right:0">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           继 9 月 17 日发布的
           <a href="https://www.infoworld.com/article/2336682/jdk-23-the-new-features-in-java-23.html/%22" rel="nofollow" title="Java 开发工具包 （JDK） 23">
            Java 开发工具包 （JDK） 23
           </a>
           之后，现在的工作重点是计划的后续版本
           <a href="https://openjdk.org/projects/jdk/24/" rel="nofollow" title="JDK 24">
            JDK 24
           </a>
           ，该版本已膨胀到 21 个提议的功能。最近添加的两个提案将通过提供抗量子模块网格的数字签名算法和抗量子模块网格的密钥封装机制的 Java 实现，提高 Java 对量子计算攻击的抵抗力。
          </span>
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           以前提议的功能包括删除 32 位 x86 端口;同步虚拟线程而不固定;简单的源文件和实例 main 方法;永久禁用安全管理器;模块导入声明;Compact Headers 的实验版本;patterns、instanceof 和 switch 中的基元类型;链接没有 JMOD 的运行时映像;世代相传的 Shenandoah 垃圾收集器;范围值;密钥派生函数 API;删除了 Z Garbage Collector 中的非分代模式;溪流收集器;一个矢量 API;类文件 API;警告，让开发人员为将来对
           <a href="https://www.infoworld.com/article/2518132/openjdk-plan-calls-for-restricting-jni-usage.html/%22" rel="nofollow" title="JNI （Java Native Interface）">
            JNI （Java Native Interface）
           </a>
           的使用限制做好准备;以及 G1 垃圾回收器的后期屏障扩展。
          </span>
         </span>
        </span>
       </p>
       <h3 style="margin-left:0px; margin-right:0px">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           JDK 24 优于 JDK 23，后者列出了 12 个官方功能
          </span>
         </span>
        </span>
       </h3>

       <p style="margin-left:0; margin-right:0">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           JDK 24 已指定为非长期支持 （LTS） 版本，截止日期为 2025 年 3 月 18 日。与 JDK 23 一样，JDK 24 将仅获得 Oracle 提供的六个月
           <a href="https://www.oracle.com/support/premier" rel="nofollow" title="高级支持">
            高级支持
           </a>
           。JDK 24 的抢先体验版本可以在
           <a href="https://jdk.java.net/24/" rel="nofollow" title="jdk.java.net">
            jdk.java.net
           </a>
           上找到。
          </span>
         </span>
        </span>
       </p>
       <div style="margin-left:0; margin-right:0">
        <div id="jwplayer--floatingVideo" style="margin-left:0px; margin-right:0px">

       <p style="margin-left:0; margin-right:0">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           为通过抗量子性提高 Java 安全性而提出的两个功能包括抗
           <a href="https://openjdk.org/jeps/496" rel="nofollow" title="量子模块晶格的密钥封装机制">
            量子模块晶格的密钥封装机制
           </a>
           （ML-KEM） 和
           <a href="https://openjdk.org/jeps/497" rel="nofollow" title="抗量子量子晶格的基于数字签名算法">
            抗量子量子晶格的基于数字签名算法
           </a>
           （ML-DSA）。ML-DSA 将使用数字签名来检测对数据的未经授权的修改并验证签名者的身份，从而抵御未来的量子计算攻击。密钥封装机制 （KEM） 用于使用公钥加密技术在不安全的通信通道上保护对称密钥。这两项功能都旨在防范未来的量子计算攻击。
          </span>
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           灵活的构造函数主体在
           <a href="https://www.infoworld.com/article/2335077/jdk-22-the-new-features-in-java-22.html" rel="nofollow" title="JDK 22">
            JDK 22
           </a>
           和
           <a href="https://www.infoworld.com/article/2336682/jdk-23-the-new-features-in-java-23.html/%22" rel="nofollow" title="JDK 23">
            JDK 23
           </a>
           中出现后，现在处于第三个预览阶段，尽管在 JDK 22 中具有不同的名称，当时该功能在 super（...） 之前称为 statements。该功能旨在重新构想构造函数在对象初始化过程中的角色，让开发人员更自然地将他们当前必须考虑的逻辑放入辅助静态方法、辅助中间构造函数或构造函数参数中。
          </span>
         </span>
        </span>

        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           <a href="https://openjdk.org/jeps/483" rel="nofollow" title="Ahead-of-time class loading and linking">
            Ahead-of-time class loading and linking
           </a>
           旨在通过在 HotSpot Java 虚拟机启动时使应用程序的类立即处于加载和链接状态来缩短启动时间。这将通过在一次运行期间监视应用程序并将所有类的加载和链接形式存储在缓存中以供后续运行使用来实现。
          </span>
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           <a href="https://openjdk.org/jeps/479" rel="nofollow" title="Windows 32 位 x86 端口">
            Windows 32 位 x86 端口
           </a>
           在
           <a href="https://openjdk.org/jeps/449/%22" rel="nofollow" title="JDK 21 中已弃用，无法删除">
            JDK 21 中已弃用，无法删除
           </a>
           ，并打算在未来发行版中删除它。计划要求删除源代码并构建对 Windows 32 位 x86 端口的支持。目标包括删除仅适用于 Windows 32 位 x86 的所有代码路径，停止所有针对 Windows 32 位 x86 平台的测试和开发工作，以及简化 JDK 的构建和测试基础结构。该提案指出，Windows 10 是最后一个支持 32 位操作的 Windows 操作系统，将于
           <a href="https://learn.microsoft.com/lifecycle/products/windows-10-home-and-pro" rel="nofollow" title="2025 年 10 月">
            2025 年 10 月
           </a>
           结束其生命周期。
          </span>
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           <a href="https://openjdk.org/jeps/491" rel="nofollow" title="同步虚拟线程而不固定">
            同步虚拟线程而不固定
           </a>
           涉及提高使用同步方法和语句的 Java 代码的可伸缩性，方法是安排在此类构造中阻塞的虚拟线程释放其底层平台供其他线程使用。这将消除几乎所有虚拟线程被固定到平台线程的情况，这严重限制了可用于处理应用程序工作负载的虚拟线程的数量。
          </span>
         </span>
        </span>
       </p>

           <a href="https://openjdk.org/jeps/495" rel="nofollow" title="简单源文件和实例 main 方法">
            简单源文件和实例 main 方法
           </a>
           的第四个预览版将发展 Java 语言，因此初学者可以编写他们的第一个程序，而无需了解为大型程序设计的语言功能。该功能之前在
           <a rel="nofollow">
            JDK 21
           </a>
           、
           <a rel="nofollow">
            JDK 22

           <a rel="nofollow">
            JDK 23

           中预览过。目标是允许初级 Java 程序员为单类程序编写简化的声明，然后随着技能的增长无缝扩展他们的程序以使用更高级的功能。
          </span>
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           <a rel="nofollow">
            永久禁用安全管理器
           </a>
           涉及修改 Java 平台规范，以便开发人员无法启用安全管理器，而其他平台类则不引用它。该提案指出，多年来，安全管理器一直不是保护客户端 Java 代码的主要手段，很少用于保护服务器端代码，并且维护成本高昂。安全管理器在 Java 17 中已被弃用，无法删除。
          </span>
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          <span style="color:#0c0c0c">
           之前在 JDK 23 中预览的
           <a href="https://openjdk.org/jeps/494" rel="nofollow" title="模块导入声明">
            模块导入声明
           </a>
           增强了 Java 编程语言，使其能够简洁地导入模块导出的所有包。这简化了模块化库的重用，但不需要将代码导入为模块本身。
          </span>
         </span>
        </span>
       </p>

       <h3>
        Java 24 减小对象标头大小并节省内存
       </h3>
       <p>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          <a href="https://openjdk.org/jeps/450" rel="nofollow" title="JEP 450（Compact Object Headers）">
           JEP 450（Compact Object Headers）
          </a>
          已计划在 JDK 24 中交付，并且已经合并到 main。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          这个目前的实验性功能通过缩小 HotSpot 中强制对象头的大小来优化堆利用率。这应该会减小总体堆大小，提高部署密度，并增加数据局部性。
         </span>
        </span>
       </p>
       <h4 style="margin-left:0px; margin-right:0px; text-align:left">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          当前实施概述
         </span>
        </span>
       </h4>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          HotSpot 将所有对象存储在 Java 堆中，Java 堆是进程的 “C heap” 的连续区域。Java 中的对象始终通过引用处理，例如：
         </span>
        </span>
       </p>
       <ol style="margin-left:0; margin-right:0">
        <li>
         引用对象的局部变量包含从 Java 方法的堆栈帧到 Java 堆的指针。
        </li>
        <li>
         引用类型的对象字段从一个 Java 堆位置指向另一个 Java 堆位置。
        </li>
       </ol>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          Java 引用的目标地址始终是对象头的开头（这在当前版本的 HotSpot 中是必需的）。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          每个对象上都存在 Headers（数组有一个额外的 32 位 Headers 来存储数组的长度）。标记字是前 64 位，用于特定于实例的元数据，即支持以下功能：
         </span>
        </span>
       </p>
       <ul style="margin-left:0; margin-right:0">
        <li>
         垃圾收集 - 存储对象的年龄（可能还有转发指针）
        </li>
        <li>
         哈希代码 - 存储对象的稳定身份哈希代码
        </li>
        <li>
         Locks - 存储对象的锁 / 监视器
        </li>
       </ul>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          在某些情况下，标记字将被覆盖并替换为引用更复杂数据结构的指针。这使得紧凑对象标头的实现稍微复杂一些。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          标记字后面是类（或 klass）字，该字用于计算指向此类类型的每个对象共享的元数据的指针。这用于方法调用、反射、类型检查等。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          klass 元数据（或 klass）保存在元空间中，元空间位于 Java 堆之外，但不在 JVM 进程的 C 堆之外。由于它们存在于 Java 堆之外，因此 klasses 不需要 Java 对象头，并且它们与反射中使用的类对象（它们是真正的 Java 对象）不是一回事。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          klass 字最初是 header 的全机器字，但这在 64 位架构上是浪费的，因此引入了一种称为“压缩类指针”的技术。这会将类指针编码为 32 位（通过使用缩放和偏移方法），这适用于加载少于 4 GB 的类文件的任何应用程序。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          因此，除了边缘情况外，64 位版本的 HotSpot 上的非数组对象还支付了 96 位的“头税”。相比之下，这是轻量级的：直到最近，Python 的头税是 308 字节，但 JEP 450 的目的是做得更好，并将头的总大小减少到 64 位。
         </span>
        </span>
       </p>
       <h4 style="margin-left:0px; margin-right:0px; text-align:left">
        <span style="color:#000000">
         <span style="background-color:#ffffff">
          Compact Object Headers 简介
         </span>
        </span>
       </h4>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          新实现是作为 OpenJDK 的“Project Lilliput”的一部分开发的，它减小了两个目标 64 位平台（x64 和 AArch64）上的对象头大小。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          总体目标是：
         </span>
        </span>
       </p>
       <ol style="margin-left:0; margin-right:0">
        <li>
         在目标平台上将吞吐量和延迟开销限制为 5%，并且仅在不常见的情况下接近该限制
        </li>
        <li>
         不会在非目标平台上引入可测量的吞吐量或延迟开销
        </li>
       </ol>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          事实上，测试目前只显示了极少数的回归（JDK 24 正在进行一些修复）。到目前为止，Amazon 的测试表明，许多工作负载实际上在吞吐量方面受益匪浅，有时甚至非常明显 - 某些工作负载的 CPU 利用率下降了高达 30%。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          该项目试图利用观察到的事实，即许多 Java 工作负载的平均对象大小很小，为 32 到 64 字节。这相当于 ~20% 的头税。因此，即使对象头大小略有改进，也可以显著减少堆占用空间。反过来，这可以提高数据局部性并降低 GC 压力，从而进一步获得潜在的性能优势。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          为了实现这种 Headers 缩减，mark 和 class 字被组合成一个 64 位字，布局如下：
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
       </p>
       <p class="img-center">
        <img alt="" height="400" src="https://img-blog.csdnimg.cn/img_convert/075ecb3236b77a0399e97af7ef4e34c2.png" width="1545"/>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          我们应该注意以下几个方面：
         </span>
        </span>
       </p>
       <ol style="margin-left:0; margin-right:0">
        <li>
         现在有 22 位（而不是 32 位）用于标识对象类类型。这意味着我们可以加载到 JVM 进程中的不同类类型的数量约为 ~400 万个。
        </li>
        <li>
         哈希代码的大小不会改变。
        </li>
        <li>
         锁定操作不再覆盖标记字。这将保留压缩的类指针。
        </li>
        <li>
         GC 转发操作变得更加复杂，以便保留对压缩类指针的直接访问。
        </li>
        <li>
         有四个未使用的位保留用于将来的增强（例如 Project Valhalla）。
        </li>
       </ol>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          如果争用 Java 锁，则新实现需要查找保存锁信息的
          <a href="https://wiki.openjdk.org/display/HotSpot/Synchronization+Using+The+ObjectMonitorTable" rel="nofollow" title="辅助数据结构">
           辅助数据结构
          </a>
          的地址。这种方法称为 “对象监视器表” ，在
          <a href="https://github.com/openjdk/jdk/pull/20067" title="JDK 22">
           JDK 22
          </a>
          中实现，由默认打开的新开关激活。Compact 对象标头依赖于此机制。
          <code>
           UseObjectMonitorTable
          </code>
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          如果未找到阻碍因素，此功能将作为 JDK 24 的一部分发布（最初作为实验性功能），预计将于 2025 年 3 月发布。长期目标是让此机制成为受支持平台上的唯一标头表示形式，但这可能需要更多版本。它还取决于对实际工作负载的广泛测试，以及缺乏性能和其他回归。甚至还有正在进行的探索工作，看看是否有可能将标头大小减少到 32 位。
         </span>
        </span>
       </p>
       <p style="margin-left:0; margin-right:0; text-align:left">
        <span style="color:#222222">
         <span style="background-color:#ffffff">
          在 JDK 24（测试版或最终版）中推出后，应用程序团队可以通过命令行开关激活此新功能来测试其工作负载，并查找与之相关的性能差异，从而提供帮助。
          <code>
           -XX:UseCompactObjectHeaders
          </code>
         </span>
        </span>
       </p>
