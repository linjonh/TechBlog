    <h3>
     <a id="111__6">
     </a>
     1.1.1 软件危机（日子没法过了）
    </h3>
    <p>
     <strong>
      软件危机
     </strong>
     （Software Crisis）是指在计算机软件开发、运行、维护和管理过程中所遇到的一系列严重问题。(木星危机，太阳氦闪危机)
     <br/>
     1）软件开发
     <strong>
      进度和成本
     </strong>
     难以控制（不能科学估算）
     <br/>
     2）软件开发难以满足用户的
     <strong>
      需求
     </strong>
     （不是要什么给什么）
     <br/>
     3）软件
     <strong>
      质量
     </strong>
     难以保证（降落伞打不开的宇航员）
     <br/>
     4）软件产品难以
     <strong>
      维护
     </strong>
     （维护成本不断上升，维护难度不断增加
    </p>
    <h3>
     <a id="112__14">
     </a>
     1.1.2 软件危机出现的原因
    </h3>
    <p>
     <strong>
      产生软件危机的主要原因有：
     </strong>
     <br/>
     (1)软件开发缺乏正确的
     <strong>
      理论指导
     </strong>
     ；软件本身属性、需要有限条件多人分工高质量产出
     <br/>
     (2)软件人员与用户
     <strong>
      缺乏充分交流
     </strong>
     ；了解但难以描述需求、知识背景有别、曲解需求
     <br/>
     (3)对软件开发过程
     <strong>
      缺乏整体认识
     </strong>
     ；经验未沉淀总结、忽视文档管理、不重视测试…（重实现）
     <br/>
     (4)软件产品缺乏有效一致的质量评价标准。
    </p>
    <h2>
     <a id="12__22">
     </a>
     1.2 软件工程的概念
    </h2>
    <h3>
     <a id="121__24">
     </a>
     1.2.1 软件工程的定义
    </h3>
    <p>
     软件工程是把
     <strong>
      系统的、规范的、可度量
     </strong>
     的途径应用于软件开发、运行和维护的全过程，以及对上述方法的研究。（IEEE-1993）
     <strong>
      （整体、有规范、各个过程可以度量）
     </strong>
    </p>
    <p>
     软件工程三要素：
     <strong>
      方法、工具、过程
     </strong>
    </p>

    <h3>
     <a id="131__34">

     1.3.1 软件和软件的组成
    </h3>
    <p>
     软件是计算机中与硬件相互依存的另一部分，它包括
     <strong>
      程序、数据以及相关文档
     </strong>
    </p>
    <p>
     <strong>
      软件=程序+数据+文档+知识
     </strong>
    </p>
    <h3>
     <a id="_40">
     </a>
     软件自身的特点
    </h3>
    <ul>
     <li>
      逻辑实体而非物理实体
     </li>
     <li>
      智力产品，生成过程主要集中在研发上
     </li>
     <li>
      永不磨损，会蜕化直至被放弃使用
     </li>
     <li>
      软件开发远未达到软件工程目标提及的产业化生产
     </li>
     <li>
      软件规模和内容越来越复杂
     </li>
    </ul>

    <h3>

     6.1边界值分析
    </h3>
    <p>
     边界值分析法是对等价分类技术的补充，即在一个等价类中不是任选一个元素作为此等价类的代表进行测试，而是
     <strong>
      选择此等价类边界上的值。
     </strong>
    </p>
    <p>
     首先确定边界情况。通
     <strong>
      常输入或输出等价类的边界就是应该着重测试的边界情况。
     </strong>
    </p>
    <p>
     选取正好
     <strong>
      等于、刚刚大于或刚刚小于
     </strong>
     边界的值作为测试数据，而不是选取等价类中的典型值或任意值。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/79d7ed719d9e69d05a13648eb88ac471.png"/>
     <br/>
     实践表明：
     <strong>
      大多数故障往往发生在输入定义域或输出值域的边界上，而不是在其内部
     </strong>
     因此，针对各种边界情况设计测试用例，通常会取得很好的测试效果。
    </p>

    <h3>
     <a id="__272">
     </a>
     第九章 项目风险管理
    </h3>
    <p>
     软件风险概念：风险不同于软件工程过程中的其它事件，它有其自身的特点：
    </p>
    <p>
     ⑴ 可能性。项目风险是将来要发生的事情。虽然可以采取措施避免或减低风险发生所导致的损失，
     <strong>
      但完全消除项目风险是不太可能的
     </strong>
     。
    </p>
    <p>
     ⑵ 偶然性。首先，软件项目，特别是有较强领域特征的软件应用项目，是一次特定的开发过程，其产生的风险属于个别事件，有很大的不确定性。其次，软件开发的智力活动产生的风险难以预料和控制，并且人员受环境、心理、任务等因素的影响较大，更增加风险发生的偶然性。 疫情、核心骨干打球崴脚了。
    </p>
    <p>
     ⑶ 复杂性。软件项目各部分间的非线性关系，其复杂度远远超过由少数几人就能掌握的程度，需要通过有效的技术分解和团队式管理才能胜任。
    </p>
    <p>
     ⑷ 需求的变动。需求的不断变化，导致软件项目过程的不断变化，
     <strong>
      风险平衡状态
     </strong>
     的出现是动态而短暂的，但其造成的后果可能是严重的，甚至是灾难性的。
    </p>

     （一）、UML题目练习
    </h3>
    <p>
     某教育公司决定开发一个管理所有客户信息的交互式网络系统。系统的功能如下:
     <br/>
     (1)浏览客户信息:任何使用Internet的网络用户都可以浏览公司所有的客户信息(包括 姓名、住址、电话号码等)。
    </p>
    <p>
     (2)登录:公司授予每个客户一个帐号。拥有授权帐号的客户，可以使用系统提供的页面 设置个人密码，并使用该帐号和密码向系统注册。
    </p>
    <p>
     (3)修改个人信息:客户向系统注册后，可以发送电子邮件或者使用系统提供的页面，对个人 信息进行修改。
    </p>
    <p>
     (4)删除客户信息:只有公司的管理人员才能删除不再接受公司服务的客户的信息。 系统采用面向对象方法进行开发。
    </p>
    <p>
     <strong>
      【问题1】
     </strong>
     <br/>
     在需求分析阶段，采用UML的用例图描述系统功能需求，如下图所示。请指出图中的A、B、 C和D分别是哪个用例?
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/35b7d01a9426883e2c2587227389c5eb.png"/>
     <br/>
     【问题2】类的关联关系中关联的直线两端标明重数，例如：例如，图中的类网络用户和客户信息表，网络用户端的“0…*”表示:一个客户信息表的实例可以与0个或多个网络用户的实例相关联。 请指出图中(1)到(4)处的值分别为多少?
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/da166f86addeaa4189fd7a70ff521e3e.png"/>
     <br/>
     <strong>
      【问题3】类通常不会单独存在，因此当对系统建模时，不仅要识别出类，还必须对类之间的相互关系建模。在本题中，提供了三种关系:泛化关系、聚合关系、关联关系。请分别说明这三种关系的含义，并说明关联关系和聚合关系之间的主要区别。
     </strong>
    </p>
    <p>
     <strong>
      泛化关系
     </strong>
     ：也可以视为继承，表示类与类(或者接口与接口) 之间的父子关系。
     <br/>
     <strong>
      关联关系
     </strong>
     ：是类和类之间最常见的关系。即一个类的对象和另外一个类有联系。关联表示类与类之间的连接，它使一个类知道另一个类的属性和方法。
     <br/>
     <strong>
      聚合关系
     </strong>
     ：聚合是整体与部分的关系，但个体有独立的生命周期。
    </p>
    <p>
     <strong>
      关联关系和聚合关系之间的主要区别：聚合是一种特殊的关联，它表示整体/部分关系。
     </strong>
    </p>
