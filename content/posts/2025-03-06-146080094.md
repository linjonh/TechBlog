    <h2>
     <a id="_1">
     </a>
     一、计数器算法
    </h2>
    <p>
     <strong>
      原理
     </strong>
     ：
    </p>
    <ul>
     <li>
      使用一个固定窗口（时间段）内的计数器，记录请求的数量。
     </li>
     <li>
      若计数器超过设定的阈值，则拒绝后续请求。
     </li>
    </ul>
    <p>
     <strong>
      实现
     </strong>
     ：
    </p>
    <ul>
     <li>
      定义一个时间窗口（如1秒）。
     </li>
     <li>
      在窗口内维护一个计数器，接收到请求时计数器+1。
     </li>
     <li>
      若计数器超过阈值，则限流；否则允许。
     </li>
    </ul>
    <p>
     <strong>
      优点
     </strong>
     ：
     <br/>
     简单易实现，适用于低精度的限流需求。
    </p>
    <p>
     <strong>
      缺点
     </strong>
     ：
     <br/>
     边界问题：窗口边界处可能产生突发流量。例如，当前窗口接近阈值，新窗口开始时流量激增。
    </p>
    <h2>
     <a id="_16">
     </a>
     二、滑动窗口计数器
    </h2>
    <p>
     <strong>
      原理
     </strong>
     ：
    </p>
    <ul>
     <li>
      将固定窗口进一步细分为更小的时间片段。
     </li>
     <li>
      根据当前时间的滑动，将多个时间片段内的请求数量进行累加。
     </li>
    </ul>
    <p>
     <strong>
      实现
     </strong>
     ：
    </p>
    <ul>
     <li>
      将1秒分为多个时间片段（如100毫秒）。
     </li>
     <li>
      使用一个队列存储各时间片段的计数。
     </li>
     <li>
      请求时移除过期的时间片段，并累计最新的请求数。
     </li>
     <li>
     </li>
    </ul>
    <p>
     <strong>
      优点
     </strong>
     ：
    </p>
    <ul>
     <li>
      缓解固定窗口算法的边界问题。
     </li>
     <li>
      流量控制更加平滑。
     </li>
     <li>
     </li>
    </ul>
    <p>
     <strong>
      缺点
     </strong>
     ：
     <br/>
     实现较为复杂，存储和计算开销增加。
    </p>
    <h2>
     <a id="_34">
     </a>
     三、漏桶算法
    </h2>
    <p>
     <strong>
      原理
     </strong>
     ：
    </p>
    <ul>
     <li>
      将请求存入一个“桶”中，以恒定速率从桶中取出处理。
     </li>
     <li>
      若请求到达速度超过出水速度，则多余的请求被丢弃或排队。
     </li>
     <li>
     </li>
    </ul>
    <p>
     <strong>
      实现
     </strong>
     ：
    </p>
    <ul>
     <li>
      维护一个队列表示漏桶。
     </li>
     <li>
      按固定速率处理请求。
     </li>
     <li>
      若队列满，则丢弃新请求。
     </li>
     <li>
     </li>
    </ul>
    <p>
     <strong>
      优点
     </strong>
     ：
    </p>
    <ul>
     <li>
      能平滑突发流量，将其转化为稳定输出。
     </li>
     <li>
      控制请求处理速率。
     </li>
     <li>
     </li>
    </ul>
    <p>
     <strong>
      缺点
     </strong>
     ：
     <br/>
     丢弃超量请求可能导致用户体验下降。
    </p>
    <h2>
     <a id="_51">
     </a>
     四、令牌桶算法
    </h2>
    <p>
     <strong>
      原理
     </strong>
     ：
    </p>
    <ul>
     <li>
      系统按固定速率向桶中加入“令牌”。
     </li>
     <li>
      每次请求需要消耗一个令牌，若令牌不足，则拒绝请求。
     </li>
    </ul>
    <p>
     <strong>
      实现
     </strong>
     ：
    </p>
    <ul>
     <li>
      维护一个桶存储令牌，设定令牌生成速率和桶容量。
     </li>
     <li>
      请求时，检查桶内是否有足够令牌：
     </li>
     <li>
      有：扣除相应令牌，允许请求。
     </li>
     <li>
      无：拒绝请求。
     </li>
    </ul>
    <p>
     <strong>
      优点
     </strong>
     ：
    </p>
    <ul>
     <li>
      支持突发流量处理（通过允许提前消费桶中积累的令牌）。
     </li>
     <li>
      限流效果较平滑。
     </li>
     <li>
     </li>
    </ul>
    <p>
     <strong>
      缺点
     </strong>
     ：
     <br/>
     相比漏桶算法实现稍复杂。
    </p>
    <h2>
     <a id="Sliding_Log_69">
     </a>
     五、Sliding Log（滑动日志算法）
    </h2>
    <p>
     <strong>
      原理
     </strong>
     ：
    </p>
    <ul>
     <li>
      记录每个请求的时间戳。
     </li>
     <li>
      根据当前时间窗口计算历史请求数量，决定是否限流。
     </li>
     <li>
     </li>
    </ul>
    <p>
     <strong>
      实现
     </strong>
     ：
    </p>
    <ul>
     <li>
      使用一个有序集合（如Redis的zset）存储时间戳。
     </li>
     <li>
      请求时清理超出时间窗口的过期时间戳，并统计剩余时间戳数量。
     </li>
     <li>
      若数量未超限，则允许请求；否则限流。
     </li>
     <li>
     </li>
    </ul>
    <p>
     <strong>
      优点
     </strong>
     ：
     <br/>
     精度高，适合精细化限流。
    </p>
    <p>
     <strong>
      缺点
     </strong>
     ：
     <br/>
     存储开销大，尤其在高并发下可能性能较低。
    </p>
