    <h2>
     <a id="_1">
     </a>
     概述
    </h2>
    <p>
     基于体系结构（架构）的软件设计（Architecture-Based Software Design，ABSD）方法。
    </p>
    <p>
     ABSD方法时
     <strong>
      体系结构驱动
     </strong>
     ，即指构成体系结构的商业、质量和功能需求的组合驱动的。在基于体系结构的软件设计方法中，采用
     <strong>
      视角与视图
     </strong>
     来描述软件架构，采用
     <strong>
      用例
     </strong>
     来描述功能需求，采用
     <strong>
      质量场景
     </strong>
     来描述质量需求。
    </p>
    <p>
     ABSD方法是一个
     <strong>
      自顶向下、递归细化
     </strong>
     的方法，软件系统的体系结构通过该方法得到细化，直到能产生软件构件和类。
    </p>
    <p>
     ABSD方法有三个基础：
    </p>
    <ol>
     <li>
      <strong>
       功能的分解
      </strong>
      ：ABSD方法使用已有的、基于模块的内聚和耦合技术。
     </li>
     <li>
      <strong>
       通过选择体系结构风格来实现质量和商业需求
      </strong>
      。
     </li>
     <li>
      <strong>
       软件模板的使用
      </strong>
      。
     </li>
    </ol>
    <h2>
     <a id="ABSDM_14">
     </a>
     基于体系结构的开发模型-ABSDM
    </h2>
    <p>
     ABSDM把整个基于体系结构的软件过程划分为
     <strong>
      体系结构需求、体系结构设计、体系结构文档化、体系结构复审、体系结构实现、体系结构演化
     </strong>
     的子过程，直到能产生软件构件和类。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/9a895789f2174b65bb0b084e67c3640c.png"/>
    </p>
    <h3>
     <a id="_18">
     </a>
     体系结构需求
    </h3>
    <p>
     需求是指用户对目标软件系统在功能、行为、性能、设计约束等方面的期望。
     <br/>
     体系结构需求过程主要是
     <strong>
      获取用户需求、标识系统中所要用到的构件，可以复用需求库中已有的构件
     </strong>
     。
     <br/>
     主要过程：
    </p>
    <ol>
     <li>
      需求获取
     </li>
     <li>
      标识构件：为系统生成初始逻辑结构，包含大致的构件。
      <ol>
       <li>
        第一步：生成类图。
       </li>
       <li>
        第二步：对类进行分组。在生成的类图的基础上，使用一些标准对类进行分组可以大大简化类图结构，使之更清晰。
       </li>
       <li>
        第三步：把类打包成构件。
       </li>
      </ol>
     </li>
     <li>
      需求评审
      <ol>
       <li>
        组织一个由不同代表（如分析人员、客户、设计人员、测试人员）组成的小组，对体系结构需求及相关构件进行仔细的审查。
       </li>
       <li>
        审查的主要内容包括所获取的需求是否真实反映了用户的要求，类的分组是否合理，构件合并是否合理等。
       </li>
       <li>
        必要时，可以再“需求获取—标识构件—需求评审”之间进行迭代。
        <br/>
        <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/144dfe600a6342ab9ad449ee25034c31.png"/>
       </li>
      </ol>
     </li>
    </ol>
    <h3>
     <a id="_34">
     </a>
     体系结构设计
    </h3>
    <p>
     主要过程：
    </p>
    <ol>
     <li>
      提出体系结构模型：提出架构风格
     </li>
     <li>
      映射构件：按照架构风格的思路映射构件
     </li>
     <li>
      分析构件相互作用
     </li>
     <li>
      产生体系结构
     </li>
     <li>
      设计评审
     </li>
    </ol>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/1cfd019fdb6940e3a8ec850698f2434a.png"/>
    </p>
    <h3>
     <a id="_45">
     </a>
     体系结构文档化
    </h3>
    <p>
     主要输出结果是
     <strong>
      体系结构规格说明
     </strong>
     和测试体系结构需求的
     <strong>
      质量设计说明书
     </strong>
     这两个文件。
    </p>
    <h3>
     <a id="_47">
     </a>
     体系结构复审
    </h3>
    <p>
     体系结构复审是一个迭代过程，目的是标识出
     <strong>
      潜在的风险
     </strong>
     尽早发现体系结构设计中的
     <strong>
      缺陷
     </strong>
     和
     <strong>
      错误
     </strong>
     、
    </p>
    <p>
     从这个方面来说，在一个主版本的软件体系结构分析之后，要安排一次由外部人员（
     <strong>
      用户代表
     </strong>
     和
     <strong>
      领域专家
     </strong>
     ）参加的复审。
    </p>
    <h3>
     <a id="_51">
     </a>
     体系结构实现
    </h3>
    <p>
     主要过程：
    </p>
    <ol>
     <li>
      复审后的文档化的体系结构
     </li>
     <li>
      分析与设计
     </li>
     <li>
      构件实现
     </li>
     <li>
      构件组装
     </li>
     <li>
      系统测试
     </li>
     <li>
      体系结构演化
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/9a855f2f3a874d43a532eff76b684350.png"/>
     </li>
    </ol>
    <h3>
     <a id="_62">
     </a>
     体系结构演化
    </h3>
    <p>
     主要过程：
    </p>
    <ol>
     <li>
      需求变化归类
     </li>
     <li>
      体系结构演化计划
     </li>
     <li>
      构件变动：更新构件库
     </li>
     <li>
      更新构件的相互作用
     </li>
     <li>
      构件组装与测试
     </li>
     <li>
      技术评审
     </li>
     <li>
      演化后的体系结构
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/815950aec6e2484283ea697b7fdd59b7.png"/>
     </li>
    </ol>
