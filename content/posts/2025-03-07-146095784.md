     将 一个分支的提交 合并到 到另一个上分支上去

    <h3 id="%E4%B8%80%E3%80%81git%20merge%EF%BC%9A%E4%BF%9D%E7%95%99%E4%BA%86%E5%8E%9F%E6%9C%89%E5%88%86%E6%94%AF%E7%9A%84%E6%8F%90%E4%BA%A4%E7%BB%93%E6%9E%84" name="%E4%B8%80%E3%80%81git%20merge%EF%BC%9A%E4%BF%9D%E7%95%99%E4%BA%86%E5%8E%9F%E6%9C%89%E5%88%86%E6%94%AF%E7%9A%84%E6%8F%90%E4%BA%A4%E7%BB%93%E6%9E%84">
     一、git merge：保留了原有分支的提交结构
    </h3>
    <p>
     <img alt="" height="383" src="https://i-blog.csdnimg.cn/direct/39a98a7faa3b49a3819c7482caef2b27.png" width="587"/>
    </p>
    <p>
     现有一个模型如上所示：master 分支有1 2 3 5 7节点， develop 分支有4 6节点；
    </p>
    <pre><code>在 master 分支上执行 git merge 分支名  

含义：将 develop 上的分支提交合并到 master 上，合并的时候是基于3 7 6节点合并，生成最新的提交记录 ——&gt; 8节点；</code></pre>
    <h3 id="%E4%BA%8C%E3%80%81git%20rebase%EF%BC%9A%E6%8F%90%E4%BA%A4%E5%88%86%E6%94%AF%E6%9B%B4%E5%8A%A0%E6%95%B4%E6%B4%81" name="%E4%BA%8C%E3%80%81git%20rebase%EF%BC%9A%E6%8F%90%E4%BA%A4%E5%88%86%E6%94%AF%E6%9B%B4%E5%8A%A0%E6%95%B4%E6%B4%81">
     二、git rebase：提交分支更加整洁
    </h3>
    <p>
     <img alt="" height="406" src="https://i-blog.csdnimg.cn/direct/2f72542e0a9d4e5db1685b6337f2215c.png" width="625"/>
    </p>
    <pre><code>git rebase origin/分支名

一般：
git rebase origin/master</code></pre>
    <h3 id="%E4%B8%89%E3%80%81git%20stash" name="%E4%B8%89%E3%80%81git%20stash">
     三、git stash
    </h3>
    <p>
     <img alt="" height="469" src="https://i-blog.csdnimg.cn/direct/d69e31cb07594f2f9a603269fe1367c3.png" width="678"/>
    </p>
    <p>
     <strong>
      git stash：
     </strong>
     将工作区中已经保存的 和 暂存区的代码进行压栈保存；
    </p>
    <h4 id="%E4%BD%BF%E7%94%A8%E5%9C%BA%E6%99%AF%EF%BC%9A" name="%E4%BD%BF%E7%94%A8%E5%9C%BA%E6%99%AF%EF%BC%9A">
     <strong>
      使用场景：
     </strong>
    </h4>
    <p>
     场景一：在分支 A 上正在工作，分支 B 上有个 bug 要处理；
    </p>
    <pre><code>git stash            // 在分支 A 上执行该操作，将分支A上的修改代码进行保存
git checkout B       // 切到分支 B，并处理 B 分支上的 bug
git checkout A       // 切回到分支 A
git stash pop        // 将 A 分支之前保存的修改代码进行恢复操作</code></pre>
    <p>
     场景二：冲突：代码开发完了准备提交，在 commit 之前，建议拉一下远端的代码；
    </p>
    <pre><code>git stash                 // 将本地修改代码进行保存
git pull                  // 拉取远端代码
git stash pop             // 将保存的代码弹出
本地处理可能出现的冲突
git commit                // 提交操作
git push</code></pre>
    <p>
     <strong>
      实例演示：
     </strong>
    </p>
    <p>
     <img alt="" height="664" src="https://i-blog.csdnimg.cn/direct/b748b0034be14f54bcc028851b824e79.png" width="767"/>
    </p>
    <h2 id="%E5%9B%9B%E3%80%81git%20cherry-pick" name="%E5%9B%9B%E3%80%81git%20cherry-pick">
     四、git cherry-pick
    </h2>
    <p>
     将一个分支的提交记录，合并到另外一个分支；
    </p>
    <p>
     <img alt="" height="422" src="https://i-blog.csdnimg.cn/direct/a6b32d935ef04294ae9bbb1d3ca99d3e.png" width="696"/>
    </p>
    <pre><code>git cherry-pick commitid                   // 提交一个commit记录
git cherry-pick commitid1 commitid2        // 提交多个commit记录
git cherry-pick commitid1..commitid2       // 提交多个连续commit记录
git cherry-pick commitid1^..commitid2      // 提交多个连续commit记录，不包含commitid1记录</code></pre>
