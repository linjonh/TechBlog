     单片机OTA升级中Bootloader怎么判断APP有没有问题？
    </h1>
   </div>
  </div>
 </div>

    <p>
     没开发过OTA的工程师，职业生涯是不完整的。因为它能让设备远程更新功能，太方便了，产品有了这个功能，再也不会跟硬件工程师一起背锅了。
    </p>
    <p>
    </p>
    <p>
     不过，新手玩OTA，搞不好，也会翻车，比如下载过程中网络掉了，新固件不完整，设备可能直接"变砖"了。
    </p>
    <p>
    </p>
    <p>
     今天就教你，怎么在Bootloader加靠谱的验证机制，这样哪怕网络断了，数据错了，都不会显得这么狼狈。
    </p>
    <p>
    </p>
    <p>
     在单片机OTA（空中升级）中，Bootloader是个很重要的角色，它得确保新下载的固件（也就是APP）没问题，才能放心加载运行。
    </p>
    <p>
    </p>
    <p>
     如果APP有毛病，比如下载不全、数据坏了或者被篡改，设备可能会罢工甚至出大乱子。
    </p>
    <p>
    </p>
    <p>
     所以，Bootloader得有一套办法来检查APP到底靠不靠谱。
    </p>
    <p>
    </p>
    <p>
     下面我就用尽量通俗的语言，讲讲Bootloader常见的判断方法，顺便加点代码示例帮你更好理解。
    </p>
    <p>
    </p>
    <h5>
     一、为什么要检查APP有没有问题？
    </h5>
    <p>
     简单来说，Bootloader检查APP是为了避免这些麻烦：
    </p>
    <ul>
     <li>
      <p>
       <strong>
        数据坏了
       </strong>
       ：下载时信号不好或者存储器出问题，APP内容可能出错。
      </p>
     </li>
     <li>
      <p>
       <strong>
        没下全
       </strong>
       ：网络断了一会儿，APP可能只下了一半。
      </p>
     </li>
     <li>
      <p>
       <strong>
        被人动了手脚
       </strong>
       ：固件被恶意改过，可能会恶意破坏/控制设备。
      </p>
     </li>
     <li>
      <p>
       <strong>
        版本不对
       </strong>
       ：新APP跟硬件或者Bootloader不搭，跑不起来。
      </p>
     </li>
     <li>
      <p>
       <strong>
        放错地方了
       </strong>
       ：APP没存到该存的地方，Bootloader找不到。
      </p>
     </li>
    </ul>
    <p>
     所以，Bootloader相当于是个"质检员"，把APP好好检查一遍。
    </p>
    <p>
    </p>
    <h5>
     二、Bootloader怎么检查APP？
    </h5>
    <p>
     下面是Bootloader常用的几种方法，咱一个一个来看。
    </p>
    <p>
    </p>
    <h6>
     1. 用校验和或者CRC查数据的“身份证”
    </h6>
    <ul>
     <li>
      <p>
       <strong>
        啥意思
       </strong>
       ：算个“校验值”跟APP自带的值比对，看数据有没有丢包或者坏掉。
      </p>
     </li>
     <li>
      <p>
       <strong>
        咋做
       </strong>
       ：APP里带个校验值，Bootloader收到后自己再算一遍，比对一下。
      </p>
     </li>
     <li>
      <p>
       <strong>
        代码示例
       </strong>
      </p>
     </li>
     <li>
      <p>
       <img alt="" height="526" src="https://i-blog.csdnimg.cn/direct/f0862240b2974481be8c9baebe2a6cb7.png" width="1280"/>
      </p>
     </li>
    </ul>
    <p>
     这是无际单片机项目6的CRC16算法例子。
    </p>
    <p>
    </p>
    <p>
     我们在服务器下载APP固件数据的时候，会把数据分包，比如一帧数据是200-255个字节，每帧数据都会进行CRC16校验。
    </p>
    <p>
     <img alt="" height="516" src="https://i-blog.csdnimg.cn/direct/fc76a8467dde453e96d543db934084bd.png" width="1280"/>
    </p>
    <ul>
     <li>
      <p>
       <strong>
        CRC16好处
       </strong>
       ：简单，算得快，单片机也能轻松搞定。
      </p>
     </li>
     <li>
      <p>
       <strong>
        缺点
       </strong>
       ：只能查数据坏没坏，挡不住故意篡改。
      </p>
     </li>
    </ul>
    <p>
    </p>
    <h6>
     2. 用数字签名验“出身”
    </h6>
    <ul>
     <li>
      <p>
       <strong>
        啥意思
       </strong>
       ：就像查身份证真假，得确认APP是正宗的，没被改过。
      </p>
     </li>
     <li>
      <p>
       <strong>
        咋做
       </strong>
       ：发送方用私钥给APP“盖章”，Bootloader用公钥验这个章。
      </p>
     </li>
     <li>
      <p>
       <strong>
        代码示例
       </strong>
       （简单模拟一下）：
      </p>
     </li>
    </ul>
    <pre><code class="language-cpp">#define SIGN_ADDR (APP_START + APP_SIZE + 4)  // 签名放的位置

uint8_t signature[64];
memcpy(signature, (uint8_t *)SIGN_ADDR, 64);  // 取出签名

if (!verify_signature(app_data, APP_SIZE, signature)) {  // 假设有验证函数
    // 签名不对，可能是假货！
    printf("签名挂了，APP不靠谱！\n");
}</code></pre>
    <ul>
     <li>
      <p>
       <strong>
        好处
       </strong>
       ：安全性高，篡改不了。
      </p>
     </li>
     <li>
      <p>
       <strong>
        缺点
       </strong>
       ：算起来费劲，单片机可能有点吃力。
      </p>
     </li>
    </ul>
    <p>
    </p>
    <h6>
     3. 看版本号对不对
    </h6>
    <ul>
     <li>
      <p>
       <strong>
        啥意思
       </strong>
       ：检查APP的版本，确保跟硬件和Bootloader能搭上。
      </p>
     </li>
     <li>
      <p>
       <strong>
        咋做
       </strong>
       ：从APP里读出版本号，比对一下。
      </p>
     </li>
     <li>
      <p>
       <strong>
        代码示例
       </strong>
       ：
      </p>
     </li>
     <li>
      <pre><code class="language-cpp">#define MIN_VERSION 0x0200  // 最低版本要求

uint16_t app_version = *(uint16_t *)(APP_START + 8);  // 版本号位置
if (app_version &lt; MIN_VERSION) {
    // 版本太老，不行！
    printf("APP版本太低，跑不了！\n");
}</code></pre>
     </li>
     <li>
      <p>
       <strong>
        好处
       </strong>
       ：简单，能避免版本不匹配。
      </p>
     </li>
     <li>
      <p>
       <strong>
        缺点
       </strong>
       ：只管版本，数据坏不坏它不管。
      </p>
     </li>
    </ul>
    <p>
    </p>
    <p>
    </p>
    <h6>
     4. 检查APP放的位置对不对
    </h6>
    <ul>
     <li>
      <p>
       <strong>
        啥意思
       </strong>
       ：确保APP在正确的存储区里。
      </p>
     </li>
     <li>
      <p>
       <strong>
        咋做
       </strong>
       ：看看地址和大小有没有跑偏。
      </p>
     </li>
     <li>
      <p>
       <strong>
        代码示例
       </strong>
       ：
      </p>
     </li>
    </ul>
    <pre><code class="language-cpp">#define FLASH_BASE 0x08000000
#define FLASH_END  0x08080000

if (APP_START &lt; FLASH_BASE || (APP_START + APP_SIZE) &gt; FLASH_END) {
    // 位置不对，APP有问题！
    printf("APP放错地方了！\n");
}</code></pre>
    <ul>
     <li>
      <p>
       <strong>
        好处
       </strong>
       ：快，能马上发现存错了。
      </p>
     </li>
     <li>
      <p>
       <strong>
        缺点
       </strong>
       ：只管位置，不管内容。
      </p>
     </li>
    </ul>
    <p>
    </p>
    <p>
    </p>
    <h6>
     5. 用个“通关密码”
    </h6>
    <ul>
     <li>
      <p>
       <strong>
        啥意思
       </strong>
       ：APP里有个特定标记，证明它没问题。
      </p>
     </li>
     <li>
      <p>
       <strong>
        咋做
       </strong>
       ：Bootloader下载完后写个标记，启动时检查。
      </p>
     </li>
     <li>
      <p>
       <strong>
        代码示例
       </strong>
       ：
      </p>
     </li>
    </ul>
    <pre><code class="language-cpp">#define FLAG_ADDR 0x08007FFC
#define FLAG_VALUE 0xA5A5

uint32_t flag = *(uint32_t *)FLAG_ADDR;
if (flag != FLAG_VALUE) {
    // 没通关密码，APP不靠谱！
    printf("标记不对，APP有问题！\n");
}</code></pre>
    <ul>
     <li>
      <p>
       <strong>
        好处
       </strong>
       ：简单，检查快。
      </p>
     </li>
     <li>
      <p>
       <strong>
        缺点
       </strong>
       ：容易被仿造，安全性不高。
      </p>
     </li>
    </ul>
    <p>
    </p>
    <h5>
     三、完整的检查流程
    </h5>
    <p>
     一般来说，Bootloader不会只用一种方法，而是把几种组合起来，比如：
    </p>
    <ol>
     <li>
      <p>
       先看看位置对不对。
      </p>
     </li>
     <li>
      <p>
       再算算CRC，确认数据没坏。
      </p>
     </li>
     <li>
      <p>
       用签名验一下真假。
      </p>
     </li>
     <li>
      <p>
       检查版本能不能用。
      </p>
     </li>
     <li>
      <p>
       最后看标记对不对。
      </p>
     </li>
    </ol>
    <p>
     每一步都得过关，APP才能算“合格”，不然Bootloader就得用老固件或者停下来。
    </p>
    <p>
    </p>
    <p>
    </p>
    <h5>
     四、单片机资源少咋办？
    </h5>
    <p>
     单片机不像电脑那么强，资源有限，所以得有点小聪明：
    </p>
    <ul>
     <li>
      <p>
       <strong>
        用硬件帮忙
       </strong>
       ：有的单片机有CRC模块，能省不少力。
      </p>
     </li>
     <li>
      <p>
       <strong>
        分块检查
       </strong>
       ：别一次算太多，分成小块慢慢验。
      </p>
     </li>
     <li>
      <p>
       <strong>
        留条后路
       </strong>
       ：可以划分2个内存区域，分别存老版本和新版本的APP，万一新APP有问题，能退回老版本。
      </p>
     </li>
    </ul>
    <p>
    </p>
    <h5>
     五、总结一下
    </h5>
    <p>
     Bootloader要判断APP靠不靠谱，常用的招数有校验和/CRC、数字签名、版本检查、位置检查和标记验证。这些方法各有各的强项，搭配起来用，能确保APP没毛病，既安全又好用。希望这些例子和解释能让你明白咋回事儿。
    </p>
