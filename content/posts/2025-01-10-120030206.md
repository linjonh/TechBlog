     <h2>
      彻底搞懂 Jest 单元测试框架
     </h2>
     <p>
      本文主要给大家深入了解 Jest 背后的运行原理，并从零开始简单实现一个 Jest 单元测试的框架，方便了解单元测试引擎是如何工作的，Jest 编写单测相信我们已经很熟悉了，但 Jest 是如何工作的我们可能还很陌生，那让我们一起走进 Jest 内心，一同探究单元测试引擎是如何工作的。
     </p>
     <p>
      先附上 Jest 核心引擎的代码实现给有需要的同学，欢迎关注和交流：https://github.com/Wscats/jest-tutorial
     </p>
     <h2>
      什么是 Jest
     </h2>
     <p>
      Jest 是 Facebook 开发的 Javascript 测试框架，用于创建、运行和编写测试的 JavaScript 库。
     </p>
     <p>
      Jest 作为 NPM 包发布，可以安装并运行在任何 JavaScript 项目中。Jest 是目前前端最流行的测试库之一。
     </p>
     <h2>
      测试意味着什么
     </h2>
     <p>
      在技术术语中，测试意味着检查我们的代码是否满足某些期望。例如：一个名为求和(
      <code>
       sum
      </code>
      )函数应该返回给定一些运算结果的预期输出。
     </p>
     <p>
      有许多类型的测试，很快你就会被术语淹没，但长话短说的测试分为三大类：
     </p>
     <ul>
      <li>
       <p>
        单元测试
       </p>
      </li>
      <li>
       <p>
        集成测试
       </p>
      </li>
      <li>
       <p>
        E2E 测试
       </p>
      </li>
     </ul>
     <h2>
      我怎么知道要测试什么
     </h2>
     <p>
      在测试方面，即使是最简单的代码块也可能使初学者也可能会迷惑。最常见的问题是“我怎么知道要测试什么？”。
     </p>
     <p>
      如果您正在编写网页，一个好的出发点是测试应用程序的每个页面和每个用户交互。但是网页其实也需要测试的函数和模块等代码单元组成。
     </p>
     <p>
      大多数时候有两种情况：
     </p>
     <ul>
      <li>
       <p>
        你继承遗留代码，其自带没有测试
       </p>
      </li>
      <li>
       <p>
        你必须凭空实现一个新功能
       </p>
      </li>
     </ul>
     <p>
      那该怎么办？对于这两种情况，你可以通过将测试视为：检查该函数是否产生预期结果。最典型的测试流程如下所示：
     </p>
     <ul>
      <li>
       <p>
        导入要测试的函数
       </p>
      </li>
      <li>
       <p>
        给函数一个输入
       </p>
      </li>
      <li>
       <p>
        定义期望的输出
       </p>
      </li>
      <li>
       <p>
        检查函数是否产生预期的输出
       </p>
      </li>
     </ul>
     <p>
      一般，就这么简单。掌握以下核心思路，编写测试将不再可怕：
     </p>
     <blockquote>
      <p>
       输入 -&gt; 预期输出 -&gt; 断言结果。
      </p>
     </blockquote>
     <h2>
      测试块，断言和匹配器
     </h2>
     <p>
      我们将创建一个简单的 Javascript 函数代码，用于 2 个数字的加法，并为其编写相应的基于 Jest 的测试
     </p>
     <pre class="has"><code class="language-go">const sum = (a, b) =&gt; a + b;
</code></pre>
     <p>
      现在，为了测试在同一个文件夹中创建一个测试文件，命名为
      <code>
       test.spec.js
      </code>
      ，这特殊的后缀是 Jest 的约定，用于查找所有的测试文件。我们还将导入被测函数，以便执行测试中的代码。Jest 测试遵循 BDD 风格的测试，每个测试都应该有一个主要的
      <code>
       test
      </code>
      测试块，并且可以有多个测试块，现在可以为
      <code>
       sum
      </code>
      方法编写测试块，这里我们编写一个测试来添加 2 个数字并验证预期结果。我们将提供数字为 1 和 2，并期望输出 3。
     </p>
     <p>
      <code>
       test
      </code>
      它需要两个参数：一个用于描述测试块的字符串，以及一个用于包装实际测试的回调函数。
      <code>
       expect
      </code>
      包装目标函数，并结合匹配器
      <code>
       toBe
      </code>
      用于检查函数计算结果是否符合预期。
     </p>
     <p>
      这是完整的测试：
     </p>
     <pre class="has"><code class="language-go">test("sum test", () =&gt; {
  expect(sum(1, 2)).toBe(3);
});
</code></pre>
     <p>
      我们观察上面代码有发现有两点：
     </p>
     <ul>
      <li>
       <p>
        <code>
         test
        </code>
        块是单独的测试块，它拥有描述和划分范围的作用，即它代表我们要为该计算函数
        <code>
         sum
        </code>
        所编写测试的通用容器。
       </p>
      </li>
      <li>
       <p>
        <code>
         expect
        </code>
        是一个断言，该语句使用输入 1 和 2 调用被测函数中的
        <code>
         sum
        </code>
        方法，并期望输出 3。
       </p>
      </li>
      <li>
       <p>
        <code>
         toBe
        </code>
        是一个匹配器，用于检查期望值，如果不符合预期结果则应该抛出异常。
       </p>
      </li>
     </ul>
     <h3>
      如何实现测试块
     </h3>
     <p>
      测试块其实并不复杂，最简单的实现不过如下，我们需要把测试包装实际测试的回调函数存起来，所以封装一个
      <code>
       dispatch
      </code>
      方法接收命令类型和回调函数：
     </p>
     <pre class="has"><code class="language-go">const test = (name, fn) =&gt; {
  dispatch({ type: "ADD_TEST", fn, name });
};
</code></pre>
     <p>
      我们需要在全局创建一个
      <code>
       state
      </code>
      保存测试的回调函数，测试的回调函数使用一个数组存起来。
     </p>
     <pre class="has"><code class="language-go">global["STATE_SYMBOL"] = {
  testBlock: [],
};
</code></pre>
     <p>
      <code>
       dispatch
      </code>
      方法此时只需要甄别对应的命令，并把测试的回调函数存进全局的
      <code>
       state
      </code>
      即可。
     </p>
     <pre class="has"><code class="language-go">const dispatch = (event) =&gt; {
  const { fn, type, name } = event;
  switch (type) {
    case "ADD_TEST":
      const { testBlock } = global["STATE_SYMBOL"];
      testBlock.push({ fn, name });
      break;
  }
};
</code></pre>
     <h3>
      如何实现断言和匹配器
     </h3>
     <p>
      断言库也实现也很简单，只需要封装一个函数暴露匹配器方法满足以下公式即可：
     </p>
     <blockquote>
      <p>
       <code>
        expect(A).toBe(B)
       </code>
      </p>
     </blockquote>
     <p>
      这里我们实现
      <code>
       toBe
      </code>
      这个常用的方法，当结果和预期不相等，抛出错误即可：
     </p>
     <pre class="has"><code class="language-go">const expect = (actual) =&gt; ({
    toBe(expected) {
        if (actual !== expected) {
            throw new Error(`${actual} is not equal to ${expected}`);
        }
    }
};
</code></pre>
     <p>
      实际在测试块中会使用
      <code>
       try/catch
      </code>
      捕获错误，并打印堆栈信息方面定位问题。
     </p>
     <p>
      在简单情况下，我们也可以使用 Node 自带的
      <code>
       assert
      </code>
      模块进行断言，当然还有很多更复杂的断言方法，本质上原理都差不多。
     </p>
     <h3>
      CLI 和配置
     </h3>
     <p>
      编写完测试之后，我们则需要在命令行中输入命令运行单测，正常情况下，命令类似如下：
     </p>
     <blockquote>
      <p>
       <code>
        node jest xxx.spec.js
       </code>
      </p>
     </blockquote>
     <p>
      这里本质是解析命令行的参数。
     </p>
     <pre class="has"><code class="language-go">const testPath = process.argv.slice(2)[0];
const code = fs.readFileSync(path.join(process.cwd(), testPath)).toString();
</code></pre>
     <p>
      复杂的情况可能还需要读取本地的 Jest 配置文件的参数来更改执行环境等，Jest 在这里使用了第三方库
      <code>
       yargs
      </code>
      <code>
       execa
      </code>
      和
      <code>
       chalk
      </code>
      等来解析执行并打印命令。
     </p>
     <h2>
      模拟
     </h2>
     <p>
      在复杂的测试场景，我们一定绕不开一个 Jest 术语：模拟(
      <code>
       mock
      </code>
      )
     </p>
     <p>
      在 Jest 文档中，我们可以找到 Jest 对模拟有以下描述：”模拟函数通过抹去函数的实际实现、捕获对函数的调用，以及在这些调用中传递的参数，使测试代码之间的链接变得容易“
     </p>
     <p>
      简而言之，可以通过将以下代码片段分配给函数或依赖项来创建模拟：
     </p>
     <pre class="has"><code class="language-go">jest.mock("fs", {
  readFile: jest.fn(() =&gt; "wscats"),
});
</code></pre>
     <p>
      这是一个简单模拟的示例，模拟了 fs 模块 readFile 函数在测试特定业务逻辑的返回值。
     </p>
     <h3>
      怎么模拟一个函数
     </h3>
     <p>
      接下来我们就要研究一下如何实现，首先是
      <code>
       jest.mock
      </code>
      ，它第一个参数接受的是模块名或者模块路径，第二个参数是该模块对外暴露方法的具体实现
     </p>
     <pre class="has"><code class="language-go">const jest = {
  mock(mockPath, mockExports = {}) {
    const path = require.resolve(mockPath, { paths: ["."] });
    require.cache[path] = {
      id: path,
      filename: path,
      loaded: true,
      exports: mockExports,
    };
  },
};
</code></pre>
     <p>
      我们方案其实跟上面的
      <code>
       test
      </code>
      测试块实现一致，只需要把具体的实现方法找一个地方存起来即可，等后续真正使用改模块的时候替换掉即可，所以我们把它存到
      <code>
       require.cache
      </code>
      里面，当然我们也可以存到全局的
      <code>
       state
      </code>
      中。
     </p>
     <p>
      而
      <code>
       jest.fn
      </code>
      的实现也不难，这里我们使用一个闭包
      <code>
       mockFn
      </code>
      把替换的函数和参数给存起来，方便后续测试检查和统计调用数据。
     </p>
     <pre class="has"><code class="language-go">const jest = {
  fn(impl = () =&gt; {}) {
    const mockFn = (...args) =&gt; {
      mockFn.mock.calls.push(args);
      return impl(...args);
    };
    mockFn.originImpl = impl;
    mockFn.mock = { calls: [] };
    return mockFn;
  },
};
</code></pre>
     <h2>
      执行环境
     </h2>
     <p>
      有些同学可能留意到了，在测试框架中，我们并不需要手动引入
      <code>
       test
      </code>
      、
      <code>
       expect
      </code>
      和
      <code>
       jest
      </code>
      这些函数，每个测试文件可以直接使用，所以我们这里需要创造一个注入这些方法的运行环境。
     </p>
     <h3>
      作用域隔离
     </h3>
     <p>
      由于单测文件运行时候需要作用域隔离。所以在设计上测试引擎是跑在 node 全局作用域下，而测试文件的代码则跑在 node 环境里的 vm 虚拟机局部作用域中。
     </p>
     <ul>
      <li>
       <p>
        全局作用域
        <code>
         global
        </code>
       </p>
      </li>
      <li>
       <p>
        局部作用域
        <code>
         context
        </code>
       </p>
      </li>
     </ul>
     <p>
      两个作用域通过
      <code>
       dispatch
      </code>
      方法实现通信。
     </p>
     <p>
      <code>
       dispatch
      </code>
      在 vm 局部作用域下收集测试块、生命周期和测试报告信息到 node 全局作用域
      <code>
       STATE_SYMBOL
      </code>
      中，所以
      <code>
       dispatch
      </code>
      主要涉及到以下各种通信类型：
     </p>
     <ul>
      <li>
       <p>
        测试块
       </p>
      </li>
      <li>
       <ul>
        <li>
         <p>
          <code>
           ADD_TEST
          </code>
         </p>
        </li>
       </ul>
      </li>
      <li>
       <p>
        生命周期
       </p>
      </li>
      <li>
       <ul>
        <li>
         <p>
          <code>
           BEFORE_EACH
          </code>
         </p>
        </li>
        <li>
         <p>
          <code>
           BEFORE_ALL
          </code>
         </p>
        </li>
        <li>
         <p>
          <code>
           AFTER_EACH
          </code>
         </p>
        </li>
        <li>
         <p>
          <code>
           AFTER_ALL
          </code>
         </p>
        </li>
       </ul>
      </li>
      <li>
       <p>
        测试报告
       </p>
      </li>
      <li>
       <ul>
        <li>
         <p>
          <code>
           COLLECT_REPORT
          </code>
         </p>
        </li>
       </ul>
      </li>
     </ul>
     <h3>
      V8 虚拟机
     </h3>
     <p>
      既然万事俱备只欠东风，我们只需要给 V8 虚拟机注入测试所需的方法，即注入测试局部作用域即可。
     </p>
     <pre class="has"><code class="language-go">const context = {
  console: console.Console({ stdout: process.stdout, stderr: process.stderr }),
  jest,
  expect,
  require,
  test: (name, fn) =&gt; dispatch({ type: "ADD_TEST", fn, name }),
};
</code></pre>
     <p>
      注入完作用域，我们就可以让测试文件的代码在 V8 虚拟机中跑起来，这里我传入的代码是已经处理成字符串的代码，Jest 这里会在这里做一些代码加工，安全处理和 SourceMap 缝补等操作，我们示例就不需要搞那么复杂了。
     </p>
     <pre class="has"><code class="language-go">vm.runInContext(code, context);
</code></pre>
     <p>
      在代码执行的前后可以使用时间差算出单测的运行时间，Jest 还会在这里预评估单测文件的大小数量等，决定是否启用 Worker 来优化执行速度
     </p>
     <pre class="has"><code class="language-go">const start = new Date();
const end = new Date();
log("\x1b[32m%s\x1b[0m", `Time: ${end - start} ms`);
</code></pre>
     <h3>
      运行单测回调
     </h3>
     <p>
      V8 虚拟机执行完毕之后，全局的
      <code>
       state
      </code>
      就会收集到测试块中所有包装好的测试回调函数，我们最后只需要把所有的这些回调函数遍历取出来，并执行。
     </p>
     <pre class="has"><code class="language-go">testBlock.forEach(async (item) =&gt; {
  const { fn, name } = item;
  await fn.apply(this);
});
</code></pre>
     <h3>
      钩子函数
     </h3>
     <p>
      我们还可以在单测执行过程中加入生命周期，例如
      <code>
       beforeEach
      </code>
      ，
      <code>
       afterEach
      </code>
      ，
      <code>
       afterAll
      </code>
      和
      <code>
       beforeAll
      </code>
      等钩子函数。
     </p>
     <p>
      在上面的基础架构上增加钩子函数，其实就是在执行 test 的每个过程中注入对应回调函数，比如
      <code>
       beforeEach
      </code>
      就是放在
      <code>
       testBlock
      </code>
      遍历执行测试函数前，
      <code>
       afterEach
      </code>
      就是放在
      <code>
       testBlock
      </code>
      遍历执行测试函数后，非常的简单，只需要位置放对就可以暴露任何时期的钩子函数。
     </p>
     <pre class="has"><code class="language-go">testBlock.forEach(async (item) =&gt; {
  const { fn, name } = item;
  beforeEachBlock.forEach(async (beforeEach) =&gt; await beforeEach());
  await fn.apply(this);
  afterEachBlock.forEach(async (afterEach) =&gt; await afterEach());
});
</code></pre>
     <p>
      而
      <code>
       beforeAll
      </code>
      和
      <code>
       afterAll
      </code>
      就可以放在，
      <code>
       testBlock
      </code>
      所有测试运行完毕前和后。
     </p>
     <pre class="has"><code class="language-go">beforeAllBlock.forEach(async (beforeAll) =&gt; await beforeAll());
testBlock.forEach(async (item) =&gt; {})
afterAllBlock.forEach(async (afterAll) =&gt; await afterAll());
</code></pre>

     </h2>
     <p>

        </h2>
        <p>
         紧接着
         <code>
          runTestInternal
         </code>
         中的
         <code>
          testFramework
         </code>
         会接受传入的 runtime 调用单测文件运行，
         <code>
          testFramework
         </code>
         方法来自于一个名字比较有意思的库
         <code>
          packages/jest-circus/src/legacy-code-todo-rewrite/jestAdapter.ts
         </code>
         ，其中
         <code>
          legacy-code-todo-rewrite
         </code>
         意思为
         <strong>
          遗留代码待办事项重写
         </strong>
         ，
         <code>
          jest-circus
         </code>
         主要会把全局
         <code>
          global
         </code>
         的一些方法进行重写，涉及这几个：
        </p>
        <ul>
         <li>
          <p>
           afterAll
          </p>
         </li>
         <li>
          <p>
           afterEach
          </p>
         </li>
         <li>
          <p>
           beforeAll
          </p>
         </li>
         <li>
          <p>
           beforeEach
          </p>
         </li>
         <li>
          <p>
           describe
          </p>
         </li>
         <li>
          <p>
           it
          </p>
         </li>
         <li>
          <p>
           test
          </p>
         </li>
        </ul>
        <img src="https://i-blog.csdnimg.cn/blog_migrate/0746271bf46b509f0b14eb3b75e90aa1.png" title="image">
         <figcaption>
          image
         </figcaption>
         <p>
          这里调用单测前会在
          <code>
           jestAdapter
          </code>
          函数中，也就是上面提到的
          <code>
           runtime.requireModule
          </code>
          加载
          <code>
           xxx.spec.js
          </code>
          文件，这里执行之前已经使用
          <code>
           initialize
          </code>
          预设好了执行环境
          <code>
           globals
          </code>
          和
          <code>
           snapshotState
          </code>
          ，并改写
          <code>
           beforeEach
          </code>
          ，如果配置了
          <code>
           resetModules
          </code>
          ，
          <code>
           clearMocks
          </code>
          ，
          <code>
           resetMocks
          </code>
          ，
          <code>
           restoreMocks
          </code>
          和
          <code>
           setupFilesAfterEnv
          </code>
          则会分别执行下面几个方法：
         </p>
         <ul>
          <li>
           <p>
            runtime.resetModules
           </p>
          </li>
          <li>
           <p>
            runtime.clearAllMocks
           </p>
          </li>
          <li>
           <p>
            runtime.resetAllMocks
           </p>
          </li>
          <li>
           <p>
            runtime.restoreAllMocks
           </p>
          </li>
          <li>
           <p>
            runtime.requireModule 或者 runtime.unstable_importModule
           </p>
          </li>
         </ul>
         <p>
          当运行完
          <code>
           initialize
          </code>
          方法初始化之后，由于
          <code>
           initialize
          </code>
          改写了全局的
          <code>
           describe
          </code>
          和
          <code>
           test
          </code>
          等方法，这些方法都在
          <code>
           /packages/jest-circus/src/index.ts
          </code>
          这里改写，这里注意
          <code>
           test
          </code>
          方法里面有一个
          <code>
           dispatchSync
          </code>
          方法，这是一个关键的方法，这里会在全局维护一份
          <code>
           state
          </code>
          ，
          <code>
           dispatchSync
          </code>
          就是把
          <code>
           test
          </code>
          代码块里面的函数等信息存到
          <code>
           state
          </code>
          里面，
          <code>
           dispatchSync
          </code>
          里面使用
          <code>
           name
          </code>
          配合
          <code>
           eventHandler
          </code>
          方法来修改
          <code>
           state
          </code>
          ，这个思路非常像 redux 里面的数据流。
         </p>
         <pre class="has"><code class="language-go">const test: Global.It = () =&gt; {
  return (test = (testName, fn, timeout) =&gt; (testName, mode, fn, testFn, timeout) =&gt; {
    return dispatchSync({
      asyncError,
      fn,
      mode,
      name: "add_test",
      testName,
      timeout,
    });
  });
};
</code></pre>
         <p>
          而单测
          <code>
           xxx.spec.js
          </code>
          即 testPath 文件会在
          <code>
           initialize
          </code>
          之后会被引入并执行，注意这里引入就会执行这个单测，由于单测
          <code>
           xxx.spec.js
          </code>
          文件里面按规范写，会有
          <code>
           test
          </code>
          和
          <code>
           describe
          </code>
          等代码块，所以这个时候所有的
          <code>
           test
          </code>
          和
          <code>
           describe
          </code>
          接受的回调函数都会被存到全局的
          <code>
           state
          </code>
          里面。
         </p>
         <pre class="has"><code class="language-go">const esm = runtime.unstable_shouldLoadAsEsm(testPath);
if (esm) {
  await runtime.unstable_importModule(testPath);
} else {
  runtime.requireModule(testPath);
}
</code></pre>
         <h2>
          jest-runtime
         </h2>
         <p>
          这里的会先判断是否 esm 模块，如果是则使用
          <code>
           unstable_importModule
          </code>
          的方式引入，否则使用
          <code>
           requireModule
          </code>
          的方式引入，具体会进入下面吗这个函数。
         </p>
         <pre class="has"><code class="language-go">this._loadModule(localModule, from, moduleName, modulePath, options, moduleRegistry);
</code></pre>
         <p>
          \_loadModule 的逻辑只有三个主要部分
         </p>
         <ul>
          <li>
           <p>
            判断是否 json 后缀文件，执行 readFile 读取文本，用 transformJson 和 JSON.parse 转格输出内容。
           </p>
          </li>
          <li>
           <p>
            判断是否 node 后缀文件，执行 require 原生方法引入模块。
           </p>
          </li>
          <li>
           <p>
            不满足上述两个条件的文件，执行 \_execModule 执行模块。
           </p>
          </li>
         </ul>
         <p>
          \_execModule 中会使用 babel 来转化 fs 读取到的源代码，这个
          <code>
           transformFile
          </code>
          就是
          <code>
           packages/jest-runtime/src/index.ts
          </code>
          的
          <code>
           transform
          </code>
          方法。
         </p>
         <pre class="has"><code class="language-go">const transformedCode = this.transformFile(filename, options);
</code></pre>
         <img src="https://i-blog.csdnimg.cn/blog_migrate/4e23ad2b468da0d83a5bd0b380c637bb.png" title="image">
          <figcaption>
           image
          </figcaption>
          <p>
           \_execModule 中会使用
           <code>
            createScriptFromCode
           </code>
           方法调用 node 的原生 vm 模块来真正的执行 js，vm 模块接受安全的源代码，并用 V8 虚拟机配合传入的上下文来立即执行代码或者延时执行代码，这里可以接受不同的作用域来执行同一份代码来运算出不同的结果，非常合适测试框架的使用，这里的注入的 vmContext 就是上面全局改写作用域包含 afterAll，afterEach，beforeAll，beforeEach，describe，it，test，所以我们的单测代码在运行的时候就会得到拥有注入作用域的这些方法。
          </p>
          <pre class="has"><code class="language-go">const vm = require("vm");
const script = new vm().Script(scriptSourceCode, option);
const filename = module.filename;
const vmContext = this._environment.getVmContext();
script.runInContext(vmContext, {
  filename,
});
</code></pre>
          <img src="https://i-blog.csdnimg.cn/blog_migrate/1fea3b987cdbfefe8020d9f52cea94fc.png" title="image"/>
          <figcaption>
           image
          </figcaption>
          <p>
           当上面复写全局方法和保存好
           <code>
            state
           </code>
           之后，会进入到真正执行
           <code>
            describe
           </code>
           的回调函数的逻辑里面，在
           <code>
            packages/jest-circus/src/run.ts
           </code>
           的
           <code>
            run
           </code>
           方法里面，这里使用
           <code>
            getState
           </code>
           方法把
           <code>
            describe
           </code>
           代码块取出来，然后使用
           <code>
            _runTestsForDescribeBlock
           </code>
           执行这个函数，然后进入到
           <code>
            _runTest
           </code>
           方法，然后使用
           <code>
            _callCircusHook
           </code>
           执行前后的钩子函数，使用
           <code>
            _callCircusTest
           </code>
           执行。
          </p>
          <pre class="has"><code class="language-go">const run = async (): Promise&lt;Circus.RunResult&gt; =&gt; {
  const { rootDescribeBlock } = getState();
  await dispatch({ name: "run_start" });
  await _runTestsForDescribeBlock(rootDescribeBlock);
  await dispatch({ name: "run_finish" });
  return makeRunResult(getState().rootDescribeBlock, getState().unhandledErrors);
};

const _runTest = async (test, parentSkipped) =&gt; {
  // beforeEach
  // test 函数块，testContext 作用域
  await _callCircusTest(test, testContext);
  // afterEach
};
</code></pre>
          <p>
           这是钩子函数实现的核心位置，也是 Jest 功能的核心要素。
          </p>
