    <h2>

     (一)概念梳理
    </h2>
    <h3>
     <a id="1_9">
     </a>
     1.什么是协议?
    </h3>
    <p>
     在现实生活中,
     <code>
      协议
     </code>
     是一种约定,经过多方的同意最终合作完成一件事情。
     <br/>
     在网络的中,计算机要帮助用户完成各种工作,包括用户之间的
     <code>
      远程通信
     </code>
     !
     <br/>
     具体来说，协议就是通信双方都认识的结构化的数据类型，比如一个
     <code>
      结构体类型
     </code>
     ，由于双方都事先通知了对方，所以一方可以结构化的向对方发送数据，并且对方也可以把收到的结构化的数据正确的解析出来。
     <br/>
     协议一般都有报头和有效载荷，报头是协议规定好的特定位置存储特定结构数据的头部；有效载荷就是将要传输的数据本身。
     <br/>
     <code>
      **举一个形象的例子：**报头和有效载荷的关系其实就像你发的快递的快递单和快递的关系（报头其实就是一个快递单，它记录的这个快递的相关信息，比如发送方是谁，接受方是谁；而有效载荷就是快递的内容）
     </code>
     <br/>
     <strong>
      报头和有效载荷共同构成了一个报文！
     </strong>
    </p>
    <h3>
     <a id="2_16">
     </a>
     2.什么是应用层?
    </h3>
    <p>
     说到应用层就不得不提到网络协议栈的分层。
     <code>
      **网络协议栈的5层（4层）协议分层是对OSI的七层模型的简化版本。**
     </code>
     <br/>
     <strong>
      OSI的七层模型：
     </strong>
     <br/>
     <img alt="OSI的七层分层" src="https://i-blog.csdnimg.cn/direct/bc948b0c96194f2f9467c9c8e38d1e62.png">
      <br/>
      OSI的七层分层是非常完善的，它综合考虑了各行各业的，各个厂商在实现的时候按照OSI的七层协议进行。
      <br/>
      但是经过实践，人们发现七层协议太过冗余，于是把OSI的七层分层
      <code>
       简化为5(或者4)层
      </code>
      ，形成了如今的
      <code>
       网络协议栈
      </code>
      。
     </img>
    </p>
    <p>
     <strong>
      网络协议栈的5（4）层分层：
     </strong>
     <br/>
     <img alt="网络协议栈的5层（4层）协议分层" src="https://i-blog.csdnimg.cn/direct/7d9edd894c34488c931813ccc92c8c60.png">
      而应用层就是网络协议栈的最上面的一层，因为它最贴近实际应用，因而被称为
      <code>
       引用层
      </code>
      。
     </img>
    </p>
    <h3>
     <a id="3__26">
     </a>
     3. 为什么要进行分层？
    </h3>
    <p>
     协议本质上是软件，在设计上的分层是为了更好的进行
     <code>
      模块化设计
     </code>
     ，便于
     <code>
      解耦和
     </code>
     ，设计为层状的结构优势在于：
     <br/>
     一层出现问题，可以快速定位，便于快速找到问题所在，同时分层后的软件层之间相互不影响，这也就让软件的维护成本更低。
    </p>
    <h2>
     <a id="HTTP_30">
     </a>
     （二）HTTP协议
    </h2>
    <h3>
     <a id="21_HTTP_31">
     </a>
     2.1 初识HTTP协议
    </h3>
    <p>
     协议看似“高大上”，其实我们每个人都可以定义一个协议，但是由于我们每一个人定义协议时考虑的内容不同，所以有很大的可能定义出来的协议五花八门。
     <br/>
     但实际上, 已经有大佬们定义了一些现成的, 又非常好用的应用层协议, 供我们直接参考使用. HTTP(超文本传输协议)就是其中之一。
     <br/>
     <code>
      HTTP
     </code>
     （
     <strong>
      H
     </strong>
     yper
     <strong>
      T
     </strong>
     ext
     <strong>
      T
     </strong>
     ransfer
     <strong>
      P
     </strong>
     rotocol，
     <strong>
      超文本传输协议
     </strong>
     ） 是一个应用广泛所以至关重要的协议。通过这个“超文本”传输协议我们不仅可以传输
     <code>
      文字内容（txt）
     </code>
     ，也可传输
     <code>
      图片（gpj）
     </code>
     ，
     <code>
      音频（mp3)
     </code>
     ，甚至是
     <code>
      视频（mp4）
     </code>
     文件。
     <br/>
     此外，HTTP协议是
     <code>
      无连接，无状态
     </code>
     的协议，即每次请求都需要建立新的链接，且服务器不会保存客户端信息。
    </p>
    <h3>
     <a id="22HTTPURL_36">
     </a>
     2.2HTTP协议的URL
    </h3>
    <p>
     随便打开一个网页，我们就可以看到网址一栏特有的内容：
     <br/>
     <img alt="HTTPS" src="https://i-blog.csdnimg.cn/direct/e5652fc49d6840cab3beac406593ae6e.png">
      这个网页使用的是
      <code>
       HTTPS
      </code>
      协议，但是HTTPS是对HTTP的完善，分析HTTPS的URL和讲解HTTP的URL并不冲突。
     </img>
    </p>
    <h4>
     <a id="221_39">
     </a>
     2.2.1域名
    </h4>
    <p>
     如图，这一串内容就是
     <code>
      URL
     </code>
     ：
     <br/>
     其中前面部分被称为域名，域名本质是IP地址，HTTP的接收方会自动把接收到的域名转化为IP地址，这个过程称为
     <code>
      DNS
     </code>
     。
     <br/>
     <img alt="域名" src="https://i-blog.csdnimg.cn/direct/c425e60f4ae941c981d80ee86bb43521.png"/>
    </p>
    <h4>
     <a id="222_43">
     </a>
     2.2.2端口号
    </h4>
    <p>
     我们知道，
     <code>
      网络通信本质就是进程间的通信
     </code>
     ，只有通过
     <code>
      IP+端口
     </code>
     才能确定两个特定的进程。
     <br/>
     HTTP协议由于广为人知，所以其端口号被规定为
     <code>
      80
     </code>
     ，为了方便起见，一般的浏览器会自动在域名后面加上端口号80。
    </p>
    <p>
     一般而言，协议名称和端口号是强关联的，比如HTTP的知名端口号为80；这意味着：当浏览器发起请求时，会自动拼接端口号80。
    </p>
    <h4>
     <a id="223_web_49">
     </a>
     2.2.3 web根目录
    </h4>
    <p>
     与Linux的根目录不同，每一台web服务器都可以设置一个
     <code>
      web根目录
     </code>
     ，用于存储服务器的资源。
     <br/>
     web根目录的名称一般设置为
     <code>
      wwwroot
     </code>
     ，以这个目录为根，形成了一颗目录树，这就给用户一种错觉：
     <strong>
      仿佛在访问服务器的根目录下的资源。
     </strong>
     <br/>
     <img alt="web根目录图解" src="https://i-blog.csdnimg.cn/direct/b4fd8eb0c153491db8000f010f874e22.png"/>
    </p>
    <h4>
     <a id="224__53">
     </a>
     2.2.4 什么是资源
    </h4>
    <p>
     资源指的是网页，图片，视频，音频等超文本内容，这些资源在被获取之前，存储在服务端。（即Linux的某目录的文件）
    </p>
    <h4>
     <a id="225_URL_56">
     </a>
     2.2.5 URL总结
    </h4>
    <p>
     URL前半部分是
     <code>
      IP和端口
     </code>
     。
     <br/>
     URL后半部分（域名和端口号后面部分）是服务端主机上的一个资源的
     <code>
      相对路径
     </code>
     。（
     <strong>
      这个路径标识了服务端机器上的某一资源的唯一性
     </strong>
     ）。此外，这个路径不是服务器主机的根目录，而是
     <code>
      web根目录
     </code>
     。
    </p>
    <h3>
     <a id="23_HTTP_60">
     </a>
     2.3 HTTP协议的报文
    </h3>
    <p>
     在网络传输的时候，HTTP协议的报文有自己的约定好的格式。
    </p>
    <p>
     HTTP报文是客户端与服务器之间通信的核心载体，主要分为
     <strong>
      请求报文
     </strong>
     和
     <strong>
      响应报文
     </strong>
     。
    </p>
    <hr/>
    <h4>
     <a id="231_HTTP_67">
     </a>
     2.3.1 HTTP请求报文结构
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/81a429c35555407387edd0afa5b79252.png">
      <strong>
       以上是
       <code>
        HTTP请求报文
       </code>
       结构的基本格式，接下来我将会分别讲解这个格式的各个部分的内容，以及它们在网络传输中的作用。
      </strong>
     </img>
    </p>
    <hr/>
    <h5>
     <a id="i_Request_Line_71">
     </a>
     i. 请求行（Request Line）
    </h5>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/88f1fc634c5c44d88c37653c932fe5a9.png"/>
    </p>
    <ul>
     <li>
      <strong>
       格式
      </strong>
      ：
      <code>
       请求方法 空格 URI 空格 HTTP版本 换行符
      </code>
     </li>
     <li>
      <strong>
       作用
      </strong>
      ：定义请求的基本操作、目标资源和协议版本。
     </li>
     <li>
      <strong>
       示例
      </strong>
      ：
      <pre><code class="prism language-http">GET /index.html?name=ddsm HTTP/1.1\r\n
</code></pre>
      <ul>
       <li>
        <strong>
         请求方法
        </strong>
        ：
        <code>
         GET
        </code>
        （获取资源）。
       </li>
       <li>
        <strong>
         URI
        </strong>
        ：
        <code>
         /index.html?name=ddsm
        </code>
        （请求路径和查询参数）。
       </li>
       <li>
        <strong>
         HTTP版本
        </strong>
        ：
        <code>
         HTTP/1.1
        </code>
        。
       </li>
      </ul>
     </li>
    </ul>
    <p>
     <strong>
      常见请求方法
     </strong>
     ：
    </p>
    <ul>
     <li>
      <code>
       GET
      </code>
      ：获取资源（无请求正文）。
     </li>
     <li>
      <code>
       POST
      </code>
      ：提交数据（有请求正文）。
     </li>
     <li>
      <code>
       PUT
      </code>
      ：更新服务器上的资源。
     </li>
     <li>
      <code>
       DELETE
      </code>
      ：删除服务器上的资源。
     </li>
    </ul>
    <p>
     其中，在网络传输中，最重要最常用（
     <code>
      95%以上
     </code>
     ）的方法就是
     <code>
      GET
     </code>
     和
     <code>
      POST
     </code>
     ；
     <strong>
      其他的方法都基本上不使用。
     </strong>
    </p>
    <hr/>
    <h5>
     <a id="ii_Request_Headers_94">
     </a>
     ii. 请求字段（Request Headers）
    </h5>
    <ul>
     <li>
      <strong>
       格式
      </strong>
      ：
      <code>
       Key:[空格]Value 换行符
      </code>
      (
      <strong>
       key-value键值对
      </strong>
      )
     </li>
     <li>
      <strong>
       作用
      </strong>
      ：传递附加信息（如客户端信息、内容类型、认证等）。
     </li>
     <li>
      <strong>
       示例
      </strong>
      ：
      <pre><code class="prism language-http">Host: www.ddsm.com\r\n
User-Agent: Mozilla/5.0 (Windows NT 10.0)\r\n
Accept: text/html\r\n
Content-Type: application/json\r\n
Content-Length: 28\r\n
\r\n
</code></pre>
      <ul>
       <li>
        <strong>
         Host
        </strong>
        ：目标服务器域名（HTTP/1.1必须）。
       </li>
       <li>
        <strong>
         User-Agent
        </strong>
        ：客户端信息（浏览器或工具标识），OS信息（包括版本）。
       </li>
       <li>
        <strong>
         Accept
        </strong>
        ：客户端可接收的响应类型。
       </li>
       <li>
        <strong>
         Content-Type
        </strong>
        ：请求正文的数据类型（如JSON、表单）。
       </li>
       <li>
        <strong>
         Content-Length
        </strong>
        ：请求正文的字节长度（POST/PUT需指定）。
       </li>
      </ul>
     </li>
    </ul>

       </td>
       <td>
        <code>
         Referer: https://www.google.com
        </code>
       </td>
       <td>
        表示当前请求的来源页面URL（用于防盗链或统计）。
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         Cache-Control
        </strong>
       </td>
       <td>
        <code>
         Cache-Control: no-cache
        </code>
       </td>
       <td>
        控制缓存行为（如
        <code>
         max-age=3600
        </code>
        、
        <code>
         no-store
        </code>
        ）。
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         Accept-Language
        </strong>
       </td>
       <td>
        <code>
         Accept-Language: en-US, zh-CN
        </code>
       </td>
       <td>
        声明客户端优先接收的语言类型。
       </td>
      </tr>
     </tbody>
    </table>
    <hr/>
    <h5>
     <a id="iii_Empty_Line_134">

       <li>

        </strong>

        、

        </code>

      </ul>
     </li>
    </ul>

    <h4>
     <a id="HTTP_187">

     HTTP请求报文注意事项：
    </h4>
    <ol>
     <li>
      <strong>
       换行符
      </strong>
      ：HTTP规范要求使用
      <code>

      </code>
      （CRLF）作为换行符。
     </li>
     <li>
      <strong>
       Host头
      </strong>
      ：HTTP/1.1中必须包含，用于指定服务器域名。
     </li>
     <li>
      <strong>
       Content-Length
      </strong>
      ：正文长度必须与实际数据一致，否则可能引发错误。
     </li>
    </ol>

    <h4>
     <a id="233_HTTP_196">
     </a>
     2.3.3 HTTP响应报文结构
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/3341468464524e059e53663f29e6bcb9.png"/>
     HTTP响应报文是服务器对客户端请求的回复，主要包含
     <strong>
      状态行、响应头、空行和响应正文
     </strong>
     。
    </p>
    <hr/>
    <h5>
     <a id="i_Status_Line_199">
     </a>
     i. 状态行（Status Line）
    </h5>
    <ul>
     <li>
      <strong>
       格式
      </strong>
      ：
      <code>
       HTTP版本 空格 状态码 空格 状态码描述 换行符
      </code>
     </li>
     <li>
      <strong>
       作用
      </strong>
      ：告知客户端请求的处理结果（成功、重定向、客户端错误或服务器错误）。
     </li>
     <li>
      <strong>
       示例
      </strong>
      ：
      <pre><code class="prism language-http">HTTP/1.1 200 OK\r\n
</code></pre>
      <ul>
       <li>
        <strong>
         HTTP版本
        </strong>
        ：
        <code>
         HTTP/1.1
        </code>
        （服务器使用的协议版本）。
       </li>
       <li>
        <strong>
         状态码
        </strong>
        ：
        <code>
         200
        </code>
        （表示请求成功）。
       </li>
       <li>
        <strong>
         状态码描述
        </strong>
        ：
        <code>
         OK
        </code>
        （人类可读的简短描述）。
       </li>
      </ul>
     </li>
    </ul>
    <p>
     <strong>
      状态码分类
     </strong>
     ：
    </p>
    <ul>
     <li>
      <code>
       1xx
      </code>
      ：信息性状态码（如
      <code>
       101 Switching Protocols
      </code>
      ）。
     </li>
     <li>
      <code>
       2xx
      </code>
      ：成功状态码（如
      <code>
       200 OK
      </code>
      、
      <code>
       201 Created
      </code>
      ）。
     </li>
     <li>
      <code>
       3xx
      </code>
      ：重定向状态码（如
      <code>
       301 Moved Permanently
      </code>
      、
      <code>
       302 Found
      </code>
      ）。
     </li>
     <li>
      <code>
       4xx
      </code>
      ：客户端错误（如
      <code>
       404 Not Found
      </code>
      、
      <code>
       400 Bad Request
      </code>
      ）。
     </li>
     <li>
      <code>
       5xx
      </code>
      ：服务器错误（如
      <code>
       500 Internal Server Error
      </code>
      、
      <code>
       503 Service Unavailable
      </code>
      ）。
     </li>
    </ul>
    <p>
     但是一般而言，这些状态码并不会被服务器严格遵守。
    </p>
    <hr/>
    <h5>
     <a id="ii_Response_Headers_222">
     </a>
     ii. 响应头（Response Headers）
    </h5>
    <ul>
     <li>
      <strong>
       格式
      </strong>
      ：
      <code>
       Key: Value 换行符
      </code>
     </li>
     <li>
      <strong>
       作用
      </strong>
      ：提供响应的元数据（如内容类型、服务器信息、缓存策略等）。
     </li>
     <li>
      <strong>
       示例
      </strong>
      ：
      <pre><code class="prism language-http">Server: nginx/1.18.0\r\n
Date: Mon, 01 Jan 2024 12:00:00 GMT\r\n
Content-Type: text/html\r\n
Content-Length: 127\r\n
Set-Cookie: session_id=abc123; Path=/\r\n
\r\n
</code></pre>
      <ul>
       <li>
        <strong>
         Server
        </strong>
        ：服务器软件信息（如Nginx、Apache）。
       </li>
       <li>
        <strong>
         Content-Type
        </strong>
        ：响应正文的数据类型（如
        <code>
         text/html
        </code>
        、
        <code>
         application/json
        </code>
        ）。
       </li>
       <li>
        <strong>
         Content-Length
        </strong>
        ：响应正文的字节长度。
       </li>
       <li>
        <strong>
         Set-Cookie
        </strong>
        ：向客户端设置Cookie。
       </li>
      </ul>
     </li>
    </ul>
    <hr/>
    <h5>
     <a id="iii_Empty_Line_241">
     </a>
     iii. 空行（Empty Line）
    </h5>
    <ul>
     <li>
      <strong>
       格式
      </strong>
      ：
      <code>
       换行符
      </code>
     </li>
     <li>
      <strong>
       作用
      </strong>
      ：分隔响应头和响应正文，表示头部结束。
     </li>
     <li>
      <strong>
       示例
      </strong>
      ：
      <pre><code class="prism language-http">\r\n
</code></pre>
     </li>
    </ul>
    <hr/>
    <h5>
     <a id="iv_Response_Body_251">
     </a>
     iv. 响应正文（Response Body）
    </h5>
    <ul>
     <li>
      <strong>
       格式
      </strong>
      ：任意数据（如HTML页面、JSON数据、文件）。
     </li>
     <li>
      <strong>
       作用
      </strong>
      ：包含服务器返回的实际内容。
     </li>
     <li>

       示例
      </strong>

      <pre><code class="prism language-html"><span class="token doctype"><span class="token punctuation">&lt;!</span><span class="token doctype-tag">DOCTYPE</span> <span class="token name">html</span><span class="token punctuation">&gt;</span></span>
<span class="token tag"><span class="token tag"><span class="token punctuation">&lt;</span>html</span><span class="token punctuation">&gt;</span></span>
  <span class="token tag"><span class="token tag"><span class="token punctuation">&lt;</span>body</span><span class="token punctuation">&gt;</span></span>Hello, World!<span class="token tag"><span class="token tag"><span class="token punctuation">&lt;/</span>body</span><span class="token punctuation">&gt;</span></span>
<span class="token tag"><span class="token tag"><span class="token punctuation">&lt;/</span>html</span><span class="token punctuation">&gt;</span></span>
</code></pre>
     </li>
    </ul>

    <h4>
     <a id="_301">
     </a>
     关键注意事项
    </h4>
    <ol>
     <li>
      <strong>
       状态码是固定的
      </strong>
      ：状态码描述（如
      <code>
       OK
      </code>
      ）可能因服务器而异，但状态码（如
      <code>
       200
      </code>
      ）是标准化的。
     </li>
     <li>
      <strong>
       Content-Type必须准确
      </strong>
      ：若响应是JSON但声明为
      <code>
       text/html
      </code>
      ，客户端可能解析失败。
     </li>
     <li>
      <strong>
       Content-Length一致性
      </strong>
      ：需与正文实际字节数一致，否则可能截断数据或读取错误。
     </li>
    </ol>
    <p>
     通过分析响应报文，可以快速定位问题（如
     <code>
      4xx
     </code>
     表示客户端错误，
     <code>
      5xx
     </code>
     表示服务器错误），并优化前后端交互逻辑。
    </p>
