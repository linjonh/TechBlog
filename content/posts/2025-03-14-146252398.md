    <p>
     单元测试、系统测试、集成测试、回归测试的梳理说明
    </p>
    <ol>
     <li>
      单元测试
      <br/>
      步骤：
     </li>
    </ol>
    <ul>
     <li>
      编写测试用例，覆盖代码的各个分支和边界条件。
     </li>
     <li>
      使用测试框架（如JUnit、NUnit）执行测试。
     </li>
     <li>
      检查测试结果，确保代码按预期运行。
     </li>
     <li>
      修复发现的缺陷并重新测试。
      <br/>
      优点：
     </li>
     <li>
      早期发现缺陷，降低修复成本。
     </li>
     <li>
      提高代码质量，确保模块功能正确。
     </li>
     <li>
      便于重构，测试用例可作为回归测试的基础。
      <br/>
      缺点：
     </li>
     <li>
      编写和维护测试用例耗时。
     </li>
     <li>
      无法发现模块间交互问题。
      <br/>
      注意点：
     </li>
     <li>
      测试用例应覆盖所有代码路径。
     </li>
     <li>
      避免测试用例过于复杂。
     </li>
     <li>
      定期维护测试用例，确保其与代码同步。
     </li>
    </ul>
    <ol start="2">
     <li>
      集成测试
      <br/>
      步骤：
     </li>
    </ol>
    <ul>
     <li>
      确定集成顺序和策略（如自顶向下、自底向上）。
     </li>
     <li>
      编写集成测试用例，覆盖模块间接口和交互。
     </li>
     <li>
      执行测试，检查模块间交互是否正常。
     </li>
     <li>
      记录并修复发现的问题，重新测试。
      <br/>
      优点：
     </li>
     <li>
      发现模块间接口和交互问题。
     </li>
     <li>
      确保模块组合后系统功能正常。
      <br/>
      缺点：
     </li>
     <li>
      定位问题较复杂，涉及多个模块。
     </li>
     <li>
      需要更多资源和时间。
      <br/>
      注意点：
     </li>
     <li>
      明确集成顺序和策略。
     </li>
     <li>
      确保模块单元测试通过后再进行集成测试。
     </li>
     <li>
      关注模块间接口和数据传递。
     </li>
    </ul>
    <ol start="3">
     <li>
      系统测试
      <br/>
      步骤：
     </li>
    </ol>
    <ul>
     <li>
      制定系统测试计划，明确测试范围和目标。
     </li>
     <li>
      编写系统测试用例，覆盖功能、性能、安全等方面。
     </li>
     <li>
      执行测试，检查系统是否满足需求。
     </li>
     <li>
      记录并修复问题，重新测试。
      <br/>
      优点：
     </li>
     <li>
      全面验证系统功能和性能。
     </li>
     <li>
      确保系统满足用户需求和预期。
      <br/>
      缺点：
     </li>
     <li>
      需要大量资源和时间。
     </li>
     <li>
      发现问题时修复成本较高。
      <br/>
      注意点：
     </li>
     <li>
      测试用例应覆盖所有需求。
     </li>
     <li>
      模拟真实环境进行测试。
     </li>
     <li>
      关注系统性能和安全性。
     </li>
    </ul>
    <ol start="4">
     <li>
      回归测试
      <br/>
      步骤：
     </li>
    </ol>
    <ul>
     <li>
      确定回归测试范围，选择相关测试用例。
     </li>
     <li>
      执行回归测试，检查修改是否引入新问题。
     </li>
     <li>
      记录并修复问题，重新测试。
      <br/>
      优点：
     </li>
     <li>
      确保代码修改未影响现有功能。
     </li>
     <li>
      提高软件质量，减少回归缺陷。
      <br/>
      缺点：
     </li>
     <li>
      需要维护大量测试用例。
     </li>
     <li>
      执行回归测试耗时。
      <br/>
      注意点：
     </li>
     <li>
      回归测试范围应覆盖可能受影响的区域。
     </li>
     <li>
      自动化回归测试以提高效率。
     </li>
     <li>
      定期更新和维护测试用例。
      <br/>
      总结
     </li>
     <li>
      单元测试
      <br/>
      ：验证单个模块功能，早期发现缺陷。
     </li>
     <li>
      集成测试
      <br/>
      ：验证模块间交互，确保接口正常。
     </li>
     <li>
      系统测试
      <br/>
      ：全面验证系统功能和性能，确保满足需求。
     </li>
     <li>
      回归测试
      <br/>
      ：确保代码修改未影响现有功能，减少回归缺陷。
      <br/>
      每种测试方法各有优缺点，需根据项目需求合理选择和应用。
