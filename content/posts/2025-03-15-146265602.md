     一.锁策略
    </h2>
    <h3>
     <a id="11_6">
     </a>
     1.1各种锁策略介绍
    </h3>
    <p>
     <strong>
      乐观锁&amp;悲观锁
     </strong>
    </p>
    <h4>
     <a id="111_8">
     </a>
     1.1.1乐观锁&amp;悲观锁
    </h4>
    <p>
     乐观锁和悲观锁只是锁的一种
     <strong>
      策略
     </strong>
     ，并不是具体实现
    </p>
    <p>
     <strong>
      乐观锁：
     </strong>
     <br/>
     假设冲突概率低，先操作，更新时检查数据有没有被修改过，比如用版本号机制。更新数据时带上版本号，如果版本号不匹配，说明数据被修改过，这时候需要处理冲突，可以重试或者报错
    </p>
    <p>
     <strong>
      悲观锁:
     </strong>
     <br/>
     总是假设最坏的情况，每次在处理数据的时候都认为别人会修改，所以每次处理数据之前都会上锁，防止干扰
    </p>
    <p>
     <strong>
      synchronized
     </strong>
     初始使用
     <strong>
      乐观锁
     </strong>
     策略，当发现锁竞争比较频繁的时候,，就会自动切换成
     <strong>
      悲观锁
     </strong>
     策略
    </p>
    <h4>
     <a id="112_18">
     </a>
     1.1.2轻量级锁&amp;重量级锁
    </h4>
    <p>
     锁的核心特性
     <strong>
      “原子性”
     </strong>
     ，这样的机制追根溯源是
     <strong>
      CPU
     </strong>
     这样的硬件设备提供的
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a1038c22c9a44557b4c701ebef046079.png#pic_center"/>
    </p>
    <ol>
     <li>
      <pre><code> 重量级锁：严重依赖操作系统内核提供的互斥机制(mutex)
</code></pre>
     </li>
     <li>
      <pre><code> 轻量级锁：尽量不依赖mutex，能在用户态解决就不切换内核态
</code></pre>
     </li>
    </ol>
    <p>
     当线程遇到轻量级锁时，会使用CAS指令快速获取锁，当获取成功后，该线程继续执行，如果获取失败，线程不会直接陷入阻塞，而是进入自旋状态，持续探测锁是否被释放了。但不会无限制的自旋下去，达到一定自旋次数时，停止自旋，进入阻塞状态。所以，轻量级锁和重量级锁是可以相互切换的。
     <br/>
     重量级锁开销大的原因
    </p>
    <ol>
     <li>
      <pre><code> 上下文切换开销：线程被挂起和唤醒时，需要进行上下文切换，这涉及到保存和恢复线程的寄存器状态，程序计数器等，开销较大
</code></pre>
     </li>
     <li>
      <pre><code> 操作系统内核介入：重量级锁的实现依赖于操作系统的内核函数调用，这会增加系统的调用开销。
</code></pre>
     </li>
    </ol>
    <h4>
     <a id="113_27">
     </a>
     1.1.3自旋锁&amp;挂起等待锁
    </h4>
    <p>
     <strong>
      自旋锁是轻量级锁的具体实现，挂起等待锁是重量级锁的具体实现
     </strong>
     <br/>
     <strong>
      自旋锁
     </strong>
     ：自旋锁是一种忙等待的锁，当某线程尝试获取自旋锁时，如果该锁已经被其他线程持有，该线程
     <strong>
      不会陷入阻塞
     </strong>
     ，而是会在一个循环中不断地检查锁是否被释放，这个过程不涉及内核态和用户态的切换，是轻量级锁的实现
    </p>
    <p>
     <strong>
      挂起等待锁
     </strong>
     (也叫阻塞等待锁)：当一个线程尝试获取一个已经被其他线程持有的锁时，该线程会被挂起，这个操作在
     <strong>
      内核态
     </strong>
     进行。这个过程涉及用户态和内核态的切换，线程的阻塞和唤醒，并且要保存该线程的上下文信息，会消耗性能，所以挂起等待锁是重量级锁的实现
    </p>
    <h4>
     <a id="114_32">
     </a>
     1.1.4读写锁
    </h4>
    <p>
     多线程之间，数据的
     <strong>
      读取方
     </strong>
     之间不会产生线程安全问题，但数据的写入方互相之间以及和读者之间都需要进行互斥。如果两种场景下都用同一个锁，就会产生极大的性能损耗，所以读写锁因此而产生
     <br/>
     读写锁就是把读操作和写操作区分对待. Java 标准库提供了ReentrantReadWriteLock 类, 实现了读写锁.
    </p>
    <ol>
     <li>
      <pre><code> ReentrantReadWriteLock.ReadLock类表示一个读锁，这个对象提供了 lock / unlock 方法进行加锁解锁
</code></pre>
     </li>
     <li>
      <pre><code> ReentrantReadWriteLock.WriteLock 类表示一个写锁. 这个对象也提供了 lock /unlock 方法进行加锁解锁
</code></pre>
     </li>
    </ol>
    <p>
     其中：
     <br/>
     读加锁和读加锁之间, 不互斥
     <br/>
     写加锁和写加锁之间, 互斥
     <br/>
     读加锁和写加锁之间, 互斥
    </p>
    <h4>
     <a id="115_41">
     </a>
     1.1.5公平锁/非公平锁
    </h4>
    <p>
     公平锁: 线程陷入阻塞后进入阻塞队列(这需要额外的数据结构来实现，比如记录线程阻塞的时间，放进优先级队列中)，在锁释放后按照先来后到的顺序获取锁
     <br/>
     非公平锁：锁释放后操作系统会唤醒一个或者多个线程，这些线程同时竞争锁，不关心是哪个线程能获取锁
    </p>
    <h4>
     <a id="116_44">
     </a>
     1.1.6可重入锁/不可重入锁
    </h4>
    <p>
     可重入锁：线程已经持有某个对象的锁，那么它可以再次获取该对象的锁，不会被阻塞。可重入锁通常会维护一个计数器，记录当前线程获取锁的次数。每次获取锁时，计数器加一；释放锁时，计数器减一。当计数器为零时，锁才真正被释放
    </p>
    <p>
     不可重入锁：线程已经持有某个对象的锁，那么它可以再次获取该对象的锁，会被阻塞
    </p>

    </h3>
    <p>
     CAS是Compare And Swap的缩写，意思是比较并交换。
     <br/>

    <h3>
     <a id="23ABA_140">
     </a>
     2.3ABA问题
    </h3>
    <p>
     它指的是在多线程环境下，一个变量在某个时间点被修改为另一个值，然后
     <strong>
      又修改回原来的值
     </strong>
     。尽管变量的值最终没有变化，但这个过程中变量的状态可能已经发生了实质性的改变，而CAS操作无法检测到这种变化。
     <br/>
     例如：
     <br/>
     假设滑稽老哥有100存款，滑稽想从ATM取50 块钱，取款机创建了两个线程，并发的来执行-50
     <br/>
     操作
     <br/>
     我们期望一个线程执行-50成功，另一个线程-50失败
     <br/>
     如果使用CAS的方式来完成这个扣款过程就可能出现问题
     <br/>
     取款过程如下：
    </p>
    <ol>
     <li>
      <pre><code> 存款100：线程1获取到当前存款值为100， 期望更新为50；线程2获取到当前存款值为100，期望更新为 50
</code></pre>
     </li>
     <li>
      <pre><code> 线程1执行扣款成功，存款被改成50；线程2阻塞等待中
</code></pre>
     </li>
     <li>
      <pre><code> 在线程2执行之前，滑稽的朋友给他转账50，余额又变回100
</code></pre>
     </li>
     <li>
      <pre><code> 轮到线程2执行了，发现当前存款为100，和之前读到的100相同，再次执行扣款操作
</code></pre>
     </li>
    </ol>
    <p>
     解决方案：引入版本号
     <br/>
     版本1：
     <br/>
     存款100：线程1获取到当前存款值为100， 期望更新为50；线程2获取到当前存款值为100，期望更新为 50
     <br/>
     版本2：
     <br/>
     线程1执行扣款成功，存款被改成50；线程2阻塞等待中
     <br/>
     版本3：
     <br/>
     在线程2执行之前，滑稽的朋友给他转账50，余额又变回100
    </p>
    <p>
     轮到线程2 执行了, 发现当前存款为100，和之前读到的 100 相同。但是当前版本号为3, 之前读
     <br/>
     到的版本号为1，版本小于当前版本，认为操作失败
    </p>
