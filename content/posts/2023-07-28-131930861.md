    <h2 id="%E4%B8%80.%E6%96%87%E4%BB%B6%E7%BB%93%E6%9E%84%E5%92%8C%E5%B7%A5%E5%85%B7%E5%8A%9F%E8%83%BD">
     一.文件结构和工具功能
    </h2>
    <h4 id="1.%E5%B0%8F%E7%A8%8B%E5%BA%8F%E7%BC%96%E8%AF%91%E7%9A%84%E6%96%87%E4%BB%B6%E7%BB%93%E6%9E%84">
     1.小程序编译的文件结构
    </h4>
    <p>
     非常必要推荐了解小程序文件结构，对于稍大的项目，对于包的精简会起到柳暗花明又一村的效果 。众所周知，微信小程序分为“
     <strong>
      主包
     </strong>
     ”和“
     <strong>
      分包
     </strong>
     ”模块，
     <strong>
      每个包不得超过2M最多分10个包共20M
     </strong>
     。而微信小程序的文件结构，只有主包和分包：
    </p>
    <p class="img-center">
     <img alt="" height="254" src="https://i-blog.csdnimg.cn/blog_migrate/f04be009877df27833aa1e3f5aae3359.png" width="246"/>
    </p>
    <p>
     因此，我们的代码中，除了定义的分包外，
     <strong>
      所有的其它内容都在主包中
     </strong>
     ：
    </p>
    <p class="img-center">
     <img alt="" height="271" src="https://i-blog.csdnimg.cn/blog_migrate/66926877c1a2377738b2cf880db55d7e.png" width="151"/>
    </p>
    <p>
     这给我们了启示，在包容量过大情况下，尽量将组件分在各分包内，同时检查除分包外的全局配置。
    </p>
    <h4 id="2.%E5%B7%A5%E5%85%B7%E5%8A%9F%E8%83%BD%E4%BD%BF%E7%94%A8">
     2.工具功能使用
    </h4>
    <h5 id="A.%E5%8F%91%E8%A1%8C">
     A.发行
    </h5>
    <p>
     使用HbuilderX运转到微信开发者工具编译成其语言，可经过“
     <strong>
      运行
     </strong>
     ”和“
     <strong>
      发行
     </strong>
     ”两种。但“发行”对编译包更加精简，
     <strong>
      若要“上传”小程序，一定要点“发行”功能。
     </strong>
    </p>
    <p class="img-center">
     <img alt="" height="312" src="https://i-blog.csdnimg.cn/blog_migrate/add89b244bb03aa6607256902a74cdb8.png" width="334"/>
    </p>
    <h5 id="B.%E4%BB%A3%E7%A0%81%E4%BE%9D%E8%B5%96%E5%88%86%E6%9E%90">
     B.代码依赖分析
    </h5>
    <p>
     代码依赖分析中，
     <strong>
      详细展示了每个包及文件的大小
     </strong>
     ，可以清晰找出比较大的文件夹和文件，进而优化。
    </p>
    <p>
     <img alt="" height="335" src="https://i-blog.csdnimg.cn/blog_migrate/3ba39ca0fe383bdc72de233cd8c30683.png" width="240">
      <img alt="" height="331" src="https://i-blog.csdnimg.cn/blog_migrate/91f1c06d54ac80f001fbd3d275310de4.png" width="327"/>
     </img>
    </p>
    <p>
     其中还有一个“无依赖代码文件数”功能，在“全部”下面，这个功能比较鸡肋，不能够完全准确地分析出依赖与否，罗列的文件是有可能存在依赖的，只是没有检测出来，不过还是能在一定程度上帮助我们分析出可能无用的文件，自己进行尝试筛选即可（我也要吐槽下，如果这个功能按照文件夹分类不更好吗？）
    </p>
    <p>
     <img alt="" height="144" src="https://i-blog.csdnimg.cn/blog_migrate/f74be8a38d07fb62c62bad05dba207a8.png" width="1174"/>
    </p>
    <h2 id="%E4%BA%8C.%E6%8E%A7%E5%88%B6%E6%96%B9%E6%B3%95">
     二.控制方法
    </h2>
    <h4 id="1.%E8%87%AA%E7%94%A8%E7%BB%84%E4%BB%B6%2F%E5%B7%A5%E5%85%B7">
     1.自用组件/工具
    </h4>
    <p>
     我们会在编写中制作
     <strong>
      全局自用组件
     </strong>
     和
     <strong>
      分包自用组件
     </strong>
     。
    </p>
    <p>
     对于一般的项目来说，比较通用的自用组件，放在
     <strong>
      全局components下
     </strong>
     即可，不会给主包占用太多的空间。
    </p>
    <p>
     对于迫切精简的项目，则考虑把通用组件分别
     <strong>
      分配在每个调用的分包中
     </strong>
     以减轻主包压力。基于这种思想，我们可以预估项目的整体大小、考虑组件的全局通用频率进行折衷考虑。
    </p>
    <p>
     除了组件外，不管哪里的图片，都要放在
     <strong>
      图片服务器
     </strong>
     上，仅保留常用的，譬如tabbar图标等几kb的图片。
    </p>
    <h4 id="2.%E5%AE%98%E6%96%B9%E7%BB%84%E4%BB%B6">
     2.官方组件
    </h4>
    <p>
     我说的官方组件，特指
     <strong>
      uni_modules
     </strong>
     和
     <strong>
      vant
     </strong>
     组件。这两款都是非常优秀和完整的组件包，里面包含各种组件（包含一些根本不需要的组件）。而它们的内部调用和引入方式却不太相同，因此也造就了不同的分包和精简方法。
    </p>
    <h5 id="A.uni_modules">
     A.uni_modules
    </h5>
    <p>
     对于uni_modules组件，
     <strong>
      内部依赖性很低
     </strong>
     ，样式尚可，正如官网所说，其独立性强，无需配置就可使用。
    </p>
    <p>
     对于一般的项目，会把整个
     <strong>
      uni_modules放在外面
     </strong>
     ，在页面中直接使用即可。
    </p>
    <p>
     对于迫切精简的项目，可以考虑，将
     <strong>
      外部的uni_modules
     </strong>
     除
     <strong>
      “uni-icons”，“uni-scss”
     </strong>
     保留外，全部删除，然后在各分包内直接引入所需组件，在页面使用时需要
     <strong>
      单独import引入
     </strong>
     注册为component。“uni-icons”和"uni-scss"保证了各组件的图标和样式不出问题，已是最大程度地保留了uni_modules组件功能性，当然譬如"uni-transition"的缺失，会造成某些组件的“来去”不再平滑。
    </p>
    <p class="img-center">
     <img alt="" height="66" src="https://i-blog.csdnimg.cn/blog_migrate/8b47c28565b2adf25affe1a0fd748691.png" width="367"/>
    </p>
    <p class="img-center">
     <img alt="" height="169" src="https://i-blog.csdnimg.cn/blog_migrate/177838b226b951f70da5e9fe1a7041b2.png" width="276"/>
    </p>
    <pre><code class="language-javascript">// uni_modules组件可像普通自定义组件般，放到分包内，vue文件中注册组件使用。
import uniDataPicker from '@/pagesE/components/uni-data-picker/components/uni-data-picker/uni-data-picker.vue'
 components: { uniDataPicker }</code></pre>
    <h5 id="B.vant">
     B.vant
    </h5>
    <p>
     对于vant组件，在小程序中叫vant-weapp，
     <strong>
      内部依赖性高
     </strong>
     ，除此之外，还
     <strong>
      不太兼容、偶出问题
     </strong>
     （诸如双向绑定不会自动绑定问题最多）。
    </p>
    <p>
     对于一般的项目来说，将整个vant组件包放在外面，引入方式建议为按需引入，在
     <strong>
      pages.json
     </strong>
     中，在使用组件的
     <strong>
      style-usingComponents
     </strong>
     中引入，当然如果自己的小程序容量较小并且性能较好，可以放在
     <strong>
      globalStyle-usingComponents
     </strong>
     。
    </p>
    <p>
     值得注意的是，vant组件内部依赖性高，
     <strong>
      不能像uni_modules般分在各个包中单独引用
     </strong>
     ，只能放在工程外部，才能被import找到，即放在分包中，无论以何种方式引入，是不会被找到的。
    </p>
    <pre><code class="language-javascript">  // 分包加载vant组件
   {
		"path": "pages/index",
		"style": {
		    "navigationBarTitleText": "首页",
			"usingComponents": {
				"van-field": "/wxcomponents/vant/dist/field/index",
				"van-popup": "/wxcomponents/vant/dist/popup/index",
				"van-button": "/wxcomponents/vant/dist/button/index"
			}
		}
	}             
    // 全局加载vant组件(不推荐)
    "globalStyle": {
		 "navigationBarTextStyle": "black",		
		 "navigationBarBackgroundColor": "#FFFFFF"
		 "usingComponents": {
		     "van-button": "/wxcomponents/vant/dist/button/index",
		 	 "van-cell": "/wxcomponents/vant/dist/cell/index",
		 	 "van-cell-group": "/wxcomponents/vant/dist/cell-group/index",
		  }
	 }</code></pre>
    <p>
     对于迫切精简的项目，目前博主在迁移vant组件上失败，即上面所说的分包引入的方法，因此只能采用删去不需要的组件来精简项目，对于常用的组件进行保留，可以参考“无依赖代码文件”进行删除，但根据经验，“good-action”系列,“minxins”,"overlay"会有某些常用组件内部调用的，不可以删除。以下是我精简后的vant组件，凡事重在尝试，当然自己根据所需精简即可。
    </p>
    <p class="img-center">
     <img alt="" height="421" src="https://i-blog.csdnimg.cn/blog_migrate/faa7497b70d333895d32ca4700075d8f.png" width="188"/>
    </p>
    <p>
     官方组件的完整包的大小约在300kb左右，对本就不太富裕空间其实就是一种累赘了。uni_modules和vant的功能范围都大差不差，
     <strong>
      uni_modules更符合微信官方
     </strong>
     的风格，大部分情况下，都可以在其中找到我们所需要的，找不到的可以去
     <strong>
      插件市场
     </strong>
     ，我在
     <a href="https://blog.csdn.net/qq_43638732/article/details/131517947" title="微信小程序运行、编译、发布等开发概括_韵泽月光的博客-CSDN博客">
      微信小程序运行、编译、发布等开发概括_韵泽月光的博客-CSDN博客
     </a>
     中也泛讲了相关开发准备。vant组件在小程序上可就不如前者和PC端了（小程序vant是用微信语言写的），会和PC端的使用有些不一样、不生效的，但是其中的某些组件确实可以弥补uni_modules中的不符合html和js的交互功能的不足。所以其中取舍，我也差不多和大家分享了经验了。
    </p>
    <h4 id="3.%E5%85%A8%E5%B1%80%E9%85%8D%E7%BD%AE">
     3.全局配置
    </h4>
    <p>
     全局配置是很容易忽略的，为追求方便，项目开发者会直接在
     <strong>
      main.js
     </strong>
     或者
     <strong>
      App.vue
     </strong>
     中引入全局js文件或者css文件。
    </p>
    <p>
     <strong>
      @font-face
     </strong>
    </p>
    <p>
     有些项目在初次建立时，就引入的全局的css文件，比如
     <strong>
      uni.scss
     </strong>
     ，或者自定义的
     <strong>
      字体css文件
     </strong>
     。而字体引入都是通过base64编码引入，编码本身就已经很大了，或已经超过正常需求的代码大小了，而且在每个编译后的wxss中都存在，代码大小直接翻倍是没有问题了，我已经吃过两次亏了。
    </p>
    <p>
     这可能需要自己主动去微信开发者工具观察每个文件的大小了，若出现了
     <strong>
      批量的异常大小的相同后缀文件
     </strong>
     ，那大概就是被全局配置了，进入文件内容可自行查看和解决。
    </p>
    <h2 id="%E4%B8%89.%E6%80%BB%E7%BB%93">
     三.总结
    </h2>
    <p>
     博文主要讲解了对于实际大包且可能出现超包的项目的精简项目方法，提供了分包、精简项目的出发角度和考虑方向，即根据微信开发者工具中提示，从
     <strong>
      自用组件
     </strong>
     、
     <strong>
      官方组件
     </strong>
     、
     <strong>
      项目配置
     </strong>
     等方面提出了见解。我在所参与的两个项目中，都出现了爆炸容量！！！，最后分去主包0.02Mb的内容都花费不小时间，由于分包经验少，所以可能会出现绝对的错误和相对的经验错误，请大家在评论区指正。
    </p>
