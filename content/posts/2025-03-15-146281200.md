    <h2>
     <a id="RESTful_1">
     </a>
     什么是RESTful
    </h2>
    <blockquote>
     <ul>
      <li>
       RESTful（Representational State Transfer）是一种基于HTTP协议设计网络应用程序接口（API）的架构风格，由Roy Fielding在2000年的博士论文中提出。其核心思想是以资源为中心，通过统一的接口对资源进行操作，实现客户端与服务器的解耦
      </li>
     </ul>
     <blockquote>
      <ul>
       <li>
        也就是RESTful是基于http定义了一组接口格式规范，用来规范所有http请求
       </li>
      </ul>
     </blockquote>
    </blockquote>
    <h2>
     <a id="RESTful__4">
     </a>
     RESTful 设计原则
    </h2>
    <p>
     <strong>
      核心原则
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       <strong>
        资源导向（Resource-Based）
       </strong>
      </li>
     </ul>
     <blockquote>
      <ul>
       <li>
        所有事物抽象为资源（如用户、订单、文章），每个资源有唯一的标识符（URI，如 /users/123）。
       </li>
       <li>
        资源通过JSON与客户端交互。
       </li>
      </ul>
     </blockquote>
     <ul>
      <li>
       <strong>
        统一接口（Uniform Interface）
       </strong>
      </li>
     </ul>
     <blockquote>
      <ul>
       <li>
        定义了HTTP方法明确操作类型：
        <br/>
        GET：获取资源
        <br/>
        POST：创建资源
        <br/>
        PUT：更新资源（全量替换）
        <br/>
        PATCH：更新资源（部分修改）
        <br/>
        DELETE：删除资源
       </li>
       <li>
        HTTP状态码标识结果
        <br/>
        如200 OK、404 Not Found等。
       </li>
      </ul>
     </blockquote>
     <ul>
      <li>
       <strong>
        无状态（Stateless）
       </strong>
      </li>
     </ul>
     <blockquote>
      <ul>
       <li>
        每个请求必须包含所有必要信息，服务器不保存客户端状态（如Session），提升扩展性和可靠性。
       </li>
      </ul>
     </blockquote>
     <ul>
      <li>
       <strong>
        可缓存（Cacheable）
       </strong>
      </li>
     </ul>
     <blockquote>
      <ul>
       <li>
        响应需明确是否可缓存（如通过Cache-Control头），减少重复请求
       </li>
       <li>
        客户端无需关注服务器架构细节（如负载均衡、代理层）
       </li>
      </ul>
     </blockquote>
     <ul>
      <li>
       <strong>
        URI命名
       </strong>
      </li>
     </ul>
     <blockquote>
      <ul>
       <li>
        使用名词复数表示资源（如/users而非/getUser）。
       </li>
       <li>
        避免动词，用HTTP方法表达操作（如DELETE /users/123）。
       </li>
       <li>
        层级关系用/分隔（如/users/123/orders）。
       </li>
       <li>
        URI中体现版本：/api/v1/users
       </li>
      </ul>
     </blockquote>
    </blockquote>
    <h3>
     <a id="RESTful__29">
     </a>
     RESTful 接口类型
    </h3>
    <p>
     <strong>
      GET
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       用途：获取资源（查询数据）。
      </li>
      <li>
       特性：
       <br/>
       安全：不会修改服务器资源。
       <br/>
       幂等：多次请求结果相同。
       <br/>
       可缓存：响应可被浏览器或代理缓存。
      </li>
      <li>
       响应状态码：
       <br/>
       200 OK（成功）
       <br/>
       404 Not Found（资源不存在）
       <br/>
       304 Not Modified（缓存未更新）
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-c">GET <span class="token operator">/</span>users<span class="token operator">/</span><span class="token number">123</span>         # 获取ID为<span class="token number">123</span>的用户信息
GET <span class="token operator">/</span>users<span class="token operator">?</span>role<span class="token operator">=</span>admin  # 过滤角色为admin的用户
</code></pre>
    <p>
     <strong>
      POST
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       用途：创建新资源（或提交非幂等操作）。
      </li>
      <li>
       特性：
       <br/>
       非安全：会修改服务器资源。
       <br/>
       非幂等：多次调用可能产生不同结果（如重复提交订单）。
      </li>
      <li>
       响应状态码：
       <br/>
       201 Created（成功创建，响应头包含 Location: /users/456）
       <br/>
       400 Bad Request（请求数据不合法）
       <br/>
       409 Conflict（资源冲突，如重复创建）
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-bash">POST /users  
Body: <span class="token punctuation">{<!-- --></span> <span class="token string">"name"</span><span class="token builtin class-name">:</span> <span class="token string">"Alice"</span>, <span class="token string">"age"</span><span class="token builtin class-name">:</span> <span class="token number">30</span> <span class="token punctuation">}</span>  <span class="token comment"># 创建新用户</span>
</code></pre>
    <p>
     <strong>
      PUT
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       用途：全量更新资源（替换整个资源）。
      </li>
      <li>
       特性：
       <br/>
       幂等：多次调用结果一致（如重复更新同一数据，最终状态相同）。
       <br/>
       若资源不存在，可创建（需由服务端决定是否支持）。
      </li>
      <li>
       响应状态码：
       <br/>
       200 OK 或 204 No Content（更新成功）
       <br/>
       201 Created（资源被创建）
       <br/>
       404 Not Found（资源不存在且服务端不支持创建）
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-c">PUT <span class="token operator">/</span>users<span class="token operator">/</span><span class="token number">123</span>  
Body<span class="token operator">:</span> <span class="token punctuation">{<!-- --></span> <span class="token string">"name"</span><span class="token operator">:</span> <span class="token string">"Bob"</span><span class="token punctuation">,</span> <span class="token string">"age"</span><span class="token operator">:</span> <span class="token number">25</span> <span class="token punctuation">}</span>  # 替换用户<span class="token number">123</span>的所有字段
</code></pre>
    <p>
     <strong>
      DELETE
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       用途：删除资源。
      </li>
      <li>
       特性：
       <br/>
       幂等：删除后再次调用无效果。
      </li>
      <li>
       响应状态码：
       <br/>
       204 No Content（删除成功，无返回内容）
       <br/>
       404 Not Found（资源不存在）
       <br/>
       403 Forbidden（无权删除）
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-c">DELETE <span class="token operator">/</span>users<span class="token operator">/</span><span class="token number">123</span>  # 删除ID为<span class="token number">123</span>的用户
</code></pre>
    <p>
     <strong>
      PATCH
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       用途：部分更新资源（仅修改指定字段）。
      </li>
      <li>
       特性：
       <br/>
       非幂等：连续相同请求可能导致不同结果（如递增计数器）。
       <br/>
       需明确传递修改的字段。
      </li>
      <li>
       响应状态码：
       <br/>
       200 OK 或 204 No Content
       <br/>
       400 Bad Request（修改格式错误）
       <br/>
       404 Not Found
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-c">PATCH <span class="token operator">/</span>users<span class="token operator">/</span><span class="token number">123</span>  
Body<span class="token operator">:</span> <span class="token punctuation">{<!-- --></span> <span class="token string">"age"</span><span class="token operator">:</span> <span class="token number">26</span> <span class="token punctuation">}</span>  # 仅更新用户<span class="token number">123</span>的年龄
</code></pre>
    <h3>
     <a id="RESTful__103">
     </a>
     RESTful 状态码
    </h3>
    <p>
     <strong>
      1xx（信息响应）
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       <code>
        100
       </code>
       Continue
       <br/>
       客户端应继续发送请求的剩余部分（用于大文件上传前的确认）。
      </li>
     </ul>
    </blockquote>
    <p>
     <strong>
      2xx（成功）
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       <code>
        200 OK
       </code>
       : 请求成功，
       <br/>
       返回响应数据（如 GET 或 PUT 请求成功）。
      </li>
      <li>
       <code>
        201 Created
       </code>
       <br/>
       资源创建成功（如 POST 请求后返回新资源的URI，响应头包含 Location: /users/123）。
      </li>
      <li>
       <code>
        202 Accepted
       </code>
       <br/>
       请求已接受但未处理完成（常用于异步任务）。
      </li>
      <li>
       <code>
        204 No Content
       </code>
       <br/>
       请求成功，但无返回内容（如 DELETE 成功或 PUT 更新后无需返回数据）。
      </li>
     </ul>
    </blockquote>
    <p>
     <strong>
      3xx（重定向）
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       <code>
        301 Moved Permanently
       </code>
       <br/>
       资源永久迁移到新URI（需更新书签）。
      </li>
      <li>
       <code>
        302 Found
       </code>
       <br/>
       资源临时重定向到新URI（浏览器默认行为是后续请求仍用原URI）。
      </li>
      <li>
       <code>
        304 Not Modified
       </code>
       <br/>
       资源未修改，客户端可使用缓存（配合 If-Modified-Since 头使用）。
      </li>
     </ul>
    </blockquote>
    <p>
     <strong>
      4xx（客户端错误）
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       <code>
        400 Bad Request
       </code>
       <br/>
       请求格式错误（如参数缺失或JSON语法错误）。
      </li>
      <li>
       <code>
        401 Unauthorized
       </code>
       <br/>
       未认证（需提供有效身份凭证，如JWT失效）。
      </li>
      <li>
       <code>
        403 Forbidden
       </code>
       <br/>
       已认证但无权限访问资源（如普通用户访问管理员接口）。
      </li>
      <li>
       <code>
        404 Not Found
       </code>
       <br/>
       资源不存在（URI无效或资源已删除）。
      </li>
      <li>
       <code>
        405 Method Not Allowed
       </code>
       <br/>
       HTTP方法不支持（如对只允许 GET 的URI发送 POST）。
      </li>
      <li>
       <code>
        409 Conflict
       </code>
       <br/>
       资源状态冲突（如重复创建同名用户）。
      </li>
      <li>
       <code>
        429 Too Many Requests
       </code>
       <br/>
       请求频率超出限制（常见于API限流）。
      </li>
     </ul>
    </blockquote>
    <p>
     <strong>
      5xx（服务器错误）
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       <code>
        500 Internal Server Error
       </code>
       <br/>
       服务器内部错误（通用兜底错误，需排查日志）。
      </li>
      <li>
       <code>
        502 Bad Gateway
       </code>
       <br/>
       网关或代理服务器收到无效响应（如后端服务崩溃）。
      </li>
      <li>
       <code>
        503 Service Unavailable
       </code>
       <br/>
       服务暂时不可用（如维护或过载，可配合 Retry-After 头提示重试时间）。
      </li>
      <li>
       <code>
        504 Gateway Timeout
       </code>
       <br/>
       网关或代理服务器超时（后端服务未及时响应）
      </li>
     </ul>
    </blockquote>
    <h3>
     <a id="RESTful_Uri_151">
     </a>
     RESTful Uri设计原则
    </h3>
    <p>
     <strong>
      对于Api的uri命名要遵守以下设计
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       使用名词复数表示资源（如/users而非/getUser）。
      </li>
      <li>
       避免动词，用HTTP方法表达操作
      </li>
     </ul>
     <blockquote>
      <ul>
       <li>
        如DELETE /users/123，而不是DELETE /delete-users/123）。
       </li>
      </ul>
     </blockquote>
     <ul>
      <li>
       层级关系用/分隔
      </li>
     </ul>
     <blockquote>
      <ul>
       <li>
        如/users/123/orders, 而不是 /users/orders/123
       </li>
      </ul>
     </blockquote>
     <ul>
      <li>
       URI中体现版本：/api/v1/users
      </li>
      <li>
       推荐使用小驼峰，比如/pet/petId
      </li>
      <li>
       uri应该大小写敏感，并且推荐使用全小写
      </li>
     </ul>
    </blockquote>
    <blockquote>
     <ul>
      <li>
       以上只是建议，具体根据实际情况而定
      </li>
     </ul>
    </blockquote>

    <h3>
     <a id="ApiQueryString__UriPath_168">
     </a>
     Api传参：QueryString 和 UriPath
    </h3>
    <p>
     <strong>
      使用 URI Path（路径参数）的场景
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       URI Path 用于标识 资源的主键 或 资源间的层级关系，
       <code>
        通常对应服务端的 核心数据模型。
       </code>
      </li>
     </ul>
    </blockquote>
    <p>
     适用场景：
    </p>
    <blockquote>
     <ul>
      <li>
       唯一资源标识
       <br/>
       通过路径参数直接定位唯一资源（如用户ID、订单ID）。
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-c">GET <span class="token operator">/</span>users<span class="token operator">/</span><span class="token punctuation">{<!-- --></span>id<span class="token punctuation">}</span>          # 获取指定ID的用户
DELETE <span class="token operator">/</span>orders<span class="token operator">/</span><span class="token punctuation">{<!-- --></span>orderId<span class="token punctuation">}</span> # 删除指定ID的订单
</code></pre>
    <blockquote>
     <ul>
      <li>
       资源层级关系
       <br/>
       表示资源之间的从属关系（如用户的所有订单）。
       <br/>
       示例：
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-c">GET <span class="token operator">/</span>users<span class="token operator">/</span><span class="token punctuation">{<!-- --></span>userId<span class="token punctuation">}</span><span class="token operator">/</span>orders  # 获取用户的订单列表
</code></pre>
    <p>
     特点：
    </p>
    <blockquote>
     <ul>
      <li>
       必要性：路径参数通常是 必填 的（缺少会导致404）。
      </li>
      <li>
       语义明确：路径体现资源的唯一性和结构（如 /users/123 明确指向用户123）。
      </li>
      <li>
       缓存友好：路径参数不同的 URL 会被视为不同资源，适合缓存（如 CDN 缓存）
      </li>
     </ul>
    </blockquote>
    <p>
     <strong>
      使用 Query String（查询参数）的场景
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       Query String 用于 过滤、排序、分页 或 附加操作，
       <code>
        通常是 可选 的非核心参数。
       </code>
      </li>
     </ul>
    </blockquote>
    <p>
     适用场景：
    </p>
    <blockquote>
     <ul>
      <li>
       过滤数据
       <br/>
       筛选符合特定条件的资源列表。
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-c">示例：
GET <span class="token operator">/</span>users<span class="token operator">?</span>role<span class="token operator">=</span>admin<span class="token operator">&amp;</span>status<span class="token operator">=</span>active  # 获取角色为admin且状态为活跃的用户
</code></pre>
    <blockquote>
     <ul>
      <li>
       分页与排序
       <br/>
       控制返回数据的范围和顺序。
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-c">示例：
GET <span class="token operator">/</span>articles<span class="token operator">?</span>page<span class="token operator">=</span><span class="token number">2</span><span class="token operator">&amp;</span>limit<span class="token operator">=</span><span class="token number">20</span><span class="token operator">&amp;</span>sort<span class="token operator">=</span><span class="token operator">-</span>created_at  # 获取第二页文章，按创建时间倒序
</code></pre>
    <blockquote>
     <ul>
      <li>
       附加操作
       <br/>
       触发特定功能（如统计、导出格式）。
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-cpp">示例：
GET <span class="token operator">/</span>reports<span class="token operator">/</span>sales<span class="token operator">?</span>format<span class="token operator">=</span>csv     # 导出CSV格式的销售报表
GET <span class="token operator">/</span>products<span class="token operator">?</span>fields<span class="token operator">=</span>name<span class="token punctuation">,</span>price  # 仅返回名称和价格字段（字段选择）
</code></pre>
    <blockquote>
     <ul>
      <li>
       非资源标识参数
       <br/>
       不影响资源定位，仅修饰响应结果。
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-c">示例：
GET <span class="token operator">/</span>search<span class="token operator">?</span>q<span class="token operator">=</span>restful<span class="token operator">&amp;</span>lang<span class="token operator">=</span>en  # 搜索关键词<span class="token string">"restful"</span>，语言为英文
</code></pre>
    <p>
     特点：
    </p>
    <blockquote>
     <ul>
      <li>
       可选性：
       <strong>
        <code>
         缺少查询参数时，通常返回默认结果
        </code>
       </strong>
       （如不分页）。
      </li>
      <li>
       灵活性：参数组合自由，适合动态需求（如多种过滤条件）。
      </li>
      <li>
       幂等性：同一 Query String 的 GET 请求是幂等的（结果可缓存）
      </li>
     </ul>
    </blockquote>
    <h2>
     <a id="RESTfulHTTP_236">
     </a>
     RESTful和HTTP的区别
    </h2>
    <blockquote>
     <ul>
      <li>
       RESTful并不等于HTTP，它只是基于HTTP的一些特性设定了一组设计规范
      </li>
      <li>
       类似的还有GraphQL，也是基于HTTP的特性设定了一组规范
      </li>
     </ul>
    </blockquote>
    <h2>
     <a id="_239">
     </a>
     注意事项
    </h2>
    <blockquote>
     <ul>
      <li>
       <strong>
        以上只是标准的设计规范，并没有技术上的限制
       </strong>
       ，开发者完全可以POST当PUT用，PUT当POST用，甚至POST当GET用等等，只是这样不规范，会导致程序可维护性降低
      </li>
      <li>
       现实开发中需要根据实际情况调整，最好和以前的开发风格保持一致
      </li>
      <li>
       大部分Api的设计都没有严格的遵守以上规则，比如Api的uri里没有版本信息，返回的状态码用的不准确等，使用动词等等
      </li>
     </ul>
    </blockquote>
