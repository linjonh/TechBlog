    <h2>
     epoll的两种工作模式
    </h2>
    <h4>
     <span style="color:#ff9900">
      水平触发
     </span>
     （Level Triggered，LT）
    </h4>
    <h5>
     原理：
    </h5>
    <h5>
     水平触发是
     <code>
      epoll
     </code>
     的默认工作模式。在这种模式下，只要文件描述符对应的 I/O 条件满足（例如，有数据可读或者可写），
     <code>
      epoll_wait
     </code>
     就会一直返回该文件描述符，直到该条件不再满足。也就是说，只要缓冲区中有数据，就会一直触发事件。
    </h5>
    <h4>
     <span style="color:#ff9900">
      边缘触发
     </span>
     （Edge Triggered，ET）
    </h4>
    <h5>
     原理：
    </h5>
    <p>
     <strong>
      边缘触发是一种更为高效的工作模式。在这种模式下，只有当文件描述符的 I/O 条件发生变化（例如，从无数据变为有数据，或者从不可写变为可写）时，
      <code>
       epoll_wait
      </code>
      才会返回该文件描述符。也就是说，只有在数据状态发生变化的 “边缘” 时刻才会触发事件。
     </strong>
    </p>

    <h4>
     其他占比相关信息
    </h4>
    <ul>
     <li>
      <strong>
       %CPU
      </strong>
      ：表示进程占用 CPU 的百分比。该数值反映了进程在 CPU 资源分配中所占的份额，百分比越高，说明该进程对 CPU 的使用越频繁。例如，如果一个进程的 % CPU 值为 50%，意味着在统计的时间间隔内，该进程占用了 CPU 一半的运算资源。
     </li>
     <li>
      <strong>
       us（用户空间 CPU 使用率）
      </strong>
      ：显示了用户进程占用 CPU 的时间百分比。通常是应用程序、命令等在用户态下运行所占用的 CPU 时间比例。
     </li>
     <li>
      <strong>
       sy（系统空间 CPU 使用率）
      </strong>
      ：代表系统内核进程占用 CPU 的时间百分比。包括内核态下的系统服务、中断处理等操作所占用的 CPU 时间。
     </li>
     <li>
      <strong>
       ni（低优先级用户进程 CPU 使用率）
      </strong>
      ：表示以较低优先级运行的用户进程占用 CPU 的时间百分比。这些进程的优先级相对较低，在系统资源紧张时，可能会被优先抢占 CPU 资源。
     </li>
     <li>
      <strong>
       id（空闲 CPU 时间百分比）
      </strong>
      ：显示 CPU 处于空闲状态的时间百分比。该值越高，说明系统的 CPU 资源越充裕，反之则表示 CPU 负载较高。
     </li>
     <li>
      <strong>
       wa（等待 I/O 的 CPU 时间百分比）
      </strong>
      ：表示 CPU 等待 I/O 操作完成所花费的时间百分比。如果这个值较高，通常意味着系统存在 I/O 瓶颈，例如硬盘读写速度慢、网络延迟高等问题，导致 CPU 需要等待 I/O 操作完成才能继续执行其他任务。
     </li>
     <li>
      <strong>
       %MEM
      </strong>
      ：指进程占用物理内存的百分比。它体现了进程在系统内存资源分配中所占用的份额。比如，一个进程的 % MEM 为 20%，表示它占用了系统 20% 的可用物理内存。
     </li>
     <li>
      <strong>
       VIRT（虚拟内存大小）
      </strong>
      ：进程使用的虚拟内存总量，包括所有代码、数据和共享库占用的空间，以及可能尚未实际分配物理内存的部分。
     </li>
     <li>
      <strong>
       RES（常驻内存大小）
      </strong>
      ：进程实际占用的物理内存大小，即进程当前在内存中驻留的部分，不包括已经换出到磁盘交换空间（swap）的内存。
     </li>
     <li>
      <strong>
       SHR（共享内存大小）
      </strong>
      ：进程使用的共享内存大小，这部分内存可以被多个进程共享，用于进程间通信或共享数据。
     </li>
     <li>
      <strong>
       PR（优先级）
      </strong>
      ：表示进程的优先级，数值越小优先级越高。优先级高的进程在竞争 CPU 和内存等资源时会更有优势。
     </li>
     <li>
      <strong>
       NI（谦让度）
      </strong>
      ：用于调整进程的优先级，范围通常是 - 20 到 19，-20 表示最高优先级，19 表示最低优先级。用户可以通过调整 NI 值来改变进程获取 CPU 资源的优先级。
     </li>
    </ul>

    <p>
     但当两个或多个
     <code>
      std::shared_ptr
     </code>

     <code>
      std::shared_ptr
     </code>
     替换为
     <code>
      std::weak_ptr
     </code>
     来打破循环。
    </p>

    <h2>
     虚函数和纯虚函数有什么区别
    </h2>
    <p>
     <strong>
      <span style="color:#ff9900">
       虚函数
      </span>
     </strong>
     主要用于实现运行时多态。当通过基类指针或引用调用虚函数时，会根据指针或引用实际指向的对象类型来决定调用哪个版本的函数，这种机制称为动态绑定。
    </p>
    <p>
     <span style="color:#ff9900">
      <strong>
       纯虚函数
      </strong>
     </span>
     的主要作用是
     <strong>
      <span style="color:#ff9900">
       定义一个接口规范，要求派生类必须实现该函数。抽象类不能实例化对象
      </span>
     </strong>
     ，它的主要用途是作为基类，为派生类提供一个统一的接口，派生类必须实现抽象类中的所有纯虚函数，否则该派生类也会成为抽象类。
    </p>

    <h2>
     锁的种类，底层实现机制
    </h2>
    <h4>
     Linux 锁的种类及原理
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        自旋锁（Spinlock）
       </strong>
       <br/>
       通过原子操作（如
       <code>
        test-and-set
       </code>
       ）实现，未获取锁时线程忙等待，适用于短临界区。内核中通过
       <code>
        spin_lock()
       </code>
       实现，依赖底层原子指令（如x86的
       <code>
        lock cmpxchg
       </code>
       ）。
      </p>
     </li>
     <li>
      <p>
       <strong>
        互斥锁（Mutex）
       </strong>
       <br/>
       基于内核的
       <code>
        futex
       </code>
       （快速用户态互斥锁），未获取锁时线程休眠。内核态实现依赖信号量和等待队列，用户态通过
       <code>
        pthread_mutex_t
       </code>
       封装。
      </p>
     </li>
     <li>
      <p>
       <strong>
        读写锁（rwlock）
       </strong>
       <br/>
       通过原子变量区分读写状态，读共享、写独占。内核中
       <code>
        rwlock_t
       </code>
       使用自旋锁保护计数器，用户态
       <code>
        pthread_rwlock_t
       </code>
       结合条件变量实现。
      </p>
     </li>
     <li>
      <p>
       <strong>
        RCU（Read-Copy-Update）
       </strong>
       <br/>
       无锁读，写操作通过副本替换和延迟释放旧数据实现同步。依赖内存屏障和原子指针操作。
      </p>
     </li>
    </ol>
    <hr/>
    <h4>
     C++ 锁的种类及原理
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        std::mutex
       </strong>
       <br/>
       底层通常封装
       <code>
        pthread_mutex_t
       </code>
       ，基于操作系统的互斥锁（如Linux的
       <code>
        futex
       </code>
       ），通过原子操作和内核等待队列实现阻塞。
      </p>
     </li>
     <li>
      <p>
       <strong>
        std::recursive_mutex
       </strong>
       <br/>
       允许同一线程重复加锁，内部记录持有线程ID和计数器，释放需匹配加锁次数。
      </p>
     </li>
     <li>
      <p>
       <strong>
        std::shared_mutex（C++17）
       </strong>
       <br/>
       读写锁实现，用户态通过原子变量和条件变量区分读写模式，类似
       <code>
        pthread_rwlock_t
       </code>
       。
      </p>
     </li>
     <li>
      <p>
       <strong>
        std::timed_mutex
       </strong>
       <br/>
       支持超时加锁，底层通过系统调用（如
       <code>
        pthread_mutex_timedlock
       </code>
       ）实现时间判断。
      </p>
     </li>
    </ol>
    <h4>
     递归锁（Recursive Lock
     <strong>
      std::recursive_mutex
     </strong>
     ）
    </h4>
    <p>
     递归锁是一种
     <strong>
      允许同一线程多次加锁
     </strong>
     的特殊互斥锁，主要用于解决嵌套调用场景中的死锁问题。
    </p>
    <hr/>
    <h5>
     核心特性
    </h5>
    <ol>
     <li>
      <p>
       <strong>
        重入性
       </strong>
       <br/>
       同一线程可重复加锁，内部通过计数器记录加锁次数，每次解锁计数器减1，直到归零时真正释放锁。
       <br/>
       <strong>
        底层实现
       </strong>
       ：记录持有线程ID和计数器（如C++的
       <code>
        std::recursive_mutex
       </code>
       ）。
      </p>
     </li>
     <li>
      <p>
       <strong>
        避免自死锁
       </strong>
       <br/>
       普通互斥锁（如
       <code>
        std::mutex
       </code>
       ）若同一线程重复加锁会直接死锁，递归锁则允许这种嵌套操作。
      </p>
     </li>
    </ol>
    <hr/>
    <h5>
     典型使用场景
    </h5>
    <ul>
     <li>
      <strong>
       递归函数
      </strong>
      ：函数内部调用自身时需重复加锁。
     </li>
     <li>
      <strong>
       多层级调用
      </strong>
      ：多个函数嵌套调用且共享同一锁资源。
     </li>
    </ul>
