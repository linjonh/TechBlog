    <h4>
     逆向工程与重构工程
    </h4>
    <ul>
     <li>
      <strong>
       定义
      </strong>
      ：
      <ul>
       <li>
        <strong>
         逆向工程
        </strong>
        ：分析已有的程序，寻求比源代码更高级的抽象表现形式。主要在软件生命周期内，将软件某种形式的描述转换成更为抽象的形式。
       </li>
       <li>
        <strong>
         重构
        </strong>
        ：指在同一抽象级别上转换系统描述形式。
       </li>
       <li>
        <strong>
         设计恢复
        </strong>
        ：借助工具从已有程序中抽象出有关数据设计、总体结构设计和过程设计的信息。
       </li>
       <li>
        <strong>
         重构工程
        </strong>
        ：在逆向工程获得的信息基础上修改或重构已有的系统，产生系统的一个新版本。
       </li>
      </ul>
     </li>
    </ul>
    <ol>
     <li>
      <p>
       <strong>
        完备性
       </strong>
       ：逆向工程的完备性指的是在特定抽象层次上提供的信息的详细程度。通常情况下，抽象层次越高，提供的设计信息越少，即完备性越低。
      </p>
     </li>
     <li>
      <p>
       <strong>
        抽象层次
       </strong>
       ：
      </p>
      <ul>
       <li>
        <strong>
         底层抽象
        </strong>
        ：例如，过程的设计模型，这是逆向工程中最直接的信息提取。
       </li>
       <li>
        <strong>
         稍高层次的抽象
        </strong>
        ：例如，程序和数据结构的信息。
       </li>
       <li>
        <strong>
         相对高层的抽象
        </strong>
        ：例如，对象模型、数据和控制流模型。
       </li>
       <li>
        <strong>
         高层抽象
        </strong>
        ：例如，UML图、状态图和部署图。
       </li>
      </ul>
     </li>
    </ol>
    <h4>
     恢复信息的级别
    </h4>
    <ol>
     <li>
      <strong>
       实现级
      </strong>
      ：程序的抽象语法树、符号表等。
     </li>
     <li>
      <strong>
       结构级
      </strong>
      ：反映程序分量之间相互依赖关系的信息，如调用图、结构图。
     </li>
     <li>
      <strong>
       功能级
      </strong>
      ：反映程序段功能及程序段之间关系的信息。
     </li>
     <li>
      <strong>
       领域级
      </strong>
      ：反映程序分量或程序诸实体与应用领域概念之间对应关系的信息。
     </li>
    </ol>
    <h4>
     恢复信息的方法
    </h4>
    <ol>
     <li>
      <strong>
       用户指导下的搜索与变换 (User-Directed Search and Transformation):
      </strong>
      用于导出实现级和结构级信息。
     </li>
     <li>
      <strong>
       变换式方法 (Transformational Approaches):
      </strong>
      <ul>
       <li>
        <strong>
         自动分析法:
        </strong>
        如静态分析和调用图生成。
       </li>
       <li>
        <strong>
         基于特定库的用户指导变换法:
        </strong>
        需要用户指定规则库。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       基于领域知识的方法 (Domain Knowledge-Based):
      </strong>
      适用于功能级和领域级信息恢复。
     </li>
     <li>
      <strong>
       铅板恢复法 (Lead Plate Recovery):
      </strong>
      识别程序设计中的简单算法和公共结构。
     </li>
    </ol>
    <h4>
     选择实施逆向工程的条件
    </h4>
    <ul>
     <li>
      决定或移植、或重新设计、或为重用而需验证正确性的程序。
     </li>
     <li>
      缺乏足够的时机、支持工具或经济上得不偿失的情况下，一般不会选择实施逆向工程。
     </li>
    </ul>
    <h4>
     举例说明
    </h4>
    <p>
     假设有一个已经上线的电商平台软件，但没有详细的开发文档。
    </p>
    <ul>
     <li>
      <strong>
       问题定义:
      </strong>
      该电商平台需要增加支持多语言的功能。
     </li>
     <li>
      <strong>
       可行性研究:
      </strong>
      确认多语言功能的市场需求和实施可行性。
     </li>
     <li>
      <strong>
       需求分析:
      </strong>
      制定详细的需求规格说明书。
     </li>
    </ul>
    <p>
     但是，开发团队发现原来的代码结构复杂，难以直接添加新功能。这时，可以使用逆向工程来恢复原来的设计信息。
    </p>
    <ol>
     <li>
      <p>
       <strong>
        实现级恢复：
       </strong>
      </p>
      <ul>
       <li>
        使用工具分析代码结构，生成符号表、抽象语法树等。
       </li>
       <li>
        例如，通过工具生成整个电商平台的调用关系图，了解代码的具体执行路径。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        结构级恢复：
       </strong>
      </p>
      <ul>
       <li>
        识别模块之间的依赖关系，生成调用图、结构图。
       </li>
       <li>
        例如，通过工具分析，可以找出电商平台中各个模块（如用户管理模块、商品管理模块）之间的调用关系。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        功能级恢复：
       </strong>
      </p>
      <ul>
       <li>
        分析代码实现的功能，理解程序段的作用。
       </li>
       <li>
        例如，通过检查代码逻辑，可以确定某个函数处理用户登录逻辑。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        领域级恢复：
       </strong>
      </p>
      <ul>
       <li>
        理解代码与业务领域的对应关系，即代码实现的具体业务功能。
       </li>
       <li>
        例如，识别代码中实现的推荐算法如何与电商平台的推荐业务相联系。
       </li>
      </ul>
     </li>
    </ol>
