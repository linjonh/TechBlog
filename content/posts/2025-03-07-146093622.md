    <h4 style="background-color:transparent">
     Redis 事务的特性
    </h4>
    <p>
     Redis 事务（Transaction）是一组命令的集合，可以一次性、顺序地执行多个命令，并且在执行过程中不会被其他客户端的请求打断。Redis 事务的特性主要包括以下几点：
    </p>
    <ol>
     <li>
      <strong>
       原子性（Atomicity）
      </strong>
      ：事务中的所有命令都会按顺序执行，但 Redis 事务
      <strong>
       不支持回滚
      </strong>
      ，即如果某个命令执行失败，不会影响之前已执行的命令。
     </li>
     <li>
      <strong>
       隔离性（Isolation）
      </strong>
      ：事务在执行过程中，不会受到其他客户端的并发干扰，所有命令按顺序执行，不会被其他命令插入。
     </li>
     <li>
      <strong>
       一致性（Consistency）
      </strong>
      ：Redis 不支持回滚，但如果某个命令写入了错误的数据，会影响数据的正确性，因此一致性需要业务逻辑保证。
     </li>
     <li>
      <strong>
       持久性（Durability）
      </strong>
      ：如果 Redis 开启了 AOF（Append-Only File）或 RDB（Redis Database）持久化机制，事务的修改可以持久化到磁盘，否则数据只存在于内存中。
     </li>
    </ol>
    <h4 style="background-color:transparent">
     Redis 事务的使用
    </h4>
    <p>
     Redis 事务的基本使用方式如下：
    </p>
    <ol>
     <li>
      <strong>
       开启事务
      </strong>
      ：使用
      <code>
       MULTI
      </code>
      命令开启事务。
     </li>
     <li>
      <strong>
       命令入队
      </strong>
      ：在
      <code>
       MULTI
      </code>
      之后，所有输入的命令都会进入队列，而不会立即执行。
     </li>
     <li>
      <strong>
       执行事务
      </strong>
      ：使用
      <code>
       EXEC
      </code>
      命令执行所有入队的命令。
     </li>
     <li>
      <strong>
       放弃事务
      </strong>
      ：使用
      <code>
       DISCARD
      </code>
      命令取消事务，清空事务队列。
     </li>
    </ol>
    <h5 style="background-color:transparent">
     示例代码：
    </h5>
    <pre><code class="hljs">127.0.0.1:6379&gt; MULTI
OK
127.0.0.1:6379&gt; SET key1 value1
QUEUED
127.0.0.1:6379&gt; SET key2 value2
QUEUED
127.0.0.1:6379&gt; EXEC
1) OK
2) OK
</code></pre>
    <p>
     如果
     <code>
      EXEC
     </code>
     之前执行
     <code>
      DISCARD
     </code>
     ，事务队列会被清空，不执行任何命令。
    </p>
    <h4 style="background-color:transparent">
     Redis 事务的适用场景
    </h4>
    <p>
     Redis 事务适用于一些简单的原子操作场景，例如：
    </p>
    <ol>
     <li>
      <strong>
       批量执行多个命令
      </strong>
      ：如一次性设置多个 key 或删除多个 key。
     </li>
     <li>
      <strong>
       确保操作的有序性
      </strong>
      ：事务中的命令会按顺序执行，不会被其他命令打断。
     </li>
     <li>
      <strong>
       配合
       <code>
        WATCH
       </code>
       实现乐观锁
      </strong>
      ：
      <ul>
       <li>
        <code>
         WATCH
        </code>
        监视某个 key，在
        <code>
         EXEC
        </code>
        执行前如果该 key 发生变化，事务会被取消，适用于类似数据库的
        <strong>
         CAS（Compare-And-Swap）
        </strong>
        操作。
       </li>
      </ul>
     </li>
    </ol>
    <p>
     示例：
    </p>
    <pre><code class="hljs">127.0.0.1:6379&gt; WATCH counter
OK
127.0.0.1:6379&gt; MULTI
OK
127.0.0.1:6379&gt; INCR counter
QUEUED
127.0.0.1:6379&gt; EXEC
(integer) 1  # 如果 counter 没有被其他客户端修改，则成功执行
</code></pre>
    <p>
     如果
     <code>
      counter
     </code>
     在
     <code>
      EXEC
     </code>
     之前被其他客户端修改，事务会被取消。
    </p>
    <h4 style="background-color:transparent">
     事务的局限性
    </h4>
    <ul>
     <li>
      <strong>
       不支持回滚
      </strong>
      ：如果事务中某条命令执行失败，已执行的命令不会回滚。
     </li>
     <li>
      <strong>
       单线程执行
      </strong>
      ：Redis 是单线程的，事务中的所有命令都会按照顺序执行，不会被其他命令打断，但不会提高执行效率。
     </li>
     <li>
      <strong>
       不支持部分执行
      </strong>
      ：如果
      <code>
       EXEC
      </code>
      失败，整个事务都不会执行，但如果事务中的某个命令失败，不影响其他命令执行。
     </li>
    </ul>
    <p>
     在实际开发中，Redis 事务适用于以下场景：
    </p>
    <h4>
     1.
     <strong>
      计数器的原子更新
     </strong>
    </h4>
    <p>
     适用于高并发场景，如点赞、浏览量、订单编号等计数操作。例如，一个网站的文章浏览量可以使用 Redis 事务进行更新：
    </p>
    <pre><code class="hljs">WATCH article:1001:views
MULTI
INCR article:1001:views
EXEC
</code></pre>
    <p>
     这样可以保证在高并发情况下，计数操作不会出现丢失更新的问题。
    </p>
    <h4>
     2.
     <strong>
      抢购/秒杀系统
     </strong>
    </h4>
    <p>
     在电商系统中，Redis 事务可以用于限量抢购，防止超卖。例如：
    </p>
    <pre><code class="hljs">WATCH stock:product_123
MULTI
GET stock:product_123  # 读取库存
DECR stock:product_123  # 扣减库存
EXEC
</code></pre>
    <p>
     如果
     <code>
      stock:product_123
     </code>
     在
     <code>
      EXEC
     </code>
     之前被其他用户修改，事务将会失败，从而避免超卖。
    </p>
    <h4>
     3.
     <strong>
      账户余额转账
     </strong>
    </h4>
    <p>
     在金融系统中，Redis 事务可以用于保证转账操作的原子性：
    </p>
    <pre><code class="hljs">WATCH balance:user_1 balance:user_2
MULTI
DECR balance:user_1 100  # 从用户1账户扣款
INCR balance:user_2 100  # 给用户2账户加款
EXEC
</code></pre>
    <p>
     如果
     <code>
      EXEC
     </code>
     之前
     <code>
      balance:user_1
     </code>
     或
     <code>
      balance:user_2
     </code>
     发生变化，事务会取消，避免出现数据不一致的问题。
    </p>
    <h4 style="background-color:transparent">
     总结
    </h4>
    <p>
     Redis 事务通过
     <code>
      MULTI
     </code>
     、
     <code>
      EXEC
     </code>
     、
     <code>
      DISCARD
     </code>
     和
     <code>
      WATCH
     </code>
     来实现，适用于批量执行命令、保证操作的顺序性和实现乐观锁机制。但由于不支持回滚，在需要强一致性的场景下，建议配合 Lua 脚本或其他机制来确保数据的完整性。
    </p>
