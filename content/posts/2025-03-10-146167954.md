     <h2>
      深入再谈智能指针、
      <code>
       AsRef
      </code>
      引用与
      <code>
       Borrow
      </code>
      借用
     </h2>
     <p style="text-align:left;">
      这是一个具有深度的技术主题。每次重温其理论知识，都会有新的领悟。大约 2 年前，我曾就这一技术方向撰写过另一篇短文《从类型转换视角，浅谈
      <code>
       Deref&lt;Target = T&gt;
      </code>
      ,
      <code>
       AsRef&lt;T&gt;
      </code>
      ,
      <code>
       Borrow&lt;T&gt;
      </code>
      和
      <code>
       From&lt;T&gt; trait
      </code>
      差异》。在那篇文章中，我依据当时的经验知识，归纳了自定义智能指针、引用和借用的代码实现方式，并罗列了一些原则。但那种“快餐式”的知识分享存在不少遗憾。两年后的今天，我凭借更多的知识积累，利用春节长假的时间，重拾此话题，力求通过一篇长文将（广义的）
      <code>
       Rust
      </code>
      “引用”阐述清楚。
     </p>
     <h3>
      名词解释
     </h3>
     <p style="text-align:left;">
      为了统一对非具名抽象概念的理解，我们先对几个
      <code>
       Rust
      </code>
      常见项的中文表述进行约定。
     </p>
     <h4>
      @Rustacean
     </h4>
     <p style="text-align:left;">
      一般发音为
      <code>
       /ˌrʌstəˈsiːən/
      </code>
      。它是由单词
      <code>
       Rust
      </code>
      和词根
      <code>
       -acean
      </code>
      （表示“……的人”的后缀）构成。它被用作
      <code>
       Rust
      </code>
      技术栈程序员的自称。
     </p>
     <h4>
      自定义引用
     </h4>
     <p style="text-align:left;">
      代表
      <code>
       std::convert::AsRef&lt;F&gt;
      </code>
      或
      <code>
       std::convert::AsMut&lt;F&gt;
      </code>
      的特征实现类。例如，泛型类型参数
      <code>
       &lt;T: AsRef&lt;F&gt;&gt;
      </code>
      表示类型
      <code>
       T
      </code>
      是
      <code>
       F
      </code>
      的自定义引用。普通引用、自定义引用与智能指针之间的对比关系可以被归纳为
     </p>
     <img alt="输入图片说明" src="https://i-blog.csdnimg.cn/img_convert/45c9c287c4c9f1c000cdfaa3cb162f69.png" title="屏幕截图">
      <figcaption>
       输入图片说明
      </figcaption>
      <h4>
       自定义借用
      </h4>
      <p style="text-align:left;">
       代表
       <code>
        std::borrow::Borrow&lt;F&gt;
       </code>
       或
       <code>
        std::borrow::BorrowMut&lt;F&gt;
       </code>
       特征的实现类。比如，泛型类型参数
       <code>
        &lt;T: Borrow&lt;F&gt;&gt;
       </code>
       表示类型
       <code>
        T
       </code>
       是
       <code>
        F
       </code>
       的自定义借用，也可读作“
       <code>
        T
       </code>
       被借入作为
       <code>
        F
       </code>
       ”。
      </p>
      <h4>
       胖引用
      </h4>
      <p style="text-align:left;">
       代表动态分派特征对象
       <code>
        trait Object
       </code>
       的普通引用。例如，
       <code>
        &amp;dyn AsRef&lt;std::path::Path&gt;
       </code>
       。
      </p>
      <h4>
       胖智能指针
      </h4>
      <p style="text-align:left;">
       代表动态分派特征对象
       <code>
        trait Object
       </code>
       的智能指针。例如，
       <code>
        Box&lt;dyn AsRef&lt;std::path::Path&gt;&gt;
       </code>
       。
      </p>
      <h4>
       胖指针
      </h4>
      <p style="text-align:left;">
       是对胖引用与胖智能指针的统称。
      </p>
      <h4>
       泛型覆盖实现
      </h4>
      <p style="text-align:left;">
       英文全称是
       <code>
        Blanket Implementation
       </code>
       ，它是作用于泛型类型参数的特征实现块。简单来说，泛型覆盖实现允许 @Rustacean 为一批满足特定泛型条件的类型统一实现某个特征，而无需为每个具体类型单独实现，这样可以减少代码重复。例如，
       <code>
        impl&lt;T: ?Sized&gt; Borrow&lt;T&gt; for T { /* 特征成员方法的实现 */ }
       </code>
       。该定义的关键在于特征实现块的“受体”不是一个具体类型，而是满足泛型参数限定条件的一批具体类型。
      </p>
      <h4>
       特征成员方法
      </h4>
      <p style="text-align:left;">
       对应英文词条是
       <code>
        trait method
       </code>
       ，指在特征定义内声明的成员方法以及在特征实现块内实现的成员方法。
      </p>
      <h4>
       智能指针内部值的数据类型
      </h4>
      <p style="text-align:left;">
       以
       <code>
        &lt;T: Deref&lt;Target = F&gt;&gt;
       </code>
       为例，在文章中，
      </p>
      <ul>
       <li>
        <p style="text-align:left;">
         要么，将内部值数据类型记作
         <code>
          F
         </code>
         ，出于文字简洁的目的。
        </p>
       </li>
       <li>
        <p style="text-align:left;">
         要么，将其记作
         <code>
          &lt;T as Deref&gt;::Target
         </code>
         关联类型，以强调当前是在智能指针讨论上下文内。
        </p>
       </li>
      </ul>
      <h4>
       <code>
        &lt;T: ?Copy&gt;
       </code>
       特征限定条件
      </h4>
      <p style="text-align:left;">
       首先，这个“泛型（类型）参数限定条件”记法从语法规则上肯定是
       <strong>
        错
       </strong>
       的，因为即便最新版
       <code>
        rustc
       </code>
       也仅只认识
       <code>
        ?Sized
       </code>
       一个“不确定”限定条件。
      </p>
      <p style="text-align:left;">
       其次，
       <code>
        &lt;T: ?Sized&gt;
       </code>
       代表“直至编译时，还不能确定被限定的泛型（类型）参数
       <code>
        T
       </code>
       是否有可度量的字节大小”。即，类型
       <code>
        T
       </code>
       是
       <code>
        DST
       </code>
       的。
      </p>
      <p style="text-align:left;">
       模仿
       <code>
        ?Sized
       </code>
       ，我想用
       <code>
        &lt;T: ?Copy&gt;
       </code>
       表示“直至编译时，还不能确定被限定的泛型（类型）参数
       <code>
        T
       </code>
       是否可复制”。注：【可克隆】不等同于【可复制】 — 这是两码事。
      </p>

       概述
      </h3>

       <h3>
        解引用的触发方式不同
       </h3>
       <p style="text-align:left;">
        首先，对“自定义引用”与“自定义借用”的解引用操作都要求 @Rustacean 必须手动调用对应的特征成员方法（
       </p>
       <ul>
        <li>
         <p style="text-align:left;">
          解引用自定义引用
         </p>
         <ul>
          <li>
           <p style="text-align:left;">
            <code>
             &lt;T as AsMut&gt;::as_mut(&amp;mut T) ➜ &amp;mut F
            </code>
            — 来自【标准库】对
            <code>
             trait AsMut
            </code>
            的直接实现
           </p>
          </li>
          <li>
           <p style="text-align:left;">
            <code>
             &lt;&amp;mut T as AsMut&gt;::as_mut(&amp;mut &amp;mut T) ➜ &amp;mut F
            </code>
            — 来自【标准库】对
            <code>
             trait AsMut
            </code>
            的
            <strong>
             泛型覆盖实现
            </strong>
           </p>
          </li>
         </ul>
         <ul>
          <li>
           <p style="text-align:left;">
            <code>
             &lt;T as AsRef&gt;::as_ref(&amp;T) ➜ &amp;F
            </code>
            — 来自【标准库】对
            <code>
             trait AsRef
            </code>
            的直接实现
           </p>
          </li>
          <li>
           <p style="text-align:left;">
            <code>
             &lt;&amp;T as AsRef&gt;::as_ref(&amp;&amp;T) ➜ &amp;F
            </code>
            — 来自【标准库】对
            <code>
             trait AsRef
            </code>
            的
            <strong>
             泛型覆盖实现
            </strong>
           </p>
          </li>
          <li>
           <p style="text-align:left;">
            <code>
             &lt;&amp;mut T as AsRef&gt;::as_ref(&amp;&amp;mut T) ➜ &amp;F
            </code>
            — 来自【标准库】对
            <code>
             trait AsRef
            </code>
            的
            <strong>
             泛型覆盖实现
            </strong>
           </p>
          </li>
         </ul>
         <ul>
          <li>
           <p style="text-align:left;">
            只读
           </p>
          </li>
          <li>
           <p style="text-align:left;">
            可变
           </p>
          </li>
         </ul>
        </li>
        <li>
         <p style="text-align:left;">
          解引用自定义借用
         </p>
         <ul>
          <li>
           <p style="text-align:left;">
            [只读]
            <code>
             &lt;T as Borrow&gt;::borrow(&amp;T) ➜ &amp;F
            </code>
           </p>
          </li>
          <li>
           <p style="text-align:left;">
            [可变]
            <code>
             &lt;T as BorrowMut&gt;::borrow_mut(&amp;mut T) ➜ &amp;mut F
            </code>
           </p>
          </li>
         </ul>
        </li>
       </ul>
       <p style="text-align:left;">
        ）才能完成从
        <code>
         T
        </code>
        /
        <code>
         &amp;T
        </code>
        /
        <code>
         &amp;mut T
        </code>
        至
        <code>
         &amp;F
        </code>
        /
        <code>
         &amp;mut F
        </code>
        的（纯手动）解引用类型转换。举个[例程 1]
       </p>
       <pre class="has"><code class="language-go">use ::std::path::{ Path, PathBuf };
fn print&lt;K: AsRef&lt;Path&gt;&gt;(file_path: K) {
    // 注意：由于编译器不会自动生成对 file_path.as_ref() 成员方法的调用语句，
    //      因此开发者必须显式地手动调用 &lt;K as AsRef&gt;::as_ref(&amp;K)。
    let file_path: &amp;Path = file_path.as_ref();
    println!("文件路径= {}", file_path.display());
}
fn main() {
    let string_2_path = String::from("/etc/&lt;String&gt;");
    let path_buf_2_path = PathBuf::from("/etc/&lt;PathBuf&gt;");
    // &amp;T ➜ &amp;F 在 print() 函数调用之后，保留变量所有权不被消费掉。
    print(&amp;string_2_path);   // &amp;String ➜ &amp;Path
    print(&amp;path_buf_2_path); // &amp;PathBuf ➜ &amp;Path
    // &amp;&amp;T ➜ &amp;F
    print(&amp;&amp;string_2_path); // &amp;&amp;String ➜ &amp;Path
    print(&amp;&amp;path_buf_2_path); // &amp;&amp;PathBuf ➜ &amp;Path
    // T ➜ &amp;F 在 print() 函数调用之后，消耗掉了变量所有权。
    print(string_2_path); // String ➜ &amp;Path
    print(path_buf_2_path); // PathBuf ➜ &amp;Path
}</code></pre>
       <p style="text-align:left;">
        前面我们介绍了解引用自定义引用和自定义借用的方式，接下来看看对“智能指针”
        <strong>
         引用
        </strong>
        的解引用处理有什么不同。
       </p>
       <p style="text-align:left;">
        对“智能指针”
        <strong>
         引用
        </strong>
        的解引用处理（
        <code>
         &amp;T ➜ &amp;F
        </code>
        或
        <code>
         &amp;mut T ➜ &amp;mut F
        </code>
        ）则是由（前端）编译器
        <code>
         rustc
        </code>
        <strong>
         自动完成
        </strong>
        的。具体地讲，编译器会在语义分析过程中
       </p>
       <ol>
        <li>
         <p style="text-align:left;">
          先识别出
         </p>
        </li>
       </ol>
       <ul>
        <li>
         <p style="text-align:left;">
          在成员方法调用语句中，对应
          <code>
           &amp;self
          </code>
          或
          <code>
           &amp;mut self
          </code>
          形参的智能指针
          <strong>
           引用
          </strong>
          实参
          <code>
           &amp;T
          </code>
          或
          <code>
           &amp;mut T
          </code>
          ，以及
         </p>
        </li>
        <li>
         <p style="text-align:left;">
          在函数、闭包、成员方法调用语句中，对应其它
          <strong>
           普通引用
          </strong>
          形参的智能指针
          <strong>
           引用
          </strong>
          实参
          <code>
           &amp;T
          </code>
          或
          <code>
           &amp;mut T
          </code>
         </p>
        </li>
       </ul>
       <p style="text-align:left;">
        再将
        <code>
         &amp;T
        </code>
        或
        <code>
         &amp;mut T
        </code>
        原地替换为对它们的
        <strong>
         解引用
        </strong>
        特征成员方法调用表达式
       </p>
       <ul>
        <li>
         <p style="text-align:left;">
          <code>
           &lt;T as Deref&gt;::deref(&amp;T) ➜ &amp;F
          </code>
          或
         </p>
        </li>
        <li>
         <p style="text-align:left;">
          <code>
           &lt;T as DerefMut&gt;::deref_mut(&amp;mut T) ➜ &amp;mut F
          </code>
         </p>
        </li>
       </ul>
       <p style="text-align:left;">
        接着，判断上一步返回值中的
        <code>
         F
        </code>
        是否又是智能指针？
       </p>
       <ul>
        <li>
         <p style="text-align:left;">
          要么，解引用后的
          <code>
           &amp;F
          </code>
          已匹配于函数形参的数据类型
         </p>
        </li>
        <li>
         <p style="text-align:left;">
          要么，
          <code>
           F
          </code>
          已不再是智能指针，且不能进一步解引用了
         </p>
        </li>
       </ul>
       <ol>
        <li>
         <p style="text-align:left;">
          若是，则绕回第二步将
          <code>
           F
          </code>
          当作
          <code>
           T
          </code>
          接着递归解引用
          <code>
           F
          </code>
          。递归处理会持续进行，直至
         </p>
         <p style="text-align:left;">
          若递归解引用被持续执行多次，那么在原来
          <code>
           &amp;T
          </code>
          实参位置上会出现一条对
          <code>
           T
          </code>
          的
          <code>
           .deref()
          </code>
          成员方法调用链条，或在原来
          <code>
           &amp;mut T
          </code>
          实参位置上会出现一条对
          <code>
           T
          </code>
          的
          <code>
           .deref_mut()
          </code>
          成员方法调用链条。
         </p>
        </li>
        <li>
         <p style="text-align:left;">
          否则，则直接进行下一步。
         </p>
        </li>
       </ol>
       <p style="text-align:left;">
        最后，判断
        <code>
         &amp;F
        </code>
        是否匹配于函数形参的数据类型
       </p>
       <ol>
        <li>
         <p style="text-align:left;">
          若匹配，则此段编译成功，并执行后续编译处理。
         </p>
        </li>
        <li>
         <p style="text-align:left;">
          否则，
          <strong>
           整体编译失败
          </strong>
          。
         </p>
        </li>
       </ol>

        <code>
         rustc
        </code>
        ，而不是 @Rustacean。
       </p>

        <pre class="has"><code class="language-go">use ::std::path::{ Path, PathBuf };
fn print(file_path: &amp;Path) {
    println!("文件路径= {}", file_path.display());
}
fn main() {
    let mut path_buf = PathBuf::from("/etc/&lt;&amp;PathBuf&gt;");
    // 场景一：在成员方法调用中，对 &amp;self 与 &amp;mut self 的自动解引用
    // (1) &amp;mut T ➜ &amp;mut F 修改智能指针内部值的内部状态信息
    path_buf.push("usr");

            </strong>
            就不一定成立了，无论它是动态分派的特征对象（例如，
            <code>
             Box&lt;dyn AsRef&lt;F&gt;&gt;
            </code>
            ），还是静态分派的泛型（例如，
            <code>
             Box&lt;T&gt; where T: AsRef&lt;F&gt;
            </code>
            ）。至少由【标准库】直供的智能指针数据结构都定义了自己专属的
            <code>
             trait AsRef&lt;F&gt;
            </code>
            ，
            <code>
             trait AsMut&lt;F&gt;
            </code>
            ，
            <code>
             trait Borrow&lt;F&gt;
            </code>
            与
            <code>
             trait BorrowMut&lt;F&gt;
            </code>
            实现块和为它们定义了独立解释的语义功能。于是，智能指针实例自身的特征成员方法（比如，
            <code>
             &lt;Box&lt;T&gt; as AsRef&gt;::as_ref(&amp;Box&lt;T&gt;)
            </code>
            ）就会遮蔽掉其内部值上的同名成员方法
            <code>
             &lt;Deref::Target as AsRef&gt;::as_ref(&amp;Deref::Target)
            </code>
            ，进而阻断模拟“自动解引用”的泛型匹配。再举个[例程7]
           </p>

           <h5>
            智能指针的语义功能
           </h5>
           <p style="text-align:left;">
            虽然智能指针可作为模仿
            <code>
             OOP
            </code>
            编程风格（比如，继承）的
            <strong>
             反模式
            </strong>
            语法糖，但它的首要任务却是从如下两个维度（同时或之一地）增强其
            <code>
             Deref::Target
            </code>
            类型内部值的语义功能：
           </p>
           <ul>
            <li>
             <p style="text-align:left;">
              所有权关系 ownership。例如，
              <code>
               Rc&lt;T&gt;
              </code>
              被当作其内部值
              <code>
               T
              </code>
              的“引用”，和
              <strong>
               按所有权
              </strong>
              传值，并摆脱普通引用规则的诸多限制。
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              内存存储位置 storage。还是以
              <code>
               Rc&lt;T&gt;
              </code>
              为例，它腾挪内部值
              <code>
               T
              </code>
              从【栈】内存至【堆】内存。然后，构造多个指向相同【堆】数据的【栈】（所有权）“引用”变量。
             </p>
            </li>
           </ul>
           <h4>
            自定义借用
           </h4>
           <p style="text-align:left;">
            对“自定义
            <strong>
             借用
            </strong>
            ”的解引用处理也是建立在
            <code>
             Rust
            </code>
            泛型类型系统的“通用底盘”基础之上的。但它的首要用途已不再是模仿函数重载多态性的语法糖，而是（以
            <code>
             &lt;T: Borrow&lt;F&gt;&gt;
            </code>
            为例）
           </p>
           <ol>
            <li>
             <p style="text-align:left;">
              强制【借用
              <code>
               T
              </code>
              】与【被借用的值
              <code>
               F
              </code>
              】都对外呈现相同的：
             </p>
            </li>
           </ol>
           <ul>
            <li>
             <p style="text-align:left;">
              哈希值 --- 意味着处理逻辑一致的
              <code>
               trait std::hash::Hash
              </code>
              实现
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              等价关系 --- 意味着处理逻辑一致的
              <code>
               trait std::cmp::Eq
              </code>
              实现
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              排序关系 --- 意味着处理逻辑一致的
              <code>
               trait std::cmp::Ord
              </code>
              实现
             </p>
            </li>
           </ul>
           <p style="text-align:left;">
            督促 @Rustacean 对【借用
            <code>
             T
            </code>
            】与【被借用的值
            <code>
             F
            </code>
            】编写处理逻辑一致的特征实现块，当需要对它们实现除
            <code>
             std::borrow::Borrow
            </code>
            与
            <code>
             std::borrow::BorrowMut
            </code>
            之外的特征时。比如，我们一般预期【借用
            <code>
             T
            </code>
            】与【被借用的值
            <code>
             F
            </code>
            】都能被
            <code>
             print!
            </code>
            宏打印输出
            <strong>
             相同的内容
            </strong>
            ，通过给它们编写处理逻辑一致的
            <code>
             trait std::fmt::Display
            </code>
            特征实现块。
           </p>
           <p style="text-align:left;">
            换句话说，只要某个类型
            <code>
             T
            </code>
            实现了
            <code>
             trait Borrow&lt;F&gt;
            </code>
            或
            <code>
             trait BorrowMut&lt;F&gt;
            </code>
            ，那么类型
            <code>
             T
            </code>
            与
            <code>
             F
            </code>
           </p>
           <ul>
            <li>
             <p style="text-align:left;">
              【必有】相同的“哈希值”和“判等+排序”偏好。
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              【可选但有理由期望】对其它
              <code>
               trait
              </code>
              的实现，也有处理逻辑一致的特征实现块。
             </p>
            </li>
           </ul>
           <p style="text-align:left;">
            这是一套非常重要的约束规则。
           </p>
           <p style="text-align:left;">
            同时从概念冠名上，
           </p>
           <ul>
            <li>
             <p style="text-align:left;">
              <code>
               T
              </code>
              是
              <code>
               F
              </code>
              的自定义借用，和
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              <code>
               T
              </code>
              被借用作为
              <code>
               F
              </code>
             </p>
            </li>
           </ul>
           <h5>
            现实意义
           </h5>
           <p style="text-align:left;">
            令我恍然大悟的是，普通引用
            <code>
             &amp;T
            </code>
            /
            <code>
             &amp;mut T
            </code>
            与被引用值
            <code>
             T
            </code>
            之间处理行为的高度一致性也是源于这套【自定义借用】约束规则，因为【标准库】为任何普通引用都预置了如下对
            <code>
             trait Borrow&lt;F&gt;
            </code>
            与
            <code>
             trait BorrowMut&lt;F&gt;
            </code>
            的
            <strong>
             泛型覆盖实现
            </strong>
            ：
           </p>
           <ul>
            <li>
             <p style="text-align:left;">
              <code>
               impl&lt;T: ?Sized&gt; Borrow&lt;T&gt; for &amp;T { .. }
              </code>
             </p>
             <p style="text-align:left;">
              读作：任何类型
              <code>
               T
              </code>
              的
              <strong>
               只读引用
              </strong>
              <code>
               &amp;T
              </code>
              同时也
              <code>
               T
              </code>
              <strong>
               自身的
              </strong>
              只读
              <strong>
               自定义借用
              </strong>
              。
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              <code>
               impl&lt;T: ?Sized&gt; BorrowMut&lt;T&gt; for &amp;mut T { .. }
              </code>
             </p>
             <p style="text-align:left;">
              读作：任何类型
              <code>
               T
              </code>
              的
              <strong>
               可变引用
              </strong>
              <code>
               &amp;mut T
              </code>
              同时也
              <code>
               T
              </code>
              <strong>
               自身的
              </strong>
              可变
              <strong>
               自定义借用
              </strong>
              。
             </p>
            </li>
           </ul>
           <p style="text-align:left;">
            于是才有我凭经验知识与死记硬背才掌握的经验法则：
           </p>
           <p style="text-align:left;">
            <strong>
             “比较两个值的引用是否相等”就等效于“比较该引用背后的所有权值是否相等”，而不是匹配这两个引用是否指向同一处内存地址。即，
             <code>
              assert!(&amp;1 == &amp;1)
             </code>
             等效于
             <code>
              assert!(1 == 1)
             </code>
             ，而不是
             <code>
              assert!(std::ptr::eq(&amp;1, &amp;1))
             </code>
             。
            </strong>
           </p>
           <p style="text-align:left;">
            举个[例程8]更形象。
           </p>
           <pre class="has"><code class="language-go">use ::std::{ path::PathBuf, ptr };
fn main() {
    let path1 = PathBuf::from("/a/b/c");
    let path2 = PathBuf::from("/a/b/c");
    // 根据自定义借用的限定规则，比较引用就相当于比较被引用的值，
    assert!(&amp;path1 == &amp;path2); // 断言成功
    assert!(path1 == path2); // 断言成功
    // 而不是匹配引用的内存地址是否是指向的同一处。
    assert!(ptr::eq(&amp;path1, &amp;path2)); // 断言失败
}</code></pre>
           <p style="text-align:left;">
            到这，发散思维的读者必定又要发问：“这套约束规则对【智能指针】有啥影响呀？”。我快速回答：“没影响，因为【标准库】未提供面向
            <code>
             Deref(Mut)
            </code>
            限定条件的
            <code>
             Borrow(Mut)
            </code>
            泛型覆盖实现”。另外，只要
            <code>
             Deref(Mut)
            </code>
            实现类不定义与其关联类型
            <code>
             Deref::Target
            </code>
            重名的成员方法，那么
            <code>
             rustc
            </code>
            自动解引用机制就能保证：
           </p>
           <ol>
            <li>
             <p style="text-align:left;">
              智能指针不仅能点出其内部值的全部
              <code>
               pub
              </code>
              成员方法，更会保持与内部值的
              <code>
               trait method
              </code>
              完全一致的处理行为。但，很可惜,
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              当面对泛型类型匹配时，智能指针却不能“透明”呈现出其内部值的
              <code>
               trait
              </code>
              “形状”。还是讲，办法总比问题多。采用ambassador crate，借助 crate 作者预定义的过程宏，便可：
             </p>
             <ol>
              <li>
               <p style="text-align:left;">
                给智能指针数据结构增补实现其内部值才实现的
                <code>
                 trait
                </code>
                。
               </p>
              </li>
              <li>
               <p style="text-align:left;">
                将智能指针的
                <code>
                 trait method
                </code>
                实现委托给内部值实现的
                <code>
                 trait method
                </code>
                。
               </p>
               <p style="text-align:left;">
                于是，从宏观效果来看，智能指针与它的内部值既对外呈现相同的泛型
                <code>
                 trait
                </code>
                “形状”，还保持了一致的
                <code>
                 trait method
                </code>
                处理逻辑，这简直是完美致极！可是，这一切就已经与
                <code>
                 rustc
                </code>
                自动解引用没关系了。
               </p>
              </li>
             </ol>
            </li>
           </ol>
           <p style="text-align:left;">
            故事依旧未结束。甚至，一个惊艳接着另一个惊艳。【自定义借用】的约束规则还大幅提升了
            <code>
             Map
            </code>
            和
            <code>
             Set
            </code>
            类“可检索”数据结构的
            <strong>
             查询效率
            </strong>
            。简单地讲，【自定义借用】允许 @Rustacean
           </p>
           <ul>
            <li>
             <p style="text-align:left;">
              既能，将
              <strong>
               所有权值
              </strong>
              作为
              <strong>
               键
              </strong>
              数据保存于
              <code>
               Map
              </code>
              和
              <code>
               Set
              </code>
              数据结构中，以满足容器
              <strong>
               占有
              </strong>
              子元素的要求。
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              又可，使用
              <strong>
               更轻量级
              </strong>
              的自定义借用（算是广义引用的一种）作为对键数据
              <strong>
               匹配查询
              </strong>
              的搜索条件。
             </p>
            </li>
           </ul>
           <p style="text-align:left;">
            进而避免，为每次检索操作，都重新构造一个所有权值作为【键】的查询条件 — 内存效率极低。举个[例程10]，让读者更形象地体会一下
           </p>
           <pre class="has"><code class="language-go">use ::std::collections::HashMap;
fn main() {
    let mut map: HashMap&lt;String, i32&gt; = HashMap::new();
    // 向 Map 内保存字符串的所有权作为【键】
    let key123 = String::from("123");
    let key124 = String::from("124");
    map.insert(key123, 123);
    map.insert(key124, 124);
    // 在这一步涉及了 trait Borrow&lt;F&gt; 的两个知识点：
    // 1. 因为 String 是 &amp;str 的自定义借用，所以 String 与 &amp;str
    //    有相同的等价偏好与 hash 值。于是，由 String 为内容保存
    //    的键，就能由 &amp;str 为检索条件给匹配出来。
    // 2. 因为 &amp;i32 就是 i32 的自定义借用，&amp;i32 与 i32 就具备相
    //    同的等价偏好，所以就允许由 &amp;i32 引用之间的判等来断言其
    //    背后 i32 值是否相等。

           <p style="text-align:left;">
            写到这里，我有感而发：“
            <strong>
             哪有什么天生的易用体质，只是有【标准库】替我们负重前行
            </strong>
            ”。
           </p>
           <h6>
            反身性
            <code>
             Reflexivity
            </code>
           </h6>
           <p style="text-align:left;">
            相比于自定义引用，自定义借用还具备“反身性Reflexivity”，因为【标准库】为任何类型都预置了如下对
            <code>
             trait Borrow&lt;F&gt;
            </code>
            与
            <code>
             trait BorrowMut&lt;F&gt;
            </code>
            的
            <strong>
             泛型覆盖实现
            </strong>
            ：
           </p>
           <ul>
            <li>
             <p style="text-align:left;">
              <code>
               impl&lt;T: ?Sized&gt; Borrow&lt;T&gt; for T { .. }
              </code>
             </p>
             <p style="text-align:left;">
              读作：任何
              <strong>
               类型
              </strong>
              <code>
               T
              </code>
              就是它
              <strong>
               自身的
              </strong>
              只读
              <strong>
               自定义借用
              </strong>
              。
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              <code>
               impl&lt;T: ?Sized&gt; BorrowMut&lt;T&gt; for T { .. }
              </code>
             </p>
             <p style="text-align:left;">
              读作：任何
              <strong>
               类型
              </strong>
              <code>
               T
              </code>
              就是它
              <strong>
               自身的
              </strong>
              可变
              <strong>
               自定义借用
              </strong>
              。
             </p>
            </li>
           </ul>
           <p style="text-align:left;">
            为了证明反身性的存在，我再举个[例程9]佐证一下
           </p>
           <pre class="has"><code class="language-go">use ::std::{ borrow::Borrow, path::PathBuf };
// 注意下面函数形参的类型不是引用 &amp;PathBuf 哟，而是像所有权值的类型！
fn print_fst&lt;T: Borrow&lt;PathBuf&gt;&gt;(file_path: T) {
    let file_path: &amp;PathBuf = file_path.borrow(); // 手动解引用，而不是自动解引用
    println!("文件路径fst= {}", file_path.display());
}
fn main() {
    let path_buf = PathBuf::from("/etc/&lt;PathBuf&gt;");
    // 1. 任何类型 T 的普通引用 &amp;T 同时也是该类型 T 自身的自定义借用。
    //    所以，即便函数的形参不是引用，我们也能将引用作为它的实参。这是兼容的不违和！
    print_fst(&amp;path_buf);
    // 2. trait Borrow&lt;F&gt; 支持【反身性】。即，
    //    任何类型 T 就是它自身的"自定义借用"
    print_fst(path_buf);
}</code></pre>
           <p style="text-align:left;">
            在上段代码中，第3行的函数签名
            <strong>
             未
            </strong>
            以引用
            <code>
             &amp;PathBuf
            </code>
            为形参类型，而是将
            <strong>
             所有权
            </strong>
            的泛型类型
            <code>
             T
            </code>
            作为形参类型。但
           </p>
           <ol>
            <li>
             <p style="text-align:left;">
              第11行既能传递引用
              <code>
               &amp;path_buf
              </code>
              作为实参 — 自定义借用的泛型覆盖实现。同时，
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              第14行又能传所有权变量
              <code>
               path_buf
              </code>
              作为实参 — 自定义借用的反身性
             </p>
            </li>
           </ol>
           <p style="text-align:left;">
            因为它们都是原始变量
            <code>
             path_buf
            </code>
            的【自定义借用】。
           </p>
           <h3>
            智能指针的条件化
            <code>
             AsRef
            </code>
            特征实现块
           </h3>
           <p style="text-align:left;">
            不同于普通引用，智能指针被允许定义它自己的（特征）实现块和实现它自己的（特征）成员方法。于是，智能指针内部值（
            <code>
             Deref::Target
            </code>
            ）的同名成员方法就会被智能指针自身的成员方法给遮蔽掉和失效，因为
            <code>
             rustc
            </code>
            在对
            <code>
             self
            </code>
            /
            <code>
             &amp;self
            </code>
            /
            <code>
             &amp;mut self
            </code>
            的实参执行解引用处理前，就检索到了“目标”成员方法和提前进入函数调用处理流程 — 只匹配名，而不匹配参数列表。这不仅造成程序设计上的难点，更导致【普通引用】与【智能指针】对
            <strong>
             被引用的
            </strong>
            【自定义引用】处理逻辑的不一致。以自定义引用
            <code>
             &lt;T: AsRef&lt;F&gt;&gt;
            </code>
            例，
           </p>
           <ul>
            <li>
             <p style="text-align:left;">
              <code>
               &amp;T
              </code>
              的
              <code>
               as_ref()
              </code>
              特征成员方法返回
              <code>
               &amp;F
              </code>
              ，而
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              <code>
               Box&lt;T&gt;
              </code>
              的
              <code>
               as_ref()
              </code>
              特征成员方法却返回
              <code>
               &amp;T
              </code>
             </p>
            </li>
           </ul>
           <p style="text-align:left;">
            它们虽同为
            <code>
             T
            </code>
            的“引用”但同名成员方法却返回不同类型的解引用值。对此，标准库的技术选择是放任此“不和谐的”存在。但，我忍不了。我要把【智能指针】对【自定义引用】内部值的处理逻辑“掰直”对齐于【普通引用】。具体做法也不难，
           </p>
           <p style="text-align:left;">
            第一步，给每个
            <strong>
             自定义本地
            </strong>
            智能指针（数据结构），都增补如下一段对
            <code>
             trait AsRef&lt;F&gt;
            </code>
            与
            <code>
             trait AsMut&lt;F&gt;
            </code>
            的【条件化特征实现块】
           </p>
           <ul>
            <li>
             <p style="text-align:left;">
              欲了解更多“条件化实现块”的精彩内容，请请移步至我的另一篇主题文章《在 Rust 中令人印象深刻的三大【编译时】条件处理》
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              这里突出强调“
              <strong>
               本地
              </strong>
              智能指针”是因为
              <code>
               Rust
              </code>
              编译沙箱的
              <strong>
               孤儿原则
              </strong>
              导致“给任何
              <strong>
               非
              </strong>
              当前
              <code>
               crate
              </code>
              的数据结构实现标准库的
              <code>
               AsRef&lt;F&gt;
              </code>
              或
              <code>
               AsMut&lt;F&gt;
              </code>
              特征都会被编译器
              <strong>
               拒绝
              </strong>
              ”。
             </p>
            </li>
           </ul>
           <pre class="has"><code class="language-go">type SmartPointer = /* 前文代码定义的"智能指针"结构体类名 */;
impl&lt;F&gt; AsRef&lt;F&gt; for SmartPointer
where &lt;SmartPointer as Deref&gt;::Target: AsRef&lt;F&gt; {
    fn as_ref(&amp;self) -&gt; &amp;F {
        self.deref().as_ref()
    }
}
impl&lt;F&gt; AsMut&lt;F&gt; for SmartPointer
where &lt;SmartPointer as Deref&gt;::Target: AsMut&lt;F&gt; {
    fn as_mut(&amp;mut self) -&gt; &amp;mut F {
        self.deref_mut().as_mut()
    }
}</code></pre>
           <p style="text-align:left;">
            这段增补程序块所完成的任务可概括为：
           </p>
           <ol>
            <li>
             <p style="text-align:left;">
              若智能指针关联类型
              <code>
               Deref::Target
              </code>
              <strong>
               同时不满足
              </strong>
              <code>
               AsRef&lt;F&gt;
              </code>
              和
              <code>
               AsMut&lt;F&gt;
              </code>
              特征限定条件，那就什么也不做，也
              <strong>
               不添加新特征实现块
              </strong>
              。否则，继续。
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              若智能指针关联类型
              <code>
               Deref::Target
              </code>
              满足
              <code>
               AsRef&lt;F&gt;
              </code>
              特征限定条件，那就给智能指针数据结构增补
              <code>
               trait AsRef&lt;F&gt;
              </code>
              特征实现块，和实现特征成员方法
              <code>
               &lt;SmartPointer as AsRef&lt;F&gt;&gt;::as_ref(&amp;SmartPointer)
              </code>
              返回
              <code>
               &amp;F
              </code>
              。
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              若智能指针关联类型
              <code>
               Deref::Target
              </code>
              满足
              <code>
               AsMut&lt;F&gt;
              </code>
              特征限定条件，那就给智能指针数据结构增补
              <code>
               trait AsMut&lt;F&gt;
              </code>
              特征实现块，和实现特征成员方法
              <code>
               &lt;SmartPointer as AsMut&lt;F&gt;&gt;::as_mut(&amp;mut SmartPointer)
              </code>
              返回
              <code>
               &amp;mut F
              </code>
              。
             </p>
            </li>
           </ol>
           <p style="text-align:left;">
            此外，即便上例中的
            <code>
             SmartPointer
            </code>
            智能指针已实现过面向关联类型
            <code>
             &lt;SmartPointer as Deref&gt;::Target
            </code>
            的自定义引用特征
           </p>
           <ul>
            <li>
             <p style="text-align:left;">
              <code>
               trait AsRef&lt;&lt;SmartPointer as Deref&gt;::Target&gt;
              </code>
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              <code>
               trait AsMut&lt;&lt;SmartPointer as Deref&gt;::Target&gt;
              </code>
             </p>
            </li>
           </ul>
           <p style="text-align:left;">
            之一（或全部）也
            <strong>
             不会
            </strong>
            与此处新增补的
            <code>
             trait AsRef&lt;F&gt;
            </code>
            与
            <code>
             trait AsMut&lt;F&gt;
            </code>
            特征实现块
            <strong>
             冲突
            </strong>
            ，因为它们的
            <strong>
             泛型类型参数不一样
            </strong>
            。
           </p>
           <p style="text-align:left;">
            第二步，在调用智能指针实例上的
            <code>
             as_ref()
            </code>
            与
            <code>
             as_mut()
            </code>
            特征成员方法时，有一点儿麻烦，因为需要分辨两种情况：
           </p>
           <ol>
            <li>
             <p style="text-align:left;">
              智能指针数据结构上
              <strong>
               未定义
              </strong>
              过面向其它类型的
              <code>
               trait AsRef&lt;_&gt;
              </code>
              和
              <code>
               trait AsMut&lt;_&gt;
              </code>
              特征实现块，那么从智能指针实例直接点出新增补的
              <code>
               as_ref()
              </code>
              或
              <code>
               as_mut()
              </code>
              特征成员方法即可，不会有任何的歧义。
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              否则，
             </p>
            </li>
           </ol>
           <ul>
            <li>
             <p style="text-align:left;">
              要么，采用
              <code>
               TurboFish
              </code>
              语法，从成员方法调用表达式，标注泛型参数值和关联目标特征实现块。例如，
              <code>
               println!("{}", &lt;SmartPointer as AsMut&lt;i32&gt;&gt;::as_mut(&amp;mut smartPointer));
              </code>
              。
             </p>
            </li>
            <li>
             <p style="text-align:left;">
              要么，定义独立的赋值语句，从赋值变量的类型声明，标注泛型参数值和关联目标特征实现块。例如，
              <code>
               let mut value: i32 = smartPointer.as_mut();
              </code>
              。
             </p>
            </li>
           </ul>
           <h4>
            可复用的宏定义
           </h4>
           <p style="text-align:left;">
            因为上面那段增补代码几乎伴随着我本地定义的每个智能指针类，所以它还被特意零成本抽象为如下一段宏定义，以方便复用。因为这个功能太小，所以我还未将其发包于
            <code>
             crates.io
            </code>
            仓库。
           </p>
           <pre class="has"><code class="language-go">// 宏定义

            }

    #[derive(Debug)]

    impl AsRef&lt;i32&gt; for Wrapping {
        fn as_ref(&amp;self) -&gt; &amp;i32 {
            &amp;self.0
        }
    }
    impl AsMut&lt;i32&gt; for Wrapping {
        fn as_mut(&amp;mut self) -&gt; &amp;mut i32 {
            &amp;mut self.0
        }
    }
    impl Wrapping {
        pub fn new(value: i32) -&gt; Self {
            Wrapping(value)
        }
    }
}
// 定义本地智能指针类型
mod smart_pointer {

    use super::wrapping::Wrapping;
    #[derive(Debug)]
    pub struct SmartPointer {
        value: Wrapping
    }
    impl Deref for SmartPointer {
        type Target = Wrapping;
        fn deref(&amp;self) -&gt; &amp;Self::Target {
            &amp;self.value
        }

    impl DerefMut for SmartPointer {

        }
    }
    impl SmartPointer {

        }
    }
    // 下面是【标准库】对【智能指针】与`AsRef&lt;F&gt;`/`AsMut&lt;F&gt;`特征的惯例处理
