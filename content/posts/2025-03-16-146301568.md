    <p>
     Kubernetes（简称 K8s）是一个用于管理容器化应用的系统，它的架构主要分为两部分：
     <strong>
      控制平面（Control Plane）
     </strong>
     和
     <strong>
      工作节点（Worker Nodes）
     </strong>
     。它们协同工作，确保应用能够高效运行。
    </p>
    <h3>
     <a id="1_Control_Plane_19">
     </a>
     1. 控制平面（Control Plane）
    </h3>
    <p>
     控制平面是 Kubernetes 的大脑，负责管理和协调整个集群。它包含以下核心组件：
    </p>
    <h4>
     <a id="11_kubeapiserver_23">
     </a>
     1.1 kube-apiserver
    </h4>
    <ul>
     <li>
      <strong>
       作用
      </strong>
      ：作为集群的“总指挥”，接收所有的操作指令（比如部署应用、更新配置等），并验证这些指令是否合法。
     </li>
     <li>
      <strong>
       简单理解
      </strong>
      ：就像公司的前台，所有请求都必须经过它。
     </li>
    </ul>
    <h4>
     <a id="12_etcd_28">
     </a>
     1.2 etcd
    </h4>
    <ul>
     <li>
      <strong>
       作用
      </strong>
      ：存储集群的所有状态信息（比如 Pod 的位置、配置等），确保这些信息不会丢失。
     </li>
     <li>
      <strong>
       简单理解
      </strong>
      ：就像一个“记事本”，记录集群的每一件事。
     </li>
    </ul>
    <h4>
     <a id="13_kubescheduler_33">
     </a>
     1.3 kube-scheduler
    </h4>
    <ul>
     <li>
      <strong>
       作用
      </strong>
      ：决定每个应用（Pod）应该运行在哪个节点上，根据资源需求和规则进行分配。
     </li>
     <li>
      <strong>
       简单理解
      </strong>
      ：就像一个“分配员”，把任务分配到最适合的机器上。
     </li>
    </ul>
    <h4>
     <a id="14_kubecontrollermanager_38">
     </a>
     1.4 kube-controller-manager
    </h4>
    <ul>
     <li>
      <strong>
       作用
      </strong>
      ：运行各种控制器，确保集群的实际状态与期望状态一致（比如自动重启失败的 Pod）。
     </li>
     <li>
      <strong>
       简单理解
      </strong>
      ：就像一个“监督员”，确保一切按计划进行。
     </li>
    </ul>
    <h3>
     <a id="2_Worker_Nodes_43">
     </a>
     2. 工作节点（Worker Nodes）
    </h3>
    <p>
     工作节点是实际运行应用的机器。每个节点包含以下组件：
    </p>
    <h4>
     <a id="21_kubelet_47">
     </a>
     2.1 kubelet
    </h4>
    <ul>
     <li>
      <strong>
       作用
      </strong>
      ：管理节点上的 Pod，确保它们正常运行。
     </li>
     <li>
      <strong>
       简单理解
      </strong>
      ：就像一个“管家”，负责照顾每个应用的“家”。
     </li>
    </ul>
    <h4>
     <a id="22_kubeproxy_52">
     </a>
     2.2 kube-proxy
    </h4>
    <ul>
     <li>
      <strong>
       作用
      </strong>
      ：管理网络流量，确保应用之间可以互相通信。
     </li>
     <li>
      <strong>
       简单理解
      </strong>
      ：就像一个“交通警察”，指挥网络流量。
     </li>
    </ul>
    <h4>
     <a id="23__Docker__containerd_57">
     </a>
     2.3 容器运行时（如 Docker 或 containerd）
    </h4>
    <ul>
     <li>
      <strong>
       作用
      </strong>
      ：运行容器，是应用的实际运行环境。
     </li>
     <li>
      <strong>
       简单理解
      </strong>
      ：就像一个“容器”，装着你的应用。
     </li>
    </ul>
    <h3>
     <a id="3__62">
     </a>
     3. 架构优势
    </h3>
    <ul>
     <li>
      <strong>
       高可用性
      </strong>
      ：控制平面可以部署多个实例，避免单点故障。
     </li>
     <li>
      <strong>
       弹性扩展
      </strong>
      ：可以随时增加或减少节点和应用副本。
     </li>
     <li>
      <strong>
       自动化管理
      </strong>
      ：自动检测故障并恢复，减少人工干预。
     </li>
    </ul>
