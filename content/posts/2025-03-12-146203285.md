    <h2>
     <a id="_3">
     </a>
     进程的状态
    </h2>
    <p>
     <mark>
      在内核中，进程状态的表示，就是进程PCB中的一个int类型的成员变量，状态不同就存储着不同的数字
     </mark>
    </p>
    <hr/>
    <h3>
     <a id="R_8">
     </a>
     R状态：
    </h3>
    <p>
     就是运行状态
    </p>
    <hr/>
    <h3>
     <a id="S_14">
     </a>
     S状态：
    </h3>
    <p>
     <mark>
      就是
      <code>
       普通的阻塞状态
      </code>
      ，进程处于浅睡眠
     </mark>
     <br/>
     <strong>
      处于S状态的进程可以被操作系统杀死
     </strong>
    </p>
    <hr/>
    <h3>
     <a id="D_21">
     </a>
     D状态：
    </h3>
    <p>
     <mark>
      <code>
       特殊的阻塞状态
      </code>
      ，进程处于深度睡眠
     </mark>
     <br/>
     <strong>
      处于D状态的进程不可以被操作系统杀死
     </strong>
    </p>
    <p>
     <strong>
      为什么需要 D 状态？
     </strong>
     <br/>
     不可中断的 I/O 操作
     <br/>
     当进程执行某些必须完成的底层操作（如
     <code>
      磁盘写入
     </code>
     、
     <code>
      网络数据包接收
     </code>
     或
     <code>
      与硬件设备的交互
     </code>
     ）时，会进入 D 状态。这些操作通常由内核发起，进程必须等待其完成。
     <br/>
     <strong>
      不能被信号（如
      <code>
       kill -9
      </code>
      ）中断
     </strong>
     。若强行中断，可能导致数据损坏或硬件状态不一致。
    </p>
    <p>
     例
     <br/>
     进程A的任务是向磁盘中写入银行的100万条交易数据
     <br/>
     因为数据量较大，而且磁盘速度慢，所以会花费一些时间
     <br/>
     <mark>
      所以A会在磁盘的等待队列中等待磁盘写完，获取磁盘进行写入的结果（即写入成功还是失败）
     </mark>
    </p>
    <p>
     此时如果操作系统因为一些原因（
     <mark>
      内存不足或者用户使用指令杀进程A等
     </mark>
     ）
     <br/>
     把进程A杀死了
     <br/>
     如果刚好磁盘也写入失败了，就会告知进程A：
     <code>
      写入失败，询问是否重新写入
     </code>
     <br/>
     但是此时进A已经没了，自然就没人给磁盘应答，
     <mark>
      此时磁盘就可能会丢弃这些数据，就可能导致数据丢失
     </mark>
    </p>
    <p>
     所以为了保护重要数据，正在与硬件交互重要数据的进程就不能被杀死
     <br/>
     也就有了状态D
    </p>
    <hr/>
    <h3>
     <a id="T_51">
     </a>
     T状态
    </h3>
    <p>
     <mark>
      暂停状态
     </mark>
     <br/>
     进入暂停状态的两种情况：
     <br/>
     ①
     <code>
      进程做了非法但是不致命的操作
     </code>
     <br/>
     操作系统就会暂停进程，并把情况汇报给用户，让用户决定
     <br/>
     ②
     <code>
      用户操作，让进程进入暂停状态
     </code>
     （比如：双击屏幕暂停视频的播放）
    </p>
    <hr/>
    <h3>
     <a id="t_61">
     </a>
     t状态
    </h3>
    <p>
     <mark>
      可追踪的暂停状态
     </mark>
    </p>
    <p>
     例
     <br/>
     使用gdb调试器，调试可执行程序test，当在gdb中打了断点，再r了一下之后
     <br/>
     gdb就会启动test，生成进程
     <br/>
     <mark>
      此是gdc的调试进度就会停在打的断点处，此时由test生成的进程就处于t状态
     </mark>
     <br/>
     因为我们如果在gdb上按一下n或者s，test生成的进程就会被执行一下（因为要跑一下那句代码，看会怎么样）
     <br/>
     此时这个进程的状态会非常短暂地变成运行状态（因为只有运行能被CPU执行代码），然后又会变成t状态
     <br/>
     <mark>
      这就是可追踪的含义
     </mark>
    </p>
    <hr/>
    <h3>
     <a id="Z_75">
     </a>
     Z状态：
    </h3>
    <p>
     当进程任务完成后
     <br/>
     <mark>
      需要向父进程[
      <code>
       一般是父进程，也有可能是操作系统
      </code>
      ]汇报任务情况（是否正确完成/错误码）
     </mark>
     <br/>
     这个过程进程就处于Z状态
    </p>
    <p>
     <strong>
      进程创建的时候，是先创建它对应的内核数据结构[PCB]再加载它的代码和数据
     </strong>
     <br/>
     就像考进一个学校，再拿到录取通知书之前你的信息就已经被学校记录在学生管理系统里面了，所以在你人进入学校之前，就有了你对应的数据对象了
    </p>
    <p>
     <strong>
      而进程退出是则是反过来，即先释放代码和数据，再处理PCB
     </strong>
     <br/>
     因为进程退出后，就不会再执行代码了，所以代码和数据没有存在的必要了
     <br/>
     但是由于
     <code>
      要在PCB里面存储退出信息（描述任务执行状态）
     </code>
     ，给父进程/操作系统读取，所以PCB还不能直接释放
    </p>
    <hr/>
    <p>
     处于Z状态的进程不能再被杀死，因为它本质上已经死了
     <br/>
     <mark>
      所以只能让父进程去
      <code>
       回收它
      </code>
     </mark>
    </p>
    <p>
     如果父进程一直不去回收处于Z状态的子进程就会出现
     <code>
      僵尸问题
     </code>
     ：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/db32631e7367499e95f99165945285aa.png#pic_center"/>
    </p>
    <p>
     进程的退出状态必须被维持下去，因为他要告诉关心它的进程(父进程)，你交给我的任务，我办的怎么样了。
     <mark>
      父进程如果一直不读取
     </mark>
     ，
     <code>
      那子进程就一直处于Z状态
     </code>
    </p>
    <p>
     维护退出状态本身就是要用数据维护，也属于进程基本信息，所以保存在task. struct(PCB)中， 换句话说，
     <mark>
      Z状态一直不退出，内核就必须一直维护它的PCB
     </mark>
    </p>
    <p>
     <mark>
      那一个父进程创建了很多子进程，就是不回收,是不是就会造成内存资源的浪费
     </mark>
     即
     <code>
      内存泄漏
     </code>
     <br/>
     是的!
     <br/>
     因为数据结构对象本身就要占用内存，想想C语言中定义一个结构体变量(对象)，是要在内存的某个位置进行开辟空间!
    </p>
    <p>
     如何避免?后面会讲。
    </p>
    <hr/>
    <h3>
     <a id="_109">
     </a>
     孤儿进程
    </h3>
    <p>
     如果父进程创建子进程之后，子进程还在运行时，父进程就挂了
    </p>
    <p>
     此时因为必须要有人在子进程退出时，获取子进程的退出信息以及对子进程进行回收
    </p>
    <p>
     所有子进程必须再找一个爹，这个爹就是操作系统
    </p>
    <p>
     <mark>
      所以子进程运行时父进程挂了，子进程被操作系统领养的
     </mark>
     <br/>
     此时，这个子进程就叫
     <code>
      孤儿进程
     </code>
    </p>
    <hr/>
    <h3>
     <a id="X_122">
     </a>
     X状态：
    </h3>
    <p>
     进程完成Z状态的汇报任务，并且父进程回收完成之后
     <br/>
     <mark>
      进程真正死亡时，就处于X状态
     </mark>
    </p>
    <hr/>
    <h2>
     <a id="_132">
     </a>
     进程的优先级
    </h2>
    <p>
     什么是优先级，为什么要有优先级？
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/13863869b68149dcb27761e15d3af3d0.png#pic_center">
      <br/>
      需要进程竞争，但是所有进程一拥而上是肯定不行的
      <br/>
      <mark>
       这个时候就得排队，那么谁排在前面就是优先级的体现
      </mark>
     </img>
    </p>
    <hr/>
    <p>
     进程优先级的实现：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/24d5d3ba3c2548a1a6210fdadbb66ecd.jpeg#pic_center"/>
    </p>
    <hr/>
    <h3>
     <a id="_146">
     </a>
     如果我们要修改一个进程的优先级
    </h3>
    <p>
     一般修改的是它的nice（NI）值，通过这个间接修改进程的优先级
    </p>
    <p>
     为什么？
     <br/>
     因为如果没有nice值，直接修改PRI值
     <br/>
     如果该进程正在运行，就不好判断是否要继续执行
    </p>
    <p>
     <mark>
      但是修改nice值，就可以等运行暂时结束之后（时间片耗尽等），操作系统就会自动加上NI中存储的值，借此改变优先级
     </mark>
    </p>
    <hr/>
    <h3>
     <a id="_158">
     </a>
     重置进程优先级
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/c042b6f33b7948ac9d8b65c29dfced80.png#pic_center"/>
    </p>
    <p>
     <strong>
      为什么每次修改进程的优先级时，pri固定是80？
     </strong>
     <br/>
     因为这样可以让用户更方便地修改，就不需要在修改前查看进程之前的优先级了，直接就可以修改
    </p>
    <p>
     <strong>
      为什么用户可以修改的nice值拥有一个范围？
     </strong>
     <br/>
     这是因为我们使用的操作系统绝大部分都是分时操作系统
     <br/>
     <mark>
      分时操作系统追求的就是进程调度尽量公平
     </mark>
     <br/>
     如果一个用户把自己的一个进程的优先级调的特别高，那么其他用户的进程就有可能一直无法调度
     <br/>
     这显然并不公平
    </p>
    <hr/>
    <h2>
     <a id="_175">
     </a>
     进程切换
    </h2>
    <p>
     切换过程的理解
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/e9e64b7773e64371b58cd41aebe8390f.jpeg#pic_center"/>
    </p>
    <p>
     注意：
     <br/>
     <mark>
      上图中被红色半圈圈起来的临时数据，指的就是进程的
      <code>
       上下文数据
      </code>
     </mark>
     <br/>
     它们在这个进程运行到时候，存储在CPU里面的各种各样的寄存器中，并且随着进程运行不断地发生变化
    </p>
    <p>
     eip（pc）：是一个寄存器
     <br/>
     里面存储了
     <mark>
      正在运行的那句代码（指令）的下一句代码
     </mark>
     <br/>
     方便CPU获取
    </p>
    <p>
     ir：也是寄存器，
     <mark>
      存储的就是正在CPU中运行的代码（指令）
     </mark>
    </p>
    <p>
     <strong>
      所以CPU运行进程的简单过程就是：
     </strong>
     <br/>
     ①把进程接下来要运行的代码的地址，加载到eip寄存器中
     <br/>
     ②CPU通过eip中的地址找到汇编代码
     <br/>
     ③把找到的汇编代码放进ir寄存器中
     <br/>
     ④让eip存储下一条汇编代码的地址
     <br/>
     ⑤把ir中的汇编代码加载到控制器中运行
    </p>
    <p>
     <mark>
      第①步只在进程刚开始被CPU调度时执行，②③④⑤则是不断循环，借此实现进程运行
     </mark>
    </p>
    <p>
     进程切换的时候，
     <code>
      把CPU中相关寄存器中的上下文数据保存在了自己的PCB中
     </code>
     <br/>
     切换回来的时候，
     <mark>
      直接再把上下文数据加载到CPU中对应的寄存器中，就可以继续运行了
     </mark>
    </p>
    <hr/>
    <h2>
     <a id="_210">
     </a>
     进程的调度
    </h2>
    <p>
     运行队列（runqueue）中的调度队列（runqueue中的queue中的queue[140]）的结构：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/deb7bac246d44a22926e0cdbd171b8b3.png#pic_center"/>
    </p>
    <p>
     如上图所示，
     <mark>
      调度队列是一个哈希桶的结构
     </mark>
     ，
     <code>
      相同优先级的进程会被放进同一个桶中
     </code>
    </p>
    <p>
     CPU调用进程的时候从左往右进行调用，如果一个桶中有进程，就直接拿着桶里的第一个元素调度（插入进程也是头插）
     <br/>
     如果没有就继续向后遍历
     <br/>
     这样调用就可以体现出优先级了
    </p>
    <hr/>
    <p>
     而runqueue中有两个类型为queue的结构体变量
     <br/>
     如图：
     <br/>
     <img alt="请添加图片描述" src="https://i-blog.csdnimg.cn/direct/0d56386e66164f61b49d5fb64481d29b.jpeg"/>
    </p>
    <hr/>
    <p>
     这两个结构体变量中的
     <br/>
     task_struct* queue[140]，就是上面提到的存放进程的PCB的哈希桶
    </p>
    <p>
     这两个结构体在runqueue中被放在一个数组中
     <br/>
     并且定义了两个同类型的指针（active和expired）指向分别指向这两个结构体变量
    </p>
    <p>
     具体示意图如下：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/712d4c8a7b0e44c2bf36c8cf672337d1.png#pic_center"/>
    </p>
    <p>
     <mark>
      所以进程调度只需要从
      <code>
       active
      </code>
      指向的queue里面的哈希桶中从上到下遍历调用即可
     </mark>
    </p>
    <hr/>
    <p>
     调度过程中，会出现如下3种情况
     <br/>
     ①进程的代码被全部执行完，
     <code>
      进程退出
     </code>
     <br/>
     <mark>
      那么这个进程的PCB当然也就不会再回到调度队列中
     </mark>
    </p>
    <p>
     ②进程的时间片耗尽
    </p>
    <p>
     ③新来了进程
    </p>
    <p>
     ②和③情况的进程肯定是要进入调度队列的，
     <mark>
      但是它们都只会进入
      <code>
       expired
      </code>
      指向queue里面的哈希桶中
     </mark>
    </p>
    <p>
     <strong>
      为什呢？
     </strong>
     <br/>
     如果②和③情况的进程再进入active指向的queue里面的哈希桶中
     <br/>
     此时如果②和③情况的进程的优先级很高，而且这样的进程很多
     <br/>
     那它们就会一直插在哈希桶的最前面
     <br/>
     <mark>
      那CPU每次调用的时候就都只会调用到它们
     </mark>
     <br/>
     <mark>
      比它们优先级更低的进程就根本没有机会被调度了
     </mark>
    </p>
    <p>
     这不符合分时操作系统的调度原则：
     <code>
      进程调度尽可能公平
     </code>
    </p>
    <p>
     <mark>
      所以新进程和时间片耗尽的进程就根据自己的优先级，插入
      <code>
       expired
      </code>
      指向queue里面的哈希桶中
     </mark>
    </p>
    <hr/>
    <p>
     所以
     <br/>
     <mark>
      <code>
       active
      </code>
      指向的queue里面的进程，因为没有补充，所以会一直减少，直到减到0
     </mark>
    </p>
    <p>
     此时只需要
     <code>
      交换
     </code>
     active和expired的指向的队列，
     <strong>
      之前队列里面积压的进程就给了active
     </strong>
     <br/>
     就可以实现
     <code>
      调度轮转[
     </code>
     <mark>
      因为CPU只从active指向的queue里面的哈希桶中选进程
     </mark>
     ]
    </p>
    <p>
     这样调度就既可以保证公平，也体现了进程的优先级
