    <h3>
     <a id="_0">
     </a>
     合并冲突
    </h3>
    <p>
     可是，在实际分支合并的时候，并不是想合并就能合并成功的，有时候可能会遇到代码冲突的问题。
    </p>
    <p>
     为了演示这问题，创建一个新的分支
     <code>
      dev1
     </code>
     ，并切换至目标分支，我们可以使用
     <code>
      git checkout -b dev1
     </code>
     一步完成创建并切换的动作，示例如下：
    </p>
    <pre><code>$ git checkout -b dev1
Switched to a new branch 'dev1'

$ git branch
* dev1
  master
</code></pre>
    <p>
     在
     <code>
      dev1
     </code>
     分支下修改
     <code>
      ReadMe
     </code>
     文件，更改文件内容如下，并进行一次提交，如：
    </p>
    <pre><code>$ cat ReadMe
hello bit
hello git
hello world
hello version1
hello version2
hello version3
write bbb for new branch # 将aaa该为bbb


$ git add.
$ git commit -m"modify ReadMe"
[dev1 0854245] modify ReadMe
 1 file changed, 1 insertion(+), 1 deletion(-)
</code></pre>
    <p>
     切换至
     <code>
      master
     </code>
     分支，观察
     <code>
      ReadMe
     </code>
     文件内容：
    </p>
    <pre><code>$ git checkout master
Switched to branch'master'

$ cat ReadMe
hello bit
hello git
hello world
hello version1
hello version2
hello version3
write aaa for new branch
</code></pre>
    <p>
     我们发现，切回来之后，文件内容变成了老的版本，这种现象很正常，我们现在也完全能理解。
     <br/>
     此时在
     <code>
      master
     </code>
     分支上，我们对
     <code>
      ReadMe
     </code>
     文件再进行一次修改，并进行提交，如下：
    </p>
    <pre><code>$ git branch
dev1
* master

$ vim ReadMe
$ cat ReadMe
hello bit
hello git
hello world
hello version1
hello version2
hello version3
write ccc for new branch

$ git add.
$ git commit -m"modify ReadMe"
[master 1c10f6d] modify ReadMe
 1 file changed, 1 insertion(+), 1 deletion(-)
</code></pre>
    <p>
     现在，
     <code>
      master
     </code>
     分支和
     <code>
      dev1
     </code>
     分支各自都分别有新的提交，变成了这样：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/52f4d3c4a1c9482d969c2ea07751deb6.jpeg"/>
    </p>
    <p>
     这种情况下，Git 只能试图把各自的修改合并起来，但这种合并就可能会有冲突，如下所示：
    </p>
    <pre><code>$ git merge dev1
Auto - merging ReadMe
CONFLICT (content): Merge conflict in ReadMe
Automatic merge failed; fix conflicts and then commit the result.

$ git status
On branch master
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)
Unmerged paths:
  (use "git add &lt;file&gt;..." to mark resolution)
        both modified:   ReadMe
no changes added to commit (use "git add" and/or "git commit -a")
</code></pre>
    <p>
     发现
     <code>
      ReadMe
     </code>
     文件有冲突后，可以直接查看文件内容，要说的是 Git 会用
     <code>
      &lt;&lt;&lt;&lt;&lt;&lt;&lt;
     </code>
     ，
     <code>
      =======
     </code>
     ，
     <code>
      &gt;&gt;&gt;&gt;&gt;&gt;&gt;
     </code>
     来标记出不同分支的冲突内容，如下所示：
    </p>
    <pre><code>$ cat ReadMe
hello bit
hello git
hello world
hello version1
hello version2
hello version3
&lt;&lt;&lt;&lt;&lt;&lt;&lt; HEAD
write ccc for new branch
=======
write bbb for new branch
&gt;&gt;&gt;&gt;&gt;&gt;&gt; dev1
</code></pre>
    <p>
     此时我们
     <strong>
      必须要手动调整冲突代码
     </strong>
     ，并需要再次提交修正后的结果！！（再次提交很重要，切勿忘记）
    </p>
    <pre><code>$ cat ReadMe
hello bit
hello git
hello world
hello version1
hello version2
hello version3
write bbb for new branch

$ git add.
$ git commit -m"merge ReadMe"
[master 2976afc] merge ReadMe
</code></pre>
    <p>
     到这里冲突就解决完成，此时的状态变成了
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/f24692031ff54f9db58a3db9004807ed.jpeg"/>
    </p>
    <p>
     用带参数的
     <code>
      git log
     </code>
     也可以看到分支的合并情况，具体大家可以自行搜索
     <code>
      git log
     </code>
     的用法:
    </p>
    <pre><code>$ git log --graph --pretty=oneline --abbrev-commit
* 2976afc (HEAD -&gt; master) merge ReadMe
|\
| * c59f4d1 (dev1) modify ReadMe
* | c10f6d0 modify ReadMe
|/
</code></pre>
    <p>
     最后，不要忘记
     <code>
      dev1
     </code>
     分支使用完毕后就可以删除了：
    </p>
    <pre><code>$ git branch
* master

$ git branch -d dev1
Deleted branch dev1 (was c59f4d1).
</code></pre>
    <h3>
     <a id="_140">
     </a>
     分支管理策略
    </h3>
    <p>
     通常合并分支时，如果可能，Git 会采用
     <code>
      Fast forward
     </code>
     模式。还记得如果我们采用
     <code>
      Fast forward
     </code>
     模式之后，形成的合并结果是什么样呢？回顾一下
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/dbb1970217104e8b9916b5dcb985d8ef.jpeg"/>
    </p>
    <p>
     在这种
     <code>
      Fast forward
     </code>
     模式下，删除分支后，查看分支历史时，会丢掉分支信息，看不出来最新提交到底是从哪个分支过来的。
    </p>
    <p>
     但在合并时如果发生了冲突，我们还是正常提交解决冲突问题，会再进行一次新的提交，得到的最终状态为：
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/1c01c79336a341b798e99e368d4d8d18.jpeg"/>
    </p>
    <p>
     那么就不是
     <code>
      Fast forward
     </code>
     模式了，这样的好处是，从分支历史上就可以看出分支信息。
     <br/>
     例如我们现在已经删除了在合并冲突部分创建的
     <code>
      dev1
     </code>
     分支，但依旧能看到
     <code>
      master
     </code>
     其实是由其他分支合并得来的。
    </p>
    <pre><code>$ git log --graph --pretty=oneline --abbrev-commit
* 2976afc (HEAD -&gt; master) merge ReadMe
|\
| * c59f4d1 modify ReadMe
* | c10f6d0 modify ReadMe
|/
</code></pre>
    <p>
     Git 支持我们强制禁用
     <code>
      Fast forward
     </code>
     模式，那么就会在
     <code>
      merge
     </code>
     时生成一个新的
     <code>
      commit
     </code>
     ，这样，从分支历史上就可以看出分支信息。
     <br/>
     下面我们实战一下
     <code>
      --no-ff
     </code>
     方式的
     <code>
      git merge
     </code>
     。首先，创建新的分支
     <code>
      dev2
     </code>
     ，并切换至新的分支：
    </p>
    <pre><code>$ git checkout -b dev2
Switched to a new branch 'dev2'
</code></pre>
    <p>
     修改
     <code>
      ReadMe
     </code>
     文件，并提交一个新的
     <code>
      commit
     </code>
     ：
    </p>
    <pre><code>$ cat ReadMe
hello bit
hello world
hello git
hello version1
hello version2
hello version3
write bbb for new branch

$ git add.
$ git commit -m"modify ReadMe"
[dev2 41b82f7] modify ReadMe
 1 file changed, 1 insertion(+)
</code></pre>
    <p>
     切回
     <code>
      master
     </code>
     分支，开始合并：
    </p>
    <pre><code>$ git checkout master
Switched to branch'master'

$ git merge --no-ff -m "merge with --no-ff" dev2
Merge made by the 'recursive' strategy.
ReadMe | 1 +
1 file changed, 1 insertion(+)

$ cat ReadMe
hello bit
hello world
hello git
hello version1
hello version2
hello version3
write bbb for new branch
a,b,c,d
</code></pre>
    <p>
     请注意
     <code>
      --no-ff
     </code>
     参数，表示禁用
     <code>
      Fast forward
     </code>
     模式。禁用
     <code>
      Fast forward
     </code>
     模式后合并会创建一个新的
     <code>
      commit
     </code>
     ，所以加上
     <code>
      -m
     </code>
     参数，把
     <code>
      merge
     </code>
     信息写进去。
    </p>
    <p>
     合并后，查看分支历史：
    </p>
    <pre><code>$ git log --graph --pretty=oneline --abbrev-commit
*   5bde1b4 (HEAD -&gt; master) merge with --no-ff
|\
| * 41b82f7 (dev2) modify ReadMe
* | c10f6d0 modify ReadMe
|/
</code></pre>
    <p>
     可以看到，不使用
     <code>
      Fast forward
     </code>
     模式，
     <code>
      merge
     </code>
     后像这样：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a592f87d8bac40e3a61b2d985bd333b3.jpeg"/>
    </p>
    <p>
     所以在合并分支时，加上
     <code>
      --no-ff
     </code>
     参数就可以保留分支信息，合并后的历史有分支，能看出来曾经做过合并，而
     <code>
      fast forward
     </code>
     合并就看不出来曾经做过合并。
    </p>
