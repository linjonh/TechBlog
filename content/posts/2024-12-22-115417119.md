    <h3>
     <a id="_7">
     </a>
     一、关系数据结构
    </h3>
    <p>
     <strong>
      1、关系的定义和性质
     </strong>
     <br/>
     <strong>
      （1）、 关系的数学定义：
     </strong>
    </p>
    <p>
     <strong>
      域
     </strong>
     ： 一组有相同数据类型的值得集合
     <br/>
     <strong>
      笛卡尔积
     </strong>
     ： 设任意的N个域D1,D2,…,Dn。定义D1,D2,…,Dn的笛卡尔积为：D1*D2*****Dn={(d1,d2,…,dn)|di属于Di,i=1,2,…,n}。例如（D1的取值类型都为动物，D2的取值类型都为菜食物）：D1={猫，狗，猪}，D2={鱼，骨头，白菜}，则D1乘D2={（猫，鱼），（猫，骨头），（猫，白菜），（狗，鱼），（狗，骨头），（狗，白菜），（猪，鱼），（猪，骨头），（猪，白菜）}
    </p>
    <p>
     <strong>
      关系
     </strong>
     ： D1******Dn中有关系的行形成的一个子集称为D1******Dn的一个关系，用R（D1,D2…Dn）表示，R表示关系名，n表示关系的元或目
    </p>
    <p>
     <strong>
      （2）、关系的性质：
     </strong>
    </p>
    <p>
     同一列的数据类型必须一致
     <br/>
     同一关系中，不同列可以有相同的数据类型，但列名必须不一样
     <br/>
     同一关系中，任意两行不能重复，且元组（行）的排行不分先后顺序
     <br/>
     同一关系中，列的排列部分先后顺序
     <br/>
     关系中属性不可再分，满足原子性
    </p>
    <p>
     <strong>
      2、关系数据库的基本概念：
      <br/>
      （1）关系模式
     </strong>
     <br/>
     在惯性型数据库中，关系模式是型，关系是值，
     <strong>
      关系模式是对关系的描述
     </strong>
     <br/>
     关系模式的表示：R（U,D,dom,F），R表示关系名；U是组成该关系的属性名集合；D是属性组U中属性来自的域；dom是属性向域的映像关系；F是属性间的数据依赖关系集合
     <br/>
     简记：R（U）或者R（A1,A2,A3…An），例如：
     <strong>
      学生情况（姓名，班级，学号，身份证号等）
     </strong>
     <br/>
     <strong>
      （2）关系数据库
     </strong>
     采用关系模式的数据库
     <br/>
     <strong>
      3、关系的完整性
     </strong>
     ： 保证行不重复（唯一）
     <br/>
     <strong>
      （1）实体完整性约束
     </strong>
    </p>
    <ul>
     <li>
      主键约束（primary key）：非空且唯一
     </li>
     <li>
      唯一键约束（unique）：唯一
     </li>
    </ul>
    <p>
     <strong>
      （2）引用完整性约束：
     </strong>
    </p>
    <ul>
     <li>
      存在两个关系（表）R和S，F是一个属性或者属性集合；F属于R但不是R的主键，同时F属于S并且F是S的主键，则称F是基本关系R的外码，R为外键表，S为主键表
     </li>
     <li>
      外键约束（foreign key）：表与表之间建立联系
     </li>
    </ul>
    <p>
     <strong>
      （3）域完整性约束
     </strong>
     ： 用户定义的完整性规则
     <br/>
     由用户根据实际情况对数据库中的数据内容进行规定，对列的约束
     <br/>
     检查约束（check）：添加范围
     <br/>
     数据类型约束（int、char等）
     <br/>
     默认约束（default）
     <br/>
     非空约束（not null）
     <br/>
     <strong>
      3、关系运算：

     <strong>
      象集
     </strong>
     ： 给定关系R（A,B），A,B为属性组。表示A属性上a在B上的分量就是象集
     <br/>
     例如：
     <br/>
     | A|B |
     <br/>
     | a1|b1 |
     <br/>
     | a2|b2|
     <br/>
     | a2|b3 |
     <br/>
     a1对应的象集为（b1）
     <br/>
     a2对应的象集为：（b2,b3）
     <br/>
     步骤：R/S
     <br/>
     stop1：找到两个关系中属性相同的列
     <br/>
     step2：R/S就是相同的列中属于R但不属于S的列
     <br/>
     step3：找到R中列的元素所对应的象集，和S在和R相同列的投影
     <br/>
     step4：象集包含S中得到的投影的元素就是除法的结果

    <h3>
     <a id="_98">
     </a>
     四、关系的规范化
    </h3>
    <p>
     1、
     <strong>
      数据依赖
     </strong>
     ： 函数依赖 、多值依赖、连接依赖
     <br/>
     （1）、
     <strong>
      函数依赖
     </strong>
     ： 关系模式中各个属性之间的一种依赖关系
    </p>
    <p>
     <strong>
      函数依赖定义
     </strong>
     ： 简单点说就是，关系R（X,Y），如果指导属性X的值就能得到属性Y的值，就能说Y函数依赖于X
     <br/>
     （2）、
     <strong>
      几种特定的函数依赖：
     </strong>
    </p>
    <ul>
     <li>
      <p>
       <strong>
        平凡和非平方函数依赖
       </strong>
       ： 关系模式R（X,Y），如果 X—&gt;Y，Y是X的子集，则称为平凡函数依赖；如果X—&gt;Y，Y不是X的子集，则称为非平凡函数依赖。例如：
       <strong>
        学号，课程号—&gt;成绩为非平凡函数依赖
       </strong>
      </p>
     </li>
     <li>
      <p>
       <strong>
        完全函数依赖和部分函数依赖
       </strong>
       ： 关系模式R（X,Y）,如果X—&gt;Y，并且X的任意一个真子集Z都不满足Z—&gt;Y，则Y完全函数依赖 X；如果X—&gt;Y，但对于X的某一个真子集Z，有Z—&gt;Y满足，则称Y部分依赖于X。例如：
       <strong>
        成绩表（学号，课程号，成绩），学号，课程号为主键，学号和课程号共同决定成绩，但是学号和课程号分开不能决定成绩，所以成绩完全函数 依赖学号和课程号
       </strong>
      </p>
     </li>
     <li>
      <p>
       <strong>
        传递函数依赖
       </strong>
       ： 关系模式R（X,Y），X—&gt;Y,Y—&gt;X（不成立），有Y—&gt;Z,则称X—&gt;Z称为传递函数依赖。例如：
       <strong>
        学生关系（学号，所在系，系主任），学号——&gt;所在系，所在系——&gt;系主任，学号——&gt;系主任
       </strong>
      </p>
     </li>
    </ul>
    <p>
     <strong>
      （3）、码的函数依赖表示：
     </strong>
    </p>
    <ul>
     <li>
      候选码： 设K为关系模式R（U）中的一个属性或属性集合，若K——&gt;U，则K称为R的一个候选码
     </li>
     <li>
      主码： 选定R中一个候选码作为主码
     </li>
     <li>
     </li>
    </ul>
    <p>
     <strong>
      2、关系规范化的目的：
     </strong>
    </p>
    <p>
     <strong>
      数据冗余
     </strong>
     ：数据库中重复存放数据的情况。造成浪费空间，修改数据时会发生数据不一致情况
     <br/>
     <strong>
      更新异常
     </strong>
     ：存在数据冗余，更新数据时代价会增高
     <br/>
     <strong>
      插入数据
     </strong>
     ：信息插不进去
     <br/>
     <strong>
      删除异常
     </strong>
     ：不该删除的数据不得不删除
    </p>
    <p>
     <strong>
      3、关系规范化的过程：
     </strong>
    </p>
    <p>
     <strong>
      （1）、第一范式： 目标是确保每列的原子性（即不可再分的最小数据单元）
     </strong>
    </p>
    <p>
     关系模式的属性不可再分解
     <br/>
     缺点：不能排除数据冗余和异常情况
     <br/>
     <strong>
      （2）、第二范式： 要求每个表只描述一个实体的属性
     </strong>
    </p>
    <p>
     主属性： 包含在任何候选码中的属性
     <br/>
     非主属性： 不包含在任何候选码中的属性
     <br/>
     定义：满足第一范式的条件下，且它的每一个 非主属性 都完全函数依赖于码（候选码）
     <br/>
     例如：学生关系（学号，姓名，性别，课程号，学分），其中学号和课程号的组合为主码，姓名、性别、学分为非主属性
     <br/>
     <strong>
      （3）、第三范式： 要求实体各属性之间不能具有直接的依赖关系
     </strong>
    </p>
    <p>
     满足第二范式，且没有一个非主属性传递函数依赖于码
     <br/>
     <strong>
      （4）、BC范式：
     </strong>
    </p>
    <p>
     定义：对于任何非平凡函数依赖X——&gt;Y，X均包含码
     <br/>
     所有非主属性完全函数依赖于每一个候选码
     <br/>
     所有主属性完全函数依赖于每个不包含它的候选码
     <br/>
     没有任何属性完全函数依赖于非码的任何一组属性
    </p>
