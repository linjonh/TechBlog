         <span style="FONT-SIZE: 12pt; COLOR: black; FONT-FAMILY: 宋体; mso-font-kerning: 0pt; mso-bidi-font-family: 宋体">
          而众多的软件系统中，银行业务软件以其高复杂性、高安全性、高准确性、高效率性给软件测试带来了一系列难度，尤其是在测试评价方面，仍然停留在比较初级的阶段，常见的方法有缺陷走势图、缺陷严重程度分布图等等。这些方法可以很清楚看出软件缺陷的走向和分布情况，对于测试评价有一定帮助。但是这些方法存在指标量化不够，评价指标过于单一的缺点，重复操作性低，在实际使用中对进行测试结果评价的人员要求很高等问题，从而影响评价的效果。

         <span style="FONT-SIZE: 12pt; COLOR: black; FONT-FAMILY: 宋体; mso-font-kerning: 0pt; mso-bidi-font-family: 宋体">
          一、评价指标体系建立
          <span lang="EN-US">
          </span>
         </span>
        </span>

      </span>
     </span>
    </p>
    <p align="left" class="MsoNormal" style="MARGIN: 0cm 0cm 0pt; LAYOUT-GRID-MODE: char; WORD-BREAK: break-all; TEXT-INDENT: 24pt; LINE-HEIGHT: 14.3pt; TEXT-ALIGN: left; mso-line-height-rule: exactly; mso-char-indent-count: 2.0; mso-pagination: widow-orphan">
     <span style="mso-bookmark: _Toc51083911">
      <span style="mso-bookmark: _Toc62016463">
       <span style="mso-bookmark: _Toc70439795">
        <span style="mso-bookmark: _Toc86483235">
         <span style="FONT-SIZE: 12pt; COLOR: black; FONT-FAMILY: 宋体; mso-font-kerning: 0pt; mso-bidi-font-family: 宋体">
          要对软件测试结果进行评价，软件的行为特性，如可靠性、正确性、健壮性、可扩展性和性能等，是最基本的指标。但由于银行业务软件的行为特性比较抽象，难以量化，不能直接作为评价指标，必须对之建立量化的子指标。在实际测试中最容易量化的就是测试用例数、程序代码行数、问题修改时间、程序响应时间和测试问题数以及数据量、交易量。
          <span lang="EN-US">
          </span>
         </span>
        </span>
       </span>
      </span>
     </span>

    <p align="left" class="MsoNormal" style="MARGIN: 0cm 0cm 0pt; LAYOUT-GRID-MODE: char; WORD-BREAK: break-all; TEXT-INDENT: 24pt; LINE-HEIGHT: 14.3pt; TEXT-ALIGN: left; mso-line-height-rule: exactly; mso-char-indent-count: 2.0; mso-pagination: widow-orphan">
     <span style="mso-bookmark: _Toc51083911">
      <span style="mso-bookmark: _Toc62016463">
       <span style="mso-bookmark: _Toc70439795">
        <span style="mso-bookmark: _Toc86483235">
         <span style="FONT-SIZE: 12pt; COLOR: black; FONT-FAMILY: 宋体; mso-font-kerning: 0pt; mso-bidi-font-family: 宋体">
          程序响应时间和数据量、交易量可以对程序性能作出评价，而问题数量及问题的修改时间则可以对程序的可靠性、健壮性、可扩展性、正确性进行很好的评价。不同严重程度的问题对测试结果的影响是不一样的，比如一般问题可能对程序正确性影响不大，而紧急问题却可能导致程序无法运行从而对程序正确性造成严重影响。所以对于程序正确性的评价必须区分一般问题、重要问题和紧急问题。
          <span lang="EN-US">
          </span>
         </span>
        </span>
       </span>
      </span>
     </span>
    </p>

     <span style="mso-bookmark: _Toc51083911">
      <span style="mso-bookmark: _Toc62016463">

        <span style="mso-bookmark: _Toc86483235">
         <span style="FONT-SIZE: 12pt; COLOR: black; FONT-FAMILY: 宋体; mso-font-kerning: 0pt; mso-bidi-font-family: 宋体">
          每一种数量还要区分绝对数量和相对数量。这是因为在一个
          <span lang="EN-US">
           100
          </span>
          行的程序和一个
          <span lang="EN-US">
           10000
          </span>
          行的程序之间直接比较问题个数是没有意义的。相对数量是指某类问题数占问题总数的百分比。还有估计相对数量是指某类问题数占估计问题总数的百分比。至于使用绝对数量还是使用相对数量作为评价指标，可视具体情况而定，或者综合考虑两方面指标。
          <span lang="EN-US">
          </span>
