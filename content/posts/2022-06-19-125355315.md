    <h2 id="1.%20%E8%99%9A%E6%8B%9F%E5%86%85%E5%AD%98%E6%A6%82%E8%BF%B0">
     1. 虚拟内存概述
    </h2>
    <blockquote>
     <p>
      传统存储管理方式同时将多个进程保存在内存中以便允许多道程序设计。它们都具有以下两个共同的特征：
     </p>
     <ul>
      <li>
       <strong>
        <span style="color:#0d0016;">
         一次性
        </span>
        ：
       </strong>
       作业必须一次性
       <strong>
        全部装入内存
       </strong>
       后，方能开始运行。这会导致两种情况发生：
       <br/>
       1）当作业很大，不能全部被装入内存时，将使该作业无法运行；
       <br/>
       2）当大量作业要求运行时，由于内存不足以容纳所有作业，只能使少数作业先运行，导致多道程序度的下降。
      </li>
      <li>
       <strong>
        驻留性：
       </strong>
       作业被装入内存后，就一直
       <strong>
        驻留在内存
       </strong>
       中，其任何部分都不会被换出，
       <strong>
        直至作业运行结束
       </strong>
       。运行中的进程，会因等待 I/O 而被阻塞，可能处于长期等待状态。
      </li>
     </ul>
     <p>
      因此，许多在程序运行中不用或暂时不用的程序（数据）占据了大量的内存空间，而一些需要运行的作业又无法装入运行，显然浪费了宝贵的内存资源。
     </p>
    </blockquote>
    <h3 id="%C2%A01.1%20%E8%99%9A%E6%8B%9F%E5%AD%98%E5%82%A8%E5%99%A8%E7%9A%84%E5%AE%9A%E4%B9%89%E5%92%8C%E7%89%B9%E5%BE%81">
     1.1 虚拟存储器的定义和特征
    </h3>
    <p>
     基于
     <strong>
      局部性
     </strong>
     原理，在程序装入时，可以将程序的
     <strong>
      一部分装入内存
     </strong>
     ，而将其余部分留在外存，就可以启动程序执行。在程序执行过程中，当所访问的信息不在内存时，由操作系统
     <strong>
      将所需要的部分调入内存
     </strong>
     ，然后继续执行程序。另一方面，操作系统将内存中
     <strong>
      暂时不使用的内容换出到外存上
     </strong>
     ，从而腾出空间存放将要调入内存的信息。
    </p>
    <p>
     这样，由于系统提供了
     <span style="color:#fe2c24;">
      <strong>
       部分装入、请求调入和置换功能
      </strong>
     </span>
     后（对用户完全透明），给用户的感觉是好像存在一个比实际物理内存大得多的存储器，称为
     <strong>
      虚拟存储器
     </strong>
     。
    </p>
    <p>
     虚拟存储器的
     <strong>
      大小由计算机的地址结构决定
     </strong>
     ，并非是内存和外存的简单相加。
    </p>
    <p>
     <strong>
      虚拟存储器有以下三个主要特征：
     </strong>
    </p>
    <ul>
     <li>
      <strong>
       多次性
      </strong>
      ：无需在作业运行时一次性地全部装入内存，而是允许被分成多次调入内存运行。
     </li>
     <li>
      <strong>
       对换性
      </strong>
      ：无需在作业运行时一直常驻内存，而是允许在作业的运行过程中，进行换进和换出。
     </li>
     <li>
      <strong>
       虚拟性
      </strong>
      ：从逻辑上扩充内存的容量，使用户所看到的内存容量，远大于实际的内存容量。
     </li>
    </ul>
    <h3 id="%C2%A01.2%20%E8%99%9A%E6%8B%9F%E5%86%85%E5%AD%98%E6%8A%80%E6%9C%AF%E7%9A%84%E5%AE%9E%E7%8E%B0">
     1.2 虚拟内存技术的实现
    </h3>
    <p>
     虚拟内存中，允许将
     <strong>
      一个作业分多次调入内存
     </strong>
     。
    </p>
    <p>
     釆用连续分配方式时，会使相当一部分内存空间都处于暂时或“永久”的空闲状态，造成内存资源的严重浪费，而且也无法从逻辑上扩大内存容量。
    </p>
    <p>
     因此，虚拟内存的实现需要建立在
     <strong>
      <span style="color:#fe2c24;">
       离散分配
      </span>
      的内存管理方式
     </strong>
     的基础上。虚拟内存的实现有以下三种方式：
    </p>
    <ul>
     <li>
      <strong>
       请求分页
      </strong>
      存储管理
     </li>
     <li>
      <strong>
       请求分段
      </strong>
      存储管理
     </li>
     <li>
      <strong>
       段页式
      </strong>
      存储管理
     </li>
    </ul>
    <p>
     不管哪种方式，都需要有一定的
     <strong>
      硬件支持
     </strong>
     。一般需要的支持有以下几个方面：
    </p>
    <ul>
     <li>
      一定容量的内存和外存。
     </li>
     <li>
      页表机制（或段表机制），作为主要的数据结构。
     </li>
     <li>
      中断机构，当用户程序要访问的部分尚未调入内存，则产生中断。
     </li>
     <li>
      地址变换机构，逻辑地址到物理地址的变换。
     </li>
    </ul>
    <blockquote>
     <p>
      <strong>
       <span style="background-color:#a2e043;">
        连续分配方式
       </span>
       ：
      </strong>
      指为一个用户程序分配一个
      <strong>
       连续的内存空间
      </strong>
      。
     </p>
     <ul>
      <li>
       <strong>
        固定分区分配：
       </strong>
       将内存空间划分为若干个固定大小的区域，在每个分区中只装入一道作业，便可以有多道作业并发执行。缺乏灵活性，会产生大量的
       <strong>
        内部碎片
       </strong>
       ，
       <strong>
        内存的利用率很低
       </strong>
       。
      </li>
      <li>
       <strong>
        动态分区分配：
       </strong>
       根据进程的实际需要，动态地为之分配内存空间。作业装入内存时，把可用内存分出一个连续区域给作业，且分区的大小正好合适作业大小的需要。会产生很多
       <strong>
        外部碎片。
       </strong>
      </li>
     </ul>
     <p>
     </p>
     <p class="img-center">
      <img alt="" height="284" src="https://i-blog.csdnimg.cn/blog_migrate/1f2731da59baf86b6abad9833fa7565e.png" width="331"/>
     </p>
     <p>
     </p>
     <p>
      <strong>
       <span style="background-color:#a2e043;">
        离散分配方式
       </span>
       ：
      </strong>
      将一个进程
      <strong>
       分散地装入
      </strong>
      到许多
      <strong>
       不相邻的分区中
      </strong>
      ，便可充分地利用内存。
     </p>
    </blockquote>
    <blockquote>
     <p>
      分页存储的概念：
     </p>
     <ul>
      <li>
       页面、页框和块。
       <strong>
        进程
       </strong>
       中的块称为
       <strong>
        页或页面(Page)
       </strong>
       ，具有页号；
       <span style="color:#1a439c;">
        <strong>
         内存
        </strong>
       </span>
       中的块称为
       <span style="color:#1a439c;">
        <strong>
         页框
        </strong>
        （Page Frame，页框=内存块=物理块=物理页面
       </span>
       <span style="color:#1a439c;">
        ）
       </span>
       ，具有页框号。
       <span style="color:#98c091;">
        <strong>
         外存
        </strong>
       </span>
       也以同样的单位进行划分，直接称为
       <span style="color:#98c091;">
        <strong>
         块(Block)
        </strong>
       </span>
       。进程在执行时需要申请主存空间，就是要为每个页面分配主存中的可用页框，这就产生了页和页框的一一对应。各个页面不必连续存放，可以放到不相邻的各个页框中。
      </li>
      <li>
       <strong>
        地址结构
       </strong>
       ：前一部分为
       <strong>
        页号P
       </strong>
       ，后一部分为
       <strong>
        页内偏移量W
       </strong>
       。地址长度为32 位，其中0~11位为页内地址，即每页大小为4KB；12~31位为页号，地址空间最多允许有2^20页。
      </li>
      <li>
       <strong>
        页表
       </strong>
       。为了便于在内存中
       <strong>
        找到进程的每个页面所对应的物理块
       </strong>
       ，系统为每个进程建立一张页表，记录页面在内存中对应的物理块号，
       <strong>
        页表一般存放在内存中
       </strong>
       。在配置了页表后，进程执行时，通过查找该表，即可找到每页在内存中的物理块号。可见，页表的作用是
       <strong>
        实现从页号到物理块号的地址映射
       </strong>
       。
      </li>
     </ul>
     <p class="img-center">
      <img alt="" src="https://i-blog.csdnimg.cn/blog_migrate/a2a7b698a8d7b9e4620681c3682b17ee.png"/>
     </p>
    </blockquote>
    <h2 id="%C2%A02.%20%E8%AF%B7%E6%B1%82%E5%88%86%E9%A1%B5%E7%AE%A1%E7%90%86%E6%96%B9%E5%BC%8F%E5%AE%9E%E7%8E%B0%E8%99%9A%E6%8B%9F%E5%86%85%E5%AD%98">
     2. 请求分页管理方式实现虚拟内存
    </h2>
    <p>
     <strong>
      请求分页
     </strong>
     是目前最常用的一种实现虚拟存储器的方法。
    </p>
    <p>
     请求分页系统建立在
     <strong>
      基本分页系统
     </strong>
     基础之上，为了支持虚拟存储器功能而增加了
     <strong>
      请求调页
     </strong>
     功能和
     <strong>
      页面置换
     </strong>
     功能。
    </p>
    <p>
     在请求分页系统中，只要求将当前需要的一部分页面装入内存，便可以启动作业运行。
     <br/>
     在作业执行过程中，当所要访问的页面不在内存时，再通过调页功能将其调入，同时还可以通过置换功能将暂时不用的页面换出到外存上，以便腾出内存空间。
    </p>
    <p>
     为了实现请求分页，系统必须提供一定的硬件支持。除了需要
     <span style="color:#fe2c24;">
      <strong>
       一定容量的内存及外存
      </strong>
     </span>
     的计算机系统，还需要有
     <span style="color:#fe2c24;">
      <strong>
       页表机制、缺页中断机构、地址变换机构
      </strong>
     </span>
     。
    </p>
    <h3 id="2.1%20%E9%A1%B5%E8%A1%A8%E6%9C%BA%E5%88%B6">
     2.1 页表机制
    </h3>
    <p>
     请求分页系统的页表机制不同于基本分页系统，请求分页系统在一个作业运行之前不要求全部一次性调入内存。
    </p>
    <p>
     因此在作业的运行过程中，必然会出现要访问的页面不在内存的情况，如何发现和处理这种情况是请求分页系统必须解决的两个基本问题。为此，在请求页表项中增加了四个字段，分别为：
    </p>
    <table align="center" border="1" cellpadding="1" cellspacing="1" style="width:500px;">
     <caption>
      请求分页系统中的页表项
     </caption>
     <tbody>
      <tr>
       <td>
        <strong>
         页号
        </strong>
       </td>
       <td>
        <strong>
         物理块号
        </strong>
       </td>
       <td>
        <strong>
         状态位 P
        </strong>
       </td>
       <td>
        <strong>
         访问字段 A
        </strong>
       </td>
       <td>
        <strong>
         修改位 M
        </strong>
       </td>
       <td>
        <strong>
         外存地址
        </strong>
       </td>
      </tr>
     </tbody>
    </table>
    <ul>
     <li>
      <strong>
       状态位 P
      </strong>
      ：用于指示该页是否已调入内存，供程序访问时参考。
     </li>
     <li>
      <strong>
       访问字段 A
      </strong>
      ：用于记录本页在一段时间内被访问的次数，或记录本页最近己有多长时间未被访问，供置换算法换出页面时参考。
     </li>
     <li>
      <strong>
       修改位 M
      </strong>
      ：标识该页在调入内存后是否被修改过。
     </li>
     <li>
      <strong>
       外存地址
      </strong>
      ：用于指出该页在外存上的地址，通常是物理块号，供调入该页时参考。
     </li>
    </ul>
    <h3 id="2.1%20%E7%BC%BA%E9%A1%B5%E4%B8%AD%E6%96%AD%E6%9C%BA%E6%9E%84">
     <strong>
      2.2 缺页中断机构
     </strong>
    </h3>
    <p>
     在请求分页系统中，每当所要
     <strong>
      访问的页面不在内存
     </strong>
     时，便产生一个
     <strong>
      缺页中断
     </strong>
     ，请求操作系统将所缺的页调入内存。
    </p>
    <p>
     此时应将缺页的进程阻塞（调页完成唤醒)，如果内存中有空闲块，则分配一个块，将要调入的页装入该块，并修改页表中相应页表项；若此时内存中没有空闲块，则要淘汰某页（若被淘汰页在内存期间被修改过，则要将其写回外存)。
    </p>
    <p>
     缺页中断作为中断同样要经历，诸如保护CPU环境、分析中断原因、转入缺页中断处理程序、恢复CPU环境等几个步骤。但与一般的中断相比，它有以下两个明显的区别：
    </p>
    <ul>
     <li>
      在指令执行期间产生和处理中断信号，而非一条指令执行完后，属于内部中断。
     </li>
     <li>
      一条指令在执行期间，可能产生多次缺页中断。
     </li>
    </ul>
    <h3>
     <strong>
      2.3 地址变换机构
     </strong>
    </h3>
    <p>
     请求分页系统中的地址变换机构，是在分页系统地址变换机构的基础上，为实现虚拟内存，又增加了某些功能而形成的。
    </p>
    <div class="img-center">
     <figure class="image">
      <img alt="请求分页中的地址变换过程" src="https://i-blog.csdnimg.cn/blog_migrate/3fe3fd3726bc2783020ad4ce1c3cfb30.png">
       <figcaption>
        请求分页中的地址变换过程
       </figcaption>
      </img>
     </figure>
    </div>
    <p>
     在进行地址变换时，先检索
     <strong>
      快表
     </strong>
     ：
    </p>
    <ul>
     <li>
      若找到要访问的页，便修改页表项中的访问位（写指令则还须重置修改位)，然后利用页表项中给出的物理块号和页内地址形成物理地址。
     </li>
     <li>
      若未找到该页的页表项，应到内存中去查找页表，再对比页表项中的状态位P，看该页是否已调入内存，未调入则产生缺页中断，请求从外存把该页调入内存。
     </li>
    </ul>
    <blockquote>
     <p>
      <strong>
       页表
      </strong>
      指出逻辑地址中的页号与所占主存物理块号的对应关系。页式存储管理在用动态重定位方式装入作业时，要利用页表做地址转换工作。
     </p>
     <p>
      <br/>
      <strong>
       快表(TLB，Translation Lookaside Buffer)
      </strong>
      就是存放在高速缓冲存储器的部分页表。作为当前进程页表的Cache，它的作用与页表相似，但
      <strong>
       加快了地址映射速度
      </strong>
      ，提高了访问速率。
     </p>
     <p>
     </p>
     <p>
      由于采用页表做地址转换，读写内存数据时CPU要访问两次主存（查询页表、访问目的地址）。
     </p>
     <p>
      有了快表，有时只要访问一次高速缓冲存储器，一次主存，这样可加速查找并提高指令执行速度。
     </p>
    </blockquote>
    <h2>
     3. 页面置换算法
    </h2>
    <p>
     进程运行时，若其访问的页面不在内存而需将其调入，但
     <strong>
      内存已无空闲空间
     </strong>
     时，就需要从内存中调出一页程序或数据，送入磁盘的对换区。
     <br/>
     <br/>
     <strong>
      选择调出页面的算法就称为页面置换算法
     </strong>
     。好的页面置换算法应有
     <strong>
      较低的页面更换频率
     </strong>
     ，也就是说，应将以后不会再访问或者以后较长时间内不会再访问的页面先调出。
    </p>
    <h3>
     3.1 最佳置换算法（OPT）
    </h3>
    <p>
     <strong>
      最佳(Optimal, OPT)置换算法
     </strong>
     所选择的被淘汰页面将是以后永不使用的，或者是在最长时间内不再被访问的页面，这样可以保证获得最低的缺页率。
    </p>
    <p>
     但由于人们目前
     <strong>
      无法预知
     </strong>
     进程在内存下的若千页面中哪个是未来最长时间内不再被访问的，因而
     <strong>
      该算法无法实现
     </strong>
     ，但最佳置换算法可以用来
     <strong>
      评价其他算法
     </strong>
     。
    </p>
    <p>
     假定系统为某进程分配了三个物理块，并考虑有以下页面号引用串：
     <br/>

    <h3>
     <strong>
      3.2 先进先出(FIFO)页面置换算法
     </strong>
    </h3>
    <p>
     <strong>
      优先淘汰最早进入内存的页面
     </strong>
     ，亦即在内存中驻留时间最久的页面。
    </p>
    <p>
     该算法实现简单，只需把调入内存的页面根据先后次序链接成
     <strong>
      队列
     </strong>
     ，设置一个指针总指向最早的页面。但该算法
     <strong>
      与进程实际运行时的规律不适应
     </strong>
     ，因为在进程中，有的页面经常被访问。

       </td>
      </tr>
     </tbody>
    </table>
    <p>
     <strong>
      只有FIFO算法可能出现Belady异常，而LRU和OPT算法永远不会出现Belady异常。
     </strong>
    </p>
    <h3>
     <strong>
      3.3 最近最久未使用(LRU)置换算法
     </strong>
    </h3>
    <p>
     <strong>
      最近最久未使用(LRU，Least Recently Used)
     </strong>
     置换算法选择
     <strong>
      最近最长时间未访问过的页面予以淘汰
     </strong>
     ，它认为过去一段时间内未访问过的页面，在最近的将来可能也不会被访问。
    </p>
    <p>
     该算法为每个页面设置一个
     <strong>
      访问字段
     </strong>
     ，来记录页面
     <strong>
      自上次被访问以来所经历的时间
     </strong>
     ，淘汰页面时选择现有页面中值最大的予以淘汰。
    </p>

    </p>
    <p>
     LRU是
     <strong>
      堆栈类
     </strong>
     的算法。理论上可以证明，堆栈类算法不可能出现Belady异常。FIFO算法基于队列实现，不是堆栈类算法。
    </p>
    <h3>
     <strong>
      3.4 时钟(CLOCK)置换算法
     </strong>
    </h3>
    <p>
     LRU算法的性能接近于OPT,但是实现起来比较困难，且开销大；FIFO算法实现简单，但性能差。所以操作系统的设计者尝试了很多算法，试图用比较小的开销接近LRU的性能，这类算法都是CLOCK算法的变体。
    </p>
    <p>
     简单的CLOCK算法是给每一帧关联一个附加位，称为使用位。当某一页首次装入主存，以及后续被访问时，使用位被置为1。
    </p>
    <p>
     对于页替换算法，用于替换的
     <strong>
      候选帧集合
     </strong>
     看做一个
     <strong>
      循环缓冲区
     </strong>
     ，并且有一个指针与之相关联。当某一页被替换时，该指针被设置成指向缓冲区中的下一帧。
    </p>
    <p>
     当需要替换一页时，操作系统扫描缓冲区，以查找第一个使用位为0的一帧。每当遇到一个使用位为1的帧时，操作系统就将该位重新置为0；如果所有帧的使用位均为1，则指针在缓冲区中完整地循环一周，把所有使用位都置为0，并且停留在最初的位置上，替换该帧中的页。由于该算法循环地检查各页面的情况，故称为CLOCK算法，又称为
     <strong>
      最近未用(Not Recently Used, NRU)
     </strong>
     算法。
    </p>
    <p>
     CLOCK算法的性能比较接近LRU，而通过增加使用的位数目，可以使得CLOCK算法更加高效。在
     <strong>
      使用位used
     </strong>
     的基础上再增加一个
     <strong>
      修改位modified
     </strong>
     ，则得到改进型的CLOCK置换算法。
    </p>
    <p>
     每一帧都处于以下四种情况之一：
    </p>
    <ol>
     <li>
      最近未被访问，也未被修改(u=0, m=0)。
     </li>
     <li>
      最近被访问，但未被修改(u=1, m=0)。
     </li>
     <li>
      最近未被访问，但被修改(u=0, m=1)。
     </li>
     <li>
      最近被访问，被修改(u=1, m=1)。
     </li>
    </ol>
    <p>
     算法执行如下操作步骤：
    </p>
    <ol>
     <li>
      从指针的当前位置开始，扫描帧缓冲区。在这次扫描过程中，对使用位不做任何修改。选择遇到的第一个帧(u=0, m=0)用于替换。
     </li>
     <li>
      如果第1步失败，则重新扫描，查找(u=0, m=1)的帧。选择遇到的第一个这样的帧用于替换。在这个扫描过程中，对每个跳过的帧，把它的使用位设置成0。
     </li>
     <li>
      如果第2步失败，指针将回到它的最初位置，并且集合中所有帧的使用位均为0。重复第1步，并且如果有必要，重复第2步。这样将可以找到供替换的帧。
     </li>
    </ol>
    <p>
     改进型的CLOCK算法优于简单CLOCK算法之处在于
     <strong>
      替换时首选没有变化的页
     </strong>
     。由于
     <strong>
      修改过的页在被替换之前必须写回
     </strong>
     ，因而这样做会节省时间。
    </p>
    <h2>
     4. 页面分配策略
    </h2>
    <h3>
     4.1 驻留集大小
    </h3>
    <p>
     对于分页式的虚拟内存，在准备执行时，不需要也不可能把一个进程的所有页都读取到主存，因此，操作系统必须决定读取多少页。也就是说，
     <strong>
      给特定的进程分配多大的主存空间
     </strong>
     ，这需要考虑以下几点：
    </p>
    <ol>
     <li>
      分配给一个进程的存储量越小，在任何时候驻留在主存中的进程数就越多，从而可以提高处理机的时间利用效率。
     </li>
     <li>
      如果一个进程在主存中的
      <strong>
       页数过少
      </strong>
      ，尽管有局部性原理，
      <strong>
       页错误率
      </strong>
      仍然会相对较高。
     </li>
     <li>
      如果
      <strong>
       页数过多
      </strong>
      ，由于局部性原理，给特定的进程分配更多的主存空间对该进程的错误率没有明显的影响。
     </li>
    </ol>
    <p>
     基于这些因素，现代操作系统通常釆用三种策略：
    </p>
    <ol>
     <li>
      <strong>
       固定分配局部置换
      </strong>
      ：它为每个进程分配一定数目的物理块，
      <strong>
       在整个运行期间都不改变
      </strong>
      。若进程在运行中发生缺页，则只能从
      <strong>
       该进程在内存中的页面中选出一页换出
      </strong>
      ，然后再调入需要的页面。实现这种策略难以确定为每个进程应分配的物理块数目：太少会频繁出现缺页中断，太多又会使CPU和其他资源利用率下降。
     </li>
     <li>
      <strong>
       可变分配全局置换
      </strong>
      ：这是最易于实现的物理块分配和置换策略，为系统中的每个进程分配一定数目的物理块，操作系统自身也保持一个空闲物理块队列。当某进程发生缺页时，系统从
      <strong>
       空闲物理块队列中取出一个物理块分配给该进程
      </strong>
      ，并将欲调入的页装入其中。
     </li>
     <li>
      <strong>
       可变分配局部置换
      </strong>
      ：它为每个进程分配一定数目的物理块，当某进程发生缺页时，只允许从该进程在内存的页面中选出一页换出，这样就不会影响其他进程的运行。如果进程在运行中
      <strong>
       频繁地缺页
      </strong>
      ，系统
      <strong>
       再分配
      </strong>
      若干物理块给该进程，直至该进程缺页率趋于适当程度； 反之，若进程在运行中
      <strong>
       缺页率特别低
      </strong>
      ，则可适当
      <strong>
       减少分配
      </strong>
      给该进程的物理块。
     </li>
    </ol>
    <h3>
     <strong>
      4.2 调入页面的时机
     </strong>
    </h3>
    <p>
     为确定系统将进程运行时所缺的页面调入内存的时机，可釆取以下两种调页策略：
    </p>
    <ol>
     <li>
      <strong>
       <span style="color:#fe2c24;">
        预调页
       </span>
       策略
      </strong>
      ：根据局部性原理，一次调入若干个相邻的页可能会比一次调入一页更高效。但如果调入的一批页面中大多数都未被访问，则又是低效的。所以就需要釆用
      <strong>
       以预测为基础
      </strong>
      的预调页策略，将预计在不久之后便会被访问的页面预先调入内存。但目前预调页的成功率仅约50%。故这种策略主要用于进程的首次调入时，由程序员指出应该先调入哪些页。
     </li>
     <li>
      <strong>
       <span style="color:#fe2c24;">
        请求调页
       </span>
       策略
      </strong>
      ：进程在运行中需要访问的
      <strong>
       页面不在内存而提出请求
      </strong>
      ，由系统将所需页面调入内存。由这种策略调入的页一定会被访问，且这种策略比较易于实现，故在目前的虚拟存储器中大多釆用此策略。它的缺点在于
      <strong>
       每次只调入一页，调入调出页面数多时会花费过多的I/O开销。
      </strong>
     </li>
    </ol>
    <h3>
     <strong>
      4.3 从何处调入页面
     </strong>
    </h3>
    <p>
     请求分页系统中的
     <strong>
      外存
     </strong>
     分为两部分：用于存放文件的
     <span style="color:#fe2c24;">
      <strong>
       文件区
      </strong>
     </span>
     和用于存放对换页面的
     <span style="color:#fe2c24;">
      <strong>
       对换区
      </strong>
     </span>
     。
     <strong>
      对换区通常是釆用连续分配方式
     </strong>
     ，而
     <strong>
      文件区釆用离散分配方式
     </strong>
     ，故
     <strong>
      对换区的磁盘I/O速度比文件区的更快
     </strong>
     。这样从何处调入页面有三种情况：
    </p>
    <ol>
     <li>
      <strong>
       系统拥有足够的对换区空间
      </strong>
      ：可以
      <strong>
       全部从对换区调入
      </strong>
      所需页面，以提髙调页速度。为此，在进程运行前，需将与该进程有关的文件从文件区
      <strong>
       复制
      </strong>
      到对换区。
     </li>
     <li>
      <strong>
       系统缺少足够的对换区空间
      </strong>
      ：凡
      <strong>
       不会被修改
      </strong>
      的文件都直接从
      <strong>
       文件区
      </strong>
      调入（换出时不必写回）。但对于那些可能被修改的部分，在将它们换出时须调到对换区，以后需要时再从对换区调入。
     </li>
     <li>
      <strong>
       UNIX方式
      </strong>
      ：与进程有关的文件都放在文件区，故
      <strong>
       未运行过的页面，都应从文件区调入
      </strong>
      。曾经运行过但又被换出的页面，由于是被放在对换区，因此
      <strong>
       下次调入时应从对换区调入
      </strong>
      。进程请求的共享页面若被其他进程调入内存，则无需再从对换区调入。
     </li>
    </ol>
    <h2>
     5. 页面抖动(颠簸)和工作集(驻留集)
    </h2>
    <h3>
     <strong>
      5.1 页面抖动（颠簸）
     </strong>
    </h3>
    <p>
     在页面置换过程中的一种最糟糕的情形是，刚刚换出的页面马上又要换入主存，刚刚换入的页面马上就要换出主存，这种
     <strong>
      频繁的页面调度行为称为抖动，或颠簸
     </strong>
     。如果一个进程在
     <strong>
      换页上用的时间多于执行时间
     </strong>
     ，那么这个进程就在颠簸。
     <br/>
     <br/>
     频繁的发生缺页中断（抖动），其主要原因是
     <strong>
      某个进程频繁访问的页面数目高于可用的物理页帧数目
     </strong>
     。虚拟内存技术可以在内存中保留更多的进程以提高系统效率。在稳定状态，几乎主存的所有空间都被进程块占据，处理机和操作系统可以直接访问到尽可能多的进程。但如果管理不当，处理机的大部分时间都将用于交换块，即请求调入页面的操作，而不是执行进程的指令，这就会大大降低系统效率。
    </p>
    <h3>
     <strong>
      5.2 工作集（驻留集）
     </strong>
    </h3>
    <p>
     工作集（或驻留集）是指在
     <strong>
      某段时间间隔内，进程要访问的页面集合
     </strong>
     。经常被使用的页面需要在工作集中，而长期不被使用的页面要从工作集中被丢弃。为了防止系统出现抖动现象，需要选择合适的工作集大小。
     <br/>
     <br/>
     工作集模型的原理是：让操作系统跟踪每个进程的工作集，并为进程分配大于其工作集的物理块。如果还有空闲物理块，则可以再调一个进程到内存以增加多道程序数。如果所有工作集之和增加以至于超过了可用物理块的总数，那么操作系统会暂停一个进程，将其页面调出并且将其物理块分配给其他进程，防止出现抖动现象。
     <br/>
     <br/>
     正确选择工作集的大小，对
     <strong>
      存储器的利用率和系统吞吐量
     </strong>
     的提高，都将产生重要影响。
    </p>
    <h2>
     6. 总结
    </h2>

      分页
     </strong>

     <strong>
      分段
     </strong>
