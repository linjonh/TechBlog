    <p>
     在 Kubernetes 中，metrics-server 是一个轻量级的指标采集器，用于提供集群节点和 Pod 的资源使用情况（如 CPU 和内存）。它主要用于支撑 Kubernetes 的自动扩展功能，如 Horizontal Pod Autoscaler (HPA) 和 Vertical Pod Autoscaler (VPA)。
    </p>
    <h4>
     <a id="font_stylecolorrgb0_0_01_fontfont_stylecolorrgb0_0_0font_13">
     </a>
     1.
     <strong>
      采集周期
     </strong>
    </h4>
    <ul>
     <li>
      <strong>
       默认采集周期
      </strong>
      : metrics-server 的默认采集周期通常为
      <strong>
       60 秒
      </strong>
      ，即每隔 60 秒从各节点采集一次指标数据。
     </li>
     <li>
      <strong>
       可配置采集周期
      </strong>
      : 你可以通过设置 –metric-resolution 参数来自定义这个采集周期。例如，设置 –metric-resolution=30s 可以将采集周期调整为 30 秒。
     </li>
    </ul>
    <h4>
     <a id="font_stylecolorrgb0_0_02_fontfont_stylecolorrgb0_0_0font_17">
     </a>
     2.
     <strong>
      采集链路
     </strong>
    </h4>
    <p>
     metrics-server 的采集链路涉及几个关键组件：Kubelet、API Server、和 metrics-server 本身。以下是它的详细采集流程：
    </p>
    <h5>
     <a id="font_stylecolorrgb0_0_0_1_Kubelet__Pod_font_20">
     </a>
     <strong>
      步骤 1: Kubelet 采集节点和 Pod 级别的资源使用情况
     </strong>
    </h5>
    <ul>
     <li>
      每个 Kubernetes 节点上都运行着 kubelet，它负责监控该节点上运行的所有容器的状态和资源使用情况。
     </li>
     <li>
      kubelet 从容器运行时（如 Docker 或 containerd）和 cgroups 中采集资源使用数据（例如 CPU 和内存使用量），并将这些数据保存在本地的 /metrics 或 /stats/summary 端点上。
     </li>
    </ul>
    <h5>
     <a id="font_stylecolorrgb0_0_0_2_metricsserver_font_24">
     </a>
     <strong>
      步骤 2: metrics-server 拉取数据
     </strong>
    </h5>
    <ul>
     <li>
      metrics-server 周期性地从集群中每个节点的 kubelet 上的 /metrics/resource/v1alpha1 端点（或者 /stats/summary 端点）拉取节点和 Pod 的资源使用数据。
     </li>
     <li>
      这种通信是通过 HTTPS 加密的，metrics-server 使用节点证书进行认证，并且 metrics-server 必须被配置为可以访问这些节点端点。
     </li>
    </ul>
    <h5>
     <a id="font_stylecolorrgb0_0_0_3_metricsserver_font_28">
     </a>
     <strong>
      步骤 3: metrics-server 聚合数据
     </strong>
    </h5>
    <ul>
     <li>
      metrics-server 收集到的数据包括每个节点上所有 Pod 的 CPU 和内存使用量。
     </li>
     <li>
      它将这些数据进行聚合和处理，并在内存中存储最近一次的采集数据。这意味着 metrics-server 不会持久化历史数据，而是只保留当前的最新状态。
     </li>
    </ul>
    <h5>
     <a id="font_stylecolorrgb0_0_0_4__API_Serverfont_32">
     </a>
     <strong>
      步骤 4: 提供数据给 API Server
     </strong>
    </h5>
    <ul>
     <li>
      一旦数据被处理好，metrics-server 通过 Kubernetes API 端点提供这些数据。API Server 可以通过 apis/metrics.k8s.io/v1beta1 端点访问这些指标数据。
     </li>
     <li>
      这个数据端点可以被 Kubernetes 内部组件（如 HPA 控制器）或外部用户查询，以获得当前集群中各个 Pod 和节点的资源使用情况。
     </li>
    </ul>
    <h4>
     <a id="font_stylecolorrgb0_0_03_fontfont_stylecolorrgb0_0_0font_36">
     </a>
     <strong>
      3. 总结的采集链路图示化
     </strong>
     ：
    </h4>
    <ol>
     <li>
      <strong>
       Kubelet
      </strong>
      采集节点及容器的资源使用数据（CPU、内存）。
     </li>
     <li>
      <strong>
       metrics-server
      </strong>
      周期性地通过 HTTPS 从每个节点的 Kubelet 拉取数据。
     </li>
     <li>
      <strong>
       metrics-server
      </strong>
      聚合并处理这些数据，保存在内存中。
     </li>
     <li>
      <strong>
       metrics-server
      </strong>
      通过 metrics.k8s.io API 端点将数据提供给 Kubernetes API Server。
     </li>
     <li>
      <strong>
       用户或组件（如 HPA）
      </strong>
      通过 API Server 查询这些实时资源使用数据，以做出扩展决策。
     </li>
    </ol>
    <h4>
     <a id="font_stylecolorrgb0_0_04font_43">
     </a>
     4.配置和优化
    </h4>
    <ul>
     <li>
      <strong>
       采集周期
      </strong>
      ：你可以通过 –metric-resolution 参数配置 metrics-server 的采集周期。如果集群对实时性要求较高，可以设置更短的采集周期，但这可能会增加系统的开销。
     </li>
     <li>
      <strong>
       安全性
      </strong>
      ：确保 metrics-server 与 kubelet 之间的通信是安全的，通常通过 TLS 证书认证。此外，配置 metrics-server 使其具有足够的权限去访问节点数据。
     </li>
    </ul>
    <p>
     通过这些配置和机制，metrics-server 能够为 Kubernetes 提供关键的资源使用数据，帮助集群进行资源管理和自动扩展。
    </p>
