    <p>
     在当今的网络环境中，数据的共享和存储变得至关重要。NFS（Network File System），即网络文件系统，作为一种通过网络共享目录的重要工具，正发挥着不可或缺的作用。它能够让网络上的其他服务器方便地挂载访问共享目录的数据，尤其适用于存放视频、附件、图片等静态数据。本文将深入探讨 NFS 的工作原理、服务安装等相关内容，帮助了解和使用 NFS。
    </p>
    <hr/>
    <ul>
     <li>
      NFS简单介绍
     </li>
    </ul>
    <p>
     NFS（Network File System）中文意思是网络文件系统。主要功能是通过网络共享目录，让网络上的其他服务器能够挂载访问共享目录的数据（NFS一般都是用来存放视频、附件、图片等静态数据的）。NFS客户端（一般为应用服务器，如 Web）可以通过挂载方式将NFS服务器端共享的数据目录挂载到NFS本地客户端（就是某一个挂载点下）。从本地客户端来看，NFS服务器端共享的目录就好像是客户端自己的磁盘分区或者目录一样，而实际上却是远端的NFS服务器的目录。
    </p>
    <h2>
     <a id="1_NFS_12">
     </a>
     1. NFS工作原理
    </h2>
    <h3>
     <a id="11__14">
     </a>

      <code>

     </p>

    <h3>
     <a id="12_NFS_41">
     </a>
     1.2 NFS的工作原理
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ac75e9bb5e874a49901d41cb00822dd1.png"/>
    </p>
    <blockquote>
     <p>
      工作流程：
     </p>
     <ol>
      <li>
       首先服务端会启动
       <strong>
        RPC服务
       </strong>
       ，并开启111端口
      </li>
      <li>
       服务端启动
       <strong>
        NFS服务
       </strong>
       ，并向RPC注册端口
      </li>
      <li>
       客户端启动RPC服务，向服务端的RPC服务请求获取NFS端口
      </li>
      <li>
       服务端的RPC服务反馈端口给客户端的
      </li>
      <li>
       客户端通过服务端的RPC反馈回来的端口和服务端进行数据传输
      </li>
     </ol>
    </blockquote>
    <ul>
     <li>
      RPC服务的引入
     </li>
    </ul>
    <p>
     NFS支持的功能相当多，而不同的功能会使用不同的程序来启动，每启动一个功能就会启用一些端口来传输数据，因此，NFS的功能所对应的端口无法固定，会随机取用一些未被使用的端口作为传输之用。
     <br/>
     因为端口不固定，这样一来就会造成NFS客户端与NFS服务器端的通信障碍，需要通过远程过程调用RPC服务来帮忙了，NFS的RPC服务最主要的功能就是记录每个NFS 功能所对应的端口号，并且在NFS客户端发出请求时将该端口和功能对应的信息传递给请求数据的NFS客户端，从而确保客户端可以连接到正确的NFS端口上去，达到实现数据传输、交互数据目的。
    </p>
    <ul>
     <li>
      RPC 服务又是如何知道每个NFS端口的呢?
     </li>
    </ul>
    <p>
     NFS服务器端启动服务时会随机取用若干端口，并主动向RPC服务注册取用的相关端口及功能信息，如此一来，RPC服务就知道NFS每个端口对应的NFS功能了，然后RPC服务使用固定111端口来监听NFS客户端提交的请求，并将正确的NFS端口信息回复给请求的NFS客户端，这样一来，NFS客户端就可以与NFS服务器端进行数据传输了。
     <br/>
     所以在启动NFS服务之前，首先要启动RPC服务，否则NFS服务就无法向RPC服务注册了。另外，如果RPC服务重新启动，原来已经注册好的NFS端口数据就会丢失，因此，此时RPC服务管理的NFS程序也需要重新启动以重新向RPC注册。要特别注意，修改NFS配置文件后是不需要重启NFS的，直接在命令行执行
     <code>
      systemctl reload nfs
     </code>
     或
     <code>
      exportfs -rv
     </code>
     即可使修改生效。
    </p>
    <ul>
     <li>
      NFS启动介绍
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ee1dc73af8d9449ba3f2336aa6b8cd22.png"/>
    </p>
    <blockquote>
     <p>
      当访问程序通过NFS客户端向NFS服务器端存取文件时，其请求数据流程大致如下:
     </p>
     <ol>
      <li>
       用户访问网站程序，由程序在NFS 客户端上发出存取NPS 文件请求，这时NFS客户端（执行程序的服务器）的RPC服务（ rpcbind服务）就会通过网络向NFS服务器端的RPC 服务的111端口发出NFS文件存取功能的询问请求。
      </li>
      <li>
       NFS服务器RPC服务找到对应已注册的NFS端口后，通知NFS客户端RPC服务。
      </li>
      <li>
       此时NFS客户端获取到正确的端口，并与 NFS daemon 联机存取数据。
      </li>
      <li>
       NFS客户端将数据存取成功后返回给前端访问程序，告知给用户存取结果，就完成了一次存取操作。
      </li>
     </ol>
    </blockquote>
    <ul>
     <li>
      总结
     </li>
    </ul>
    <p>
     NFS的各项功能都需要向RPC服务注册，所以只有RPC服务才能获取到NFS服务的各项功能对应的端口号(port number)、PID、NFS在主机所监听的IP等信息，而NFS客户端也只能通过向RPC服务询问才能找到正确的端口。
    </p>
    <p>
     也就是说，
     <strong>
      NFS需要有RPC服务的协助才能成功对外提供服务。
     </strong>
    </p>
    <p>
     <strong>
      所以无论是NFS客户端还是NFS服务器端，当要使用NFS时都需要首先启动RPC服务，NFS服务必须在RPC 服务启动之后启动，客户端无须启动NFS服务，但需要启动RPC服务。
     </strong>
    </p>
    <h2>
     <a id="2_NFS_82">
     </a>
     2. NFS服务安装
    </h2>
    <h3>
     <a id="21_NFS_84">
     </a>
     2.1 NFS软件列表
    </h3>
    <p>
     要部署NFS服务，需要安装以下软件包。
    </p>
    <ul>
     <li>
      <code>
       nfs-utils
      </code>
      ：NFS服务的主程序，包括
      <code>
       rpc.nfsd
      </code>
      、
      <code>
       rpc.mountd
      </code>
      这两个daemons和相关文档说明，以及执行命令文件等。
     </li>
     <li>
      <code>
       rpcbind
      </code>
      ：CentOS6/7环境下RPC主程序。NFS可视为一个RPC程序，在启动任何一个RPC程序之前，需要做好端口和功能的映射工作，这个映射工作就是由rpcbind服务来完成的。因此，在提供NFS服务之前必须先启动rpcbind服务才行。
     </li>
    </ul>
    <blockquote>
     <p>
      使用如下命令查看默认情况下NFS软件的安装情况。
     </p>
     <pre><code class="prism language-bash"><span class="token punctuation">[</span>root@nfs ~<span class="token punctuation">]</span><span class="token comment"># rpm -qa nfs-utils rpcbind</span>

nfs-utils-1.3.0-0.68.el7.2.x86_64
</code></pre>
     <p>
      CentOS7默认没有安装 NFS软件包，可以使用
      <code>
       yum install nfs-utils rpcbind -y
      </code>
      命令来安装NFS软件。
     </p>
    </blockquote>

    </h2>
    <h3>
     <a id="41_NFS_401">
     </a>
     4.1 NFS服务的重点知识梳理
    </h3>
    <p>
     当多个NFS客户端访问服务器端读写文件时，需要具有以下几个权限:
    </p>
    <ul>
     <li>
      NFS 服务器
      <code>
       /etc/exports
      </code>
      设置需要开放可写入的权限，即服务器端的共享权限。
     </li>
     <li>
      NFS服务器实际要共享的NFS目录权限具有写入的权限，即服务器端本地目录的安全权限。
     </li>
     <li>
      每台机器都对应存在和NFS默认配置UID的相同UID 65534的
      <code>
       nfsnobody
      </code>
      用户（确保所有客户端的访问权限统一，否则每个机器需要同时建立相同UID的用户，并覆盖NFS的默认用户配置）。
     </li>
    </ul>
    <blockquote>
     <p>
      只有满足上述3个条件，多个NFS客户端才能具有查看、修改、删除其他任意NFS客户端上传的文件的权限，这在大规模的集群环境中作为集群共享存储时尤为重要。
     </p>
    </blockquote>

     <a id="42_NFS_410">

    <ul>
     <li>
      <strong>
       mount -o 的参数说明
      </strong>
     </li>
    </ul>
    <blockquote>
     <p>
      注意，有些选项只有出现在
      <code>
       /etc/fstab
      </code>
      里才有效，可以应用在绝大多数文件系统上，但是sync仅适合ext2、ext3、fat、vfat 和 ufs等文件系统。
     </p>
     <ul>
      <li>
       <strong>
        async
       </strong>
       ：涉及文件系统IO的操作都是异步处理，会提高性能，但是降低数据安全性。一般生产环境不推荐使用。
      </li>
      <li>
       sync：有IO操作时会同步处理IO，会牺牲一些IO性能，但是在断电后数据会安全。
      </li>
      <li>
       atime：每次数据访问时，会同步更新访问文件夹的inode时间戳，默认选项。
      </li>
      <li>
       <strong>
        noatime
       </strong>
       ：访问文件时不更新文件的inode时间戳，高并发环境下推荐使用，可以提高IO性能。
      </li>
      <li>
       ro：以只读方式挂载一个文件系统。
      </li>
      <li>
       <strong>
        rw
       </strong>
       ：以可写的方式挂载一个文件系统。
      </li>
      <li>
       auto：能够通过 -a 参数实现自动挂载。
      </li>
      <li>
       noauto：不能实现自动挂载。
      </li>
      <li>
       defaults：是fstab文件中的默认值，包括（rw, suid, dev, exec, auto, nouser, async）.
      </li>
      <li>
       exec：允许文件系统执行二进制文件，取消这个参数可以提高安全性。
      </li>
      <li>
       <strong>
        noexec
       </strong>
       ：不允许直接执行二进制的程序。
      </li>
      <li>
       dirsync：目录更新时同步写入磁盘。
      </li>
      <li>
       <strong>
        nodiratime
       </strong>
       ：不更新文件系统上目录的inode时间戳，高并发环境下推荐使用，可以提高IO性能。
      </li>
      <li>
       suid：允许特殊权限为suid和sgid生效。
      </li>
      <li>
       <strong>
        nosuid
       </strong>
       ：不允许设置特殊权限位生效，提升安全性。
      </li>
      <li>
       nouser：禁止一个普通用户挂载该文件系统，这是默认挂载时的默认选项。
      </li>
      <li>
       <strong>
        remount
       </strong>
       ：尝试重新挂载一个已经挂载了的文件系统。
      </li>
     </ul>
    </blockquote>
    <p>
     一般情况下，安全参数与性能参数是对立的，越安全性能越差。
    </p>
    <h3>

     4.4 客户端挂载优化
    </h3>
    <p>
     一般来讲，NFS服务器共享的只是普通静态数据（图片、附件、视频），不需要执行
     <code>
      suid
     </code>
     和
     <code>
      exec
     </code>
     等权限，挂载的这个文件系统只能作为数据存储之用，无法执行程序，对于客户端来讲增加了安全性。
    </p>
    <pre><code class="prism language-bash"><span class="token function">mount</span> <span class="token parameter variable">-t</span> nfs <span class="token parameter variable">-o</span> nosuid,noexec,nodev,rw <span class="token number">172.16</span>.1.31:/data /mnt              
</code></pre>
    <p>
     通过
     <code>
      mount -o
     </code>
     指定挂载参数和在
     <code>
      /etc/fstab
     </code>
     里指定挂载参数的效果是一样的。网络文件系统和本地的文件系统效果也是一样的。
    </p>
    <p>
     默认挂载方式：
    </p>
    <pre><code class="prism language-bash"><span class="token function">mount</span> <span class="token parameter variable">-t</span> nfs <span class="token number">172.16</span>.1.31:/data /mnt        
</code></pre>
    <p>
     禁止更新目录及文件的挂载方式：
    </p>
    <pre><code class="prism language-bash"><span class="token function">mount</span> <span class="token parameter variable">-t</span> nfs <span class="token parameter variable">-o</span> noatime,nodiratime <span class="token number">172.16</span>.1.31:/data /mnt                  
</code></pre>
    <p>
     安全加优化的挂载方式：
    </p>
    <pre><code class="prism language-bash"><span class="token function">mount</span> <span class="token parameter variable">-t</span> nfs <span class="token parameter variable">-o</span> nosuid,noexec,nodev,noatime,nodiratime,intr,rsize<span class="token operator">=</span><span class="token number">131072</span>,wsize<span class="token operator">=</span><span class="token number">131072</span> <span class="token number">172.16</span>.1.31:/data /mnt     
</code></pre>
    <p>
     在实际的工作中还是需要根据自己的实际测试效果选择挂载参数。
    </p>
    <h3>
     <a id="45_NFS_520">
     </a>
     4.5 NFS客户端开机自启动挂载
    </h3>
    <p>
     根据以往的知识我们可以知道可实现开机自启动（挂载）有两种实现方式：
    </p>
    <ul>
     <li>
      <strong>
       方法1：对/etc/rc.local文件进行编辑
      </strong>
     </li>
    </ul>
    <pre><code class="prism language-bash"><span class="token builtin class-name">echo</span> <span class="token string">"/usr/bin/mount -t nfs 172.16.1.31:/data /mnt"</span> <span class="token operator">&gt;&gt;</span>/etc/rc.local 
<span class="token comment"># 或者直接对文件进行编辑       </span>

<span class="token function">chmod</span> +x /etc/rc.d/rc.local    
</code></pre>
    <p>
     根据文件提示注意，在使用该文件实现开机自启动时，需要执行命令开放权限。
    </p>
    <blockquote>
     <p>
      <strong>
       缺点
      </strong>
      ：可能偶尔开机挂载不上，工作中除了开机自启动配置，还要对是否挂载进行监控。
     </p>
    </blockquote>
    <ul>
     <li>
      <strong>
       方法2：对/etc/fstab文件进行编辑
      </strong>
     </li>
    </ul>
    <pre><code class="prism language-bash"><span class="token assign-left variable">UUID</span><span class="token operator">=</span>1d1e4734-e83b-408d-9c6d-2b79b36ffcbb  /         xfs    defaults    <span class="token number">0</span> <span class="token number">0</span>
<span class="token assign-left variable">UUID</span><span class="token operator">=</span>767454a3-0fbe-4d9a-9fc4-39d6f19dae4e  /boot     xfs    defaults    <span class="token number">0</span> <span class="token number">0</span>
<span class="token assign-left variable">UUID</span><span class="token operator">=</span>1c1b7499-345f-466e-97b5-bfb64e265f0b  swap      swap   defaults    <span class="token number">0</span> <span class="token number">0</span>
<span class="token number">172.16</span>.1.31:/data                          /mnt      nfs    defaults    <span class="token number">0</span> <span class="token number">0</span>
挂载设备                                    挂载点    文件系统      
</code></pre>
    <blockquote>
     <ul>
      <li>
       <strong>
        问题
       </strong>
       ：编辑完成后依然无法实现自动挂载服务，是因为开机启动加载顺序的原因。
      </li>
      <li>
       <strong>
        解释
       </strong>
       ：正常情况下，加载时依次是挂载磁盘（/etc/fstab）、防火墙、网卡。因为NFS是一个 网络共享服务，只有网络通畅之后才能挂载磁盘，与正常的开机加载顺序有冲突。
      </li>
      <li>
       <strong>
        解决
       </strong>
       ：所以这里需要一个延迟启动服务来实现NFS开机自动挂载。在CentOS 7中是
       <code>
        remote-fs.target
       </code>
       服务
      </li>
     </ul>
    </blockquote>
    <pre><code class="prism language-bash"><span class="token punctuation">[</span>root@nfs01 /data1<span class="token punctuation">]</span><span class="token comment"># systemctl start remote-fs.target </span>
<span class="token punctuation">[</span>root@nfs01 /data1<span class="token punctuation">]</span><span class="token comment"># systemctl enable remote-fs.target </span>
</code></pre>
    <hr/>
    <p>
     综上所述，NFS 在网络数据共享方面有着重要的地位和作用。通过了解其工作原理，我们明白 NFS 需要有 RPC 服务的协助才能成功对外提供服务，无论是客户端还是服务器端，使用 NFS 时都需先启动 RPC 服务。同时，掌握 NFS 服务的安装步骤，包括安装必要的软件包以及正确启动相关服务，能够让我们顺利搭建起 NFS 服务环境。希望本文的内容能在实际应用 NFS 时提供有益的参考和帮助。
    </p>
