      <h3>
       <span class="mw-headline" id=".E4.B8.BB.E9.94.AE">
        主键
       </span>
      </h3>
      <div class="thumb tright">
       <div class="thumbinner" style="width:282px">
        <a class="image" href="http://zh.wikipedia.org/wiki/File:PrimaryKey_zht.svg" rel="nofollow noopener noreferrer" target="_blank">
         <img alt="" class="thumbimage" height="160" src="http://upload.wikimedia.org/wikipedia/commons/thumb/0/0f/PrimaryKey_zht.svg/280px-PrimaryKey_zht.svg.png" width="280"/>
        </a>
        <div class="thumbcaption">
         <div class="magnify">
          <a class="internal" href="http://zh.wikipedia.org/wiki/File:PrimaryKey_zht.svg" rel="nofollow noopener noreferrer" target="_blank" title="放大">
           <img alt="" height="11" src="http://bits.wikimedia.org/static-1.23wmf17/skins/common/images/magnify-clip.png" width="15"/>
          </a>
         </div>
         数据库理论中, 超键、候选键与主键之间的关系:
         <br/>
         • 候选键是超键的一种：不含有多余属性的超键
         <br/>
         • 某个候选键被选为主键
        </div>
       </div>
      </div>
      <p>
       <strong>
        <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E4%B8%BB%E9%94%AE" rel="nofollow noopener noreferrer" target="_blank" title="主键">
         主键
        </a>
       </strong>
       ，又称
       <strong>
        主码
       </strong>
       （
       <span class="LangWithName">
        <a href="http://zh.wikipedia.org/wiki/%E8%8B%B1%E8%AF%AD" rel="nofollow noopener noreferrer" target="_blank" title="英语">
         英语
        </a>
        ：
        <span lang="en">
         primary key或unique key
        </span>
       </span>
       ）。
       <a href="http://zh.wikipedia.org/wiki/%E6%95%B0%E6%8D%AE%E5%BA%93%E8%A1%A8" rel="nofollow noopener noreferrer" target="_blank" title="数据库表">
        数据库表
       </a>
       中对储存数据对象予以唯一和完整标识的
       <a class="new" href="http://zh.wikipedia.org/w/index.php?title=%E6%95%B0%E6%8D%AE%E5%88%97&amp;action=edit&amp;redlink=1" rel="nofollow noopener noreferrer" target="_blank" title="数据列（页面不存在）">
        数据列
       </a>
       或属性的组合。一个数据列只能有一个主键，且主键的取值不能缺失，即不能为
       <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E7%A9%BA%E5%80%BC" rel="nofollow noopener noreferrer" target="_blank" title="空值">
        空值
       </a>
       （Null）。
      </p>
      <p>
       从技术的角度来看，primary key和unique key有很多相似之处。但还是有以下区别：
      </p>
      <ol>
       <li>
        作为primary key的域／域组不能为Null。而unique key可以。
       </li>
       <li>
        在一个表中只能有一个primary key，而多个unique key可以同时存在。
       </li>
       <li>
        更大的区别在逻辑设计上。primary key一般在逻辑设计中用作记录标识，这也是设置primary key的本来用意。而unique key只是为了保证域／域组的唯一性。
       </li>
      </ol>
      <h3>
       <span class="mw-headline" id=".E8.B6.85.E9.94.AE">
        超键
       </span>
      </h3>
      <p>
       超键（
       <span class="LangWithName">
        <a href="http://zh.wikipedia.org/wiki/%E8%8B%B1%E8%AF%AD" rel="nofollow noopener noreferrer" target="_blank" title="英语">
         英语
        </a>
        ：
        <span lang="en">
         superkey
        </span>
       </span>
       ），有的文献称“超码”，是在数据库
       <a class="new" href="http://zh.wikipedia.org/w/index.php?title=%E5%85%B3%E7%B3%BB%E6%A8%A1%E5%BC%8F&amp;action=edit&amp;redlink=1" rel="nofollow noopener noreferrer" target="_blank" title="关系模式（页面不存在）">
        关系模式
       </a>
       设计中能够唯一标示
       <a href="http://zh.wikipedia.org/wiki/%E5%A4%9A%E5%85%83%E7%BB%84" rel="nofollow noopener noreferrer" target="_blank" title="多元组">
        多元组
       </a>
       的属性集。
      </p>
      <p>
       包含
       <strong>
        所有
       </strong>
       属性的集叫做
       <strong>
        <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E6%98%8E%E9%A1%AF_%28%E6%95%B8%E5%AD%B8%29" rel="nofollow noopener noreferrer" target="_blank" title="明显 (数学)">
         明显
        </a>
        超键
       </strong>
       。
      </p>
      <h3>
       <span class="mw-headline" id=".E5.80.99.E9.80.89.E9.94.AE">
        候选键
       </span>
      </h3>
      <p>
       在
       <a href="http://zh.wikipedia.org/wiki/%E5%85%B3%E7%B3%BB%E6%A8%A1%E5%9E%8B" rel="nofollow noopener noreferrer" target="_blank" title="关系模型">
        关系模型
       </a>
       中，
       <strong>
        候选键
       </strong>
       或
       <strong>
        候选码
       </strong>
       （
       <span class="LangWithName">
        <a href="http://zh.wikipedia.org/wiki/%E8%8B%B1%E8%AF%AD" rel="nofollow noopener noreferrer" target="_blank" title="英语">
         英语
        </a>
        ：
        <span lang="en">
         candidate key
        </span>
       </span>
       ）是某个
       <a class="new" href="http://zh.wikipedia.org/w/index.php?title=%E5%85%B3%E7%B3%BB%E5%8F%98%E9%87%8F&amp;action=edit&amp;redlink=1" rel="nofollow noopener noreferrer" target="_blank" title="关系变量（页面不存在）">
        关系变量
       </a>
       的一组属性所组成的
       <a href="http://zh.wikipedia.org/wiki/%E9%9B%86%E5%90%88" rel="nofollow noopener noreferrer" target="_blank" title="集合">
        集合
       </a>
       ，它需要同时满足下列两个条件：
      </p>
      <ol>
       <li>
        这个属性集合始终能够确保在关系中能唯一标识
        <a href="http://zh.wikipedia.org/wiki/%E5%A4%9A%E5%85%83%E7%BB%84" rel="nofollow noopener noreferrer" target="_blank" title="多元组">
         元组
        </a>
       </li>
       <li>
        在这个属性集合中找不出合适的
        <a href="http://zh.wikipedia.org/wiki/%E5%AD%90%E9%9B%86" rel="nofollow noopener noreferrer" target="_blank" title="子集">
         子集
        </a>
        能够满足条件(1)
       </li>
      </ol>
      <p>
       满足第一个条件的属性集合称为
       <strong>
        <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E8%B6%85%E9%94%AE" rel="nofollow noopener noreferrer" target="_blank" title="超键">
         超键
        </a>
       </strong>
       ，因此我们也可以把候选键定义为“最小超键”，也就是不含有多余属性的超键。
      </p>
      <p>
       候选键的重要性是它们能够在关系中唯一标识出不同的元组，因此超键也是在设计
       <a class="new" href="http://zh.wikipedia.org/w/index.php?title=%E6%95%B0%E6%8D%AE%E5%BA%93%E6%A8%A1%E5%BC%8F&amp;action=edit&amp;redlink=1" rel="nofollow noopener noreferrer" target="_blank" title="数据库模式（页面不存在）">
        数据库模式
       </a>
       时需要指定的最重要的约束之一。由于在关系模型中，每个关系都是一个集合（没有重复的元素），所以每个关系都至少有一个候选键（因为所有属性组合必然是个超键）。但是在某些
       <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E5%85%B3%E7%B3%BB%E5%9E%8B%E6%95%B0%E6%8D%AE%E5%BA%93" rel="nofollow noopener noreferrer" target="_blank" title="关系型数据库">
        关系型数据库
       </a>
       中
       <a href="http://zh.wikipedia.org/wiki/%E6%95%B0%E6%8D%AE%E5%BA%93%E8%A1%A8" rel="nofollow noopener noreferrer" target="_blank" title="数据库表">
        表
       </a>
       也能代表
       <a href="http://zh.wikipedia.org/wiki/%E5%A4%9A%E9%87%8D%E9%9B%86" rel="nofollow noopener noreferrer" target="_blank" title="多重集">
        多重集
       </a>
       ，所以在每个关系中都显式地定义至少一个候选键是一条很重要的设计原则。
       <a href="http://zh.wikipedia.org/wiki/%E6%95%B0%E6%8D%AE%E5%BA%93%E7%AE%A1%E7%90%86%E7%B3%BB%E7%BB%9F" rel="nofollow noopener noreferrer" target="_blank" title="数据库管理系统">
        数据库管理系统
       </a>
       通常都需要将每个关系中的某个候选键定义为
       <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E4%B8%BB%E9%94%AE" rel="nofollow noopener noreferrer" target="_blank" title="主键">
        主键
       </a>
       ，亦即这个候选键是区分不同元组时首选的识别方式，例如
       <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E5%A4%96%E9%94%AE" rel="nofollow noopener noreferrer" target="_blank" title="外键">
        外键
       </a>
       通常就是引用
       <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E4%B8%BB%E9%94%AE" rel="nofollow noopener noreferrer" target="_blank" title="主键">
        主键
       </a>
       而非其他候选键。
      </p>
      <h3>
       <span class="mw-headline" id=".E5.A4.96.E9.94.AE">
        外键
       </span>
      </h3>
      <p>
       <strong>
        外键
       </strong>
       （
       <span class="LangWithName">
        <a href="http://zh.wikipedia.org/wiki/%E8%8B%B1%E8%AF%AD" rel="nofollow noopener noreferrer" target="_blank" title="英语">
         英语
        </a>
        ：
        <span lang="en">
         foreign key
        </span>
       </span>
       ，台湾译作
       <strong>
        外来键
       </strong>
       ），又称
       <strong>
        外部键
       </strong>
       。其实在关系数据库中，每个数据表都是由关系来连系彼此的关系，父数据表（
       <span lang="en">
        Parent Entity
       </span>
       ）的
       <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E4%B8%BB%E9%94%AE" rel="nofollow noopener noreferrer" target="_blank" title="主键">
        主键
       </a>
       （
       <span lang="en">
        primary key
       </span>
       ）会放在另一个数据表，当做属性以创建彼此的关系，而这个属性就是外键。
      </p>
      <p>
       比如，学生跟老师之间是教学的关系，学生数据表会有个属性叫指导老师(FK)，而这个值就是对应到老师数据表的老师代号(PK)，学生的指导老师就是外键。
      </p>
      <h3>
       <span class="mw-headline" id=".E4.BB.A3.E7.90.86.E9.8D.B5">
        代理键
       </span>
      </h3>
      <p>
       在
       <a class="mw-redirect" href="http://zh.wikipedia.org/wiki/%E5%85%B3%E7%B3%BB%E5%9E%8B%E6%95%B0%E6%8D%AE%E5%BA%93" rel="nofollow noopener noreferrer" target="_blank" title="关系型数据库">
        关系型数据库
       </a>
       设计中，
       <strong>
        代理键
       </strong>
       是在当数据表中的候选键都不适合当主键时，例如数据太长，或是意义层面太多，就会请一个无意义的但唯一的字段来代为作主键。
      </p>
      <p>
       代理键是：
      </p>
      <dl>
       Surrogate (1) – Hall, Owlett and Codd (1976)
       <dd>
        一个代理键值确定了外部世界的一个实体。代理键值是数据库生成的，从来不显示给用户或应用程序看。
       </dd>
       Surrogate (2) – Wieringa and De Jonge (1991)
       <dd>
        一个代理键值确定了数据库中的一个对象。代理键值是数据库生成的，用户或应用程序看不到它。
       </dd>
      </dl>
      <p>
       在实践中，代理键值通常是个自动递增的数字。在Sybase或SQL Server，用identity column标识代理键，PostgreSQL里用serial，Oracle里用SEQUENCE，在MySQL里用一个标记有AUTO_INCREMENT的字段。
      </p>
      <h4>
       <span class="mw-headline" id=".E4.BD.95.E6.97.B6.E4.BD.BF.E7.94.A8.E4.BB.A3.E7.90.86.E9.94.AE">
        何时使用代理键
       </span>
      </h4>
      <p>
       以中国大陆的十八位
       <a href="http://zh.wikipedia.org/wiki/%E4%B8%AD%E5%8D%8E%E4%BA%BA%E6%B0%91%E5%85%B1%E5%92%8C%E5%9B%BD%E5%B1%85%E6%B0%91%E8%BA%AB%E4%BB%BD%E8%AF%81" rel="nofollow noopener noreferrer" target="_blank" title="中华人民共和国居民身份证">
        身份证号
       </a>
       为例，从左往右为六位数字地址码，八位数字出生日期码，三位数字顺序码和一位数字校验码。
      </p>
      <p>
       一家公司想要将它的客户记入数据库，以客户的身份证号作为主键当然是可以的；但是这18位身份证号是用于标识大陆13多亿人口的，一家公司的客户显然没有这么多，所以用18位的数字作为主键有点浪费空间。
      </p>
      <p>
       另外，身份证号中包含了地区、生日信息，若以身份证号为主键，要不要另开字段记录客户的地区、生日也是个问题。如果不另开字段，从主键（身份证号）中提取地区、生日有点麻烦；如果开字段，主键和地区、生日字段的数据存在冗余。
      </p>
      <p>
       所以，一般的做法是，根本不记录客户的身份证号（除非有其他需求），用一个代理键作为主键，另开字段记录客户的地区、生日等信息。
      </p>
      <h3>
       <span class="mw-headline" id=".E8.87.AA.E7.84.B6.E9.94.AE">
        自然键
       </span>
      </h3>
      <p>
       自然键与代理键相反，它是在自然生活中唯一确定一个事物的标识。身份证号就是一个自然键，用于确定一个人。
      </p>

        外部链接

      </h3>
      <ul>
       <li>
        <a class="external text" href="http://structedtext.appspot.com/db/fk.html" rel="nofollow noopener noreferrer" target="_blank">
         一个详细介绍外键的网站
        </a>

      <h2 class="firstHeading" id="firstHeading" lang="en">
       <span dir="auto">
        Natural key
       </span>
      </h2>
      <div id="siteSub">
       From Wikipedia, the free encyclopedia
      </div>
      <div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en">
       <p>
        In
        <a href="http://en.wikipedia.org/wiki/Relational_model" rel="nofollow noopener noreferrer" target="_blank" title="Relational model">
         relational model
        </a>
        <a href="http://en.wikipedia.org/wiki/Database" rel="nofollow noopener noreferrer" target="_blank" title="Database">
         database
        </a>
        design, a
        <strong>
         natural key
        </strong>
        is a key that is formed of attributes that already exist in the real world. For example, a USA citizen's social security number could be used as a natural key. In other words, a natural key is a
        <a href="http://en.wikipedia.org/wiki/Candidate_key" rel="nofollow noopener noreferrer" target="_blank" title="Candidate key">
         candidate key
        </a>
        that has a logical relationship to the attributes within that
        <a href="http://en.wikipedia.org/wiki/Row_%28database%29" rel="nofollow noopener noreferrer" target="_blank" title="Row (database)">
         row
        </a>
        . A natural key is sometimes called
        <em>
         domain key
        </em>
        .
       </p>
       <p>
        The main advantage of a natural key over a
        <a href="http://en.wikipedia.org/wiki/Surrogate_key" rel="nofollow noopener noreferrer" target="_blank" title="Surrogate key">
         surrogate key
        </a>
        , which has no meaning outside the database environment, is that it already exists; there is no need to add a new, artificial
        <a href="http://en.wikipedia.org/wiki/Column_%28database%29" rel="nofollow noopener noreferrer" target="_blank" title="Column (database)">
         column
        </a>
        to the
        <a href="http://en.wikipedia.org/wiki/Logical_schema" rel="nofollow noopener noreferrer" target="_blank" title="Logical schema">
         schema
        </a>
        . Using a natural key (when one can be identified) also simplifies data quality: It ensures that there can only be one row for a key; this "one version of the truth" can be verified, because the natural key is based on a real-world observation.
       </p>
       <p>
        The main disadvantage of choosing a natural key is that its value may change and the relational database engine may not be able to propagate that change across the related
        <a href="http://en.wikipedia.org/wiki/Foreign_key" rel="nofollow noopener noreferrer" target="_blank" title="Foreign key">
         foreign keys
        </a>
        . For example, if
        <tt>
         person_name
        </tt>
        is used as the primary key for the person table, and a person gets married and changes name, then all of the
        <a href="http://en.wikipedia.org/wiki/Multivalued_function" rel="nofollow noopener noreferrer" target="_blank" title="Multivalued function">
         one-to-many
        </a>
        related tables need to be updated also. The secondary disadvantage of choosing a natural key is identifying
        <a href="http://en.wikipedia.org/wiki/Unique_key" rel="nofollow noopener noreferrer" target="_blank" title="Unique key">
         uniqueness
        </a>
        . The primary key must consist of the attributes that uniquely identify a row. However, it may be difficult (or it may add constraints) to create a natural key on a table. For example, if
        <tt>
         person_name
        </tt>
        is used as a primary key for the person table, many persons may share the same name and all but the first entry will be rejected as a duplication. The uniqueness constraint may be overcome by adding an additional column to the primary key, like
        <tt>
         street_address
        </tt>
        , to increase the likelihood of uniqueness.
       </p>
