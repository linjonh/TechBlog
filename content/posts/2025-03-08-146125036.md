    <h2>
     <a id="_1">
     </a>
     需求分析阶段
    </h2>
    <p>
     从软件需求模型向软件架构模型的转换主要关注两个问题：
    </p>
    <ol>
     <li>
      <strong>
       如何根据需求模型构建软件架构模型
      </strong>
     </li>
     <li>
      <strong>
       如何保证模型转换的可追踪性
      </strong>
     </li>
    </ol>
    <h2>
     <a id="_7">
     </a>
     设计阶段
    </h2>
    <ol>
     <li>
      <strong>
       软件架构模型的描述
      </strong>
     </li>
     <li>
      <strong>
       软件架构模型的设计与分析方法
      </strong>
     </li>
     <li>
      <strong>
       对软件架构设计经验的总结与复用
      </strong>
     </li>
    </ol>
    <h3>
     <a id="Software_ArchitectureSA_12">
     </a>
     软件架构模型描述（Software Architecture，SA）
    </h3>
    <h4>
     <a id="SA_13">
     </a>
     SA的基本概念
    </h4>
    <p>
     传统的设计概念只包括构件（组件、模块），以及一些基本的模块互联机制。
     <br/>
     对着研究的深入，构件间的互联机制逐渐独立出来，成为与构件同等级别的实体，称为
     <strong>
      连接子（连接件）
     </strong>
     。
    </p>
    <h4>
     <a id="Architecture_Description_LanguageADL_16">
     </a>
     体系结构描述语言（Architecture Description Language，ADL）
    </h4>
    <p>
     支持构件、连接子及其配置的描述语言，就是如今所说的体系结构描述语言。
    </p>
    <h4>
     <a id="_18">
     </a>
     软件架构模型的多视图表示
    </h4>
    <p>
     从不同的视角描述特定系统的体系结构，从而得到多个视图，并将这些视图组织起来以描述整体的软件架构模型。
    </p>
    <p>
     典型的包括4+1视图。
    </p>
    <table>
     <thead>
      <tr>
       <th>
        名称
       </th>
       <th>
        基本内容
       </th>
       <th>
        图类型
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        逻辑视图（Logical View）
       </td>
       <td>
        设计的对象模型（使用面向对象的设计方法时）。
       </td>
       <td>
        系统架构图
       </td>
      </tr>
      <tr>
       <td>
        过程视图（Process View）
       </td>
       <td>
        捕捉设计的并发和同步特征。
       </td>
       <td>
        时序图
       </td>
      </tr>
      <tr>
       <td>
        物理视图（Physical View）
       </td>
       <td>
        描述了软件到硬件的映射，反映了分布式特性。
       </td>
       <td>
        部署图
       </td>
      </tr>
      <tr>
       <td>
        开发视图（Development View）
       </td>
       <td>
        描述了再开发环境中软件的静态组织结构。
       </td>
       <td>
        类图
       </td>
      </tr>
      <tr>
       <td>
        场景视图（Scenarios View）
       </td>
       <td>
        围绕上述4个视图来组织，然后由一些用例（Use Case）或者场景（Scenarios）来说明，从而形成了第五个视图。
       </td>
       <td>
        用例图
       </td>
      </tr>
     </tbody>
    </table>
    <h2>
     <a id="_32">
     </a>
     实现阶段
    </h2>
    <ol>
     <li>
      研究基于软件架构的开发过程支持，如项目组织结构、配置管理等。
     </li>
     <li>
      寻求从软件架构向实现过渡的途径，如将程序设计语言元素引入软件架构阶段、模型映射、构件组装、复用中间件平台等。
     </li>
     <li>
      研究基于软件架构的测试技术。
     </li>
    </ol>
    <h2>
     <a id="_37">
     </a>
     构件组装阶段
    </h2>
    <ol>
     <li>
      如何支持可复用构件的互联，即对SA设计模型中规约的连接子的实现提供支持。
     </li>
     <li>
      组装过程中，如何检测并消除体系结构失配问题。
     </li>
    </ol>
    <h3>
     <a id="_41">
     </a>
     体系结构失配问题类型
    </h3>
    <ol>
     <li>
      由
      <strong>
       构件
      </strong>
      引起的失配，包括由于系统对构件基础设施、构件控制模型和构件数据模型的假设存在冲突引起的失配。
     </li>
     <li>
      由
      <strong>
       连接子
      </strong>
      引起的失配，包括由于系统对构件交互协议、连接子数据模型的假设存在冲突引起的失配。
     </li>
     <li>
      由于
      <strong>
       系统成分对全局体系结构的假设存在冲突
      </strong>
      引起的失配等。
      <ol>
       <li>
        架构设计中对业务的理解，同实际情况存在偏差。
       </li>
      </ol>
     </li>
    </ol>
    <p>
     要解决失配问题，首先要检测出失配问题，并在此基础上通过适当的手段消除检测出的失配问题。
    </p>
    <h2>
     <a id="_48">
     </a>
     部署阶段
    </h2>
    <p>
     部署阶段的软件架构对软件部署作用如下：
    </p>
    <ol>
     <li>
      提供高层的体系结构视图描述部署阶段的软硬件模型（部署图）。
     </li>
     <li>
      基于软件架构模型可以分析部署方案的质量属性，从而选择合理的部署方案。
     </li>
    </ol>
    <h2>
     <a id="_54">
     </a>
     后开发阶段
    </h2>
    <p>
     后开发阶段是指软甲部署安装之后的阶段。
     <br/>
     这一阶段的软件架构研究主要围绕维护、演化、复用等方面来进行。
     <br/>
     典型的研究方向包括动态软件体系结构、体系结构恢复与重建等。
    </p>
