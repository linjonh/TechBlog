    <h2 id="%E4%B8%80%E3%80%81%E7%B4%A2%E5%BC%95" name="%E4%B8%80%E3%80%81%E7%B4%A2%E5%BC%95">
     一、索引
    </h2>
    <p>
     索引是数据库中用于加快数据检索速度的数据结构，类似于书籍的目录。它通过建立特字段的引用，帮助数据库快速定位，而不必扫描整个表。
    </p>
    <h4 id="%E7%B4%A2%E5%BC%95%E7%9A%84%E4%B8%BB%E8%A6%81%E4%BD%9C%E7%94%A8" name="%E7%B4%A2%E5%BC%95%E7%9A%84%E4%B8%BB%E8%A6%81%E4%BD%9C%E7%94%A8">
     索引的主要作用
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        加速查询
       </strong>
       ：显著提高
       <code>
        SELECT
       </code>
       查询的速度。
      </p>
     </li>
     <li>
      <p>
       <strong>
        加速排序和分组
       </strong>
       ：提升
       <code>
        ORDER BY
       </code>
       和
       <code>
        GROUP BY
       </code>
       操作的效率。
      </p>
     </li>
     <li>
      <p>
       <strong>
        保证唯一性
       </strong>
       ：唯一索引确保某列的值不重复。
      </p>
     </li>
     <li>
      <p>
       <strong>
        加速连接操作
       </strong>
       ：提高表连接（
       <code>
        JOIN
       </code>
       ）的速度。
      </p>
     </li>
    </ol>
    <h4 id="%E5%B8%B8%E8%A7%81%E7%9A%84%E7%B4%A2%E5%BC%95%E7%B1%BB%E5%9E%8B" name="%E5%B8%B8%E8%A7%81%E7%9A%84%E7%B4%A2%E5%BC%95%E7%B1%BB%E5%9E%8B">
     常见的索引类型
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        单列索引
       </strong>
       ：基于单个列创建。
      </p>
      <pre>CREATE INDEX idx_name ON table_name (column_name);</pre>
     </li>
     <li>
      <p>
       <strong>
        复合索引
       </strong>
       ：基于多个列创建。
      </p>
      <pre>CREATE INDEX idx_name ON table_name (column1, column2);</pre>
     </li>
     <li>
      <p>
       <strong>
        唯一索引
       </strong>
       ：确保列中的值唯一。
      </p>
      <pre>CREATE UNIQUE INDEX idx_name ON table_name (column_name);</pre>
     </li>
     <li>
      <p>
       <strong>
        主键索引
       </strong>
       ：特殊的唯一索引，不允许
       <code>
        NULL
       </code>
       值。
      </p>
      <pre>ALTER TABLE table_name ADD PRIMARY KEY (column_name);</pre>
     </li>
     <li>
      <p>
       <strong>
        全文索引
       </strong>
       ：用于全文搜索，适用于文本数据。
      </p>
      <pre>CREATE FULLTEXT INDEX idx_name ON table_name (column_name);</pre>
     </li>
     <li>
      <p>
       <strong>
        空间索引
       </strong>
       ：用于地理空间数据。
      </p>
      <pre>CREATE SPATIAL INDEX idx_name ON table_name (column_name);</pre>
     </li>
    </ol>
    <h4 id="%E7%B4%A2%E5%BC%95%E7%9A%84%E4%BC%98%E7%BC%BA%E7%82%B9" name="%E7%B4%A2%E5%BC%95%E7%9A%84%E4%BC%98%E7%BC%BA%E7%82%B9">
     索引的优缺点
    </h4>
    <p>
     <strong>
      优点
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       显著提高查询速度。
      </p>
     </li>
     <li>
      <p>
       加速排序和分组操作。
      </p>
     </li>
     <li>
      <p>
       保证数据唯一性。
      </p>
     </li>
    </ul>
    <p>
     <strong>
      缺点
     </strong>
     ：
    </p>
    <ul>
     <li>
      <p>
       占用额外存储空间。
      </p>
     </li>
     <li>
      <p>
       降低数据插入、更新、删除的速度，因为索引需要同步更新
      </p>
     </li>
    </ul>
    <h3 id="%E7%B4%A2%E5%BC%95%E5%85%B7%E4%BD%93%E5%A6%82%E4%BD%95%E5%B7%A5%E4%BD%9C%E7%9A%84%EF%BC%9F%E4%B8%BA%E4%BB%80%E4%B9%88%E5%8A%A0%E4%BA%86%E7%B4%A2%E5%BC%95%E5%90%8E%E5%AF%B9%E6%95%B0%E6%8D%AE%E7%9A%84%E6%93%8D%E4%BD%9C%E5%8F%98%E5%BF%AB%E4%BA%86%EF%BC%9F" name="%E7%B4%A2%E5%BC%95%E5%85%B7%E4%BD%93%E5%A6%82%E4%BD%95%E5%B7%A5%E4%BD%9C%E7%9A%84%EF%BC%9F%E4%B8%BA%E4%BB%80%E4%B9%88%E5%8A%A0%E4%BA%86%E7%B4%A2%E5%BC%95%E5%90%8E%E5%AF%B9%E6%95%B0%E6%8D%AE%E7%9A%84%E6%93%8D%E4%BD%9C%E5%8F%98%E5%BF%AB%E4%BA%86%EF%BC%9F">
     索引具体如何工作的？为什么加了索引后对数据的操作变快了？
    </h3>
    <p>
    </p>
    <h4 id="%E5%8E%9F%E6%9C%AC%E7%9A%84%E5%AD%98%E5%82%A8%E6%96%B9%E5%BC%8F%EF%BC%88%E6%97%A0%E7%B4%A2%E5%BC%95%EF%BC%89" name="%E5%8E%9F%E6%9C%AC%E7%9A%84%E5%AD%98%E5%82%A8%E6%96%B9%E5%BC%8F%EF%BC%88%E6%97%A0%E7%B4%A2%E5%BC%95%EF%BC%89">
     原本的存储方式（无索引）
    </h4>
    <p>
     在没有索引的情况下，数据库通常以
     <strong>
      堆表（Heap Table）
     </strong>
     的方式存储数据。数据行按插入顺序存储，查询时需要通过
     <strong>
      全表扫描（Full Table Scan）
     </strong>
     逐行查找匹配的数据。
    </p>
    <h5 id="%E5%AD%98%E5%82%A8%E7%89%B9%E7%82%B9%EF%BC%9A" name="%E5%AD%98%E5%82%A8%E7%89%B9%E7%82%B9%EF%BC%9A">
     存储特点：
    </h5>
    <ol>
     <li>
      <p>
       <strong>
        无序存储
       </strong>
       ：数据按插入顺序存储，物理上没有特定的顺序。
      </p>
     </li>
     <li>
      <p>
       <strong>
        全表扫描
       </strong>
       ：查询时，数据库需要逐行扫描整个表，直到找到匹配的数据。
      </p>
     </li>
     <li>
      <p>
       <strong>
        时间复杂度
       </strong>
       ：查找操作的时间复杂度为
       <strong>
        O(n)
       </strong>
       ，其中 n 是表中的行数。当数据量很大时，查询效率会显著下降。
      </p>
     </li>
    </ol>
    <h5 id="%E7%A4%BA%E4%BE%8B%EF%BC%9A" name="%E7%A4%BA%E4%BE%8B%EF%BC%9A">
     示例：
    </h5>
    <p>
     假设有一个
     <code>
      users
     </code>
     表，包含以下数据：
    </p>
    <table>
     <thead>
      <tr>
       <th>
        id
       </th>
       <th>
        name
       </th>
       <th>
        age
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        1
       </td>
       <td>
        Alice
       </td>
       <td>
        25
       </td>
      </tr>
      <tr>
       <td>
        2
       </td>
       <td>
        Bob
       </td>
       <td>
        30
       </td>
      </tr>
      <tr>
       <td>
        3
       </td>
       <td>
        Charlie
       </td>
       <td>
        22
       </td>
      </tr>
      <tr>
       <td>
        4
       </td>
       <td>
        David
       </td>
       <td>
        28
       </td>
      </tr>
     </tbody>
    </table>
    <p>
     如果执行以下查询：
    </p>
    <pre>SELECT * FROM users WHERE name = 'Bob';</pre>
    <p>
     数据库需要逐行扫描整个表，直到找到
     <code>
      name = 'Bob'
     </code>
     的行。
    </p>
    <hr/>
    <h4 id="%E5%8A%A0%E4%BA%86%E7%B4%A2%E5%BC%95%E4%B9%8B%E5%90%8E%E7%9A%84%E5%AD%98%E5%82%A8%E6%96%B9%E5%BC%8F" name="%E5%8A%A0%E4%BA%86%E7%B4%A2%E5%BC%95%E4%B9%8B%E5%90%8E%E7%9A%84%E5%AD%98%E5%82%A8%E6%96%B9%E5%BC%8F">
     加了索引之后的存储方式
    </h4>
    <p>
     索引是一种
     <strong>
      数据结构
     </strong>
     ，它通过额外的存储空间来维护表中某些列的快速查找路径。常见的索引类型是
     <strong>
      B+树索引
     </strong>
     。
    </p>
    <h5 id="%E5%AD%98%E5%82%A8%E7%89%B9%E7%82%B9%EF%BC%9A" name="%E5%AD%98%E5%82%A8%E7%89%B9%E7%82%B9%EF%BC%9A">
     存储特点：
    </h5>
    <ol>
     <li>
      <p>
       <strong>
        有序存储
       </strong>
       ：索引会根据索引列的值对数据进行排序，并建立一种高效的查找结构（如 B+树）。
      </p>
     </li>
     <li>
      <p>
       <strong>
        快速查找
       </strong>
       ：查询时，数据库可以通过索引快速定位到目标数据，而不需要扫描整个表。
      </p>
     </li>
     <li>
      <p>
       <strong>
        时间复杂度
       </strong>
       ：查找操作的时间复杂度为
       <strong>
        O(log n)
       </strong>
       ，其中 n 是表中的行数。对于大数据集，索引可以显著提高查询效率。
      </p>
     </li>
    </ol>
    <h5 id="%E7%A4%BA%E4%BE%8B%EF%BC%9A" name="%E7%A4%BA%E4%BE%8B%EF%BC%9A">
     示例：
    </h5>
    <p>
     在
     <code>
      users
     </code>
     表的
     <code>
      name
     </code>
     列上创建索引：
    </p>
    <pre>CREATE INDEX idx_name ON users (name);</pre>
    <p>
     索引会生成一个类似以下的结构（以 B+树为例）：
    </p>
    <pre>Root
├── Alice
├── Bob
├── Charlie
└── David</pre>
    <p>
     当执行以下查询时：
    </p>
    <pre>SELECT * FROM users WHERE name = 'Bob';</pre>
    <p>
     数据库会使用索引
     <code>
      idx_name
     </code>
     快速定位到
     <code>
      name = 'Bob'
     </code>
     的行，而不需要扫描整个表。
    </p>
    <hr/>
    <h4 id="%E4%B8%BA%E4%BB%80%E4%B9%88%E5%8A%A0%E4%BA%86%E7%B4%A2%E5%BC%95%E4%B9%8B%E5%90%8E%E5%8F%98%E5%BF%AB%E4%BA%86%EF%BC%9F" name="%E4%B8%BA%E4%BB%80%E4%B9%88%E5%8A%A0%E4%BA%86%E7%B4%A2%E5%BC%95%E4%B9%8B%E5%90%8E%E5%8F%98%E5%BF%AB%E4%BA%86%EF%BC%9F">
     为什么加了索引之后变快了？
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        减少数据扫描量
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         无索引时，数据库需要扫描整个表（全表扫描）。
        </p>
       </li>
       <li>
        <p>
         有索引时，数据库只需扫描索引结构，快速定位目标数据。
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        高效的数据结构
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         索引通常使用 B+树等高效数据结构，支持快速查找、插入和删除操作。
        </p>
       </li>
       <li>
        <p>
         B+树的时间复杂度为 O(log n)，远低于全表扫描的 O(n)。
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        排序和范围查询优化
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         索引对数据进行了排序，支持高效的等值查询和范围查询。
        </p>
       </li>
       <li>
        <p>
         例如，查询
         <code>
          WHERE age BETWEEN 20 AND 30
         </code>
         时，索引可以快速定位到范围内的数据。
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        减少磁盘 I/O
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         索引通常比表数据小，可以加载到内存中，减少磁盘 I/O 操作。
        </p>
       </li>
       <li>
        <p>
         无索引时，数据库可能需要多次读取磁盘来扫描整个表。
        </p>
       </li>
      </ul>
     </li>
    </ul>
    <h2 id="%E4%BA%8C%E3%80%81%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8" name="%E4%BA%8C%E3%80%81%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8">
     二、事务数据字典
    </h2>
    <p>
     事务数据字典是数据库管理系统（DBMS）中用于存储和管理元数据（Metadata）的系统表或数据结构。元数据是描述数据库对象（如表、索引、列、约束等）的信息。事务数据字典确保这些元数据在事务处理过程中保持一致性和完整性。
    </p>
    <h4 id="%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8%E7%9A%84%E4%B8%BB%E8%A6%81%E5%8A%9F%E8%83%BD" name="%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8%E7%9A%84%E4%B8%BB%E8%A6%81%E5%8A%9F%E8%83%BD">
     事务数据字典的主要功能
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        存储元数据
       </strong>
       ：保存数据库对象的定义和结构信息。
      </p>
     </li>
     <li>
      <p>
       <strong>
        支持事务
       </strong>
       ：确保元数据操作（如创建、修改、删除表）具有事务性，要么全部成功，要么全部回滚。（原子性）
      </p>
     </li>
     <li>
      <p>
       <strong>
        一致性维护
       </strong>
       ：保证元数据与数据的一致性。
      </p>
     </li>
     <li>
      <p>
       <strong>
        权限管理
       </strong>
       ：存储用户权限和角色信息，支持权限管理。
      </p>
     </li>
    </ol>
    <h4 id="%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8%E7%9A%84%E5%B8%B8%E8%A7%81%E5%86%85%E5%AE%B9" name="%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8%E7%9A%84%E5%B8%B8%E8%A7%81%E5%86%85%E5%AE%B9">
     事务数据字典的常见内容
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        表信息
       </strong>
       ：表名、列名、数据类型、约束等。
      </p>
     </li>
     <li>
      <p>
       <strong>
        索引信息
       </strong>
       ：索引名、索引类型、索引列等。
      </p>
     </li>
     <li>
      <p>
       <strong>
        视图信息
       </strong>
       ：视图定义、依赖的表和列等。
      </p>
     </li>
     <li>
      <p>
       <strong>
        约束信息
       </strong>
       ：主键、外键、唯一约束、检查约束等。
      </p>
     </li>
     <li>
      <p>
       <strong>
        用户和权限
       </strong>
       ：用户账号、角色、权限分配等。
      </p>
     </li>
     <li>
      <p>
       <strong>
        存储信息
       </strong>
       ：表空间、文件路径、存储参数等。
      </p>
     </li>
    </ol>
    <h4 id="%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8%E7%9A%84%E5%AE%9E%E7%8E%B0" name="%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8%E7%9A%84%E5%AE%9E%E7%8E%B0">
     事务数据字典的实现
    </h4>
    <p>
     不同数据库管理系统有各自的实现方式，常见的有：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        系统表
       </strong>
       ：如 MySQL 的
       <code>
        information_schema
       </code>
       和
       <code>
        mysql
       </code>
       数据库。
      </p>
     </li>
     <li>
      <p>
       <strong>
        数据字典表
       </strong>
       ：如 Oracle 的
       <code>
        DBA_*
       </code>
       、
       <code>
        ALL_*
       </code>
       、
       <code>
        USER_*
       </code>
       视图。
      </p>
     </li>
     <li>
      <p>
       <strong>
        内存结构
       </strong>
       ：部分数据库将数据字典信息加载到内存以提高访问速度。
      </p>
     </li>
    </ol>
    <h4 id="%E7%A4%BA%E4%BE%8B%EF%BC%9AMySQL%20%E4%B8%AD%E7%9A%84%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8" name="%E7%A4%BA%E4%BE%8B%EF%BC%9AMySQL%20%E4%B8%AD%E7%9A%84%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8">
     示例：MySQL 中的事务数据字典
    </h4>
    <p>
     MySQL 8.0 引入了事务性数据字典，将元数据存储在 InnoDB 表中，确保元数据操作具有事务性。
    </p>
    <ol>
     <li>
      <p>
       <strong>
        查看表信息
       </strong>
       ：
      </p>
      <pre>SELECT * FROM information_schema.TABLES WHERE table_schema = 'your_database_name';</pre>
     </li>
     <li>
      <p>
       <strong>
        查看列信息
       </strong>
       ：
      </p>
      <pre>SELECT * FROM information_schema.COLUMNS WHERE table_schema = 'your_database_name' AND table_name = 'your_table_name';</pre>
     </li>
     <li>
      <p>
       <strong>
        查看索引信息
       </strong>
       ：
      </p>
      <pre>SELECT * FROM information_schema.STATISTICS WHERE table_schema = 'your_database_name' AND table_name = 'your_table_name';</pre>
     </li>
    </ol>
    <h4 id="%E7%A4%BA%E4%BE%8B%EF%BC%9AOracle%20%E4%B8%AD%E7%9A%84%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8" name="%E7%A4%BA%E4%BE%8B%EF%BC%9AOracle%20%E4%B8%AD%E7%9A%84%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8">
     示例：Oracle 中的事务数据字典
    </h4>
    <p>
     Oracle 使用数据字典视图来管理元数据。
    </p>
    <ol>
     <li>
      <p>
       <strong>
        查看表信息
       </strong>
       ：
      </p>
      <pre>SELECT * FROM DBA_TABLES WHERE owner = 'YOUR_SCHEMA';</pre>
     </li>
     <li>
      <p>
       <strong>
        查看列信息
       </strong>
       ：
      </p>
      <pre>SELECT * FROM DBA_TAB_COLUMNS WHERE owner = 'YOUR_SCHEMA' AND table_name = 'YOUR_TABLE';</pre>
     </li>
     <li>
      <p>
       <strong>
        查看索引信息
       </strong>
       ：
      </p>
      <pre>SELECT * FROM DBA_INDEXES WHERE owner = 'YOUR_SCHEMA' AND table_name = 'YOUR_TABLE';</pre>
     </li>
    </ol>
    <h4 id="%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8%E7%9A%84%E4%BC%98%E5%8A%BF" name="%E4%BA%8B%E5%8A%A1%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8%E7%9A%84%E4%BC%98%E5%8A%BF">
     事务数据字典的优势
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        事务性
       </strong>
       ：确保元数据操作具有原子性、一致性、隔离性和持久性（ACID）。
      </p>
     </li>
     <li>
      <p>
       <strong>
        一致性
       </strong>
       ：保证元数据与数据的一致性。
      </p>
     </li>
     <li>
      <p>
       <strong>
        安全性
       </strong>
       ：通过权限管理保护元数据。
      </p>
     </li>
    </ol>
    <p>
    </p>
    <h2 id="%E4%B8%89%E3%80%81%E5%85%83%E6%95%B0%E6%8D%AE" name="%E4%B8%89%E3%80%81%E5%85%83%E6%95%B0%E6%8D%AE">
     三、元数据
    </h2>
    <p>
     元数据是有关存储在RDBMS中的数据的信息，如列定义、索引定义、约束等。
    </p>
    <p>
     数据字典是RDBMS中所有的数据的元数据集中资料档案库。
    </p>
    <h2 id="%E5%9B%9B%E3%80%81%E5%8D%95%E8%A1%A8%E6%96%87%E4%BB%B6%E8%A1%A8%E7%A9%BA%E9%97%B4%E5%92%8C%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4" name="%E5%9B%9B%E3%80%81%E5%8D%95%E8%A1%A8%E6%96%87%E4%BB%B6%E8%A1%A8%E7%A9%BA%E9%97%B4%E5%92%8C%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4">
     四、单表文件表空间和通用表空间
    </h2>
    <p>
     在 MySQL 中，表空间（Tablespace）是用于存储表数据和索引的物理文件。MySQL 支持多种表空间类型，其中
     <strong>
      单表文件表空间
     </strong>
     和
     <strong>
      通用表空间
     </strong>
     是两种常见的表空间管理方式。它们的主要区别在于如何组织和管理表的存储。
    </p>
    <hr/>
    <h4 id="1.%20%E5%8D%95%E8%A1%A8%E6%96%87%E4%BB%B6%E8%A1%A8%E7%A9%BA%E9%97%B4%EF%BC%88File-Per-Table%20Tablespace%EF%BC%89" name="1.%20%E5%8D%95%E8%A1%A8%E6%96%87%E4%BB%B6%E8%A1%A8%E7%A9%BA%E9%97%B4%EF%BC%88File-Per-Table%20Tablespace%EF%BC%89">
     1. 单表文件表空间（File-Per-Table Tablespace）
    </h4>
    <p>
     单表文件表空间是 MySQL 的默认表空间类型（InnoDB 存储引擎）。每个表都有一个独立的
     <code>
      .ibd
     </code>
     文件来存储表的数据和索引。
    </p>
    <h5 id="%E7%89%B9%E7%82%B9%EF%BC%9A" name="%E7%89%B9%E7%82%B9%EF%BC%9A">
     特点：
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        每个表一个文件
       </strong>
       ：每个表的数据和索引存储在一个独立的
       <code>
        .ibd
       </code>
       文件中。
      </p>
     </li>
     <li>
      <p>
       <strong>
        文件位置
       </strong>
       ：默认情况下，
       <code>
        .ibd
       </code>
       文件位于数据库目录下（如
       <code>
        /var/lib/mysql/database_name/
       </code>
       ）。
      </p>
     </li>
     <li>
      <p>
       <strong>
        灵活性
       </strong>
       ：可以单独管理每个表的文件，例如移动、备份或删除。
      </p>
     </li>
     <li>
      <p>
       <strong>
        空间回收
       </strong>
       ：删除表时，对应的
       <code>
        .ibd
       </code>
       文件会被删除，空间会立即释放。
      </p>
     </li>
    </ul>
    <h5 id="%E5%90%AF%E7%94%A8%E6%96%B9%E5%BC%8F%EF%BC%9A" name="%E5%90%AF%E7%94%A8%E6%96%B9%E5%BC%8F%EF%BC%9A">
     启用方式：
    </h5>
    <p>
     单表文件表空间是 MySQL 5.6 及以上版本的默认行为。可以通过以下配置确认：
    </p>
    <pre>SHOW VARIABLES LIKE 'innodb_file_per_table';</pre>
    <p>
     如果值为
     <code>
      ON
     </code>
     ，则表示启用单表文件表空间。
    </p>
    <h5 id="%E4%BC%98%E7%82%B9%EF%BC%9A" name="%E4%BC%98%E7%82%B9%EF%BC%9A">
     优点：
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        独立性
       </strong>
       ：每个表的文件独立，便于管理和维护。
      </p>
     </li>
     <li>
      <p>
       <strong>
        空间回收
       </strong>
       ：删除表时，空间会立即释放。
      </p>
     </li>
     <li>
      <p>
       <strong>
        备份灵活
       </strong>
       ：可以单独备份或恢复某个表。
      </p>
     </li>
    </ul>
    <h5 id="%E7%BC%BA%E7%82%B9%EF%BC%9A" name="%E7%BC%BA%E7%82%B9%EF%BC%9A">
     缺点：
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        文件数量多
       </strong>
       ：如果表数量很多，会导致文件数量过多，可能影响文件系统的性能。
      </p>
     </li>
     <li>
      <p>
       <strong>
        空间浪费
       </strong>
       ：每个表都有自己的文件，可能导致一定的空间浪费（如文件系统的最小分配单元）。
      </p>
     </li>
    </ul>
    <h5 id="%E7%A4%BA%E4%BE%8B%EF%BC%9A" name="%E7%A4%BA%E4%BE%8B%EF%BC%9A">
     示例：
    </h5>
    <p>
     创建一个使用单表文件表空间的表：
    </p>
    <pre>CREATE TABLE example_table (
    id INT PRIMARY KEY,
    name VARCHAR(100)
) ENGINE=InnoDB;</pre>
    <p>
     对应的
     <code>
      .ibd
     </code>
     文件会存储在数据库目录下，如：
    </p>
    <pre>/var/lib/mysql/database_name/example_table.ibd</pre>
    <hr/>
    <h4 id="2.%20%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4%EF%BC%88General%20Tablespace%EF%BC%89" name="2.%20%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4%EF%BC%88General%20Tablespace%EF%BC%89">
     2. 通用表空间（General Tablespace）
    </h4>
    <p>
     通用表空间是 MySQL 5.7 引入的一种表空间类型。它允许将多个表的数据和索引存储在同一个表空间文件中。
    </p>
    <h5 id="%E7%89%B9%E7%82%B9%EF%BC%9A" name="%E7%89%B9%E7%82%B9%EF%BC%9A">
     特点：
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        共享文件
       </strong>
       ：多个表可以共享同一个表空间文件。
      </p>
     </li>
     <li>
      <p>
       <strong>
        文件位置
       </strong>
       ：通用表空间文件可以存储在任意位置，由用户指定。
      </p>
     </li>
     <li>
      <p>
       <strong>
        灵活性
       </strong>
       ：可以将多个表的数据集中存储，减少文件数量。
      </p>
     </li>
     <li>
      <p>
       <strong>
        空间管理
       </strong>
       ：删除表时，空间不会立即释放，而是可以供其他表复用。
      </p>
     </li>
    </ul>
    <h5 id="%E5%88%9B%E5%BB%BA%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4%EF%BC%9A" name="%E5%88%9B%E5%BB%BA%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4%EF%BC%9A">
     创建通用表空间：
    </h5>
    <pre>CREATE TABLESPACE general_ts
ADD DATAFILE 'general_ts.ibd'
ENGINE=InnoDB;</pre>
    <h5 id="%E5%B0%86%E8%A1%A8%E6%B7%BB%E5%8A%A0%E5%88%B0%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4%EF%BC%9A" name="%E5%B0%86%E8%A1%A8%E6%B7%BB%E5%8A%A0%E5%88%B0%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4%EF%BC%9A">
     将表添加到通用表空间：
    </h5>
    <pre>CREATE TABLE example_table (
    id INT PRIMARY KEY,
    name VARCHAR(100)
) TABLESPACE general_ts;</pre>
    <h5 id="%E4%BC%98%E7%82%B9%EF%BC%9A" name="%E4%BC%98%E7%82%B9%EF%BC%9A">
     优点：
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        减少文件数量
       </strong>
       ：多个表共享一个表空间文件，减少文件系统压力。
      </p>
     </li>
     <li>
      <p>
       <strong>
        集中管理
       </strong>
       ：可以集中管理多个表的存储。
      </p>
     </li>
     <li>
      <p>
       <strong>
        灵活存储
       </strong>
       ：表空间文件可以存储在任意位置。
      </p>
     </li>
    </ul>
    <h5 id="%E7%BC%BA%E7%82%B9%EF%BC%9A" name="%E7%BC%BA%E7%82%B9%EF%BC%9A">
     缺点：
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        空间回收
       </strong>
       ：删除表时，空间不会立即释放，而是保留在表空间文件中。
      </p>
     </li>
     <li>
      <p>
       <strong>
        管理复杂
       </strong>
       ：需要手动管理表空间文件。
      </p>
     </li>
    </ul>
    <h5 id="%E7%A4%BA%E4%BE%8B%EF%BC%9A" name="%E7%A4%BA%E4%BE%8B%EF%BC%9A">
     示例：
    </h5>
    <ol>
     <li>
      <p>
       创建通用表空间：
      </p>
      <pre>CREATE TABLESPACE general_ts
ADD DATAFILE '/path/to/general_ts.ibd'
ENGINE=InnoDB;</pre>
     </li>
     <li>
      <p>
       创建表并指定通用表空间：
      </p>
      <pre>CREATE TABLE example_table (
    id INT PRIMARY KEY,
    name VARCHAR(100)
TABLESPACE general_ts;</pre>
     </li>
     <li>
      <p>
       查看表空间信息：
      </p>
      <pre>SELECT * FROM INFORMATION_SCHEMA.INNODB_TABLESPACES;</pre>
     </li>
    </ol>
    <hr/>
    <h4 id="%E5%8D%95%E8%A1%A8%E6%96%87%E4%BB%B6%E8%A1%A8%E7%A9%BA%E9%97%B4%20vs%20%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4" name="%E5%8D%95%E8%A1%A8%E6%96%87%E4%BB%B6%E8%A1%A8%E7%A9%BA%E9%97%B4%20vs%20%E9%80%9A%E7%94%A8%E8%A1%A8%E7%A9%BA%E9%97%B4">
     单表文件表空间 vs 通用表空间
    </h4>
    <table>
     <thead>
      <tr>
       <th>
        特性
       </th>
       <th>
        单表文件表空间
       </th>
       <th>
        通用表空间
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         文件数量
        </strong>
       </td>
       <td>
        每个表一个文件
       </td>
       <td>
        多个表共享一个文件
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         空间回收
        </strong>
       </td>
       <td>
        删除表时立即释放空间
       </td>
       <td>
        删除表时空间不立即释放
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         管理灵活性
        </strong>
       </td>
       <td>
        每个表独立管理
       </td>
       <td>
        集中管理多个表
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         文件位置
        </strong>
       </td>
       <td>
        默认在数据库目录下
       </td>
       <td>
        可指定任意位置
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         适用场景
        </strong>
       </td>
       <td>
        表数量较少，需要独立管理的场景
       </td>

       </td>
      </tr>
     </tbody>

    <h2 id="%C2%A0%E4%BA%94%E3%80%81MVCC" name="%C2%A0%E4%BA%94%E3%80%81MVCC">
     五、MVCC
    </h2>
    <p>
     MVCC(多版本并发控制)是一种用于数据库管理系统的并发控制机制，旨在提高数据库的并发性能，同时避免读写冲突。MVCC通过维护数据的多个版本来实现非阻塞的读操作，从而允许多个事务并发执行不会相互阻塞。
    </p>
    <h4 id="MVCC%20%E7%9A%84%E6%A0%B8%E5%BF%83%E6%80%9D%E6%83%B3" name="MVCC%20%E7%9A%84%E6%A0%B8%E5%BF%83%E6%80%9D%E6%83%B3">
     MVCC 的核心思想
    </h4>
    <p>
     MVCC 的核心思想是为每个事务提供数据的“快照”（Snapshot），使得每个事务在读取数据时看到的是事务开始时的数据状态，而不受其他并发事务的影响。这样，读操作不会阻塞写操作，写操作也不会阻塞读操作，从而提高了并发性能。
    </p>
    <h4 id="MVCC%20%E7%9A%84%E5%B7%A5%E4%BD%9C%E5%8E%9F%E7%90%86" name="MVCC%20%E7%9A%84%E5%B7%A5%E4%BD%9C%E5%8E%9F%E7%90%86">
     MVCC 的工作原理
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        版本链
       </strong>
       ：每行数据可能有多个版本，每个版本包含一个时间戳或事务 ID，用于标识该版本的创建时间和有效时间。
      </p>
     </li>
     <li>
      <p>
       <strong>
        读操作
       </strong>
       ：事务读取数据时，系统会根据事务的开始时间或事务 ID 选择合适的版本，确保事务看到一致的数据视图。
      </p>
     </li>
     <li>
      <p>
       <strong>
        写操作
       </strong>
       ：事务修改数据时，系统会创建数据的新版本，而不是直接覆盖旧版本。旧版本仍然保留，供其他事务读取。
      </p>
     </li>
     <li>
      <p>
       <strong>
        垃圾回收
       </strong>
       ：当某个数据版本不再被任何事务需要时，系统会将其标记为可回收，并在适当的时候进行清理。
      </p>
     </li>
    </ul>
    <h4 id="MVCC%20%E7%9A%84%E4%BC%98%E7%82%B9" name="MVCC%20%E7%9A%84%E4%BC%98%E7%82%B9">
     MVCC 的优点
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        高并发
       </strong>
       ：读操作不会阻塞写操作，写操作也不会阻塞读操作，从而提高了数据库的并发性能。
      </p>
     </li>
     <li>
      <p>
       <strong>
        一致性
       </strong>
       ：每个事务看到的数据视图是一致的，避免了脏读、不可重复读和幻读等问题。
      </p>
     </li>
     <li>
      <p>
       <strong>
        非阻塞
       </strong>
       ：读操作不需要加锁，减少了锁争用和死锁的可能性。
      </p>
     </li>
    </ul>
    <h4 id="MVCC%20%E7%9A%84%E7%BC%BA%E7%82%B9" name="MVCC%20%E7%9A%84%E7%BC%BA%E7%82%B9">
     MVCC 的缺点
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        存储开销
       </strong>
       ：需要维护数据的多个版本，增加了存储空间的开销。
      </p>
     </li>
     <li>
      <p>
       <strong>
        垃圾回收
       </strong>
       ：需要定期清理不再使用的数据版本，增加了系统的复杂性。
      </p>
     </li>
     <li>
      <p>
       <strong>
        写冲突
       </strong>
       ：虽然读操作不会阻塞写操作，但写操作之间仍然可能发生冲突，需要进行冲突检测和解决。
      </p>
     </li>
    </ul>
    <h4 id="MVCC%20%E7%9A%84%E5%AE%9E%E7%8E%B0" name="MVCC%20%E7%9A%84%E5%AE%9E%E7%8E%B0">
     MVCC 的实现
    </h4>
    <p>
     不同的数据库管理系统对 MVCC 的实现方式有所不同。以下是一些常见的实现方式：
    </p>
    <p>
    </p>
    <ol>
     <li>
      <p>
       <strong>
        MySQL（InnoDB）
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         使用回滚段（Undo Log）来存储数据的历史版本。
        </p>
       </li>
       <li>
        <p>
         每个事务在修改数据时，会创建一个新的版本，并将旧版本存储在回滚段中。
        </p>
       </li>
       <li>
        <p>
         通过事务 ID 和回滚指针（Rollback Pointer）来管理数据版本。
        </p>
       </li>
      </ul>
     </li>
    </ol>
    <h4 id="%E7%A4%BA%E4%BE%8B%EF%BC%9AMYSQL%20%E4%B8%AD%E7%9A%84%20MVCC" name="%E7%A4%BA%E4%BE%8B%EF%BC%9AMYSQL%20%E4%B8%AD%E7%9A%84%20MVCC">

    <h2 id="%E5%85%AD%E3%80%81%E9%87%8D%E5%81%9A%E6%97%A5%E5%BF%97%E5%92%8C%E8%BF%98%E5%8E%9F%E6%97%A5%E5%BF%97" name="%E5%85%AD%E3%80%81%E9%87%8D%E5%81%9A%E6%97%A5%E5%BF%97%E5%92%8C%E8%BF%98%E5%8E%9F%E6%97%A5%E5%BF%97">
     六、重做日志和还原日志
    </h2>
    <p>
     作用：确保数据的一致性和事务的完整性
    </p>
    <p>
     主要作用如下：
    </p>
    <h4 id="1.%C2%A0%E9%87%8D%E5%81%9A%E6%97%A5%E5%BF%97%EF%BC%88Redo%20Log%EF%BC%89" name="1.%C2%A0%E9%87%8D%E5%81%9A%E6%97%A5%E5%BF%97%EF%BC%88Redo%20Log%EF%BC%89">
     1.
     <strong>
      重做日志（Redo Log）
     </strong>
    </h4>
    <ul>
     <li>
      <strong>
       作用
      </strong>
      ：重做日志用于记录对数据库的所有
      <span style="background-color:#fbd4d0">
       修改操作（如插入、更新、删除）
      </span>
      ，以便在系统崩溃或故障后能够重新应用这些操作，恢复数据库到故障前的状态。
     </li>
     <li>
      <strong>
       工作原理
      </strong>
      ：
      <ul>
       <li>
        当事务提交时，系统会先将修改操作写入重做日志，然后再将数据写入磁盘。
       </li>
       <li>
        如果系统崩溃，恢复过程会读取重做日志，重新执行其中的操作，确保已提交的事务被持久化。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       特点
      </strong>
      ：
      <ul>
       <li>
        重做日志是顺序写入的，性能较高。
       </li>
       <li>
        主要用于保证数据的持久性（Durability）。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       应用场景
      </strong>
      ：系统崩溃后的恢复、数据备份的恢复等。
     </li>
    </ul>
    <h4 id="2.%C2%A0%E8%BF%98%E5%8E%9F%E6%97%A5%E5%BF%97%EF%BC%88Undo%20Log%EF%BC%89" name="2.%C2%A0%E8%BF%98%E5%8E%9F%E6%97%A5%E5%BF%97%EF%BC%88Undo%20Log%EF%BC%89">
     2.
     <strong>
      还原日志（Undo Log）
     </strong>
    </h4>
    <ul>
     <li>
      <strong>
       作用
      </strong>
      ：还原日志用于记录事务执行前的
      <span style="background-color:#ed7976">
       数据状态
      </span>
      ，以便在事务回滚或系统需要撤销某些操作时，能够将数据恢复到事务开始前的状态。
     </li>
     <li>
      <strong>
       工作原理
      </strong>
      ：
      <ul>
       <li>
        当事务执行修改操作时，系统会先将修改前的数据状态记录到还原日志中。
       </li>
       <li>
        如果事务需要回滚，系统会根据还原日志将数据恢复到事务开始前的状态。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       特点
      </strong>
      ：
      <ul>
       <li>
        还原日志主要用于保证事务的原子性（Atomicity）和一致性（Consistency）。
       </li>
       <li>
        支持事务的回滚操作。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       应用场景
      </strong>
      ：事务回滚、并发控制中的一致性维护等。
     </li>
    </ul>
    <h4 id="3.%C2%A0%E9%87%8D%E5%81%9A%E6%97%A5%E5%BF%97%E4%B8%8E%E8%BF%98%E5%8E%9F%E6%97%A5%E5%BF%97%E7%9A%84%E5%8C%BA%E5%88%AB" name="3.%C2%A0%E9%87%8D%E5%81%9A%E6%97%A5%E5%BF%97%E4%B8%8E%E8%BF%98%E5%8E%9F%E6%97%A5%E5%BF%97%E7%9A%84%E5%8C%BA%E5%88%AB">
     3.
     <strong>
      重做日志与还原日志的区别
     </strong>
    </h4>
    <p>
     表格
    </p>
    <table>
     <thead>
      <tr>
       <th>
        <strong>
         特性
        </strong>
       </th>
       <th>
        <strong>
         重做日志（Redo Log）
        </strong>
       </th>
       <th>
        <strong>
         还原日志（Undo Log）
        </strong>
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         目的
        </strong>
       </td>
       <td>
        确保已提交事务的持久化
       </td>
       <td>
        支持事务回滚和一致性恢复
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         记录内容
        </strong>
       </td>
       <td>
        修改后的数据状态
       </td>
       <td>
        修改前的数据状态
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         使用场景
        </strong>
       </td>
       <td>
        系统崩溃后的恢复
       </td>
       <td>
        事务回滚、并发控制
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         与事务的关系
        </strong>
       </td>
       <td>
        记录已提交事务的修改
       </td>
       <td>
        记录未提交事务的修改
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         持久性
        </strong>
       </td>
       <td>
        必须持久化到磁盘
       </td>
       <td>
        通常也持久化，但可能被清理
       </td>
      </tr>
     </tbody>
    </table>
    <h4 id="4.%C2%A0%E5%8D%8F%E5%90%8C%E5%B7%A5%E4%BD%9C" name="4.%C2%A0%E5%8D%8F%E5%90%8C%E5%B7%A5%E4%BD%9C">
     4.
     <strong>
      协同工作
     </strong>
    </h4>
    <ul>
     <li>
      在事务处理中，重做日志和还原日志通常会协同工作：
      <ul>
       <li>
        当事务提交时，重做日志确保修改被持久化。
       </li>
       <li>
        当事务回滚时，还原日志确保数据恢复到事务开始前的状态。
       </li>
      </ul>
     </li>
     <li>
      在系统恢复过程中，重做日志用于重新应用已提交的事务，而还原日志用于撤销未提交的事务。
     </li>
    </ul>
    <h4 id="5.%C2%A0%E5%AE%9E%E9%99%85%E5%BA%94%E7%94%A8" name="5.%C2%A0%E5%AE%9E%E9%99%85%E5%BA%94%E7%94%A8">
     5.
     <strong>
      实际应用
     </strong>
    </h4>
    <ul>
     <li>
      <strong>
       数据库恢复
      </strong>
      ：在数据库崩溃后，系统会先使用还原日志回滚未提交的事务，然后使用重做日志重新应用已提交的事务。
     </li>
     <li>
      <strong>
       事务管理
      </strong>
      ：在并发事务中，还原日志用于实现多版本并发控制（MVCC），确保事务的隔离性。
     </li>
    </ul>
