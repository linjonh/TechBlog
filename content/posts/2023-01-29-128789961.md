    <h2>
     前言
    </h2>
    <p>
     大家新年快乐，过年的时候一直在忙也没时间更新，今天好不容易闲了下来就给大家总结一下自动化测试面试的一些常问问题，堪称自动化测试的面试宝典。方便大家年后面试找工作。我已经讲面试题目总结成word文档，需要的请添加我文章底部群聊领取。
    </p>
    <h2 id="1.1%20%E4%BB%80%E4%B9%88%E6%98%AF%C2%A0API%EF%BC%9F" style="margin-left:5pt;text-align:justify;">
     <strong>
      <span style="color:#3e1d5a;">
       1.1 什么是
      </span>
     </strong>
     <strong>
     </strong>
     <strong>
      <span style="color:#3e1d5a;">
       API？
      </span>
     </strong>
    </h2>
    <p style="margin-left:5pt;text-align:justify;">
     API 是(Application Programming Interface)首字母缩略词，即应用程序编程接口。API 是一组用于 构建软件应用程序的规程，协议和工具。API 充当软件应用程序之间的接口，并允许两个软件应用程 序相互通信。API 是一组软件功能，可以由其他软件执行。
    </p>
    <h2 id="1.2%20%E4%BB%80%E4%B9%88%E6%98%AF%20API%C2%A0%E6%B5%8B%E8%AF%95%EF%BC%9F" style="margin-left:5pt;text-align:justify;">
     <strong>
      <span style="color:#3e1d5a;">
       1.2 什么是 API
      </span>
     </strong>
     <strong>
     </strong>
     <strong>
      <span style="color:#3e1d5a;">
       测试？
      </span>
     </strong>
    </h2>
    <p style="margin-left:5pt;text-align:justify;">
     API 测试是一种软件测试，涉及直接测试 API，也是集成测试的一部分，用于检查 API 是否满足应用 程序的功能，可靠性，性能和安全性方面的期望。在 API 测试中，我们主要关注软件架构的业务逻辑 层。可以在包含多个 API 的任何软件系统上执行 API 测试。
    </p>
    <h2 id="1.3%20%E5%B8%B8%E8%A7%81%E7%9A%84%20API%C2%A0%E6%B5%8B%E8%AF%95%E7%B1%BB%E5%9E%8B%E6%9C%89%E5%93%AA%E4%BA%9B%EF%BC%9F" style="margin-left:5pt;text-align:justify;">

    </h2>
    <p style="margin-left:5pt;text-align:justify;">
     API 测试通常涉及以下实践：
    </p>
    <p style="margin-left:5pt;">
     单元测试、功能测试、负载测试、运行时/错误检测、安全测试、UI 测试、互操作性和 WS 一致性测 试、渗透测试、模糊测试
    </p>
    <h2 id="1.4%20%E5%88%97%E4%B8%BE%20API%C2%A0%E6%B5%8B%E8%AF%95%E4%B8%AD%E4%BD%BF%E7%94%A8%E7%9A%84%E4%B8%80%E4%BA%9B%E5%B8%B8%E7%94%A8%E5%8D%8F%E8%AE%AE%EF%BC%9F" style="margin-left:5pt;text-align:justify;">

     <h2 id="1.9API%C2%A0%E5%B8%B8%E8%A7%81%E6%B5%8B%E8%AF%95%E6%9C%89%E5%93%AA%E4%BA%9B%EF%BC%9F" style="margin-left:5pt;">

     </p>
     <p style="margin-left:5pt;">
      验证 API 是否触发其他事件或请求其他 API、 在没有返回值时验证 API 的行为、
     </p>
     <h2 id="1.10API%C2%A0%E6%B5%8B%E8%AF%95%E6%9C%89%E5%93%AA%E4%BA%9B%E4%BC%98%E5%8A%BF%EF%BC%9F" style="margin-left:5pt;">
      <a name="1.10API测试有哪些优势？">
      </a>
      <strong>
       <span style="color:#3e1d5a;">
        1.10API
       </span>
      </strong>
      <strong>
      </strong>
      <strong>
       <span style="color:#3e1d5a;">
        测试有哪些优势？
       </span>
      </strong>
     </h2>
     <p style="margin-left:5pt;">
      更快及更高的测试覆盖率。
     </p>
     <p style="margin-left:5pt;">
      API 测试有助于我们降低测试成本。通过 API 测试，我们可以在 GUI 测试之前找到小错误。在 GUI 测试期间，这些小错误将变得更大。因此，在 API 测试中发现这些错误将对公司具有成本效益。
     </p>
     <p style="margin-left:5pt;">
      API 测试与语言无关。
     </p>
     <p style="margin-left:5pt;">
      API 测试在测试核心功能方面非常有用。我们可以在没有用户界面的情况下测试 API。在 GUI 测试中， 我们需要等到应用程序可用于测试核心功能。
     </p>
     <p style="margin-left:5pt;">
      API 测试有助于我们降低风险。
     </p>

       </span>
      </strong>
     </h2>
     <p style="margin-left:5pt;">

     </p>
     <p style="margin-left:5pt;">

      API 返回任何错误时的错误代码 授权检查 非功能测试，如性能测试，安全测试
     </p>

       1.27

     <strong>
     </strong>
     <strong>
      <span style="color:#3e1d5a;">
       什么是分层测试？
      </span>
     </strong>
    </h2>
    <p>
     <strong>
      首先，分层即是有明显的层级关系。
     </strong>
    </p>
    <p style="margin-left:5pt;">
     例如： 国企中的领导级别，公务员级别，严格的层级关系；
    </p>
    <p style="margin-left:5pt;">
     面包也有层级，不同层次不同馅的面包或者蛋糕是不是好吃点？ 那测试的层级是什么呢？就是不同的时间段，不同的团队(当然也可能是同一团队)使用不同的测试用 例对产品不同的关注点进行测试。
    </p>
    <p style="margin-left:5pt;text-align:justify;">
     一个系统/产品，最先看到的是 UI 层，也就是外观或者说整体，这些是最上层，最上层依赖下面的服 务层，也就是接口或者模块，最底层就是单元，这个单元是函数或者方法。按照这三层选择不同时间 段，不同团队不同测试用例进行的测试就是分层测试。
    </p>
    <p style="margin-left:5pt;">
     <strong>
      其次，说下怎么分层测试？
     </strong>
     以一个同行的比喻，拿造一辆车来说，从最重要的发动机开始，单元测试就是零部件检测，若是组装 发动机后发现某个零部件质量不过关，那不是要拆了后修理然后再组装，这不仅麻烦还占时间，
    </p>
    <p style="margin-left:5pt;">
     所以 轻易想到的一个方法就是在零部件制作好后，使用某些工具对零部件单独检测，那软件测试中的单元 测试就是对函数或者方法进行测试，这个需要开发来完成，有两种方法： 第一种就是代码走查，依赖个人或者团队能力，且不可重复； 第二种方法就是写单元自动化测试用例，可重复性高，但是这个要求覆盖率要高，否则也没多大用处， 所以底层的单元自动化测试用例量最多，打好地基，同时发现问题解决成本最小，不需要搭建环境， 简单快速执行。
    </p>
    <p style="margin-left:5pt;">
     零部件检查完毕后就需要组装成发动机，发动机提供一个接口给仪表盘提供实时车速， 那就需要对这个接口进行测试了，选用相应的工具。 同理，软件测试的服务层也是采用自动化测试，更注重组件测试/集成测试等，比单元测试难度大点， 运行速度相对慢点，需要搭建环境，因此测试用例比单元测试要少。发动机接口测试完毕后组装成整 个车辆，从外观整体各方面进行测试，那软件在最上层的也是界面测试，
    </p>
    <p style="margin-left:5pt;">
     这个层级只有很少可以进行自动化测试，其余部分依靠人力，执行慢，搭建环境复杂，用例更少，所以三个层级组成一个正三角形，按照这种方式编写用例并执行相应用例，是最稳固的，质量也是最高的。
     <strong>
      然后，说一下分层的优势。
     </strong>
     尽量测试前移，在开发前期发现问题解决问题，开发成本会迅速下降。 不同时间段关注不同，分重点测试，层层防护。
    </p>
    <p style="margin-left:5pt;">
     后续的测试根据前面的测试内容进行补充或者有计划的交叉，能在有限的测试时间里覆盖更多的风险 点。
    </p>
    <p style="margin-left:5pt;">
     有些功能点只适合白盒测试，开发前期有效分层，能解决后期无法测试的问题。
     <strong>
      最后，说说实施时的注意点。
     </strong>
     层级如何划分要设计好，相应的用例就好设计了，用例执行时要持续追踪，前面的工作要为后面的工 作起到实际作用。
    </p>
