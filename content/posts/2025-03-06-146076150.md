     1.1 场景1
    </h3>
    <p>
     如何将分支hotfix/bug-123的提交 abc，拿到 feature/bug-123分支?
     <br/>
     假设你当前在feature分支
    </p>
    <p>
     有三种方法：
    </p>
    <ol>
     <li>
      使用 git cherry-pick
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> checkout feature/bug-123
<span class="token function">git</span> cherry-pick abc
<span class="token comment"># 解决冲突（如果有）</span>
<span class="token function">git</span> <span class="token function">add</span> <span class="token operator">&lt;</span>冲突文件<span class="token operator">&gt;</span>
<span class="token function">git</span> cherry-pick <span class="token parameter variable">--continue</span>
<span class="token comment"># 此时提交 abc 现在已经被应用到 feature/bug-123 分支</span>
</code></pre>
    <ol start="2">
     <li>
      使用 git merge
      <br/>
      如果 hotfix/bug-123 分支只有提交 abc 是你需要的，可以直接合并整个分支
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> checkout feature/bug-123
<span class="token function">git</span> merge hotfix/bug-123
<span class="token comment"># 解决冲突（如果有）</span>
<span class="token function">git</span> <span class="token function">add</span> <span class="token operator">&lt;</span>冲突文件<span class="token operator">&gt;</span>
<span class="token function">git</span> commit
<span class="token comment"># 此时hotfix/bug-123 分支的提交（包括 abc）现在已经被合并到 feature/bug-123 分支</span>
</code></pre>
    <ol start="3">
     <li>
      使用 git rebase
      <br/>
      如果 feature/bug-123 分支需要保持线性历史，可以使用 rebase
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> checkout feature/bug-123
<span class="token function">git</span> rebase hotfix/bug-123
<span class="token comment"># 解决冲突（如果有）</span>
<span class="token function">git</span> <span class="token function">add</span> <span class="token operator">&lt;</span>冲突文件<span class="token operator">&gt;</span>
<span class="token function">git</span> rebase <span class="token parameter variable">--continue</span>
<span class="token comment"># 此时feature/bug-123 分支现在包含了 hotfix/bug-123 分支的提交（包括 abc），并且提交历史是线性的</span>
</code></pre>
    <p>
     <strong>
      推荐方法
     </strong>
     ：使用 git cherry-pick，因为它只提取指定提交，不会引入其他无关的更改
    </p>
    <h3>
     <a id="12_2_38">
     </a>
     1.2 场景2
    </h3>
    <p>
     你刚提交的 commit message 写错了，如何快速修改?
    </p>
    <ol>
     <li>
      执行交互式 rebase：
      <br/>
      在终端中输入以下命令，打开交互式 rebase 界面，HEAD~1 表示最近的一次提交：
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> rebase <span class="token parameter variable">-i</span> HEAD~1
</code></pre>
    <ol start="2">
     <li>
      编辑提交信息
      <br/>
      这时会打开一个编辑器，显示类似以下内容：
     </li>
    </ol>
    <pre><code class="prism language-bash">pick <span class="token operator">&lt;</span>commit_hash<span class="token operator">&gt;</span> <span class="token operator">&lt;</span>commit_message<span class="token operator">&gt;</span>
</code></pre>
    <p>
     pick 代表要保持这个提交不变。将 pick 改为 reword，然后保存退出编辑器
    </p>
    <pre><code class="prism language-bash">reword <span class="token operator">&lt;</span>commit_hash<span class="token operator">&gt;</span> <span class="token operator">&lt;</span>commit_message<span class="token operator">&gt;</span>
</code></pre>
    <ol start="3">
     <li>
      修改提交信息
      <br/>
      执行完 rebase 后，Git 会再次打开编辑器，让你修改提交信息。你可以在这里编辑提交的 message
     </li>
     <li>
      完成 rebase
      <br/>
      修改完提交信息后，保存并关闭编辑器。Git 会自动完成 rebase，并更新提交信息
     </li>
    </ol>

     1.7 场景7
    </h3>
    <p>
     如何 clone 所有子模块?假设主项目已克隆在本地
    </p>
    <ol>
     <li>
      初始化子模块
      <br/>
      首先，进入到你已经克隆的主项目目录，然后使用 git submodule update --init 来初始化子模块。这个命令会根据主项目中的 .gitmodules 配置文件初始化子模块
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token builtin class-name">cd</span> /path/to/your/repo  <span class="token comment"># 进入你的主项目目录</span>
<span class="token function">git</span> submodule update <span class="token parameter variable">--init</span>
</code></pre>
    <ol start="2">
     <li>
      更新所有子模块（包括嵌套的子模块）
      <br/>
      如果主项目中有子模块嵌套子模块（即子模块也有自己的子模块），你可以使用 --recursive 选项来递归地初始化和更新所有子模块
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> submodule update <span class="token parameter variable">--init</span> <span class="token parameter variable">--recursive</span>
</code></pre>
    <ol start="3">
     <li>
      克隆子模块的最新版本
      <br/>
      如果你希望更新到子模块的最新版本，或者确保它们与主项目同步，可以使用以下命令
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> submodule update <span class="token parameter variable">--remote</span> <span class="token parameter variable">--recursive</span>
</code></pre>
    <ol start="4">
     <li>
      查看子模块的状态
      <br/>
      你可以使用 git submodule status 来查看当前子模块的状态，确保它们已经正确地初始化和更新
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> submodule status
</code></pre>

     1.12 场景12
    </h3>
    <p>
     如何将当前所在的本地分支名称 hotfix/1234修改为hotfix/bug-1234？
    </p>
    <ol>
     <li>
      确保你在 hotfix/1234 分支上
      <br/>
      首先，确认你当前在 hotfix/1234 分支上。如果你还没有切换到该分支，可以使用以下命令
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> checkout hotfix/1234
</code></pre>
    <ol start="2">
     <li>
      重命名本地分支
      <br/>
      使用 git branch -m 命令来重命名当前的本地分支
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> branch <span class="token parameter variable">-m</span> hotfix/bug-1234
</code></pre>
    <p>
     这将把 hotfix/1234 本地分支重命名为 hotfix/bug-1234
    </p>
    <ol start="3">
     <li>
      更新远程分支（如果需要）
      <br/>
      如果你已经将 hotfix/1234 推送到了远程仓库，并且想要将远程分支也重命名，你需要执行以下步骤：
      <br/>
      删除远程旧分支（hotfix/1234）
     </li>
    </ol>
    <pre><code class="prism language-bash"><span class="token function">git</span> push origin <span class="token parameter variable">--delete</span> hotfix/1234
</code></pre>
    <p>
     推送新分支（hotfix/bug-1234）
    </p>
    <pre><code class="prism language-bash"><span class="token function">git</span> push origin hotfix/bug-1234
</code></pre>
    <p>
     设置上游分支（用于跟踪远程分支）
    </p>
    <pre><code class="prism language-bash"><span class="token function">git</span> push --set-upstream origin hotfix/bug-1234
</code></pre>

    <h2>
     <a id="2_git_cherrypick_commithash__git_checkout_branch__file_254">
     </a>
     2 git cherry-pick &lt;commit-hash&gt; 和 git checkout branch -- file
    </h2>
    <ol>
     <li>
      git cherry-pick
     </li>
    </ol>
    <ul>
     <li>
      作用：将某个
      <strong>
       提交
      </strong>
      （commit）应用到当前分支
     </li>
     <li>
      使用场景：当你需要将另一个分支的某个特定提交（包括其更改）应用到当前分支时使用
     </li>
     <li>
      操作对象：提交（commit），而不是单个文件
     </li>
     <li>
      命令格式：
      <code>
       git cherry-pick &lt;commit-hash&gt;
      </code>
     </li>
     <li>
      特点：
      <ul>
       <li>
        会将指定提交的更改应用到当前分支，并生成一个新的提交。
       </li>
       <li>
        适用于跨分支复制某个提交的更改。
       </li>
       <li>
        如果提交涉及多个文件，所有文件的更改都会被应用。
       </li>
      </ul>
     </li>
    </ul>
    <ol start="2">
     <li>
      git checkout branch – file
     </li>
    </ol>
    <ul>
     <li>
      作用：将某个分支中的特定
      <strong>
       文件
      </strong>
      覆盖到当前工作目录。
     </li>
     <li>
      使用场景：当你需要将另一个分支的某个文件的状态复制到当前分支时使用。
     </li>
     <li>
      操作对象：单个
      <strong>
       文件
      </strong>
      ，而不是提交。
     </li>
     <li>
      特点：
      <ul>
       <li>
        只操作单个文件，不会影响其他文件。
       </li>
       <li>
        不会生成新的提交，
        <strong>
         你需要手动 git add 和 git commit
        </strong>
        。
       </li>
       <li>
        适用于从另一个分支恢复某个文件的状态。
       </li>
      </ul>
     </li>
    </ul>
    <p>
     <strong>
      总结
     </strong>
     ：
    </p>
    <ul>
     <li>
      如果你需要复制某个提交的所有更改，使用 git cherry-pick。
     </li>
     <li>
      如果你只需要复制另一个分支的某个文件，使用 git checkout branch – file
     </li>
    </ul>
    <h2>
     <a id="3_git_checkout_commitIDgit_reset_hard_commitID_277">
     </a>
     3 git checkout commitID和git reset --hard commitID的区别
    </h2>
    <p>
     <code>
      git checkout commitID
     </code>
     和
     <code>
      git reset --hard commitID
     </code>
     都可以将 Git 仓库的状态恢复到指定的提交，但是它们的工作方式和影响有所不同。具体区别如下：
    </p>
    <h3>
     <a id="31_git_checkout_commitID_279">
     </a>
     3.1 git checkout commitID
    </h3>
    <ul>
     <li>
      作用：切换到指定的提交
      <code>
       commitID
      </code>
      ，使工作目录和暂存区的文件恢复到该提交的状态
     </li>
     <li>
      行为：
      <ul>
       <li>
        该命令会让你的工作目录变为与 commitID 对应的文件内容一致，但是不会修改当前分支的历史，你依然停留在当前分支上
       </li>
       <li>
        它会进入 分离头指针（detached HEAD） 状态，也就是说，Git 不再指向任何分支，而是指向了一个特定的提交。如果在这种状态下进行新的提交，Git 会创建一个新的分支，除非你显式地创建一个新分支
       </li>
       <li>
        不会改变暂存区（index）的状态，因此不会丢失任何暂存的更改
       </li>
      </ul>
     </li>
     <li>
      适用场景
      <ul>
       <li>
        你想查看某个特定提交的内容，或者临时检查某个历史版本
       </li>
       <li>
        你只想查看一个特定的提交，而不想修改当前分支
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="32_git_reset_hard_commitID_288">
     </a>
     3.2 git reset --hard commitID
    </h3>
    <ul>
     <li>
      作用：将当前分支的指针重置到指定的 commitID，同时会重置工作目录和暂存区，使它们与该提交一致
     </li>
     <li>
      行为：
      <ul>
       <li>
        改变当前分支的历史：它会把当前分支的指针（例如 master 或 main）指向 commitID，从而“丢弃”该提交之后的所有更改和提交
       </li>
       <li>
        更改工作目录和暂存区：它会将工作目录中的所有文件和暂存区的文件恢复到 commitID 所对应的状态，所有未提交的更改都会丢失
       </li>
       <li>
        删除历史提交：所有在 commitID 之后的提交都会被删除，分支的历史会被重写
       </li>
      </ul>
     </li>
     <li>
      适用场景：
      <ul>
       <li>
        你想彻底丢弃一些提交并重置代码状态，或者将分支回滚到某个特定提交
       </li>
       <li>
        要小心使用，因为会删除所有本地更改，并且无法恢复被丢弃的提交（除非你有备份）
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="33__297">
     </a>
     3.3 使用场景对比
    </h3>
    <ul>
     <li>
      如果你只是想查看某个提交，不打算修改当前分支，可以使用 git checkout commitID
     </li>
     <li>
      如果你想删除当前分支上的提交，并回到某个历史点，丢弃所有后续的更改，则使用 git reset --hard commitID
     </li>
    </ul>
    <h2>
     <a id="4_git_rebase_i_301">
     </a>
     4 git rebase -i
    </h2>
    <p>
     启动交互式 rebase
    </p>
    <ul>
     <li>
      命令形式：
      <code>
       git rebase -i &lt;commit-hash&gt;
      </code>
      或
      <code>
       git rebase -i HEAD~n
      </code>
     </li>
     <li>
      编辑 rebase 的操作:
      <ul>
       <li>
        示例：
        <ul>
         <li>
          <code>
           pick 1234567 Commit message 1
          </code>
         </li>
         <li>
          <code>
           pick 2345678 Commit message 2
          </code>
         </li>
        </ul>
       </li>
       <li>
        以下是一些常用的命令和作用：
        <ul>
         <li>
          pick：保留该提交
         </li>
         <li>
          reword：保留提交，但修改提交信息
         </li>
         <li>
          edit：保留提交，但让你修改该提交的内容（例如，修改代码或文件）
         </li>
         <li>
          squash 或 s：将当前提交与前一个提交合并，并保留两者的提交信息
         </li>
         <li>
          fixup 或 f：将当前提交与前一个提交合并，并丢弃当前提交的提交信息（只保留前一个提交的信息）
         </li>
         <li>
          drop：删除当前提交
         </li>
         <li>
          exec：在该提交时执行一个命令
         </li>
        </ul>
       </li>
      </ul>
     </li>
    </ul>
    <p>
     例如，你想将第 2 和第 3 个提交合并到第一个提交中，并修改它们的提交信息，可以将文件编辑为如下所示
    </p>
    <pre><code class="prism language-bash">pick <span class="token number">1234567</span> Commit message <span class="token number">1</span>
squash <span class="token number">2345678</span> Commit message <span class="token number">2</span>
squash <span class="token number">3456789</span> Commit message <span class="token number">3</span>
</code></pre>
    <p>
     编辑完成后，保存并关闭编辑器，Git 会继续执行 rebase 操作，应用你选择的操作。
    </p>
    <ul>
     <li>
      如果有冲突，Git 会暂停 rebase，要求你解决冲突。你需要手动解决冲突，
      <strong>
       解决完冲突后，要add到暂存区，但不需要提交
      </strong>
      ，然后使用
      <code>
       git rebase --continue
      </code>
      继续 rebase
     </li>
     <li>
      如果你不想继续 rebase，可以使用
      <code>
       git rebase --abort
      </code>
      来放弃 rebase 操作
     </li>
    </ul>
    <p>
     编辑提交信息
     <br/>
     如果你选择了
     <code>
      squash
     </code>
     或
     <code>
      reword
     </code>
     ，Git 会打开一个新的编辑器，允许你编辑提交信息。你可以选择保留原始信息，或者将提交信息合并为新的内容
    </p>
