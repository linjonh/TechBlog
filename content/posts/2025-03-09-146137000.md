    <h2>
     网关实现路由
    </h2>
    <p>
     问题：每个微服务都有不同的地址或端口，入口不同，请求不同数据时要访问不同的入口，需要维护多个入口地址，前端无法调用nacos，无法实时更新服务列表。
    </p>
    <p>
     解决方案：采用微服务网关，数据在网络间传输，从一个网络传输到另一网络时就需要经过网关来做数据的
     <strong>
      路由
     </strong>
     <strong>
      和转发。
     </strong>
    </p>
    <p>
     在微服务中新建一个网关模块，作为网关微服务
    </p>
    <p>
     引入依赖（需要加入nacos依赖，让nacos管理）
    </p>
    <pre><code class="language-java"> &lt;dependencies&gt;
        &lt;!--common--&gt;
        &lt;dependency&gt;
            &lt;groupId&gt;com.heima&lt;/groupId&gt;
            &lt;artifactId&gt;hm-common&lt;/artifactId&gt;
            &lt;version&gt;1.0.0&lt;/version&gt;
        &lt;/dependency&gt;
        &lt;!--网关--&gt;
        &lt;dependency&gt;
            &lt;groupId&gt;org.springframework.cloud&lt;/groupId&gt;
            &lt;artifactId&gt;spring-cloud-starter-gateway&lt;/artifactId&gt;
        &lt;/dependency&gt;
        &lt;!--nacos discovery--&gt;
        &lt;dependency&gt;
            &lt;groupId&gt;com.alibaba.cloud&lt;/groupId&gt;
            &lt;artifactId&gt;spring-cloud-starter-alibaba-nacos-discovery&lt;/artifactId&gt;
        &lt;/dependency&gt;
        &lt;!--负载均衡--&gt;
        &lt;dependency&gt;
            &lt;groupId&gt;org.springframework.cloud&lt;/groupId&gt;
            &lt;artifactId&gt;spring-cloud-starter-loadbalancer&lt;/artifactId&gt;
        &lt;/dependency&gt;
    &lt;/dependencies&gt;</code></pre>
    <p>
     <code>
      在
      <strong>
       application.yaml
      </strong>
     </code>
     文件，配置路由
    </p>
    <pre><code class="language-java">  gateway:
      routes:
        - id: item # 路由规则id，自定义，唯一
          uri: lb://item-service # 路由的目标服务，lb代表负载均衡，会从注册中心拉取服务列表
          predicates: # 路由断言，判断当前请求是否符合当前规则，符合则路由到目标服务
            - Path=/items/**,/search/** # 这里是以请求路径作为判断规则</code></pre>
    <p>
     经测试成功
    </p>
    <h2>
     网关登录鉴权
    </h2>
    <p>
     问题：单体架构时我们只需要完成一次用户登录、身份校验，就可以在所有业务中获取到用户信息。但是，在微服务中，每个微服务都独立部署，一般只有用户微服务能校验登录信息，事实上，这是不安全的。
    </p>
    <p>
     解决方案：既然网关是所有微服务的入口，一切请求都需要先经过网关。我们完全可以把登录校验的工作放到网关去做。
    </p>
    <h3>
     鉴权思路
    </h3>
    <p>
     登录是基于JWT来实现的，校验JWT的算法复杂，而且需要用到秘钥。我们不可能让个微服务都需要知道JWT的秘钥，不安全。也不可能每个微服务重复编写登录校验代码、权限校验代码，麻烦。
    </p>
    <p>
     所以，我们在网关和用户服务保存秘钥，开发登录校验功能。
    </p>
    <h3>
     登录校验流程图
    </h3>
    <p>
     <img src="https://i-blog.csdnimg.cn/direct/4c55f5a61253493fac4f1f0c364221d0.png"/>
    </p>
    <p>
     我们可以看到：前端——网关——后端服务
    </p>
    <p>
     我们在网关层去实现过滤请求。
    </p>
    <h3>
     网关过滤器详解
    </h3>
    <p>
     <strong>
      <code>
       Gateway
      </code>
     </strong>
     内部工作的基本原理：
    </p>
    <p>
     <img src="https://i-blog.csdnimg.cn/direct/c57c372e34004d0ba859406a8fe3634d.png"/>
    </p>
    <p>
     如图所示：
    </p>
    <ol>
     <li>
      <p>
       客户端请求进入网关后由
       <strong>
        <code>
         HandlerMapping
        </code>
       </strong>
       对请求做判断，找到与当前请求匹配的路由规则（
       <strong>
        <code>
         Route
        </code>
       </strong>
       ），然后将请求交给
       <strong>
        <code>
         WebHandler
        </code>
       </strong>
       去处理
      </p>
     </li>
     <li>
      <p>
       <strong>
        <code>
         WebHandler
        </code>
       </strong>
       则会加载当前路由下需要执行的过滤器链（
       <strong>
        <code>
         Filter chain
        </code>
       </strong>
       ），然后按照顺序逐一执行过滤器（后面称为
       <strong>
        <code>
         Filter
        </code>
       </strong>
       ）
      </p>
     </li>
     <li>
      <p>
       图中
       <strong>
        <code>
         Filter
        </code>
       </strong>
       被虚线分为左右两部分，是因为
       <strong>
        <code>
         Filter
        </code>
       </strong>
       内部的逻辑分为
       <strong>
        <code>
         pre
        </code>
       </strong>
       和
       <strong>
        <code>
         post
        </code>
       </strong>
       两部分，分别会在请求路由到微服务
       <strong>
        之前
       </strong>
       和
       <strong>
        之后
       </strong>
       被执行
      </p>
     </li>
     <li>
      <p>
       只有所有
       <strong>
        <code>
         Filter
        </code>
       </strong>
       的
       <strong>
        <code>
         pre
        </code>
       </strong>
       逻辑都依次顺序执行通过后，请求才会被路由到微服务
      </p>
     </li>
     <li>
      <p>
       微服务返回结果后，再倒序执行
       <strong>
        <code>
         Filter
        </code>
       </strong>
       的
       <strong>
        <code>
         post
        </code>
       </strong>
       逻辑
      </p>
     </li>
     <li>
      <p>
       最终把响应结果返回
      </p>
     </li>
    </ol>
    <p>
     反正就是：我们需要在
     <strong>
      <code>
       NettyRoutingFilter过滤器之前，在发起Request时，即pre时，定义一个过滤器，进行网关登录校验。
      </code>
     </strong>
    </p>
    <h3>
     实现网关过滤器
    </h3>
    <p>
     我们采用全局过滤器，作用范围是所有路由，即
     <strong>
      <code>
       GlobalFilter。
      </code>
     </strong>
    </p>
    <pre><code class="language-java">package com.hmall.gateway.filters;

import com.hmall.common.exception.UnauthorizedException;
import com.hmall.gateway.config.AuthProperties;
import com.hmall.gateway.utils.JwtTool;
import lombok.RequiredArgsConstructor;
import org.springframework.cloud.gateway.filter.GatewayFilterChain;
import org.springframework.cloud.gateway.filter.GlobalFilter;
import org.springframework.core.Ordered;
import org.springframework.http.HttpStatus;
import org.springframework.http.server.reactive.ServerHttpRequest;
import org.springframework.http.server.reactive.ServerHttpResponse;
import org.springframework.stereotype.Component;
import org.springframework.util.AntPathMatcher;
import org.springframework.web.server.ServerWebExchange;
import reactor.core.publisher.Mono;

import java.util.List;

@RequiredArgsConstructor
@Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {

    private final AuthProperties authProperties;

    private final JwtTool jwtTool;

    private final AntPathMatcher antPathMatcher = new AntPathMatcher();

    /**
     * 过滤器
     * @param exchange
     * @param chain
     * @return
     */
    @Override
    public Mono&lt;Void&gt; filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1.获取请求对象
        ServerHttpRequest request = exchange.getRequest();
        // 2.判断是否需要拦截
        if (isExclude(request.getPath().toString())) {
            // 放行
            return chain.filter(exchange);
        }
        // 3.获取token
        String token = null;
        List&lt;String&gt; headers = request.getHeaders().get("authorization");
        if (headers != null &amp;&amp; !headers.isEmpty()) {
         token = headers.get(0);
        }
        // 4.解析token
        Long userId = null;
        try {
            userId = jwtTool.parseToken(token);
        } catch (UnauthorizedException e) {
            // 如果无效，拦截
            ServerHttpResponse response = exchange.getResponse();
            response.setStatusCode(HttpStatus.UNAUTHORIZED);
            return response.setComplete();
        }
        // 5.获取用户信息
        String userInfo = userId.toString();
        ServerWebExchange swe = exchange.mutate().request(builder -&gt; builder.header("user-info",userInfo ))
                .build();
        // 6.放行
        return chain.filter(swe);
    }

    /**
     * 判断路径是否需要拦截
     * @param antPath
     * @return
     */
    private boolean isExclude(String antPath) {
        for (String pathPattern : authProperties.getExcludePaths()) {
            if(antPathMatcher.match(pathPattern, antPath)){
                return true;
            }
        }
        return false;
    }

    // 优先级
    @Override
    public int getOrder() {
        return 0;
    }
}
</code></pre>
    <p>
     因为我采用了JWT令牌进行校验，所以引入了JWT工具类，进行令牌的获取与解析。
    </p>
    <p>
     至于
     <strong>
      AuthProperties
     </strong>
     是配置了不需要鉴权就能访问的路径。
    </p>
    <p>
     经测试，网关已经可以完成登录校验并获取登录用户身份信息。
    </p>
    <hr/>
    <p>
     问题：当网关将请求转发到微服务时，微服务如何获取用户身份，我们不可能每个微服务都写一个拦截器去得到用户身份信息。
    </p>
    <p>
     解决方案：将用户信息以请求头的方式传递到下游微服务。然后微服务可以从请求头中获取登录用户信息（上述代码第五步已经完成了）。考虑到微服务内部可能很多地方都需要用到登录用户信息，因此我们可以利用SpringMVC的拦截器来实现登录用户信息获取，并存入ThreadLocal。
    </p>
    <h3>
     拦截器流程图
    </h3>
    <p>
     <img src="https://i-blog.csdnimg.cn/direct/f362ae5358334459b051f071e81c6c84.png"/>
    </p>
    <p>
     提供一个用于保存登录用户的ThreadLocal工具UserContext：
    </p>
    <pre><code class="language-java">public class UserContext {
    private static final ThreadLocal&lt;Long&gt; tl = new ThreadLocal&lt;&gt;();

    /**
     * 保存当前登录用户信息到ThreadLocal
     * @param userId 用户id
     */
    public static void setUser(Long userId) {
        tl.set(userId);
    }

    /**
     * 获取当前登录用户信息
     * @return 用户id
     */
    public static Long getUser() {
        return tl.get();
    }

    /**
     * 移除当前登录用户信息
     */
    public static void removeUser(){
        tl.remove();
    }
}</code></pre>
    <p>
     在公用模块下定义一个拦截器：
    </p>
    <pre><code class="language-java">public class UserInfoInterceptor implements HandlerInterceptor {

    /**
     * 请求拦截器
     * @param request
     * @param response
     * @param handler
     * @return
     * @throws Exception
     */
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        // 1.获取请求头中的 用户信息
        String userInfo = request.getHeader("user-info");
        // 2.判断是否为空
        if (StringUtils.isNotBlank(userInfo)) {
            UserContext.setUser(Long.valueOf(userInfo));
        }
        // 3.放行
        return true;
    }

    /**
     * 响应拦截器
     * @param request
     * @param response
     * @param handler
     * @param ex
     * @throws Exception
     */
    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        UserContext.removeUser();
    }
}
</code></pre>
    <p>
     编写
     <strong>
      <code>
       SpringMVC
      </code>
     </strong>
     的配置类，配置登录拦截器：
    </p>
    <pre><code class="language-java">@Configuration
@ConditionalOnClass(DispatcherServlet.class)
public class MvcConfig implements WebMvcConfigurer {
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new UserInfoInterceptor());
    }
}</code></pre>
    <p>
     注意：这个配置类默认是不会生效的，基于SpringBoot的自动装配原理，我们要将其添加到
     <strong>
      <code>
       resources
      </code>
     </strong>
     目录下的
     <strong>
      <code>
       META-INF/spring.factories
      </code>
     </strong>
     文件中：
    </p>
    <pre><code class="language-java">org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  com.hmall.common.config.MyBatisConfig,\
  com.hmall.common.config.MvcConfig</code></pre>
    <p>
     经测试，服务得到网关传递的用户身份信息。
    </p>
    <h2>
     服务信息鉴权
    </h2>
    <p>
     问题：因为之前编写的过滤器和拦截器功能，微服务可以轻松获取登录用户信息。但是，有时候请求到达微服务后还需要调用其它多个微服务。我们没有实现服务之间的用户身份信息的传递。
    </p>
    <p>
     解决方案：由于微服务获取用户信息是通过拦截器在请求头中读取，因此要想实现微服务之间的用户信息传递，就
     <strong>
      必须在微服务发起调用时把用户信息存入请求头
     </strong>
     。
    </p>
    <p>
     因为我们之前微服务之间调用是基于OpenFeign来实现的，并不是我们自己发送的请求。所以我们可以采用Feign中提供的一个拦截器接口：
     <strong>
      <code>
       feign.RequestInterceptor。
      </code>
     </strong>
    </p>
    <p>
     在公用模块下定义一个
     <strong>
      userInfoRequestInterceptor
     </strong>
    </p>
    <pre><code class="language-java">    /**
     * feign请求拦截器 微服务之间的远程调用时，将当前登录用户的userId传递给目标服务
     * @return
     */
    @Bean
    public RequestInterceptor userInfoRequestInterceptor() {
        return new RequestInterceptor() {
            public void apply(RequestTemplate requestTemplate) {
                Long userId = UserContext.getUser();
                if (userId != null) {
                    requestTemplate.header("user-info", userId.toString());
                }
            }
        };
    }</code></pre>
    <p>
     总结：
    </p>
    <ul>
     <li>
      为了实现网关处简便的登录校验，我们采用了
      <strong>
       GlobalFilter
      </strong>
      ；
     </li>
     <li>
      为了实现网关传递用户信息到多个微服务，我们采用了
      <strong>
       UserInfoInterceptor
      </strong>
      ；
     </li>
     <li>
      为了实现微服务之间用户身份信息传递，我们采用了
      <strong>
       userInfoRequestInterceptor。
      </strong>
     </li>
    </ul>
