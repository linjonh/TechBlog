    <h2 id="1.%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F" name="1.%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F" style="background-color:transparent">
     1.单例模式
    </h2>
    <p>
     <span style="color:#fe2c24">
      <strong>
       先回答问题：
      </strong>
     </span>
     <img alt="" height="146" src="https://i-blog.csdnimg.cn/direct/0e2dae293164403f841907c629bd73b1.png" width="143"/>
    </p>
    <p>
     <strong>
      提问：
     </strong>
    </p>
    <blockquote>
     <p>
      单例模式是什么呢？我们在某一个特定的场景下只能new一个对象。
     </p>
     <p>
      譬如代码里面用来管理对象就应该是单例的
     </p>
     <p>
      像：MySQL中的JDBC编程中的DataSource（描述了mysql服务器的位置，这个服务器的描述信息就应该是单例的）
     </p>
    </blockquote>
    <blockquote>
     <p>
      那我就要问了，那我new一次对象不就好了，还整个这个模式干嘛呢？
     </p>
     <p>
     </p>
     <p>
      这个不靠谱呀，一不小心我们new多个对象了不就完蛋了？所以这个时候我们就需要一个模式
      <strong>
       <span style="background-color:#ffd900">
        让编译器帮我们做检查
       </span>
      </strong>
     </p>
    </blockquote>
    <blockquote>
     <p>
      就像final关键字（我们不能修改这个变量，不然编译器就会报错）同理：interface，@Override，throws也是相同的道理
     </p>
    </blockquote>
    <p>
     <strong>
      <span style="background-color:#ffd900">
       但是：java在语法层面上没有对单例模式作出支持，所以我们只能通过一些编程技巧来实现类似的效果
      </span>
     </strong>
    </p>
    <h3 id="1.1%E9%A5%BF%E6%B1%89%E6%A8%A1%E5%BC%8F" name="1.1%E9%A5%BF%E6%B1%89%E6%A8%A1%E5%BC%8F" style="background-color:transparent">
     <strong>
      1.1饿汉模式
     </strong>
    </h3>
    <p>
     <span style="background-color:#ffd900">
      <strong>
       第一种方式（饿汉模式）：先类里面创建出有一个实例，然后把构造方法给隐藏起来（你就new不了了，哈哈）
      </strong>
     </span>
    </p>
    <pre><code class="language-java">class Singleton {
    private static Singleton instance = new Singleton();
  //静态成员变量，类加载的时候就被创建好了
    public static Singleton getInstance(){  // 一定要是静态方法，不然一开始别人都拿不到这个对象，又没办法调用这个方法，岂不是贻笑大方
        return instance;
    }
    private Singleton(){} // 啥也不用干，也干了呀，把构造方法给藏起来了

}
public class SingletonDemo{
    public static void main(String[] args) {
       // Singleton singleton = new Singleton();
  // 直接报错，编译器帮我们检查
        Singleton singleton1 = Singleton.getInstance();  // 都通过getInstance创建实例
        Singleton singleton2 = Singleton.getInstance();
        System.out.println(singleton1 == singleton2); // true,俩对象引用都一致
    }
}</code></pre>
    <p>
     <strong>
      <span style="background-color:#a2e043">
       这种方式也叫做饿汉模式，（这么急切）类一加载对象就被创建好了
      </span>
      <span style="background-color:#ffd900">
       （创建的时机比较早）
      </span>
     </strong>
    </p>
    <p>
     <strong>
      <span style="background-color:#ffd900">
       那我们能不能不先创建呢？我想用的时候再去创建，当然可以，
      </span>
     </strong>
     <span style="color:#0d0016">
      <strong>
       <span style="background-color:#fe2c24">
        这个就叫做懒汉模式
       </span>
      </strong>
     </span>
    </p>
    <h3 id="1.2.%E6%87%92%E6%B1%89%E6%A8%A1%E5%BC%8F" name="1.2.%E6%87%92%E6%B1%89%E6%A8%A1%E5%BC%8F" style="background-color:transparent">
     <span style="color:#0d0016">
      <strong>
       1.2.懒汉模式
      </strong>
     </span>
    </h3>
    <pre><code class="language-java">class SingletonLazy{
    private static SingletonLazy instance = null; // 先置为null
    public static SingletonLazy getInstance(){
        if(instance == null){  // 如果不为空直接返回instance
            instance = new SingletonLazy(); //  为null 再new
        }
        return instance;
    }
    private SingletonLazy(){} // 私有构造方法
}</code></pre>
    <h4 id="1.3.%E9%A5%BF%E6%B1%89%E6%A8%A1%E5%BC%8F%E5%92%8C%E6%87%92%E6%B1%89%E6%A8%A1%E5%BC%8F%E6%9C%89%E4%B8%80%E4%B8%AA%E6%98%AF%E7%BA%BF%E7%A8%8B%E4%B8%8D%E5%AE%89%E5%85%A8%E7%9A%84" name="1.3.%E9%A5%BF%E6%B1%89%E6%A8%A1%E5%BC%8F%E5%92%8C%E6%87%92%E6%B1%89%E6%A8%A1%E5%BC%8F%E6%9C%89%E4%B8%80%E4%B8%AA%E6%98%AF%E7%BA%BF%E7%A8%8B%E4%B8%8D%E5%AE%89%E5%85%A8%E7%9A%84" style="background-color:transparent">
     1.3.饿汉模式和懒汉模式有一个是线程不安全的
    </h4>
    <p>
     我们第一次说的线程不安全的时候是在count++的问题上面，这个
     <strong>
      <span style="background-color:#a2e043">
       count++这个操作再cpu上面是分三步进行的
      </span>
      <span style="background-color:#ffd900">
       （两个线程同时修改一个变量就很容易出现线程安全问题）
      </span>
     </strong>
     （非原子当时加锁解决的，当然还有其他解决办法）。
    </p>
    <p>
     <strong>
      <span style="background-color:#a2e043">
       解释：
      </span>
     </strong>
    </p>
    <blockquote>
     <p>
      <strong>
       <span style="background-color:#a2e043">
        1.饿汉模式：
       </span>
      </strong>
      （我们在类加载的时候就把对象创建好了，大家调用getInstance（）
      <strong>
       <span style="background-color:#ffd900">
        就只是涉及到读操作）
       </span>
      </strong>
     </p>
     <p>
     </p>
     <p>
      <strong>
       <span style="background-color:#a2e043">
        2.懒汉模式：
       </span>
       我们需要的时候才去创建这个对象，
       <span style="background-color:#ffd900">
        这个时候就会涉及到两个线程同时修改一个变量。
       </span>
      </strong>
     </p>
     <p>
      <span style="color:#0d0016">
       <strong>
        <span style="background-color:#fe2c24">
         并且new对象的操作也不是原子的
        </span>
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       <strong>
        1.申请内存空间
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       <strong>
        2.在内存上面构造对象
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       <strong>
        3.把内存的地址赋值给instance引用（一共这三步）
       </strong>
      </span>
     </p>
    </blockquote>
    <h5 id="1.3.1.%E9%82%A3%E6%88%91%E4%BB%AC%E6%8A%8Anew%E8%BF%99%E4%B8%AA%E6%93%8D%E4%BD%9C%E5%8F%98%E6%88%90%E5%8E%9F%E5%AD%90%E7%9A%84%E5%8F%AF%E4%BB%A5%E4%BA%86%E5%90%97%EF%BC%9F" name="1.3.1.%E9%82%A3%E6%88%91%E4%BB%AC%E6%8A%8Anew%E8%BF%99%E4%B8%AA%E6%93%8D%E4%BD%9C%E5%8F%98%E6%88%90%E5%8E%9F%E5%AD%90%E7%9A%84%E5%8F%AF%E4%BB%A5%E4%BA%86%E5%90%97%EF%BC%9F" style="background-color:transparent">
     <span style="color:#0d0016">
      <strong>
       <span style="background-color:#ffd900">
        1.3.1.那我们把new这个操作变成原子的可以了吗？
       </span>
      </strong>
     </span>
    </h5>
    <pre><code class="language-java">class SingletonLazy{
    private static SingletonLazy instance = null; // 先置为null
    public static SingletonLazy getInstance(){
        if(instance == null){  // 如果不为空直接返回instance
            synchronized(SingletonLazy.class){
                instance = new SingletonLazy(); // 把这一步变成原子的
            }
        }
        return instance;
    }
    private SingletonLazy(){} // 私有构造方法
}</code></pre>
    <p>
     <strong>
      现在确实new对象的时候不会被穿插了，但是这个锁的位置不对
     </strong>
    </p>
    <p>
     <strong>
      我们判断这个引用是否为null的时候也被穿插的了呢。另一个线程这个时候也来判断这个引用是否为null。
      <span style="color:#0d0016">
       <span style="background-color:#fe2c24">
        好了这两个线程判断的都是null，又创建了两个对象（图解如下）
       </span>
      </span>
     </strong>
    </p>
    <p>
     <img alt="" height="649" src="https://i-blog.csdnimg.cn/direct/f75234a0d8ad41359501475e6fd006b3.png" width="1200"/>
    </p>
    <blockquote>
     <p>
      总结：​​​​​​加锁的关键是：
     </p>
     <p>
      <strong>
       <span style="background-color:#a2e043">
        不是锁一个操作在cpu上是多个指令就只把这个操作锁起来，
       </span>
       <span style="background-color:#ffd900">
        而是要综合代码仔细分析，看看哪些操作是要一起被锁起来的
       </span>
      </strong>
     </p>
     <p>
     </p>
     <p>
      <span style="color:#0d0016">
       <strong>
        <span style="background-color:#fe2c24">
         特别是这种判断语句：
        </span>
        <br/>
        <span style="background-color:#fe2c24">
         这个地方最容易出问题     （小编也是写博客的时候才意识到）
        </span>
       </strong>
      </span>
      <img alt="" height="209" src="https://i-blog.csdnimg.cn/direct/4edf071ecf9c4b229f4dfc713ecf3989.png" width="209">
       <strong>
        （好开心哈哈）
       </strong>
      </img>
     </p>
    </blockquote>
    <h5 id="1.3.2.%E6%89%80%E4%BB%A5%E8%AF%B4%E8%BF%99%E4%B8%AA%E9%94%81%E7%9A%84%E5%8A%A0%E6%B3%95%E5%BE%88%E6%9C%89%E8%AF%B4%E6%B3%95%E7%9A%84%EF%BC%81" name="1.3.2.%E6%89%80%E4%BB%A5%E8%AF%B4%E8%BF%99%E4%B8%AA%E9%94%81%E7%9A%84%E5%8A%A0%E6%B3%95%E5%BE%88%E6%9C%89%E8%AF%B4%E6%B3%95%E7%9A%84%EF%BC%81" style="background-color:transparent">
     <strong>
      <span style="background-color:#a2e043">
       1.3.2.所
      </span>
     </strong>
     <strong>
      <span style="background-color:#a2e043">
       以说这个锁的加法很有说法的！
      </span>
     </strong>
    </h5>
    <p>
     <span style="background-color:#a2e043">
      <strong>
       我们这个时候就可以把这个判断语句一起加上锁了
      </strong>
     </span>
    </p>
    <pre><code class="language-java"> public static SingletonLazy getInstance(){
        synchronized (SingletonLazy.class){
            if(instance == null){
                instance = new SingletonLazy();
            }
        }
         return instance;
    }</code></pre>
    <p>
     <strong>
      现在是解决了刚才的问题但是，又引发了新的问题
     </strong>
    </p>
    <blockquote>
     <p>
      一旦我们这样写：
      <strong>
       <span style="background-color:#ffd900">
        后续每次调用getInstance（）都会进行加锁
       </span>
      </strong>
      ，但是懒汉模式就只有在最开始的时候（后面都是读的操作了，线程就安全了）
     </p>
    </blockquote>
    <p>
     <strong>
      <span style="background-color:#ffd900">
       两个很大的问题：
      </span>
     </strong>
    </p>
    <blockquote>
     <p>
      1.我们一直加锁，加锁就会涉及到锁冲突，然后在阻塞等待（鬼知道你下次上cpu是啥时候，时间相对是很大被浪费了）
     </p>
     <p>
      2.一个代码一旦涉及到加锁（那么这个代码基本就和高性能无缘了，而且我们还是每一次都加锁）
     </p>
    </blockquote>
    <h5 id="1.3.3.%E9%82%A3%E6%88%91%E4%BB%AC%E8%83%BD%E4%B8%8D%E8%83%BD%E6%97%A2%E5%8F%AF%E4%BB%A5%E8%AE%A9%E7%BA%BF%E7%A8%8B%E5%AE%89%E5%85%A8%EF%BC%8C%E5%8F%88%E4%B8%8D%E4%BC%9A%E5%AF%B9%E6%95%88%E7%8E%87%E6%9C%89%E5%A4%AA%E5%A4%A7%E7%9A%84%E5%BD%B1%E5%93%8D%E5%91%A2%EF%BC%88%E5%8F%AA%E5%8A%A0%E7%AC%AC%E4%B8%80%E6%AC%A1%E7%9A%84%E9%94%81%EF%BC%89%EF%BC%9F" name="1.3.3.%E9%82%A3%E6%88%91%E4%BB%AC%E8%83%BD%E4%B8%8D%E8%83%BD%E6%97%A2%E5%8F%AF%E4%BB%A5%E8%AE%A9%E7%BA%BF%E7%A8%8B%E5%AE%89%E5%85%A8%EF%BC%8C%E5%8F%88%E4%B8%8D%E4%BC%9A%E5%AF%B9%E6%95%88%E7%8E%87%E6%9C%89%E5%A4%AA%E5%A4%A7%E7%9A%84%E5%BD%B1%E5%93%8D%E5%91%A2%EF%BC%88%E5%8F%AA%E5%8A%A0%E7%AC%AC%E4%B8%80%E6%AC%A1%E7%9A%84%E9%94%81%EF%BC%89%EF%BC%9F" style="background-color:transparent">
     1.3.3.那我们能不能既可以让线程安全，又不会对效率有太大的影响呢（只加第一次的锁）？
    </h5>
    <blockquote>
     <p>
      <strong>
       <span style="background-color:#ffd900">
        当然可以！！！：我们现在无非就是担心后面每次getInstance都要加锁，那我们再加一层if（instance == null ）不就行了？后面就不需要进入到加锁的那一步了
       </span>
      </strong>
     </p>
    </blockquote>
    <pre><code class="language-java"> public static SingletonLazy getInstance(){
        if(instance == null){
            synchronized (SingletonLazy.class){
                if(instance == null){
                    instance = new SingletonLazy();
                }
            }
        }
        return instance;
    }</code></pre>
    <p>
     <img alt="" height="326" src="https://i-blog.csdnimg.cn/direct/a78607be7a664abab06add605141dea3.png" width="1200"/>
    </p>
    <p>
     我们的代码看似已经很完美了
    </p>
    <p>
     <strong>
      <span style="background-color:#ffd900">
       但是但是但是（还是有问题）！！！
      </span>
     </strong>
    </p>
    <h5 id="1.3.4.%E6%8C%87%E4%BB%A4%E9%87%8D%E6%8E%92%E5%BA%8F%E9%97%AE%E9%A2%98" name="1.3.4.%E6%8C%87%E4%BB%A4%E9%87%8D%E6%8E%92%E5%BA%8F%E9%97%AE%E9%A2%98" style="background-color:transparent">
     <strong>
      <span style="background-color:#ffd900">
       1.3.4.指令重排序问题
      </span>
     </strong>
    </h5>
    <p>
     <strong>
      <span style="background-color:#ffd900">
       指令重排序也会对上述的代码产生影响
      </span>
     </strong>
    </p>
    <p>
     <strong>
      <span style="background-color:#a2e043">
       （也是编译器为了执行效率，帮助我们调整代码的执行顺序，调整的前提是保证代码的逻辑不变）
      </span>
     </strong>
    </p>
    <p>
     <strong>
      <span style="background-color:#a2e043">
       上一次编译器是帮助我们
      </span>
      把一些要读内存的操作优化成读寄存器,我们用volatile就能解决这个问题，我们这一次也是用volatile解决这个问题。我给大家分析一下。
     </strong>
    </p>
    <p>
     <img alt="" height="623" src="https://i-blog.csdnimg.cn/direct/6e31ba741c5b45cc9af735b5e53fc5af.png" width="1200"/>
    </p>
    <p>
     <strong>
      <span style="background-color:#a2e043">

     2.阻塞队列
    </h2>
    <p>
     我们接下来来讲解一下阻塞队列吧~~那什么是阻塞队列呢？
    </p>
    <p>
     <strong>
      <span style="background-color:#a2e043">
       阻塞队列是一种特殊的队列（线程安全的队列）
      </span>
     </strong>
    </p>
    <blockquote>
     <p>
      1.如果队列为
      <strong>
       <span style="background-color:#ffd900">
        空
       </span>
      </strong>
      ，继续出队列，就会发生阻塞，阻塞到其他线程
      <strong>
       <span style="background-color:#ffd900">
        往队列里面添加
       </span>
      </strong>
      元素为止
     </p>
     <p>
      2.如果队列为
      <strong>
       <span style="background-color:#ffd900">
        满
       </span>
      </strong>
      ，继续入队列，就会发生阻塞，阻塞到其他线程
      <strong>
       <span style="background-color:#ffd900">
        从队列里面取出
       </span>
      </strong>
      元素为止
     </p>
    </blockquote>
    <blockquote>
     <p>
      <strong>
       <span style="background-color:#ffd900">
        这个队列最大的用处：
       </span>
      </strong>
      用来实现
      <strong>
       <span style="background-color:#a2e043">
        “生产者消费者模型”（一种常见的多线程代码编写方式
       </span>
      </strong>
      <strong>
       <span style="background-color:#a2e043">
        ）
       </span>
      </strong>
     </p>
    </blockquote>
    <p>
     <strong>
      <span style="background-color:#a2e043">
       那生产者消费者模型有什么意义呢？
      </span>
     </strong>
    </p>
    <blockquote>
     <p>
      <span style="color:#0d0016">
       1.解耦合
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       2.削峰填谷：阻塞队列就相当于一个缓冲区，平衡了生产者和消费者的处理能力
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       （
      </span>
      <span style="color:#494949">
       在 "秒杀" 场景下,消费者服务器同一时刻可能会收到大量的支付请求. 如果直接处理这些支付请求, 服务器可能扛不住，这个时候就可以把这些请求都放到一个阻塞队列中, 然后再由
       <strong>
        消费者
       </strong>
       线程慢慢的来处理每个支付请求
      </span>
      <span style="color:#333333">
       ）
      </span>
     </p>
    </blockquote>

 // 两个主要的方法；

    <pre><code class="language-java">class MyBlockingQueue{
    String[] data = new String[1000];
    volatile private int head; // 头  volatile
    volatile private int tail;  // 尾巴进头出
    volatile private int size;// 有效元素个数
    //take 和 put方法几乎每一步都涉及到修改值，而且还有判断（这个最容易）被
    synchronized public void put(String elem) throws InterruptedException {  // 扔出异常还是？？？
        while(size == data.length){  // 有效元素个数 = 数组长度
            this.wait();  // 队列满了，等待别的线程取走元素，然后再put元素
        }
        data[tail] = elem;

        tail++;
        // 数组比实际元素个数 -1; 下标为1000时说明刚好转了一圈回来了，同时下标置0
        if(tail == data.length){
            tail = 0;
        }
        this.notify(); //唤醒（小编解释了在上文中）

        size++; // 有效元素++
    }

       synchronized  public String take() throws InterruptedException {
        while (size == 0){
            this.wait();
        }
        // 队列不为空，就把对首元素删除，先保存一下
        String ret = data[head];
        head++;
        if(head == data.length){
            head = 0;
        }
        size--;
        this.notify();
        return ret;
    }
