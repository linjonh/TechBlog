    <h3>
     <a id="HarmonyOS_0">
     </a>
     在学习HarmonyOS时，看到了原子化服务的概念，完整整理下它在这个系统中的意义，以下内容均来自官方文档
    </h3>
    <h4>
     <a id="_1">
     </a>
     含义
    </h4>
    <p>
     <strong>
      什么是原子化服务？
     </strong>
    </p>
    <p>
     <strong>
      背景
     </strong>
     ：随着万物互联时代的到来，设备和场景的多样性，应用开发变得复杂、应用入口更加丰富。在这个背景下，需要一种新的服务提供方式，让服务的提供者和用户获取和使用服务变得更加便捷。因此，“原子化服务”，也就是提供免于安装而提供服务的应用应运而生。
    </p>
    <p>
     <strong>
      定义
     </strong>
     ：原子化服务是HarmonyOS提供的一种面向未来的服务提供方式，是有独立入口的（用户可通过点击方式直接触发）、免安装的（无需显示安装，由系统程序框架后台安装后即可使用）、可为用户提供一个或多个便捷服务的用户应用程序形态。
     <br/>
     原子化服务由1个或多个HAP包组成，1个HAP包对应1个FA或1个PA。每个FA或PA均可独立运行，完成特定功能；1个或多个功能（对应FA或PA）完成1个特定的便捷服务。（下图来自
     <a href="https://developer.harmonyos.com/cn/docs/documentation/doc-guides/atomic-service-definition-0000001090840664" rel="nofollow">
      官方文档
     </a>
     ）
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/ab1256639ca4f6e089a5f7d5ca79824e.png#pic_center"/>
    </p>
    <h4>
     <a id="_9">
     </a>
     原子化服务设计
    </h4>
    <p>
     以下内容均只提取关键字，详细内容请查看
     <a href="https://developer.harmonyos.com/cn/docs/design/des-guides/service-overview-0000001139795693" rel="nofollow">
      设计文档
     </a>
     。
     <br/>
     <strong>
      原子化服务基本要素
     </strong>
    </p>
    <ul>
     <li>
      基础信息：每个原子化服务有独立的图标、名称、描述、快照。基础信息将根据场景在服务中心、系统设置等界面展示。
     </li>
     <li>
      服务卡片：为了给用户提供便捷、智能的服务体验，每个原子化服务都需要开发至少一个服务卡片，每个应用可选配置服务卡片。卡片作为服务的轻量承载，需要做到易用可见、智能可选和多端可变。
      <br/>
      <strong>
       体验特性
      </strong>
     </li>
     <li>
      多种入口：在HarmonyOS Connect标签的支持下，原子化服务可以通过线上和线下发现。用户也可以在设备的服务中心、桌面等界面发现并管理原子化服务。
     </li>
     <li>
      服务流转：在HarmonyOS中泛指涉及多端的分布式操作。流转能力打破设备界限，多设备联动，使原子化服务可分可合、可流转，实现如邮件跨设备编辑、多设备协同健身、多屏游戏等分布式业务。
     </li>
     <li>
      服务分享：原子化服务支持通过面对面或网络途径分享给其他用户。
     </li>
     <li>
      设备控制：HarmonyOS尽力为用户打造简单易用、自然流畅的设备控制功能。用户可通过手机便捷地连接和控制HarmonyOS设备，用手机“碰一碰”HarmonyOS设备的NFC标签，识别后手机将运行对应的原子化服务。
     </li>
    </ul>
    <h4>
     <a id="_19">
     </a>
     三方应用调用管控机制
    </h4>
    <p>
     <strong>
      为什么要管控？因为浪费资源
     </strong>
     <br/>
     <strong>
      相关概念
     </strong>
    </p>
    <ul>
     <li>
      前台：用户应用程序有可见的FA正在显示，则认为用户应用程序在前台。
     </li>
     <li>
      用户应用程序内调用：同一用户应用程序内的FA、PA之间的访问。
     </li>
    </ul>
    <p>
     <strong>
      调用管控总体思路
     </strong>
    </p>
    <ul>
     <li>
      用户应用程序内部调用不管控。
     </li>
     <li>
      三方用户应用程序间调用严格管控：禁止三方用户应用程序在后台调用其他三方应用的PA；严格管控三方用户应用程序在前台调用其他用户应用程序的PA。
     </li>
    </ul>
    <p>
     <strong>
      管控规则
     </strong>
    </p>
    <ul>
     <li>
      用户应用程序内调用不管控。
     </li>
     <li>
      三方用户应用程序间调用。三方应用程序A调用三方应用程序B的PA，具体限制如下：
      <br/>
      1.禁止A在后台调用B的PA。
      <br/>
      2.当B有进程存活时，允许A在前台调用B的PA；当B无进程存活时，禁止A的调用。
     </li>
    </ul>
    <h4>
     <a id="_34">
     </a>
     原子化服务总体开发要求
    </h4>
    <p>
     <strong>
      总体开发要求
     </strong>
    </p>
    <ol>
     <li>
      原子化服务内所有HAP包（包括Entry HAP和Feature HAP）均需满足免安装要求。（说明：原子化服务由一个或多个HAP包组成，1个HAP包对应1个FA或1个PA。）
     </li>
    </ol>
    <ul>
     <li>
      免安装的HAP包不能超过10MB，以提供秒开体验。超过此大小的HAP包不符合免安装要求，也无法在服务中心露出。
     </li>
     <li>
      DS(DevEco Stdio)创建原子化服务时，Project Type字段选择Service。
     </li>
     <li>
      现有原子化服务的新版本不可失去免安装属性，否则，不允许上架。
     </li>
     <li>
      支持免安装HAP包的设备类型如下：
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/bbdfceeb29e26af7296d256564250e07.png#pic_center"/>
     </li>
    </ul>
    <ol start="2">
     <li>
      如果某便捷服务的入口需要在服务中心露出，则该服务对应HAP包必须包含FA，且FA中必须指定一个唯一的mainAbility（定位为用户操作入口），mainAbility必须为Page Ability。同时，mainAbility中至少配置2*2（小尺寸）规格的默认服务卡片（也可以同时提供其他规格的卡片）及该便捷服务对应的基础信息（包括图标、名称、描述、快照）。
     </li>
    </ol>
    <ul>
     <li>
      通过DS创建工程时，Project Type字段选择Service，同时勾选“show in Service Center”。将自动指定mainActivity，并添加卡片信息。
     </li>
     <li>
      config.json配置。
     </li>
    </ul>
    <h4>
     <a id="_46">
     </a>
     服务卡片概述
    </h4>
    <p>
     服务卡片就是以卡片的形式为服务提供调用入口的一种展示形式，例如，常见手机上的天气卡片，显示在launcher上，点击后进入天气app。重要的角色有：
    </p>
    <ul>
     <li>
      卡片提供方
     </li>
     <li>
      卡片使用方
     </li>
     <li>
      卡片管理服务
     </li>
    </ul>
    <p>
     <strong>
      运作机制
     </strong>
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/1fd540452fd12fd61b48c3b9d93d48bd.png#pic_center"/>
    </p>
    <ul>
     <li>
      卡片管理服务包含以下模块：周期性刷新、卡片缓存管理、卡片生命周期管理、卡片使用方对象管理、通信适配层（RPC通信）
     </li>
     <li>
      卡片提供方包含以下模块：卡片服务、卡片提供方实例管理模块、通信适配层。
      <br/>
      <a href="https://developer.harmonyos.com/cn/docs/documentation/doc-guides/ability-service-widget-overview-0000001062607955" rel="nofollow">
       详见文档
      </a>
     </li>
    </ul>
    <h4>
     <a id="_57">
     </a>
     流转概述
    </h4>
    <p>
     <strong>
      介绍
     </strong>
     <br/>
     就是不同类型设备取长补短，整合为一个超级终端。
     <br/>
     <strong>
      基本概念
     </strong>
    </p>
    <ul>
     <li>
      流转：在HarmonyOS中泛指多设备分布式操作。流转按照体验可以分为跨端迁移和多端协同。
     </li>
     <li>
      跨端迁移：也就是在A端运行的FA迁移到B端上，完成迁移后B端继续执行任务，A端应用退出。
     </li>
     <li>
      多端协同：多端上的不同FA/PA同时运行、或者交替运行完成整个的业务；或者，多端上相同的FA/PA同时运行实现完整的业务。
     </li>
    </ul>
