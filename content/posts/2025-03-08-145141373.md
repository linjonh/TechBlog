    <h2 id="%E4%B8%80%EF%BC%9A%E7%A3%81%E7%9B%98%E4%B8%8E%E7%89%A9%E7%90%86%E5%86%85%E5%AD%98%E7%9A%84%E6%95%B0%E6%8D%AE%E4%BA%A4%E4%BA%92" name="%E4%B8%80%EF%BC%9A%E7%A3%81%E7%9B%98%E4%B8%8E%E7%89%A9%E7%90%86%E5%86%85%E5%AD%98%E7%9A%84%E6%95%B0%E6%8D%AE%E4%BA%A4%E4%BA%92">
     一：磁盘与物理内存的数据交互
    </h2>
    <p>
     <img alt="" height="653" src="https://i-blog.csdnimg.cn/direct/3641937de79d46f8907e3151e53c5551.png" width="1249"/>
    </p>
    <p>
     物理内存与磁盘之间的数据交换一般是以
     <span style="background-color:#fef2f0">
      4KB
     </span>
     为单位来交互的。
     <strong>
      页框
     </strong>
     是物理内存的划分单位，用于实际存储数据。
     <strong>
      页帧
     </strong>
     是虚拟内存的划分单位，通过页表映射到页框。那么为什么要使用
     <span style="background-color:#fef2f0">
      4 KB
     </span>
     的大小来交互呢？
    </p>
    <ul>
     <li>
      <strong>
       硬件层面上，减少 IO 的次数，即减少访问外设的次数来提高效率。
      </strong>
      例如，一次访问 4KB 数据和 4 次访问 1KB 数据相比，明显的前者效率更加高效。因为磁盘是一个机械设备，访问周期很长。一次访问4KB的 CPU 只与磁盘交互一次，磁头和盘面只需进行一次定位就可读取 4KB 的内容，后者4次读取 1KB 的内容，且这四次可能不连续，意味着磁头和盘面要定位4次，大量的机械运动就导致了效率的下降。
     </li>
     <li>
      <strong>
       软件层面上，基于局部性原理的预加载机制。
      </strong>
      尽管当前 CPU 和磁盘只需要访问50字节数据，但是 OS 和磁盘之间还是以 4KB 为单位来数据交互将数据加载到内存。
     </li>
    </ul>
    <h2 id="%E4%BA%8C%EF%BC%9A%E6%93%8D%E4%BD%9C%E7%B3%BB%E7%BB%9F%E5%AF%B9%E7%89%A9%E7%90%86%E5%86%85%E5%AD%98%E7%9A%84%E7%AE%A1%E7%90%86" name="%E4%BA%8C%EF%BC%9A%E6%93%8D%E4%BD%9C%E7%B3%BB%E7%BB%9F%E5%AF%B9%E7%89%A9%E7%90%86%E5%86%85%E5%AD%98%E7%9A%84%E7%AE%A1%E7%90%86">
     二：操作系统对物理内存的管理
    </h2>
    <p>
     操作系统是一定能看到物理内存的，即能看到物理内存的物理地址的。而对物理内存的管理，既然要管理，还是要
     <strong>
      "先描述，再组织"
     </strong>
     ，在OS内核中有一个结构体 struct page 用来描述物理内存的。
    </p>
    <p>
     一个 struct page 对象就对应一个 4KB 的内存空间，对象内部页记录了一些
     <span style="background-color:#fef2f0">
      4KB
     </span>
     的相关属性信息【当前页框状态，引用计数等等】。所以操作系统就可以将物理内存划分为一个个的 struct page 对象，用数组的形式组织起来，数组的下标为对应的页号。假设共有 4GB 的内存，那么
     <span style="background-color:#fef2f0">
      【4*1024*1024*1024*4 / (4*1024)
     </span>
     】，大约等于100w左右，即在
     <span style="background-color:#fef2f0">
      4GB
     </span>
     的物理内存中大约含有100w左右个
     <span style="background-color:#fef2f0">
      struct page
     </span>
     对象。
    </p>
    <p>
     随便给一个地址【一个地址一个字节】，求它所对应的页号。
     <span style="background-color:#fef2f0">
      物理地址 / (4KB = 4*1024字节)
     </span>
     ，即使用该地址按位与上
     <span style="background-color:#fef2f0">
      0xFFFFF000
     </span>
     ，将低12为清零就是该地址所对应的页号。找到对应的页号了，就能在系统中找到对应的物理页框。所有申请内存的动作，都是在访问
     <span style="background-color:#fef2f0">
      page
     </span>
     数据，对这个数组进行增删查改。
    </p>
    <p>
     <span style="background-color:#fef2f0">
      struct page
     </span>
     对象内部有一个
     <span style="background-color:#fef2f0">
      flags
     </span>
     属性，通过
     <span style="background-color:#fef2f0">
      flags
     </span>
     属性来判断当前页是否被使用，即获取当前页框的状态。若没有被使用，申请内存时，flags置为1，那么就表示当前页框已经被申请成功了。
    </p>
    <h2 id="%E4%B8%89%EF%BC%9A%E5%86%8D%E6%AC%A1%E7%90%86%E8%A7%A3%E6%96%87%E4%BB%B6%E7%9A%84%E6%89%93%E5%BC%80" name="%E4%B8%89%EF%BC%9A%E5%86%8D%E6%AC%A1%E7%90%86%E8%A7%A3%E6%96%87%E4%BB%B6%E7%9A%84%E6%89%93%E5%BC%80">
     三：再次理解文件的打开
    </h2>
    <h3 id="3.1%EF%BC%9A%E6%96%87%E4%BB%B6%E9%A1%B5%E7%BC%93%E5%86%B2%E5%8C%BA%E7%9A%84%E5%BC%95%E5%85%A5" name="3.1%EF%BC%9A%E6%96%87%E4%BB%B6%E9%A1%B5%E7%BC%93%E5%86%B2%E5%8C%BA%E7%9A%84%E5%BC%95%E5%85%A5">
     3.1：文件页缓冲区的引入
    </h3>
    <p>
     因为
     <span style="background-color:#fef2f0">
      struct file
     </span>
     是内核中的一个数据结构，用来描述一个被打开的文件。
     <span style="background-color:#fef2f0">
      inode
     </span>
     是磁盘那里引入的。磁盘上一个文件只对应一个
     <span style="background-color:#fef2f0">
      inode
     </span>
     ，内部包含文件的属性。那么它们两者之间有什么关系呢？
    </p>
    <p>
     <span style="background-color:#fef2f0">
      struct file
     </span>
     只记录了文件的少量属性，所以在内核中还有一个
     <span style="background-color:#fef2f0">
      struct inode
     </span>
     结构体专门描述一个文件的所有属性。即在
     <span style="background-color:#fef2f0">
      struct file
     </span>
     中有一个指针字段来指向该文件的
     <span style="background-color:#fef2f0">
      struct inode
     </span>
     对象。
    </p>
    <p>
     <img alt="" height="954" src="https://i-blog.csdnimg.cn/direct/bf593fb25b5142bc96583451bcca5f27.png" width="1564"/>
    </p>
    <p>
     <img alt="" height="888" src="https://i-blog.csdnimg.cn/direct/3bb9400d4e8c48b4bab590a88861ad5a.png" width="1688"/>
    </p>
    <p>
     文件都是内容+属性的。在磁盘上文件属性使用
     <span style="background-color:#fef2f0">
      inode
     </span>
     来存储，内容使用
     <span style="background-color:#fef2f0">
      DataBlock
     </span>
     来存储，所以在内核中，使用
     <span style="background-color:#fef2f0">
      struct inode
     </span>
     来存储文件属性，使用
     <span style="background-color:#fef2f0">
      struct address_space
     </span>
     来存储文件的内容。【采用页缓冲区的方式来存储文件的内容，
     <span style="background-color:#fef2f0">
      struct file
     </span>
     结构体中有一个
     <span style="background-color:#fef2f0">
      struct address_space
     </span>
     字段的指针】，
     <span style="background-color:#fef2f0">
      struct address_space
     </span>
     内部有一个
     <span style="background-color:#fef2f0">
      struct radix_tree_root
     </span>
     的结构体对象，本质上就是一个字典树状结构。树内的每一个结点都是 struct radix_tree_node 类型。类型中有个
     <span style="background-color:#fef2f0">
      void* slots
     </span>
     属性。内部存的是
     <span style="background-color:#fef2f0">
      struct page
     </span>
     对象的地址。故此，在
     <span style="background-color:#fef2f0">
      struct file
     </span>
     结构体中的有物理内存的指针。就将这些物理内存称作是
     <strong>
      文件页缓冲区。
     </strong>
    </p>
    <p>
     <img alt="" height="825" src="https://i-blog.csdnimg.cn/direct/00ce4fae597748cc82df21ad261449ec.png" width="2017"/>
    </p>
    <h3 id="3.2%EF%BC%9A%E5%90%91%E6%96%87%E4%BB%B6%E4%B8%AD%E5%86%99%E5%85%A5%E6%95%B0%E6%8D%AE" name="3.2%EF%BC%9A%E5%90%91%E6%96%87%E4%BB%B6%E4%B8%AD%E5%86%99%E5%85%A5%E6%95%B0%E6%8D%AE">
     3.2：向文件中写入数据
    </h3>
    <p>
     在向文件中写入的时候，首先将内容写入到用户缓冲区。然后再在合适的是时候将内容刷新到页缓冲区中。之后再在合适的时候刷新到外设。这些操作全部都是由 IO 子系统来执行的，进程不会关心。所以在 OS 上肯定会存在着非常多的 IO 操作，因为可能会有很多进程都要将数据写入磁盘中，所以操作系统会将所有的 IO 操作管理起来（"先描述，再组织"）。内核中有 struct request 结构专门来管理 IO 操作的，有时为了提高效率，OS 还会将 IO排序和 IO 合并，以减少磁盘和盘面的定位次数。
    </p>
    <blockquote>
     <p>
      <strong>
       在 Linux 中，我们的每一个进程，打开的每一个文件都要有自己的 inode 属性和自己的文件页缓冲区。
      </strong>
     </p>
    </blockquote>
