    <h3>
     <a id="_0">
     </a>
     概述
    </h3>
    <blockquote>
     <p>
      <strong>
       桥接模式
      </strong>
      ：
      <strong>
       Bridge Pattern
      </strong>
      ： 是一种结构型设计模式。
      <br/>
      旨在将
      <code>
       抽象部分
      </code>
      与
      <code>
       实现部分
      </code>
      分离，使它们可以独立变化。
      <br/>
      它通过组合代替继承，
      <strong>
       解决类爆炸
      </strong>
      问题，并提高系统的灵活性和可扩展性。
     </p>
    </blockquote>
    <h3>
     <a id="_5">
     </a>
     组成部分
    </h3>
    <blockquote>
     <p>
      <strong>
       【抽象部分】（Abstraction）
      </strong>
      ：定义高层控制逻辑，并包含一个对实现部分的引用。
      <br/>
      <strong>
       扩展抽象部分（Refined Abstraction）
      </strong>
      ：对抽象部分的扩展，提供更多控制逻辑。
      <br/>
      <strong>
       【实现部分】（Implementor）
      </strong>
      ：定义底层操作的接口，供抽象部分调用。
      <br/>
      <strong>
       具体实现部分（Concrete Implementor）
      </strong>
      ：实现实现部分的接口，提供具体操作。
     </p>
    </blockquote>
    <h3>
     <a id="_10">
     </a>
     一个背景案例
    </h3>
    <blockquote>
     <p>
      画图 ： 画 不同形状的、不同颜色的 图形。
      <br/>
      例如 ： 三角形-红色的；三角形-绿色的；圆形-红色的；圆形-绿色的；
      <br/>
      分析 ： 把
      <code>
       形状定位为 抽象的部分
      </code>
      ，把
      <code>
       填充颜色定位为具体实现部分
      </code>
      ；画图形的时候，组合颜色一并填充。
      <br/>
      <strong>
       当需要新增 形状 or 颜色 的时候，只需要新增一个类即可，在使用时可以自由组合。
      </strong>
      <br/>
      实际上，
      <code>
       此处的这个 Shap 抽象类，就相当于一个桥的作用
      </code>
      。
      <br/>
      类图结构如下：
     </p>
    </blockquote>
