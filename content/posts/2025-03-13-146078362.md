    <h2>
     <a id="RPC_0">
     </a>
     RPC
    </h2>
    <p>
     <strong>
      Remote Procedure Call
     </strong>
     ，远程过程调用，是用来屏蔽分布式计算中的各种调用细节，使得调用远端的方法就像调用本地的一样。
    </p>
    <h3>
     <a id="_2">
     </a>
     客户端与服务端沟通的过程
    </h3>
    <ol>
     <li>
      客户端发送数据(以字节流的方式)；
      <strong>
       （
       <mark>
        编码
       </mark>
       ）
      </strong>
     </li>
     <li>
      服务端接受并解析，根据约定知道要执行什么，然后把结果返回给客户。
      <strong>
       （
       <mark>
        解码
       </mark>
       ）
      </strong>
     </li>
    </ol>
    <blockquote>
     <p>
      <strong>
       RPC
      </strong>
      ：
     </p>
     <ol>
      <li>
       RPC就是将上述过程封装下，使其操作更加优化；
      </li>
      <li>
       使用一些大家都认可的协议，使其规范化；
      </li>
      <li>
       做成一些框架，直接或间接产生利益。
      </li>
     </ol>
    </blockquote>
    <h3>
     <a id="RPC__HTTP__10">
     </a>
     RPC 和 HTTP 区别
    </h3>
    <p>
     RPC 和 HTTP
     <mark>
      不是对等
     </mark>
     的概念。
    </p>
    <ul>
     <li>
      RPC 是一个
      <strong>
       完整的远程调用方案
      </strong>
      ，它包括了：接口规范 + 序列化反序列化规范 + 通信协议等。
     </li>
     <li>
      HTTP 只是一个
      <strong>
       通信协议
      </strong>
      ，工作在 OSI 的第七层，不是一个完整的远程调用方案。
     </li>
    </ul>
    <blockquote>
     <p>
      RPC是能够基于 HTTP 实现，也可以不基于，基于更下一层的 TCP/ UDP协议。
     </p>
    </blockquote>
    <h3>
     <a id="RPC__17">
     </a>
     RPC 原理
    </h3>
    <ul>
     <li>
      调用方为
      <code>
       client
      </code>
      ，被调用方为
      <code>
       server
      </code>
      。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/47b32a178fb54b6e951336dad5980328.png">
       <br/>
       ① 服务调用方（client）以本地调用方式调用服务；
      </img>
     </li>
    </ul>
    <p>
     ② client stub 接收到调用后负责将方法、参数等组装成能够进行网络传输的消息体；
    </p>
    <p>
     ③ client stub 找到服务地址，并将消息发送到服务端；
    </p>
    <p>
     ④ server 端接收到消息；
    </p>
    <p>
     ⑤ server stub 收到消息后进行解码；
    </p>
    <p>
     ⑥ server stub 根据解码结果调用本地的服务；
    </p>
    <p>
     ⑦ 本地服务执行并将结果返回给 server stub；
    </p>
    <p>
     ⑧ server stub 将返回结果打包成能够进行网络传输的消息体；
    </p>
    <p>
     ⑨ 按地址将消息发送至调用方；
    </p>
    <p>
     ⑩ client 端接收到消息；
    </p>
    <p>
     ⑪ client stub 收到消息并进行解码；
    </p>
    <p>
     ⑫ 调用方得到最终结果。
    </p>
    <h3>
     <a id="RPC__http__44">
     </a>
     RPC 接口和传统的 http 接口的区别
    </h3>
    <h4>
     <a id="_45">
     </a>
     传输协议
    </h4>
    <ul>
     <li>
      <strong>
       RPC
      </strong>
      ：可以基于TCP协议，也可以基于HTTP协议。
     </li>
     <li>
      <strong>
       HTTP
      </strong>
      ：基于HTTP协议。
     </li>
    </ul>
    <h4>
     <a id="_49">
     </a>
     传输效率
    </h4>
    <ul>
     <li>
      <strong>
       RPC
      </strong>
      ：使用自定义的TCP协议，可以让请求报文体积更小，或者使用HTTP2协议，也可以很好的减少报文的体积，提高传输效率。
     </li>
     <li>
      <strong>
       HTTP
      </strong>
      ：如果是基于HTTP1.1的协议，请求中会包含很多无用的内容，如果是基于HTTP2.0，那么简单的封装以下是可以作为一个RPC来使用的，这时标准RPC框架更多的是服务治理。
     </li>
    </ul>
    <h4>
     <a id="_53">
     </a>
     服务治理
    </h4>
    <ul>
     <li>
      <strong>
       RPC
      </strong>
      ：能做到
      <mark>
       自动通知
      </mark>
      ，不影响上游。
     </li>
     <li>
      <strong>
       HTTP
      </strong>
      ：需要
      <mark>
       事先通知
      </mark>
      ，修改Nginx/HAProxy配置。
     </li>
    </ul>
    <h2>
     <a id="GRPC_57">
     </a>
     GRPC
    </h2>
    <p>
     gRPC 是一个高性能的、开源的、通用的 RPC 框架。
    </p>
    <blockquote>
     <ul>
      <li>
       grpc 只是一个
       <strong>
        框架
       </strong>
       ，是
       <strong>
        对rpc的封装
       </strong>
       ，类似的框架还有thrift等。
      </li>
      <li>
       目标是只需要关心第1步和最后1步，中间的其他步骤统统封装起来，让使用者无需关心。
      </li>
     </ul>
    </blockquote>
    <h3>
     <a id="grpc__rpc__62">
     </a>
     grpc 解决的 rpc 三大问题
    </h3>
    <h4>
     <a id="_63">
     </a>
     协议约定
    </h4>
    <p>
     gRPC 的协议是 Protocol Buffers，是一种压缩率极高的序列化协议，Google 在 2008 年开源了 Protocol Buffers，支持多种编程语言，所以
     <strong>
      gRPC 支持客户端与服务端可以用不同语言实现
     </strong>
     。
    </p>
    <h4>
     <a id="_66">
     </a>
     传输协议
    </h4>
    <p>
     gRPC 的数据传输用的是 Netty Channel， Netty 是一个高效的基于异步 IO 的网络传输架构。Netty Channel 中，每个 gRPC 请求封装成 HTTP 2.0 的 Stream。
    </p>
    <h4>
     <a id="_69">
     </a>
     服务发现
    </h4>
    <p>
     gRPC 本身没有提供服务发现的机制，需要通过其他组件。
    </p>
    <h3>
     <a id="Protocol_Buffs_72">
     </a>
     Protocol Buffs
    </h3>
    <p>
     Protocol Buffss 是谷歌开源的一套成熟的数据结构序列化机制。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/5403f2ebd1c7433f984637883fa6467a.png"/>
    </p>
    <blockquote>
     <ul>
      <li>
       序列化：将数据结构或对象转换成二进制串的过程。
      </li>
      <li>
       反序列化：将在序列化过程中所产生的二进制串转换成数据结构或者对象的过程。
      </li>
     </ul>
    </blockquote>
    <p>
     protobuf 是谷歌开源的一种数据格式，适合高性能，对响应速度有要求的数据传输场景。因为profobuf是
     <strong>
      二进制数据格式
     </strong>
     ，需要编码和解码。数据本身不具有可读性。因此只能反序列化之后得到真正可读的数据。
    </p>
    <blockquote>
     <p>
      优势
     </p>
     <ol>
      <li>
       序列化后体积相比Json和XML很小，适合网络传输
      </li>
      <li>
       支持跨平台多语畜
      </li>
      <li>
       消息格式升级和兼容性还不错
      </li>
      <li>
       序列化反序列化速度很快
      </li>
     </ol>
    </blockquote>

    <h3>
     <a id="proto_126">
     </a>
     proto文件介绍
    </h3>
    <h4>
     <a id="message_127">
     </a>
     message
    </h4>
    <ul>
     <li>
      protobuf 中定义一个消息类型式是通过关键字 message 字段指定的。
     </li>
     <li>
      <mark>
       消息就是需要传输的数据格式的定义。
      </mark>
     </li>
     <li>
      message 关键字类似于 C++ 中的 class，JAVA 中的 class，go 中的 struct。
     </li>
     <li>
      在消息中承载的数据分别对应于每一个字段，其中每个字段都有一个名字和一种类型。
     </li>
     <li>
      一个 proto 文件中可以定义多个消息类型。
     </li>
    </ul>
    <p>
     <code>
      hello.proto
     </code>
     ：
    </p>
    <pre><code class="prism language-protobuf">// message关键字，可以理解为Golang中的结构体
// 这里比较特别的是变量后面的“赋值”。
// 注意，这里并不是赋值，而是在定义这个变量在这个message中的位置。
message HelloRequest {
  string requestName = 1;
//  int64 age = 2;
}

message HelloResponse {
  string responseMsg = 1;
}
</code></pre>
    <h4>
     <a id="_148">
     </a>
     字段规则
    </h4>
    <ul>
     <li>
      <strong>
       required
      </strong>
      ：消息体中必填字段，不设置会导致编码异常。在 protobuf2 中使用，在 protobuf3 中被删去。
     </li>
     <li>
      <strong>
       optional
      </strong>
      ：消息体中可选字段。protobuf3 没有了required，optional 等说明关键字，都默认为optional 。
     </li>
     <li>
      <strong>
       repeate
      </strong>
      ：消息体中可重复字段，重复的值的顺序会被保留在go中重复的会被定义为切片。
     </li>
    </ul>
    <p>
     <code>
      hello.proto
     </code>
     ：
    </p>
    <pre><code class="prism language-protobuf">message HelloRequest {
  string requestName = 1;
  int64 age = 2;
  repeated string name = 3;
}
</code></pre>
    <p>
     执行
     <code>
      protoc --go_out=. hello.proto
     </code>
     <br/>
     <code>
      hello.pb.go
     </code>
     ：
    </p>

    <h4>
     <a id="_173">
     </a>
     消息号
    </h4>
    <p>
     在消息体的定义中，每个字段都必须要有一个唯一的标识号，标识号是 [1，2^29-1] 范围内的
     <strong>
      一个整数
     </strong>
     。
    </p>
    <h4>
     <a id="_176">
     </a>
     嵌套消息
    </h4>
    <p>
     可以在其他消息类型中定义、使用消息类型。
    </p>
    <pre><code class="prism language-protobuf">// Person 消息定义在 PersonInfo 消息内
message PersonInfo {
	    message Person{
  			     string name = 1;
  				 int32 height = 2;
  				 repeated int32 weight = 3;
}
repeated Person info = 1;
</code></pre>
    <p>
     如果要在它的父消息类型的外部重用这个消息类型，需要Personlnfo.Person的形式使用它。
    </p>
    <pre><code class="prism language-protobuf">message PersonMessage {
	    message Person{
  			     PersonInfo.Person info = 1;
}
</code></pre>
    <h4>
     <a id="_195">
     </a>
     服务定义
    </h4>
    <p>
     如果想要将消息类型用在 RPC 系统中，可以在
     <mark>
      .proto 文件中定义一个 RPC 服务接口
     </mark>
     ，protocol buffer编译器将会根据所选择的不同语言生成服务接口代码及存根。
    </p>
    <pre><code class="prism language-properties">service searchservicef 
	  # rpc 服务函数名 (参数) 返回 (返回参数)
	  rpc search(searchRequest) returns (searchResponse)
}
</code></pre>
    <p>
     定义了一个RPC服务，该方法接受 SearchRequest 返回 SearchResponse 。
    </p>

    <h2>
     <a id="_288">
     </a>
     认证-安全传输
    </h2>
    <p>
     gRPC 是一个典型的 C/S 模型，需要开发客户端和服务端，客户端与服务端需要达成协议，使用某一个确认的传输协议来传输数据，
     <strong>
      gRPC 通常默认是使用 protobuf 来作为传输协议
     </strong>
     ，当然也是可以使用其他自定义的。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/7b20df1c8b13459695bf8b1282101055.png"/>
    </p>
    <p>
     客户端与服务端要通信之前，客户端如何知道自己的数据是发给哪一个明确的服务端呢？反过来，服务端是不是也需要有一种方式来弄个清楚自己的数据要返回给谁呢？
     <br/>
     那么就不得不提 gRPC 的认证。
     <br/>
     此处说到的认证，不是用户的身份认证，而是指
     <strong>
      多个 server 和多个 client 之间，如何识别对方是谁，并且可以安全的进行数据传输
     </strong>
     。
    </p>
    <ul>
     <li>
      SSL/TLS 认证方式 (采用 http2 协议)
     </li>
     <li>
      基于 Token 的认证方式 (基于安全连接)
     </li>
     <li>
      不采用任何措施的连接，这是不安全的连接 (默认采用 http1)
     </li>
     <li>
      自定义的身份认证
     </li>
    </ul>
    <p>
     <strong>
      客户端和服务端之间调用，可以通过加入证书的方式，实现调用的安全性。
     </strong>
    </p>
    <h3>
     <a id="SSLTLS_302">
     </a>
     SSL/TLS认证方式
    </h3>
    <p>
     TLS (Transport Layer Security，安全传输层)，TLS 是建立在传输层 TCP 协议之上的协议，服务于应用层，它的前身是 SSL (Secure Socket Layer 安全套接字层)，它实现了将应用层的报文进行加密后再交由 TCP 进行传输的功能。
    </p>
    <p>
     TLS 协议主要解决如下三个网络安全问题。
    </p>
    <ol>
     <li>
      <strong>
       保密 (message privacy)
      </strong>
      ，保密通过加密 encryption 实现，所有信息都加密传输，第三方无法嗅探；
     </li>
     <li>
      <strong>
       完整性 (message integrity)
      </strong>
      ，通过 MAC 校验机制，一旦被篡改，通信双方会立刻发现；
     </li>
     <li>
      <strong>
       认证(mutual authentication)
      </strong>
      ，双方认证,双方都可以配备证书，防止身份被冒充；
     </li>
    </ol>
    <blockquote>
     <p>
      生产环境可以购买证书（eg. 阿里云、腾讯）或者使用一些平台发放的免费证书（eg. 宝塔）
     </p>
    </blockquote>
    <ul>
     <li>
      <mark>
       key
      </mark>
      ：服务器上的私钥文件，用于对发送给客户端数据的加密，以及对从客户端接收到数据的解密。
     </li>
     <li>
      <mark>
       csr
      </mark>
      ：证书签名请求文件，用于提交给证书颁发机构(CA)对证书签名。
     </li>
     <li>
      <mark>
       crt
      </mark>
      ：由证书颁发机构(CA)签名后的证书，或者是开发者自签名的证书，包含证书持有人的信息，持有人的公钥，以及签署者的签名等信息。
     </li>
     <li>
      <mark>
       pem
      </mark>
      ：是基于 Base64 编码的证书格式，扩展名包括 PEM、CRT 和 CER 。
     </li>
    </ul>
    <p>
     首先通过 openssl 生成证书和私钥。
    </p>

    <h2>
     <a id="Token_441">
     </a>
     Token认证
    </h2>
    <p>
     先看一个 gRPC 提供的一个接口，这个接口中有两个方法，接口位于 credentials 包下，这个接口需要客户端来实现
    </p>
    <pre><code class="prism language-powershell"><span class="token function">type</span> PerRPCCredentials interface <span class="token punctuation">{<!-- --></span>
	GetRequestMetadata<span class="token punctuation">(</span>ctx context<span class="token punctuation">.</span>Context<span class="token punctuation">,</span> uri <span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">.</span>string<span class="token punctuation">)</span><span class="token punctuation">(</span>map<span class="token namespace">[string]</span>string<span class="token punctuation">,</span> error<span class="token punctuation">)</span>
	RequireTransportSecurity bool
<span class="token punctuation">}</span>
</code></pre>
    <ul>
     <li>
      第一个方法作用是获取元数据信息，也就是客户端提供的key,value对，context 用于控制超时和取消，uri 是请求入口处的 uri
     </li>
     <li>
      第二个方法的作用是否需要基于 TLS 认证进行安全传输，如果返回值是 true，则必须加上 TLS 验证，返回值是 false 则不用
     </li>
    </ul>
    <p>
     gRPC 将各种认证方式浓缩统一到一个凭证(credentials)上，可以单独使用一种凭证，比如只使用 TLS 凭证或者只使用自定义凭证，也可以多种凭证组合，gRPC 提供统一的 API 验证机制，使研发人员使用方便，这也是 gRPC 设计的巧妙之处。
    </p>
