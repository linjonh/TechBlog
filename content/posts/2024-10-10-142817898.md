    <h3>
     <a id="_2">
     </a>
     管理流程
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ecef41691035480387103e852490c502.png"/>
    </p>
    <p>
     信息系统软件运维服务的四个关键要素是：
     <code>
      人员、资源、技术和过程
     </code>
     ，每个要素通过关键 指标反映运维服务的能力。
    </p>
    <h4>
     <a id="_11">
     </a>
     人员
    </h4>
    <p>
     确保提供信息系统软件运维服务的相关人员具备应有的运维服务能力，主要从人员管理、 岗位结构以及知识、技能和经验等方面进行衡量。
    </p>
    <h5>
     <a id="_15">
     </a>
     人员管理
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        人员规划
       </strong>
       。针对新的或扩展的运维服务人员配备需求，考虑因使用新技术，或因预期 的人员离职所导致的人员缺口，制定相应的策略。
      </p>
     </li>
     <li>
      <p>
       <strong>
        人员培训
       </strong>
       。运维服务的人员应充分了解运维体系相关的知识，而且可能还会不断涉及 新的理念、新的流程、新的工具。
      </p>
      <p>
       培训的内容如下。
      </p>
      <ul>
       <li>
        <code>
         理论培训
        </code>
        ：对运维服务的理论模型和标准规范进行培训。
       </li>
       <li>
        <code>
         流程培训
        </code>
        ：对实际运维过程中的各种具体流程的操作实践进行培训。
       </li>
       <li>
        <code>
         工具培训
        </code>
        ：对运维过程中使用的各种工具进行培训。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        绩效考核
       </strong>
       。应建立与运行维护服务相关的绩效考核体系或机制，并能够有效组织实施。
      </p>
     </li>
    </ul>
    <h5>
     <a id="_29">
     </a>
     岗位结构
    </h5>
    <p>
     应设置专职团队负责信息系统软件运维服务的工作，对运行维护服务中的不同角色有明确 分工和职责定义。 一个完整的运行维护服务团队应包括管理、技术支持、操作等主要岗位。
    </p>
    <ul>
     <li>
      <p>
       管理岗职责如下。
      </p>
      <ul>
       <li>
        在运行维护服务中负责管理运行维护服务；
       </li>
       <li>
        与需方建立顺畅的沟通渠道，准确地将需方的需求传递到运行维护服务团队；
       </li>
       <li>
        规划、检查运行维护服务的各个过程，对运行维护服务能力的策划、实施、检查、改进的范围、过程、信息安全和成果负责。
       </li>
      </ul>
     </li>
     <li>
      <p>
       技术支持岗职责如下。
      </p>
      <ul>
       <li>
        <p>
         在运行维护服务中负责技术支持，包括网络、操作系统、数据库、中间件、应用开发、 信息安全等；
        </p>
       </li>
       <li>
        <p>
         对运行维护服务过程中的问题做出响应，保障信息安全并对处理结果负责。
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       操作岗职责如下。
      </p>
     </li>
     <li>
      <p>
       在运行维护服务中负责日常操作的实施；
      </p>
     </li>
     <li>
      <p>
       根据规范和手册，执行运行维护服务的过程，并对其执行结果负责。
      </p>
     </li>
    </ul>
    <h5>
     <a id="_50">
     </a>
     知识、技能和经验
    </h5>
    <p>
     保证信息系统运维服务人员应具备运维服务相关的知识、技能和经验。
    </p>
    <h4>
     <a id="_54">
     </a>
     资源
    </h4>
    <p>
     资源配置为信息系统运维服务提供基础保障，主要包括
     <code>
      服务台管理、运维工具管理和知识库管理
     </code>
     三个方面。
    </p>
    <ul>
     <li>
      <strong>
       服务台管理
      </strong>
      <ul>
       <li>
        <code>
         设置专门的沟通渠道作为与客户的联络点
        </code>
        ；
       </li>
       <li>
        <code>
         设定专人负责接收服务请求
        </code>
        ；
       </li>
       <li>
        <code>
         建立服务台的管理制度
        </code>
        ，包括服务请求的接收、记录、跟踪和反馈等机制，以及日常 工作的监督和考核。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       运维工具
      </strong>
      <ul>
       <li>
        <code>
         监控工具
        </code>
        ：对信息系统软件的运行状态、配置参数，操作系统的CPU 、内存、磁盘、网络状态等进行实时监控，使运维服务由被动型向主动型发展；
       </li>
       <li>
        <code>
         过程管理工具
        </code>
        ：使信息系统软件运维服务组织能够对运维服务过程进行系统化管理， 包括日常运行维护管理、记录、测量、监督和评估等功能。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       知识库管理
      </strong>
      <ul>
       <li>
        <code>
         提升服务台解决问题的能力
        </code>
        。
       </li>
       <li>
        <code>
         降低对关键技术人员的依赖
        </code>
        。
       </li>
       <li>
        <code>
         提高故障、缺陷恢复的效率
        </code>
        。
       </li>
       <li>
        <code>
         有助于团队整体技能水平的提高
        </code>
        。
       </li>
      </ul>
     </li>
    </ul>
    <h4>
     <a id="_71">
     </a>
     技术
    </h4>
    <p>
     在信息系统软件运维服务实施过程中，可能面临各种问题、风险以及新技术和前沿技术用所提出的新要求。应根据需求或技术发展趋势，具备发现和解决问题、风险控制、技术储备 以及研发、应用新技术和前沿技术的能力。
    </p>
    <h4>
     <a id="_75">
     </a>
     过程
    </h4>
    <p>
     信息系统软件运维的过程主要包括：
     <code>
      日常运维、缺陷诊断与修复、配置管理、变更管理、 系统恢复管理、发布管理等
     </code>
     。
    </p>
    <h4>
     <a id="_79">
     </a>
     运维策划
    </h4>
    <p>
     信息系统软件运维策划是指对信息系统软件运维活动过程中的服务内容、组织、资源、标 准、改进等进行全局策划，以确保信息系统软件运维活动顺利高效完成，具体内容如下：
    </p>
    <ul>
     <li>
      <p>
       <strong>
        内容
       </strong>
       ：内容策划是根据信息系统软件所涉及的业务定位和管理范围，策划信息系统软 件运维服务对象的业务内容与要求，并形成服务目录。
      </p>
     </li>
     <li>
      <p>
       <strong>
        组织
       </strong>
       ：软件运维和软件开发一样，技术性强，要有完善的组织管理作为保证。
      </p>
     </li>
     <li>
      <p>
       <strong>
        资源
       </strong>
       ：资源策划是指对信息系统软件运维所涉及的人力资源、环境资源、财务资源、 技术资源、时间资源等的分析。
      </p>
     </li>
     <li>
      <p>
       <strong>
        标准
       </strong>
       ：信息系统软件运维工作涉及范围广，影响因素多，所以要用软件工程的方法， 结合信息系统软件运维的实际，制定出一套运维标准，包括运维流程、运维安全、运维各阶段 所要完成的文档、考核评估体系等。
      </p>
     </li>
     <li>
      <p>
       <strong>
        改进
       </strong>
       ：策划如何管理、审核并改进运维服务质量，并建立内部审核评估机制。
      </p>
     </li>
    </ul>
    <h4>
     <a id="_93">
     </a>
     运维实施
    </h4>
    <p>
     信息系统软件运维策划完成后，就应该按照整体策划实施。在具体实施过程中应保证：
    </p>
    <ul>
     <li>
      <p>
       <strong>
        制定满足整体策划的实施计划，并按计划实施
       </strong>
       。
      </p>
     </li>
     <li>
      <p>
       <strong>
        建立与客户的沟通协调机制
       </strong>
       。
      </p>
     </li>
     <li>
      <p>
       <strong>
        按照运维规范要求实施管理活动并记录，确保服务过程实施可追溯，服务结果可计量 或可评估
       </strong>
       。
      </p>
     </li>
     <li>
      <p>
       <strong>
        提交满足质量要求的交付物
       </strong>
       。
      </p>
     </li>
    </ul>
    <h4>
     <a id="_105">
     </a>
     运维检查
    </h4>
    <p>
     要做好以下工作:
    </p>
    <ul>
     <li>
      <p>
       <strong>
        定期评审运维过程及相关管理体系，以确保运维能力的适宜和有效
       </strong>
       。
      </p>
     </li>
     <li>
      <p>
       <strong>
        调查客户满意度，并对运维结果进行统计分析
       </strong>
       。
      </p>
     </li>
     <li>
      <p>
       <strong>
        检查各项指标的达成情况
       </strong>
       。
      </p>
     </li>
    </ul>
    <h4>
     <a id="_114">
     </a>
     运维改进
    </h4>
    <p>
     具体包括以下内容。
    </p>
    <ul>
     <li>
      <p>
       <code>
        建立信息系统运维管理改进机制
       </code>
       。
      </p>
     </li>
     <li>
      <p>
       <code>
        对不符合策划要求的运维行为进行总结分析
       </code>
       。
      </p>
     </li>
     <li>
      <p>
       <code>
        对未达成的运维指标进行调查分析
       </code>
       。
      </p>
     </li>
     <li>
      <p>
       <code>
        根据分析结果确定改进措施，分析评估结果中需要改进的项，确定改进目标，制定信 息系统软件运维管理改进计划，按照计划对改进结果和改进过程执行监控管理、评审并记录， 保留记录文档，以评估改进的有效性和持续性
       </code>
       。
      </p>
     </li>
    </ul>
    <h4>
     <a id="_126">
     </a>
     文档管理
    </h4>
    <p>
     信息系统软件运维文档是对运维服务参与各方主体从事信息系统软件运维实施及运维管 理提供决策支持的一种载体。
    </p>
    <p>
     信息系统软件运维文档管理应注意如下方面。
    </p>
    <ul>
     <li>
      <p>
       <strong>
        文档管理制度化
       </strong>
       。
      </p>
      <p>
       形成一整套完善的文档管理制度，根据这一套制度来协调控制、评 价信息系统软件运维中各类人员的工作。
      </p>
     </li>
     <li>
      <p>
       <strong>
        文档标准化
       </strong>
       、
       <strong>
        规范化
       </strong>
       。
      </p>
      <p>
       在信息系统软件运维前要选择或制定文档标准，在统一的标准 约束下来规范地建立各类文档。
      </p>
     </li>
     <li>
      <p>
       <strong>
        落实文档管理人员
       </strong>
       。
      </p>
      <p>
       应设专人负责集中保管与信息系统软件运维相关的文档，他人可 按一定的流程向文档管理员借阅文档。
      </p>
     </li>
     <li>
      <p>
       <strong>
        保持文档的一致性
       </strong>
       。
      </p>
      <p>
       信息系统软件在运维过程中如果修改了原来的需求和设计，但是 文档却没有进行同步修改，造成交付的文档与实际信息系统软件不一致，使用户在使用信息系 统软件参考文档对软件进行维护时出现许多误解，这将严重影响系统的质量和维护的效率。所 以，在信息系统软件运维过程中，如果修改部分涉及设计文档或用户手册的， 一定要及时更改， 这样才能达到事半功倍的效果。
      </p>
     </li>
     <li>
      <p>
       <strong>
        维护文档的可追踪性
       </strong>
       。
      </p>
      <p>
       由于信息系统软件运维的动态性，软件的某种修改最终是否有 效要经过一定的时间检验，所以运维文档也应与相应的信息系统软件一样要分版本进行管理， 这样软件和文档就具有可追踪性，便于持续地运维与改进。
      </p>
     </li>
    </ul>
