    <h4>
     测试大纲如下：
    </h4>
    <p>
     <img alt="" height="668" src="https://i-blog.csdnimg.cn/blog_migrate/22f69e5bbea9510654207fd14d897346.png" width="1170"/>
    </p>
    <p>
     1.功能性测试：
     <br/>
     ——根据产品需求文档编写测试用例。
     <br/>
     ——软件设计文档编写用例。
     <br/>
     注意：就是根据产品需求文档编写测试用例而进行测试。
    </p>
    <p>
     2.兼容性测试:
     <br/>
     ——android版本的兼容性
     <br/>
     ——手机分辨率兼容性
     <br/>
     ——网络的兼容性：2G\3G\4G\WIFI,弱网下、断网时
     <br/>
     ——app跨版本的兼容性
     <br/>
     1.适配性测试：
     <br/>
     1&gt;.手机不同分辨率支持：客户端支持的分辨率等
     <br/>
     2&gt;.手机不同版本的支持：2.34.04.4等；在测试计划中：需要安排单独的时间用于android不同系统的兼容性测试，包括2.0以下版本和4.0以上等
     <br/>
     3&gt;.手机不同厂家系统的支持：不同厂家会有不同android系统，例如：小米，华为，锤子对市面上主流手机的支持
     <br/>
     4&gt;.手机不同尺寸的支持：3.5到5.0屏幕在UI显示有区别，要支持最大到最小。
     <br/>
     2.安装、卸载测试：
     <br/>
     1&gt;.生成apk文件在真机上可以安装及卸载；
     <br/>
     2&gt;.Android手机端通用安装工具。如：豌豆荚
     <br/>
     3.在线升级测试：
     <br/>
     1&gt;.验证数字签名
     <br/>
     2&gt;.升级后可以正常使用。
     <br/>
     3&gt;.在线跨版本升级。
     <br/>
     3.性能测试：
     <br/>
     ——压力测试：
     <br/>
     ——电量流量测试：
     <br/>
     ——cup、内存消耗：
     <br/>
     ——app启动时长
     <br/>
     ——crash率
     <br/>
     ——内存泄漏
     <br/>
     4.网络测试：
     <br/>
     1.外网测试主要现实模拟客户使用网络环境，检验客户单程序在实际网若环境中使用情况及进行业务操作。
     <br/>
     2.外网测试主要覆盖到wifi\2G\3G\4G,.net\wap、电信\移动\联通、所有可能的组合进行测试。
     <br/>
     原则：
     <br/>
     1.尽可能全面覆盖用户的使用场景，测试用例中需要包含不同网络排列组合的各种可能。
     <br/>
     2.还有模拟信号被屏蔽时候。客户端的影响等。还有做外包场景测试，在高山、丘陵、火车上等特殊环境下进行全面测试
     <br/>
     5.接口性测试：
     <br/>
     ——client端和service端的交互
     <br/>
     ——client端的数据更新和service端的数据是否一致
     <br/>
     ——client端更新时断开了。
     <br/>
     ——client端更新时service端挂了。
     <br/>
     6.业务逻辑测试：
     <br/>
     1.业务逻辑测试：主要测试客户端业务能否正常完成。
     <br/>
     2.功能点测试：主要测试客户端功能点是否正常使用
     <br/>
     3.关联性测试：主要测试客户端与pc端的交互，客户端处理完后，pc端与客户端数据一致
     <br/>
     7.异常测试：
     <br/>
     1.交互异常性测试：客户端作为手机特性测试，包括被打扰的情况；如来电、来短信、低电量测试等，还要注意手机端硬件上，如：待机，插拔数据线、耳机等操作不会影响客户端。
     <br/>
     2.异常性测试：主要包含了断网、断电、服务器异常等情况下，客户端能否正常处理，保证数据正确性。
     <br/>
     客户端侧性能测试：
     <br/>
     1.基准性能测试：主要通过压服务器端接口及客户端在不同网络环境下响应速度。
     <br/>
     2.大数量的测试：主要在特定环境下，客户端一次性更新大量的数据及人员列表时，客户端能否正常处理，分为三种情况：
     <br/>
     ——客户端第一次使用，第一次就更新大量数据及人员列表。
     <br/>
     ——客户端在平时更新中，更新大量的数据
     <br/>
     ——客户端已经在手机本地下载很多数据后，再次更新大量
     <br/>
     如果想要在测试方面获得进一步的提升，那么你就需要学会使用App测试工具。一方面，通过测试工具可以代替你做重复繁琐的部分工作，你节省出的是更多的学习时间，另一方面，这些工具还会为你提供大量的游戏运行数据和日志，有了这些数据你就能更方便的判断问题发生的原因，这写数据的解读能力将是你未来的最大竞争力。
     <br/>
     1、安全测试（权限）
     <br/>
     1）软件权限：其中包括发送信息，拨打电话，链接网络，访问手机信息，联系人信息等等
     <br/>
     2）数据在本地的存储、传输等
     <br/>
     3）执行某些操作时导致的输入有效性验证、授权、数据加密等方面
     <br/>
     4）基于各种通信协议或者行业标准来检查
     <br/>
     2、安装运行卸载测试
     <br/>
     1）验证app能否正确安装运行卸载，以及操作过程和操作前后对系统资源的占有情况
     <br/>
     2）安装运行卸载的提示，报告等
     <br/>
     3）检查安装路径，文件是否合理，组件是否正确注册等
     <br/>
     3、UI测试
     <br/>
     1）用户界面（菜单、对话框、窗口）等布局，风格是否满足用户需求，文字位置，描述是否正确，界面美观程度，文字图片组合是否合理
     <br/>
     2）用户友好性、人性化、便于操作等
     <br/>
     4、功能测试
     <br/>
     1）评审需求，多方面考虑，整理出内在外在以及非功能性的直接间接功能点，对比需求，提取测试点
     <br/>
     2）根据常用的一些分析方法，等价类边界值判定表因果图场景法等方法，设计测试用例，对提取的功能点进行覆盖
     <br/>
     3）测试各个阶段不断跟踪缺陷，做好用例的更新迭代和不断变更需求所带来的业务或者需求的错误
     <br/>
     5、性能测试
     <br/>
     1）极限测试：各种边界情况下验证app的响应能力
     <br/>
     如：低电量、储存满。弱网等情况
     <br/>
     2）响应能力测试：验证各种情况下不同操作能否满足用户响应需求
     <br/>
     3）压力测试：反复长期操作下，系统该资源的使用情况
     <br/>
     6、中断测试（干扰）
     <br/>
     比如：前后台运行时来电话，短信，下载文件，听音乐看电影等不同情况下的表现
     <br/>
     7、兼容测试
     <br/>
     1）不同网络环境（WiFi、2G、3G、4G等）
     <br/>
     2）各种设备品牌机型系统版本等兼容
     <br/>
     苹果、安卓（不同品牌，不同安卓系统版本）等
     <br/>
     8、回归测试
     <br/>
     bug修复后的回归测试，上线交付前进行全部的回归，验证
     <br/>
     9、升级更新测试
     <br/>
     每次app版本迭代更新时，配合不同网络环境，及不同更新权限（强制更新，不强制更新），进行下载、安装、更新、启动运行等测试
     <br/>
     10、支付测试
     <br/>
     1）支付结果的确认，数据库查询
     <br/>
     2）请求报文是否加密
     <br/>
     3）不同场景的支付
     <br/>
     金额足够、金额不足、重复支付、无网支付、弱网支付、同账号多平台一起支付、余额宝微信信用卡等多种支付方式、不同支付方式的组合、密码正确/错误、支付上限等情况
     <br/>
     2.1安全测试
     <br/>
     2.1.1软件权限
     <br/>
     1）扣费风险：包括发送短信、拨打电话、连接网络等
     <br/>
     2）隐私泄露风险：包括访问手机信息、访问联系人信息等
     <br/>
     3）对App的输入有效性校验、认证、授权、敏感数据存储、数据加密等方面进行检测
     <br/>
     4）限制/允许使用手机功能接人互联网
     <br/>
     5）限制/允许使用手机发送接受信息功能
     <br/>
     6）限制/允许应用程序来注册自动启动应用程序
     <br/>
     7）限制或使用本地连接
     <br/>
     8）限制/允许使用手机拍照或录音
     <br/>
     9）限制/允许使用手机读取用户数据
     <br/>
     10)限制/允许使用手机写人用户数据
     <br/>
     11)检测App的用户授权级别、数据泄漏、非法授权访问等
     <br/>
     2.1.2安装与卸载安全性
     <br/>
     1）应用程序应能正确安装到设备驱动程序上
     <br/>
     2）能够在安装设备驱动程序上找到应用程序的相应图标
     <br/>
     3）是否包含数字签名信息
     <br/>
     4）JAD文件和 JAR包中包含的所有托管属性及其值必需是正确的
     <br/>
     5）JAD文件显示的资料内容与应用程序显示的资料内容应一致
     <br/>
     6）安装路径应能指定
     <br/>
     7）没有用户的允许,应用程序不能预先设定自动启动
     <br/>
     8）卸载是否安全,其安装进去的文件是否全部卸载
     <br/>
     9）卸载用户使用过程中产生的文件是否有提示
     <br/>
     10）其修改的配置信息是否复原
     <br/>
     11）卸载是否影响其他软件的功能
     <br/>
     12）卸载应该移除所有的文件
     <br/>
     2.1.3数据安全性
     <br/>
     1）当将密码或其他的敏感数据输人到应用程序时,其不会被储存在设备中,同时密码也不会
     <br/>
     被解码
     <br/>
     2）输人的密码将不以明文形式进行显示
     <br/>
     3）密码,信用卡明细,或其他的敏感数据将不被储存在它们预输人的位置上
     <br/>
     4）不同的应用程序的个人身份证或密码长度必需至少在 4一 8个数字长度之间
     <br/>
     5）当应用程序处理信用卡明细,或其他的敏感数据时,不以明文形式将数据写到其它单独的
     <br/>
     文件或者临时文件中。以 6）防止应用程序异常终止而又没有侧除它的临时文件,文件可能
     <br/>
     遭受人侵者的袭击,然后读取这些数据信息。
     <br/>
     7）当将敏感数据输人到应用程序时,其不会被储存在设备中
     <br/>
     8）备份应该加密,恢复数据应考虑恢复过程的异常�通讯中断等,数据恢复后再使用前应该
     <br/>
     经过校验
     <br/>
     9）应用程序应考虑系统或者虚拟机器产生的用户提示信息或安全替告
     <br/>
     10）应用程序不能忽略系统或者虚拟机器产生的用户提示信息或安全警告,更不能在安全警
     <br/>
     告显示前,，利用显示误导信息欺骗用户，应用程序不应该模拟进行安全警告误导用户
     <br/>
     11）在数据删除之前，应用程序应当通知用户或者应用程序提供一个“取消”命令的操作
     <br/>
     12）“取消”命令操作能够按照设计要求实现其功能
     <br/>
     13）应用程序应当能够处理当不允许应用软件连接到个人信息管理的情况
     <br/>
     14）当进行读或写用户信息操作时,应用程序将会向用户发送一个操作错误的提示信息
     <br/>
     15）在没有用户明确许可的前提下不损坏侧除个人信息管理应用程序中的任何内容Μ
     <br/>
     16）应用程序读和写数据正确。
     <br/>
     17）应用程序应当有异常保护。
     <br/>
     18）如果数据库中重要的数据正要被重写,应及时告知用户
     <br/>
     19）能合理地处理出现的错误
     <br/>
     20）意外情况下应提示用户
     <br/>
     2.1.4通讯安全性
     <br/>
     1）在运行其软件过程中,如果有来电、SMS、EMS、MMS、蓝牙、红外等通讯或充电时,是
     <br/>
     否能暂停程序，优先处理通信,并在处理完毕后能正常恢复软件,继续其原来的功能
     <br/>
     2）当创立连接时,应用程序能够处理因为网络连接中断,进而告诉用户连接中断的情况
     <br/>
     3）应能处理通讯延时或中断
     <br/>
     4）应用程序将保持工作到通讯超时,进而发送给用户一个错误信息指示有连接错误
     <br/>
     5）应能处理网络异常和及时将异常情况通报用户
     <br/>
     6）应用程序关闭或网络连接不再使用时应及时关闭)断开
     <br/>
     7) HTTP、HTTPS覆盖测试
     <br/>
     –App和后台服务一般都是通过 HTTP来交互的，验证 HTTP环境下是否正常；
     <br/>
     –公共免费网络环境中（如：麦当劳、星巴克等）都要输入用户名和密码，通过 SSL认证
     <br/>
     来访问网络，需要对使用 HTTP Client的 library异常作捕获处理。
     <br/>
     2.1.5人机接口安全性
     <br/>
     1）返回菜单总保持可用
     <br/>
     2）命令有优先权顺序
     <br/>
     3）声音的设置不影响应用程序的功能
     <br/>
     4）应用程序必需利用目标设备适用的全屏尺寸来显示上述内容
     <br/>
     5）应用程序必需能够处理不可预知的用户操作,例如错误的操作和同时按下多个键
     <br/>
     2.2安装、卸载测试
     <br/>
     验证 App是否能正确安装、运行、卸载
     <br/>
     2.2.1安装
     <br/>
     1）软件在不同操作系统（Palm OS、Symbian、Linux、Android、iOS、Black Berry OS 6.0、
     <br/>
     Windows Phone 7）下安装是否正常。
     <br/>
     2）软件安装后的是否能够正常运行，安装后的文件夹及文件是否写到了指定的目录里。
     <br/>
     3）软件安装各个选项的组合是否符合概要设计说明
     <br/>
     4)）软件安装向导的 UI测试
     <br/>
     5）软件安装过程是否可以取消，点击取消后，写入的文件是否如概要设计说明处理
     <br/>
     6）软件安装过程中意外情况的处理是否符合需求（如死机，重启，断电）
     <br/>
     7）安装空间不足时是否有相应提示
     <br/>
     8）安装后没有生成多余的目录结构和文件
     <br/>
     9）对于需要通过网络验证之类的安装，在断网情况下尝试一下
     <br/>
     10）还需要对安装手册进行测试，依照安装手册是否能顺利安装
     <br/>
     2.2.2卸载
     <br/>
     1）直接删除安装文件夹卸载是否有提示信息。
     <br/>
     2）测试系统直接卸载程序是否有提示信息。
     <br/>
     3）测试卸载后文件是否全部删除所有的安装文件夹。
     <br/>
     4）卸载过程中出现的意外情况的测试（如死机、断电、重启）。
     <br/>
     5）卸载是否支持取消功能，单击取消后软件卸载的情况。
     <br/>
     6）系统直接卸载 UI测试，是否有卸载状态进度条提示。
     <br/>
     2.3 UI测试
     <br/>
     测试用户界面(如菜单、对话框、窗口和其它可规控件)布局、风格是否满足客户要求、文字
     <br/>
     是否正确、页面是否美观、文字、图片组合是否完美、操作是否友好等。
     <br/>
     UI测试的目标是确保用户界面会通过测试对象的功能来为用户提供相应的访问或浏觅功能。
     <br/>
     确保用户界面符合公司或行业的标准。包括用户友好性、人性化、易操作性测试。
     <br/>
     2.3.1导航测试
     <br/>
     1）按钮、对话框、列表和窗口等；或在不同的连接页面之间需要导航
     <br/>
     2）是否易于导航，导航是否直观
     <br/>
     3）是否需要搜索引擎
     <br/>
     4）导航帮助是否准确直观
     <br/>
     5）导航与页面结构、菜单、连接页面的风格是否一致
     <br/>
     2.3.2图形测试
     <br/>
     1）横向比较。各控件操作方式统一
     <br/>
     2）自适应界面设计，内容根据窗口大小自适应
     <br/>
     3）页面标签风格是否统一
     <br/>
     4）页面是否美观
     <br/>
     5）页面的图片应有其实际意义而要求整体有序美观
     <br/>
     6）图片质量要高且图片尺寸在设计符合要求的情况下应尽量小
     <br/>
     7）界面整体使用的颜色不宜过多
     <br/>

      2.5性能测试
      <br/>
      评估App的时间和空间特性：
      <br/>
      1)极限测试：在各种边界压力情况下，如电池、存储、网速等，验证App是否能正确响应。
      <br/>
      –内存满时安装 App
      <br/>
      –运行 App时手机断电
      <br/>
      –运行 App时断掉网络
      <br/>
      2)响应能力测试：测试App中的各类操作是否满足用户响应时间要求。
      <br/>
      –App安装、卸载的响应时间
      <br/>
      –App各类功能性操作的影响时间
      <br/>
      3)压力测试：反复/长期操作下、系统资源是否占用异常。
      <br/>
      –App反复进行安装卸载，查看系统资源是否正常
      <br/>
      –其他功能反复进行操作，查看系统资源是否正常
      <br/>
      4)性能评估：评估典型用户应用场景下，系统资源的使用情况。
      <br/>
      5)Benchmark测试（基线测试）：与竞争产品的Benchmarking,产品演变对比测试等。
      <br/>

      <br/>
      冲突测试，是指一个功能正在执行过程中，同时另外一个事件或操作对该过程进行干扰的测
      <br/>
      试。如；App在前/后台运行状态时与来电、文件下载、音乐收听等关键运用的交互情况测
      <br/>
      试等。交叉事件测试非常重要，能发现很多应用中潜在的性能问题。
      <br/>
      1）多个 App同时运行是否影响正常功能
      <br/>
      2）App运行时前/后台切换是否影响正常功能
      <br/>
      3）App运行时拨打/接听电话
      <br/>
      4）App运行时发送/接收信息
      <br/>
      5）App运行时发送/收取邮件
      <br/>
      6）App运行时切换网络（2G、3G、wifi）
      <br/>
      7）App运行时浏览网络
      <br/>
      8）App运行时使用蓝牙传送/接收数据
      <br/>
      9）App运行时使用相机、计算器等手机自带设备

      <br/>

      <br/>

      1）是否有空数据界面设计，引导用户去执行操作。

      2）是否滥用用户引导。

      <br/>
      掉按钮，否则会给用户误导
      <br/>
      4）菜单层次是否太深
      <br/>
      5）交互流程分支是否太多
      <br/>
      6）相关的选项是否离得很远
      <br/>
      7）一次是否载入太多的数据
      <br/>
      8）界面中按钮可点击范围是否适中
      <br/>
      9）标签页是否跟内容没有从属关系，当切换标签的时候，内容跟着切换
      <br/>
      10）操作应该有主次从属关系

      11）是否定义 Back的逻辑。涉及软硬件交互时，Back键应具体定义

      12）是否有横屏模式的设计，应用一般需要支持横屏模式，即自适应设计

      2.11.2网络环境
      <br/>
      手机的网络目前主要分为2G、3G、wifi。目前2G的网络相对于比较慢，测试时尤其要注意此
      <br/>
      块的测试。
      <br/>
      1)无网络时，执行需要网络的操作，给予友好提示，确保程序不出现crash。
      <br/>
      2)内网测试时，要注意选择到外网操作时的异常情况处理。
      <br/>
      3)在网络信号不好时，检查功能状态是否正常，确保不因提交数据失败而造成crash。
      <br/>
      4)在网络信号不好时，检查数据是否会一直处于提交中的状态，有无超时限制。如遇数据
      <br/>
      交换失败时要给予提示。
      <br/>
      5)在网络信号不好时，执行操作后，在回调没有完成的情况下，退出本页面或者执行其他
      <br/>
      操作的情况，有无异常情况。此问题也会经常出现程序crash。

      2.11.3服务器宕机或出现404、502等情况下的测试
      <br/>
      后台服务牵涉到 DNS、空间服务商的情况下会影响其稳定性，如：当出现域名解析故障时，
      <br/>
      你对后台 API的请求很可能就会出现 404错误，抛出异常。这时需要对异常进行正确的处
      <br/>
      理，否则可能会导致程序不能正常工作。

      2.12接口测试
      <br/>
      服务端一般会提供JSON格式的数据给客户端，所以我们在服务端需要进行接口测试，确保
      <br/>
      服务端提供的接口并转换的JSON内容正确，对分支、异常流有相应的返回值。此块测试可
      <br/>
      以采用itest框架进行测试。最方便的是采用httpclient进行接口测试。
      <br/>
      进行服务端测试时，需要开发提供一份接口文档。
      <br/>
      2.13客户端数据库测试
      <br/>
      1）一般的增、删、改、查测试。
      <br/>
      2）当表不存在时是否能自动创建，当数据库表被删除后能否再自建，数据是否还能自动从
      <br/>
