     1. 什么是测试用例
    </h2>

     </strong>

      设计测试⽤例原则⼀：

      <li>
       测试⽤例中⼀个必需部分是对预期输出或结果进⾏定义
      </li>
     </ol>
    </blockquote>

    <h2>
     <a id="2__37">
     </a>
     2. 设计测试用例的万能公式
    </h2>
    <p>
     设计测试用例的万能公式：
     <strong>
      功能测试+界面测试+性能测试+兼容性测试+易用性测试+安全测试
     </strong>
    </p>
    <ol>
     <li>
      功能测试：从产品功能角度出发，验证功能是否正确；
     </li>
     <li>
      界面测试：肉眼可以看到的称之为界面，界面所有的元素（大小，颜色，形状，材质）都需要测试；
     </li>
     <li>
      性能测试：通常为一些
      <strong>
       极端的情况
      </strong>
      会产生一些性能的区别；
     </li>
     <li>
      兼容性测试：不同的版本（软件、系统…）、浏览器的版本、不同的浏览器；
     </li>
     <li>
      易用性测试：具有简单易上手的属性（引导教程）
     </li>
     <li>
      安全测试：是否具备危险材质、气味；
      <br/>
      接口响应数据也要考虑到用户数据的安全性（加密）；
      <br/>
      登录场景也需要将秘密进行加密展示；
      <br/>
      数据存储用户隐私数据是否加密。
      <br/>
      比如：SQL注入、越权
     </li>
    </ol>

     3. 特殊的测试场景

    <h3>
     <a id="31__58">
     </a>
     3.1 弱网测试
    </h3>
    <blockquote>
     <p>
      <strong>
       弱网测试的目的就是尽可能保证用户体验（覆盖更多的网络场景），关注的关键点包括：
      </strong>
     </p>
     <ol>
      <li>
       页面响应时间是否可以接受，关注包括热启动、冷启动时间、页面切换、前后台切换、首字时间，首屏时间等。
      </li>
      <li>
       页面呈现是否完成一致。
      </li>
      <li>
       超时文案是否符合定义，异常信息是否显示正常。
      </li>
      <li>
       是否有超时重连。
      </li>
      <li>
       安全角度：是否会发生 dns 劫持、登陆 ip 更换频繁、单点登陆异常等。
      </li>
      <li>
       大流量事件风险：是否会在弱网下进行更新 apk 包、下载文件等大流量动作。
      </li>
     </ol>
    </blockquote>

     4. 设计测试用例的方法
    </h2>
    <h3>
     <a id="41__89">
     </a>
     4.1 基于需求的设计方法写出初步的测试用例
    </h3>
    <p>
     <strong>
      <code>
       测试和开发工作开展的依据：需求文档 / 产品规格说明书（用户需求通过需求分析形成软件需求）
      </code>
     </strong>
    </p>
    <p>
     测试人员接到需求之后，要对需求进行分析和验证，从合理的需求中进一步分析细化需求，从细化的需求中找出测试点，根据这些测试点再去设计测试用例。

    <h3>
     <a id="42__102">
     </a>
     4.2 具体的设计方法
    </h3>
    <h4>
     <a id="421__103">
     </a>
     4.2.1 等价类
    </h4>
    <p>
     依据需求将输入（特殊情况下会考虑输出）划分为若干个等价类从等价类中选出一个测试用例，如果这个测试用例测试通过，则认为所代表的等价类测试通过，这样就可以用较少的测试用例达到尽量多的功能覆盖，解决了不能穷举测试的问题。
    </p>
    <ul>
     <li>
      <p>
       <strong>
        姓名：必填，6 - 15位字符
       </strong>
       <br/>
       若测试的姓名位上述字符，那么我们应该测试多少位的字符？6789…15?
       <br/>
       那么小于 6 和 大于 15 的需要吗，那么使用穷举法就太耗时耗力了。
      </p>
     </li>
     <li>
      <p>
       <strong>
        那我们应该怎么做 ？
       </strong>
       <br/>
       我们可以考虑等价类：等价类就是一个可以了，那么这个等价的范围里也就都可以了
      </p>
     </li>
    </ul>
    <blockquote>
     <p>
      <strong>
       等价类分类：
      </strong>
     </p>
     <ol>
      <li>
       有效等价类：对于程序的需求文档来说是有意义的（6 ~ 15 位字符）
      </li>
      <li>
       无效等价类：对于程序的需求文档来说是无意义的（&lt; 6 或 &gt; 15 位字符）
      </li>
     </ol>
    </blockquote>

    <h4>
     <a id="422__122">
     </a>
     4.2.2 边界值
    </h4>
    <p>
     边界值分析法就是对输入或输出的边界值进行测试的一种黑盒测试方法。通常边界值分析法是作为对等价类划分法的补充，这种情况下，其测试用例来自等价类的边界。
    </p>
    <p>
     边界值包括：边界值 + 次边界值
    </p>
    <blockquote>
     <ul>
      <li>
       边界值：给定返回的左数据和右数据
      </li>
      <li>
       次边界值：需要根据边界值的有效无效情况来定
       <ol>
        <li>
         若边界值为有效等价类中的数据，则次边界值为无效等价类中的边界
        </li>
        <li>
         若边界值为无效等价类中的数据，则次边界值为有效等价类中的边界
        </li>
       </ol>
      </li>
     </ul>
    </blockquote>

    <h4>
     <a id="424__139">

     4.2.4 等价表法
    </h4>
    <p>
     正交法的目的是为了减少用例数目。用尽量少的用例覆盖输入的两两组合。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/2d572272187f4f26a9af14a2155d2c21.png"/>
     <br/>
     正交试验设计是研究多因素多水平的一种设计方法，它是根据正交性，由试验因素的全部水平组合中挑选出部分有代表性的点进行试验，通过对这部分试验结果的分析了解全面试验的情况，找出最优的水平组合。正交试验设计是一种基于正交表的、高效率、快速、经济的试验。
    </p>

    <h4>
     <a id="426__153">
     </a>
     4.2.6 错误猜测法
    </h4>
    <p>
     错误猜测法是对被测试软件设计的理解，过往经验以及个人直觉，推测出软件可能存在的缺陷，从而针对性地设计测试用例的方法。
    </p>
    <p>
     这个方法强调的是对被测试软件的需求理解以及设计实现的细节把握，还有个人的经验和直觉。
    </p>
    <p>
     错误推测法和目前流行的“探索式测试方法”的基本思想一致，这类方法在敏捷开发模式下的投入产出比很高，被广泛应用于测试。
    </p>
