    <h2 id="14.%E4%BB%80%E4%B9%88%E6%98%AF%E8%80%A6%E5%90%88%EF%BC%9F">

    <h2 id="16.%E5%BC%80%E5%8F%91%E8%BD%AF%E4%BB%B6%E5%AF%B9%E8%80%A6%E5%90%88%E4%B8%8E%E5%86%85%E8%81%9A%E8%BF%BD%E6%B1%82%E7%9A%84%E5%87%86%E5%88%99%E6%98%AF%E4%BB%80%E4%B9%88%EF%BC%9F">

    </h2>
    <blockquote>
     <p>
      开发软件追求“
      <strong>
       <span style="color:#fe2c24;">
        低耦合、高内聚
       </span>

      ”的准则，实践表明内聚更重要
     </p>
    </blockquote>
    <h2 id="17.%E8%80%A6%E5%90%88%E7%A8%8B%E5%BA%A6%E7%94%B1%E4%BD%8E%E5%88%B0%E9%AB%98%E6%8E%92%E5%BA%8F%EF%BC%9F%E5%B9%B6%E5%88%86%E5%88%AB%E4%B8%BE%E4%BE%8B">
     17.耦合程度由低到高排序？并分别举例
    </h2>
    <blockquote>
     <ol>
      <li>
       <span style="color:#fe2c24;">
        数据耦合
       </span>
       ：两个模块间通过参数交换信息，而且交换的信息仅仅是数据。比如：两个模块，A模块的运算结果作为参数传入B模块中参与计算
      </li>
      <li>
       <span style="color:#fe2c24;">
        控制耦合
       </span>
       ：传递信息中有控制信息，控制耦合把模块适当分解可以用数据耦合来代替。比如：空调遥控器模块和空调外机模块之间的耦合
      </li>
      <li>
       <span style="color:#fe2c24;">
        特征耦合
       </span>
       ：把整个数据结构作为参数进行传递，而被调用模块只用其中一部分元素。比如：一个系统需要身份证信息，另一个只需要姓名信息，但把整个用户信息传入就会出现特征耦合
      </li>
      <li>
       <span style="color:#fe2c24;">
        公共环境耦合
       </span>
       ：两个或多个模块通过一个公共数据环境相互作用。比如：多个模块共同使用一个全局数组，不同模块可以读、写该数组
      </li>
      <li>
       <span style="color:#fe2c24;">
        内容耦合
       </span>
       ：指如果一个模块与另一个模块的内部属性有关，不经过调用而直接使用另一个模块的程序代码或者内部数据就叫做内容耦合。要坚决避免，比如GOTO语句等。
      </li>
     </ol>
     <p>
      【注1】无连接耦合的耦合程度最低，但是几乎不可能实现
     </p>
     <p>
      【注2】设计原则：尽量使用数据耦合，少用控制耦合和特征耦合，限制公告环境耦合的范围，完全不用内容耦合
     </p>
    </blockquote>
    <h2 id="18.%E5%86%85%E8%81%9A%E7%A8%8B%E5%BA%A6%E7%94%B1%E9%AB%98%E5%88%B0%E4%BD%8E%E6%8E%92%E5%BA%8F%EF%BC%9F%E5%B9%B6%E5%88%86%E5%88%AB%E4%B8%BE%E4%BE%8B%EF%BC%9F">
     18.内聚程度由高到低排序？并分别举例？
    </h2>
    <blockquote>
     <ol>
      <li>
       <span style="color:#fe2c24;">
        功能内聚
       </span>
       ：模块内所有处理元素属于一个整体，完成单一的功能。比如：一个模块中所有的操作都是为了计算出一个人的年龄
      </li>
      <li>
       <span style="color:#fe2c24;">
        顺序内聚
       </span>
       ：模块内处理元素和同一个功能密切相关，且处理必须顺序执行。比如：计算成绩平均分，先要输入全部的成绩，计算总分之后再做除法计算平均分。计算总分的输出作为计算平均分的输入，有必须的数据传递，限制了先后关系。
      </li>
      <li>
       <span style="color:#fe2c24;">
        通信内聚
       </span>
       ：模块中所有元素都使用同一个输入数据和（或）产生同一个输出数据。比如：有一个子程序负责将输入的信息处理生产报告，同时利用输入数据更新已有的数据，两个操作使用同一个数据源。
      </li>
      <li>
       <span style="color:#fe2c24;">
        过程内聚
       </span>
       ：模块内的处理元素相关，且必须以特定次序执行。比如：计算各班同学的平均成绩，要求以先后顺序输出123班的成绩。
      </li>
      <li>
       <span style="color:#fe2c24;">
        时间内聚
       </span>
       ：一个模块内包含的任务必须在同一段时间内执行。比如：将多个变量的初始化放在同一个模块中实现。
      </li>
      <li>
       <span style="color:#fe2c24;">
        逻辑内聚
       </span>
       ：一个模块完成的任务在逻辑上属于相同或相似的一类。比如：一个模块产生各种类型的输出，打印数字，打印姓名，打印数组等等，他们的功能都是打印输出信息，功能相同。
      </li>
      <li>
       <span style="color:#fe2c24;">
        偶然内聚
       </span>
       ：一个模块完成一组任务，完成任务彼此之间即使有关系，关系也很松散。比如：A模块中有n条语句，但是这些语句彼此之间没有关系。
      </li>
     </ol>
    </blockquote>
    <h2 id="19.%E5%90%AF%E5%8F%91%E5%BC%8F%E8%A7%84%E5%88%99%E6%9C%89%E5%93%AA%E4%BA%9B%EF%BC%9F">
     19.启发式规则有哪些？
    </h2>
    <blockquote>
     <ol>
      <li>
       改进软件结构，
       <span style="color:#fe2c24;">
        提高模块独立性
       </span>
      </li>
      <li>
       模块
       <span style="color:#fe2c24;">
        规模
       </span>
       应该
       <span style="color:#fe2c24;">
        适中
       </span>
      </li>
      <li>
       深度、宽度、扇入、扇出都应该适当
      </li>
      <li>
       模块作用域应该在控制域之内
      </li>
      <li>
       力争
       <span style="color:#fe2c24;">
        降低
       </span>
       模块
       <span style="color:#fe2c24;">
        接口的复杂程度
       </span>
      </li>
      <li>
       设计
       <span style="color:#fe2c24;">
        单入口
       </span>
       和
       <span style="color:#fe2c24;">
        单出口
       </span>
       的模块
      </li>
      <li>
       模块功能应该可以
       <span style="color:#fe2c24;">
        预测
       </span>
      </li>
     </ol>
    </blockquote>
    <h2 id="20.%E6%A8%A1%E5%9D%97%E7%9A%84%E4%BD%9C%E7%94%A8%E5%9F%9F%E5%92%8C%E6%8E%A7%E5%88%B6%E5%9F%9F%E5%88%86%E5%88%AB%E6%98%AF%E4%BB%80%E4%B9%88%EF%BC%9F">
     20.模块的作用域和控制域分别是什么？
    </h2>
    <blockquote>
     <p>
      <span style="color:#fe2c24;">
       作用域
      </span>
      ：受该模块内一个判定影响的所有模块的集合
     </p>
     <p>
      <span style="color:#fe2c24;">
       控制域
      </span>
      ：这个模块本身以及所有直接或从属于它的模块的集合
     </p>
     <p style="text-align:center;">
      <img alt="" height="361" src="https://i-blog.csdnimg.cn/blog_migrate/dda8aac5f5479ad34e2668eda8cf65c2.jpeg" width="509"/>
     </p>
     <p>
      【注】以上图为例，A模块的控制域就是A、B、C、D、E、F
     </p>
    </blockquote>
    <h2 id="21.%E6%B7%B1%E5%BA%A6%E3%80%81%E5%AE%BD%E5%BA%A6%E3%80%81%E6%89%87%E5%85%A5%E3%80%81%E6%89%87%E5%87%BA%E5%88%86%E5%88%AB%E6%98%AF%E4%BB%80%E4%B9%88%EF%BC%9F">
     21.深度、宽度、扇入、扇出分别是什么？
    </h2>
    <blockquote>
     <ol>
      <li>
       深度：表示软件结构中
       <span style="color:#fe2c24;">
        控制的层数
       </span>
       ，它往往能粗略地表明一个系统的大小和复杂程度
      </li>
      <li>
       宽度：表示软件结构内
       <span style="color:#fe2c24;">
        同一个层次上的模块总数的最大值
       </span>
       。一般来说，宽度越大系统越复杂。对宽度影响最大的因素是模块的扇出
      </li>
      <li>
       扇出：表示一个模块
       <span style="color:#fe2c24;">
        直接控制（调用）的模块数目
       </span>
       。好的系统平均扇出是3~4个
      </li>
      <li>
       扇入：表示
       <span style="color:#fe2c24;">
        有多少个上级模块调用它
       </span>
       ，扇入多表面该模块被调用的次数多，比较好
      </li>
     </ol>
    </blockquote>
    <h2 id="22.%E5%B1%82%E6%AC%A1%E5%9B%BE%E4%B8%8E%E5%B1%82%E6%AC%A1%E6%96%B9%E6%A1%86%E5%9B%BE%E4%B8%AD%E8%BF%9E%E7%BA%BF%E6%89%80%E4%BB%A3%E8%A1%A8%E7%9A%84%E5%90%AB%E4%B9%89%E6%9C%89%E4%BB%80%E4%B9%88%E5%8C%BA%E5%88%AB%EF%BC%9F">
     22.层次图与层次方框图中连线所代表的含义有什么区别？
    </h2>
    <blockquote>
     <p>
      层次图的连线表示
      <span style="color:#fe2c24;">
       调用关系
      </span>
      ，而层次方框图中连线表示
      <span style="color:#fe2c24;">
       组成关系
      </span>
     </p>
     <p style="text-align:center;">
      <span style="color:#fe2c24;">
       <img alt="" height="295" src="https://i-blog.csdnimg.cn/blog_migrate/329d97ce6520f178b950a79b1eccbd5d.jpeg" width="554"/>
      </span>
     </p>
     <p style="text-align:center;">
      <img alt="" height="377" src="https://i-blog.csdnimg.cn/blog_migrate/98f5c7dc3fa3feb3a128b8cb389d5725.png" width="555"/>
     </p>
     <p>
      【注】第2张图是层次方框图，它是以
      <span style="color:#fe2c24;">
       实体
      </span>
      的角度出发进行分析，而层次图是以
      <span style="color:#fe2c24;">
       功能
      </span>
      的角度
     </p>
    </blockquote>
    <h2 id="23.%E5%B1%82%E6%AC%A1%E5%9B%BE%E9%80%82%E7%94%A8%E4%BA%8E%E5%93%AA%E7%A7%8D%E8%AE%BE%E8%AE%A1%E8%BD%AF%E4%BB%B6%E7%9A%84%E8%BF%87%E7%A8%8B%E4%B8%AD%E4%BD%BF%E7%94%A8%EF%BC%9F">
     23.层次图适用于哪种设计软件的过程中使用？
    </h2>
    <blockquote>
     <p>
      自顶向下
     </p>
    </blockquote>
    <h2 id="24.%E7%BB%93%E6%9E%84%E5%9B%BE%E4%B8%AD%E7%AC%A6%E5%8F%B7%E5%90%AB%E4%B9%89%EF%BC%9F">
     24.结构图中符号含义？
    </h2>
    <blockquote>
     <p>
      结构图是进行软件设计的另一个有力工具
     </p>
     <p>
      一个方框代表一个模块，之间的箭头表示模块
      <span style="color:#fe2c24;">
       调用关系
      </span>
      ，带注释的箭头代表模块调用过程中来回传递的消息
     </p>
     <p style="text-align:center;">
      <img alt="" height="298" src="https://i-blog.csdnimg.cn/blog_migrate/2dd33ef15d1bd80f68fdd069282ca330.jpeg" width="532"/>
     </p>
     <p>
     </p>
    </blockquote>
    <h2 id="25.%E5%B1%82%E6%AC%A1%E5%9B%BE%E5%92%8C%E7%BB%93%E6%9E%84%E5%9B%BE%E8%83%BD%E8%A1%A8%E7%A4%BA%E6%A8%A1%E5%9D%97%E9%97%B4%E7%9A%84%E8%B0%83%E7%94%A8%E6%AC%A1%E5%BA%8F%E5%90%97%EF%BC%9F">
     25.层次图和结构图能表示模块间的调用次序吗？
    </h2>
    <blockquote>
     <p>
      层次图和结构图
      <span style="color:#fe2c24;">
       不能
      </span>
      严格表示模块间的调用次序，他们并不指明什么时候调用下层模块，而
      <span style="color:#fe2c24;">
       只是表明一个模块调用哪些模块
      </span>
     </p>
    </blockquote>
    <h2 id="26.%E9%9D%A2%E5%90%91%E6%95%B0%E6%8D%AE%E6%B5%81%E7%9A%84%E8%AE%BE%E8%AE%A1%E6%96%B9%E6%B3%95%E7%9B%AE%E6%A0%87%E6%98%AF%E4%BB%80%E4%B9%88%EF%BC%9F%E4%BF%A1%E6%81%AF%E6%B5%81%E5%8C%85%E6%8B%AC%E4%BB%80%E4%B9%88%EF%BC%9F">
     26.面向数据流的设计方法目标是什么？信息流包括什么？
    </h2>
    <blockquote>
     <p>
      面向数据流的设计方法目标是
      <span style="color:#fe2c24;">
       给出设计软件结构的一个系统化的途径
      </span>
     </p>
     <p>
      信息流包括
      <span style="color:#fe2c24;">
       变换流与事物流
      </span>
     </p>
     <p style="text-align:center;">
      <img alt="" height="280" src="https://i-blog.csdnimg.cn/blog_migrate/69021988657fc7e9a43021c962dce3d2.jpeg" width="609"/>
     </p>
     <p>
     </p>
    </blockquote>
    <h2 id="27.%E5%88%B6%E5%AE%9A%E6%B5%8B%E8%AF%95%E8%AE%A1%E5%88%92%E5%92%8C%E8%AE%BE%E8%AE%A1%E6%B5%8B%E8%AF%95%E7%94%A8%E4%BE%8B%E4%B8%80%E6%A0%B7%E5%90%97%EF%BC%9F%E4%B8%BA%E4%BB%80%E4%B9%88">
     27.制定测试计划和设计测试用例一样吗？为什么
    </h2>
    <blockquote>
     <p>
      二者
      <span style="color:#fe2c24;">
       不一样
      </span>
      ，因为在总体设计阶段制定测试计划，但此时还
      <span style="color:#fe2c24;">
       没有具体的代码实现
      </span>
      ，所以无法具体的写出每条测试用例。
     </p>
    </blockquote>
    <h2 id="28.%E6%80%BB%E4%BD%93%E8%AE%BE%E8%AE%A1%E7%9A%84%E5%BF%85%E8%A6%81%E6%80%A7%EF%BC%9F">
     28.总体设计的必要性？
    </h2>
    <blockquote>
     <p>
      可以站在
      <span style="color:#fe2c24;">
       全局高度
      </span>
      上，花较少的成本，从比较
      <span style="color:#fe2c24;">
       抽象
      </span>
      的层次上分析对比
      <span style="color:#fe2c24;">
       多种
      </span>
      可能的系统实现方案和软件结构，从中
      <span style="color:#fe2c24;">
       选出最佳方案和最合理的软件结构
      </span>
      ，从而用较低成本开发出较高质量的软件系统。
     </p>
    </blockquote>
    <h2 id="%E7%AB%A0%E6%9C%AB%E5%B0%8F%E7%BB%93">
     章末小结
    </h2>
    <blockquote>
     <p style="margin-left:.0001pt;text-align:left;">
      总体设计阶段的基本目的是
      <span style="color:#fe2c24;">
       用比较抽象概括的方式确定系统如何完成预定的任务
      </span>
      ，也就是说：应该确定系统的物理配置方案，并且进而确定组成系统的每个程序的结构。因此，总体设计阶段主要由两个小阶段组成。首先需要进行
      <span style="color:#fe2c24;">
       系统设计
      </span>
      ，从数据流图出发设想完成系统功能的若干种合理的物理方案，分析员应该仔细分析比较这些方案,并且和用户共同选定一个最佳方案。然后进行软件
      <span style="color:#fe2c24;">
       结构设计
      </span>
      ，确定软件由哪些模块组成以及这些模块之间的动态调用关系。
      <span style="color:#fe2c24;">
       层次图和结构图
      </span>
      是描绘软件结构的常用工具。
      <br/>
      在进行软件结构设计时应该遵循的
      <span style="color:#fe2c24;">
       最主要的原理是模块独立原理
      </span>
      ，也就是说，软件应该由一组完成
      <span style="color:#fe2c24;">
       相对独立的子功能的模块
      </span>
      组成这此模块彼此之间的接口关系应该尽量简单。
      <br/>
      抽象和求精是一对
      <span style="color:#fe2c24;">
       互补
      </span>
      的概念,也是人类解决复杂问题时最常用、最有效的方法。在进行软件结构设计时一种有效的方法就是，由抽象到县体地构造出软件的层次结构。
     </p>
     <p style="margin-left:.0001pt;text-align:left;">
      软件工程师在开发软件的长期实践中积累了丰富的经验，总结这些经验得出一些很有参考价值的启发式规则，它们往往能对如何改进软件设计给出宝贵的提示。在软件开发过程中既要充分重视和利用这些启发式规则，又要从实际情况出发
      <span style="color:#fe2c24;">
       避免生搬硬套
      </span>
      。
     </p>
     <p style="margin-left:.0001pt;text-align:left;">
      <span style="color:#fe2c24;">
       自顶向下逐步求精
      </span>
      是进行软件结构设计的常用途径；但是，如果已经有了详细的数据流图，也可以使用面向数据流的设计方法，用形式化的方法由数据流图
      <span style="color:#fe2c24;">
       映射出软件结构
      </span>
      。应该记住这样映射出来的只是软件的
      <span style="color:#fe2c24;">
       初步结构
      </span>
      ，还必须根据设计原理并且参考启发式规则，认真分析和改进软件的初步结构以得到质量更高的模块和更合理的软件结构。
      <br/>
      在进行详细的过程设计和编写程序之前，首先进行结构设计,其好处正在于可以在软件开发的早期站在
      <span style="color:#fe2c24;">
       全局高度
      </span>
      对软件结构进行优化。在这个时期进行优化付出的
      <span style="color:#fe2c24;">
       代价不高
      </span>
      ，却可以使软件质量得到
      <span style="color:#fe2c24;">
       重大改进
      </span>
      。
     </p>
    </blockquote>
