    <h2>
     <a id="_1">
     </a>
     一、分布式版本控制器和集中式
    </h2>
    <p>
     Git、SVN 和 CVS 都是常用的版本控制系统，以下是它们的相关介绍及对比：
    </p>
    <h3>
     <a id="1_5">
     </a>
     1.基本概念
    </h3>
    <ul>
     <li>
      <strong>
       Git
      </strong>
      ：是一种
      <strong>
       分布式版本控制系统
      </strong>
      ，它允许每个开发者在自己的本地机器上拥有完整的版本库，包括所有的提交历史和文件版本。开发者可以在本地进行提交、分支管理等操作，然后再将本地的更改推送到远程服务器或与其他开发者共享。
     </li>
     <li>
      <strong>
       SVN
      </strong>
      ：即
      <code>
       Subversion
      </code>
      ，是一种
      <strong>
       集中式版本控制系统
      </strong>
      。它有一个中央服务器，所有的版本信息都存储在服务器上，开发者通过网络连接到服务器进行提交、更新等操作，本地只保留最新版本的文件和一些临时工作目录。
     </li>
     <li>
      <strong>
       CVS
      </strong>
      ：即
      <code>
       Concurrent Versions System
      </code>
      ，是最早的广泛使用的版本控制系统之一，也是
      <strong>
       集中式
      </strong>
      的。它以类似的方式工作，有一个中心服务器来存储版本数据，开发人员从服务器获取和提交更改。
     </li>
    </ul>
    <h3>
     <a id="2_11">
     </a>
     2.主要特点
    </h3>
    <ul>
     <li>
      Git
      <ul>
       <li>
        <strong>
         分布式
        </strong>
        ：每个**
        <mark>
         开发者的本地环境都是一个完整的版本库
        </mark>
        **，这使得开发者可以在没有网络的情况下进行开发、提交等操作，之后再与远程仓库同步。
       </li>
       <li>
        <strong>
         高效的分支管理
        </strong>
        ：创建、切换和合并分支非常快速和高效，使得开发人员可以轻松地进行并行开发，例如同时进行新功能开发、bug 修复等不同任务，互不干扰。
       </li>
       <li>
        <strong>
         数据安全
        </strong>
        ：采用哈希算法对数据进行校验和存储，确保数据的完整性和一致性。如果数据在传输或存储过程中发生损坏，Git 能够检测到并提示错误。
       </li>
      </ul>
     </li>
     <li>
      SVN
      <ul>
       <li>
        <strong>
         集中式管理
        </strong>
        ：所有版本数据集中存储在服务器上，便于统一管理和维护，管理员可以方便地控制用户的访问权限，确保代码的安全性和规范性。
       </li>
       <li>
        <strong>
         简单易用
        </strong>
        ：对于小型团队或简单项目，SVN 的操作相对简单，容易上手。开发人员只需要关注与服务器之间的交互，进行提交和更新操作即可。
       </li>
       <li>
        <strong>
         支持原子提交
        </strong>
        ：可以将多个文件的修改作为一个原子操作提交到服务器，要么全部成功，要么全部失败，保证了版本库的一致性。
       </li>
      </ul>
     </li>
     <li>
      CVS
      <ul>
       <li>
        <strong>
         历史悠久
        </strong>
        ：具有很长的发展历史，有大量的开源项目使用 CVS 进行版本控制，积累了丰富的使用经验和相关工具。
       </li>
       <li>
        <strong>
         基本功能完备
        </strong>
        ：提供了版本控制的基本功能，如提交、更新、查看历史记录等，能够满足一般项目的版本管理需求。
       </li>
       <li>
        <strong>
         广泛的系统支持
        </strong>
        ：可以在多种操作系统上运行，包括 UNIX、Windows 等，具有较好的跨平台性。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="3_26">
     </a>
     3.使用场景
    </h3>
    <ul>
     <li>
      <strong>
       Git
      </strong>
      ：适用于大型分布式开发项目，尤其是**
      <mark>
       开源项目和需要频繁进行分支和合并的项目
      </mark>
      **。像 Linux 内核开发、GitHub 上的众多项目等都使用 Git 进行版本控制。
     </li>
     <li>
      <strong>
       SVN
      </strong>
      ：适合团队规模较小、对权限管理要求较高、工作流程相对简单的项目。例如一些企业内部的小型项目，开发人员需要严格按照规定的权限和流程进行代码管理。
     </li>
     <li>
      <strong>
       CVS
      </strong>
      ：对于一些历史遗留项目或者对版本控制功能要求不高、追求简单实用的项目，CVS 仍然可以发挥作用。一些传统的科研项目、小型软件公司的老项目可能还在使用 CVS。
     </li>
    </ul>
    <h3>
     <a id="4_32">
     </a>
     4.操作命令示例
    </h3>
    <ul>
     <li>
      Git
      <ul>
       <li>
        <strong>
         克隆仓库
        </strong>
        ：
        <code>
         git clone &lt;repository_url&gt;
        </code>
       </li>
       <li>
        <strong>
         提交更改
        </strong>
        ：
        <code>
         git add &lt;file&gt;
        </code>
        然后
        <code>
         git commit -m "Commit message"
        </code>
       </li>
       <li>
        <strong>
         推送更改
        </strong>
        ：
        <code>
         git push origin &lt;branch_name&gt;
        </code>
       </li>
      </ul>
     </li>
     <li>
      SVN
      <ul>
       <li>
        <strong>
         克隆仓库
        </strong>
        ：
        <code>
         svn checkout &lt;repository_url&gt;
        </code>
       </li>
       <li>
        <strong>
         提交更改
        </strong>
        ：
        <code>
         svn add &lt;file&gt;
        </code>
        然后
        <code>
         svn commit -m "Commit message"
        </code>
       </li>
       <li>
        <strong>
         更新文件
        </strong>
        ：
        <code>
         svn update
        </code>
       </li>
      </ul>
     </li>
     <li>
      CVS
      <ul>
       <li>
        <strong>
         检出模块
        </strong>
        ：
        <code>
         cvs checkout &lt;module_name&gt;
        </code>
       </li>
       <li>
        <strong>
         提交更改
        </strong>
        ：
        <code>
         cvs commit -m "Commit message"
        </code>
       </li>
       <li>
        <strong>
         更新文件
        </strong>
        ：
        <code>
         cvs update
        </code>
       </li>
      </ul>
     </li>
    </ul>
    <h2>
     <a id="git_47">
     </a>
     二、git基本概念
    </h2>
    <p>
     Git 是一个分布式版本控制系统，以下为你详细介绍其一些基本概念：
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/e0389543d2704a2ca393bd596040f946.jpeg#pic_center"/>
    </p>
    <h3>
     <a id="1Repository_54">
     </a>
     1.仓库（Repository）
    </h3>
    <ul>
     <li>
      <strong>
       定义
      </strong>
      ：仓库是存储项目所有文件和版本历史的地方。它包含了项目从创建以来的所有提交记录、分支信息等。
     </li>
     <li>
      <strong>
       分类
      </strong>
      <ul>
       <li>
        <strong>
         本地仓库
        </strong>
        ：存在于开发者本地计算机上的仓库，开发者可以在本地仓库进行各种版本控制操作，
        <strong>
         如提交、分支管理等，
         <mark>
          无需网络连接
         </mark>
        </strong>
        。
       </li>
       <li>
        <strong>
         远程仓库
        </strong>
        ：存储在远程服务器上的仓库，通常用于团队协作开发，多个开发者可以通过网络与远程仓库进行交互，共享代码和版本信息。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       创建仓库
      </strong>
      <ul>
       <li>
        创建本地仓库：使用
        <code>
         git init
        </code>
        命令可以将一个普通的本地目录初始化为一个 Git 仓库。
       </li>
       <li>
        克隆远程仓库：使用
        <code>
         git clone &lt;repository_url&gt;
        </code>
        命令可以将远程仓库复制到本地。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="2Commit_64">
     </a>
     2.提交（Commit）
    </h3>
    <ul>
     <li>
      <strong>
       定义
      </strong>
      ：提交是将本地工作目录中的更改保存到本地仓库的操作。每次提交都会生成一个唯一的哈希值（如 SHA - 1）来标识该提交，同时包含提交的作者、提交时间、提交说明等元数据。
     </li>
     <li>
      <strong>
       操作流程
      </strong>
      :
      <ul>
       <li>
        首先使用
        <code>
         git add &lt;file&gt;
        </code>
        命令将工作目录中修改的文件添加到暂存区。
       </li>
       <li>
        然后使用
        <code>
         git commit -m "commit message"
        </code>
        命令将暂存区的文件更改提交到本地仓库，其中
        <code>
         "commit message"
        </code>
        是对本次提交的简要描述。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="3Staging_Area_71">
     </a>
     3.暂存区（Staging Area）
    </h3>
    <ul>
     <li>
      <strong>
       定义
      </strong>
      ：
      <strong>
       暂存区也称为索引
       <code>
        （Index）
       </code>
      </strong>
      ，它是工作目录和本地仓库之间的一个中间区域。开发者可以选择将工作目录中的
      <strong>
       部分或全部更改添加到暂存区，然后再将暂存区的更改一次性提交到本地仓库
      </strong>
      。
     </li>
     <li>
      <strong>
       作用
      </strong>
      ：暂存区提供了一种灵活的方式来组织和管理提交内容，允许开发者将不同类型的更改分别提交，使提交记录更加清晰和有意义。
     </li>
     <li>
      操作命令
      <ul>
       <li>
        <code>
         git add &lt;file&gt;
        </code>
        ：将指定文件的更改添加到暂存区。
       </li>
       <li>
        <code>
         git add .
        </code>
        ：将当前工作目录下所有更改的文件添加到暂存区。
       </li>
       <li>
        <code>
         git reset &lt;file&gt;
        </code>
        ：将指定文件从暂存区移除，但不影响工作目录中的文件。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="4Branch_80">
     </a>
     4.分支（Branch）
    </h3>
    <ul>
     <li>
      <strong>
       定义
      </strong>
      ：分支是 Git 中非常重要的概念，它允许开发者在不影响主分支的情况下，独立地进行新功能开发、bug 修复等工作。每个分支都有自己独立的提交历史，可以在需要时将分支合并到其他分支。
     </li>
     <li>
      <strong>
       主分支
      </strong>
      ：通常情况下，每个仓库都有一个默认的主分支，命名为
      <code>
       master
      </code>
      或
      <code>
       main
      </code>
      ，它代表项目的稳定版本。
     </li>
     <li>
      操作命令
      <ul>
       <li>
        <code>
         git branch
        </code>
        ：查看本地分支列表。
       </li>
       <li>
        <code>
         git branch &lt;branch_name&gt;
        </code>
        ：创建一个新的分支。
       </li>
       <li>
        <code>
         git checkout &lt;branch_name&gt;
        </code>
        ：切换到指定的分支。
       </li>
       <li>
        <code>
         git checkout -b &lt;branch_name&gt;
        </code>
        ：创建并切换到一个新的分支。
       </li>
       <li>
        <code>
         git merge &lt;branch_name&gt;
        </code>
        ：将指定分支合并到当前分支。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="5Tag_91">
     </a>
     5.标签（Tag）
    </h3>
    <ul>
     <li>
      <strong>
       定义
      </strong>
      ：标签是对某个特定提交的一个固定引用，通常用于标记项目的重要版本，如发布版本、里程碑等。标签类似于分支，但它是静态的，不会随着新的提交而移动。
     </li>
     <li>
      操作命令
      <ul>
       <li>
        <code>
         git tag
        </code>
        ：查看所有标签。
       </li>
       <li>
        <code>
         git tag &lt;tag_name&gt;
        </code>
        ：为当前提交创建一个轻量级标签。
       </li>
       <li>
        <code>
         git tag -a &lt;tag_name&gt; -m "tag message"
        </code>
        ：为当前提交创建一个带注释的标签。
       </li>
       <li>
        <code>
         git push origin &lt;tag_name&gt;
        </code>
        ：将本地标签推送到远程仓库。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="6Remote_100">
     </a>
     6.远程（Remote）
    </h3>
    <ul>
     <li>
      <strong>
       定义
      </strong>
      ：远程是指远程仓库的别名，通过远程可以方便地与远程仓库进行交互。默认情况下，克隆远程仓库时会自动创建一个名为
      <code>
       origin
      </code>
      的远程。
     </li>
     <li>
      操作命令
      <ul>
       <li>
        <code>
         git remote -v
        </code>
        ：查看当前仓库的远程信息。
       </li>
       <li>
        <code>
         git remote add &lt;remote_name&gt; &lt;repository_url&gt;
        </code>
        ：添加一个新的远程仓库。
       </li>
       <li>
        <code>
         git push &lt;remote_name&gt; &lt;branch_name&gt;
        </code>
        ：将本地分支的更改推送到远程仓库。
       </li>
       <li>
        <code>
         git pull &lt;remote_name&gt; &lt;branch_name&gt;
        </code>
        ：从远程仓库拉取并合并更改到本地分支。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="7HEAD_109">
     </a>
     7.HEAD
    </h3>
    <ul>
     <li>
      <strong>
       定义
      </strong>
      ：HEAD 是一个特殊的引用，它指向当前所在的分支或提交。当切换分支时，HEAD 会随之移动到新的分支或提交上。
     </li>
     <li>
      <strong>
       作用
      </strong>
      ：HEAD 用于标识当前工作的位置，在进行提交、合并等操作时，Git 会根据 HEAD 的指向来确定操作的目标。
     </li>
    </ul>
    <h2>
     <a id="git_114">
     </a>
     三、git优缺点
    </h2>
    <h3>
     <a id="1Git__116">
     </a>
     1.Git 的优点
    </h3>
    <h4>
     <a id="_118">
     </a>
     分布式特性
    </h4>
    <ul>
     <li>
      <strong>
       离线工作
      </strong>
      ：
      <strong>
       <mark>
        每个开发者的本地环境都有完整的版本库
       </mark>
      </strong>
      ，包含所有提交历史和文件版本。这意味着开发者可以在没有网络连接的情况下进行日常开发工作，如提交代码、创建分支、查看历史记录等。例如，在飞机上、偏远地区等没有网络的环境中，开发者依然能够继续手中的项目开发，等网络恢复后再将本地的修改同步到远程仓库。
     </li>
     <li>
      <strong>
       数据安全
      </strong>
      ：由于每个开发者的本地都有完整的版本库副本，即使远程仓库出现故障（如服务器硬件损坏、网络攻击等），
      <strong>
       <mark>
        也可以从多个开发者的本地仓库中恢复数据
       </mark>
      </strong>
      。而且，Git 使用哈希算法对数据进行校验和存储，能有效防止数据在传输或存储过程中发生损坏。
     </li>
     <li>
      <strong>
       高效协作
      </strong>
      ：
      <strong>
       <mark>
        分布式的架构使得多个开发者可以并行地进行开发工作，各自在本地仓库进行修改和提交，然后再将本地的更改推送到远程仓库进行合并
       </mark>
      </strong>
      。这种方式大大提高了开发效率，减少了开发者之间的等待时间。
     </li>
    </ul>
    <h4>
     <a id="_124">
     </a>
     强大的分支管理
    </h4>
    <ul>
     <li>
      <strong>
       快速创建和切换
      </strong>
      ：Git 创建和切换分支的操作非常快速，几乎是瞬间完成。这使得开发者可以轻松地在不同的功能开发、bug 修复等任务之间进行切换，而不会影响其他工作。例如，当开发者正在进行一个新功能的开发时，突然发现一个紧急的 bug 需要修复，此时可以快速切换到一个新的分支进行 bug 修复，修复完成后再切换回原来的功能开发分支继续工作。
     </li>
     <li>
      <strong>
       并行开发
      </strong>
      ：多个开发者可以在不同的分支上同时进行开发，互不干扰。每个分支都有自己独立的提交历史，开发完成后可以将这些分支合并到主分支上。这种并行开发的方式极大地提高了团队的开发效率，尤其适用于大型项目的开发。
     </li>
    </ul>
    <h4>
     <a id="_129">
     </a>
     灵活的工作流程
    </h4>
    <ul>
     <li>
      <strong>
       多种工作流支持
      </strong>
      ：Git 支持多种不同的工作流程，如 GitFlow、GitHub Flow、GitLab Flow 等。团队可以根据项目的特点和需求选择适合的工作流程。例如，GitFlow 适用于有明确发布周期的项目，它定义了
      <strong>
       主分支（master）、开发分支（develop）、功能分支（feature）、发布分支（release）和热修复分支（hotfix）等不同类型的分支
      </strong>
      ，每个分支都有特定的职责和使用规则；而 GitHub Flow 则更简单轻量级，适用于快速迭代的项目，强调快速将功能合并到主分支。
     </li>
     <li>
      <strong>
       自定义工作流
      </strong>
      ：除了上述常见的工作流程，Git 还允许团队根据自身的实际情况自定义工作流程，以满足特定的开发需求。
     </li>
    </ul>
    <h4>
     <a id="_134">
     </a>
     社区支持和生态系统
    </h4>
    <ul>
     <li>
      <strong>
       广泛使用
      </strong>
      ：Git 是目前最流行的版本控制系统，被广泛应用于各种开源项目和商业项目中。大量的开发者使用 Git 进行项目开发，这意味着在遇到问题时可以很容易地在社区中找到解决方案。
     </li>
     <li>
      <strong>
       丰富的工具和服务
      </strong>
      ：围绕 Git 形成了丰富的生态系统，有许多优秀的工具和服务可供选择。例如，GitHub、GitLab、Bitbucket 等代码托管平台，提供了代码托管、协作开发、问题跟踪、持续集成等一系列功能；还有许多可视化的 Git 客户端工具，如 SourceTree、GitKraken 等，方便开发者进行操作。
     </li>
    </ul>
    <h3>
     <a id="2Git__139">
     </a>
     2.Git 的缺点
    </h3>
    <h4>
     <a id="_141">
     </a>
     学习曲线较陡
    </h4>
    <ul>
     <li>
      <strong>
       概念复杂
      </strong>
      ：Git 有许多复杂的概念，如仓库、提交、暂存区、分支、标签、远程等，对于初学者来说，理解这些概念并掌握它们之间的关系需要花费一定的时间和精力。例如，暂存区的概念可能会让初学者感到困惑，不清楚为什么需要将文件添加到暂存区再进行提交。
     </li>
     <li>
      <strong>
       命令繁多
      </strong>
      ：Git 的命令非常多，而且有些命令的使用方式和参数比较复杂。例如，
      <code>
       git rebase
      </code>
      命令可以用于整理提交历史，但使用不当可能会导致数据丢失或冲突难以解决。对于初学者来说，记住这些命令并正确使用它们是一个挑战。
     </li>
    </ul>
    <h4>
     <a id="_146">
     </a>
     冲突解决复杂
    </h4>
    <ul>
     <li>
      <strong>
       冲突频繁
      </strong>
      ：由于 Git 支持并行开发，多个开发者在不同的分支上进行修改，当合并分支时，很容易出现冲突。尤其是在多人同时修改同一文件的同一部分时，冲突的解决会比较麻烦。
     </li>
     <li>
      <strong>
       解决难度大
      </strong>
      ：解决冲突需要开发者对代码有深入的理解，并且要手动合并冲突的部分。在一些复杂的项目中，冲突可能涉及多个文件和多个提交，解决起来非常耗时且容易出错。
     </li>
    </ul>
    <h4>
     <a id="_151">
     </a>
     占用磁盘空间大
    </h4>
    <ul>
     <li>
      <strong>
       完整副本存储
      </strong>
      ：由于每个开发者的本地都有完整的版本库副本，包含了项目的所有历史记录和文件版本，这会占用大量的磁盘空间。对于一些大型项目，版本库可能会达到几百兆甚至几个吉字节，这对于磁盘空间有限的开发者来说可能是一个问题。
     </li>
     <li>
      <strong>
       数据冗余
      </strong>
      ：在分布式环境中，多个开发者的本地仓库可能会存在大量的数据冗余，进一步增加了磁盘空间的占用。
     </li>
    </ul>
    <h4>
     <a id="_156">
     </a>
     网络带宽要求高
    </h4>
    <ul>
     <li>
      <strong>
       克隆和推送数据量大
      </strong>
      ：当克隆一个远程仓库时，需要下载整个版本库的内容，包括所有的提交历史和文件版本，这会消耗大量的网络带宽。同样，当推送本地的更改到远程仓库时，如果更改的内容较多，也会占用较多的网络带宽。在网络带宽有限的情况下，克隆和推送操作可能会非常缓慢。
     </li>
    </ul>
    <h2>
     <a id="git_160">
     </a>
     四、git常用指令
    </h2>
    <p>
     Git 有众多指令，下面按照不同的使用场景为你介绍常用指令：
    </p>
    <h4>
     <a id="_164">
     </a>
     仓库操作
    </h4>
    <h5>
     <a id="1__166">
     </a>
     1. 初始化仓库
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> init
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：在当前目录下初始化一个新的 Git 仓库，会在该目录下创建一个
      <code>
       .git
      </code>
      隐藏文件夹，用于存储版本控制相关的信息。
     </li>
    </ul>
    <h5>
     <a id="2__174">
     </a>
     2. 克隆远程仓库
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> clone <span class="token operator">&lt;</span>repository_url<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：将远程仓库（如 GitHub、GitLab 上的仓库）复制到本地。
      <code>
       &lt;repository_url&gt;
      </code>
      是远程仓库的地址，可以是 HTTPS 或 SSH 格式。例如：
     </li>
    </ul>
    <pre><code class="prism language-bash"><span class="token function">git</span> clone https://github.com/user/repo.git
</code></pre>
    <h4>
     <a id="_186">
     </a>
     提交操作
    </h4>
    <h5>
     <a id="1__190">
     </a>
     1. 添加文件到暂存区
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> <span class="token function">add</span> <span class="token operator">&lt;</span>file<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：将指定的文件添加到暂存区。如果要添加当前目录下的所有文件，可以使用：
     </li>
    </ul>
    <pre><code class="prism language-bash"><span class="token function">git</span> <span class="token function">add</span> <span class="token builtin class-name">.</span>
</code></pre>
    <h5>
     <a id="2__202">
     </a>
     2. 提交暂存区的文件到本地仓库
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> commit <span class="token parameter variable">-m</span> <span class="token string">"commit message"</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：将暂存区的文件更改提交到本地仓库，
      <code>
       -m
      </code>
      后面跟着本次提交的描述信息，用于说明本次提交做了哪些更改。
     </li>
    </ul>
    <h5>
     <a id="3__210">
     </a>
     3. 一次完成添加和提交操作
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> commit <span class="token parameter variable">-am</span> <span class="token string">"commit message"</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：此命令仅适用于已被 Git 跟踪的文件，它会将所有已跟踪文件的更改添加到暂存区并提交到本地仓库。
     </li>
    </ul>
    <h4>
     <a id="_218">
     </a>
     分支操作
    </h4>
    <h5>
     <a id="1__220">
     </a>
     1. 查看分支
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> branch
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：查看本地仓库的所有分支，当前所在分支会以绿色显示并带有
      <code>
       *
      </code>
      号。
     </li>
    </ul>
    <h5>
     <a id="2__228">
     </a>
     2. 创建新分支
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> branch <span class="token operator">&lt;</span>branch_name<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：创建一个名为
      <code>
       &lt;branch_name&gt;
      </code>
      的新分支，但不会切换到该分支。
     </li>
    </ul>
    <h5>
     <a id="3__236">
     </a>
     3. 切换分支
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> checkout <span class="token operator">&lt;</span>branch_name<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：切换到指定的分支。
     </li>
    </ul>
    <h5>
     <a id="4__244">
     </a>
     4. 创建并切换到新分支
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> checkout <span class="token parameter variable">-b</span> <span class="token operator">&lt;</span>branch_name<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：这是一个组合命令，相当于先执行
      <code>
       git branch &lt;branch_name&gt;
      </code>
      再执行
      <code>
       git checkout &lt;branch_name&gt;
      </code>
      。
     </li>
    </ul>
    <h5>
     <a id="5__252">
     </a>
     5. 合并分支
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> merge <span class="token operator">&lt;</span>branch_name<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：将指定的
      <code>
       &lt;branch_name&gt;
      </code>
      分支合并到当前所在的分支。
     </li>
    </ul>
    <h5>
     <a id="6__260">
     </a>
     6. 删除分支
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> branch <span class="token parameter variable">-d</span> <span class="token operator">&lt;</span>branch_name<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：删除指定的本地分支，前提是该分支已经合并到其他分支。如果要强制删除未合并的分支，可以使用
      <code>
       -D
      </code>
      选项。
     </li>
    </ul>
    <h4>
     <a id="_268">
     </a>
     远程仓库操作
    </h4>
    <h5>
     <a id="1__270">
     </a>
     1. 查看远程仓库信息
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> remote <span class="token parameter variable">-v</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：显示当前仓库配置的所有远程仓库的名称和对应的 URL。
     </li>
    </ul>
    <h5>
     <a id="2__278">
     </a>
     2. 添加远程仓库
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> remote <span class="token function">add</span> <span class="token operator">&lt;</span>remote_name<span class="token operator">&gt;</span> <span class="token operator">&lt;</span>repository_url<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：添加一个新的远程仓库，
      <code>
       &lt;remote_name&gt;
      </code>
      是为远程仓库指定的别名，通常使用
      <code>
       origin
      </code>
      作为默认别名。
     </li>
    </ul>
    <h5>
     <a id="3__286">
     </a>
     3. 从远程仓库拉取代码
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> pull <span class="token operator">&lt;</span>remote_name<span class="token operator">&gt;</span> <span class="token operator">&lt;</span>branch_name<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：从指定的远程仓库的指定分支拉取代码，并将其合并到当前本地分支。如果不指定
      <code>
       &lt;branch_name&gt;
      </code>
      ，默认拉取当前分支对应的远程分支。
     </li>
    </ul>
    <h5>
     <a id="4__294">
     </a>
     4. 将本地代码推送到远程仓库
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> push <span class="token operator">&lt;</span>remote_name<span class="token operator">&gt;</span> <span class="token operator">&lt;</span>branch_name<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：将本地的指定分支推送到远程仓库的对应分支。如果是第一次推送该分支，可以使用
      <code>
       -u
      </code>
      选项，这样之后再推送该分支时就可以直接使用
      <code>
       git push
      </code>
      命令，例如：
     </li>
    </ul>
    <pre><code class="prism language-bash"><span class="token function">git</span> push <span class="token parameter variable">-u</span> origin <span class="token operator">&lt;</span>branch_name<span class="token operator">&gt;</span>
</code></pre>
    <h4>
     <a id="_306">
     </a>
     查看历史记录
    </h4>
    <h5>
     <a id="1__308">
     </a>
     1. 查看提交历史
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> log
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：显示当前分支的提交历史，包括提交的哈希值、作者、提交时间和提交说明等信息。
     </li>
    </ul>
    <h5>
     <a id="2__316">
     </a>
     2. 简洁查看提交历史
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> log <span class="token parameter variable">--oneline</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：以简洁的一行格式显示提交历史，只显示提交的哈希值前几位和提交说明。
     </li>
    </ul>
    <h4>
     <a id="_324">
     </a>
     撤销操作
    </h4>
    <h5>
     <a id="1__326">
     </a>
     1. 撤销暂存区的文件
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> reset <span class="token operator">&lt;</span>file<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：将指定文件从暂存区移除，但不会改变工作目录中的文件内容。
     </li>
    </ul>
    <h5>
     <a id="2__334">
     </a>
     2. 撤销工作目录的修改
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> checkout -- <span class="token operator">&lt;</span>file<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：丢弃工作目录中指定文件的修改，使其恢复到上一次提交时的状态。
     </li>
    </ul>
    <h5>
     <a id="3__342">
     </a>
     3. 回退到指定提交
    </h5>
    <pre><code class="prism language-bash"><span class="token function">git</span> reset <span class="token parameter variable">--hard</span> <span class="token operator">&lt;</span>commit_hash<span class="token operator">&gt;</span>
</code></pre>
    <ul>
     <li>
      <strong>
       说明
      </strong>
      ：将本地仓库的 HEAD、分支指针和工作目录都回退到指定的提交，
      <code>
       &lt;commit_hash&gt;
      </code>
      是提交的哈希值。使用
      <code>
       --hard
      </code>
      选项会彻底丢弃回退点之后的所有更改，请谨慎使用。
     </li>
    </ul>
