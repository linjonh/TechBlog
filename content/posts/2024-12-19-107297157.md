    <h2>
     <a id="2__3">
     </a>
     2 时间戳
    </h2>
    <p>
     音视频采样后会给每个音频采样、视频帧打一个时间戳，打包成RTP后放在RTP头中，称为RTP时间戳，RTP时间戳的单位依赖于音视频流各自的采样率。
    </p>
    <p>
     RTP Header格式如下：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/643af90624a47706001e2d87226e5e0e.png"/>
    </p>
    <h3>
     <a id="21__8">
     </a>
     2.1 视频时间戳
    </h3>
    <p>
     视频时间戳的单位为1/90000秒，但是90000并不是视频的采样率，而只是一个单位，帧率才是视频的采样率。
    </p>
    <p>
     不同打包方式下的时间戳：
    </p>
    <ul>
     <li>
      Single Nalu：如果一个视频帧包含1个NALU，可以单独打包成一个RTP包，那么RTP时间戳就对应这个帧的采集时间；
     </li>
     <li>
      FU-A：如果一个视频帧的NALU过大(超过MTU)需要拆分成多个包，可以使用FU-A方式来拆分并打到不同的RTP包里，那么这几个包的RTP时间戳是一样的；
     </li>
     <li>
      STAP-A：如果某帧较大不能单独打包，但是该帧内部单独的NALU比较小，可以使用STAP-A方式合并多个NALU打包发送，但是这些NALU的时间戳必须一致，打包后的RTP时间戳也必须一致。
     </li>
    </ul>
    <h3>
     <a id="22__16">
     </a>
     2.2 音频时间戳
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/9e8d11f61fba8df0122b0b133b5611d9.png">
      <br/>
      音频时间戳的单位就是采样率的倒数，例如采样率48000，那么1秒就有48000个采样，每个采样1/48ms，每个采样对应一个时间戳。RTP音频包一般打包20ms的数据，对应的采样数为 48000 * 20 / 1000 = 960，也就是说每个音频包里携带960个音频采样，因为1个采样对应1个时间戳，那么相邻两个音频RTP包的时间戳之差就是960。
     </img>
    </p>
    <h3>
     <a id="23_NTP_19">
     </a>
     2.3 NTP时间戳
    </h3>
    <p>
     RTP的标准并没有规定音频、视频流的第一个包必须同时采集、发送，也就是说开始的一小段时间内可能只有音频或者视频，再加上可能的网络丢包，音频或者视频流的开始若干包可能丢失，那么不能简单认为接收端收到的第一个音频包和视频包是对齐的，需要一个共同的时间基准来做时间对齐，这就是NTP时间戳的作用。
    </p>
    <p>
     NTP时间戳是从1900年1月1日00:00:00以来经过的秒数，发送端以一定的频率发送SR(Sender Report)这个RTCP包，分为视频SR和音频SR，SR包内包含一个RTP时间戳和对应的NTP时间戳，接收端收到后就可以确定某个流的RTP时间戳和NTP时间戳的对应关系，这样音频、视频的时间戳就可以统一到同一个时间基准下。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/2bedcba98db77a68784dc29871e24367.png">
      <br/>
      如上图，发送端的音视频流并没有对齐，但是周期地发送SR包，接收端得到音视频SR包的RTP时间戳、NTP时间戳后通过
      <strong>
       线性回归
      </strong>
      得到NTP时间戳T
      <sub>
       ntp
      </sub>
      和RTP时间戳T
      <sub>
       rtp
      </sub>
      时间戳的对应关系：
     </img>
    </p>
    <ul>
     <li>
      T
      <sub>
       ntp_audio
      </sub>
      = f(T
      <sub>
       rtp_audio
      </sub>
      )
     </li>
     <li>
      T
      <sub>
       ntp_video
      </sub>
      = f(T
      <sub>
       rtp_video
      </sub>
      )
     </li>
    </ul>
    <p>
     其中T
     <sub>
      ntp
     </sub>
     = f(T
     <sub>
      rtp
     </sub>
     ) = kT
     <sub>
      rtp
     </sub>
     + b 为线性函数，这样接收端每收到一个RTP包，都可以将RTP时间戳换算成NTP时间戳，从而在同一时间基准下进行音视频同步。
    </p>
    <h2>
     <a id="2__30">
     </a>
     2 延迟
    </h2>
    <p>
     视频延迟的单位为ms，对音频来说，由于采样跟时间戳一一对应，所有时间延迟都会被换算成了缓存大小(音频包的个数)，其值为：
    </p>
    <pre><code>音频延迟 = 时间延迟 &lt;&lt; 8 / 20
</code></pre>
    <p>
     也就是说，对48000的采样率，960个采样对应一个20ms包，时间延迟 / 20ms等于延迟了几个包，左移8(乘以256)也就是所谓的Q8，是为了用定点数表示一定精度的浮点数。
    </p>
    <h2>
     <a id="3__37">
     </a>
     3 同步
    </h2>
    <h3>
     <a id="31__38">
     </a>
     3.1 一张图看懂音视频同步
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/2d89f276b454623c11e17f3c582bf367.png">
      <br/>
      首先接收端需要按照音、视频各自的帧率来解码、渲染，保证流畅地播放，在这个基础上，需要计算音视频两个流目前的相对延迟，分别给音、视频两个流施加一定的延迟，保证音视频的同步。
     </img>
    </p>
    <blockquote>
     <p>
      延迟播放，也就意味着在缓存中暂时存放数据，延迟换流畅。
     </p>
    </blockquote>
    <p>
     对音频来说，施加的延迟直接影响到音频缓存的大小，音频缓存的大小就体现了音频的播放延迟。
    </p>
    <p>
     对视频来说，施加的延迟影响到视频帧的渲染时间，通过比较渲染时间和当前时间来决定解码后的视频帧需要等待还是需要立刻渲染。
    </p>
    <p>
     正确设置好音视频各自的播放延迟后，音视频达到同步的效果。
    </p>
    <p>
     可以看到，音视频同步中主要需要做到三点：
    </p>
    <ul>
     <li>
      正确计算音视频相对延迟；
     </li>
     <li>
      正确计算音视频各自的网络目标时延；
     </li>
     <li>
      正确设置音视频各自的播放延迟。
     </li>
    </ul>
    <h3>
     <a id="32__55">
     </a>
     3.2 音视频相对延迟
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/66751023c65b77a2c96942737063a2ca.png">
      <br/>
      如上图：
     </img>
    </p>
    <p>
     最近一对音视频包的相对延迟 = (T
     <sub>
      video_recv
     </sub>
     - T
     <sub>
      audio_recv
     </sub>
     ) - (T
     <sub>
      video_send
     </sub>
     - T
     <sub>
      audio_send
     </sub>
     )
    </p>
    <p>
     其中T
     <sub>
      video_recv
     </sub>
     、T
     <sub>
      audio_recv
     </sub>
     分别是接收端收到视频包、音频包记录的本地时间，可以直接获取，而T
     <sub>
      video_send
     </sub>
     ，T
     <sub>
      audio_send
     </sub>
     作为视频包、音频包的发送时间无法直接获取，因为接收到的RTP包只有RTP时间戳，无法直接作为本地时间来与T
     <sub>
      video_recv
     </sub>
     、T
     <sub>
      audio_recv
     </sub>
     进行运算，这时候就需要SR包中携带的NTP时间戳和RTP的对应关系来进行换算。
    </p>
    <p>
     通过SR包中的NTP时间戳和RTP时间戳做线性回归(通过采样归纳映射关系)得到两者的线性关系：
     <br/>
     T
     <sub>
      ntp
     </sub>
     = f(T
     <sub>
      rtp
     </sub>
     ) = kT
     <sub>
      rtp
     </sub>
     + b
    </p>
    <p>
     这样RTP时间戳就可以直接转化为NTP时间戳，也就是发送端本地时间。从最近一对音视频包相对延迟的计算公式可以看出，分别对发送端和接收端的时间做运算，两者都在同一时间基准，可以排除NTP时间同步问题的影响。
    </p>
    <blockquote>
     <p>
      stream_synchronization.cc:34
      <br/>
      StreamSynchronization::ComputeRelativeDelay
     </p>
    </blockquote>
    <h3>
     <a id="33__70">
     </a>
     3.3 期望目标延迟
    </h3>
    <p>
     期望目标延迟就是保证音频流、视频流各自流畅播放的
     <strong>
      期望延迟
     </strong>
     。
    </p>
    <p>
     从3.1的图可以看出，对视频来说，期望目标延迟 = 网络延迟 + 解码延迟 + 渲染延迟，对音频来说，期望目标延迟 = 前后两个音频包之间的到达间隔的期望值。在接收时间的基础上，加上各自的期望目标延迟进行播放，可以保证音频、视频流可以按照各自的步调进行流畅无卡顿的播放。
    </p>
    <p>
     既要流畅播放又要进行同步，这就是为什么在计算音视频流相对延迟的时候要同时考虑最近一对音视频包的相对延迟又要考虑音视频目标延迟差的原因。
    </p>
    <blockquote>
     <p>
      stream_synchronization.cc:34
      <br/>
      StreamSynchronization::ComputeRelativeDelay
     </p>
    </blockquote>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/b9146e65d9d8078693e4a9f141db6a3d.png"/>
    </p>
    <pre><code>当前音视频流相对延迟 = 最近一对音视频包的相对延迟 + 音视频目标延迟之差
</code></pre>
    <h4>
     <a id="331__85">
     </a>
     3.3.1 期望视频目标延迟
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/b532446de60ddcd636c8790701fc18a6.png"/>
    </p>
    <pre><code>期望视频目标延迟 = 网络延迟 + 解码延迟 + 渲染延迟
</code></pre>
    <p>
     网络延迟其实就是视频JittterBuffer输出的延迟googJitterBufferMs，可以参考我的文章
     <a href="https://blog.csdn.net/sonysuqin/article/details/106629343">
      《WebRTC视频JitterBuffer详解》
     </a>
     7.1节[抖动计算]，简单说就是通过卡尔曼滤波器计算视频帧的到达延迟差(抖动)，作为网络的延迟。
    </p>
    <p>
     解码时间的统计方法：统计最近最多10000次解码的时间消耗，计算其95百分位数T
     <sub>
      decode
     </sub>
     ，也就是说最近95%的帧的解码时间都小于T
     <sub>
      decode
     </sub>
     ，以之作为解码时间。
    </p>
    <p>
     视频渲染延迟默认是一个定值：10ms。
    </p>
    <blockquote>
     <p>
      timing.cc:210
      <br/>
      VCMTiming::TargetVideoDelay
     </p>
    </blockquote>
    <h4>
     <a id="332__99">
     </a>
     3.3.2 期望音频目标延迟
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/ff059e76f75655f11e3d35e1a8822f86.png">
      <br/>
      期望音频目标延迟的算法和视频解码时间的算法类似，但是用直方图来存放最近的65个音频包的到达间隔，取95百分位数T
      <sub>
       audio_target_delay
      </sub>
      ，也就是说最近一段时间内，有95%的音频包的到达间隔都小于T
      <sub>
       audio_target_delay
      </sub>
      。同时考虑到网络突发的可能，增加了峰值检测，去掉异常的时间间隔。
     </img>
    </p>
    <p>
     取这个值作为期望目标延迟来影响音频的播放，可以保证绝大多数情况下音频流的流畅。
    </p>
    <blockquote>
     <p>
      neteq_impl.cc:311
      <br/>
      NetEqImpl::FilteredCurrentDelayMs
     </p>
    </blockquote>
    <h3>
     <a id="34__107">
     </a>
     3.4 音视频同步
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/8e32713d40ffd6ea1b7b927e7be17de9.png">
      <br/>
      同步器的外部输入有：
     </img>
    </p>
    <ul>
     <li>
      期望音频目标延迟，以该延迟播放，音频是流畅的；
     </li>
     <li>
      期望视频目标延迟，以该延迟播放，视频是流畅的；
     </li>
     <li>
      最近一对音视频包的相对延迟。
     </li>
    </ul>
    <p>
     最近一对音视频包的相对延迟与音视频的目标延迟差之和，得到当前时刻的音视频相对延迟，也就是音、视频流目前的时间偏差。
    </p>
    <ul>
     <li>
      当相对延迟 &gt; 0，说明视频比较慢，视频延迟与基准(base_target_delay_ms_，默认0)比较：
      <ul>
       <li>
        extra_video_delay_ms &gt; base_target_delay_ms_，减小视频流延迟，设置音频延迟为基准；
       </li>
       <li>
        extra_video_delay_ms &lt;= base_target_delay_ms_，增大音频流延迟，设置视频延迟为基准；
       </li>
      </ul>
     </li>
     <li>
      当相对延迟 &lt; 0，说明音频比较慢，音频延迟与基准(base_target_delay_ms_，默认0)比较：
      <ul>
       <li>
        extra_audio_delay_ms &gt; base_target_delay_ms_，减小音频流延迟，设置视频延迟为基准；
       </li>
       <li>
        extra_audio_delay_ms &lt;= base_target_delay_ms_，增大视频流延迟，设置音频延迟为基准。
       </li>
      </ul>
     </li>
    </ul>
    <p>
     使用这个算法，可以保证音、频流的延迟都趋向于逼近基准，不会出现无限增加、减小的情况。同时，一次延迟增大、减小的延迟diff_ms被设置为相对延迟的一半，并限制在80ms范围之内，也就是说WebRTC对一次同步的追赶时间做了限制，一次延迟增大、减小最大只能是80ms，因此如果某个时刻某个流发生了较大抖动，需要一段时间另外一个流才能同步。
    </p>
    <p>
     经过了以上校准之后，输出了同步后音频、视频流各自的最小播放延迟。
    </p>
    <pre><code>extra_audio_delay_ms -&gt; 音频最小播放延迟
extra_video_delay_ms -&gt; 视频最小播放延迟
</code></pre>
    <p>
     理论上将这两个播放延迟分别施加到音、视频流后，这两个流就是同步的，再与音、视频流各自期望目标延迟取最大值，得到音、视频流的最优目标延迟(googTargetDelayMs)，施加在音、视频流上，可以保证做到既同步、又流畅。
    </p>
    <blockquote>
     <p>
      stream_synchronization.cc:64
      <br/>
      StreamSynchronization::ComputeDelays
     </p>
    </blockquote>

    <h5>
     <a id="3511__136">
     </a>
     3.5.1.1 期望接收时间
    </h5>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/6bf1d6d7db843d5c864d8dc65f747ac6.png">
      <br/>
      TimestampExtrapolator类负责期望接收时间的产生，视频JitterBuffer(的FrameBuffer)每收到一帧，会记录该帧的RTP时间戳T
      <sub>
       frame_rtp
      </sub>
      和本地接收时间T
      <sub>
       frame_rcv
      </sub>
      ，其中第一帧的RTP时间戳为T
      <sub>
       first_frame_rtp
      </sub>
      和本地接收时间T
      <sub>
       first_frame_rcv
      </sub>
      。
      <br/>
      记帧RTP时间戳之差：T
      <sub>
       frame_rtp_delta
      </sub>
      = T
      <sub>
       frame_rtp
      </sub>
      - T
      <sub>
       first_frame_rtp
      </sub>
      <br/>
      帧本地接收时间之差：T
      <sub>
       frame_recv_delta
      </sub>
      = T
      <sub>
       frame_recv
      </sub>
      - T
      <sub>
       first_frame_rcv
      </sub>
      <br/>
      两者为线性关系，期望RTP时间戳之差T
      <sub>
       frame_rtp_delta
      </sub>
      = _w[0] * T
      <sub>
       frame_recv_delta
      </sub>
      + _w[1]
      <br/>
      通过卡尔曼滤波器得到线性系数_w[0]、_w[1]，进而得到期望接收时间的值：
      <br/>
      T
      <sub>
       frame_recv
      </sub>
      = T
      <sub>
       first_frame_rcv
      </sub>
      + (T
      <sub>
       frame_rtp_delta
      </sub>
      - _w[1]) / _w[0]
     </img>
    </p>
    <p>
     也就是说，卡尔曼滤波器输入视频帧的RTP时间戳和本地接收时间观测值，得到视频帧最优的期望接收时间，用于平滑网络的抖动。
    </p>

    <h5>
     <a id="3512___googCurrentDelayMs_149">
     </a>
     3.5.1.2 视频当前延迟 - googCurrentDelayMs
    </h5>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/99835b1bf540342bd0dd3e5c430e0d15.png"/>
     <br/>
     解码器通过视频JitterBuffer的NextFrame方法获取一帧去解码时会设置该帧的期望渲染时间T
     <sub>
      expect_render
     </sub>
     ，以及该帧的实际开始解码时间T
     <sub>
      actual_decode
     </sub>
     。
    </p>
    <p>
     该帧的期望开始解码时间为期望渲染时间减去解码、渲染的延迟：
     <br/>
     T
     <sub>
      expect_decode
     </sub>
     = T
     <sub>
      expect_render
     </sub>
     - T
     <sub>
      decode_delay
     </sub>
     - T
     <sub>
      render_delay
     </sub>
    </p>
    <p>
     那么该帧产生的延迟为实际开始解码时间减去期望开始解码时间：
     <br/>
     T
     <sub>
      frame_delay
     </sub>
     = T
     <sub>
      actual_decode
     </sub>
     - T
     <sub>
      expect_decode
     </sub>
    </p>
    <p>
     该帧延迟和上一个时刻的视频当前延迟叠加，如果仍然小于目标延迟，则增长视频当前延迟。
     <br/>
     T
     <sub>
      current_delay
     </sub>
     = max(T
     <sub>
      current_delay
     </sub>
     + T
     <sub>
      frame_delay
     </sub>
     , T
     <sub>
      target_delay
     </sub>
     )
    </p>
    <p>
     也就是视频当前延迟以目标延迟为上限逼近目标延迟。
    </p>

     取同步后的延迟作为视频的实际延迟，也就是当前延迟和最小播放延迟的最大者：

      actual_delay

    <h4>
     <a id="352__174">
     </a>
     3.5.2 音频渲染时间
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/47a09300dd4b8b01ac88f59aff8b3fe3.png"/>
     <br/>
     NetEQ中有若干缓存用来暂存数据，主要的是JitterBuffer(PacketBuffer)、SyncBuffer，分别存放解码前和解码后的数据，这些缓存的大小就体现了音频当前的延迟。
    </p>
    <p>
     NetEQ的BufferLevelFilter类维护音频的当前延迟，音频渲染器每取一次音频数据都根据当前剩余的缓存大小设置一次音频的当前延迟并进行平滑，得到平滑后的当前延迟(googCurrentDelayMs)。
    </p>
    <blockquote>
     <p>
      buffer_level_filter.cc:29
      <br/>
      BufferLevelFilter::Update
     </p>
    </blockquote>
    <p>
     NetEQ的DecisionLogic类比较下一个音频包的时间戳与SynBuffer中的结尾时间戳，如果不相等，也就是不连续，那么需要进行丢包隐藏(Expand/PLC)或者融合(Merge)；如果相等，也就是连续，则根据当前缓存的大小与目标延迟大小来决定是对音频数据进行加速、减速，或者正常播放。
    </p>

     </li>
     <li>

    </p>
