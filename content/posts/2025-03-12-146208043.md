    <h2>
     <a id="2_BUG_12">
     </a>
     2. BUG
    </h2>
    <h3>
     <a id="21__bug_13">
     </a>
     2.1 什么是 bug
    </h3>
    <ol>
     <li>
      当且仅当需求文档是存在的且正确的，程序与其不匹配才是错误的
     </li>
     <li>
      需求文档未提及的功能，判断标准以用户未标准，若不符合用户要求的就是 bug
     </li>
    </ol>
    <h3>
     <a id="22__bug_17">
     </a>
     2.2 如何提 bug
    </h3>
    <blockquote>
     <p>
      <strong>
       描述bug的基本要素：
      </strong>
     </p>
     <ol>
      <li>
       问题出现的版本
      </li>
      <li>
       问题出现的环境
      </li>
      <li>
       问题出现的步骤
      </li>
      <li>
       预期结果
      </li>
      <li>
       实际结果
      </li>
     </ol>
    </blockquote>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/657d0772697c42c88e13c24eebc40932.png"/>
    </p>

    <h3>
     <a id="23_bug__30">
     </a>
     2.3 bug 的级别
    </h3>
    <p>
     bug级别一般分为：
     <strong>
      崩溃、严重、一般、次要
     </strong>
    </p>
    <table>
     <thead>
      <tr>
       <th>
        崩溃
       </th>
       <th>
        严重
       </th>
       <th>
        一般
       </th>
       <th>
        次要
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         核心特征
        </strong>
        ：
        <br/>
        • 系统崩溃/死循环
        <br/>
        • 数据库数据丢失
        <br/>
        • 主要功能丧失
        <br/>
        <br/>
        <strong>
         典型案例
        </strong>
        ：
        <br/>
        • 代码死循环
        <br/>
        • 一级菜单功能失效
        <br/>
        <br/>
        <strong>
         处理策略
        </strong>
        ：
        <br/>
        立即中止版本测试
       </td>
       <td>
        <strong>
         核心特征
        </strong>
        ：
        <br/>
        • 部分功能失效
        <br/>
        • 数据保存错误
        <br/>
        • 安全稳定性问题
        <br/>
        <br/>
        <strong>
         典型案例
        </strong>
        ：
        <br/>
        • 程序自动退出
        <br/>
        • 接口调用冲突
        <br/>
        <br/>
        <strong>
         处理策略
        </strong>
        ：
        <br/>
        可继续其他功能测试
       </td>
       <td>
        <strong>
         核心特征
        </strong>
        ：
        <br/>
        • 功能未完全实现
        <br/>
        • 非稳定性缺陷
        <br/>
        <br/>
        <strong>
         典型案例
        </strong>
        ：
        <br/>
        • 边界条件错误
        <br/>
        • 查询响应超时
        <br/>
        <br/>
        <strong>
         处理策略
        </strong>
        ：
        <br/>
        测试中最高发问题
       </td>
       <td>
        <strong>
         核心特征
        </strong>
        ：
        <br/>
        • 界面显示异常
        <br/>
        • 性能优化建议
        <br/>
        <br/>
        <strong>
         典型案例
        </strong>
        ：
        <br/>
        • 文字排版错位
        <br/>
        • 提示语缺失
        <br/>
        <br/>

         处理策略

        <br/>

    <h3>
     <a id="25__41">
     </a>
     2.5 与开发起争执怎么办（高频面试题）
    </h3>
    <blockquote>
     <ol>
      <li>
       检查自身，是否 bug 描述不清楚
      </li>
      <li>
       站在用户角度考虑并抛出问题
      </li>
      <li>
       bug 的定级要有理有据，根据企业的定级规范
      </li>
      <li>
       提升技术能力和业务水平，做到不仅能提出问题，最好也要能给出解决方案给开放借鉴（不可以以命令的语气）
      </li>
      <li>
       如果确实是 Bug，开发比较犟，那就召开 Bug 评审，需要有相关的代表来参加：产品代表、开发代表、测试代表等
       <br/>
       （1. 如何解决 Bug 2. 如何预防类似的 Bug 再发生）
      </li>
     </ol>
    </blockquote>
    <blockquote>
     <p>
      bug评审主要解决两个问题：
     </p>
     <ol>
      <li>
       决定如何处理bug
      </li>
      <li>
       分析缺陷产生的原因，找出预防的对策
      </li>
     </ol>
    </blockquote>
