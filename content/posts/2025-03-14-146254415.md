# App测试的学习笔记

* 功能测试
* 兼容性测试
  + 手机型号
    - 覆盖市场主流机型
    - 考虑APP线上用户机型排名
  + 手机系统版本
    - 安卓：4.4，5.1，6.0，7.0， 8.0，9.0
    - iOS： 9.x, 10.x, 11.X, 12.x
  + 屏幕尺寸
  + 分辨率
  + 网络
    - 2G，3G，4G，WI-FI
  + 应用兼容性
    - 与手机硬件兼容：home键，电源键，音量键
    - 与外部硬件设备兼容：耳机，蓝牙
    - 与操作系统软件兼容： wlan, 系统时间调节，LBS定位
    - 与其他APP兼容：后台播放音乐时，进入动态页面点击动态视频的播放
* 安装，卸载，升级
  + 安装
    - 正常场景
      * 在不同系统上安装
      * 从不同渠道安装：APP商城，手机助手，直接下载包
      * 不同的安装路径：安装到手机上，安装到SD卡上
    - 异常场景
      * 安装时出现异常（关机，断网），恢复后能否继续安装
      * 安装时存储空间不足
      * 安装时取消后再次安装
      * 正在运行时覆盖安装
      * 低版本覆盖安装高版本
      * 卸载后安装
  + 卸载
    - 正常卸载（App手动卸载，工具卸载）
    - 运行时卸载
    - 取消卸载
    - 卸载异常终断
    - 卸载后无数据残留
  + 升级
    - 从临近版本升级
    - 跨版本升级
    - 不同渠道升级（应用商城，手机助手）
    - 升级提醒成功（可不提醒，可以提升升级，强制升级）
    - 应用内升级时非Wi-Fi提醒
    - 升级后要观察升级前的数据是否正常
* 交叉事件测试
  + app运行的时候，接电话，拍照，查看推送消息，切换网络等
* 用户体验测试
  + UI界面
  + 易用性
  + 横竖屏
  + 其他辅助功能：调整字体大小，语音转换，多点触碰
* 稳定性测试 Monkey
* 弱网测试
* 性能测试 GT
  + **CPU**

    - CPU    整体CPU的使用水平
      * 用户态：表示CPU处于应用程序执行的时间
      * 系统态：表示系统内核执行的时间
      * 空闲态：表示空闲系统进程执行的时间
      * CPU使用率=CPU执行非系统空闲进程时间 / CPU总的执行时间
    - Jiffies
      * 表示开机以来，应用程序消耗的CPU时间片的总数
    - CPU问题产生的影响：
      * CPU使用长时间处于90%以上
      * 手机发热，耗电量增加
      * 反应变慢，引起ANR（Application not responding)
  + 内存

    - 内存
      * 私有内存：进程独占的内存，在进程销毁的时候同步回收私有内存
      * 实际使用内存PSS：除了私有内存外，与其他进程共享的内存空间，在进程销毁的时候不回收
    - 常见的内存问题
      * 内存泄露: 是指程序在申请内存后，无法释放已申请的内存空间
      * 内存溢出： 是指程序在申请内存时，没有足够的内存空间供其使用，出现out of memory
    - 内存问题产生的影响
      * 程序实际使用的内存PSS持续增长
      * 程序出现Crash
  + 流畅度

    - FPS（Frames per second) GPU 在一秒内绘制的帧数
    - 流畅度问题产生的影响
      * 想要让大脑觉得动作是连续的，至少是每秒10-12帧的速度
      * 要想达到流畅的效果，至少需要每秒24帧
      * 60帧每秒的流畅度是最佳的
  + 流量测试

    - 流量：手机通过运营商上网时，运营商的网络替我们转发报文，报文包括上下行的报文
    - 测试流量的方法
      * 流量统计（统计上下行报文）
      * 抓包（fiddler, tcpdump)
    - 流量优化方法
      * 数据的压缩
      * 不同数据格式的采用
      * 控制访问的频次
      * 只获取必要的数据
      * 缓存机制
      * 针对不同的网络类型设置不同的访问策略
  + 电量测试

    - 电量测试：测试移动设备电量消耗快慢的一种测试方法，一般是平均电流
    - 常见的耗电场景
      * 定位，尤其是调用GPS
      * 网络传输，尤其是非wifi环境
      * 屏幕亮度
      * CPU频率
      * 内存调度频率
      * wake_locker时间和次数
    - 测试结果分析
      * 与基准对比
      * 横向对比，与竞品对比
  + 启动速度测试

    - App启动分为冷启动，热启动
      * 冷启动：app被后台杀死后，在这个状态下打开app
      * 热启动：app没被后台杀死，仍然在后台运行，打开app
    - 测试方法
      * adb shell am start -W -n 包名/Activity命， 查看app启动耗时
      * 该命令获取3个关键指标
        + thisTime:表示一连串启动activity，最后一个activity的时间
        + totalTime: 应用的启动时间，包括创建进程，App初始化，activity初始化到界面显示
        + waitTime:前一个应用activity pause的时间+totalTime
      * 一般需要测试多次，比如10组，取平均值
    - 测试结果分析
      * 与基准对比
      * 横向对比，与竞品对比
