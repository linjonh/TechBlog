    <h2>
     前言
    </h2>
    <p>
     上篇文章主要介绍的是网络通信方面的一些知识，也算是为以后的学习先做一下铺垫。从本章开始我们就要正式进入网络编程的学习了。
    </p>
    <p>
     但是在开始网络编程之前，我们还需要进行一些准备工作，需要先了解一些概念。
    </p>
    <h3 id="1%E3%80%81%E6%BA%90ip%E5%9C%B0%E5%9D%80%E5%92%8C%E7%9B%AE%E7%9A%84ip%E5%9C%B0%E5%9D%80" name="1%E3%80%81%E6%BA%90ip%E5%9C%B0%E5%9D%80%E5%92%8C%E7%9B%AE%E7%9A%84ip%E5%9C%B0%E5%9D%80">
     1、源ip地址和目的ip地址
    </h3>
    <p>
     从上篇文章我们知道不同的局域网的主机之间的通信是靠ip地址进行的，所以
     <span style="color:#511b78">
      <strong>
       源ip地址
      </strong>
     </span>
     <span style="color:#fe2c24">
      即指发送的主机的ip地址
     </span>
     ，
     <strong>
      <span style="color:#511b78">
       目的ip地址
      </span>
     </strong>
     <span style="color:#fe2c24">
      指接收主机的ip地址
     </span>
     <span style="color:null">
      。就像车辆的始发站和终点站一样，即使中途经过了好多地方，但是一般情况下，终点站是不会变的。
     </span>
    </p>
    <h3 id="2%E3%80%81%E7%AB%AF%E5%8F%A3%E5%8F%B7%20%E5%92%8C%20socket%E5%A5%97%E6%8E%A5%E5%AD%97" name="2%E3%80%81%E7%AB%AF%E5%8F%A3%E5%8F%B7%20%E5%92%8C%20socket%E5%A5%97%E6%8E%A5%E5%AD%97">
     <span style="color:null">
      2、端口号 和 socket套接字
     </span>
    </h3>
    <p>
     <span style="color:null">
      网络通信可以看作是两台主机在通信，不过在通信过程中我们不仅仅需要考虑两台主机之间的数据交互。
     </span>
    </p>
    <p>
     <span style="color:null">
      网络通信、数据交互说到底都是为用户提供的交互，无论是朋友之间的线上聊天还是网络游戏。在上篇中我们讲到网络协议栈，我们知道网络层和传输层是属于操作系统的，更上层的应用层才是属于用户使用的。
     </span>
    </p>
    <p>
     <span style="color:null">
      虽然操作系统是由用户操作的，但是在本质上，在操作系统层面，用户的身份是由程序体现的，要实现通信，程序一定是在运行中，也就是
     </span>
     <span style="color:#fe2c24">
      <strong>
       进程
      </strong>
     </span>
     <span style="color:null">
      。那么在操作系统看来，两台主机进行通信就是通过运行应用层程序进行通信，也就是说是
     </span>
     <span style="color:#fe2c24">
      <strong>
       进程在通信和交换数据
      </strong>
     </span>
     <span style="color:null">
      。
     </span>
    </p>
    <p>
     <span style="color:null">
      所以网络通信的本质实际上是进程间的通信，只不过是不同主机之间的进程。而
     </span>
     <span style="color:#fe2c24">
      <strong>
       端口号
      </strong>
     </span>
     <span style="color:null">
      ，其实就是用来
     </span>
     <span style="color:#fe2c24">
      <strong>
       表示唯一进程的标识符
      </strong>
     </span>
     <span style="color:null">
      ，是
     </span>
     <span style="color:#fe2c24">
      <strong>
       传输层协议的內容
      </strong>
     </span>
     <span style="color:null">
      。
     </span>
    </p>
    <p>
     <span style="color:#fe2c24">
      <strong>
       所以对当前主机来说，ip地址保证了主机的唯一性，端口号（PORT）保证了主机内进程的唯一性。
      </strong>
     </span>
    </p>
    <blockquote>
     <p>
      <span style="color:null">
       当端口号和IP地址按照如下方式使用就可以标识到
      </span>
      <span style="color:#fe2c24">
       <strong>
        网络中的唯一进程
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#fe2c24">
       <strong>
        IP地址:PORT
       </strong>
      </span>
      <span style="color:null">
       ,该组合也被称为 socket套接字
      </span>
     </p>
     <p>
      <span style="color:null">
       注：端口号是一个2字节16位的整数，一个端口号只能被一个进程占用，即一个端口号只能标识一个进程。
      </span>
     </p>
    </blockquote>
    <p>
     那么我们思考一个问题，就是在系统中每个进程都有自己的PID作为自己的唯一标识符，为什么还需要端口号呢？
    </p>
    <p>
     <strong>
      首先，利用进程的PID来进行标识网络中的唯一进程理论和操作上当然是可以实现的，但是这样做不好，因为进程的PID就是来方便进程管理的，进程并不都是需要进行网络通信的，如果使用了PID来标识网络中的唯一进程，就相当于将操作系统和网络强耦合起来了，这并不好。
     </strong>
    </p>
    <h2 id="3%E3%80%81%E6%BA%90%E7%AB%AF%E5%8F%A3%E5%8F%B7%E5%92%8C%E7%9B%AE%E7%9A%84%E7%AB%AF%E5%8F%A3%E5%8F%B7" name="3%E3%80%81%E6%BA%90%E7%AB%AF%E5%8F%A3%E5%8F%B7%E5%92%8C%E7%9B%AE%E7%9A%84%E7%AB%AF%E5%8F%A3%E5%8F%B7">
     3、源端口号和目的端口号
    </h2>
    <p>
     源IP地址和目的IP地址是用来确定主机的，但是网络通信的本质实际上是进程间的通信，所以在网络间通信的时候，除了需要确定对应的主机IP还需要知道主机上的进程，因此就需要知道 源端口号和目的端口号来确定源主机的发送进程和目的主机的进程。
    </p>
    <blockquote>
     <p>
      <span style="color:#fe2c24">
       <strong>
        源IP：源端口号
       </strong>
      </span>
      <strong>
       和
       <span style="color:#fe2c24">
        目的IP：目的端口号
       </span>
       就组成了一个socket对。
      </strong>
     </p>
    </blockquote>
    <h2 id="4%E3%80%81%E8%AE%A4%E8%AF%86TCP%E5%8D%8F%E8%AE%AE%E7%9A%84%E5%9F%BA%E6%9C%AC%E7%89%B9%E7%82%B9" name="4%E3%80%81%E8%AE%A4%E8%AF%86TCP%E5%8D%8F%E8%AE%AE%E7%9A%84%E5%9F%BA%E6%9C%AC%E7%89%B9%E7%82%B9">
     4、认识TCP协议的基本特点
    </h2>
    <p>
     先对TCP的传输控制协议做一个简单的认识
    </p>
    <ol>
     <li>
      <strong>
       传输层协议
      </strong>
     </li>
     <li>
      <strong>
       有链接
      </strong>
      。如我们使用的SSH连接服务器主机，SSH使用的就是TCP协议，必须要与服务器主机连接上之后才能正常地与服务器主机进行通信。
     </li>
     <li>
      <strong>
       可靠传输
      </strong>
      。在传输数据时，TCP会使用各种防止数据丢失或者损坏，但是并不表示TCP传输一定是可靠的。
     </li>
     <li>
      <strong>
       面向字节流
      </strong>
      。指它将应用层传递下来的数据视为一个无结构的字节序列，即字节流。这意味着应用程序通过TCP发送数据时，这些数据可以被视为是连续的比特或字节序列，而不是一个个独立的消息或数据包。
     </li>
    </ol>
    <h2 id="5%E3%80%81UDP%E5%8D%8F%E8%AE%AE%E7%9A%84%E5%9F%BA%E6%9C%AC%E7%89%B9%E7%82%B9" name="5%E3%80%81UDP%E5%8D%8F%E8%AE%AE%E7%9A%84%E5%9F%BA%E6%9C%AC%E7%89%B9%E7%82%B9">
     5、UDP协议的基本特点
    </h2>
    <ol>
     <li>
      <strong>
       传输层协议
      </strong>
     </li>
     <li>
      <strong>
       无连接。指
      </strong>
      通信前，双方不需要建立连接，如直播和短视频等。
     </li>
     <li>
      <strong>
       不可靠传输
      </strong>
      。UDP协议不会保证数据发送的完整性，如果在传输过程中数据出现丢失或者损坏，UDP也不会管，更不会重新发送。例如直播的时候 出现的卡顿模糊等。
     </li>
     <li>
      <strong>
       面向数据包
      </strong>
      。指在传输数据的时候，是将数据视为独立的、不可分割的数据包或数据报进行传输。
     </li>
    </ol>
    <h2 id="6%E3%80%81%E7%BD%91%E7%BB%9C%E5%AD%97%E8%8A%82%E5%BA%8F" name="6%E3%80%81%E7%BD%91%E7%BB%9C%E5%AD%97%E8%8A%82%E5%BA%8F">
     6、网络字节序
    </h2>
    <p>
     内存中的多字节数据相对于内存地址都有着大端和小端之分，磁盘文件中的多字节数据相对于文件中的偏移地址也有大端和小端之分。
    </p>
    <p>
     先来回忆一下什么是大端和小端，我们以内存中的存储数据为例，长字节数据在内存中存储时，不同的平台可能会有着不同的存储方式。
    </p>
    <ol>
     <li>
      <span style="color:#511b78">
       <strong>
        大端字节序
       </strong>
      </span>
      ：数据的高位字节存储在内存的低地址处，低位字节存储在内存的高地址处。
     </li>
     <li>
      <strong>
       <span style="color:#511b78">
        小端字节序
       </span>
      </strong>
      ：数据的低位字节存储在内存的低地址处，高位字节存储在内存的高地址处。这里需要注意的是该存储方式并不是将数据倒序存储，而是以字节为单位，从低位数据到高位数据存储到内存的低地址处到高地址处。
     </li>
    </ol>
    <p>
     <img alt="" height="741" src="https://i-blog.csdnimg.cn/direct/602e9963b5ab48ab95aeb8520a3d8dd8.png" width="1898"/>
    </p>
    <p>
     所以数据的存储方式不同就导致我们在传输数据时读取数据的方式也不同。
    </p>
    <p>
     使用大端字节序的平台，读取数据的时候需要从内存的低地址处开始读到高地址处。
    </p>
    <p>
     而小端字节序的平台，读取数据的时候需要从内存的高地址处开始读到低地址处。
    </p>
    <blockquote>
     <p>
      <span style="color:#511b78">
       <strong>
        需要注意的是 ，CPU在读取数据的时候默认是从有效低地址开始的，所以小端字节序的平台默认读取到的数据顺序是错误的。
       </strong>
      </span>
     </p>
    </blockquote>
    <p>
     所以我们在网络通信中
     <strong>
      网络字节流的地址
     </strong>
     就需要有规定
    </p>
    <ul>
     <li>
      发送主机通常将发送缓冲区的数据按内存地址从低到高的顺序发出。
     </li>
     <li>
      接受主机把从网络中接收到的数据也是按照从低到高的顺序保存在接收缓冲区中。
     </li>
    </ul>
    <p>
     <strong>
      因此我们规定网络字节流的地址应该是 先发出的是低地址，后发出的是高地址。并且
      <span style="color:#fe2c24">
       TCP/IP协议规定，网络数据流应该采用大端字节序，即低地址处是高位字节
      </span>
      。
     </strong>
     不论当前的主机是大端机还是小端机都要按照TCP/IP规定的这个网络字节序来发送/接收数据。如果当前主机是小端机，那么就需要先将数据转成大端，否则就忽略。因此，
     <span style="color:#fe2c24">
      <strong>
       大端字节序也称为 网络字节序
      </strong>
     </span>
     。
    </p>
    <p>
     经过对上面的对网络字节流的规定，我们现在知道网络中的所有数据都是大端字节序，那么在接收端读取这些数据的时候就按照大端字节序的方式读取，或者对于小端机器先将接受到的数据的存储顺序调整成小端字节序再读取。
    </p>
    <blockquote>
     <p>
      系统中提供了 主机字节序（
      <strong>
       Host Byte Order
      </strong>
      ） 和 网络字节序（
      <strong>
       Network Byte Order
      </strong>
      ） 的转换接口如下:
     </p>
     <p>
      <img alt="" height="736" src="https://i-blog.csdnimg.cn/direct/2a3853b8d1204a768742a24e6129534e.png" width="1660"/>
     </p>
     <ul>
      <li>
       <span style="color:#fe2c24">
        <strong>
         <code>
          htonl()
         </code>
        </strong>
       </span>
       ：将32位整数从主机字节序转换为网络字节序（"host to network long"）。
      </li>
      <li>
       <span style="color:#fe2c24">
        <strong>
         <code>
          htons()
         </code>
        </strong>
       </span>
       ：将16位整数从主机字节序转换为网络字节序（"host to network short"）。
      </li>
      <li>
       <span style="color:#fe2c24">
        <strong>
         <code>
          ntohl()
         </code>
        </strong>
       </span>
       ：将32位整数从网络字节序转换为主机字节序（"network to host long"）。
      </li>
      <li>
       <strong>
        <span style="color:#fe2c24">
         <code>
          ntohs()
         </code>
        </span>
       </strong>
       ：将16位整数从网络字节序转换为主机字节序（"network to host short"）。
      </li>
     </ul>
     <p>
      转换之后会通过返回值返回，如果本机就是大端字节序存储，则这些接口就不会发生转换。
     </p>
    </blockquote>
    <h2 id="7%E3%80%81socket%E7%BC%96%E7%A8%8B%E6%8E%A5%E5%8F%A3%C2%A0" name="7%E3%80%81socket%E7%BC%96%E7%A8%8B%E6%8E%A5%E5%8F%A3%C2%A0">
     7、socket编程接口
    </h2>
    <p>
     先见一见一些常见的
     <strong>
      API（应用程序编程接口，Application Programming Interface）
     </strong>
     接口
    </p>
    <pre><code class="language-cpp">// 创建 socket 文件描述符 (TCP/UDP, 客户端 + 服务器)
int socket(int domain, int type, int protocol);

// 绑定端口号 (TCP/UDP, 服务器)
int bind(int socket, const struct sockaddr* address, socklen_t address_len);

// 开始监听socket (TCP, 服务器)
int listen(int socket, int backlog);

// 接收请求 (TCP, 服务器)
int accept(int socket, struct sockaddr* address, socklen_t* address_len);

// 建立连接 (TCP, 客户端)
int connect(int sockfd, const struct sockaddr* addr, socklen_t addrlen);

// 发送报文 (UDP)
ssize_t sendto(int sockfd, const void *buf, size_t len, int flags, const struct sockaddr* dest_addr, socklen_t addrlen);

// 接收报文 (UDP)
ssize_t recvfrom(int socket, void* restrict buffer, size_t length, int flags, struct sockaddr* restrict address, socklen_t* restrict address_len);
</code></pre>
    <p>
     我们可以看到大部分的接口的参数都包含一个参数类型：
     <span style="color:#fe2c24">
      <strong>
       struct sockaddr
      </strong>
     </span>
    </p>
    <hr/>
    <h3 id="struct%20sockaddr%C2%A0" name="struct%20sockaddr%C2%A0" style="background-color:transparent">
     <span style="color:null">
      <strong>
       struct sockaddr
      </strong>
     </span>
    </h3>
    <p>
     <strong>
      <span style="background-color:#a2e043">
       sockaddr
      </span>
     </strong>
     是一个结构体。套接字通信在设计的时候不仅实现了网络间通信还实现了本机内进程的通信。所以套接字我们通常分为两类：
     <span style="color:#fe2c24">
      <strong>
       网络套接字
      </strong>
     </span>
     和
     <span style="color:#fe2c24">
      <strong>
       域间套接字，
      </strong>
     </span>
     分别用于网络通信和本地（域间）通信。
    </p>
    <p>
     域间套接字通信可以实现本地进程的通信，与我们之前介绍的管道通信、共享内存通信的功能大致相同。
     <strong>
      域间通信也可以称为双向管道
     </strong>
     。其使用起来要比网络套接字简单，因为不需要跨主机进行通信，所以没有IP，也没有端口号的概念。在使用时，只需要提供一个文件的路径，与命名管道听起来是一样的，但是操作是有着区别的。
    </p>
    <blockquote>
     <p>
      思考：套接字的设计分成了两类，那么使用套接字实现进程通信也分别实现了网络通信和本机通信两种吗？
     </p>
     <p>
      <strong>
       实际上不是的，要是设计成两种就太过于麻烦了。既然都是套接字，那么只需要将两种的通信接口统一起来就好了。那么该如何做呢？两种通信所需要的资源都是不一样的
      </strong>
      。
     </p>
     <ul>
      <li>
       网络间的通信需要的资源是
       <span style="color:#fe2c24">
        <strong>
         IP地址、端口号
        </strong>
       </span>
       等资源，所以设计了
       <span style="color:null">
        <strong>
         <span style="background-color:#38d8f0">
          struct sockaddr_in
         </span>
        </strong>
       </span>
       等结构来描述网络通信所需要的资源。
      </li>
      <li>
       本机间进程的通信需要的资源是
       <span style="color:#fe2c24">
        <strong>
         路径名
        </strong>
       </span>
       等资源，所以设计了
       <strong>
        <span style="background-color:#38d8f0">
         struct sockaddr_un
        </span>
       </strong>
       结构体来描述域间通信所需要的资源。
      </li>
     </ul>
     <p>
      <img alt="" height="1213" src="https://i-blog.csdnimg.cn/direct/72b44c79dcd14f3aa8b0bcb8151be7ce.png" width="1554"/>
     </p>
     <p>
      <strong>
       struct sockaddr_in
      </strong>
      的
      <strong>
       前16位是一个宏
      </strong>
      ，
      <span style="color:#fe2c24">
       <strong>
        AF_INET
       </strong>
      </span>
     </p>
     <p>
      <strong>
       struct sockaddr_un
      </strong>
      的
      <strong>
       前16位也是一个宏
      </strong>
      ，
      <span style="color:#fe2c24">
       <strong>
        AF_UNIX
       </strong>
      </span>
     </p>
    </blockquote>
    <p>
     既然我们统一了接口，但是不同的通信所需要的资源是不同的，所以同一个接口怎么接收不同类型的资源呢？
    </p>
    <p>
     接口既然要接收不同类型的数据，那么就不能将接口的参数设置成 上面的具体的描述的资源的结构体。而且在上面接口参数中，我们也没有看到
     <span style="color:null">
      <span style="background-color:#38d8f0">
      </span>
      <strong>
       <span style="background-color:#38d8f0">
        struct sockaddr_in
       </span>
      </strong>
      <span style="background-color:#38d8f0">
      </span>
     </span>
     和
     <strong>
      <span style="background-color:#38d8f0">
       struct sockaddr_un
      </span>
     </strong>
     类型的，只有一个
     <strong>
      <span style="background-color:#38d8f0">
       struct sockaddr
      </span>
     </strong>
     。下面我们研究一下这个结构体是什么。
    </p>
    <p>
     <img alt="" height="1215" src="https://i-blog.csdnimg.cn/direct/9b8ca333f5b74450824cd09296742207.png" width="1631"/>
    </p>
    <blockquote>
     <p>
      我们将他们三个放在一起对比着看，可以发现他们有着共同点，这三个结构体的首16位都是 地址类型。
     </p>
     <p>
      <strong>
       地址类型，不同的宏可以区分协议 以及 区分网络通信还是域间通信。
      </strong>
     </p>
     <p>
      IPv4和IPv6是互联网协议（IP）的两个主要版本，它们定义了如何在网络上定位设备并进行数据包的路由。IPv4使用32位地址,IPv6使用128位地址.
     </p>
     <p>
      AF_INET 和AF_INET6 都是用来指定IP地址类型的两个常量，AF_INET 用于 IPv4  AF_INET6 用于 IPv6
     </p>
     <p>
      如果是域间通信则地址类型为AF_UNIX.
     </p>
    </blockquote>
    <p>
     再看看 sockaddr_in 结构体中都是什么
    </p>
    <p>
     <img alt="" height="332" src="https://i-blog.csdnimg.cn/direct/a61f48ae240c4bbf9e57ab33afd8119e.png" width="624"/>
    </p>
    <p>
     所以
     <strong>
      <span style="background-color:#38d8f0">
       struct sockaddr
      </span>
      <span style="color:#fe2c24">
       实际上是设计出来的一个抽象的中间的结构体，就是为了让接口能接收不同类型的数据资源
      </span>
     </strong>
     。
    </p>
    <h3 id="%E4%BD%BF%E7%94%A8" name="%E4%BD%BF%E7%94%A8" style="background-color:transparent">
     使用
    </h3>
    <ol>
     <li>
      <strong>
       在使用该接口的时候，需要先将
       <span style="background-color:#38d8f0">
        struct sockaddr_in*
       </span>
       或者
       <span style="color:null">
        <span style="background-color:#38d8f0">
         struct sockaddr_un*
        </span>
       </span>
       等类型的结构体先强转成
       <span style="background-color:#38d8f0">
        struct sockaddr*
       </span>
       然后再传给接口使用。这是因为上面所说的他们的首16位都表示地址类型。
      </strong>
     </li>
     <li>
      <strong>
       接口接收到数据之后，会根据 前16 的地址类型来区分协议及通信方式，也会根据地址类型判断出数据的原结构类型，然后将
       <span style="background-color:#38d8f0">
        sockaddr
       </span>
       结构体强转回原结构体类型以获取完成的信息。
      </strong>
     </li>
    </ol>
    <h2 id="%E7%AE%80%E5%8D%95%E7%9A%84UDP%E6%8E%A5%E5%8F%A3" name="%E7%AE%80%E5%8D%95%E7%9A%84UDP%E6%8E%A5%E5%8F%A3" style="background-color:transparent">
     简单的UDP接口
    </h2>
    <p>
     学习了上面的知识，我们在演示简单的UDP通信接口前再来介绍一个接口：
     <strong>
      <span style="background-color:#38d8f0">
       int socket()
      </span>
     </strong>
    </p>
    <p>
     <img alt="" height="253" src="https://i-blog.csdnimg.cn/direct/b95aa6e593314c48812fe2c7cc447e60.png" width="843"/>
    </p>
    <p>
     该接口的作用是
     <span style="color:#fe2c24">
      <strong>
       创建一个 通信的端点
      </strong>
     </span>
     。
    </p>
    <p>
     成功则返回
     <span style="color:#fe2c24">
      <strong>
       一个新的socket文件描述符（一个整数）
      </strong>
     </span>
     ，这个描述符用于
     <strong>
      <span style="color:#fe2c24">
       标识网络连接
      </span>
     </strong>
     。
     <strong>
      实际上在操作系统中，套接字操作都是通过文件描述符来实现的
     </strong>
     。
    </p>
    <p>
     失败返回 -1并设置全局变量 errno以指示错误类型。
    </p>
    <p>
     下来看看三个参数：
    </p>
    <p>
     <strong>
      1、int domain
     </strong>
    </p>
    <p>
     该参数用于指定一个通信域，它选择将用于通信的协议族。这些家族定义在
     <span style="color:#fe2c24">
      <strong>
       <code>
        &lt;sys/socket.h&gt;
       </code>
      </strong>
     </span>
     头文件中。目前理解的格式包括如下：
    </p>
    <p>
     <img alt="" height="286" src="https://i-blog.csdnimg.cn/direct/3bb9be828e964880a6451d754ed6789a.png" width="1813"/>
    </p>
    <ul>
     <li>
      <span style="color:#fe2c24">
       AF_UNIX（或 AF_LOCAL）
      </span>
      ：本地通信协议族，用于同一台机器上的进程间通信（IPC）。使用文件系统的路径名作为地址。
     </li>
     <li>
      <span style="color:#fe2c24">
       AF_INET
      </span>
      <span style="color:null">
       ：IPv4
      </span>
      网络协议族，用于通过互联网协议版本4进行通信。这是最常用的类型之一，适用于广域网和局域网中的通信。
     </li>
     <li>
      <span style="color:#fe2c24">
       AF_INET6
      </span>
      ：IPv6 网络协议族，用于通过互联网协议版本6进行通信。随着IPv4地址空间的耗尽，越来越多的应用开始支持IPv6。
     </li>
     <li>
      <span style="color:#fe2c24">
       AF_IPX
      </span>
      ：Novell IPX 协议族，用于NetWare环境下的网络通信。不过，由于IP逐渐成为主流，这种协议族现在较少见。
     </li>
     <li>
      <span style="color:#fe2c24">
       AF_PACKET
      </span>
      ：直接访问低层网络协议。这个选项通常用于需要直接处理数据链路层协议的应用程序，比如网络嗅探器。
     </li>
    </ul>
    <p>
     <strong>
      2、int type
     </strong>
    </p>
    <p>
     在创建socket时，type 参数指定 socket 的类型，它决定了通信的语义。当前定义的主要类型包括
    </p>
    <p>
     <img alt="" height="282" src="https://i-blog.csdnimg.cn/direct/ff09430571c54e0386a242011732feb3.png" width="1797"/>
    </p>
    <ul>
     <li>
      <span style="color:#fe2c24">
       SOCK_STREAM
      </span>
      ：提供一个有序、可靠、基于连接的字节流服务。数据被视为无边界的数据流，通常使用TCP协议。这意味着你可以发送和接收任意大小的数据流，并且这些数据将按照发送顺序到达另一端。
     </li>
     <li>
      <span style="color:#fe2c24">
       SOCK_DGRAM
      </span>
      ：支持无连接的数据报服务，每个数据报都是独立的，具有固定的最大长度，并且不保证送达顺序或可靠性。通常使用UDP协议。这种类型的socket适合于那些对实时性要求较高但对可靠性要求不那么严格的场景。
     </li>
     <li>
      <span style="color:#fe2c24">
       SOCK_RAW
      </span>
      ：允许直接访问底层网络协议。这类sockets主要用于高级用户、开发者或需要直接处理IP数据包的应用程序。它们可以用来构建自定义的协议或者进行网络研究等。即
      <p>
       <span style="color:#1c7331">
        原始套接字. 使用此套接字, 通信可以直接绕过传输层的协议, 直接访问IP协议.不过, 绕过传输层协议, 就表示需要自己实现一些传输协议的内容. 一般用于网络诊断等方面
       </span>
      </p>
     </li>
     <li>
      <span style="color:#fe2c24">
       SOCK_SEQPACKET
      </span>
      ：提供了一个面向数据报的接口，但是数据报是有序的并且可靠的，类似于SOCK_STREAM，但是数据边界被保留（不像
      <span style="color:#fe2c24">
       SOCK_STREAM
      </span>
      那样是一个无结构的字节流）。并不是所有协议族都支持这种类型。
     </li>
     <li>
      <span style="color:#fe2c24">
       SOCK_RDM
      </span>
      ：一种可靠的分组交付服务，保证消息按顺序到达且不会重复或丢失，但并不保证消息的边界。这个类型不如其他几种常见，支持也较为有限。
     </li>
    </ul>
    <p>
     <strong>
      3、int protocol
     </strong>
    </p>
    <p>
     该参数用来选择协议类型，此参数的选择与第二个参数密切相关。
    </p>
    <p>
     如 type 选择了
     <span style="color:#fe2c24">
      SOCK_STREAM
     </span>
     <span style="color:null">
      此参数就需要传入
     </span>
     <span style="color:#fe2c24">
      IPPROTO_TCP
     </span>
     <span style="color:null">
      ,即选择了TCP协议。
     </span>
    </p>
    <p>
     <span style="color:null">
      但是实际上我们不用手动使用宏去选择协议，网络通信时，选定type并且只需要使用一种协议时， 该参数可以直接传入 0 ，表示使用默认协议，其实就是操作系统根据前面的参数自动选择最合适的协议。
     </span>
    </p>
