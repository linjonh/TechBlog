    <h2>
     前言
    </h2>
    <p>
     上篇文章中我们对udp网络通信有了初步的认识，我们在之前已经知道了UDP通信和TCP通信的不同
    </p>
    <ul>
     <li>
      UDP非连接，面向数据包。
     </li>
     <li>
      TCP连接，面向字节流。
     </li>
    </ul>
    <p>
     这篇文章我们来介绍和演示一下TCP套接字的接口。
    </p>
    <h2 id="TCP%E5%A5%97%E6%8E%A5%E5%AD%97%E6%8E%A5%E5%8F%A3" name="TCP%E5%A5%97%E6%8E%A5%E5%AD%97%E6%8E%A5%E5%8F%A3">
     TCP套接字接口
    </h2>
    <pre><code class="language-cpp">// 创建 socket 文件描述符 (TCP/UDP, 客户端 + 服务器)
int socket(int domain, int type, int protocol);

// 绑定端口号 (TCP/UDP, 服务器)
int bind(int socket, const struct sockaddr* address, socklen_t address_len);

// 开始监听socket (TCP, 服务器)
int listen(int socket, int backlog);

// 接收请求 (TCP, 服务器)
int accept(int socket, struct sockaddr* address, socklen_t* address_len);

// 建立连接 (TCP, 客户端)
int connect(int sockfd, const struct sockaddr* addr, socklen_t addrlen);

// 发送报文 (UDP)
ssize_t sendto(int sockfd, const void *buf, size_t len, int flags, const struct sockaddr* dest_addr, socklen_t addrlen);

// 接收报文 (UDP)
ssize_t recvfrom(int socket, void* restrict buffer, size_t length, int flags, struct sockaddr* restrict address, socklen_t* restrict address_len);
</code></pre>
    <p>
     上篇文中我们已经演示过了里面的一些接口，剩下的三个用于连接的接口就在下面演示一遍。
    </p>

    <h3 id="TCP%E6%9C%8D%E5%8A%A1%E7%AB%AF%E5%AE%9E%E7%8E%B0" name="TCP%E6%9C%8D%E5%8A%A1%E7%AB%AF%E5%AE%9E%E7%8E%B0">
     TCP服务端实现
    </h3>
    <p>
     我们还是由浅入深，实现一个简单点的TCP服务器，服务器包含一个对客户端发送来的消息进行小写转大写的功能。要实现TCP网络通信服务端都需要哪些步骤，与UDP的区别具体是什么呢？
    </p>
    <ol>
     <li>
      <strong>
       创建套接字
      </strong>
      ：使用
      <code>
       <span style="background-color:#cccccc">
        socket()
       </span>
      </code>
      函数创建一个新的套接字。
     </li>
     <li>
      <strong>
       绑定地址和端口
      </strong>
      ：通过
      <span style="background-color:#cccccc">
      </span>
      <code>
       <span style="background-color:#cccccc">
        bind()
       </span>
      </code>
      <span style="background-color:#d7d8d9">
      </span>
      将该套接字绑定到一个特定的IP地址和端口号上。
     </li>
     <li>
      <strong>
       监听连接请求
      </strong>
      ：调用
      <span style="color:null">
       <code>
        <span style="background-color:#cccccc">
         listen()
        </span>
       </code>
       <span style="background-color:#cccccc">
       </span>
      </span>
      函数让套接字进入监听状态，准备接受连接请求。
     </li>
     <li>
      <strong>
       接受连接
      </strong>
      ：使用
      <code>
       <span style="background-color:#cccccc">
        accept()
       </span>
      </code>
      函数接受来自客户端的连接请求。这将返回一个新的套接字描述符，用于与特定客户端进行通信。
     </li>
     <li>
      <strong>
       数据交换
      </strong>
      ：使用
      <span style="background-color:#cccccc">
      </span>
      <code>
       <span style="background-color:#cccccc">
        send()
       </span>
      </code>
      和
      <span style="background-color:#cccccc">
      </span>
      <code>
       <span style="background-color:#cccccc">
        recv()
       </span>
      </code>
      （或
      <span style="background-color:#cccccc">
      </span>
      <code>
       <span style="background-color:#cccccc">
        write()
       </span>
      </code>
      和
      <span style="background-color:#cccccc">
      </span>
      <code>
       <span style="background-color:#cccccc">
        read()
       </span>
      </code>
      ）函数来发送和接收数据。
     </li>
     <li>
      <strong>
       关闭连接
      </strong>
      ：通信完成后，使用
      <code>
       <span style="background-color:#cccccc">
        close()
       </span>
      </code>
      关闭连接。
     </li>
    </ol>
    <p>
     可以看到TCP通信服务端与UDP通信服务端的主要差别在于监听连接请求、接收连接这两步。
    </p>
    <p>
     先来分别介绍一下这几个新的接口
    </p>
    <h4 id="1%E3%80%81listen()" name="1%E3%80%81listen()">
     1、
     <strong>
      <span style="background-color:#cccccc">
       listen()
      </span>
     </strong>
    </h4>
    <blockquote>
     <pre><code class="language-cpp">int listen(int sockfd, int backlog);</code></pre>
     <p>
      <span style="color:#fe2c24">
       <strong>
        与UDP不同，TCP网络通信是双方要以建立连接为基础的
       </strong>
      </span>
      <span style="color:#511b78">
       <strong>
        。在TCP通信中，
        <span style="background-color:#cccccc">
         listen
        </span>
        是服务器端用于开始监听传入连接请求的一个重要函数。它是网络编程中实现服务端逻辑的关键步骤之一。
       </strong>
      </span>
     </p>
     <hr/>
     <p>
      接口参数：
     </p>
     <ul>
      <li>
       <strong>
        <span style="background-color:#cccccc">
         sockfd
        </span>
       </strong>
       : 这是一个整数类型的文件描述符，指向之前通过调用 socket() 创建的套接字。这个套接字需要绑定到一个本地地址（使用 bind() 函数）。
      </li>
      <li>
       <strong>
        <span style="background-color:#cccccc">
         backlog
        </span>
       </strong>
       : 它指定了待处理连接请求队列的最大长度。也就是说，它决定了系统可以为该套接字排队等待接受的最大未处理连接数。如果连接请求数超过了这个值，新的请求可能会被拒绝。（类似餐馆就像是服务器，而座位和等候区加起来就像是 backlog 参数。）
      </li>
     </ul>
     <p>
      返回值：
     </p>
     <ul>
      <li>
       成功时，listen() 返回 0。
      </li>
      <li>
       错误时，返回 -1，并设置全局变量 errno 来指示错误类型。
      </li>
     </ul>
     <hr/>
     <p>
      <strong>
       <span style="background-color:null">
        执行
       </span>
       <code>
        <span style="background-color:#cccccc">
         listen()
        </span>
       </code>
       <span style="background-color:null">
        , 服务器会自动进入监听状态. 之后
       </span>
       <span style="color:#fe2c24">
        <span style="background-color:null">
         会一直监听 来自客户端 向服务器 发送的连接请求
        </span>
       </span>
       <span style="background-color:null">
        .
       </span>
       <span style="color:#fe2c24">
        <span style="background-color:null">
         实际上监听的就是服务器的套接字
        </span>
       </span>
       <span style="background-color:null">
        . 监听的过程是
       </span>
       <span style="color:#fe2c24">
        <span style="background-color:null">
         非阻塞的
        </span>
       </span>
       <span style="background-color:null">
        .
       </span>
      </strong>
     </p>
    </blockquote>
    <h4 id="2%E3%80%81%20accept()%C2%A0" name="2%E3%80%81%20accept()%C2%A0">
     2、
     <strong>
      <span style="background-color:#cccccc">
       accept()
      </span>
     </strong>
    </h4>
    <blockquote>
     <pre><code class="language-cpp">int accept(int sockfd, struct sockaddr *addr, socklen_t *addrlen);</code></pre>
     <p>
      TCP是
      <strong>
       面向连接
      </strong>
      的. 其他客户端想要将信息发送到服务器, 就必须先于服务器建立连接.
     </p>
     <p>
      而能否与服务器建立连接 是服务器说了算的. 只有
      <strong>
       服务器接受了来自客户端的连接请求 才是连接成功
      </strong>
      了。
     </p>
     <hr/>
     <p>
      接口参数：
     </p>
     <ul>
      <li>
       <strong>
        <span style="background-color:#cccccc">
         sockfd
        </span>
       </strong>
       : 这是一个整数类型的文件描述符，指向之前通过调用
       <span style="background-color:#cccccc">
        socket()
       </span>
       创建并且已经通过
       <span style="background-color:#cccccc">
        bind()
       </span>
       和
       <span style="color:#0d0016">
        <span style="background-color:#cccccc">
         listen()
        </span>
       </span>
       设置为监听状态的套接字。
      </li>
      <li>
       <strong>
        <span style="background-color:#cccccc">
         addr
        </span>
       </strong>
       : 这是一个指向
       <span style="background-color:#cccccc">
        struct sockaddr
       </span>
       类型对象的指针，用于存储发起连接的客户端地址信息（如果不需要获取客户端地址信息，可以传递 NULL）。
      </li>
      <li>
       <strong>
        <span style="background-color:#cccccc">
         addrlen
        </span>
       </strong>
       : 这是一个指向
       <span style="background-color:#cccccc">
        socklen_t
       </span>
       类型变量的指针，指示
       <span style="background-color:#cccccc">
        addr
       </span>
       参数指向的对象的大小。在调用
       <span style="background-color:#cccccc">
        accept
       </span>
       函数前，应设置为该结构体的大小；函数返回时，会更新为实际写入的地址长度。
      </li>
     </ul>
     <p>
      返回值：
     </p>
     <ul>
      <li>
       成功时，
       <strong>
        <span style="background-color:#cccccc">
         accept()
        </span>
       </strong>
       返回一个新的套接字描述符，用于和客户端进行通信。
      </li>
      <li>
       错误时，返回 -1，并设置全局变量
       <code>
        errno
       </code>
       来指示错误类型。
      </li>
     </ul>
     <hr/>
     <p>
      <strong>
       <span style="color:#511b78">
        <span style="background-color:#cccccc">
         accept
        </span>
        是在服务器端用于接受来自客户端的连接请求的一个重要函数。它通常与
        <span style="background-color:#cccccc">
        </span>
        <code>
         <span style="background-color:#cccccc">
          listen
         </span>
        </code>
        <span style="background-color:#cccccc">
        </span>
        函数配合使用，用来从已完成连接队列中提取第一个连接请求，创建一个新的套接字，并返回引用这个套接字的文件描述符，这个新的套接字不会被监听，服务器原来的套接字，不会被此次调用影响。
       </span>
      </strong>
     </p>
     <p>
     </p>
     <p>
      <span style="color:#511b78">
       也就是说，这个TCP服务器会创建很多个套接字，而UDP只会创建一个属于服务器的套接字。这个新创建的套接字和原本的套接字有什么关系呢？
      </span>
     </p>
     <p>
      <span style="color:#511b78">
       <strong>
        服务器原本的套接字已经被listen接口作用，处于监听状态，是为了能够接收到连接信息。
       </strong>
      </span>
      <span style="color:#511b78">
       <strong>
        当accept接受了客户端的连接请求时，会创建一个存储有 服务器本地网络信息 以及 客户端网络信息的新的套接字，服务器可以通过此套接字与客户端进行通信。
       </strong>
      </span>
     </p>
     <p>
     </p>
     <p>
      <span style="color:#511b78">
       <strong>
        所以原来的服务器创建的套接字 只用来与客户端建立连接，所以被称为
       </strong>
      </span>
      <span style="color:#fe2c24">
       <strong>
        监听套接字
       </strong>
      </span>
      <span style="color:#511b78">
       <strong>
        。
       </strong>
      </span>
     </p>
     <p>
     </p>
     <p>
      <span style="color:#511b78">
       <strong>
        创建的新的套接字
       </strong>
      </span>
      <span style="color:#511b78">
       <strong>
        存储有
        <span style="background-color:#cccccc">
         服务器本地网络信息
        </span>
        以及
        <span style="background-color:#cccccc">
         客户端网络信息
        </span>
        ，并且 此套接字也会被绑定到系统内核中，并于服务器原套接字相独立。此套接字主要是
       </strong>
      </span>
      <span style="color:#fe2c24">
       <strong>
        用来给客户端提供服务的套接字
       </strong>
      </span>
      <span style="color:#511b78">
       <strong>
        。
       </strong>
      </span>
     </p>
    </blockquote>
    <h4 id="%E5%AE%8C%E6%95%B4%E4%BB%A3%E7%A0%81%C2%A0" name="%E5%AE%8C%E6%95%B4%E4%BB%A3%E7%A0%81%C2%A0">
     完整代码
    </h4>
    <pre><code class="language-cpp">//tcpServer.cc
#include"tcp.hpp"

using std::string;
using std::cerr;
using std::endl;
class tcpServer
{
    public:
        tcpServer(uint16_t port,const string&amp; ip="")//按引用传递提高效率
            :_port(port)
            ,_ip(ip){}
        ~tcpServer()
        {}
        void Init()
        {//面向字节流参数选择SOCK_STREAM
            _LiSockFd=socket(AF_INET,SOCK_STREAM,0);//创建套接字（先创建一个listen的套接字）
            if(_LiSockFd&lt;0)
            {
                logMessage(FATAL,"socket() failed::%s : %d",strerror(errno),_LiSockFd);
                exit(SOCKET_ERR);
            }
            logMessage(DEBUG,"socket() success::%d",_LiSockFd);
            struct sockaddr_in local;
            memset(&amp;local,0,sizeof(local));//初始化结构体
            local.sin_family=AF_INET;//填充
            local.sin_port=htons(_port);
            _ip.empty()?(local.sin_addr.s_addr=htonl(INADDR_ANY)):(inet_aton(_ip.c_str(),&amp;local.sin_addr));
            
            if(bind(_LiSockFd,(const struct sockaddr*)&amp;local,sizeof(local))==-1)
            {
                logMessage(FATAL,"socket() failed::%s : %d",strerror(errno),_LiSockFd);
                exit(BIND_ERR);
            }
            logMessage(DEBUG,"bind() success::%d",_LiSockFd);

            if(listen(_LiSockFd,5)==-1)//执行listen(), 服务器会自动进入监听状态. 之后会一直监听 来自客户端 向服务器 发送的连接请求. 实际上监听的就是服务器的套接字. 监听的过程是非阻塞的.
            {
                logMessage(FATAL,"listen() failed::%s : %d",strerror(errno),_LiSockFd);
                exit(LISTEN_ERR);
            }
            logMessage(DEBUG,"listen() success::%d",_LiSockFd);
        }
        void to_upper(int sock,const string&amp; clientIP,const uint16_t &amp;clientPort)//小写转大写功能函数
        {
            assert(sock&gt;0);
            assert(!clientIP.empty());
            
            char inbuffer[BUFFER_SIZE];//用于存储来自客户端信息的数组
            while(true)
            {
                // TCP获取来自客户端的信息的操作就是 read
                // 从 服务器与客户端连接 的文件描述符中 读取来自客户端的信息
                // 可看作 通过文件描述符 从文件读取内容
                ssize_t s=read(sock,inbuffer,sizeof(inbuffer)-1);
                if(s&gt;0)
                {
                    inbuffer[s]='\0';
                     // 我们实现一个操作, 如果 客户端传输过来的信息是 quit 这个单词, 就表示客户端请求退出
                    // 就可以退出 服务循环了
                    if(strcasecmp(inbuffer,"quit")==0)
                    {
                        logMessage(DEBUG,"Client requests to quit: [%s: %d]",clientIP.c_str(),clientPort);
                        break;
                    }
                    logMessage(DEBUG,"to_upper before:[%s:%d]&gt;&gt; %s",clientIP.c_str(),clientPort,inbuffer);
                    for(int i=0;i&lt;s;i++)
                    {
                        if(isalpha(inbuffer[i])&amp;&amp;islower(inbuffer[i]))
                            inbuffer[i]=toupper(inbuffer[i]);
                    }
                    logMessage(DEBUG,"to_upper after:[%s:%d]&gt;&gt;%s",clientIP.c_str(),clientPort,inbuffer);
                    // 上面做的都是对获取到的信息 进行转换
                    // 最后需要做的就是 将转换后的信息 再重新回应给客户端
                    // 而 回应给客户端 则是用 write, 可看做 通过文件描述符像文件写入内容
                    write(sock,inbuffer,strlen(inbuffer));
                }
                else if(s==0)
                {
                    logMessage(DEBUG,"Client has quited:[%s:%d],clientIP,clientPort");
                    break;
                }
                else{
                    // 到这里 本次 read() 出错
                    logMessage(DEBUG,"Client [%s:%d] read::%s",clientIP.c_str(),clientPort,strerror(errno));
                    break; 
                }
            }
            // 走到这里 循环已经退出了, 表示 client 也已经退出了
            // 所以 此时需要关闭文件描述符, 因为一个主机上的文件描述符数量是一定的, 达到上限之后 就无法再创建
            // 已经无用但没有被归还的文件描述符, 文件描述符泄漏
            close(sock);
            logMessage(DEBUG,"Service close %d sockFd",sock);
        }
        void start()
        {
            while(true)
            {
                struct sockaddr_in peer;
                socklen_t peerlen=sizeof(peer);
                
                int serviceSockFd=accept(_LiSockFd,(struct sockaddr*)&amp;peer,&amp;peerlen);
                if(serviceSockFd==-1)
                {
                    logMessage(WARINING,"accept() failed::%s : %d",strerror(errno),_LiSockFd);
                    continue;
                }
                string peerIp=inet_ntoa(peer.sin_addr);
                uint16_t peerPort=ntohs(peer.sin_port);
                logMessage(DEBUG,"accept() success:: [%s: %d] | %d",peerIp.c_str(),peerPort,serviceSockFd);
                to_upper(serviceSockFd,peerIp,peerPort);//执行转换功能，小写换大写
            }
        }
    private:
        uint16_t _port;//存储服务器端口号
        string _ip;//存储服务器ip
        int _LiSockFd;//存储服务器进程的监听套接字
};

}

     但是以上的版本是单进程版本，当多客户尝试连接服务器时，会发现服务器只会对第一个连接到客户端进行响应。当客户端连接到服务器的时候，会调用 to_upper() 这个函数，但是它本质上是一个死循环，所以单进程进去之后就出不来了，后面的客户无论怎么连接都是无法响应的， 因为进程的唯一一个执行流在死循环内，这里就需要用到多进程或者多线程来处理这个问题了。
    </p>

     <p>
      需要注意的问题：
     </p>
     <ol>
      <li>
       子进程是继承父进程的文件描述符表的，也就是说父进程已经打开的文件在子进程继承的时候是默认打开的。
      </li>
      <li>
       <strong>
        _LiSockFd
       </strong>
       是监听套接字，对于子进程来说是没用的，所以提前关掉以免子进程的执行影响到父进程的监听。
      </li>
      <li>
       serviceSockFd是服务端与客户端用来通信的新的套接字，使用多进程后变成了子进程实现通信功能，该套接字的文件描述符的关闭操作原本是在 to_upper() 函数中的，现在在子进程中调用此函数，所以子进程内不用再手动关闭了。但是, 父进程已经不再执行
       <strong>
        <span style="background-color:#cccccc">
         to_upper()
        </span>
       </strong>
       , 所以必须要
       <strong>
        <span style="background-color:#cccccc">
         close(serviceSockFd)
        </span>
       </strong>
       . 否则父进程会发生文件描述符泄漏
      </li>
      <li>
       子进程退出时, 是需要父进程回收的.如果父进程使用
       <strong>
        <span style="background-color:#cccccc">
         wait()
        </span>
       </strong>
       进行等待, 默认是阻塞式的 无法使用. 如果使用
       <span style="background-color:#cccccc">
        waitpid()
       </span>

       <code>

       </code>

       <strong>
        <span style="background-color:#cccccc">
         start()
        </span>
       </strong>

       <strong>
        <span style="background-color:#cccccc">
         signal(SIGCHLD,SIG_IGN)
        </span>
       </strong>
       <code>
        <strong>
         <span style="background-color:#cccccc">
          ;
         </span>
        </strong>
       </code>

       <span style="background-color:#cccccc">
        多进程版本二
       </span>
      </span>
     </strong>
    </h3>
    <p>
     这个版本不在忽略子进程的退出信号. 而是利用了进程的一个特性.
    </p>
    <p>
     如果子进程还没有退出, 但是其父进程退出了, 那么此子进程就成了孤儿进程, 会被操作系统领养. 退出时会自动被操作系统回收
    </p>
    <p>
     即, 主父进程创建子进程之后, 子进程又创建了孙子进程. 然后子进程退出, 让孙子进程与客户端通信. 主父进程直接回收退出的子进程. 也不会发生一直阻塞等待的情况.
    </p>
    <pre><code class="language-cpp">#include"tcp.hpp"

using std::string;
using std::cerr;
using std::endl;
class tcpServer
{
    public:
        tcpServer(uint16_t port,const string&amp; ip="")
            :_port(port)
            ,_ip(ip){}
        ~tcpServer()
        {}
        void Init()
        {
            _LiSockFd=socket(AF_INET,SOCK_STREAM,0);
            if(_LiSockFd&lt;0)
            {
                logMessage(FATAL,"socket() failed::%s : %d",strerror(errno),_LiSockFd);
                exit(SOCKET_ERR);
            }
            logMessage(DEBUG,"socket() success::%d",_LiSockFd);
            struct sockaddr_in local;
            memset(&amp;local,0,sizeof(local));
            local.sin_family=AF_INET;
            local.sin_port=htons(_port);
            _ip.empty()?(local.sin_addr.s_addr=htonl(INADDR_ANY)):(inet_aton(_ip.c_str(),&amp;local.sin_addr));
            
            if(bind(_LiSockFd,(const struct sockaddr*)&amp;local,sizeof(local))==-1)
            {
                logMessage(FATAL,"socket() failed::%s : %d",strerror(errno),_LiSockFd);
                exit(BIND_ERR);
            }
            logMessage(DEBUG,"bind() success::%d",_LiSockFd);

            if(listen(_LiSockFd,5)==-1)
            {
                logMessage(FATAL,"listen() failed::%s : %d",strerror(errno),_LiSockFd);
                exit(LISTEN_ERR);
            }
            logMessage(DEBUG,"listen() success::%d",_LiSockFd);
        }
        void to_upper(int sock,const string&amp; clientIP,const uint16_t &amp;clientPort)
        {
            assert(sock&gt;0);
            assert(!clientIP.empty());
            
            char inbuffer[BUFFER_SIZE];
            while(true)
            {
                ssize_t s=read(sock,inbuffer,sizeof(inbuffer)-1);
                if(s&gt;0)
                {
                    inbuffer[s]='\0';
                    if(strcasecmp(inbuffer,"quit")==0)
                    {
                        logMessage(DEBUG,"Client requests to quit: [%s: %d]",clientIP.c_str(),clientPort);
                        break;
                    }
                    logMessage(DEBUG,"to_upper before:[%s:%d]&gt;&gt; %s",clientIP.c_str(),clientPort,inbuffer);
                    for(int i=0;i&lt;s;i++)
                    {
                        if(isalpha(inbuffer[i])&amp;&amp;islower(inbuffer[i]))
                            inbuffer[i]=toupper(inbuffer[i]);
                    }
                    logMessage(DEBUG,"to_upper after:[%s:%d]&gt;&gt;%s",clientIP.c_str(),clientPort,inbuffer);
                    write(sock,inbuffer,strlen(inbuffer));
                }
                else if(s==0)
                {
                    logMessage(DEBUG,"Client has quited:[%s:%d],clientIP,clientPort");
                    break;
                }
                else{
                    logMessage(DEBUG,"Client [%s:%d] read::%s",clientIP.c_str(),clientPort,strerror(errno));
                    break; 
                }
            }
            close(sock);
            logMessage(DEBUG,"Service close %d sockFd",sock);
        }
        void start()
        {
            while(true)
            {
                struct sockaddr_in peer;
                socklen_t peerlen=sizeof(peer);
                
                int serviceSockFd=accept(_LiSockFd,(struct sockaddr*)&amp;peer,&amp;peerlen);
                if(serviceSockFd==-1)
                {
                    logMessage(WARINING,"accept() failed::%s : %d",strerror(errno),_LiSockFd);
                    continue;
                }
                string peerIp=inet_ntoa(peer.sin_addr);
                uint16_t peerPort=ntohs(peer.sin_port);
                logMessage(DEBUG,"accept() success:: [%s: %d] | %d",peerIp.c_str(),peerPort,serviceSockFd);
                pid_t id=fork();
                if(id==-1)
                {
                    logMessage(FATAL,"Server fork() failed:%s",strerror(errno));
                    exit(FORK_ERR);
                }
                else if(id==0)
                {
                    close(_LiSockFd);
                    if(fork()&gt;0)
                    {
                        exit(0);
                    }
                    to_upper(serviceSockFd,peerIp,peerPort);
                    exit(0);
                }
                close(serviceSockFd);
                int ret=waitpid(id,nullptr,peerPort);
                if(ret==-1)
                {
                    logMessage(FATAL,"Server waitpid failed:%s",strerror(errno));
                    exit(WAIT_ERR);
                }
                (void)ret;
            }
        }
    private:
        uint16_t _port;
        string _ip;
        int _LiSockFd;
};

void Usage(string proc)
{
    cerr&lt;&lt;"Usage::\n\t"&lt;&lt;proc&lt;&lt;"port ip"&lt;&lt;endl;
    cerr&lt;&lt;"example::\n\t"&lt;&lt;proc&lt;&lt;"8080 127.0.0.1"&lt;&lt;endl;
}

int main(int argc,char* argv[])
{
    if(argc!=3&amp;&amp;argc!=2)
    {
        Usage(argv[0]);
        exit(USE_ERR);
    }
    uint16_t port=atoi(argv[1]);
    string ip;
    if(argc==3)
    {
        ip=argv[2];
    }
    tcpServer Usvr(port,ip);
    Usvr.Init();
    Usvr.start();
    return 0;
