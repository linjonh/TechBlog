    <h4>
     <a id="_spring_security_166">
     </a>
     配置
     <code>
      spring security
     </code>
    </h4>
    <p>
     示例代码：
    </p>
    <pre><code>package hello;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.annotation.web.configuration.WebSecurityConfigurerAdapter;
import org.springframework.security.core.userdetails.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.provisioning.InMemoryUserDetailsManager;

@Configuration
@EnableWebSecurity
public class WebSecurityConfig extends WebSecurityConfigurerAdapter {
    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/", "/home").permitAll()
                .anyRequest().authenticated()
                .and()
            .formLogin()
                .loginPage("/login")
                .permitAll()
                .and()
            .logout()
                .permitAll();
    }

    @Bean
    @Override
    public UserDetailsService userDetailsService() {
        UserDetails user =
             User.withDefaultPasswordEncoder()
                .username("user")
                .password("password")
                .roles("USER")
                .build();

        return new InMemoryUserDetailsManager(user);
    }
}
</code></pre>
    <p>
     我们在
     <code>
      WebSecurityConfig
     </code>
     类使用了
     <code>
      @EnableWebSecurity
     </code>
     注解 ，该注解提供
     <code>
      spring security
     </code>
     的支持以及
     <code>
      springMvc
     </code>
     的集成支持，配合
     <code>
      @Configuration
     </code>
     注解，即可构成一个
     <code>
      spring security
     </code>
     的配置；
    </p>
    <p>
     其中我们自己写的
     <code>
      WebSecurityConfig
     </code>
     类必须是扩展了
     <code>
      WebSecurityConfigurerAdapter
     </code>
     抽象类的类；我们就是通过它，告诉
     <code>
      spring security
     </code>
     ，哪些用户需要经过身份验证，通过何种方式验证；
    </p>
    <p>
     其中我们选择覆写一些方法，来做一些安全的细节设置，达到上面的目的，比如，配置 ：
     <strong>
      拦截什么URL、设置什么权限，检验表单数据
     </strong>
     等 ；
    </p>
    <p>
     <code>
      userDetailsService()
     </code>
     方法，在内存中添加了一个用户，并设置了其角色、用户名、密码，不多讲；
    </p>

    <h4>
     <a id="configureHttpSecurity__225">
     </a>
     <code>
      configure(HttpSecurity)
     </code>
     方法
    </h4>
    <p>
     其中
     <code>
      configure(HttpSecurity)
     </code>
     方法里面定义了哪些路径需要被保护，那些路径不需要被保护。简单而言，只有
     <code>
      /
     </code>
     、
     <code>
      /home
     </code>
     不需要保护，可以被任何权限、角色查看，其他的路径都需要进行验证；
    </p>
    <p>
     代码中配置：
    </p>
    <pre><code>protected void configure(HttpSecurity http) throws Exception {
    http
        .authorizeRequests()
            .anyRequest().authenticated()
            .and()
        .formLogin()
            .and()
        .httpBasic();
}
</code></pre>
    <p>
     其实上面的配置，就是一个
     <code>
      xml
     </code>
     配置：
    </p>
    <pre><code>&lt;http&gt;
    &lt;intercept-url pattern="/**" access="authenticated"/&gt;
    &lt;form-login /&gt;
    &lt;http-basic /&gt;
&lt;/http&gt;
</code></pre>
    <p>
     每个方法或多或少都对应着一个标签，想象成这样以后，就便于我们看懂代码的配置；
    </p>
    <p>
     配置规则都是使用方法，诸多方法的具体含义如下（图内容来自
     <a href="http://www.spring4all.com" rel="nofollow">
      spring4All 社区
     </a>
     ）：
    </p>
    <p>
     其中有一个特殊的方法
     <code>
      and()
     </code>
     ，类比于
     <code>
      xml
     </code>
     的标签结束符；
    </p>
    <p>
     方法
    </p>
    <p>
     说明
    </p>
    <p>
     openidLogin()
    </p>
    <p>
     用于基于
     <code>
      OpenId
     </code>
     的验证
    </p>
    <p>
     headers()
    </p>
    <p>
     将安全标头添加到响应
    </p>
    <p>
     cors()
    </p>
    <p>
     配置跨域资源共享（
     <code>
      CORS
     </code>
     ）
    </p>
    <p>
     sessionManagement()
    </p>
    <p>
     允许配置会话管理
    </p>
    <p>
     portMapper()
    </p>
    <p>
     允许配置一个
     <code>
      PortMapper(HttpSecurity#(getSharedObject(class)))
     </code>
     ，其他提供
     <code>
      SecurityConfigurer
     </code>
     的对象使用
     <code>
      PortMapper
     </code>
     从
     <code>
      HTTP
     </code>
     重定向到
     <code>
      HTTPS
     </code>
     或者从
     <code>
      HTTPS
     </code>
     重定向到
     <code>
      HTTP
     </code>
     。 默认情况下，
     <code>
      Spring Security
     </code>
     使用一个
     <code>
      PortMapperImpl
     </code>
     映射
     <code>
      HTTP
     </code>
     端口
     <code>
      8080
     </code>
     到
     <code>
      HTTPS
     </code>
     端口
     <code>
      8443
     </code>
     ，
     <code>
      HTTP
     </code>
     端口
     <code>
      80
     </code>
     到
     <code>
      HTTPS
     </code>
     端口
     <code>
      443
     </code>
    </p>
    <p>
     jee()
    </p>
    <p>
     配置基于容器的预认证。 在这种情况下，认证由Servlet容器管理
    </p>
    <p>
     x509()
    </p>
    <p>
     配置基于
     <code>
      x509
     </code>
     的认证
    </p>
    <p>
     rememberMe
    </p>
    <p>
     允许配置
     <code>
      “记住我”
     </code>
     的验证
    </p>
    <p>
     authorizeRequests()
    </p>
    <p>
     允许基于使用
     <code>
      HttpServletRequest
     </code>
     限制访问
    </p>
    <p>
     requestCache()
    </p>
    <p>
     允许配置请求缓存
    </p>
    <p>
     exceptionHandling()
    </p>
    <p>
     允许配置错误处理
    </p>
    <p>
     securityContext()
    </p>
    <p>
     在
     <code>
      HttpServletRequests
     </code>
     之间的
     <code>
      SecurityContextHolder
     </code>
     上设置
     <code>
      SecurityContext
     </code>
     的管理。 当使用
     <code>
      WebSecurityConfigurerAdapter
     </code>
     时，这将自动应用
    </p>
    <p>
     servletApi()
    </p>
    <p>
     将
     <code>
      HttpServletRequest
     </code>
     方法与在其上找到的值集成到
     <code>
      SecurityContext
     </code>
     中。 当使用
     <code>
      WebSecurityConfigurerAdapter
     </code>
     时，这将自动应用
    </p>
    <p>
     csrf()
    </p>
    <p>
     添加
     <code>
      CSRF
     </code>
     支持，使用
     <code>
      WebSecurityConfigurerAdapter
     </code>
     时，默认启用
    </p>
    <p>
     logout()
    </p>
    <p>
     添加退出登录支持。当使用
     <code>
      WebSecurityConfigurerAdapter
     </code>
     时，这将自动应用。 默认情况是，访问
     <code>
      "URL/logout”
     </code>
     ，使
     <code>
      HTTP Session
     </code>
     无效来清除用户，清除已配置的任何
     <code>
      #rememberMe()
     </code>
     身份验证，清除
     <code>
      SecurityContextHolder
     </code>
     ，然后重定向到
     <code>
      ”/login?success”
     </code>
    </p>
    <p>
     anonymous()
    </p>
    <p>
     允许配置匿名用户的表示方法。 当与
     <code>
      WebSecurityConfigurerAdapter
     </code>
     结合使用时，这将自动应用。 默认情况下，匿名用户将使用
     <code>
      org.springframework.security.authentication.AnonymousAuthenticationToken
     </code>
     表示，并包含角色
     <code>
      “ROLE_ANONYMOUS”
     </code>
    </p>
    <p>
     formLogin()
    </p>
    <p>
     指定支持基于表单的身份验证。如果未指定
     <code>
      FormLoginConfigurer#loginPage(String)
     </code>
     ，则将生成默认登录页面
    </p>
    <p>
     oauth2Login()
    </p>
    <p>
     根据外部
     <code>
      OAuth 2.0
     </code>
     或
     <code>
      OpenID Connect 1.0
     </code>
     提供程序配置身份验证
    </p>
    <p>
     requiresChannel()
    </p>
    <p>
     配置通道安全。为了使该配置有用，必须提供至少一个到所需信道的映射
    </p>
    <p>
     httpBasic()
    </p>
    <p>
     配置
     <code>
      Http Basic
     </code>
     验证
    </p>
    <p>
     addFilterAt()
    </p>
    <p>
     在指定的
     <code>
      Filter
     </code>
     类的位置添加过滤器
    </p>
    <hr/>
    <h4>
     <a id="URL_347">
     </a>
     自定义URL身份验证
    </h4>
    <ul>
     <li>
      <p>
       缺省登陆路径
      </p>
      <pre><code>protected void configure(HttpSecurity http) throws Exception {
    http
    // 表示允许使用HttpServletRequest限制访问
        .authorizeRequests()
        // 对任何请求都进行身份验证
            .anyRequest().authenticated()
            .and()
        // 启动基于表单验证
        .formLogin()
            // 该路径允许所有人访问到
            .permitAll();        
}
</code></pre>
      <p>
       上面的
       <code>
        .formLogin()
       </code>
       没有配置登陆页面，
       <code>
        spring security
       </code>
       会默认的生成一个页面
      </p>
      <p>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/4d7ca11e12751b5da9bcc4115f6c67d2.png"/>
      </p>
      <p>
       配置自己的登陆页面，使用
       <code>
        .loginPage("路径")
       </code>
       ：
      </p>
      <pre><code> 	.formLogin()
	    .loginPage("/login")
	    .permitAll();  
</code></pre>
      <p>
       自定义的页面表单中，需要包含下面的代码：
      </p>
      <pre><code>&lt;input type="hidden"  name="${_csrf.parameterName}"   value="${_csrf.token}"/&gt;
</code></pre>
      <p>
       防止
       <code>
        CSRF
       </code>
       攻击；
      </p>
     </li>
     <li>
      <p>
       授权请求
      </p>
      <p>
       上面配置的只是，一刀切，对除了登陆路径的其他任何路径都进行身份验证，实际开发中，我们应该有细粒度的配置，比如权限控制；
      </p>
      <pre><code>protected void configure(HttpSecurity http) throws Exception {
    http
        .authorizeRequests()                                                                
        	// 如果URL以 "/resources/**", "/signup", "/about" 开头，则任何用户都可以访问
            .antMatchers("/resources/**", "/signup", "/about").permitAll()         
             // 如果访问 "/admin/**" ，则必须具有 ”ADMIN" 角色才可以访问 ；
             // 其中，因为调用的就是 hasRole 方法，所以 前缀 ROLE_ 可以省略不写；
            .antMatchers("/admin/**").hasRole("ADMIN")      
            // 如果访问 /db/** 开头的 URL ，则必须同时拥有 "ADMIN" "DBA" 两个角色；                        
            // 对于这样多角色判断的，使用 access 连接        
            .antMatchers("/db/**").access("hasRole('ADMIN') and hasRole('DBA')")            
            // 任何URL ，经过上面几道工序，如果还没有得到匹配的，则只要求进行身份的验证；
            // 这里说明下，匹配 URL 的规则是从上到下，按照顺序匹配的，如果前面的已经匹配了，则后面的不再进行匹配；
            .anyRequest().authenticated()                                                   
            .and()
        // ...
        .formLogin();
}
</code></pre>
     </li>
     <li>
      <p>
       注销操作
      </p>
      <p>
       因为我们使用的是
       <code>
        WebSecurityConfigurerAdapter
       </code>
       ，
       <code>
        logout
       </code>
       功能默认就有，不需要我们自己编码，只需要访问
       <code>
        /logout
       </code>
       即可，然后
       <code>
        spring security
       </code>
       会自动的帮我们完成注销操作；
      </p>
      <p>
       并且
       <code>
        spring security
       </code>
       会完成以下几件事：
      </p>
      <p>
       1、销毁对应的会话
       <code>
        session
       </code>
       ；
       <br/>
       2、清楚任何已经配置的
       <code>
        记住我
       </code>
       身份验证；
       <br/>
       3、清除
       <code>
        SecurityContextHolder
       </code>
       ；
       <br/>
       4、重定向到
       <code>
        /login?logout
       </code>
       ；
      </p>
      <p>
       上面都是
       <code>
        spring security
       </code>
       默认的行为（
       <em>
        前提是使用
        <code>
         WebSecurityConfigurerAdapter
        </code>
       </em>
       ），我们还可以自己定制更细节的行为 ：
      </p>
      <pre><code>protected void configure(HttpSecurity http) throws Exception {
    http
    	// 提供注销支持，如果使用 `WebSecurityConfigurerAdapter` 则默认提供支持，不需要显式配置了
        .logout()       
        	// 触发注销行为的 URL ，默认是 /logout ,如果启用了 CSRF ，则该请求必须是 post ；但是默认就是启用 CSRF ，so 默认就必须使用 post 访问 ；                                                        
            .logoutUrl("/my/logout")   
            //  触发注销操作以后，重定到具体的页面，默认是 /login?logout                                             
            .logoutSuccessUrl("/my/index")   
            //   我们指定一个自定义的LogoutSuccessHandler。如果指定了此参数，则忽略logoutSuccessUrl 的配置                                              
            .logoutSuccessHandler(logoutSuccessHandler)    
            // 指定在注销时是否使HttpSession无效。默认情况下为 真                   
            .invalidateHttpSession(true)                   
            //   添加LogoutHandler。默认情况下，SecurityContextLogoutHandler被添加为最后一个LogoutHandler。                       
            .addLogoutHandler(logoutHandler)    
            //  指定注销成功以后，要删除的cookie的名字   
            //  同时这里也是显式添加 CookieClearingLogoutHandler 处理器的快捷方式                     
            .deleteCookies(cookieNamesToClear)                                      
            .and()
        ...
}
</code></pre>
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="_and__445">
     </a>
     身份验证 and 授权
    </h4>
    <p>
     应用程序的安全问题，或许可以简单的归结为两类：
     <code>
      身份验证 and 授权
     </code>
     ，
     <code>
      Spring Security
     </code>
     致力于将二者分开，为两者提供各自的问题解决方法；
    </p>
    <hr/>
    <h4>
     <a id="Authentication_451">
     </a>
     Authentication
    </h4>
    <p>
     <code>
      AuthenticationManager
     </code>
     是用于身份认证的主要策略接口，它仅有一个方法；
    </p>
    <pre><code>public interface AuthenticationManager {

  Authentication authenticate(Authentication authentication)
    throws AuthenticationException;

}
</code></pre>
    <p>
     该接口的方法，可以返回下面的三种数据的任何一个：
    </p>
    <ul>
     <li>
      <p>
       <code>
        authenticated=true
       </code>
      </p>
      <p>
       如果认为输入的主体（可以理解为用户），是有效的，即账号密码角色都对，则验证通过，返回
       <code>
        authenticated=true
       </code>
       ；
      </p>
     </li>
     <li>
      <p>
       <code>
        AuthenticationException
       </code>
      </p>
      <p>
       如果验证没有通过没，则抛出一个
       <code>
        AuthenticationException
       </code>
       异常，这个异常是一个运行时异常，官网文档里面建议，不需要我们代码中捕捉然后处理，我们应该做一个统一处理，让其跳到特定页面（
       <em>
        网站全局异常页面
       </em>
       ）
      </p>
     </li>
     <li>
      <p>
       <code>
        null
       </code>
      </p>
      <p>
       如果无法决定是否验证成功，则返回
       <code>
        null
       </code>
       ，为什么会出现无法决定呢，因为一个
       <code>
        AuthenticationManager
       </code>
       实例，可能只管理一部分身份的验证，另外一些身份验证有其他
       <code>
        AuthenticationManager
       </code>
       实例验证，因此这时候，就会返回一个
       <code>
        null
       </code>
       ；
      </p>
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="AuthenticationProvider_480">
     </a>
     <code>
      AuthenticationProvider
     </code>
    </h4>
    <p>
     <code>
      AuthenticationManager
     </code>
     最常用的实现是
     <code>
      ProviderManager
     </code>
     ，它委托一系列
     <code>
      AuthenticationProvider
     </code>
     实例。
     <code>
      AuthenticationProvider
     </code>
     有点像
     <code>
      AuthenticationManager
     </code>
     ，但它有一个额外的方法允许调用者查询它是否支持给定的身份验证类型：
    </p>
    <pre><code>public interface AuthenticationProvider {

	Authentication authenticate(Authentication authentication)
			throws AuthenticationException;

	boolean supports(Class&lt;?&gt; authentication);

}
</code></pre>
    <p>
     一个
     <code>
      ProviderManager
     </code>
     支持多种不同验证机制，通过委托多个小弟
     <code>
      AuthenticationProvider
     </code>
     来完成，小弟的方法
     <code>
      supports(Class&lt;?&gt; authentication)
     </code>
     ，用于校验传给它的验证身份类型是否支持，如果不支持，则返回
     <code>
      false
     </code>
     ， 然后
     <code>
      ProviderManager
     </code>
     就会跳过该小弟，询问下一位小弟；（有待考证）
    </p>
    <p>
     <code>
      ProviderManager
     </code>
     有一个可选的父级，如果所有小弟都返回
     <code>
      null
     </code>
     ，它可以参考。如果父级不可用，则
     <code>
      null
     </code>
     验证会导致
     <code>
      AuthenticationException
     </code>
     。
    </p>
    <p>
     有时候，
     <code>
      web
     </code>
     应用有多个受保护的逻辑组，比如：
     <code>
      /db/**
     </code>
     、
     <code>
      /login/**
     </code>
     等等，每个受保护的逻辑组可能有他们自己的专用的
     <code>
      AuthenticationManager
     </code>
     ，通常情况下这些
     <code>
      AuthenticationManager
     </code>
     ，都是
     <code>
      ProviderManager
     </code>
     ，他们共享一个父级——这个父级是一个全局资源，充当所有的
     <code>
      ProviderManager
     </code>
     的后备资源，也是一个
     <code>
      ProviderManager
     </code>
     ，走投无路的时候，就去找老爹；
    </p>
    <hr/>
    <h4>
     <a id="_502">
     </a>
     自定义身份验证器
    </h4>
    <p>
     <code>
      spring security
     </code>
     提供了配置帮助程序，用于快速获取程序中设置的身份管理；其中，最常用的是
     <code>
      AuthenticationManagerBuilder
     </code>
     ，它很适用于
     <code>
      内存
     </code>
     、
     <code>
      JDBC
     </code>
     、
     <code>
      LDAP
     </code>
     ，或者增加一个
     <code>
      UserDetailsService
     </code>
     ；
    </p>
    <p>
     配置全局的
     <code>
      AuthenticationManager
     </code>
     ：
    </p>
    <pre><code>@Configuration
public class ApplicationSecurity extends WebSecurityConfigurerAdapter {

   ... // web stuff here

  @Autowired
  public initialize(AuthenticationManagerBuilder builder, DataSource dataSource) {
    builder.jdbcAuthentication().dataSource(dataSource).withUser("dave")
      .password("secret").roles("USER");
  }

}
</code></pre>
    <p>
     上面的配置中，关于
     <code>
      AuthenticationManagerBuilder
     </code>
     的用法是一种广泛的使用方式；它被使用
     <code>
      @Autowired
     </code>
     注入到方法中，这将导致它可以构建全局的
     <code>
      AuthenticationManager
     </code>
     ；
    </p>
    <p>
     当然也有另外一种写法：
    </p>
    <pre><code>@Configuration
public class ApplicationSecurity extends WebSecurityConfigurerAdapter {

  @Autowired
  DataSource dataSource;

   ... // web stuff here

  @Override
  public configure(AuthenticationManagerBuilder builder) {
    builder.jdbcAuthentication().dataSource(dataSource).withUser("dave")
      .password("secret").roles("USER");
  }

}
</code></pre>
    <p>
     这里是覆盖父类方法，这样的
     <code>
      AuthenticationManagerBuilder
     </code>
     只能被用来构建局部的
     <code>
      AuthenticationManager
     </code>
     ，构建出的
     <code>
      AuthenticationManager
     </code>
     将是全局
     <code>
      AuthenticationManager
     </code>
     的小弟；
    </p>
    <p>
     在
     <code>
      springBoot
     </code>
     中可以使用
     <code>
      @Autowired
     </code>
     将一个全局的
     <code>
      bean
     </code>
     直接注入到另外一个
     <code>
      bean
     </code>
     里面 ，但是不能对局部的
     <code>
      bean
     </code>
     使用这种操作；
    </p>
    <p>
     <code>
      springBoot
     </code>
     中，如果自己没有全局的自定义身份验证，它默认提供一个全局的
     <code>
      AuthenticationManager
     </code>
     ，默认的
     <code>
      AuthenticationManager
     </code>
     的安全性不需要我们去担心；我们可以在局部的
     <code>
      AuthenticationManager
     </code>
     里面做配置，从而不去影响到全局的
     <code>
      AuthenticationManager
     </code>
     ；
    </p>
    <hr/>
    <h4>
     <a id="_551">
     </a>
     授权
    </h4>
    <p>
     身份认证成功以后，就需要我们进行授权了；
    </p>
    <p>
     在
     <code>
      spring Security
     </code>
     中的对应的核心类是
     <code>
      AccessDecisionManager
     </code>
     ，框架本身有三个默认实现，并且这三个实现都委托给
     <code>
      AccessDecisionVoter
     </code>
     管理，跟身份验证的
     <code>
      AuthenticationProvider
     </code>
     委托给
     <code>
      ProviderManager
     </code>
     ；
    </p>
    <p>
     <code>
      AccessDecisionVoter
     </code>
     如下：
    </p>
    <pre><code>public interface AccessDecisionVoter&lt;S&gt; {
    int ACCESS_GRANTED = 1;
    int ACCESS_ABSTAIN = 0;
    int ACCESS_DENIED = -1;

  	boolean supports(ConfigAttribute attribute);

	boolean supports(Class&lt;?&gt; clazz);

	int vote(Authentication authentication, S object,
        Collection&lt;ConfigAttribute&gt; attributes);
}
</code></pre>
    <p>
     其中
     <code>
      S object
     </code>
     的
     <code>
      Object
     </code>
     对象，在
     <code>
      AccessDecisionManager 、AccessDecisionVoter
     </code>
     中是通用的，代表用户想访问的东西，大部分时候都代表一个
     <code>
      web
     </code>
     资源，或者一个
     <code>
      java
     </code>
     类的方法 ；
    </p>
    <p>
     <code>
      ConfigAttribute
     </code>
     接口也是通用的，是安全对象的封装，里面封装一些元数据，用于确定访问
     <code>
      Object
     </code>
     所需要的权限 ；它只有一个方法，返回
     <code>
      String
     </code>
     ，代表开发者制定的规则，谁可以访问
     <code>
      Object
     </code>
     ；其中返回的字符串格式，一般都是以
     <code>
      ROLE_
     </code>
     为前缀的角色名称，比如
     <code>
      ROLE_ADMIN
     </code>
     、
     <code>
      ROLE_AUDIT
     </code>
     等；
    </p>
    <p>
     使用
     <code>
      SPEL
     </code>
     表达式 的
     <code>
      ConfigAttribute
     </code>
     是很常见的，比如
     <code>
      isFullyAuthenticated（）&amp;&amp; hasRole（'FOO'）
     </code>
     ，这些表达式被
     <code>
      AccessDecisionVoter
     </code>
     所支持，它可以处理这些表达式，并且创建对应的上下文；
    </p>
    <p>
     如果你想扩展
     <code>
      SPEL
     </code>
     的表达式范围，需要自定义实现
     <code>
      SecurityExpressionRoot
     </code>
     ，有时还需要
     <code>
      SecurityExpressionHandler
     </code>
     。
    </p>
    <p>
     大部分人都使用默认的
     <code>
      AccessDecisionManager
     </code>
     —
     <code>
      AffirmativeBased
     </code>
     ，它的内部是只要有一个投票通过，即可进行授权访问，要想进行细致的投票定制，可以自定义实现
     <code>
      AccessDecisionManager
     </code>
     ；
    </p>
    <hr/>
    <h4>
     <a id="Spring_Security__586">
     </a>
     <code>
      Spring Security
     </code>
     的过滤器
    </h4>
    <p>
     <code>
      Spring Security
     </code>
     是基于
     <code>
      Filter
     </code>
     进行实现的；
    </p>
    <p>
     当客户端向应用程序发送请求，
     <code>
      Spring Security
     </code>
     根据请求
     <code>
      URI
     </code>
     的路径决定哪些过滤器和哪个
     <code>
      servlet
     </code>
     处理它 ；一个
     <code>
      servlet
     </code>
     处理一个请求，所有的
     <code>
      Filter
     </code>
     构建成一个 拦截链 ，它们是有序的，其中的任何一个都可以中断请求，或者修改掉
     <code>
      request 、response
     </code>
     ；
    </p>
    <p>
     <code>
      Spring Security
     </code>
     本身是在
     <code>
      Spring
     </code>
     容器中是作为作为一个
     <code>
      Filter
     </code>
     的，其类型是
     <code>
      FilterChainProxy
     </code>
     ，但是其本身之中，又包含多个
     <code>
      Filter
     </code>
     ；
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/53fe133e562fce7ea0a13ea5a57695a9.png"/>
    </p>
    <p>
     <code>
      Spring Security
     </code>
     在
     <code>
      Spring
     </code>
     容器中仅仅是一个物理的过滤器，也就是它本身其实不进行任何拦截，都是交给它背后的过滤器；
    </p>
    <p>
     实际上在
     <code>
      Spring Security
     </code>
     的内部还有一个间接层，它通常作为
     <code>
      DelegatingFilterProxy
     </code>
     安装在
     <code>
      Spring Security
     </code>
     容器中，它不需要在
     <code>
      Spring
     </code>
     中进行注册，这个间接层最后委托
     <code>
      FilterChainProxy
     </code>
     在
     <code>
      spring
     </code>
     容器中进行注册，并且名字必须是
     <code>
      springSecurityFilterChain
     </code>
     ； 。
    </p>
    <p>
     <code>
      DelegatingFilterProxy
     </code>
     背后的那一堆
     <code>
      Filter
     </code>
     都实现了相同接口，只是具体逻辑不同，并且都可以决定后续的
     <code>
      Filter
     </code>
     是否进行拦截 ；
    </p>
    <p>
     <code>
      Spring Security
     </code>
     背后可以管理多个过滤器，但是这些过滤器对
     <code>
      Spring
     </code>
     都是不可见的，并且这些过滤器可以进行分组，然后
     <code>
      Spring Security
     </code>
     将
     <code>
      request
     </code>
     进行派发给第一个匹配的分组；
    </p>
    <p>
     如图所示：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/9c5791d1e1a2fc64def588fe9f7ddcb1.png"/>
    </p>
    <p>
     这只是
     <code>
      Spring Security
     </code>
     分配请求的一种方法，用此种方法分配时，只有一组拦截链可以处理请求 ；
    </p>
    <hr/>
    <h4>
     <a id="__611">
     </a>
     创建和自定义 拦截链
    </h4>
    <p>
     在
     <code>
      SpringBoot
     </code>
     中有一个默认的最低级别的拦截器 ，它拦截
     <code>
      /**
     </code>
     请求，预定义的顺序为
     <code>
      SecurityProperties.BASIC_AUTH_ORDER
     </code>
     ，如果不想使用它的话，设置
     <code>
      SecurityProperties.BASIC_AUTH_ORDER = false
     </code>
     即可 ；
    </p>
    <p>
     但是一般都将其作为一个基准，便于我们自己创建拦截器的时候，定义顺序 ；
    </p>
    <p>
     比如新建一个拦截器，只需要继承
     <code>
      WebSecurityConfigurerAdapter 或 WebSecurityConfigurer
     </code>
     即可，然后定义一个顺序，这时候就可以使用
     <code>
      SecurityProperties.BASIC_AUTH_ORDER
     </code>
     作为基准；
    </p>
    <p>
     使用
     <code>
      @Order(SecurityProperties.BASIC_AUTH_ORDER - 10)
     </code>
     定义顺序，数值越少，级别越高 ；
    </p>
    <pre><code>@Configuration
@Order(SecurityProperties.BASIC_AUTH_ORDER - 10)
public class ApplicationConfigurerAdapter extends WebSecurityConfigurerAdapter {
  @Override
  protected void configure(HttpSecurity http) throws Exception {
    http.antMatcher("/foo/**")
     ...;
  }
}
</code></pre>
    <p>
     其中对于同一组资源可能有多个拦截器设置匹配规则，此时匹配规则重叠，则哪一个拦截器的优先级高，就使用谁 ；
    </p>
    <hr/>
    <h4>
     <a id="_636">
     </a>
     请求匹配调度和授权
    </h4>
    <p>
     一个
     <code>
      WebSecurityConfigurerAdapter
     </code>
     就是一组拦截链，使用
     <code>
      http.antMatcher("/foo/**")
     </code>
     决定匹配什么请求；
    </p>
    <p>
     然后再使用
     <code>
      antMatchers("/foo/bar").hasRole("BAR")
     </code>
     ，设置访问该组拦截链拦截的URL的角色 ；
    </p>
    <pre><code>@Configuration
@Order(SecurityProperties.BASIC_AUTH_ORDER - 10)
public class ApplicationConfigurerAdapter extends WebSecurityConfigurerAdapter {
  @Override
  protected void configure(HttpSecurity http) throws Exception {
    http.antMatcher("/foo/**")
      .authorizeRequests()
        .antMatchers("/foo/bar").hasRole("BAR")
        .antMatchers("/foo/spam").hasRole("SPAM")
        .anyRequest().isAuthenticated();
  }
}
</code></pre>
    <hr/>
    <h4>
     <a id="_658">
     </a>
     方法安全
    </h4>
    <p>
     前面讲的都是基于
     <code>
      url
     </code>
     级别的授权，下面说的授权，粒度更细，是基于方法的 ；
    </p>
    <p>
     对于
     <code>
      Spring Security
     </code>
     来说，方法也是一种受保护的资源类型 ;
    </p>
    <p>
     在应用程序的顶级配置中，启动方法安全性
     <code>
      @EnableGlobalMethodSecurity(securedEnabled = true)
     </code>
     ；
    </p>
    <pre><code>@SpringBootApplication
@EnableGlobalMethodSecurity(securedEnabled = true)
public class SampleSecureApplication {
}
</code></pre>
    <p>
     然后在需要保护的方法上配置：
    </p>
    <pre><code>@Service
public class MyService {

  @Secured("ROLE_USER")
  public String secure() {
    return "Hello Security";
  }

}
</code></pre>
    <p>
     这样配置以后，如果
     <code>
      Spring
     </code>
     创建了这种类型的
     <code>
      Bean
     </code>
     对象，那么它将被代理，并且调用者必须在实际执行该方法之前通过安全拦截器。如果访问被拒绝，则调用者将获得
     <code>
      AccessDeniedException
     </code>
     而不是实际的方法结果 ；
    </p>
    <p>
     还有其他注释可用于强制执行安全约束的方法，特别是
     <code>
      @PreAuthorize
     </code>
     和
     <code>
      @PostAuthorize
     </code>
     ，它们允许您编写包含对方法参数和返回值的引用的表达式。
    </p>
    <hr/>
    <h4>
     <a id="spring_Security__691">
     </a>
     spring Security 线程问题
    </h4>
    <ul>
     <li>
      <p>
       <code>
        SecurityContext
       </code>
      </p>
      <p>
       <code>
        Spring Security
       </code>
       是线程绑定的，最基本的组件是
       <code>
        SecurityContext
       </code>
       里面包含
       <code>
        Authentication
       </code>
       ，可以通过下面的方式，获取和修改
       <code>
        SecurityContext
       </code>
       ；
      </p>
      <pre><code>SecurityContext context = SecurityContextHolder.getContext();
Authentication authentication = context.getAuthentication();
assert(authentication.isAuthenticated);
</code></pre>
     </li>
     <li>
      <p>
       <code>
        @AuthenticationPrincipal
       </code>
       注解
      </p>
      <pre><code>@RequestMapping("/foo")
public String foo(@AuthenticationPrincipal User user) {
  ... // do stuff with user
}
</code></pre>
      <p>
       此注解可以将当前的
       <code>
        Authentication
       </code>
       从
       <code>
        SecurityContext
       </code>
       中抽取出来，然后调用
       <code>
        getPrincipal()
       </code>
       填充方法参数；至于调用
       <code>
        getPrincipal()
       </code>
       获取的
       <code>
        Principal
       </code>
       类型，取决于
       <code>
        AuthenticationManager
       </code>
       使用什么类型来进行验证 Authentication ；
      </p>
      <p>
       如果使用
       <code>
        Spring Security
       </code>
       的
       <code>
        HttpServletRequest
       </code>
       中的
       <code>
        Principal
       </code>
       ，那么
       <code>
        Principal
       </code>
       将是
       <code>
        Authentication
       </code>
       类型，我们可以向下面这样操作，获取
       <code>
        User
       </code>
       ：
      </p>
      <pre><code>@RequestMapping("/foo")
public String foo(Principal principal) {
  Authentication authentication = (Authentication) principal;
  User = (User) authentication.getPrincipal();
  ... // do stuff with user
}
</code></pre>
     </li>
     <li>
      <p>
       <code>
        异步处理安全方法
       </code>
      </p>
      <p>
       由于
       <code>
        SecurityContext
       </code>
       是线程绑定的，因此如果要进行任何调用安全方法的后台处理，例如使用
       <code>
        @Async
       </code>
       （
       <code>
        Runnable, Callable
       </code>
       等等异步执行方法），需要确保传播上下文。
      </p>
      <p>
       要将
       <code>
        SecurityContext
       </code>
       传播到
       <code>
        @Async
       </code>
       方法，需要提供
       <code>
        AsyncConfigurer
       </code>
       并确保
       <code>
        Executor
       </code>
       的类型正确：
      </p>
      <pre><code>@Configuration
public class ApplicationConfiguration extends AsyncConfigurerSupport {

  @Override
  public Executor getAsyncExecutor() {
    return new DelegatingSecurityContextExecutorService(Executors.newFixedThreadPool(5));
  }

}
</code></pre>
     </li>
    </ul>
