     1. STM32程序升级方法
    </h2>
    <h3>
     <a id="11_STLink__Jlink_5">
     </a>
     1.1 ST-Link / J-link下载
    </h3>
    <p>
     这个应该是最基本的方法，只要自己写过程序的应该都会，将编译生成的
     <code>
      hex
     </code>
     文件使用
     <font color="red">
      <strong>
       ST-Link
      </strong>
     </font>
     工具或者
     <font color="red">
      <strong>
       J-Link
      </strong>
     </font>
     工具直接下载进 Flash 即可。Keil中点击下载也能一键下载。
    </p>
    <p>
     下载时可以看到地址是从
     <font color="red">
      <strong>
       0x0800 0000
      </strong>
     </font>
     ，即 Flash 的起始地址开始下载的。
    </p>
    <p>
     <strong>
      优点
     </strong>
     ：简单，插上下载器直接下载即可。
     <br/>
     <strong>
      缺点
     </strong>
     ：在产品中嵌入式板卡封装起来之后，因为下载口没有实际功能，所以很多时候下不拆机是没办法插上下载器的。这时候就不方便。
    </p>
    <p>
     简单补充一句，
     <strong>
      bin文件
     </strong>
     和
     <strong>
      hex文件
     </strong>
     的区别：
    </p>
    <ul>
     <li>
      bin文件不带地址信息，因此下载的时候需要指定下载地址。
     </li>
     <li>
      hex文件自带地址信息，直接点击下载自己会找到要下载到的地址（默认0x0800 0000）。
     </li>
    </ul>
    <h3>
     <a id="12_ISPIn_System_Programing_18">
     </a>
     1.2 ISP（In System Programing）
    </h3>
    <p>
     我们常见的
     <font color="red">
      <strong>
       一键下载电路
      </strong>
     </font>
     就是用的这种方式。这个是利用了 STM32 自带的 Bootloader 升级程序。
    </p>
    <p>
     在用户参考手册中，可以看到下表，关于启动模式设置的。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/675bbbeb90025de5443801627b824eff.png">
      <br/>
      中文版的如下：
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/3434a53ec0ab526a32769fbba1e05f3c.png">
       <br/>
       <strong>
        程序运行时的启动过程
       </strong>
       ：
      </img>
     </img>
    </p>
    <p>
     一般在我们的程序运行的时候，BOOT0 是接地的，也就是
     <code>
      BOOT0 = 0
     </code>
     。也就是程序是从
     <strong>
      主存储器Flash
     </strong>
     开始启动的，启动的地址为
     <code>
      0x08000000
     </code>
     。
    </p>
    <p>
     <strong>
      使用ISP下载时的启动过程
     </strong>
     ：
    </p>
    <ol>
     <li>
      首先，硬件上将 STM32 的 BOOT0 引脚拉高、BOOT1 引脚拉低，即
      <code>
       BOOT0 = 1
      </code>
      、
      <code>
       BOOT1 = 0
      </code>
      。
     </li>
     <li>
      此时，程序会从系统存储器中的程序启动，这段程序会接收串口数据（我们编译好的程序文件），并将这写数据放到主闪存存储器（Flash）当中。
     </li>
     <li>
      最后，硬件上重新将 BOOT0 接地，也就是
      <code>
       BOOT0 = 0
      </code>
      ，然后复位引脚拉低，程序复位重启，从 Flash 中开始运行程序。
     </li>
    </ol>
    <blockquote>
     <p>
      可以这么理解：芯片出厂时，系统存储器中已经存储了一段程序，这段程序的功能是将
      <strong>
       串口1
      </strong>
      （固定的）收到的数据，放到主闪存存储器（Flash）中，从0x0800 0000地址处开始。
     </p>
    </blockquote>
    <p>
     这样也就完成了我们的一键下载过程。
     <br/>
     比如，正点原子的一键下载电路如下，左侧的三极管就是用来完成 BOOT引脚和复位引脚的操作的。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/e8eec45993d9c40f498f29ce6a301f45.jpeg"/>
    </p>
    <blockquote>
     <p>
      具体的硬件引脚电平如何变化百度搜一下，分析应该挺多的。我也没详细看过。
     </p>
    </blockquote>
    <p>
     沁恒官网也有专门的一键下载芯片，原理上其实一样的。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/dd98762f5c2a429eca09c11631ab5ed7.jpeg"/>
    </p>
    <blockquote>
     <p>
      链接：https://www.wch.cn/application/575.html
     </p>
    </blockquote>
    <p>
     ISP的下载方式：
    </p>
    <p>
     <strong>
      优点
     </strong>
     ：提供了一种升级方式，无需代码支持。
     <br/>
     <strong>
      缺点
     </strong>
     ：需要相应的硬件支持，成本增加；使用的接口也是固定的，并且很多时候串口可能用于其他功能了已经。
    </p>
    <h3>
     <a id="13_IAPIn_Applicating_Programing_52">
     </a>
     1.3 IAP（In Applicating Programing）
    </h3>
    <p>
     IAP 和 ISP 其实基本上是一样的。
    </p>
    <p>
     但是：ISP 是由厂商已经提供好的，因此接口固定；IAP可以自定义使用任何接口接收应用程序。也正是因为这一点，使得用户可以用多种不同的方式进行升级。
    </p>
    <h4>
     <a id="131__58">
     </a>
     1.3.1 正常程序运行流程
    </h4>
    <p>
     在看IAP之前，要先看一下正常情况下，程序从Flash启动时的启动流程，如下图所示：
    </p>
    <ol>
     <li>
      首先程序从Flash启动，根据
      <strong>
       中断向量表
      </strong>
      找到
      <strong>
       复位中断处理函数
      </strong>
      的地址（0x0800 0004处是中断向量表的起始地址，记录了复位中断处理函数的地址）。
     </li>
     <li>
      执行复位中断处理函数，初始化系统环境之后，该函数最后会跳转到
      <strong>
       main函数
      </strong>
      继续运行。
     </li>
     <li>
      在main函数的死循环中一直运行，直到
      <strong>
       有中断发生
      </strong>
      时（外设中断等等），重新跳转到
      <strong>
       中断向量表起始
      </strong>
      处。
     </li>
     <li>
      在中断向量表中根据
      <strong>
       中断信号源
      </strong>
      来判断要执行的中断处理函数，然后跳转到
      <strong>
       相应的中断处理函数
      </strong>
      。
     </li>
     <li>
      中断处理函数运行完成之后继续跳转到
      <strong>
       main函数
      </strong>
      处继续运行。
     </li>
    </ol>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/53b1e2600c4b0e4607383394a9c88afe.png"/>
    </p>
    <h4>
     <a id="132_IAP_69">
     </a>
     1.3.2 有IAP时程序运行流程
    </h4>
    <p>
     接下来看下有了IAP之后的启动流程，如下图所示，下图中可以看到，在Flash中存储了 两“套” 程序（套的意思是，不仅只有用户程序，配套的中断向量等也都有）。
    </p>
    <p>
     其中第一套为：Bootloader程序，该程序的功能是，接收某个接口的数据，并把这些数据存储到Flash中。
    </p>
    <blockquote>
     <p>
      这些数据其实就是后面的
      <strong>
       一套
      </strong>
      APP程序，这套应用程序可以通过Bootlaoder程序保存到Flash中，这样就不用再使用专门的下载器。
     </p>
    </blockquote>
    <p>
     第二套才是我们真正的应用程序。
    </p>
    <ol>
     <li>
      首先程序从Flash启动，根据
      <strong>
       中断向量表
      </strong>
      找到
      <strong>
       复位中断处理函数
      </strong>
      的地址（0x0800 0004处是中断向量表的起始地址，记录了复位中断处理函数的地址）。
     </li>
     <li>
      执行复位中断处理函数，该函数最后会跳转到
      <strong>
       Bootloader的main函数
      </strong>
      继续运行。这个main函数的任务就是，判断是否接收新的APP程序，如果有，就把新的APP程序文件保存到Flash（就是第二套程序的位置）然后跳转到第二套程序中运行。如果无，就直接跳转到第二套程序中运行。
     </li>
     <li>
      跳转之后的过程也是和正常程序运行的流程一样，一旦进入新的APP程序的起始地址，就会根据中断向量表找到复位中断处理函数，然后进入
      <strong>
       App程序的main函数
      </strong>
      运行。
     </li>
     <li>
      但是如果发生中断，是强制跳转到
      <strong>
       Bootloader程序的中断向量表
      </strong>
      进行查询的，而我们需要的肯定是需要跳转到
      <strong>
       APP程序的中断处理函数
      </strong>
      处运行。
      <strong>
       <font color="red">
        所以，在进行到APP程序后，APP程序一定要修改中断向量表的偏移，让查找对应中断处理函数的时候偏移一段地址到App程序的中断处理函数处，否则APP程序中的中断发生时，就无法跳转到APP程序的中断处理函数了。
       </font>
      </strong>
     </li>
    </ol>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/50aacf2496b665a00ca982bfa4ec971b.png">
      <br/>
      到这，我们就知道Bootloader程序是干啥的了。
     </img>
    </p>
    <blockquote>
     <p>
      这个分析过程，是2.1小节中的方式一。其他方式跟这种原理上基本是一样一样的。
     </p>
    </blockquote>
    <p>
     对比：
    </p>
    <table>
     <thead>
      <tr>
       <th>
        ISP
       </th>
       <th>
        IAP
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        在系统存储器中存储了一套接收串口1数据的程序
       </td>
       <td>
        IAP是将Flash分成了两份，在第一份中存储了一套接收某个接口的程序
       </td>
      </tr>
      <tr>
       <td>
        使用硬件BOOT引脚设置进行跳转
       </td>
       <td>
        使用软件（直接修改PC指针）进行跳转
       </td>
      </tr>
      <tr>
       <td>
        仅能使用芯片厂商设置好的接口（串口1）
       </td>
       <td>
        用户自定义，理论上只要能接收数据的接口都可以用
       </td>
      </tr>
     </tbody>
    </table>
