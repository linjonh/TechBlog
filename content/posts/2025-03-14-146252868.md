     经典架构风格对比
    </h2>
    <h3>
     <a id="_2">
     </a>
     面向对象架构风格/显示调用风格
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/edf42daf463a4a2eb0473d2f2c11eafb.png"/>
    </p>
    <h4>
     <a id="_5">
     </a>
     优点
    </h4>
    <ol>
     <li>
      一个对象对外隐藏了自己的详细信息。
     </li>
     <li>
      对象将数据和操作封装在一起。
     </li>
     <li>
      继承和封装方法为对象复用提供了技术支持。
     </li>
    </ol>
    <h4>
     <a id="_10">
     </a>
     缺点
    </h4>
    <ol>
     <li>
      如果一个对象要调用另一个对象，则必须知道它的标识和名称。
     </li>
    </ol>
    <h4>
     <a id="_13">
     </a>
     举例
    </h4>
    <p>
     一般业务系统
    </p>
    <h3>
     <a id="_17">
     </a>
     事件驱动的系统/隐式调用风格
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/5ec1ca9fbcae49bb937a5aa02b009dc3.png"/>
    </p>
    <h4>
     <a id="_20">
     </a>
     优点
    </h4>
    <ol>
     <li>
      事件发布者不需要知道哪些构件会响应事件。
     </li>
    </ol>
    <h4>
     <a id="_23">
     </a>
     缺点
    </h4>
    <ol>
     <li>
      构件放弃了对计算的控制权，完全由系统来决定。
     </li>
     <li>
      存在数据传输的问题。
     </li>
    </ol>
    <h4>
     <a id="_27">
     </a>
     举例
    </h4>
    <p>
     消息队列
    </p>
    <h3>
     <a id="_30">
     </a>
     基于规则的系统架构风格
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/f990035e5c304cdb83662e2a4d167406.png"/>
    </p>
    <h4>
     <a id="_34">
     </a>
     优点
    </h4>
    <ol>
     <li>
      规则数据可以动态更改，灵活性好。
     </li>
     <li>
      定义新的规则，即可扩展。
     </li>
    </ol>
    <h4>
     <a id="_38">
     </a>
     缺点
    </h4>
    <ol>
     <li>
      实时解释规则，性能较差。
     </li>
    </ol>
    <h4>
     <a id="_41">
     </a>
     举例
    </h4>
    <p>
     规则引擎
    </p>
    <h3>
     <a id="_44">
     </a>
     管道过滤器风格
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/d94ff2eb2d364c07b97cd4786de9b1d4.png"/>
    </p>
    <h4>
     <a id="_48">
     </a>
     优点
    </h4>
    <ol>
     <li>
      简单性。
     </li>
     <li>
      支持复用。
     </li>
     <li>
      便于系统分析。
     </li>
    </ol>
    <h4>
     <a id="_53">
     </a>
     缺点
    </h4>
    <ol>
     <li>
      不适合用来设计交互式应用系统。
     </li>
     <li>
      由于没有通用的数据传输标准，因此每个过滤器都需要解析输入数据和合成数据。
     </li>
     <li>
      难以进行错误处理。
     </li>
    </ol>
    <h4>
     <a id="_58">
     </a>
     举例
    </h4>
    <p>
     流程引擎
    </p>
    <h3>
     <a id="_61">
     </a>
     仓库风格
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/225f1332241847aba36223f25ee42e50.png"/>
    </p>
    <h4>
     <a id="_64">
     </a>
     优点
    </h4>
    <ol>
     <li>
      便于多客户共享大量数据，而不必关心数据时何时产生的，由谁提供的，通过何种途径来提供。
     </li>
    </ol>
    <h4>
     <a id="_67">
     </a>
     缺点
    </h4>
    <ol>
     <li>
      对共享数据结构，不同知识源要达成一致。
     </li>
     <li>
      需要同步机制和加锁机制来保证数据的完整性和一致性，增大了设计的复杂度。
     </li>
    </ol>
    <h4>
     <a id="_71">
     </a>
     举例
    </h4>
    <p>
     数据仓库
    </p>
    <h3>
     <a id="_74">
     </a>
     解释器风格
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/b5c13e90d6c142ffa7dabe92c4272df6.png"/>
    </p>
    <h4>
     <a id="_78">
     </a>
     优点
    </h4>
    <ol>
     <li>
      可移植性好。
     </li>
    </ol>
    <h4>
     <a id="_81">
     </a>
     缺点
    </h4>
    <ol>
     <li>
      由于使用了特定语言和自定义操作规则，因此增加了系统运行的开销。
     </li>
     <li>
      解释器系统难以设计和测试。
     </li>
    </ol>
    <h4>
     <a id="_85">
     </a>
     举例
    </h4>
    <p>
     编译器
    </p>
    <h3>
     <a id="_88">
     </a>
     分层架构风格
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/79a4a0b45f324bbda0210da423061a64.png"/>
    </p>
    <h4>
     <a id="_91">
     </a>
     优点
    </h4>
    <ol>
     <li>
      设计者可以将系统分解为一个增量的步骤序列，从而完成复杂的业务逻辑。
     </li>
     <li>
      每一层最多和相邻的上下两层进行交互，只给相邻层提供相同的接口。
     </li>
    </ol>
    <h4>
     <a id="_95">
     </a>
     缺点
    </h4>
    <ol>
     <li>
      并非所有系统都能够按照层次来进行划分。
     </li>
     <li>
      很难找到一种合适和正确的层次划分方法。
     </li>
     <li>
      传输数据需要经过多个层次。
     </li>
     <li>
      多层结构难以调试。
     </li>
    </ol>
    <h4>
     <a id="_101">
     </a>
     举例
    </h4>
    <p>
     OSI协议，MVC架构，DDD架构，Cola架构。
    </p>
