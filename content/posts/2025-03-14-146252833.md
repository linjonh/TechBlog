     为什么List、Set集合无法在遍历的时候修改内部元素
    </h1>
   </div>
  </div>
 </div>

    <p>
     以常用集合ArrayList为例，
     <code>
      ArrayList
     </code>
     在遍历过程中无法直接修改内部元素的结构（例如通过
     <code>
      remove()
     </code>
     或
     <code>
      add()
     </code>
     方法修改元素），是因为
     <strong>
      遍历的过程中修改结构
     </strong>
     可能会导致
     <strong>
      不一致的行为
     </strong>
     、
     <strong>
      并发修改异常
     </strong>
     或
     <strong>
      逻辑错误
     </strong>
     。
     <br/>
     注意：和
     <code>
      ArrayList
     </code>
     一样，
     <code>
      Set
     </code>
     集合也不允许在遍历时直接修改其结构（如添加或删除元素），否则会抛出
     <code>
      ConcurrentModificationException
     </code>
     异常。为了安全地修改结构，可以使用
     <code>
      Iterator
     </code>
     的
     <code>
      remove()
     </code>
     方法或者使用
     <code>
      removeIf
     </code>
     方法。在需要修改集合结构时，最好先完成遍历操作，避免在遍历过程中进行结构修改，确保程序的稳定性和一致性。
    </p>
    <h4>
     <a id="1_ConcurrentModificationException_3">
     </a>
     1.
     <strong>
      ConcurrentModificationException（并发修改异常）
     </strong>
    </h4>
    <p>
     <code>
      ArrayList
     </code>
     在内部维护了一个修改计数器（
     <code>
      modCount
     </code>
     ），每次修改
     <code>
      ArrayList
     </code>
     （如删除或添加元素）时，该计数器会增加。在遍历
     <code>
      ArrayList
     </code>
     时，
     <code>
      Iterator
     </code>
     会检查该计数器。如果在遍历过程中，集合的结构发生了变化（例如删除或添加元素），
     <code>
      Iterator
     </code>
     会发现计数器的值发生变化，因此抛出
     <code>
      ConcurrentModificationException
     </code>
     异常。
    </p>
    <p>
     这个机制的目的是为了防止多线程环境下发生集合的并发修改，从而导致无法预料的行为。在单线程情况下，虽然没有并发问题，但仍然需要保持结构一致性。
    </p>
    <h4>
     <a id="2__8">
     </a>
     2.
     <strong>
      修改结构导致遍历不一致
     </strong>
    </h4>
    <p>
     在遍历过程中修改
     <code>
      ArrayList
     </code>
     结构（比如添加、删除元素）可能会导致遍历过程中的不一致性。例如：
    </p>
    <ul>
     <li>
      <strong>
       删除元素：
      </strong>
      如果在遍历过程中删除元素，剩下的元素会向前移动，导致后续的元素错位。可能会导致漏掉一些元素。
     </li>
     <li>
      <strong>
       添加元素：
      </strong>
      如果在遍历过程中添加新元素，新的元素会加入到集合的末尾，或者加入到指定位置，这会影响元素顺序，导致遍历时无法保证按预期的顺序访问。
     </li>
    </ul>
    <p>
     这种行为非常不稳定，并且容易导致程序错误，因此 Java 中的
     <code>
      ArrayList
     </code>
     和其他集合类通常会避免在遍历过程中修改结构。
    </p>
    <h4>
     <a id="3__15">
     </a>
     3.
     <strong>
      线程安全性问题
     </strong>
    </h4>
    <p>
     如果
     <code>
      ArrayList
     </code>
     在多个线程中共享使用，且在遍历时对其进行了结构性修改（如添加或删除元素），这可能会导致多个线程间的竞争条件和数据不一致。为了避免此类问题，Java 设计上默认会禁止这种行为，尤其是在多线程环境中。
    </p>
    <h4>
     <a id="4__18">
     </a>
     4.
     <strong>
      遍历顺序和数据一致性
     </strong>
    </h4>
    <p>
     遍历
     <code>
      ArrayList
     </code>
     时假定集合的结构是固定的。如果在遍历过程中修改了元素（比如删除或添加元素），会破坏集合的状态，使得遍历结果变得不可预测。为了保证数据一致性，Java 设计者决定不允许在遍历时修改集合的结构。
    </p>
