    <h4>
     <a id="1__read_3">
     </a>
     <strong>
      1. 阻塞
      <code>
       read()
      </code>
     </strong>
    </h4>
    <ul>
     <li>
      <strong>
       阻塞
       <code>
        read()
       </code>
      </strong>
      就像
      <strong>
       你打电话问饭好了吗，但如果对方不接电话（没数据），你就一直守在电话边等，什么别的事情都做不了
      </strong>
      。
     </li>
    </ul>
    <blockquote>
     <p>
      你打电话问外卖员“饭好了吗？”，如果外卖还没送到，
      <strong>
       你只能一直等着，啥都干不了
      </strong>
      。
      <br/>
      <strong>
       （阻塞式等待）
      </strong>
     </p>
    </blockquote>
    <ul>
     <li>
      <strong>
       优点
      </strong>
      ：实现简单，适用于单个连接。
     </li>
     <li>
      <strong>
       缺点
      </strong>
      ：等待期间 CPU 无法做别的事，
      <strong>
       如果有多个连接，性能极差
      </strong>
      。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="2__read_13">
     </a>
     <strong>
      2. 轮询
      <code>
       read()
      </code>
     </strong>
    </h4>
    <ul>
     <li>
      <strong>
       非阻塞
       <code>
        read()
       </code>
       + 轮询
      </strong>
      就是
      <strong>
       你每秒钟打一次电话问“饭好了吗？”
      </strong>
      ，如果没好就挂断，等 1 秒再打一次，重复操作，浪费时间和精力。
     </li>
    </ul>
    <blockquote>
     <p>
      你每秒钟打一次电话问“饭好了吗？”
      <br/>
      <strong>
       如果没好，就挂断，过 1 秒再打一次
      </strong>
      ，一直循环到饭做好。
      <br/>
      <strong>
       （非阻塞 + 轮询）
      </strong>
     </p>
    </blockquote>
    <ul>
     <li>
      <strong>
       优点
      </strong>
      ：不会完全卡住，但 CPU 仍然会被轮询占用。
     </li>
     <li>
      <strong>
       缺点
      </strong>
      ：
      <strong>
       即使没有饭做好，你还是要不停地打电话（轮询
       <code>
        read()
       </code>
       ），CPU 资源浪费严重
      </strong>
      。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="3_select__poll_24">
     </a>
     <strong>
      3.
      <code>
       select()
      </code>
      /
      <code>
       poll()
      </code>
     </strong>
    </h4>
    <blockquote>
     <p>
      你在餐厅前台登记了一个电话，并告诉服务员：
      <br/>
      <strong>
       “有饭做好了，打电话通知我。”
      </strong>
      <br/>
      服务员真的通知你了，但他说：
      <br/>
      <strong>
       “有几道菜做好了，你自己去检查是哪道菜。”
      </strong>
      <br/>
      <strong>
       （你必须自己挨个检查）
      </strong>
     </p>
    </blockquote>
    <ul>
     <li>
      <strong>
       优点
      </strong>
      ：不用一直打电话（轮询），CPU 资源消耗减少。
     </li>
     <li>
      <strong>
       缺点
      </strong>
      ：
      <strong>
       仍然要遍历所有连接（菜品）检查哪些是好的，连接数量大时很慢
      </strong>
      。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="4_epoll_36">
     </a>
     <strong>
      4.
      <code>
       epoll()
      </code>
     </strong>
    </h4>
    <blockquote>
     <p>
      你用高科技 GPS 追踪器，每道菜都装上一个传感器，
      <strong>
       只要菜做好了，GPS 直接告诉你“5 号菜、8 号菜已经完成！”
      </strong>
      <br/>
      <strong>
       你不用自己挨个检查，而是直接处理完成的菜品。
      </strong>
     </p>
    </blockquote>
    <ul>
     <li>
      <strong>
       优点
      </strong>
      ：只有有数据的连接才会被通知，不用遍历所有 socket，
      <strong>
       处理速度是 O(1)
      </strong>
      。
     </li>
     <li>
      <strong>
       缺点
      </strong>
      ：Linux 独有，Windows 不支持（Windows 用
      <code>
       IOCP
      </code>
      代替
      <code>
       epoll
      </code>
      ）。
     </li>
    </ul>
    <hr/>
    <h3>
     <a id="_45">
     </a>
     <strong>
      最终结论
     </strong>
    </h3>
    <table>
     <thead>
      <tr>
       <th>
        <strong>
         方式
        </strong>
       </th>
       <th>
        <strong>
         比喻
        </strong>
       </th>
       <th>
        <strong>
         效率
        </strong>
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         阻塞
         <code>
          read()
         </code>
        </strong>
       </td>
       <td>
        你打电话问饭好了吗？没人接就一直等
       </td>
       <td>
        <strong>
         低，卡住 CPU
        </strong>
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         轮询
         <code>
          read()
         </code>
        </strong>
       </td>
       <td>
        你每秒打一次电话问“饭好了吗？”
       </td>
       <td>
        <strong>
         很低，CPU 白白浪费
        </strong>
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         <code>
          select()
         </code>
         /
         <code>
          poll()
         </code>
        </strong>
       </td>
       <td>
        你让服务员通知你，但仍然要自己检查每道菜
       </td>
       <td>
        <strong>
         中等，仍然要遍历所有连接
        </strong>
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         <code>
          epoll()
         </code>
        </strong>
       </td>
       <td>
        你用 GPS 追踪，服务员直接告诉你哪道菜好了
       </td>
       <td>
        <strong>
         高效，适用于大规模高并发
        </strong>
       </td>
      </tr>
     </tbody>
    </table>
    <p>
     所以，在写高性能 HTTP 服务器时，
     <code>
      epoll
     </code>
     是最佳选择。
    </p>
