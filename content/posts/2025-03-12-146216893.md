    <h3>
     <a id="_23">
     </a>
     框架的概念与理解
    </h3>
    <p>
     框架（Framework）是一个集成了基本结构、规范、设计模式、编程语言和程序库等基础组件的软件系统，它可以用来构建更高级别的应用程序。框架的设计和实现旨在解决特定领域中的常见问题，帮助开发人员更高效、更稳定地实现软件开发目标。
    </p>
    <p>
     框架的优点包括以下几点：
    </p>
    <ul>
     <li>
      <strong>
       提高开发效率
      </strong>
      ：框架提供了许多预先设计好了的组件和工具，能够帮助开发人员快速进行开发。相较于传统手写代码，在框架提供的规范化环境中，开发者可以更快地实现项目的各种要求。
     </li>
     <li>
      <strong>
       降低开发成本
      </strong>
      ：框架的提供标准化的编程语言、数据操作等代码片段，避免了重复开发的问题，降低了开发成本，提供深度优化的系统，降低了维护成本，增强了系统的可靠性。
     </li>
     <li>
      <strong>
       提高应用程序的稳定性
      </strong>
      ：框架通常经过了很长时间的开发和测试，其中的许多组件、代码片段和设计模式都得到了验证。重复利用这些组件有助于减少 bug 的出现，从而提高了应用程序的稳定性。
     </li>
     <li>
      <strong>
       提供标准化的解决方案
      </strong>
      ：框架通常是针对某个特定领域的，通过提供标准化的解决方案，可以为开发人员提供一种共同的语言和思想基础，有助于更好地沟通和协作。
     </li>
    </ul>
    <p>
     框架的缺点包括以下几个方面：
    </p>
    <ul>
     <li>
      <strong>
       学习成本高
      </strong>
      ：框架通常具有特定的语言和编程范式。对于开发人员而言，需要花费时间学习其背后的架构、模式和逻辑，这对于新手而言可能会耗费较长时间。
     </li>
     <li>
      <strong>
       可能存在局限性
      </strong>
      ：虽然框架提高了开发效率并可以帮助开发人员解决常见问题，但是在某些情况下，特定的应用需求可能超出框架的范围，从而导致应用程序无法满足要求。开发人员可能需要更多的控制权和自由度，同时需要在框架和应用程序之间进行权衡取舍。
     </li>
     <li>
      <strong>
       版本变更和兼容性问题
      </strong>
      ：框架的版本发布和迭代通常会导致代码库的大规模变更，进而导致应用程序出现兼容性问题和漏洞。当框架变更时，需要考虑框架是否向下兼容，以及如何进行适当的测试、迁移和升级。
     </li>
     <li>
      <strong>
       架构风险
      </strong>
      ：框架涉及到很多抽象和概念，如果开发者没有足够的理解和掌握其架构，可能会导致系统出现设计和架构缺陷，从而影响系统的健康性和安全性。
     </li>
    </ul>
    <p>
     站在文件结构的角度理解框架，可以将框架总结：
     <strong>
      框架 = jar包 + 配置文件
     </strong>
    </p>

    <h2>
     <a id="Spring__48">
     </a>
     Spring 框架介绍
    </h2>
    <p>
     在学习 Spring 框架之前，我们需要搞懂 Spring 和 Spring Framework 之前的区别。
    </p>
    <p>
     实际上，Spring 框架和 Spring Framework 实际上是同一个概念的不同表达方式。Spring Framework 是 Spring 框架的官方名称。
    </p>
    <blockquote>
     <p>
      <strong>
       广义的 Spring：Spring 技术栈
      </strong>
      （全家桶）
     </p>
    </blockquote>
    <p>
     广义上的 Spring 泛指以 Spring Framework 为基础的 Spring 技术栈。
    </p>
    <p>
     经过十多年的发展，Spring 已经不再是一个单纯的应用框架，而是逐渐发展成为一个由多个不同子项目（模块）组成的成熟技术，例如 Spring Framework、Spring MVC、SpringBoot、Spring Cloud、Spring Data、Spring Security 等，其中 Spring Framework 是其他子项目的基础。
    </p>
    <p>
     这些子项目涵盖了从企业级应用开发到云计算等各方面的内容，能够帮助开发人员解决软件发展过程中不断产生的各种实际问题，给开发人员带来了更好的开发体验。
    </p>
    <blockquote>
     <p>
      <strong>
       狭义的 Spring：Spring Framework
      </strong>
      （基础框架）
     </p>
    </blockquote>
    <p>
     狭义的 Spring 特指 Spring Framework，通常我们将它称为 Spring 框架。
    </p>
    <p>
     Spring Framework（Spring 框架）是一个开源的应用程序框架，由 Pivotal 公司（现为 VMware 旗下）开发，最初是为了解决企业级开发中各种常见问题而创建的。它提供了很多功能，例如：依赖注入（Dependency Injection）、面向切面编程（AOP）、声明式事务管理（TX）等。其主要目标是使企业级应用程序的开发变得更加简单和快速，并且 Spring 框架被广泛应用于 Java 企业开发领域。
    </p>
    <p>
     Spring 全家桶的其他框架都是以 Spring Framework 框架为基础。
    </p>

    <h3>
     <a id="Spring__89">
     </a>
     Spring 的主要特点
    </h3>
    <ol>
     <li>
      <strong>
       轻量级和非侵入性
      </strong>
      ：
      <ul>
       <li>
        Spring 框架是轻量级的，它的核心容器不需要依赖任何特定的应用服务器。
       </li>
       <li>
        Spring 的应用代码不依赖于 Spring 的特定类和接口，这使得应用代码更加干净和可移植。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       依赖注入（DI）
      </strong>
      ：
      <ul>
       <li>
        Spring 通过依赖注入机制来管理对象之间的依赖关系。开发者只需要定义好对象的依赖关系，Spring 容器会自动完成对象的创建和依赖注入。
       </li>
       <li>
        这种方式减少了代码的耦合性，使得代码更加模块化和易于测试。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       面向切面编程（AOP）
      </strong>
      ：
      <ul>
       <li>
        Spring 提供了强大的 AOP 支持，允许开发者将横切关注点（如日志、事务管理、安全性等）从业务逻辑中分离出来。
       </li>
       <li>
        通过 AOP，开发者可以更好地实现关注点分离，提高代码的可维护性和可重用性。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       事务管理
      </strong>
      ：
      <ul>
       <li>
        Spring 提供了一个一致的事务管理抽象，支持编程式和声明式事务管理。
       </li>
       <li>
        开发者可以通过简单的配置来管理事务，而不需要直接处理底层的事务 API。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       MVC 框架
      </strong>
      ：
      <ul>
       <li>
        Spring MVC 是 Spring 框架中的一个模块，用于构建基于 MVC 设计模式的 Web 应用程序。
       </li>
       <li>
        它提供了灵活的配置和强大的数据绑定、验证、国际化等功能。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       集成支持
      </strong>
      ：
      <ul>
       <li>
        Spring 框架提供了与多种其他技术和框架的集成支持，如 Hibernate、JPA、JMS、Quartz、JDBC 等。
       </li>
       <li>
        这使得开发者可以轻松地将 Spring 与其他技术栈结合使用。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       测试支持
      </strong>
      ：
      <ul>
       <li>
        Spring 提供了强大的测试支持，开发者可以轻松地进行单元测试和集成测试。
       </li>
       <li>
        Spring 的测试框架支持 JUnit 和 TestNG，并且提供了 Mock 对象和上下文配置的支持。
       </li>
      </ul>
     </li>
    </ol>
    <h3>
     <a id="Spring__115">
     </a>
     Spring 的优势
    </h3>
    <ol>
     <li>
      丰富的生态系统：Spring 生态系统非常丰富，支持许多模块和库，如 Spring Boot、Spring Security、Spring Cloud 等等，可以帮助开发人员快速构建高可靠性的企业应用程序。
     </li>
     <li>
      模块化的设计：框架组件之间的松散耦合和模块化设计使得 Spring 具有良好的可重用性、可扩展性和可维护性。开发人员可以轻松地选择自己需要的模块，根据自己的需求进行开发。
     </li>
     <li>
      简化 Java 开发：Spring 简化了 Java 开发，提供了各种工具和 API，可以降低开发复杂度和学习成本。同时，Spring 支持各种应用场景，包括 Web 应用程序、RESTful API、消息传递、批处理等等。
     </li>
     <li>
      不断创新和发展：Spring 开发团队一直在不断创新和发展，保持与最新技术的接轨，为开发人员提供更加先进和优秀的工具和框架。
     </li>
    </ol>
    <p>
     因此，这些优点使得 Spring 成为了一个稳定、可靠、且创新的框架，为企业级 Java 开发提供了一站式的解决方案。
    </p>
    <p>
     Spring 使创建 Java 企业应用程序变得容易。它提供了在企业环境中采用 Java 语言所需的一切，支持 Groovy 和 Kotlin 作为 JVM 上的替代语言，并且可以根据应用程序的需求灵活地创建多种架构。从Spring 6.0.6开始，Spring 需要 Java 17+。
    </p>
    <h3>
     <a id="Spring__128">
     </a>
     Spring 的两大核心
    </h3>
    <p>
     <strong>
      Spring 框架的两大核心：Spring IoC（控制反转）和 Spring AOP（面向切面编程）。
     </strong>
    </p>
    <h2>
     <a id="Spring_IoC__134">
     </a>
     Spring IoC 框架和核心概念
    </h2>
    <p>
     Spring IoC（Inversion of Control，控制反转）是 Spring 框架的核心特性之一，也是 Spring 实现依赖注入（Dependency Injection，DI）的基础。IoC 是一种设计原则，它将对象的创建、依赖关系的管理以及生命周期控制从应用程序代码中转移到框架或容器中，从而降低了代码的耦合性，提高了代码的灵活性和可维护性。
    </p>
    <h3>
     <a id="_140">
     </a>
     组件和组件管理
    </h3>
    <h4>
     <a id="_142">
     </a>
     什么是组件
    </h4>
    <p>
     <strong>
      组件（Component）
     </strong>
     是软件系统中可独立部署、可替换的功能单元。它通常是一个封装了特定功能或行为的模块，具有清晰的接口和明确的职责。组件可以是类、模块、服务、库等，具体取决于上下文。
    </p>
    <h4>
     <a id="_148">
     </a>
     组件的特点
    </h4>
    <ul>
     <li>
      <strong>
       独立性
      </strong>
      ：组件可以独立开发、测试和部署。
     </li>
     <li>
      <strong>
       可复用性
      </strong>
      ：组件可以在不同的项目或系统中重复使用。
     </li>
     <li>
      <strong>
       接口清晰
      </strong>
      ：组件通过定义良好的接口与外部交互，隐藏内部实现细节。
     </li>
     <li>
      <strong>
       职责单一
      </strong>
      ：每个组件通常只负责一个特定的功能或任务。
     </li>
    </ul>
    <p>
     简单来说：组件是映射到应用程序中所有可重用组件的 Java 对象，应该是可复用的功能对象！
    </p>
    <p>
     注意：
     <strong>
      组件一定是对象，但对象不一定是组件
     </strong>
     。
    </p>
    <h4>
     <a id="_161">
     </a>
     组件的例子
    </h4>
    <ul>
     <li>
      在 Spring 框架中，一个
      <code>
       Service
      </code>
      类、
      <code>
       Repository
      </code>
      类或
      <code>
       Controller
      </code>
      类都可以被视为组件。
     </li>
     <li>
      在前端开发中，一个按钮、表单或导航栏也可以被视为组件。
     </li>
    </ul>
    <p>
     整个项目其实是由一个个组件组成的：
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/5cfc16a0d3134899ae317204264adc75.png#pic_center"/>
    </p>
    <h4>
     <a id="_171">
     </a>
     组件管理
    </h4>
    <p>
     <strong>
      组件管理（Component Management）
     </strong>
     是指对组件的创建、配置、依赖注入、生命周期管理以及组织的过程。组件管理的目标是确保组件能够高效、灵活地协同工作，同时降低系统的复杂性和耦合性。
    </p>
    <p>
     在 Spring 框架中，组件管理是通过
     <strong>
      IoC 容器
     </strong>
     来实现的。Spring 容器负责创建组件，管理它们的依赖关系，并控制它们的生命周期。
    </p>
    <p>
     在 Spring 框架中，组件通常被称为
     <strong>
      Bean
     </strong>
     。
    </p>
    <h4>
     <a id="_Spring__181">
     </a>
     组件由 Spring 管理优势
    </h4>
    <ul>
     <li>
      降低了组件之间的耦合性：Spring IoC 容器通过依赖注入机制，将组件之间的依赖关系削弱，减少了程序组件之间的耦合性，使得组件更加松散地耦合。
     </li>
     <li>
      提高了代码的可重用性和可维护性：将组件的实例化过程、依赖关系的管理等功能交给 Spring IoC 容器处理，使得组件代码更加模块化、可重用、更易于维护。
     </li>
     <li>
      方便了配置和管理：Spring IoC 容器通过 xml 文件或者注解，轻松的对组件进行配置和管理，使得组件的切换、替换等操作更加的方便和快捷。
     </li>
     <li>
      交给 Spring 管理的对象（组件），方可享受 Spring 框架的其他功能（AOP、声明事务管理）等。
     </li>
    </ul>
    <h3>
     <a id="_190">
     </a>
     普通容器和复杂容器
    </h3>
    <h4>
     <a id="_192">
     </a>
     普通容器
    </h4>
    <p>
     普通容器只能用来存储数据，没有其它功能。
    </p>
    <p>

    <h3>
     <a id="Spring_IoC__225">
     </a>
     Spring IoC 容器
    </h3>
    <p>
     IoC 的核心思想是
     <strong>
      将控制权从应用程序代码反转给框架或容器
     </strong>
     。在传统的编程模式中，对象的创建和依赖关系的管理是由开发者手动完成的，而在 IoC 模式下，这些工作由 Spring 容器负责。
    </p>
    <p>
     Spring IoC 容器，负责实例化、配置和组装 bean（组件）。容器通过读取配置元数据来获取有关要实例化、配置和组装组件的指令。配置元数据以
     <strong>
      xml、Java 注解或 Java 代码
     </strong>
     形式表现。它允许表达组成应用程序的组件以及这些组件之间丰富的相互依赖关系。
    </p>
    <h4>
     <a id="IoC__233">
     </a>
     IoC 容器实现
    </h4>
    <p>
     Spring IoC 容器是 Spring 框架的核心组件，负责管理对象的生命周期和依赖关系。Spring 提供了两种主要的 IoC 容器实现：
    </p>
    <ol>
     <li>
      <strong>
       <code>
        BeanFactory
       </code>
      </strong>
      ：
      <ul>
       <li>
        是 Spring 最基础的容器，提供了基本的依赖注入功能。
       </li>
       <li>
        适合资源受限的环境，延迟加载 Bean（只有在需要时才创建 Bean）。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       <code>
        ApplicationContext
       </code>
      </strong>
      ：
      <ul>
       <li>
        是
        <code>
         BeanFactory
        </code>
        的扩展，提供了更多的企业级功能，如国际化、事件传播、AOP 支持等。
       </li>
       <li>
        在应用启动时就会初始化所有的 Bean，适合大多数应用场景。
       </li>
      </ul>
     </li>
    </ol>
    <p>
     简而言之，
     <code>
      BeanFactory
     </code>
     提供了配置框架和基本功能，而
     <code>
      ApplicationContext
     </code>
     添加了更多特定于企业的功能。
     <code>
      ApplicationContext
     </code>
     是
     <code>
      BeanFactory
     </code>
     的完整超集。
    </p>
    <p>
     再往下，
     <code>
      ApplicationContext
     </code>
     主要有4个容器实现类：
    </p>
    <table>

       </th>
      </tr>

       <td>
        <code>
         ClassPathXmlApplicationContext
        </code>
       </td>
       <td>

       </td>
      </tr>
      <tr>
       <td>
        <code>
         FileSystemXmlApplicationContext
        </code>
       </td>
       <td>
        通过文件系统路径读取 XML 格式的配置文件创建 IoC 容器对象
       </td>
      </tr>
      <tr>
       <td>
        <code>
         AnnotationConfigApplicationContext
        </code>
       </td>
       <td>
        通过读取 Java 配置类创建 IoC 容器对象
       </td>
      </tr>
      <tr>
       <td>
        <code>

        </code>
       </td>

    <h4>
     <a id="IoC__257">
     </a>
     IoC 的工作机制
    </h4>
    <p>
     Spring IoC 容器通过以下步骤实现对象的创建和依赖注入：
    </p>
    <ol>
     <li>
      <strong>
       配置元数据
      </strong>
      ：
      <ul>
       <li>
        Spring 容器需要知道如何创建和管理对象，这些信息通过配置元数据提供。
       </li>
       <li>
        配置元数据可以通过 XML 文件、Java 注解或 Java 代码（基于配置类）来定义。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       实例化 Bean
      </strong>
      ：
      <ul>
       <li>
        容器根据配置元数据创建 Bean 的实例。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       依赖注入
      </strong>
      ：
      <ul>
       <li>
        容器根据 Bean 之间的依赖关系，将所需的依赖注入到 Bean 中。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       管理 Bean 的生命周期
      </strong>
      ：
      <ul>
       <li>
        容器负责 Bean 的初始化、使用和销毁。
       </li>
      </ul>
     </li>
    </ol>

    <h3>
     <a id="IoC__DI_423">
     </a>
     IoC 和 DI
    </h3>
    <p>
     IoC 主要是针对对象的创建和调用控制而言的，也就是说，当应用程序需要使用一个对象时，不再是应用程序直接创建该对象，而是由 IoC 容器来创建和管理，即控制权由应用程序转移到 IoC 容器中，也就是“反转”了控制权。这种方式基本上是通过依赖查找的方式来实现的，即 IoC 容器维护着构成应用程序的对象，并负责创建这些对象。
    </p>
    <blockquote>
     <p>
      传统模式 vs IoC 模式
     </p>
    </blockquote>
    <ul>
     <li>
      <p>
       <strong>
        传统模式
       </strong>
       ：
       <br/>
       在传统编程模式中，对象的创建和依赖关系的管理是由开发者手动完成的。例如：
      </p>
      <pre><code class="prism language-java"><span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">UserService</span> <span class="token punctuation">{<!-- --></span>
    <span class="token keyword">private</span> <span class="token class-name">UserRepository</span> userRepository <span class="token operator">=</span> <span class="token keyword">new</span> <span class="token class-name">UserRepository</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">;</span> <span class="token comment">// 手动创建依赖对象</span>
<span class="token punctuation">}</span>
</code></pre>
      <p>
       这种方式的缺点是代码耦合性高，难以测试和维护。
      </p>
     </li>
     <li>
      <p>
       <strong>
        IoC 模式
       </strong>
       ：
       <br/>
       在 IoC 模式下，对象的创建和依赖关系的管理由框架或容器负责。例如：
      </p>
      <pre><code class="prism language-java"><span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">UserService</span> <span class="token punctuation">{<!-- --></span>
    <span class="token keyword">private</span> <span class="token class-name">UserRepository</span> userRepository<span class="token punctuation">;</span>

    <span class="token comment">// 通过构造函数注入依赖</span>
    <span class="token keyword">public</span> <span class="token class-name">UserService</span><span class="token punctuation">(</span><span class="token class-name">UserRepository</span> userRepository<span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
        <span class="token keyword">this</span><span class="token punctuation">.</span>userRepository <span class="token operator">=</span> userRepository<span class="token punctuation">;</span>
    <span class="token punctuation">}</span>
<span class="token punctuation">}</span>
</code></pre>
      <p>
       这种方式降低了代码的耦合性，提高了灵活性和可维护性。
      </p>
     </li>
    </ul>
    <blockquote>
     <p>
      IoC 的优点
     </p>
    </blockquote>
    <ul>
     <li>
      <strong>
       降低耦合性
      </strong>
      ：对象之间的依赖关系由容器管理，代码更加松耦合。
     </li>
     <li>
      <strong>
       提高可测试性
      </strong>
      ：依赖注入使得单元测试更加容易，可以通过 Mock 对象替换真实依赖。
     </li>
     <li>
      <strong>
       增强灵活性
      </strong>
      ：通过配置可以动态调整对象的创建和依赖关系。
     </li>
    </ul>
    <p>
     DI 是 IoC 的一种实现方式，是指在组件之间传递依赖关系的过程中，将依赖关系在容器内部进行处理，这样就不必在应用程序代码中硬编码对象之间的依赖关系，实现了对象之间的解耦合。
    </p>
    <p>
     依赖注入的方式包括构造函数注入、Setter 注入和字段注入。
    </p>
    <ol>
     <li>
      <p>
       <strong>
        构造函数注入
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         通过构造函数注入依赖。
        </p>
       </li>
       <li>
        <p>
         示例：
        </p>
        <pre><code class="prism language-java"><span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">UserService</span> <span class="token punctuation">{<!-- --></span>
    <span class="token keyword">private</span> <span class="token keyword">final</span> <span class="token class-name">UserRepository</span> userRepository<span class="token punctuation">;</span>

    <span class="token keyword">public</span> <span class="token class-name">UserService</span><span class="token punctuation">(</span><span class="token class-name">UserRepository</span> userRepository<span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
        <span class="token keyword">this</span><span class="token punctuation">.</span>userRepository <span class="token operator">=</span> userRepository<span class="token punctuation">;</span>
    <span class="token punctuation">}</span>

</code></pre>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        Setter 注入
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         通过 Setter 方法注入依赖。
        </p>
       </li>
       <li>
        <p>
         示例：
        </p>
        <pre><code class="prism language-java"><span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">UserService</span> <span class="token punctuation">{<!-- --></span>

        <span class="token keyword">this</span><span class="token punctuation">.</span>userRepository <span class="token operator">=</span> userRepository<span class="token punctuation">;</span>
    <span class="token punctuation">}</span>

      </ul>
     </li>

      <p>
       <strong>
        字段注入
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         通过字段直接注入依赖。
        </p>
       </li>
       <li>
        <p>

        </p>
        <pre><code class="prism language-java"><span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">UserService</span> <span class="token punctuation">{<!-- --></span>

       </li>
      </ul>
     </li>
    </ol>
    <blockquote>
     <p>
      DI 的优点
     </p>
    </blockquote>
    <ul>
     <li>
      <strong>
       解耦
      </strong>
      ：对象不需要知道如何创建或查找依赖，只需声明依赖关系。
     </li>
     <li>
      <strong>
       可测试性
      </strong>
      ：依赖注入使得单元测试更加容易，可以通过 Mock 对象替换真实依赖。
     </li>
     <li>
      <strong>
       可维护性
      </strong>
      ：依赖关系集中管理，便于修改和扩展。
     </li>
    </ul>
    <h2>
     <a id="IoC__519">
     </a>
     IoC 实践
    </h2>
    <h3>
     <a id="IoC__521">
     </a>
     IoC 实现的几个步骤
    </h3>
    <ol>
     <li>
      <p>
       <strong>
        定义组件（Bean）
       </strong>
       ：
      </p>
      <p>
       在 Spring 中，组件通常被称为
       <strong>
        Bean
       </strong>
       。首先需要定义哪些类需要由 Spring 容器管理。
      </p>
      <blockquote>
       <p>
        定义方式：
       </p>
      </blockquote>
      <ul>
       <li>
        <strong>
         XML 配置
        </strong>
        ：在 XML 文件中定义 Bean。
       </li>
       <li>
        <strong>
         注解配置
        </strong>
        ：使用
        <code>
         @Component
        </code>
        、
        <code>
         @Service
        </code>
        、
        <code>
         @Repository
        </code>
        、
        <code>
         @Controller
        </code>
        等注解标记类。
       </li>
       <li>
        <strong>
         Java 配置类
        </strong>
        ：使用
        <code>
         @Bean
        </code>
        注解在配置类中定义 Bean。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        配置元数据
       </strong>
       ：
      </p>
      <p>
       Spring 容器需要知道如何创建和管理 Bean，这些信息通过配置元数据提供。
      </p>
     </li>
     <li>
      <p>
       <strong>
        初始化 IoC 容器
       </strong>
       ：
      </p>
      <p>
       Spring 容器负责管理 Bean 的创建和依赖注入。常见的 IoC 容器实现包括
       <code>
        ClassPathXmlApplicationContext
       </code>
       、
       <code>
        AnnotationConfigApplicationContext
       </code>
       等。
      </p>
     </li>
     <li>
      <p>
       <strong>
        实例化 Bean
       </strong>
       ：
      </p>
      <p>
       Spring 容器根据配置元数据创建 Bean 的实例。
      </p>
      <blockquote>
       <p>
        实例化过程：
       </p>
      </blockquote>
      <ol>
       <li>
        容器读取配置元数据。
       </li>
       <li>
        根据配置创建 Bean 的实例。
       </li>
       <li>
        如果 Bean 有依赖关系，容器会递归创建依赖的 Bean。
       </li>
      </ol>
     </li>
     <li>
      <p>
       <strong>
        依赖注入
       </strong>
       ：
      </p>
     </li>
    </ol>
    <p>
     ​ Spring 容器根据 Bean 之间的依赖关系，将所需的依赖注入到 Bean 中。
    </p>
    <ol start="6">
     <li>
      <p>
       <strong>
        管理 Bean 的生命周期
       </strong>
       ：
      </p>
      <p>
       Spring 容器负责管理 Bean 的整个生命周期，包括初始化、使用和销毁。
      </p>
      <blockquote>
       <p>
        生命周期阶段：
       </p>
      </blockquote>
      <ol>
       <li>
        <strong>
         实例化
        </strong>
        ：容器创建 Bean 的实例。
       </li>
       <li>
        <strong>
         属性赋值
        </strong>
        ：容器注入 Bean 的依赖。
       </li>
       <li>
        <strong>
         初始化
        </strong>
        ：调用 Bean 的初始化方法（如
        <code>
         @PostConstruct
        </code>
        或
        <code>
         init-method
        </code>
        ）。
       </li>
       <li>
        <strong>
         使用
        </strong>
        ：Bean 可以被应用程序使用。
       </li>
       <li>
        <strong>
         销毁
        </strong>
        ：容器关闭时，调用 Bean 的销毁方法（如
        <code>
         @PreDestroy
        </code>
        或
        <code>
         destroy-method
        </code>
        ）。
       </li>
      </ol>
     </li>
     <li>
      <p>
       <strong>
        使用 Bean
       </strong>
       ：
      </p>
      <p>
       通过 Spring 容器获取 Bean 并使用。
      </p>
     </li>
    </ol>
    <h3>

    </h6>
    <p>
     <code>
      &lt;bean&gt;
     </code>
     标签声明 bean，只是将 bean 的信息配置给 Spring IoC 容器。
    </p>
    <p>
     在 IoC 容器中，这些
     <code>
      &lt;bean&gt;
     </code>
     标签对应的信息转成 Spring 内部
     <code>
      BeanDefinition
     </code>
     对象，
     <code>
      BeanDefinition
     </code>
     对象内，包含定义的信息（id、class、属性等等）。
    </p>
    <p>
     这意味着，
     <code>
      BeanDefinition
     </code>
     与
     <strong>
      类
     </strong>
     概念一样，Spring IoC 容器可以可以根据
     <code>
      BeanDefinition
     </code>
     对象反射创建多个 bean 对象实例。
    </p>
    <p>
     具体创建多少个 bean 的实例对象，由 bean 的作用域 Scope 属性指定。
    </p>

    <h4>
     <a id="FactoryBean__1283">
     </a>
     FactoryBean 的特性及使用
    </h4>
    <p>
     <code>
      FactoryBean
     </code>
     是 Spring 框架中的一个特殊接口，用于创建复杂的 Bean 实例。与普通的 Bean 不同，
     <code>
      FactoryBean
     </code>
     本身是一个 Bean，但它负责创建和管理另一个 Bean 实例。通过
     <code>
      FactoryBean
     </code>
     ，开发者可以自定义 Bean 的创建逻辑，从而实现更灵活的对象管理。
    </p>
    <ol>
     <li>
      <strong>
       双重身份
      </strong>
     </li>
    </ol>
    <ul>
     <li>
      <code>
       FactoryBean
      </code>
      本身是一个 Bean，由 Spring 容器管理。
     </li>
     <li>
      它负责创建另一个 Bean 实例（称为目标 Bean）。
     </li>
    </ul>
    <ol start="2">
     <li>
      <strong>
       延迟创建
      </strong>
     </li>
    </ol>
    <ul>
     <li>
      目标 Bean 的创建可以延迟到第一次使用时，而不是在容器启动时立即创建。
     </li>
    </ul>
    <ol start="3">
     <li>
      <strong>
       复杂对象的创建
      </strong>
     </li>
    </ol>
    <ul>
     <li>
      <code>
       FactoryBean
      </code>
      适用于创建复杂的对象，例如：
      <ul>
       <li>
        需要动态生成的对象。
       </li>
       <li>
        需要依赖外部资源的对象（如数据库连接、线程池等）。
       </li>
       <li>
        需要代理或装饰的对象。
       </li>
      </ul>
     </li>
    </ul>
    <ol start="4">
     <li>

      </strong>
     </li>
    </ol>
    <ul>
     <li>
      通过
      <code>

      </code>

      <code>
       FactoryBean
      </code>

      </code>

     </li>

    <h5>
     <a id="FactoryBean__1309">
     </a>
     FactoryBean 的核心方法
    </h5>
    <p>
     <code>
      FactoryBean
     </code>
     接口定义了以下三个方法：
    </p>
    <ol>
     <li>
      <strong>
       <code>
        T getObject()
       </code>
      </strong>
     </li>
    </ol>
    <ul>
     <li>
      返回由
      <code>
       FactoryBean
      </code>
      创建的目标 Bean 实例。
     </li>
     <li>
      这是
      <code>
       FactoryBean
      </code>
      的核心方法。
     </li>
    </ul>
    <ol start="2">
     <li>
      <strong>
       <code>
        Class&lt;?&gt; getObjectType()
       </code>
      </strong>
     </li>
    </ol>
    <ul>
     <li>
      返回目标 Bean 的类型。
     </li>
     <li>
      如果类型未知，可以返回
      <code>
       null
      </code>
      。
     </li>
    </ul>
    <ol start="3">
     <li>
      <strong>
       <code>
        boolean isSingleton()
       </code>
      </strong>
     </li>
    </ol>
    <ul>
     <li>
      返回目标 Bean 是否是单例。
     </li>
     <li>
      如果返回
      <code>
       true
      </code>
      ，表示目标 Bean 是单例；如果返回
      <code>
       false
      </code>
      ，表示目标 Bean 是原型（每次请求都会创建一个新实例）。
     </li>
    </ul>

    <h5>
     <a id="FactoryBean__BeanFactory__1430">
     </a>
     FactoryBean 和 BeanFactory 区别
    </h5>
    <p>
     **
     <code>
      FactoryBean
     </code>
     **是 Spring 中一种特殊的 bean，可以在
     <code>
      getObject()
     </code>
     工厂方法自定义的逻辑创建 Bean！是一种能够生产其他 Bean 的 Bean。
     <code>
      FactoryBean
     </code>
     在容器启动时被创建，而在实际使用时则是通过调用
     <code>
      getObject()
     </code>
     方法来得到其所生产的 Bean。因此，
     <code>
      FactoryBean
     </code>
     可以自定义任何所需的初始化逻辑，生产出一些定制化的 bean。
    </p>
    <p>
     一般情况下，整合第三方框架，都是通过定义
     <code>
      FactoryBean
     </code>
     实现！！！
    </p>
    <p>
     <strong>
      <code>
       BeanFactory
      </code>
     </strong>
     是 Spring 框架的基础，其作为一个顶级接口定义了容器的基本行为，例如管理 bean 的生命周期、配置文件的加载和解析、bean 的装配和依赖注入等。
     <code>
      BeanFactory
     </code>
     接口提供了访问 bean 的方式，例如
     <code>
      getBean()
     </code>
     方法获取指定的 bean 实例。它可以从不同的来源（例如 Mysql 数据库、XML 文件、Java 配置类等）获取 bean 定义，并将其转换为 bean 实例。同时，
     <code>
      BeanFactory
     </code>
     还包含很多子类（例如，
     <code>
      ApplicationContext
     </code>
     接口）提供了额外的强大功能。
    </p>
    <p>
     总的来说，
     <code>
      FactoryBean
     </code>
     和
     <code>
      BeanFactory
     </code>
     的区别主要在于前者是用于创建 bean 的接口，它提供了更加灵活的初始化定制功能，而后者是用于管理 bean 的框架基础接口，提供了基本的容器功能和 bean 生命周期管理。
    </p>

    <ol>
     <li>
      <p>
       <code>
        @Scope
       </code>
      </p>
      <ul>
       <li>
        <strong>
         作用
        </strong>
        ：用于定义 Bean 的作用域，不同的作用域会影响 Bean 的生命周期。
       </li>
       <li>
        <strong>
         Singleton（默认）
        </strong>
        ：Bean 在 Spring 容器中只有一个实例，生命周期与容器一致。
       </li>
       <li>
        <strong>
         Prototype
        </strong>
        ：每次请求时都会创建一个新的 Bean 实例，Spring 不管理其销毁。
       </li>
       <li>
        <strong>
         Request / Session / Application
        </strong>
        ：用于 Web 应用，分别对应请求、会话和应用级别的作用域。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <code>
        @PostConstruct
       </code>
       ：也可使用
       <code>
        @Bean
       </code>
       注解的
       <code>
        initMethod
       </code>
       属性指定。
      </p>
      <ul>
       <li>
        <strong>
         作用
        </strong>
        ：标记一个方法为 Bean 初始化后的回调方法。该方法会在 Bean 的依赖注入完成后执行。
       </li>
       <li>
        <strong>
         使用场景
        </strong>
        ：执行初始化逻辑，如数据加载、资源初始化等。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <code>
        @PreDestroy
       </code>
       ：也可使用
       <code>
        @Bean
       </code>
       注解的
       <code>
        destroyMethod
       </code>
       属性指定。
      </p>
      <ul>
       <li>
        <strong>
         作用
        </strong>
        ：标记一个方法为 Bean 销毁前的回调方法。该方法会在 Bean 被销毁之前执行。
       </li>
       <li>
        <strong>
         使用场景
        </strong>
        ：执行清理逻辑，如释放资源、关闭连接等。
       </li>
      </ul>
     </li>
    </ol>

      @Controller
     </code>
     、
     <code>
      @Service
     </code>
     、
     <code>
      @Repository
     </code>

    <blockquote>
     <p>
      关于 bean id 问题
     </p>
    </blockquote>
    <p>

    </p>
    <p>
     默认情况：
    </p>
    <p>
     类名首字母小写就是 bean 的 id。例如：
     <code>
      UserController
     </code>
     类对应的 bean 的 id 就是
     <code>
      userController
     </code>
     。

              使用
              <code>
               @Qualifier
              </code>
              注解：根据
              <code>
               @Qualifier
              </code>
              注解中指定的名称作为 bean 的id进行匹配
              <ul>
               <li>
                能够找到：执行装配
               </li>
               <li>
                找不到：装配失败
               </li>
              </ul>
             </li>
            </ul>
           </li>
          </ul>
         </li>
        </ul>
        <p>
         示例：当有两个 service 实现类时
        </p>
        <pre><code class="prism language-java"><span class="token comment">// 实现类1</span>
<span class="token annotation punctuation">@Service</span>
<span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">UserServiceImpl</span> <span class="token keyword">implements</span> <span class="token class-name">UserService</span> <span class="token punctuation">{<!-- --></span>

	<span class="token annotation punctuation">@Autowired</span>
	<span class="token keyword">private</span> <span class="token class-name">UserDao</span> userDao<span class="token punctuation">;</span>

	<span class="token annotation punctuation">@Override</span>
	<span class="token keyword">public</span> <span class="token keyword">void</span> <span class="token function">getUser</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
		userDao<span class="token punctuation">.</span><span class="token function">save</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
		<span class="token class-name">System</span><span class="token punctuation">.</span>out<span class="token punctuation">.</span><span class="token function">println</span><span class="token punctuation">(</span><span class="token string">"user service"</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
	<span class="token punctuation">}</span>
<span class="token punctuation">}</span>

<span class="token comment">// 实现类2</span>
<span class="token annotation punctuation">@Service</span>
<span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">UserService2Impl</span> <span class="token keyword">implements</span> <span class="token class-name">UserService</span> <span class="token punctuation">{<!-- --></span>

	<span class="token annotation punctuation">@Autowired</span>
	<span class="token keyword">private</span> <span class="token class-name">UserDao</span> userDao<span class="token punctuation">;</span>
	
	<span class="token annotation punctuation">@Override</span>
	<span class="token keyword">public</span> <span class="token keyword">void</span> <span class="token function">getUser</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
		userDao<span class="token punctuation">.</span><span class="token function">save</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
		<span class="token class-name">System</span><span class="token punctuation">.</span>out<span class="token punctuation">.</span><span class="token function">println</span><span class="token punctuation">(</span><span class="token string">"user service"</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
	<span class="token punctuation">}</span>
<span class="token punctuation">}</span>
</code></pre>
        <p>
         controller 层注入的 service 如果没有指定是哪个bean 就会报错。
        </p>
        <p>
         解决方法：
        </p>
        <ol>
         <li>
          <p>
           service 层注入容器的 bean id 默认为首字母小写的方法名，controller 层的成员属性名可以设置为这个方法名实现精确匹配。
          </p>
          <pre><code class="prism language-java"><span class="token annotation punctuation">@Autowired</span>
<span class="token keyword">private</span> <span class="token class-name">UserService</span> userServiceImpl<span class="token punctuation">;</span>
</code></pre>
          <p>
           由于实际开放中，属性名是很关键的信息，所以不推荐使用这种方法。
          </p>
         </li>
         <li>
          <p>
           使用
           <code>
            @Qualifier
           </code>
           注解匹配。
          </p>
          <pre><code class="prism language-java"><span class="token annotation punctuation">@Autowired</span>
<span class="token annotation punctuation">@Qualifier</span><span class="token punctuation">(</span><span class="token string">"userServiceImpl"</span><span class="token punctuation">)</span>
<span class="token keyword">private</span> <span class="token class-name">UserService</span> userService<span class="token punctuation">;</span>
</code></pre>
         </li>
         <li>
          <p>
           使用
           <code>
            @Resource
           </code>
           注解匹配，这个注解的 name 属性根据 bean id 进行匹配，等价于
           <code>
            @Autowired
           </code>
           +
           <code>
            @Qualifier
           </code>
          </p>
          <pre><code class="prism language-java"><span class="token annotation punctuation">@Resource</span><span class="token punctuation">(</span>name <span class="token operator">=</span> <span class="token string">"userServiceImpl"</span><span class="token punctuation">)</span>
<span class="token keyword">private</span> <span class="token class-name">UserService</span> userService<span class="token punctuation">;</span>
</code></pre>
         </li>
        </ol>
        <p>
         <strong>
          <code>
           @Resource
          </code>
          和
          <code>
           @Autowired
          </code>
          注解对比
         </strong>
         ：
        </p>
        <p>
         <code>
          @Resource
         </code>
         和
         <code>
          @Autowired
         </code>
         都用于实现依赖注入，但它们在所属规范、注入方式、查找顺序、依赖处理、适用场景等方面存在一些差异，以下是详细对比：
        </p>
        <ol>
         <li>
          <p>
           <strong>
            所属规范
           </strong>
          </p>
          <ul>
           <li>
            <p>
             <code>
              @Resource
             </code>
             ：它是 Java 标准注解，来自 JSR - 250 规范。这意味着使用该注解编写的代码具有更好的可移植性，不依赖于特定的框架，能在支持 JSR - 250 的不同 Java 环境中使用。
            </p>
           </li>
           <li>
            <p>
             <code>
              @Autowired
             </code>
             ：是 Spring 框架提供的注解，与 Spring 框架紧密绑定。如果项目不使用 Spring 框架，就无法使用该注解。
            </p>
           </li>
          </ul>
         </li>
         <li>
          <p>
           <strong>
            注入方式
           </strong>
          </p>
          <ul>
           <li>
            <code>
             @Resource
            </code>
            ：支持按名称和按类型两种注入方式。
            <ul>
             <li>
              当指定
              <code>
               name
              </code>
              属性时，会按照名称进行注入。
             </li>
             <li>
              若未指定
              <code>
               name
              </code>
              属性，会先尝试按名称查找，若找不到则按类型查找。
             </li>
            </ul>
           </li>
           <li>
            <code>
             @Autowired
            </code>
            ：默认按类型进行注入。当容器中存在多个相同类型的 Bean 时，仅使用
            <code>
             @Autowired
            </code>
            会抛出异常，此时通常需要结合
            <code>
             @Qualifier
            </code>
            注解指定 Bean 的名称来明确使用哪个 Bean。
           </li>
          </ul>
         </li>
         <li>
          <p>
           <strong>
            查找顺序
           </strong>
          </p>
          <ul>
           <li>
            <p>
             <code>
              @Resource
             </code>
             ：先根据
             <code>
              name
             </code>
             属性的值查找对应的 Bean，如果
             <code>
              name
             </code>
             未指定，则使用字段名或方法名作为查找的名称。若按名称找不到匹配的 Bean，再按类型查找。
            </p>
            <pre><code class="prism language-java"><span class="token keyword">import</span> <span class="token import"><span class="token namespace">javax<span class="token punctuation">.</span>annotation<span class="token punctuation">.</span></span><span class="token class-name">Resource</span></span><span class="token punctuation">;</span>

<span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">MyService</span> <span class="token punctuation">{<!-- --></span>
    <span class="token comment">// 先按名称 "myRepository" 查找 Bean，如果没指定 name 属性则用字段名 "repository" 查找，找不到再按类型查找</span>
    <span class="token annotation punctuation">@Resource</span><span class="token punctuation">(</span>name <span class="token operator">=</span> <span class="token string">"myRepository"</span><span class="token punctuation">)</span> 
    <span class="token keyword">private</span> <span class="token class-name">MyRepository</span> repository<span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
           </li>
           <li>
            <p>
             <code>
              @Autowired
             </code>
             ：先根据类型查找匹配的 Bean，若找到多个相同类型的 Bean，会根据字段名或参数名作为 Bean 的名称进行匹配，如果还无法确定则需要使用
             <code>
              @Qualifier
             </code>
             注解。
            </p>
            <pre><code class="prism language-java"><span class="token keyword">import</span> <span class="token import"><span class="token namespace">org<span class="token punctuation">.</span>springframework<span class="token punctuation">.</span>beans<span class="token punctuation">.</span>factory<span class="token punctuation">.</span>annotation<span class="token punctuation">.</span></span><span class="token class-name">Autowired</span></span><span class="token punctuation">;</span>
<span class="token keyword">import</span> <span class="token import"><span class="token namespace">org<span class="token punctuation">.</span>springframework<span class="token punctuation">.</span>beans<span class="token punctuation">.</span>factory<span class="token punctuation">.</span>annotation<span class="token punctuation">.</span></span><span class="token class-name">Qualifier</span></span><span class="token punctuation">;</span>

<span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">MyService</span> <span class="token punctuation">{<!-- --></span>
    <span class="token annotation punctuation">@Autowired</span>
    <span class="token annotation punctuation">@Qualifier</span><span class="token punctuation">(</span><span class="token string">"myRepository"</span><span class="token punctuation">)</span> <span class="token comment">// 当有多个 MyRepository 类型的 Bean 时，指定使用名为 "myRepository" 的 Bean</span>
    <span class="token keyword">private</span> <span class="token class-name">MyRepository</span> repository<span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
           </li>
          </ul>
         </li>
         <li>
          <p>
           <strong>
            依赖处理
           </strong>
          </p>
          <ul>
           <li>
            <p>
             <code>
              @Resource
             </code>
             ：如果找不到匹配的 Bean，会抛出
             <code>
              NoSuchBeanDefinitionException
             </code>
             异常，要求依赖必须存在。
            </p>
           </li>
           <li>
            <p>
             <code>
              @Autowired
             </code>
             ：默认情况下，依赖必须存在，若找不到匹配的 Bean 会抛出异常。但可以通过设置
             <code>
              required = false
             </code>
             来允许依赖为
             <code>
              null
             </code>
             。
            </p>
            <pre><code class="prism language-java"><span class="token keyword">import</span> <span class="token import"><span class="token namespace">org<span class="token punctuation">.</span>springframework<span class="token punctuation">.</span>beans<span class="token punctuation">.</span>factory<span class="token punctuation">.</span>annotation<span class="token punctuation">.</span></span><span class="token class-name">Autowired</span></span><span class="token punctuation">;</span>

<span class="token keyword">public</span> <span class="token keyword">class</span> <span class="token class-name">MyService</span> <span class="token punctuation">{<!-- --></span>
    <span class="token annotation punctuation">@Autowired</span><span class="token punctuation">(</span>required <span class="token operator">=</span> <span class="token boolean">false</span><span class="token punctuation">)</span> <span class="token comment">// 允许 repository 为 null</span>
    <span class="token keyword">private</span> <span class="token class-name">MyRepository</span> repository<span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
           </li>
          </ul>
         </li>
         <li>
          <p>
           <strong>
            适用场景
           </strong>
          </p>
          <ul>
           <li>
            <code>
             @Resource
            </code>
            ：适用于需要明确指定依赖名称的场景，或者在跨框架开发中，为了保持代码的兼容性和可移植性，优先选择
            <code>
             @Resource
            </code>
            。
           </li>
           <li>
            <code>
             @Autowired
            </code>
            ：在纯 Spring 项目中广泛使用，尤其是当依赖的类型是唯一确定的时候，使用
            <code>
             @Autowired
            </code>
            可以更简洁地完成依赖注入。
           </li>
          </ul>
         </li>
        </ol>
       </li>
      </ol>
     </li>
    </ol>

    <h4>

    </h4>
    <p>
     Spring AOP（Aspect-Oriented Programming，面向切面编程）是 Spring 框架的一个重要模块，用于将横切关注点（如日志记录、事务管理、安全检查等）与核心业务逻辑分离。通过 AOP，你可以将这些横切关注点模块化，并在不修改核心代码的情况下将其应用到多个模块中。
    </p>
    <p>
     简单来说，在大型项目开发时，我们对于项目的每个操作都要记录日志，正常开发下，需要在每个方法都记录日志输出，一旦方法过多，非常麻烦，我们可以把日志的输出提取成一个公共的方法，然后让 AOP 帮我们把这个日志方法插入到每个方法中，AOP 就是做这种插入的工作。
    </p>
    <h4>
     <a id="AOP__2055">
     </a>
     AOP 的核心概念
    </h4>
    <ul>
     <li>
      <strong>
       Aspect（切面）
      </strong>
      ：横切关注点的模块化单元，通常是一个类，包含多个通知（Advice）和切点（Pointcut）。
     </li>
     <li>
      <strong>
       Join Point（连接点）
      </strong>
      ：程序执行过程中的某个特定点，如方法调用、异常抛出等。
     </li>
     <li>
      <strong>
       Advice（通知）
      </strong>
      ：在连接点执行的动作，如前置通知、后置通知等。
     </li>
     <li>
      <strong>
       Pointcut（切点）
      </strong>
      ：用于匹配连接点的表达式，决定哪些连接点会触发通知。
     </li>
     <li>
      <strong>
       Weaving（织入）
      </strong>
      ：将切面应用到目标对象并创建代理对象的过程。
     </li>
    </ul>
    <h4>
     <a id="AOP__2065">
     </a>
     AOP 的实现方式
    </h4>
    <p>
     Spring AOP 基于动态代理实现，支持两种代理方式：
    </p>
    <ul>
     <li>
      <strong>
       JDK 动态代理
      </strong>
      ：适用于实现了接口的类。
     </li>
     <li>
      <strong>
       CGLIB 代理
      </strong>
      ：适用于没有实现接口的类。
     </li>
    </ul>
    <h4>
     <a id="AOP__2074">
     </a>
     AOP 的通知类型
    </h4>
    <p>
     Spring AOP 支持以下五种通知类型：
    </p>
    <ul>
     <li>
      <strong>
       @Before
      </strong>
      ：在目标方法执行之前执行。
     </li>
     <li>
      <strong>
       @After
      </strong>
      ：在目标方法执行之后执行，无论是否抛出异常。
     </li>
     <li>
      <strong>
       @AfterReturning
      </strong>
      ：在目标方法成功执行并返回结果后执行。
     </li>
     <li>
      <strong>
       @AfterThrowing
      </strong>
      ：在目标方法抛出异常后执行。
     </li>
     <li>
      <strong>
       @Around
      </strong>
      ：在目标方法执行前后执行，可以控制是否执行目标方法。
     </li>
    </ul>
    <h4>
     <a id="AOP__2086">
     </a>
     AOP 的应用场景
    </h4>
    <ul>
     <li>
      <strong>
       日志记录
      </strong>
      ：记录方法调用参数、执行时间等。
     </li>
     <li>
      <strong>
       事务管理
      </strong>
      ：通过
      <code>
       @Transactional
      </code>
      注解管理数据库事务。
     </li>
     <li>
      <strong>
       权限校验
      </strong>
      ：在方法执行前检查用户权限。
     </li>
     <li>
      <strong>
       性能监控
      </strong>
      ：统计方法执行耗时。
     </li>
     <li>
      <strong>
       异常处理
      </strong>
      ：统一捕获并处理异常。
     </li>
    </ul>
    <h4>
     <a id="AOP__2096">
     </a>
     AOP 的优缺点
    </h4>
    <p>
     <strong>
      优点
     </strong>
     ：
    </p>
    <ul>
     <li>
      与 Spring 容器无缝集成。
     </li>
     <li>
      配置简单，学习成本低。
     </li>
     <li>
      无需额外编译步骤。
     </li>
    </ul>
    <p>
     <strong>
      缺点
     </strong>
     ：
    </p>
    <ul>
     <li>
      仅支持方法级别的拦截（无法拦截字段修改）。
     </li>
     <li>
      自调用问题：同一类内部方法调用不触发 AOP。
     </li>
     <li>
      功能弱于 AspectJ（如不支持编译时织入）。
     </li>
    </ul>

    <p>
     以日志输出为例，步骤：
    </p>
    <ol>

    <h5>
     <a id="_2524">
     </a>
     环绕通知
    </h5>
    <p>
     环绕通知（
     <code>
      @Around
     </code>
     ）是
     <strong>
      AOP（面向切面编程）中最强大且灵活的通知类型
     </strong>
     。它允许开发者完全控制目标方法的执行流程，包括拦截方法调用、修改参数、处理返回值或异常，甚至阻止方法执行。以下是其核心要点：
    </p>
    <ol>
     <li>
      <strong>
       完全控制方法执行
      </strong>
      ：环绕通知通过
      <code>
       ProceedingJoinPoint
      </code>
      参数直接决定是否（及何时）执行目标方法。
     </li>
     <li>
      <strong>
       唯一能修改参数和返回值的通知
      </strong>
      ：
      <ul>
       <li>
        <strong>
         修改参数
        </strong>
        ：通过
        <code>
         pjp.getArgs()
        </code>
        获取参数数组并修改。
       </li>
       <li>
        <strong>
         替换返回值
        </strong>
        ：在
        <code>
         pjp.proceed()
        </code>
        后修改返回的
        <code>
         result
        </code>
        。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       异常处理能力
      </strong>
      ：可捕获目标方法抛出的异常，决定是否重新抛出或静默处理。
     </li>
    </ol>
    <p>
     总结来说，
     <code>
      @Around
     </code>
     包含上面所有注解的所有功能。
    </p>
    <p>
     使用这个注解，必须在通知方法里定义返回值（目标方法的返回值），定义参数（
     <code>
      ProceedingJoinPoint joinPoint
     </code>
     ），抛出异常或直接
     <code>
      try...catch
     </code>
     处理，执行目标方法。
    </p>

    <h4>
     <a id="_2563">
     </a>
     切面优先级设置
    </h4>
    <p>
     相同目标方法上同时存在多个切面时，切面的优先级控制切面的内外嵌套顺序。
    </p>
    <ul>
     <li>
      优先级高的切面：外面
     </li>
     <li>
      优先级低的切面：里面
     </li>
    </ul>
    <p>
     使用
     <code>
      @Order
     </code>
     注解可以控制切面的优先级：数越小，优先级越高。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a523689ed1514fba94ce83f42752a40d.png#pic_center"/>
    </p>
    <p>
     实际开发时，如果有多个切面嵌套的情况，要慎重考虑。例如：如果事务切面优先级高，那么在缓存中命中数据的情况下，事务切面的操作都浪费了。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/8abc7509081b44c693445cf5f0641961.png#pic_center"/>
    </p>
    <p>
     此时应该将缓存切面的优先级提高，在事务操作之前先检查缓存中是否存在目标数据。
    </p>

    <h3>
     <a id="_2595">
     </a>
     编程式事务
    </h3>
    <p>
     编程式事务是指手动编写程序来管理事务，即通过编写代码的方式直接控制事务的提交和回滚。在 Java 中，通常使用事务管理器（如 Spring 中的
     <code>
      PlatformTransactionManager
     </code>
     ）来实现编程式事务。
    </p>
    <p>
     编程式事务的主要优点是灵活性高，可以按照自己的需求来控制事务的粒度、模式等等。但是，编写大量的事务控制代码容易出现问题，对代码的可读性和可维护性有一定影响。
    </p>
    <p>
     编程式的实现方式存在缺陷：
    </p>
    <ul>
     <li>
      细节没有被屏蔽：具体操作过程中，所有细节都需要程序员自己来完成，比较繁琐。
     </li>
     <li>
      代码复用性不高：如果没有有效抽取出来，每次实现功能都需要自己编写代码，代码就没有得到复用。
     </li>
    </ul>
    <h3>
     <a id="_2608">
     </a>
     声明式事务
    </h3>
    <p>
     Spring 的
     <strong>
      声明式事务管理
     </strong>
     是一种基于 AOP 的事务管理方式，开发者通过配置（而非硬编码）定义事务规则，将事务逻辑与业务代码解耦。其核心思想是
     <strong>
      “约定优于配置”
     </strong>
     ，通过注解或 XML 定义事务边界和属性，由 Spring 框架自动管理事务的提交、回滚等操作，使用声明式事务可以将事务的控制和业务逻辑分离开来，提高代码的可读性和可维护性。
    </p>
    <p>
     Spring 的声明式事务是由 Spring tx 模块实现的。
    </p>
    <p>
     区别：
    </p>
    <ul>
     <li>
      <strong>
       编程式事务需要手动编写代码来管理事务
      </strong>
      。
     </li>
     <li>
      <strong>
       而声明式事务可以通过配置文件或注解来控制事务
      </strong>
      。
     </li>
    </ul>
    <h4>
     <a id="_2621">
     </a>
     核心优势
    </h4>
    <ol>
     <li>
      <strong>
       非侵入性
      </strong>
      ：业务代码无需包含事务管理逻辑（如
      <code>
       try-catch
      </code>
      或 JDBC 事务 API），只需通过注解声明事务规则。
     </li>
     <li>
      <strong>
       可维护性高
      </strong>
      ：事务配置集中管理，修改时无需改动业务代码。
     </li>
     <li>
      <strong>
       灵活配置
      </strong>
      ：支持细粒度的事务属性控制（如传播行为、隔离级别、超时时间等）。
     </li>
    </ol>

    <h5>
     <a id="PROPAGATION_2644">
     </a>
     事务传播行为（PROPAGATION）
    </h5>
    <p>
     定义多个事务方法相互调用时的处理规则：
    </p>
    <table>
     <thead>
      <tr>
       <th>
        传播行为
       </th>
       <th>
        说明
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         REQUIRED
        </strong>
       </td>
       <td>

       </td>
      </tr>
      <tr>
       <td>
        <strong>
         REQUIRES_NEW
        </strong>
       </td>
       <td>
        始终新建事务，若当前存在事务则挂起
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         NESTED
        </strong>
       </td>
       <td>

       </td>
      </tr>
      <tr>
       <td>
        <strong>
         SUPPORTS
        </strong>
       </td>
       <td>
        当前存在事务则加入，否则以非事务方式执行
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         NOT_SUPPORTED
        </strong>
       </td>
       <td>
        以非事务方式执行，若当前存在事务则挂起
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         MANDATORY
        </strong>
       </td>
       <td>
        必须在事务中执行，否则抛出异常
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         NEVER
        </strong>
       </td>
       <td>
        必须在非事务中执行，否则抛出异常
       </td>
      </tr>

    <h3>
     <a id="Spring__2690">
     </a>
     Spring 事务管理器
    </h3>
    <h4>
     <a id="Spring_2692">
     </a>
     Spring声明式事务对应依赖
    </h4>
    <ul>
     <li>
      <code>
       spring-tx
      </code>
      ：包含声明式事务实现的基本规范（事务管理器规范接口和事务增强等等）。
     </li>
     <li>
      <code>
       spring-jdbc
      </code>
      ：包含 DataSource 方式事务管理器实现类
      <code>
       DataSourceTransactionManager
      </code>
      。
     </li>
     <li>
      <code>
       spring-orm
      </code>
      ：包含其他持久层框架的事务管理器实现类例如：Hibernate、Jpa 等。
     </li>
    </ul>
    <h4>
     <a id="Spring_2700">
     </a>
     Spring声明式事务对应事务管理器接口
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/b28aff531f144e7fa1dab42557e4d2c6.png#pic_center"/>
    </p>
    <p>
     我们现在要使用的事务管理器是
     <code>
      org.springframework.jdbc.datasource.DataSourceTransactionManager
     </code>
     ，将来整合 JDBC 方式、JdbcTemplate 方式、Mybatis 方式的事务实现！
    </p>
    <p>
     <code>
      DataSourceTransactionManager
     </code>
     类中的主要方法：
    </p>
    <ul>
     <li>
      <code>
       doBegin()
      </code>
      ：开启事务。
     </li>
     <li>
      <code>
       doSuspend()
      </code>
      ：挂起事务。
     </li>
     <li>
      <code>
       doResume()
      </code>
      ：恢复挂起的事务。
     </li>
     <li>
      <code>
       doCommit()
      </code>
      ：提交事务。
     </li>
     <li>
      <code>
       doRollback()
      </code>
      ：回滚事务。
     </li>
    </ul>
