     K8S中MySQL配置持久化存储
    </h1>
   </div>
  </div>
 </div>
 <article class="baidu_pl">
  <div class="article_content clearfix" id="article_content">
   <link href="../../assets/css/kdoc_html_views-1a98987dfd.css" rel="stylesheet"/>

    <p>
     在 Kubernetes 中为有状态应用（如 MySQL）配置持久化存储，需结合
     <strong>
      StatefulSet
     </strong>
     、
     <strong>
      PersistentVolumeClaim (PVC)
     </strong>
     和
     <strong>
      StorageClass
     </strong>
     ，确保数据在 Pod 重启或迁移时不会丢失。以下是详细步骤和最佳实践：
    </p>
    <hr/>
    <h4>
     <strong>
      一、核心组件选择
     </strong>
    </h4>
    <table>
     <thead>
      <tr>
       <th>
        <strong>
         组件
        </strong>
       </th>
       <th>
        <strong>
         作用
        </strong>
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         StatefulSet
        </strong>
       </td>
       <td>
        为每个 Pod 提供唯一的网络标识和稳定的持久化存储（按顺序创建/删除）。
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         PVC
        </strong>
       </td>
       <td>
        声明 Pod 所需的存储容量和访问模式（如
        <code>
         ReadWriteOnce
        </code>
        ）。
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         StorageClass
        </strong>
       </td>
       <td>
        定义动态创建 PV 的规则（如云厂商存储类型、回收策略）。
       </td>
      </tr>
     </tbody>
    </table>
    <hr/>
    <h4>
     <strong>
      二、配置步骤（以 MySQL 为例）
     </strong>
    </h4>
    <h5>
     <strong>
      1. 创建 StorageClass（可选，若需动态供应）
     </strong>
    </h5>
    <p>
     如果是云环境（如 AWS、GCP），通常已有默认 StorageClass。若需自定义（例如本地存储），需手动配置：
    </p>
    <pre>apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: mysql-storage
provisioner: kubernetes.io/aws-ebs  # 根据环境调整（如 rook-ceph、local）
parameters:
  type: gp3                         # AWS EBS 类型（或 SSD、HDD）
reclaimPolicy: Retain               # 删除 PVC 后保留 PV 和数据
volumeBindingMode: WaitForFirstConsumer  # 延迟绑定到 Pod 调度节点</pre>
    <h5>
     <strong>
      2. 定义 StatefulSet 和 PVC 模板
     </strong>
    </h5>
    <pre>apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql-headless     # Headless Service 用于 Pod 发现
  replicas: 1                     # 单节点示例（生产环境建议集群化）
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8.0
          env:
            - name: MYSQL_ROOT_PASSWORD
              value: "your-root-password"
          ports:
            - containerPort: 3306
          volumeMounts:
            - name: mysql-data
              mountPath: /var/lib/mysql  # MySQL 数据目录
  volumeClaimTemplates:           # 自动为每个 Pod 创建 PVC
    - metadata:
        name: mysql-data
      spec:
        accessModes: [ "ReadWriteOnce" ]
        storageClassName: mysql-storage  # 使用自定义 StorageClass
        resources:
          requests:
            storage: 20Gi               # 根据需求调整容量</pre>
    <h5>
     <strong>
      3. 创建 Headless Service
     </strong>
    </h5>
    <pre>apiVersion: v1
kind: Service
metadata:
  name: mysql-headless
spec:
  clusterIP: None   # Headless Service 无 ClusterIP
  selector:
    app: mysql
  ports:
    - port: 3306</pre>
    <hr/>
    <h4>
     <strong>
      三、关键配置说明
     </strong>
    </h4>
    <h5>
     <strong>
      1. 存储访问模式 (
      <code>
       accessModes
      </code>
      )
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        <code>
         ReadWriteOnce (RWO)
        </code>
       </strong>
       ：单节点读写（适合单实例 MySQL）。
      </p>
     </li>
     <li>
      <p>
       <strong>
        <code>
         ReadWriteMany (RWX)
        </code>
       </strong>
       ：多节点读写（适用于分布式数据库如 Cassandra）。
      </p>
     </li>
     <li>
      <p>
       <strong>
        <code>
         ReadOnlyMany (ROX)
        </code>
       </strong>
       ：多节点只读。
      </p>
     </li>
    </ul>
    <h5>
     <strong>
      2. 持久化路径
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       MySQL 默认数据目录为
       <code>
        /var/lib/mysql
       </code>
       ，必须挂载到持久化卷。
      </p>
     </li>
     <li>
      <p>
       挂载其他路径（如配置文件）需额外配置。
      </p>
     </li>
    </ul>
    <h5>
     <strong>
      3. 动态供应 vs 静态供应
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        动态供应
       </strong>
       （推荐）：通过 StorageClass 自动创建 PV。
      </p>
     </li>
     <li>
      <p>
       <strong>
        静态供应
       </strong>
       ：手动创建 PV 并绑定 PVC（适合本地存储或无云厂商环境）。
      </p>
     </li>
    </ul>
    <hr/>
    <h4>
     <strong>
      四、生产环境优化
     </strong>
    </h4>
    <h5>
     <strong>
      1. 高可用配置
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        MySQL 集群
       </strong>
       ：使用 Galera Cluster 或 Group Replication，配合多副本 StatefulSet。
      </p>
     </li>
     <li>
      <p>
       <strong>
        跨可用区存储
       </strong>
       ：配置 StorageClass 支持多 AZ（如 AWS EBS 需结合
       <code>
        topologyKey: topology.kubernetes.io/zone
       </code>
       ）。
      </p>
     </li>
    </ul>
    <h5>
     <strong>
      2. 备份与恢复
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        定期快照
       </strong>
       ：使用云厂商快照工具（如 AWS EBS Snapshot）或 Velero。
      </p>
     </li>
     <li>
      <p>
       <strong>
        逻辑备份
       </strong>
       ：通过 CronJob 执行
       <code>
        mysqldump
       </code>
       并存储到对象存储（如 S3）。
      </p>
     </li>
    </ul>
    <h5>
     <strong>
      3. 监控与告警
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       监控 PV/PVC 使用率（Prometheus +
       <code>
        kubelet
       </code>
       指标）。
      </p>
     </li>
     <li>
      <p>
       设置 PVC 容量不足的告警规则。
      </p>
     </li>
    </ul>
    <hr/>
    <h4>
     <strong>
      五、验证流程
     </strong>
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        部署验证
       </strong>
      </p>
      <pre>kubectl apply -f mysql-statefulset.yaml
kubectl get statefulset,pvc,pod -l app=mysql</pre>
      <ul>
       <li>
        <p>
         确认 PVC 状态为
         <code>
          Bound
         </code>
         ，Pod 运行正常。
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        数据持久化测试
       </strong>
      </p>
      <ul>
       <li>
        <p>
         进入 MySQL Pod 写入测试数据：
        </p>
        <pre>kubectl exec -it mysql-0 -- mysql -u root -p
CREATE DATABASE test;</pre>
       </li>
       <li>
        <p>
         删除 Pod 并观察 StatefulSet 自动重建：
        </p>
        <pre>kubectl delete pod mysql-0
kubectl exec -it mysql-0 -- mysql -u root -p -e "SHOW DATABASES;"  # 确认 test 库存在</pre>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        故障恢复测试
       </strong>
      </p>
      <ul>
       <li>
        <p>
         模拟节点故障，观察 Pod 重新调度到其他节点后数据是否保留。
        </p>
       </li>
      </ul>
     </li>
    </ol>
    <hr/>
    <h4>
     <strong>
      六、常见问题
     </strong>
    </h4>
    <h5>
     <strong>
      1. PVC 无法绑定
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       检查 StorageClass 是否存在且配置正确。
      </p>
     </li>
     <li>
      <p>
       确认 PV 的容量和访问模式与 PVC 匹配。
      </p>
     </li>
    </ul>
    <h5>
     <strong>
      2. 数据写入性能差
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       使用 SSD 类型 StorageClass（如 AWS
       <code>
        gp3
       </code>
       或
       <code>
        io2
       </code>
       ）。
      </p>
     </li>
     <li>
      <p>
       调整文件系统挂载参数（如
       <code>
        ext4
       </code>
       的
       <code>
        discard
       </code>
       选项）。
      </p>
     </li>
    </ul>
    <h5>
     <strong>
      3. 存储资源不足
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       设置
       <code>
        ResourceQuota
       </code>
       限制命名空间存储使用量。
      </p>
     </li>
     <li>
      <p>
       启用 PVC 扩容功能（需存储驱动支持）。
      </p>
     </li>
    </ul>
    <hr/>
    <h4>
     <strong>
      总结
     </strong>
    </h4>
    <p>
     通过
     <strong>
      StatefulSet + PVC + StorageClass
     </strong>
     的组合，可为 MySQL 等有状态应用提供稳定的持久化存储，同时结合高可用、备份和监控策略，确保生产环境的可靠性和数据安全。实际部署时需根据业务需求调整存储类型、容量和访问模式。
    </p>
