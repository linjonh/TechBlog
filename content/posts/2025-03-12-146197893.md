    <p>
     在 Android 开发中，
     <strong>
      MVC
     </strong>
     、
     <strong>
      MVP
     </strong>
     和
     <strong>
      MVVM
     </strong>
     是三种常见的架构模式，它们的目标都是通过分层解耦代码，提升可维护性和可测试性。以下是它们的核心区别和实际应用对比：
    </p>
    <h4>
     <strong>
      1. 核心职责划分
     </strong>
    </h4>
    <table>
     <thead>
      <tr>
       <th>
        <strong>
         架构模式
        </strong>
       </th>
       <th>
        <strong>
         分层结构
        </strong>
       </th>
       <th>
        <strong>
         各层职责
        </strong>
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         MVC
        </strong>
       </td>
       <td>
        <strong>
         Model-View-Controller
        </strong>
       </td>
       <td>
        -
        <strong>
         Model
        </strong>
        : 数据逻辑（数据库、网络请求）
        <br/>
        -
        <strong>
         View
        </strong>
        : UI 显示（XML布局）
        <br/>
        -
        <strong>
         Controller
        </strong>
        : 业务逻辑（Activity/Fragment）
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         MVP
        </strong>
       </td>
       <td>
        <strong>
         Model-View-Presenter
        </strong>
       </td>
       <td>
        -
        <strong>
         Model
        </strong>
        : 数据逻辑
        <br/>
        -
        <strong>
         View
        </strong>
        : UI 显示 + 接口（Activity/Fragment 实现）
        <br/>
        -
        <strong>
         Presenter
        </strong>
        : 业务逻辑（纯 Java/Kotlin 类）
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         MVVM
        </strong>
       </td>
       <td>
        <strong>
         Model-View-ViewModel
        </strong>
       </td>
       <td>
        -
        <strong>
         Model
        </strong>
        : 数据逻辑
        <br/>
        -
        <strong>
         View
        </strong>
        : UI 显示（XML + Activity/Fragment）
        <br/>
        -
        <strong>
         ViewModel
        </strong>
        : 业务逻辑 + 数据驱动（通过 LiveData/Flow 暴露数据）
       </td>
      </tr>
     </tbody>
    </table>
    <p>
    </p>
    <h4>
     <strong>
      2. 数据流向
     </strong>
    </h4>
    <table>
     <thead>
      <tr>
       <th>
        <strong>
         架构模式
        </strong>
       </th>
       <th>
        <strong>
         数据流
        </strong>
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         MVC
        </strong>
       </td>
       <td>
        <strong>
         单向
        </strong>
        ：用户操作 → Controller → 更新 Model → 通知 View 刷新（但 Controller 可能直接操作 View）
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         MVP
        </strong>
       </td>
       <td>
        <strong>
         双向
        </strong>
        ：用户操作 → View → Presenter → Model → Presenter → View
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         MVVM
        </strong>
       </td>
       <td>
        <strong>
         双向
        </strong>
        ：用户操作 → View → ViewModel → Model → ViewModel → View（通过数据绑定自动更新）
       </td>
      </tr>
     </tbody>
    </table>
    <p>
    </p>
    <h4>
     <strong>
      3. 核心优缺点对比
     </strong>
    </h4>
    <h5>
     <strong>
      MVC（传统 Android 开发模式）
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        优点
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         结构简单，适合小型项目。
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        缺点
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         <strong>
          Controller 臃肿
         </strong>
         ：Activity/Fragment 同时承担 View 和 Controller 职责。
        </p>
       </li>
       <li>
        <p>
         <strong>
          测试困难
         </strong>
         ：业务逻辑和 UI 耦合，难以单元测试。
        </p>
       </li>
      </ul>
     </li>
    </ul>
    <h5>
     <strong>
      MVP（解耦业务逻辑）
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        优点
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         <strong>
          职责清晰
         </strong>
         ：Presenter 独立于 Android API，便于测试。
        </p>
       </li>
       <li>
        <p>
         <strong>
          View 被动
         </strong>
         ：通过接口定义交互，降低耦合。
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        缺点
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         <strong>
          接口冗余
         </strong>
         ：需要为每个 View 定义大量接口。
        </p>
       </li>
       <li>
        <p>
         <strong>
          手动绑定
         </strong>
         ：需在 View 中主动调用 Presenter 方法。
        </p>
       </li>
      </ul>
     </li>
    </ul>
    <h5>
     <strong>
      MVVM（Jetpack 推荐模式）
     </strong>
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        优点
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         <strong>
          自动更新
         </strong>
         ：通过数据绑定（如
         <code>
          LiveData
         </code>
         +
         <code>
          DataBinding
         </code>
         ）实现 UI 自动刷新。
        </p>
       </li>
       <li>
        <p>
         <strong>
          生命周期感知
         </strong>
         ：ViewModel 自动管理数据，避免内存泄漏。
        </p>
       </li>
       <li>
        <p>
         <strong>
          代码简洁
         </strong>
         ：减少手动更新 UI 的代码。
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        缺点
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         <strong>
          学习成本
         </strong>
         ：需掌握 Jetpack 组件（ViewModel、LiveData 等）。
        </p>
       </li>
       <li>
        <p>
         <strong>
          过度绑定
         </strong>
         ：不当使用可能导致逻辑分散到 XML。
        </p>
       </li>
       <li>
        <p>
        </p>
       </li>
      </ul>
     </li>
    </ul>
    <p>
    </p>
    <p>
    </p>
    <h4>
     <strong>
      4. 代码示例对比
     </strong>
    </h4>
    <h5>
     <strong>
      MVC（典型问题）
     </strong>
    </h5>
    <pre><code class="hljs">// Activity 承担 Controller + View 职责
class UserActivity : AppCompatActivity() {
    private lateinit var textView: TextView
    private val userModel = UserModel()

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_user)
        textView = findViewById(R.id.text_view)
        
        // 直接操作 Model 和 View
        userModel.fetchUser { user -&gt;
            textView.text = user.name // Controller 直接更新 View
        }
    }
}</code></pre>
    <h5>
     <strong>
      MVP（解耦实现）
     </strong>
    </h5>
    <pre><code class="hljs">// View 接口
interface UserView {
    fun showUserName(name: String)
}

// Presenter 类
class UserPresenter(private val view: UserView) {
    private val userModel = UserModel()

    fun loadUser() {
        userModel.fetchUser { user -&gt;
            view.showUserName(user.name)
        }
    }
}

// Activity 实现 View 接口
class UserActivity : AppCompatActivity(), UserView {
    private lateinit var presenter: UserPresenter
    private lateinit var textView: TextView

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_user)
        textView = findViewById(R.id.text_view)
        presenter = UserPresenter(this)
        presenter.loadUser()
    }

    override fun showUserName(name: String) {
        textView.text = name
    }
}</code></pre>
    <h5>
     <strong>
      MVVM（Jetpack 实现）
     </strong>
    </h5>
    <pre><code class="hljs">// ViewModel
class UserViewModel : ViewModel() {
    private val _user = MutableLiveData&lt;User&gt;()
    val user: LiveData&lt;User&gt; = _user

    init {
        loadUser()
    }

    private fun loadUser() {
        UserModel.fetchUser { user -&gt;
            _user.value = user
        }
    }
}

// Activity + DataBinding
class UserActivity : AppCompatActivity() {
    private lateinit var binding: ActivityUserBinding
    private val viewModel: UserViewModel by viewModels()

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        binding = DataBindingUtil.setContentView(this, R.layout.activity_user)
        binding.lifecycleOwner = this
        binding.viewModel = viewModel // XML 中直接绑定 user.name
    }
}</code></pre>
    <h4>
     <strong>
      5. 适用场景
     </strong>
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        MVC
       </strong>
       ：快速原型开发或简单页面，不推荐复杂项目。
      </p>
     </li>
     <li>
      <p>
       <strong>
        MVP
       </strong>
       ：需要高测试覆盖率的项目（Presenter 可独立测试）。
      </p>
     </li>
     <li>
      <p>
       <strong>
        MVVM
       </strong>
       ：中大型项目，结合 Jetpack 组件（推荐 Google 官方方案）。
      </p>
     </li>
    </ul>

    <h4>
     <strong>
      6. 总结
     </strong>
    </h4>
    <table>
     <thead>
      <tr>
       <th>
        <strong>
         维度

       </th>

         MVP

         MVVM

     <tbody>

        </strong>

       <td>

       </td>
      </tr>

         测试难度

    <h4>
     <strong>
      7. 现代 Android 开发推荐
     </strong>
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        优先选择 MVVM
       </strong>
       ：结合 Jetpack 组件（ViewModel + LiveData + DataBinding/Room）实现高效开发。
      </p>
     </li>
     <li>
      <p>
       <strong>
        辅助工具
       </strong>
       ：
      </p>
      <ul>
       <li>
        <p>
         <strong>
          Dagger/Hilt
         </strong>
         ：依赖注入，进一步解耦。
        </p>
       </li>
       <li>
        <p>
         <strong>
          Jetpack Compose
         </strong>
         ：声明式 UI，天然适合 MVVM。
        </p>
       </li>
      </ul>
     </li>
    </ul>
    <p>
     通过理解这三种架构的核心差异，开发者可以更合理地选择适合项目需求的模式。对于新项目，强烈建议采用
     <strong>
      MVVM + Jetpack
     </strong>
     的现代化架构方案。
    </p>
