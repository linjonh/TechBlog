    <h3>
     <a id="header_29">
     </a>
     请求参数化（请求参数、header参数）
    </h3>
    <p>
     其实请求参数和header参数化都是一样的道理，原理和操作都是一样的.
     <br/>
     <strong>
      注意：参数化的话，我们一定要在实现参数化之前，引用的参数是之前执行过的那个用例，不然会导致参数化找不到对应的编号用例。
     </strong>
    </p>
    <h5>
     <a id="_32">
     </a>
     请求参数
    </h5>
    <p>
     先说请求参数化，在需要对应参数的那个用例已经完成的前提下，按照一定的格式进行编写就OK了
    </p>
    <pre><code>{<!-- -->{ $.17.body.message }} //id为17的用例的返回里的message字段的值
{<!-- -->{ $.17.params.type }} //id为17的用例的请求里的type字段的值
</code></pre>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/6c0db8e2c7d1ad354074f8eb4ede7a8a.png"/>
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/e856eb7438d7dce1c32c24f857ea3495.png"/>
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/94db2a92b648394568c7da23bbe2f00b.png"/>
     <br/>
     上面3张图片就是我参数化的操作及思路，首先我们在测试集合中每个测试用例都有一个对应的id，然后我们在需要参数化的那个测试用例中去找我们在之前的那个用例中需要响应中的哪些参数，然后以固定的格式进行请求就OK了！
    </p>
    <h5>
     <a id="header_44">
     </a>
     header参数化
    </h5>
    <p>
     header参数化同理，也是一样的操作。公司的研发人员为了接口的安全性考虑，不能让随便那个人都有权限进行接口执行，肯定会设置一定的权限，所以就肯定会涉及到token，但是token又有一定的时效性，所以我们就要想啊，这个token是哪里来的，后面的接口怎么引用的。
     <br/>
     以我们公司的项目为例，有权限的接口里面的token都是以一下方式实现的
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/b8886ff4be2777efae8bfbdae688983a.png"/>
     <br/>
     存在header中，对应key：Authorization，value为：Bearer +token
     <br/>
     token呢，又是在登录后拿到的，我找到登录接口，对登录接口的响应数据进行分析
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/8e52d841741a8151bc605ec60750613a.png"/>
     <br/>
     是对应在body的access_token中，所以就有了下面的参数化
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/7af511618c92baa6e13b85a30dfd2bdf.png"/>
     <br/>
     就此我们的参数化完成
     <br/>
     另外，还有一点，我们总体的生成的阶段性测试报告在这个蓝色的地址中
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/8b0f0b06268e2d77abba993206c05b2b.png"/>
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/168244ef15b495e6d951dea36cf2fcd6.png"/>
     <br/>
     注意：我这里踩了一个坑，我加了一个新的环境配置，每当我手动批量执行用例时，选择对应的环境，都可以成功，但是看我总体的测试报告的时候却失败了，经过分析，原来是我的自动化参数的环境没有在默认环境的位置，所以我就直接把这个有参数的环境拖到最上面，让他变成默认环境，就此OK~！
