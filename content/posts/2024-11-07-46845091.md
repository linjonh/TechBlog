     <span style="font-size:14px; color:#333333">
      <strong>
       背景
      </strong>
      <br/>
      <br/>
      测试是完善的研发体系中不可或缺的一环。前端同样需要测试，你的css改动可能导致页面错位、js改动可能导致功能不正常。
      <br/>
      由于前端偏向GUI软件的特殊性，尽管测试领域工具层出不穷，在前端的自动化测试上面却实施并不广泛，很多人依旧以手工测试
      <br/>
      为主。本文试图探讨前端自动化测试领域的工具和实践。
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      <br/>
      <br/>
      <strong>
       为什么需要自动化测试
      </strong>
      <br/>
      <br/>
     </span>
     <span style="font-size:14px; color:#333333">
      一个项目最终会经过快速迭代走向以维护为主的状态，在合理的时机以合理的方式引入自动化测试能有效减少人工维护成本。
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      自动
     </span>
     <span style="font-size:14px; color:#333333">
      化测试的收益可以简单总结为：
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      自动化的收益 = 迭代次数 * 全手动执行成本 - 首次自动化成本 - 维护次数 * 维护成本
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      <br/>
      对于自动化测试来说，相对于发现未知的问题，更
     </span>
     <span style="font-size:14px; color:#333333">
      倾向于避免可能的问题。
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      <br/>
      <br/>
      <strong>
       可测试方向
      </strong>
      <br/>
      <br/>
      首先本文不会探讨单元测试方向，因为单测已经有完善的工具体系。但前端开发中，除了一些框架和库，愿意去写单测的
      <br/>
      少之又少。另外单测维护成本较高，而且也没法满足前端测试的所有需求。
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      <br/>
      前端自动化测试可以在几个方向进行尝试：
      <br/>
      <br/>
      <strong>
       界面回归测试：
      </strong>
      测试界面是否正常，这是前端测试最基础的环节
      <br/>
      <br/>
      <strong>
       功能测试：
      </strong>
      测试功能操作是否正常，由于涉及交互，这部分测试比界面测试会更复杂
      <br/>
      <br/>
      <strong>
       性能测试：
      </strong>
      页面性能越来越受到关注，并且性能需要在开发过程中持续关注，否则很容易随着业务迭代而下降
      <br/>
      <br/>
      <strong>
       页面特征检测：
      </strong>
      有些动态区域无法通过界面对比进行测试、也没有功能上的异常，但可能不符合需求。例如性能测试中移动端大图
      <br/>
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      素材检测就是一种特征检测，另外常见的还有页面区块静态资源是否符合预期等等。
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      <br/>
      <br/>
      <strong>
       业界开源工具
      </strong>
      <br/>
      <br/>
      工欲善其事，必先利其器。业界在自动化测试领域已经有不少优秀的框架和库，善于利用能事半功倍。
      <br/>
      <br/>
      <br/>
      <br/>
      <strong>
       界面回归测试
      </strong>
      <br/>
      <br/>
      界面回归测试常见的做法有
      <strong>
       像素对比
      </strong>
      和
      <strong>
       dom结构
      </strong>
      对比两个方向。
      <br/>
      <br/>
      <br/>
      <strong>
       像素对比
      </strong>
      <br/>
      <br/>
      像素对比基本的思想认为，如果网站没有因为你的改动而界面错乱，那么在截图上测试页面应当跟正常页面保持一致。可以跟
      <br/>
      线上正常页面对比或者页面历史记录对比。像素对比能直观的显示图像上的差异，如果达到一定阈值则页面可能不正常。
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      <br/>
      <br/>
      <strong>
       PhantomCSS
      </strong>
      <br/>
      <br/>
      像素对比比较出名的工具是PhantomCSS。 PhantomCSS结合了 Casperjs截图和ResembleJs 图像对比分析。单纯从易用性和
      <br/>
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      对比
     </span>
     <span style="font-size:14px; color:#333333">
      效果来说还是不错的。

       不支持PhantomJS 2.0的问题
      </strong>
      <br/>
      <br/>
      由于PhantomJS 2.0暂时禁用了文件上传，PhantomCSS默认不支持PhantomJS 2.0 。如果还是想使用可以修改源码中获取图片
      <br/>
      文件的方式，改为通过ajax获取同域名下文件的方式，具体可以参考ResembleJs官网示例。
      <br/>
      <br/>
      <br/>
      <strong>
       如何测试多浏览器
      </strong>
      <br/>
      <br/>
      如果想测试多浏览器下的兼容性情况，只需要拿到多个浏览器下的截图即可。多浏览器测试最出名的当属selenium，selenium
      <br/>
      可以自动化的获取多个浏览器下的截图，前端工程师来说还可以借助Node的webdriver 来轻松开发测试脚本。
      <br/>
      <br/>
      但selenium的安装和上手成本要稍大些，而且对于多浏览器来说，各个浏览器之间的兼容性对比容易出错。不同浏览器截图可能
      <br/>
      一像素的偏差就导致截屏对比失败，多浏览器可能更适用回归性测试。
      <br/>
      <br/>
      <br/>
      <strong>
       响应式页面测试
      </strong>
      <br/>
      <br/>
      国外有人将像素对比应用到了响应式页面上，如果您针对PC和移动设备使用同一个网页，响应式测试可以很快的回归你的页面
      <br/>
      在不同尺寸上的页面是否正常。与单纯针对移动端开发的响应式不同，同时支持PC移动的页面更容易发生错乱。
      <br/>
      <br/>
      例如BackstopJS 项目，便是通过PhantomJS、capserJS等工具在不同尺寸下截图然后根据resemberJS进行像素比对判断是否

     <span style="font-size:14px; color:#333333">
      <strong>
       像素对比需要注意的问题
      </strong>
      <br/>
      <br/>
      不建议对网站所有页面进行测试 这只会导致很容易出现告警，但不一定是错误。针对重复使用的组件和样式、容易出问题的
      <br/>
      区域测试更加有效。
      <br/>
      <br/>
      推荐测试区域而不是整个页面 整个页面的测试导致任何一点文字、图像等动态的改变都可能导致不通过，而且真正的错误可能
      <br/>
      由于图像太大而被阈值忽略。图像越大对比也越容易超时。
      <br/>
      <br/>
      隐藏动态区域 在选择器对应的区域如果有动态元素，可以同样通过选择器来隐藏。
      <br/>
      <br/>
      界面对比只是一个环节，需与其他测试相结合 没有银弹，合理结合才是关键。
      <br/>
      <br/>
      <br/>

      <br/>
      <br/>
      <br/>
      <strong>
       用户操作测试
      </strong>
      <br/>
      <br/>
      上面提到界面回归测试无法取代功能测试。即便是界面正常，功能正常也是必须关注的部分。最直接的功能测试就是模拟用户
      <br/>
      操作，通过模拟正常的操作流程来判断页面展现是否符合预期。
      <br/>
      <br/>
      <br/>
      <strong>
       Phantomjs、CasperJS
      </strong>
      <br/>
      <br/>
      大名鼎鼎的PhantomJS当然要隆重介绍啦！前面界面对比测试基本都是基于PhantomJS开发的， Phantom JS是一个服务器端
      <br/>

      <br/>
      捕获、页面自动化访问等等方面可以说是信手拈来。
     </span>

      <strong>
       不想维护case?
      </strong>
      <br/>
      <br/>
      除非有足够的QA同学来帮你完成测试工作，否则通过人工来回归肯定会消耗更多的精力。在项目功能基本稳定期，维护case
      <br/>
      会简单的多，而且同样建议针对网站核心功能而不是所有功能来添加case。
      <br/>
      <br/>
      <br/>
      <strong>
       浏览器兼容测试
      </strong>
      <br/>
      <br/>
      当然selenium同样支持操作测试，类似的工具还有dalekjs等，如果想专门针对IE测试，可以考虑[triflejs]http://triflejs.org/，它
      <br/>
      提供了与PhantomJS基本类似的API。
      <br/>
      <br/>
      <strong>
       PhantomFlow操作对比测试
      </strong>
      <br/>
      <br/>
      有没有像图像对比一样直观，又能比较简单的写case的工具呢？可以考虑PhantomFlow, PhantomFlow假定如果页面正常，那么
      <br/>
      在相同的操作下，测试页面与正常页面的展现应该是一样的。 基于这点，用户只需要定义一系列操作流程和决策分支，然后利用
      <br/>
      PhantomCSS进行截图和图像对比。最后在一个很赞的可视化报表中展现出来。可以看下作者所在公司进行的测试可视化图表：

      <br/>
      <strong>
       性能测试
      </strong>
      <br/>
      <br/>
      网站展现性能也越来越成为人们关注的点，尤其是移动端性能始终是一个影响体验的重要因素。一般开发者都会利用自动化工具
      <br/>
      对资源进行合并压缩等优化，很多大公司也都搭建自己的性能监控系统指导优化工作。性能监控可以参考我的另一篇文章七天打造
      <br/>
      前端性能监控系统。
      <br/>
      <br/>
      需要注意的是性能并不是一个目标，而是开发、测试过程中需要持续关注的问题。我们有自动化的工具和框架在开发时进行
      <br/>
      优化，同样可以借助工具在测试时进行性能测试。
      <br/>
      <br/>
      这里推荐一个同样是基于PhantomJS的工具Phantomas,它能运行测试页面获取很多性能指标，加载时间、页面请求数、资源
      <br/>
      大小、是否开启缓存和Gzip、选择器性能、dom结构等等诸多指标都能一次性得到，并且有相应的grunt插件。你也可以对检测指标
      <br/>
      进行二次开发，例如移动端定义一个最大图片大小的规则，在开发的时候如果使用了超过限制的大图则进行告警。不过如果把加载
      <br/>
      过程中的时间点作为常规的测试监控，则最好模拟移动端网络环境
      <strong>
       。
       <br/>
       <br/>
       <br/>
       页面特征检测与实践
      </strong>
      <br/>
      <br/>
      前面讲到性能测试中测试资源大小其实就属于一种资源特征，诸如此类我们还可以开发一些通用的测试规则，以测试页面是否正
      <br/>
      常。这种测试主要适用于在界面和操作上无法直接进行判断的元素。例如页面中广告部署是否正常。
      <br/>
      <br/>
      <br/>
      <strong>
       广告部署检测实践
      </strong>
      <br/>
      <br/>
      第三方部署广告以及物料配置的时候容易出现问题，例如代码脚本升级出错、部署错误、物料尺寸格式不对、广告容器未适配
      <br/>
      多种屏幕大小、广告是否可见、时效广告是否展现等。已知的问题就有很多，如果出现问题时由广告系统的人员挨个检测是一个很
      <br/>
      耗费人力的过程。而这些特征都是跟实际运行环境相关的，大部分都可以通过casperjs之类的工具来进行检测。
      <br/>
      <br/>

      <br/>

      <br/>

      <br/>
      <br/>
      谈起ci肯定要介绍jenkins，稳定可靠，是很多大公司ci的首选。只是在前端的眼中它看起来会感觉。。丑了点和难用了点。
      <br/>
      如果能像travis-ci那样小清新和直观易用该多好哈哈。

      <br/>
      测试任务然后将运行结果反馈给用户即可。当然前端测试可能对于自定义的报表输出要求更高点。如果你想实现一套，使用laravel

      轻量级且支持自定义展现的ci系统了。这方面有很多教程，可以自行搜索

       实践经验
      </strong>
      <br/>
      <br/>
      前端自动化测试可以说还是一个在不断探索的领域，实施过程中也难免遇到问题。有些需要注意的点可以作为经验参考。
      <br/>
      <br/>
      <br/>
      <strong>
       减小使用和维护成本
      </strong>
      <br/>
      <br/>
      自动化测试为人诟病的地方无外乎使用效果和使用成本，使用效果可以对症下药选择合适的工具，而使用成本则可以通过一系列
      <br/>
      措施来减小到合理程度：
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      <br/>
      <br/>
      <strong>
       与构建工具结合
      </strong>
      <br/>
      <br/>
      grunt、FIS，将自动化测试与构建工具结合能更早的发现问题，也能减小使用和维护成本。
      <br/>
      <br/>
      <br/>
      <strong>
       与持续基础结合
      </strong>
      <br/>
      <br/>
      与CI系统的结合能更大范围更有效的发挥自动化测试的作用。
      <br/>
      <br/>
      <br/>
      <strong>
       与工作流结合
      </strong>
      <br/>
      <br/>
      与日常工作流结合同样是为了减少使用成本，如将结果通过自定义的方式反馈给用户等。
      <br/>
      <br/>
      <br/>
      <strong>
       测试配置化
      </strong>
      <br/>
      <br/>
      测试配置化能让用户使用和维护更加简单、大部分情况下只需要维护配置脚本即可。
      <br/>
      <br/>
      <br/>
      <strong>
       注重细节提高问题定位能力
      </strong>
      <br/>
      <br/>
      每个产品都有自身的特点，如果只是粗略的使用这些开源工具，可能达不到想要的效果，需要根据自身的情况选择合理的工具
      <br/>
      并进行一定的调优。只有不断提高自动化测试的问题定位能力，才能真正发挥自动化的价值。
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      <br/>
      <br/>
      <strong>
       利用开源力量、合理搭配使用
      </strong>
      <br/>
      <br/>
      如果遇到问题，请寻找解决思路。
      <br/>
      根据思路寻找开源支持。
      <br/>
      如果找不到请参照第一条。
      <br/>
      开源世界已经有很多优秀的资源，不建议从头开开始造轮子，除非你能很好的维护下去。基于现有的优秀工具、库、平台，针对
      <br/>
      自身产品的特点进行。
     </span>
    </div>
    <div style="text-align:left">
     <span style="font-size:14px; color:#333333">
      优化和二次开发更有利于工具本身的发展。
