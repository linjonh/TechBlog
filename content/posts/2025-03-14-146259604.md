    <h2>
     <span style="color:#0d0016">
      <strong>
       1. 引言
      </strong>
     </span>
    </h2>
    <p>
     <span style="color:#0d0016">
      在现代 Web 应用中，
      <strong>
       WebSocket
      </strong>
      以其
      <strong>
       全双工通信、低延迟、减少 HTTP 开销
      </strong>
      等优势，被广泛应用于即时通讯、在线游戏、实时数据推送等场景。然而，在涉及
      <strong>
       用户认证
      </strong>
      时，WebSocket 存在一个常见问题——
      <strong>
       每次刷新页面都会重新建立 WebSocket 连接，导致用户需要重新认证
      </strong>
      。
     </span>
    </p>
    <h3>
     <span style="color:#0d0016">
      <strong>
       1.1 WebSocket 连接与用户认证的挑战
      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">
      WebSocket 连接是
      <strong>
       长连接
      </strong>
      ，与传统 HTTP 请求不同，它并不会在每次通信时都自动携带身份认证信息（如 Cookie、Session）。这意味着：
     </span>
    </p>
    <ol>
     <li>
      <span style="color:#0d0016">
       <strong>
        页面刷新后，WebSocket 连接会断开并重新建立
       </strong>
       ，服务器无法自动识别用户身份，必须重新进行认证。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        传统的 WebSocket 认证流程较长
       </strong>
       ，通常是：
      </span>
      <ul>
       <li>
        <span style="color:#0d0016">
         前端建立 WebSocket 连接；
        </span>
       </li>
       <li>
        <span style="color:#0d0016">
         连接成功后，前端发送 Token 进行身份验证；
        </span>
       </li>
       <li>
        <span style="color:#0d0016">
         服务器验证 Token，返回认证结果。
        </span>
       </li>
      </ul>
     </li>
    </ol>
    <p>
     <span style="color:#0d0016">
      这样下来，整个认证过程涉及
      <strong>
       三次数据往返
      </strong>
      （RTT，Round Trip Time），不仅增加了服务器压力，还会影响用户体验，尤其是在网络延迟较高的环境下，认证速度会明显变慢。
     </span>
    </p>
    <h3>
     <span style="color:#0d0016">
      <strong>
       1.2 目标：减少认证回合，提高用户体验
      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">
      为了解决这个问题，我们希望优化 WebSocket 认证流程，目标是：
     </span>
    </p>
    <ul>
     <li>
      <span style="color:#0d0016">
       <strong>
        减少认证数据往返次数
       </strong>
       ，从三次缩减为两次甚至一次；
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        在 WebSocket 连接建立时就完成认证
       </strong>
       ，避免额外的认证消息交互；
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        确保认证方式安全可靠
       </strong>
       ，防止 Token 泄露或被劫持。
      </span>
     </li>
    </ul>
    <p>
     <span style="color:#0d0016">
      通过对 WebSocket 连接建立过程的深入研究，我们发现可以在
      <strong>
       WebSocket 握手阶段
      </strong>
      （仍然是 HTTP 请求）直接传递 Token，从而让服务器
      <strong>
       在握手时就完成身份认证
      </strong>
      。这样可以大大减少通信回合，提高用户体验。
     </span>
    </p>
    <h2>
     <span style="color:#0d0016">
      <strong>
       2. WebSocket 认证优化思路
      </strong>
     </span>
    </h2>
    <p>
     <span style="color:#0d0016">
      为了减少 WebSocket 认证的额外开销，我们需要优化传统的认证流程，让用户在刷新页面时不需要额外发送认证请求，而是
      <strong>
       在 WebSocket 连接建立的同时完成认证
      </strong>
      。本节将详细介绍 WebSocket 认证的优化方向，并对比不同方案的优缺点。
     </span>
    </p>
    <h3>
     <span style="color:#0d0016">
      <strong>
       2.1 传统 WebSocket 认证方式及其问题
      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">
      在标准的 WebSocket 连接流程中，前端通常按照以下方式进行身份认证：
     </span>
    </p>
    <ol>
     <li>
      <span style="color:#0d0016">
       <strong>
        建立 WebSocket 连接
       </strong>
       （
       <code>
        new WebSocket(url)
       </code>
       ）。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        WebSocket 连接建立后，前端发送 Token
       </strong>
       （
       <code>
        ws.send(token)
       </code>
       ）。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        服务器接收 Token 并进行身份验证
       </strong>
       。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        服务器返回认证结果，前端决定是否继续通信
       </strong>
       。
      </span>
     </li>
    </ol>
    <p>
     <span style="color:#0d0016">
      这种方式的主要问题是：
     </span>
    </p>
    <ul>
     <li>
      <span style="color:#0d0016">
       <strong>
        增加了不必要的通信回合
       </strong>
       。连接建立后，还需要额外的消息往返进行身份认证，导致
       <strong>
        三次通信
       </strong>
       （RTT3）。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        影响用户体验
       </strong>
       。网络延迟高时，用户可能需要等待较长时间才能完成认证，体验不够流畅。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        安全性问题
       </strong>
       。Token 通过 WebSocket 发送，可能被恶意中间人劫持，存在一定的安全风险（虽然 WebSocket 连接一般是
       <code>
        wss://
       </code>
       加密的，但仍需谨慎）。
      </span>
     </li>
    </ul>
    <h3>
     <span style="color:#0d0016">
      <strong>
       2.2 目标：让 WebSocket 认证更高效
      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">
      为了优化 WebSocket 认证，我们希望达到以下目标：
     </span>
    </p>
    <ul>
     <li>
      <span style="color:#0d0016">
       <strong>
        减少通信回合
       </strong>
       ，最好在
       <strong>
        WebSocket 连接建立的同时完成认证
       </strong>
       ，避免额外的认证数据交互。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        不依赖额外的 WebSocket 消息
       </strong>
       进行认证，而是在
       <strong>
        握手阶段（WebSocket Upgrade 请求）
       </strong>
       直接完成 Token 认证。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        兼容性强，安全性高
       </strong>
       ，避免 Token 通过 WebSocket 发送，减少被劫持的风险。
      </span>
     </li>
    </ul>
    <p>
     <span style="color:#0d0016">
      基于以上目标，我们提出两种优化方案：
     </span>
    </p>
    <h3>
     <span style="color:#0d0016">
      <strong>
       2.3 方案一：通过 URL 传递 Token（推荐方案）
      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">
      <strong>
       实现思路：
      </strong>
      <br/>
      在 WebSocket 连接 URL 中直接拼接 Token，例如：
     </span>
    </p>
    <pre><code class="language-java">const ws = new WebSocket("wss://example.com/ws?token=your_token_here");
</code></pre>
    <p>
     <span style="color:#0d0016">
      服务器在 Netty 中解析 WebSocket 握手请求的 URL，提取 Token 进行身份认证，认证成功后绑定 Token 到用户的 WebSocket 会话中。
     </span>
    </p>
    <p>
     <span style="color:#0d0016">
      <strong>
       优点：
      </strong>
      <br/>
      ✅
      <strong>
       减少一次往返
      </strong>
      ，从
      <strong>
       RTT3 降低为 RTT2
      </strong>
      （握手 + 服务器返回认证结果）。
      <br/>
      ✅
      <strong>
       无需额外 WebSocket 消息
      </strong>
      进行认证，认证逻辑清晰。
      <br/>
      ✅
      <strong>
       易于实现
      </strong>
      ，只需在 WebSocket 握手阶段解析 URL 参数即可。
     </span>
    </p>
    <p>
     <span style="color:#0d0016">
      <strong>
       注意事项：
      </strong>
      <br/>
      ⚠
      <strong>
       安全性问题
      </strong>
      ：
     </span>
    </p>
    <ul>
     <li>
      <span style="color:#0d0016">
       <strong>
        避免 Token 长期存留在 URL 记录
       </strong>
       （如浏览器历史、服务器日志等）。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        可以在服务器解析 Token 后立即从 URL 中删除
       </strong>
       ，避免 Token 泄露。
      </span>
     </li>
    </ul>
    <h3>
     <span style="color:#0d0016">
      <strong>
       2.4 方案二：利用 HTTP 头传递 Token（更安全但兼容性稍差）
      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">
      <strong>
       实现思路：
      </strong>
      <br/>
      WebSocket 的初始握手是基于 HTTP 进行的，因此可以利用
      <strong>
       HTTP 头部（Authorization）
      </strong>
      传递 Token，例如：
     </span>
    </p>
    <pre><code class="language-java">const ws = new WebSocket("wss://example.com/ws", [], {
  headers: { Authorization: "Bearer your_token_here" }
});
</code></pre>
    <p>
     <span style="color:#0d0016">
      然后，在 Netty 服务器中解析
      <code>
       Authorization
      </code>
      头，提取 Token 进行身份认证。
     </span>
    </p>
    <p>
     <span style="color:#0d0016">
      <strong>
       优点：
      </strong>
      <br/>
      ✅
      <strong>
       安全性更高
      </strong>
      ，Token 不会出现在 URL 中，避免泄露风险。
      <br/>
      ✅
      <strong>
       减少 WebSocket 消息认证，直接在握手时完成身份校验
      </strong>
      。
     </span>
    </p>
    <p>
     <span style="color:#0d0016">
      <strong>
       缺点：
      </strong>
      <br/>
      ⚠
      <strong>
       前端受限
      </strong>
      ：浏览器原生
      <code>
       WebSocket
      </code>
      API
      <strong>
       不支持自定义 HTTP 头
      </strong>
      ，必须通过
      <strong>
       自定义 WebSocket 客户端
      </strong>
      （如 Node.js 或 JavaScript WebSocket 库）实现。
      <br/>
      ⚠
      <strong>
       兼容性问题
      </strong>
      ：部分 WebSocket 服务器和代理可能不会转发自定义 HTTP 头，需要额外配置。
     </span>
    </p>
    <h3>
     <span style="color:#0d0016">
      <strong>
       2.5 方案三（失败方案）：利用 WebSocket
       <code>
        protocols
       </code>
       传递 Token
      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">
      WebSocket 连接时支持
      <code>
       protocols
      </code>
      参数，原本希望使用它来传递 Token，例如：
     </span>
    </p>
    <pre><code class="language-java">const ws = new WebSocket("wss://example.com/ws", ["token_your_token_here"]);
</code></pre>
    <p>
     <span style="color:#0d0016">
      然而，WebSocket
      <code>
       protocols
      </code>
      的作用是
      <strong>
       指定子协议
      </strong>
      （如
      <code>
       graphql-ws
      </code>
      ），并不是设计来传递 Token，因此
      <strong>
       服务器端 Netty 解析
       <code>
        protocols
       </code>
       时无法正确获取 Token
      </strong>
      。
     </span>
    </p>
    <p>
     <span style="color:#0d0016">
      <strong>
       最终结论：
      </strong>
      该方案不可行，不推荐使用。
     </span>
    </p>

    <h3>
     <span style="color:#0d0016">
      <strong>
       2.7 结论与下一步
      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">
      通过在
      <strong>
       WebSocket 连接握手阶段
      </strong>
      直接传递 Token，我们可以有效减少不必要的认证回合，优化用户体验。在接下来的章节，我们将深入讲解如何在
      <strong>
       Netty 服务器端
      </strong>
      解析 Token 并完成身份认证，确保 WebSocket 认证的安全性和稳定性。
     </span>
    </p>
    <h2>
     <span style="color:#0d0016">
      <strong>
       3. WebSocket 握手阶段传递 Token 的技术实现
      </strong>
     </span>
    </h2>
    <p>
     <span style="color:#0d0016">
      在上一节中，我们确定了
      <strong>
       在 WebSocket 连接握手阶段传递 Token
      </strong>
      是优化 WebSocket 认证的最佳方案。本节将详细讲解如何
      <strong>
       在前端传递 Token
      </strong>
      ，以及
      <strong>
       如何在 Netty 服务器端解析和验证 Token
      </strong>
      ，确保 WebSocket 连接的安全性和高效性。
     </span>
    </p>
    <h3>
     <span style="color:#0d0016">
      <strong>
       3.1 前端如何在 WebSocket 握手时传递 Token
      </strong>
     </span>
    </h3>
    <h4>
     <span style="color:#0d0016">
      <strong>
       方案一：通过 URL 传递 Token（推荐）
      </strong>
     </span>
    </h4>
    <p>
     <span style="color:#0d0016">
      在 WebSocket 连接 URL 中直接拼接 Token，例如：
     </span>
    </p>

     <span style="color:#0d0016">
      <strong>

      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">
      无论 Token 是通过
      <strong>
       URL 传递
      </strong>
      还是
      <strong>
       HTTP 头传递
      </strong>

     </span>
    </p>
    <ol>
     <li>
      <span style="color:#0d0016">
       <strong>

     </li>
     <li>
      <span style="color:#0d0016">
       <strong>

       </strong>
       。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        认证成功后，将 Token 绑定到 WebSocket 连接（Channel）
       </strong>

      </span>
     </li>

    <h4>
     <span style="color:#0d0016">
      <strong>
       3.2.1 Netty 服务器端代码实现
      </strong>
     </span>
    </h4>
    <h5>
     <span style="color:#0d0016">
      <strong>
       （1）自定义 WebSocket 握手处理器
      </strong>
     </span>
    </h5>
    <p>
     <span style="color:#0d0016">
      我们需要在 WebSocket 握手阶段解析 Token，并存储到 Channel 的
      <code>
       Attributes
      </code>
      中，方便后续使用。
     </span>
    </p>
    <pre><code class="language-java">import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.SimpleChannelInboundHandler;
import io.netty.handler.codec.http.FullHttpRequest;
import io.netty.handler.codec.http.QueryStringDecoder;
import java.util.List;
import java.util.Map;

public class WebSocketHandshakeHandler extends SimpleChannelInboundHandler&lt;FullHttpRequest&gt; {

    @Override
    protected void channelRead0(ChannelHandlerContext ctx, FullHttpRequest req) {
        // 解析 Token
        String token = extractToken(req);

        if (token == null || !isValidToken(token)) {
            // 认证失败，关闭连接
            ctx.close();
            return;
        }

        // 认证成功，将 Token 绑定到 Channel
        ctx.channel().attr(AttributeKey.valueOf("token")).set(token);

        // 继续处理 WebSocket 握手
        ctx.fireChannelRead(req.retain());
    }

    // 解析 URL 中的 Token
    private String extractToken(FullHttpRequest req) {
        QueryStringDecoder decoder = new QueryStringDecoder(req.uri());
        Map&lt;String, List&lt;String&gt;&gt; params = decoder.parameters();
        if (params.containsKey("token")) {
            return params.get("token").get(0);
        }
        return null;
    }

    // 这里可以调用自己的 Token 认证逻辑，例如 JWT 解析
    private boolean isValidToken(String token) {
        return "your_token_here".equals(token); // 这里替换成实际的 Token 验证逻辑
    }
}
</code></pre>

    <p>
     <span style="color:#0d0016">

    <pre><code class="language-java">import io.netty.channel.ChannelHandlerContext;

import io.netty.handler.codec.http.websocketx.TextWebSocketFrame;

public class WebSocketFrameHandler extends SimpleChannelInboundHandler&lt;TextWebSocketFrame&gt; {

    @Override
    protected void channelRead0(ChannelHandlerContext ctx, TextWebSocketFrame msg) {

        String token = (String) ctx.channel().attr(AttributeKey.valueOf("token")).get();

        if (token == null) {
            ctx.close();
            return;
        }

        // 处理 WebSocket 消息
        System.out.println("Received message: " + msg.text());
        ctx.writeAndFlush(new TextWebSocketFrame("Server received your message: " + msg.text()));
    }
}

      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016">

    <pre><code class="language-java">import io.netty.channel.ChannelInitializer;

import io.netty.handler.codec.http.HttpObjectAggregator;

public class WebSocketServerInitializer extends ChannelInitializer&lt;SocketChannel&gt; {

    <p>
     <span style="color:#0d0016">

     <li>

     </li>
     <li>
      <span style="color:#0d0016">
       <strong>

      <span style="color:#0d0016">

     <span style="color:#0d0016">

    <h3>
     <span style="color:#0d0016">
      <strong>
       4.1 Token 认证逻辑优化
      </strong>
     </span>
    </h3>
    <h4>
     <span style="color:#0d0016">
      <strong>
       🔹 问题：每次握手都要解析 Token，导致性能损耗
      </strong>
     </span>
    </h4>
    <p>
     <span style="color:#0d0016">
      在当前实现中，每次 WebSocket 握手时，服务器都需要解析并验证 Token。如果 Token 解析逻辑复杂（如 JWT 解析、数据库查询等），可能会影响服务器性能。
     </span>
    </p>
    <h4>
     <span style="color:#0d0016">
      <strong>
       ✅ 解决方案：引入缓存，加速 Token 解析
      </strong>
     </span>
    </h4>
    <p>
     <span style="color:#0d0016">
      可以使用
      <strong>
       本地缓存（如 Caffeine）或分布式缓存（如 Redis）
      </strong>
      ，存储已验证的 Token，避免重复解析。例如：
     </span>
    </p>
    <pre><code class="language-java">import com.github.benmanes.caffeine.cache.Cache;
import com.github.benmanes.caffeine.cache.Caffeine;

import java.util.concurrent.TimeUnit;

public class TokenCache {
    // 使用 Caffeine 作为本地缓存，Token 过期时间设为 10 分钟
    private static final Cache&lt;String, Boolean&gt; tokenCache = Caffeine.newBuilder()
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .maximumSize(10_000)
            .build();

    // 校验 Token
    public static boolean isValidToken(String token) {
        // 先查缓存
        Boolean isValid = tokenCache.getIfPresent(token);
        if (isValid != null) {
            return isValid;
        }

        // 若缓存中不存在，则进行 Token 解析（可以是 JWT 解析、数据库查询等）
        boolean valid = verifyTokenFromDatabaseOrJWT(token);

        // 缓存结果，避免重复解析
        tokenCache.put(token, valid);
        return valid;
    }

    // 这里模拟一个 Token 验证逻辑
    private static boolean verifyTokenFromDatabaseOrJWT(String token) {
        return "valid_token".equals(token); // 实际上应调用数据库或 JWT 解析库
    }
}
</code></pre>
    <p>
     <span style="color:#0d0016">
      📌
      <strong>
       优化效果
      </strong>
      ：
     </span>
    </p>
    <ul>
     <li>
      <span style="color:#0d0016">
       第一次验证 Token 仍然执行完整的解析逻辑。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       之后的请求将直接从缓存读取，减少计算量，提高认证效率。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       10 分钟后 Token 过期，防止长期缓存导致安全隐患。
      </span>
     </li>
    </ul>

     <span style="color:#0d0016">
      <strong>
       4.2 Token 续期机制
      </strong>
     </span>
    </h3>
    <h4>
     <span style="color:#0d0016">

    <pre><code class="language-java">import java.util.concurrent.ConcurrentHashMap;
import io.netty.channel.Channel;

public class WebSocketSessionManager {
    private static final ConcurrentHashMap&lt;String, Channel&gt; userSessions = new ConcurrentHashMap&lt;&gt;();

    public static void bindUser(String userId, Channel channel) {
        if (userSessions.containsKey(userId)) {
            // 踢掉之前的连接
            userSessions.get(userId).close();

        userSessions.put(userId, channel);
    }

    public static void removeUser(String userId) {

      <span style="color:#0d0016">

    <h2>
     <span style="color:#0d0016">
      <strong>
       5. 认证通过后的用户信息推送与刷新
      </strong>
     </span>
    </h2>
    <p>
     <span style="color:#0d0016">
      在 WebSocket 认证成功后，用户的身份信息已经确定，此时可以进行
      <strong>
       用户信息的推送和刷新
      </strong>
      。这一环节主要涉及：
     </span>
    </p>
    <ol>
     <li>
      <span style="color:#0d0016">
       <strong>
        用户连接后，主动推送用户的最新信息
       </strong>
       ，确保前端拿到最新状态。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        用户信息发生变化（如昵称、头像、权限等）时，自动推送更新
       </strong>
       ，避免前端数据滞后。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       <strong>
        支持前端主动请求刷新用户信息
       </strong>
       ，保证客户端在需要时能够获取最新数据。
      </span>
     </li>
    </ol>
    <p>
     <span style="color:#0d0016">
      下面，我们逐步分析如何在

      </strong>
      实现这些功能。
     </span>
    </p>
    <h3>
     <span style="color:#0d0016">
      <strong>
       5.1 用户连接后主动推送信息
      </strong>
     </span>
    </h3>
    <h4>
     <span style="color:#0d0016">
      <strong>
       🔹 问题：用户 WebSocket 连接成功后，前端需要获取最新用户信息
      </strong>
     </span>
    </h4>
    <p>
     <span style="color:#0d0016">
      当 WebSocket 连接建立后，前端可能需要：
     </span>
    </p>
    <ul>
     <li>
      <span style="color:#0d0016">
       获取用户的基本信息（如用户名、头像、权限等）。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       获取用户的未读消息数量。
      </span>
     </li>
     <li>
      <span style="color:#0d0016">
       获取其他应用状态数据（如好友在线状态、系统通知等）。
      </span>
     </li>
    </ul>
    <h4>
     <span style="color:#0d0016">
      <strong>
       ✅ 解决方案：在 WebSocket 握手成功后，服务器主动推送用户信息
      </strong>
     </span>
    </h4>
    <p>
     <span style="color:#0d0016">
      我们可以在
      <strong>
       WebSocket 握手完成时
      </strong>
      ，从数据库或缓存中读取用户信息，并主动推送给前端。
     </span>
    </p>
    <h5>
     <span style="color:#0d0016">
      <strong>
       Netty 服务器端实现

     </span>
    </h5>
    <pre><code class="language-java">import io.netty.channel.Channel;

            e.printStackTrace();
        }
    }

    private UserInfo getUserInfoFromDatabase(String userId) {

    }
}

     </span>

    <h4>
     <span style="color:#0d0016">
      <strong>
       🔹 问题：如果用户信息在其他地方修改了，WebSocket 连接的前端不会自动更新
      </strong>
     </span>

    <p>
     <span style="color:#0d0016">
      例如：
     </span>
    </p>
    <ul>
     <li>
      <span style="color:#0d0016">
       用户修改了头像或昵称，前端不会自动更新。

     </li>
     <li>
      <span style="color:#0d0016">

    <p>

        userChannelMap.put(userId, channel);
