     Nodejs中npm i 与npm ci的区别

    <p>
     在 Node.js 里，
     <code>
      npm i
     </code>
     （也就是
     <code>
      npm install
     </code>
     ）和
     <code>
      npm ci
     </code>
     都是用来管理项目依赖的命令，但它们在功能和使用场景上存在差别，下面为你详细说明：
    </p>
    <h4>
     <a id="1_npm_i_3">
     </a>
     1.
     <code>
      npm i
     </code>
    </h4>
    <ul>
     <li>
      <strong>
       功能
      </strong>
      ：它的主要功能是安装项目依赖，会依据项目根目录下的
      <code>
       package.json
      </code>
      文件来安装所有依赖包。要是项目里有
      <code>
       package-lock.json
      </code>
      或者
      <code>
       npm-shrinkwrap.json
      </code>
      文件，
      <code>
       npm i
      </code>
      会尽量依照这些文件里记录的版本来安装依赖；若没有这些文件，
      <code>
       npm
      </code>
      会去下载最新的兼容版本。
     </li>
     <li>
      <strong>
       使用场景
      </strong>
      ：在项目开发过程中使用较为频繁。当你要添加新依赖、更新已有依赖或者在新环境中初始化项目时，都可以使用
      <code>
       npm i
      </code>
      。
     </li>
     <li>
      <strong>
       示例
      </strong>
      ：
      <ul>
       <li>
        安装所有依赖：
       </li>
      </ul>
     </li>
    </ul>
    <pre><code class="prism language-bash"><span class="token function">npm</span> i
</code></pre>
    <pre><code>- 安装单个依赖：
</code></pre>
    <pre><code class="prism language-bash"><span class="token function">npm</span> i lodash
</code></pre>
    <h4>
     <a id="2_npm_ci_16">
     </a>
     2.
     <code>
      npm ci
     </code>
    </h4>
    <ul>
     <li>
      <strong>
       功能
      </strong>
      ：
      <code>
       npm ci
      </code>
      专门用于持续集成（CI）环境，它会严格按照
      <code>
       package-lock.json
      </code>
      或者
      <code>
       npm-shrinkwrap.json
      </code>
      文件里的内容来安装依赖。如果
      <code>
       package.json
      </code>
      里的依赖版本与
      <code>
       package-lock.json
      </code>
      不匹配，
      <code>
       npm ci
      </code>
      会报错，不会尝试安装其他版本。并且，在安装之前，它会先删除
      <code>
       node_modules
      </code>
      目录，以确保安装的依赖是干净且一致的。
     </li>
     <li>
      <strong>
       使用场景
      </strong>
      ：通常在自动化构建、测试或者部署流程中使用，保证每次构建时使用的依赖版本完全相同，从而避免因依赖版本不一致导致的问题。
     </li>
     <li>
      <strong>
       示例
      </strong>
      ：
     </li>
    </ul>
    <pre><code class="prism language-bash"><span class="token function">npm</span> ci
</code></pre>
    <h4>
     <a id="3__24">
     </a>
     3. 两者区别总结
    </h4>
    <ul>
     <li>
      <strong>
       依赖安装的灵活性
      </strong>
      ：
      <code>
       npm i
      </code>
      较为灵活，在没有锁文件时会安装最新兼容版本，有锁文件时也可能因版本范围不同而安装不同版本；
      <code>
       npm ci
      </code>
      严格按照锁文件安装，确保环境的一致性。
     </li>
     <li>
      <strong>
       安装速度
      </strong>
      ：由于
      <code>
       npm ci
      </code>
      会先删除
      <code>
       node_modules
      </code>
      目录，且不进行额外的版本检查和更新操作，所以在大多数情况下比
      <code>
       npm i
      </code>
      安装速度快。
     </li>
     <li>
      <strong>
       依赖管理
      </strong>
      ：
      <code>
       npm i
      </code>
      可以用于添加、更新或删除依赖；
      <code>
       npm ci
      </code>
      只能安装锁文件里指定的依赖，不能用于添加或更新依赖。
     </li>
     <li>
      <strong>
       错误处理
      </strong>
      ：
      <code>
       npm i
      </code>
      在依赖版本不匹配时会尝试解决问题；
      <code>
       npm ci
      </code>
      遇到版本不匹配会直接报错，保证构建环境的稳定性。
     </li>
    </ul>
