    <p>
     Spring Security 是一个强大且可扩展的框架，用于保护 Java 应用程序，尤其是基于 Spring 的应用。它提供了身份验证（验证用户身份）、授权（管理用户权限）和防护机制（如 CSRF 保护和防止会话劫持）等功能。
    </p>
    <p>
     Spring Security 允许开发者通过灵活的配置实现安全控制，确保应用程序的数据和资源安全。通过与其他 Spring 生态系统的无缝集成，Spring Security 成为构建安全应用的理想选择。
    </p>
    <p>
     <strong>
      核心概念
     </strong>
    </p>
    <ul>
     <li>
      <strong>
       身份验证 (Authentication)
      </strong>
      : 验证用户的身份（例如，用户名/密码）。
     </li>
     <li>
      <strong>
       授权 (Authorization)
      </strong>
      : 确定用户是否有权限访问特定资源。
     </li>
     <li>
      <strong>
       安全上下文 (Security Context)
      </strong>
      : 存储已认证用户的详细信息，应用程序中可以访问。
     </li>
    </ul>

    <pre><code>@Bean
public UserDetailsService userDetailsService() {
	// 创建基于内存的用户信息管理器
    InMemoryUserDetailsManager manager = new InMemoryUserDetailsManager();

	
    manager.createUser(
	    // 创建UserDetails对象，用于管理用户名、用户密码、用户角色、用户权限等内容

    // 如果自己配置的有账号密码, 那么上面讲的 user 和 随机字符串 的默认密码就不能用了

}

    <pre><code>@Service
public class UserDetailsServiceImpl implements UserDetailsService {

    @Autowired
    UserMapper userMapper;

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        QueryWrapper&lt;User&gt; queryWrapper = new QueryWrapper&lt;User&gt;();
        queryWrapper.eq("username", username);	// 这里不止可以用username，你可以自定义，主要根据你自己写的查询逻辑
        User user = userMapper.selectOne(queryWrapper);
        if (user == null) {
            throw new UsernameNotFoundException(username);
        }
        return new UserDetailsImpl(user);	// UserDetailsImpl 是我们实现的类
    }
}
</code></pre>

    <h5>
     <a id="13__187">

     1.3 基本的配置
    </h5>
    <p>
     下面这个是
     <code>
      security
     </code>
     的默认配置。我们可以修改并把它加到spring容器中，完成我们特定的需求。
    </p>
    <pre><code>@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
            // 开启授权保护
            .authorizeHttpRequests(authorize -&gt; authorize
                    // 不需要认证的地址有哪些
                    .requestMatchers("/blog/**", "/public/**", "/about").permitAll()	// ** 通配符
                    // 对所有请求开启授权保护
                    .anyRequest().
                    // 已认证的请求会被自动授权
                    authenticated()
            )
            // 使用默认的登陆登出页面进行授权登陆
            .formLogin(Customizer.withDefaults())
            // 启用“记住我”功能的。允许用户在关闭浏览器后，仍然保持登录状态，直到他们主动注销或超出设定的过期时间。
            .rememberMe(Customizer.withDefaults());
    // 关闭 csrf CSRF（跨站请求伪造）是一种网络攻击，攻击者通过欺骗已登录用户，诱使他们在不知情的情况下向受信任的网站发送请求。
    http.csrf(csrf -&gt; csrf.disable());

    return http.build();
}
</code></pre>

     <code>

     <code>
      /api

     <code>

     <code>

     <code>
      /test

     1.4 常用配置
    </h5>
    <p>
     下面这个是我常用的配置，配置了jwt，加密的类，过滤器启用 jwt，而不是session。（jwt的类会在下面讲到）
    </p>
    <pre><code>@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Autowired
    private JwtAuthenticationTokenFilter jwtAuthenticationTokenFilter;

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public AuthenticationManager authenticationManager(AuthenticationConfiguration authConfig) throws Exception {
        return authConfig.getAuthenticationManager();
    }

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.csrf(CsrfConfigurer::disable) // 基于token，不需要csrf
                .sessionManagement((session) -&gt; session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)) // 基于token，不需要session
                .authorizeHttpRequests((authz) -&gt; authz
                        .requestMatchers("/login/",  "/getPicCheckCode").permitAll() // 放行api
                        .requestMatchers(HttpMethod.OPTIONS).permitAll()
                        .anyRequest().authenticated()
                )
                .addFilterBefore(jwtAuthenticationTokenFilter, UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}
</code></pre>
    <h4>
     <a id="2_258">
     </a>
     2、加密
    </h4>
    <p>
     Spring Security 提供了多种加密和安全机制来保护用户的敏感信息，尤其是在用户身份验证和密码管理方面。（我们只讲默认的
     <code>
      BCryptPasswordEncoder
     </code>
     ）
    </p>
    <h5>
     <a id="1__262">
     </a>
     1. 密码加密的重要性
    </h5>
    <p>
     在应用程序中，用户密码是最敏感的数据之一。为了防止密码泄露，即使数据库被攻击者获取，密码也应以加密形式存储。加密可以保护用户的隐私，并在一定程度上增加安全性。
    </p>
    <h5>
     <a id="2_Spring_Security__266">
     </a>
     2. Spring Security 的加密机制
    </h5>
    <h6>
     <a id="21_PasswordEncoder__268">
     </a>
     2.1 PasswordEncoder 接口
    </h6>
    <p>
     Spring Security 提供了
     <code>
      PasswordEncoder
     </code>
     接口，用于定义密码的加密和验证方法。主要有以下几种实现：
    </p>
    <ul>
     <li>
      <strong>
       BCryptPasswordEncoder
      </strong>
      ：基于 BCrypt 算法，具有适应性和强大的加密强度。它可以根据需求自动调整加密的复杂性。
     </li>
     <li>
      <strong>
       NoOpPasswordEncoder
      </strong>
      ：不执行加密，适用于开发和测试环境，不建议在生产环境中使用。
     </li>
     <li>
      <strong>
       Pbkdf2PasswordEncoder
      </strong>
      、
      <strong>
       Argon2PasswordEncoder
      </strong>
      等：这些都是基于不同算法的实现，具有不同的安全特性。
     </li>
    </ul>
    <h6>
     <a id="22_BCrypt__276">
     </a>
     2.2 BCrypt 算法
    </h6>
    <p>
     BCrypt 是一种安全的密码哈希算法，具有以下特点：
    </p>
    <ul>
     <li>
      <strong>
       盐值（Salt）
      </strong>
      ：每次加密时都会生成一个随机盐值，确保相同的密码在每次加密时生成不同的哈希值。
     </li>
     <li>
      <strong>
       适应性
      </strong>
      ：通过增加计算复杂性（如工作因子），可以提高密码的加密强度。
     </li>
    </ul>
    <h5>
     <a id="3__PasswordEncoder_283">
     </a>
     3. 使用 PasswordEncoder
    </h5>
    <h6>
     <a id="31__PasswordEncoder_285">
     </a>
     3.1 配置 PasswordEncoder
    </h6>
    <p>
     可以在 Spring Security 的配置类中定义
     <code>
      PasswordEncoder
     </code>
     bean，例如：
    </p>
    <pre><code>@Configuration
public class SecurityConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        // 也可用有参构造，取值范围是 4 到 31，默认值为 10。数值越大，加密计算越复杂
        return new BCryptPasswordEncoder();	
    }
}
</code></pre>
    <h6>
     <a id="32__300">
     </a>
     3.2 加密密码
    </h6>
    <p>
     在注册用户或更新密码时，可以使用
     <code>
      PasswordEncoder
     </code>
     来加密密码：
    </p>
    <pre><code>@Autowired
private PasswordEncoder passwordEncoder;

public void registerUser(String username, String rawPassword) {
    String encryptedPassword = passwordEncoder.encode(rawPassword);
    // 保存用户信息到数据库，包括加密后的密码
}
</code></pre>
    <h6>
     <a id="33__313">
     </a>
     3.3 验证密码
    </h6>
    <p>
     在用户登录时，可以使用
     <code>
      matches
     </code>
     方法验证输入的密码与存储的加密密码是否匹配：
    </p>
    <pre><code>public boolean login(String username, String rawPassword) {
    // 从数据库中获取用户信息，包括加密后的密码
    String storedEncryptedPassword = // 从数据库获取;
    return passwordEncoder.matches(rawPassword, storedEncryptedPassword);
}
</code></pre>

    <h4>
     <a id="6_758">
     </a>
     6、授权
    </h4>
    <p>
     Spring Security 的授权机制用于控制用户对应用程序资源的访问权限。授权通常是基于用户角色或权限的，以下是对 Spring Security 授权的详细讲解。
    </p>
    <h5>
     <a id="1__762">
     </a>
     1. 授权的基本概念
    </h5>
    <ul>
     <li>
      <strong>
       授权（Authorization）
      </strong>
      ：指的是决定用户是否有权限访问特定资源的过程。在用户认证成功后，系统会根据预定义的规则来判断用户是否可以访问某些资源。
     </li>
    </ul>
    <h5>
     <a id="2__766">
     </a>
     2. 授权的主要组成部分
    </h5>
    <h6>
     <a id="21__768">
     </a>
     2.1 权限与角色
    </h6>
    <ul>
     <li>
      <strong>
       权限（Permission）
      </strong>
      ：通常指对特定资源的操作能力，如“读取”、“写入”或“删除”。
     </li>
     <li>
      <strong>
       角色（Role）
      </strong>
      ：一组权限的集合。例如，管理员角色可能具有所有权限，而普通用户角色可能只有读取权限。
     </li>
    </ul>
    <h6>
     <a id="22_GrantedAuthority__773">
     </a>
     2.2 GrantedAuthority 接口
    </h6>
    <ul>
     <li>
      Spring Security 使用
      <code>
       GrantedAuthority
      </code>
      接口表示用户的权限。每个用户的权限可以通过实现该接口的对象来表示。
     </li>
    </ul>
    <h5>
     <a id="3__777">
     </a>
     3. 授权方式
    </h5>
    <p>
     Spring Security 支持多种授权方式，主要包括：
    </p>
    <h6>
     <a id="31__781">
     </a>
     3.1 基于角色的授权
    </h6>
    <ul>
     <li>
      <p>
       <strong>
        定义角色
       </strong>
       ：通常在用户注册或用户管理中定义。
      </p>
     </li>
     <li>
      <p>
       <strong>
        配置角色授权
       </strong>
       ：可以在 Spring Security 配置类中设置基于角色的访问控制。
      </p>
      <p>
       @Override
       <br/>
       protected void configure(HttpSecurity http) throws Exception {
       <!-- -->
       <br/>
       http
       <br/>
       .authorizeRequests()
       <br/>
       .antMatchers(“/admin/
       <strong>
        “).hasRole(“ADMIN”) // 只有 ADMIN 角色可以访问
        <br/>
        .antMatchers(”/user/
       </strong>
       ”).hasAnyRole(“USER”, “ADMIN”) // USER 和 ADMIN 角色可以访问
       <br/>
       .anyRequest().authenticated(); // 其他请求需要认证
       <br/>
       }
      </p>
     </li>
    </ul>
    <h6>
     <a id="32__796">
     </a>
     3.2 基于权限的授权
    </h6>
    <ul>
     <li>
      <p>
       <strong>
        定义权限
       </strong>
       ：与角色类似，定义更细粒度的权限。
      </p>
     </li>
     <li>
      <p>
       <strong>
        配置权限授权
       </strong>
       ：在配置类中设置基于权限的访问控制。
      </p>
      <p>
       @Override
       <br/>
       protected void configure(HttpSecurity http) throws Exception {
       <!-- -->
       <br/>
       http
       <br/>
       .authorizeRequests()
       <br/>
       .antMatchers(“/edit/**”).hasAuthority(“EDIT_PRIVILEGE”) // 仅具有 EDIT_PRIVILEGE 权限的用户可以访问
       <br/>
       .anyRequest().authenticated();
       <br/>
       }
      </p>
     </li>
    </ul>
    <h5>
     <a id="4__810">
     </a>
     4. 自定义授权逻辑
    </h5>
    <p>
     如果需要更复杂的授权逻辑，可以实现自定义的
     <code>
      AccessDecisionVoter
     </code>
     或
     <code>
      AccessDecisionManager
     </code>
     。
    </p>
    <h6>
     <a id="41_AccessDecisionVoter_814">
     </a>
     4.1 AccessDecisionVoter
    </h6>
    <ul>
     <li>
      负责评估用户的访问请求，返回授权决策。
     </li>
    </ul>
    <h6>
     <a id="42_AccessDecisionManager_818">
     </a>
     4.2 AccessDecisionManager
    </h6>
    <ul>
     <li>
      管理多个
      <code>
       AccessDecisionVoter
      </code>
      的调用，确定用户是否有权访问请求的资源。
     </li>
    </ul>
