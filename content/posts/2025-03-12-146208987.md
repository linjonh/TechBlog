    <h2>
     <a id="Kubernetes__YAML__0">
     </a>
     Kubernetes 中 YAML 文件详解
    </h2>
    <p>
     在 Kubernetes（简称 K8s）的世界里，YAML（Yet Another Markup Language）文件是核心配置工具。K8s 作为一个强大的容器编排系统，允许用户通过 YAML 文件来定义和管理各种资源，如 Pods、Deployments、Services 等。使用 YAML 文件可以实现声明式配置，清晰地描述系统的期望状态，而 K8s 会自动确保实际状态与期望状态一致。下面将详细介绍 K8s 中 YAML 文件的结构、常用字段以及不同资源类型的 YAML 示例。
    </p>
    <h2>
     <a id="YAML__2">
     </a>
     二、YAML 文件基础结构
    </h2>
    <p>
     所有 K8s 资源的 YAML 文件通常包含以下四个基本部分：
    </p>
    <h3>
     <a id="1_apiVersion_4">
     </a>
     1. apiVersion
    </h3>
    <p>
     指定使用的 Kubernetes API 版本，不同的资源类型可能使用不同的 API 版本。例如，v1 是核心 API 版本，常用于 Pod、Service 等资源；apps/v1 用于 Deployment、StatefulSet 等工作负载资源。
    </p>
    <h3>
     <a id="2_kind_6">
     </a>
     2. kind
    </h3>
    <p>
     明确资源的类型，如 Pod、Deployment、Service、ConfigMap 等。
    </p>
    <h3>
     <a id="3_metadata_8">
     </a>
     3. metadata
    </h3>
    <p>
     包含资源的元数据，如名称（name）、命名空间（namespace）、标签（labels）和注解（annotations）等。
     <br/>
     name：资源的唯一标识符，在同一命名空间内必须唯一。
     <br/>
     namespace：可选字段，用于将资源划分到不同的逻辑组中，默认是 default 命名空间。
     <br/>
     labels：键值对形式，用于对资源进行分类和选择，例如可以根据应用名称、环境等进行标记。
     <br/>
     annotations：也是键值对，但主要用于存储非标识性的额外信息，供工具或系统使用。
    </p>
    <h3>
     <a id="4_spec_14">
     </a>
     4. spec
    </h3>
    <p>
     定义资源的具体规格，不同的资源类型其 spec 部分差异很大。例如，对于 Pod 资源，spec 会包含容器的定义；对于 Deployment 资源，spec 会包含副本数量、选择器和 Pod 模板等信息。
     <br/>
     以下是一个简单的 Pod 资源 YAML 文件示例：
    </p>
    <pre><code class="prism language-bash">apiVersion: v1
kind: Pod
metadata:
  name: my-pod
  labels:
    app: my-app
spec:
  containers:
  - name: my-container
    image: nginx:1.19
    ports:
    - containerPort: <span class="token number">80</span>
</code></pre>
    <h2>
     <a id="_YAML__33">
     </a>
     三、常用资源类型的 YAML 配置
    </h2>
    <h3>
     <a id="1_Pod_34">
     </a>
     1. Pod
    </h3>
    <p>
     Pod 是 K8s 中最小的可部署单元，一个 Pod 可以包含一个或多个紧密相关的容器。除了上述示例中的基本配置外，还可以添加更多字段，如资源请求与限制、环境变量、健康检查等。
    </p>
    <pre><code class="prism language-bash">apiVersion: v1
kind: Pod
metadata:
  name: my-pod
  labels:
    app: my-app
spec:
  containers:
  - name: my-container
    image: nginx:1.19
    ports:
    - containerPort: <span class="token number">80</span>
    resources:
      requests:
        cpu: <span class="token string">"200m"</span>
        memory: <span class="token string">"512Mi"</span>
      limits:
        cpu: <span class="token string">"500m"</span>
        memory: <span class="token string">"1Gi"</span>
    env:
    - name: DB_HOST
      value: <span class="token string">"db.example.com"</span>
    livenessProbe:
      httpGet:
        path: /healthz
        port: <span class="token number">80</span>
      initialDelaySeconds: <span class="token number">15</span>
      periodSeconds: <span class="token number">5</span>
</code></pre>
    <p>
     resources：用于设置容器对 CPU 和内存的请求和限制，requests 表示容器运行所需的最小资源，limits 表示容器最多能使用的资源。
     <br/>
     env：定义容器的环境变量，这里设置了一个名为 DB_HOST 的环境变量。
     <br/>
     livenessProbe：存活探针，用于检查容器是否正常运行。这里使用 HTTP GET 请求 /healthz 路径来判断容器的健康状态。
    </p>
    <h3>
     <a id="2_Deployment_72">
     </a>
     2. Deployment
    </h3>
    <p>
     Deployment 用于管理 Pod 的副本数量，并支持滚动更新、回滚等操作。
    </p>
    <pre><code class="prism language-bash">apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-deployment
spec:
  replicas: <span class="token number">3</span>
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-container
        image: nginx:1.19
        ports:
        - containerPort: <span class="token number">80</span>
</code></pre>
    <p>
     replicas：指定要运行的 Pod 副本数量。
     <br/>
     selector：用于选择要管理的 Pod，通过匹配标签来确定。
     <br/>
     template：定义 Pod 的模板，即创建 Pod 时使用的配置。
    </p>
    <h3>
     <a id="3_Service_101">
     </a>
     3. Service
    </h3>
    <p>
     Service 用于为一组 Pod 提供统一的访问入口，实现服务发现和负载均衡。
    </p>
    <pre><code class="prism language-bash">apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app: my-app
  ports:
    - protocol: TCP
      port: <span class="token number">80</span>
      targetPort: <span class="token number">80</span>
  type: ClusterIP
</code></pre>
    <p>
     selector：同样用于选择要关联的 Pod。
     <br/>
     ports：定义服务的端口映射，port 是服务对外暴露的端口，targetPort 是 Pod 内部容器的端口。
     <br/>
     type：指定服务的类型，常见的有 ClusterIP（默认，只能在集群内部访问）、NodePort（通过节点的端口暴露服务）、LoadBalancer（使用云提供商的负载均衡器）等。
    </p>
    <h3>
     <a id="4_ConfigMap_122">
     </a>
     4. ConfigMap
    </h3>
    <p>
     ConfigMap 用于存储非敏感的配置数据，可将其注入到 Pod 中作为环境变量或配置文件。
    </p>
    <pre><code class="prism language-bash">apiVersion: v1
kind: ConfigMap
metadata:
  name: my-config
data:
  app.properties: <span class="token operator">|</span>
    <span class="token assign-left variable">db.host</span><span class="token operator">=</span>db.example.com
    <span class="token assign-left variable">db.port</span><span class="token operator">=</span><span class="token number">3306</span>
yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
  - name: my-container
    image: my-app-image
    env:
    - name: DB_HOST
      valueFrom:
        configMapKeyRef:
          name: my-config
          key: app.properties
          optional: <span class="token boolean">false</span>
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config
  volumes:
  - name: config-volume
    configMap:
      name: my-config
</code></pre>
    <p>
     在 ConfigMap 中，data 字段存储配置数据，可以是键值对或文件内容。
     <br/>
     在 Pod 中，可以通过 env 字段将 ConfigMap 中的数据作为环境变量注入，也可以通过 volumeMounts 和 volumes 将其挂载为配置文件。
    </p>
    <h3>
     <a id="5_Secret_161">
     </a>
     5. Secret
    </h3>
    <p>
     Secret 用于存储敏感数据，如密码、令牌等，其使用方式与 ConfigMap 类似，但数据会进行加密存储。
    </p>
    <pre><code class="prism language-bash">apiVersion: v1
kind: Secret
metadata:
  name: my-secret
type: Opaque
data:
  username: base64_encoded_username
  password: base64_encoded_password
yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
  - name: my-container
    image: my-app-image
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: my-secret
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: my-secret
          key: password
</code></pre>
    <p>
     在 Secret 中，data 字段的值需要进行 Base64 编码。
     <br/>
     在 Pod 中，可以通过 env 字段将 Secret 中的数据作为环境变量注入。
    </p>
    <h2>
     <a id="YAML__197">
     </a>
     四、YAML 文件的编写和使用注意事项
    </h2>
    <h3>
     <a id="1__198">
     </a>
     1. 语法规则
    </h3>
    <p>
     YAML 文件使用缩进表示层级关系，必须使用空格而不是制表符进行缩进，否则会导致解析错误。同时，注意冒号后面需要有一个空格。
    </p>
    <h3>
     <a id="2__200">
     </a>
     2. 验证和调试
    </h3>
    <p>
     可以使用 kubectl 命令来验证 YAML 文件的正确性，例如 kubectl create --dry-run=client -f your-file.yaml 可以在不实际创建资源的情况下检查文件的语法和有效性。如果出现错误，K8s 会给出相应的错误信息，帮助你定位问题。
    </p>
    <h3>
     <a id="3__202">
     </a>
     3. 多资源定义
    </h3>
    <p>
     一个 YAML 文件中可以定义多个资源，使用 — 分隔不同的资源定义。例如：
    </p>
    <pre><code class="prism language-bash">apiVersion: v1
kind: Pod
metadata:
  name: pod-1
spec:
  containers:
  - name: container-1
    image: nginx:1.19
---
apiVersion: v1
kind: Service
metadata:
  name: service-1
spec:
  selector:
    app: my-app
  ports:
    - protocol: TCP
      port: <span class="token number">80</span>
      targetPort: <span class="token number">80</span>
  type: ClusterIP
</code></pre>
    <h3>
     <a id="4__229">
     </a>
     4. 版本兼容性
    </h3>
    <p>
     不同版本的 K8s 对 API 版本和字段的支持可能有所不同，在编写 YAML 文件时要确保使用的 API 版本和字段与当前 K8s 集群版本兼容。可以参考 K8s 官方文档来获取最新的版本信息和使用说明。
    </p>
