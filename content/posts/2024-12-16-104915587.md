    <h2>
     <a id="_1">
     </a>
     单元测试过程
    </h2>
    <p>
     定义：单元测试是对软件
     <strong>
      基础组成单元
     </strong>
     进行的测试
     <br/>
     时机：一般在
     <strong>
      代码完成后由开发人员完成
     </strong>
     ，QA人员辅助
     <br/>
     对象：类、模块、组件、单元
    </p>
    <h3>
     <a id="_7">
     </a>
     单元测试
    </h3>
    <ol>
     <li>
      单元测试的依据是软件的
      <strong>
       详细设计描述、源程序清单、编码标准
      </strong>
      等。
     </li>
     <li>
      单元测试一般应该由编程人员完成，有时测试人员也加入进来，但编程人员扔会起到主要作用。
     </li>
     <li>
      多个被测试模块之间的单元测试可同时进行，以提高单元测试效率。
     </li>
     <li>
      单元测试是对软件组成的基本单元测试。
      <ul>
       <li>
        在传统的结构化编程语言如c语言中，单元一般是模块，也就是函数或子过程。
       </li>
       <li>
        在象c++中，单元是类和类的方法
       </li>
       <li>
        在Ada语言中，单元可为独立的过程、函数或Ada包
       </li>
       <li>
        在第四代语言（4GL）中，单元对应为一个菜单或显示界面。
       </li>
      </ul>
     </li>
    </ol>
    <h3>
     <a id="_17">
     </a>
     单元测试的目的
    </h3>
    <ul>
     <li>
      验证
      <strong>
       代码
      </strong>
      是否达到详细设计的预期要求（概要设计-&gt;集成测试）
     </li>
     <li>
      发现代码中不符合
      <strong>
       编码规范
      </strong>
      的地方
     </li>
     <li>
      准确定位发现的错误，以便排除错误
     </li>
    </ul>
    <h3>
     <a id="_22">
     </a>
     单元测试的优点
    </h3>
    <ul>
     <li>
      单元测试在编码过程中（在所有测试前），若发现一个错误，不论是从做回归测试的角度，还是对错误原因理解的深刻性的角度，修复错误的成本远小于集成测试阶段，更小于系统测试阶段（
      <strong>
       效益更优
      </strong>
      ）
     </li>
     <li>
      在编码过程中考虑单元测试的问题，有助于编程人员养成更良好的
      <strong>
       编程习惯
      </strong>
      （
      <strong>
       规范
      </strong>
      ），提高源代码质量
     </li>
    </ul>
    <h3>
     <a id="_26">
     </a>
     单元测试的步骤
    </h3>
    <p>
     实施应遵循一定的步骤。
    </p>
    <ol>
     <li>
      <strong>
       计划
      </strong>
      单元测试
     </li>
     <li>
      <strong>
       设计
      </strong>
      单元测试
     </li>
     <li>
      <strong>
       实现
      </strong>
      单元测试
     </li>
     <li>
      <strong>
       执行
      </strong>
      单元测试
     </li>
     <li>
      结果分析并提交
      <strong>
       测试报告
      </strong>
     </li>
    </ol>
    <h3>
     <a id="_34">
     </a>
     单元测试环境构成
    </h3>
    <p>
     单元测试中，单元测试一般为编码步骤的附属部分，模块不是独立的程序，自己不能运行，要靠其他部分来驱动，要为每个单元模块开发来两个软件：
    </p>
    <ul>
     <li>
      驱动模块（Driver）:调用模块的运行
     </li>
     <li>
      桩模块（Stub）:测试接口
      <br/>
      驱动一般少于桩，尽量避免开发桩
      <br/>
      若采用自底向上的方式开发，底层的单元先开发并测试，则避免了底层桩模块的开发。
     </li>
    </ul>
    <h4>
     <a id="_41">
     </a>
     如何构建单元测试的环境
    </h4>
    <ul>
     <li>
      构造最小运行调度系统，即构造被测单元的驱动模块
     </li>
     <li>
      模拟被测单元的接口，即构造桩模块
     </li>
     <li>
      模拟相应测试集中的测试数据
     </li>
    </ul>
    <h3>
     <a id="_46">
     </a>
     单元测试的内容
    </h3>
    <ul>
     <li>
      模块接口
     </li>
     <li>
      局部数据结构测试
     </li>
     <li>
      路径测试
     </li>
     <li>
      错误处理测试
     </li>
     <li>
      边界测试
     </li>
    </ul>
    <h3>
     <a id="_53">
     </a>
     模块接口测试
    </h3>
    <ul>
     <li>
      调用所测试模块的输入参数与模块的形式参数在个数、属性、顺序上是否匹配
     </li>
     <li>
      所测试模块调用子模块时，它输入子模块的参数和子模块的形式参数在个数、属性、顺序上是否匹配
     </li>
     <li>
      是否修改了只做输入用的形式参数
     </li>
     <li>
      输出给标准函数的参数在个数上、属性上、顺序上是否匹配
     </li>
     <li>
      全局变量的定义在各模块中是否一致
     </li>
     <li>
      限制是否通过形式参数来传送
     </li>
    </ul>
    <h3>
     <a id="_62">
     </a>
     局部数据结构测试
    </h3>
    <ul>
     <li>
      检查不正确或不一致的数据类型说明；
     </li>
     <li>
      使用尚未赋值或尚未初始化的变量；
     </li>
     <li>
      错误的初始值或错误的默认值；
     </li>
     <li>
      变量名拼写错误或书写错误；
     </li>
     <li>
      不一致的数据类型。
     </li>
    </ul>
    <h3>
     <a id="_68">
     </a>
     路径测试
    </h3>
    <p>
     <strong>
      1. 常见的不正确的计算有：
     </strong>
    </p>
    <ul>
     <li>
      运算的优先次序不正确或误解了运算的优先次序；
     </li>
     <li>
      运算的方式错误（运算的对象彼此在类型上不相容）；
     </li>
     <li>
      算法错误；
     </li>
     <li>
      初始化不正确；
     </li>
     <li>
      运算精度不够；
     </li>
     <li>
      表达式的符号表示不正确等。
     </li>
    </ul>
    <p>
     <strong>
      2. 常见的比较和控制流错误有：
     </strong>
    </p>
    <ul>
     <li>
      不同数据类型的比较；
     </li>
     <li>
      不正确的逻辑运算符或优先次序；
     </li>
     <li>
      因浮点运算精度问题而造成的两值比较不等；
     </li>
     <li>
      关系表达式中不正确的变量和比较符；
     </li>
     <li>
      “差1错”，即不正确地多循环或少循环一次；
     </li>
     <li>
      错误的或不可能的循环终止条件；
     </li>
     <li>
      当遇到发散的迭代时不能终止循环；
     </li>
     <li>
      不适当地修改了循环变量等。
     </li>
    </ul>
    <h3>
     <a id="_86">
     </a>
     错误处理测试（容错）
    </h3>
    <ul>
     <li>
      出错的描述难以理解；
     </li>
     <li>
      出错的描述不足以对错误定位和确定出错的原因；显示的错误与实际的错误不符；
     </li>
     <li>
      对错误条件的处理不正确；在对错误进行处理之前，错误条件已经引起系统的干预；
     </li>
     <li>
      如果出错情况不予考虑，那么检查恢复正常后模块可否正常工作。
     </li>
    </ul>
    <h3>
     <a id="_91">
     </a>
     边界测试
    </h3>
    <ul>
     <li>
      在n次循环的第0次、1次、n次是否有错误；
     </li>
     <li>
      运算或判断中取最大最小值时是否有错误；
     </li>
     <li>
      数据流、控制流中刚好等于、大于、小于确定的比较值时是否出现错误。
     </li>
    </ul>
    <h3>
     <a id="_95">
     </a>
     单元测试类型
    </h3>
    <ul>
     <li>
      逻辑单元测试；
     </li>
     <li>
      集成单元测试；
     </li>
     <li>
      功能单元测试。
      <br/>
      单元测试应试应从
      <strong>
       各个层次
      </strong>
      来对单元
      <strong>
       内部算法
      </strong>
      、
      <strong>
       外部功能
      </strong>
      实现等进行检验，包括对
      <strong>
       程序代码的评审
      </strong>
      和通过运行单元程序来验证其
      <strong>
       功能特性
      </strong>
      等内容。一般单元测试应紧接在编码之后，当源程序编制完成并通过复审和编译检查，便可开始单元测试。测试用例的设计应与复审工作相结合，根据设计信息选取测试数据，将增大发现上述各类错误的可能性。
      <em>
       在确定测试用例的同时，应给出期望结果。
      </em>
      <br/>
      进行单元测试时，常用的方法是采用
      <strong>
       白盒测试
      </strong>
      ，辅之以
      <strong>
       黑盒测试
      </strong>
      。
     </li>
    </ul>
    <p>
     根据测试对象的其内部结构的逻辑关系、测试的方法，按照由小到大、由单一到组合、又简单到复杂，单元测试可以逻辑单元测试、集成单元测试和功能单元测试。

    <h2>
     <a id="_105">
     </a>
     断言
    </h2>
    <h5>
     <a id="_106">
     </a>
     定义：
    </h5>
    <p>
     简单的方法调用，判断一个语句、一个函数或对象的一个方法所产生的结果是否符合你期望的那个结集（为真）。
    </p>
    <h5>
     <a id="_108">
     </a>
     时机：
    </h5>
    <ol>
     <li>
      用错误处理代码来处理预期会发生的状况，用断言来处理绝不应该发生的状
     </li>
     <li>
      用断言注解并验前（置），条件和后（置）条件赶链的花构，应该先使用断管再处理错要
     </li>
    </ol>
    <h5>
     <a id="_111">
     </a>
     应用场景：
    </h5>
    <ol>
     <li>
      在在功能代码开发阶段，可以逐步添加断言测试是否获取自己想要的数据结果
     </li>
     <li>
      写单元测试时，可用到断言，主要目的：测试这个功能片段的代码能否返回预期的结果
     </li>
     <li>
      自己提供接口供他人使用时，可先断言使用者传递过来的参数是否符合要求，如果不符合要求，将以AssertionError 的方式告知使用者。
     </li>
    </ol>
    <ul>
     <li>
      assertion（断言）是软件开发中一种常用的调试方式，很多开发语言中都支持这种机制。
     </li>
     <li>
      assertion 就是在程序中的一条语句，它对一个布尔表达式进行检查，必须保证这个表达式的值为true；如果为false，则说明程序已经处于不正确的状态，系统将给出警告或退出。
     </li>
     <li>
      assertion 用于保证程序最基本、关键的正确性。
     </li>
     <li>
      assertion 通常在开发和测试时开启。为了提高性能，在软件发布后，assertion通常是关闭的。
      <br/>

     </li>
    </ul>
    <h3>
     <a id="_123">
     </a>
     单元测试的目的
    </h3>
    <ul>
     <li>
      验证代码能否达到详细设计的预期要求。
     </li>
     <li>
      发现代码中不符合编码规范的地方。口准确定位发现的错误，以便排除错误。
     </li>
    </ul>
    <h3>
     <a id="_126">
     </a>
     单元测试的作用
    </h3>
    <ul>
     <li>
      编写单元测试可以帮助开发人员书写高质量的代码。
     </li>
     <li>
      编写单元测试可以使开发人员更有信心重构应用程序，去拥抱变化。
     </li>
    </ul>
