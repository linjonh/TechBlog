    <h2>
     <a id="_10">

     性能测试理论知识
    </h2>
    <h3>
     <a id="_11">
     </a>
     什么是性能？
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/b5e8a37eac284dc89d7a97404b13a7c6.png">
      <br/>
      通俗的讲， 时间特性指的就是软件系统处理用户请求的响应时间。资源特性指的是在软件系统运行过程中，系统资源的消耗情况。系统资源大致包括：CPU、内存、磁盘IO等
     </img>
    </p>
    <h3>
     <a id="_15">
     </a>
     什么是性能测试？
    </h3>
    <p>
     性能测试就是测试人员借助性能测试工具，模拟系统在不同场景下，对系统的各项性能指标进行测试。
    </p>
    <p>
     以电商软件为例，系统会出现以下常见场景：
    </p>
    <ol>
     <li>
      日常在线用户500万（稳定运行情况下）
     </li>
     <li>
      双11当天，在线用户3亿（超大量用户）
     </li>
     <li>
      双11零点，有一千万用户同时秒杀抢购（大量用户同时使用）
     </li>
    </ol>
    <h3>
     <a id="_25">
     </a>
     性能测试的作用
    </h3>
    <ol>
     <li>
      系统上线时， 评估系统能力
     </li>
     <li>
      系统上线后， 寻找性能瓶颈， 优化性能
     </li>
     <li>
      评估软件能否满足未来的需要
     </li>
    </ol>
    <h3>
     <a id="_31">
     </a>
     性能测试与功能测试的区别
    </h3>
    <p>
     <strong>
      功能测试：
     </strong>
     <br/>
     功能测试主要关注系统是否按照需求规格说明书中定义的功能进行正常运行，并符合用户的期望。其目的是验证软件的有效性或正确性，即确认软件在各种输入情况下能够按照业务逻辑正确地处理数据，并产生预期的输出结果。
    </p>
    <p>
     <strong>
      性能测试：
     </strong>
     <br/>
     性能测试主要关注系统在不同负载和压力下的表现和响应能力。它通过模拟实际使用场景或特定负载情况，评估系统的性能指标，例如响应时间、吞吐量、并发用户数等。性能测试的目的是评估系统的效率、可靠性和可扩展性，确保系统能够在预期的负载下稳定运行。
    </p>
    <p>
     <strong>
      测试方法：
     </strong>
     <br/>
     功能测试通常采用黑盒测试方法，测试人员基于需求规格说明书或用户需求进行测试，验证软件是否满足特定功能要求。
    </p>
    <p>
     性能测试通常采用白盒测试方法，测试人员需要了解系统的内部机制和架构，并使用性能测试工具进行负载模拟和性能指标的监测与分析。
    </p>
    <p>
     <strong>
      一般是功能测试通过之后做性能测试。
     </strong>
    </p>
    <h3>
     <a id="_46">
     </a>
     性能测试常见术语
    </h3>
    <ol>
     <li>
      <p>
       并发：并发就是大量用户在同一时间访问同一系统。
       <br/>
       业务层面的并发用户数：指的是同时向服务器发送请求的用户数量。
       <br/>
       后端服务器层面的并发用户数：指的是同时向服务器发送请求的请求数量。
      </p>
     </li>
     <li>
      <p>
       用户数
       <br/>
       系统用户数：系统注册的用户
       <br/>
       在线用户数：成功登陆系统用户数
       <br/>
       并发用户数：同时使用系统的用户数量。
      </p>
     </li>
    </ol>
    <h3>
     <a id="_57">
     </a>
     性能测试的策略
    </h3>
    <h4>
     <a id="_58">
     </a>
     基准测试
    </h4>
    <p>
     基准测试狭义上又称单用户测试 ，当性能测试环境确定后，通常选取业务模型中的重要业务做基准测试，对被测系统施加⼀定压力，从而获取被测系统在单用户运行情况下的各项性能指标，并作为后续性能优化和比较、多用户并发测试和混合场景测试等的参考依据。
    </p>
    <p>
     广义上讲， 基准测试是一种评估软件性能指标的活动。通过基准测试确定系统的性能基准线，当系统的软硬件发生变化后再进行测试，对比基准线来确定变化对性能的影响。
    </p>
    <h4>
     <a id="_64">
     </a>
     负载测试
    </h4>
    <p>
     负载测试通过逐步加载的方式，确定在满足系统性能指标的情况下，找出系统所能够承受的最大负载量的测试。
    </p>
    <p>
     负载测试类似于举重运动，通过不断给运动员增加重量，确定运动员在其身体状况保持正常的情况下所能举起的最大重量。
    </p>
    <p>
     例如，⼀个软件系统的响应时间要求不超过2秒，如果在这个前提下不断增加用户访问量，系统的响应时间就会变长。假设当访问量超过1万⼈时系统的响应时间超过2秒，那么就可以确定在系统响应时间不超过2秒的前提下，系统的最大负载量是1万⼈。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/619dcb069d384c4fbdc029d29bae9f0c.png"/>
    </p>
    <h4>
     <a id="_73">
     </a>
     稳定性测试
    </h4>
    <p>
     稳定性测试是在服务器稳定运行（日常使用的业务负载）的情况下进行长时间测试，并保证服务器能满足业务需求。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ad8ae3aaac454529853473075cf5d76c.png">
      <br/>
      场景一： 实际测试的最大负载达不到用户正常使用的15人，那么就说明设计不合理，不进行稳定性测试。 类比到软件上就应该回去检查代码是否出现了BUG、或者设计不合理等情况。
      <br/>
      场景二：稳定运行的负载量为10。 要根据需求进行设定， 稳定性测试并不是在最大负载情况下进行长时间测试。
     </img>
    </p>
    <h4>
     <a id="_80">
     </a>
     压力测试
    </h4>
    <p>
     压力测试用于评估被测系统在高于预期、高于指定容量负载需求或低于最少需求资源的条件下，查看系统是否有良好的容错能力和可恢复能力。
    </p>
    <p>
     进行压力测试时通常采用逐步增加系统负载的方式，使系统某些资源达到饱和甚至失效，从而发现那些只有在高负载条件下才会出现的缺陷，如同步问题、内存泄漏等。
    </p>
    <p>
     通过对被测系统进行压力测试，也能找出被测系统的性能拐点，获得系统所能提供的最大服务级别（系统所能承受的最大压力），评估系统在峰值负载或超出最大负载情况下的处理能力。
    </p>
    <p>
     <strong>
      压力测试的方式：
     </strong>
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/c16d283153bf4297935a6b58cdf2dad0.png">
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/c2034335b90e492ca91d700640ba46fc.png">
       <br/>
       如上图所示，随着用户数（负载）的逐步上升， 系统资源、吞吐量、系统响应时间的变化。
       <br/>
       在A-B区间，随着系统用户数的增加，系统的处理能力会增大。（低负载区）
       <br/>
       在B-C区间，随着系统用户数的增加，系统的处理能力基本不变。（高负载区）
       <br/>
       在C-D区间，随着系统用户数的增加，系统的处理能力减小。（崩溃区）
      </img>
     </img>
    </p>
    <p>
     <strong>
      压力测试和负载测试的区别：
     </strong>
     <br/>
     负载测试是在保持性能指标要求的前提下测试系统能够承受的最大负载。
    </p>
    <p>
     压力测试则是测试系统性能达到极限的状态。
    </p>
    <p>
     例如，软件系统要求的响应时间不超过2秒。进行负载测试时发现，当访问量达到1万时，系统响应时间不超过2秒，而当访问量超过1万时，系统响应时间则会超过2秒。
    </p>
    <p>
     负载测试： 在满足系统响应时间指标的前提下，该系统能够承受的最大访问量是1万。
    </p>
    <p>
     压力测试 ：继续增加系统的访问量，并观察系统的性能变化。例如，当系统访问量增加到2万时，发现系统响应时间延迟到5秒，当访问量增加到3万时，系统崩溃，⽆法做出响应。由此可以确定系统能达到的极限访问量是3万
    </p>
    <h4>
     <a id="_107">
     </a>
     并发测试
    </h4>
    <p>
     并发测试用于评估被测系统的某些操作同时发生时的性能表现.。
     <br/>
     例如，测试系统多个用户同时登录时的响应能力，或系统的某⼀功能被多个用户同时操作时的性能表现。
     <br/>
     例如，通过模拟多个用户同时访问某⼀条件数据，或模拟多个用户同时更新数据，可能发现系统的数据库访问错误、写⼊错误等。
    </p>
    <h3>
     <a id="_112">
     </a>
     常见性能测试指标
    </h3>
    <h4>
     <a id="_114">
     </a>
     响应时间
    </h4>
    <p>
     响应时间(Response Time)，是指用户从客户端发起一个请求开始，到客户端接收到从服务器端返回的结果，整个过程所耗费的时间。不包括前端页面的处理和渲染时间。直观上看，这个指标与人对软件性能的主观感受是非常一致的，因为它完整地记录了整个计算机系统处理请求的时间。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/f9e3f153383c417fb0992347378fcb6f.png"/>
    </p>
    <h4>
     <a id="_118">
     </a>
     并发数
    </h4>
    <p>
     并发数：某一时刻同时向服务器发送请求的用户数
    </p>
    <h4>
     <a id="_122">
     </a>
     吞吐量
    </h4>
    <p>
     吞吐量：指系统在单位时间内处理请求的数量。
     <br/>
     从业务角度看：业务数/小时、业务数/天、页面访问量/天、访问人数/天
     <br/>
     从网络角度看：字节数/天、字节数/小时
    </p>
    <p>
     <strong>
      从技术角度看
     </strong>
     <br/>
     吞吐量的表现形式有TPS和QPS：
     <br/>
     QPS（Query per Second）每秒查询数：即服务器每秒处理的请求数量。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/f16090c64a2746d0acea56a106e4b4c4.png">
      <br/>
      TPS（Transactions per Second）每秒事务数：服务器每秒处理的事务请求的数量。
      <br/>
      事务：即业务，一次操作可能会产生多个请求，这一次操作称为一次事务。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/93c64fe5b42a41cba4ec9b8e412b25ce.png">
       <br/>
       页面上的一次支付操作会有三个请求，这一次操作称为事务。QPS=TPS*n个请求，此处为45
      </img>
     </img>
    </p>
    <h4>
     <a id="_137">
     </a>
     点击数和错误率
    </h4>
    <p>
     点击数：客户端向服务器发送请求时，所有的请求总数量。点击数不是指页面上的一次点击，包括页面资源元素（图片、链接、CSS、JS等）的请求总数量。
     <br/>
     点击量:一般指用户点击数量
    </p>
    <p>
     错误率：系统在负载情况下，失败业务的概率。错误率=（失败业务/总业务数）*100%
    </p>
    <ul>
     <li>
      绝大多数系统要求错误率接近于0
     </li>
     <li>
      错误率是性能测试的指标，是由于高负载导致的错误，而不是代码BUG导致的。
     </li>
    </ul>
    <h4>
     <a id="_146">
     </a>
     资源使用率
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ccfa03fd71034ec8a9e6675720a91622.png"/>
    </p>
    <h3>
     <a id="_150">
     </a>
     性能测试的流程
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/2a781d86f161470e954115ea4b43190a.png"/>
    </p>
    <ol>
     <li>
      <p>
       需求分析：熟悉系统的业务功能和技术架构。了解系统的用户需求和预期负载，明确性能测试的目标和内容。
      </p>
     </li>
     <li>
      <p>
       制定测试计划：制定详细的测试计划，包括测试用例、测试数据和测试时间表。明确每个测试场景的输入、操作步骤和预期输出。
       <strong>
        关于测试计划的格式，可自行查阅资料
       </strong>
       <br/>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/71425b57001244a39f7e0b1032d521f4.png"/>
      </p>
     </li>
     <li>
      <p>
       测试用例设计，
       <strong>
        关于测试用例的格式，可自行查阅资料
       </strong>
       <br/>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/2e3311bc1e3c4d91b695f51b55c65b14.png"/>
      </p>
     </li>
     <li>
      <p>
       搭建测试环境：搭建与生产环境相似的测试环境，包括服务器、数据库、缓存等组件的设置和配置。确保测试环境能够支持预期的负载和数据量。
      </p>
     </li>
     <li>
      <p>
       编写测试脚本、执行性能测试脚本
      </p>
     </li>
     <li>
      <p>
       收集性能数据：收集性能测试期间产生的数据，包括日志、性能指标记录等，用于后续的分析和评估。
      </p>
     </li>
     <li>
      <p>
       分析测试结果：根据收集到的性能数据，进行性能分析，找出性能瓶颈、潜在问题以及系统的弱点，并提出优化建议。
      </p>
     </li>
     <li>
      <p>
       撰写测试报告：总结性能测试的结果和分析，撰写测试报告，包括测试过程、测试结果、性能问题和优化建议等内容。
       <strong>
        关于测试报告的格式，可自行查阅资料
       </strong>
      </p>
     </li>
     <li>
      <p>
       优化和改进：根据测试报告中的性能问题和优化建议，对系统进行优化和改进，如代码优化、数据库调优、服务器配置调整等。
      </p>
     </li>
     <li>
      <p>
       再次测试：在优化和改进后，再次进行性能测试，检验优化效果，确保系统在实际负载下的性能能够满足要求。
      </p>
     </li>
    </ol>
    <h2>
     <a id="Jmeter_167">
     </a>
     Jmeter模拟性能测试
    </h2>
    <p>
     Jmeter的基本操作请看另外一篇博客，或自行查阅自行、查看Jmeter自带文档。除另外一篇博客外，在此再介绍一些Jmeter组件用于性能测试。
    </p>
    <h3>
     <a id="Jmeter_169">
     </a>
     Jmeter组件
    </h3>
    <h4>
     <a id="_171">
     </a>
     事务控制器
    </h4>
    <p>
     作用：将多个请求汇总成一个事务，易于查看整个事务的性能指标。
     <br/>
     位置：线程组-&gt;右键-&gt;添加-&gt;逻辑控制器-&gt;事务控制器
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/3ef34ddc46e140c0b9e63a526756e151.png"/>
     <br/>
     如果，电商网站整个下单业务包括登录、加入购物车、结算、下订单的操作，如果我们想收集整个业务的性能指标，就需要用到事务控制器。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/97b1470af38343cb8b023963ba06aeda.png"/>
    </p>
    <h4>
     <a id="_179">
     </a>
     聚合报告
    </h4>
    <p>
     作用：收集性能测试中， 系统的各项性能指标。如：响应时间、吞吐量、错误率等。
     <br/>
     位置：测试计划-&gt;右键-&gt;添加-&gt;监听器-&gt;聚合报告
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/d2fffbb5bbaa41a2be0dc716045a0245.png"/>
    </p>
    <h4>
     <a id="PerfMon_183">
     </a>
     第三方插件PerfMon（仅作参考）
    </h4>
    <p>
     作用：监控系统的资源指标（CPU、内存、磁盘IO、网络）
     <br/>
     操作步骤：
    </p>
    <ol>
     <li>
      <p>
       下载Jmeter插件管理工具jar包 jmeter-plugins.org
       <br/>
       <a href="https://jmeter-plugins.org/install/Install/" rel="nofollow">
        jmeter-plugins.org
       </a>
       <br/>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a0e5d0c0794b4b2a9b2e6620003a4fc3.png"/>
      </p>
     </li>
     <li>
      <p>
       将下载好的插件管理工具jar包 jmeter-plugins.org 放到Jmeter文件夹lib\ext目录下
      </p>
     </li>
     <li>
      <p>
       重启Jmeter，在选项下面可以看到插件管理器
       <br/>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a8387fc32f5448fa881708d1ef4661c4.png"/>
      </p>
     </li>
     <li>
      <p>
       打开 Plugins Manager
      </p>
     </li>
     <li>
      <p>
       选择Available Plugins
      </p>
     </li>
     <li>
      <p>
       搜索 PerfMon插件 勾选
      </p>
     </li>
     <li>
      <p>
       点击右下角安装按钮
      </p>
     </li>
    </ol>
    <p>
     PerfMon组件添加方法：线程组-&gt;监听器-&gt;jp@gc - PerfMon Metrics Collector
     <br/>
     PerfMon组件工作原理：需要在服务器端安装资源监听服务程序，启动并收集资源指标，使用PerfMon汇总资源指标并展示。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/b691279a1ce84178b27606200d5a89e5.png"/>
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/1d8a3b2985504ded949c144b9a571579.png"/>
     <br/>
     随后，我们要对jp@gc - PerfMon Metrics Collector进行配置，才能接收到服务端的资源指标信息。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/4d0391c98dd942b7a8fa6f06e665be5c.png"/>
    </p>
    <h3>
     <a id="_205">
     </a>
     压力测试场景（仅供参考）
    </h3>
    <p>
     假设有一个电商网站的性能需求如下（具体性能指标仅作演示），峰值场景（高负载场景）相比于日常场景下的下订单业务，对TPS和响应时间要求不同，我们要对峰值场景（高负载场景）进行性能测试，该如何进行呢？
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/e356f581c417446c9bfac577eb18af37.png"/>
     <br/>
     按照压力测试策略的设计方案，我们进行如下测试步骤。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a0aada4288b442bdb1308e402a3fa436.png"/>
     <br/>
     Jmeter脚本大致设计
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/39cbc2deca8e4f16bbce6cceee899594.png"/>
    </p>
    <ol>
     <li>
      添加线程组
     </li>
     <li>
      添加事务控制器
     </li>
     <li>
      在事务控制器下添加HTTP请求-登录，我们要测试整个下单业务的性能指标，所以使用事务控制器
     </li>
     <li>
      在事务控制器下添加HTTP请求-添加购物车
     </li>
     <li>
      在事务控制器下添加HTTP请求-结算
     </li>
     <li>
      在事务控制器下添加HTTP请求-下订单
     </li>
     <li>
      HTTP请求-下订单下添加常数吞吐量定时器，由于高负载场景下的吞吐量为50，在逐步增加负载的情况下，最后将吞吐量设为50。
     </li>
     <li>
      添加聚合报告和jp@gc - PerfMon Metrics Collector
     </li>
     <li>
      执行，通过聚合报告和PerfMon查看性能指标
     </li>
    </ol>
    <h3>
     <a id="_222">
     </a>
     并发测试场景
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/37a0d02509064dbebf4c2b8e69fff359.png"/>

     Jmeter脚本大致设计
    </p>
    <ol>
     <li>
      添加线程组，线程数设为50
     </li>
     <li>
      添加HTTP请求-下订单， 秒杀活动肯定是已经提前完成了登录、物品加入购物车等操作。
     </li>
     <li>
      测试数据提前准备
     </li>
     <li>
      HTTP请求-下订单下添加同步定时器，用于模拟50个用户同时发送请求。
     </li>
     <li>
      添加聚合报告和jp@gc - PerfMon Metrics Collector
     </li>
     <li>
      执行，通过聚合报告和PerfMon查看性能指标
