    <h2>
     <a id="_1">
     </a>
     概述
    </h2>
    <p>
     面向服务的架构（service-oriented Architecture，SOA），就是如何“基于服务”或“基于一堆服务”来实现的业务架构。
    </p>
    <p>
     面向服务是在面向对象的基础上发展而来的，但服务和对象有所不同：
    </p>
    <ul>
     <li>
      对象主要是面向系统的，而服务是面向业务的。
     </li>
     <li>
      对象的粒度级别主要集中在类级，而服务是粗粒度的。
     </li>
     <li>
      类是以函数调用为基础来交互的，而服务是以网络请求为基础来交互的。
     </li>
    </ul>
    <h3>
     <a id="SOA_10">
     </a>
     SOA
    </h3>
    <p>
     关于服务，一些常见的设计原则有：
     <strong>
      明确定义的接口
     </strong>
     、
     <strong>
      自包含
     </strong>
     和
     <strong>
      模块化
     </strong>
     、
     <strong>
      粗粒度
     </strong>
     、
     <strong>
      松耦合
     </strong>
     、
     <strong>
      互操作性
     </strong>
     。
    </p>
    <p>
     SOA紧密相关的基数主要有
     <strong>
      UDDI
     </strong>
     、
     <strong>
      WSDL
     </strong>
     、
     <strong>
      SOAP
     </strong>
     ，而这些技术都是以XML为基础而发展起来的。
    </p>
    <ul>
     <li>
      UDDI：一种用于描述、发现、集成Web Service的技术。
      <ul>
       <li>
        通过UDDI，企业可以根据自己的需要动态查找并使用Web服务。
       </li>
       <li>
        也可以将自己的Web服务动态地发布到UDDI注册中心，供其他用户使用。
       </li>
      </ul>
     </li>
     <li>
      WSDL：Web服务描述语言，是为描述Web服务发布的XML格式。
     </li>
     <li>
      SOAP：简单对象访问协议是交换数据的一种协议规范，是一种轻量的、简单的、基于XML的协议，它被设计成在Web上交互结构化的和固化的信息。
     </li>
    </ul>

    <h2>
     <a id="_24">
     </a>
     实现方式
    </h2>
    <h3>
     <a id="_25">
     </a>
     服务注册表模式
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/3272e593c5e048d29e20e20e6483534e.png"/>
    </p>
    <h4>
     <a id="SOA_VS_SOAP_28">
     </a>
     SOA VS SOAP
    </h4>
    <p>
     SOA指的是架构方法及流程，WebService（服务提供者）、UDDI（注册中心）、业务方（调用方、消费者）三者的地位、作用以及需要遵从的执行流程：
    </p>
    <ul>
     <li>
      WebService将自己的WSDL注册给UDDI。
     </li>
     <li>
      业务方先从UDDI获取到WSDL，进而才能访问WebService。
     </li>
    </ul>
    <p>
     SOAP指的是SOA的三个组件互相访问时遵从的网络协议，即：用HTTP请求承载，XML为组织格式，来传递输入输出的有约束的数据。
    </p>
    <h3>
     <a id="ESB_36">
     </a>
     企业服务总线模式（ESB）
    </h3>
    <p>
     Enterprise Service Bus，企业服务总线架构模式在于整合，可能是公司内部各团队的服务，也可能是公司之间的不同服务，组成一个整体，来向外提供服务。
    </p>
    <p>
     业务方只跟业务总线交互：
    </p>
    <ul>
     <li>
      业务方将请求提交给业务总线。
     </li>
     <li>
      由业务总线转发请求给相应的后端服务。
     </li>
     <li>
      业务总线收到回复后再转发给业务方。
     </li>
    </ul>
    <p>
     即，业务总线也兼具“代理”的功能。业务方只需要知晓业务总线的地址，无需耦合所有服务。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/399e7ecc110148daaaac014a25a36241.png"/>
    </p>
    <h4>
     <a id="_48">
     </a>
     主要作用
    </h4>
    <ol>
     <li>
      SOA的一种实现方式，ESB在面向服务的架构中起到的是总线作用，将各种服务进行连接与整合。
     </li>
     <li>
      描述服务的元数据和服务注册管理。
     </li>
     <li>
      在服务请求者和提供者之间传递数据，以及对这些数据进行转化的能力，并支持由实践中总结出来的一些模式如同步模式、异步模式等。
     </li>
     <li>
      发现、路由、匹配和选择的能力，以支持服务之间的动态交互，解耦服务请求者和服务提供者。高级一些的能力，包括对安全的支持、服务质量保证、可管理性和负载平衡等。
     </li>
    </ol>
