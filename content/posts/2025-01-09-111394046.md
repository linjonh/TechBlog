    <h2>
     <a id="_22">
     </a>
     一.实验背景
    </h2>
    <p>
     对病毒进行逆向分析，可以彻底弄清楚病毒的行为，从而采取更有效的针对手段。为了节省篇幅，在这里我不打算将“熊猫烧香”进行彻底的分析，只会讲解一些比较重要的部分，大家只要掌握了这些思想，那么就可以处理很多的恶意程序了。
    </p>
    <p>
     这里主要使用的工具包括：
    </p>
    <ul>
     <li>
      <strong>
       PEiD：病毒加壳、脱壳基础性分析
      </strong>
     </li>
     <li>
      <strong>
       IDA Pro：静态分析
      </strong>
     </li>
     <li>
      <strong>
       OllyDbg：动态分析
      </strong>
     </li>
    </ul>
    <p>
     实验文件：
    </p>
    <ul>
     <li>
      <strong>
       setup.exe：熊猫烧香病毒
      </strong>
     </li>
    </ul>
    <p>
     基本流程：
    </p>
    <ul>
     <li>
      利用查壳工具检查病毒是否带壳
     </li>
     <li>
      利用OD动态分析病毒
     </li>
     <li>
      利用IDA静态分析病毒
     </li>
    </ul>
    <p>
     <mark>
      注意：由于OD工具会将程序运行起来，所以我们在进行恶意代码分析时尽量在搭建好的虚拟机中操作。如果病毒传播性较强如WannaCry，建议断开网络和断开共享分析。
     </mark>
    </p>
    <p>
     <img alt="在这里插入图片描述" height="300" src="https://i-blog.csdnimg.cn/blog_migrate/c0bf30826b957f5ccbb7c72ef2a85010.png#pic_center" width="300"/>
    </p>
    <p>
     <strong>
      实验思路：
     </strong>
    </p>
    <ul>
     <li>
      1.利用OD分析每一个CALL的功能
     </li>
     <li>
      2.使用IDA Pro在宏观上把握程序的功能并及时对函数进行重命名操作
     </li>
    </ul>
    <p>
     <strong>
      上篇回顾：
     </strong>
    </p>
    <ul>
     <li>
      <mark>
       <strong>
        0x0040CB7E call sub_403C98
       </strong>
      </mark>
      <br/>
      – 重命名为：AllocStackAndCopyString
      <br/>
      – sub_403D08：分配内存空间
      <br/>
      – sub_402650：字符串拷贝功能
     </li>
     <li>
      <mark>
       <strong>
        0x0040CB9F call sub_00405360
       </strong>
      </mark>
      <br/>
      – 重命名为：DecodeString
      <br/>
      – 0x004053CC：循环入口点
      <br/>
      – 0x004053E8：获取“xboy”解密字符
      <br/>
      – 0x00405408：完成异或解密操作
     </li>
     <li>
      <mark>
       <strong>
        0x0040CBAC call sub_404018
       </strong>
      </mark>
      <br/>
      – 重命名为：CMPString
      <br/>
      – 0x00404041：循环入口点
      <br/>
      – 功能：字符串比较“武汉男生感染下载者”
     </li>
     <li>
      <mark>
       <strong>
        0x0040CBCC loc_40CBBC
       </strong>
      </mark>
      <br/>
      – 功能：字符串解密和比较操作，解密字符“whboy”
      <br/>
      – DecodeString
      <br/>
      – CMPString
     </li>
    </ul>

    <h2>
     <a id="sub_408024_71">
     </a>
     二.sub_408024核心函数分析
    </h2>
    <p>
     上一篇文章我们详细介绍了熊猫烧香病毒起始阶段的初始化操作，这篇文章将进入该病毒的核心功能函数，进行相关的逆向分析。Let’s Go！！！
    </p>
    <h3>
     <a id="1sub_40277C_74">
     </a>
     1.sub_40277C子函数
    </h3>
    <p>
     <strong>
      第一步，打开IDA Pro载入我们的病毒样本。
     </strong>
    </p>
    <p>
     <img alt="在这里插入图片描述" height="380" src="https://i-blog.csdnimg.cn/blog_migrate/8fa1425b8ae7daec52816dcb72e757b7.png#pic_center" width="650"/>
    </p>
    <p>
     <strong>
      第二步，定位到loc_40CBE6位置。
     </strong>
     <br/>
     这里看到了三个call函数，它们又是什么功能呢？这三个call是熊猫烧香病毒最重要的功能，也是我们接下来要深入逆向分析的内容。
    </p>
    <ul>
     <li>
      <mark>
       sub_408024
      </mark>
     </li>
     <li>
      <mark>
       sub_40CA5C
      </mark>
     </li>
     <li>
      <mark>
       sub_40C97C
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="220" src="https://i-blog.csdnimg.cn/blog_migrate/ed453359dd89442cc78f5a88238cafef.png#pic_center" width="500"/>
    </p>
    <p>
     <strong>
      第三步，查看函数sub_408024内容。
     </strong>
     <br/>
     在这个函数的最开始位置我们看到了将84h赋值给ecx，84h表示循环的次数。循环的主体是下面两个push，其中每一个push能开辟8个字节（32位）的空间，两个字节能获得16个字节的空间。这样的空间一共申请了ecx（84h）次。
    </p>
    <ul>
     <li>
      <mark>
       mov ecx, 84h
      </mark>
     </li>
     <li>
      <mark>
       push 0
      </mark>
     </li>
     <li>
      <mark>
       push 0
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="220" src="https://i-blog.csdnimg.cn/blog_migrate/27596d233e1bd2d58f45d2bebf0581ae.png#pic_center" width="500"/>
    </p>
    <hr/>
    <p>
     <strong>
      第四步，我们主要分析call函数，往下看到第一个函数sub_40277c。
     </strong>
    </p>
    <p>
     <img alt="在这里插入图片描述" height="260" src="https://i-blog.csdnimg.cn/blog_migrate/7c0bf233ddc8ad7efea29cb91947e0f9.png#pic_center" width="500"/>
    </p>
    <p>
     <strong>
      第五步，接着用OD载入exe程序，进行动态的分析，直接跳转到该地址。
     </strong>
    </p>
    <ul>
     <li>
      <mark>
       0x0040804D call sub_40277C
      </mark>
     </li>
    </ul>
    <p>
     基本流程为右键“转到”-&gt;“表达式”-&gt;输入“0040804D”，按下F2增加断点。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/287cedc9ba96f149367023b0b92ece98.png#pic_center" width="650"/>
    </p>
    <p>
     按F9直接执行到该位置。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="300" src="https://i-blog.csdnimg.cn/blog_migrate/092611baa3b5bb92c93442dc6cc1b812.png#pic_center" width="550"/>
    </p>
    <p>
     该函数我们可以在IDA中先双击查看，会发现该函数首先调用了GetModuleFileNameA。
    </p>
    <ul>
     <li>
      <mark>
       0x004027A0 call GetModuleFileNameA
      </mark>
     </li>
     <li>
      <mark>
       该函数功能主要是获取当前进程已加载模块文件的完整路径
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="300" src="https://i-blog.csdnimg.cn/blog_migrate/e85e079237b19a41a77dff0d29814674.png#pic_center" width="500"/>
    </p>
    <hr/>
    <p>
     <strong>
      第六步，回到OD中sub_40277C函数，按下F7单步进入函数。
     </strong>
     <br/>
     接着进入函数后按F8单步步过，可以看到GetModuleFileNameA右下角的内容，在这个PathBuffer里，会保存即将获取的路径信息。
    </p>
    <ul>
     <li>
      <mark>
       0x004027A0 call 00401100
      </mark>
     </li>
     <li>
      <mark>
       PathBuffer
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/21fbc912b1d4ca12370e319d84902547.png#pic_center" width="650"/>
    </p>
    <p>
     接着选中PathBuffer值，右键点击“数据窗口中跟随”。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="420" src="https://i-blog.csdnimg.cn/blog_migrate/cb81cf338fdb8b403ad994ff5d3b5370.png#pic_center" width="500"/>
    </p>
    <p>
     <font color="red">
      <strong>
       注意：按下F8，大家可以在数据窗口中留意该值的前后变化。发现它成功获取了该样本的本地路径。
      </strong>
     </font>
    </p>
    <ul>
     <li>
      <mark>
       C:\Users\14551\Desktop\setup.exe
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="320" src="https://i-blog.csdnimg.cn/blog_migrate/6d7b460048d913cbb431761ad37a9a32.png#pic_center" width="500"/>
    </p>
    <hr/>
    <p>
     <strong>
      第七步，我们打开IDA给sub_40277C函数重命名。
     </strong>
    </p>
    <blockquote>
     <p>
      <strong>
       小技巧
      </strong>
      <br/>
      IDA中双击会进入对应的函数，那么如何返回上一层呢？按下ESC即可返回，如下图所示，从GetModuleFileNameA中按ESC返回sub_40277C位置。而按F5键会逆向出C语言程序。
     </p>
    </blockquote>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/f9e43b19adf0796f2e6edb94caf1aa4f.png#pic_center" width="650"/>
    </p>
    <p>
     然后选中函数重命名，快捷键为N。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/9891bd9de0f6f1f5ee684a6ff979d169.png#pic_center" width="650"/>
    </p>
    <p>
     <font color="red">
      <strong>
       总结，重命名及对应功能如下：
      </strong>
     </font>
    </p>
    <ul>
     <li>
      <mark>
       sub_40277C -&gt; GetFilePathAndName
      </mark>
     </li>
     <li>
      <mark>
       功能：获取文件的完整路径及文件名称
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="260" src="https://i-blog.csdnimg.cn/blog_migrate/35b324e504a4502b73091ef3d6419d6b.png#pic_center" width="250"/>
    </p>

     2.sub_405684子函数
    </h3>
    <p>

    </p>
    <ul>
     <li>
      <mark>
       0x0040805E call sub_405684
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="260" src="https://i-blog.csdnimg.cn/blog_migrate/9ef3ec0246ab5d861af9a7c93056b480.png#pic_center" width="450"/>
    </p>
    <p>
     <strong>
      第一步，继续在OD中进行跟进。
     </strong>
     <br/>
     首先我们表达式跳转到“00408052”位置，然后按下F2增加断点，按下F9执行过来。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="260" src="https://i-blog.csdnimg.cn/blog_migrate/2d7fd846cf45a7a3188d6cc4781b767e.png#pic_center" width="550"/>

    <p>
     <strong>
      第二步，在数据窗口中跟随EAX。
     </strong>
     <br/>
     按下F7单步执行，在寄存器中选中EAX，右键“数据窗口中跟随”。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/c233fdf447ea4e9f7ba93348cf97aff9.png#pic_center" width="650"/>

     <img alt="在这里插入图片描述" height="260" src="https://i-blog.csdnimg.cn/blog_migrate/65f7c73e47fccf965cbfa0f5aa3c45db.png#pic_center" width="550"/>
    </p>
    <p>
     <strong>
      第一步，我们在IDA中大致看下函数的内容。

     我们可以看到这个函数非常的长，且也调用了很多其它的函数，那么看上去很难分析清楚这个函数到底是做什么的。这里我们先不管其具体的实现细节，只看看该函数执行完后，这些参数或寄存器位置发生了什么变化，主要关注该函数执行前后，程序使用了哪些寄存器。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="260" src="https://i-blog.csdnimg.cn/blog_migrate/b5e4d3246513bcab8d74f2e13d0473da.png#pic_center" width="550"/>

     <strong>
      第二步，打开OD跳转到40811D位置，按下F2增加断点，再按F9执行过来。

    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/45269b29b6e9583c718dc4dc092dafc3.png#pic_center" width="650"/>

    <p>
     我们现在按F8来步过这个call函数，再观察数据窗口中存在什么变化。可以看到执行完call函数后，这个函数用了很大一片区域写入了非常多看似无意义的字符，结合右边的ASCII码，可以将这些内容理解为暴力P解的字典。
    </p>
    <ul>
     <li>
      <mark>
       <strong>
        病毒的编写者企图利用暴力P解的方式来攻破计算机中某些验证机制
       </strong>
      </mark>
     </li>

     <img alt="在这里插入图片描述" height="380" src="https://i-blog.csdnimg.cn/blog_migrate/20127fe9d5e8fda81d1a1bc933fe73e0.png#pic_center" width="450"/>

      </mark>
     </li>

    <h3>
     <a id="7sub_403C44_439">
     </a>
     7.sub_403C44子函数
    </h3>
    <p>
     继续在OD中分析sub_403C44子函数。
    </p>
    <p>
     <strong>
      第一步，查看[EBP-8]地址对应的值。
     </strong>
     <br/>
     在数据窗口中，可以看到eax的值为ebp+var_8的地址，通过OD可以知道，这个地址中保存的是0，可以理解为没有数据。
    </p>
    <ul>
     <li>
      <mark>
       00 00 00 00
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/f3704545e45f0f77dce195afd0b79ced.png#pic_center" width="550"/>

    <p>
     <strong>
      第二步，我们按下F7进入这个call，查看具体内容。
     </strong>
    </p>
    <ul>
     <li>
      <mark>
       0x0040812E call 00403C44
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/f1355d2692746b089dcffdb5539ee478.png#pic_center" width="650"/>

     可以看到，它首先是将EAX的内容赋值给EDX，我们按下F8可以看到EDX也变成了0值。
    </p>
    <ul>
     <li>
      <mark>
       mov edx, dword ptr ds:[eax]
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/9ed143d4de4606ee871dc9e8f708a2a5.png#pic_center" width="450"/>
    </p>
    <p>
     之后的test运算，使得ZF变为1，满足跳转条件，直接跳转至retn，那么本段函数也就结束了。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="300" src="https://i-blog.csdnimg.cn/blog_migrate/79aa0e2dbb2f17811b428cc043ad6880.png#pic_center" width="550"/>

     刚才我们通过test edx, edx发现ZeroFlag标志位变成了1。这个标志位变为1，说明这个值是一个0，这段函数似乎并没有实现什么特别清晰具体的功能，那么不妨将这个操作理解为某种标志的设置。

    <h3>
     <a id="8sub_403ECC_478">
     </a>
     8.sub_403ECC子函数
    </h3>
    <p>
     回到OD，我们分析sub_403ECC子函数。
    </p>
    <p>
     <strong>
      第一步，查看并分析[EBP-4]保存的内容。
     </strong>
    </p>
    <ul>
     <li>
      <mark>
       mov eax, dword ptr ss:[ebp-0x4]
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="260" src="https://i-blog.csdnimg.cn/blog_migrate/9c8f45747a91f7e4cd92716fd5b98c42.png#pic_center" width="550"/>
    </p>
    <p>
     在数据窗口中跟随，可以看到“MZP”，这里大致是保存了一个PE文件，事实上就是将这个PE文件的首地址赋值给这个EAX，接着看看这个call。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/83ec40eff6599a3ffa5242fe141b5421.png#pic_center" width="550"/>
    </p>
    <p>
     <strong>
      第二步，分析call 403ECC函数。
     </strong>
     <br/>
     按下F7进入这个call函数，首先调用“TEST EAX, EAX”验证PE文件是否存在。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="260" src="https://i-blog.csdnimg.cn/blog_migrate/d0c9f43ce0607d89bfb402a23cb9a49a.png#pic_center" width="550"/>
    </p>
    <p>
     按下F8继续，这里是跳转不成立，然后出现在EAX-4位置，我们在数据窗口中跟随内容。
    </p>
    <ul>
     <li>
      <mark>
       mov eax, dword ptr ds:[eax-0x4]
      </mark>
     </li>
    </ul>
    <blockquote>
     <p>
      <strong>
       小技巧
      </strong>
      <br/>
      这里需要特别强调的是，由于本病毒程序是使用Delphi编写的，因此字符串的首地址减去4后，所取出的4个字节的内容就是此字符串的长度。
     </p>
    </blockquote>
    <p>
     换句话说，这段代码中的[eax-4]就是eax所指向的文件长度，字符串长度就是EC00，这条语句就是将EC00保存在寄存器里面。于是可以将sub_403ECC重命名。
    </p>
    <ul>
     <li>
      <mark>
       字符串长度：EC00
      </mark>
     </li>
     <li>
      <mark>
       现在EAX保存的就是字符串的长度，即PE文件的长度
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/fcb6e4f03440636eb702a3181f2e9bc6.png#pic_center" width="550"/>
    </p>
    <p>
     <img alt="在这里插入图片描述" height="300" src="https://i-blog.csdnimg.cn/blog_migrate/6f1aa53e2725f892f1d17515c221a581.png#pic_center" width="450"/>
    </p>
    <p>
     <font color="red">
      <strong>
       总结，重命名及对应功能如下：
      </strong>
     </font>
    </p>
    <ul>
     <li>
      <mark>
       sub_403ECC -&gt; GetFileLen
      </mark>
     </li>
     <li>
      <mark>
       功能：获取PE文件的长度
      </mark>
     </li>
    </ul>
    <p>
     之前这个文件的长度是保存在EAX里面，这里又将文件长度赋值给EBX。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="320" src="https://i-blog.csdnimg.cn/blog_migrate/9cbf09a2a892e3a058190d57b527877e.png#pic_center" width="550"/>
    </p>
    <p>
     接下来，在获取文件长度后，程序会跳转到loc_408163处执行。首先它会验证EBX是否为零（文件长度是否为0），正常来说文件长度是不为0的，也就是这个文件是真实存在的，所以接下来的跳转不成立，会继续往下执行。
    </p>
    <ul>
     <li>
      <mark>
       test ebx, ebx
      </mark>
     </li>
     <li>
      <mark>
       jle short loc_408171
      </mark>
     </li>
     <li>
      <mark>
       mov eax, [ebp+var_4]
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="260" src="https://i-blog.csdnimg.cn/blog_migrate/046c185a515391774cac536457b72d93.png#pic_center" width="550"/>
    </p>
    <p>
     接着我们尝试用OD进行分析，跳转到0x00408163的位置。按下F2增加断点，再按F9让它执行过来。
    </p>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/9021064521d701fe7840372f54214353.png#pic_center" width="550"/>
    </p>
    <p>
     按F8跳转不成立继续执行，可以看到将[EBP-4]赋值给EAX，事实上就是让EAX重新指向这个PE文件的起始位置。
    </p>
    <ul>
     <li>
      <mark>
       mov eax, dword ptr ss:[ebp-0x4]
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/da10e37e7d279c055fca726a9efc115d.png#pic_center" width="550"/>
    </p>
    <p>
     按下F8继续执行，这里是将 [EAX+EBX-1]，由于EAX指向的是文件起始地址，而EBX是文件的长度，减一是获取整个PE文件最后一个字符它是什么。事实上，CMP语句就是验证这个文件最后一个数据它是否为零，如果为0，则接下来的跳转不成立。
    </p>
    <ul>
     <li>
      <mark>
       cmp byte ptr [eax+ebx-0x1], 0x0
      </mark>
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" height="360" src="https://i-blog.csdnimg.cn/blog_migrate/c1aa79af991166deb5fb01cf028d69d0.png#pic_center" width="450"/>
    </p>
    <p>
     经过OD的动态分析发现，因为这里显示跳转未实现，故文件尾端的值为0，所以不执行跳转，继续向下执行。
    </p>

     <img alt="在这里插入图片描述" height="320" src="https://i-blog.csdnimg.cn/blog_migrate/3c2a97a039272766112d438cff6db453.png#pic_center" width="450"/>
