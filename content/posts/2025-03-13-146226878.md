     GC 频率和触发条件
    </h1>

    <p>
     在 Java 中，垃圾回收（GC）的
     <strong>
      频率
     </strong>
     和
     <strong>
      触发条件
     </strong>
     取决于
     <strong>
      GC算法、堆内存分配、对象生命周期
     </strong>
     以及
     <strong>
      JVM参数
     </strong>
     的配置。下面详细介绍这些影响因素：
    </p>
    <h2>
     <a id="1_GC__1">
     </a>
     1.
     <strong>
      GC 触发条件
     </strong>
    </h2>
    <p>
     GC 主要触发的情况如下：
    </p>
    <h3>
     <a id="1__GCMinor_GC__Young_GC_3">
     </a>
     <strong>
      (1) 年轻代 GC（Minor GC / Young GC）
     </strong>
    </h3>
    <p>
     <strong>
      触发条件
     </strong>
     ：
    </p>
    <ul>
     <li>
      <strong>
       Eden 区满了
      </strong>
      ：当新对象分配到 Eden 区，如果 Eden 区没有足够的空间分配新对象，就会触发 Minor GC。
     </li>
     <li>
      <strong>
       Survivor 空间不足
      </strong>
      ：当存活对象从 Eden 复制到 Survivor，但 Survivor 空间不够时，也可能导致 Minor GC。
     </li>
    </ul>
    <p>
     <strong>
      特点
     </strong>
     ：
    </p>
    <ul>
     <li>
      仅回收
      <strong>
       年轻代（Young Generation）
      </strong>
      ，不会影响老年代（Old Generation）。
     </li>
     <li>
      采用
      <strong>
       复制算法
      </strong>
      （如 Serial、Parallel、G1 的 YGC）。
     </li>
     <li>
      <strong>
       停顿时间短
      </strong>
      ，但回收频率较高。
     </li>
     <li>
      <strong>
       Minor GC 之后，存活对象可能晋升到老年代
      </strong>
      。
     </li>
    </ul>
    <h3>
     <a id="2__GCMajor_GC__Old_GC_13">
     </a>
     <strong>
      (2) 老年代 GC（Major GC / Old GC）
     </strong>
    </h3>
    <p>
     <strong>
      触发条件
     </strong>
     ：
    </p>
    <ul>
     <li>
      <strong>
       老年代空间不足
      </strong>
      ：当对象从 Survivor 晋升到老年代，或者大对象直接进入老年代，导致老年代空间不够时，会触发 Major GC。
     </li>
     <li>
      <strong>
       CMS GC 的 concurrent mode failure
      </strong>
      ：CMS GC 在并发回收过程中如果老年代空间不足，会触发 STW 的 Full GC。
     </li>
     <li>
      <strong>
       G1 GC 触发 Mixed GC
      </strong>
      ：G1 在一定条件下会触发回收老年代的 Mixed GC。
     </li>
    </ul>
    <p>
     <strong>
      特点
     </strong>
     ：
    </p>
    <ul>
     <li>
      主要清理
      <strong>
       老年代（Old Generation）
      </strong>
      ，回收存活时间较长的对象。
     </li>
     <li>
      <strong>
       相比 Minor GC，Major GC 的停顿时间更长
      </strong>
      ，但一般回收频率较低。
     </li>
     <li>
      某些 GC（如 CMS）不会 STW，而是并发执行（Concurrent Mark-Sweep）。
     </li>
    </ul>
    <h3>
     <a id="3_Full_GC_23">
     </a>
     <strong>
      (3) Full GC
     </strong>
    </h3>
    <p>
     <strong>
      触发条件
     </strong>
     ：
    </p>
    <ul>
     <li>
      <strong>
       显式调用
       <code>
        System.gc()
       </code>
      </strong>
      （不推荐，因为 JVM 可能会忽略）。
     </li>
     <li>
      <strong>
       老年代空间不足
      </strong>
      ：当老年代没有足够空间存放新对象时，Major GC 可能变成 Full GC。
     </li>
     <li>
      <strong>
       Metaspace/元空间溢出
      </strong>
      （如类加载过多，导致
      <code>
       java.lang.OutOfMemoryError: Metaspace
      </code>
      ）。
     </li>
     <li>
      <strong>
       CMS GC 失败
      </strong>
      ：如果 CMS GC 过程中发生
      <code>
       concurrent mode failure
      </code>
      ，会触发 Full GC。
     </li>
     <li>
      <strong>
       G1 GC 触发 Full GC
      </strong>
      ：当 G1 发现回收无法跟上对象分配速度时，会进行 STW 的 Full GC。
     </li>
    </ul>
    <p>
     <strong>
      特点
     </strong>
     ：
    </p>
    <ul>
     <li>
      <strong>
       回收整个堆（包括年轻代 + 老年代 + 元空间）
      </strong>
      。
     </li>
     <li>
      <strong>
       停顿时间长
      </strong>
      ，影响系统吞吐量和响应时间。
     </li>
     <li>
      <strong>
       一般不希望频繁发生 Full GC
      </strong>
      ，需要调优。
     </li>
    </ul>
    <h2>
     <a id="2_GC__35">
     </a>
     2.
     <strong>
      GC 频率的影响因素
     </strong>
    </h2>
    <p>
     GC 的触发频率取决于以下几个因素：
    </p>
    <h3>
     <a id="1__37">
     </a>
     <strong>
      (1) 对象分配速率
     </strong>
    </h3>
    <ul>
     <li>
      <strong>
       短生命周期对象多（临时变量、业务请求数据）
      </strong>
      →
      <strong>
       Minor GC 频繁
      </strong>
      。
     </li>
     <li>
      <strong>
       大量大对象（如
       <code>
        byte[]
       </code>
       ）
      </strong>
      → 可能直接进入老年代，
      <strong>
       加速 Major/Full GC
      </strong>
      。
     </li>
    </ul>
    <h3>
     <a id="2_GC__40">
     </a>
     <strong>
      (2) GC 算法
     </strong>
    </h3>
    <p>
     不同 GC 算法对 GC 频率的影响不同：
    </p>
    <ul>
     <li>
      <strong>
       Serial GC
      </strong>
      （单线程、适用于小内存） → GC 频率高，暂停时间长。
     </li>
     <li>
      <strong>
       Parallel GC
      </strong>
      （多线程 GC，吞吐量优先） → GC 频率较低，适用于高吞吐场景。
     </li>
     <li>
      <strong>
       G1 GC
      </strong>
      （区域化分代、回收预测） → 控制 GC 停顿时间，适用于大内存。
     </li>
     <li>
      <strong>
       ZGC、Shenandoah GC
      </strong>
      （低延迟 GC） → 减少 GC 影响，适用于大内存应用。
     </li>
    </ul>
    <h3>
     <a id="3_JVM__46">
     </a>
     <strong>
      (3) JVM 参数
     </strong>
    </h3>
    <p>
     JVM 相关参数直接影响 GC 频率：
    </p>
    <ul>
     <li>
      <code>
       -Xms
      </code>
      /
      <code>
       -Xmx
      </code>
      （堆内存大小）：
      <ul>
       <li>
        <strong>
         较小的堆内存
        </strong>
        → GC 触发更频繁。
       </li>
       <li>
        <strong>
         较大的堆内存
        </strong>
        → GC 触发较少，但可能增加 Full GC 停顿时间。
       </li>
      </ul>
     </li>
     <li>
      <code>
       -XX:NewRatio
      </code>
      （年轻代与老年代的比例）：
      <ul>
       <li>
        <strong>
         较大年轻代
        </strong>
        → Minor GC 频率降低，但可能加速老年代填满导致 Major GC 。
       </li>
       <li>
        <strong>
         较小年轻代
        </strong>
        → Minor GC 频率上升，但老年代增长较慢。
       </li>
      </ul>
     </li>
     <li>
      <code>
       -XX:SurvivorRatio
      </code>
      （Eden 和 Survivor 的比例）：
      <ul>
       <li>
        <strong>
         Survivor 较小
        </strong>
        → 对象更容易晋升老年代，加快 Major GC 触发。
       </li>
       <li>
        <strong>
         Survivor 较大
        </strong>
        → Minor GC 次数可能减少，但 Survivor 可能浪费空间。
       </li>
      </ul>
     </li>
     <li>
      <code>
       -XX:MaxTenuringThreshold
      </code>
      （晋升老年代的阈值）：
      <ul>
       <li>
        <strong>
         较低阈值
        </strong>
        → 对象更快晋升老年代，可能增加 Major GC 频率。
       </li>
       <li>
        <strong>
         较高阈值
        </strong>
        → 对象更长时间停留在 Survivor，可能增加 Minor GC 频率。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="4_GC__60">
     </a>
     <strong>
      (4) GC 负担
     </strong>
    </h3>
    <ul>
     <li>
      <strong>
       对象回收速率低
      </strong>
      → GC 触发频率更高。
     </li>
     <li>
      <strong>
       对象生命周期较长（长生命周期的缓存对象等）
      </strong>
      → 老年代更容易被填满，增加 Major/Full GC 频率。
     </li>
    </ul>
    <h2>
     <a id="3__GC__63">
     </a>
     3.
     <strong>
      如何优化 GC 频率
     </strong>
    </h2>
    <h3>
     <a id="1__64">
     </a>
     <strong>
      (1) 调整堆内存大小
     </strong>
    </h3>
    <ul>
     <li>
      增大
      <code>
       -Xmx
      </code>
      （最大堆内存），减少 GC 触发频率。
     </li>
     <li>
      增大
      <code>
       -Xms
      </code>
      （初始堆内存），减少动态扩展导致的 Full GC。
     </li>
    </ul>
    <h3>
     <a id="2__GC__67">
     </a>
     <strong>
      (2) 调整 GC 参数
     </strong>
    </h3>
    <ul>
     <li>
      <strong>
       增加年轻代大小
      </strong>
      （
      <code>
       -XX:NewRatio=1
      </code>
      ）：减少 Minor GC 触发频率，但可能影响老年代回收。
     </li>
     <li>
      <strong>
       调整 Survivor 空间
      </strong>
      （
      <code>
       -XX:SurvivorRatio=6
      </code>
      ）：减少对象晋升到老年代，降低 Major GC 频率。
     </li>
     <li>
      <strong>
       调高
       <code>
        -XX:MaxTenuringThreshold
       </code>
      </strong>
      （如 10），避免短生命周期对象过早进入老年代。
     </li>
    </ul>
    <h3>
     <a id="3__GC__71">
     </a>
     <strong>
      (3) 选择合适的 GC 算法
     </strong>
    </h3>
    <ul>
     <li>
      <strong>
       吞吐量优先
      </strong>
      （如并发任务多、批量计算） →
      <strong>
       Parallel GC
      </strong>
      （
      <code>
       -XX:+UseParallelGC
      </code>
      ）。
     </li>
     <li>
      <strong>
       低延迟场景
      </strong>
      （如微服务、高并发请求） →
      <strong>
       G1 GC
      </strong>
      （
      <code>
       -XX:+UseG1GC
      </code>
      ）。
     </li>
     <li>
      <strong>
       极低延迟需求（如金融系统）
      </strong>
      →
      <strong>
       ZGC/Shenandoah GC
      </strong>
      （
      <code>
       -XX:+UseZGC
      </code>
      或
      <code>
       -XX:+UseShenandoahGC
      </code>
      ）。
     </li>
    </ul>
    <h3>
     <a id="4__GC_75">
     </a>
     <strong>
      (4) 监控 GC
     </strong>
    </h3>
    <ul>
     <li>
      <strong>
       开启 GC 日志
      </strong>
      （
      <code>
       -Xlog:gc*
      </code>
      或
      <code>
       -XX:+PrintGCDetails
      </code>
      ）观察 GC 频率。
     </li>
     <li>
      <strong>
       使用
       <code>
        jstat
       </code>
       分析 GC
      </strong>
      ：
      <pre><code class="prism language-bash">jstat <span class="token parameter variable">-gcutil</span> <span class="token operator">&lt;</span>pid<span class="token operator">&gt;</span> <span class="token number">1000</span>
</code></pre>
     </li>
     <li>
      <strong>
       使用 VisualVM、Arthas 监控 GC 状态
      </strong>
      。
     </li>
    </ul>
    <h2>
     <a id="4__82">
     </a>
     4.
     <strong>
      总结
     </strong>
    </h2>
    <table>
     <thead>
      <tr>
       <th>
        GC 类型
       </th>
       <th>
        触发条件
       </th>
       <th>
        影响
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         Minor GC (Young GC)
        </strong>
       </td>
       <td>
        Eden 区满，Survivor 区空间不足
       </td>
       <td>
        频率高，暂停时间短，对业务影响小
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         Major GC (Old GC)
        </strong>
       </td>
       <td>
        老年代空间不足
       </td>
       <td>
        频率较低，暂停时间长，对吞吐量影响较大
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         Full GC
        </strong>
       </td>
       <td>
        老年代不足、Metaspace 溢出、CMS 失败等
       </td>
       <td>
        影响最大，应尽量避免
       </td>
      </tr>
     </tbody>
    </table>
    <p>
     <strong>
      优化 GC 频率的核心
     </strong>
     是合理分配堆内存、调整 GC 策略，并监控 GC 运行情况。
    </p>
