    <h2>
     <a id="11__3">
     </a>
     1.1 数据库系统概述
    </h2>
    <h3>
     <a id="111_4_4">
     </a>
     1.1.1 数据库的4个基本概念
    </h3>
    <ul>
     <li>
      数据（Data）:是数据库中存储的基本对象
     </li>
     <li>
      <strong>
       数据库（Database）是长期储存在计算机内、有组织的、可共享的大量数据的集合
      </strong>
      。
      <ul>
       <li>
        数据库的基本特征
        <ul>
         <li>
          数据按一定的数据模型组织、描述和储存
         </li>
         <li>
          可为各种用户共享
         </li>
         <li>
          冗余度较小
         </li>
         <li>
          数据独立性较高
         </li>
         <li>
          易扩展
         </li>
        </ul>
       </li>
      </ul>
     </li>
     <li>
      数据库管理系统（DBMS）:位于用户与操作系统之间的一层数据管理软件
     </li>
     <li>
      数据库系统（DBS）
      <ul>
       <li>
        数据库系统的构成：数据库、数据库管理系统（及其应用开发工具）、应用程序、数据库管理员
        <br/>
        <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/9c82c40ff692463e8b32287a59189ca0.png"/>
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="113__18">
     </a>
     1.1.3 数据库系统的特点
    </h3>
    <ul>
     <li>
      数据结构化
     </li>
     <li>
      数据的共享性高，冗余度低且易扩充
     </li>
     <li>
      数据独立性高
     </li>
     <li>
      数据由数据库管理系统统一管理和控制
     </li>
    </ul>
    <h2>
     <a id="12___23">
     </a>
     1.2 数据模型
    </h2>
    <p>
     通俗地讲数据模型就是现实世界的模拟
    </p>
    <h3>
     <a id="121___25">
     </a>
     1.2.1 两类数据模型
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/63cadef572374fedb5bdfbf1541b5c4a.png"/>
    </p>
    <ul>
     <li>
      <strong>
       概念模型
      </strong>
      ，也称信息模型，它是按用户的观点来对数据和信息建模，用于数据库设计。
     </li>
     <li>
      <strong>
       逻辑模型和物理模型
      </strong>
      <ul>
       <li>
        逻辑模型主要包括网状模型、层次模型、关系模型、面向对象数据模型、对象关系数据模型、半结构化数据模型等。按计算机系统的观点对数据建模，用于DBMS实现。
       </li>
       <li>
        物理模型是对数据最底层的抽象，描述
        <strong>
         数据在系统内部的表示方式和存取方法
        </strong>
        ，在磁盘或磁带上的存储方式和存取方法。
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="122___32">
     </a>
     1.2.2 概念模型
    </h3>
    <ul>
     <li>
      概念模型的一种表示方法：
      <strong>
       实体-联系方法
      </strong>
      （Entity-Relationship Approach）
      <br/>
      用E-R图来描述现实世界的概念模型，E-R方法也称为E-R模型
     </li>
    </ul>
    <blockquote>
     <p>
      （1）实体（Entity） :客观存在并可相互区别的事物称为实体。可以是具体的人、事、物或抽象的概念。
      <br/>
      （2）属性（Attribute） 实体所具有的某一特性称为属性。一个实体可以由若干个属性来刻画。
      <br/>
      （3）码（Key） 唯一标识实体的属性集称为码。
      <br/>
      （4）实体型（Entity Type） 用实体名及其属性名集合来抽象和刻画同类实体称为实体型
      <br/>
      （5）实体集（Entity Set） 同一类型实体的集合称为实体集
      <br/>
      （6）联系（Relationship） 现实世界中事物内部以及事物之间的联系在信息世界中反映为实体（型）内部的联系和实体（型）之间的联系。
     </p>
     <ul>
      <li>
       <strong>
        实体内部的联系
       </strong>
       通常是指组成实体的各属性之间的联系
      </li>
      <li>
       <strong>
        实体之间的联系
       </strong>
       通常是指不同实体集之间的联系
      </li>
      <li>
       实体之间的联系有
       <strong>
        一对一、一对多和多对多
       </strong>
       等多种类型
      </li>
     </ul>
    </blockquote>
    <h3>
     <a id="123___48">
     </a>
     1.2.3 数据模型的组成要素
    </h3>
    <ul>
     <li>
      <strong>
       数据结构（静态）
      </strong>
      ：描述数据库的组成对象，以及对象之间的联系
     </li>
     <li>
      <strong>
       数据操作 （动态）
      </strong>
      ：对数据库中各种对象（型）的实例（值）允许执行的操作的集合，包括操作及有关的操作规则
     </li>
     <li>
      <strong>
       数据的完整性约束条件
      </strong>
      :反映和规定必须遵守的基本的通用的完整性约束条件。
      <blockquote>
       <p>
        完整性规则：给定的数据模型中数据及其联系所具有的制约和依存规则
       </p>
      </blockquote>
     </li>
    </ul>
    <h3>
     <a id="124___56">
     </a>
     1.2.4 常用的数据模型
    </h3>
    <ul>
     <li>
      层次模型（Hierarchical Model）
     </li>
     <li>
      网状模型（Network Model）
     </li>
     <li>
      关系模型（Relational Model)）
     </li>
     <li>
      面向对象数据模型（Object Oriented Data Model）
     </li>
     <li>
      对象关系数据模型（Object Relational Data Model）
     </li>
     <li>
      半结构化数据模型（Semistruture Data Model）
     </li>
    </ul>
    <h3>
     <a id="125___63">
     </a>
     1.2.5 层次模型
    </h3>
    <h4>
     <a id="_65">
     </a>
     （一）层次模型的数据结构
    </h4>
    <p>
     层次模型用
     <strong>
      树形结构
     </strong>
     来表示各类
     <strong>
      实体以及实体间的联系
     </strong>
     <br/>
     <strong>
      层次模型的特点
     </strong>
     ：
    </p>
    <ul>
     <li>
      结点的双亲是唯一的
     </li>
     <li>
      只能直接处理一对多的实体联系
     </li>
     <li>
      每个记录类型可以定义一个排序字段，也称为码字段
     </li>
     <li>
      任何记录值只有按其路径查看时，才能显出它的全部意义
     </li>
     <li>
      没有一个子女记录值能够脱离双亲记录值而独立存在
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/33e38803cf034453b95235c743c01ba2.png"/>
     </li>
    </ul>
    <h4>
     <a id="_74">
     </a>
     （二）层次模型的数据操纵
    </h4>
    <p>
     层次模型的数据操纵
    </p>
    <ul>
     <li>
      查询
     </li>
     <li>
      插入
     </li>
     <li>
      删除
     </li>
     <li>
      更新
     </li>
    </ul>
    <h4>
     <a id="_80">
     </a>
     （三）层次模型的完整性约束
    </h4>
    <p>
     层次模型的完整性约束条件
    </p>
    <ul>
     <li>
      无相应的双亲结点值就不能插入子女结点值
     </li>
     <li>
      如果删除双亲结点值，则相应的子女结点值也被同时删除
     </li>
     <li>
      更新操作时，应更新所有相应记录，以保证数据的一致性
     </li>
    </ul>
    <h3>
     <a id="126___86">
     </a>
     1.2.6 网状模型
    </h3>
    <h4>
     <a id="__87">
     </a>
     (一) 网状模型的数据结构
    </h4>
    <p>
     满足下面两个条件的基本层次联系的集合：
    </p>
    <ol>
     <li>
      允许一个以上的结点无双亲；
     </li>
     <li>
      一个结点可以有多于一个的双亲。
     </li>
    </ol>
    <p>
     网状模型与层次模型的区别
    </p>
    <ul>
     <li>
      网状模型允许多个结点没有双亲结点
     </li>
     <li>
      网状模型允许结点有多个双亲结点
     </li>
     <li>
      网状模型允许两个结点之间有多种联系（复合联系）
     </li>
     <li>
      网状模型可以更直接地描述现实世界
     </li>
     <li>
      层次模型实际上是网状模型的一个特例
     </li>
    </ul>
    <h4>
     <a id="_101">
     </a>
     (二)网状模型的数据操纵与完整性约束
    </h4>
    <p>
     网状数据库系统（如DBTG）对数据操纵加了一些限制，提供了一定的完整性约束
    </p>
    <ul>
     <li>
      码：唯一标识记录的数据项的集合
     </li>
     <li>
      一个联系中双亲记录与子女记录之间是一对多联系
     </li>
     <li>
      支持双亲记录和子女记录之间某些约束条件
     </li>
    </ul>
    <h3>
     <a id="127___107">
     </a>
     1.2.7 关系模型
    </h3>
    <h4>
     <a id="__109">
     </a>
     （一） 关系模型的数据结构
    </h4>
    <p>
     在用户观点下，关系模型中数据的逻辑结构是
     <mark>
      一张二维表
     </mark>
     ，它由行和列组成
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/415a52da8bcc49f4bdfd09125c139178.png"/>
    </p>
    <blockquote>
     <p>
      关系（Relation）：一个关系对应通常说的一张表
      <br/>
      元组（Tuple）表中的
      <strong>
       一行
      </strong>
      即为一个元组
      <br/>
      属性（Attribute）表中的
      <strong>
       一列
      </strong>
      即为一个属性，给每一个属性起一个名称即属性名
      <br/>
      主码（Key）也称码键。表中的某个属性组，
      <strong>
       它可以唯一确定一个元组
      </strong>
      <br/>
      域（Domain）是一组具有相同数据类型的值的集合。
      <strong>
       属性的取值范围来自某个域
      </strong>
      。
      <br/>
      分量：元组中的一个属性值。
      <br/>
      <strong>
       关系模式
      </strong>
      ：对关系的描述，关系名（属性1，属性2，…，属性n）
      <font color="grey">
       学生（学号，姓名，年龄，性别，系名，年级）
      </font>
     </p>
    </blockquote>
    <ul>
     <li>
      <strong>
       关系必须是规范化的，满足一定的规范条件
      </strong>
      <br/>
      最基本的规范条件：
      <strong>
       关系的每一个分量必须是一个不可分的
      </strong>
      (数据项, 不允许表中还有表)
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/f33cad05245542cda7493ce3414da704.png"/>
     </li>
    </ul>
    <h4>
     <a id="_124">
     </a>
     （二）关系模型的数据操纵
    </h4>
    <p>
     数据操作是集合操作，操作对象和操作结果都是关系
    </p>
    <ul>
     <li>
      查询
     </li>
     <li>
      插入
     </li>
     <li>
      删除
     </li>
     <li>
      更新
     </li>
    </ul>
    <h4>
     <a id="_130">
     </a>
     （三）关系模型的完整性约束
    </h4>
    <ul>
     <li>
      <strong>
       实体完整性
      </strong>
     </li>
     <li>
      <strong>
       参照完整性
      </strong>
     </li>
     <li>
      <strong>
       用户定义的完整性
      </strong>
     </li>
    </ul>
    <h2>
     <a id="13___135">
     </a>
     1.3 数据库系统的结构
    </h2>
    <h3>
     <a id="131___136">
     </a>
     1.3.1 数据库系统模式的概念
    </h3>
    <ul>
     <li>
      <strong>
       模式（Schema）:数据库逻辑结构和特征的描述
      </strong>
      <ul>
       <li>
        是型的描述，不涉及具体值
       </li>
       <li>
        反映的是数据的结构及其联系
       </li>
       <li>
        模式是相对稳定的
       </li>
      </ul>
     </li>
     <li>
      实例（Instance）:模式的一个具体值
      <ul>
       <li>
        反映数据库某一时刻的状态
       </li>
       <li>
        同一个模式可以有很多实例
       </li>
       <li>
        实例随数据库中的数据的更新而变动
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="132___145">
     </a>
     1.3.2 数据库系统的三级模式结构
    </h3>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/899d02e787f248168194d1e92acbd106.png"/>
    </p>
    <h4>
     <a id="Schema_148">
     </a>
     （一）模式（Schema）
    </h4>
    <ul>
     <li>
      模式（也称逻辑模式）：数据库中
      <strong>
       全体数据的逻辑结构和特征的描述
      </strong>
      ，
      <strong>
       所有用户的公共数据视图
      </strong>
     </li>
     <li>
      <strong>
       一个数据库只有一个模式
      </strong>
     </li>
     <li>
      <strong>
       模式的地位
      </strong>
      ：是数据库系统模式结构的中间层
      <br/>
      与数据的物理存储细节和硬件环境无关
      <br/>
      与具体的应用程序、开发工具及高级程序设计语言无关
     </li>
     <li>
      <strong>
       模式的定义
      </strong>
      <br/>
      数据的逻辑结构（数据项的名字、类型、取值范围等）
      <br/>
      数据之间的联系
      <br/>
      数据有关的安全性、完整性要求
     </li>
    </ul>
    <h4>
     <a id="External_Schema_158">
     </a>
     （二）外模式（External Schema）
    </h4>
    <ul>
     <li>
      <p>
       外模式：数据库用户（包括应用程序员和最终用户）使用的
       <strong>
        局部数据的逻辑结构和特征的描述
       </strong>
      </p>
     </li>
     <li>
      <p>
       外模式的地位：介于模式与应用之间
      </p>
      <ul>
       <li>
        <strong>
         模式与外模式的关系：一对多
        </strong>
        <ul>
         <li>
          外模式通常是模式的子集
         </li>
         <li>
          一个数据库可以有多个外模式。反映了不同的用户的应用需求、看待数据的方式、对数据保密的要求
         </li>
         <li>
          对模式中同一数据，在外模式中的结构、类型、长度、保密级别等都可以不同
         </li>
        </ul>
       </li>
       <li>
        <strong>
         外模式与应用的关系：一对多
        </strong>
        <ul>
         <li>
          同一外模式也可以为某一用户的多个应用系统所使用
         </li>
         <li>
          <strong>
           但一个应用程序只能使用一个外模式
          </strong>
         </li>
        </ul>
       </li>
      </ul>
     </li>
     <li>
      <p>
       外模式的用途
      </p>
      <ul>
       <li>
        保证数据库安全性的一个有力措施
       </li>
       <li>
        每个用户只能看见和访问所对应的外模式中的数据
       </li>
      </ul>
     </li>
    </ul>
    <h4>
     <a id="Internal_Schema_174">
     </a>
     （三）内模式（Internal Schema）
    </h4>
    <ul>
     <li>
      内模式（也称存储模式）：
      <strong>
       是数据物理结构和存储方式的描述，是数据在数据库内部的表示方式
      </strong>
      <ul>
       <li>
        记录的存储方式（例如，顺序存储，按照B树结构存储， 按hash方法存储等）
       </li>
       <li>
        索引的组织方式
       </li>
       <li>
        数据是否压缩存储
       </li>
       <li>
        数据是否加密
       </li>
       <li>
        数据存储记录结构的规定
       </li>
      </ul>
     </li>
     <li>
      <strong>
       一个数据库只有一个内模式
      </strong>
     </li>
    </ul>
    <h3>
     <a id="133___182">
     </a>
     1.3.3 数据库的二级映像功能与数据独立性
    </h3>
    <p>
     二级映象在数据库管理系统内部实现这三个抽象层次的联系和转换
    </p>
    <ul>
     <li>
      外模式／模式映像
     </li>
     <li>
      模式／内模式映像
     </li>
    </ul>
    <h4>
     <a id="_186">
     </a>
     （一）外模式／模式映像
    </h4>
    <ul>
     <li>
      模式：描述的是数据的全局逻辑结构
     </li>
     <li>
      外模式：描述的是数据的局部逻辑结构
     </li>
     <li>
      <strong>
       同一个模式可以有任意多个外模式
      </strong>
     </li>
     <li>
      <strong>
       每一个外模式，数据库系统都有一个外模式／模式映象
      </strong>
      ，定义外模式与模式之间的对应关系
     </li>
     <li>
      映象定义通常包含在各自外模式的描述中
      <br/>
      <br/>
     </li>
     <li>
      <strong>
       保证数据的逻辑独立性
      </strong>
      <ul>
       <li>
        当模式改变时，数据库管理员对外模式／模式映象作相应改变，使外模式保持不变
       </li>
       <li>
        应用程序是依据数据的外模式编写的，应用程序不必修改，保证了数据与程序的逻辑独立性，简称数据的逻辑独立性
       </li>
      </ul>
     </li>
    </ul>
    <h4>
     <a id="_197">
     </a>
     （二）模式／内模式映像
    </h4>
    <ul>
     <li>
      <p>
       模式／内模式映象定义了数据全局逻辑结构与存储结构之间的对应关系。
       <br/>
       <font color="grey">
        例如，说明逻辑记录和字段在内部是如何表示的
       </font>
      </p>
     </li>
     <li>
      <p>
       <strong>
        数据库中模式／内模式映象是唯一的
       </strong>
      </p>
     </li>
     <li>
      <p>
       该映象定义通常包含在模式描述中
      </p>
     </li>
     <li>
      <p>
       <strong>
        保证数据的物理独立性
       </strong>
      </p>
      <ul>
       <li>
        当数据库的存储结构改变了（例如选用了另一种存储结构），数据库管理员修改模式／内模式映象，使模式保持不变。
       </li>
       <li>
        应用程序不受影响。保证了数据与程序的物理独立性，简称数据的物理独立性
       </li>
      </ul>
     </li>
    </ul>
