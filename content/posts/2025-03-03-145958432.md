    <h2 id="%E5%89%8D%E8%A8%80" name="%E5%89%8D%E8%A8%80">

     前言
    </h2>
    <p>
    </p>
    <p>
     本文重点介绍开展软件需求分析的方法。
    </p>
    <hr/>
    <h2 id="1.1%20%E5%B8%A6%E5%8F%98%E5%8E%8B%E5%99%A8%E7%9A%84%E5%8F%8C%E7%BB%9E%E7%BA%BF%E6%80%BB%E7%BA%BF" name="1.1%20%E5%B8%A6%E5%8F%98%E5%8E%8B%E5%99%A8%E7%9A%84%E5%8F%8C%E7%BB%9E%E7%BA%BF%E6%80%BB%E7%BA%BF">
     需求分析方法
    </h2>
    <h3 id="%E5%B7%A5%E5%85%B7%E5%92%8C%E6%96%B9%E6%B3%95" name="%E5%B7%A5%E5%85%B7%E5%92%8C%E6%96%B9%E6%B3%95" style="margin-left:0; margin-right:0; text-align:justify">
     工具和方法
    </h3>
    <p style="margin-left:0; margin-right:0; text-align:justify">
     软件需求可以维护在ALM系统中，譬如：doors，codeBeamer等，JIRA适合互联网行业，并不合适颗粒度较细的汽车级控制器开发。同时可以使用 UML、Visio 和 EA 作为辅助工具进行软件需求分析，方法如下：
    </p>
    <ul>
     <li style="text-align:left">
      例图
     </li>
     <li style="text-align:left">
      序列图
     </li>
     <li style="text-align:left">
      活动图
     </li>
     <li style="text-align:left">
      状态图
     </li>
    </ul>
    <p style="margin-left:0; margin-right:0; text-align:justify">
     注：SRS 将以文字和图表两种形式完成，以使软件架构师和软件开发工程师进一步理解 SRS。
    </p>
    <h3 id="%E5%85%B7%E4%BD%93%E5%88%86%E6%9E%90%E6%96%B9%E6%B3%95" name="%E5%85%B7%E4%BD%93%E5%88%86%E6%9E%90%E6%96%B9%E6%B3%95" style="margin-left:0px; margin-right:0px; text-align:justify">
     具体分析方法
    </h3>
    <p style="margin-left:0; margin-right:0; text-align:justify">
     从系统分析方面来看，需求分析方法可以分为四类：
    </p>
    <ul>
     <li style="text-align:left">
      功能分解法。
     </li>
     <li style="text-align:left">
      结构分析法。
     </li>
     <li style="text-align:left">
      信息建模方法。
     </li>
     <li style="text-align:left">
      面向对象分析方法。
     </li>
    </ul>
    <p style="margin-left:0; margin-right:0; text-align:justify">
     在 ecu 的软件开发过程中，当采用功能分解和结构分析方法时，通常将 UML 应用于软件需求分析中。
    </p>
    <p style="margin-left:0; margin-right:0; text-align:justify">
     在分析过程中，同步编写符合 2.3 规则的文本化需求，并按照以下方法对需求进行结构化描述：
    </p>
    <ul>
     <li style="text-align:left">
      功能用例的定义。
     </li>
     <li style="text-align:left">
      操作场景和顺序分析。
     </li>
     <li style="text-align:left">
      根据功能用例进行结构功能分解。
     </li>
     <li style="text-align:left">
      ecu 软件接口分析与描述。
     </li>
    </ul>
    <p style="margin-left:0; margin-right:0; text-align:justify">
     文本需求对于具体细节是有效的，但当需要提供需求的相互关系和上下文时，则无效。因此，图或模型（用例、UML、Simulink 模型）是软件需求规范中促进可读性和完整性所必需的部分。
    </p>
    <p style="margin-left:0; margin-right:0; text-align:justify">
     分析过程完成后，就可以得到软件需求和接口需求。
    </p>
    <h3 id="%E5%AF%B9%E8%BF%90%E8%A1%8C%E7%8E%AF%E5%A2%83%E7%9A%84%E5%BD%B1%E5%93%8D" name="%E5%AF%B9%E8%BF%90%E8%A1%8C%E7%8E%AF%E5%A2%83%E7%9A%84%E5%BD%B1%E5%93%8D" style="margin-left:0; margin-right:0; text-align:justify">
     对运行环境的影响
    </h3>
    <p style="margin-left:0; margin-right:0; text-align:justify">
     对运行环境影响的分析，既包括对范围内软件的影响，也包括对其他软件部件、其他系统或整车考虑以下可能部件的影响：
    </p>
    <ul>
     <li>
      接口
     </li>
    </ul>
    <blockquote>
     <ol>
      <li style="text-align:left">
       信号及信号质量
      </li>
      <li style="text-align:left">
       电压和电流
      </li>
     </ol>
    </blockquote>
    <ul>
     <li style="text-align:left">
      环境
     </li>
    </ul>
    <blockquote>
     <ol>
      <li style="text-align:left">
       温度
      </li>
      <li style="text-align:left">
       EMC
      </li>
     </ol>
    </blockquote>
    <ul>
     <li style="text-align:left">
      性能
     </li>
    </ul>
    <blockquote>
     <ol>
      <li style="text-align:left">
       接口响应时间（信号响应、采样时间、周期时间、总线负载、信号延迟、抖动）
      </li>
      <li style="text-align:left">
       微控制器响应时间（处理时间）
      </li>
     </ol>
    </blockquote>
    <ul>
     <li style="text-align:left">
      资源
     </li>
    </ul>
    <blockquote>
     <ol>
      <li style="text-align:left">
       RAM / ROM 内存使用情况
      </li>
      <li style="text-align:left">
       EEPROM / DataFlash 内存使用情况
      </li>
     </ol>
    </blockquote>
    <p style="margin-left:0; margin-right:0; text-align:justify">
     软件/系统交互的其他系统或系统元素（如硬件），构成软件/系统的操作环境。它可以被看作是“固定的”。
    </p>
