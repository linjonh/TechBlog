    <h3>
     <a id="_select_3">
     </a>
     关于 select
    </h3>
    <h4>
     <a id="select__4">
     </a>
     select 的缺点
    </h4>
    <p>
     select 复用方法太老，性能不适合以 Web 服务器端开发为主流的现代开发环境。
    </p>
    <p>
     在第 12 章的基于 select 的 I/O 复用服务器端的代码中，有这样一段代码：
    </p>
    <pre><code class="prism language-cpp">    <span class="token keyword">while</span> <span class="token punctuation">(</span><span class="token number">1</span><span class="token punctuation">)</span>

    <p>
     里面有 2 点不合理的设计：
    </p>
    <ol>
     <li>
      每次调用 select 函数前都要复制并保存原有监控对象信息，每次调用 select 函数时都要传递新的监控对象信息。
     </li>
     <li>
      调用 select 函数后对所有文件描述符循环遍历一次。
     </li>
    </ol>
    <p>
     应用程序向操作系统传递数据将对程序造成很大负担，而且无法通过代码优化，每次调用 select 函数时都要传递新的监控对象信息将成为性能上的致命弱点。
    </p>
    <blockquote>
     <p>
      select 函数与文件描述符有关，是监视套接字变化的函数。而套接字由操作系统管理，所以 select 函数必须借助于操作系统才能完成功能。
     </p>
    </blockquote>
    <p>
     select 函数的缺点可以通过如下方式弥补：“仅向操作系统传递 1 次监视对象，监视范围或内容发生变化时只通知发送变化的事项。”
    </p>
    <p>
     Linux 的支持方式是 epoll，Windows 的支持方式是 IOCP。
    </p>
    <h4>
     <a id="select__81">
     </a>
     select 的优点
    </h4>
    <p>
     select 的兼容性比较高，这样就可以支持很多的操作系统，不受平台的限制，满足以下两个条件使可以使用 select 函数：
    </p>
    <ul>
     <li>
      服务器接入者少
     </li>
     <li>
      程序应该具有兼容性
     </li>
    </ul>
    <h3>
     <a id="epoll__89">
     </a>
     epoll 理解及应用
    </h3>
    <h4>
     <a id="_epoll__91">
     </a>
     实现 epoll 时必要的函数和结构体
    </h4>
    <p>
     能够克服 select 函数缺点的 epoll 函数具有以下优点，这些优点正好与之前的 select 函数缺点相反。
    </p>
    <ul>
     <li>
      无需编写以监视状态变化为目的的针对所有文件描述符的循环语句。
     </li>
     <li>
      调用对应于 select 函数的 epoll_wait 函数时无需每次传递监视对象信息。
     </li>
    </ul>
    <p>
     下面是 epoll 服务器端实现中需要的 3 个函数的功能：
    </p>
    <ul>
     <li>
      epoll_create：创建保存 epoll 文件描述符的空间
     </li>
     <li>
      epoll_ctl：向空间注册并注销文件描述符
     </li>
     <li>
      epoll_wait：与 select 函数类似，等待文件描述符发生变化
     </li>
    </ul>
    <p>
     select 函数中为了保存监视对象的文件描述符，直接声明了 fd_set 变量，但 epoll 方式下的操作系统负责保存监视对象文件描述符，因此需要向操作系统请求创建保存文件描述符的空间，此时用的函数就是 epoll_create。
    </p>
    <p>
     此外，为了添加和删除监视对象文件描述符，select 方式中需要 FD_SET、FD_CLR 函数。但在 epoll 方式中，通过 epoll_ctl 函数请求操作系统完成。
    </p>
    <p>
     最后，select 方式下调用 select 函数等待文件描述符的变化，而 epoll 方式则调用 epoll_wait 函数。
    </p>
    <p>
     还有，select 方式中通过 fd_set 变量查看监视对象的状态变化，而 epoll 方式通过如下结构体 epoll_event 将发生变化的（发生事件的）文件描述符单独集中在一起。
    </p>
    <pre><code class="prism language-cpp"><span class="token keyword">struct</span> <span class="token class-name">epoll_event</span>
<span class="token punctuation">{<!-- --></span>
    __uint32_t events<span class="token punctuation">;</span>
    epoll_data_t data<span class="token punctuation">;</span>
<span class="token punctuation">}</span><span class="token punctuation">;</span>

<span class="token keyword">typedef</span> <span class="token keyword">union</span> epoll_data <span class="token punctuation">{<!-- --></span>
    <span class="token keyword">void</span> <span class="token operator">*</span>ptr<span class="token punctuation">;</span>
    <span class="token keyword">int</span> fd<span class="token punctuation">;</span>
    __uint32_t u32<span class="token punctuation">;</span>
    __uint64_t u64<span class="token punctuation">;</span>
<span class="token punctuation">}</span> epoll_data_t<span class="token punctuation">;</span>
</code></pre>
    <p>
     声明足够大的 epoll_event 结构体数组后，传递给 epoll_wait 函数时，发生变化的文件描述符信息将被填入数组。因此，无需像 select 函数那样针对所有文件描述符进行循环。
    </p>
    <h4>
     <a id="epoll_create_130">
     </a>
     epoll_create
    </h4>
    <p>
     下面是 epoll_create 函数的原型：
    </p>
    <pre><code class="prism language-cpp"><span class="token macro property"><span class="token directive-hash">#</span><span class="token directive keyword">include</span> <span class="token string">&lt;sys/epoll.h&gt;</span></span>

<span class="token keyword">int</span> <span class="token function">epoll_create</span><span class="token punctuation">(</span><span class="token keyword">int</span> size<span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     参数 size 表示 epoll 实例的大小。
    </p>
    <p>
     成功时返回 epoll 的文件描述符，失败时返回 -1。
    </p>
    <p>
     调用 epoll_create 函数时创建的文件描述符保存空间称为「epoll 例程」。
    </p>
    <p>
     通过参数 size 传递的值决定 epoll 例程的大小，但该值只是向操作系统提出的建议。换言之，size 并不用来决定 epoll 的大小，而仅供操作系统参考。
    </p>
    <blockquote>
     <p>
      Linux 2.6.8 之后的内核将完全忽略传入 epoll_create 函数的 size 参数，因为内核会根据情况调整 epoll 例程大小。
     </p>
    </blockquote>
    <p>
     epoll_create 函数创建的资源与套接字相同，也由操作系统管理。因此，该函数和创建套接字的情况相同，也会返回文件描述符，也就是说返回的文件描述符主要用于区分 epoll 例程。需要终止时，与其他文件描述符相同，也要调用 close 函数。
    </p>
    <h4>
     <a id="epoll_ctl_154">
     </a>
     epoll_ctl
    </h4>
    <p>
     生成 epoll 例程后，应在其内部注册监视对象文件描述符，此时使用 epoll_ctl 函数。
    </p>
    <pre><code class="prism language-cpp"><span class="token macro property"><span class="token directive-hash">#</span><span class="token directive keyword">include</span> <span class="token string">&lt;sys/epoll.h&gt;</span></span>

<span class="token keyword">int</span> <span class="token function">epoll_ctl</span><span class="token punctuation">(</span><span class="token keyword">int</span> epfd<span class="token punctuation">,</span> <span class="token keyword">int</span> op<span class="token punctuation">,</span> <span class="token keyword">int</span> fd<span class="token punctuation">,</span> <span class="token keyword">struct</span> <span class="token class-name">epoll_event</span> <span class="token operator">*</span>event<span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     参数：
    </p>
    <ul>
     <li>
      epfd：用于注册监视对象的 epoll 例程的文件描述符
     </li>
     <li>
      op：用于指定监视对象的添加、删除或更改等操作
     </li>
     <li>
      fd：需要注册的监视对象文件描述符
     </li>
     <li>
      event：监视对象的事件类型
     </li>
    </ul>
    <p>
     成功时返回 0 ，失败时返回 -1。
    </p>
    <p>
     与其他 epoll 函数相比，该函数看起来有些复杂，但通过调用语句就很容易理解，假设按照如下形式调用 epoll_ctl 函数：
    </p>
    <pre><code class="prism language-cpp"><span class="token function">epoll_ctl</span><span class="token punctuation">(</span>A<span class="token punctuation">,</span> EPOLL_CTL_ADD<span class="token punctuation">,</span> B<span class="token punctuation">,</span> C<span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     第二个参数 EPOLL_CTL_ADD 意味着「添加」，上述语句有如下意义：“epoll 例程 A 中注册文件描述符 B ，主要目的是为了监视参数 C 中的事件。”
    </p>
    <p>
     再介绍一个调用语句：
    </p>
    <pre><code class="prism language-cpp"><span class="token function">epoll_ctl</span><span class="token punctuation">(</span>A<span class="token punctuation">,</span> EPOLL_CTL_DEL<span class="token punctuation">,</span> B<span class="token punctuation">,</span> <span class="token constant">NULL</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     上述语句中第二个参数意味着「删除」，有以下含义：“从 epoll 例程 A 中删除文件描述符 B。”
    </p>
    <p>
     从上述示例中可以看出，从监视对象中删除时，不需要监视类型，因此向第四个参数可以传递为 NULL。
    </p>
    <p>
     下面是 epoll_ctl 函数的第二个参数的类型及其含义：
    </p>
    <ul>
     <li>
      EPOLL_CTL_ADD：将文件描述符注册到 epoll 例程
     </li>
     <li>
      EPOLL_CTL_DEL：从 epoll 例程中删除文件描述符
     </li>
     <li>
      EPOLL_CTL_MOD：更改注册的文件描述符的关注事件发生情况
     </li>
    </ul>
    <p>
     epoll_event 结构体用于保存事件的文件描述符结合。但也可以在 epoll 例程中注册文件描述符时，用于注册关注的事件。该函数中 epoll_event 结构体的定义并不显眼，因此通过调用语句说明该结构体在 epoll_ctl 函数中的应用。
    </p>
    <pre><code class="prism language-cpp"><span class="token keyword">struct</span> <span class="token class-name">epoll_event</span> event<span class="token punctuation">;</span>
<span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">.</span>
event<span class="token punctuation">.</span>events <span class="token operator">=</span> EPOLLIN<span class="token punctuation">;</span> <span class="token comment">// 发生需要读取数据的情况时</span>
event<span class="token punctuation">.</span>data<span class="token punctuation">.</span>fd <span class="token operator">=</span> sockfd<span class="token punctuation">;</span>
<span class="token function">epoll_ctl</span><span class="token punctuation">(</span>epfd<span class="token punctuation">,</span> EPOLL_CTL_ADD<span class="token punctuation">,</span> sockfd<span class="token punctuation">,</span> <span class="token operator">&amp;</span>event<span class="token punctuation">)</span><span class="token punctuation">;</span>
<span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">.</span>
</code></pre>
    <p>
     上述代码将sockfd 注册到 epoll 例程 epfd 中，并在需要读取数据的情况下产生相应事件。接下来给出 epoll_event 的成员 events 中可以保存的常量及所指的事件类型：
    </p>
    <ul>
     <li>
      EPOLLIN：需要读取数据的情况
     </li>
     <li>
      EPOLLOUT：输出缓冲为空，可以立即发送数据的情况
     </li>
     <li>
      EPOLLPRI：收到 OOB 数据的情况
     </li>
     <li>
      EPOLLRDHUP：断开连接或半关闭的情况，这在边缘触发方式下非常有用
     </li>
     <li>
      EPOLLERR：发生错误的情况
     </li>
     <li>
      EPOLLET：以边缘触发的方式得到事件通知
     </li>
     <li>
      EPOLLONESHOT：发生一次事件后，相应文件描述符不再收到事件通知。因此需要向 epoll_ctl 函数的第二个参数传递 EPOLL_CTL_MOD ，再次设置事件
     </li>
    </ul>
    <p>
     可通过位或运算同时传递多个上述参数。
    </p>

    <p>
     总结一下epoll的流程：
    </p>
    <ol>
     <li>
      <strong>
       创建套接字
      </strong>
      ：首先，创建一个套接字（一般是服务器套接字），用于监听连接请求或数据传输。
     </li>
     <li>
      <strong>
       创建 epoll 实例
      </strong>
      ：通过调用 epoll_create() 函数创建一个 epoll 实例，用于管理文件描述符的事件。
     </li>
     <li>
      <strong>
       注册事件
      </strong>
      ：将要监听的套接字（文件描述符）注册到 epoll 实例中，通过调用 epoll_ctl() 函数，并设置感兴趣的事件类型（如读事件、写事件等）以及相关的数据（如文件描述符本身）。
     </li>
     <li>
      <strong>
       进入事件循环
      </strong>
      ：进入一个循环，在循环中调用 epoll_wait() 函数来等待事件的发生。epoll_wait() 会阻塞程序直到有事件发生或超时。
     </li>
     <li>
      <strong>
       处理事件
      </strong>
      ：一旦有事件发生，epoll_wait() 函数会返回发生事件的文件描述符集合。在事件循环内，遍历这些事件，根据文件描述符的类型进行不同的处理：
      <ul>
       <li>
        如果是服务器套接字上的事件，表示有新的连接请求，使用 accept() 函数接受连接，然后将新的客户端套接字注册到 epoll 实例中。
       </li>
       <li>
        如果是客户端套接字上的事件，表示有数据传输事件发生，可以使用 read() 函数读取数据，并根据需要进行处理，然后使用 write() 函数回传数据。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       移除事件
      </strong>
      ：如果需要，可以在事件处理后将文件描述符从 epoll 实例中移除，通过调用 epoll_ctl() 函数，通常是在客户端关闭连接时执行。
     </li>
     <li>
      <strong>
       关闭套接字和 epoll 实例
      </strong>
      ：在程序结束时，确保关闭所有的套接字和释放分配的内存，包括关闭 epoll 实例。
     </li>
    </ol>
    <h3>
     <a id="_381">
     </a>
     条件触发和边缘触发
    </h3>
    <h4>
     <a id="_383">
     </a>
     条件触发和边缘触发的区别
    </h4>
    <p>
     条件触发和边缘触发的区别在于发生事件的时间点。
    </p>
    <p>
     <strong>
      条件触发方式中，只要输入缓冲有数据就会一直通知该事件
     </strong>
     。例如，服务器端输入缓冲收到 50 字节数据时，服务器端操作系统将通知该事件（注册到发生变化的文件描述符）。但是服务器端读取 20 字节后还剩下 30 字节的情况下，仍会注册事件。也就是说，条件触发方式中，只要输入缓冲中还剩有数据，就将以事件方式再次注册。
    </p>
    <p>
     <strong>
      边缘触发中输入缓冲收到数据时仅注册 1 次该事件。即使输入缓冲中还留有数据，也不会再进行注册。
     </strong>
    </p>

    <h4>
     <a id="_516">
     </a>
     边缘触发的服务器端实现中必知的两点
    </h4>
    <ul>
     <li>
      <p>
       通过 errno 变量验证错误原因
      </p>
     </li>
     <li>
      <p>
       为了完成非阻塞（Non-blocking）I/O ，更改套接字特性
      </p>
      <p>
       Linux 套接字相关函数一般通过 -1 通知发生了错误。虽然知道发生了错误，但仅凭这些内容无法得知产生错误的原因。因此，为了在发生错误的时候提额外的信息，Linux 声明了如下全局变量：
      </p>
     </li>
    </ul>
    <pre><code class="prism language-cpp"><span class="token keyword">int</span> errno<span class="token punctuation">;</span>
</code></pre>
    <p>
     为了访问该变量，需要引入 error.h 头文件，因为此头文件有上述变量的 extern 声明。另外，每种函数发生错误时，保存在 errno 变量中的值都不同，没必须强记所有可能的值。
    </p>
    <p>
     本节只介绍一种错误：read 函数发现输入缓冲中没有数据可读时返回 -1，同时在 errno 中保存 EAGAIN 常量。
    </p>
    <p>
     下面是 Linux 中提供的读取或更改文件属性的办法：
    </p>
    <pre><code class="prism language-cpp"><span class="token macro property"><span class="token directive-hash">#</span><span class="token directive keyword">include</span> <span class="token string">&lt;fcntl.h&gt;</span></span>

<span class="token keyword">int</span> <span class="token function">fcntl</span><span class="token punctuation">(</span><span class="token keyword">int</span> filedes<span class="token punctuation">,</span> <span class="token keyword">int</span> cmd<span class="token punctuation">,</span> <span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     参数：
    </p>
    <ul>
     <li>
      filedes : 属性更改目标的文件描述符
     </li>
     <li>
      cmd : 表示函数调用目的
     </li>
    </ul>
    <p>
     成功时返回 cmd 参数相关值，失败时返回 -1。
    </p>
    <p>
     从上述声明可以看出 fcntl 有可变参数的形式。如果向第二个参数传递 F_GETFL ，可以获得第一个参数所指的文件描述符属性（int 型）。反之，如果传递 F_SETFL ，可以更改文件描述符属性。若希望将文件（套接字）改为非阻塞模式，需要如下 2 条语句：
    </p>
    <pre><code class="prism language-cpp"><span class="token keyword">int</span> flag <span class="token operator">=</span> <span class="token function">fcntl</span><span class="token punctuation">(</span>fd<span class="token punctuation">,</span> F_GETFL<span class="token punctuation">,</span> <span class="token number">0</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
<span class="token function">fcntl</span><span class="token punctuation">(</span>fd<span class="token punctuation">,</span> F_SETFL<span class="token punctuation">,</span> flag <span class="token operator">|</span> O_NONBLOCK<span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     通过第一条语句，获取之前设置的属性信息，通过第二条语句在此基础上添加非阻塞 O_NONBLOCK 标志。调用 read &amp; write 函数时，无论是否存在数据，都会形成非阻塞文件（套接字）。fcntl 函数的适用范围很广。
    </p>
    <h4>
     <a id="_558">
     </a>
     实现边缘触发的回声服务器端
    </h4>
    <p>
     通过 errno 确认错误的原因是：边缘触发方式中，接收数据仅注册一次该事件。
    </p>
    <p>
     因为这种特点，一旦发生输入相关事件时，就应该读取输入缓冲中的全部数据。因此需要验证输入缓冲是否为空。
    </p>
    <p>

    <h4>
     <a id="_691">
     </a>
     条件触发和边缘触发孰优孰劣
    </h4>
    <p>
     边缘触发方式可以做到：分离接收数据和处理数据的时间点。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/9da4517717e14b2ebd4d830465fed7c5.png"/>
    </p>
    <p>
     运行流程如下：
    </p>
    <ol>
     <li>
      服务器端分别从 A B C 接收数据
     </li>
     <li>
      服务器端按照 A B C 的顺序重新组合接收到的数据
     </li>
     <li>
      组合的数据将发送给任意主机。
     </li>
    </ol>
    <p>
     为了完成这个过程，如果可以按照如下流程运行，服务端的实现并不难：
    </p>
    <ol>
     <li>
      客户端按照 A B C 的顺序连接服务器，并且按照次序向服务器发送数据
     </li>
     <li>
      需要接收数据的客户端应在客户端 A B C 之前连接到服务器端并等待
     </li>
    </ol>
    <p>
     但是实际情况中可能是下面这样：
    </p>
    <ol>
     <li>
      客户端 C 和 B 正在向服务器发送数据，但是 A 并没有连接到服务器
     </li>
     <li>
      客户端 A B C 乱序发送数据
     </li>
     <li>
      服务端已经接收到数据，但是要接收数据的目标客户端并没有连接到服务器端。
     </li>
    </ol>
    <p>
     因此，即使输入缓冲收到数据，服务器端也能决定读取和处理这些数据的时间点，这样就给服务器端的实现带来很大灵活性。
    </p>
    <h3>
     <a id="_720">
     </a>
     习题
    </h3>
    <h4>
     <a id="1_select__721">
     </a>
     （1）利用 select 函数实现服务器端时，代码层面存在的两个缺点是？
    </h4>
    <ol>
     <li>
      每次调用 select 函数前都要复制并保存原有监控对象信息，每次调用 select 函数时都要传递新的监控对象信息。
     </li>
     <li>
      调用 select 函数后对所有文件描述符循环遍历一次。
     </li>
    </ol>
    <p>
     换一种说法：
    </p>
    <ol>
     <li>
      文件描述符集合需手动维护，代码繁琐
      <br/>
      每次调用 select 后，原有的文件描述符集合会被内核修改（通过值-结果参数），因此必须在每次循环中重新初始化并重新添加所有需要监视的文件描述符。
     </li>
     <li>
      需线性遍历所有描述符以检测事件，效率低下
      <br/>
      select 返回后，需遍历所有被监视的文件描述符（通过 FD_ISSET 检查），即使仅有少数描述符处于就绪状态。在高并发场景（如数万个连接）下，这种 O(n) 的遍历会显著降低性能。
     </li>
    </ol>
    <h4>
     <a id="2_select__epoll__732">
     </a>
     （2）无论是 select 方式还是 epoll 方式，都需要将监视对象文件描述符信息通过函数调用传递给操作系统。请解释传递该信息的原因。
    </h4>
    <p>
     文件描述符（如套接字）的底层状态变化（如数据到达、缓冲区可写等）由内核管理，用户程序无法直接访问硬件或内核的网络协议栈。因此，必须通过系统调用将需要监控的文件描述符信息传递给内核，委托内核进行事件监听。若用户程序不传递文件描述符集合，内核无从得知哪些描述符需要被监视，也就无法在事件发生时通知用户程序。
    </p>
    <p>
     用户程序运行在用户空间，无法直接访问内核空间的数据结构。因此，文件描述符集合必须通过系统调用（如 select、epoll_ctl）从用户空间拷贝到内核空间，供内核处理。
    </p>
    <h4>
     <a id="3select__epoll__737">
     </a>
     （3）select 方式和 epoll 方式的最大差异在于监视对象文件描述符传递给操作系统的方式。请说明具体的差异，并解释为何存在这种差异。
    </h4>
    <p>
     select 函数每次调用都要传递所有的监视对象信息，而 epoll 函数仅向操作系统传递 1 次监视对象，监视范围或内容发生变化时只通知发生变化的事项。
    </p>
    <p>
     差异背后的设计原因：
    </p>
    <ol>
     <li>
      select 诞生较早，未考虑高并发场景；epoll 专为高并发优化，解决 select 的性能瓶颈。
     </li>
     <li>
      内核数据结构的不同
      <ul>
       <li>
        select 的朴素实现：内核通过线性遍历用户传递的 fd 集合，检查每个 fd 的状态。时间复杂度为 O(n)，性能随 fd 数量增长急剧下降。
       </li>
       <li>
        epoll 的高效数据结构：内核使用红黑树组织监控的 fd（增删改查时间复杂度为 O(log n)），并通过就绪链表直接返回活跃事件，时间复杂度为 O(1)。
       </li>
      </ul>
     </li>
     <li>
      事件通知机制的改进
      <ul>
       <li>
        select 的无状态性：每次调用都是独立的，内核不保留任何上下文，因此必须全量传递 fd 集合。这导致频繁的用户态-内核态切换和数据拷贝。
       </li>
       <li>
        epoll 的有状态性：内核通过 epoll_create 创建上下文，长期维护监控列表，用户程序通过 epoll_ctl 动态管理。仅需在 fd 状态变化时更新内核，epoll_wait 仅获取就绪事件，减少系统调用和数据拷贝。
       </li>
      </ul>
     </li>
    </ol>
    <h4>
     <a id="4_epoll__select__select__select__750">
     </a>
     （4）虽然 epoll 是 select 的改进方案，但 select 也有自己的优点。在何种情况下使用 select 更加合理？
    </h4>
    <ol>
     <li>
      跨平台兼容性要求高
     </li>
     <li>
      服务器或客户端仅需处理少量并发连接（如嵌入式设备、简单工具）
     </li>
     <li>
      运行在老旧内核（如 Linux 2.4 之前）或资源受限的嵌入式系统
     </li>
     <li>
      短连接或临时性任务，例如一个 DNS 查询服务器，每个请求仅需处理一次 UDP 数据包
     </li>
    </ol>
    <h4>
     <a id="5epoll__759">
     </a>
     （5）epoll 是以条件触发和边缘触发方式工作。二者有何差别？从输入缓冲的角度说明这两种方式通知事件的时间点差异。
    </h4>
    <p>
     条件触发方式中，只要输入缓冲有数据就会一直通知该事件。
    </p>
    <p>
     边缘触发中输入缓冲收到数据时仅注册 1 次该事件。即使输入缓冲中还留有数据，也不会再进行注册。
    </p>
    <h4>
     <a id="6_766">
     </a>
     （6）采用边缘触发时可以分离数据的接收和处理时间点。请说明其优点和原因。
    </h4>
    <p>
     优点：给服务器端的实现带来很大灵活性。
    </p>
    <p>
     原因：边缘触发仅在输入缓冲区的数据状态发生变更时（如从空变为非空，或新增数据到达）通知一次事件。
    </p>
    <p>
     以客户端发送 2KB 数据 为例：
    </p>
    <ol>
     <li>
      数据到达输入缓冲区 → epoll_wait 立即通知可读事件（状态从空→非空）。
     </li>
     <li>
      服务器调用 read(fd, buf, 1KB) → 读取 1KB，缓冲区剩余 1KB。
     </li>
     <li>
      再次调用 epoll_wait → 不会通知可读事件（状态未发生新变化）。
     </li>
     <li>
      客户端再发送 0.5KB 数据 → 缓冲区变为 1.5KB，状态变更（新增数据）。
     </li>
     <li>
      epoll_wait 再次通知可读事件。
     </li>
    </ol>
