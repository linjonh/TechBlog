     第十关（双写绕过）：
    </h3>
    <p>
     源码：
     <br/>
    </p>
    <blockquote>
     <pre><code>$is_upload = false;
$msg = null;
if (isset($_POST['submit'])) {
    if (file_exists(UPLOAD_PATH)) {
        $deny_ext = array("php","php5","php4","php3","php2","html","htm","phtml","pht","jsp","jspa","jspx","jsw","jsv","jspf","jtml","asp","aspx","asa","asax","ascx","ashx","asmx","cer","swf","htaccess");

        $file_name = trim($_FILES['upload_file']['name']);
        $file_name = str_ireplace($deny_ext,"", $file_name);
        $temp_file = $_FILES['upload_file']['tmp_name'];
        $img_path = UPLOAD_PATH.'/'.$file_name;        
        if (move_uploaded_file($temp_file, $img_path)) {
            $is_upload = true;
        } else {
            $msg = '上传出错！';
        }
    } else {
        $msg = UPLOAD_PATH . '文件夹不存在,请手工创建！';
    }
}
</code></pre>
    </blockquote>
    <p>
     源码分析：先来看一下这几个函数
    </p>
    <p>
     <img alt="" height="162" src="https://i-blog.csdnimg.cn/direct/89e43efef0af4b4db19ac25fd7eb2e9c.png" width="518"/>
    </p>
    <p>
     先使用此函数去掉两边的空白字符以及转义的字符，然后下一个函数再将上传的文件名跟黑名单里面的作比较，如果相同，则从$file_name中删除对应部分
    </p>
    <p>
     <strong>
      <code>
       $file_name = str_ireplace($deny_ext,"", $file_name);
      </code>
     </strong>
     这段代码试图替换掉不允许的扩展名，但问题是它只是简单地删除了这些扩展名，而不是验证文件名是否合法。
    </p>
    <p>
     str_ireplace这个函数的工作模式是直接匹配这个字符串中的php字符串，像我们的pphphp中他把"php"匹配走了过后，还是仍然保留的是php所以就是此漏洞形成

     第十一关：（%00截断，此漏洞在5.2版本中）
    </h3>
    <pre><code>$img_path = $_GET['save_path']."/".rand(10, 99).date("YmdHis").".".$file_ext;</code></pre>
    <p>
     这关的主要一个漏洞来自这里这串代码，我们知道get传参是可以手动修改的，我们先来介绍一下%00截断的原理：
    </p>
    <p>
     0x00是字符串的结束标识符，攻击者可以利用手动添加字符串标识符的方式来将后面的内容进行截断，而后面的内容又可以帮助我们绕过检测。
    </p>
    <h4 id="%E6%AD%A3%E7%A1%AE%E7%94%A8%E6%B3%95" name="%E6%AD%A3%E7%A1%AE%E7%94%A8%E6%B3%95">
     正确用法
    </h4>
    <p>
     只有数据包中必须含有上传路径的情况下才可以用，比如本次测试中数据包中存在
     <code>
      save_path: ../uploads/
     </code>
     ，那么攻击者可以通过修改path的值来构造paylod，并且
     <code>
      PHP&lt;5.3.29，且magic_quotes_gpc关闭
     </code>
    </p>
    <p>
     <img alt="" height="189" src="https://i-blog.csdnimg.cn/direct/23b493c3f1fa450bbb23e68fb34aa1ad.png" width="852"/>
    </p>
    <h4 id="%E9%94%99%E8%AF%AF%E7%94%A8%E6%B3%95" name="%E9%94%99%E8%AF%AF%E7%94%A8%E6%B3%95">
     错误用法
    </h4>
    <p>
     只有文件名的情况下，是不能让%00截断生效的，因为只是文件名可控，即使通过00截断
     <code>
      test.php%00.txt
     </code>
     ，但是此时经过服务器处理后，文件名仍然为
     <code>
      test.php
     </code>
     ，不能通过对后缀的检测
    </p>
    <h4 id="%E6%80%9D%E8%B7%AF%EF%BC%9A" name="%E6%80%9D%E8%B7%AF%EF%BC%9A">
     思路：
    </h4>
    <p>
     使用
     <code>
      %00
     </code>
     可以绕过这种检测机制，这是因为 PHP 在处理字符串时会将
     <code>
      %00
     </code>
     解释为 NUL 字符，这可能导致某些函数提前终止字符串处理。所以当文件名为
     <code>
      test.php%00.jpg
     </code>
     时，PHP 会将
     <code>
      %00
     </code>
     解释为 NUL 字符，并在处理字符串时将其视为字符串的结束。因此
     <code>
      substr($_FILES['upload_file']['name'], strrpos($_FILES['upload_file']['name'], ".") + 1)
     </code>
     会提取到
     <code>
      .jpg
     </code>
     而不是
     <code>
      .php
     </code>
     。
    </p>

     第十二关（都是00截断，只不过一个是post，一个是get）：
    </h2>
    <p>
     post传参跟get传参的区别就在于编码的格式不同，一个是传到地址栏进行url编码，%00=/0；而post传参是hex，需要抓包，然后直接修改他的16进制编码：
    </p>
    <p>
     <img alt="" height="878" src="https://i-blog.csdnimg.cn/direct/9cc028290243419d8dfb9f4d78a6b9c0.png" width="1262"/>
    </p>
    <p>
     找20字段，因为我在上传之前在upload/后面打了个空格，URL编码是一种将字符集中的字符编码为仅包含字母、数字以及一些安全符号的方式。具体来说，空格在URL编码中被替换为%20，这里的20是空格字符在ASCII表中的16进制表示。
    </p>
