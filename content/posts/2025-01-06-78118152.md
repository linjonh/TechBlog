    <ul>
     <li>
      <p>
       <font color="#0099ff">
        <strong>
         1. 软件测试按项目流程分类
        </strong>
       </font>
      </p>
      <ul>
       <li>
        <p>
         一般来说，测试分为5个阶段：
         <strong>
          单元测试
         </strong>
         +
         <strong>
          集成测试
         </strong>
         +
         <strong>
          确认测试
         </strong>
         +
         <strong>
          系统测试
         </strong>
         +
         <strong>
          验收测试
         </strong>
         。在测试过程中如果有需要还要进行回归测试
        </p>
        <ul>
         <li>
          [1] 单元测试：单元就是最小的被测功能模块，
          <font color="#00dd">
           <strong>
            模块测试
           </strong>
          </font>
         </li>
         <li>
          [2] 集成测试：模块组装成完整的软件系统，排除
          <font color="#00dd">
           <strong>
            接口错误
           </strong>
          </font>
          等
         </li>
         <li>
          [3] 确认测试：
          <strong>
           软件
          </strong>
          的功能和性能是否如
          <font color="#00dd">
           <strong>
            用户所合理期待
           </strong>
          </font>
          的那样
         </li>
         <li>
          [4] 系统测试：将已确认的
          <strong>
           软件、硬件、网络等结合起来
          </strong>
          进行
          <font color="#00dd">
           <strong>
            组装和确认测试
           </strong>
          </font>
         </li>
         <li>
          [5] 验收测试：由用户或者独立的测试人员根据
          <strong>
           测试计划和结果
          </strong>
          进行
          <font color="#00dd">
           <strong>
            测试和验收
           </strong>
          </font>
         </li>
        </ul>
       </li>
       <li>
        <p>
         <font color="#ff00">
          <strong>
           单元测试
          </strong>
         </font>
         的测试对象，目的，测试依据，测试方法
        </p>
        <ul>
         <li>
          测试对象：模块内部的
          <font color="#ee88">
           <strong>
            程序
           </strong>
          </font>
          ，可能是一个C语言的函数或过程，C++的一个类等
         </li>
         <li>
          目的：消除模块的
          <font color="#ee88">
           <strong>
            逻辑
           </strong>
          </font>
          和
          <font color="#ee88">
           <strong>
            功能
           </strong>
          </font>
          缺陷
         </li>
         <li>
          测试依据：模块的详细设计（比如模块设计文档）
         </li>
         <li>
          测试方法：
          <font color="#00dd">
           <strong>
            白盒测试
           </strong>
          </font>
         </li>
        </ul>
       </li>
       <li>
        <p>
         <font color="#ff00">
          <strong>
           集成测试
          </strong>
         </font>
         的测试对象，目的，测试依据，测试方法
        </p>
        <ul>
         <li>
          测试对象：模块间的
          <font color="#ee88">
           <strong>
            组装
           </strong>
          </font>
          和
          <font color="#ee88">
           <strong>
            调用
           </strong>
          </font>
          关系
         </li>
         <li>
          目的：找出模块
          <font color="#ee88">
           <strong>
            调用
           </strong>
          </font>
          和模块
          <font color="#ee88">
           <strong>
            接口
           </strong>
          </font>
          方面的问题
         </li>
         <li>
          测试依据：概要设计（如模块间的通信要求等）
         </li>
         <li>
          测试方法：
          <font color="#00dd">
           <strong>
            灰盒测试
           </strong>
          </font>
         </li>
        </ul>
       </li>
       <li>
        <p>
         <font color="#ff00">
          <strong>
           系统测试
          </strong>
         </font>
         的测试对象，目的，测试依据，测试方法
        </p>
        <ul>
         <li>
          测试对象：
          <font color="#ee88">
           <strong>
            整个系统
           </strong>
          </font>
         </li>
         <li>
          目的：找出
          <font color="#ee88">
           <strong>
            系统与需求不符合
           </strong>
          </font>
          的地方
         </li>
         <li>
          测试依据：
          <font color="#ee88">
           <strong>
            需求规格说明书
           </strong>
          </font>
         </li>
         <li>
          测试方法：
          <font color="#00dd">
           <strong>
            黑盒测试
           </strong>
          </font>
         </li>
        </ul>
       </li>
       <li>
        <p>
         <font color="#ff00">
          <strong>
           验收测试
          </strong>
         </font>
         的分类
        </p>
        <ul>
         <li>
          [1] 非正式验收测试
          <br/>
          <ul>
           <li>
            α测试：
            <font color="#ee88">
             <strong>
              内测
             </strong>
            </font>
            。由用户、测试人员、开发人员共同参与的内部测试
           </li>
           <li>
            β测试：内测后的
            <font color="#ee88">
             <strong>
              公测
             </strong>
            </font>
            。即完全交给最终用户测试
           </li>
          </ul>
         </li>
         <li>
          [2] 正式验收测试
         </li>
        </ul>
       </li>
      </ul>
     </li>
    </ul>
    <p>
    </p>
    <center>
     <br/>
    </center>
    <p>
    </p>
    <ul>
     <li>
      <p>
       <font color="#0099ff">
        <strong>
         2. 按测试工作对代码的可见程度分类
        </strong>
       </font>
      </p>
      <p>
       上面讲的单元测试、集成测试、确认测试、系统测试和验收测试，是从
       <font color="#00aa">
        <strong>
         项目开发流程
        </strong>
       </font>
       来进行划分的，如果从
       <font color="#00aa">
        <strong>
         测试工作对代码的可见程度
        </strong>
       </font>
       来划分，可将软件测试划分为：
      </p>
      <ul>
       <li>
        <p>
         <font color="#00dd">
          <strong>
           [1] 黑盒测试
          </strong>
         </font>
        </p>
        <ul>
         <li>
          <p>
           黑盒测试，指的是把被测的软件看作是一个黑盒子，我们
           <strong>
            不去关心盒子里面的结构
           </strong>
           是什么样子的，只关心软件的输入数据和输出结果
          </p>
         </li>
         <li>
          <p>
           黑盒测试也称为
           <font color="#00eeff">
            <strong>
             功能测试
            </strong>
           </font>
           、
           <font color="#00eeff">
            <strong>
             数据驱动测试
            </strong>
           </font>
           、或
           <font color="#00eeff">
            <strong>
             基于规格说明书的测试
            </strong>
           </font>
           ，是一种从用户观点出发的测试
          </p>
         </li>
         <li>
          <p>
           主要检测的错误类型有：
          </p>
          <ul>
           <li>
            <strong>
             不正确或遗漏的功能
            </strong>
            +
            <strong>
             接口界面错误
            </strong>
            (能否正确输入正确输出) +
            <strong>
             性能错误
            </strong>
            (性能能否满足要求) +
            <strong>
             数据结构或外部数据访问错误
            </strong>
            +
            <strong>
             初始化或终止条件错误
            </strong>
           </li>
          </ul>
         </li>
         <li>
          <p>
           常用的黑盒测试方法：
          </p>
          <ul>
           <li>
            <strong>
             等价类划分法
            </strong>
            +
            <strong>
             边界值分析法
            </strong>
            +
            <strong>
             因果图法
            </strong>
            +
            <strong>
             场景法
            </strong>
            +
            <strong>
             正交实验设计法
            </strong>
            +
            <strong>
             判定表驱动分析法
            </strong>
            +
            <strong>
             错误推测法
            </strong>
            +
            <strong>
             功能图分析法
            </strong>
            。
           </li>
          </ul>
         </li>
        </ul>
       </li>
       <li>
        <p>
         <font color="#00dd">
          <strong>
           [2] 白盒测试
          </strong>
         </font>
        </p>
        <ul>
         <li>
          <p>
           白盒测试，指的是把盒子盖子打开，去研究
           <strong>
            里面的源代码和程序结果
           </strong>
           。按照程序内部的结构测试程序，通过测试来检测产品内部动作是否按照设计规格说明书的规定正常进行，检验程序中的每条通路是否都能按预定要求正确工作
          </p>
         </li>
         <li>
          <p>
           白盒测试也称为
           <font color="#00eeff">
            <strong>
             结构测试
            </strong>
           </font>
           、
           <font color="#00eeff">
            <strong>
             透明盒测试
            </strong>
           </font>
           、
           <font color="#00eeff">
            <strong>
             逻辑驱动测试
            </strong>
           </font>
           或
           <font color="#00eeff">
            <strong>
             基于代码的测试
            </strong>
           </font>
          </p>
         </li>
         <li>
          <p>
           白盒测试需要遵循的原则：
          </p>
          <ul>
           <li>
            (1)
            <strong>
             保证一个模块中的所有独立路径至少被测试一次
            </strong>
            ；
           </li>
           <li>
            (2)
            <strong>
             所有逻辑值均需要测试真（true）和假（false）两种情况
            </strong>
            ；
           </li>
           <li>
            (3)
            <strong>
             检查程序的内部数据结构，保证其结构的有效性
            </strong>
            ；
           </li>
           <li>
            (4)
            <strong>
             在上下边界及可操作范围内运行所有循环
            </strong>
            。
           </li>
          </ul>
         </li>
         <li>
          <p>
           白盒测试的主要方法有：
          </p>
          <ul>
           <li>
            <p>
             <strong>
              静态测试&amp;动态测试
             </strong>
             +
             <strong>
              单元测试
             </strong>
             +
             <strong>
              代码检查
             </strong>
             +
             <strong>
              同行评审
             </strong>
             +
             <strong>
              技术评审
             </strong>
            </p>
           </li>
           <li>
            <p>
             <strong>
              静态测试和动态测试
             </strong>
             ：静态测试是不用运行程序的测试，包括代码检查、静态结构分析、代码质量度量、文档测试等等，它可以由人工进行，充分发挥人的逻辑思维优势，也可以借助软件工具（Fxcop）自动进行。动态测试则需要执行代码，也是我们用的最多的一种测试，通过运行程序找到问题，包括功能确认与接口测试、覆盖率分析、性能分析、内存分析等。
            </p>
           </li>
          </ul>
         </li>
        </ul>
       </li>
       <li>
        <p>
         <font color="#00dd">
          <strong>
           [3] 灰盒测试
          </strong>
         </font>
        </p>
        <ul>
         <li>
          <p>
           灰盒测试更像是白盒测试和黑盒测试的
           <strong>
            混合测试
           </strong>
           。更多时候测试做的就是灰盒测试，既有白盒测试也有黑盒测试
          </p>
         </li>
         <li>
          <p>
           灰盒测试关注输出对于输入的正确性，同时也关注内部表现，但这种关注不象白盒那样详细、完整，只是通过一些表征性的现象、事件、标志来判断内部的运行状态。有时候输出是正确的，但内部其实已经错误了，这种情况非常多，如果每次都通过白盒测试来操作，效率会很低，因此需要采取灰盒的方法。
          </p>
         </li>
        </ul>
       </li>
      </ul>
     </li>
    </ul>
    <p>
    </p>
    <center>
     <br/>
    </center>
    <p>
    </p>
    <ul>
     <li>
      <p>
       <font color="#0099ff">
        <strong>
         3. 按使用的测试方法分类
        </strong>
       </font>
      </p>
      <ul>
       <li>
        <p>
         <font color="#00dd">
          <strong>
           [1] 手工测试
          </strong>
         </font>
        </p>
        <ul>
         <li>
          由人去执行测试用例，观察返回结果是否符合预期
         </li>
        </ul>
       </li>
       <li>
        <p>
         <font color="#00dd">
          <strong>
           [2] 自动化测试
          </strong>
         </font>
        </p>
        <ul>
         <li>
          <p>
           把人为驱动的测试行为转换为机器执行。
          </p>
         </li>
         <li>
          <p>
           自动化测试又可分为：
           <font color="#ee88">
            <strong>
             功能自动化测试
            </strong>
           </font>
           +
           <font color="#ee88">
            <strong>
             性能自动化测试
            </strong>
           </font>
          </p>
         </li>
         <li>
          <p>
           <strong>
            (1) 功能自动化测试
           </strong>
           ：一般所说的自动化测试就是指功能自动化测试。用一段程序来测试软件功能，重复执行进行重复测试，如果软件发生小部分改动，只需要改动小部分测试代码，就可继续对软件进行功能测试
          </p>
         </li>
         <li>
          <strong>
           (2) 性能自动化测试
          </strong>
          ：现在的性能测试工作都是通过
          <strong>
           性能测试工具
          </strong>
          辅助完成的。测试工具可以模拟成千上万的用户向系统发送请求，用来验证系统的处理能力。
         </li>
        </ul>
       </li>
      </ul>
     </li>
    </ul>
    <p>
    </p>
    <center>
     <br/>
    </center>
    <p>
    </p>
    <ul>
     <li>
      <p>
       <font color="#0099ff">
        <strong>
         4. 性能测试的分类
        </strong>
       </font>
      </p>
      <ul>
       <li>
        [1] 性能测试
       </li>
       <li>
        [2] 负载测试：在
        <strong>
         一定的工作负荷
        </strong>
        下，系统的负荷及响应时间
       </li>
       <li>
        [3] 容量测试：系统在其
        <strong>
         极限值状态
        </strong>
        下没有出现任何软件故障或还能保持主要功能正常运行
       </li>
       <li>
        [4] 压力测试：确定系统瓶颈的测试
       </li>
       <li>
        [5] 强度测试：对异常情况的抵抗能力
       </li>
      </ul>
      <p>
       压力和强度测试通常与性能测试结合进行。
      </p>
      <p>
       针对一个网站进行测试，模拟10到50个用户就是在进行常规
       <strong>
        性能测试
       </strong>
       ，用户增加到1000乃至上万就变成了
       <strong>
        压力/负载测试
       </strong>
       。如果同时对系统进行大量的数据查询操作，就包含了
       <strong>
        强度测试
       </strong>
       。
      </p>
     </li>
    </ul>
    <p>
    </p>
    <center>
     <br/>
    </center>
    <p>
    </p>
    <ul>
     <li>
      <p>
       <font color="#0099ff">
        <strong>
         5. 设计测试用例最重要的原则
        </strong>
       </font>
      </p>
      <ul>
       <li>
        <p>
         <strong>
          准确性
         </strong>
         ：准确对应需求来设计测试用例
        </p>
       </li>
       <li>
        <p>
         <font color="#00dd">
          <strong>
           完整性
          </strong>
         </font>
         ：完整覆盖应有的功能需求
        </p>
       </li>
       <li>
        <p>
         <font color="#00dd">
          <strong>
           可重复性
          </strong>
         </font>
         ：测试用例应该是可重复的，排除测试情形的特殊性
        </p>
       </li>
       <li>
        <p>
         <strong>
          独立性
         </strong>
         ：一个测试用例最好能独立运行，不依赖于其他测试用例的输出结果
        </p>
       </li>
      </ul>
      <p>
       除了上述原则，还有其他一些原则比如：
      </p>
      <ul>
       <li>
        经济性：测试没有冗余步骤
       </li>
       <li>
        可追踪：测试用例应该追溯到具体需求
       </li>
       <li>
        自我清理：测试结束后，恢复到原有干净的状态，不应该对原有系统造成影响
       </li>
       <li>
        结构化和可测试性：测试用例应该是结构化。一般可以根据一个横向维度，对测试用例进行
        <strong>
         功能模块
        </strong>
        的划分；同时纵向维度上可以根据
        <strong>
         测试类别
        </strong>
        对测试用例进行纵向结构的划分。测试同时应该是可测试性的。对于无法执行的测试用例是没有意义的。
       </li>
       <li>
        规范性：包含测试用例的说明-命名、环境、步骤、结果等必要说明
       </li>
       <li>
        简洁性：尽量简洁，精悍，执行时间不要过长，粒度不要太大（大型系统不太适用）
       </li>
       <li>
        有效性：测试用例是否符合商业用例
       </li>
      </ul>
     </li>
    </ul>
    <p>
    </p>
    <center>
     <br/>
    </center>
    <p>
    </p>
    <ul>
     <li>
      <p>
       <font color="#0099ff">
        <strong>
         6. 测试报告包括哪些内容
        </strong>
       </font>
      </p>
      <p>
       测试报告的通常阅读者：开发人员、测试经理、产品经理、项目负责人
       <br/>
       不同人员有不同的阅读需求：
      </p>
      <blockquote>
       <p>
        开发人员往往希望能从报告中了解缺陷的情况
        <br/>
        测试经理还关心用例的执行情况及覆盖率
        <br/>
        项目责任人则最关心还有多少问题、此次版本是否测试通过
       </p>
      </blockquote>
      <p>
       测试报告有：功能测试报告，性能测试报告，兼容性测试报告等
      </p>
      <p>
       <strong>
        功能测试报告
       </strong>
       根据内容的侧重点，分为
       <font color="00dd">
        <strong>
         版本测试报告
        </strong>
       </font>
       和
       <font color="#00dd">
        <strong>
         总结测试报告
        </strong>
       </font>
      </p>
      <ul>
       <li>
        <p>
         [1]
         <strong>
          版本测试报告
         </strong>
         的主要内容
        </p>
        <ul>
         <li>
          测试简介：测试目的，测试消耗的资源
         </li>
         <li>
          测试环境：软硬件环境，系统拓扑图等
         </li>
         <li>
          测试方法
         </li>
         <li>
          测试用例：用例分析，用例执行情况
         </li>
         <li>
          测试过程：缺陷统计，问题摘要
         </li>
         <li>
          测试结果：资源占用，测试结论(详细数据支持)
         </li>
         <li>
          备注：用例执行记录，资源监控记录等
         </li>
        </ul>
       </li>
       <li>
        <p>
         [2]
         <strong>
          总结测试报告
         </strong>
         的主要内容
        </p>
        <ul>
         <li>
          测试简介：条目同上
         </li>
         <li>
          测试环境：条目同上
         </li>
         <li>
          测试过程：各版本测试状况，各版本bug统计
         </li>
         <li>
          测试分析：缺陷分析，遗留问题
         </li>
         <li>
          测试小结：资源占用，风险分析等
         </li>
        </ul>
       </li>
      </ul>
     </li>
    </ul>
    <p>
    </p>
    <center>
     <br/>
    </center>
    <p>
    </p>
    <ul>
     <li>
      <font color="#0099ff">
       <strong>
        7. 对测试框架的了解
       </strong>
      </font>
     </li>
    </ul>
    <p>
     四种常用的测试框架：
     <strong>
      数据驱动测试框架
     </strong>
     +
     <strong>
      测试脚本模块化框架
     </strong>
     +
     <strong>
      测试库架构框架
     </strong>
     +
     <strong>
      关键字驱动测试框架
     </strong>
    </p>
    <ul>
     <li>
      <p>
       [1] 数据驱动测试框架：
      </p>
      <ul>
       <li>
        <p>
         所有测试框架中最简单的一种。
        </p>
       </li>
       <li>
        <p>
         数据驱动测试是测试从数据文件（数据池，ODBC源，cvs文件，Excel文件，DAO对象等）中读取输入和输出数值并载入到捕获的或手工编码的脚本中变量里的一种框架。将测试数据从测试脚本中分离出来。
        </p>
       </li>
       <li>
        <p>
         <strong>
          优点
         </strong>
         ：
        </p>
        <ul>
         <li>
          至少测试数据可以单独维护了
         </li>
         <li>
          可以快速增加相似测试，测试者增加新的测试不必掌握测试工具语言
         </li>
        </ul>
       </li>
       <li>
        <p>
         <strong>
          缺点
         </strong>
         ：
        </p>
        <ul>
         <li>
          维护成本非常高。任何测试程序的变更所导致的工作量是所有测试框架中最多的
         </li>
        </ul>
       </li>
      </ul>
     </li>
     <li>
      <p>
       [2] 测试脚本模块化框架
      </p>
      <ul>
       <li>
        <p>
         测试脚本模块化框架应用了抽象或封装的原则。
        </p>
       </li>
       <li>
        <p>
         测试脚本模块化框架需要创建能够代表测试下应用程序（application-under-test）的模块，零件（section）和函数的小的、独立的
         <strong>
          脚本
         </strong>
         。然后用一种
         <strong>
          分级
         </strong>
         的方式将这些小脚本组成更大的测试，实现一个特定的测试用例。
        </p>
       </li>
       <li>
        (1) 测试数据由测试工程师负责
       </li>
       <li>
        <p>
         (2) 测试脚本由自动化工程师负责维护，必须懂业务逻辑
        </p>
       </li>
       <li>
        <p>
         <strong>
          优点
         </strong>
         ：
        </p>
        <ul>
         <li>
          控件和业务逻辑一旦发生变化，要进行修改和维护的是底层的
          <strong>
           测试脚本
          </strong>
          （比无任何抽象封装的自动化测试程序稍好一些）
         </li>
        </ul>
       </li>
       <li>
        <p>
         <strong>
          缺点
         </strong>
         ：
        </p>
        <ul>
         <li>
          控件识别和业务逻辑本身属于不同的领域，没有很好进行抽象封装
         </li>
         <li>
          几乎所有大的变更引起的工作量都由自动化测试开发工程师完成
         </li>
        </ul>
       </li>
      </ul>
     </li>
     <li>
      <p>
       [3] 测试库架构框架
      </p>
      <ul>
       <li>
        <p>
         测试库构架框架和测试脚本模块化框架非常相似，有着同样的优势，但是它把测试下的应用程序分成过程和函数，而不是脚本。
        </p>
       </li>
       <li>
        <p>
         这种框架要求创建代表测试下应用程序模块，零件和函数的库文件（SQABasic libraries, APIs, DLLs等等）。然后这些库文件被测试用例脚本直接调用。
        </p>
       </li>
       <li>
        (1) 测试数据由测试工程师负责
       </li>
       <li>
        (2) 测试脚本由自动化工程师负责，必须懂业务逻辑
       </li>
       <li>
        <p>
         (3) 测试库由自动化工程师负责，无须懂业务，负责控件的维护
        </p>
       </li>
       <li>
        <p>
         <strong>
          优点
         </strong>
         ：
        </p>
        <ul>
         <li>
          完成了
          <strong>
           控件识别操作
          </strong>
          和
          <strong>
           业务逻辑
          </strong>
          的抽象分离
         </li>
         <li>
          被测试系统无论是哪层发生变化，只需要相应的人员进行变更维护即可
         </li>
        </ul>
       </li>
       <li>
        <strong>
         缺点
        </strong>
        ：
        <br/>
        <ul>
         <li>
          变更引起的工作量还是附加在自动化测试开发工程师身上
         </li>
        </ul>
       </li>
      </ul>
     </li>
     <li>
      <p>
       [4] 关键字驱动测试框架/表格驱动测试
      </p>
      <ul>
       <li>
        <p>
         关键字驱动和表格驱动测试是一种独立于应用程序的自动化框架，它们是可以互相替换的术语。这种框架要求开发于用来运行的自动化工具，驱动测试下应用程序和数据的测试脚本代码相独立的数据表和关键字。关键字驱动测试看上去非常象手工测试。在关键字测试里，应用程序的功能特性被写在表格和每个测试的详细指引里了。
        </p>
       </li>
       <li>
        <p>
         控件和业务逻辑以关键字的形式在EXCEL里面进行调用，普通的测试工程师不需要了解
         <strong>
          框架
         </strong>
         和
         <strong>
          工具
         </strong>
         的知识就能维护好控件和业务逻辑
        </p>
       </li>
       <li>
        <p>
         <strong>
          优点
         </strong>
         ：
        </p>
        <ul>
         <li>
          极大的减少了自动化开发工程师维护量，毕竟在测试团队中，自动化开发工程师占的比较少
         </li>
         <li>
          普通测试工程师，可以很好的维护自身负责的模块中涉及的测试case和测试数据
         </li>
        </ul>
       </li>
       <li>
        <strong>
         缺点
        </strong>
        ：
        <br/>
        <ul>
         <li>
          框架的抽象程度比较高，对自动化测试工程师的开发能力比较高
         </li>
        </ul>
       </li>
      </ul>
     </li>
    </ul>
    <p>
     以上测试框架图可参考博客：
     <a href="http://www.cnblogs.com/x10322/p/6024618.html" rel="nofollow noopener noreferrer" target="_blank">
      http://www.cnblogs.com/x10322/p/6024618.html
     </a>
    </p>
    <p>
     除了上述四种测试框架，可能还有
     <strong>
      混合的测试自动化框架
     </strong>
    </p>
    <ul>
     <li>
      [5] 混合的测试自动化框架
      <br/>
      <ul>
       <li>
        最常见的已实现的框架是上述技术的组合，抽取它们的优点，剔除其弱点。这种混合的测试自动化框架是发展时间较长且应用项目最多的框架
       </li>
      </ul>
     </li>
    </ul>
