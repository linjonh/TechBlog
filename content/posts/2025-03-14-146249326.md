    <p id="ue151451a">
     在ArkUI中，UI显示的内容均为组件，由框架直接提供的称为系统组件，由开发者定义的称为自定义组件。在进行 UI 界面开发时，通常不是简单的将系统组件进行组合使用，而是需要考虑代码可复用性、业务逻辑与UI分离，后续版本演进等因素。因此，将UI和部分业务逻辑封装成自定义组件是不可或缺的能力。
    </p>
    <p id="u33758d47">
     自定义组件具有以下特点：
    </p>
    <ul>
     <li id="ua2337b65">
      可组合：允许开发者组合使用系统组件、及其属性和方法。
     </li>
     <li id="u1027f89e">
      可重用：自定义组件可以被其他组件重用，并作为不同的实例在不同的父组件或容器中使用。
     </li>
     <li id="uf6f70a83">
      数据驱动UI更新：通过状态变量的改变，来驱动UI的刷新。
     </li>
    </ul>
    <h3 id="自定义组件的基本用法" name="自定义组件的基本用法">
     自定义组件的基本用法
    </h3>
    <p id="ud1ee4843">
     以下示例展示了自定义组件的基本用法。
    </p>
    <pre id="NmroA"><code>@Component
struct HelloComponent {
  @State message: string = 'Hello, World!';


  build() {
    // HelloComponent自定义组件组合系统组件Row和Text
    Row() {
      Text(this.message)
        .onClick(() =&gt; {
          // 状态变量message的改变驱动UI刷新，UI从'Hello, World!'刷新为'Hello, ArkUI!'
          this.message = 'Hello, ArkUI!';
        })
    }
  }
}</code></pre>
    <p id="u5e237014">
     <strong>
      注意
     </strong>
    </p>
    <p id="u64838441">
     如果在另外的文件中引用该自定义组件，需要使用export关键字导出，并在使用的页面import该自定义组件。
    </p>
    <p id="u2bd6fe55">
     HelloComponent可以在其他自定义组件中的build()函数中多次创建，实现自定义组件的重用。
    </p>
    <pre id="mHkSw"><code>class HelloComponentParam {
  message: string = ""
}


@Entry
@Component
struct ParentComponent {
  param: HelloComponentParam = {
    message: 'Hello, World!'
  }


  build() {
    Column() {
      Text('ArkUI message')
      HelloComponent(this.param);
      Divider()
      HelloComponent(this.param);
    }
  }
}</code></pre>

    <h3 id="自定义组件的基本结构" name="自定义组件的基本结构">

    </h3>
    <p id="ud5b914c6">
     struct：自定义组件基于struct实现，struct + 自定义组件名 + {...}的组合构成自定义组件，不能有继承关系。对于struct的实例化，可以省略new。

      <strong>
       说明
      </strong>
     </li>

    </p>
    <p id="u6564f25b">
     @Component：@Component装饰器仅能装饰struct关键字声明的数据结构。struct被@Component装饰后具备组件化的能力，需要实现build方法描述UI，一个struct只能被一个@Component装饰。@Component可以接受一个可选的bool类型参数。

    <h3 id="成员函数变量" name="成员函数变量">
     成员函数/变量
    </h3>
    <p id="u8c9289ee">
     自定义组件除了必须要实现build()函数外，还可以实现其他成员函数，成员函数具有以下约束：
    </p>
    <ul>
     <li id="u898ecb8b">
      自定义组件的成员函数为私有的，且不建议声明成静态函数。
     </li>
    </ul>
    <p id="uf0879172">
     自定义组件可以包含成员变量，成员变量具有以下约束：
    </p>
    <ul>
     <li id="u1e2d3a8c">
      自定义组件的成员变量为私有的，且不建议声明成静态变量。
     </li>
     <li id="u46a0edfd">
      自定义组件的成员变量本地初始化有些是可选的，有些是必选的。具体是否需要本地初始化，是否需要从父组件通过参数传递初始化子组件的成员变量，请参考
      <a href="https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V5/arkts-state-management-overview-V5" rel="nofollow" title="状态管理">
       状态管理
      </a>
      。
     </li>
    </ul>
    <h3 id="自定义组件的参数规定" name="自定义组件的参数规定">
     自定义组件的参数规定
    </h3>
    <p id="u24223245">
     从上文的示例中，我们已经了解到，可以在build方法里创建自定义组件，在创建自定义组件的过程中，根据装饰器的规则来初始化自定义组件的参数。
    </p>
    <pre id="fVyer"><code>@Component
struct MyComponent {
  private countDownFrom: number = 0;
  private color: Color = Color.Blue;


  build() {
  }
}


@Entry
@Component
struct ParentComponent {
  private someColor: Color = Color.Pink;


  build() {
    Column() {
      // 创建MyComponent实例，并将创建MyComponent成员变量countDownFrom初始化为10，将成员变量color初始化为this.someColor
      MyComponent({ countDownFrom: 10, color: this.someColor })
    }
  }
}</code></pre>
    <p id="u2c08a1d8">
     下面的示例代码将父组件中的函数传递给子组件，并在子组件中调用。
    </p>
    <pre id="pwMXF"><code>@Entry
@Component
struct Parent {
  @State cnt: number = 0
  submit: () =&gt; void = () =&gt; {
    this.cnt++;
  }


  build() {
    Column() {
      Text(`${this.cnt}`)
      Son({ submitArrow: this.submit })
    }
  }
}


@Component
struct Son {
  submitArrow?: () =&gt; void


  build() {
    Row() {
      Button('add')
        .width(80)
        .onClick(() =&gt; {
          if (this.submitArrow) {
            this.submitArrow()
          }
        })
    }
    .justifyContent(FlexAlign.SpaceBetween)
    .height(56)
  }
}</code></pre>
    <h3 id="build函数" name="build函数">
     build()函数
    </h3>
    <p id="u6b569553">
     所有声明在build()函数的语句，我们统称为UI描述，UI描述需要遵循以下规则：
    </p>
    <ul>
     <li id="udb07137a">
      @Entry装饰的自定义组件，其build()函数下的根节点唯一且必要，且必须为容器组件，其中ForEach禁止作为根节点。
     </li>
    </ul>
    <p id="u4c5582ad">
     @Component装饰的自定义组件，其build()函数下的根节点唯一且必要，可以为非容器组件，其中ForEach禁止作为根节点。
    </p>
    <pre id="OYwVM"><code>@Entry
@Component
struct MyComponent {
  build() {
    // 根节点唯一且必要，必须为容器组件
    Row() {
      ChildComponent() 
    }
  }
}


@Component
struct ChildComponent {
  build() {
    // 根节点唯一且必要，可为非容器组件
    Image('test.jpg')
  }
}</code></pre>
    <ul>
     <li id="u3b65198a">
      不允许声明本地变量，反例如下。
     </li>
    </ul>
    <pre id="Q4ERz"><code>build() {
  // 反例：不允许声明本地变量
  let a: number = 1;
}</code></pre>
    <ul>
     <li id="ue802c4e0">
      不允许在UI描述里直接使用console.info，但允许在方法或者函数里使用，反例如下。
     </li>
    </ul>
    <pre id="cZ2a2"><code>build() {
  // 反例：不允许console.info
  console.info('print debug log');
}</code></pre>
    <ul>
     <li id="u4ed27b4b">
      不允许创建本地的作用域，反例如下。
     </li>
    </ul>
    <pre id="Tl1IV"><code>build() {
  // 反例：不允许本地作用域
  {
    ...
  }
}</code></pre>
    <ul>
     <li id="u80aa1311">
      不允许调用没有用@Builder装饰的方法，允许系统组件的参数是TS方法的返回值。
     </li>
    </ul>
    <pre id="y1pnW"><code>@Component
struct ParentComponent {
  doSomeCalculations() {
  }


  calcTextValue(): string {
    return 'Hello World';
  }


  @Builder doSomeRender() {
    Text(`Hello World`)
  }


  build() {
    Column() {
      // 反例：不能调用没有用@Builder装饰的方法
      this.doSomeCalculations();
      // 正例：可以调用
      this.doSomeRender();
      // 正例：参数可以为调用TS方法的返回值
      Text(this.calcTextValue())
    }
  }
}</code></pre>
    <ul>
     <li id="u7d0ba2db">
      不允许使用switch语法，如果需要使用条件判断，请使用if。示例如下。
     </li>
    </ul>
    <pre id="gKY2n"><code>build() {
  Column() {
    // 反例：不允许使用switch语法
    switch (expression) {
      case 1:
        Text('...')
        break;
      case 2:
        Image('...')
        break;
      default:
        Text('...')
        break;
    }
    // 正例：使用if
    if(expression == 1) {
      Text('...')
    } else if(expression == 2) {
      Image('...')
    } else {
      Text('...')
    }
  }
}</code></pre>
    <ul>
     <li id="u8e8ba00b">
      不允许使用表达式，反例如下。
     </li>
    </ul>
    <pre id="SHXD8"><code>build() {
  Column() {
    // 反例：不允许使用表达式
    (this.aVar &gt; 10) ? Text('...') : Image('...')
  }
}</code></pre>
    <ul>
     <li id="u02a70327">
      不允许直接改变状态变量，反例如下。详细分析见
      <a href="https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V5/arkts-state-V5#%E4%B8%8D%E5%85%81%E8%AE%B8%E5%9C%A8build%E9%87%8C%E6%94%B9%E7%8A%B6%E6%80%81%E5%8F%98%E9%87%8F" rel="nofollow" title="@State常见问题：不允许在build里改状态变量">
       @State常见问题：不允许在build里改状态变量
      </a>
     </li>
    </ul>
    <pre id="k9jeI"><code>@Component
struct CompA {
  @State col1: Color = Color.Yellow;
  @State col2: Color = Color.Green;
  @State count: number = 1;
  build() {
    Column() {
      // 应避免直接在Text组件内改变count的值
      Text(`${this.count++}`)
        .width(50)
        .height(50)
        .fontColor(this.col1)
        .onClick(() =&gt; {
          this.col2 = Color.Red;
        })
      Button("change col1").onClick(() =&gt;{
        this.col1 = Color.Pink;
      })
    }
    .backgroundColor(this.col2)
  }
}</code></pre>
    <p id="ubfbf96ae">
     在ArkUI状态管理中，状态驱动UI更新。
    </p>
    <p id="uacef0fa5">
    </p>
    <p class="img-center">
     <img alt="" height="89" id="u05808364" src="https://i-blog.csdnimg.cn/img_convert/7a8014177bb318ec9b78e0f9dd3a2b52.png" width="265"/>
    </p>
    <p id="u2c931c68">
     所以，不能在自定义组件的build()或@Builder方法里直接改变状态变量，这可能会造成循环渲染的风险。Text('${this.count++}')在全量更新或最小化更新会产生不同的影响：
    </p>
    <ul>
     <li>
      <ul>
       <li id="u66971208">
        全量更新（API8及以前版本）： ArkUI可能会陷入一个无限的重渲染的循环里，因为Text组件的每一次渲染都会改变应用的状态，就会再引起下一轮渲染的开启。 当 this.col2 更改时，都会执行整个build构建函数，因此，Text(${this.count++})绑定的文本也会更改，每次重新渲染Text(${this.count++})，又会使this.count状态变量更新，导致新一轮的build执行，从而陷入无限循环。
       </li>
       <li id="u19fb6c41">
        最小化更新（API9-至今版本）： 当 this.col2 更改时，只有Column组件会更新，Text组件不会更改。 只当 this.col1 更改时，会去更新整个Text组件，其所有属性函数都会执行，所以会看到Text(${this.count++})自增。因为目前UI以组件为单位进行更新，如果组件上某一个属性发生改变，会更新整体的组件。所以整体的更新链路是：this.col1 = Color.Pink -&gt; Text组件整体更新-&gt;this.count++ -&gt;Text组件整体更新。值得注意的是，这种写法在初次渲染时会导致Text组件渲染两次，从而对性能产生影响。
       </li>
      </ul>
     </li>
    </ul>
    <p id="u864bbe92">
     build函数中更改应用状态的行为可能会比上面的示例更加隐蔽，比如：
    </p>
    <ul>
     <li>
      <ul>
       <li id="u627ce92a">
        在@Builder，@Extend或@Styles方法内改变状态变量 。
       </li>
       <li id="ue68dbde9">
        在计算参数时调用函数中改变应用状态变量，例如 Text('${this.calcLabel()}')。
       </li>
       <li id="u77f27877">
        对当前数组做出修改，sort()改变了数组this.arr，随后的filter方法会返回一个新的数组。
       </li>
      </ul>
     </li>
    </ul>
    <pre id="vkY8j"><code>// 反例
@State arr : Array&lt;...&gt; = [ ... ];
ForEach(this.arr.sort().filter(...), 
  item =&gt; { 
  ...
})
// 正确的执行方式为：filter返回一个新数组，后面的sort方法才不会改变原数组this.arr
ForEach(this.arr.filter(...).sort(), 
  item =&gt; { 
  ...
})</code></pre>
