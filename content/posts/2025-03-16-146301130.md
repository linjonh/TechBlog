    <h4>
     <a id="1_0">
     </a>
     1、‌状态逻辑复用困难
    </h4>
    <p>
     在Hooks出现之前，React组件间的状态逻辑复用主要依赖
     <code>
      高阶组件（HOC）
     </code>
     和
     <code>
      render props
     </code>
     。
     <br/>
     <code>
      Hooks
     </code>
     通过允许创建自定义Hook，使得状态逻辑的复用变得简单而直接‌
    </p>
    <blockquote>
     <p>
      缺点
     </p>
    </blockquote>
    <ul>
     <li>
      HOC 可能导致 props 命名冲突
     </li>
     <li>
      render props 则可能导致组件树嵌套过深，使得代码难以阅读和维护
     </li>
    </ul>
    <blockquote>
     <p>
      <strong>
       HOC 以组件作为参数并返回一个新组件的函数
      </strong>
     </p>
    </blockquote>

    <h4>
     <a id="2_32">
     </a>
     2、复杂组件难以理解
    </h4>
    <p>
     类组件中，特别是当组件包含多个生命周期方法时，代码逻辑可能会分散在各个生命周期方法中，导致同一块功能的代码被拆分到了不同的地方
    </p>
    <h4>
     <a id="3_36">
     </a>
     3、‌副作用的管理‌
    </h4>
    <ul>
     <li>
      <p>
       在类组件中，副作用（如数据获取、订阅事件等）通常需要在生命周期方法中进行管理
      </p>
     </li>
     <li>
      <p>
       Hooks 提供了
       <code>
        useEffect
       </code>
       等API
      </p>
     </li>
    </ul>
    <h4>
     <a id="4_42">
     </a>
     4、性能优化
    </h4>
    <ul>
     <li>
      类组件中，为了避免不必要的渲染，需要使用
      <code>
       shouldComponentUpdate
      </code>
      或
      <code>
       PureComponent
      </code>
     </li>
     <li>
      Hooks中，memo 来优化函数组件的渲染性能，避免不必要的重新渲染。
     </li>
    </ul>
    <h4>
     <a id="_48">
     </a>
     拓展
    </h4>
    <p>
     hooks 本质是闭包
    </p>
    <p>
     Hooks 也保证了它们的
     <code>
      状态
     </code>
     能在多次渲染中保持不变。每次组件函数调用时，React 都会为 Hooks 创建一个新的闭包环境来实现的。
    </p>
