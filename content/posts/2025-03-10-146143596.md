    <p>
     适配器模式是一种结构型设计模式，用于将不兼容的接口转换为客户端期望的接口，使原本因接口不匹配而无法协同工作的类能够一起工作。类似于电源适配器将220V电压转换为5V供手机充电。
    </p>
    <h4>
     结构
    </h4>
    <h5>
     <strong>
      1.角色划分：
     </strong>
    </h5>
    <p>
     ◦
     <strong>
      目标接口（Target）
     </strong>
     ：客户端期望调用的接口。
     <br/>
     ◦
     <strong>
      被适配者（Adaptee）
     </strong>
     ：已存在的、需要被适配的类（接口不兼容）。
     <br/>
     ◦
     <strong>
      适配器（Adapter）
     </strong>
     ：通过继承或组合方式，将Adaptee转换为Target接口。
    </p>
    <h5>
     <strong>
      2. 实现方式：
     </strong>
    </h5>
    <p>
     ◦
     <strong>
      类适配器：
     </strong>
     通过继承Adaptee并实现Target接口（需支持多继承，Java中不适用）。
     <br/>
     ◦
     <strong>
      对象适配器：
     </strong>
     通过组合Adaptee对象并实现Target接口（推荐方式）。
    </p>
    <h4>
     实现方式
    </h4>
    <p>
     <strong>
      场景描述：
     </strong>
    </p>
    <p>
     假设现有旧版文件读取器LegacyFileReader，其接口为readFile(String path)，而客户端期望调用统一的文件处理器接口FileProcessor的process(String path)方法。通过适配器将二者对接。

    <h4>
     使用场景
    </h4>
    <p>
     1.
     <strong>
      整合旧系统
     </strong>
     ：将遗留代码或第三方库接口转换为新系统的接口。
     <br/>
     2.
     <strong>
      统一接口规范
     </strong>
     ：多个类功能相似但接口不同，需统一调用方式。
     <br/>
     3.
     <strong>
      兼容不同版本
     </strong>
     ：如系统升级时保留旧接口的适配层。
     <br/>
     4.
     <strong>
      框架扩展
     </strong>
     ：适配不同数据源（如数据库驱动、文件格式）。
    </p>
    <h4>
     适配器模式 vs. 其他模式
     <img alt="" height="271" src="https://i-blog.csdnimg.cn/direct/9a107fe239db481bb0b5addad821454f.png" width="1002"/>
    </h4>
    <h4>
     实际应用案例
    </h4>
    <p>
     <strong>
      1、Java I/O流：
     </strong>
     <br/>
     ◦ InputStreamReader 和 OutputStreamWriter 是适配器，将字节流适配为字符流。
     <img alt="" height="70" src="https://i-blog.csdnimg.cn/direct/789bfa990fe145edaf211d5678a5e422.png" width="904"/>
    </p>
    <p>
     <strong>
      2、集合框架：
     </strong>
     <br/>
     ◦ Collections.list(Enumeration) 将旧版Enumeration适配为List。
    </p>
    <p>
     <strong>
      3、Spring MVC：
     </strong>
     <br/>
     ◦ HandlerAdapter 将不同类型的Controller（如Servlet、POJO）适配为统一的处理接口。
    </p>
    <h4>
     优缺点
    </h4>
    <p>
     •
     <strong>
      优点
     </strong>
     ：
     <br/>
     ◦
     <strong>
      解耦客户端与被适配者
     </strong>
     ：客户端仅依赖目标接口。
     <br/>
     ◦
     <strong>
      复用现有类
     </strong>
     ：无需修改原有代码即可整合功能。
    </p>
    <p>
     •
     <strong>
      缺点
     </strong>
     ：
     <br/>
     ◦
     <strong>
      增加代码复杂度
     </strong>
     ：过多适配器会降低代码可读性。
     <br/>
     ◦
     <strong>
      可能掩盖设计缺陷
     </strong>
     ：过度使用可能延迟系统重构。
    </p>
    <h4>
     总结
    </h4>
    <p>
     适配器模式是解决接口不兼容问题的利器，尤其在以下场景中效果显著：
    </p>
    <p>
     • 系统需要集成外部组件或遗留代码。
     <br/>
     • 需要统一多个类的接口调用方式。
     <br/>
     • 希望在不修改原有代码的前提下扩展功能。
    </p>
    <p>
     通过合理使用适配器模式，可以提升系统的灵活性和可维护性，降低模块间的耦合度。
    </p>
