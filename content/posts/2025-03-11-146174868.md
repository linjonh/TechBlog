    <h2>
     <a id="_2">
     </a>
     缓存功能
    </h2>
    <h3>
     <a id="_4">
     </a>
     实现思路
    </h3>
    <p>
     <img alt="image.png" src="https://i-blog.csdnimg.cn/img_convert/3ec149f295cdc72c74d303d3b022eab1.png"/>
    </p>
    <p>
     <strong>
      整体的思路：
     </strong>
     <br/>
     应用服务器访问数据的时候，先查询
     <code>
      Redis
     </code>
    </p>
    <ul>
     <li>
      如果
      <code>
       Redis
      </code>
      上数据存在了，就直接从
      <code>
       Redis
      </code>
      读取数据交给应用服务器，不继续访问数据库了
     </li>
     <li>
      如果
      <code>
       Redis
      </code>
      上数据不存在，再读取
      <code>
       MySQL
      </code>
      ，把读到的结果，返回给应用服务器。同时，把这个数据也写入到
      <code>
       Redis
      </code>
      中
     </li>
    </ul>
    <p>
     <code>
      Redis
     </code>
     这样的缓存，经常用来存储“
     <strong>
      热点
     </strong>
     ”数据（高频被使用的数据）
    </p>
    <blockquote>
     <p>
      刚才上述描述的过程，相当于是把最近使用到的数据作为热点数据
     </p>
     <ul>
      <li>
       暗含了一层假设：某个数据一旦被用到了，那么很可能在最近这段时间就会被反复用到
      </li>
     </ul>
    </blockquote>
    <h3>
     <a id="_19">
     </a>
     存在的问题
    </h3>
    <p>
     上述策略，存在一个明显的问题：
     <br/>
     随着时间的推移，肯定是会有越来越多的
     <code>
      key
     </code>
     在
     <code>
      Redis
     </code>
     上访问不到，从而从
     <code>
      MySQL
     </code>
     读取并写入
     <code>
      Redis
     </code>
     中，此时
     <code>
      Redis
     </code>
     里面的数据不就越来越多了吗？
    </p>
    <ol>
     <li>
      在把数据写给
      <code>
       Redis
      </code>
      的时候，给这个
      <code>
       key
      </code>
      设置一个过期时间
     </li>
     <li>
      <code>
       Redis
      </code>
      也在内存不足的时候，提供了“
      <strong>
       淘汰策略
      </strong>
      ”
     </li>
    </ol>
    <h3>
     <a id="_28">
     </a>
     伪代码实现
    </h3>
    <ol>
     <li>
      假设业务是根据用户
      <code>
       uid
      </code>
      获取用户信息
     </li>
    </ol>
    <pre><code class="prism language-java"><span class="token class-name">UserInfo</span> <span class="token function">getUserInfo</span><span class="token punctuation">(</span><span class="token keyword">long</span> uid<span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
	<span class="token punctuation">.</span><span class="token punctuation">.</span><span class="token punctuation">.</span>

    <blockquote>
     <p>
      <code>
       uid
      </code>
      还需要拼接一些前缀：
      <code>
       user:info
      </code>
     </p>
     <ul>
      <li>
       因为 Redis 里面存的信息有很多种，不仅仅只有用户信息
      </li>
      <li>
       所以为了后续进行区分，例如
       <code>
        grade:info
       </code>
       ，
       <code>
        admin:info
       </code>
       … 需要拼接一个前缀
      </li>
     </ul>
    </blockquote>
    <ol start="2">
     <li>
      首先从
      <code>
       Redis
      </code>
      获取用户信息，我们假设用户信息保存在“
      <code>
       user:info&lt;uid&gt;
      </code>
      ”
     </li>
    </ol>

    <h2>
     <a id="_88">
     </a>
     记数功能
    </h2>
    <p>
     许多应⽤都会使⽤
     <code>
      Redis
     </code>
     作为计数的基础⼯具，它可以实现快速计数、查询缓存的功能，同时数据可以异步处理或者落地到其他数据源。
    </p>
    <p>
     如图下图所⽰，例如视频⽹站的视频播放次数可以使⽤
     <code>
      Redis
     </code>
     来完成：⽤⼾每播放⼀次视频，相应的视频播放数就会
     <strong>
      ⾃增 1
     </strong>
     。
    </p>
    <h3>
     <a id="_93">
     </a>
     实现思路
    </h3>
    <p>
     <img alt="image.png|435" src="https://i-blog.csdnimg.cn/img_convert/3d9ccc18c72bbe34958c9abe9526968a.png"/>
    </p>
    <blockquote>
     <p>
      企业为什么老乐意收集用户的数据？
     </p>
     <ul>
      <li>
       因为统计可以进一步明确用户拒的需求，然后根据需求改进和迭代产品
      </li>
     </ul>
    </blockquote>
    <h3>
     <a id="_100">
     </a>
     统计
    </h3>
    <p>
     <code>
      Redis
     </code>
     可以计数，但是不擅长统计
    </p>
    <p>
     比如，想在上述的
     <code>
      Redis
     </code>
     中，统计播放量前
     <code>
      100
     </code>
     的视频有哪些
    </p>
    <ul>
     <li>
      基于
      <code>
       Redis
      </code>
      搞，就很麻烦
     </li>
     <li>
      相比之下，如果是
      <code>
       MySQL
      </code>
      来存储上述数据，一个
      <code>
       SQL
      </code>
      就搞定了
      <br/>
      所以在 Redis 计数之后，还需要将播放量同步到“
      <strong>
       统计数据仓库
      </strong>
      ”中
     </li>
    </ul>
    <p>
     异步的方式：这里写入统计数据仓库（MySQL/HDFS…）的步骤是
     <strong>
      异步
     </strong>
     的
    </p>
    <ul>
     <li>
      不是说，来一个播放请求，这里就必须立马写一个数据
     </li>
     <li>
      不要求两边是同时完成的，只要最后都完成了就行
     </li>
    </ul>
