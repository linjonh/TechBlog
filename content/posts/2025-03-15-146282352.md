    <h2>
     <a id="Kubernetes__0">
     </a>
     Kubernetes 调度机制：亲和性、污点与容忍全解析
    </h2>
    <p>
     在 Kubernetes 的调度机制中，有三个重要概念：
     <strong>
      节点亲和性
     </strong>
     、
     <strong>
      污点
     </strong>
     和 **容忍 **。它们是 Kubernetes 调度的核心，决定了 Pod 是如何被调度到具体节点的。理解这三者之间的关系，可以帮助我们更精确地控制集群中的 Pod 调度。
    </p>
    <h3>
     <a id="1__4">
     </a>
     1. 节点亲和性
    </h3>
    <h4>
     <a id="_6">
     </a>
     什么是节点亲和性？
    </h4>
    <p>
     简单来说，节点亲和性就是让
     <strong>
      Pod 主动选择节点
     </strong>
     的机制。通过设置亲和性规则，你可以让 Pod 更倾向于调度到某些符合条件的节点。
    </p>
    <h4>
     <a id="_10">
     </a>
     适用场景：
    </h4>
    <ul>
     <li>
      需要特定硬件支持的任务，比如数据库 Pod 只想运行在带 SSD 存储的节点上。
     </li>
     <li>
      性能要求高的业务，比如某些关键业务 Pod 只想跑在高性能节点上。
     </li>
    </ul>
    <h4>
     <a id="_15">
     </a>
     示例：
    </h4>
    <p>
     假设你想把某个业务 Pod 只调度到类型为 “high-performance” 的节点上，可以这么做：
    </p>
    <pre><code>nodeAffinity:
  requiredDuringSchedulingIgnoredDuringExecution:
    nodeSelectorTerms:
    - matchExpressions:
      - key: "node-type"
        operator: In
        values:
        - high-performance
</code></pre>
    <hr/>
    <h3>
     <a id="2__Taint_32">
     </a>
     2. 污点 (Taint)
    </h3>
    <h4>
     <a id="_34">
     </a>
     什么是污点？
    </h4>
    <p>
     污点是节点主动设置的条件，表示
     <strong>
      节点不愿意接受 Pod
     </strong>
     。你可以给节点加上污点，防止其他 Pod 被调度到这个节点上。
    </p>
    <h4>
     <a id="_38">
     </a>
     适用场景：
    </h4>
    <ul>
     <li>
      GPU 节点通常只给 AI 训练任务使用，其他普通 Pod 需要被拒绝调度到这些节点。
     </li>
     <li>
      节点出现故障时，可以通过污点避免新 Pod 调度到这个节点。
     </li>
    </ul>
    <h4>
     <a id="_43">
     </a>
     示例：
    </h4>
    <p>
     在 GPU 节点上设置污点，防止普通 Pod 调度到该节点：
    </p>
    <pre><code>kubectl taint nodes gpu-node type=gpu:NoSchedule
</code></pre>
    <hr/>
    <h3>
     <a id="3__Toleration_53">
     </a>
     3. 容忍 (Toleration)
    </h3>
    <h4>
     <a id="_55">
     </a>
     什么是容忍？
    </h4>
    <p>
     容忍就是
     <strong>
      Pod 声明“我能忍受污点”
     </strong>
     的机制。被污点标记的节点，不是所有的 Pod 都会被拒绝，只有那些声明了“容忍”这个污点的 Pod，才能顺利调度到这些节点上。
    </p>
    <h4>
     <a id="_59">
     </a>
     适用场景：
    </h4>
    <ul>
     <li>
      如果某个 Pod 想调度到 GPU 节点，必须声明容忍 GPU 的污点。
     </li>
     <li>
      即便是有故障的节点，只要某个 Pod 有容忍该节点的污点，它仍然可以在该节点运行。
     </li>
    </ul>
    <h4>
     <a id="_64">
     </a>
     示例：
    </h4>
    <p>
     如果 Pod 要调度到被污点标记的 GPU 节点，它需要添加以下容忍设置：
    </p>
    <pre><code>tolerations:
  - key: "type"
    operator: "Equal"
    value: "gpu"
    effect: "NoSchedule"
</code></pre>

    <h3>
     <a id="_88">
     </a>
     总结
    </h3>
    <p>
     Kubernetes 的调度机制可以用
     <strong>
      “亲和性 + 污点 + 容忍”
     </strong>
     三个词来概括：
    </p>
    <ul>
     <li>
      <strong>
       亲和性
      </strong>
      ：Pod 主动选择节点
     </li>
     <li>
      <strong>
       污点
      </strong>
      ：节点主动拒绝 Pod
     </li>
     <li>
      <strong>
       容忍
      </strong>
      ：Pod 声明“我能忍受这个污点”
     </li>
    </ul>
    <p>
     这三者结合起来，就能灵活地控制 Pod 的调度方式，解决了 Kubernetes 中 Pod 调度的“正向匹配 + 反向隔离”问题。
    </p>
