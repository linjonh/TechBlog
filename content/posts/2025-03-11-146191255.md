     回顾与今天的计划

    <p>
     今天的任务是继续推进游戏开发的进程，主要目标是完成资产文件合并的工作。之前，我们在资产系统上取得了一些不错的进展，所以我们打算利用今天的时间将其做得更完善，以便下周能够顺利进入新的开发阶段。
    </p>
    <p>
     我们已经为接下来的任务做了一些准备工作，接下来就是解决目前存在的问题，并为下周的工作打下良好的基础。这就像篮球比赛中的进攻配合一样——我们现在是准备好了，下一步就是通过这些努力让整个项目进入一个新的高度。
    </p>
    <p>
     不过，有一个小插曲需要先处理，就是清理一下之前为了帮助理解指针传递和按值传递的代码。之前有人提出过关于指针传递的疑问，为了帮助理解，我们编写了相关的示例代码。但现在这个代码已经不再需要，因此我们首先删除这些不再使用的部分。
    </p>
    <p>
     今天的目标主要是：
    </p>
    <ol>
     <li>
      清理不再需要的示例代码。
     </li>

     </li>
    </ol>
    <h2>
     <a id="_13">
     </a>
     回顾之前的代码
    </h2>
    <p>
     昨天的工作主要集中在开发一个可以打开多个资产文件并将它们的数据合并在一起的系统。目的是模拟最终游戏启动时加载和处理资产文件的过程。这一步是为了帮助设计一个平台层的API，即最终游戏发布时我们希望在所有平台上支持的文件API，而这个API需要能够加载和处理资产文件。
    </p>
    <p>
     在这个过程中，首先设计了一个名为“平台文件句柄”的系统。这个文件句柄的作用是保持对游戏中资产文件的引用，使得在游戏运行过程中，可以持续操作这些文件。在启动时，我们会检查目录中有哪些资产文件，然后逐个打开这些文件，并保持对它们的文件句柄，从而可以在后续操作中读取和处理这些文件的内容。
    </p>
    <p>
     此外，平台文件句柄还需要具备错误检测功能。当文件发生错误时，文件句柄会记录该错误，这样在后续的操作中，开发者只需做一次错误检查就能知道是否出现了问题。如果检测到错误，程序可以优雅地关闭相关文件，否则继续进行正常操作。
    </p>
    <p>
     为了进一步完善这个系统，计划在错误发生时为游戏用户提供反馈，例如弹出对话框来通知用户发生了错误。考虑到如果资产文件未能加载，我们可能无法显示通知内容（因为可能没有可用的字体），因此可能需要在程序中嵌入一个调试字体，仅用于这种紧急情况。这种方案在实际操作中需要进一步细化和测试。
    </p>
    <p>
     在确保文件句柄功能完善后，我们会分配内存来存储所有要加载的资产的元数据。这里加载的只是资产的元数据，而不是整个资产文件本身，因为资产文件通常非常大，而且会通过流式加载来优化内存使用。元数据存储可以帮助我们快速查找资产并决定何时加载相应的内容，尤其是当游戏中有大量资产时。
    </p>
    <p>
     接下来，我们还计划编写代码来实际加载这些资产。具体方法是遍历所有文件，检查每个文件是否包含特定类型的资产。如果发现符合条件的资产，就会从文件中读取相应的资产数据，并将其存储在预分配的内存区域中。每读取完一部分数据，就会更新相应资产类型的计数器。这一过程是为了确保所有需要的资产都被加载到内存中，虽然这种方法可能效率不高，但目前来看它的效率足够满足需求。
    </p>
    <p>
     总体来说，昨天的工作主要集中在资产文件的加载和处理机制上，确保能够有效地读取并合并多个文件的数据，并为后续的工作做出合理的规划。

     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/6bebf03c572549428ace0841508574b4.png#pic_center"/>
    </p>
    <h2>
     <a id="_30">
     </a>
     加载资产数组
    </h2>
    <p>
     首先，我们需要做的第一件事是计算每种类型的资产数量。计算方法是使用上一次传递时记录的最后一个资产索引，并减去当前数组中指定的第一个资产索引，这样就可以得到实际的资产数量，也就是我们需要加载的资产数量。
    </p>
    <p>
     接下来，我们需要将这些资产加载到正确的位置，确保它们按预期进行排列和使用。这部分我们稍后会处理，但在计算出需要加载的资产数量之后，我们要做的就是按照该数量进行推进，也就是跳过这些已经计算并加载的资产，继续处理下一个资产类型。
    </p>
    <p>
     因此，整体流程就是：首先计算当前类型的资产数量，然后确保将这些资产加载到正确的位置，最后通过增加该数量的步长继续处理后续资产类型。这是当前我们所需要完成的主要任务。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/b4dbd770d48c4a2a8c36347bcf147695.png#pic_center"/>
    </p>
    <h2>
     <a id="_41">
     </a>
     标签索引重定基机制概述
    </h2>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/1f03668f37384a758ab24f6bdb7a2c2d.png#pic_center"/>
    </p>
    <p>
     首先，我们需要处理的一个重要问题是重新基准化（rebase）标签索引，这一点非常关键，所以我们先解决这个问题，然后再讨论其他内容。
    </p>
    <p>
     我们现在拥有一组标签数组（Tags arrays），也就是一组资产标签（asset tags），这些标签是存储在文件格式中的。在文件格式中，我们使用 HHA 标签进行标识，而 HHA 标签中包含两个重要的字段：ID 和 value。这两个字段用于唯一标识和描述每个资产。
    </p>
    <pre><code class="prism language-cpp"><span class="token keyword">struct</span> <span class="token class-name">hha_tag</span> <span class="token punctuation">{<!-- --></span>
    uint32 ID<span class="token punctuation">;</span>
    real32 Value<span class="token punctuation">;</span>
<span class="token punctuation">}</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     然而，资产引用标签的方式是通过索引进行的。也就是说，HHA 资产中会记录两个索引：第一个标签的索引（FirstAssetIndex）以及
     <strong>
      超出最后一个标签的索引
     </strong>
     （OnePastLastAssetIndex）。这些索引是
     <strong>
      绝对索引
     </strong>
     ，意味着它们直接对应标签表中的位置。例如，如果某个资产的标签索引是 5 到 6，那么它表示该资产关联了标签表中的第 5 和第 6 个标签。
    </p>
    <p>
     在只有一个文件的情况下，这种映射关系是完全正确的。但如果我们有多个文件需要合并（concatenate），那么问题就出现了。
    </p>
    <p>
     假设我们有两个文件：
    </p>
    <ul>
     <li>
      第一个文件包含 100 个标签。
     </li>
     <li>
      第二个文件包含 50 个标签。
     </li>
    </ul>
    <p>
     在合并时，标签数组会直接拼接在一起，即第一个文件的 100 个标签在前，第二个文件的 50 个标签紧随其后。此时：
    </p>
    <ul>
     <li>
      第一个文件的标签索引是完全正确的，因为它们从 0 开始，直接对应它们的标签位置。
     </li>
     <li>
      <strong>
       但第二个文件中的标签索引就会完全错误
      </strong>
      ，因为它的标签索引仍然是从 0 开始计算的，而实际合并后的标签位置已经发生了变化。
     </li>
    </ul>
    <p>
     举个例子：
    </p>
    <ul>
     <li>
      假设第二个文件中某个资产引用标签索引为 0 和 1，但在合并后的数组中，这两个标签的实际位置已经变成了 100 和 101。
     </li>
     <li>
      因此，
      <strong>
       我们需要将第二个文件中的所有标签索引都加上前一个文件的标签总数
      </strong>
      ，这样才能确保它们引用的是正确的标签位置。
     </li>
    </ul>
    <p>
     我们需要做的事情就是：
    </p>
    <ol>
     <li>
      遍历第二个文件中的所有资产。
     </li>
     <li>
      将它们的
      <strong>
       第一个标签索引
      </strong>
      和
      <strong>
       oOnePastLastAssetIndex 标签索引
      </strong>
      <strong>
       加上前一个文件的标签总数
      </strong>
      ，使它们的索引基准与合并后的数组对齐。
     </li>
     <li>
      如果有第三个、第四个文件，继续重复上述操作，使所有文件的标签索引都基于完整合并后的标签表进行重新基准化（rebase）。
     </li>
    </ol>
    <p>
     这个过程本身是非常简单的，我们只需要按照前面文件的标签数量进行偏移修正即可。所以我们接下来要做的，就是优先解决这个索引重基准化的问题，确保所有资产在合并后引用的标签是正确的。
    </p>
    <h2>
     <a id="_TagBase_76">
     </a>
     设置文件的 TagBase
    </h2>
    <p>
     在这里我们需要处理的主要任务是加载多个文件中的标签数组（tag arrays），并将它们合并成一个连续的大标签数组，同时确保每个文件中的标签索引保持正确。
    </p>
    <p>
     我们目前的文件格式中，每个文件都包含一个标签数组（tag array），而每个标签数组中包含若干个标签（tag）。每个标签本质上是一个 HHA 标签，里面包含 ID 和 value，用于描述资产的属性。然而，多个文件的标签数组在合并时会带来一些索引错位的问题，因此我们需要通过设置**标签基准（TagBase）**来解决这个问题。
    </p>
    <h4>
     <a id="1_tagbase_81">
     </a>
     1. 设置文件的标签基准（tagbase）
    </h4>
    <p>
     我们首先要做的，是确保在加载每个文件时，记录该文件的
     <strong>
      标签基准
     </strong>
     。
    </p>
    <ul>
     <li>
      第一个文件的标签基准起始位置为 0，因为还没有加载任何标签。
     </li>
     <li>
      第二个文件的标签基准起始位置应该是
      <strong>
       第一个文件中标签总数的下一位
      </strong>
      ，即
      <strong>
       one-past-last
      </strong>
      标签的位置。
     </li>
     <li>
      第三个文件的标签基准起始位置则是前两个文件标签总数的下一位，以此类推。
     </li>
    </ul>
    <p>
     通过设置这个
     <strong>
      标签基准（TagBase）
     </strong>
     ，我们确保了每个文件在加载标签时，能够将它们放置到正确的位置，并且不会覆盖已经加载的标签数据。
    </p>
    <h4>
     <a id="2__89">
     </a>
     2. 循环加载所有文件的标签数组
    </h4>
    <p>
     我们需要遍历所有文件，将它们的标签数组
     <strong>
      连续加载
     </strong>
     到一块大的标签数组中。具体过程如下：
    </p>
    <ul>
     <li>
      <strong>
       遍历文件
      </strong>
      ，确保文件没有错误的情况下，开始加载标签数组。
     </li>
     <li>
      <strong>
       计算标签数组的大小
      </strong>
      ，即：
      <br/>
      <span class="katex--display">
       KaTeX parse error: Expected 'EOF', got '_' at position 42: …text{sizeof(hha_̲tag)} \times \t…
      </span>
      <br/>
      这里的 sizeof(hha_tag) 是标签结构的大小，而 TagCount 是当前文件的标签数量。
     </li>
     <li>
      将标签数据加载到
      <strong>
       当前文件的标签基准偏移量
      </strong>
      位置：
      <ul>
       <li>
        第一个文件从索引 0 开始加载。
       </li>
       <li>
        第二个文件从第一个文件的标签总数索引处开始加载。
       </li>
       <li>
        第三个文件从前两个文件标签总数索引处开始加载。
       </li>
      </ul>
     </li>
     <li>
      每加载完一个文件的标签数组后，
      <strong>
       更新标签基准
      </strong>
      ，使其指向下一个文件标签数组的起始位置。
     </li>
    </ul>
    <h4>
     <a id="3__103">
     </a>
     3. 保证标签数组连续排列
    </h4>
    <p>
     通过上述操作，我们确保了所有文件的标签数组都被加载到一个
     <strong>
      连续的内存区域
     </strong>
     中。具体示例如下：
    </p>
    <table>
     <thead>
      <tr>
       <th>
        文件序号
       </th>
       <th>
        标签数量
       </th>
       <th>
        标签基准 (TagBase)
       </th>
       <th>
        标签在内存中的位置
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        文件0
       </td>
       <td>
        100
       </td>
       <td>
        0
       </td>
       <td>
        [0 - 99]
       </td>
      </tr>
      <tr>
       <td>
        文件1
       </td>
       <td>
        50
       </td>
       <td>
        100
       </td>
       <td>
        [100 - 149]
       </td>
      </tr>
      <tr>
       <td>
        文件2
       </td>
       <td>

       </td>
       <td>
        150
       </td>
       <td>
        [150 - 179]
       </td>
      </tr>
     </tbody>
    </table>
    <p>
     这种做法确保了我们最终得到的标签数组是
     <strong>
      连续且无重叠
     </strong>
     的，并且每个文件中的标签都按照正确的索引排列在大数组中。
    </p>
    <h4>
     <a id="4__114">
     </a>
     4. 加载时的核心操作
    </h4>
    <p>
     在读取数据时，我们会使用平台提供的
     <code>
      PlatformReadDataFromFile
     </code>
     接口进行数据加载。加载操作如下：
    </p>
    <ul>
     <li>
      将标签数组数据读取到内存中，读取的起始位置为
      <strong>
       当前文件的标签基准
      </strong>
      （TagBase）。
     </li>
     <li>
      将标签数组的大小设置为
      <strong>
       当前文件标签数量 * HHA 标签大小
      </strong>
      。
     </li>
     <li>
      每加载完一个文件的数据后，更新标签基准，使其指向下一块空闲内存区域。
     </li>
    </ul>
    <h4>
     <a id="5__120">
     </a>
     5. 为什么标签数据本身不需要修改
    </h4>
    <p>
     值得注意的是，标签数据本身并不需要修改。因为标签本身（ID 和 value）没有引用其他数据，所以无需进行任何调整。我们所需要做的仅仅是保证
     <strong>
      标签索引在合并后的大数组中保持正确
     </strong>
     。
    </p>
    <p>
     例如：
    </p>
    <ul>
     <li>
      文件 0 的第一个标签，索引为 0，合并后索引仍然是 0。
     </li>
     <li>
      文件 1 的第一个标签，原索引为 0，但合并后索引应该变成 100（假设文件 0 有 100 个标签）。
     </li>
     <li>
      文件 2 的第一个标签，原索引为 0，但合并后索引应该变成 150（假设前两个文件共 150 个标签）。
     </li>
    </ul>
    <p>
     这个过程中，
     <strong>
      我们不修改标签内容本身
     </strong>
     ，只需要通过标签基准偏移量确保它们加载到正确的位置即可。
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/829620a17c9744e892c432a93aa30997.png#pic_center"/>
    </p>
    <h2>
     <a id="_133">
     </a>
     重定基资产数组
    </h2>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/7d64339b776c4577a00be2f3205a1a05.png#pic_center"/>
    </p>
    <p>
     在这里，我们需要做的是对资产（assets）进行重定位（rebase），以确保它们的标签索引正确地指向合并后的标签数组中的相应位置。
    </p>
    <h4>
     <a id="1__138">
     </a>
     1. 资产重定位的目标
    </h4>
    <p>
     由于每个文件在合并标签数组时，标签的起始索引会发生偏移，因此每个文件中的资产引用的标签索引也需要调整。具体来说，每个资产的标签索引需要加上该文件的
     <strong>
      标签基准（tag base）
     </strong>
     ，以确保它指向合并后的标签数组中的正确位置。
    </p>
    <h4>
     <a id="2__141">
     </a>
     2. 重定位的具体步骤
    </h4>
    <p>
     我们需要按照以下步骤来完成资产的重定位：
    </p>
    <ul>
     <li>
      <strong>
       循环遍历当前文件中的资产
      </strong>
      。
      <ul>
       <li>
        资产的遍历起始索引是当前已加载的资产数量
        <code>
         AssetCount
        </code>
        。
       </li>
       <li>
        结束索引是
        <code>
         AssetCount + 当前文件的资产数量
        </code>
        ，即确保覆盖当前文件的所有资产。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       获取当前资产的指针
      </strong>
      。
      <ul>
       <li>
        资产指针的计算方式是
        <code>
         assets数组 + 当前资产索引
        </code>
        ，这可以快速找到内存中该资产的位置。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       调整资产的标签索引
      </strong>
      。
      <ul>
       <li>
        每个资产都有一个指向其标签的索引字段。
       </li>
       <li>
        该索引需要加上该文件的标签基准（
        <code>
         TagBase
        </code>
        ），以将其从原本的
        <strong>
         局部索引
        </strong>
        转换为
        <strong>
         全局索引
        </strong>
        。
       </li>
      </ul>
     </li>
    </ul>

     <a id="_hha_asset__166">

     读取 hha_asset 数据
    </h2>
    <p>
     现在我们需要做的最后一件事就是
     <strong>

     </strong>
     ，并进行相应的计算，使这些资产数据能够正确地填充到内存中的合适位置。
    </p>
    <h4>
     <a id="1__170">
     </a>
     1. 读取数据的总体思路
    </h4>
    <p>
     我们的主要目标是：
    </p>
    <ul>
     <li>
      <strong>
       从文件中读取资产数据
      </strong>
      。
     </li>
     <li>
      <strong>

      </strong>

     <li>
      <strong>

     <li>

    </h2>
    <p>
     现在我们需要开始
     <strong>
      真正定义这些平台函数
     </strong>
     ，也就是之前我们在开发过程中预设的那些函数。之前我们在编写代码时，只是
     <strong>

     </strong>

     </strong>

    </p>
    <p>
     我们面临的主要问题是：
    </p>
    <ul>
     <li>

     </li>

      </strong>
      ，比如先通过原始的文件 IO 例程加载数据，再逐步过渡到新的平台 API，但目前看来这样做的工作量可能更大，所以决定直接一步到位。
     </li>
    </ul>

    <h4>
     <a id="1__252">
     </a>
     1. 确定平台函数的目标和路径
    </h4>
    <p>
     我们首先明确一点：
    </p>
    <ul>
     <li>
      这些平台 API 主要提供文件操作、内存管理等功能。
     </li>
     <li>
      我们希望尽量使这些 API 保持简单易用，同时兼容 C 语言。
     </li>
     <li>
      为了保证可读性和一致性，我们在命名时使用
      <code>
       platform_
      </code>
      作为前缀，这样就能确保平台特定的 API 有明确的标识。
     </li>
    </ul>

    <h2>
     <a id="_platform_file_handle__platform_file_group__260">
     </a>
     定义 platform_file_handle 和 platform_file_group 结构体
    </h2>
    <p>
     我们现在需要定义
     <strong>
      平台文件句柄 (platform_file_handle)
     </strong>
     和
     <strong>
      平台文件组 (platform_file_group)
     </strong>
     ，以支持我们在平台层中操作文件。我们之前已经定义了
     <code>
      platform_file_handle
     </code>
     ，但现在我们需要进一步调整它，并创建
     <code>
      platform_file_group
     </code>
     ，以便我们能够通过文件组来访问一系列的文件。
    </p>
    <p>
     我们面临的几个主要问题是：
    </p>
    <ol>
     <li>
      <strong>
       platform_file_handle 的数据结构设计
      </strong>
      ：我们希望平台句柄是
      <strong>
       不透明的 (opaque)
      </strong>
      ，即我们不希望调用方知道文件句柄内部的具体内容，这样可以确保不同平台的文件句柄结构可以自由变化。
     </li>
     <li>
      <strong>
       platform_file_group 的数据结构设计
      </strong>
      ：平台文件组将表示某个目录下的文件集合，它允许我们通过索引访问文件并打开它们，但我们不想暴露文件组的内部结构。
     </li>
     <li>
      <strong>
       保持平台 API 的一致性
      </strong>
      ：我们需要确保所有文件操作都通过平台 API 完成，而不会直接操作 HHA 文件。
     </li>
    </ol>
    <hr/>
    <h3>
     <a id="_platform_file_handle__270">
     </a>
     一、调整 platform_file_handle 的定义
    </h3>
    <p>
     我们最初的
     <code>
      platform_file_handle
     </code>
     是这样定义的：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">struct</span> <span class="token class-name">platform_file_handle</span>
<span class="token punctuation">{<!-- --></span>
    <span class="token keyword">void</span><span class="token operator">*</span> handle<span class="token punctuation">;</span>
<span class="token punctuation">}</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     但是现在我们希望它
     <strong>
      完全不透明
     </strong>

    </p>

    <h2>
     <a id="_586">
     </a>
     使用新工作指令类型和旧版文件加载例程交换，以最小化调试量
    </h2>
    <p>
     现在我们的目标是对现有的加载位图的功能进行改进，使其更加通用化，同时减少重复代码并优化加载流程。之前的实现中，每种类型的资产（如位图、声音等）的加载工作是分开进行的，并且加载位图的任务有一个特定的
     <code>
      load_bitmap_work
     </code>
     来处理所有和位图相关的加载和初始化操作。但是现在我们计划将其简化，创建一个通用的
     <code>
      load_asset_work
     </code>
     ，该任务不关心加载的是哪种资产，只负责从文件中读取指定大小的数据到指定的内存位置，并将该任务的所有完成状态回传。
    </p>
    <p>
     我们的第一个目标是确保在功能切换前，现有的加载功能完全正常，这样我们在重构过程中就不会因为同时引入多种修改而导致难以排查的错误。我们计划分步进行这个修改，避免将新文件加载例程的开发和位图加载的重构混在一起，以减少调试难度。
    </p>
    <p>
     接下来我们计划创建一个通用的
     <code>
      load_asset_work
     </code>
     任务，它只包含三个主要参数：
    </p>
    <ul>
     <li>
      <strong>
       偏移量（offset）
      </strong>
      ：表示从文件的哪个位置开始读取数据。
     </li>
     <li>
      <strong>
       加载大小（size）
      </strong>
      ：表示需要读取的数据大小。
     </li>
     <li>
      <strong>
       目标地址（destination）
      </strong>
      ：表示读取的数据应该存放在内存中的哪个地址。
     </li>
    </ul>
    <p>
     这个任务的职责非常简单，只需要调用平台层的
     <code>
      read_data_from_file
     </code>
     函数，按照指定的偏移量和大小，将数据读取到指定的内存位置即可。它不会关心当前加载的是位图、声音还是其他资产，也不会做任何资产解析或数据填充的操作。它的核心任务就是
     <strong>
      加载文件数据
     </strong>
     ，其余的一切由外部调用者决定。
    </p>
    <p>
     在实现这个任务时，我们还需要提供一个文件句柄（file handle），确保我们在读取数据时没有文件错误。在数据读取完成后，我们也需要更新任务状态，确保任务结束后能够正确通知其他部分加载已完成。
    </p>
    <p>
     <strong>
      接下来我们需要注意几个问题
     </strong>
     ：
    </p>
    <ol>
     <li>
      <strong>
       文件句柄错误处理
      </strong>
      ：我们需要检查文件句柄是否有效，如果文件句柄无效，我们不能错误地设置任务完成状态，而是需要有一种处理策略。例如，可以选择填充一块无意义的数据（如全零），或者让该任务保持未完成状态，等待后续加载。
     </li>
     <li>
      <strong>
       任务状态更新
      </strong>
      ：在任务完成后，需要明确设置任务状态，标志其已完成并可供后续使用。
     </li>
     <li>
      <strong>
       保持接口通用性
      </strong>
      ：在
      <code>
       load_asset_work
      </code>
      任务中，不涉及任何与位图、声音等资产相关的操作，仅执行数据加载任务。
     </li>
    </ol>
    <p>
     接下来我们将修改原有的
     <code>
      load_bitmap_work
     </code>
     任务，将其重构为使用新的
     <code>
      load_asset_work
     </code>
     任务。在重构过程中，我们需要将原本在
     <code>
      load_bitmap_work
     </code>
     中的所有初始化操作前置，在创建任务时直接处理。这样，
     <code>
      load_asset_work
     </code>
     任务将完全不需要接触资产相关信息，确保其通用性。
    </p>
    <p>
     我们还需要处理一些原本特定于位图的状态更新和数据分配操作。这部分工作将移至任务创建前进行，而不再依赖任务内部处理。例如，原本在任务内部处理的
     <strong>
      位图尺寸计算
     </strong>
     、
     <strong>
      内存分配
     </strong>
     等工作，现在都提前在创建任务时完成，这样任务内部只需关注数据的加载即可。
    </p>
    <p>
     最后，我们将原本的
     <code>
      load_bitmap_work
     </code>
     任务完全替换为
     <code>
      load_asset_work
     </code>
     ，并确保编译和运行正常。我们还需要做一些测试，确保新任务加载位图时不会出现任何问题，并验证在文件读取失败的情况下，是否会如预期填充默认数据或保持未加载状态。
    </p>
    <p>
     通过这样的重构：
    </p>
    <ul>
     <li>
      <strong>
       代码复用性更高
      </strong>
      ：不再需要为每种资产写一个特定的加载任务。
     </li>
     <li>
      <strong>
       错误隔离性更好
      </strong>
      ：将数据加载和数据解析分离，减少了加载任务中的复杂逻辑。
     </li>
     <li>
      <strong>
       调试更方便
      </strong>
      ：由于修改被拆分为两个阶段（先重构加载任务，后更换文件读取例程），调试过程将变得更容易。
     </li>
     <li>
      <strong>
       未来扩展性更强
      </strong>
      ：如果未来需要加载更多类型的资产（如模型、纹理等），只需调用
      <code>
       load_asset_work
      </code>
      即可，无需增加新的任务类型。
     </li>
    </ul>
    <p>
     下一步就是进行加载位图的全面重构，并确保它完全依赖新的
     <code>
      load_asset_work
     </code>
     进行数据加载。

    <h2>
     <a id="_621">
     </a>
     为位图预留空间
    </h2>
    <p>
     现在我们需要做的就是为即将加载的位图预留内存空间，以便在文件数据加载完成后能够直接将位图数据存储到内存中。因此，我们只需要在内存区域中进行内存分配，提前为位图的原始数据保留足够的空间。
    </p>
    <p>
     在分配内存时，我们选择使用
     <code>
      push
     </code>
     操作，将指定大小的内存块从内存区域（arena）中分配出来。这里的内存区域通常是为加载资产（assets）预留的，所以我们直接在该区域中进行分配操作。分配的大小直接取决于位图的尺寸，即宽度、高度和每行像素占用的字节数。我们可以通过计算
     <strong>
      pitch * height
     </strong>
     的方式来得到需要分配的内存大小。
    </p>
    <p>
     这里存在一个潜在的“先有鸡还是先有蛋”的问题。即：我们在分配内存之前，需要知道位图的大小，但在加载数据之前我们还没有这些信息。因此，我们需要先通过位图信息结构（info）来获取
     <strong>
      pitch
     </strong>
     和
     <strong>
      height
     </strong>
     ，从而计算出内存大小。
    </p>
    <p>
     具体计算方法如下：
    </p>
    <ul>
     <li>
      <strong>
       pitch
      </strong>
      表示位图数据中每一行像素占用的内存大小（以字节为单位）。
     </li>
     <li>
      <strong>
       height
      </strong>
      表示位图的总行数。
     </li>
     <li>
      二者相乘即可得到总的内存需求大小，即
      <code>
       pitch * height
      </code>
      。
     </li>
    </ul>
    <p>
     我们将计算结果赋值给一个名为
     <code>
      memory_size
     </code>
     的变量，该变量代表为该位图所需要分配的内存大小。随后我们在内存区域中调用
     <code>
      push
     </code>
     操作，预留这块内存空间，确保稍后文件数据加载完成后，能够直接将数据填充到这里。
    </p>
    <p>
     在分配完内存后，我们接下来就可以填充其他一些基本信息，例如：
    </p>
    <ul>
     <li>
      将内存地址设置为位图的存储地址。
     </li>
     <li>
      将任务状态标记为“等待加载完成”。
     </li>
     <li>
      设置文件读取的偏移量和数据大小等。
     </li>
    </ul>
    <p>
     目前我们还没有真正触碰到文件加载部分，所有的工作都是在为加载任务做前期准备。我们在这一阶段确保内存分配是正确的，并且任务结构中的内存地址已经准备就绪。这样等到文件加载完成时，数据可以无缝地填充到该内存中。
    </p>
    <p>
     我们在这里还有一个考虑点，即如果文件读取失败怎么办？目前有两种策略：
    </p>
    <ol>
     <li>
      <strong>
       保守策略
      </strong>
      ：如果文件加载失败，则不改变任务状态，避免将无效数据暴露给其他模块。
     </li>
     <li>
      <strong>
       宽容策略
      </strong>
      ：如果文件加载失败，我们仍然填充这块内存，但用特定的标识数据（如全零或全粉色）表示该位图加载失败，确保游戏不会崩溃。
     </li>
    </ol>
    <p>
     当前我们暂时保持灵活，先专注于确保内存分配的正确性，稍后再决定如何处理加载失败的情况。接下来我们就可以继续配置任务参数，让文件加载功能将数据填充到该内存中。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ce9c43ae7d7341d3882ec5eac3ccfcc0.png#pic_center"/>
    </p>
    <h2>
     <a id="_650">
     </a>
     填充工作指令结构体
    </h2>
    <p>
     现在我们需要做的是填充工作任务（work order）的所有字段，并将该任务放入任务队列中，以便稍后执行加载操作。这些任务将被放置在**任务内存区域（task arena）**中，因为这是临时内存，因此在加载完成之后就不需要保留这些数据了。我们创建一个新的
     <code>
      load_asset_work
     </code>
     ，并开始填充其所有必要的字段。
    </p>
    <p>
     首先，我们要填充
     <strong>
      工作任务结构
     </strong>
     中的
     <code>
      Work-&gt;Slot
     </code>
     ，即对应的资产槽位（Assets-&gt;Slots）。该槽位将指向我们正在加载的位图，我们直接将这个槽位的指针赋值给
     <code>
      Work-&gt;Slot
     </code>
     字段。这么做的好处是，当加载完成后，数据会直接填充到该槽位中，而不需要额外的映射过程。
    </p>
    <p>
     接下来，我们需要填充
     <strong>
      文件句柄（file handle）
     </strong>
     ，以便读取文件数据。我们之前已经有了一个平台文件句柄
     <code>
      platform_file_handle
     </code>
     ，所以直接将其赋值即可。
    </p>
    <p>
     然后，我们需要填写
     <strong>
      文件偏移量（offset）
     </strong>
     。文件偏移量决定了从文件的哪个位置开始读取数据。在前面的流程中，我们已经通过
     <code>
      HHAAsset-&gt;DataOffset
     </code>
     获取了该信息，所以直接将其赋值给任务即可。
    </p>
    <p>
     **数据大小（size）**也需要填充。这部分数据大小就是我们之前计算出来的
     <code>
      MemorySize
     </code>
     ，即位图数据所占用的总字节数，我们直接将其填入任务的
     <code>
      size
     </code>
     字段中。
    </p>
    <p>
     **内存地址（destination）**是数据最终需要存放的位置，我们之前已经通过
     <code>
      push
     </code>
     操作在内存中预留了一块空间，用来存储位图数据。因此，这里我们将那块内存地址赋值给任务的
     <code>
      destination
     </code>
     字段，这样当文件加载完成时，数据会直接填充到这块内存中。
    </p>
    <pre><code class="prism language-cpp">Bitmap<span class="token operator">-&gt;</span>Memory <span class="token operator">=</span> Assets<span class="token operator">-&gt;</span>HHAContents <span class="token operator">+</span> HHAAsset<span class="token operator">-&gt;</span>DataOffset<span class="token punctuation">;</span>
</code></pre>
    <p>
     至此，我们的工作任务已经基本设置完成，包含了：
    </p>
    <ul>
     <li>
      <strong>
       work slot
      </strong>
      ：对应的资产槽位。
     </li>
     <li>
      <strong>
       file handle
      </strong>
      ：平台文件句柄，用于读取数据。
     </li>
     <li>
      <strong>
       offset
      </strong>
      ：文件中数据的起始偏移量。
     </li>
     <li>
      <strong>
       size
      </strong>
      ：要读取的数据大小。
     </li>
     <li>
      <strong>
       Destination
      </strong>
      ：数据的目标内存地址。
     </li>
    </ul>
    <p>
     我们接着发现一个问题：在某些情况下，
     <code>
      work slot
     </code>
     的指针并没有提前设置，而是留在了加载过程的线程中设置的，这样就导致线程中不得不执行一些
     <strong>
      非必要的任务
     </strong>
     。但实际上，这种设置完全可以提前在主线程中完成，避免在多线程中做一些不必要的工作，减少线程竞争的风险。因此，我们决定直接在主线程中
     <strong>
      预先设置
      <code>
       work slot
      </code>
      的指针
     </strong>
     ，并避免在子线程中修改它，从而降低线程安全性问题。
    </p>
    <p>
     我们同时还优化了
     <strong>
      返回位图的逻辑
     </strong>
     ，通过判断
     <code>
      slot state
     </code>
     是否处于
     <code>
      asset loaded
     </code>
     状态来决定是否返回有效数据。我们直接将
     <code>
      slot state
     </code>
     与
     <code>
      asset loaded
     </code>
     进行比较，如果状态满足，就直接返回位图，否则返回空指针（null），这样就避免了子线程中频繁检查或更新指针的需求。
    </p>
    <p>
     接下来我们注意到，在填充任务时，存在一些多余的工作，比如在工作任务中填充
     <code>
      work bitmap
     </code>
     的过程。实际上，这个过程也可以提前在主线程中完成，而不必放到子线程中去执行。我们认为这样做更加清晰，避免了在多线程中操作内存地址，从而减少潜在的竞争风险。
    </p>
    <p>
     在调整这些内容之后，我们还修复了一些小问题：
    </p>
    <ol>
     <li>
      <strong>
       移除了多余的状态检查
      </strong>
      ：原本在子线程中会检查
      <code>
       work bitmap
      </code>
      是否为空，现在直接在主线程中将其设置好，避免了重复检查。
     </li>
     <li>
      <strong>
       统一了指针设置的时机
      </strong>
      ：我们在主线程中就直接设置好
      <code>
       work slot
      </code>
      的指针，确保子线程无需再次操作，减少线程竞争。
     </li>
     <li>
      <strong>
       确保任务结构简洁
      </strong>
      ：我们剔除了不必要的字段，仅保留了加载数据的必需信息。
     </li>
    </ol>
    <p>
     接下来我们需要做的就是确保文件加载完成后，数据能顺利填充到内存中。因此，我们的下一步就是在文件加载完成的回调中，确认数据已成功加载，并将位图标记为已加载状态（asset loaded）。这样，当其他模块请求位图时，就可以直接从内存中获取而无需重复加载。
    </p>
    <p>
     我们稍后还需要处理加载失败的情况。当前的策略是：
    </p>
    <ul>
     <li>
      <strong>
       如果文件加载失败
      </strong>
      ，我们可以选择用特定颜色（如粉色或全零）填充内存。
     </li>
     <li>
      或者，我们可以保留该任务，但不标记为
      <code>
       asset loaded
      </code>
      ，让其保持未加载状态，直到下次加载请求再次触发。
     </li>
    </ul>
    <p>
     目前我们倾向于保留任务失败状态，避免使用错误数据，但最终方案可以根据实际情况调整。
    </p>
    <p>
     现在我们的任务结构已经完成，下一步就是调整文件读取流程，将文件数据填充到内存中，并确保数据在加载完成后状态正确。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ac960a665f594ee7bf95134423907fbf.png#pic_center"/>
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/3b6b80bfbc204510a4affb471d9ed20e.png#pic_center"/>
    </p>
    <h2>
     <a id="Win32__698">
     </a>
     Win32 文件例程仍未完成，但我们的时间到了，所以让我们保持代码在运行状态
    </h2>
    <p>
     我们现在需要做的就是
     <strong>
      暂时禁用任务的实际操作
     </strong>
     ，即让任务在执行时
     <strong>
      不做任何实际的数据加载
     </strong>
     ，只保留完成任务的操作 (
     <code>
      complete
     </code>
     )。这样我们可以快速验证整个流程是否正常工作，而不需要等待文件加载的实际完成。我们的目标是
     <strong>
      通过跳过文件加载
     </strong>
     ，观察内存指针是否正确设置，从而验证加载任务流程是否正确。
    </p>
    <p>
     首先，我们进入任务处理的核心部分，将
     <strong>
      任务中的所有文件加载逻辑去除
     </strong>
     ，保留任务完成 (
     <code>
      complete
     </code>
     ) 的部分。在这里，我们不会真正执行
     <code>
      platform.read_data_from_file
     </code>
     ，而是直接
     <strong>
      将内存指针指向正确的数据地址
     </strong>
     ，以观察流程是否通畅。换句话说，我们暂时欺骗任务系统，让其看起来像是加载成功，但实际上并没有真正加载任何数据。
    </p>
    <p>
     我们还遇到一个问题：如果文件句柄 (
     <code>
      file handle
     </code>
     ) 不可用，那么任务无法继续推进。因此，我们需要在这里
     <strong>
      添加一个条件分支
     </strong>
     ，如果句柄无效 (
     <code>
      file handle == null
     </code>
     )，那么直接跳过任务执行，但仍然标记任务已完成 (
     <code>
      complete
     </code>
     )。这确保了任务队列不会卡死在无效句柄的任务上。
    </p>
    <p>
     接下来，我们需要解决
     <strong>
      任务回调的指针设置
     </strong>
     问题。在任务完成后，我们希望指针能够指向正确的数据内存，但由于目前任务没有真正加载数据，因此我们需要
     <strong>
      手动设置指针
     </strong>
     ，确保任务完成后能访问数据。我们直接将
     <code>
      work slot
     </code>
     中的内存指针设置为我们预留的内存地址 (
     <code>
      destination
     </code>
     )，以便在测试时可以直接访问数据内存。
    </p>
    <p>
     接下来我们发现，由于没有真正加载数据，因此我们不能调用文件读取函数 (
     <code>
      platform.read_data_from_file
     </code>
     )。因此，我们需要将这一部分代码用
     <code>
      #if 0
     </code>
     宏屏蔽掉，避免调用无效文件句柄。将来当我们恢复文件加载逻辑时，只需要移除
     <code>
      #if 0
     </code>
     宏即可。
    </p>
    <p>
     我们接着检查任务状态，确保任务在完成后能正常进入
     <strong>
      任务完成状态
     </strong>
     (
     <code>
      complete
     </code>
     )。我们发现任务结构体中的
     <code>
      final_state
     </code>
     仍然需要填写，以确保任务在完成后能被识别为已完成状态。我们直接将
     <code>
      final_state
     </code>
     设置为
     <code>
      asset_loaded
     </code>
     ，确保即使没有实际加载数据，也能将任务标记为已完成。
    </p>

    <h2>
     <a id="_hha_IDs__717">
     </a>
     使用此系统，是否可以在不修改原始文件的情况下，在新的 .hha 文件中添加更新版的资产？在这种情况下，IDs 会怎么处理？
    </h2>
    <p>
     我们现在讨论的是在不修改原始HHA文件的情况下，
     <strong>
      添加一个更新版本的资源
     </strong>
     到新的HHA文件中，并确保系统能够优先使用新的资源。这种情况在
     <strong>
      资源更新、补丁更新或MOD制作
     </strong>
     时非常常见，而我们需要解决的核心问题是：
     <strong>
      如何处理资源ID
     </strong>
     ，确保加载时优先使用新的资源而不是旧的资源。
    </p>
    <h4>
     <a id="_720">
     </a>
     <strong>
      当前资源查询的工作方式
     </strong>
    </h4>
    <p>
     目前，我们的资源查询系统基于**标签匹配 (tag matching)**工作。当我们调用
     <code>
      GetBestMatchBitmapFrom
     </code>
     时，内部会基于提供的标签向量 (tag vector) 执行查询，并返回最匹配的资源。查询过程大致如下：
    </p>
    <ol>
     <li>
      <strong>
       遍历所有已加载的HHA文件
      </strong>
      。
     </li>
     <li>
      <strong>
       在每个HHA文件内遍历资源
      </strong>
      ，根据标签匹配程度选出最佳匹配。
     </li>
     <li>
      <strong>
       如果有多个匹配
      </strong>
      ，目前的逻辑是
      <strong>
       选择第一个找到的资源
      </strong>
      。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/52d504fd03b342a5b7c6ed48d0e9fbfc.png#pic_center"/>
     </li>
    </ol>
    <p>
     <strong>
      问题在于：
     </strong>
    </p>
    <ul>
     <li>
      <strong>
       如果我们添加了新的HHA文件
      </strong>
      ，即使它包含了更新版本的资源，由于查询逻辑
      <strong>
       优先选择最早匹配到的资源
      </strong>
      ，因此不会加载新资源。
     </li>
     <li>
      即使新HHA文件内存在ID相同的资源，系统也无法识别应该替换原始资源。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="_733">
     </a>
     <strong>
      基于加载顺序优先替换的方案
     </strong>
    </h4>
    <p>
     我们可以在
     <strong>
      不修改原始HHA文件
     </strong>
     的前提下，让
     <strong>
      新HHA文件中的资源优先级更高
     </strong>
     ，方法是：
    </p>
    <ul>
     <li>
      <strong>
       调整加载顺序
      </strong>
      ：在加载HHA文件时，
      <strong>
       优先加载新文件
      </strong>
      ，确保新文件内的资源排在前面。
     </li>
     <li>
      <strong>
       修改匹配逻辑
      </strong>
      ：将匹配到相同标签的多个资源时，始终选择
      <strong>
       最后加载的资源
      </strong>
      。
     </li>
    </ul>
    <p>
     目前的匹配代码类似于：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">if</span> <span class="token punctuation">(</span>current_match_score <span class="token operator">&gt;</span> best_match_score<span class="token punctuation">)</span>
<span class="token punctuation">{<!-- --></span>
    best_match <span class="token operator">=</span> current_asset<span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
    <p>
     我们只需要修改成：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">if</span> <span class="token punctuation">(</span>current_match_score <span class="token operator">&gt;</span> best_match_score <span class="token operator">||</span>
    <span class="token punctuation">(</span>current_match_score <span class="token operator">==</span> best_match_score <span class="token operator">&amp;&amp;</span> current_asset<span class="token operator">-&gt;</span>load_time <span class="token operator">&gt;</span> best_match<span class="token operator">-&gt;</span>load_time<span class="token punctuation">)</span><span class="token punctuation">)</span>
<span class="token punctuation">{<!-- --></span>
    best_match <span class="token operator">=</span> current_asset<span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
    <p>
     通过引入
     <code>
      load_time
     </code>
     字段，确保
     <strong>
      后加载的资源优先
     </strong>
     ，这样就可以在不修改原始HHA文件的前提下
     <strong>
      优先使用新HHA文件中的资源
     </strong>
     。
    </p>
    <hr/>
    <h4>
     <a id="_757">
     </a>
     <strong>
      但这个方法有局限性
     </strong>
    </h4>
    <p>
     上述方法只能在
     <strong>
      标签匹配
     </strong>
     的情况下工作，但如果资源是通过
     <strong>
      直接ID引用
     </strong>
     的，比如：
    </p>
    <pre><code class="prism language-c">bitmap <span class="token operator">=</span> <span class="token function">get_bitmap_by_id</span><span class="token punctuation">(</span>asset_id<span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     那么新的HHA文件中即使存在更新资源，系统仍然会
     <strong>
      通过ID直接加载旧资源
     </strong>
     。因此，这种方式对于
     <strong>
      直接ID查询
     </strong>
     场景无法覆盖。
    </p>
    <hr/>
    <h4>
     <a id="ID_766">
     </a>
     <strong>
      针对ID查询的资源替换方案
     </strong>
    </h4>
    <p>
     如果我们希望
     <strong>
      无论是标签匹配还是直接ID查询
     </strong>
     都能优先加载新资源，我们需要
     <strong>
      添加一张资源替换表
     </strong>
     。流程如下：
    </p>
    <h5>
     <a id="1_HHA_769">
     </a>
     <strong>
      1. 在新HHA文件内添加资源替换表
     </strong>
    </h5>
    <p>
     每个新的HHA文件可以内置一个
     <code>
      资源替换表 (AssetRemovalTable)
     </code>
     ，结构类似于：
    </p>
    <pre><code class="prism language-plaintext">Asset Replacement Table:
{
    Original File ID: 1234,  // 要替换的资源ID
    Replacement File ID: 5678 // 替换的新资源ID
}
</code></pre>
    <p>
     如果资源ID
     <code>
      1234
     </code>
     存在于原HHA文件内，而新的HHA文件内包含
     <code>
      5678
     </code>
     作为替代资源，则加载时会优先加载
     <code>
      5678
     </code>
     。
    </p>
    <h5>
     <a id="2__780">
     </a>
     <strong>
      2. 在加载过程中处理替换表
     </strong>
    </h5>
    <p>
     在加载HHA文件时，检测是否存在资源替换表，如果有，则记录替换映射：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">if</span> <span class="token punctuation">(</span>new_hha<span class="token operator">-&gt;</span>has_replacement_table<span class="token punctuation">)</span>
<span class="token punctuation">{<!-- --></span>
    <span class="token keyword">for</span> <span class="token punctuation">(</span><span class="token keyword">int</span> i <span class="token operator">=</span> <span class="token number">0</span><span class="token punctuation">;</span> i <span class="token operator">&lt;</span> new_hha<span class="token operator">-&gt;</span>replacement_count<span class="token punctuation">;</span> i<span class="token operator">++</span><span class="token punctuation">)</span>
    <span class="token punctuation">{<!-- --></span>
        asset_replacement_table<span class="token punctuation">[</span>new_hha<span class="token operator">-&gt;</span>replacements<span class="token punctuation">[</span>i<span class="token punctuation">]</span><span class="token punctuation">.</span>original_id<span class="token punctuation">]</span> <span class="token operator">=</span> new_hha<span class="token operator">-&gt;</span>replacements<span class="token punctuation">[</span>i<span class="token punctuation">]</span><span class="token punctuation">.</span>replacement_id<span class="token punctuation">;</span>

     此时，
     <code>
      asset_replacement_table
     </code>
     记录的是
     <strong>
      旧资源ID
     </strong>
     到
     <strong>
      新资源ID
     </strong>
     的映射关系。
    </p>
    <h5>
     <a id="3__793">
     </a>
     <strong>
      3. 在查询资源时优先检查替换表
     </strong>
    </h5>
    <p>
     每当通过ID获取资源时 (
     <code>
      get_bitmap_by_id
     </code>
     )，先检查替换表：
    </p>
    <pre><code class="prism language-c"><span class="token keyword">int</span> real_asset_id <span class="token operator">=</span> asset_id<span class="token punctuation">;</span>
<span class="token keyword">if</span> <span class="token punctuation">(</span>asset_replacement_table<span class="token punctuation">.</span><span class="token function">contains</span><span class="token punctuation">(</span>asset_id<span class="token punctuation">)</span><span class="token punctuation">)</span>
<span class="token punctuation">{<!-- --></span>
    real_asset_id <span class="token operator">=</span> asset_replacement_table<span class="token punctuation">[</span>asset_id<span class="token punctuation">]</span><span class="token punctuation">;</span>
<span class="token punctuation">}</span>

</code></pre>
    <p>
     这样，无论是
     <strong>
      通过ID
     </strong>
     还是
     <strong>
      通过标签匹配
     </strong>
     ，都会
     <strong>
      优先加载新资源
     </strong>
     ，彻底解决资源替换问题。
    </p>

     <a id="_807">

     </strong>
    </h4>
    <p>
     我们还可以允许在新HHA文件内
     <strong>
      完全禁用旧资源
     </strong>
     ，即不提供替换资源，而是直接禁用旧资源。这种情况适用于：
    </p>
    <ul>
     <li>
      <strong>
       移除错误资源
      </strong>
      。
     </li>
     <li>
      <strong>
       删除某些不再需要的资源
      </strong>
      。
     </li>
    </ul>
    <p>
     <strong>
      实现方式
     </strong>
     ：
    </p>
    <ul>
     <li>
      在
      <code>
       Asset Replacement Table
      </code>
      中，只提供
      <code>
       Original File ID
      </code>
      而不提供
      <code>
       Replacement File ID
      </code>
      ，即可标记为禁用：
     </li>
    </ul>
    <pre><code class="prism language-plaintext">Asset Replacement Table:
{
    Original File ID: 1234,  // 要禁用的资源ID
    Replacement File ID: 0   // 无替代资源，直接禁用
}
</code></pre>
    <ul>
     <li>
      在加载过程中检测到
      <code>
       Replacement File ID = 0
      </code>
      ，直接跳过该资源。
     </li>
    </ul>

    <h2>
     <a id="_834">
     </a>
     启动时，怎么去除角色的初始闪烁？
    </h2>
    <p>
     目前我们的
     <strong>
      资源加载系统
     </strong>
     存在一个明显的问题：
     <strong>
      在游戏启动或遇到新资源时，资源的首次加载会导致屏幕闪烁
     </strong>
     。这是由于
     <strong>
      资源加载系统
     </strong>
     的工作机制决定的：
    </p>
    <h4>
     <a id="_837">
     </a>
     <strong>
      当前资源加载的机制
     </strong>
    </h4>
    <ol>
     <li>
      <strong>
       资源不会提前加载
      </strong>
      ：当前系统不会在游戏启动时主动加载任何资源，而是
      <strong>
       等到资源第一次被使用时
      </strong>
      ，才会通过线程加载。
     </li>
     <li>
      <strong>
       线程加载有延迟
      </strong>
      ：在首次使用资源时，系统会将加载任务提交给线程
      <strong>
       异步加载
      </strong>
      ，而此时资源尚未可用，因此会导致
      <strong>
       资源未加载完成时屏幕出现闪烁
      </strong>
      的情况。
     </li>
     <li>
      <strong>
       遇到新资源就会闪烁
      </strong>
      ：每次角色进入一个新场景，遇到新怪物、新物品、新特效等时，如果这些资源之前没有被加载过，就会出现短暂的
      <strong>
       空白或错误显示
      </strong>
      ，直到加载完成。
     </li>
    </ol>
    <hr/>
    <h4>
     <a id="PreCaching_844">
     </a>
     <strong>
      解决方案：引入资源预加载（Pre-Caching）
     </strong>
    </h4>
    <p>
     为了解决
     <strong>
      首次使用资源导致的闪烁问题
     </strong>
     ，我们计划
     <strong>
      增加一个资源预加载机制
     </strong>
     ，即在
     <strong>
      游戏启动或进入新场景
     </strong>
     时，提前将需要的资源加载到内存中，从而避免资源加载延迟带来的视觉闪烁。
    </p>
    <hr/>
    <h4>
     <a id="_849">
     </a>
     <strong>
      预加载的基本策略
     </strong>
    </h4>
    <p>
     我们不打算采用非常复杂的全量预加载（因为这会导致长时间的加载过程和占用大量内存），而是通过
     <strong>
      基于视野的预加载
     </strong>
     策略，以
     <strong>
      最小化加载延迟
     </strong>
     的方式保证资源可用性。
    </p>
    <h5>
     <a id="1__852">
     </a>
     1.
     <strong>
      基于屏幕视野的资源预加载
     </strong>
    </h5>
    <ul>
     <li>
      当角色进入新的场景时，我们不仅加载
      <strong>
       当前屏幕内可见的资源
      </strong>
      ，还会加载
      <strong>
       屏幕周围的一定范围内的资源
      </strong>
      ，这样在角色移动时，新资源几乎已经加载完成，避免出现闪烁。
     </li>
     <li>
      比如：
      <ul>
       <li>
        <strong>
         当前屏幕
        </strong>
        ：加载所有可见资源。
       </li>
       <li>
        <strong>
         屏幕外边缘
        </strong>
        ：提前加载一定范围内的资源。
       </li>
       <li>
        <strong>
         一定缓冲区
        </strong>
        ：加载一些资源（如常用特效、角色动画等），确保在短时间内可用。
       </li>
      </ul>
     </li>
     <li>
      这样，即使角色快速移动，也不会出现
      <strong>
       资源未加载完成的空白或错误贴图
      </strong>
      。
     </li>
    </ul>
    <h5>
     <a id="_860">
     </a>
     <strong>
      伪代码示意
     </strong>
    </h5>
    <pre><code class="prism language-c"><span class="token keyword">void</span> <span class="token function">PrecacheAssets</span><span class="token punctuation">(</span>WorldRegion <span class="token operator">*</span>region<span class="token punctuation">)</span>

    <p>
     该方案通过在
     <strong>
      角色进入新区域时
     </strong>

     <strong>

     </strong>

     <strong>
      视觉闪烁
     </strong>

    </p>

     <a id="2__882">
     </a>
     <strong>
      2. 基于场景上下文的资源预加载
     </strong>
    </h4>
    <p>
     除了基于视野的预加载外，还可以通过
     <strong>
      场景上下文
     </strong>
     进行资源预加载。例如：
    </p>
    <ul>
     <li>
      <strong>
       角色进入村庄
      </strong>
      时，提前加载村民的贴图、背景音乐、交互物品等。
     </li>
     <li>
      <strong>
       进入地下城
      </strong>
      时，提前加载怪物、战斗特效、地下城背景等。
     </li>
     <li>
      <strong>
       进入BOSS战
      </strong>
      时，提前加载BOSS模型、BOSS技能特效、战斗音乐等。
     </li>
    </ul>
    <p>
     这种
     <strong>
      场景上下文预加载
     </strong>
     可以极大减少重要场景的
     <strong>
      资源加载延迟
     </strong>
     ，让玩家感受更加流畅。
    </p>
    <h5>
     <a id="_890">

     <strong>

     <strong>
      3. 关键角色、关键资源的强制预加载

    </h4>
    <p>
     对于某些资源，比如：
    </p>
    <ul>
     <li>
      <strong>
       主角模型、主角动画
      </strong>
     </li>
     <li>
      <strong>
       主角技能特效
      </strong>
     </li>
     <li>
      <strong>

      </strong>

    </p>

    <h4>
     <a id="4__930">
     </a>
     <strong>
      4. 资源加载优先级机制
     </strong>
    </h4>
    <p>
     我们还可以设置
     <strong>
      资源加载优先级
     </strong>
     ，确保
     <strong>
      高优先级资源
     </strong>
     （如角色模型、UI等）优先加载，而
     <strong>
      低优先级资源
     </strong>
     （如装饰物、背景音乐等）可以稍后加载：
    </p>
    <ul>
     <li>
      <strong>
       高优先级
      </strong>
      ：角色模型、动画、UI界面、主线任务物品。
     </li>
     <li>
      <strong>
       中优先级
      </strong>
      ：场景背景、环境音效、次要NPC模型。
     </li>
     <li>
      <strong>
       低优先级
      </strong>
      ：装饰物品、非关键场景资源、稀有出现的怪物模型等。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="5__938">
     </a>
     <strong>
      5. 异步加载和同步加载的结合
     </strong>
    </h4>
    <p>
     我们仍然保持
     <strong>
      异步加载
     </strong>
     的方式，但在
     <strong>
      特定场合
     </strong>
     下（如进入场景、加载主角等），可以采用
     <strong>
      同步加载
     </strong>
     的策略：
    </p>
    <ul>
     <li>
      <strong>
       同步加载
      </strong>
      ：进入新场景时，强制加载场景资源，确保无闪烁。
     </li>
     <li>
      <strong>
       异步加载
      </strong>
      ：其他非关键资源，继续保持线程加载，确保不卡住主线程。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="_945">
     </a>
     <strong>
      预加载系统的内存消耗控制
     </strong>
    </h4>
    <p>
     预加载会增加内存占用，因此我们还需要设计
     <strong>
      缓存回收机制
     </strong>
     ：
    </p>
    <ul>
     <li>
      <strong>
       最近未使用的资源
      </strong>
      优先被释放。
     </li>
     <li>
      <strong>
       当前区域资源
      </strong>
      持续保留。
     </li>
     <li>
      <strong>
       核心资源
      </strong>
      不释放。
     </li>
    </ul>
    <p>

     </strong>

    <p>

    </p>

      </strong>
      。
     </li>
     <li>
      <strong>
       在角色快速移动时，保证资源已经提前加载，避免延迟
      </strong>
      。
     </li>
     <li>
      <strong>
       进入BOSS战或重要场景时，确保资源无卡顿加载
      </strong>
      。
     </li>
     <li>
      <strong>
       确保内存使用合理，避免加载过多资源导致内存溢出
      </strong>
      。
     </li>
    </ol>

    <h2>
     <a id="_986">
     </a>
     作为一个开发简单图形游戏的程序员，如果你不认识一个好的数字艺术家，有什么建议去寻找别人来委托艺术资产或类似的工作吗？
    </h2>
    <p>
     在开发一款具有相对简单图形风格的游戏时，如果需要
     <strong>
      委托他人绘制美术资源
     </strong>
     ，但又
     <strong>
      不认识合适的数字艺术家
     </strong>
     ，寻找合适的艺术家或美术资源的过程确实是一个难题。
    </p>
    <p>
     目前来说，我们可以考虑以下几种方法来
     <strong>
      寻找适合的美术资源或艺术家
     </strong>
     ：
    </p>
    <hr/>
    <h4>
     <a id="_1__993">
     </a>
     ✅
     <strong>
      1. 使用专门的美术外包平台
     </strong>
    </h4>
    <p>
     目前有很多
     <strong>
      专门用于游戏美术外包
     </strong>
     的平台或网站，可以直接找到
     <strong>
      适合自己项目的数字艺术家
     </strong>
     或
     <strong>
      购买现成的美术资源
     </strong>
     。
    </p>
    <h5>
     <a id="__996">
     </a>
     📌 常用的美术外包平台
    </h5>
    <ol>
     <li>
      <strong>
       Fiverr
      </strong>
      ：这是一个非常流行的自由职业平台，里面有大量的
      <strong>
       像素风、手绘风、卡通风
      </strong>
      等艺术家可供选择，价格从
      <strong>
       几美元到几百美元不等
      </strong>
      。
     </li>
     <li>
      <strong>
       Upwork
      </strong>
      ：这是一个更偏向于专业人才的平台，可以找到
      <strong>
       更具专业能力的游戏美术设计师
      </strong>
      ，适合预算稍微充足的项目。
     </li>
     <li>
      <strong>
       ArtStation
      </strong>
      ：如果想直接找到专业的
      <strong>
       游戏原画设计师
      </strong>
      ，ArtStation是非常理想的选择，这里聚集了大量专业游戏美术人员。可以直接联系他们
      <strong>
       进行私下合作
      </strong>
      或委托。
     </li>
     <li>
      <strong>
       DeviantArt
      </strong>
      ：DeviantArt是一个全球知名的艺术家社区，里面有大量的独立艺术家和插画师。可以浏览作品并联系作者进行
      <strong>
       定制化合作
      </strong>
      。
     </li>
    </ol>
    <h5>
     <a id="__1002">
     </a>
     📜
     <strong>
      寻找艺术家的策略
     </strong>
    </h5>
    <ul>
     <li>
      <strong>
       如果预算有限
      </strong>
      ：优先选择Fiverr或DeviantArt，成本较低且可控。
     </li>
     <li>
      <strong>
       如果追求专业美术
      </strong>
      ：优先选择ArtStation或Upwork，能够找到具备
      <strong>
       游戏美术设计经验
      </strong>
      的专业人才，但价格相对较高。
     </li>
     <li>
      <strong>
       如果需要现成资源
      </strong>
      ：可以在
      <strong>
       Unity Asset Store
      </strong>
      、
      <strong>
       itch.io
      </strong>
      等平台上直接购买美术资源，节省时间。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="_2__1009">
     </a>
     ✅
     <strong>
      2. 使用素材市场直接购买资源
     </strong>
    </h4>
    <p>
     如果不想直接委托艺术家，也可以考虑
     <strong>
      直接购买现成的美术资源
     </strong>
     ，例如角色、场景、UI等素材。
    </p>
    <h5>
     <a id="__1012">
     </a>
     📌 常见的素材市场
    </h5>
    <ol>
     <li>
      <strong>
       Unity Asset Store
      </strong>
      ：如果使用Unity引擎开发，可以直接在Unity官方商店购买各种
      <strong>
       美术素材
      </strong>
      ，包括
      <strong>
       2D/3D模型、UI资源、特效等
      </strong>
      。
     </li>
     <li>
      <strong>
       itch.io Asset Store
      </strong>
      ：itch.io是一个非常受欢迎的
      <strong>
       独立游戏开发者社区
      </strong>
      ，同时提供
      <strong>
       大量的免费或收费游戏资源
      </strong>
      ，价格非常便宜，甚至有很多免费资源可用。
     </li>
     <li>
      <strong>
       OpenGameArt
      </strong>
      ：这是一个专注于
      <strong>
       开源游戏美术资源
      </strong>
      的网站，所有资源都可以免费使用，适合预算非常有限的开发者。
     </li>
    </ol>
    <h5>
     <a id="__1017">
     </a>
     📜
     <strong>
      优先购买哪些资源
     </strong>
    </h5>
    <ul>
     <li>
      <strong>
       角色设计
      </strong>
      ：如果需要简单的角色，可以在这些平台找到
      <strong>
       2D像素角色、卡通风角色、RPG角色等
      </strong>
      资源。
     </li>
     <li>
      <strong>
       UI界面
      </strong>
      ：游戏的UI界面通常可以直接使用现成的UI Kit，节省大量设计时间。
     </li>
     <li>
      <strong>
       背景资源
      </strong>
      ：如果是2D游戏，可以直接购买像素风背景或者平台背景，避免自己制作。
     </li>
    </ul>

      <strong>

     </li>
     <li>
      预算范围。
     </li>
     <li>
      项目风格参考。
     </li>
    </ul>

     例如：
    </p>
    <pre><code>我们正在开发一款2D像素风RPG游戏，现需要1名美术设计师，负责：
- 制作主角和敌人的像素角色。
- 设计游戏内的UI界面。

     <li>
      <strong>

      确保所有资源都有合法授权，避免版权纠纷。
     </li>
    </ul>
    <p>
     这样可以
     <strong>

    <p>

     <strong>
      重要的工作
     </strong>
     需要处理，主要集中在
     <strong>
      音频资源的加载统一
     </strong>
     以及
     <strong>
      资源内存管理

     这两个方面。
    </p>
    <hr/>
    <h4>
     <a id="__1119">
     </a>
     ✅
     <strong>
      一、将音频加载逻辑统一合并
     </strong>
    </h4>
    <p>
     目前在我们的资源系统中，
     <strong>
      音频（Audio）
      <strong>
       和
      </strong>
      位图（Bitmap）
      <strong>
       的加载是通过
      </strong>
      两个不同的回调函数
     </strong>
     进行的，即：
    </p>
    <ul>
     <li>
      <strong>
       位图资源
      </strong>
      使用
      <code>
       LoadAsset
      </code>
      函数通过工作队列回调加载；
     </li>
     <li>
      <strong>
       音频资源
      </strong>
      使用
      <code>
       LoadSound
      </code>
      函数通过另一个工作队列回调加载。
     </li>
    </ul>
    <p>
     但是，这种
     <strong>
      分离的加载方式
     </strong>
     存在一定的问题：
    </p>
    <ol>
     <li>
      <strong>
       代码冗余
      </strong>
      ：两个回调函数的逻辑实际上非常相似，分别处理资源加载，但没有共用相同的逻辑。
     </li>
     <li>
      <strong>
       资源系统的不一致性
      </strong>
      ：所有的资源都应当通过同一个通道加载，这样可以确保资源加载逻辑更加一致，便于未来扩展。
     </li>
     <li>
      <strong>
       维护成本增加
      </strong>
      ：如果未来要修改资源加载逻辑，需要修改两个回调函数，增加了开发和维护成本。
     </li>
    </ol>
    <h5>
     <a id="__1129">
     </a>
     ✅
     <strong>
      解决方案
     </strong>
    </h5>
    <p>
     我们计划将
     <strong>
      音频资源加载逻辑
     </strong>
     直接合并到**
     <code>
      LoadAsset
     </code>
     **回调中，取消
     <code>
      LoadSound
     </code>
     的专用回调。
    </p>
    <ul>
     <li>
      <strong>
       即：位图和音频资源统一通过同一个工作队列回调进行加载
      </strong>
      ；
     </li>
     <li>
      <strong>
       不再为音频资源单独设置回调函数
      </strong>
      ，而是直接通过
      <code>
       LoadAsset
      </code>
      函数进行加载。
     </li>
    </ul>
    <p>
     这样做的
     <strong>
      好处
     </strong>
     ：
    </p>
    <ul>
     <li>
      ✅
      <strong>
       简化代码结构
      </strong>
      ，只需维护一个资源加载回调；
     </li>
     <li>
      ✅
      <strong>
       减少维护成本
      </strong>
      ，未来调整资源加载逻辑时只需修改一处；
     </li>
     <li>
      ✅
      <strong>
       提升加载一致性
      </strong>
      ，避免资源加载过程中的差异性问题。
     </li>
    </ul>
    <p>
     我们下一步要做的就是
     <strong>
      重构音频加载逻辑
     </strong>
     ，让音频资源与位图资源共用同一条加载管道，确保
     <strong>
      资源加载流程
     </strong>
     的一致性。
    </p>
    <hr/>
    <h4>
     <a id="__Win32__1143">
     </a>
     ✅
     <strong>

    </h4>
    <p>

     <strong>

     <strong>
      完善一些底层支持
     </strong>

     <li>
      <strong>

      </strong>
      ，让平台层可以无差别地加载资源。

    <p>

     <strong>

    </p>

    <h4>
     <a id="__1154">
     </a>
     ✅
     <strong>
      三、解决超大资源占用导致的内存溢出问题
     </strong>
    </h4>
    <p>
     完成资源加载逻辑的统一后，我们需要开始解决一个
     <strong>
      更加棘手的问题
     </strong>
     ：
    </p>
    <blockquote>
     <p>
      <strong>
       如果资源体积过大，超出内存限制时该如何处理？
      </strong>
     </p>
    </blockquote>
    <h5>
     <a id="__1158">
     </a>
     📌
     <strong>
      问题描述
     </strong>
    </h5>
    <p>
     在大型游戏开发中，游戏的
     <strong>
      资源文件
     </strong>
     （如音频、位图、模型等）的总大小
     <strong>
      通常会远超设备内存
     </strong>
     。
     <br/>
     例如：
    </p>
    <ul>
     <li>
      假设玩家的设备物理内存只有
      <strong>
       2GB
      </strong>
      ；
     </li>
     <li>
      而我们的游戏资源体积达到
      <strong>
       4GB
      </strong>
      ；
     </li>
     <li>
      此时游戏加载过多资源，将会导致
      <strong>
       内存溢出
      </strong>
      ，引发游戏崩溃。
     </li>
    </ul>
    <p>
     目前我们还没有实现资源溢出后的内存管理机制，因此
     <strong>
      这将是接下来最难的工作
     </strong>
     。
    </p>
    <hr/>
    <h4>
     <a id="__1169">
     </a>
     ✅
     <strong>
      四、设计资源内存管理系统
     </strong>
    </h4>
    <p>
     我们需要
     <strong>
      设计一个类似虚拟内存的资源管理系统
     </strong>
     ，确保在
     <strong>
      内存不足时
     </strong>
     ，能够自动卸载不需要的资源，从而腾出内存空间加载新的资源。
    </p>
    <h5>
     <a id="__1172">
     </a>
     📜
     <strong>
      设计核心思路
     </strong>
    </h5>
    <ol>
     <li>
      <strong>
       内存占用限制
      </strong>
      ：我们需要设定一个内存占用阈值，例如：
      <strong>
       1.5GB
      </strong>
      ，超过这个值后就主动卸载一部分资源。
     </li>
     <li>
      <strong>
       资源优先级
      </strong>
      ：我们需要给资源
      <strong>
       分配优先级
      </strong>
      ，优先加载
      <strong>
       当前屏幕内可见资源
      </strong>
      ，延迟加载
      <strong>
       屏幕外资源
      </strong>
      。
     </li>
     <li>
      <strong>
       资源替换策略
      </strong>
      ：当内存即将耗尽时，主动卸载
      <strong>
       一段时间未使用的资源
      </strong>
      ，为新资源腾出空间。
     </li>
     <li>
      <strong>
       缓存策略
      </strong>
      ：尽量缓存
      <strong>
       高使用率资源
      </strong>
      ，避免反复加载相同资源导致的性能消耗。
     </li>
    </ol>
    <hr/>
    <h4>
     <a id="__1180">
     </a>
     ✅
     <strong>
      五、实现内存管理的技术方案
     </strong>
    </h4>
    <p>
     我们计划实现一个
     <strong>
      类似虚拟内存的资源管理机制
     </strong>
     ，主要分为以下几个步骤：
    </p>
    <h5>
     <a id="_1__1183">
     </a>

    </h5>

     <strong>

     </strong>

     <strong>
      加载到内存中的资源
     </strong>
     。

     缓存表格式大致如下：
    </p>
    <table>
     <thead>
      <tr>

       </th>

       </th>
       <th>
        加载时间戳
       </th>
       <th>
        访问次数
       </th>
       <th>
        是否可丢弃
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>

       </td>

       </td>

       <td>
        56
       </td>
       <td>
        ❌
       </td>
      </tr>
      <tr>
       <td>

       </td>
       <td>
        音频
       </td>
       <td>
        12:31:10
       </td>
       <td>
        10

       <td>
        ✅
       </td>
      </tr>

       </td>

       </td>
       <td>
        12:35:00
       </td>
       <td>
        4
       </td>
       <td>
        ✅
       </td>
      </tr>
     </tbody>
    </table>
    <p>
     我们通过以下策略管理内存：
    </p>
    <ul>
     <li>
      <strong>
       优先保留高访问率的资源
      </strong>
      ；
     </li>
     <li>
      <strong>
       优先卸载长时间未访问的资源
      </strong>
      ；
     </li>
     <li>
      <strong>
       设置内存阈值
      </strong>
      ，保证不超出设备内存。
     </li>
    </ul>
    <hr/>
    <h5>
     <a id="_2__1200">
     </a>
     📌 2. 资源丢弃策略
    </h5>
    <p>
     当内存占用
     <strong>
      接近阈值
     </strong>
     时，我们需要丢弃一些
     <strong>
      长时间未使用
     </strong>
     的资源。
    </p>
    <p>
     丢弃策略：
    </p>
    <ul>
     <li>
      <strong>
       优先丢弃低优先级资源
      </strong>
      ；
     </li>
     <li>
      <strong>
       优先丢弃长时间未访问资源
      </strong>
      ；
     </li>
     <li>
      <strong>
       避免丢弃主角、UI等核心资源
      </strong>
      。
     </li>
    </ul>
    <p>
     例如：
    </p>
    <ul>
     <li>
      当内存占用达到1.5GB时，丢弃30分钟未使用的资源；
     </li>
     <li>
      当内存占用达到1.8GB时，强制丢弃低优先级资源；
     </li>
    </ul>
    <hr/>
    <h5>
     <a id="_3__1214">
     </a>
     📌 3. 自动内存回收
    </h5>
    <p>
     我们需要实现一个
     <strong>
      内存回收线程
     </strong>
     ，定期检查内存占用：
    </p>
    <ul>
     <li>
      <strong>
       内存不足时
      </strong>
      ，自动丢弃低优先级资源；
     </li>
     <li>
      <strong>
       内存充足时
      </strong>
      ，允许保留更多资源；
     </li>
     <li>
      <strong>
       持续监控
      </strong>
      内存状态，动态调整资源缓存大小。
     </li>
    </ul>

     <strong>

     </strong>
    </h5>

      </strong>

     </li>
     <li>

      <strong>

      </strong>
      ；
     </li>
    </ul>
    <h5>
     <a id="__1303">
     </a>

     <strong>
      资源回收策略
     </strong>
    </h5>
    <ul>
     <li>

      <strong>
       长时间未使用
      </strong>

       低优先级
      </strong>

      <strong>

      </strong>

     </strong>
    </h5>
    <ul>
     <li>

      <strong>
       内存管理线程

     <li>

      <strong>

      </strong>

     </li>

     <strong>

     </strong>
    </h5>
    <ul>
     <li>

      <strong>

      <strong>

      </strong>

     </li>
     <li>
      确保内存使用始终处于
      <strong>
       可控状态
      </strong>
      。
     </li>
    </ul>
