    <p>
     Git的分支（Branch）是
     <a href="https://so.csdn.net/so/search?q=Git%E7%89%88%E6%9C%AC%E6%8E%A7%E5%88%B6&amp;spm=1001.2101.3001.7020" title="Git版本控制">
      Git版本控制
     </a>
     系统中的一个核心概念，允许创建代码的多个并行版本，使得团队成员可以在不影响主开发线路的情况下进行功能开发、错误修复或实验性尝试。
    </p>
    <p>
     每次提交，Git都把它们串成一条
     <a href="https://so.csdn.net/so/search?q=%E6%97%B6%E9%97%B4%E7%BA%BF&amp;spm=1001.2101.3001.7020" title="时间线">
      时间线
     </a>
     ，这条时间线就是一个分支。截止 到目前，只有一条时间线，在Git里，这个分支叫主分支，即master分支。
     <code>
      master
     </code>
     是指向提交的，而
     <code>
      HEAD
     </code>
     指向的是当前分支的最新提交。
    </p>
    <p>
     开始的时候，master分支是一条线，Git 用 master 指向最新的提交，再用 HEAD 指向 master，就能确定当前分支，以及当前分支的提交点。每次提 交，master分支都会向前移动一步，这样，随着你不断提交，
     <strong>
      master分支 的线
     </strong>
     也越来越长。
    </p>
    <p>
     <img alt="" height="342" src="https://i-blog.csdnimg.cn/direct/f5850870db6d476292a80df06ce17838.png" width="784"/>
    </p>
    <p>
     分支管理:
     <code>
      GitLab
     </code>
     支持基于分支的开发工作流。在
     <code>
      Repository” &gt; “Branches
     </code>
     中，可以创建新分支、设置保护分支、查看分支活动等。
    </p>
    <h4 id="%E5%88%86%E6%94%AF%E7%9A%84%E6%96%B0%E5%BB%BA%E4%B8%8E%E5%90%88%E5%B9%B6" name="%E5%88%86%E6%94%AF%E7%9A%84%E6%96%B0%E5%BB%BA%E4%B8%8E%E5%90%88%E5%B9%B6">
     分支的新建与合并
    </h4>
    <p>
     让我们来看一个简单的分支新建与分支合并的例子，实际工作中你可能会用到如下步骤：
    </p>
    <ol>
     <li>
      <p>
       开发某个网站。为实现某个新的用户需求，创建一个分支。在这个分支上开展工作。
      </p>
     </li>
    </ol>
    <p>
     正在此时，你突然接到一个电话说有个很严重的问题需要紧急修补。 你将按照如下方式来处理：
    </p>
    <ol>
     <li>
      <p>
       切换到你的线上分支（production branch）。
      </p>
     </li>
     <li>
      <p>
       为这个紧急任务新建一个分支，并在其中修复它。
      </p>
     </li>
     <li>
      <p>
       在测试通过之后，切换回线上分支，然后合并这个修补分支，最后将改动推送到线上分支。
      </p>
     </li>
     <li>
      <p>
       切换回你最初工作的分支上，继续工作。
      </p>
     </li>
    </ol>
    <h5 id="%E6%96%B0%E5%BB%BA%E5%88%86%E6%94%AF" name="%E6%96%B0%E5%BB%BA%E5%88%86%E6%94%AF">
     新建分支
    </h5>
    <p>
     假设你正在项目上工作，并且在
     <code>
      master
     </code>
     分支上已经有了一些提交。
    </p>
    <p>
     <img alt="" height="259" src="https://i-blog.csdnimg.cn/direct/44b35ccb61654c619b02bc41dddbcecf.png" width="1043"/>
    </p>
    <h6 id="Figure%2018.%20%E4%B8%80%E4%B8%AA%E7%AE%80%E5%8D%95%E6%8F%90%E4%BA%A4%E5%8E%86%E5%8F%B2" name="Figure%2018.%20%E4%B8%80%E4%B8%AA%E7%AE%80%E5%8D%95%E6%8F%90%E4%BA%A4%E5%8E%86%E5%8F%B2">
     Figure 18. 一个简单提交历史
    </h6>
    <p>
     你已经决定要解决公司使用的问题追踪系统中的 #53 问题。 想要新建一个分支并同时切换到那个分支上，你可以运行
     <code>
      git checkout
     </code>
     命令
    </p>
    <pre><code class="hljs">$ git branch iss53  #创建一个新的分支
$ git checkout iss53  # 切换到该分支

$ git branch         # 查看当前分支
  * iss53      # 哪个分支前面有*号，就在那个分支
  master</code></pre>
    <p>
     <img alt="" height="596" src="https://i-blog.csdnimg.cn/direct/0ace2848a312454b85523272a4874fd7.png" width="1305"/>
    </p>
    <h6 id="Figure%2019.%20%E5%88%9B%E5%BB%BA%E4%B8%80%E4%B8%AA%E6%96%B0%E5%88%86%E6%94%AF%E6%8C%87%E9%92%88" name="Figure%2019.%20%E5%88%9B%E5%BB%BA%E4%B8%80%E4%B8%AA%E6%96%B0%E5%88%86%E6%94%AF%E6%8C%87%E9%92%88">
     Figure 19. 创建一个新分支指针
    </h6>
    <p>
     你继续在 #53 问题上工作，并且做了一些提交。 在此过程中，
     <code>
      iss53
     </code>
     分支在不断的向前推进，因为你已经检出
     <code>
      checkout
     </code>
     到该分支 （即你的
     <code>
      HEAD
     </code>
     指针指向了
     <code>
      iss53
     </code>
     分支）
    </p>
    <pre><code class="hljs">$ vim index.html
$ git commit -a -m 'added a new footer [issue 53]'   #新增一个错误</code></pre>
    <p>
     <img alt="" height="453" src="https://i-blog.csdnimg.cn/direct/af64c51de6a548feb1188b6f978168b9.png" width="1329"/>
    </p>
    <h6 id="Figure%2020.%20iss53%20%E5%88%86%E6%94%AF%E7%9A%84%E7%8B%AC%E7%AB%8B%E6%BC%94%E8%BF%9B" name="Figure%2020.%20iss53%20%E5%88%86%E6%94%AF%E7%9A%84%E7%8B%AC%E7%AB%8B%E6%BC%94%E8%BF%9B">
     Figure 20.
     <code>
      iss53
     </code>
     分支的独立演进
    </h6>
    <p>
     有了 Git 的帮助，你不必把这个紧急问题和
     <code>
      iss53
     </code>
     的修改混在一起， 也不需要花大力气来还原关于 53# 问题的修改，然后再添加关于这个紧急问题的修改，最后将这个修改提交到线上分支。
     <strong>
      你所要做的仅仅是切换回
      <code>
       master
      </code>
      分支。
     </strong>
    </p>
    <p>
     但如果你的工作目录或暂存区存在未提交的修改时，直接切换分支可能导致以下问题：
    </p>
    <ol>
     <li>
      <p>
       <strong>
        冲突风险
       </strong>
       ：目标分支可能包含与当前修改冲突的文件，Git 会阻止切换以避免数据丢失。
      </p>
     </li>
     <li>
      <p>
       <strong>
        污染分支
       </strong>
       ：未提交的修改会被带到新分支，可能破坏目标分支的代码状态
      </p>
     </li>
    </ol>
    <blockquote>
     <p>
      最好的方法是，在你切换分支之前，保持好一个干净的状态。 有一些方法可以绕过这个问题
     </p>
    </blockquote>
    <ol>
     <li>
      <p>
       <strong>
        贮藏（stashing）
       </strong>
      </p>
     </li>
    </ol>
    <pre><code class="hljs"># 将iss53分支的未提交修改（包括已暂存和未暂存的文件）保存到一个独立于分支的“贮藏栈”中，并将工作目录恢复到 iss53 分支最后一次提交的状态
git stash save "描述信息"
贮藏内容与分支无关，可以跨分支恢复。例如，你可以切换到 master 分支后应用贮藏，但此时修改的内容会应用到 master 分支的当前工作目录中

# 切换到目标分支
git checkout master
# 恢复贮藏的修改（保留贮藏记录）
git stash apply
# 或恢复并删除贮藏记录
git stash pop</code></pre>
    <p>
     2.
     <strong style="color:#4d4d4d; font-size:16px">
      修补提交（commit amending）
     </strong>
    </p>
    <pre><code class="hljs"># 将暂存区修改合并到上一次提交
git add .
git commit --amend -m "新的提交信息"     #其实就是提交到仓库

# 切换到目标分支
git checkout target-branch</code></pre>
    <p>
     现在，我们假设你已经把你的修改全部提交了，这时你可以切换回
     <code>
      master
     </code>
     分支了：
     <code>
      git checkout master
     </code>
    </p>
    <p>
     这个时候，你的工作目录和你在开始 #53 问题之前一模一样，现在你可以专心修复紧急问题了。 请牢记：当你切换分支的时候，Git 会重置你的工作目录，使其看起来像回到了你在那个分支上最后一次提交的样子。
    </p>
    <p>
     接下来，你要修复这个紧急问题。 我们又建立一个
     <code>
      hotfix
     </code>
     分支，然后在这个分支上把问题解决
    </p>
    <pre><code class="hljs">$ git branch hotfix  #创建一个新的分支
$ git checkout hotfix  # 切换到该分支

$ vim index.html
$ git commit -a -m 'fixed the broken email address'    #修复这个错误</code></pre>
    <p>
     <img alt="" height="603" src="https://i-blog.csdnimg.cn/direct/4837eeca264c4f208b63b8cd735a6f94.png" width="1312"/>
    </p>
    <h6 id="Figure%2021.%20%E5%9F%BA%E4%BA%8E%20master%20%E5%88%86%E6%94%AF%E7%9A%84%E7%B4%A7%E6%80%A5%E9%97%AE%E9%A2%98%E5%88%86%E6%94%AF%20hotfix%20branch" name="Figure%2021.%20%E5%9F%BA%E4%BA%8E%20master%20%E5%88%86%E6%94%AF%E7%9A%84%E7%B4%A7%E6%80%A5%E9%97%AE%E9%A2%98%E5%88%86%E6%94%AF%20hotfix%20branch">
     Figure 21. 基于
     <code>
      master
     </code>
     分支的紧急问题分支
     <code>
      hotfix branch
     </code>
    </h6>
    <p>
     运行测试，确保修改是正确的，然后将
     <code>
      hotfix
     </code>
     分支合并回你的
     <code>
      master
     </code>
     分支来部署到线上。 你可以使用
     <code>
      git merge
     </code>
     命令来达到上述目的：
    </p>
    <pre><code class="hljs">$ git checkout master   #切换到master 
$ git merge hotfix      #合并hotfix分支
​
Updating f42c576..3a0874c
Fast-forward
 index.html | 2 ++
 1 file changed, 2 insertions(+)</code></pre>
    <p>
     在合并的时候， 由于你想要合并的分支
     <code>
      hotfix
     </code>
     所指向的提交
     <code>
      C4
     </code>
     是你所在的提交
     <code>
      C2
     </code>
     的直接后继，
     <strong>
      因此 Git 会直接将指针向前移动
     </strong>
     。
    </p>
    <p>
     即合并两个分支时， 如果顺着一个分支走下去能够到达另一个分支，那么 Git 在合并两者的时候， 只会简单的将指针向前推进（指针右移），因为这种情况下的合并操作没有需要解决的分歧---这就是
     <strong>
      快进（fast-forward）
     </strong>
     这个词。
    </p>
    <p>
     现在，最新的修改已经在
     <code>
      master
     </code>
     分支所指向的提交快照中，你可以着手发布该修复了。
    </p>
    <p>
     <img alt="" height="765" src="https://i-blog.csdnimg.cn/direct/4ee9dc8923cb462ca9d57292a49f5a63.png" width="1318"/>
    </p>
    <h6 id="Figure%2022.%20master%20%E8%A2%AB%E5%BF%AB%E8%BF%9B%E5%88%B0%20hotfix" name="Figure%2022.%20master%20%E8%A2%AB%E5%BF%AB%E8%BF%9B%E5%88%B0%20hotfix">
     Figure 22.
     <code>
      master
     </code>
     被快进到
     <code>
      hotfix
     </code>
    </h6>
    <p>
     这个问题的解决发布之后，你准备回到被打断之前时的工作中。 这时应该先删除
     <code>
      hotfix
     </code>
     分支，因为你已经不再需要它了 ——
     <code>
      master
     </code>
     分支已经指向了同一个位置。 你可以使用带
     <code>
      -d
     </code>
     选项的
     <code>
      git branch
     </code>
     命令来删除分支：
    </p>
    <pre>$ git branch -d hotfix      #删除hotfix分支
Deleted branch hotfix (3a0874c).</pre>
    <p>
     现在就可以切换回正在工作的
     <code>
      iss53
     </code>
     分支
    </p>
    <pre><code class="hljs">$ git checkout iss53
Switched to branch "iss53"
$ vim index.html
$ git commit -a -m 'finished the new footer [issue 53]'  #完成这个错误修改
[iss53 ad82d7a] finished the new footer [issue 53]    
1 file changed, 1 insertion(+)</code></pre>
    <p>
     <img alt="" height="603" src="https://i-blog.csdnimg.cn/direct/bc97eb510fdb4ea0bf657a7ee1300895.png" width="1295"/>
    </p>
    <h6 id="Figure%2023.%20%E7%BB%A7%E7%BB%AD%20iss53%20%E5%88%86%E6%94%AF%E4%B8%8A%E7%9A%84%E5%B7%A5%E4%BD%9C" name="Figure%2023.%20%E7%BB%A7%E7%BB%AD%20iss53%20%E5%88%86%E6%94%AF%E4%B8%8A%E7%9A%84%E5%B7%A5%E4%BD%9C">
     Figure 23. 继续
     <code>
      iss53
     </code>
     分支上的工作
    </h6>
    <p>
     你在
     <code>
      hotfix
     </code>
     分支上所做的工作并没有包含到
     <code>
      iss53
     </code>
     分支中。 如果你需要拉取
     <code>
      hotfix
     </code>
     所做的修改，你可以使用
     <code>
      git merge master
     </code>
     命令将
     <code>
      master
     </code>
     分支合并入
     <code>
      iss53
     </code>
     分支，或者你也可以等到
     <code>
      iss53
     </code>
     分支完成其使命，再将其合并回
     <code>
      master
     </code>
     分支。
    </p>
    <h5 id="%E5%88%86%E6%94%AF%E7%9A%84%E5%90%88%E5%B9%B6" name="%E5%88%86%E6%94%AF%E7%9A%84%E5%90%88%E5%B9%B6">
     分支的合并
    </h5>
    <p>
     假设已经修正了 #53 问题，并且打算将
     <code>
      iss53
     </code>
     合并入
     <code>
      master
     </code>
     分支。 这和之前合并
     <code>
      hotfix
     </code>
     分支所做的工作差不多。
    </p>
    <pre><code class="hljs">$ git checkout master   #切换到master
Switched to branch 'master'

$ git merge iss53   # 合并iss53到master
Merge made by the 'recursive' strategy.
index.html |    1 +
1 file changed, 1 insertion(+)</code></pre>
    <ul>
     <li>
      <p>
       但这和之前合并
       <code>
        hotfix
       </code>
       分支有点不一样。因为两个分支并不是相邻的。 Git 不得不做一些额外的工作。
      </p>
     </li>
     <li>
      <p>
       出现这种情况的时候，Git 会使用两个分支的末端所指的快照（
       <code>
        C4
       </code>
       和
       <code>
        C5
       </code>
       ）以及这两个分支的
       <strong>
        公共祖先
       </strong>
       （
       <code>
        C2
       </code>
       ），做一个简单的三方合并。
      </p>
     </li>
    </ul>
    <p>
     <img alt="" height="599" src="https://i-blog.csdnimg.cn/direct/269904459b17464aa27c6ecc4154721d.png" width="1310"/>
    </p>
    <h6 id="Figure%2024.%20%E5%85%B8%E5%9E%8B%E5%90%88%E5%B9%B6%E4%B8%AD%E6%89%80%E7%94%A8%E5%88%B0%E7%9A%84%E4%B8%89%E4%B8%AA%E5%BF%AB%E7%85%A7" name="Figure%2024.%20%E5%85%B8%E5%9E%8B%E5%90%88%E5%B9%B6%E4%B8%AD%E6%89%80%E7%94%A8%E5%88%B0%E7%9A%84%E4%B8%89%E4%B8%AA%E5%BF%AB%E7%85%A7">
     Figure 24. 典型合并中所用到的三个快照
    </h6>
    <p>
     和之前将分支指针向前推进所不同的是，Git 将此次三方合并的结果为一个新的快照并且自动创建一个新的提交指向它。 这个被称作一次
     <strong>
      合并提交
     </strong>
     ，它的特别之处在于他有不止一个父提交。
    </p>
    <p>
     <img alt="" height="499" src="https://i-blog.csdnimg.cn/direct/ca88b263418a4b12aff5b5b79665e7d7.png" width="1303"/>
    </p>
    <h6 id="Figure%2025.%20%E4%B8%80%E4%B8%AA%E5%90%88%E5%B9%B6%E6%8F%90%E4%BA%A4" name="Figure%2025.%20%E4%B8%80%E4%B8%AA%E5%90%88%E5%B9%B6%E6%8F%90%E4%BA%A4">
     Figure 25. 一个合并提交
    </h6>
    <p>
     既然你的修改已经合并进来了，就不再需要
     <code>
      iss53
     </code>
     分支了。 现在你可以在任务追踪系统中关闭此项任务，并删除这个分支。
    </p>
    <pre>$ git branch -d iss53</pre>
    <blockquote>
     <p>
      遇到冲突时的分支合并
     </p>
    </blockquote>
    <p>
     有时候合并操作不会如此顺利。 如果两个不同的分支中，
     <strong>
      对同一个文件的同一个部分进行了不同的修改
     </strong>
     ，Git 就没法干净的合并它们。 如果你对 #53 问题的修改和有关
     <code>
      hotfix
     </code>
     分支的修改都涉及到同一个文件的同一处，在合并它们的时候就会产生合并冲突：
    </p>
    <pre><code class="hljs">$ git merge iss53
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.</code></pre>
    <ul>
     <li>
      <p>
       此时 Git 做了合并，但是没有自动地创建一个新的合并提交。 Git 会暂停下来，
       <strong>
        等待你去解决合并产生的冲突
       </strong>
       。
      </p>
     </li>
     <li>
      <p>
       你可以在合并冲突后的任意时刻使用
       <code>
        git status
       </code>
       命令来查看那些因包含合并冲突而处于未合并（unmerged）状态的文件
      </p>
     </li>
     <li>
      <p>
       任何因包含合并冲突而有待解决的文件，都会以未合并状态标识出来。
      </p>
     </li>
    </ul>
    <pre><code class="hljs">&lt;!--head分支，因为前面合并时已经切换到master分支了--&gt;
&lt;&lt;&lt;&lt;&lt;&lt;&lt; HEAD:index.html    
&lt;div id="footer"&gt;contact : email.support@github.com&lt;/div&gt;
=======
&lt;div id="footer"&gt;
 please contact us at support@github.com
&lt;/div&gt;
&lt;!--要合并的jss53分支--&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt; iss53:index.html</code></pre>
    <p>
     解决冲突也很简单，选择冲突部分，这两个其中的一个
    </p>
    <ul>
     <li>
      <p>
       等你退出合并工具之后，Git 会询问刚才的合并是否成功。 如果你回答是，Git 会暂存那些文件以表明冲突已解决
      </p>
     </li>
     <li>
      <p>
       可以再次运行
       <code>
        git status
       </code>
       来确认所有的合并冲突都已被解决：
      </p>
     </li>
    </ul>
    <pre><code class="hljs">$ git status
On branch master
All conflicts fixed but you are still merging.
  (use "git commit" to conclude merge)

Changes to be committed:

    modified:   index.html</code></pre>
    <p>
     如果对结果满意，并且确定了，这时可以输入
     <code>
      git commit
     </code>
     来完成合并提交。
    </p>
    <blockquote>
     <p>
      <a href="https://git-scm.com/book/zh/v2/Git-%E5%88%86%E6%94%AF-%E5%8F%98%E5%9F%BA" rel="nofollow" title="Git 分支 - 变基">
       Git 分支 - 变基
      </a>
     </p>
    </blockquote>
    <p>
     在 Git 中整合来自不同分支的修改主要有两种方法：
     <code>
      merge
     </code>
     以及
     <code>
      rebase
     </code>
     。
    </p>
    <h4 id="%E5%8F%98%E5%9F%BA%E7%9A%84%E5%9F%BA%E6%9C%AC%E6%93%8D%E4%BD%9C" name="%E5%8F%98%E5%9F%BA%E7%9A%84%E5%9F%BA%E6%9C%AC%E6%93%8D%E4%BD%9C">
     变基的基本操作
    </h4>
    <p>
     请回顾之前在
     <a href="https://git-scm.com/book/zh/v2/ch00/_basic_merging" rel="nofollow" title="分支的合并">
      分支的合并
     </a>
     中的一个例子，你会看到开发任务分叉到两个不同分支，又各自提交了更新。
    </p>
    <p>
     <img alt="" height="607" src="https://i-blog.csdnimg.cn/direct/f6370fb30c4c4812b4a4619d17510111.png" width="1299"/>
    </p>
    <h5 id="Figure%2035.%20%E5%88%86%E5%8F%89%E7%9A%84%E6%8F%90%E4%BA%A4%E5%8E%86%E5%8F%B2" name="Figure%2035.%20%E5%88%86%E5%8F%89%E7%9A%84%E6%8F%90%E4%BA%A4%E5%8E%86%E5%8F%B2">
     Figure 35. 分叉的提交历史
    </h5>
    <p>
     之前提到，合并分支的方法是
     <code>
      merge
     </code>
     命令。 它会把两个分支的最新快照（
     <code>
      C3
     </code>
     和
     <code>
      C4
     </code>
     ）以及二者最近的共同祖先（
     <code>
      C2
     </code>
     ）进行三方合并，合并的结果是生成一个新的快照（并提交）
    </p>
    <p>
     <img alt="" height="480" src="https://i-blog.csdnimg.cn/direct/b69d2acaf17f40acad71df7147ce8a6e.png" width="1293"/>
    </p>
    <h5 id="Figure%2036.%20%E9%80%9A%E8%BF%87%E5%90%88%E5%B9%B6%E6%93%8D%E4%BD%9C%E6%9D%A5%E6%95%B4%E5%90%88%E5%88%86%E5%8F%89%E7%9A%84%E5%8E%86%E5%8F%B2" name="Figure%2036.%20%E9%80%9A%E8%BF%87%E5%90%88%E5%B9%B6%E6%93%8D%E4%BD%9C%E6%9D%A5%E6%95%B4%E5%90%88%E5%88%86%E5%8F%89%E7%9A%84%E5%8E%86%E5%8F%B2">
     Figure 36. 通过合并操作来整合分叉的历史
    </h5>
    <p>
     其实，还有一种方法：可以提取在
     <code>
      C4
     </code>
     中引入的补丁和修改，然后在
     <code>
      C3
     </code>
     的基础上应用一次。
    </p>
    <p>
     在 Git 中，这种操作叫做
     <strong>
      变基（rebase）
     </strong>
     可以使用
     <code>
      rebase
     </code>
     命令将提交到某一分支上的所有修改都移至另一分支上，就好像“重新播放”一样。
    </p>
    <p>
     在这个例子中，你可以检出
     <code>
      experiment
     </code>
     分支，然后将它变基到
     <code>
      master
     </code>
     分支上：
    </p>
    <pre><code class="hljs">$ git checkout experiment  #切换到experiment
$ git rebase master   #变基到master上
First, rewinding head to replay your work on top of it...
Applying: added staged command</code></pre>
    <p>
     <img alt="" height="345" src="https://i-blog.csdnimg.cn/direct/c4ed980f26c345769c2c6377b346c4aa.png" width="1283"/>
    </p>
    <p>
     原理：首先找到这两个分支（即当前分支 experiment、变基的目标分支 master） 的
     <strong>
      最近共同祖先 C2
     </strong>
     ，然后对比当前分支相对于该祖先的历次提交，
     <strong>
      提取相应的修改并存为临时文件， 然后将当前分支指向目标基底 C3, 最后以此将之前另存为临时文件的修改依序应用。
     </strong>
    </p>
    <p>
     <strong>
      变基
     </strong>
     是将一系列提交按照原有次序依次应用到另一分支上，而
     <strong>
      合并
     </strong>
     是把最终结果合在一起。
    </p>
    <h5 id="Figure%2037.%20%E5%B0%86%20C4%20%E4%B8%AD%E7%9A%84%E4%BF%AE%E6%94%B9%20%E5%8F%98%E5%9F%BA%20%E5%88%B0%20C3%20%E4%B8%8A" name="Figure%2037.%20%E5%B0%86%20C4%20%E4%B8%AD%E7%9A%84%E4%BF%AE%E6%94%B9%20%E5%8F%98%E5%9F%BA%20%E5%88%B0%20C3%20%E4%B8%8A">
     Figure 37. 将
     <code>
      C4
     </code>
     中的修改 变基 到
     <code>
      C3
     </code>
     上
    </h5>
    <p>
     $ git checkout master   # 切换到master
     <br/>
     $ git merge experiment  #进行快进合并
    </p>
    <p>
     <img alt="" height="371" src="https://i-blog.csdnimg.cn/direct/05f339e6b0ca4cc0b45474263c6bd67c.png" width="1283"/>
    </p>
    <h5 id="Figure%2038.%20master%20%E5%88%86%E6%94%AF%E7%9A%84%E5%BF%AB%E8%BF%9B%E5%90%88%E5%B9%B6" name="Figure%2038.%20master%20%E5%88%86%E6%94%AF%E7%9A%84%E5%BF%AB%E8%BF%9B%E5%90%88%E5%B9%B6">
     Figure 38.
     <code>
      master
     </code>
     分支的快进合并
    </h5>
    <p>
     这两种整合方法的最终结果没有任何区别，但变基使得提交历史更加整洁。 你在查看一个经过变基的分支的历史记录时会发现，尽管实际的开发工作是并行的， 但它们看上去就像是串行的一样，提交历史是一条直线没有分叉。
    </p>
    <p>
     变基的目的是为了确保在向远程分支推送时能保持提交历史的整洁——例如向某个其他人维护的项目贡献代码时。
    </p>
    <p>
     在这种情况下，你首先在自己的分支里进行开发，
     <strong>
      当开发完成时你需要先将你的代码变基到
      <code>
       origin/master
      </code>
      上，然后再向主项目提交修改。
     </strong>
     这样的话，该项目的维护者就不再需要进行整合工作，只需要快进合并便可。
    </p>
    <blockquote>
     <p>
      如果我执行了变基操作，但是后续没有进行快速合并会怎么样
     </p>
    </blockquote>
    <p>
     此时会停留在Figure 36 图中的效果，即一条线性链
    </p>
    <p>
     若后续需要将
     <code>
      experiment
     </code>
     的修改整合到
     <code>
      master
     </code>
     ，必须显式执行
     <code>
      git merge
     </code>
     或
     <code>
      git rebase
     </code>
     。此时合并可能会产生冲突，需要手动解决
    </p>
    <p>
     即此时
     <code>
      git merge
     </code>
     等效于
     <code>
      git rebase
     </code>
     ，都是快速合并
    </p>
    <blockquote>
     <p>
      如果进行了快速合并，那我master分支里面的代码就改变了，我怎么样回滚回之前的状态
     </p>
    </blockquote>
    <pre><code class="hljs">#1.找到合并提交的哈希值
git log --oneline
#2.撤销该提交
git revert -m 1 &lt;合并提交的哈希值&gt;
#3.推送回滚
git push origin master

# 效果：生成一个反向提交（如 C5），撤销合并引入的更改，保留原合并记录</code></pre>
    <h5 id="Figure%2039.%20%E4%BB%8E%E4%B8%80%E4%B8%AA%E4%B8%BB%E9%A2%98%E5%88%86%E6%94%AF%E9%87%8C%E5%86%8D%E5%88%86%E5%87%BA%E4%B8%80%E4%B8%AA%E4%B8%BB%E9%A2%98%E5%88%86%E6%94%AF%E7%9A%84%E6%8F%90%E4%BA%A4%E5%8E%86%E5%8F%B2" name="Figure%2039.%20%E4%BB%8E%E4%B8%80%E4%B8%AA%E4%B8%BB%E9%A2%98%E5%88%86%E6%94%AF%E9%87%8C%E5%86%8D%E5%88%86%E5%87%BA%E4%B8%80%E4%B8%AA%E4%B8%BB%E9%A2%98%E5%88%86%E6%94%AF%E7%9A%84%E6%8F%90%E4%BA%A4%E5%8E%86%E5%8F%B2">
     Figure 39. 从一个主题分支里再分出一个主题分支的提交历史
    </h5>
    <p>
     <img alt="" height="586" src="https://i-blog.csdnimg.cn/direct/1e61006e5f5c4a58ab8dcc59ecbebd6d.png" width="1062"/>
    </p>
    <p>
     假设你希望将
     <code>
      client
     </code>
     中的修改合并到主分支并发布，但并不想合并
     <code>
      server
     </code>
     中的修改， 这时，可以使用
     <code>
      git rebase
     </code>
     命令的
     <code>
      --onto
     </code>
     选项， 选中在
     <code>
      client
     </code>
     分支里但不在
     <code>
      server
     </code>
     分支里的修改（即
     <code>
      C8
     </code>
     和
     <code>
      C9
     </code>
     ），将它们在
     <code>
      master
     </code>
     分支上重放：
    </p>
    <pre>$ git rebase --onto master server client</pre>
    <p>
     以上命令的意思是：“取出
     <code>
      client
     </code>
     分支，找出它从
     <code>
      server
     </code>
     分支分歧之后的补丁， 然后把这些补丁在
     <code>
      master
     </code>
     分支上重放一遍，让
     <code>
      client
     </code>
     看起来像直接基于
     <code>
      master
     </code>
     修改一样”。这理解起来有一点复杂，不过效果非常酷。
    </p>
    <p>
     <img alt="" height="514" src="https://i-blog.csdnimg.cn/direct/222f17b4a4ae4a99913d6b5f097c598a.png" width="1308"/>
    </p>
    <p>
     现在可以快进合并
     <code>
      master
     </code>
     分支了
    </p>
    <pre><code class="hljs">$ git checkout master
$ git merge client</code></pre>
    <p>
     <img alt="" height="413" src="https://i-blog.csdnimg.cn/direct/673117f7b1294a6da25bf32f0eb8590b.png" width="1301"/>
    </p>
    <h4 id="%E5%8F%98%E5%9F%BA%E7%9A%84%E9%A3%8E%E9%99%A9" name="%E5%8F%98%E5%9F%BA%E7%9A%84%E9%A3%8E%E9%99%A9">
     变基的风险
    </h4>
    <ol>
     <li>
      <p>
       <strong>
        不要对共享提交进行变基
       </strong>
       ：如果你的提交已经被推送到一个公共仓库，并且其他人可能已经基于这些提交进行了开发工作，那么不应该对这些提交执行变基操作。因为这会改变提交的历史记录，导致他人需要重新整合他们的工作，造成混乱。
      </p>
     </li>
     <li>
      <p>
       <strong>
        重复提交问题
       </strong>
       ：如果有人通过变基覆盖了远程仓库上的提交历史，而你又从这个远程仓库拉取更新并尝试合并或再次变基的话，可能会引入重复的提交。这是因为变基操作会生成新的提交对象，即使它们的内容与之前的提交相同。
      </p>
     </li>
     <li>
      <p>
       <strong>
        强制推送带来的混乱
       </strong>
       ：当你使用
       <code>
        git push --force
       </code>
       强制推送经过变基的提交时，可能会覆盖其他人的工作，特别是那些基于旧提交进行开发的人的工作，从而引发冲突和额外的整合工作。
      </p>
     </li>
    </ol>
    <h5 id="%E8%A7%A3%E5%86%B3%E6%96%B9%E5%BC%8F" name="%E8%A7%A3%E5%86%B3%E6%96%B9%E5%BC%8F">
     解决方式
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        检查修改差异
       </strong>
       ：如果你发现自己的工作基于的提交被覆盖了，可以通过比较“patch-id”来识别哪些是你所做的修改以及哪些是被覆盖的修改。Git 能够自动检测你的修改并尝试将它们应用到新的分支上。
      </p>
     </li>
     <li>
      <p>
       <strong>
        变基而非合并
       </strong>
       ：在某些情况下，与其直接合并他人的变基后的工作，不如选择对自己的分支执行变基（例如
       <code>
        git rebase teamone/master
       </code>
       ），这样可以避免不必要的合并提交，同时保持提交历史的清晰。
      </p>
     </li>
    </ul>
    <p>
     总的原则是，只对尚未推送或分享给别人的本地修改执行变基操作清理历史， 从不对已推送至别处的提交执行变基操作
    </p>
