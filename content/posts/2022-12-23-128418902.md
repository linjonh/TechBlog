    <h2>
     <a id="web_5">
     </a>
     一、移动开发和web开发的区别
    </h2>
    <p>
     移动前端开发和web开发都属于前端开发的范围。
    </p>
    <ul>
     <li>
      一般我们在开发pc端web项目时，需要考虑这么几点：
      <ul>
       <li>
        语言和UI库选型
       </li>
       <li>
        浏览器的兼容性
       </li>
       <li>
        构建速度
       </li>
       <li>
        打包体积
       </li>
       <li>
        SEO优化
       </li>
       <li>
        …
       </li>
      </ul>
     </li>
    </ul>
    <p>
     不难发现，PC端web开发工作重点主要都在
     <strong>
      代码
     </strong>
     和
     <strong>
      工程化
     </strong>
     上。
    </p>
    <ul>
     <li>
      移动开发时，就目前来说，需要关注的内容有所增加：
      <ul>
       <li>
        浏览器兼容性上我们只需要考虑webkit内核的浏览器和chrome，uc，qq，小米手机浏览器就好了
       </li>
       <li>
        打包发布需要有规范化的流程（特别是IOS的发布，我们并不会像传统web一样可以频繁发版）
       </li>
       <li>
        前期投入上需要更慎重些，比如说技术选型，在Flutter被炒的如火如荼的今天，我们是更想要uni-app的一套代码跨平台快速开发的特性来外包项目或者小程序，还是更想要React Native的这种原生应用做属于自己的产品？
       </li>
      </ul>
     </li>
    </ul>
    <p>
     总之，移动端的开发不仅需要投入更多的学习成本在
     <strong>
      代码
     </strong>
     和
     <strong>
      工程化
     </strong>
     上，还需要重视
     <strong>
      跨平台
     </strong>
     、
     <strong>
      版本管理
     </strong>
     等。
    </p>
    <h2>
     <a id="React_Native_24">
     </a>
     二、什么是React Native?
    </h2>
    <p>
     React Native (RN)是一个用于构建跨平台应用程序的 JavaScript 框架。2015年由facebook开源，社区驱动。RN使用 React 为 Android、iOS 等创建原生应用，现在甚至已经探索出了包括React Native Windows、React Native macOS和React Native Web等新的项目，支持更多的平台。
    </p>
    <h2>
     <a id="IOS_26">
     </a>
     三、如何实现安卓和IOS用一套代码开发
    </h2>
    <p>
     <code>
      React Native
     </code>
     提供了一组与平台无关的核心原生组件，例如
     <code>
      View
     </code>
     、
     <code>
      Text
     </code>
     和
     <code>
      Image
     </code>
     直接映射到平台的原生 UI 构建块。
     <br/>
     <img alt="React Native" src="https://i-blog.csdnimg.cn/blog_migrate/66b82c2637535a0237c7229318ece3e4.png"/>
    </p>
    <p>
     简单讲，就是
     <code>
      RN
     </code>
     在Android端构建时会把
     <code>
      View
     </code>
     编译为
     <code>
      AndroidView
     </code>
     ，IOS端构建时会把
     <code>
      View
     </code>
     编译为
     <code>
      IOSView
     </code>
     ，在开发人员视角，我们只需要知道RN包装的
     <code>
      View
     </code>
     组件即可。
    </p>

    <h3>
     <a id="2__150">
     </a>
     2. 中型项目
    </h3>
    <p>
     随着我们的应用程序的增长，我们通常会将“screens”组件分离到一个
     <code>
      screens
     </code>
     目录中，并开始进一步对
     <code>
      components
     </code>
     目录中的文件进行分类。“screens”组件应该占据整个屏幕，例如个人资料屏幕或设置屏幕。
    </p>
    <p>
     如果使用着导航的库，例如：
     <code>
      react-navigation
     </code>
     ，我们可以将导航器/路由器组织到一个单独的
     <code>
      navigation
     </code>
     目录中（也可以叫
     <code>
      navigators
     </code>
     或
     <code>
      routing
     </code>
     ）。
    </p>
    <p>
     其他一些常见的文件类别：
     <br/>
     <code>
      api
     </code>
     ：网络 API 调用，通常由提供商或路由组织
     <br/>
     <code>
      assets
     </code>
     ：要与应用捆绑的图像和其他文件
     <br/>
     <code>
      hooks
     </code>
     : 自定义挂钩
     <br/>
     <code>
      reducers
     </code>
     : reducer 函数，用于使用useReducer钩子（或其他库）管理应用程序数据
     <br/>
     <code>
      theme
     </code>
     ：共享颜色和文本样式（有时称为styles）
     <br/>
     <code>
      utils
     </code>
     : 字符串格式化等杂项工具
    </p>

    <h3>
     <a id="3__204">
     </a>
     3. 大型项目
    </h3>
    <p>
     当项目增长到包含许多不同的功能或 UI 流程时，通常会在顶层按功能对文件进行分类。如果多个团队在开发该应用程序，功能名称将经常与团队名称对齐：例如帐户、演化、隐私。
    </p>
    <p>
     功能目录通常分组在
     <code>
      modules
     </code>
     、
     <code>
      packages
     </code>
     或
     <code>
      apps
     </code>
     目录下。在多个功能/团队之间共享的组件或实用程序通常位于其中的特殊目录
     <code>
      shared
     </code>
     或
     <code>
      core
     </code>
     目录中，并被视为公共 API。一个功能应该只引用它自己的目录或共享目录中的文件，而不是另一个功能目录中的文件——这表明该文件是移动到共享目录的候选者，并将其视为公共 API。
    </p>
    <p>
     这个项目结构旨在为每个文件清楚地描述：它应该存在的地方，谁维护它，以及它是否被允许导入到其他功能中。
    </p>
