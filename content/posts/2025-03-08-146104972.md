    <h2>
     <a id="41___2">
     </a>
     4.1 数据库安全性概述
    </h2>
    <h3>
     <a id="411___3">
     </a>
     4.1.1 数据库的不安全因素
    </h3>
    <ol>
     <li>
      非授权用户对数据库的恶意存取和破坏：一些黑客（Hacker）和犯罪分子在用户存取数据库时猎取用户名和用户口令，然后假冒合法用户偷取、修改甚至破坏用户数据。
      <ul>
       <li>
        数据库管理系统提供的安全措施主要包括
        <strong>
         用户身份鉴别、存取控制和视图
        </strong>
        等技术。
       </li>
      </ul>
     </li>
     <li>
      数据库中重要或敏感的数据被泄露：黑客和敌对分子千方百计盗窃数据库中的重要数据，一些机密信息被暴露。
      <ul>
       <li>
        数据库管理系统提供的主要技术有
        <strong>
         强制存取控制、数据加密存储和加密传输
        </strong>
        等。
       </li>
       <li>
        <strong>
         审计日志分析
        </strong>
       </li>
      </ul>
     </li>
     <li>
      安全环境的脆弱性：数据库的安全性与计算机系统的安全性紧密联系
      <ul>
       <li>
        建立一套
        <strong>
         可信（Trusted）计算机系统
        </strong>
        的概念和标准
       </li>
      </ul>
     </li>
    </ol>
    <h2>
     <a id="42___13">
     </a>
     4.2 数据库安全性控制
    </h2>
    <ul>
     <li>
      计算机系统中，安全措施是一级一级层层设置
      <ul>
       <li>
        系统根据用户标识鉴定用户身份，合法用户才准许进入计算机系统
       </li>
       <li>
        数据库管理系统还要进行存取控制，只允许用户执行合法操作
       </li>
       <li>
        操作系统有自己的保护措施
       </li>
       <li>
        数据以密码形式存储到数据库中
        <br/>
        <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/0c74bf3c22fe420b8cafb3b1f91ca910.png"/>
       </li>
      </ul>
     </li>
     <li>
      <strong>
       存取控制流程
      </strong>
      <ul>
       <li>
        首先，数据库管理系统对提出SQL访问请求的数据库
        <strong>
         用户进行身份鉴别
        </strong>
        ，防止不可信用户使用系统。
       </li>
       <li>
        然后，在
        <strong>
         SQL处理层
        </strong>
        进行
        <strong>
         自主存取控制和强制存取控制，进一步可以进行推理控制
        </strong>
        。
       </li>
       <li>
        还可以对
        <strong>
         用户访问行为和系统关键操作进行审计
        </strong>
        ，对
        <strong>
         异常用户行为进行简单入侵检测
        </strong>
        。
        <br/>
        <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/868bb6c2d25c45b69614dcc53ed1c956.png"/>
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="421__25">
     </a>
     4.2.1 用户身份鉴别
    </h3>
    <ul>
     <li>
      <strong>
       用户身份鉴别
      </strong>
      （Identification &amp; Authentication）
      <strong>
       系统提供的最外层安全保护措施
      </strong>
     </li>
     <li>
      <strong>
       用户身份鉴别的方法
      </strong>
      <ul>
       <li>
        静态口令鉴别：静态口令一般由用户自己设定，这些口令是静态不变的
       </li>
       <li>
        动态口令鉴别口令是动态变化的，每次鉴别时均需使用动态产生的新口令登录数据库管理系统，即采用一次一密的方法
       </li>
       <li>
        生物特征鉴别：通过生物特征进行认证的技术，生物特征如指纹、虹膜和掌纹等
       </li>
       <li>
        智能卡鉴别：智能卡是一种不可复制的硬件，内置集成电路的芯片，具有硬件加密功能
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="422__32">
     </a>
     4.2.2 存取控制
    </h3>
    <ul>
     <li>
      <p>
       <strong>
        用户权限定义
       </strong>
       和
       <strong>
        合法权检查机制
       </strong>
       一起组成了数据库管理系统的存取控制子系统
      </p>
     </li>
     <li>
      <p>
       <strong>
        常用存取控制方法
       </strong>
      </p>
      <ul>
       <li>
        <strong>
         自主存取控制（Discretionary Access Control ，简称DAC）
        </strong>
        <ul>
         <li>
          用户对不同的数据对象有不同的存取权限
         </li>
         <li>
          不同的用户对同一对象也有不同的权限
         </li>
         <li>
          用户还可将其拥有的存取权限转授给其他用户
         </li>
        </ul>
       </li>
      </ul>
      <blockquote>
       <p>
        你有一个文件夹，里面放着一些文件。你可以决定谁可以看这些文件，谁可以修改它们。比如，你可以让你的朋友A只能看文件，而让朋友B既能看又能修改。你还可以把你的权限转给朋友C，让他也能看或修改文件。这就是自主存取控制（DAC），你作为文件的主人，可以自由决定谁有什么权限。
        <br/>
        <strong>
         DAC
        </strong>
        ：你作为主人，可以自由决定谁可以看或修改你的文件。
       </p>
      </blockquote>
      <ul>
       <li>
        <strong>
         强制存取控制（Mandatory Access Control，简称 MAC）
        </strong>
        <ul>
         <li>
          每一个数据对象被标以一定的密级
         </li>
         <li>
          每一个用户也被授予某一个级别的许可证
         </li>
         <li>
          对于任意一个对象，只有具有合法许可证的用户才可以存取
         </li>
        </ul>
       </li>
      </ul>
      <blockquote>
       <p>
        你和你的朋友们在一个军队或政府机构工作。每个文件都有一个保密级别，比如“机密”或“绝密”。每个人也有一个安全级别，比如“普通员工”或“高级官员”。只有你的安全级别足够高，才能看或修改相应保密级别的文件。你不能随便把权限给别人，因为这是由系统或上级强制规定的。这就是强制存取控制（MAC），权限是由系统或规则强制管理的，你不能随意更改。
        <br/>
        <strong>
         MAC
        </strong>
        ：文件的访问权限是由系统或规则强制规定的，你不能随意更改。
       </p>
      </blockquote>
     </li>
    </ul>
    <h3>
     <a id="423__DAC_52">
     </a>
     4.2.3 自主存取控制方法 DAC
    </h3>
    <ul>
     <li>
      定义用户存取权限：定义用户可以在哪些数据库对象上进行哪些类型的操作，
      <strong>
       定义存取权限称为授权
      </strong>
     </li>
     <li>
      通过 SQL 的
      <code>
       GRANT
      </code>
      语句和
      <code>
       REVOKE
      </code>
      语句实现
     </li>
    </ul>
    <h3>
     <a id="424__55">
     </a>
     4.2.4 授权：授予与回收
    </h3>
    <h4>
     <a id="1GRANT_56">
     </a>
     1.GRANT
    </h4>
    <p>
     语义：将对指定操作对象的指定操作权限授予指定的用户
    </p>
    <pre><code> GRANT &lt;权限&gt;[,&lt;权限&gt;]... 
 ON &lt;对象类型&gt; &lt;对象名&gt;[,&lt;对象类型&gt; &lt;对象名&gt;]…
 TO &lt;用户&gt;[,&lt;用户&gt;]...
 [WITH GRANT OPTION];
</code></pre>
    <ul>
     <li>
      <p>
       发出GRANT：数据库管理员，数据库对象创建者（即属主Owner），拥有该权限的用户
      </p>
     </li>
     <li>
      <p>
       按受权限的用户 ：一个或多个具体用户，PUBLIC（即全体用户）
      </p>
     </li>
     <li>
      <p>
       <code>
        WITH GRANT OPTION
       </code>
       子句:
       <br/>
       指定：可以再授予
       <br/>
       没有指定：不能传播
      </p>
      <pre><code> [例4.6] GRANT INSERT 
             ON TABLE SC 
             TO U6
             WITH GRANT OPTION;
      同样，U6还可以将此权限授予U7：
 [例4.7] GRANT INSERT 
	         ON TABLE SC 
             TO U7;
      但U7不能再传播此权限。
</code></pre>
     </li>
     <li>
      <p>
       <strong>
        不允许循环授权
       </strong>
       <br/>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/c68245605c3b4ca3b6109687c8b0b0bb.png"/>
      </p>
     </li>
    </ul>
    <h4>
     <a id="2REVOKE_84">
     </a>
     2.REVOKE
    </h4>
    <p>
     授予的权限可以由数据库管理员或其他授权者用REVOKE语句收回
    </p>
    <pre><code>REVOKE &lt;权限&gt;[,&lt;权限&gt;]... 
    ON &lt;对象类型&gt; &lt;对象名&gt;[,&lt;对象类型&gt;&lt;对象名&gt;]…
    FROM &lt;用户&gt;[,&lt;用户&gt;]...[CASCADE | RESTRICT];
</code></pre>

     4.2.5 数据库角色
    </h3>
    <ul>
     <li>

      </strong>

      <ul>
       <li>
        <strong>
         角色是权限的集合
        </strong>
       </li>
       <li>
        <strong>
         可以为一组具有相同权限的用户创建一个角色
        </strong>
       </li>
       <li>
        简化授权的过程
       </li>
      </ul>
     </li>
    </ul>
    <table>
     <thead>
      <tr>
       <th>
        <strong>
         命令
        </strong>
       </th>
       <th>
        <strong>

        </strong>
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        <strong>
         CREATE USER
        </strong>
       </td>
       <td>
        创建一个数据库用户，用户可以直接登录数据库并执行操作。
       </td>
      </tr>
      <tr>
       <td>
        <strong>
         CREATE ROLE
        </strong>
       </td>
       <td>
        创建一个角色，角色是一组权限的集合，不能直接登录数据库，
        <strong>
         但可以分配给用户
        </strong>
        。
       </td>
      </tr>
     </tbody>
    </table>
    <h3>
     <a id="426__MAC_138">
     </a>
     4.2.6 强制存取控制方法 MAC
    </h3>
    <ul>
     <li>
      <p>
       在强制存取控制中，数据库管理系统所管理的
       <strong>
        全部实体被分为主体和客体两大类
       </strong>
      </p>
      <ul>
       <li>
        <strong>
         主体是系统中的活动实体
        </strong>
        ： 数据库管理系统所管理的实际用户 ，代表用户的各进程
       </li>
       <li>
        <strong>
         客体是系统中的被动实体，受主体操纵
        </strong>
        ： 文件、基本表、索引、视图
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        敏感度标记（Label）
       </strong>
       ：对于主体和客体，DBMS为它们每个实例（值）指派一个敏感度标记（Label）
      </p>
      <ul>
       <li>
        <p>
         敏感度标记分成若干级别
         <br/>
         绝密（Top Secret，TS）
         <br/>
         机密（Secret，S）
         <br/>
         可信（Confidential，C）
         <br/>
         公开（Public，P）
         <br/>
         TS&gt;=S&gt;=C&gt;=P
        </p>
       </li>
       <li>
        <p>
         <strong>
          主体的敏感度标记称为许可证级别
         </strong>
         （Clearance Level）
        </p>
       </li>
       <li>
        <p>
         <strong>
          客体的敏感度标记称为密级
         </strong>
         （Classification Level）
        </p>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        强制存取控制规则
       </strong>
       <br/>
       （1）仅当主体的许可证级别大于或等于客体的密级时，该主体才能
       <strong>
        读
       </strong>
       取相应的客体
      </p>
      <blockquote>
       <p>
        如果文件是“机密”级别，那么只有“机密”级别或更高级别（如“绝密”）的人才能看。
        <br/>
        如果你是“普通员工”，就不能看“机密”文件。
        <br/>
        <strong>
         总结：
        </strong>
        <br/>
        <strong>
         “你想看文件，你的级别必须够高。”
        </strong>
       </p>
      </blockquote>
      <p>
       （2）仅当主体的许可证级别小于或等于客体的密级时，该主体才能
       <strong>
        写
       </strong>
       相应的客体
      </p>
      <blockquote>
       <p>
        如果文件是“机密”级别，那么只有“机密”级别或更低级别（如“普通”）的人才能修改。
        <br/>
        如果你是“绝密”级别的人，就不能修改“机密”文件，因为你的级别太高了。
        <br/>
        <strong>
         为什么这样规定？
        </strong>
        <br/>
        这是为了防止高级别的人不小心或故意修改低级别的文件，导致信息泄露或错误。
        <br/>
        <strong>
         总结：
        </strong>
        <br/>
        <strong>
         “你想改文件，你的级别不能太高。”
        </strong>
       </p>
      </blockquote>
     </li>
     <li>
      <p>
       <strong>
        实现强制存取控制时要首先实现自主存取控制
       </strong>
       <br/>
       原因：较高安全性级别提供的安全保护要包含较低级别的所有保护

       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/aeaf84693d6943ef97f1905cb4cdd1de.png">
        <br/>
        先进行自主存取控制检查，通过自主存取控制检查的数据对象再由系统进行强制存取控制检查，只有通过强制存取控制检查的数据对象方可存取。
       </img>
      </p>
     </li>
    </ul>

    <h2>
     <a id="44__Audit_181">
     </a>
     4.4 审计（Audit）
    </h2>
    <ul>
     <li>
      <strong>
       什么是审计
      </strong>
      <ul>
       <li>
        <strong>
         启用一个专用的审计日志（Audit Log）： 将用户对数据库的所有操作记录在上面
        </strong>
       </li>
       <li>
        审计员利用审计日志，监控数据库中的各种行为，找出非法存取数据的人、时间和内容
       </li>
      </ul>
     </li>
    </ul>
    <h3>
     <a id="1_187">
     </a>
     1.审计事件
    </h3>
    <ul>
     <li>
      服务器事件：审计数据库服务器发生的事件
     </li>
     <li>
      系统权限：对系统拥有的结构或模式对象进行操作的审计，要求该操作的权限是通过系统权限获得的
     </li>
     <li>
      语句事件：对SQL语句，如DDL、DML、DQL及DCL语句的审计
     </li>
     <li>
      模式对象事件：对特定模式对象上进行的SELECT或DML操作的审计
     </li>
    </ul>
    <h3>
     <a id="3_AUDITNOAUDIT_192">
     </a>
     3. AUDIT语句和NOAUDIT语句
    </h3>
    <p>
     AUDIT语句：设置审计功能
     <br/>
     NOAUDIT语句：取消审计功能
    </p>

    <h2>
     <a id="45___209">
     </a>
     4.5 数据加密
    </h2>
    <ul>
     <li>
      <p>
       数据加密：防止数据库中数据在存储和传输中失密的有效手段
      </p>
     </li>
     <li>
      <p>
       加密的基本思想：根据一定的算法将原始数据—明文（Plain text）变换为不可直接识别的格式­—密文（Cipher text）
      </p>
     </li>
     <li>
      <p>
       <strong>
        加密方法：存储加密和传输加密
       </strong>
      </p>
      <ul>
       <li>
        <strong>
         存储加密
        </strong>
        <ul>
         <li>
          <strong>
           透明存储加密
          </strong>
          <br/>
          内核级加密保护方式，对用户完全透明
          <br/>
          将数据在写到磁盘时对数据进行加密，授权用户读取数据时再对其进行解密
          <br/>
          数据库的应用程序不需要做任何修改，只需在创建表语句中说明需加密的字段即可
          <br/>
          *
          <em>
           内核级加密方法: 性能较好，安全完备性较高
          </em>
         </li>
         <li>
          <strong>
           非透明存储加密
          </strong>
          ：通过多个加密函数实现
         </li>
        </ul>
       </li>
       <li>
        <strong>
         传输加密
        </strong>
        <ul>
         <li>
          <strong>
           链路加密
          </strong>
          ：在链路层进行加密
          <br/>
          传输信息由报头和报文两部分组成
          <br/>
          <strong>
           报文和报头均加密
          </strong>
         </li>
         <li>
          <strong>
           端到端加密
          </strong>
          <br/>
          在发送端加密，接收端解密
          <br/>
          <strong>
           只加密报文不加密报头
          </strong>
          <br/>
          所需密码设备数量相对较少，容易被非法监听者发现并从中获取敏感信息
         </li>
        </ul>
       </li>
      </ul>
     </li>
    </ul>
