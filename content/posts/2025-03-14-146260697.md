    <h4>
     <a id="2__30">
     </a>
     2. 设计测试用例的万能公式
    </h4>
    <p>
     <strong>
      设计测试用例的万能公式：
     </strong>
     <br/>
     <mark>
      功能测试 + 界面测试 + 性能测试 + 兼容性测试 + 易用性测试 + 安全测试
     </mark>
    </p>
    <h5>
     <a id="21__35">
     </a>
     2.1 功能测试
    </h5>
    <p>
     功能测试是一个试图发现程序与外部规格说明之间存在不一致的过程。外部规格说明是一份从最终用户的角度对程序行为的精确描述。
    </p>
    <p>
     <strong>
      需求不明确时, 如何进行功能测试?
     </strong>
    </p>
    <ul>
     <li>
      查找其他相关文档，帮助理解产品目标
     </li>
     <li>
      参加项目组内的会议，加深对产品的理解
     </li>
     <li>
      召集相关人员，评审整理的结果
     </li>
     <li>
      使用已上线产品，了解历史bug
     </li>
    </ul>
    <h5>
     <a id="22__44">
     </a>
     2.2 界面测试
    </h5>
    <p>
     对软件界面上所有的内容都需要进行测试。
    </p>
    <p>
     <strong>
      要求：
     </strong>
    </p>
    <ul>
     <li>
      整体界面测试：界面的实现与设计图要求一致
     </li>
     <li>
      界面元素测试：控件操作验证
     </li>
    </ul>
    <h5>
     <a id="23__51">
     </a>
     2.3 性能测试
    </h5>
    <p>
     性能测试和功能测试的区别是：功能测试检查软件是否做了，而性能测试测试软件做的好不好。
    </p>
    <h5>
     <a id="24__54">
     </a>
     2.4 兼容性测试
    </h5>
    <p>
     软件是否能够在不同的环境下正确运行需要测试人员进行验证。
    </p>
    <p>
     <strong>
      兼容性测试的选取标准：
     </strong>
    </p>
    <ul>
     <li>
      优先选择使用当前产品top级别的机型进行测试
     </li>
     <li>
      选择主流的浏览器/机型进行测试
     </li>
    </ul>
    <h5>
     <a id="25__61">
     </a>
     2.5 易用性测试
    </h5>
    <p>
     易用性测试的标准是检查产品是否具备简单易上手的属性。
    </p>
    <h5>
     <a id="26__64">
     </a>
     2.6 安全测试
    </h5>
    <p>
     安全测试和性能测试一样都是比较大的领域。常见的安全问题如：
    </p>
    <ul>
     <li>
      隐私数据明文显示
     </li>
     <li>
      参数未强校验导致SQL注入
     </li>
     <li>
      越权：普通用户也可以执行管理员权限的操作
     </li>
     <li>
      除了万能公式之外，还有⼀个⽐较常⽤的测试类型：弱⽹测试、安装卸载测试
     </li>
    </ul>
    <h5>
     <a id="27__70">
     </a>
     2.7 弱网测试
    </h5>
    <p>
     弱网测试的目的就是尽可能保证用户体验，关注的关键点包括：
    </p>
    <ul>
     <li>
      页面响应时间是否可以接受，关注包括热启动、冷启动时间、页面切换、前后台切换、首字时间，首屏时间等。
     </li>
     <li>
      页面呈现是否完成一致。
     </li>
     <li>
      超时文案是否符合定义，异常信息是否显⽰正常。
     </li>
     <li>
      是否有超时重连。
     </li>
     <li>
      安全角度：是否会发生dns劫持、登陆ip更换频繁、单点登陆异常等。
     </li>
     <li>
      大流量事件风险：是否会在弱网下进⾏更新apk包、下载文件等大流量动作。
     </li>
    </ul>

    <h5>
     <a id="28__85">
     </a>
     2.8 安装卸载测试
    </h5>
    <ul>
     <li>
      针对需要进行部署的软件，除了软件功能外，我们还需要关注软件的能够成功安装和卸载
     </li>
    </ul>

     <a id="32__101">

     3.2 具体的设计方法
    </h5>
    <h6>
     <a id="321__103">
     </a>
     3.2.1 等价类
    </h6>
    <p>
     等价类法依据需求将输入（特殊情况下会考虑输出）划分为若干个等价类，从等价类中选出一个测试用例，如果这个测试用例测试通过，则认为所代表的等价类测试通过。
    </p>
    <p>
     <strong>
      等价类分类：
     </strong>
    </p>
    <ul>
     <li>
      有效等价类：对于程序的规格说明书是合理的、有意义的输入数据构成的集合
     </li>
     <li>
      无效等价类：根据需求说明书，不满足需求的集合
     </li>
    </ul>
    <p>
     <strong>
      根据等价类设计测试用例的方式：
     </strong>
    </p>
    <ol>
     <li>
      确定有效等价类和无效等价类
     </li>
     <li>
      编写测试用例，设计具体测试数据

      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a74b90c66edf4edca59332490bc05e07.png"/>

    </p>
    <p>
     <strong>
      边界值包含：
     </strong>
    </p>
    <ul>

     <strong>
      正交试验设计：
     </strong>
     <br/>
     正交试验设计是研究多因素多水平的一种设计方法，它是根据正交性，由试验因素的全部水平组合中挑选出部分有代表性的点进行试验。
    </p>
    <p>
     <strong>
      正交表的构成：
     </strong>
    </p>

     <li>

     </li>
     <li>

     </li>
     <li>
      行数
     </li>
    </ul>

     3.2.4 判定表法
    </h6>
    <p>
     判定表是一种表达逻辑判断的工具，通过判定表可以清晰地表达出所有条件对应的结果。
    </p>
    <p>
     <strong>
      根据判定表法设计测试用例的步骤：
     </strong>
    </p>
    <ol>
     <li>
      确认需求中输入条件和输出条件
     </li>
     <li>
      找出输入条件和输出条件之间的关系
     </li>
     <li>
      画判定表
     </li>
     <li>
      根据判定表编写测试用例
     </li>
    </ol>
    <p>
     假设存在以下需求:
     <br/>
     用户输入的账号中包含admin字符，或者通过内部链接进入注册页面，提交注册按钮成为管理员身份；反之无管理员身份。
    </p>
    <ol>
     <li>
      <p>

       <br/>
       输入条件：账号包含admin字符（a）、内部注册链接(b)、点击注册按钮©

     </li>
     <li>
      <p>
       找出输入条件和输出条件之间的关系
       <br/>
       输入条件：ac ab bc abc a b c 非abc

    <h6>
     <a id="325__188">

     3.2.5 场景法
    </h6>
    <p>
     场景法是通过运用场景对系统的功能点或业务流程的描述，从而提高测试效果的一种方法。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/3275a15e06c54e75970f5f9a088e794a.png"/>
    </p>
    <p>
     <strong>
      场景法设计测试用例的步骤：
     </strong>
    </p>
    <ol>
     <li>
      确定基本流
     </li>
     <li>
      确定备选流
     </li>
     <li>
      根据备选流补充测试用例
     </li>
     <li>
      编写测试用例
     </li>
    </ol>
    <h6>
     <a id="326__198">
     </a>
     3.2.6 错误猜测法
    </h6>
    <p>
     错误猜测法是对被测试软件设计的理解，过往经验以及个人直觉，推测出软件可能存在的缺陷，从而针对性地设计测试用例的方法。
    </p>
    <p>
     <strong>
      错误猜测法的缺点：
     </strong>
    </p>
    <ul>
     <li>
      难以系统化
     </li>
     <li>
      过度依赖个人能力
     </li>
    </ul>
    <p>
     以注册为例
     <br/>
     1、校验中特殊字符空格的处理?
     <br/>
     2、密码校验中的大小写？
     <br/>
     3、姓名中的特殊字符？
     <br/>
     4、密码发送是否明文
    </p>
