    <h2>
     <a id="_6">
     </a>
     可执行文件
    </h2>
    <ul>
     <li>
      <p>
       可执行文件的格式
       <br/>
       在裸机环境下,得到的是HEX/BIN格式的文件,而使用操作系统时，操作的往往是ELF格式的文件
       <br/>
       BIN/HEX文件是纯指令文件，没有其他杂七杂八的辅助信息，BIN文件最纯粹,而HEX文件会有一些附加的信息比如校验什么的;而ELF文件除了基本的代码段、数据段，还有文件头、符号表、program header table等用来辅助程序运行的信息
      </p>
     </li>
     <li>
      <p>
       裸机环境下程序的运行
       <br/>
       在ARM架构下 bin文件的头部的第一个存放的是SP指针的值 第二个存放的是RestHandler的地址 这样的话一上电SP指针和PC指针得到了相关的值就知道怎么运行了,同时在进入main函数之前会有一个__main函数，__main函数解析了bin文件中关于ZI段，数据段等信息，并对相应的SRAM内存做了操作，比如分配堆/栈，清空ZI段，加载数据段等等
      </p>
     </li>
     <li>
      <p>
       ELF的加载与运行
       <br/>
       ELF通过加载器加载到内存当中
       <br/>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/bc694a12e5444641b3a960b2634bee8f.png"/>
      </p>
      <p>
       .text段:代码段
       <br/>
       .bss段: 未初始化的全局变量(实际上此时只记录了符号 并没有分配内存,只有使用的时候才会加载)
       <br/>
       .data段: 初始化的全局变量
       <br/>
       .rodata段: 字符串/const修饰的常量
       <br/>
       .elf headler:记录了一些关键信息不可缺少
      </p>
     </li>
     <li>
      <p>
       可执行文件加载的过程
       <br/>
       如何把一个可执行文件转换为进程呢
       <br/>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ee7f1dd28d3b48ba80e7e56b63cc4fe9.png"/>
      </p>
     </li>
     <li>
      <p>
       静态链接
       <br/>
       静态链接就是在程序链接的过程中 就把引用的函数和变量编入可执行文件里面了 这样做的话会大大增加可执行文件的体积,并且假设多个文件都使用了printf 每个人一个副本的话也
       <strong>
        浪费内存
       </strong>
       ,关键是C语言
       <strong>
        是以文件为基本单位的
       </strong>
       ,所以实际上多的不止printf而是整个的xx.c文件
      </p>
      <ul>
       <li>
        ar命令可以用来生成静态库
       </li>
       <li>
        gcc -L -l用来指定链接的库
       </li>
      </ul>
     </li>
     <li>
      <p>
       动态链接
       <br/>
       动态链接就是为了解决静态链接的缺点而存在的 同样的动态链接有一个动态链接器。
       <br/>
       动态链接器通过解析ELF,做这样几件事
      </p>
      <ol>
       <li>
        对动态链接库进行
        <strong>
         重定位
        </strong>
        :调整库内代码和数据的地址以适应实际加载地址的过程
       </li>
       <li>
        对用到动态库的符号重定位(
        <strong>
         更新GOT表
        </strong>
        ):(也不知道动态库被加载到哪里了 所以动态库加载完毕后，需要重新找然后告诉程序更新了)
       </li>
       <li>
        然后再把控制权交给程序本身
        <br/>
        <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/02087a9380a5400fa7914437e0832540.png"/>
       </li>
      </ol>
      <ul>
       <li>
        <p>
         与地址无关的代码与GOT表
         <br/>
         你不能每个进程都用printf就重新加载一遍动态链接库到对应进程,这不白操作了吗 我们希望的是虽然多个进程都是用了printf 但实际上printf的代码执被加载了一次,这个事情需要动态链接库和应用程序双方的努力。
        </p>
        <ul>
         <li>
          动态链接库:生成与地址无关的代码(比如偏移量靠绝对的)
         </li>
         <li>
          应用程序: GOT表 把使用动态链接库的符号都取出来,到时候动态库加载了就实时更新这个表
          <br/>
          <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a070aa51a4bc4ec0bf54b452a486b24e.png"/>
         </li>
        </ul>
       </li>
       <li>
        <p>
         延迟绑定
         <br/>
         没有必要一次性把所有动态库都加载到内存,用到谁就加载谁
        </p>
       </li>
       <li>
        <p>
         模块的加载流程
        </p>
        <ol>
         <li>
          初始化 init_module–放在特殊的section段
         </li>
         <li>
          拷贝到内核 copy_from_user
         </li>
         <li>
          地址空间分配
         </li>
         <li>
          符号解析
         </li>
         <li>
          重定位
         </li>
         <li>
          执行
         </li>
        </ol>
       </li>
      </ul>
     </li>
    </ul>
