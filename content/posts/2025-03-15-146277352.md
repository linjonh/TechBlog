      网络基础概念–socket编程预备

    <h3>
     <a id="1__ip__IP____4">
     </a>
     1. 理解 ip 地址和目的 IP 地址 -&gt; 区分主机唯一性
    </h3>
    <p>
     <strong>
      IP 在网络当中, 用来标识主机的唯一性.
     </strong>
     <br/>
     是的, 我们就是用两个不同的 ip 地址来标识两台在网络当中不同的主机来进行网络通信的.
    </p>
    <h3>
     <a id="2__8">
     </a>
     2. 认识端口号
    </h3>
    <p>
     思考: 数据传输到主机是目的吗? 不是的. 因为数据目的不是给机器看的, 是要给人看的, 所以说发数据给主机是一种手段.
     <br/>
     而人怎么看聊条信息呢? 如何进行下载任务呢? 都是通过各种应用来做的, 所以说, 各种应用在系统的进程是人的代表.
    </p>
    <blockquote>
     <p>
      <font color="ff00">
       <strong>
        结论: 数据传输到主机不是目的, 而是手段. 到达主机内部, 交给对应的进程才是目的(进程代表的就是人).
       </strong>
      </font>
     </p>
    </blockquote>
    <p>
     但是一个系统内的进程是大量的, 那数据如何确定到达哪个进程呢? -&gt; 网络视角上区分进程的唯一性 -&gt; 我们引入
     <strong>
      端口号
     </strong>
     .
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/3af183150747440e93f0940355e0964e.png">
      <br/>
      为了在任意一个主机内部, 标识进程的
      <strong>
       唯一性
      </strong>
      , 我们引入
      <strong>
       端口号
      </strong>
      .
     </img>
    </p>
    <ul>
     <li>
      <strong>
       端口号是一个 2 字节 16 位的整数;
      </strong>
     </li>
     <li>
      <strong>
       端口号用来标识一个进程, 告诉操作系统, 当前的这个数据要交给哪一个进程来处理;
      </strong>
     </li>
     <li>
      <strong>
       IP 地址 + 端口号能够标识网络上的某一台主机的某一个进程;
      </strong>
     </li>
     <li>
      <font color="ff0000">
       <strong>
        一个端口号只能被一个进程占用.
       </strong>
      </font>
      <br/>
      这个很好理解的, 我们 port 是用来标识一个主机内唯一的一个进程的, 如果一个端口号同时被多个进程占用, 这肯定不唯一了吧!
     </li>
    </ul>
    <p>
     端口号是传输层的概念, 当数据报到达目标主机的传输层的时候, 传输层会根据提前封装好的端口号信息来给到对应的进程. 这就要求我们的特定软件绑定对应的端口号.
    </p>
    <h3>
     <a id="3_socket__27">
     </a>
     3. socket 套接字
    </h3>
    <p>
     实际通信的是网络当中的两个进程, 为了区分主机和进程, 我们用
     <code>
      IP + PORT
     </code>
     的形式来标识互联网当中唯一的一个进程, 对于目标进程也是同理.
    </p>
    <blockquote>
     <p>
      <font color="ff00">
       <strong>
        结论: 网络通信的本质是进程之间的通信.
       </strong>
      </font>
     </p>
    </blockquote>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/86b7d6a13fd94df6a2fac361ed67a0e5.png">
      <br/>
      进程之间的通信需要两个条件:
     </img>
    </p>
    <ol>
     <li>
      <strong>
       两个进程独立 -&gt; 当然独立
      </strong>
     </li>
     <li>
      <strong>
       公共资源 -&gt; 网络
      </strong>
     </li>
    </ol>
    <p>
     所以说, 网络通信对于进程之间的通信条件全部满足,
     <strong>
      本质上, 网络通信就是一种进程间通信(只不过物理距离变长了~)
     </strong>
    </p>
    <p>
     从今天开始, 我们称
     <code>
      IP + PORT
     </code>
     的这种网络通信方式称之为
     <code>
      socket 通信
     </code>
     .
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/d9638a61dc864e02a259006a24b8e8c6.png"/>
    </p>
    <p>
     强调三个点:
    </p>
    <ol>
     <li>
      通常来说, ip 的概念在网络层, port 的概念在传输层, 而 mac 地址的概念在链路层, 因此每一层都有对应的关键概念.
     </li>
     <li>
      因此, 服务端启动的时候, 就一定要和一个 port 关联起来(不关联数据到了传输层, 系统如何清楚要交给哪个进程, 总不能让 qq 发的消息, 让微信来显示吧?)
     </li>
     <li>
      <font color="ff0000">
       在传输层咋不用进程 pid 标识进程的唯一性呢? -&gt; 为了解耦, 一旦系统层面变了, 网络也必须改变! 当然, 还有比如进程的 pid 会变化, 以及端口号可以自己指定等等各种原因, 但是最本质的还是解开强耦合问题.
       <br/>
       系统当中, 并不是所有的进程都需要网络通信, 因此单独拿出一个 port 而非直接使用 pid 是十分合理的. -&gt; 在学校咋不用身份证标识而是用学号呢?
      </font>
     </li>
    </ol>
    <p>
     <strong>
      端口号的划分范围:
     </strong>
    </p>
    <ul>
     <li>
      0 - 1023: 知名端口号, HTTP, FTP, SSH 等这些广为使用的应用层协议, 他们的端口号都是固定的.
      <br/>
      这个知名端口号类似 110 和警察的关系.
     </li>
     <li>
      1024 - 65535: 操作系统动态分配的端口号. 客户端程序的端口号, 就是由操作系统从这个范围分配的.
      <br/>
      一般来说我们用的是这个普通的端口号, 如果想要用知名端口号在云服务器上默认需要 root 权限才可以.
     </li>
    </ul>
    <p>
     <strong>
      理解源端口号和目的端口号
     </strong>
     <br/>
     传输层协议(TCP 和 UDP)的数据段中有两个端口号, 分别叫做源端口号和目的端口号.
     <br/>
     就是在描述 “数据是谁发的, 要发给谁”;
    </p>
    <p>
     <strong>
      理解 socket
     </strong>
    </p>
    <ul>
     <li>
      综上，IP 地址用来标识互联网中唯一的一台主机，port 用来标识该主机上
      <font color="ff0000">
       唯一的一个网络进程
      </font>
     </li>
     <li>
      <code>
       IP+Port
      </code>
      就能表示互联网中唯一的一个进程
     </li>
     <li>
      所以，通信的时候，本质是两个互联网进程代表人来进行通信，{srcIp，srcPort ，dstIp，dstPort}这样的 4 元组就能标识互联网中唯二的两个进程
     </li>
     <li>
      <strong>
       <font color="ff00">
        所以，网络通信的本质，也是进程间通信
       </font>
      </strong>
     </li>
     <li>
      我们把 ip+port 叫做
      <strong>
       套接字 socket
      </strong>
     </li>
    </ul>
    <blockquote>
     <p>
      <strong>
       理解: 网络操作都被转换为IO 操作.
      </strong>
     </p>
    </blockquote>
    <p>
     所有的网络行为, 就俩行为 -&gt; 封装一下自己的数据, 发到网络上, 或者从网络上拿到数据, 就这俩行为. 因此, 所有的网络行为就是输入和输出操作, 也基本就是我们的 IO 操作了, 一个主机输入, 一个主机输出, 网络联系两台主机, 基本算是一个 网络版的冯诺依曼体系结构.
    </p>
    <h3>
     <a id="4___tcp__udp_69">
     </a>
     4. 传输层的代表协议 -&gt;
     <code>
      tcp
     </code>
     &amp;
     <code>
      udp
     </code>
    </h3>
    <p>
     如果我们了解了系统，也了解了网络协议栈，我们就会清楚，传输层是属于内核的，那么我们要通过网络协议栈进行通信，必定调用的是传输层提供的系统调用，来进行的网络通信。
     <br/>
     <strong>
      解释一下网络协议栈
     </strong>
     : 为啥叫做栈呢? 这是因为整个 tcp/ip 协议是分层的, 数据包在整个过程中封装的时候都是"压入"报文, 解包的时候都是"弹出"报文, 很类似栈, 就叫做网络协议栈咯.
     <br/>
     说白了, 这句话就想表达一个事情:
     <strong>
      程序员你想用网络通信? 那你得调用系统接口.
     </strong>
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a8908d8d8640499d9a7064867d9c8145.png"/>
    </p>
    <p>
     下面我们简单的认识一下传输层的两个典型的代表协议.
    </p>
    <ul>
     <li>
      <p>
       <strong>
        认识 TCP 协议
       </strong>
       <br/>
       此处我们先对
       <code>
        TCP(Transmission Control Protocol 传输控制协议)
       </code>
       有一个直观的认识;后面我们再详细讨论 TCP 的一些细节问题.
      </p>
      <ul>
       <li>
        传输层协议
       </li>
       <li>
        有连接
       </li>
       <li>
        可靠传输
       </li>
       <li>
        面向字节流
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        认识 UDP 协议
       </strong>
       <br/>
       此处我们也是对
       <code>
        UDP(User Datagram Protocol 用户数据报协议)
       </code>
       有一个直观的认识; 后面再详细讨论.
      </p>
      <ul>
       <li>
        传输层协议
       </li>
       <li>
        无连接
       </li>
       <li>
        不可靠传输
       </li>
       <li>
        面向数据报
       </li>
      </ul>
     </li>
    </ul>
    <p>
     因为我们暂时还没有深入了解
     <code>
      tcp、udp
     </code>
     协议，此处只做了解即可.
    </p>
    <p>
     <strong>
      兄弟们, 可能你们会有个疑惑, 就是整体解决这个长物理距离通信问题的解决方案叫做
      <code>
       tcp/ip
      </code>
      协议, 那这个传输层除了
      <code>
       tcp
      </code>
      咋还有个
      <code>
       udp
      </code>
      呢?
     </strong>
     <br/>
     这里咱们来解释一下, 就是
     <code>
      tcp/ip
     </code>
     协议这个名词表示的是整个网络问题的解决方案, 然后这个网络问题是很大的一个难题, 势必有很多的难题, 每个难题都对应一个小的解决方案, 这个小的解决方案可能就是我们具体的某种协议了, 比如说
     <code>
      ip协议
     </code>
     ,
     <code>
      tcp协议
     </code>
     ,
     <code>
      udp协议
     </code>
     … 之后呢, 我们的
     <code>
      tcp/ip协议
     </code>
     代表的是这些许多小方案的一个集合, 当然也就包含
     <code>
      udp协议
     </code>
     咯.
    </p>
    <p>
     上面我们仅仅是对这个传输层代表协议的两个典型协议做一个初步的认识, 在这里了解一下有这个东西即可.
    </p>
    <p>
     <strong>
      然后, 我们来解释一下两个协议当中提到的"可靠 与 不可靠"的理解:
     </strong>
     <br/>
     兄弟, 你可
     <code>
      不要误解 可靠当作是好的, 不可靠是不好的
     </code>
     , 这两个词表示的是两种协议的一种性质, 特征, 可靠的含义是当发生数据丢包的情况时, 两种协议对数据丢包问题的解决方式, 可靠的就是去修补一些数据, 不可靠呢则直接对这个丢包不做处理, 但是这并不意味着一个好一个坏, 因为
     <code>
      可靠
     </code>
     往往以为着需要做更多的工作, 而
     <code>
      不可靠
     </code>
     则相对简单.
    </p>
    <h3>
     <a id="5____103">
     </a>
     5. 网络字节序列 -&gt; 大端序列
    </h3>
    <p>
     我们的数字是有高低权值位的. 比如
     <code>
      int
     </code>
     有4个字节.
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/876fac1e73c64c48a3ddd66e2df72ec9.png"/>
    </p>
    <p>
     我们的电脑因为历史原因, 在数据存储上没有做到统一,
     <strong>
      有的电脑是把数值高权值位放到高地址处, 低权值位放到低地址处(
      <code>
       大端机
      </code>
      ), 有的电脑呢则把高权值位放到低地址处, 低权值位放到高地址处(
      <code>
       小端机
      </code>
      )
     </strong>
     . 也就是
     <strong>
      大端机
     </strong>
     和
     <strong>
      小端机
     </strong>
     的区别.
     <br/>
     兄弟们/姐妹们有没有想过一个问题,
     <font color="ff0000">
      就是一个大端机和一个小端机在跨网络通信的时候, 会不会造成数据解析错误的情况呢? 毕竟数据的存储方式不一样啊!
     </font>
     <br/>
     实际上, 网络设计者早就想到了这个问题, 他们规定:
     <strong>
      所有机器向网络传输数据时, 全部是大端字节序.
     </strong>
     <br/>
     <strong>
      有同学可能会好奇, 为啥不把自己的大小端信息放到数据报里向目标主机发送呢?
     </strong>
     实际上我感觉这样做是可以的, 但是处理起来会很麻烦. 最典型的一个问题就是数据报需要解析吧? 数据报需要解析之前就得需要知道是大端还是小端, 但是想要知道大端还是小端, 就必须解析数据报, 这个类似于一个
     <code>
      先有鸡还是先有蛋问题
     </code>
     .
    </p>
    <p>
     下面是一些解释:
    </p>
    <ul>
     <li>
      发送主机通常将发送缓冲区中的数据按内存地址
      <strong>
       从低到高
      </strong>
      的顺序发出;
     </li>
     <li>
      接收主机把从网络上接到的字节依次保存在接收缓冲区中,也是按内存地址
      <strong>
       从低到高
      </strong>
      的顺序保存;
     </li>
     <li>
      因此,网络数据流的地址应这样规定:先发出的数据是
      <strong>
       低地址
      </strong>
      ,后发出的数据是
      <strong>
       高地址
      </strong>
      .
     </li>
     <li>
      <code>
       TCP/IP 协议
      </code>
      规定,网络数据流应采用
      <strong>
       大端字节序
      </strong>
      ,即低地址高字节.
     </li>
     <li>
      不管这台主机是大端机还是小端机, 都会按照这个
      <code>
       TCP/IP
      </code>
      规定的网络字节序来发送/接收数据;
     </li>
     <li>
      如果当前发送主机是小端, 就需要先将数据转成大端; 否则就忽略, 直接发送即可;
     </li>
    </ul>
    <p>
     <br/>
     当然, 我们笔记本电脑通常就是
     <code>
      小端机
     </code>
     , 如果在编码时候还需要自己转换
     <strong>
      高低字节序
     </strong>
     是比较麻烦的, OS给我们提供了对应的转换接口:
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/96471cac2f2d4b6588b7733da204db0a.png"/>
    </p>
    <ul>
     <li>
      这些函数名很好记,h 表示 host,n 表示 network,l 表示 32 位长整数,s 表示 16 位短整数。
     </li>
     <li>
      例如 htonl 表示将 32 位的长整数从主机字节序转换为网络字节序,例如将 IP 地址转换后准备发送。
     </li>
     <li>
      如果主机是小端字节序,这些函数将参数做相应的大小端转换然后返回;
     </li>
     <li>
      如果主机是大端字节序,这些函数不做转换,将参数原封不动地返回。
     </li>
    </ul>
    <h3>
     <a id="6_socket__127">
     </a>
     6. socket 编程接口
    </h3>
    <p>
     常用的 socket 编程接口一览:
    </p>
    <pre><code class="prism language-cpp"><span class="token comment">// 创建 socket 文件描述符 (TCP/UDP, 客户端 + 服务器)</span>
<span class="token keyword">int</span> <span class="token function">socket</span><span class="token punctuation">(</span><span class="token keyword">int</span> domain<span class="token punctuation">,</span> <span class="token keyword">int</span> type<span class="token punctuation">,</span> <span class="token keyword">int</span> protocol<span class="token punctuation">)</span><span class="token punctuation">;</span>
<span class="token comment">// 绑定端口号 (TCP/UDP, 服务器)</span>
<span class="token keyword">int</span> <span class="token function">bind</span><span class="token punctuation">(</span><span class="token keyword">int</span> socket<span class="token punctuation">,</span> <span class="token keyword">const</span> <span class="token keyword">struct</span> <span class="token class-name">sockaddr</span> <span class="token operator">*</span>address<span class="token punctuation">,</span>socklen_t address_len<span class="token punctuation">)</span><span class="token punctuation">;</span>
<span class="token comment">// 开始监听 socket (TCP, 服务器)</span>
<span class="token keyword">int</span> <span class="token function">listen</span><span class="token punctuation">(</span><span class="token keyword">int</span> socket<span class="token punctuation">,</span> <span class="token keyword">int</span> backlog<span class="token punctuation">)</span><span class="token punctuation">;</span>

</code></pre>
    <p>
     我们这些接口也有一些内容, 我们放到一下节编码的时候再去介绍各个接口的用法.
    </p>
    <p>
     <code>
      socket API
     </code>
     是一层抽象的网络编程接口,适用于各种底层网络协议,如
     <code>
      IPv4
     </code>
     、
     <code>
      IPv6
     </code>
     ,以及后面要讲的
     <code>
      UNIX Domain Socket
     </code>
     . 然而, 各种网络协议的地址格式并不相同.
     <br/>
     <code>
      socket
     </code>
     通信也是一种进程间通信, 他既可以网络通信和本地通信.
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/71b36585684c43e8aa235105eab4659b.png"/>
    </p>
    <ul>
     <li>
      <strong>
       解释第二个结构体
      </strong>
      :
      <code>
       Pv4
      </code>
      和
      <code>
       IPv6
      </code>
      的地址格式定义在
      <code>
       netinet/in.h
      </code>
      中,
      <code>
       IPv4
      </code>
      地址用
      <code>
       sockaddr_in
      </code>
      结构体表示,包括 16 位地址类型, 16 位端口号和 32 位 IP 地址…
      <code>
       IPv4
      </code>
      、
      <code>
       IPv6
      </code>
      地址类型分别定义为常数
      <code>
       AF_INET, AF_INET6
      </code>
      . 这样,只要取得某种
      <code>
       sockaddr
      </code>
      结构体的首地址,不需要知道具体是哪种类型的
      <code>
       sockaddr
      </code>
      结构体,就可以根据地址类型字段确定结构体中的内容.
     </li>
     <li>
      <strong>
       第一个结构体干啥的?
      </strong>
      <code>
       socket API
      </code>
      可以都用
      <code>
       struct sockaddr *
      </code>
      类型表示, 在使用的时候需要强制转化成
      <code>
       sockaddr_in
      </code>
      ; 这样的好处是程序的通用性, 可以接收
      <code>
       IPv4
      </code>
      ,
      <code>
       IPv6
      </code>
      , 以及
      <code>
       UNIX DomainSocket
      </code>
      各种类型的
      <code>
       sockaddr
      </code>
      结构体指针做为参数; -&gt; 这仨玩意类似于一种
      <font color="ff0000">
       <strong>
        多态
       </strong>
      </font>
      的关系!
     </li>
     <li>
      <strong>
       解释第三个结构体
      </strong>
      : 第三个结构体是用于本地进程间通信的一个结构体, 这个类似于我们之前学过的"命名管道".
      <code>
       sockaddr_un
      </code>
      是类似于一种
      <strong>
       命名管道
      </strong>
      的东西.
     </li>
    </ul>
    <p>
     <strong>
      问题: 上面的接口咋不用 void
      <em>
       而是 sockaddr
      </em>
      呢?
     </strong>
     <br/>

     <strong>
      历史原因
     </strong>
     , 并且本身也没啥问题, 写法上需要用到
     <strong>
      强制转换
     </strong>
     就是了~
