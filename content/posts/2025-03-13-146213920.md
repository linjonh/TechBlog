    <p>
     Go语言（Golang）的垃圾回收（GC）机制经历了多个重要的发展阶段，每个阶段都在性能、停顿时间等方面进行了改进和优化，以下是各阶段的详细介绍：
     <br/>
    </p>
    <p>
    </p>
    <h3>
     <a id="GC_2">
     </a>
     一.GC简单介绍
    </h3>
    <h4>
     <a id="_GC_4">
     </a>
     什么是 GC（垃圾回收）
    </h4>
    <p>
     GC 即垃圾回收（Garbage Collection），是一种自动内存管理机制。在程序运行过程中，会不断地分配内存来存储各种数据，当这些数据不再被程序使用时，其所占用的内存就变成了“垃圾”。如果不及时回收这些内存，会导致内存泄漏，最终使程序耗尽系统资源而崩溃。垃圾回收机制的主要目的就是自动检测并回收这些不再使用的内存，让开发者无需手动管理内存的分配和释放，从而降低了编程的复杂度，提高了程序的稳定性和可维护性。
    </p>
    <p>
     不同的垃圾回收算法有不同的方式来找到可回收对象，主要分为：引用计数法和可达性分析法（在这篇文章种介绍了这两篇方法：
     <a href="https://blog.csdn.net/ydl1128/article/details/146225219">
      垃圾回收GC介绍
     </a>
     ）。Go语言采用可达性分析法来找到可回收的对象。
    </p>
    <h4>
     <a id="GC_Roots_11">
     </a>
     GC Roots（根对象）是什么？
    </h4>
    <p>
     <strong>
      GC Roots（垃圾回收根对象）
     </strong>
     是
     <strong>
      GC 在进行可达性分析（Reachability Analysis）时的起点
     </strong>
     ，用来查找存活对象。
    </p>
    <p>
     <strong>
      简单理解
     </strong>
     ：
    </p>
    <ul>
     <li>
      GC 需要决定哪些对象存活、哪些对象应该被回收。
     </li>
     <li>
      GC 不会扫描所有对象，而是
      <strong>
       从特定的根对象（GC Roots）出发，遍历整个可达对象图
      </strong>
      。
     </li>
     <li>
      <strong>
       如果某个对象没有被 GC Roots 访问到，它就会被视为“垃圾”，并被回收
      </strong>
      。
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="GC_Roots__21">
     </a>
     GC Roots 具体包含哪些内容？
    </h4>
    <p>
     GC Roots 主要包括以下几类：
    </p>
    <h5>
     <a id="1__24">
     </a>
     1. 栈上的局部变量
    </h5>
    <ul>
     <li>
      <strong>
       函数调用栈中的变量
      </strong>
      被视为 GC Roots，因为它们是程序当前执行的重要数据。
     </li>
     <li>
      这些变量通常存放在**线程栈（Thread Stack）**中。
     </li>
     <li>
      例如：
      <pre><code class="prism language-go"><span class="token keyword">func</span> <span class="token function">main</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
    a <span class="token operator">:=</span> <span class="token operator">&amp;</span>SomeStruct<span class="token punctuation">{<!-- --></span><span class="token punctuation">}</span> <span class="token comment">// 变量 `a` 作为局部变量存储在栈上</span>
    fmt<span class="token punctuation">.</span><span class="token function">Println</span><span class="token punctuation">(</span>a<span class="token punctuation">)</span>  <span class="token comment">// `a` 是 GC Roots，指向的对象不会被回收</span>
<span class="token punctuation">}</span>
</code></pre>
     </li>
    </ul>
    <h5>
     <a id="2__35">
     </a>
     2. 全局变量和静态变量
    </h5>
    <ul>
     <li>
      在
      <strong>
       全局作用域中声明的变量
      </strong>
      不会被 GC 轻易回收，因为它们可以在整个程序生命周期内访问。
     </li>
     <li>
      例如：
      <pre><code class="prism language-go"><span class="token keyword">var</span> globalVar <span class="token operator">*</span><span class="token builtin">int</span> <span class="token comment">// 全局变量</span>
<span class="token keyword">func</span> <span class="token function">main</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
    num <span class="token operator">:=</span> <span class="token number">42</span>
    globalVar <span class="token operator">=</span> <span class="token operator">&amp;</span>num <span class="token comment">// `globalVar` 持有 `num` 的地址</span>
<span class="token punctuation">}</span>
</code></pre>
      <ul>
       <li>
        <code>
         globalVar
        </code>
        是 GC Roots，因此
        <code>
         num
        </code>
        也不会被回收。
       </li>
      </ul>
     </li>
    </ul>
    <h5>
     <a id="3__Goroutine_47">
     </a>
     3. 进行中的 Goroutine
    </h5>
    <ul>
     <li>
      <strong>
       所有活动的 Goroutine 及其栈上的变量都属于 GC Roots
      </strong>
      ，因为它们可能在后续执行过程中访问对象。
     </li>
     <li>
      例如：
      <pre><code class="prism language-go"><span class="token keyword">func</span> <span class="token function">worker</span><span class="token punctuation">(</span>ch <span class="token keyword">chan</span> <span class="token builtin">int</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
    val <span class="token operator">:=</span> <span class="token operator">&lt;-</span>ch <span class="token comment">// `val` 存在于 Goroutine 栈中</span>
    fmt<span class="token punctuation">.</span><span class="token function">Println</span><span class="token punctuation">(</span>val<span class="token punctuation">)</span>
<span class="token punctuation">}</span>

<span class="token keyword">func</span> <span class="token function">main</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
    ch <span class="token operator">:=</span> <span class="token function">make</span><span class="token punctuation">(</span><span class="token keyword">chan</span> <span class="token builtin">int</span><span class="token punctuation">,</span> <span class="token number">1</span><span class="token punctuation">)</span>
    <span class="token keyword">go</span> <span class="token function">worker</span><span class="token punctuation">(</span>ch<span class="token punctuation">)</span> <span class="token comment">// 启动 Goroutine</span>
    ch <span class="token operator">&lt;-</span> <span class="token number">42</span>
<span class="token punctuation">}</span>
</code></pre>
      <ul>
       <li>
        <code>
         worker
        </code>
        函数中的
        <code>
         val
        </code>
        变量
        <strong>
         存活于 Goroutine 的栈中
        </strong>
        ，因此它属于 GC Roots。
       </li>
      </ul>
     </li>
    </ul>
    <h5>
     <a id="4__JNI__64">
     </a>
     4. 运行中的 JNI 引用
    </h5>
    <ul>
     <li>
      在 Java、Go 等语言中，可能会与 C/C++ 代码进行交互（如
      <code>
       cgo
      </code>
      ）。
     </li>
     <li>
      任何
      <strong>
       被 C 代码持有的 Go 对象引用
      </strong>
      ，在 GC 运行时也不会立即被回收。
     </li>
    </ul>
    <h5>
     <a id="5__68">
     </a>
     5. 内存管理系统中的特殊对象
    </h5>
    <ul>
     <li>
      <strong>
       Go 运行时维护的一些特殊对象，如反射（reflect）、内部缓存（比如
       <code>
        map
       </code>
       ）、runtime 数据结构
      </strong>
      等，都会被视为 GC Roots。
     </li>
     <li>
      例如：
      <pre><code class="prism language-go"><span class="token keyword">var</span> cache <span class="token operator">=</span> <span class="token function">make</span><span class="token punctuation">(</span><span class="token keyword">map</span><span class="token punctuation">[</span><span class="token builtin">string</span><span class="token punctuation">]</span><span class="token builtin">string</span><span class="token punctuation">)</span> <span class="token comment">// `cache` 作为全局变量，不会被 GC 直接回收</span>
</code></pre>
     </li>
    </ul>
    <hr/>
    <h4>
     <a id="GC_Roots__77">
     </a>
     GC Roots 是否等同于“扫描栈”？
    </h4>
    <p>
     <strong>
      不完全等同，但“扫描栈”是 GC Roots 识别的一部分！
     </strong>
    </p>
    <ul>
     <li>
      <strong>
       GC Roots 本身不是目标
      </strong>
      ，而是
      <strong>
       GC 进行可达性分析的起点
      </strong>
      。GC 主要从 GC Roots
      <strong>
       开始扫描对象引用
      </strong>
      ，
      <strong>
       栈上的局部变量
      </strong>
      是 GC Roots 之一。
     </li>
     <li>
      <strong>
       不仅仅扫描栈，还包括全局变量、Goroutine、JNI 引用等
      </strong>
      。
     </li>
     <li>
      <strong>
       GC 主要关注堆上的对象
      </strong>
      ，因为栈上的数据通常会随着函数调用的结束而自动释放，不需要 GC 处理。
      <br/>
      <strong>
       示例：
      </strong>
     </li>
    </ul>
    <pre><code class="prism language-go"><span class="token keyword">var</span> globalVar <span class="token operator">*</span><span class="token builtin">int</span> <span class="token comment">// GC Roots</span>

<span class="token keyword">func</span> <span class="token function">main</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
    localVar <span class="token operator">:=</span> <span class="token number">42</span>  <span class="token comment">// 仅存于栈上</span>
    globalVar <span class="token operator">=</span> <span class="token operator">&amp;</span>localVar  <span class="token comment">// 使 `globalVar` 指向 `localVar`</span>
<span class="token punctuation">}</span> <span class="token comment">// `localVar` 作用域结束，但由于 `globalVar` 仍然指向它，它不会被 GC 回收</span>
</code></pre>
    <ul>
     <li>
      <code>
       globalVar
      </code>
      是 GC Roots，因此
      <code>
       localVar
      </code>
      <strong>
       即使超出作用域，也不会被 GC 立即回收
      </strong>
      。
     </li>
     <li>
      <strong>
       如果
       <code>
        globalVar = nil
       </code>
       ，那么
       <code>
        localVar
       </code>
       将在下一次 GC 时被回收
      </strong>
      。
     </li>
    </ul>
    <h4>
     <a id="GC__94">
     </a>
     GC 最终回收的是栈上引用还是堆上的对象？
    </h4>
    <p>
     GC
     <strong>
      最终回收的是堆上的对象，而不是栈上的引用
     </strong>
     。
    </p>
    <ol>
     <li>
      <p>
       <strong>
        栈上的变量（引用）会随函数返回自动释放
       </strong>
      </p>
      <ul>
       <li>
        Go 语言的栈管理是自动的，局部变量在函数执行完后自动出栈，不需要 GC 处理。
       </li>
       <li>
        例如：
        <pre><code class="prism language-go"><span class="token keyword">func</span> <span class="token function">foo</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
    a <span class="token operator">:=</span> <span class="token number">42</span>  <span class="token comment">// `a` 存在于栈上</span>
<span class="token punctuation">}</span> <span class="token comment">// `a` 在函数结束后会自动销毁，不需要 GC 介入</span>
</code></pre>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        GC 主要负责管理堆上的对象
       </strong>
      </p>
      <ul>
       <li>
        只有
        <strong>
         在堆上分配的对象才需要 GC 处理
        </strong>
        ，因为它们的生命周期不受函数作用域限制。
       </li>
       <li>
        例如：
        <pre><code class="prism language-go"><span class="token keyword">func</span> <span class="token function">foo</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token operator">*</span><span class="token builtin">int</span> <span class="token punctuation">{<!-- --></span>
    a <span class="token operator">:=</span> <span class="token function">new</span><span class="token punctuation">(</span><span class="token builtin">int</span><span class="token punctuation">)</span>  <span class="token comment">// `a` 指向堆上的整数</span>
    <span class="token operator">*</span>a <span class="token operator">=</span> <span class="token number">42</span>
    <span class="token keyword">return</span> a  <span class="token comment">// `a` 逃逸到堆上</span>
<span class="token punctuation">}</span>
</code></pre>
        <ul>
         <li>
          <code>
           a
          </code>
          指向的整数是在
          <strong>
           堆上分配
          </strong>
          的，而
          <code>
           a
          </code>
          本身是
          <strong>
           栈上的引用
          </strong>
          。
         </li>
         <li>
          GC 只会回收
          <strong>
           堆上的整数
          </strong>
          ，不会管
          <code>
           a
          </code>
          ，因为
          <code>
           a
          </code>
          会随函数结束自动销毁。
         </li>
        </ul>
       </li>
      </ul>
     </li>
    </ol>
    <hr/>
    <ul>
     <li>
      示例：GC 仅回收堆上的对象
     </li>
    </ul>
    <pre><code class="prism language-go"> <span class="token keyword">package</span> main

     <span class="token keyword">import</span> <span class="token string">"fmt"</span>

     <span class="token keyword">type</span> Data <span class="token keyword">struct</span> <span class="token punctuation">{<!-- --></span>
         value <span class="token builtin">int</span>
     <span class="token punctuation">}</span>

     <span class="token keyword">func</span> <span class="token function">main</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
         <span class="token keyword">var</span> p <span class="token operator">*</span>Data  <span class="token comment">// 栈上的引用变量</span>
         p <span class="token operator">=</span> <span class="token operator">&amp;</span>Data<span class="token punctuation">{<!-- --></span>value<span class="token punctuation">:</span> <span class="token number">42</span><span class="token punctuation">}</span>  <span class="token comment">// 在堆上创建对象</span>
         fmt<span class="token punctuation">.</span><span class="token function">Println</span><span class="token punctuation">(</span>p<span class="token punctuation">.</span>value<span class="token punctuation">)</span>
     
         <span class="token comment">// 之后 p 设为 nil，原来的 Data 对象如果没有其他引用，就会被 GC 回收</span>
         p <span class="token operator">=</span> <span class="token boolean">nil</span>
     <span class="token punctuation">}</span>
</code></pre>
    <ul>
     <li>
      <code>
       p
      </code>
      是
      <strong>
       栈上的变量
      </strong>
      ，它本身不会被 GC 处理。
     </li>
     <li>
      <code>
       Data{value: 42}
      </code>
      <strong>
       存储在堆上
      </strong>
      ，当
      <code>
       p = nil
      </code>
      并且没有其他引用后，GC 就会回收这个对象。
     </li>
    </ul>
    <hr/>
    <h3>
     <a id="GoGC_145">
     </a>
     二.Go的GC历程和细节
    </h3>
    <h4>
     <a id="1_Mark_and_SweepGo_10__Go_13_146">
     </a>
     1. 标记清除（Mark and Sweep）阶段（Go 1.0 - Go 1.3）
    </h4>
    <ul>
     <li>
      <strong>
       设计原理
      </strong>
      <ul>
       <li>
        <strong>
         标记阶段
        </strong>
        ：从根对象（如全局变量、栈上的变量等）开始，递归遍历所有可达对象，并将这些对象标记为存活。这个过程需要暂停程序的执行（Stop The World，简称 STW），因为在标记过程中如果程序继续修改对象的引用关系，会导致标记结果不准确。
       </li>
       <li>
        <strong>
         清除阶段
        </strong>
        ：遍历整个堆内存，将未标记的对象标记为可回收，并释放其占用的内存空间。这个阶段可以在程序继续执行的情况下进行，不会造成 STW。
       </li>
      </ul>
     </li>
     <li>
      <strong>
       存在的问题
      </strong>
      ：
      <code>
       标记阶段的 STW 会导致程序出现明显的停顿，尤其是在堆内存较大、对象较多的情况下，停顿时间会很长，影响程序的响应性能
      </code>
      。
     </li>
    </ul>
    <h4>
     <a id="2_Go_15_152">
     </a>
     2. 三色标记法（Go 1.5）(核心)
    </h4>
    <h5>
     <a id="_153">
     </a>
     三色标记设计（最初）
    </h5>
    <ul>
     <li>
      <strong>
       三色抽象
      </strong>
      ：将对象分为白色、灰色和黑色三种颜色。白色表示对象未被访问过，灰色表示对象已被访问但它的子对象还未全部访问，黑色表示对象及其子对象都已被访问。
     </li>
     <li>
      <strong>
       标记过程
      </strong>
      ：初始时所有对象都是白色，从根对象开始遍历，将根对象及其直接引用的对象标记为灰色。然后不断从灰色对象集合中取出对象，将其标记为黑色，并将其引用的白色对象标记为灰色，直到灰色对象集合为空。此时，白色对象即为不可达对象，可以被回收。
      <ul>
       <li>
        <p>
         第一步：初始时，所有的对象标记为白色。注意，这里从栈上引出的对象最终是指向堆上的对象！
         <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/5981fad9fc30437d932f4f5876ce4bc4.png"/>
        </p>
       </li>
       <li>
        <p>
         第二步：从根对象开始遍历，将根对象及其直接引用的对象标记为灰色。
         <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/f36c4e6fc70148ec86e00fb358db079c.png"/>
        </p>
       </li>
       <li>
        <p>
         第三步：不断从灰色对象集合中取出对象，将其标记为黑色，并将其引用的白色对象标记为灰色。
         <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/acbeee6bf2df4bc7ae9d47ba980860e3.png"/>
        </p>
       </li>
       <li>
        <p>
         第四步：重复第三步，直到灰色对象集合为空。此时的白色集合就是本次要回收的对象。
         <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ac732fb304894cf1960c963029ac24a9.png"/>
        </p>
       </li>
      </ul>
     </li>
    </ul>
    <h5>
     <a id="_163">
     </a>
     三色标记存在的问题
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        问题1-黑色对象引用白色对象
       </strong>
       ：由于三色标记法在堆区不存在STW，在线程并发的情况下，可能存在黑色对某个白色对象的引用。例如下图中：在某个时刻，对象4断开了对对象3的引用，同时，对象5引用了对象4。由于对象5已经在黑色集合，而对象3无法被扫描到灰色集合中。那么，对象4就会被当作垃圾回收。从而产生了误删的情况。
       <br/>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/a043288253f64371a04a0c6002f76c9f.png"/>
      </p>
     </li>
     <li>
      <p>
       <strong>
        问题2-灰色对象丢失下游的引用对象
       </strong>
       ：下图中，对象5在引用对象3的时候，此时对象2恰好删除对对象3的引用。导致对象3不能被标记为灰色。
       <br/>
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/0d972172f80840e49a6bc9f28d106e80.png">
        <br/>
        当上述两个问题同时发生时，就会发生对象丢失的情况。最简单的解决办法就是加入STW，但是加入STW会带来GC的效率降低。那么如何进行优化呢？Go-GC定义了强弱三色不变式的条件来解决上述两个问题。
       </img>
      </p>
     </li>
    </ul>
    <h5>
     <a id="_171">
     </a>
     强/弱三色不变式
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        强三色不变式
       </strong>
       ：不允许黑色对象引用白色对象。解决问题1。
      </p>
     </li>
     <li>
      <p>
       <strong>
        弱三色不变式
       </strong>
       ：黑色对象可以引用白色对象，但是白色对象必须存在被灰色对象引用，或者白色对象的上游存在可达它的灰色对象。解决问题2。
      </p>
     </li>
    </ul>
    <p>
     那么具体如何实现呢？Golang引入了插入屏障机制和删除屏障机制来保证强弱三色不变式的条件成立。
    </p>
    <h5>
     <a id="_178">
     </a>
     插入写屏障、删除屏障机制
    </h5>
    <ul>
     <li>
      <p>
       <strong>
        插入写屏障
       </strong>
      </p>
      <ul>
       <li>
        <strong>
         原理
        </strong>
        ：插入屏障是在新的引用关系建立时触发的一种机制。当一个黑色对象引用一个白色对象时，插入屏障会将这个白色对象标记为灰色，确保这个白色对象不会因为在并发标记过程中被遗漏而被错误地回收。
       </li>
       <li>
        <strong>
         过程介绍
        </strong>
        ：如下图，对象5引用了对象4，对象4被标记为灰色。在之后会从灰色变为黑色。此时，栈空间将其gc root的所有对象进行STW保护，从而删除白色对象。
        <br/>
        <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/5476be111aa646d98d8fc26e0e4b6329.png"/>
       </li>
       <li>
        <strong>
         不足
        </strong>
        ：在一次完整的遍历结束以后，栈空间需要一次STW来清除白色对象，大约10-100ms，造成效率低下。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        删除写屏障
       </strong>
      </p>
      <ul>
       <li>
        原理：删除屏障是在引用关系被删除时触发的机制。当一个灰色对象或黑色对象删除对一个白色对象的引用时，删除屏障会将这个白色对象标记为灰色，保证这个白色对象在后续的标记过程中仍然会被处理，避免被错误地回收。
       </li>
       <li>
        过程介绍：如下图，对象1删除引用对象2，对象5删除引用对象6，被删除的对象均会被标记为灰色。以保证下一次能够从灰色集合中标记为黑色。
        <br/>
        <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/30b80a4761b1489c9eb32a6cbac28be7.png"/>
       </li>
       <li>
        <strong>
         不足
        </strong>
        ：在一次完整的遍历结束以后，本来应该在本次GC后删除的对象，只能等到下一次GC扫描的时候才会被标记为白色从而被清除。
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        改进之处
       </strong>
       ：在 Go 1.8 中引入了混合写屏障，它结合了插入写屏障和删除写屏障的优点。
      </p>
     </li>
    </ul>
    <h4>
     <a id="3_Go_18_195">
     </a>
     3. 混合写屏障（Go 1.8）(核心)
    </h4>
    <ul>
     <li>
      <strong>
       混合写屏障机制
      </strong>
      <ul>
       <li>
        <strong>
         主要规则
        </strong>
        ：（1）从在栈上的gcroot出发，能可达的对象在初始扫描时全部标记为黑色，并且创建的新对象均为黑色。（2）堆上的对象按照插入屏障、删除屏障来进行。
       </li>
       <li>
        <strong>
         流程
        </strong>
        ：首先，GC优先扫描栈，从栈可达的对象(1，2，3)均被标记为黑色，此时，栈上不开启插入写屏障。这就意味着，栈上任何新增和删除的对象都不会开启屏障，即保持新增黑色，删除不变的思想。然后，GC扫描堆上，开始将对象5标记为灰色，之后的流程遵循三色标记，并且堆上开启屏障。这就意味着，任何从堆上进行的添加和删除操作均会开启屏障，需要将其下游的对象进行灰色标记处理。
        <br/>
        <img alt="- **改进之处**：混合写屏障使得标记过程更加高效，不需要对栈进行重新扫描(STW)，提高了垃圾回收的性能。" src="https://i-blog.csdnimg.cn/direct/98c203aab5534815badc4a79e2b6400d.png"/>
       </li>
      </ul>
     </li>
     <li>
      <strong>
       总结
      </strong>
      ：在 Golang 里，混合写屏障机制遵循弱三色不变式，巧妙融合了删除写屏障与插入写屏障的优势。在垃圾回收过程中，它只需在开始阶段并发地扫描各个 goroutine 的栈，将栈上对象标记为黑色，并且能保证其一直维持黑色状态。这一扫描过程无需暂停程序（即无需 STW）。由于栈在完成扫描后始终保持黑色，所以在标记操作结束时，就无需再次对栈进行重新扫描（re - scan）。如此一来，大大减少了因 STW 而导致的程序停顿时间，有效提升了程序的并发性能和整体执行效率。
     </li>
    </ul>
    <h4>
     <a id="4_Go_19__202">
     </a>
     4. 并发标记和清理优化（Go 1.9 及以后）
    </h4>
    <p>
     Go 1.9 在垃圾回收（GC）方面进行了一系列优化，主要围绕减少停顿时间、提升并发性能和增强内存管理效率等方面展开，以下是详细介绍：
    </p>
    <ul>
     <li>
      <p>
       <strong>
        （1）减少 STW 时间
       </strong>
      </p>
      <ul>
       <li>
        <strong>
         混合写屏障的进一步优化
        </strong>
        <ul>
         <li>
          Go 1.8 引入了混合写屏障机制，Go 1.9 对其进行了进一步的调整和优化。混合写屏障结合了插入写屏障和删除写屏障的优点，减少了标记阶段的 STW 时间。在 Go 1.9 中，对写屏障的逻辑进行了更精细的控制，减少了不必要的标记操作，从而降低了写屏障带来的性能开销，进一步缩短了 STW 时间。
         </li>
         <li>
          例如，在并发标记阶段，对堆上对象引用关系的修改处理更加高效，能够更准确地标记对象，避免了一些误标记和重复标记的情况。
         </li>
        </ul>
       </li>
       <li>
        <strong>
         优化栈扫描
        </strong>
        <ul>
         <li>
          Go 1.9 改进了栈扫描的算法和策略。栈扫描是 GC 过程中的一个重要环节，因为栈上的变量可能引用堆上的对象。在 Go 1.9 中，采用了更高效的栈扫描方式，减少了栈扫描的时间。
         </li>
         <li>
          比如，通过对栈上对象的快速分类和过滤，只扫描那些可能包含堆对象引用的栈帧，避免了对整个栈的无差别扫描，从而提高了扫描效率，减少了 STW 时间。
         </li>
        </ul>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        （2）提升并发性能
       </strong>
      </p>
      <ul>
       <li>
        <strong>
         细粒度的并发控制
        </strong>
        <ul>
         <li>
          Go 1.9 采用了更细粒度的并发控制机制，允许 GC 与程序在更多的场景下并发执行。在之前的版本中，部分操作可能需要暂停程序来保证 GC 的正确性，但在 Go 1.9 中，通过优化并发控制逻辑，将这些操作拆分成更小的任务，使得这些任务可以与程序并发执行，减少了对程序的阻塞。
         </li>
         <li>
          例如，在标记阶段，对不同区域的内存标记可以并行进行，提高了标记的并发度，从而提升了整个 GC 过程的并发性能。
         </li>
        </ul>
       </li>
       <li>
        <strong>
         减少锁竞争
        </strong>
        <ul>
         <li>
          在 GC 过程中，锁的使用会影响并发性能。Go 1.9 对锁的使用进行了优化，减少了锁竞争的情况。通过使用更细粒度的锁或者无锁算法，降低了不同 goroutine 之间对锁的争用，提高了并发性能。
         </li>
         <li>
          比如，在内存分配和回收的过程中，采用了更高效的锁机制，使得多个 goroutine 可以更高效地同时进行内存操作，减少了等待锁的时间。
         </li>
        </ul>
       </li>
      </ul>
     </li>
     <li>
      <p>
       <strong>
        （2）增强内存管理效率
       </strong>
      </p>
      <ul>
       <li>
        <strong>
         优化内存分配器
        </strong>
        <ul>
         <li>
          Go 1.9 对内存分配器进行了优化，提高了内存分配和回收的效率。内存分配器是 GC 系统中的重要组成部分，它的性能直接影响到程序的整体性能。
         </li>
         <li>
          例如，采用了更智能的内存分配策略，根据对象的大小和使用频率进行分类管理，减少了内存碎片的产生，提高了内存的利用率。
         </li>
        </ul>
       </li>
       <li>
        <strong>
         改进内存清理策略
        </strong>
        <ul>
         <li>
          在 GC 的清扫阶段，Go 1.9 改进了内存清理策略。通过优化清扫算法和数据结构，提高了清扫的效率，减少了清扫过程中的性能开销。
         </li>
         <li>
          比如，采用了更高效的空闲内存块合并算法，将相邻的空闲内存块合并成更大的块，方便后续的内存分配，提高了内存管理的效率。
         </li>
        </ul>
       </li>
      </ul>
     </li>
    </ul>
