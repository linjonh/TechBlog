    <h2>
     采用左外连接
    </h2>
    <p>
     下面开始 EXPLAIN 分析
    </p>
    <pre><code class="language-sql">EXPLAIN SELECT SQL_NO_CACHE * FROM `type` LEFT JOIN book ON type.card = book.card;</code></pre>
    <blockquote>
     <p>
      结论：type 有All  ,代表着全表扫描，效率较差
     </p>
    </blockquote>
    <p>
     <img alt="" height="153" src="https://i-blog.csdnimg.cn/direct/c58a9f3d2c7741e29d6adefe5e915251.png" width="1155"/>
    </p>
    <p>
     添加索引优化
    </p>
    <pre><code class="language-sql">ALTER TABLE book ADD INDEX Y ( card); #【被驱动表】，可以避免全表扫描
EXPLAIN SELECT SQL_NO_CACHE * FROM `type` LEFT JOIN book ON type.card = book.card;</code></pre>
    <blockquote>
     <p>
      可以看到第二行的 type 变为了 ref，rows 也变成了优化比较明显。
      <strong>
       这是由左连接特性决定的。LEFT JOIN 条件用于确定如何从右表搜索行
      </strong>
      ，左边一定都有，所以
      <code>
       右边是我们的关键点,一定需要建立索引
      </code>
      。
      <strong>
       也就是left join 右边所关联的表的关联字段一定要建立索引
      </strong>
      。
     </p>
    </blockquote>
    <p>
     <img alt="" height="175" src="https://i-blog.csdnimg.cn/direct/2bbf76d089da441787f6e24d883bdc1c.png" width="1157"/>
    </p>
    <blockquote>
     <p>
      只是对左边的表建立索引的话，是没有效果的，可以通过rows这一列看到，type表要读取的记录仍然是20条。
     </p>
    </blockquote>
    <pre><code class="language-sql">ALTER TABLE `type` ADD INDEX X (card); #【驱动表】，无法避免全表扫描
EXPLAIN SELECT SQL_NO_CACHE * FROM `type` LEFT JOIN book ON type.card = book.card;</code></pre>
    <p>
     <img alt="" height="174" src="https://i-blog.csdnimg.cn/direct/9feb52eb741545e1b9670ef9a5dc6b02.png" width="1157"/>
    </p>
    <blockquote>
     <p>
      把右边的表的索引删除，可以发现现在又要走全表扫描了
     </p>
    </blockquote>
    <pre><code class="language-sql">DROP INDEX Y ON book;
EXPLAIN SELECT SQL_NO_CACHE * FROM `type` LEFT JOIN book ON type.card = book.card;</code></pre>
    <p>
     <img alt="" height="157" src="https://i-blog.csdnimg.cn/direct/14fa103c23f345f089c34b0be351b9d0.png" width="1151"/>
    </p>
    <h2>
     采用内连接
    </h2>
    <p>
     删除先前的索引
    </p>
    <pre><code class="language-sql">drop index X on type;
drop index Y on book;（如果已经删除了可以不用再执行该操作）</code></pre>
    <p>
     换成 inner join（MySQL自动选择驱动表）
    </p>
    <pre><code class="language-sql">EXPLAIN SELECT SQL_NO_CACHE * FROM type INNER JOIN book ON type.card=book.card;</code></pre>
    <p>
     <img alt="" height="159" src="https://i-blog.csdnimg.cn/direct/4323800be2694c419758acef64f970c4.png" width="1158"/>
    </p>
    <p>
     添加索引优化
    </p>
    <blockquote>
     <p>
      向book表添加索引后，book自动成为被驱动表，提高了查询效率。
     </p>
    </blockquote>
    <pre><code class="language-sql">ALTER TABLE book ADD INDEX Y (card);
EXPLAIN SELECT SQL_NO_CACHE * FROM type INNER JOIN book ON type.card=book.card;</code></pre>
    <p>
     <img alt="" height="183" src="https://i-blog.csdnimg.cn/direct/70f439e8d45e446c8f41f113423b9a53.png" width="1153"/>
    </p>
    <blockquote>
     <p>
      如果新增了type表的索引，此时两个表都有索引，优化器会选择小数据量的表作为驱动表，用来驱动大表。
     </p>
    </blockquote>
    <pre><code class="language-sql">ALTER TABLE type ADD INDEX X (card);
EXPLAIN SELECT SQL_NO_CACHE * FROM type INNER JOIN book ON type.card=book.card;</code></pre>
    <p>
     <img alt="" height="176" src="https://i-blog.csdnimg.cn/direct/41d08cd6da284e4c990025a92cb31e10.png" width="1154"/>
    </p>
    <blockquote>
     <p>
      对于内连接来说，查询优化器可以决定谁作为驱动表，谁作为被驱动表出现的，接下来把type表的索引删了。可以看到有索引的book表又作为了被驱动表
     </p>
    </blockquote>
    <pre><code class="language-sql">DROP INDEX X ON `type`;
EXPLAIN SELECT SQL_NO_CACHE * FROM TYPE INNER JOIN book ON type.card=book.card;</code></pre>
    <p>
     <img alt="" height="134" src="https://i-blog.csdnimg.cn/direct/9082e5e0e7934085866115344a8b67fb.png" width="1157"/>
    </p>
    <blockquote>
     <p>
      向type表里面添加索引，此时又变成了被驱动表了
     </p>
    </blockquote>
    <pre><code class="language-sql">ALTER TABLE `type` ADD INDEX X (card);
EXPLAIN SELECT SQL_NO_CACHE * FROM `type` INNER JOIN book ON type.card=book.card;</code></pre>
    <p>
     <img alt="" height="176" src="https://i-blog.csdnimg.cn/direct/1b2c6dfa5b024a5da8696fe9a8f6ecdd.png" width="1154"/>
    </p>
    <h2>
     join语句原理
    </h2>
    <blockquote>
     <p>
      join方式连接多个表，
      <strong>
       本质就是各个表之间数据的循环匹配
      </strong>
      。MySQL5.5版本之前，MySQL只支持一种表间关联方式，就是嵌套循环(Nested Loop Join)。如果关联表的数据量很大，则join关联的执行时间会很长。在MySQL5.5以后的版本中，MySQL通过引入BNLJ算法来优化嵌套执行。
     </p>
    </blockquote>
    <h3>
     驱动表和被驱动表
    </h3>
    <p>
     驱动表就是主表，被驱动表就是从表、非驱动表。
    </p>
    <ul>
     <li>
      对于内连接来说：
     </li>
    </ul>
    <blockquote>
     <p>
      A一定是驱动表吗？不一定，优化器会根据你查询语句做优化，决定先查哪张表。先查询的那张表就是驱动表，反之就是被驱动表。通过explain关键字可以查看。
     </p>
    </blockquote>
    <pre><code class="language-sql">SELECT * FROM A JOIN B ON ...</code></pre>
    <ul>
     <li>
      对于外连接来说：
     </li>
    </ul>
    <pre><code class="language-sql">SELECT * FROM A LEFT JOIN B ON ...
# 或
SELECT * FROM B RIGHT JOIN A ON ... </code></pre>
    <h3>
     Simple Nested-Loop Join (简单嵌套循环连接)
    </h3>
    <p>
     算法相当简单，从表A中取出一条数据1，遍历表B，将匹配到的数据放到result.. 以此类推，驱动表A中的每一条记录与被驱动表B的记录进行判断：
    </p>
    <p>
     <img alt="" height="400" src="https://i-blog.csdnimg.cn/direct/873a01e7f3fc4fb4935022be2e590e77.png" width="922"/>
    </p>
    <blockquote>
     <p>
      可以看到这种方式效率是非常低的，以上述表A数据100条，表B数据1000条计算，则A*B=10万次。开销统计如下。当然mysql肯定不会这么粗暴的去进行表的连接，所以就出现了后面的两种对Nested-Loop Join优化算法。
     </p>
    </blockquote>
    <p>
     <img alt="" height="332" src="https://i-blog.csdnimg.cn/direct/7cdd8e48946d4c309c1fae9ef73a5350.png" width="1185"/>
    </p>
    <h3>
     Index Nested-Loop Join （索引嵌套循环连接）
    </h3>
    <blockquote>
     <p>
      Index Nested-Loop Join其优化的思路主要是为了
      <code>
       减少内存表数据的匹配次数
      </code>
      ，所以要求被驱动表上必须
      <code>
       有索引
      </code>
      才行。通过外层表匹配条件直接与内层表索引进行匹配，避免和内存表的每条记录去进行比较，这样极大的减少了对内存表的匹配次数。
      <strong>
       就是利用索引来提高匹配效率
      </strong>
     </p>
    </blockquote>
    <p>
     <img alt="" height="343" src="https://i-blog.csdnimg.cn/direct/320645c971d243f094a3cd3a2cd26fae.png" width="797"/>
    </p>
    <blockquote>
     <p>
      驱动表中的每条记录通过被驱动表的索引进行访问，因为索引查询的成本是比较固定的，故mysql优化器都倾向于使用记录数少的表作为驱动表（外表）。如果被驱动表加索引，效率是非常高的，但如果索引不是主键索引，所以还得进行一次回表查询。相比，
      <strong>
       被驱动表的索引是主键索引，效率会更高。
      </strong>
     </p>
    </blockquote>
    <h3>
     <img alt="" height="329" src="https://i-blog.csdnimg.cn/direct/4d27d834b2844c9ba8b473bb109c039e.png" width="1184"/>
     Block Nested-Loop Join（块嵌套循环连接）
    </h3>
    <blockquote>
     <p>
      之前是将驱动表逐条与非驱动表的记录进行匹配，现在是引入join buffer缓冲区，将驱动表的记录缓冲到缓冲区，然后进行批量匹配，而不是逐条匹配。
     </p>
    </blockquote>
    <p>
     <img alt="" height="244" src="https://i-blog.csdnimg.cn/direct/51352e9c08fa455fad40f90e1971a4f1.png" width="870"/>
    </p>
    <p>
     <img alt="" height="370" src="https://i-blog.csdnimg.cn/direct/628b7fc49a754dc69b30d2b2a0f83839.png" width="857"/>
    </p>
    <p>
     <img alt="" height="243" src="https://i-blog.csdnimg.cn/direct/dc3660e9401e401790aa9cd6dda1e255.png" width="872"/>
    </p>
    <h3>
     Join小结
    </h3>
    <blockquote>
     <p>
      1、
      <strong>
       整体效率比较：INLJ &gt; BNLJ &gt; SNLJ
      </strong>
     </p>
     <p>
      2、永远用小结果集驱动大结果集（其本质就是减少外层循环的数据数量）（小的度量单位指的是表行数 * 每行大小）
     </p>
     <pre><code class="language-sql">select t1.b,t2.* from t1 straight_join t2 on (t1.b=t2.b) where t2.id&lt;=100; # 推荐
select t1.b,t2.* from t2 straight_join t1 on (t1.b=t2.b) where t2.id&lt;=100; # 不推荐</code></pre>
     <p>
      3、为被驱动表匹配的条件增加索引(减少内存表的循环匹配次数)
     </p>
     <p>
      4、增大join buffer size的大小（一次索引的数据越多，那么内层包的扫描次数就越少）
     </p>
     <p>
      5、减少驱动表不必要的字段查询（
      <strong>
       字段越少，join buffer所缓存的数据就越多
      </strong>
      ）
     </p>
    </blockquote>
    <h3>
     Hash Join
    </h3>
    <p>
     <strong>
      从MySQL的8.0.20版本开始将废弃BNLJ，因为从MySQL8.0.18版本开始就加入了hash join默认都会使用hash join
     </strong>
    </p>
    <blockquote>
     <ul>
      <li>
       <p>
        Nested Loop:
       </p>
       <p>
        对于被连接的数据子集较小的情况，Nested Loop是个较好的选择。
       </p>
      </li>
      <li>
       <p>
        Hash Join是做
        <code>
         大数据集连接
        </code>
        时的常用方式，优化器使用两个表中较小（相对较小）的表
        <strong>
         利用Join Key在内存中建立
         <code>
          散列表
         </code>
        </strong>
        ，然后扫描较大的表并探测散列表，找出与Hash表匹配的行。
       </p>
       <ul>
        <li>
         <p>
          这种方式适合于较小的表完全可以放于内存中的情况，
          <strong>
           这样总成本就是访问两个表的成本之和。
          </strong>
         </p>
        </li>
        <li>
         <p>
          在表很大的情况下并不能完全放入内存，这时优化器会将它分割成
          <code>
           若干不同的分区
          </code>
          ，不能放入内存的部分就把该分区写入磁盘的临时段，此时要求有较大的临时段从而尽量提高I/O的性能。
         </p>
        </li>
        <li>
         <p>
          它能够很好的工作于没有索引的大表和并行查询的环境中，并提供最好的性能。大多数人都说它是Join的重型升降机。
          <strong>
           Hash Join只能应用于等值连接
          </strong>
          （如WHERE A.COL1 = B.COL2），这是由Hash的特点决定的。
         </p>
        </li>
       </ul>
      </li>
     </ul>
    </blockquote>
    <p>
     <img alt="" height="483" src="https://i-blog.csdnimg.cn/direct/658c009511f541d9b11a006df04b57b9.png" width="872"/>
    </p>
    <h2>
     小结
    </h2>
    <blockquote>
     <ul>
      <li>
       <p>
        保证被驱动表的JOIN字段已经创建了索引
       </p>
      </li>
      <li>
       <p>
        需要JOIN 的字段，
        <strong>
         数据类型保持绝对一致
        </strong>
        。
       </p>
      </li>
      <li>
       <p>
        LEFT JOIN 时，选择小表作为驱动表， 大表作为被驱动表 。减少外层循环的次数。
       </p>
      </li>
      <li>
       <p>
        INNER JOIN 时，MySQL会自动将
        <strong>
         小结果集的表选为驱动表
        </strong>
        。选择相信MySQL优化策略。
       </p>
      </li>
      <li>
       <p>
        能够直接多表关联的
        <strong>
         尽量直接关联，不用子查询。(减少查询的趟数)
        </strong>
       </p>
      </li>
      <li>
       <p>
        不建议使用子查询，
        <strong>
         建议将子查询SQL拆开结合程序多次查询，或使用 JOIN 来代替子查询。
        </strong>
       </p>
      </li>
      <li>
       <p>
        衍生表建不了索引
       </p>
      </li>
     </ul>
    </blockquote>
