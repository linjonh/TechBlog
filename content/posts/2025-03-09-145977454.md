    <h2>
     <a id="WorkQueue_26">
     </a>
     WorkQueue(多消费)
    </h2>
    <blockquote>
     <p>
      一个队列，多个消费者
      <br/>
      消费者之间属于竞争关系(队列中的一个数据只能被一个消费者获取)
      <br/>
      这点和交换机与队列的关系不同（由交换）
     </p>
    </blockquote>
    <p>
     <mark>
      消费者从queue中获取数据，进行处理，如果不设置其他操作，每一个消费者只需要处理n/m个数据（n为生产者生成数，m为消费者消费数）
     </mark>
    </p>
    <blockquote>
     <p>
      下面我们在一个类建了两个消费者方法，一般不会这样做（因为他们消耗的是同一个机器资源）
      <br/>
      正常我们会写一个消费者方法，复制多个springboot项目实例部署，形成集群
      <br/>
      启动多个消费者，绑定到这个队列，这样里面的代码逻辑也相同，减轻单个消费者服务器的压力
     </p>
    </blockquote>

    <h3>
     <a id="_44">
     </a>
     能者多劳配置
    </h3>
    <p>
     新需求
     <br/>
     让消费者1每秒处理40条消息，消费者2每秒处理5条消息
     <br/>
     代码改造
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/934dd88ae67e458f9fdb8e13ab4632c4.png"/>
     <br/>
     出现问题
     <br/>
     我们消息消费速度跟不上消息分配速度
     <br/>
     我们消息分配还是一人一条这样分配，先给1再给2
     <br/>
     这样没有考虑到谁消费的快的问题，这样的话，2处理的慢，你还给人分50条就太离谱了，影响整体性能
     <br/>
     如果可以让1消费完去帮助2消费就可以了
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/fde701d101c84226a2c369bce85b4461.png"/>
    </p>
    <p>
     那么如何让先消费完的消费者优先获取消息呢？
    </p>
    <blockquote>
     <p>
      如何发挥我们MQ消费者最大性能（处理完的先分配）
      <br/>
      而不是轮询，做以下设置
      <br/>
      保证同一时刻给一条信息，处理完就会再给，不空闲
      <br/>
      这样处理快的可能处理2条的同时慢的处理1条，发挥最大效能
      <br/>
      在消费者项目的配置文件中进行配置
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ad8242e8d0ca4ee092a92fea4e6d498d.png"/>
      <br/>
      这样就实现了最大效率
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/5f25e657ad99403e93c639271b29b277.png"/>
     </p>
    </blockquote>

    <h3>
     <a id="fanout_74">
     </a>
     fanout交换机
    </h3>
    <p>
     队列发的消息，只可能被一个消费者处理
     <br/>
     而通过我们这个交互机可以达到同一个消费被多个连接不同队列的消费者处理（因为队列没有广播性质）
    </p>
    <blockquote>
     <p>
      应用场景：比如说我们之前的支付案例
      <br/>
      我们会干1.通知订单服务2.通知短信服务3.通知积分服务
      <br/>
      三件事都要做，不同的微服务相当于消费者，不同服务连接不同队列
      <br/>
      如果你只能发送一条信息，只有三者之一可以实现，如果可以广播，三者都可以实现
     </p>
    </blockquote>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/ad81105be9f8487ead98b900ada5959d.png"/>

     业务改造（实际例子）
    </h2>
    <p>
     从OpenFegin的同步调用改成基于MQ的异步调用
    </p>
    <blockquote>
     <p>
      订单状态在支付失败时候更新为支付失败,通知服务和积分服务就不会执行
