    <h3>
     <br/>
     make/Makefile的重要性
    </h3>
    <p>
    </p>
    <ul>
     <li>
      会不会写Makefile，从侧面说明了一个人是否具备完成大型工程的能力。
     </li>
     <li>
      一个工程的源文件不计其数，按照其类型、功能、模块分别放在若干个目录当中，Makefile定义了一系列的规则来指定：哪些文件需要先编译，哪些文件需要后编译，甚至于进行更复杂的功能操作。
     </li>
     <li>
      Makefile带来的好处就是“自动化编译”，一旦写好，只需一个make命令，整个工程完全自动编译，极大的提高了软件开发的效率。
     </li>
     <li>
      mak是一个命令工具，是一个解释Makefile当中指令的命令工具，一般来说，大多数的IDE都有这个命令，例如：Delphi的make，Visual
     </li>
     <li>
      C++的nmake，Linux下GNU的make。可见，Makefile都成为了一种在工程方面的编译方法。
     </li>
     <li>
      <strong>
       make是一条命令，Makefile是一个文件
      </strong>
      ，两个搭配使用，完成项目自动化构建。
     </li>
    </ul>
    <p>
     <strong>
      依赖关系和依赖方法
     </strong>
     <br/>
     在使用make/Makefile前我们首先应该理解各个文件之间的依赖关系以及它们之间的依赖方法。
    </p>
    <p>
     <strong>
      依赖关系
     </strong>
     ： 文件A的变更会影响到文件B，那么就称文件B依赖于文件A。
    </p>
    <p>
     例如，test.o文件是由test.c文件通过预处理、编译以及汇编之后生成的文件，所以test.c文件的改变会影响test.o，所以说test.o文件依赖于test.c文件。
     <br/>
     <strong>
      依赖方法
     </strong>
     ： 如果文件B依赖于文件A，那么通过文件A得到文件B的方法，就是文件B依赖于文件A的依赖方法。
    </p>
    <p>
     例如，test.o依赖于test.c，而test.c通过gcc -c test.c -o
     <br/>
     test.o指令就可以得到test.o，那么test.o依赖于test.c的依赖方法就是gcc -c test.c -o test.o。
    </p>
    <blockquote>
     <p>
      <strong>
       就相当于月末没钱，找父母要，其中依赖关系是我是你孩子，依赖方法就是要钱，缺一不可。
      </strong>
     </p>
    </blockquote>
    <p>
     <br/>
     多文件编译
     <br/>
     当你的工程当中有源文件的时候，应该如何进行编译生成可执行程序呢？
    </p>
    <h4>
     gcc
    </h4>
    <p>
     首先，我们可以直接使用gcc指令对源文件进行编译，进而生成可执行程序。
    </p>
    <p>
     <img alt="" height="78" src="https://i-blog.csdnimg.cn/direct/4868b58aa73a4392bdeef8d74699a8c4.png" width="608"/>
    </p>
    <p>
     <strong>
      单个文件
     </strong>
     ：使用源文件直接生成可执行程序
    </p>
    <p>
     <img alt="" height="117" src="https://i-blog.csdnimg.cn/direct/1e5cdfdf598a41108fef8050d5108ba6.png" width="881"/>
    </p>
    <p>
     但进行
     <strong>
      多文件编译
     </strong>
     的时候一般不使用源文件直接生成可执行程序，而是先用每个源文件各自生成自己的二进制文件，然后再将这些二进制文件通过链接生成可执行程序。
    </p>
    <p>
     <img alt="" height="196" src="https://i-blog.csdnimg.cn/direct/d7ebd0a6e2824e74a9fd2a09f3c57807.png" width="937"/>
    </p>
    <p>
     1.先各自生成二进制文件
    </p>
    <p>
     2.将二进制文件通过链接生成可执行程序
    </p>
    <p>
     原因：
    </p>
    <p>
     若是直接使用源文件生成可执行程序，那么其中一个源文件进行了修改，再生成可执行程序的时候就需要将所以的源文件重新进行编译链接。
     <br/>
     而若是先用每个源文件各自生成自己的二进制文件，那么其中一个源文件进行了修改，就只需重新编译生成该源文件的二进制文件，然后再将这些二进制文件通过链接生成可执行程序即可。
    </p>
    <h4>
     <br/>
     make 和Makefile
    </h4>
    <p>
     但是随着源文件个数的增加，我们每次重新生成可执行程序时，所需输入的gcc指令的长度与个数也会随之增加。这时我们就需要使用
     <strong>
      make和Makefile（m可以大写或小写）
     </strong>
     了，这将大大减少我们的工作量
    </p>
    <p>
     第一步：在源文件所在目录下创建一个名为Makefile/makefile的文件。
    </p>
    <p>
     <img alt="" height="160" src="https://i-blog.csdnimg.cn/direct/9944ac802eec40cbbcf3ec21c38b1d92.png" width="899"/>
    </p>
    <p>
     第二步 ：用
     <a class="link-info" href="https://blog.csdn.net/LJY_CF/article/details/143665528" title="vim">
      vim
     </a>
     编写格式
    </p>
    <p>
     第一种
    </p>
    <p>
     <img alt="" height="616" src="https://i-blog.csdnimg.cn/direct/22119d220b334834bc9cb61ff4e1a903.png" width="999"/>
    </p>
    <p>
     <strong>
      黄色的为依赖关系白色的为方法
     </strong>
    </p>
    <p>
    </p>
    <p>
     <img alt="" height="297" src="https://i-blog.csdnimg.cn/direct/5186a76b02824a2eb4bba7546473b74a.png" width="1280"/>
    </p>
    <p>
     make是如何⼯作的,在默认的⽅式下，也就是我们只输⼊make命令。
    </p>
    <p>
     那么： 1. make会在当前⽬录下找名字叫“Makefile”或“makefile”的⽂件。
    </p>
    <p>
     2. 如果找到，它会找⽂件中的第⼀个⽬标⽂件（target），在上⾯的例⼦中，他会找到 myproc 这 个⽂件，并把这个⽂件作为最终的⽬标⽂件。
    </p>
    <p>
     3. 如果 myproc ⽂件不存在，或是 myproc 所依赖的后⾯的 myproc.o ⽂件的⽂件修改时间要 ⽐ myproc 这个⽂件新（可以⽤ touch 测试），那么，他就会执⾏后⾯所定义的命令来⽣成 myproc 这个⽂件。
    </p>
    <p>
     4. 如果 myproc 所依赖的 myproc.o ⽂件不存在，那么 make 会在当前⽂件中找⽬标为 myproc.o ⽂件的依赖性，如果找到则再根据那⼀个规则⽣成 myproc.o ⽂件。（这有点像⼀ 个堆栈的过程） 5. 当然，你的C⽂件和H⽂件是存在的啦，于是 make 会⽣成myproc.o ⽂件，然后再⽤myproc.o ⽂件声明 make 的终极任务，也就是执⾏⽂件 hello 了。
    </p>
    <p>
     6. 这就是整个make的依赖性，make会⼀层⼜⼀层地去找⽂件的依赖关系，直到最终编译出第⼀个 ⽬标⽂件。
    </p>
    <p>
     7. 在找寻的过程中，如果出现错误，⽐如最后被依赖的⽂件找不到，那么make就会直接退出，并 报错，⽽对于所定义的命令的错误，或是编译不成功，make根本不理。
    </p>
    <p>
     第二种
    </p>
    <p>
     <img alt="" height="443" src="https://i-blog.csdnimg.cn/direct/0d1d710fae7948b087504d68ce3f2e03.png" width="1183"/>
    </p>
    <p>
     <strong>
      Makefile文件的简写方式：
     </strong>
    </p>
    <p>
     <img alt="" height="495" src="https://i-blog.csdnimg.cn/direct/511397beb3634f4dbe53f1d683cdc829.png" width="503"/>
    </p>
    <h4>
     变量定义部分
    </h4>
    <ol>
     <li>
      <code>
       BIN=test
      </code>
      ：定义变量
      <code>
       BIN
      </code>
      ，代表最终生成的可执行文件名，这里设置为
      <code>
       test
      </code>
      。
     </li>
     <li>
      <code>
       SRC=$(wildcard *.c)
      </code>
      ：使用
      <code>
       wildcard
      </code>
      函数获取当前目录下所有扩展名为
      <code>
       .c
      </code>
      的源文件，将结果赋值给变量
      <code>
       SRC
      </code>
      。
     </li>
     <li>
      <code>
       OBJ=$(SRC:.c=.o)
      </code>
      ：利用变量替换功能，把
      <code>
       SRC
      </code>
      中所有
      <code>
       .c
      </code>
      后缀的文件名替换为
      <code>
       .o
      </code>
      后缀，得到对应的目标文件列表，赋值给变量
      <code>
       OBJ
      </code>
      。
     </li>
     <li>
      <code>
       CC=gcc
      </code>
      ：定义变量
      <code>
       CC
      </code>
      ，指定使用
      <code>
       gcc
      </code>
      作为编译器。
     </li>
     <li>
      <code>
       RM=rm -f
      </code>
      ：定义变量
      <code>
       RM
      </code>
      ，表示删除文件的命令，
      <code>
       -f
      </code>
      选项表示强制删除，不询问。
     </li>
    </ol>
    <h4>
     编译规则部分
    </h4>
    <ol>
     <li>
      <code>
       $(BIN):$(OBJ)
      </code>
      ：这是一个依赖规则，表明可执行文件
      <code>
       $(BIN)
      </code>
      （即
      <code>
       test
      </code>
      ）依赖于所有的目标文件
      <code>
       $(OBJ)
      </code>
      （由
      <code>
       .c
      </code>
      文件编译生成的
      <code>
       .o
      </code>
      文件）。
     </li>
     <li>
      <code>
       $(CC) $^ -o $@
      </code>
      ：当满足上一条依赖规则需要重新生成目标时，执行此命令。
      <code>
       $(CC)
      </code>
      是前面定义的
      <code>
       gcc
      </code>
      编译器；
      <code>
       $^
      </code>
      是自动化变量，代表所有依赖文件，即
      <code>
       $(OBJ)
      </code>
      中的目标文件；
      <code>
       -o
      </code>
      是
      <code>
       gcc
      </code>
      选项，用于指定输出文件名；
      <code>
       $@
      </code>
      是自动化变量，代表当前规则的目标文件，即
      <code>
       $(BIN)
      </code>
      （
      <code>
       test
      </code>
      ）。该命令整体意思是用
      <code>
       gcc
      </code>
      将所有目标文件链接成可执行文件
      <code>
       test
      </code>
      。
     </li>
     <li>
      <code>
       echo "链接 $^ 成 $@"
      </code>
      ：执行链接命令后，输出提示信息，说明正在将哪些依赖文件链接成哪个目标文件。
     </li>
    </ol>
    <h4>
     模式规则部分
    </h4>
    <ol>
     <li>
      <code>
       %.o:%.c
      </code>
      ：这是一个模式规则，
      <code>
       %
      </code>
      是通配符，表示所有
      <code>
       .o
      </code>
      文件依赖于对应的
      <code>
       .c
      </code>
      文件，比如
      <code>
       main.o
      </code>
      依赖于
      <code>
       main.c
      </code>
      。
     </li>
     <li>
      <code>
       $(CC) -c $&lt;
      </code>
      ：针对上述模式规则，当
      <code>
       .c
      </code>
      文件发生变化需要重新编译时执行此命令。
      <code>
       $(CC)
      </code>
      是
      <code>
       gcc
      </code>
      编译器；
      <code>
       -c
      </code>
      选项表示只进行编译，不链接；
      <code>
       $&lt;
      </code>
      是自动化变量，代表当前规则的第一个依赖文件，也就是对应的
      <code>
       .c
      </code>
      源文件。该命令会将
      <code>
       .c
      </code>
      文件编译成
      <code>
       .o
      </code>
      目标文件。
     </li>
     <li>
      <code>
       echo "编译... $&lt; 成 $@"
      </code>
      ：编译完成后，输出提示信息，显示正在将哪个
      <code>
       .c
      </code>
      文件编译成哪个
      <code>
       .o
      </code>
      文件。
     </li>
    </ol>
    <h4>
     伪目标和清理规则部分
    </h4>
    <ol>
     <li>
      <code>
       .PHONY:clean
      </code>
      ：声明
      <code>
       clean
      </code>
      为伪目标。这意味着即使当前目录下存在名为
      <code>
       clean
      </code>
      的文件，
      <code>
       make
      </code>
      也会执行
      <code>
       clean
      </code>
      规则，而不是认为它已经是最新的而不执行。
     </li>
     <li>
      <code>
       clean:
      </code>
      ：
      <code>
       clean
      </code>
      规则的开始。
     </li>
     <li>
      <code>
       @$(RM) $(OBJ) $(BIN)
      </code>
      ：执行此命令，
      <code>
       @
      </code>
      表示执行命令时不显示命令本身；
      <code>
       $(RM)
      </code>
      是前面定义的删除命令；该命令会删除所有目标文件
      <code>
       $(OBJ)
      </code>
      和可执行文件
      <code>
       $(BIN)
      </code>
      （
      <code>
       test
      </code>
      ） ，起到清理编译生成文件的作用。
     </li>
    </ol>
    <h3>
     Linux小练习 ———倒计时
    </h3>
    <p>
     <img alt="" height="523" src="https://i-blog.csdnimg.cn/direct/1674aa9cfed74e2d8cf3a3fd67c783d4.png" width="601"/>
    </p>
    <p>
     <img alt="" height="69" src="https://i-blog.csdnimg.cn/direct/0735305566a34849bcfde4a056a2829d.png" width="632"/>
    </p>
    <blockquote>
     <ul>
      <li>
       <strong>
        <code>
         printf("%-2d\r", a);
        </code>
       </strong>
       ：使用
       <code>
        printf
       </code>
       函数输出变量
       <code>
        a
       </code>
       的值。
       <code>
        %-2d
       </code>
       是格式化字符串，
       <code>
        -
       </code>
       表示左对齐，
       <code>
        2
       </code>
       表示输出宽度为 2 个字符；
       <code>
        \r
       </code>
       是回车符，在 Linux 终端中，它会将光标移到当前行的开头。
      </li>
      <li>
       <strong>
        <code>
         fflush(stdout);
        </code>
       </strong>
       ：刷新标准输出缓冲区。在 Linux 系统中，为了提高 I/O 效率，输出操作往往不是立即执行的，而是先将数据存储在缓冲区中。调用
       <code>
        fflush(stdout)
       </code>
       可以强制将缓冲区中的数据立即输出到终端。
      </li>
     </ul>
    </blockquote>
    <h3>
     小练习——进度条
     <img alt="" height="85" src="https://i-blog.csdnimg.cn/direct/69b1982070444d6fbc45c9214b71a980.png" width="1833"/>
    </h3>
    <h4>
     1.
     <code>
      process.h
     </code>
     和
     <code>
      process.c
     </code>
     文件
    </h4>
    <h5>
     宏定义
    </h5>
    <pre><code>#define NUM 101
#define STYLE '='
</code></pre>
    <ul>
     <li>
      <code>
       NUM
      </code>
      定义了进度条字符数组的长度，
      <code>
       101
      </code>
      用于存储进度条字符和字符串结束符
      <code>
       '\0'
      </code>
      。
     </li>
     <li>
      <code>
       STYLE
      </code>
      定义了进度条使用的字符，这里使用
      <code>
       =
      </code>
      来表示进度。
     </li>
    </ul>
    <h5>
     <code>
      process_v1
     </code>
     函数
    </h5>
    <pre><code>void process_v1()
{
    char buffer[NUM];
    memset(buffer, 0, sizeof(buffer));
    const char *lable="|/-\\";
    int len = strlen(lable);
    int cnt = 0;
    while(cnt &lt;= 100)
    {
        printf("[%-100s][%d%%][%c]\r", buffer, cnt, lable[cnt%len]);
        fflush(stdout);
        buffer[cnt]= STYLE;
        cnt++;
        usleep(50000);
    }
    printf("\n");
}
</code></pre>
    <ul>
     <li>
      <strong>
       初始化部分
      </strong>
      ：
      <ul>
       <li>
        <code>
         char buffer[NUM];
        </code>
        ：定义一个字符数组
        <code>
         buffer
        </code>
        用于存储进度条的字符。
       </li>
       <li>
        <code>
         memset(buffer, 0, sizeof(buffer));
        </code>
        ：将
        <code>
         buffer
        </code>
        数组的所有元素初始化为
        <code>
         0
        </code>
        ，即空字符
        <code>
         '\0'
        </code>

void process_v1() {
    char buffer[NUM];
    memset(buffer, 0, sizeof(buffer));
    const char *lable = "|/-\\";
    int len = strlen(lable);
    int cnt = 0;
    while (cnt &lt;= 100) {

        buffer[cnt] = STYLE;
        cnt++;
        usleep(50000);
    }
    printf("\n");
}

    double rate = current / total;
