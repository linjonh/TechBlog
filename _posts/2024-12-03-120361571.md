    <h3>
     <a id="_1">
     </a>
     一、前言
    </h3>
    <p>
     分布式数据库（ Distributed Database ），简称DDB，即物理上分散而逻辑上集中的数据库系统。
    </p>
    <p>
     它是数据库技术与计算机网络相结合的产物，也因此，其高度依赖高速网络。
    </p>
    <p>
     分布式数据库系统（ Distributed Database System)，简称DDBS。分布式数据库管理系统简称为DDBMS。
     <strong>
      分布式数据库系统DDBS = DDB + DDBMS
     </strong>
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/188ff17a5f7ca6fe2b23c49c96035fb4.png">
      <br/>
      比较经典的一句话是，分布式数据库应该看起来完全像一个非分布式数据库。
     </img>
    </p>
    <h4>
     <a id="_10">
     </a>
     信息孤岛
    </h4>
    <p>
     信息孤岛是指相互之间在功能上不关联互助、信息不共享互换以及信息与业务流程和应用相互脱节的计算机应用系统。
    </p>
    <p>
     如果这些独立的系统之前进行交互，会出现复杂的交互网络。
    </p>
    <h4>
     <a id="_14">
     </a>
     分布式数据库系统的特点
    </h4>
    <ul>
     <li>
      物理分布性：数据不是存放在一个站点上
     </li>
     <li>
      逻辑整体性：是与分散式数据库系统的区别
     </li>
     <li>
      站点自治性：是与多处理机的系统的区别
     </li>
     <li>
      数据分布透明性：用户不必知道所操作的数据放在何处
     </li>
     <li>
      集中与自治相结合：各数据库彼此独立自治，又整体集中
     </li>
     <li>
      存在适当的数据冗余度：适当的数据冗余，在分布式情况下很多时候是方便的
     </li>
     <li>
      事务管理的分布性：分布式事务管理是一个挑战
     </li>
    </ul>

     </li>

      <br/>

    <h3>
     <a id="_37">
     </a>
     三、数据分片
    </h3>
    <p>
     当数据量过于庞大，单机难以支撑时，会面临扩展瓶颈，那么就需要将数据进行拆分，分散在多个数据库实例上。
    </p>
    <p>
     数据分片是指将数据全局划分为相关的逻辑片段，有水平切分、垂直切分、混合切分三种类型。
    </p>
    <ul>
     <li>
      <strong>
       水平切分
      </strong>
      :可以简单地理解为
      <strong>
       按照数据行
      </strong>
      进行切分，即一部分行放在某数据库，另外一部分放在另外的数据库实例。比如可以按照时间、地区拆分，亦或是根据
      <code>
       hash
      </code>
      进行拆分。
     </li>
     <li>
      <strong>
       垂直切分
      </strong>
      ：垂直拆分可以简单理解为
      <strong>
       按照表
      </strong>
      进行分类，将表分布在不同的节点上,基本目标是将使用频繁的属性聚集在一起
     </li>
    </ul>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/00f8bd65cc5569c493fe6e0ec469322c.png">
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/9a91101c3870a39804cf14076f781892.png">
       <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/2097e83e42229be79b579f95641f54cd.png"/>
      </img>
     </img>
    </p>
    <ul>
     <li>
      <strong>
       混合切分
      </strong>
      ：水平切分与垂直切合的结合。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/17be990565da2ecb8f275e92ae59d896.png"/>
     </li>
    </ul>

     </li>
     <li>
      最小的：分配的和实际应用的是密切相关的
     </li>
    </ul>

    <h3>
     <a id="_72">
     </a>
     四、分布式查询处理及优化
    </h3>
    <p>
     分布式数据库需要考虑查询问题，其需要做的就是把
     <strong>
      一个分布式数据库上的高层次查询映射为本地数据库上的操作，最后通过网络通信，将操作结果汇聚起来。
     </strong>
    </p>
    <p>
     相对于集中式数据库，分布式数据库还要考虑额外的几个问题：
    </p>
    <ul>
     <li>
      选择最优站点查询
     </li>
     <li>
      数据传送方式
     </li>
     <li>
      站点之间交换数据的问题
     </li>
    </ul>
    <p>
     相对于集中式的查询目标，分布式需要多考虑一项
     <strong>
      “通信开销代价”
     </strong>
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/9b7679714251605aba0a225c9c58849e.png"/>
    </p>
    <h4>
     <a id="_84">
     </a>
     对水平分片的优化
    </h4>
    <ul>
     <li>
      尽量
      <strong>
       把选择条件下移到分片的限定关系
      </strong>
      处，再把分片的限定关系与选择条件进行比较，去掉它们之间存在矛盾的相应片断。
     </li>
     <li>
      如果最后剩下一个水平片断，则在重构全局关系的操作中，
      <strong>
       就可去掉“并”操作.
      </strong>
     </li>
    </ul>
    <h4>
     <a id="_87">
     </a>
     对垂直分片的优化
    </h4>
    <ul>
     <li>
      把垂直分片所用到的属性集，与查询条件中的投影操作所涉及的属性相比较，
      <strong>
       去掉无关的垂直片断
      </strong>
      。
     </li>
     <li>
      如果最后只剩下一个垂直片断与查询有关时，
      <strong>
       去掉
      </strong>
      重构全局关系的**“连接”**操作（至少可以减少“连接”操作的次数）
     </li>
    </ul>
    <h4>
     <a id="_91">
     </a>
     基于半连接算法的查询优化
    </h4>
    <p>
     网络中两个站点间进行连接操作时，需要将一个站点的关系通过网络传输到另一站点与在该站点上的关系进行连接操作。在这个过程中，如果传输整个关系，那么网络传输中的数据量会很大，网络本身复杂多变，如果一次性传输较大的数据量定会产生不少问题，而在实际的连接操作中，并非所有的数据都参与连接操作。
    </p>
    <p>
     因此为减少传输数据量，可以禁止那些不参与或者无用的数据在网络中传输，而半连接操作就是针对这一问题提出来的，它要实现的目标是
     <strong>
      减少进行连接操作关系的数据量
     </strong>
     ，从而减少在网络上的传输的数据量，
     <strong>
      但同时在某种程度上会增加通信的次数以及本地处理的时间
     </strong>
     。
    </p>
    <p>
     简单来说，就是在连接之前，先消除无用数据，减少网路通信中传输的数据量。
    </p>

    <h4>
     <a id="1_126">
     </a>
     1.分布式数据库的基本封锁算法
    </h4>
    <h5>
     <a id="_127">
     </a>
     简单分布式封锁法
    </h5>
    <ul>
     <li>
      <strong>
       封锁全部副本
      </strong>
      （各站点负责各自数据的封锁管理）
     </li>
     <li>
      过程消息（请求封锁、封锁确认、请求更新、更新确认、解除封锁）需要发送n次
     </li>
     <li>
      各站点间进行相当大的数据传输。
     </li>
    </ul>
    <h5>
     <a id="_131">
     </a>
     主站点封锁法(集中封锁法)
    </h5>
    <ul>
     <li>
      选定一个站点为“主站点”，负责系统全部封锁管理。
     </li>
     <li>
      容易造成“瓶颈”，制约可靠性和可用性
     </li>
    </ul>
    <h4>
     <a id="22PL_135">
     </a>
     2.两阶段封锁协议(2PL)
    </h4>
    <ul>
     <li>
      <strong>
       所有的封锁操作必须在第一次解锁操作之前
      </strong>
     </li>
     <li>
      任何事务在操作数据之前必须获得锁
     </li>
     <li>
      第一阶段是封锁阶段，也称扩张阶段，第二阶段是解锁阶段，也称收缩阶段
     </li>
     <li>
      第一阶段和第二阶段间隔点称为封锁点
     </li>
    </ul>
    <p>
     若分布式数据库中，事务遵循
     <code>
      2PL
     </code>
     ，则分布式数据库事务处理是可串行化的。
    </p>
    <h5>
     <a id="2PL_142">
     </a>
     2PL分类
    </h5>
    <ul>
     <li>
      基本的2PL（在完成数据项的访问后立即释放锁，
      <strong>
       可能产生死锁，可能读到“脏数据”
      </strong>
      ）
     </li>
     <li>
      保守的2PL（事务在执行操作前获得所有操作数据上的锁，
      <strong>
       不会产生死锁，但难以实现
      </strong>
      ）
     </li>
     <li>
      严格的2PL（事务在提交或撤销前不能释放任何排他锁，
      <strong>
       可避免脏数据，不能避免死锁
      </strong>
      ）
     </li>
     <li>
      严酷的2PL（事务在提交或撤销前不能释放任何锁，
      <strong>
       也不能避免死锁
      </strong>
      ）
     </li>
    </ul>

    <h4>
     <a id="3__152">
     </a>
     3. 多粒度封锁
    </h4>
    <ul>
     <li>
      允许多粒度树中的每个结点被独立地加锁。
     </li>
     <li>
      对一个结点加锁意味着这个结点的所有后裔结点也被加以同样类型的锁。
     </li>
     <li>
      数据项可能以两种方式封锁：显式封锁和隐式封锁，显示封锁和隐式封锁不能冲突。
     </li>
     <li>
      检查会比较是否加锁会比较麻烦，故为了管理的方便引入
      <code>
       意向锁
      </code>
      。
     </li>
    </ul>
    <h4>
     <a id="4_157">
     </a>
     4.意向锁
    </h4>
    <ul>
     <li>
      如果对一个节点加意向锁，则说明该节点的
      <strong>
       下层节点正在被封锁
      </strong>
     </li>
     <li>
      对任一节点封锁时，必须先对它的上层节点加意向锁
     </li>
     <li>
      意向锁指出该节点的某个后代需要锁的类型
     </li>
    </ul>
    <h5>
     <a id="_162">
     </a>
     意向锁的类型
    </h5>
    <p>
     意向锁并不是真正锁数据，它是表明一种意思，即该结点的下层结点正在实际加锁
    </p>
    <ul>
     <li>
      <strong>
       意向共享锁(IS)
      </strong>
      <br/>
      对一个数据对象加IS锁，表示某些后代将会请求S锁。
      <br/>
      、例：若对元组加S锁，则对数据库和关系要加IS锁。
     </li>
     <li>
      <strong>
       意向排它锁(IX)
      </strong>
      <br/>
      对一个数据对象加IX锁，表示可能对其下层结点加X锁。例：若对元组加X锁，则对数据库和关系要加IX锁。
     </li>
     <li>
      <strong>
       共享意向排它锁(SIX=S+IX)
      </strong>
      <br/>
      对一个数据对象加SIX锁，表示当前结点处于S封锁中，但是下层某些结点将请求X锁。即事务要读下层结点中的对象，还可能要更新一些对象，因此要对当前结点加
      <code>
       IX
      </code>
      锁。
     </li>
    </ul>
    <h5>
     <a id="_173">
     </a>
     锁的相容矩阵
    </h5>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/7b6cc83fd8572c15455624747b6b8d75.png"/>
     <br/>
     需要注意的是两个事务对某个结点T的SIX和IX是不相容的，因为对于SIX=S+IX，那么根据多粒度锁树的概念，T结点的子结点是加S锁和X锁的，那么这就会出现读写锁冲突。
    </p>
    <h4>
     <a id="5_177">
     </a>
     5.可串行化调度
    </h4>
    <p>
     串行调度：
    </p>
    <ul>
     <li>
      一个事务的第一个动作是在另一个事务的最后一个动作完成后开始。即调度中事务的各个操作不会交叉，每个事务相继执行。
     </li>
     <li>
      串行调度总是可以正确执行，但是串行调度效率很低。
     </li>
    </ul>
    <p>
     如果一个调度等价于串行调度，那么称该调度为
     <strong>
      可串行化调度
     </strong>
     。
    </p>

    <h3>
     <a id="_192">
     </a>
     并发控制的时标技术
    </h3>
    <h4>
     <a id="_193">
     </a>
     基于时标的并发控制方法
    </h4>
    <ul>
     <li>
      给每个事务赋予一个唯一的时标，事务的执行等效于按时标次序串行执行。如果发生冲突，则通过撤消并重启一个事务来解决的。事务重新启动时，则赋予新的时标。
     </li>
     <li>
      时标(Time Stamp)用来唯一识别每个事务并允许排序的标识
     </li>
     <li>
      时标具有唯一性和单调性，可以采用计数器或系统时钟来产生时标，在分布式系统中有全局时标和本地时标之分。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/1f8930dae3d3dd091a9e9156b1d3e880.png"/>
     </li>
    </ul>
    <h3>
     <a id="_199">
     </a>
     七、分布式事务控制
    </h3>
    <ul>
     <li>
      事务是访问或更新各种数据项的最小逻辑工作单位
     </li>
     <li>
      一个分布式事务由
      <strong>
       主事务
      </strong>
      （负责事务的开始、提交或异常终止）和
      <strong>
       多个子事务
      </strong>
      （局部事务，
      <strong>
       完成对数据的操作
      </strong>
      ）组成。
     </li>
     <li>
      分布式事务与一般事务的不同表现在，全局事务的主事务和子事务都完成才能提交，任何一个子事务失败，都要Rollback
     </li>
    </ul>
