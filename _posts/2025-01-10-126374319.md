    <h3>
     <a id="1_ES6_constlet_14">
     </a>
     😃1. 请你说下为什么ES6中引入了 const,let
    </h3>
    <p>
     <code>
      回答思路:
     </code>
    </p>
    <ul>
     <li>
      没有
      <code>
       const、let
      </code>
      之前有什么弊端
     </li>
     <li>
      <code>
       const、let
      </code>
      解决了什么问题
     </li>
    </ul>
    <p>
     <strong>
      扩展问题：
     </strong>
    </p>
    <ul>
     <li>
      let、const 存储方式
     </li>
    </ul>
    <p>
     <strong>
      开始回答
     </strong>
    </p>
    <ol>
     <li>
      在没有
      <code>
       const、let
      </code>
      之前，我们使用
      <code>
       var
      </code>
      来定义变量，因此我们的
      <code>
       作用域
      </code>
      只分为两种，也就是全局作用域和块级作用域，因此，我们在使用像
      <code>
       for或者if
      </code>
      这种关键词时，会有很大的隐患，可能造成变量冲突。
      <br/>
      <strong>
       比如说：
      </strong>
     </li>
    </ol>
    <ul>
     <li>
      <p>
       for: 正常来说，我们在 for 循环中定义其他变量，在结束后 for 中定义的变量应该被销毁，不会修改全局变量，但是由于没有块级作用域的限制，使得本应该销毁的变量没有被销毁。
      </p>
     </li>
     <li>
      <p>
       if: 正常我们在
       <code>
        fun
       </code>
       函数中会打印x的值1，但是由于
       <code>
        var
       </code>
       变量提升，并且没有块级作用域的限制，虽然
       <code>
        if
       </code>
       中的代码不会执行，但是变量 x 会覆盖
       <code>
        fun
       </code>
       外部的值，导致
       <code>
        x打印undefined
       </code>
       。
      </p>
      <pre><code>  var i = 5;                               
  for(var i = 3; i &lt; 10; i ++){            
      // 其他操作                                 
  }                                        
  console.log(i) 
  ---------------------------
  var x = 1
  function fun (){
      console.log(x)
      if(false){
          var x = 1
      }
      console.log(x)
  }
</code></pre>
     </li>
    </ul>
    <ol start="2">
     <li>
      在引入
      <code>
       let、const
      </code>
      之后，我们引入了新的概念，块级作用域，很好的解决了变量提升和无块级作用域带来的变量冲突的问题。那是如何解决的呢？我们引入新的问题。
     </li>
    </ol>
    <p>
     <strong>
      扩展回答：
     </strong>
    </p>
    <ol>
     <li>
      在js代码执行之前会经历编译阶段，而在编译阶段会确定当前执行上下文所需要的信息，
      <code>
       变量环境
      </code>
      ,
      <code>
       词法环境
      </code>
      ,
      <code>
       this
      </code>
      ,
      <code>
       作用域链
      </code>
      ，其中，变量环境用来存放
      <code>
       var
      </code>
      和
      <code>
       function
      </code>
      定义的变量，并且初始化值为
      <code>
       undefined
      </code>
      和
      <code>
       地址值
      </code>
      ，而在执行前变量已经定义好了，也就是所谓的变量提升。而
      <code>
       const
      </code>
      和
      <code>
       let
      </code>
      定义的变量存储在词法环境中，在词法环境中维护一个小型栈，按照块的形式，将每一个块级变量压入栈顶，执行后弹出，并且不会初始化，因此在一个块级作用域中提前访问变量会报错，也就是所谓的
      <code>
       暂时性死区
      </code>
      。
     </li>
    </ol>
    <p>
     <code>
      变量查找顺序
     </code>
     ：先从词法环境中从栈顶到栈底，然后到变量环境中查找，之后沿着作用域链查找。
    </p>
    <h5>
     <a id="_50">
     </a>
     相关题型：
    </h5>
    <ul>
     <li>
      给代码看输出
     </li>
    </ul>
    <blockquote>
     <p>
      <strong>
       注意点
      </strong>
      ：
      <code>
       var
      </code>
      和
      <code>
       function
      </code>
      变量提升，并且
      <code>
       function
      </code>
      优先于
      <code>
       var
      </code>
      ,
      <code>
       const 、let
      </code>
      块级作用域中的暂时性死区。
     </p>
    </blockquote>
    <h3>
     <a id="2_Promise_53">
     </a>
     😃2. 请你说下Promise?
    </h3>
    <p>
     <code>
      回答思路
     </code>
     ：
    </p>
    <ul>
     <li>
      首先回答
      <code>
       Promise
      </code>
      是什么？
     </li>
     <li>
      <code>
       Promise
      </code>
      为什么出现，没出现前有什么问题，出现后解决了什么问题？
     </li>
    </ul>
    <p>
     <strong>
      拓展问题：
     </strong>
    </p>
    <ul>
     <li>
      <code>
       Promise
      </code>
      执行机制，微任务宏任务
     </li>
     <li>
      <code>
       Promise
      </code>
      的缺陷，
      <code>
       async/await
      </code>
      语法糖的出现
     </li>
    </ul>
    <p>
     <strong>
      开始回答：
     </strong>
    </p>
    <ol>
     <li>
      首先，从字面意思来理解，
      <code>
       Promise
      </code>
      期约， 是一个用来执行将来要发生的或者即将要发生的事件的对象，它自身有三种状态，
      <code>
       pending
      </code>
      ,
      <code>
       fulfilled
      </code>
      ，
      <code>
       rejected
      </code>
      ,同一时刻只能有一种状态，一旦状态改变，则不能再更改，也不可逆。通过构造函数的方式使用，传入一个执行器回调函数，以此来决定
      <code>
       Promise
      </code>
      的执行状态。之后通过
      <code>
       then
      </code>
      方法来决定执行什么样的操作，并且
      <code>
       then
      </code>
      方法返回一个值会自动包装为
      <code>
       Promise
      </code>
      从而实现对象的链式调用。（之后可以介绍下其他方法
      <code>
       race, all, resolve, reject
      </code>
      ）
     </li>
     <li>
      在
      <code>
       Promise
      </code>
      出现之前，我们在书写异步代码时，通过回调的方式来拿到异步返回的值，代码抒写逻辑不连续，除此之外，如果下一次的执行需要依赖上一次的执行结果，会导致代码嵌套，如果嵌套次数太多，就造成了新的问题，回调地狱，使得代码难看难以维护，
      <code>
       Promise
      </code>
      的出现，通过
      <code>
       then
      </code>
      方法，解决了函数嵌套的问题，
      <code>
       then
      </code>
      方法的链式调用，也将回调地狱的问题迎刃而解。
     </li>
    </ol>
    <blockquote>
     <p>
      上面回答，只是简单的介绍了下
      <code>
       Promise
      </code>
      的用法，没有深入的去讲
      <code>
       Promise
      </code>
      ,有能力的话我们应该扩展去讲它周边的知识
     </p>
    </blockquote>
    <p>
     <strong>
      扩展：
     </strong>
    </p>
    <h6>
     <a id="1_Promise_70">
     </a>
     1. Promise执行机制
    </h6>
    <p>
     <code>
      Promise
     </code>
     除了解决异步回调的问题之外，还有一个特性，就是它的执行时机,
     <code>
      微任务
     </code>
     ,和
     <code>
      微任务
     </code>
     对应的还有
     <code>
      宏任务
     </code>
     ，说到这我们不得不讲下浏览器的异步实现机制，为了解决 js 单线程（
     <code>
      可能涉及问题：js为什么是单线程？
     </code>
     ）同步执行的效率问题，我们引入了异步执行机制，而异步执行依赖于 v8引擎 中的
     <code>
      消息队列和事件循环
     </code>
     机制,也就是js在执行过程中遇到异步任务时，不会立即执行，而是将该事件存放到消息队列当中，而消息队列中存放的任务也就是我们所谓的
     <code>
      宏任务
     </code>
     ，然后继续执行js代码，当所有同步代码执行完毕之后，通过
     <code>
      事件循环
     </code>
     ,也就是一个循环代码
     <code>
      for或者while
     </code>
     来不停的从
     <code>
      消息队列
     </code>
     中取出一个事件来执行，这就是异步任务的执行机制。消息队列有一个缺点，就是所有任务都是按顺序执行，因此，如果我们需要执行一些时间粒度小的任务，比如监听DOM的变化去做相应的业务逻辑的时候，再使用消息队列的话会造成严重的效率问题，因为消息队列是先进先出的结构，在
     <code>
      该事件
     </code>
     添加到消息队列尾部时，消息队列内部可能已经有很多任务了，所以宏任务的执行效率不高，这是就引入了新的概念
     <code>
      微任务
     </code>
     ,在宏任务中包含一个
     <code>
      微任务队列
     </code>
     ,来存放需要执行效率高的事件，在每次宏任务执行完之后，不会着急执行下一个宏任务，而是将该宏任务中的
     <code>
      微任务
     </code>
     执行完毕，再去执行下一个
     <code>
      宏任务
     </code>
     。
    </p>
    <ul>
     <li>
      js
      <ul>
       <li>
        添加微任务：Promise, MutationObserver
       </li>
       <li>
        添加宏任务：script，定时器，I/O,交互事件
       </li>
      </ul>
     </li>
     <li>
      node
      <ul>
       <li>
        添加微任务：: nextTick
       </li>
      </ul>
     </li>
    </ul>
    <h6>
     <a id="2_asyncawait__78">
     </a>
     2. async/await 语法糖
    </h6>
    <p>
     <code>
      Promise
     </code>
     虽然可以链式调用来解决回调地狱，但是还是不够完美，依然是回调的方式书写代码，为了更加符合代码逻辑，推出了
     <code>
      async/await
     </code>
     语法糖，来用同步的方式书写异步代码。使用
     <code>
      async
     </code>
     修饰的函数代表异步函数，会自动包装返回一个
     <code>
      Promise
     </code>
     类型的对象，
     <code>
      await
     </code>
     关键字用来暂停执行逻辑。通过
     <code>
      try/cache
     </code>
     来捕获异常，介绍完后，我们来说下它的执行原理。
     <br/>
     <strong>
      执行原理：（自执行的生成器 + 协程）
     </strong>
    </p>
    <ul>
     <li>
      <strong>
       生成器
      </strong>
      (Generator)：
      <code>
       yield/*
      </code>
      ,通过调用
      <code>
       *
      </code>
      函数来返回一个对象，其中有
      <code>
       next,return,trow
      </code>
      方法。next 方法用来恢复
      <code>
       *函数
      </code>
      的执行，
      <code>
       yield
      </code>
      来暂停
      <code>
       *函数
      </code>
      的执行。详细使用大家请自己查阅，这里只做大致介绍。
     </li>
     <li>
      <strong>
       协程
      </strong>
      ：协程是一种比线程更加轻量级的存在,可以把协程看成是跑在线程上的任务，一个线程上可以存在多个协程，但是在线程上同时只能执行一个协程，协程不是被操作系统内核所管理，而完全是由程序所控制。这样带来的好处就是性能得到了很大的提升，不会像线程切换那样消耗资源。
     </li>
    </ul>
    <p>
     通过
     <code>
      *
     </code>
     函数来创建一个协程，但是该协程不会立刻执行，当调用
     <code>
      next
     </code>
     方法时，将控制权交给该协程，开始执行，遇到
     <code>
      yield
     </code>
     关键字后，停止协程，并将控制权返回给父协程，并把 yield 后的值传递给父协程，父协程通过
     <code>
      next（value）
     </code>
     传值给该子协程。这就是生成器可以实现暂停执行的原理。
    </p>

    <h3>
     <a id="_SetMap_253">
     </a>
     😃三. 你知道Set和Map吗?
    </h3>
    <p>
     <code>
      回答思路：
     </code>
    </p>
    <ul>
     <li>
      <code>
       Set和Map
      </code>
      的特点，有什么API
     </li>
     <li>
      有什么不足之处，引入了
      <code>
       WeakMap 和 WeakSet
      </code>
     </li>
    </ul>
    <p>
     <strong>
      扩展问题
     </strong>
    </p>
    <ul>
     <li>
      <code>
       Map和object
      </code>
      比较，性能对比
     </li>
     <li>
      垃圾回收机制（放在浏览器模块讲）
     </li>
    </ul>
    <p>
     <strong>
      开始回答
     </strong>
    </p>
    <ol>
     <li>
      Set 是一种类数组的数据结构，它有一个重要的特点就是Set中不会重复的存储一个值。Map是一种类似object的数据结构，通过键值对的方式进行存储，但是Map的键值可以是任意类型。Set和Map都是可迭代的。
      <br/>
      Api的的介绍就不多说了，自己可以查阅一下。
     </li>
     <li>
      Set和Map有一个问题就是对引用类型的数据是强引用，如果不主动使用
      <code>
       delete
      </code>
      删除数据，这段引用类型数据会永远存在于内存中，不会被垃圾回收机制回收，因为
      <code>
       Set和Map
      </code>
      一直保持对该引用类型数据的引用。因此引入了
      <code>
       WeakMap和WeakSet
      </code>
      ，因为是为了解决对引用类型数据的强引用类型，所以，
      <code>
       WeakMap和WeakSet
      </code>
      与
      <code>
       Set和Map
      </code>
      不同的是，键名只能是引用类型，不可以是字符串或者数组等类型，
      <code>
       WeakMap和WeakSet
      </code>
      对数据是弱引用，因此不会阻止垃圾回收机制回收。因为其弱引用类型，所以也没用
      <code>
       keys,values,entire，size
      </code>
      的api。
     </li>
    </ol>
    <p>
     <strong>
      案例
     </strong>
     <br/>
     正常来讲，btn所指向的内存中会保存该
     <code>
      dom
     </code>
     ，当
     <code>
      btn=null
     </code>
     后会断开
     <code>
      btn同保存dom内存的连接
     </code>
     会回收该段内存，由于Map的强引用，即时连接断开，保存该
     <code>
      dom
     </code>
     的内存也不会被回收。
    </p>
    <pre><code>   let btn = document.getElementById('button')
    let map = new Map([
        [btn, { count: 0 }]
    ])
  btn = null
</code></pre>
    <p>
     <strong>
      扩展回答
     </strong>
    </p>
    <ol>
     <li>
      Map和object 对比：
      <ul>
       <li>
        object 中的数据是有数序的，根据键值，非负数按顺序存储，字符串按插入顺序存储，而Map是根据插入顺序存储
       </li>
       <li>
        object 会对键值做转换，将数组转为字符串类型，而Map中不会，
        <code>
         ‘0’
        </code>
        和
        <code>
         0
        </code>
        是不一样的键值
       </li>
       <li>
        object 是不可迭代的，而Map是可迭代的
        <br/>
        性能对比：Map是一种散列表的数据结构，所以其取值和插入的时间复杂度是O(1)，因此频繁的取值和插入使用Map的性能更高。
       </li>
      </ul>
     </li>
    </ol>
    <p>
     <strong>
      散列表：
     </strong>
     <br/>
     是能够通过给定的关键字的值直接访问到具体对应的值的一个数据结构，保存在数组中，而数组是一块连续内存，可以直接根据地址值访问，因此查询速度更快，插入更快。当我们使用
     <code>
      Map
     </code>
     保存数据时，
     <code>
      Map
     </code>
     会先将我们传入的
     <code>
      key
     </code>
     经过哈希函数，生成一个哈希值（整数），而这个哈希值在内存中映射着我们传入的
     <code>
      value
     </code>
     值，因此他的访问速度很快。而
     <code>
      Set
     </code>
     是一种
     <code>
      key和value相同的散列表
     </code>
     ，因此不可以重复。
    </p>
    <p>
     <strong>
      根据能力继续扩展(自己查找)

    <p>
     <strong>
      开始回答
     </strong>
    </p>
    <ol>
     <li>
      <p>
       首先
       <code>
        Proxy
       </code>
       根据字面意思来看，它是一个对象的代理，代理就意味着我们想要为一个对象添加或者删除一些属性的时候，不能直接操作该对象，我们只需要操作代理对象就可以，通过代理对象做一些拦截就可以监测到我们对该对象的行为。
       <code>
        Proxy
       </code>
       通过构造函数的形式调用，接收两个参数，一个是需要代理的对象
       <code>
        target
       </code>
       ，另一个是一些代理的行为
       <code>
        handler
       </code>
       。
       <br/>
       <code>
        Reflect
       </code>
       是伴随
       <code>
        Proxy
       </code>
       使用的一个对象，它本身实现了同
       <code>
        Proxy
       </code>
       的 handler 相同的全部方法，它就像是我们在修改 handler 的默认行为时的一个正确指引，无论我们怎样对handler做修改，通过
       <code>
        Reflect
       </code>
       总能保持正确的输出。其上的
       <code>
        API
       </code>
       请自己查阅。
      </p>
     </li>
     <li>
      <p>
       在
       <code>
        Proxy
       </code>
       出现之前，我们有一个与之相同功能的实现方式，通过
       <code>
        Object.defineProperty
       </code>
       来为对象的某个键值实现代理，实现数据劫持，但是它有一个严重的弊端，是为对象的键值实现数据劫持，因此没有添加的键值不会实现，这也是
       <code>
        vue2
       </code>
       中响应式原理实现的一个缺陷，新添加的属性不会实现响应式，必须通过
       <code>
        $set
       </code>
       方法来单独实现响应式，而
       <code>
        vue3
       </code>
       通过
       <code>
        Proxy和Reflect
       </code>
       实现对对象的代理，也就是响应式原理，完美的解决了
       <code>
        vue2
       </code>
       的问题。这也就是
       <code>
        Proxy
       </code>
       的优势。
      </p>
     </li>
    </ol>

    <h3>
     <a id="__395">
     </a>
     😊一. 请你说下浏览器的发展（单进程浏览器和多进程浏览器）。
    </h3>
    <p>
     <code>
      答题思路:
     </code>
    </p>
    <ul>
     <li>
      早期浏览器的缺点
     </li>
     <li>
      现代浏览器的优势
     </li>
    </ul>
    <p>
     <strong>
      扩展问题
     </strong>
    </p>
    <ul>
     <li>
      什么是线程、进程
     </li>
     <li>
      进程间的通信方式
     </li>
    </ul>
    <p>
     <strong>
      开始回答：
     </strong>
    </p>
    <ol>
     <li>
      <strong>
       单进程浏览器
      </strong>
      ：早期，我们的浏览器是采用单进程架构的，意思就是，所有的任务都在一个进程中执行，网络请求、js运行环境、渲染引擎、页面、垃圾回收、用户交互、浏览器插件等。多个线程在一个独立的进程中工作，共享所有资源，因此浏览器插件可以访问操作系统上的任意资源，所以单进程浏览器是
      <strong>
       不安全的
      </strong>
      ；由于js和页面渲染等代码都是执行在一个线程上的，如果我们打开多个页面，多个页面的js和渲染机制抢夺一个线程，因此单进程浏览器是
      <strong>
       不流畅的
      </strong>
      ；再加上如果一个线程出错，会导致整个进程奔溃，进而导致整个浏览器退出无法工作，造成我们打开多个页面，有一个页面出错导致所有页面和浏览器一起关闭，所以单进程的浏览器是
      <strong>
       不稳定
      </strong>
      。\
     </li>
     <li>
      <strong>
       多进程浏览器
      </strong>
      ：
      <br/>
      现代浏览器采用多进程架构，完美的解决了单进程
      <code>
       不流畅、不稳定、不安全
      </code>
      的问题。现代浏览器包括
      <code>
       一个浏览器主进程
      </code>
      ，主要负责页面的显示、用户交互、子进程管理、存储等操作；
      <code>
       一个GPU进程
      </code>
      负责绘制UI页面和css3D效果，
      <code>
       一个网络进程
      </code>
      主要负责网络资源的加载和请求，多
      <code>
       个渲染进程
      </code>
      主要作用是将js，css，html转变为一个可视的页面，为什么是多个呢，因为谷歌浏览器默认为每一个页面开启一个渲染进程，出去安全考虑，渲染进程运行在沙箱模式下，因此js不可以直接访问操作系统资源；
      <code>
       多个插件进程
      </code>
      ，用来运行浏览器的插件，不同的插件执行在不同的进程，防止某个插件出错干扰其他插件和页面。
     </li>
    </ol>
    <p>
     <strong>
      扩展回答
     </strong>
    </p>
    <ol>
     <li>
      <strong>
       进程
      </strong>
      ：进程就是一个程序的运行实例，是资源分配的最小单位，可以理解为是为该程序运行专门创建的一块内存，用来存放程序运行的代码，运行的数据，还有一个执行任务的主线程。进程之间的资源是相互独立的，当一个进程关闭后，操作系统会回收进程所咋用的全部内存。
      <br/>
      <strong>
       线程
      </strong>
      ：线程是依附于进程的，不能独立存在，是cpu调度的最小单位，可以理解为线程是在进程上工作的，不同线程可以并行工作，不同线程之间共享所在进程的资源，一个线程出错会导致整个进程的奔溃。
     </li>
    </ol>
    <h3>
     <a id="__414">
     </a>
     😊二. 请你说下浏览器的渲染过程
    </h3>
    <p>
     <code>
      回答思路
     </code>
    </p>
    <ul>
     <li>
      没有什么思路，该问题也可以同
      <code>

      </code>

    </ul>
    <p>
     <strong>
      开始回答
     </strong>
    </p>
    <p>
     <code>
      资源请求阶段
     </code>
    </p>
    <ul>
     <li>
      首先，输入内容到地址栏之后，会判断输入的是关键字还是完整的
      <code>
       url
      </code>
      ，如果是关键字，浏览器会使用浏览器的默认搜索引擎，合成带搜索关键字的
      <code>
       URL
      </code>
      ,然后
      <code>
       浏览器进程
      </code>
      将该
      <code>
       URL
      </code>
      发送到
      <code>
       网络进程
      </code>
      。
     </li>
     <li>
      <code>
       网络进程
      </code>
      会查找本地是否缓存该网站资源，如果缓存中没有找到，那么就进入浏览器的请求流程，发送该
      <code>
       URL
      </code>
      到
      <code>
       DNS服务器
      </code>
      获取域名的服务器IP地址，之后开始与服务器建立
      <code>
       TCP
      </code>
      连接，而这里就要经历
      <code>
       TCP
      </code>
      的三次握手（
      <code>
       详细内容放在TCP模块讲
      </code>
      ）成功建立连接，如果是
      <code>
       https
      </code>
      站点还要进行安全验证（
      <code>
       放在http模块讲
      </code>
      ），之后浏览器构成请求的请求行请求头等信息，并携带本地存储的跟该域名相关的
      <code>
       Cookie
      </code>
      添加到请求头中，一起发送到服务器，服务器接收信息后，将响应数据，包括响应行、响应头、响应体发送给网络进程，等网络进程接收到响应头信息后，发起
      <code>
       提交文档
      </code>
      信息，通知
      <code>
       渲染进程
      </code>
      和
      <code>
       网络进程
      </code>
      建立
      <code>
       管道
      </code>
      准备通信。同时开始解析响应头，判断是否需要
      <code>
       重定向
      </code>
      如果状态码为
      <code>
       301、302
      </code>
      ，通过
      <code>
       Content-Type
      </code>
      判断此次响应体的文件格式，来决定如何处理响应体资源，是下载资源，还是显示图片，还是显示页面。等渲染进程拿到响应体之后，渲染进程返回
      <code>
       确认提交
      </code>
      的消息，浏览器进程接收到之后，会更新浏览器界面状态。之后准备渲染页面。
     </li>
    </ul>
    <p>
     <code>
      渲染阶段
     </code>
    </p>
    <ul>
     <li>
      <code>
       渲染进程接收到响应体后，
      </code>
      开始进入渲染流程，首先
      <code>
       HTML解析器
      </code>
      将html解析为浏览器可以识别的
      <code>
       DOM树
      </code>
      ，然后到了
      <code>
       样式计算
      </code>
      阶段
     </li>
     <li>
      <code>
       样式计算
      </code>
      分为三步
      <ol>
       <li>
        将css转换为浏览器可以识别的结构
        <code>
         styleSheets
        </code>
        。
       </li>
       <li>
        将样式表中的属性值标准化
       </li>
       <li>
        计算处DOM树中每个节点的样式，计算过程中遵循
        <code>
         CSS继承
        </code>
        和
        <code>
         层叠
        </code>
        的两个规则，
        <code>
         层叠是合并来自多个源的属性的算法
        </code>
        。
        <br/>
        之后进入布局阶段
       </li>
      </ol>
     </li>
     <li>
      <code>
       布局阶段
      </code>
      ，我们通过
      <code>
       DOM树
      </code>
      和
      <code>
       styleSheets
      </code>
      来合成
      <code>
       布局树
      </code>
      ，来确定DOM元素的几何位置，首先先遍历DOM树的可见元素，将其添加到布局树上，像
      <code>
       head
      </code>
      这种标签，还有
      <code>
       display：none
      </code>
      的元素不会出现在布局树中，然后根据
      <code>
       styleSheets
      </code>
      计算出DOM的几何位置。一切都准备好之后，进入分层阶段。
     </li>
     <li>
      <code>
       分层阶段
      </code>
      ，分层其实就是创建不同的图层，虽然浏览器页面看起来是二维的页面，但是实际上是三维的，根据
      <code>
       z-indexing
      </code>
      属性做z轴排序，生成一颗对应的图层树，但是并不是每个DOM都会生成一个单独的图层，只有具有层叠样式属性的DOM才会创建一个图层，而没有的属性附属于它的父节点的图层，所有图层重叠起来，也就构成了我们的浏览器页面。
     </li>
     <li>
      <code>
       图层绘制
      </code>
      阶段，当所有准备工作做好之后，我们就可以开始绘制，渲染进程将我们的绘制过程拆分成一个个小的
      <code>
       绘制指令
      </code>
      ,然后按照顺序组成一个
      <code>
       待绘制列表
      </code>
      ，然后交给合成线程实现真正的
      <code>
       绘制
      </code>
      。
     </li>
     <li>
      <code>
       合成线程
      </code>
      ,合成线程拿到绘制指令后，不会直接开始绘制，由于我们的页面可能很大，但是
      <code>
       视口
      </code>
      是有限的，因此
      <code>
       合成线程
      </code>
      会将图层划分为
      <code>
       图块
      </code>
      ,然后按照视口附近的
      <code>
       图块
      </code>
      优先生成
      <code>
       位图
      </code>
      ，而这个生成过程是在
      <code>
       栅格化
      </code>
      线程池中进行，将我们的图块
      <code>
       栅格化处理
      </code>
      ,就是将
      <code>
       图块转换为位图
      </code>
     </li>
     <li>
      <code>
       栅格化
      </code>
      ，栅格化过程都会使用GPU来加速生成，将生成的位图保存在GPU内存中
     </li>
     <li>
      <code>
       合成显示
      </code>
      ,一旦所有图块都被栅格化后，合成线程会生成一个绘制指令提交给浏览器进程，浏览器接收到指令后根据该绘制指令将内容绘制到
      <code>
       显存
      </code>
      中，最后将显存显示到屏幕上。
     </li>
    </ul>
    <h3>
     <a id="__441">
     </a>
     😊三. 请你说下浏览器的垃圾回收机制
    </h3>
    <p>
     <code>
      回答思路
     </code>
    </p>
    <ul>
     <li>
      栈内存的回收
     </li>
     <li>
      堆内存的回收
     </li>
    </ul>
    <p>
     <strong>
      开始回答
     </strong>
    </p>
    <ol>
     <li>
      我们都知道，js 在运行中会维护一个执行栈，每当执行一个函数，就会往执行栈中压入一个执行上下文，用来保存一些当前执行函数的数据，而函数中的原始变量会保存在栈中，而引用类型数据是保存在堆中的，变量中保存的是堆中的地址。栈顶就是维护着我们当前正在执行的函数，当函数执行完之后，有一个出栈操作，出栈也就意味着销毁当前函数的执行上下文，也就回收了该函数保存在栈中的所有内存。
     </li>
     <li>
      v8 会把堆内存分为两个部分，一部分用来存放生命周期短的对象，也就是新生代，另一部分用来存放生命周期长的对象，也就是老生代。老生代的内存比新生代大很多，老生代有1400MB，新生代有32MB。
      <ul>
       <li>
        <code>
         新生代
        </code>
        ： 我们先来说说新生代的垃圾回收。v8将新生代的内存分为两个部分，
        <code>
         from
        </code>
        区域和
        <code>
         to
        </code>
        区域，我们的数据统一保存在
        <code>
         from
        </code>
        区域，而
        <code>
         to
        </code>
        区域是空闲区域，因为新生代中大部分都是生命周期短的对象，所以在执行垃圾回收时我们将还存活的对像存放到
        <code>
         to
        </code>
        区域更高效，然后将
        <code>
         from
        </code>
        中的内存全部回收，之后将
        <code>
         from
        </code>
        和
        <code>
         to
        </code>
        区域调换，就可以继续完成垃圾回收，同时，为了防止新生代内存溢出，v8提供了一套晋升机制，将经历两次存在的内存存放到老生代区域。
       </li>
       <li>
        <code>
         老生代
        </code>
        ：采用标记清除法，因为老生代大多是生命周期长的对象，所以我们去处理生命周期短的更高效，通过遍历调用栈，查看内存中的哪些对象没有相应的变量引用了，那就将其标记，之后将已经标记的内存释放，也就完成了垃圾回收，但是这样一来导致内存中出现很多不连续的片段，与是又采用标记整理，将所有存活的对象都移动到一端，然后清理掉端边界以外的内存，进而实现垃圾回收。但是还有一个问题，js是运行在渲染进程的主线程上的，一旦执行垃圾回收算法，都需要将正在执行的 JavaScript 脚本暂停下来，待垃圾回收完毕后再恢复脚本执行。我们把这种行为叫做全停顿，新生代内存较小影响不大，但是老生代的内存空间较大，可能在回收过程中造成页面卡顿，为了降低老生代的垃圾回收而造成的卡顿，V8 将标记过程分为一个个的子标记过程，同时让垃圾回收标记和 JavaScript 执行交替进行，直到标记阶段完成。
       </li>
      </ul>
     </li>
    </ol>
    <h3>
     <a id="js_453">
     </a>
     😊四.浏览器是如何执行一段js代码的?
    </h3>
    <p>
     <code>
      回答思路
     </code>
    </p>
    <ul>
     <li>
      现在执行：将代码转为字节码
     </li>
     <li>
      过去执行：将代码转为机器码，又什么不足
     </li>
     <li>
      执行代码中的优化
      <br/>
      <strong>
       开始回答
      </strong>
     </li>
    </ul>
    <ol>
     <li>
      js 本身是
      <code>
       解释型语言
      </code>
      ，再运行之前，需要先经过解释器处理，因为计算机是不能直接读懂开发人员所写的代码的，所以再执行之前都要先
      <code>
       翻译
      </code>
      成计算机能懂的语言，无论
      <code>
       编译型语言
      </code>
      还是
      <code>
       解释型语言
      </code>
      都是如此。首先，
      <code>
       解释器
      </code>
      需要将我们写的源代码生成
      <code>
       AST语法树
      </code>
      和
      <code>
       执行上下文
      </code>
      , 先对源代码进行
      <code>
       分词
      </code>
      ,进行
      <code>
       词法分析
      </code>
      就是将代码拆分成
      <code>
       token
      </code>
      ,
      <code>
       token
      </code>
      就是指语法上不可再分的最小单位字符或者字符串，之后进行
      <code>
       解析
      </code>
      做
      <code>
       语法分析
      </code>
      生成
      <code>
       AST
      </code>
      ,有了
      <code>
       AST
      </code>
      和
      <code>
       执行上下文
      </code>
      之后，解释器根据AST生成
      <code>
       字节码
      </code>
      ，然后执行。
     </li>
     <li>
      在过去，V8引擎中并没有字节码，而是通过
      <code>
       编译器
      </code>
      直接将
      <code>
       AST
      </code>
      转为
      <code>
       机器码
      </code>
      ,因为机器码的执行效率是非常高的，所以js运行也很快，但是有一个缺点，机器码占用的内存太大，为了解决占用内存的问题，改为了
      <code>
       字节码
      </code>
      ，字节码可以远远减少系统内存的使用。
     </li>
     <li>
      生成字节码之后，解释器开始逐条解释执行，在执行过程中，会对代码进行优化，就是
      <code>
       JIT即时编译
      </code>
      技术，在执行字节码同时，收集代码信息，一段代码执行次数多了之后会把该段代码标记为
      <code>
       热点代码
      </code>
      ,将该段代码的
      <code>
       字节码转变为机器码
      </code>
      并且保存,当下次再执行时，直接执行机器码，大大的提升了执行效率。
     </li>
    </ol>

      </code>

      <code>

      </code>

      <code>

      </code>

      </code>

      </code>

    <h6>
     <a id="2_XSS__528">
     </a>
     2. XSS 跨站脚本攻击
    </h6>
    <p>
     我们先来说说什么是
     <code>
      XSS攻击
     </code>
     ，其实根据其名称跨站脚本攻击，我们就可以知道，是通过执行脚本来做一些恶意操作，比如获取用户的
     <code>
      cookie
     </code>
     等信息，
     <code>
      监听用户行为
     </code>
     ,或者通过
     <code>
      修改DOM假冒登录窗口获取用户密码
     </code>
     等。那说了半天浏览器安全策略，那这还是不安全呀，其实这并不是浏览器不安全，而实我们写代码时留下的漏洞。我们来看看恶意脚本是怎么注入的。
    </p>
    <ol>
     <li>
      <code>
       储存型攻击
      </code>
      ：这种攻击是将恶意执行的脚本保存到用户的数据库中的，比方说有一个网站有一个留言功能，当用户留言后不做任何处理把留言信息保存到数据库，然后将留言展示到网页上，这就留下了漏洞，我们可以通过留言一个
      <code>
       script
      </code>
      脚本，做任何操作，一旦打开留言面板访问到我们留言的人，都会执行这段恶意脚本，这就是
      <code>
       储存型攻击
      </code>
     </li>
     <li>
      <code>
       反射型攻击
      </code>
      ：反射型攻击是利用
      <code>
       url
      </code>
      来加入恶意脚本，比方说有一个页面有一个搜索功能，他的请求接口为这样
      <code>
       http://xxx.com/queryName='石头山'
      </code>
      ,服务器响应搜索不到后会在页面中显示找不到
      <code>
       石头山
      </code>
      相关结果，因此这也留下了漏洞，当我把请求接口改成这样
      <code>
       http://xxx.com/queryName=&lt;script src="我的网站"&gt;&lt;script&gt;
      </code>
      ，然后在用户的搜索界面诱导他点击发起这个请求，那么就会执行这段恶意代码进行任意攻击。
     </li>
     <li>
      <code>
       基于 DOM 的 XSS 攻击
      </code>
      : 其他两种攻击都是借助服务器来实现恶意脚本注入，而
      <code>
       DOM型攻击
      </code>
      是属于前端漏洞，可以通过网络劫持，在页面传输过程中修改 HTML 页面的内容，进而注入恶意代码。
     </li>
    </ol>
    <p>
     <strong>
      如何阻止XSS攻击？
     </strong>
    </p>
    <ol>
     <li>
      <code>
       过滤或者转码
      </code>
      ：因为反射型攻击和存储型攻击都会经过服务器，因此我们可以在服务器端对用户输入的内容进行过滤，将有
      <code>
       script
      </code>
      脚本的字段删除，或者将
      <code>
       &lt;、&gt;、/
      </code>
      等特殊符号进行转码防止脚本在客户端执行。
     </li>
     <li>
      <code>
       CSP
      </code>
      ：是浏览器提供的有效防止
      <code>
       XSS攻击
      </code>
      和
      <code>
       数据注入
      </code>
      的内容安全策略，我们可以通过服务器响应头中添加
      <code>
       Content-Security-Policy
      </code>
      字段，或者
      <code>
       meta
      </code>
      标签来配置，该站点资源允许从哪些域名加载并执行。可以自行去
      <code>
       MDN
      </code>
      官网详细了解。
     </li>
     <li>
      <code>
       httpOnly
      </code>
      :可以在服务器限制
      <code>
       cookie
      </code>
      的访问防止，通过
      <code>
       httpOnly
      </code>
      来限制
      <code>
       cookie
      </code>
      不可以在浏览器端通过js访问，以此来防止恶意代码偷窃
      <code>
       cookie
      </code>
      信息
     </li>
    </ol>
    <h6>
     <a id="3_CSRF_539">
     </a>
     3. CSRF攻击
    </h6>
    <p>
     <code>
      CSRF
     </code>
     英文全称是 Cross-site request forgery，所以又称为
     <code>
      跨站请求伪造
     </code>
     ,
     <code>
      CSRF
     </code>
     的本质就是诱导用户去发起某些请求，利用用户的登录状态，
     <code>
      帮助
     </code>
     用户发起某些请求，比如转账等操作。
    </p>
    <p>
     <strong>
      如何阻止CSRF攻击？
     </strong>
    </p>
    <ol>
     <li>
      <code>
       cookie
      </code>
      ：因为
      <code>
       http
      </code>
      请求是没有状态的，所以
      <code>
       cookie
      </code>
      是维护服务器和浏览器交互的一个重要数据，因此我们可以通过
      <code>
       cookie
      </code>
      本身的属性
      <code>
       SameSite
      </code>
      ，来有效防止一些
      <code>
       冒牌的请求
      </code>
      .
      <ul>
       <li>
        Lax: 携带
        <code>
         Lax
        </code>
        的
        <code>
         Cookie
        </code>
        ，没有
        <code>
         Strict
        </code>
        那么严格，允许
        <code>
         get请求、a 标签
        </code>
        在跨域情况下携带cookie
       </li>
       <li>
        Strict: 携带
        <code>
         Strict
        </code>
        的
        <code>
         Cookie
        </code>
        是十分严格的，只有在本站内允许携带cookie，任何跨域形式的请求都不允许携带
        <code>
         cookie
        </code>
        。
       </li>
      </ul>
     </li>
    </ol>
    <p>
     虽然
     <code>
      Strict
     </code>
     是绝对安全的，但是有些时候使用它是十分影响用户体验的，比方式某个网站通过
     <code>
      cookie
     </code>
     保存用户登录状态，用户登录之后打开其他网站，而其他网站有跳转该网站的链接，因为
     <code>
      Strict
     </code>
     禁止携带cookie，因此用户会失去登录状态而导致重新登录，因此我们大多时候使用
     <code>
      Lax
     </code>
     。
    </p>
    <ol start="2">
     <li>
      验证请求站点的来源：我们在服务器端验证请求的来源，由于 CSRF 攻击大多来自于第三方站点，因此服务器可以禁止来自第三方站点的请求。我们可以通过
      <code>
       请求头
      </code>
      中的
      <code>
       orign
      </code>
      字段查看请求的来源，以此来决定是否发起响应。
     </li>
     <li>
      <code>
       CSRF Token
      </code>
      ：这里提到的
      <code>
       Token
      </code>
      跟我们用来保存用户状态的
      <code>
       Token
      </code>
      不同，这个就是一个用来识别是安全请求的一个标志，后端可以同前端协商携带一个安全字段，每次发起
      <code>
       post
      </code>
      请求都将该字段携带，后端通过验证是否存在该字段来决定是否完成响应。由于伪造请求的人不知道我们的该字段，因此他的请求中不会携带该字段，也就不会请求成功。
     </li>
    </ol>
    <h6>
     <a id="4__551">
     </a>
     4. 安全沙箱
    </h6>
    <p>
     通过前边浏览器的进程中学习到，浏览器可以通过网络进程下载和接收任意资源，而这些资源的执行都是在
     <code>
      渲染进程
     </code>
     中执行，如果这些资源中有恶意程序对浏览器发起攻击，就会给用户带来极大的安全隐患，因此我们直接从根上扼制这种行为，我们将
     <code>
      渲染进程
     </code>
     独立起来，让他与操作系统隔离，因此恶意程序就访问不到除渲染进程之外的任意权限，而将渲染进程和操作系统隔离的这道墙就是
     <code>
      安全沙箱
     </code>
     。浏览器中的安全沙箱是利用操作系统提供的安全技术，让渲染进程在执行过程中无法访问或者修改操作系统中的数据，在渲染进程需要访问系统资源的时候，需要通过浏览器内核来实现，然后将访问的结果通过 IPC 转发给渲染进程.安全沙箱最小的保护单位是进程。因为单进程浏览器需要频繁访问或者修改操作系统的数据，所以单进程浏览器是无法被安全沙箱保护的，而现代浏览器采用的多进程架构使得安全沙箱可以发挥作用。
    </p>
    <h6>
     <a id="5_https_553">
     </a>
     5. https
    </h6>
    <p>
     http 协议中所有数据传输都是采用明文传输，因此传输内容很容易被黑客劫持，因此是十分不安全的。所以提出了一种更安全的协议
     <code>
      https
     </code>
     ，
     <code>
      https
     </code>
     是建立在
     <code>
      http
     </code>
     的基础上的，因此它同
     <code>
      http
     </code>
     完全一致，只不过是在
     <code>
      http应用层
     </code>
     和
     <code>
      Tcp传输层
     </code>
     直接又加了新的一层
     <code>
      安全层TLS/SSL
     </code>
     ，在建立tcp连接之后，做好加密工作，然后再发起http请求。我们来讲讲
     <code>
      https
     </code>
     的发展历程。
    </p>
    <ol>
     <li>
      <strong>
       https 加密第一版 对称加密：
      </strong>
      <ul>
       <li>
        在客户端与服务器建立TCP连接之后，开始验证加密方式，客户端发送浏览器所支持的加密方式，也就是加密套件列表，除此之外，浏览器还要携带一个随机数client-Random
       </li>
       <li>
        服务器接收到之后，选择一种加密方式，然后还会生成一个随机数 service-random，并将 service-random 和加密套件列表返回给浏览器。
       </li>
       <li>
        最后浏览器和服务器分别返回确认消息。这样浏览器端和服务器端都有相同的 client-random 和 service-random 了，然后它们再使用相同的方法将 client-random 和 service-random 混合起来生成一个密钥 master secret，有了密钥 master secret 和加密套件之后，双方就可以进行数据的加密传输了。
       </li>
      </ul>
     </li>
    </ol>
    <p>
     <strong>
      缺点
     </strong>
     ：由于clien-random 和 server-random 和 加密套件列表是明文的，所以这种方式仍然不安全，黑客仍然可以窃取用户和浏览器传递的
     <code>
      http
     </code>
     数据。
    </p>
    <ol start="2">
     <li>
      <strong>
       https 第二版 非对称加密
      </strong>
      <ul>
       <li>
        首先浏览器还是发送加密套件列表给服务器。
       </li>
       <li>
        然后服务器会选择一个加密套件, 同时，浏览器维护了一个私钥，公钥，将公钥和选择的加密套件一起发送给客户端
       </li>
       <li>
        最后就是浏览器和服务器返回确认消息
       </li>
       <li>
        这样浏览器端就有了服务器的公钥，在浏览器端向服务器端发送数据时，就可以使用该公钥来加密数据。由于公钥加密的数据只有私钥才能解密，所以即便黑客截获了数据和公钥，他也是无法使用公钥来解密数据的。
       </li>
       <li>
       </li>
      </ul>
     </li>
    </ol>
    <p>
     <strong>
      缺点
     </strong>
     ：无法保证服务器发送给浏览器的数据安全，黑客任然可以窃取服务器响应给用户的数据。虽然浏览器端可以使用公钥来加密，而黑客没有私钥，因此数据是安全的，但是服务器端只能采用私钥来加密，私钥加密只有公钥能解密，但黑客也是可以获取得到公钥的，这样就不能保证服务器端数据的安全了。
    </p>
    <ol start="3">
     <li>
      <strong>
       https 第三版 非对称加密和对称加密搭配使用
      </strong>
     </li>
    </ol>
    <p>
     <strong>
      思想
     </strong>
     ：在传输数据阶段依然使用对称加密，但是对称加密的密钥我们采用非对称加密来传输
    </p>
    <ul>
     <li>
      首先浏览器向服务器发送对称加密套件列表、非对称加密套件列表和随机数 client-random；
     </li>
     <li>
      服务器保存随机数 client-random，选择对称加密和非对称加密的套件，然后生成随机数 service-random，向浏览器发送选择的加密套件、service-random 和浏览器的公钥；
     </li>
     <li>
      浏览器保存公钥，并生成随机数 pre-master，然后利用公钥对 pre-master 加密，并向服务器发送加密后的数据；
     </li>
     <li>
      最后服务器拿出自己的私钥，解密出 pre-master 数据，并返回确认消息。
     </li>
     <li>
      到此为止，服务器和浏览器就有了共同的 client-random、service-random 和 pre-master，然后服务器和浏览器会使用这三组随机数生成对称密钥，因为服务器和浏览器使用同一套方法来生成密钥，所以最终生成的密钥也是相同的，之后的数据传输可以利用密钥加密，也可以利用密钥解密，保证了双端数据传输的绝对安全性。
     </li>
    </ul>
    <p>
     <strong>
      缺点：
     </strong>
     虽然使用对称加密和非对称加密的方式，解决了数据在客户端和目标站点传输的安全问题，防止数据在中途被黑客窃取，但是，黑客仍然可以模拟服务器，也就是直接模拟目标网站，而浏览器却无法察觉。这 样完全取得了客户端的操控权，可以轻松获取用户的信息。
    </p>
    <ol start="4">
     <li>
      <strong>
       https 第四版 添加数字证书
      </strong>
     </li>
    </ol>
    <p>
     <strong>
      解决问题
     </strong>
     ：告诉浏览器，该网站就是我要访问的真实网站。
    </p>
    <ul>
     <li>
      浏览器发起请求之后，服务器将提前
      <code>
       申请好的证书发送给浏览器端
      </code>
     </li>
     <li>
      然后浏览器验证该证书的真实性，首先通过CA 机构在签名时同样的 hash 函数来取得信息A，然后再利用证书中携带的
      <code>
       CA机构（证书办法机构，是国内获得工信部认可的组织）
      </code>
      的 公钥解密签名数据B，通过对比信息，可以确认该证书的可靠性和网站的可靠性，同时验证证书时间等等，但是这个 CA机构 可能比较小众，浏览器不知道该不该信任它，然后浏览器会继续查找给这个 CA 颁发证书的 CA，再以同样的方式验证它上级 CA 的可靠性。通常情况下，操作系统中会内置信任的顶级 CA 的证书信息（包含公钥），如果这个 CA 链中没有找到浏览器内置的顶级的 CA，证书也会被判定非法。
     </li>
     <li>
      证书验证完成后，继续第三版的操作。
     </li>
    </ul>
    <p>
     通过证书验证，非对称加密和对称加密的方式，不仅保证了数据在传输过程的安全性，也保证了服务器的可靠性。
    </p>
    <h3>
     <a id="_js__css__590">
     </a>
     😊六. 请你说下浏览器中js 和 css 是如何影响页面加载的。
    </h3>
    <p>
     <code>
      答题思路
     </code>
    </p>
    <ul>
     <li>
      首先说下浏览器的渲染流程
     </li>
     <li>
      js 如何影响 DOM 的构建的，HTML解析器是如何生成DOM的
     </li>
     <li>
      css 又是如何影响的
     </li>
     <li>
      优化解决方案
      <br/>
      <strong>
       开始回答
      </strong>
     </li>
    </ul>
    <ol>
     <li>
      浏览器的渲染流程我们已经在上文中讲过，不熟悉的可以再返回去看一下，在这不重复了，我们接着往下讲
     </li>
     <li>
      通过渲染流程我们知道，首先需要通过
      <code>
       HTML解析器
      </code>
      将html文件转换成
      <code>
       DOM
      </code>
      树，而这个过程是在
      <code>
       网络进程
      </code>
      接收到响应体，通过管道发送到
      <code>
       渲染进程
      </code>
      之后的，接收到多少文件，
      <code>
       HTML解析器
      </code>
      就解析多少文件，而不需要等待所有html加载完成之后再开始。知道这一点后我们来聊聊DOM的生成流程。
     </li>
    </ol>
    <ul>
     <li>
      <h6>
       <a id="DOM__600">
       </a>
       DOM 树的构建流程
      </h6>
     </li>
    </ul>
    <p>
     首先通过
     <code>
      分词器
     </code>
     将html文件字节流转换为
     <code>
      Token
     </code>
     ,我们用以下代码举例，生成的
     <code>
      Token
     </code>
     就是这样的：
    </p>
    <blockquote>
     <p>
      <code>
       StartTaghtml
      </code>
      -&gt;
      <code>
       StartTagbody
      </code>
      -&gt;
      <code>
       StartTagh1
      </code>
      -&gt;
      <code>
       石头山
      </code>
      -&gt;
      <code>
       EndTagh1
      </code>
      -&gt;
      <code>
       StartTagdiv
      </code>
      -&gt;
      <code>
       点个赞再走
      </code>
      -&gt;
      <code>
       EndTagdiv
      </code>
      -&gt;
      <code>
       EndTagdiv
      </code>
      -&gt;
      <code>
       EngTaghtml
      </code>
     </p>
    </blockquote>
    <pre><code class="prism language-js">    <span class="token operator">&lt;</span>html<span class="token operator">&gt;</span>
        <span class="token operator">&lt;</span>body<span class="token operator">&gt;</span>
            <span class="token operator">&lt;</span>h1<span class="token operator">&gt;</span>石头山<span class="token operator">&lt;</span><span class="token operator">/</span>h1<span class="token operator">&gt;</span>
            <span class="token operator">&lt;</span>div<span class="token operator">&gt;</span>点个赞再走<span class="token operator">&lt;</span><span class="token operator">/</span>div<span class="token operator">&gt;</span>
        <span class="token operator">&lt;</span><span class="token operator">/</span>body<span class="token operator">&gt;</span>
    <span class="token operator">&lt;</span><span class="token operator">/</span>html<span class="token operator">&gt;</span>
</code></pre>
    <p>
     然后需要将
     <code>
      Token
     </code>
     转为DOM节点并且添加到DOM中。HTML解析器维护了一个
     <code>
      Token栈结构
     </code>
     ，将生成的
     <code>
      Token
     </code>
     按顺序压入到这个栈中，如果压入到栈中的是
     <code>
      StartTag Token
     </code>
     ，HTML 解析器会为该
     <code>
      Token
     </code>
     创建一个
     <code>
      DOM 节点
     </code>
     ，然后将该节点加入到 DOM 树中，它的父节点就是位于栈顶的这个
     <code>
      Token
     </code>
     生成的DOM节点。如果是一个
     <code>
      文本Token
     </code>
     ，那么会生成一个文本节点，然后将该节点加入到 DOM 树中，文本 Token 是不需要压入到栈中，它的父节点就是当前栈顶 Token 所对应的 DOM 节点。如果分词器产生的是
     <code>
      EndTag Token
     </code>
     ，则查看栈顶是否文对应的
     <code>
      Start Token
     </code>
     ，如果是就将
     <code>
      StartTag Token
     </code>
     从栈顶弹出，表示该dom解析完成。通过该流程，可以将html标签解析成一颗完整的DOM树。注意，html解析器会提前在空的
     <code>
      Token栈
     </code>
     中压入
     <code>
      StartTagdocument
     </code>
     同时创建一个根为
     <code>
      document
     </code>
     的DOM树。了解了DOM的生成过程之后，我们回归主题，说下js如何影响ODM构建的，如果在 html 中内嵌了一个
     <code>
      script
     </code>
     标签，
     <code>
      html解析器
     </code>
     解析到这是一个
     <code>
      StartTag script
     </code>
     时候，
     <code>
      渲染进程
     </code>
     知道这是一段执行脚本，并且该脚本可能会修改已经生成的DOM树，因此会停止
     <code>
      HTML解析器
     </code>
     的解析工作，而先执行该脚本，执行完之后，在继续执行解析操作。因此，
     <code>
      js脚本会阻塞DOM树的生成过程
     </code>
     ，如果js是远程资源，还要等待资源加载完成，十分影响用户体验。
    </p>
    <ol start="3">
     <li>
      <p>
       我们了解到，js的执行会阻塞html的解析，也就是DOM树的生成，因为js有操作DOM树的能力，但是，js同时有操作
       <code>
        styleSheets
       </code>
       的能力，所以浏览器不知道该段脚本是否会操作
       <code>
        node.style
       </code>
       属性，所以在js执行前，会等待当前页面的
       <code>
        css
       </code>
       文件加载，并且等待
       <code>
        css
       </code>
       解析生成
       <code>
        CSSOM
       </code>
       ，然后去执行js，再继续
       <code>
        dom树
       </code>
       的构建，严重影响了首屏加载速度。
      </p>
     </li>
     <li>
      <p>
       优化方案：
      </p>
      <ul>
       <li>
        通过内联 JavaScript、内联 CSS 来移除这两种类型的文件下载，这样获取到 HTML 文件之后就可以直接开始渲染流程了。
       </li>
       <li>
        但并不是所有的场合都适合内联，那么还可以尽量减少文件大小，比如通过 webpack 等工具移除一些不必要的注释，并压缩 JavaScript 文件。
       </li>
       <li>
        还可以将一些不需要在解析 HTML 阶段使用的 JavaScript 标记上 async 或者 defer。
       </li>
       <li>
        <code>
         async
        </code>
        : 会在js脚本加载完成后立即执行
       </li>
       <li>
        <code>
         defer
        </code>
        : 会在html解析完成后，
        <code>
         DOMContentLoaded
        </code>
        事件之前执行
       </li>
      </ul>
     </li>
    </ol>
    <h3>
     <a id="__623">
     </a>
     😊七. 请你说说浏览器缓存
    </h3>
    <p>
     <code>
      答题思路
     </code>
    </p>
    <ul>
     <li>
      什么是缓存，有什么效果
     </li>
     <li>
      200和304区别，强缓存、弱缓存
     </li>
     <li>
      如何设置强缓存、弱缓存
     </li>
    </ul>
    <p>
     <strong>
      开始回答
     </strong>
    </p>
    <ol>
     <li>
      <p>
       浏览器缓存，是为了优化页面的二次访问速度，减少服务器请求次数的机制，通过在http响应头或者请求头中携带相关信息，告诉浏览器应该缓存哪些内容，从而下次请求时直接从本地获取，减少网络请求带来的延迟。
      </p>
     </li>
     <li>
      <p>
       缓存分为两种，
       <code>
        强缓存
       </code>
       和
       <code>
        弱缓存
       </code>
      </p>
      <ul>
       <li>
        <code>
         强缓存
        </code>
        ：将资源存在到本地，下次请求时直接从本地拿去，不需要发起请求，响应码为 200
       </li>
       <li>
        <code>
         弱缓存
        </code>
        : 将资源缓存到本地，但是请求时仍然会发起请求，但是响应只有响应头，没有响应体，响应码为304，告诉浏览器缓存没有过期，可以从本地读取资源。
       </li>
      </ul>
     </li>
     <li>
      <p>
       设置强缓存
      </p>
     </li>
    </ol>
    <ul>
     <li>
      Cache-Control
      <ul>
       <li>
        max-age：设置缓存的到期时间
       </li>
       <li>
        public：表明响应可以被任何对象（包括：发送请求的客户端，代理服务器，等等）缓存，即使是通常不可缓存的内容
       </li>
       <li>
        private: 表明响应只能被用户浏览器缓存，不能被代理服务器缓存
       </li>
       <li>
        no-cache：强制要求缓存把请求提交给原始服务器进行验证 (协商缓存验证)。
       </li>
       <li>
        no-store：不应该缓存任何响应数据
       </li>
      </ul>
     </li>
    </ul>
    <p>
     设置弱缓存
    </p>
    <ul>
     <li>
      ETag/If-None-Match:响应头中携带
      <code>
       Etag
      </code>
      ，下次请求，请求头中会自动加入
      <code>
       If-None-Match
      </code>
      字段用来校验是否过期
     </li>
     <li>
      Last-Modified/If-Modified-Since:响应头中携带
      <code>
       Last-Modified
      </code>
      ，下次请求，请求头中会自动加入
      <code>
       If-Modified-Since
      </code>
      字段用来校验是否过期
     </li>
    </ul>

     <a id="__648">

     八. 浏览器加载页面性能优化
    </h3>
    <p>
     <code>
      回答思路
     </code>
    </p>
    <ul>
     <li>
      先讲下浏览器的渲染机制
     </li>
     <li>
      根据渲染机制做出对应的性能优化措施
     </li>
    </ul>
    <p>
     <strong>
      开始回答
     </strong>
    </p>
    <ol>
     <li>
      省略讲解，详细可看前文
     </li>
    </ol>
    <blockquote>
     <p>
      <code>
       生成DOM树
      </code>
      -&gt;
      <code>
       样式计算
      </code>
      -&gt;
      <code>
       生成布局树
      </code>
      -&gt;
      <code>
       分层生成图层树
      </code>
      -&gt;
      <code>
       绘制、生成待绘制列表
      </code>
      -&gt;
      <code>
       栅格化、将块图转为位图
      </code>
      -&gt;
      <code>
       合成并显示
      </code>
     </p>
    </blockquote>
    <ol start="2">
     <li>
      性能优化
     </li>
    </ol>
    <ul>
     <li>
      加载阶段：
      <ul>
       <li>
        通过之前了解，js和css的加载和执行会阻塞DOM树的生成，所以可以通过压缩js和css资源来加快资源的加载速度。或者将其内联。
       </li>
       <li>
        让不对 dom和style 做操作的 script 异步加载，使用defer、async
       </li>
       <li>
        可以将
        <code>
         js
        </code>
        和
        <code>
         css
        </code>
        内联，减少关键资源的
        <code>
         RTT
        </code>
        个数
       </li>
      </ul>
     </li>
     <li>
      交互阶段：
      <ul>
       <li>
        减少 JavaScript 脚本执行时间，将执行时间较长的js开启多线程执行。
       </li>
       <li>
        避免强制同步布局。正常情况，我们通过
        <code>
         DOM接口
        </code>
        添加或者删除元素，都需要从新计算样式并且布局，为了避免当前的任务占用太长的主线程时间，这些操作都是在另外的任务中异步完成的。但是如果在更改节点后立刻获取修改后节点的信息，会导致立即在主线程中重新布局。
       </li>
       <li>
        合理利用 CSS 合成动画，css动画是直接在合成线程中进行的，从渲染流程来说，大大加快了动画速度，并且不会受到主线程js 或者一些布局任务影响。除此之外如果能提前知道对某个元素执行动画操作，那就最好将其标记为 will-change，这是告诉渲染引擎需要将该元素单独生成一个图层。
       </li>
       <li>
        避免频繁的垃圾回收，从垃圾回收那一节我们知道，垃圾回收任务添加到消息队列也会影响渲染进程中主线程的执行，会阻塞js的执行。
       </li>
      </ul>
     </li>
    </ul>

    <h3>
     <a id="__668">
     </a>
     一. 说下模块热加载原理
    </h3>
    <p>
     <code>
      回答思路
     </code>
    </p>
    <ul>
     <li>
      先简单说下什么是模块热加载
     </li>
     <li>
      说下原理
     </li>
    </ul>
    <p>
     <strong>
      开始回答
     </strong>
    </p>
    <ol>
     <li>
      模块热加载就是在修改了某个模块之后，不需要重新加载整个页面，就可以实现页面的部分刷新，大大提高开发效率。
     </li>
     <li>
      <ul>
       <li>
        首先在
        <code>
         webpack-dev-sever
        </code>
        启动后，会实现一个轮询机制，就是定时的去查看文件是否修改，我们可以通过配置
        <code>
         watch
        </code>
        和
        <code>
         watchOptions
        </code>
        去配置，当检测到有文件改变时会重新进行编译，并将编译后的结果保存到内存中，没有写入到文件夹中，所以不会生产
        <code>
         dist
        </code>
        文件，加快了读取速度，减少I/O带来的性能问题。
       </li>
       <li>
        服务器启动之后还会维护一个
        <code>
         ws
        </code>
        协议与服务器建立连接，并且主动向浏览器推送一个哈希值，也就是当前文件被编译之后生成的
        <code>
         hash
        </code>
        值。
       </li>
       <li>
        当轮询机制检测到有文件改变之后，会通过ws通知浏览器，有文件改变，并且携带新的编译完成后的
        <code>
         hash
        </code>
        值
       </li>
       <li>
        然后浏览器收到
        <code>
         hash
        </code>
        值之后，会将上一次的
        <code>
         hash
        </code>
        值合成请求标识
        <code>
         上次hash.hot-update.json
        </code>
        文件，发起一个
        <code>
         xhr
        </code>
        请求，服务器响应内容中包含了改变的
        <code>
         chunk
        </code>
        。
       </li>
       <li>
        然后继续合成新的请求标识
        <code>
         上次hash.hot-update.js
        </code>
        ,通过
        <code>
         jsonp
        </code>
        的方式来获取新的模块，通过
        <code>
         module.hot.accept
        </code>
        来让用户添加热模块替换规则。
       </li>
      </ul>
     </li>
    </ol>
    <h3>
     <a id="_treeshark__682">
     </a>
     二. tree-shark 原理
    </h3>
    <p>
     <code>
      回答思路
     </code>
    </p>
    <ul>
     <li>
      <code>
       tree-shark
      </code>
      是基于什么实现的？如何配置。
     </li>
     <li>
      <code>
       ES
      </code>
      和
      <code>
       Commonjs
      </code>
      模块导出有什么区别。
     </li>
     <li>
      <code>
       tree-shark
      </code>
      原理
     </li>
    </ul>
    <p>
     <strong>
      开始回答
     </strong>
    </p>
    <ol>
     <li>
      <p>
       首先
       <code>
        tree-shark
       </code>
       是基于
       <code>
        ES
       </code>
       的模块导入实现的，由于ES的模块导入是静态导入，可以在执行前静态分析导入模块是否被使用，从而不会被打包。而
       <code>
        Commonjs
       </code>
       是动态导入，无法判断是否被使用，所以要想
       <code>
        tree-shark
       </code>
       一定要使用
       <code>
        ES
       </code>
       规范。
       <br/>
       配置：
       <code>
        webpack5
       </code>
       已经在生产模式下自动实现了
       <code>
        tree-shark
       </code>
       ，我们来聊聊开发模式下如何配置。
      </p>
      <ul>
       <li>
        首先使用
        <code>
         optimization.usedExports
        </code>
        来开启标记未使用的导出值
       </li>
       <li>
        然后使用
        <code>
         Terset插件删除这些没有被使用的导出值
        </code>
        。
       </li>
      </ul>
     </li>
     <li>
      <p>
       区别：
      </p>
      <ul>
       <li>
        Commonjs支持动态导入，而ES则必须将
        <code>
         import
        </code>
        放在顶层导入，不可以使用
        <code>
         if
        </code>
        等语句包裹。虽然后续ES模块也支持动态导入
        <code>
         import（）
        </code>
       </li>
       <li>
        Commonjs 导出相当于是一个值得拷贝，因此值可以直接重新赋值，而ES是导出的是值得引用，不可以直接重新赋值，只能依赖提供的函数修改。
       </li>
       <li>
        ESModule
        <code>
         export default
        </code>
        可以与
        <code>
         export
        </code>
        混用，并且前者必须位于顶层，它们导出的模块不属于同一个对象， 而Commonjs 的
        <code>
         module.exports
        </code>
        和
        <code>
         exports
        </code>
        的导出属于导出一个对象，可以随便混入。
       </li>
      </ul>
     </li>
    </ol>

    </h3>

     <code>
      http
     </code>

     </li>

     </strong>

     <br/>
     1. 每次请求发送前都需要建立一次
     <code>
      TCP连接
     </code>
     ，极大的消耗性能
    </p>
    <h3>
     <a id="http_11_714">
     </a>
     http 1.1
    </h3>
    <p>
     为了完善
     <code>
      http 1.0
     </code>
     的缺点，做出了修改
    </p>
    <ul>
     <li>
      引入了
      <code>
       keep-live
      </code>
      长连接，使得一次
      <code>
       TCP
      </code>
      连接可以发送多个
      <code>
       http
      </code>
      请求
     </li>
     <li>
      在同一个域名下，可以建立6个TCP连接，实现并发传输，因此可以绑定多个域名来建立更多的TCP连接
     </li>
    </ul>
    <p>
     <strong>
      缺点：
     </strong>
    </p>
    <ol>
     <li>
      对头阻塞问题：由于http请求必须等待前一个请求响应才可以发送下一个请求，其他请求都需要排队，造成性能和带宽的浪费
     </li>
     <li>
      多个TCP连接，导致多个请求争夺有限的带宽
     </li>
     <li>
      每个TCP连接都需要进行慢启动
     </li>
    </ol>
    <p>
     <strong>
      拓展：
     </strong>
    </p>
    <p>
     <code>
      什么是慢启动？
     </code>
    </p>
    <p>
     在
     <code>
      TCP
     </code>
     连接时TCP内部会维护一个
     <code>
      滑动窗口
     </code>
     ，由于TCP是可靠传输，所以每次发送一段数据都会等到响应
     <code>
      ACK
     </code>
     确认收到之后发送下一段数据，但是每次确认的效率十分低下，所以出现了
     <code>
      滑动窗口
     </code>
     这个概念，发送一组数据后，只需要返回一个
     <code>
      ACK
     </code>
     就可以，加快了传输效率。并且这个窗口时可收缩的，可以根据当前网络状态实现最高效率的网络吞吐量。
    </p>
    <p>
     了解了什么是
     <code>
      滑动窗口
     </code>
     后我们继续回归主题，由于TCP第一次连接时不知道当前的网络状态，所以此时发送数据的窗口很小，如果正常响应
     <code>
      ACK
     </code>
     的话，则下次窗口会增大为上次的二倍，并且找到最大上限的窗口大小，以达到当前网络的最佳传输效率。
    </p>
    <h3>
     <a id="http_20_731">
     </a>
     http 2.0
    </h3>
    <ul>
     <li>
      为了解决http 1.1慢启动和资源争夺的问题，http 2.0将TCP连接个数减少到了一个。
     </li>
     <li>
      http 引入了
      <code>
       二进制分帧层
      </code>
      的概念，通过流来传输数据，每个流都有一个唯一的整数标识符，为了防止请求和响应的id冲突，规定客户端发起的流使用奇数ID，服务器响应的流具有偶数ID，当流传输到对面后再根据不同帧首部的流标识符进行组装。因此不同的请求可以在一个
      <code>
       TCP
      </code>
      中乱序传输，真正实现了并发传输。
     </li>
     <li>
      头部压缩，像
      <code>
       get
      </code>
      请求，大部分的数据都是请求头的占用，为了避免请求头部和响应头部的重复发送带来的性能消耗，http 2.0引入了
      <code>
       头部压缩
      </code>
      ,在服务端和客户端分别维护了一个映射表，在传输时只发送索引，大大减小了头部的数据大小。
     </li>
     <li>
      引入了数据传输的优先级，每个流都携带有一个
      <code>
       31比特
      </code>
      的优先值，根据优先级，将高优先级的帧先发给客户，但又不会绝对的按照优先级约定，防止又带来对头阻塞的问题。
     </li>
     <li>
      增加了服务器推送功能，但是又不同于
      <code>
       websocket
      </code>
      ，只能推送静态资源，提前将客户端需要的静态资源发送给客户端。
     </li>
    </ul>
    <p>
     缺点：虽然可以同时发送多个请求，但是由于tcp的特性，进入的数据包应该和出去时是一样的数据，因此如果某个数据丢包，则所有的传输会被堵塞等待重传，所以没有从根本解决了对头阻塞的问题，如果丢包率太大，这种情况会不如http1.1 6个tcp通道的效率高。
    </p>
    <h3>
     <a id="http_30_740">
     </a>
     http 3.0
    </h3>
    <ul>
     <li>
      摆脱tcp，使用udp
     </li>
     <li>
      QUIC协议：基于 UDP 实现了类似于 TCP 的多路数据流、传输可靠性等功能
     </li>
     <li>
      实现了类似 TCP 的流量控制、传输可靠性的功能。虽然 UDP 不提供可靠性的传输，但 QUIC 在 UDP 的基础之上增加了一层来保证数据可靠性传输。它提供了数据包重传、拥塞控制以及其他一些 TCP 中存在的特性。
     </li>
     <li>
      集成了 TLS 加密功能。目前 QUIC 使用的是 TLS1.3，相较于早期版本 TLS1.3 有更多的优点，其中最重要的一点是减少了握手所花费的 RTT 个数。
     </li>
     <li>
      实现了 HTTP/2 中的多路复用功能。和 TCP 不同，QUIC 实现了在同一物理连接上可以有多个独立的逻辑数据流（如下图）。实现了数据流的单独传输，就解决了 TCP 中队头阻塞的问题。
     </li>
     <li>
      实现了快速握手功能。由于 QUIC 是基于 UDP 的，所以 QUIC 可以实现使用 0-RTT 或者 1-RTT 来建立连接，这意味着 QUIC 可以用最快的速度来发送和接收数据，这样可以大大提升首次打开页面的速度。
     </li>
    </ul>
