    <h3>
     一、前言
    </h3>
    <blockquote>
     <ul>
      <li>
       伴随着云计算的第三次浪潮的滚滚而来，云原生 (CloudNative) 的概念应运而生。
      </li>
      <li>
       那究竟云原生到底是什么？我们又如何选择适合自己的开发平台？这篇文章为你做一个大概的了解。
      </li>
     </ul>
    </blockquote>
    <h3>
     二、基本概念
    </h3>
    <blockquote>
     <ul>
      <li>
       <strong>
        官方解释
       </strong>
       ：云原生是基于分布部署和统一运管的分布式云 ，以
       <strong>
        微服务
       </strong>
       、
       <strong>
        容器化
       </strong>
       、
       <strong>
        DevOps
       </strong>
       、
       <strong>
        持续交付
       </strong>
       等技术要素为基础建立的一套云技术产品体系。
      </li>
      <li>
       上面的解释，是不是觉得有点晕，云原生本质并没有确切定义。云原生是一种构建和运行应用程序的方法，是一套技术体系和方法论。云原生可拆分成
       <strong>
        云
       </strong>
       和
       <strong>
        原生
       </strong>
       两个概念。
      </li>
      <li>
       <strong>
        云
       </strong>
       和本地是相对的，现在部署的应用都跑在云端，云包含了IaaS,、PaaS和SaaS三种。
      </li>
      <li>
       <strong>
        原生
       </strong>
       就是土生土长的意思，为云而设计，在云上以最佳姿势运行，充分利用和发挥云平台的弹性+分布式优势。
      </li>
     </ul>
    </blockquote>
    <h3>
     三、微服务
    </h3>
    <blockquote>
     <ul>
      <li>
       几乎每个云原生的定义都包含微服务，而微服务是一个独立发布的应用服务，可以作为独立组件升级、灰度或复用。
      </li>
      <li>
       微服务和整体架构之间的区别，微服务由许多较小的，松散的服务组成一个应用程序，与大型紧密的应用程序的方法恰恰相反。
      </li>
     </ul>
    </blockquote>
    <h3>
     四、容器化
    </h3>
    <blockquote>
     <ul>
      <li>
       Docker是应用最为广泛的容器引擎，是软件行业最受欢迎的软件容器项目。为微服务提供实施保障，起到应用隔离作用。
      </li>
     </ul>
    </blockquote>
    <h3>
     五、DevOps
    </h3>
    <blockquote>
     <ul>
      <li>
       字面上来理解，DevOps 只是Dev（开发人员）+Ops（运维人员），突出重视软件开发人员和运维人员的沟通合作，通过自动化流程来使得软件构建、测试、发布更加快捷、频繁和可靠。
      </li>
     </ul>
    </blockquote>
    <h3>
     六、持续交付
    </h3>
    <blockquote>
     <ul>
      <li>
       持续交付可以让软件交付变得更快更频繁，即随时都可以发布，它的目标是让软件的构建、测试与发布变得更快、更频繁。
      </li>
      <li>
       持续发布的粒度一定要很小，发布频率一定要高效。
      </li>
      <li>
       整个过程来看，持续交付中整个的交付过程要保证测试、发布、开发等流程开发时间短、各项过程等待时间要少，对于出现的问题反馈要快。
      </li>
      <li>
       持续交付的质量要有所保证，做到潜在的缺陷、软件故障少。
      </li>
     </ul>
    </blockquote>
