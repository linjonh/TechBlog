    <h3>
     <a id="1_2">
     </a>
     1.四个基本概念：
    </h3>
    <p>
     <strong>
      数据(Data
     </strong>
     )是数据库中存储的基本对象。
     <br/>
     <strong>
      数据库(Database,简称DB
     </strong>
     )是长期储存在计算机内、有组织的、可共享的大量数据集合。
     <br/>
     <strong>
      数据库管理系统(Database Management System，简称DBMS
     </strong>
     )是位于用户与操作系统之间的一层数据管理软件。
     <br/>
     <strong>
      数据库系统（Database System，简称DBS
     </strong>
     ）是指在计算机系统中引入数据库后的系统构成。常常把数据库系统简称为数据库。
    </p>
    <h3>
     <a id="2_7">
     </a>
     2.数据库系统的构成：
    </h3>
    <p>
     由
     <strong>
      数据库、数据库管理系统（及其开发工具）、应用系统、数据库管理员（和用户
     </strong>
     ）构成。
    </p>
    <h3>
     <a id="3_9">
     </a>
     3.数据库系统人员
    </h3>
    <p>
     数据库管理员（简称DBA）、系统分析员和数据库设计人员、应用程序员、用户。
    </p>
    <h3>
     <a id="4DBMS_11">
     </a>
     4.数据库管理系统DBMS的功能
    </h3>
    <p>
     <strong>
      (1)数据库定义功能
     </strong>
     。DBMS提供数据定义语言(Data Definition Language, DDL),用户通过它可以方便地对数据库中的数据对象进行定义。
     <br/>
     <strong>
      (2)数据组织、存储和管理功能
     </strong>
     。通过对数据的组织和存储提高存储空间利用率和方便存取，数据库管理系統提供多种存取方法(如索引查找、Hash查找、顺序查找等)来提高存取效率。
     <br/>
     <strong>
      (3)数据操纵功能
     </strong>
     。DBMS还提供数据操纵语言(Data Manipulation Language, DML), 用户可以使用DML操纵数据。实现对数据库的基本操作，如查询、插入、删除和修改等。
     <br/>
     <strong>
      (4)数据库的事务管理和运行管理
     </strong>
     。数据库在建立、运用和维护时由数据库管理系统统一管理、统一控制，以保证数据的安全性、完整性、多用户对数据的并发使用及发生故障后的系统恢复。
     <br/>
     <strong>
      (5)数据库的建立和维护功能
     </strong>
     。数据库初始数据的输入、转换功能，数据库的转储、恢复功能，数据库的重组织功能和性能监视、分析功能等。这些功能通常是由一些实用程序或管理工具完成的。
     <br/>
     <strong>
      (6)其他功能
     </strong>
     。例如DBMS与网络中其他软件系统的通信功能:一个DBMS与另一个DBMS或文件系统的数据转换功能:异构数据库之间的互访和互操作功能等。
    </p>
    <h3>
     <a id="5DBS_18">
     </a>
     5.数据库系统DBS的特点
    </h3>
    <p>
     <strong>
      (1)数据结构化
     </strong>
     。数据库系统实现整体数据的结构化，这是数据库的主要特征之一，也是数据库系统与文件系统的本质区别。
     <br/>
     <strong>
      (2)数据的共享性高，冗余度低，易扩充
     </strong>
     。数据库的数据不再面向某个应用而是而向整个系统。因此可以被多个用户。多个应用以名种不同的语言共享使用。由于数据面向整个系统，是有结构的数据。不仅可以被多个应用共享使用，而且容易增加新的应用，这就使得数据库系统弹性大，易于扩充。
     <br/>
     <strong>
      (3)数据独立性高
     </strong>
     。数据独立性包括数据的物理独立性和数据的逻辑独立性。数据率管理系统的模式结构和二级映像功能保证了数据库中的数据具有很高的物理独立性和逻辑独立性。
     <br/>
     <strong>
      (4)数据由DBMS统一管理和控制
     </strong>
     。数据库的共享是井发的共享，即多个用户可以同时存取数据库中的数据甚至可以同时存取数据库中同一个数据。为此。DBMS必须提供统的数据控制功能，包括数据的安全性保护。数据的完整性检查、并发控制和数据库恢复。
    </p>
    <h3>
     <a id="6_23">
     </a>
     6.数据模型的分类及三要素：
    </h3>
    <p>
     （1）分类：
     <strong>
      层次模型(树形结构）、网状模型（图）、关系模型（二维表）
     </strong>
     <br/>
     （2）数据模型通常由数据结构、数据操作和完整性约束三部分组成:
     <br/>
     <strong>
      ①数据结构
     </strong>
     :它是所研究的对象类型的集合，是对系统
     <strong>
      静态
     </strong>
     特性的描述。
     <br/>
     <strong>
      ②数据操作
     </strong>
     :是指对数据库中各种对象(型)的实例(值)允许进行的操作的集合，包括操作及有关的操作规则，是对系统
     <strong>
      动态
     </strong>
     特性的描述。
     <br/>
     <strong>
      ③完整性约束条件
     </strong>
     :数据的约束条件是一组完整性规则的集合。完整性规则是给定的数据模型中数据及其联系所具有的制约和依存规则，用以限定符合数据模型的数据库状态以及状态的变化，以保证数据的正确、有效、相容。
    </p>
    <h3>
     <a id="7_29">
     </a>
     7.关系数据模型
    </h3>
    <p>
     <strong>
      关系
     </strong>
     ：一个关系对应通常说的一张表
     <br/>
     <strong>
      元组
     </strong>
     ：表中的一行即为一个元组
     <br/>
     <strong>
      属性
     </strong>
     ：表中的一列即为一个属性，给每个属性起一个名称即属性名
     <br/>
     <strong>
      码
     </strong>
     ：也称码键。表中的某个属性组，它可以唯一确定一个元组
     <br/>
     <strong>
      域
     </strong>
     ：域是一组具有相同数据类型的值的集合
     <br/>
     <strong>
      分量
     </strong>
     ：元组中的一个属性值
     <br/>
     <strong>
      关系模型
     </strong>
     ：对关系的一种描述，一般表示为
     <br/>
     关系名（属性1，属性2，…，属性n）
    </p>
    <h3>
     <a id="8_38">
     </a>
     8.数据库系统的三级模式二级映像
    </h3>
    <p>
     <strong>
      模式
     </strong>
     是相对
     <strong>
      稳定
     </strong>
     的，而
     <strong>
      实例
     </strong>
     是相对
     <strong>
      变动
     </strong>
     的
     <br/>
     <strong>
      三级模式
     </strong>
     ：模式、外模式、外模式
     <br/>
     <strong>
      二级映像
     </strong>
     ：
     <strong>
      外模式/模式映像
     </strong>
     —保证
     <strong>
      数据的逻辑独立性
     </strong>
     <br/>
     <strong>
      内模式/模式映像
     </strong>
     —保证
     <strong>
      数据的物理独立性
     </strong>
    </p>
    <h3>
     <a id="9_43">
     </a>
     9.数据独立性：
    </h3>
    <p>
     <strong>
      物理独立性、逻辑独立性
     </strong>
     <br/>
     (1)数据与程序的物理独立性:当数据库的存储结构改变，由数据库管理员对模式/内模式映像做相应改变，可以使模式保持不变，从而应用程序也不必改变，保证了数据与程序的物理独立性，简称数据的物理独立性。
     <br/>
     (2)数据与程序的逻辑独立性:当模式改变时(例如增加新的关系、新的属性、改变属性的数据类型等)。由数据库管理员对各个外模式/模式的映像做相应改变，可以使外模式保持不变。应用程序是依据数据的外模式编写的。从而应用程序不必修改，保证了数据与程序的逻辑独立性，简称数据的逻辑独立性。
    </p>
    <h2>
     <a id="_47">
     </a>
     第二章
    </h2>
    <h3>
     <a id="1_48">
     </a>
     1.关系、关系模式概念
    </h3>
    <p>
     ①关系:
     <strong>
      笛卡尔积
     </strong>
     D1XD2X…XDn的子集称叫做在域D1,D2, …, Dn上的关系，表示为R (D1，D2, … Dn)这里R表示关系的名字，n表示关系的目或度。
     <strong>
      两关系的行数相乘，列数相加
     </strong>

     <a id="7_94">

     7.视图的概念，与基本表的区别，优点，创建
    </h3>
    <p>
     (1)基本表概念：是本身独立存在的表，在sql中一个关系就对应一个表。一个(或多个)基本表对应一个存储文件，一个表可以带若干索引，索引也存放在存储文件中。
     <br/>
     (2)视图概念：
     <strong>
      是从一个或几个基本表导出的表。它本身不独立存储在数据库中，即数据库中只存放视图的定义
     </strong>
     。而不存放视图对应的数据。这些数据仍存放在导出视图的基本表中，因此视图是一个虚表。
     <br/>
     (3)基本表与视图的区别：视图本身不独立存储在数据库中，是一个虚表。即数据库中只存放视图的定义而不存放视图对应的数据，这些数据仍存放在导出视图的基本表中。
     <br/>
     (4)更新视图：
     <strong>
      插入、删除、修改数据。由于视图是不实际存储数据的虚表，因此对视图的更新最终要转换为对基本表的更新。并不是所有的视图都是可更新的
     </strong>
     <br/>
     (5)视图有以下五个优点:
     <br/>
     ①
     <strong>
      视图能够简化用户的操作
     </strong>
     :视图机制使用户可以将注意力集中在所关心的数据上，如果这些数据不是直接来自基本表。则可以通过定义视图，使数据库看起来结构简单。清晰。井且可以简化用户的数据查询操作:
     <br/>
     ②
     <strong>
      视图使用户能以多种角度看待同一数据
     </strong>
     :视图机制能使不同的用户以不同的方式看待同一数据，当许
     <br/>
     多不同种类的用户共享一个数据库时，这种灵活性是非常重要的:
     <br/>
     ③
     <strong>
      视图对重构数据库提供了一定程度的逻辑独立性
     </strong>
     :数据的逻辑独立性是指当数据库重构时，用户的应用程序不会受影响。
     <br/>
     ④
     <strong>
      视图能够对机密数据提供安全保护
     </strong>
     :有了视图机制，就可以在设计数据库应用系统时，对不同的用户定义不同的视图，使机密数据不出现在不应看到这些数据的用户视图上.这样视图机制就自动提供了对机密数据的安全保护功能。
     <br/>
     ⑤
     <strong>
      使用视图可以更清晰的表达查询。
     </strong>
    </p>

     如果对于实体集A中的每一个实体，实体集B中至多有一个(也可以没有)实体与之联系。反之亦然。则称实体集A与实体集B具有一对一联系，记为1 ：1.
     <br/>
     b.一对多联系(1：n)
     <br/>
     如果对于实体集A中的每一个实体，实体集B中有n个实体(n20) 与之联系，反之，对于实体集B中的每一个实体，实体集A中至多只有一个实体与之联系。则称实体集A与实体集B有一对多联系。记为1 ：n.
     <br/>
     c.多对多联系(m： n)
     <br/>
     如果对于实体集A中的每一个实体。实体集B中有n个实体(n&gt;10) 与之联系，反之，对于实体集B中的每一个实体，实体集A中也有m个实体(m&gt;10) 与之联系，则称实体集A与实体集B具有多对多联系，记为m:n

     <a id="1_200">
     </a>
     1.查询处理的步骤
    </h3>
    <p>
     <strong>
      (1)查询分析
     </strong>
     <br/>
     首先对查询语句进行扫描、词法分析和语法分析。从查询语句中识别出语言符号，如SQL 关键字、属性名和关系名等，进行语法检查和语法分析，即判断查询语句是否符合SQL语法规则。
     <br/>
     <strong>
      (2)查询检查
     </strong>
     <br/>
     根据数据字典中有关的模式定义检查语句中的数据库对象，如关系名、属性名是否存在和有效。如果是对视图的操作，则要用视图消解方法把对视图的操作转换成对基本表的操作。还要根据数据字典中的用户权限和完整性约束。对用户的存取权限进行检查时，如果该用户没有相应的访问权限或违反了完整性约束，就拒绝执行该查询。检查通过后便把SQL查询语句转换成内部表示，即等价的关系代数表达式。这个过程中要把數据库对象的外部:名称转换为内部表示。关系數据库管理系统- -般都用查询树(query tee,也称为语法分析树(syntax tree)来表示扩展的关系代数表达式。
     <br/>
     <strong>
      (3)查询优化
     </strong>
     <br/>
     查询优化就是选择-一个高效执行的查询处理策略。查询优化有多种方法。按照优化的层次一般可将查询优化分为代数优化和物理优化。
     <br/>
     ①代数优化:代数优化是指关系代数表达式的优化，即按照-一定的规则，通过对关系代数表达式进行等价变换，改变代数表达式中操作的次序和组合，使查询执行更高效。
     <br/>
     ②物理优化:物理优化则是指存取路径和底层操作算法的选择。
     <br/>
     <strong>
      (4)查询执行
     </strong>
     <br/>
     依据优化器得到的执行策略生成查询执行计划,由代码生成器(code generator)生成执行这个查询计划的代码。
    </p>

    <h3>
     <a id="1_218">
     </a>
     1.事务的概念及特性
    </h3>
    <p>
     <strong>
      事务是用户定 义的一个数据库操作序列，这些操作要么全做，要么全不做，是一个不可分制的工作单位。
     </strong>
     <br/>
     在SQL中，定义事务的语句一般有三条：
     <br/>
     begin transaction;
     <br/>
     commit;
     <br/>
     rollback;
     <br/>
     事务通常是以begin transaction开始，以commit或rollback结束。
     <strong>
      commit表示提交，即提交事务的所有操作
     </strong>
     。具体地说就是
     <strong>
      将事务中所有对数据库的更新写回到磁盘上的物理数据库中去
     </strong>
     ，事务正常结束。
     <strong>
      rollback表示回滚
     </strong>
     ，即在事务运行的过程中发生了某种故障，事务不能继续执行，
     <strong>
      系统将事务中对数据库的所有已完成的操作全部撤销，回滚到事务开始的状态。
     </strong>
     <br/>
     <strong>
      (1)原子性.
     </strong>
     <br/>
     事务是数据库的逻辑工作单位，事务中包括的诸操作要么都做，要么都不做。
     <br/>
     <strong>
      (2)一致性
     </strong>
     <br/>
     事务执行的结果必须是使数据库从一个-致性状态变到另一个-一致性状态。
     <br/>
     <strong>
      (3)隔离性
     </strong>
     <br/>
     一个事务的执行不能被其他事务干扰。即一个事务的内邮操作及使用的数据对其他并发事务是隔离的，并发执行的各个事务之间不能互相干扰。
     <br/>
     <strong>
      (4)持续性
     </strong>
     <br/>
     持续性也称水久性(Permanence), 指一个事务一旦提交，它对数据库中数据的改变就应该是水久性的。接下来的其他操作或故障不应该对其执行结果有任何影响。
    </p>

    <h3>
     <a id="4_250">
     </a>
     4.恢复策略
    </h3>
    <p>
     <strong>
      事务故障的恢复
     </strong>
     :
     <br/>
     (1)事务故障的恢复是由DBMS执行的。恢复步骤是自动完成的，对用户是透明的。具体过程是:
    </p>
    <ul>
     <li>
      ①反向扫描文件日志(即从后向前扫描日志文件)，查找该事务的更新操作；
     </li>
     <li>
      ②对该事务的更新操作执行逆操作，即将8志记录中“更新前的值”写入数据库:
     </li>
     <li>
      ③继续反向扫描日志文件，做同样处理；
     </li>
     <li>
      ④如此处理下去，直至读到此事务的开始标记，该事务故障的恢复就完成了。
     </li>
    </ul>
    <p>
     <strong>
      系统故障的恢复
     </strong>
     <br/>
     (2)系统故障的恢复操作是指撒销(UNDO)故障发生时未完成的事务，重做(REDO)已完成的事务。系统的恢复步骤是:
    </p>
    <ul>
     <li>
      ①正向扫描日志文件，找出在故障发生前已经提交的事务队列(REDO队列)和未完成的事务队列(UNDO队列);
     </li>
     <li>
      ②对撒销队列中的各个事务进行UNDO处理;
     </li>
     <li>
      ③对重做队列中的各个事务进行REDO处理。
     </li>
    </ul>
    <p>
     <strong>
      介质故障的恢复
     </strong>
     <br/>
     (3)介质故障是最严重的一种故障。
     <strong>
      恢复方法是重装数据库，重做已完成的事务
     </strong>
     。具体过程是:
    </p>
    <ul>
     <li>
      ① DBA装入最新的数据库后备副本(离故障发生时刻最近的转储副本) ,使数据库恢复到转储时的一致性状态:
     </li>
     <li>
      ②DBA装入转储结束时刻的日志文件副本:
     </li>
     <li>
      ③DBA启动系统恢复命令，由DBMS完成恢复功能，即重做已完成的事务。
     </li>
    </ul>

    <h3>
     <a id="1_273">
     </a>
     1.并发控制
    </h3>
    <p>
     事务是并发控制的基本单位，保证事务ACID特性是事务处理的重要任务，
     <br/>
     三类并发问题及判断
     <br/>
     并发操作带来的数据不一致性主要包括
     <strong>
      丢失修改（WW并发）、不可重复读（RW并发）和读“脏”数据（WR并发）
     </strong>
     。产生三类数据不一致性的主要原因是并发操作破坏了事务的隔离性。
     <br/>
     并发控制的主要技术有
     <strong>
      封锁
     </strong>
     （locking)、
     <strong>
      时间戳
     </strong>
     （timestamp)、
     <strong>
      乐观控制法
     </strong>
     （optimistic scheduler)和
     <strong>
      多版本开发控制
     </strong>
     （multi-version coneurreney control,MVCC)等。
    </p>
    <h3>
     <a id="2_278">
     </a>
     2.封锁
    </h3>
    <p>
     <strong>
      排他锁又称为写锁
     </strong>
     。若事务T对数据对象A加上X锁，则只允许T读取和修改A,其他任何事备都不能再对A加任何类型的锁，直到T释放A上的锁为止。这就保证了其他事务在T释放A上的锁之前不能再读取和修改A。
     <br/>
     <strong>
      共享锁又称为读锁
     </strong>
     。若事务T对数据对象A加上S锁，则事务T可以读A但不能修改A,其他事务只能再对A加S锁，而不能加X锁，直到T释放A上的S锁为止。这就保证了其他事务可以读A,但在T释放A上的S锁之前不能对A做任何修改。
    </p>
    <h3>
     <a id="3_281">
     </a>
     3.封锁协议
    </h3>
    <p>
     <strong>
      一级封锁协议
     </strong>
     :事务T在修改数据R之前必须先对其加X锁，直到事物结束才释放。
     <strong>
      解决WW丢失修改
     </strong>
     <br/>
     <strong>
      二级封锁协议
     </strong>
     :在一级封锁协议的基础上增加事务T在读取数据R之前必须先对其加S锁，读完后即可释放S锁。
     <strong>
      解决WW丢失修改和WR读脏数据
     </strong>
     <br/>
     <strong>
      三级封锁协议
     </strong>
     :在一级封锁协议的基础上增加事务T在读取数据R之前必须先对其加S锁，直到事物结束才释放。
     <strong>
      解决WW丢失修改和WR读脏数据和RW不可重复读
     </strong>
    </p>
    <h3>
     <a id="4_285">
     </a>
     4.死锁
    </h3>
    <p>
     T1在等待T2，而T2又在等待T1的局面，T1和T2两个事物永远不能结束，形成死锁。
     <br/>
     死锁的预防:一次封锁法，顺序封锁法
     <br/>
     死锁的诊断与解除:超时法，等待图法
    </p>
    <h3>
     <a id="5_289">
     </a>
     5.可串行化调度
    </h3>
    <p>
     多个事务的并发执行是正确的，当且仅当其结果与按某- -次序串行地执行这些事务时的结果相同，称这种调度策略为可串行化(serializable) 调度。可串行性srializbili)是并发事务正确调度的准则。
    </p>
    <h3>
     <a id="6_291">
     </a>
     6.两段锁协议
    </h3>
    <p>
     两段锁协议是指所有事务必须分两个阶段对数据项加锁和解锁:在对任何数据进行读、写操作之前，首先要申请并获得对该数据的封锁:在释放-一个封锁之后，事务不再申请和获得任何其他封锁。“两段"锁的含义是，事务分为两个阶段。第一阶段是获得封锁:第二阶段是释放封锁。
    </p>
