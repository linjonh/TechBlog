    <h2>
     <a id="bug_web__2">
     </a>
     日常bug web (括号内代表—目的)
    </h2>
    <h3>
     <a id="_5">
     </a>
     一。发现问题（熟悉业务）
    </h3>
    <p>
     1.了解清楚发生了什么问题,要解决成什么样子,做到能清晰的陈述出来,方便团队配合,寻求帮助
     <br/>
     2.定位是不是接口问题，是的话先让后台处理。
     <br/>
     3.要先熟悉业务，不了解需求就去问客服，问产品，问身边的同事都可以
     <br/>
     4.了解了需求之后，问清楚要做成什么 结果 再开始做。
    </p>
    <h3>
     <a id="_11">
     </a>
     二。复现问题（效率）
    </h3>
    <p>
     1.解决问题的时候一定要能复现问题，可以找测试帮忙，尽量提高效率
     <br/>
     2.越难复现的越难解决，复现不了的一般没法解决。。。
    </p>
    <h3>
     <a id="_15">
     </a>
     三。处理问题（个人能力）
    </h3>
    <p>
     1.尽快定位问题相关代码片段，只看和问题有关的地方，然后加断点跟踪问题
     <br/>
     2.在一个问题上卡了十几分钟的时候就把问题抛出来
    </p>
    <h3>
     <a id="_19">
     </a>
     四。解决问题（效率和质量）
    </h3>
    <p>
     1.要考虑后期维护的情况，尽量把逻辑写的清晰明了，加上注释
     <br/>
     2.解决完之后一定要自己测试，把可能出现问题的情况全部测试一遍，没问题之后才能提交测试，要兼顾效率和质量
    </p>
    <h3>
     <a id="_23">
     </a>
     五。其他（效率和能力）
    </h3>
    <p>
     1.项目延期的话提前说原因和更改时间
     <br/>
     2.项目经理都是有经验的，可以多问问，耐心听一下他们的建议，有工作经验肯定想的全面一点
     <br/>
     3.没把握的事儿就问问别人
     <br/>
     4.对自己处理的问题负责一点，尽量别出现逻辑方面的bug
     <br/>
     5.没事儿可以看看别人的代码，提升自己
    </p>
    <h3>
     <a id="__30">
     </a>
     六。发版 （稳定）
    </h3>
    <p>
     1.要有
     <mark>
      责任心
     </mark>
     ，要把自己的活儿干好，别让人笑话，写个功能一堆bug
     <br/>
     2.先在测试上测一遍再发正式
     <br/>
     3.只要确保自己发的每一个字母都是没问题的就好！保证没问题
     <br/>
     4.记得提交代码，最好发版之前之后十分钟内提交，不要引起不必要的麻烦
    </p>
