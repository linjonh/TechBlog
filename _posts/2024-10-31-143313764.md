    <h2>
     <a id="_0">
     </a>
     软件质量模型
    </h2>
    <p>
     应用场景：提供对于软件产品从测试角度思考的一种思路
     <br/>
     定义：实际实现的产品和需求描述是否相一致，相一致程度高说明质量满足需求（好）
     <br/>
     如何评判软件的质量？
     <br/>
     功能：软件产品是否具备某种能力 某手机是否支持5G
     <br/>
     性能：软件产品对于时间和空间的占用程度高低 速度快、占用空间小
     <br/>
     兼容性：软件兼容其他类型的软硬件的能力 著名的"3Q"大战
     <br/>
     易用性：在
     <strong>
      一定用户群
     </strong>
     的基础上，软件是否好用、容易理解 遵循专业性，例如：财务系统软件（账单、流水等）
     <br/>
     可靠性：软件是否具备持续无故障运行的能力
     <br/>
     安全性：软件运行过程中对于数据的传输和存储是否安全 属于专项测试，要求较高
     <br/>
     可移植性：软件产品从一个环境移植到另一个环境中正常运行的能力
     <br/>
     可维护性：软件出现故障后，自我修复/恢复的能力
    </p>
    <h2>
     <a id="_12">
     </a>
     软件的生命周期
    </h2>
    <p>
     软件生命周期：软件从无到有到消亡的过程
     <br/>
     软件生命周期也叫软件开发过程模型、软件生命周期模型
    </p>
    <h3>
     <a id="_15">
     </a>
     瀑布模型
    </h3>
    <p>
     描述软件生成到消亡的过程模型图
     <br/>
     该模型目前实际工作中已不常用，但是该模型是其他新型模型的鼻祖
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/f18dcec06cca4f7ab6d8001a6b7de712.png"/>
    </p>
    <h4>
     <a id="_19">
     </a>
     瀑布模型的优点
    </h4>
    <p>
     每个阶段比较清楚，并且有对应的文档产生
     <br/>
     当前一个阶段完成后，才开始后面的阶段（一次性的）
    </p>
    <h4>
     <a id="_22">
     </a>
     瀑布模型的缺点
    </h4>
    <p>
     发现问题的时机比较晚，失去提前纠错的机会
     <br/>
     测试介入比较晚
    </p>
    <h4>
     <a id="_25">
     </a>
     适用场景
    </h4>
    <p>
     适用于需求不易发生变化的大项目
    </p>
    <h3>
     <a id="_27">
     </a>
     敏捷开发模型
    </h3>
    <p>
     能够适用需求的变化，并且能够给出快速的响应
     <br/>
     小步快跑
     <br/>
     ACP
    </p>

     <a id="_31">
     </a>
     软件测试模型
    </h2>
    <h3>
     <a id="V_32">
     </a>
     V模型
    </h3>
    <p>
     作用：主要描述测试、开发之间的对应关系
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/42672d7d85294f7393435d1cc466f13a.png"/>
    </p>
    <h4>
     <a id="V_35">
     </a>
     V模型优点
    </h4>
    <p>
     每个阶段比较清楚，测试过程由底层（代码）测试到高层（应用）测试过程
    </p>
    <h4>
     <a id="V_37">
     </a>
     V模型缺点
    </h4>
    <p>
     不适用于需求的变更，发现问题的时机比较晚
    </p>
    <h3>
     <a id="W_39">
     </a>
     W模型
    </h3>
    <p>
     作用：将测试过程更加细化说明，对应测试、开发之间的关系更加清楚
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/87e7c3bfe6be4997a87d558f26c98b79.png"/>
    </p>
    <h4>
     <a id="W_42">
     </a>
     W模型优点
    </h4>
    <p>
     测试介入时间早，能够及时发现问题，降低修复成本
     <br/>
     测试伴随整个软件生产周期，除了测试软件之外，还需要验证文档
    </p>
    <h4>
     <a id="W_45">
     </a>
     W模型缺点
    </h4>
    <p>
     该模型应用起来复杂度高（具备计算机技能、业务能力、管理能力、测试素质）
    </p>

    <h2>
     <a id="_59">
     </a>
     测试用例设计方法
    </h2>
    <h3>
     <a id="_60">
     </a>
     等价类划分法
    </h3>
    <h4>
     <a id="_61">
     </a>
     等价类定义
    </h4>
    <p>
     在批量的测试数据中，选取具有共同特征的数据子集作为测试的输入
    </p>
    <h4>
     <a id="_63">
     </a>
     等价类的分类
    </h4>
    <p>
     有效等价类：满足需求的测试数据
     <br/>
     无效等价类：不满足需求的测试数据
    </p>
    <h4>
     <a id="_66">
     </a>
     等价类划分法设计用例步骤
    </h4>
    <p>
     案例：如何测试两个两位数整数之间的和（即-99到99之间数据求和）没有问题？
     <br/>
     明确需求
     <br/>
     1.测试目的 eg ： 验证两位数整数是否能够正常求和
     <br/>
     2.测试条件 eg : 不超过两位数 整数、字母、中文、特殊符号、空格、空
     <br/>
     长度 类型（来源于键盘输入） 规则
     <br/>
     3.划分等价类
     <br/>
     有效等价类：满足需求的所有条件 eg : -10 , 30
     <br/>
     无效等价类：不满足（只要不满其中一个条件即可）需求的所有输入数据 eg ：包含字母、中文、符号、空格、空
     <br/>
     4.提取数据编写用例 通过测试用模板按照要求填写内容
    </p>
    <h4>
     <a id="_76">
     </a>
     等价类适用场景
    </h4>
    <p>
     针对有批量数据输入的测试场景，无法穷举测试时适用
     <br/>
     常见代表：输入框（下拉框、选择框、下拉列表等）
    </p>
    <h4>
     <a id="_79">
     </a>
     编写用例注意事项
    </h4>
    <p>
     单模块测试中，用例标题具有唯一性
     <br/>
     必要步骤尽可能清楚
     <br/>
     预期结果尽量描述测试结论性的语句及现象（如果有具体现象最好描述）
     <br/>
     用例编号和测试项目简称对应设置
    </p>
    <h3>
     <a id="_84">
     </a>
     边界值分析法
    </h3>
    <p>
     引入的场景：开发人员常常在边界的位置容易出现问题，此时需要针对边界位置再测试
     <br/>
     常在河边走，哪有不湿鞋
    </p>
    <h4>
     <a id="_87">
     </a>
     边界范围的确定
    </h4>
    <p>
     根据需求，将数据类型和边界确定，直接可以获取对应的边界范围值
     <br/>
     上点：刚好等于边界的值 （取值不考虑开闭区间）
     <br/>
     离点：刚好小于/大于边界上的值 （取值类型看需求）
     <br/>
     内点：边界范围内的任何取值 （取中间的值）
    </p>
    <h4>
     <a id="_92">
     </a>
     边界值分析法设计用例步骤
    </h4>
    <p>
     明确需求
     <br/>
     测试目的
     <br/>
     测试条件
     <br/>
     长度
     <br/>
     类型
     <br/>
     规则
     <br/>
     划分等价类
     <br/>
     有效等价类
     <br/>
     无效等价类
     <br/>
     确认边界范围值
     <br/>
     确定边界范围值之后，结合等价类进行合并补充
     <br/>
     上点
     <br/>
     离点（可以优化）
     <br/>
     内点（可以和有效等价类的取值合并）
     <br/>
     提取数据编写用例
     <br/>
     按照用例模板编写内容即可
    </p>
    <h5>
     <a id="_109">
     </a>
     离点优化
    </h5>
    <p>
     目的：减少用例数量，由7条数据变5条数据
     <br/>
     注意事项：非必须，可以不用优化
     <br/>
     <strong>
      离点优化：离点的选取和上点取值相反的取值点即可
     </strong>
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/1138943868b04bbd9c497395ae0ed0f2.png"/>
    </p>
    <h4>
     <a id="_114">
     </a>
     边界值适用场景
    </h4>
    <p>
     对于等价类划分法的完善和补充
     <br/>
     针对有边界范围的批量数据的输入类测试（重点关注边界）
     <br/>
     典型代表：输入框（有边界范围区间）
    </p>
    <h3>
     <a id="_118">
     </a>
     判定表法
    </h3>
    <h4>
     <a id="_119">
     </a>
     判定表引入
    </h4>
    <p>
     判定表：是一种以表格形式表达多条件逻辑判断的工具
    </p>
    <h4>
     <a id="_121">
     </a>
     判定表构成
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/60a9b37035f4434aace60e7371410e31.png">
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/d81c69df58de41b4b23e667dd90a5d1e.png"/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/1f824a375e444fa59bb81dde6b50c355.png"/>
     </img>
    </p>
    <p>
     条件桩：列表需求中所有条件，次序无关
     <br/>
     灰色区域
     <br/>
     动作桩：列出需求中条件可能采取的操作（动作），可能存在多个，次序无所谓
     <br/>
     绿色区域
     <br/>
     条件项：所有条件对应取值（一般取真假值）的全组合
     <br/>
     黄色区域
     <br/>
     动作项：上述条件项对用操作结果（通过条件项得到的结果）
     <br/>
     蓝色区域
    </p>
    <h4>
     <a id="_132">
     </a>
     根据表计算测试用例
    </h4>
    <p>
     如果条件的取值只有两个，那么每种条件的组合数量为2的n次方
     <br/>
     规则：每种条件项和动作项对应的一列就是一条规则，也叫一条测试用例
     <br/>
     判定表设计用例步骤
     <br/>
     明确需求
     <br/>
     测试目的
     <br/>
     测试条件：根据需求列出
     <br/>
     画出判定表
     <br/>
     列出条件桩和动作桩（根据需求来分析提取）
     <br/>
     在条件桩前面加判定词，根据条件数量进行组合得到所有取值（条件项）
     <br/>
     根据每种条件的组合得到动作项
     <br/>
     优化合并相同的条件
     <br/>
     按照规则编写用例
     <br/>
     按照测试用例模板编写即可
    </p>
    <h4>
     <a id="_146">
     </a>
     判定表的适用场景
    </h4>
    <p>
     针对需求中有多个条件，并且条件和条件之间有组合关系，条件和结果之间有制约（因果）关系的场景
     <br/>
     常见词汇：如果…那么…；若…则…
     <br/>
     局限性：条件个数不易过多（不超过4个）
     <br/>
     注意：超过4个条件的不常见，如果出现超过4个以上条件的，可以使用因果图（网络查询）
    </p>
    <h3>
     <a id="_152">
     </a>
     场景法
    </h3>
    <p>
     也叫流程图法，通过流程图的描述用户的使用场景过程，验证整个产品的业务（用户使用过程）是否正常
     <br/>
     用户：用户使用更加关注整个系统的应用
     <br/>
     测试：测试不仅仅要关注单功能测试，还需要关注系统之间组合测试
    </p>
    <h4>
     <a id="_156">
     </a>
     适用场景
    </h4>
    <pre><code>一般在测试的后期，对于整个系统的模块功能进行全部的组合测试
测试的依据：业务流程图
</code></pre>

    <h3>
     <a id="_180">
     </a>
     错误推测法
    </h3>
    <h4>
     <a id="_181">
     </a>
     错误推测法介绍
    </h4>
    <p>
     要求：有实际项目测试经验的人使用
     <br/>
     定义：通过直觉（经验）或者智慧推测系统可能出现问题的地方进行再次测试
    </p>
    <h4>
     <a id="_184">
     </a>
     错误推测法适用场景
    </h4>
    <p>
     1.时间紧迫：通过以往类似项目的经验，提取当前项目中核心模块（出现问题较多）进行验证测试
     <br/>
     2.时间宽裕：在基础测试的基础上，将原有模块（存在问题较多）进行再次细化测试
    </p>
    <h4>
     <a id="_187">
     </a>
     问题
    </h4>
    <p>
     上述两种场景是否需要测试用例？
     <br/>
     场景1：需要提取核心模块（按照用例的优先级）用例进行测试
     <br/>
     通过xmind思维导图先整理个大概的测试点
     <br/>
     找有相关项目经验的测试人员去测试
     <br/>
     通过自动化的技术或者找能够提升测试效率的技术去实现测试
     <br/>
     场景2：需要将原有用例细化完善后，按照用例依次测试即可
    </p>
    <h2>
     <a id="_194">
     </a>
     软件的缺陷
    </h2>
    <h3>
     <a id="_195">
     </a>
     缺陷的定义
    </h3>
    <p>
     判断的依据：用户需求说明书 ----&gt; 编写测试用例
     <br/>
     软件在使用（运行）过程中存在的任何问题（如：错误、异常、失效等），都叫软件的缺陷，简称bug
    </p>
    <h3>
     <a id="_198">
     </a>
     缺陷的判定标准
    </h3>
    <p>
     如何确定一个问题是不是缺陷？
     <br/>
     软件未实现需求（规格）说明书中明确要求的功能 —— 没有做
     <br/>
     eg : 手机是否支持5G功能
     <br/>
     软件出现了需求（规格）说明书中指明不应该出现的错误 —— 做错了
     <br/>
     eg : 登录失败应该有错误提示，但是没有提示
     <br/>
     软件实现的功能超出需求（规格）说明书指明的范围 —— 做多了
     <br/>
     eg : 实现计算的软件出现聊天互动等功能
     <br/>
     软件未实现需求（规格）说明书中虽未明确指明但应该实现的要求 —— 没做完
     <br/>
     eg ：软件的产品对于金额默认保留两位有效数据，但是没有保留（需求中未写）
     <br/>
     软件难以理解，不易使用，运行缓慢，用户体验不好 —— 不完美
     <br/>
     eg ：使用过卡顿，响应速度慢、界面不好看等
    </p>
    <h3>
     <a id="1_211">
     </a>
     问题1
    </h3>
    <p>
     面试官：如果你发现了一个问题认为是bug，开发不认为是一个bug如何处理？
     <br/>
     首先确认是否是因为开发和测试的环境因素导致问题不一致
     <br/>
     其次，如果部分环境因素，可根据需求说明书确认实际产品和需求是否一致
     <br/>
     再者，需求不清楚的通过找产品确认，需求设计是否有问题，然后再和开发对接确认
    </p>
    <h3>
     <a id="_216">
     </a>
     【扩展】缺陷的级别
    </h3>
    <p>
     违反上述标准的1、2、3条件，基本上属于高严重级的bug
     <br/>
     违反上述标准的4、5条件，基本属于中低级别的bug
     <br/>
     缺陷产生原因阶段
    </p>
    <h3>
     <a id="_220">
     </a>
     为什么需要分析缺陷的原因？
    </h3>
    <p>
     能帮助测试领导确定产品出现质量问题的具体阶段，方便后续软件产品质量的优化
     <br/>
     能够帮助测试人员积累经验
     <br/>
     需求阶段：需求描述不易理解，有歧义、错误等
     <br/>
     设计阶段： 设计文档存在错误或者缺陷
     <br/>
     编码阶段：代码出现错误（语法、单词、标点符号等）
     <br/>
     运行系统：软硬件系统本身故障导致软件缺陷
     <br/>
     故障解决阶段：对于系统不熟悉修复问题时引入新bug
    </p>

    <h4>
     <a id="_238">

     缺陷的状态：描述缺陷生命周期的过程
    </h4>
    <p>
     新建（new）: 表示缺陷的产生
     <br/>
     打开（open）：表示开发确认通过
     <br/>
     拒绝（rejected）：表示开发确认不通过
     <br/>
     进行中（inprogress）：表示开发正在修复缺陷
     <br/>
     已修复（fixed）：表示开发已经修复完成
     <br/>
     延迟修复（delay/postpone）：表示开发延迟修复
     <br/>
     测试通过（closed）：表示测试通过，已关闭
     <br/>
     测试不通过（reopen/open）：表示测试不通过，重新打开
    </p>
    <h4>
     <a id="_247">

     缺陷的所属模块：类似于用例的所属项目，描述缺陷产生的模块范围

    <h4>
     <a id="__P1priority_248">
     </a>
     缺陷的优先级：告诉开发当前缺陷修改的先后次序 （ P1）priority
    </h4>
    <p>
     urgent priority : 最高优先级
     <br/>
     veryhigh priority : 次高优先级
     <br/>
     high priority : 高优先级
     <br/>
     medium priority ： 中优先级
     <br/>
     low priority：低优先级
    </p>
    <h4>
     <a id="_S1serious_254">
     </a>
     缺陷的严重级：告诉产品当前缺陷对于整个产品的破坏程度（和优先级一一对应） （S1）serious
    </h4>
    <p>
     critical ：致命的破坏
     <br/>
     major ：高的破坏性
     <br/>
     medium ： 中等破坏
     <br/>
     minor：低等破坏
     <br/>
     tiny ： 微小的破坏
    </p>

    <h3>
     <a id="_278">
     </a>
     编写缺陷报告规范
    </h3>
    <p>
     可复现：确保当前发现的bug能够复现
     <br/>
     唯一性：确保一个缺陷报告中上报一个问题
     <br/>
     eg：登录测试失败了，在数据库对应表中发现当前的账号密码信息是明文的
     <br/>
     此时描述的是两个问题，可以分开报bug
     <br/>
     规范性：遵循公司规定的报bug的要求
     <br/>
     eg : 客服人员发现的bug，需要在标题中单独标记
     <br/>
     【客服】正确的账号登录过程中出现的失败的结果
    </p>
    <h3>
     <a id="_286">
     </a>
     注意事项
    </h3>
    <p>
     确保上报的bug是准确的
     <br/>
     描述尽可能简洁易懂具体
     <br/>
     不能使用感情色彩的词语
     <br/>
     不能使用模棱两可的词汇
     <br/>
     不能使用人称代词
    </p>
    <h3>
     <a id="_292">
     </a>
     面试题
    </h3>
    <p>
     在实际测试中如果出现不可复现的bug怎么办？
     <br/>
     经过多次复现后，还是没有出现，此时在本地记录当前的问题
     <br/>
     回顾当时操作的流程及测试环境的配置要求，确认是否由于操作失误或者环境临时故障引起
     <br/>
     请开发协助（自己）查找当前测试模块是否有对应的日志信息（日志的位置可以问开发）
     <br/>
     再考虑更换一套环境查看是否能够复现上述问题
     <br/>
     在后续的版本中测试，此时需要关注当时测试该功能时是否还出现上述的问题
     <br/>
     在后续版本还出现过，需要开发协助打印日志进行分析定位，同时测试需要提交bug
    </p>
