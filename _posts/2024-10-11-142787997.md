    <h2 id="1.%20%E4%BA%91%E5%8E%9F%E7%94%9F%E6%A6%82%E5%BF%B5%E5%92%8C%E7%89%B9%E7%82%B9">
     1. 云原生概念和特点
    </h2>
    <blockquote>
     <ul>
      <li>
       <strong>
        概念
       </strong>
       <ul>
        <li>
         云原生是一种
         <strong>
          构建和运行应用程序的方法，程序
          <span style="color:#fe2c24">
           生于云端，长于云端
          </span>
          。
         </strong>
         从有构建应用的想法开始，到需求、设计、开发、测试、构建、打包、部署所有的软件生命周期全部都在云平台上面进行，从应用设计之初（技术选型、架构设计、编译机制）就充分考虑并符合了云的特征，在云平台以最佳姿态原型、为企业降本增效。
        </li>
       </ul>
      </li>
      <li>
       <strong>
        特点
       </strong>
       <ul>
        <li>
         <span style="color:#fe2c24">
          弹性扩缩容
         </span>
         ：本地部署的传统应用无法动态扩展，往往需要冗余资源以抵抗流量高峰，而云原生应用利用云的弹性自动伸缩，应用程序快速复制扩展、部署。
        </li>
        <li>
         快速启停：应用程序可以快速启停以应对流量变化
        </li>
        <li>
         隔离性强：进程级别的故障隔离
        </li>
        <li>
         CICD：持续集成、持续交付、持续部署
        </li>
       </ul>
      </li>
     </ul>
    </blockquote>

    <h2 id="3.%20%E4%BA%91%E5%AF%B9%E5%A4%96%E6%8F%90%E4%BE%9B%E6%9C%8D%E5%8A%A1%E7%9A%84%E6%9E%B6%E6%9E%84%E6%A8%A1%E5%BC%8F">
     3. 云对外提供服务的架构模式
    </h2>
    <p class="img-center">
     <img alt="" height="288" src="https://i-blog.csdnimg.cn/direct/b048d245a4354adf8dc65d8b38cc586c.png" width="412"/>
    </p>
    <h4 id="3.1%20IaaS%EF%BC%88Infrastructure-as-a-Service%EF%BC%89" style="background-color:transparent">
     3.1 IaaS（Infrastructure-as-a-Service）
    </h4>
    <blockquote>
     <p>
      <strong>
       基础设施即服务
      </strong>
     </p>
     <p>
      向外提供硬件资源等基础设施，包括计算、存储、网络等等，用户可以基于基础设施进行上层应用开发部署。
     </p>
     <p>
      拿租房比喻就是提供毛坯房，自己装水电、置办家具。
     </p>
    </blockquote>
    <h4 id="3.2%20PaaS%EF%BC%88Platform-as-a-Service%EF%BC%89">
     3.2 PaaS（Platform-as-a-Service）
    </h4>
    <p>
     <strong>
      平台即服务
     </strong>
    </p>
    <blockquote>
     <p>
      向外提供平台组件服务，如操作系统、数据库。
     </p>
     <p>
      拿租房比喻就是提供装好水电的房子，自己只需置办家具即可入住。
     </p>
    </blockquote>
    <h4 id="3.3%20SaaS%EF%BC%88SoftWare-as-a-Service%EF%BC%89">
     3.3 SaaS（SoftWare-as-a-Service）
    </h4>
    <p>
     <strong>
      软件即服务
     </strong>
    </p>
    <blockquote>
     <p>
      直接向外提供一款成品应用型服务，屏蔽了用户对软件底层的基础设施，用户只需要拿来使用即可。如钉钉、企业微信。
     </p>
     <p>
      拿租房比喻就是提供装好水电、家具的房子，直接交租金就拎包入住。
     </p>
    </blockquote>
    <h4 id="3.4%20FaaS%EF%BC%88Function-as-a-Service%EF%BC%89">
     3.4 FaaS（Function-as-a-Service）
    </h4>
    <p>
     <strong>
      功能即服务
     </strong>
    </p>
    <p>
     <a href="https://www.ibm.com/cn-zh/topics/faas" rel="nofollow" title="https://www.ibm.com/cn-zh/topics/faas">
      https://www.ibm.com/cn-zh/topics/faas
     </a>
    </p>
    <blockquote>
     <p>
      FaaS是一种云计算服务，专注于
      <span style="color:#fe2c24">
       事件驱动，
      </span>
      在有请求时自动启动服务，没有时自动关闭服务。
     </p>
     <p>
      Serverless和FaaS经常被混为一谈，我认为
      <span style="color:#fe2c24">
       <strong>
        FaaS算是无服务器的子集
       </strong>
      </span>
      。
     </p>
     <p>
      无服务器专注于所有服务类别，无论是计算、存储、数据库、消息传递还是 API 网关等。其中服务器的配置、管理和计费对最终用户不可见，用户
      <span style="color:#fe2c24">
       只需要对服务按需付费即可
      </span>
      。
     </p>
    </blockquote>
    <h2 id="4.%20%E4%BA%91%E5%8E%9F%E7%94%9F%E6%A0%B8%E5%BF%83%E6%8A%80%E6%9C%AF%E6%A0%88">
     4. 云原生核心技术栈
    </h2>
    <h3 id="4.1%20%E5%BE%AE%E6%9C%8D%E5%8A%A1">
     <span style="color:#0d0016">
      4.1 微服务
     </span>
    </h3>
    <blockquote>
     <ul>
      <li>
       <strong>
        单体架构
       </strong>
       ：把业务所有功能集中在一个项目中开发，
       <span style="color:#fe2c24">
        以整个系统为单位进行部署
       </span>
       ，这种架构简单，如果某一业务的请求量非常大，那么是无法单独扩展该业务的，只能拷贝整个单体应用，再部署一套环境，来实现集群。
      </li>
      <li>
       <strong>
        微服务架构
       </strong>
       ：根据业务把整个项目划分成多个功能模块，比如订单模块、购物车模块、支付模块、商品详情模块等等，模块之间通过http或者RPC进行通信。这种架构降低了服务耦合，有利于服务扩展，同时每个服务模块实现了
       <span style="color:#fe2c24">
        故障隔离，提高了可用性
       </span>
       ！
      </li>
     </ul>
     <p>
      <span style="color:#0d0016">
       SpringCloud就是微服务中具有代表性的一个技术栈。
      </span>
     </p>
     <p class="img-center">
      <img alt="" height="272" src="https://i-blog.csdnimg.cn/direct/4fe93ce6be54485da38788f47eb110a7.png" width="469"/>
     </p>
    </blockquote>
    <h3 id="4.2%20%E5%AE%B9%E5%99%A8%E6%8A%80%E6%9C%AF-Docker%E3%80%81K8s">
     <span style="color:#0d0016">
      4.2 容器技术-Docker、K8S
     </span>
    </h3>
    <blockquote>
     <p>
      <span style="color:#0d0016">
       所谓容器，对操作系统（通常为 Linux）进行虚拟化，具有比虚拟机更高的可移植性和资源效率
      </span>
      ，可以
      <span style="color:#0d0016">
       解决环境差异带来的部署等问题。
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       我们把单体项目拆成了微服务，各个微服务模块所需的部署环境可能大不相同，那么不妨把每个微服务模块放到容器中，这个容器包含了服务模块运行所需的除操作系统内容以外所需的函数、配置、依赖等，类似exe安装包，这就不仅
      </span>
      <span style="color:#fe2c24">
       解决了环境差异带来的应用部署问题
      </span>
      <span style="color:#0d0016">
       ，而且各个容器之间实现
      </span>
      <span style="color:#fe2c24">
       <strong>
        进程隔离
       </strong>
       ，容器启动速度也更快
      </span>
      <span style="color:#0d0016">
       。
      </span>
     </p>
    </blockquote>
    <blockquote>
     <p>
      <span style="color:#0d0016">
       以Docker容器为资源分割和调度的基本单位，封装整个软件运行时的环境，然后发布到Linux机器上。
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       按照Docker的设计方案，应用软件的交付过程如同海上运输，操作系统如同一个货轮，操作系统上的软件都如同一个集装箱。用户可以通过标准化手段自由组装运行环境，同时集装箱的内容可以由用户自定义，然后
      </span>
      <span style="color:#fe2c24">
       使用k8s编排管理容器的生命周期
      </span>
      <span style="color:#0d0016">
       。如此一来，交付一个应用软件产品，就相当于交付一系列标准化组件的集合。
      </span>
     </p>
    </blockquote>
    <h3 id="4.3%20DevOps%26CICD">
     <span style="color:#0d0016">
      4.3 DevOps&amp;CICD

     <p>
      <span style="color:#0d0016">
       Development和Operations，即开发运维一体化，涉及软件在整个开发生命周期中的
      </span>
      <span style="color:#fe2c24">
       持续开发，持续测试，持续集成，持续部署和持续监控
      </span>
      <span style="color:#0d0016">
       。简单来说是开发和运维之间地高度协同，实现全生命周期的工具全链路打通与自动化、跨团队的线上
       <strong>
        协作能力
       </strong>
       。完成高频率部署的同时，提高生产环境的可靠性、稳定性、弹性和安全性。
      </span>
     </p>
    </blockquote>

     <img alt="" height="290" src="https://i-blog.csdnimg.cn/direct/6fb8ce3fa5334d3690e0da96176f643e.png" width="479"/>
    </p>

     <p>
      <strong>
       CI：持续集成
      </strong>
     </p>
     <ul>
      <li>
       持续集成：开发团队通过将代码的不同部分通过

       集成到共享存储库中，
       <span style="color:#fe2c24">
        系统可以自动频繁地进行构建和测试

    <h3 id="4.4%20Serverless">
     <span style="color:#0d0016">
      4.4 Serverless
     </span>
    </h3>
    <p>
     <a href="https://www.redhat.com/zh/topics/cloud-native-apps/what-is-serverless" rel="nofollow" title="Serverless 是什么？无服务器架构简介-红帽">
      Serverless 是什么？无服务器架构简介-红帽
     </a>
    </p>
    <p>
     <a href="https://developer.aliyun.com/article/857454" rel="nofollow" title="一文读懂 Serverless 的起源、发展和落地实践-阿里云开发者社区">
      一文读懂 Serverless 的起源、发展和落地实践-阿里云开发者社区
     </a>
    </p>
    <blockquote>
     <ul>
      <li>
       <p class="img-center">
        <img alt="" height="229" src="https://i-blog.csdnimg.cn/direct/ee82fb15328e483aaeca87f574570d83.png" width="389"/>
       </p>
      </li>
      <li>
       Serverless并不是不需要服务器，而是将服务器全权托管给了云厂商，用户聚焦业务代码，无需关心管理服务器，只用把业务部署到平台的容器上，服务器能自动进行弹性伸缩，这些容器在
       <span style="color:#fe2c24">
        <strong>
         被调用时会自动按需启动
        </strong>
       </span>
       。
      </li>
     </ul>
    </blockquote>
    <h3 id="4.5%20%E4%B8%8D%E5%8F%AF%E5%8F%98%E5%9F%BA%E7%A1%80%E8%AE%BE%E6%96%BD">
     <span style="color:#0d0016">
      4.5 不可变基础设施
     </span>
    </h3>
    <blockquote>
     <p>
     </p>
     <p>
      <span style="color:#0d0016">
       在传统的可变服务器基础架构中，开发人员操作服务器，手动升级或降级软件包，逐个服务器地调整配置文件，服务器会不断更新和修改。
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       <strong>
        可变基础设施通常会导致以下问题：
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       持续的修改服务器，缺乏标准，易引入不稳定因素，会导致灾难发生后很难重新构建起等效的新服务。
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       而不可变基础设施，最基本的指运行服务的服务器在完成部署后，就不在进行更改，如果配置发生了改变就会
      </span>
      <span style="color:#fe2c24">
       生成新的容器，旧容器直接销毁
      </span>
      <span style="color:#0d0016">
       。这就保证了基础架构中更高的一致性和可靠性，以及更简单，更可预测的部署过程。这样云原生就有了稳定的基石！
      </span>
     </p>
    </blockquote>
    <h3 id="4.6%20%E5%A3%B0%E6%98%8E%E5%BC%8FAPI">
     <span style="color:#0d0016">
      4.6 声明式API（k8s）
     </span>
    </h3>
    <blockquote>
     <ul>
      <li>
       <em>
        在命令式 API 中，我们可以直接发出服务器要执行的命令，例如：
       </em>
       <em>
        “运行容器”、“停止容器”等。
       </em>
       通俗的说，命令式编程是第一人称，我要做什么，我要怎么做。操作系统最喜欢这种编程范式了，操作系统几乎不用“思考”, 只要一对一的将代码翻译成指令就可以了。
      </li>
      <li>
       <em>
        在声明式API中，我们声明系统要执行的操作，系统将不断向该状态驱动。
       </em>
       声明式编程类似于“第二人称”， 也就是你要做什么，这有点“”产品经理”和“开发”之间的关系，“产品经理”只负责提需求，而“开发”怎么实现的，他并不关心
      </li>
     </ul>
    </blockquote>
    <h3 id="4.7%20Service%20Mesh%EF%BC%88%E6%9C%8D%E5%8A%A1%E7%BD%91%E6%A0%BC%EF%BC%89">
     <span style="color:#0d0016">
      4.7 Service Mesh（服务网格）
     </span>
    </h3>
    <blockquote>
     <p>
      <strong>
       为什么使用服务网格
      </strong>
     </p>
     <ul>
      <li>
       <span style="color:#0d0016">
        应用程序性一定程度上能取决于服务之间通信的速度和弹性。开发人员必须跨服务监控和优化应用程序，但由于系统的分布性质，他们很难获得可见性，在没有服务网格层时，开发人员把服务间的通信逻辑编码到每个服务中，当应用程序越来越大并且在同一个服务上同时运行多个实例时，微服务之间通信将会变得愈发复杂，业务代码和非业务代码糅合在一起。
       </span>
      </li>
      <li>
       <span style="color:#0d0016">
        Service mesh可以处理应用程序中服务之间的所有通信，同时提供了监控、记录、跟踪和流量控制等新功能。
       </span>
      </li>
     </ul>
    </blockquote>
    <h4 id="4.7.1%20%E6%9C%8D%E5%8A%A1%E7%BD%91%E6%A0%BC%E5%A6%82%E4%BD%95%E5%B7%A5%E4%BD%9C">
     4.7.1 服务网格如何工作
    </h4>
    <blockquote>
     <p>
      <span style="color:#fe2c24">
       <strong>
        服务网格从单个服务中提取控制服务间通信的逻辑，并抽象到自己的基础设施层
       </strong>
      </span>
      <span style="color:#0d0016">
       （如Istio）。它使用多个网络代理来路由和跟踪服务之间的通信。
      </span>
     </p>
     <p>
      <span style="color:#0d0016">
       代理充当组织网络和微服务之间的中间网关。所有进出该服务的流量都通过代理服务器路由。代理有时被称为
       <em>
        sidecar（直译为边车），sidercar和微服务块并行运行
       </em>
       ，这些代理一起构成了服务网格层。
      </span>
      <br/>
     </p>
     <p>
      <span style="color:#0d0016">
       下面的网格中，绿色是一个个微服务，代表不同的功能模块，蓝色就是每个微服务的代理他们从绿色的微服务中提取出来下沉到Istio等设施，负责服务间的通信、监控等。
      </span>
     </p>
     <p>
      <img alt="" height="559" src="https://i-blog.csdnimg.cn/direct/74f3b94da328485a92569cdba71515b8.png" width="1200"/>
     </p>
    </blockquote>
    <h4 id="4.7.2%20%E6%9C%8D%E5%8A%A1%E7%BD%91%E6%A0%BC%E4%BC%98%E7%82%B9%C2%A0">
     4.7.2 服务网格优点
    </h4>
    <blockquote>
     <ul>
      <li>
       <strong>
        服务发现
       </strong>
       <br/>
       服务网格使用服务注册表来
       <strong>
        <span style="color:#fe2c24">
         动态发现和跟踪
        </span>
       </strong>
       网格中的所有服务，减少管理服务端点的运维负担。
      </li>
      <li>
       <strong>
        负载均衡
       </strong>
       <br/>
       服务网格使用各种算法（例如循环算法、最少连接或加权负载均衡）在多个服务实例之间
       <span style="color:#fe2c24">
        <strong>
         智能地分配请求
        </strong>
       </span>
       。负载均衡可提高资源利用率并确保高可用性和可扩展性。
      </li>
      <li>
       <strong>
        流量管理
       </strong>
       <br/>
       服务网格提供高级
       <strong>
        <span style="color:#fe2c24">
         流量管理
        </span>
       </strong>
       功能，可对请求路由和流量行为进行精细控制。
      </li>
      <li>
       <strong>
        流量分割
       </strong>
       <br/>
       将传入流量划分到不同的服务版本或配置中。网格将一些流量引导到更新后的版本，从而以受控方式逐步推出变更。这样可以实现平稳过渡，并最大限度地降低变更的影响。
      </li>
      <li>
       <strong>
        <strong>
         安全性
        </strong>
       </strong>
       <br/>
       服务网格提供安全通信功能，例如双向 TLS加密、身份验证和授权。
      </li>
      <li>
       <strong>
        <strong>
         监控
        </strong>
       </strong>
       <br/>
       服务网格提供全面的监控和可观测性功能，可深入了解服务的运行状况、性能和行为。监控还支持故障排除和性能优化。
      </li>
     </ul>
    </blockquote>
    <h4 id="4.7.3%20%E6%9C%8D%E5%8A%A1%E7%BD%91%E6%A0%BC%E6%9E%B6%E6%9E%84">
     4.7.3 服务网格架构
    </h4>
    <p>
     <strong>
      服务网格架构中有两个主要组成部分：控制面板和数据面板。
     </strong>
    </p>
    <h5 id="%E6%95%B0%E6%8D%AE%E9%9D%A2%E6%9D%BF">
     <strong>
      数据面板
     </strong>
    </h5>
    <blockquote>
     <p>
     </p>
     <p>
      <span style="color:#0d0016">
       数据面板是服务网格的数据处理组件。它包括所有 sidecar 代理及其功能。当一个服务想要与其他服务通信时，sidecar 代理会采取以下操作：
      </span>
     </p>
     <ol>
      <li>
       <span style="color:#fe2c24">
        sidecar 拦截请求
       </span>
      </li>
      <li>
       <span style="color:#fe2c24">
        它将请求封装在单独的网络连接中
       </span>
      </li>
      <li>
       <span style="color:#fe2c24">
        它在源代理和目标代理之间建立安全的加密通道
       </span>
      </li>
     </ol>
     <p>
      <span style="color:#0d0016">
       sidecar 代理处理服务之间的低级消息传递。它们还会实施断路和请求重试等功能，以增强弹性并防止服务降级。服务网格功能（例如负载均衡、服务发现和流量路由）在数据面板中实现。
      </span>
     </p>
    </blockquote>
    <h5 id="%E6%8E%A7%E5%88%B6%E9%9D%A2%E6%9D%BF">
     控制面板
    </h5>
    <blockquote>
     <ul>
      <li>
       <span style="color:#0d0016">
        控制面板是服务网格的中央管理和配置层。
       </span>
      </li>
      <li>
       <span style="color:#0d0016">
        管理员可以通过控制面板在网格内定义和配置服务。例如，指定服务端点、路由规则、负载均衡策略和安全设置等参数。定义配置后，控制面板将必要信息
       </span>
       <span style="color:#fe2c24">
        分发到服务网格的数据面板。
       </span>
      </li>
      <li>
       <span style="color:#fe2c24">
        代理使用配置信息来决定如何处理传入的请求
       </span>
       <span style="color:#0d0016">
        。它们还可以接收配置更改并
       </span>
       <span style="color:#fe2c24">
        动态调整
       </span>
       <span style="color:#0d0016">
        其行为而无需重新启动或中断服务。
       </span>
      </li>
      <li>
       <span style="color:#0d0016">
        控制面板通常包括以下功能：
       </span>
       <ol>
        <li>
         <span style="color:#0d0016">
          跟踪网格内所有服务的服务注册表
         </span>
        </li>
        <li>
         <span style="color:#0d0016">
          自动发现新服务并删除非活动服务
         </span>
        </li>
        <li>
         <span style="color:#0d0016">
          收集和聚合遥测数据，例如指标、日志和分布式跟踪信息
         </span>
        </li>
       </ol>
      </li>
     </ul>
    </blockquote>
    <h4 id="4.7.4%20%E6%9C%8D%E5%8A%A1%E7%BD%91%E6%A0%BC%E5%92%8Ck8s">
     4.7.4 服务网格和k8s
    </h4>
    <blockquote>
     <p>
      <span style="color:#0d0016">
       k8s“服务”资源是简化的service mesh，它提供服务发现和请求的轮询调度均衡。完整的service mesh则提供更丰富的功能，如管理安全策略和加密、“断路”以暂停对缓慢响应的实例的请求以及如上所述的负载均衡等。 服务网格本质上是微服务治理，把服务治理，服务通讯，服务安全，服务监控等逻辑
       <strong>
        从业务逻辑代码中提取出来形成代理并下沉到istio等基础设施中，如下图：
       </strong>
      </span>
     </p>
     <p class="img-center">
      <img alt="" height="250" src="https://i-blog.csdnimg.cn/direct/b605a1261ce94c08b1b0086c77a440ae.png" width="516"/>
     </p>
    </blockquote>
    <h4 id="4.7.5%20%E6%9C%8D%E5%8A%A1%E7%BD%91%E6%A0%BC%E9%9D%A2%E4%B8%B4%E7%9A%84%E6%8C%91%E6%88%98">
     4.7.5 服务网格面临的挑战
    </h4>
    <blockquote>
     <ul>
      <li>
       <span style="color:#0d0016">
        <strong>
         复杂性
        </strong>
       </span>
       <br/>
       <span style="color:#0d0016">
        服务网格引入了其他基础设施组件、配置要求和部署注意事项，有一定的学习难度。
       </span>
      </li>
      <li>
       <span style="color:#0d0016">
        <strong>
         运维管理费用
        </strong>
       </span>
       <br/>
       <span style="color:#0d0016">
        服务网格会带来部署、管理和监控数据面板代理和控制面板组件的额外开销。例如：
       </span>
       <ul>
        <li>
         <span style="color:#0d0016">
          确保服务网格基础设施的高可用性和可扩展性
         </span>
        </li>
        <li>
         <span style="color:#0d0016">
          监控代理的运行状况和性能
         </span>
        </li>
        <li>
         <span style="color:#0d0016">
          处理升级和兼容性问题
         </span>
        </li>
        <li>
         <span style="color:#0d0016">
          必须仔细设计和配置服务网格，以最大限度地减少对整个系统的性能影响。
         </span>
        </li>
       </ul>
      </li>
      <li>
       <span style="color:#0d0016">
        <strong>
         集成挑战
        </strong>
