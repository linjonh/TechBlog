     安装完 Git 之后，要做的第一件事就是设置自己的用户名和邮件地址。因为通过 Git 对项目进行版本管理的时候，Git 需要使用这些基本信息，来记录是谁对项目进行了操作：
    </p>
    <p>
     git config --global user.name “username” # 你的用户名
    </p>
    <p>
     git config --global user.email “email@example.com” # 你的邮箱

     😆温馨提醒😆：如果使用了
     <code>
      --global
     </code>
     选项，那么该命令只需要运行一次，即可永久生效；

    <h5>
     <a id="_12_Git__61">
     </a>
     <a href="" rel="nofollow">
     </a>
     1.2 Git 的全局配置文件
    </h5>
    <p>
     通过
     <code>
      git config --global user.name
     </code>
     和
     <code>
      git config --global user.email
     </code>
     配置的用户名和邮箱地址，会被写入到 C:/Users/用户名文件夹/.gitconfig 文件中。这个文件是 Git 的全局配置文件，配置一次即可永久生效。
    </p>

     <a href="" rel="nofollow">

    <p>
     除了使用记事本查看全局的配置信息之外，还可以运行如下的终端命令，快速的查看 Git 的全局配置信息：
    </p>

     <a id="_79">
     </a>
     查看所有的全局配置项
    </h2>
    <p>
     git config --list --global
    </p>
    <h2>
     <a id="_85">
     </a>
     查看指定的全局配置项
    </h2>
    <p>
     git config user.name
    </p>
    <p>
     git config user.email
    </p>

    <hr/>
    <h4>
     <a id="_2Git__137">
     </a>
     <a href="" rel="nofollow">
     </a>
     （2）Git 的基本操作
    </h4>
    <h5>
     <a id="_21__Git__141">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.1 获取 Git 仓库的两种方式
    </h5>
    <p>
     1️⃣ 将尚未进行版本控制的本地目录转换为 Git 仓库；
    </p>
    <p>
     2️⃣ 从其它服务器克隆一个已存在的 Git 仓库；
    </p>
    <h5>
     <a id="_22__153">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.2 在现有目录中初始化仓库
    </h5>
    <p>
     如果自己有一个尚未进行版本控制的项目目录，想要用 Git 来控制它，需要执行如下两个步骤：
    </p>
    <ol>
     <li>
      <p>
       在项目目录中，通过鼠标右键打开
       <code>
        “Git Bash”
       </code>
       ；
      </p>
     </li>
     <li>
      <p>
       执行
       <code>
        git init
       </code>
       命令将当前的目录转化为 Git 仓库；
      </p>
     </li>
    </ol>
    <p>
     <code>
      git init
     </code>
     命令会创建一个名为
     <code>
      .git
     </code>
     的隐藏目录，这个
     <code>
      .git
     </code>
     目录就是当前项目的 Git 仓库，里面包含了初始的必要文件，这些文件是 Git 仓库的必要组成部分。
    </p>
    <h5>
     <a id="_23__4__171">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.3 工作区中文件的 4 种状态
    </h5>
    <p>
     工作区中的每一个文件可能有 4 种状态，这四种状态共分为两大类，如图所示：
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/a1ff1ef8b213699fc1e9cd7853a36935.png#pic_center"/>
    </p>
    <p>
     😆Git 操作的终极结果😆：让工作区中的文件都处于“未修改”的状态；
    </p>
    <h5>
     <a id="_24__185">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.4 检查文件的状态
    </h5>
    <p>
     可以使用
     <code>
      git status
     </code>
     命令查看文件处于什么状态；
    </p>
    <h2>
     <a id="_195">
     </a>
     查看指定文件状态
    </h2>
    <p>
     git status [file-name]
    </p>
    <h2>
     <a id="_201">
     </a>
     查看所有文件状态
    </h2>
    <p>
     git status
    </p>
    <p>
     1️⃣ 如果在状态报告中可以看到有文件出现在
     <code>
      Untracked files
     </code>
     （未跟踪的文件）下面，则说明该文件为未跟踪的文件；
    </p>
    <p>
     2️⃣ 未跟踪的文件意味着 Git 在之前的快照（提交）中没有这些文件；Git 不会自动将之纳入跟踪范围，除非明确地告诉它“我需要使用 Git 跟踪管理该文件”；
    </p>
    <h5>
     <a id="_25__219">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.5 以精简的方式显示文件状态
    </h5>
    <p>
     使用
     <code>
      git status
     </code>
     输出的状态报告很详细，但有些繁琐。如果希望以精简的方式显示文件的状态，可以使用如下两条完全等价的命令，其中
     <code>
      -s
     </code>
     是 --short 的简写形式：
    </p>
    <h2>
     <a id="_229">
     </a>
     以精简的方式显示文件的状态
    </h2>
    <p>
     git status -s
    </p>
    <p>
     git status --short
    </p>
    <p>
     😆
     <strong>
      温馨提醒
     </strong>
     😆：
     <strong>
      未跟踪文件前面有红色的 ?? 标记；
     </strong>
    </p>
    <h5>
     <a id="_26__245">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.6 跟踪新文件
    </h5>
    <p>
     使用命令
     <code>
      git add
     </code>
     开始跟踪一个文件。例如要跟踪 index.html 文件，运行如下的命令即可：
    </p>
    <p>
     git add index.html
    </p>
    <p>
     此时再运行
     <code>
      git status
     </code>
     命令，会看到 index.html 文件在
     <code>
      Changes to be committed
     </code>
     这行的下面，说明已被跟踪，并处于暂存状态；
    </p>
    <p>
     😆
     <strong>
      以精简的方式显示文件的状态
     </strong>
     😆：
     <strong>
      新添加到暂存区中的文件前面有绿色的 A 标记；
     </strong>
    </p>
    <h5>
     <a id="_27__271">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.7 提交更新
    </h5>
    <p>
     现在假设暂存区中有一个 index.html 文件等待被提交到 Git 仓库中进行保存。可以执行
     <code>
      git commit
     </code>
     命令进行提交,其中
     <code>
      -m
     </code>
     选项后面是本次的提交消息，用来对提交的内容做进一步的描述：
    </p>
    <p>
     git commit -m “新建的index.html文件”
    </p>
    <p>
     提交成功之后，再次检查文件的状态，得到提示如下：
    </p>
    <p>
     <img alt="" src="https://i-blog.csdnimg.cn/blog_migrate/aa5f560d00b0a0e5e08f180d6543e449.png#pic_center"/>
    </p>
    <p>
     证明工作区中所有的文件都处于“未修改”的状态，没有任何文件需要被提交；
    </p>
    <p>
     <img alt="" src="https://i-blog.csdnimg.cn/blog_migrate/f7799c008b505a2f33b5e0328431eaf8.png#pic_center"/>
    </p>
    <h5>
     <a id="_28__301">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.8 对已提交的文件进行修改
    </h5>
    <p>
     目前，index.html 文件已经被 Git 跟踪，并且工作区和 Git 仓库中的 index.html 文件内容保持一致。当我们修改了工作区中 index.html 的内容之后，再次运行
     <code>
      git status
     </code>
     和
     <code>
      git status -s
     </code>
     命令，会看到如下图片所示：
    </p>
    <p>
     <img alt="" src="https://i-blog.csdnimg.cn/blog_migrate/c29da1075df16b2538548a510d2a7245.png#pic_center"/>
    </p>
    <p>
     文件 index.html 出现在
     <code>
      Changes not staged for commit
     </code>
     这行下面，说明已跟踪文件的内容发生了变化，但还没有放到暂存区；
    </p>
    <p>
     😆温馨提醒😆：
     <strong>
      修改过的、没有放入暂存区的文件前面有红色的 M 标记；
     </strong>
    </p>
    <h5>
     <a id="_29__317">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.9 暂存已修改的文件
    </h5>
    <p>
     目前，工作区中的 index.html 文件已被修改，如果要暂存这次修改，需要再次运行
     <code>
      git add
     </code>
     命令，这个命令是个多功能的命令，主要有如下 3 个功效：
    </p>
    <ol>
     <li>
      <p>
       可以用它开始跟踪新文件；
      </p>
     </li>
     <li>
      <p>
       把已跟踪的、且已修改的文件放到暂存区；
      </p>
     </li>
     <li>
      <p>
       把有冲突的文件标记为已解决状态；
      </p>
     </li>
    </ol>
    <p>
     <img alt="" src="https://i-blog.csdnimg.cn/blog_migrate/3d10b2ee4845b26a285f9d0640ec8fdf.png#pic_center"/>
    </p>
    <h5>
     <a id="_210__337">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.10 提交已暂存的文件
    </h5>
    <p>
     再次运行 git commit -m “提交消息” 命令，即可将暂存区中记录的 index.html 的快照，提交到 Git 仓库中进行保存：
    </p>
    <p>
     <img alt="" src="https://i-blog.csdnimg.cn/blog_migrate/d515b1c6ebff27fb6b42b667bf481a01.png#pic_center"/>
    </p>
    <p>
     <img alt="" src="https://i-blog.csdnimg.cn/blog_migrate/598264f073dcbe36d388699767b3f7e7.png#pic_center"/>
    </p>
    <h5>
     <a id="_211__353">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.11 撤销对文件的修改
    </h5>
    <p>
     撤销对文件的修改指的是：把对工作区中对应文件的修改，还原成 Git 仓库中所保存的版本。
    </p>
    <p>
     操作的结果：所有的修改会丢失，且无法恢复！危险性比较高，请慎重操作！
    </p>
    <p>
     <img alt="" src="https://i-blog.csdnimg.cn/blog_migrate/edbecdb9417c7430d670e6386c3e26fd.png#pic_center"/>
    </p>
    <p>
     😆撤销操作的本质😆：用 Git 仓库中保存的文件，覆盖工作区中指定的文件；
    </p>
    <h5>
     <a id="_212__371">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.12 向暂存区中一次性添加多个文件
    </h5>
    <p>
     如果需要被暂存的文件个数比较多，可以使用如下的命令，一次性将所有的新增和修改过的文件加入暂存区；
    </p>
    <p>
     git add
    </p>
    <p>
     😆温馨提醒😆：今后在项目开发中，会经常使用这个命令，将新增和修改过后的文件加入暂存区；
    </p>
    <h5>
     <a id="_213__393">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.13 取消暂存的文件
    </h5>
    <p>
     如果需要从暂存区中移除对应的文件，可以使用如下的命令：
    </p>
    <p>
     git reset HEAD [file-name]
    </p>
    <h5>
     <a id="_214__411">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.14 跳过使用暂存区域
    </h5>
    <p>
     Git 标准的工作流程是工作区 → 暂存区 → Git 仓库，但有时候这么做略显繁琐，此时可以跳过暂存区，直接将工作区中的修改提交到 Git 仓库，这时候 Git 工作的流程简化为了工作区 → Git 仓库。
    </p>
    <p>
     Git 提供了一个跳过使用暂存区域的方式， 只要在提交的时候，给
     <code>
      git commit
     </code>
     加上
     <code>
      -a
     </code>
     选项，Git 就会自动把所有已经跟踪过的文件暂存起来一并提交，从而跳过
     <code>
      git add
     </code>
     步骤：
    </p>
    <h2>
     <a id="_423">
     </a>
     将工作区的内容直接加入本地仓库
    </h2>
    <p>
     git commit -a -m “描述消息”
    </p>
    <h5>
     <a id="_215__433">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.15 移除文件
    </h5>
    <p>
     从 Git 仓库中移除文件的方式有两种：
    </p>
    <ol>
     <li>
      <p>
       从 Git 仓库和工作区中同时移除对应的文件；
      </p>
     </li>
     <li>
      <p>
       只从 Git 仓库中移除指定的文件，但保留工作区中对应的文件；
      </p>
     </li>
    </ol>
    <h2>
     <a id="_Git__449">
     </a>
     从 Git 仓库和工作区中同时移除对应的文件
    </h2>
    <p>
     git rm -f [file-name]
    </p>
    <h2>
     <a id="_Git__455">
     </a>
     只从 Git 仓库中移除指定的文件，但保留工作区中对应的文件
    </h2>
    <p>
     git rm --cached [file-name]
    </p>
    <h5>
     <a id="_216__465">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.16 忽略文件
    </h5>
    <p>
     一般我们总会有些文件无需纳入 Git 的管理，也不希望它们总出现在未跟踪文件列表。 在这种情况下，我们可以创建一个名为
     <code>
      .gitignore
     </code>
     的配置文件，列出要忽略的文件的匹配模式。
    </p>
    <p>
     文件
     <code>
      .gitignore
     </code>
     的格式规范如下：
    </p>
    <ol>
     <li>
      <p>
       以 # 开头的是注释；
      </p>
     </li>
     <li>
      <p>
       以 / 结尾的是目录；
      </p>
     </li>
     <li>
      <p>
       以 / 开头防止递归；
      </p>
     </li>
     <li>
      <p>
       以 ! 开头表示取反；
      </p>
     </li>
     <li>
      <p>
       可以使用 glob 模式进行文件和文件夹的匹配（glob 指简化了的正则表达式）；
      </p>
     </li>
    </ol>
    <h5>
     <a id="_217_glob__489">
     </a>
     <a href="" rel="nofollow">
     </a>
     2.17 glob 模式
    </h5>
    <p>
     所谓的 glob 模式是指简化了的正则表达式：
    </p>
    <ol>
     <li>
      <p>
       <code>
        星号 *
       </code>
       匹配零个或多个任意字符；
      </p>
     </li>
     <li>
      <p>
       <code>
        [abc]
       </code>
       匹配任何一个列在方括号中的字符 （此案例匹配一个 a 或匹配一个 b 或匹配一个 c）；
      </p>
     </li>
     <li>
      <p>
       <code>
        问号 ?
       </code>
       只匹配一个任意字符；
      </p>
     </li>
     <li>
      <p>
       在方括号中使用
       <code>
        短划线
       </code>
       分隔两个字符， 表示所有在这两个字符范围内的都可以匹配（比如 [0-9] 表示匹配所有 0 到 9 的数字）；
      </p>
     </li>
     <li>
      <p>
       <code>
        两个星号 **
       </code>
       表示匹配任意中间目录（比如 a/**/z 可以匹配 a/z 、 a/b/z 或 a/b/c/z 等）；
      </p>
     </li>
    </ol>

    <p>
     <a href="" rel="nofollow">
     </a>
     三、Git 分支
    </p>
    <hr/>
    <p>
     在进行多人协作开发的时候，为了防止互相干扰，提高协同开发的体验，所以每个开发者都基于分支进行项目功能的开发。
    </p>
    <h4>
     <a id="_1_565">
     </a>
     <a href="" rel="nofollow">
     </a>
     （1）本地分支操作
    </h4>
    <h5>
     <a id="_11_master__569">
     </a>
     <a href="" rel="nofollow">
     </a>
     1.1 master 主分支
    </h5>
    <p>
     1️⃣ 在初始化本地 Git 仓库的时候，Git 默认已经帮我们创建了一个名字叫做
     <code>
      master
     </code>
     的分支。通常我们把这个
     <code>
      master
     </code>
     分支叫做主分支；
    </p>
    <p>
     2️⃣ 在实际工作中，master 主分支的作用是：
     <strong>
      用来保存和记录整个项目已完成的功能代码
     </strong>
     ；
    </p>
    <p>
     3️⃣ 因此，不允许程序员直接在
     <code>
      master
     </code>
     分支上修改代码，因为这样做的风险太高，容易导致整个项目崩溃；
    </p>
    <h5>
     <a id="_12__585">
     </a>
     <a href="" rel="nofollow">
     </a>
     1.2 功能分支
    </h5>
    <p>
     由于程序员不能直接在 master 分支上进行功能的开发，所以就有了功能分支的概念。功能分支指的是专门用来开发新功能的分支，它是临时从 master 主分支上分叉出来的，当新功能开发且测试完毕后，最终需要合并到 master 主分支上；
    </p>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/56e2bb9bb12d7154433db876753fd348.png#pic_center"/>
    </p>

     <a id="_13__597">
     </a>
     <a href="" rel="nofollow">
     </a>
     1.3 查看分支列表
    </h5>
    <p>
     使用如下的命令，可以查看当前 Git 仓库中所有的分支列表：
    </p>
    <h2>
     <a id="_607">
     </a>
     列出所有本地分支
    </h2>
    <p>
     git branch
    </p>
    <p>
     😆温馨提醒😆：
     <strong>
      分支名字前面的 * 号表示当前所处的分支；
     </strong>
    </p>

    </h2>

     1.5 切换分支
    </h5>
    <p>
     使用如下的命令，可以切换到指定的分支上进行开发：
    </p>
    <p>
     git checkout [branchname]
    </p>

     <a id="_16__659">
     </a>

    </h5>

     <a id="_669">
     </a>

    </h2>
    <p>

    </p>
    <h5>
     <a id="_17__679">
     </a>
     <a href="" rel="nofollow">
     </a>
     1.7 合并分支
    </h5>
    <p>
     功能分支的代码开发测试完毕之后，可以使用如下的命令，将完成后的代码合并到 master 主分支上：
    </p>
    <h2>
     <a id="_689">
     </a>
     切换到主分支
    </h2>
    <p>
     git checkout master
    </p>
    <h2>
     <a id="_695">
     </a>
     合并指定分支到当前分支
    </h2>
    <p>
     git merge [branchname]
    </p>
    <p>
     😆温馨提醒😆：假设要把 C 分支的代码合并到 A 分支，则必须先切换到 A 分支上，再运行
     <code>
      git merge
     </code>
     命令，来合并 C 分支；
    </p>
    <h5>
     <a id="_18__709">
     </a>
     <a href="" rel="nofollow">
     </a>
     1.8 删除分支
    </h5>
    <p>
     当把功能分支的代码合并到 master 主分支上以后，就可以使用如下的命令，删除对应的功能分支：
    </p>
    <h2>
     <a id="_719">
     </a>
     删除分支
    </h2>
    <p>
     git branch -d [branchname]
    </p>
