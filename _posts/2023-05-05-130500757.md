    <p>
     引言：
    </p>
    <blockquote>
     <p>
      随着docker的技术普及，越来越多的企业加入了云计算发展进程，云原生产业发展迅猛，云原生建设投入比例明显，面对大规模的集群投入、部署、维护等问题也逐渐产生，越来越多的企业对云原生不断提出更高要求，同时云原生技术简化运维的效能提 升开始显著，但深化应用后的测试投入开始攀升。
     </p>
    </blockquote>
    <p>
     如今，云原生技术和理念正在不断外延和丰富，越来越多的企业正在基于云原生技术，构建适应业务快速发展的技术架构和绿色可持续发展模式。在此背景下，腾讯云推出了国内首个基于云原生技术的降本增效开源平台——FinOps Crane。
    </p>
    <p>
     FinOps定义了一系列云财务管理规则和最佳实践，强调通过数据驱动的方式，进行成本决策，使组织能够获得最大收益；腾讯云基于 FinOps 打造云资源分析与成本优化平台Crane，Crane 支持多云成本观测，优化预期测算，浪费看板以及资源优化能力，帮助分布式云架构下的云客户降本。
    </p>
    <p>
     为推进云原生用户在确保业务稳定性的基础上做到真正的极致降本，腾讯推出了国内第一个基于云原生技术的成本优化开源项目 Crane（ Cloud Resource Analytics and Economics ）。
    </p>
    <p>
     <mark>
      当前 Crane 项目主要贡献者包括有腾讯、小红书、谷歌、eBay、微软、特斯拉等知名公司的行业专家。
     </mark>
    </p>
    <hr/>

      <mark>
       国内首个获得该国家级奖项的云原生工具

    <pre><code class="prism language-js"><span class="token number">3.</span> 提供一组 Exporter 计算集群云资源的计费和账单数据并存储到你的监控系统，比如 Prometheus。

    <ul>
     <li>
      推荐框架
     </li>
    </ul>
    <pre><code class="prism language-js"><span class="token number">1.</span> 提供了一个可扩展的推荐框架以支持多种云资源的分析
<span class="token number">2.</span> 内置了多种推荐器：资源推荐，副本推荐，<span class="token constant">HPA</span> 推荐，闲置资源推荐。
</code></pre>
    <ul>
     <li>
      基于预测的水平弹性器
     </li>
    </ul>
    <pre><code class="prism language-js"><span class="token number">1.</span> EffectiveHorizontalPodAutoscaler 支持了预测驱动的弹性。
<span class="token number">2.</span> 它基于社区 <span class="token constant">HPA</span> 做底层的弹性控制，支持更丰富的弹性触发策略（预测，观测，周期），让弹性更加高效，并保障了服务的质量。
</code></pre>
    <ul>
     <li>
      负载感知的调度器
     </li>
    </ul>
    <pre><code class="prism language-js"><span class="token number">1.</span> 动态调度器根据实际的节点利用率构建了一个简单但高效的模型，并过滤掉那些负载高的节点来平衡集群。
</code></pre>
    <ul>
     <li>
      拓扑感知的调度器
     </li>
    </ul>
    <pre><code class="prism language-js"><span class="token number">1.</span> Crane Scheduler与Crane Agent配合工作，支持更为精细化的资源拓扑感知调度和多种绑核策略。
<span class="token number">2.</span> 可解决复杂场景下“吵闹的邻居问题"，使得资源得到更合理高效的利用。
</code></pre>
    <ul>
     <li>
      基于 QOS 的混部
     </li>
    </ul>
    <pre><code class="prism language-js"><span class="token number">1.</span> <span class="token constant">QOS</span>相关能力保证了运行在 Kubernetes 上的 Pod 的稳定性。
<span class="token number">2.</span> 具有多维指标条件下的干扰检测和主动回避能力，支持精确操作和自定义指标接入。
<span class="token number">3.</span> 具有预测算法增强的弹性资源超卖能力，复用和限制集群内的空闲资源。
<span class="token number">4.</span> 具备增强的旁路cpuset管理能力，在绑核的同时提升资源利用效率。
</code></pre>
    <ol start="4">
     <li>
      Crane有什么优势：
     </li>
    </ol>
    <blockquote>
     <p>
      Crane 基于两级调度能力，实现高优先级延迟敏感业务和低优先级高吞吐业务在相同节点上的混合运行：
     </p>
    </blockquote>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/0b8de5e5edf77d1810db9965142ac3a9.png"/>
    </p>
    <p>
     1.一级调度能力，确保应用的高效调度，实现真正的“用多少占多少”。
    </p>
    <ul>
     <li>
      <p>
       基于应用历史负载信息进行画像构建
      </p>
     </li>
     <li>
      <p>
       基于应用画像和节点画像实现智能调度
      </p>
     </li>
     <li>
      <p>
       基于DSP算法、AI算法等实现弹性预测
      </p>
     </li>
    </ul>
    <p>
     2.二级调度能力，在保障服务质量的同时，实现资源利用率的大幅提升。
    </p>
    <ul>
     <li>
      <p>
       节点画像和闲置资源回收。运行在每个节点的代理采集节点负载，基于 DSP等预测算法预测未来负载走势，并将闲置资源回收成为节点扩展资源供低优业务使用。
      </p>
     </li>
     <li>
      <p>
       资源隔离和服务质量保障。定义资源隔离规则，确保当混部业务发生资源竞争时，高优业务的稳定性不受影响；开源方案基于 CPU Quota 完成低优业务的资源压制；闭源方案基于腾讯 TLinux 如意内核完成高优业务的绝对资源抢占。
      </p>
     </li>
     <li>
      <p>
       干扰检测和低优主动回避。节点代理探测资源隔离策略生效后，节点是否依然有干扰发生，如有干扰则驱逐低优业务确保高优业务不受影响。
      </p>
     </li>
    </ul>
    <ol start="5">
     <li>
      Crane的战绩：
     </li>
    </ol>
    <p>
     目前，Crane 已经被
     <mark>
      腾讯、小红书、网易、思必驰、酷家乐、明源云、数数科技
     </mark>
     等公司部署在生产系统，其主要贡献者来自腾讯、小红书、谷歌、eBay、微软、特斯拉等知名公司。
    </p>
    <ul>
     <li>
      <p>
       在腾讯内部自研业务大规模落地，部署至数百个 Kubernetes 集群，管控数百万 CPU 核，全面上线一个月内，大盘总核数缩减 25%。
      </p>
     </li>
     <li>
      <p>
       以腾讯内部部门集群优化为例，通过使用FinOps Crane，该部门在保障业务稳定的情况下,资源利用率提升了3倍；腾讯另一自研业务落地FinOps后，在一个月内实现了总CPU规模40万核的节省量，相当于每月成本节约超千万元。
      </p>
     </li>
     <li>
      <p>
       只利用Crane的Request推荐等基本能力，设备资源利用率从原来的不到10%提升了现在的16.6%，从而使得成本整体降低了30%。
      </p>
     </li>
    </ul>

<p> 6.如何计算成本：
<br/>
成本计算功能是由组件 Fadvisor 实现，在安装 Crane 时会一起安装，一起提供了成本展示和成本分析功能：
</p>
<ul>
<li>
Server：收集集群 Metric 数据并计算成本
</li>
<li>
Exporter：将成本 Metric 暴露出来
</li>
</ul>
<p>
<img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/7e1ab02d34b3209bbfebd792c165ce51.png"/>
<br/>
原理
</p>
<p>
Fadvisor 成本模型提供了一个方法来估计和分析每个容器,pod 或其他资源在 Kubernetes 中的资源价格。
</p>
<p>
请注意,成本模型只是预估成本,而不是替代云订单,因为实际的计费数据取决于更多原因，比如各类计费逻辑。以下是计算理论：
</p>
<ul>
<li>
最简单的成本模型是以相同的价格估算所有节点或 pod 的资源价格。例如,在计算成本时,您可以假设所有容器的 CPU 和 RAM 单位价格相同,2/小时核心，0.3/小时 Gib
</li>
<li>
高级成本模型是通过成本分摊来估计资源价格。 这一理论的基础是不同实例类型和计费类型的每个云机器实例的价格不同，不过 CPU 和 RAM 的价格比率是相对固定的，可以通过这个价格比率来计算资源成本。
</li>
</ul>
<p>
成本分摊模型下的具体的计算公式如下：
</p>
<ul>
<li>
集群整体成本：cvm 成本之和
</li>
<li>
CPU/mem 价格比率相对固定
</li>
<li>
cvm 的成本 = CPU 成本 _ CPU 数量 + mem 成本 _ mem 数量
</li>
<li>
CPU 申请成本：整体成本 \* （CPU 占 cvm 成本的比例）得到整体 CPU 成本，再按申请的 CPU 总览占整体 CPU 总量的比例计算出 CPU 申请的成本
</li>
<li>
namespace 下的 CPU 申请成本： CPU 申请成本按 namespace 聚合
</li>
</ul>
