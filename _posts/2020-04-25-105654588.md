    <h2>
     <a id="_11">
     </a>
     数据库管理系统的主要功能是什么？
    </h2>
    <p>
     答：
     <br/>
     数据
     <mark>
      定义
     </mark>
     、数据
     <mark>
      操纵
     </mark>
     、数据库的
     <mark>
      运行管理
     </mark>
     、数据库
     <mark>
      建立和维护
     </mark>
    </p>
    <h2>
     <a id="_14">
     </a>
     数据库系统的特点是什么？
    </h2>
    <p>
     答：
     <br/>
     数据
     <mark>
      结构化
     </mark>
     ；
     <br/>
     数据
     <mark>
      共享性高、冗余度低
     </mark>
     且扩充；
     <br/>
     数据
     <mark>
      独立性高
     </mark>
     ；
     <br/>
     数据由
     <mark>
      DBMS统一管理
     </mark>
     。
    </p>
    <h2>
     <a id="DBA_20">
     </a>
     DBA的职责是什么？
    </h2>
    <p>
     答：
     <br/>
     DBA负责
     <strong>
      全面管理和控制数据库系统
     </strong>
     的职责，具体有：
     <br/>
     (1)决定数据库中要存放哪些信息；
     <br/>
     (2)决定数据库的存储结构和存取策略；
     <br/>
     (3)定义数据的安全性要求和完整性约束条件；
     <br/>
     (4)监控数据库的使用和运行：转储数据、维护日志文件、故障恢复
     <br/>
     (5)数据库的改进和重组重构：对运行情况进行记录、统计分析，以此来不断改进数据库设计。
    </p>

    <h2>
     <a id="_39">
     </a>
     三级模式结构是什么？二级映射有什么优点？
    </h2>
    <p>
     答：
     <br/>
     三级模式包括外模式、模式、内模式；
     <br/>
     <strong>
      模式
     </strong>
     描述了数据库中
     <mark>
      全体数据
     </mark>
     的逻辑结构和特征；
     <br/>
     <strong>
      外模式
     </strong>
     描述了
     <mark>
      用户可见的局部数据
     </mark>
     的逻辑结构和特征；
     <br/>
     <strong>
      内模式
     </strong>
     描述了数据库的
     <mark>
      物理结构和存储方式
     </mark>
     。
     <br/>
     <strong>
      外模式/模式映像的优点
     </strong>
     ：当模式改变时，由DBA对外模式/模式映像做相应的改变，外模式可以保持不变。应用程序是根据外模式而编写的，因而应用程序不必修改，保证了数据与逻辑的而独立性，即
     <mark>
      数据的逻辑独立性
     </mark>
     ；
     <br/>
     <strong>
      内模式/模式映像的优点
     </strong>
     ：当数据库的存储结构改变时，由DBA对模式/内模式映射作相应的改变，可以使模式保持不变，从而应用程序也不必改变。保证了数据与程序的物理独立性，即
     <mark>
      数据的物理独立性
     </mark>
     。
    </p>
    <h2>
     <a id="_48">
     </a>
     关系模型中的完整性约束是哪几类？定义？
    </h2>
    <p>
     答：
     <br/>
     实体完整性：关系模式中的
     <mark>
      主码不能为空值
     </mark>
     ；
     <br/>
     参照完整是：关系模式中的外码
     <mark>
      只能是空值或者另一关系模式的主码
     </mark>
     ；
     <br/>
     用户定义完整性：关系模式中
     <mark>
      针对某一属性的约束
     </mark>
     。
    </p>
    <h2>
     <a id="SQL_53">
     </a>
     SQL的特点？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      综合统一
     </strong>
     。集DCL，DML，DDL功能于一体；
     <br/>
     高度非过程化。使用SQL进行数据操作时，是需要提出“做什么”，而不需要指明怎么做；
     <br/>
     <strong>
      面向集合的操作方式
     </strong>
     。操作的对象、查找结果都可以是元组的集合；
     <br/>
     <strong>
      同一种语法结构提供多种使用方式
     </strong>
     。既可以作为独立的语言进行交互，又可以作为嵌入式语言嵌入到更高级的语言程序中进行操作；
     <br/>
     <strong>
      语言简洁，易学易用
     </strong>
     。
    </p>
    <h2>
     <a id="_61">
     </a>
     什么是数据库操纵语言、什么是数据库控制语言,试举例说明
    </h2>
    <p>
     答：
     <br/>
     <strong>
      数据操纵语言
     </strong>
     是指通过对表中的数据进行插入、删除和修改等操作，实现对数据库的基本操作。如：SELECT命令的作用是检查表的内容，UPDATE命令的作用是修改表等等。
     <br/>
     <strong>
      数据控制语言
     </strong>
     是用来设置或者更改数据库用户或角色权限的语句，这些语句包括GRANT、REVOKE等语句。
    </p>
    <h2>
     <a id="_66">
     </a>
     *等值连接和自然连接的区别是什么？
    </h2>
    <p>
     答：
     <br/>
     自然连接是等值连接的一种特殊情况；
     <br/>
     等值连接要求连接的是值相等的分量，两个关系中
     <mark>
      可以没有相同的属性
     </mark>
     ；进行自然连接的两个关系中必须有相同的属性。
     <br/>
     等值连接
     <mark>
      不要求去掉重复属性列
     </mark>
     ；自然连接时需要除掉重复的属性列。
    </p>
    <h2>
     <a id="_72">
     </a>
     *外连接是什么？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      外连接
     </strong>
     是指两个表在进行操作时，不仅返回符合连接和查询条件的元组，还返回不符合条件的一些元组；
     <br/>
     <strong>
      左外连接
     </strong>
     是指返回左表中仅符合连接条件不符合查询条件的元组；
     <br/>
     <strong>
      右外连接
     </strong>
     是指返回右表中仅符合连接条件不符合查询条件的元组；
     <br/>
     <strong>
      全外连接
     </strong>
     是左外连接和右外连接去掉重复项的元组集并集。
    </p>
    <h2>
     <a id="_79">
     </a>
     索引的作用？和它的优点缺点是什么？
    </h2>
    <p>
     答：
     <br/>
     当表的数据量比较大时，查询操作比较耗时，建立索引可以加快查询速度。
     <br/>
     <strong>
      优点
     </strong>
     ：加速查询速度；
     <strong>
      缺点
     </strong>
     ：索引需要占一定的存储空间，且基本表更新时需要维护索引表。
    </p>
    <h2>
     <a id="_83">
     </a>
     索引和键有什么区别？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      索引
     </strong>
     是存储在数据库中的一个物理结构，是实际存在的，
     <mark>
      相当于一本书的目录
     </mark>
     ；
     <br/>
     <strong>
      键
     </strong>
     是一个逻辑概念，不是数据库中的物理部分。键分为主键和外键，
     <mark>
      主键相当于一本书中每个页的页码
     </mark>
     。
     <br/>
     （扩展：聚簇索引：与基表的物理顺序相同，数据值的顺序总是按照顺序排列。）
    </p>
    <h2>
     <a id="_89">
     </a>
     数据库中常用的索引有哪些？
    </h2>
    <p>
     答：
     <br/>
     B+树索引：B+树是对B树优化处理了的多路搜索树，与B树的区别在于，B树的每个结点都可以存储数据，而
     <mark>
      B+树只有叶子结点可以存储数据，且每个叶子结点都用链表链接
     </mark>
     。
     <br/>
     在数据库的查询中一般都是
     <mark>
      批量查询数据
     </mark>
     ，采用B+树一次查询多条时，确定首尾位置，便可以方便的确定多条数据位置，提高查询效率。
     <br/>
     Hash索引：通过一定的算法
     <mark>
      计算出哈希值
     </mark>
     ，然后映射出对应的数据存储位置。检索速度快，但是哈希值可能存在碰撞。
    </p>
    <h2>
     <a id="_94">
     </a>
     视图是什么？有何作用？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      视图
     </strong>
     是从一个或几个基本表中导出的表，是一个虚表。数据库中只存放视图的定义，而不存放视图对应的数据。基本表中的数据发生变化时，从视图中查询出来的数据也就随之发生变化。
     <br/>
     <strong>
      作用
     </strong>
     ：
     <br/>
     能够
     <mark>
      简化用户的操作
     </mark>
     ；
     <br/>
     使用户能以
     <mark>
      多种角度看待同一数据
     </mark>
     ；
     <br/>
     在一定程度上提供了
     <mark>
      数据的逻辑独立性
     </mark>
     ；
     <br/>
     能够对秘密数据提供
     <mark>
      安全保护
     </mark>
     ；
     <br/>
     利用可以
     <mark>
      更清晰的表达查询
     </mark>
     。
    </p>
    <h2>
     <a id="_103">
     </a>
     数据库的安全性是指什么？有哪些安全性技术？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      数据库的安全性
     </strong>
     是指保护数据库以恶意破坏和非法存取。
     <br/>
     <strong>
      安全性技术
     </strong>
     ：用户标识和鉴别、多层存取控制、审计、视图、数据加密。
    </p>
    <h2>
     <a id="_108">
     </a>
     存取控制是指什么？主要包括哪两部分？有哪两类方法？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      存取控制
     </strong>
     是指确保只授权给有资格的用户访问数据库的权限，且令所有未被授权的人员无法接近数据。
     <br/>
     <strong>
      两个部分
     </strong>
     ：定义用户权限和合法权限检查；
     <br/>
     <strong>
      两种方法
     </strong>
     ：
     <br/>
     <mark>
      自主存取控制（DAC）
     </mark>
     ：用户对不同的数据库对象有不同的存取权限，不同的用户对同一对象也有不同的权限
     <br/>
     <mark>
      强制存取控制（MAC）
     </mark>
     ：每一个数据库对象被标以一定的密级，每一个用户也被授予一定级别的许可证，只有具有合法许可证的用户才可以进行存取。
    </p>
    <h2>
     <a id="_115">
     </a>
     视图机制是如何对数据库实现安全性控制的？
    </h2>
    <p>
     答：
     <br/>
     <mark>
      可以对不同的用户定义不同的视图
     </mark>
     ，也就是说，通过视图机制把要保密的数据对无权存取的用户隐藏起来。
    </p>
    <h2>
     <a id="_118">
     </a>
     审计功能是如何对数据库实现安全性控制的？
    </h2>
    <p>
     答：
     <strong>
      审计功能
     </strong>
     是把用户对数据库的一系列操作自动记录到审计日志中，审计员可以
     <mark>
      利用审计日志监控数据库中的各种行为
     </mark>
     ，找出导致数据库发生异常的事件。
    </p>
    <h2>
     <a id="_120">
     </a>
     数据库的完整性是指什么？
    </h2>
    <p>
     答：数据库的完整性是指防止数据库中存在不正确的数据。
    </p>
    <h2>
     <a id="_122">
     </a>
     什么是断言？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      断言
     </strong>
     是指
     <mark>
      更具有一般性的约束
     </mark>
     ，断言创建后，任何涉及到断言中的关系的操作都会引发数据库对断言的检查，任何
     <mark>
      使断言为假的操作都会被拒绝执行
     </mark>
     。
    </p>
    <h2>
     <a id="_125">
     </a>
     什么是触发器？触发器的作用？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      触发器
     </strong>
     是用户定义在关系表上的一类由数据驱动的一类由事物驱动的特殊过程，类似于约束，但是比约束更灵活，是保证数据库完整性的一种方法。任何用户对表进行
     <mark>
      增删改操作
     </mark>
     都会有
     <mark>
      数据库服务器
     </mark>
     自动激活相应的触发器，对数据库进行相应的检查和操作。
    </p>
    <h2>
     <a id="_129">
     </a>
     规范化过程中，逐级依次消除了何种函数依赖？
    </h2>
    <p>
     答：
     <br/>
     由INF到2NF，消除了非主属性对主属性的部分函数依赖；
     <br/>
     由2NF到3NF，消除了非主属性对主属性的传递函数依赖；
     <br/>
     由3NF到BCNF，消除了
     <mark>
      主属性对码的部分函数依赖和传递函数依赖
     </mark>
     。
    </p>
    <h2>
     <a id="_135">
     </a>
     数据库设计的基本步骤是什么？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      需求分析
     </strong>
     。了解和分析用户需求；
     <br/>
     <strong>
      概念结构设计
     </strong>
     。对用户需求进行抽象和归纳，形成一个独立于DBMS的概念模型（E-R图）；
     <br/>
     <strong>
      逻辑结构设计
     </strong>
     。将概念结构转换为数据模型，通常为关系模型；
     <br/>
     <strong>
      物理结构设计
     </strong>
     。为逻辑数据模型选取一个最适合存储结构和存取方法；
     <br/>
     <strong>
      数据库实施阶段
     </strong>
     。编写数据库，编写和调试应用程序；
     <br/>
     <strong>
      数据库运行和维护
     </strong>
     。正式投入运行。
    </p>
    <h2>
     <a id="_143">
     </a>
     概念模型有什么作用？
    </h2>
    <p>
     答：
     <br/>
     概念模型用于
     <mark>
      信息世界的建模
     </mark>
     ，是
     <mark>
      现实世界到信息世界
     </mark>
     的第一层抽象，是数据库设计人员进行数据库设计的有力工具，也是数据库设计人员和用户之间进行交流的语言。
    </p>
    <h2>
     <a id="ER_146">

    <p>

     <br/>

     </strong>
     。不同的实体对同一属性的定义不同；
     <br/>
     <strong>
      命名冲突
     </strong>
     。同名异义、异名同义；
     <br/>
     <strong>
      结构冲突
     </strong>
     。同一实体在不同的局部E-R图中有不同的抽象。
    </p>
    <h2>
     <a id="_151">
     </a>
     一个实体型转换为一个关系模式有哪几种情况？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      一个1:1联系
     </strong>
     可以转换为一个独立的关系模式，也可以与任意一端对应的关系模式合并；
     <br/>
     <strong>
      一个1：n联系
     </strong>
     可以转换为一个独立的关系模式，也可以与n端对应的关系模式合并；
     <br/>
     <strong>
      一个m：n联系
     </strong>
     可以转换为一个关系模式，与该联系相连的各实体的码以及联系本身的属性均转换为关系的属性。
    </p>
    <h2>
     <a id="SQL_156">

     </mark>

    </p>

     <mark>

     </mark>
     ，优点是可以建立非常复杂的查询，只需创建一次，在程序中即可多次调用，且比执行单纯的SQL语句要快。可以创建一个
     <mark>
      命令对象
     </mark>
     进行调用。
    </p>

     </mark>
     ，存放SQL语句的执行结果。
    </p>

     <mark>

     <br/>
     <mark>

     </mark>
     尽可能先做；
     <br/>
     <mark>
      投影操作
     </mark>
     尽可能先做；
     <br/>
     把
     <mark>
      笛卡尔积和附近的一连串选择和投影
     </mark>
     操作合并起来做。
    </p>
    <h2>
     <a id="ACID_175">
     </a>
     事物是什么？ACID特性包括？
    </h2>
    <p>
     答：
     <strong>
      事物
     </strong>
     是数据库进行操作的一个基本单位。
     <br/>
     <strong>
      ACID特性
     </strong>
     包括：
     <br/>
     隔离性：一个事务的执行不能被其他事务所干扰；
     <br/>
     原子性：事务是一个不可分割的单位，要么全做，要么全不做；
     <br/>
     一致性：事务执行的结果必须使数据库从一个一致性状态变到另一个一致性状态；
     <br/>
     永久性：一旦事务被提交，它对数据库的改变就是永久的。
    </p>
    <h2>
     <a id="_182">
     </a>
     为什么事务非正常结束时会影响数据库数据的正确性？
    </h2>
    <p>
     答：
     <br/>
     事物具有
     <mark>
      一致性
     </mark>
     ，事务执行的结果必须是使数据库从一个一致性状态变到另一个一致性状态。如果数据库系统运行中发生故障，有些事务尚未完成就被迫中断，这些未完成事务对数据库所做的修改有一部分已写入物理数据库，这时数据库就处于一种不正确的状态，或者说是不一致的状态。
    </p>
    <h2>
     <a id="_185">
     </a>
     数据库恢复的关键是什么？
    </h2>
    <p>
     答：关键是
     <mark>
      建立冗余数据
     </mark>
     ；
    </p>
    <h2>
     <a id="_188">
     </a>
     数据库恢复的基本技术有哪些？
    </h2>
    <p>
     答：数据转储和登录日志文件是数据库恢复的基本技术。
     <br/>
     <mark>
      数据转储
     </mark>
     是指DBA定期将数据库复制到其他介质上进行保存，这些备份的数据叫做后备副本；
     <br/>
     <mark>
      登记日志文件
     </mark>
     ，日志文件是用来记录事物对数据库进行更新操作的文件。
     <br/>
     当系统运行过程中发生故障，利用转储的数据库后备副本和日志文件就可以将数据库恢复到故障前的某个一致性状态。
    </p>
    <h2>
     <a id="_193">
     </a>
     登记日志文件时为什么必须先写日志文件，后写数据库？
    </h2>
    <p>
     答：
     <br/>
     把对数据的修改写到数据库中和把表示这个修改的日志记录写到日志文件中是两个不同的操作。有可能在这两个操作之间发生故障，即这两个写操作只完成了一个。
     <br/>
     如果
     <mark>
      先写了数据库修改，而在运行记录中没有登记这个修改，则以后就无法恢复这个修改了
     </mark>
     。
     <mark>
      如果先写日志，但没有修改数据库，在恢复时并不会影响数据库的正确性
     </mark>
     。所以一定要先写日志文件，即首先把日志记录写到日志文件中，然后写数据库的修改。
    </p>
    <h2>
     <a id="_197">
     </a>
     用于数据库恢复的文件是什么？
    </h2>
    <p>
     答：日志文件、后备副本。
    </p>
    <h2>
     <a id="_199">
     </a>
     *数据库故障的种类有哪几种？相应的恢复策略是什么？
    </h2>
    <p>
     答：三种：事物故障、系统故障、介质故障；
     <br/>
     事物故障是指
     <mark>
      事物在运行至终点前被中止
     </mark>
     ；
     <br/>
     事物故障恢复是由
     <mark>
      系统自动完成
     </mark>
     ，利用
     <mark>
      日志文件
     </mark>
     撤销此事务已对数据库进行的修改；
    </p>
    <p>
     系统故障是指
     <mark>
      造成系统停止运转的任何事件
     </mark>
     ，使得系统要重新启动；
     <br/>
     系统故障的恢复是在重新启动后
     <mark>
      系统自动完成
     </mark>
     ，为了防止系统故障造成的数据不一致性，必须
     <mark>
      撤销故障时未完成的事物，重做已完成的事物
     </mark>
     ；
    </p>
    <p>
     介质故障是指磁盘上的
     <mark>
      物理数据和日志文件被破坏
     </mark>
     ；
     <br/>
     介质故障的恢复方法就是
     <mark>
      重装数据库，重做已经完成的事物
     </mark>
     。
    </p>
    <h2>
     <a id="_209">
     </a>
     在数据库中为什么要并发控制？
    </h2>
    <p>
     答：
     <br/>
     数据库是共享资源，通常有多个事物在同时执行，当多个事物并发的存取数据库时就会存在同时读或写统一数据的情况，如果对并发操作不加控制，就会存在数据读取或存取错误，破坏数据库的一致性。
    </p>
    <h2>
     <a id="_212">
     </a>
     *并发造成的数据不一致性包括什么？
    </h2>
    <p>
     答：
     <br/>
     <strong>
      丢失修改
     </strong>
     ：A、B两个事物同时对一个数据进行操作，
     <mark>
      A的修改破坏了B的修改
     </mark>
     ，导致B的修改丢失；
     <br/>
     <strong>
      不可重复读
     </strong>
     ：A读取了数据后，B对数据进行了修改，导致
     <mark>
      A无法再现前一次
     </mark>
     的读取结果；
     <br/>
     <strong>
      读脏数据
     </strong>
     ：A对数据进行了修改之后，B读入该数据，然后A又因为某一原因撤销了该修改，
     <mark>
      导致数据恢复原值
     </mark>
     ，此时B读取的数据就成了脏数据。
    </p>
    <h2>
     <a id="_217">
     </a>
     *什么是锁？有哪两种锁？
    </h2>
    <p>
     答：
     <strong>
      锁
     </strong>
     是最常用的
     <mark>
      并发控制机构
     </mark>
     ，是防止其他事务访问指定资源，实现并发控制的一种手段。
     <br/>
     <strong>
      排他锁
     </strong>
     （X写锁）：当数据被加上写锁，其他事务不能对该数据进行读和写；
     <br/>
     <strong>
      共享锁
     </strong>
     （S读锁）：当数据被加上读锁，允许其他事务对该数据进行读，不允许写。
    </p>
    <h2>
     <a id="_222">
     </a>
     *简要概括一、二、三级封锁协议
    </h2>
    <p>
     答：
     <br/>
     一级封锁协议：事务在
     <mark>
      修改
     </mark>
     数据之前加
     <mark>
      写锁
     </mark>
     ，直到事务结束才释放。该协议可以
     <mark>
      防止丢失修改
     </mark>
     ；
     <br/>
     二级封锁协议：在一级封锁协议的基础上，加上了事务在读取数据之前对其加读锁，
     <mark>
      读完后即可释放读锁
     </mark>
     。该协议
     <mark>
      避免了读脏数据
     </mark>
     ；
     <br/>
     三级封锁协议：在一级封锁协议的基础上，加上了事务在读取数据之前必须加上读锁，
     <mark>
      直到事务结束才释放
     </mark>
     。该协议
     <mark>
      解决了不可重复读问题
     </mark>
     。
    </p>
    <h2>
     <a id="_227">
     </a>
     数据库保护（访问）的内容有哪些？
    </h2>
    <p>
     答：
     <br/>
     利用权限机制；利用完整性约束防止法数据进入数据库；提供故障恢复能力，提供并发访问控制。
