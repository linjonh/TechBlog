    <h3>
     <a id="1_2">

     1.简述
    </h3>
    <p>
     本文主要针对如何使用Junit编写单元测试进行描述
     <br/>
     文中的实例基于Junit 4
    </p>
    <pre><code>所谓单元测试，即是指针对程序中的一些单元进行测试的方法
这些单元在Junit中的最小单位为方法
借助单元测试，我们可以轻松地单独测试程序中的某一个逻辑片段而不需要在意程序的外部依赖和其它逻辑
</code></pre>
    <table>
     <thead>
      <tr>
       <th>
        接口测试
       </th>
       <th>
        单元测试
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        只能以接口为维度进行测试
       </td>
       <td>
        只需被测试的单元逻辑正常即可
       </td>
      </tr>
      <tr>
       <td>
        工程必须编译通过并打包进行部署
       </td>
       <td>
        可以不依赖外部，测试进度不再受制于外部条件
       </td>
      </tr>
      <tr>
       <td>
        工程的外部依赖（数据库、调用的服务等）必须就绪
       </td>
       <td>
        可以以方法为维度进行测试
       </td>
      </tr>
      <tr>
       <td>
        难以测试复杂的逻辑分支，为测试数据需要调整各个数据源（数据库、缓存、消息队列）
       </td>
       <td>
        可以根据单元的逻辑复杂程度编排测试用例数量，测试使用的数据可以自由调整
       </td>
      </tr>
     </tbody>
    </table>
    <h3>
     <a id="2_16">
     </a>
     2.创建测试用例
    </h3>
    <h4>
     <a id="21_17">
     </a>
     2.1工程准备
    </h4>
    <ol>
     <li>
      <p>
       确保工程的maven依赖中包含junit，版本至少为4.12，一般包含在spring-boot-starter-test中；
      </p>
     </li>
     <li>
      <p>
       确保工程的目录中包含src/test/java和src/test/resource；
      </p>
      <p>
       *部分工程没有src/test的相关包，需要手动创建
       <br/>
       *src/test/java主要用于存放测试用例和测试相关的类
       <br/>
       *src/test/resoutces主要用于存放测试使用的配置和资源文件
      </p>
     </li>
    </ol>
    <h4>
     <a id="22__27">
     </a>

    <h4>
     <a id="24__98">
     </a>
     2.4 使用测试专用的配置
    </h4>
    <p>
     就像在工程部署时我们可以指定不同环境的配置文件一样;
     <br/>
     执行测试时我们也可以使用不同的配置文件;
     <br/>
     使用自定义测试启动类的情况下将配置文件放置在src/test/resource下则测试类默认会使用该包下的配置和资源文件;
     <br/>
     由于测试启动类中通常会排除许多与测试目标无关的类，所以测试专用的配置文件可以更加轻量化，哪些被排除的类所引用的配置都可以省略;
    </p>
    <h4>
     <a id="25_Test_104">
     </a>
     2.5 新建Test类
    </h4>
    <pre><code>针对一个被测类创建一个测试类
测试类通常以被测类的名字+Test进行命名，并放置在与src/test/java下与被测类同包名的包中
</code></pre>
    <ol>
     <li>
      根据被测类创建测试类
     </li>
     <li>
      配置测试类的参数，通常这些参数会按照约定给出一些默认值；
     </li>
     <li>
      选择要测试的方法，注意：junit测试只能对非private方法进行测试DemoServiceTest，测试用例在src/test/java下的被测类同名包中；此时可以选择继承的父类，根据测试需要进行继承
     </li>
     <li>
      生成测试类，得到一个单元测试类的骨架，包含了选择的方法的测试用例
     </li>
    </ol>
    <h3>
     <a id="_112">
     </a>
     编写用例
    </h3>
    <h4>
     <a id="31_Mock_113">
     </a>
     3.1 了解Mock测试
    </h4>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/c186ac0505649f3869a2cf24ed5194a2.png">
      <br/>
      一个Bean通常会形成一个依赖树，这种发散的依赖结构导致我们在测试一个类的逻辑时其实是连同它所依赖的逻辑一同进行测试
     </img>
    </p>
    <p>
     此时使用Mock测试就可以解决依赖过多，逻辑复杂的问题
    </p>
    <p>
     Mock以为模拟、虚拟，就是将原有的逻辑进行模拟，使用规划的Mock方案逻辑进行替代
    </p>
    <p>
     在Mock测试中，我们通常对测试对象的所有直接依赖进行Mock，被Mock的直接依赖将变成只有方法签名的空壳
    </p>
    <p>
     被调用时它们不会再调用间接依赖，也不会执行原有的逻辑，只会根据Mock方案进行返回
    </p>
    <p>
     这样在测试测试对象时我们就不在需要关心它负载的间接依赖关系和所有依赖的内部逻辑了，只需要专注于当前测试对象的逻辑即可
    </p>
    <h4>
     <a id="32_Mock_126">
     </a>
     3.2 完全Mock依赖
    </h4>
    <p>
     这种测试仅测试测试对象内部的逻辑，屏蔽所有测试对象的依赖
     <br/>
     适用于测试内部逻辑较为复杂的对象，或依赖较为复杂的对象
     <br/>
     样例：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/e65af206043fc971ba88252b65c0e8a5.png">
      <br/>
      注意
     </img>
    </p>
    <ol>
     <li>
      @MockBean标记的依赖的方法默认变为空方法且返回值为null，一定要配合mock方案才能执行逻辑和获取返回值（参照FAQ-@MockBean与@SpyBean）
     </li>
     <li>
      mock方案的编写参照FAQ-mock方案
     </li>
     <li>
      需要调用@MockBean标记的类中无返回值的方法时可以不编写mock方案，但一定要校验调用是否执行（参照FAQ-验证调用是否执行）
     </li>
    </ol>
    <h4>
     <a id="33__135">
     </a>
     3.3 完全真实依赖
    </h4>
    <pre><code>这种类型的测试同时测试测试对象及全部其依赖的逻辑，针对实际调用流程进行测试
适用于测试一个完整的流程以及测试调用链路正确性
</code></pre>
    <p>
     样例
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/97056049c07d6a15efca76350966c1d4.png">
      <br/>
      注意
     </img>
    </p>
    <ol>
     <li>
      这种测试方式与Mock毫无关系，为常规的方法逻辑调用
     </li>
     <li>
      测试的结果与输入的数据、持久化的数据、程序逻辑直接相关，因此需要事先准备好配套得入参、数据库脚本/Redis缓存并熟悉程序的逻辑，以此推导出正确的处理结果
     </li>
    </ol>
    <h4>
     <a id="34_Mock_143">
     </a>
     3.4 Mock与真实依赖相结合
    </h4>
    <pre><code>这种类型的测试同时测试测试对象及其部分依赖的逻辑，屏蔽部分较为复杂的依赖同时对实际调用流程进行测试
适用于测试一个部分依赖较为复杂的流程以及测试调用链路正确性
</code></pre>
    <p>
     样例：
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/f2c892494ed713aa9a5b5a6795499383.png">
      <br/>
      注意
     </img>
    </p>
    <ol>
     <li>
      @SpyBean标记的依赖的方法默认为真实逻辑，若配合mock方案则在入参与mock方案一致时根据mock方案的设置进行返回（参照FAQ-@MockBean与@SpyBean）
     </li>
     <li>
      mock方案的编写参照FAQ-mock方案
     </li>
     <li>
      需要调用@SpyBean标记的类的方法的真实逻辑时可以不编写mock方案
     </li>
     <li>
      测试的结果与输入的数据、持久化的数据、程序逻辑、mock方案均相关，因此需要事先准备好配套得入参、数据库脚本/Redis缓存并熟悉程序的逻辑和mock方案的逻辑，以此推导出正确的处理结果
     </li>
     <li>
      虽然@SpyBean标记的类在没有设置mock方案的情况下原则上与原类的逻辑一致，但由于底层实现原理的种种限制，它与原生的类在某些情况下不能完全一致；若想直接测试完全真实依赖的场景，请参照完全使用真实依赖的测试章节的描述
     </li>
    </ol>
    <h4>
     <a id="35__155">
     </a>
     3.5 混合测试
    </h4>
    <pre><code>对一个测试对象中的不同方法采取不同的测试策略
对同一个测试对象的不同方法根据需求采取“完全Mock”或“完全真实”或“Mock与真实结合”的方式；没有对同一个测试	对象编写多个不同类型测试用例的需求
</code></pre>
    <p>
     注意
    </p>
    <ol>
     <li>
      此种测试是其他3种的复合形式，单个用例内的书写参照相应章节的描述
     </li>
     <li>
      此种测试的测试对象的依赖需要使用@SpyBean的方式进行标记，否则难以执行真实逻辑（参照FAQ-@MockBean与@SpyBean）
     </li>
    </ol>

    <h4>
     <a id="41_MockBeanSpyBean_165">
     </a>
     4.1 @MockBean与@SpyBean
    </h4>

       </td>
      </tr>
      <tr>
       <td>

       </td>

       </td>
       <td>

       </td>

<span class="token class-name">Mockito</span><span class="token punctuation">.</span><span class="token function">doNothing</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">when</span><span class="token punctuation">(</span>demoService<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">function</span><span class="token punctuation">(</span><span class="token class-name">Matchers</span><span class="token punctuation">.</span><span class="token function">eq</span><span class="token punctuation">(</span>arg1<span class="token punctuation">)</span><span class="token punctuation">,</span> <span class="token class-name">Matchers</span><span class="token punctuation">.</span><span class="token function">eq</span><span class="token punctuation">(</span>arg2<span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">;</span>

    <h4>
     <a id="43__194">
     </a>
     4.3 同名方法调用多次
    </h4>
    <p>
     同名方法设置多个mock方案时，不同入参预期的方案都会保留，相同入参预期的方案以最后一个为准
     <br/>
     例，设置了如下mock方案：
    </p>
    <pre><code class="prism language-java"><span class="token annotation punctuation">@MockBean</span>
<span class="token keyword">private</span> <span class="token class-name">DemoService</span> demoService<span class="token punctuation">;</span>
 
<span class="token annotation punctuation">@Test</span>
<span class="token keyword">public</span> <span class="token keyword">void</span> <span class="token function">testFunction</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
    <span class="token comment">// 省略其他步骤，此处只展示mock方案设置</span>
    <span class="token comment">// 方案1</span>
    <span class="token class-name">Mockito</span><span class="token punctuation">.</span><span class="token function">doReturn</span><span class="token punctuation">(</span>demoDataA<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">when</span><span class="token punctuation">(</span>demoService<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">callFun</span><span class="token punctuation">(</span><span class="token function">eq</span><span class="token punctuation">(</span><span class="token string">"alpha"</span><span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
    <span class="token comment">// 方案2</span>
    <span class="token class-name">Mockito</span><span class="token punctuation">.</span><span class="token function">doReturn</span><span class="token punctuation">(</span>demoDataB<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">when</span><span class="token punctuation">(</span>demoService<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">callFun</span><span class="token punctuation">(</span><span class="token function">eq</span><span class="token punctuation">(</span><span class="token string">"beta"</span><span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
    <span class="token comment">// 方案3</span>
    <span class="token class-name">Mockito</span><span class="token punctuation">.</span><span class="token function">doCallRealMethod</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">when</span><span class="token punctuation">(</span>demoService<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">callFun</span><span class="token punctuation">(</span><span class="token function">eq</span><span class="token punctuation">(</span><span class="token string">"beta"</span><span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
    <p>
     方案1,3会生效，方案2会被方案3覆盖而失效
     <br/>
     调用demoService.callFun(String)方法时：
     <br/>
     若入参为"alpha"，则返回demoDataA；
     <br/>
     若入参为"beta"，则执行该方法的真实逻辑并返回；
     <br/>
     若入参为其他，则执行空方法并返回null（@MockBean的默认方案）；
     <br/>
     在上述场景的基础上，若被测方法中存在同一方法调用多次且入参相同，但根据执行顺序返回值不同，则参照如下写法（使用链式调用依序排布多个方案）：
    </p>
    <pre><code class="prism language-java"><span class="token annotation punctuation">@MockBean</span>
<span class="token keyword">private</span> <span class="token class-name">DemoService</span> demoService<span class="token punctuation">;</span>
 
<span class="token annotation punctuation">@Test</span>
<span class="token keyword">public</span> <span class="token keyword">void</span> <span class="token function">testFunction</span><span class="token punctuation">(</span><span class="token punctuation">)</span> <span class="token punctuation">{<!-- --></span>
    <span class="token comment">// 省略其他步骤，此处只展示mock方案设置</span>
    <span class="token comment">// 方案1</span>
    <span class="token class-name">Mockito</span><span class="token punctuation">.</span><span class="token function">doReturn</span><span class="token punctuation">(</span>demoDataA<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">when</span><span class="token punctuation">(</span>demoService<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">callFun</span><span class="token punctuation">(</span><span class="token function">eq</span><span class="token punctuation">(</span><span class="token string">"alpha"</span><span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
    <span class="token comment">// 方案2（包含多次调用的方案）</span>
    <span class="token class-name">Mockito</span><span class="token punctuation">.</span><span class="token function">doReturn</span><span class="token punctuation">(</span>demoDataB<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">doCallRealMethod</span><span class="token punctuation">(</span><span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">when</span><span class="token punctuation">(</span>demoService<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">callFun</span><span class="token punctuation">(</span><span class="token function">eq</span><span class="token punctuation">(</span><span class="token string">"beta"</span><span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
<span class="token punctuation">}</span>
</code></pre>
    <p>
     调用demoService.callFun(String)方法时：
    </p>
    <p>
     若入参为"alpha"，则返回demoDataA；
     <br/>
     若入参为"beta"且是该入参的第1次调用，则返回demoDataB；
     <br/>
     若入参为"beta"且是该入参的第2次调用，执行该方法的真实逻辑并返回；
     <br/>
     若入参为"beta"且是该入参的第3次及以上调用，执行方案链中最后一个方案（例子中为调用真实逻辑）；
     <br/>
     若入参为其他，则执行空方法并返回null（@MockBean的默认方案）
    </p>
    <h4>
     <a id="44__241">
     </a>
     4.4 验证调用是否执行
    </h4>
    <p>
     在mock测试中需要验证mock方案标记的方法是否按照预期的入参执行过，若缺少这一步可能导致测试用例出现未预期的成功
     <br/>
     例，一个用例按如下步骤执行：
    </p>
    <ol>
     <li>
      测试的依赖使用@SpyBean标注
     </li>
     <li>
      执行到mock方案对应的方法，入参与mock方案预期不一致
     </li>
     <li>
      方法按照真实逻辑执行
     </li>
     <li>
      方法的返回值恰巧与预期一致
     </li>
     <li>
      测试成功
     </li>
    </ol>
    <p>
     上述例子的用例可能会在某次数据库数据变动，外部链接中断的场景下执行失败
     <br/>
     增加验证调用是否执行的步骤可以解决该问题
     <br/>
     写法如下：
    </p>
    <pre><code class="prism language-java"><span class="token class-name">Mockito</span><span class="token punctuation">.</span><span class="token function">verify</span><span class="token punctuation">(</span>demoService<span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">function</span><span class="token punctuation">(</span><span class="token class-name">Matchers</span><span class="token punctuation">.</span><span class="token function">eq</span><span class="token punctuation">(</span>arg1<span class="token punctuation">)</span><span class="token punctuation">,</span> <span class="token class-name">Matchers</span><span class="token punctuation">.</span><span class="token function">eq</span><span class="token punctuation">(</span>arg2<span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
    <p>
     与mock方案中的写法非常相似
     <br/>
     当同名方法调用多次且入参一致时采用如下写法：
    </p>
    <pre><code class="prism language-java"><span class="token class-name">Mockito</span><span class="token punctuation">.</span><span class="token function">verify</span><span class="token punctuation">(</span>demoService<span class="token punctuation">,</span> <span class="token keyword">new</span> <span class="token class-name">Times</span><span class="token punctuation">(</span><span class="token number">2</span><span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">.</span><span class="token function">function</span><span class="token punctuation">(</span><span class="token class-name">Matchers</span><span class="token punctuation">.</span><span class="token function">eq</span><span class="token punctuation">(</span>arg1<span class="token punctuation">)</span><span class="token punctuation">,</span> <span class="token class-name">Matchers</span><span class="token punctuation">.</span><span class="token function">eq</span><span class="token punctuation">(</span>arg2<span class="token punctuation">)</span><span class="token punctuation">)</span><span class="token punctuation">;</span>
</code></pre>
