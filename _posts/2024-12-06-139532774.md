    <h2>
     <a id="_4">
     </a>
     一、协议概述
    </h2>
    <p>
     LIN网络管理主要指休眠和唤醒。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/717accaf352cb4bb7f6dacc0289f5c79.png"/>
    </p>
    <h3>
     <a id="_7">
     </a>
     休眠
    </h3>
    <p>
     1、主机节点发送休眠命令进行休眠；
     <br/>
     2、或者总线静默4s~10s（
     <font color="blue">
      没有显/隐性位之间的转换
     </font>
     ）节点自动进入休眠状态。
    </p>
    <h3>
     <a id="_10">
     </a>
     唤醒
    </h3>
    <p>
     LIN网段上的主/从机节点都可以向总线上发送唤醒信号，唤醒信号持续 250μs～5ms的显性电平（5个bit位）。其余节点(除发送唤醒信号以外的节点)以大于 150μs 为阈值判定唤醒信号。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/81791d54c9c8e53acb26c78bc5b945dc.png"/>
    </p>
    <blockquote>
     <p>
      这里有几个细节点：
      <br/>
      1、唤醒信号持续 250μs～5ms，按照位时间51us计算，至少是需要
      <strong>
       5个显性bit位
      </strong>
      。
      <br/>
      2、主机节点的
      <strong>
       同步间隔段
      </strong>
      也可以充当唤醒信号（理由：同步间隔段时至少由13个显性位组成，按照位时间51us计算，此时的显性电平持续时间为51*13=663us，满足唤醒信号要求。)
      <br/>
      3、如上图，协议规定唤醒信号最多可以发送 3 次，3 次之后，必须等待至少 1.5s 之后才可以再次发送唤醒信号。
     </p>
    </blockquote>
    <h2>
     <a id="_18">
     </a>
     二、休眠测试（节点级）
    </h2>
    <h3>
     <a id="1_19">
     </a>
     1.测试场景
    </h3>
    <p>
     利用诊断帧中的主机请求帧0x3C作休眠命令，要求Byte0=0x00，其余Byte=0xFF。节点能够正常休眠。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/a767b8b6ca04723c8fc0aaec6129f224.png">
      <br/>
      当发送的0x3C报文数据场内容有误时，如下图Byte1=0x1时，节点不休眠。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/85fa097fc5b3e1538b10acd1809219a4.png"/>
     </img>
    </p>
    <h3>
     <a id="2_24">
     </a>
     2.问题描述
    </h3>
    <p>
     在发送错误的3C休眠报文时，结果出现Trace窗口报文停发，电源电流变为0，理所应当的会判断为节点休眠。与上述测试场景预期结果不符（不休眠），认为节点有故障。实际真的如此吗？
    </p>
    <h3>
     <a id="3_26">
     </a>
     3.结果分析
    </h3>
    <p>
     Trace窗口上报文停发并不一定是响应3C休眠命令导致的。发送3C休眠报文，LDF调度表跳转到LIN诊断调度表中，Trace上报文立即停发只是因为没有切换回Normal调度表。因此，对于节点休眠的判断应该回归到电流上。
    </p>
    <ul>
     <li>
      正确的3C报文发送后节点立即休眠，电流在ms级别内变为0。
      <br/>
      从波形图可以看到节点休眠的时间差为4.574ms。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/a58880f30497246502dd791b6441a8c4.png"/>
     </li>
     <li>
      错误的3C报文发送后节点没有休眠，4s后电流变为0。
      <br/>
      从波形图可以看出，错误的3C报文发出后，因调度表在诊断调度表中没有切换回Normal调度表，导致报文停发，但是LIN线电流没有降为0。报文停发后，没有显/隐性位之间的转换，4s后因总线空闲导致休眠。时间差为4.009s印证此结论。
      <br/>
      <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/9f06afd1346e3dcfdc6704f1c422aff2.png">
       <br/>
       <strong>
        注
       </strong>
       ：若没有示波器条件，可进行程控电源电流监测，立即变0还是4秒变0。
      </img>
     </li>
    </ul>
    <p>
     而总线静默4~10s后进入休眠模式的Trace效果如下图。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/a5884a7759b1d0d4e8f45ec8368ad4d9.png">
      <br/>
      综上，发送错误的3C休眠报文，被测LIN节点是没有休眠满足预期的。表现上看着像是休眠了，是因为总线静默4s~10s休眠，而不是发送休眠命令导致的休眠。
     </img>
    </p>
