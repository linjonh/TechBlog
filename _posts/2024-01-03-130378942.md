    <h2>
     小程序持续集成
    </h2>
    <p>
     Taro 小程序端构建后支持 CI（持续集成）的插件
     <code>
      @tarojs/plugin-mini-ci
     </code>
     。 目前已支持（企业）微信、京东、字节、支付宝、钉钉、百度小程序
    </p>
    <p>
     功能包括：
    </p>
    <ul>
     <li>
      构建完毕后自动唤起小程序开发者工具并打开项目
     </li>
     <li>
      上传代码作为开发版并生成预览二维码
     </li>
     <li>
      上传代码作为体验版并生成体验二维码
     </li>
     <li>
      支持 上传、预览 hooks 回调
     </li>
    </ul>
    <h3 id="使用">
     使用
     <a href="https://taro-docs.jd.com/docs/plugin-mini-ci#%E4%BD%BF%E7%94%A8" rel="nofollow" title="​">
      ​
     </a>
    </h3>
    <h4 id="安装">
     安装
     <a href="https://taro-docs.jd.com/docs/plugin-mini-ci#%E5%AE%89%E8%A3%85" rel="nofollow" title="​">
      ​
     </a>
    </h4>
    <pre><code>npm i @tarojs/plugin-mini-ci -D
</code></pre>
    <p>
    </p>

    <h4 id="作为选项配合-build-命令使用">
     作为选项配合 build 命令使用
     <a href="https://taro-docs.jd.com/docs/plugin-mini-ci#%E4%BD%9C%E4%B8%BA%E9%80%89%E9%A1%B9%E9%85%8D%E5%90%88-build-%E5%91%BD%E4%BB%A4%E4%BD%BF%E7%94%A8" rel="nofollow" title="​">
      ​
     </a>
    </h4>
    <p>
     <code>
      package.json
     </code>
     的
     <code>
      scripts
     </code>
     字段使用命令参数
    </p>
    <pre><code>{
  "scripts": {
    //  构建完后自动 “打开开发者工具”
    "build:weapp": "taro build --type weapp --open",
    //  构建完后自动 “上传代码作为开发版并生成预览二维码”
    "build:weapp:preview": "taro build --type weapp --preview",
    //  构建完后自动“上传代码作为体验版”
    "build:weapp:upload": "taro build --type weapp --upload",
    //  构建完后自动“上传 dist/xxx 目录的代码作为体验版”， `--projectPath` 参数 适用于 taro 和 原生混合的场景
    "build:weapp:upload": "taro build --type weapp --upload --projectPath dist/xxx"
  },
  "taroConfig": {
    "version": "1.0.0",
    "desc": "上传描述"
  }
}
</code></pre>
    <p>
    </p>
    <p>
     由上面的示例可知，插件为 taro cli 命令扩展了 4 个选项：
    </p>
    <ul>
     <li>
      --open 打开开发者工具，类似于网页开发中自动打开谷歌浏览器
     </li>
     <li>
      --preview 上传代码作为开发版并生成预览二维码
     </li>
     <li>
      --upload 上传代码作为体验版
     </li>
    </ul>
    <p>
     此 3 个选项在一条命令里不能同时使用（互斥）
    </p>
    <ul>
     <li>
      --projectPath 指定要操作（打开、预览、上传）的目录路径， 默认情况下是操作构建后目录路径，即
      <a href="https://taro-docs.jd.com/taro/docs/next/config-detail#outputroot" rel="nofollow" title="outputRoot 选项">
       outputRoot 选项
      </a>
      ；
     </li>
    </ul>
    <p>
     此选项必须搭配上述三个选项之一一起使用；
    </p>
    <p>
     此选项优先级为： 终端传入的
     <code>
      --projectPath
     </code>
     &gt; CI 配置的
     <code>
      projectPath
     </code>
     选项 &gt;
     <a href="https://taro-docs.jd.com/taro/docs/next/config-detail#outputroot" rel="nofollow" title="outputRoot 选项">
      outputRoot 选项
     </a>
     。
    </p>
    <h4 id="作为命令单独使用360-开始支持">
     作为命令单独使用（3.6.0 开始支持）
     <a href="https://taro-docs.jd.com/docs/plugin-mini-ci#%E4%BD%9C%E4%B8%BA%E5%91%BD%E4%BB%A4%E5%8D%95%E7%8B%AC%E4%BD%BF%E7%94%A8360-%E5%BC%80%E5%A7%8B%E6%94%AF%E6%8C%81" rel="nofollow" title="​">
      ​
     </a>
    </h4>
    <pre><code>{
  "scripts": {
    //  直接 “打开开发者工具并载入项目”
    "build:weapp": "taro open --type weapp  --projectPath dist/xxx",
    //  直接 “上传代码作为开发版并生成预览二维码”
    "build:weapp:preview": "taro preview --type weapp",
    //  直接“上传代码作为体验版”
    "build:weapp:upload": "taro upload --type weapp",
    //  上传指定目录代码作为体验版
    "build:weapp:upload2": "taro upload --type weapp --projectPath dist/xxx"
  },
  "taroConfig": {
    "version": "1.0.0",
    "desc": "上传描述"
  }
}
</code></pre>
    <p>
    </p>
    <p>
     由上面的示例可知，插件额外新增了 3 个独立命令，让你可以直接操作指定目录。适用于把
     <code>
      taro
     </code>
     作为项目一部分的使用场景。
    </p>
    <p>
     当直接作为命令使用时，有两个选项：
    </p>
    <ul>
     <li>
      --type 传入平台名称
     </li>
     <li>
      --projectPath 传入路径。 此选项优先级为： 终端传入的
      <code>
       --projectPath
      </code>
      &gt; CI 配置的
      <code>
       projectPath
      </code>
      选项 &gt;
      <a href="https://taro-docs.jd.com/taro/docs/next/config-detail#outputroot" rel="nofollow" title="outputRoot 选项">
       outputRoot 选项
      </a>
     </li>
    </ul>
    <h4 id="hooks-使用360-开始支持">
     Hooks 使用（3.6.0 开始支持）
     <a href="https://taro-docs.jd.com/docs/plugin-mini-ci#hooks-%E4%BD%BF%E7%94%A8360-%E5%BC%80%E5%A7%8B%E6%94%AF%E6%8C%81" rel="nofollow" title="​">
      ​
     </a>
    </h4>
    <p>
     在插件执行完
     <code>
      预览
     </code>
     、
     <code>
      上传
     </code>
     操作后， 插件会触发 2 个钩子事件：
    </p>
    <table>
     <thead>
      <tr>
       <th>
        事件名
       </th>
       <th>
        传递参数对象
       </th>
       <th>
        说明
       </th>
      </tr>
     </thead>
     <tbody>
      <tr>
       <td>
        onPreviewComplete
       </td>
       <td>
        详细见下文
       </td>
       <td>
        CI 执行预览后触发
       </td>
      </tr>
      <tr>
       <td>
        onUploadComplete
       </td>
       <td>
        详细见下文
       </td>
       <td>
        CI 执行上传后触发
       </td>
      </tr>
     </tbody>
    </table>
    <p>
     两个钩子被触发时传入的数据对象描述如下
    </p>
    <pre><code>interface HooksData {
  /** 是否预览、构建成功 */
  success: boolean
  data: {
    /** 当前构建的小程序平台 */
    platform: string
    /** 预览码本地路径 */
    qrCodeLocalPath: string
    /** 预览码内容 */
    qrCodeContent: string
    /** 插件传递的预览版本号 */
    version: string
    /** 插件传递的描述文本 */
    desc: string
    /** 预览或上传的目录路径 */
    projectPath: string
  }
  /** 错误对象 */
  error?: Error
}
</code></pre>
    <p>
    </p>
    <p>
     你可以写一个自定义插件，来接收上述 2 个事件传递的值：
    </p>
    <pre><code>// config/test.js
module.exports = function (ctx) {
  ctx.register({
    name: 'onPreviewComplete',
    fn: ({ success, data, error }) =&gt; {
      console.log('接收预览后数据', success, data, error)
      // 你可以在这里发送钉钉或者飞书消息
    },
  })
  ctx.register({
    name: 'onUploadComplete',
    fn: ({ success, data, error }) =&gt; {
      console.log('接收上传后数据', success, data, error)
      // 你可以在这里发送钉钉或者飞书消息
    },
  })
}
</code></pre>
    <p>
    </p>
    <p>
     然后把自己写的插件配置应用起来：
    </p>
    <pre><code>// config/index.js
const config = {
  plugins: [
    ['@tarojs/plugin-mini-ci', CI插件参数],
    // 应用自己写的插件
    require('path').join(__dirname, './test'),
  ],
  // 其他配置省略
}

module.exports = function (merge) {
  if (process.env.NODE_ENV === 'development') {
    return merge({}, config, require('./dev'))
  }
  return merge({}, config, require('./prod'))
}
</code></pre>
