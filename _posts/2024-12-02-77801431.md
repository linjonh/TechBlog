    <h3 id="一首先我先了解下flyway是如何运转的">
     一，首先我先了解下FlyWay是如何运转的。
    </h3>
    <p>
     最简单的方案是将Flyway指向一个空数据库。
     <br/>
     <img alt="这里写图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/8275d7b3b695ce3544f2aa1434ca512a.png" title="">
      <br/>
      它将尝试找到其元数据表。当数据库为空时，Flyway将不会找到它， 而是创建它。您现在拥有一个名为SCHEMA_VERSION的单个空表的数据库 ：
      <br/>
      <img alt="这里写图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/14a30e7ef2e1d5dedf3639a18bc25794.png" title="">
       <br/>
       该表将用于跟踪数据库的状态。之后，Flyway将开始扫描应用程序的文件系统或类路径进行迁移。它们可以用Sql或Java编写。
       <br/>
       然后根据其版本号对迁移进行排序，并按顺序应用：
       <br/>
       <img alt="这里写图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/4d098e8b90c7e0a20decf45e20340a41.png" title="">
        <br/>
        随着应用每个迁移，元数据表将相应更新：
        <br/>
        schema_version
        <br/>
        <img alt="这里写图片描述" src="https://img-blog.csdn.net/20170902151703404?watermark/2/text/aHR0cDovL2Jsb2cuY3Nkbi5uZXQvdTAxMDk2Mzk0OA==/font/5a6L5L2T/fontsize/400/fill/I0JBQkFCMA==/dissolve/70/gravity/SouthEast" title="">
         <br/>
         随着元数据和初始状态的到位，我们现在可以谈论迁移到较新的版本。
         <br/>
         Flyway将再次扫描应用程序的文件系统或类路径进行迁移。根据元数据表检查迁移。如果版本号低于或等于标记为当前版本的版本号，则忽略它们。
         <br/>
         剩余的迁移是挂起的迁移：可用但未应用。
         <br/>
         <img alt="这里写图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/fb624f79e355bb5260e975b9f925d71f.png" title="">
          <br/>
          然后，他们通过版本号进行排序，并依次执行：
          <br/>
          <img alt="这里写图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/a59a17e33d1b6a36e5bcbcedbecd6490.png" title="">
           <br/>
           该元数据表进行更新，因此：
           <br/>
           schema_version
          </img>
         </img>
        </img>
       </img>
      </img>
     </img>
    </p>
    <p>
     就是这样！每当需要发展数据库时，无论是结构（DDL）还是参考数据（DML），只需创建一个版本号高于当前版本的新迁移。下一次Flyway启动时，它会发现并相应地升级数据库。
    </p>
    <h3 id="二flyway在spingboot的使用">
     二，FlyWay在SpingBoot的使用
    </h3>
    <p>
     一种途径是通过Spring Boot的spring.jpa.hibernate.ddl-auto属性将hibernate.hbm2ddl.auto属性设置为create、 create-drop或update。例如，要把hibernate.hbm2ddl.auto设置为create-drop，我们可以在application.yml里加入如下内容：
    </p>
    <pre class="prettyprint"><code class="hljs sql">spring:
    jpa:
        hibernate:
            ddl-auto: <span class="hljs-operator"><span class="hljs-keyword">create</span>-<span class="hljs-keyword">drop</span></span></code></pre>
    <p>
     然而，这对生产环境来说并不理想，因为应用程序每次重启数据库， Schema就会被清空，从头开始重建。它可以设置为update，但就算这样，我们也不建议将其用于生产环境。
     <br/>
     还有一个途径。我们可以在schema.sql里定义Schema。在第一次运行时，这么做没有问题，但随后每次启动应用程序时，这个初始化脚本都会失败，因为数据表已经存在了。这就要求在书写初始化脚本时格外注意，不要重复执行那些已经做过的工作。
     <br/>
     一个比较好的选择是使用数据库迁移库（database migration library）。它使用一系列数据库脚本，而且会记录哪些已经用过了，不会多次运用同一个脚本。应用程序的每个部署包里都包含了这些脚本，数据库可以和应用程序保持一致。Spring Boot为两款流行的数据库迁移库提供了自动配置支持。
     <br/>
      Flyway（
     <a href="http://flywaydb.org" rel="nofollow noopener noreferrer" target="_blank">
      http://flywaydb.org
     </a>
     ）
     <br/>
      Liquibase（
     <a href="http://www.liquibase.org" rel="nofollow noopener noreferrer" target="_blank">
      http://www.liquibase.org
     </a>
     ）
     <br/>
     当你想要在Spring Boot里使用其中某一个库时，只需在项目里加入对应的依赖，然后编写脚本就可以了。让我们先从Flyway开始了解吧。
     <br/>
     <strong>
      1. 用Flyway定义数据库迁移过程

    <p>
     在应用程序部署并运行起来后， Spring Boot会检测到Classpath里的Flyway，自动配置所需的
     <br/>
     Bean。 Flyway会依次查看/db/migration里的脚本，如果没有执行过就运行这些脚本。每个脚本都
     <br/>
     执行过后，向schema_version表里写一条记录。应用程序下次启动时，Flyway会先看schema_version
     <br/>
     里的记录，跳过那些脚本。
    </p>
