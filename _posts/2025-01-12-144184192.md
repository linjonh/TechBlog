    <h2>
     <a id="1__5">
     </a>
     1. 测试用例基础知识
    </h2>
    <p>
     在讲解测试用例设计之前，我们先来了解了解测试用例的基础知识~
    </p>
    <h3>
     <a id="11__7">
     </a>
     1.1 测试用例概念
    </h3>
    <p>
     测试用例就是一组集合，包括了这四大要素：
    </p>
    <ul>
     <li>
      <strong>
       测试环境
      </strong>
     </li>
     <li>
      <strong>
       操作步骤
      </strong>
     </li>
     <li>
      <strong>
       测试数据
      </strong>
     </li>
     <li>
      <strong>
       预期结果
      </strong>
     </li>
    </ul>
    <h3>
     <a id="12__13">
     </a>
     1.2 测试用例好处
    </h3>
    <p>
     为什么需要测试用例呢？包括但不限于以下三点好处：
    </p>
    <ul>
     <li>
      <strong>
       提高代码的复用性，提高测试人员的测试效率
      </strong>
     </li>
     <li>
      <strong>
       便于回归测试
      </strong>
     </li>
     <li>
      <strong>
       为自动化测试提供基础
      </strong>
     </li>
    </ul>
    <h2>
     <a id="2__19">
     </a>
     2. 黑盒测试用例的设计
    </h2>
    <h3>
     <a id="21__20">
     </a>
     2.1 黑盒测试概念
    </h3>
    <p>
     黑盒测试也称为
     <strong>
      功能测试
     </strong>
     ，或者是数据驱动测试，是一种软件测试的方法。（软件测试按照不同的条件可以分成不同的类别～上期内容有介绍）
     <strong>
      黑盒测试的核心思想是
     </strong>
     将被测的软件视为一个无法打开的“黑盒子”，
     <strong>
      测试人员无需关注软件内部的逻辑结构和代码实现，仅根据软件的需求规格说明书/产品需求文档/prd
     </strong>
     ，
     <strong>
      检查软件的功能是否符合预期的功能说明
     </strong>
    </p>
    <h3>
     <a id="22__22">
     </a>
     2.2 基于需求进行测试用例的设计
    </h3>
    <p>
     基于需求设计测试用例是测试设计和开发测试用例的基础，测试用例是需要紧紧围绕需求来设计的～
    </p>
    <p>
     第一步：要分析测试需求，验证需求是否正确、完整、无二义性，并且逻辑自洽
     <br/>
     第二步：在需求正确的基础上细化测试需求，从测试需求提炼出一个个测试点或者测试项
     <br/>
     第三步：根据每一个测试点进行测试用例的设计
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/81d90cb218614d5e89d5b657d3e3a8f0.png#pic_center"/>
    </p>
    <p>
     总之，就是要根据需求文档，分析需求，根据需求文档进行测试用例的编写～
    </p>
    <p>
     在进行分析测试需求的时候，一般分为
     <strong>
      功能测试需求和非功能测试需求
     </strong>
    </p>
    <h4>
     <a id="221__33">
     </a>
     2.2.1 功能测试需求
    </h4>
    <p>
     对于功能测试中，可以借助功能框图来帮助我们进行测试的需求分析，总之，概括起来，功能测试需求通常包括以下几个方面：
    </p>
    <ol>
     <li>
      系统各功能
      <strong>
       界面
      </strong>
      的验证
     </li>
     <li>
      借助
      <strong>
       业务
      </strong>
      把功能串起来进行测试
     </li>
     <li>
      功能的
      <strong>
       一致性，交互性
      </strong>
      （即多功能相互操作）的测试
     </li>
     <li>
      系统的
      <strong>
       不同输入，结果输出
      </strong>
      的业务数据测试
     </li>
     <li>
      功能的
      <strong>
       错误、异常
      </strong>
      操作测试（属于负面测试）
     </li>
     <li>
      功能实现用到的
      <strong>
       算法验证
      </strong>
      （有的时候需要运用代码评审）
     </li>
     <li>
      用户操作的
      <strong>
       易用性
      </strong>
      ，用户体验，往往需要结合功能测试同时验证
     </li>
    </ol>
    <h4>
     <a id="222__42">
     </a>
     2.2.2 非功能测试需求
    </h4>
    <p>
     非功能测试需求，主要是涉及
     <strong>
      性能、安全性、可靠性、兼容性、易维护性、可移植性
     </strong>
     等等
    </p>
    <p>
     每一类非功能特性测试都需要根据需求
     <strong>
      单独
     </strong>
     分析，因为它们之间可能存在影响，比如，安全性越高，就可能给易用性和性能方面带来更大的挑战（有点鱼和熊掌不可兼得的意味～）
    </p>
    <h3>
     <a id="23__46">
     </a>
     2.3. 黑盒测试用例设计的具体方法
    </h3>
    <h4>
     <a id="231__47">
     </a>
     2.3.1 等价类法
    </h4>
    <p>
     【
     <strong>
      概念
     </strong>
     】依据需求将输入划分成若干个等价类，从等价类中选取一个测试用例，如果这个测试用例通过了，则认为所代表的等价类测试通过，这里可以清楚的看出等价类的好处，
     <strong>
      可以用较少的测试用例达到尽可能多的功能覆盖，解决了不能穷举测试的问题～
     </strong>
    </p>
    <p>
     【
     <strong>
      注意
     </strong>
     】这里需求将输入划分，
     <strong>
      在特殊情况下会考虑输出
     </strong>
    </p>
    <p>
     【
     <strong>
      分类
     </strong>
     】分为有效等价类和无效等价类
    </p>
    <ul>
     <li>
      <strong>
       有效等价类
      </strong>
      ：指对于程序的规格来说合理，有意义的输入数据构成的集合
     </li>
     <li>
      <strong>
       无效等价类
      </strong>
      ：指对于程序的规格说明而言不合理或者无意义的输入数据
     </li>
    </ul>
    <p>
     【
     <strong>
      步骤
     </strong>
     】
    </p>
    <ol>
     <li>
      充分理解需求
     </li>
     <li>
      划分有效等价类和无效等价类
     </li>
     <li>
      分别从有效等价类和无效等价类中抽取一个数据进行测试用例设计
     </li>
    </ol>
    <p>
     【
     <strong>
      举例
     </strong>
     】对用户名长度在6-12个字符之间，进行等价类划分，如下：

    </p>
    <h4>
     <a id="232__65">
     </a>
     2.3.2 边界值法
    </h4>
    <p>
     【
     <strong>
      概念
     </strong>
     】边界值法是一种黑盒测试方法，
     <strong>
      其核心思想是通过选择系统输入的边界值和特殊值作为测试用例
     </strong>
     ，边界值通常指的是输入变量的最小值、最大值以及临界值，特殊值则是系统能够接受的非常规输入
    </p>
    <p>
     【
     <strong>
      注意
     </strong>
     】在设计测试用例时，边界值通常与等价类划分法一起使用，这样可以实现尽可能多的用例覆盖
    </p>
    <p>
     使用边界值法进行测试用例的设计时，需要对以下三个概念进行理解，如下：
    </p>
    <ul>
     <li>
      上点：边界上的点（在闭区间中，上点位于区间内，在开区间中，上点位于区间外）
     </li>
     <li>
      离点：是指离上点最近的点（在开区间中，离点位于区间内最近的点，而在闭区间中，离点则位于区间外最近的点）
     </li>
     <li>
      内点：是指区间内任意的一点
     </li>
    </ul>

     下面通过一个例子，对上点、离点、内点，进行进一步的介绍：

    <h4>
     <a id="233__85">
     </a>
     2.3.3 判定表法
    </h4>
    <p>
     【
     <strong>
      概念
     </strong>
     】判定表法是一种黑盒测试方法，又称为决策表法，是一种表格状的图形工具，
     <strong>
      其核心思想是适用于处理判断条件较多，各条件相互组合，有多种决策方案的情况
     </strong>
     ，判定法能够把所有的输入条件、所有可能采取的动作按照表格列出来，每一种条件和动作组合构成一条规则，也就是一条用例
    </p>
    <p>
     【
     <strong>
      组成
     </strong>
     】判定表由 4 个部分组成
    </p>
    <ul>
     <li>
      条件桩：列出所有条件的名称，即系统的所有输入
     </li>
     <li>
      动作桩：列出系统可能采取的操作，即结果
     </li>
     <li>
      条件项：列出各个条件所有可能的取值
     </li>
     <li>
      动作项：列出所有可能采取的动作
     </li>
    </ul>
    <p>
     【
     <strong>
      规则
     </strong>
     】一组条件与动作的组合，一条规则对应一条测试用例
    </p>
    <p>
     【
     <strong>
      举例
     </strong>
     】使用判定表法，在打顺风车的时候，乘客订单提交，如果是优惠车或者乘客有优惠券，则由优惠，设计测试用例
    </p>

    <h4>
     <a id="234__103">
     </a>
     2.3.4 正交表法
    </h4>
    <p>
     【
     <strong>
      概念
     </strong>
     】正交表法通常用 Ln(mk) 表示，其中这 4 个字母代表的含义如下：
    </p>
    <ul>
     <li>
      L 代表正交表
     </li>
     <li>
      n 代表试验次数或正交表的行数
     </li>
     <li>
      k 代表最多可安排影响指标因素的个数或正交表的列数
     </li>
     <li>
      m 代表每个因素水平数，正交表的每一列中，不同数字出现的次
     </li>
    </ul>
    <p>
     【
     <strong>
      解释说明
     </strong>
     】
    </p>
    <ul>
     <li>
      <strong>
       因素
      </strong>
      ：输入变量
     </li>
     <li>
      <strong>
       水平
      </strong>
      ：每个输入变量的取值
     </li>
    </ul>
    <p>
     【
     <strong>
      两个重要性质
     </strong>
     】
    </p>
    <ul>
     <li>
      <strong>
       均衡分散性
      </strong>
      ：正交表的每一列，不同的数字/水平 出现的次数是相等的
     </li>
     <li>
      <strong>
       整齐可比性
      </strong>
      ：正交表中任意两列所构成的各有序数对出现的次数一样多
     </li>
    </ul>
    <p>
     【
     <strong>
      如何通过正交表设计测试用例
     </strong>
     】
    </p>
    <p>
     主要分为以下五个大步骤：
    </p>
    <p>
     充分理解需求 ——&gt; 确定因素，确定水平 ——&gt; 画正交表 ——&gt; 补充正交表 ——&gt; 将正交表转换成测试用例
    </p>
    <p>
     以注册这个需求为例子，要求对姓名，邮箱，密码，确认密码，验证码都输入正确才能注册成功
    </p>
    <p>
     1）对以上需求进行分析可知：
     <br/>
     2）
     <strong>
      因素
     </strong>
     ：姓名，邮箱，密码，确认秘密，验证码
     <br/>
     <strong>
      水平
     </strong>
     ：填写，不填写
     <br/>
     3）通过 allpairs 画正交表
    </p>
    <ul>
     <li>
      将因素和水平放到 excel 表格中
     </li>
     <li>
      将 excel 表格中内容直接复制到 txt 文本中
     </li>
     <li>
      通过 cmd 进入到 allpairs 安装路径下面
     </li>
     <li>
      生成正交表 allpairs.exe filename.txt &gt; filename_result.txt
     </li>
    </ul>
    <p>
     4）补充正交表
     <br/>
     5） 将正交表转换成测试用例
    </p>
    <h4>
     <a id="235__137">
     </a>
     2.3.5 场景设计法
    </h4>
    <p>
     我们知道，现在软件几乎都是用事件触发来控制流程的，事件触发时的情景就便形成了场景，而同一事件不同的触发顺序和处理结果就形成了事件流。
    </p>
    <p>
     其中，
     <strong>
      基本流
     </strong>
     是软件功能按照最短的事件流实现的一条正确的流程
     <br/>
     而
     <strong>
      备选流
     </strong>
     则是由于异常、缺陷或其他原因导致最终的目的不能实现或实现的流程，并非最短的流程
    </p>
    <p>
     该方法可以比较生动地描绘出事件触发时的场景，有利于测试设计者设计测试用例，使测试用例更容易理解和执行，典型的应用是用业务流把各个孤立的功能点串起来，为测试人员建立整体业务感觉，从而避免缺陷功能细节忽略业务流程要点的错误倾向~
    </p>
    <p>
     那么如何通过场景设计进行测试用例的设计呢？
    </p>
    <p>
     【
     <strong>
      步骤
     </strong>
     】
    </p>
    <p>

    <h4>
     <a id="236__164">
     </a>
     2.3.6 错误猜测法
    </h4>
    <p>
     黑盒测试的错误猜测法，也被称为
     <strong>
      错误推测法
     </strong>
     ，是一种软件测试方法，其
     <strong>
      核心思想是在测试程序的时候，测试人员根据经验、知识和直觉来推测程序可能存在的各种错误，从而有针对性地编写检查这些错误的测试用例
     </strong>
     。这种方法主要依赖于测试人员的直觉和工程经验，而并非是一种有章可循的工程设计方法。
    </p>
    <p>
     需要注意的是，错误猜测法并不是一种完全可靠的测试方法，因为它依赖于测试人员的直觉和经验，可能存在遗漏或者偏差。因此，在实际测试过程中，通常会结合其他测试方法，比如等价类划分、边界值分析等，以确保测试的全面性和有效性~
    </p>
    <p>
     继续以注册为例：
    </p>
    <ol>
     <li>
      校验中特殊字符空格的处理
     </li>
     <li>
      密码发送是否明文
     </li>
     <li>
      密码校验中的大小写
     </li>
     <li>
      姓名中的特殊字符
      <br/>
      …
     </li>
    </ol>
