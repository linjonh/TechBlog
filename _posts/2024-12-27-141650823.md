     不管是什么行业和职业，都要做好相应的了解，提前做好职业规划和学习路线，否则在现如今的大环境下，倘若摸着石头过河，很容易还没摸到石头，就被大水冲跑了。

     尤其是对于现在的应届毕业生，大环境让很多人都很难找到心仪的工作，竞争的人很多，岗位却变少了。有了清晰的职业发展方向后，便可提前做好技术积累，给面试官一个 surprise。
    </p>
    <p id="u59f98673">

    </p>

    </p>
    <p class="img-center">
     <img alt="" height="259" id="u716ead5a" src="https://i-blog.csdnimg.cn/blog_migrate/09f8d073973b4abb6912f855338a4973.png" width="414"/>

    <p id="uc38be857">
     今天咱们就来聊一聊，作为前端程序员，我们的职业发展思路和方向，可以是什么样子的呢。
    </p>
    <h2>
    </h2>
    <h2 id="fNkYD">
     Advancement Path
    </h2>
    <p id="ucf9373bc">
     先来看看晋升路线。
    </p>
    <p id="uffcbddc1">
    </p>
    <p id="u6f047d2a">
     程序员的技术职业生涯，这里咱们暂且先不提转岗为管理层，技术人的 title 可大致分为几个等级：
     <code>
      初级
     </code>
     、
     <code>
      中级
     </code>
     、
     <code>
      高级
     </code>
     、
     <code>
      资深
     </code>
     、
     <code>
      专家
     </code>
     、
     <code>
      架构师
     </code>
     、
     <code>
      CTO
     </code>
     。
     <img alt="" height="978" src="https://i-blog.csdnimg.cn/direct/05d04ec6351f4874bb2d677b850d67a7.png" width="1008"/>
    </p>
    <p id="u61ccb6f0">
     每个阶段所掌握的技术内容，和需要具备的能力，都是不尽相同的。我们需要基于当前的等级，规划未来几年需要达到什么样的层次，为了达到这个等级我需要掌握什么样的技能，而不是盲目去学习试图可以很自然的就得到了晋升。
    </p>
    <p id="udeee8655">
    </p>
    <p id="ucc2f1a8e">
     <span style="color:#1a439c;">
      <strong>
       初级前端
      </strong>
     </span>
     需要掌握怎么使用框架进行基本的开发，学习产品开发思维和提升学习能力，需要掌握基本的 HTML、CSS、JS 相关知识，例如原型链、闭包、作用域、异步编程等；
    </p>
    <p id="u35d0331d">
     <span style="color:#1a439c;">
      <strong>
       中级前端
      </strong>
     </span>
     需要接触一些更深刻的问题，例如如何进行一些基本的工程化配置和原理、了解浏览器的渲染原理、常用框架的一些实现原理、了解性能优化、了解网络协议、掌握 TS 等等；
    </p>
    <p id="u901f95a3">
     <span style="color:#1a439c;">
      <strong>
       高级前端
      </strong>
     </span>
     需要掌握浏览器运行机制、性能优化、代码重构、不同框架的底层实现原理、前端工程化、技术选型、可视化开发、熟悉 Docker、了解后端知识、算法、设计模式、项目组织结构、指导新人等等；
    </p>
    <p id="u2eb950a8">
     <span style="color:#1a439c;">
      <strong>
       资深前端
      </strong>
     </span>
     需要掌握跨端、前端的高级应用、管理一个小组、掌握大型项目的架构设计、掌握前端领域的研究方法、跟进前端最新发展趋势、掌握前端运维部署、掌握计算机底层原理等等；
    </p>
    <p id="u578fa0db">
     <span style="color:#1a439c;">
      <strong>
       专家
      </strong>
     </span>
     需要带领团队攻坚技术难题、管理前端团队、培养有潜力的团队成员、提高团队整体工作效率等等；
    </p>
    <p id="u2619b7c5">
     <span style="color:#1a439c;">
      <strong>
       架构师
      </strong>
     </span>
     一般不会仅停留在纯前端或纯后端，需要做到统筹帷幄，需要设计系统架构，保障系统的稳定性、安全性、扩展性等等；
    </p>
    <p id="ud044c2d2">
     <span style="color:#1a439c;">
      <strong>
       CTO
      </strong>
     </span>
     就是一家公司技术层面的最高负责人了，需要带领公司进行技术探索，保障整个公司的系统稳定，探索公司未来的技术发展方向等等，CTO 一般需要较高的学历或极其丰富的阅历，才能作为公司的技术招牌、稳定人心、拉到投资。
    </p>
    <p class="img-center">
     <img alt="" height="337" src="https://i-blog.csdnimg.cn/direct/31f00131c9604026b5e9ac5b1f98ed75.png" width="337"/>
    </p>
    <p id="u67739f23">
     不同的公司对于不同的职级会有不同的划分，以上仅是参考，以实际具体的公司要求为准，不同职级所需要掌握的技术内容也没有限制那么严格，毕竟学无止境，会的更多当然更好。当然也不是说一定非要学那么多，但是至少要保障有一些核心竞争力，才不会被淘汰。
    </p>
    <h2>
    </h2>
    <h2 id="nQZPH">
     Basic
    </h2>
    <p id="u6666f6b5">
     程序员有一些永恒的话题：程序员到底需不需要高学历？程序员英文不好到底行不行？前端/Java/... 到底有没有前途？...
    </p>
    <p id="ube317c37">
    </p>
    <p id="ua9d02235">
     这些问题的答案其实每个人的回答可能都不太相同，我想以我的过往经历来认真的回答一下，给大家一个参考。
    </p>
    <p class="img-center">
     <img alt="" height="214" src="https://i-blog.csdnimg.cn/direct/ffdb110c5c0544b2af8839fdf524b91a.png" width="235"/>
    </p>
    <ol>
     <li id="ue8c20c3a">
      程序员到底需不需要高学历？
     </li>
    </ol>
    <p id="uf77df17d">
     不可否认，程序员（尤其是前端）的门槛没有那么的高，不像有些职业，对于普通的岗位都设定为最低硕士，甚至是博士。程序员大部分基本上没有这个要求，在以前，有个大专、普通本科的学历就可以了，再会一点代码，应届生可以随随便便找个工作。
    </p>
    <p id="u4d618bd4">
    </p>
    <p id="ud755910a">
     但是到了现在，相信很多刚毕业的同学，已经经历过社会的第一次拷打了吧，应届生越来越难找工作，因为和你竞争的人太多了，一名专科和一名 985 毕业同学，谁的入职几率更大呢？我们没有学历歧视，但是现在的情况就是僧多粥少，企业是既要还要，现在就是一个全民降本增效的年代。
    </p>
    <p class="img-center">
     <img alt="" height="200" src="https://i-blog.csdnimg.cn/direct/dac2c3e40391452bbb93a1db2e2405cb.png" width="200"/>
    </p>
    <p id="u70a2c4f3">
     所以答案是：
     <span style="color:#1a439c;">
      <strong>
       不是一定要，但是尽量要
      </strong>
     </span>
     。虽然很多人在上学期间不知道读书的重要性，在工作之后才反应过来，然后勤奋苦学，通过自己努力也可能实现不小的成就，但是企业都是现实的，不会去赌你的未来，而且有可能你在一家企业只会呆上两三年，也等不到你实现成就，为公司带来价值。
    </p>
    <p id="ue1b09b16">
    </p>
    <p id="u460130f0">
     那么对于因为学历屡屡碰壁的同学该怎么办呢？只能前期猥琐发育，先找份工作安定下来，不眼高手低，先入行、后学习、再跳槽，当你有了很多的技术积累，还是能缩小很多差距的。
    </p>
    <p id="u3a697071">
     如果没有好项目，写在简历上不够亮眼，自己的技术积累又得不到展示，就只能另辟蹊径，让企业知道你的水平。比如去
     <span style="color:#1a439c;">
      <strong>
       贡献开源项目、开博客写文章、自己做项目、做个人网站、包装自己简历
      </strong>
     </span>
     等等，都是大家值得去试试的方法。总之，要想方设法让别人看到你的能力，给予你足够的正反馈，否则你会陷入对自己的怀疑：我学了那么多，为什么 Boss 还是已读不回，我还需要继续努力学习吗。
    </p>
    <p class="img-center">
     <img alt="" height="211" src="https://i-blog.csdnimg.cn/direct/aee3ea5a18e94dc9b3607742e0feaeff.png" width="238"/>
    </p>
    <p>
     学习相对来说本来就很痛苦，要是得不到企业的认可，很难保持动力一直坚持下去，所以一定要表现出来，至少可以争得很多面试机会，通过面试也可以检验一下自己的水平，还可以做到查缺补漏。
    </p>
    <p id="u60ae41a9">
    </p>
    <p>
     2. 程序员英文不好到底行不行？
    </p>
    <p id="udd614f56">
     我说的残酷点，不行！或者说
     <span style="color:#1a439c;">
      <strong>
       对于想要在技术生涯里有所成就的，英文一定要好，至少可以看懂英文文档、博客等
      </strong>
     </span>
     。
    </p>
    <p id="u14ddd759">
    </p>
    <p id="u49c1c558">
     有人可能会嗤之以鼻，反正我都是靠百度过来的，要是实在需要看懂英文的文档我机器翻译不就好了？
    </p>
    <p id="u172347c7">
    </p>
    <p id="u1cad1109">
     踩过坑的人都知道，很多时候，技术文档和博客，英文版都比中文版要更新更全更详细，也很少像国内一些文章那样鱼龙混杂，相对来说会更靠谱。我就遇到过一次，我平时都是看英文版的 MDN，不记得为什么有一次我看了中文版的，然后我的 bug 对照文档上的说明，根本没有问题，但是就是有 bug，调试了很长时间都不知道原因是什么，最后切换回了英文版，马上就看到了不一样的说明，英文版更加详细，立马就解决了我的问题。
    </p>
    <p id="u092e6616">
    </p>
    <p class="img-center">
     <img alt="" height="196" id="ud3d0291d" src="https://i-blog.csdnimg.cn/blog_migrate/ac384cbbc33f0379b5e80bcf545582f1.png" width="258"/>
    </p>
    <p id="u30d74d70">
    </p>
    <p id="u146fdab5">
     而且，全球🌐最大的同性交友网站
     <a href="https://github.com/" title="GitHub">
      GitHub
     </a>
     上绝大多数都是英文，全球🌐 最大的程序员问答网站
     <a href="https://stackoverflow.com/" rel="nofollow" title="Stack Overflow">
      Stack Overflow
     </a>
     也是如此，所以想要解决、查询或发帖询问一些前沿技术的问题，能读写英文还是非常有必要的。
    </p>
    <p id="u1683aa94">
    </p>
    <p class="img-center">
     <img alt="" height="260" id="u4d07e1c1" src="https://i-blog.csdnimg.cn/blog_migrate/f853edc359c75a4b5209cdd5b6f8bff3.png" width="460"/>
    </p>
    <p id="ue96bee3b">
    </p>
    <p id="u542bcca0">
     而且 GitHub 上的开源项目源码，基本上全都是英文，想让你的开源项目做到世界闻名，用英文是最基本的条件啦。像 Java 这样的语言，很多类、方法等都可以直接跳转到源码，并且会有很多注释，英文好的话阅读效率杠杠的。
    </p>
    <p id="u222caa9a">
    </p>
    <p id="u6cbee8f2">
     至于用机器翻译，个人认为还是没那么靠谱，很多专业性词汇使用机器翻译，大概率是离谱到没边儿的，终究没有自己能看得懂来的方便。
    </p>
    <p id="ubd029fc8">
    </p>
    <p id="uc3cb27c1">
     还有一点就是，大家经常会戏称，工作中大部分时间都用来取类、变量、函数名了（当然是玩笑啦），英文不太好的人需要每次都去翻译软件儿上查询，还是挺耽误时间的。更有甚者，有人变量名直接用拼音，别人看到了真的很降低专业性，就显得有点儿 low。
    </p>
    <p id="u8cca7ba6">
    </p>
    <p id="udeb35c56">
     所以，建议大家务必学好英文！！！后续我会发表的从底层官方规范讲起的系列文章也都尽量多带上一些英文，让大家多熟悉熟悉英文。
    </p>
    <p class="img-center">
     <img alt="" height="225" src="https://i-blog.csdnimg.cn/direct/a295fb7866bf47faab1aad3dfe47864f.png" width="225"/>
    </p>
    <p id="u12313a9b">
     这里也推荐一些高质量的英文网站，链接放在文末。
    </p>
    <p id="u66322cd8">
    </p>
    <p id="ufc635d92">
     再提一嘴，尽量用 Google，用过 Google 再去用百度，你会发现很多搜索结果都质量挺差的，只不过得科学上网啦。
    </p>
    <p class="img-center">
     <img alt="" height="236" src="https://i-blog.csdnimg.cn/direct/5c28b9ee963c426e9087c38ebcf8d0e1.png" width="426"/>
    </p>
    <p>
     3. 前端到底有没有前途？
    </p>
    <p id="uafe3228d">
     这个问题得分两种情况：
    </p>
    <ul>
     <li id="u7dfe948c">
      满足现状、对未来发展没有方向、觉得前端工作画画页面就可以了的工程师：很难在技术上有所为，这个世界太卷了，遍地都是培训机构，大家都是卷王，当你安心于当前的工作、不去思考未来的发展方向和学习路线，残酷的现实就是没有什么前途，分分钟就被淘汰了。所以我也建议对这一行不热爱，也不能做到坚持学习的人，还是尽量别选择做程序员了，会非常痛苦的。
     </li>
     <li id="u290e516a">
      有职业发展方向和学习路线的工程师：稳步发展，目标清晰，辛苦学习得来的技术积累，未来终究有一天会得到回报。这条路会很累，需要学很多东西，并且要一直保持学习，但也是最有前途、有竞争力的方向。我们可以做 Web、移动端、桌面应用、小程序、数据可视化、架构基建、全栈工程师等等，可选择的方向非常多。
     </li>
    </ul>
    <p id="ub7a40f47">
    </p>
    <p id="u6cb49c74">
     所以看到这篇文章，还在犹豫要不要入行的小伙伴，请先三思而后行～
    </p>
    <p id="uf7d96ec4">
    </p>
    <p class="img-center">
     <img alt="" height="288" id="u98a0ecf6" src="https://i-blog.csdnimg.cn/blog_migrate/d7be16fde5379da44264a246422f4f5c.png" width="288"/>
    </p>
    <p id="u341e9019">
    </p>
    <p>
     4. AI 时代来临，前端岗位会不会被取代？
    </p>
    <p id="ube660feb">
     不会，
     <span style="color:#1a439c;">
      <strong>
       目前 AI 最多会取代一些重复的机械性的简单前端工作
      </strong>
     </span>
     ，例如画一画简单的页面、调一调页面的样式。
    </p>
    <p id="u9e6e0955">
     对于复杂的有核心竞争力的岗位，目前来说还是不会有什么影响的，最多可以借助一下 AI，帮助提高效率，所以大家也不要太焦虑～
    </p>
    <p id="u493741b2">
    </p>
    <p id="u7bc5df0c">
     如果还是比较担心的话，可以尝试拓展自己的知识圈，例如去接触后端、运维，技多不压身，轮到淘汰也是淘汰别人。
    </p>
    <p id="uf02a4ea1">
    </p>
    <p class="img-center">
     <img alt="" height="478" id="JakLz" src="https://i-blog.csdnimg.cn/blog_migrate/851be698c9ca9be1d11f42b9054615a4.jpeg" width="395"/>
    </p>
    <h2>
    </h2>
    <h2 id="oQpbF">
     What to learn？
    </h2>
    <p id="u7a8a8fd0">
     一定要注意，在我们的职业发展中，一定不能只满足于实现了普通的需求，比如说工作内容是开发某某系统中的一些模块，这些模块虽然复杂，但是工作都是一样的内容：画页面。繁重的工作会让人觉得，我已经做了这么多模块了，我已经学到了很多了，但这样其实相当于是
     <strong>
      <code>
       同一个项目 * n
      </code>
     </strong>
     ，而不是
     <strong>
      <code>
       n 个不同的项目
      </code>
     </strong>
     。
    </p>
    <p id="u75f0eb6e">
    </p>
    <p class="img-center">
     <img alt="" height="512" id="uad73351d" src="https://i-blog.csdnimg.cn/blog_migrate/64ae7c63977eef1a0de07556ae6b4f9e.png" width="287"/>
    </p>
    <p id="u2d9f548e">
     在我们早期的职业生涯，很难有机会能遇到特别好的项目（我对于好的项目的定义是：有深度、有广度，能学到很多东西），经常只是写写组件、画画页面，甚至都不用太过考虑性能（我和代码，有一个能跑就行），更别提架构、工程化、前端高级应用（例如前端埋点、前端监控、白屏检测等）这些东西了。
    </p>
    <p id="udac7e1f0">
    </p>
    <p id="u7974a58c">
     久而久之，很多人就在一家公司里产生了舒适圈，做的都是完全一样的东西，没有任何冲出舒适圈的内容。这是国内环境催生出来的，尤其对于前端工程师，很多中小型公司在前端方面，并不需要做多少有深度的东西，他们要的是螺丝钉（说的残酷点就是纯纯工具人），能快速响应需求、迭代上线，性能方面只要不是很严重的问题，对于他们来说就够了。
    </p>
    <p id="uaf93054e">
    </p>
    <p id="ue759262a">
     我相信作为前端工程师，不管在学校还是在工作中，多多少少有时候会受到一些“歧视”，很多人对于前端的理解，仍然停留在“切图仔“的年代，这跟公司、前端工程师的职级、周边环境有很大关系，毕竟不会有人说尤雨溪、Dan Abramov 这类大佬是切图仔吧~
    </p>
    <p class="img-center">
     <img alt="" height="217" src="https://i-blog.csdnimg.cn/direct/ca81c9a6e3d24eb1b3855702bb3c1c05.png" width="232"/>
    </p>
    <p id="u1ce505e0">
     所以我们需要有危机意识，不能停留在画页面的舒适圈里。我见过工作两三年就对前后端各种原理、计算机相关知识、算法等等都非常熟练的人；也见过工作了十余年的人，还在跟刚毕业的同事一起做无聊重复的画页面的工作。在越来越严峻的大环境下，谁会更先被淘汰，无需多言。
    </p>
    <p id="u0778c868">
    </p>
    <p id="ub0741029">
     所以在最初的前两三年的工作里，可以只是画画页面，学习基础和框架，顺带着了解一些底层原理。再往后，已经不是初中级工程师了，要往高级应用的方向走，比如去做前端架构基建、前端埋点监控、前端截图、探索前端发展趋势、低代码无代码平台、大型组件库、游戏、3D 等等，这些方向都是需要运用到很多底层的相关知识，并且可能有一些方向没有很多现有的例子可以参考，需要一定的创新和突破。
    </p>
    <p class="img-center">
     <img alt="" height="244" src="https://i-blog.csdnimg.cn/direct/ed7d0e6df89a4038a61f3ae75e64f28a.png" width="347"/>
    </p>
    <p id="u4f9ddb2c">
     还可以让自己不仅仅局限在前端，可以和别的方向结合起来，
     <span style="color:#1a439c;">
      <strong>
       跨出前端的圈子，但是服务于前端
      </strong>
     </span>
     。举个例子，淘宝的代理服务器
     <code>
      Tengine
     </code>
     开发的
     <code>
      concat
     </code>
     模块，可以做到文件资源合并，比如页面有两个 css 文件，就需要发送两个 HTTP 请求，减少 HTTP 请求是很常见的性能优化的一种方式，通过
     <code>
      concat
     </code>
     模块就可以实现只发送一个 HTTP 请求，就可以把两个 css 文件都加载过来。该模块也开源给了原版
     <code>
      Nginx
     </code>
     ，以下是
     <a href="https://github.com/alibaba/nginx-http-concat" title="官方介绍">
      官方介绍
     </a>
     ：
    </p>
    <blockquote>
     <p id="u37ce232f">
      This is a module that is distributed with tengine which is a distribution of Nginx that is used by the e-commerce/auction site Taobao.com. This distribution contains some modules that are new on the Nginx scene. The
      <code>
       ngx_http_concat
      </code>
      module is one of them.
     </p>
     <p id="uab20260f">
     </p>
     <p id="u08a7035f">
      The module is inspired by Apache's modconcat. It follows the same pattern for enabling the concatenation. It uses two
      <code>
       ?
      </code>
      , like this:
     </p>
     <p id="u4fe918d5">
      <code>
       http://example.com/??style1.css,style2.css,foo/style3.css
      </code>
     </p>
    </blockquote>
    <p id="u0d88b0b5">
     举个代码例子：
    </p>
    <p id="ua89fd69d">
     在
     <code>
      Nginx
     </code>
     上下载完单独的
     <code>
      concat
     </code>
     模块后，在
     <code>
      location
     </code>
     中配置：
    </p>
    <pre id="ziE1S"><code>location /static/css {
  concat on;
  concat_max_files 20; # 一次最多请求多少文件数
}</code></pre>
    <p id="u3ccc11dc">
     并且在 HTML 文件里，原本是：
    </p>
    <pre id="ct1Nq"><code class="language-html">&lt;link href="a.css" rel="stylesheet" /&gt;
&lt;link href="b.css" rel="stylesheet" /&gt;</code></pre>
    <p id="ud5742932">
     只需要改成：
    </p>
    <pre id="qY6S3"><code class="language-html">&lt;link href="??a.css,b.css" rel="stylesheet" /&gt;</code></pre>
    <p id="u2d61d72e">
    </p>
    <p id="u5609c590">
     再举个例子，大家都知道
     <code>
      Webpack
     </code>
     ，它使用起来很方便，在以前是前端工程化必备的基建。但是随着时代的发展，前端代码量越来越庞大，
     <code>
      Webpack
     </code>
     这种使用 JS 单线程的语言，并且在构建时需要从入口文件打包整个文件依赖树，性能上其实挺差的。所以出现了
     <code>
      ESBuild
     </code>
     、
     <code>
      SWC
     </code>
     等工具来解决性能问题，
     <code>
      ESBuild
     </code>
     采用
     <code>
      go
     </code>
     语言编写，
     <code>
      SWC
     </code>
     采用
     <code>
      Rust
     </code>
     语言编写，通过别的语言的优势来弥补 JS 的劣势。
    </p>
    <p id="u47332603">
    </p>
    <p id="u2c200118">
     <strong>
      这就是典型的跨出前端，但服务于前端
     </strong>
     。越是到了职业生涯后期，越是可以考虑要不要通过突破前端来服务前端。
    </p>
    <p>
    </p>
    <h2 id="TgUAf">
     How to learn?
    </h2>
    <p id="u9d09beb8">
     最好的方式，就是能在日常工作中接触到上文所说的方向，你每天 8 个小时（可能更多😭）都在接触这些东西，对自己的提升是相当大的，也可能会成为这些领域的专家。
    </p>
    <p class="img-center">
     <img alt="" height="203" src="https://i-blog.csdnimg.cn/direct/fdac9958bf6f47ca904d887e064f7891.png" width="212"/>
    </p>
    <p>
    </p>
    <p id="u24542685">
     当然肯定有人是这样的：“我工作就是这些内容，根本接触不到那些有竞争力的项目，导致换工作的时候，有广度有深度的项目，还是轮不到我”。那么该如何破局呢？
    </p>
    <p id="ue989d5e8">
    </p>
    <p id="u5fa57195">
     确实想把这些感兴趣的方向塞到公司的项目里，基本上是不太可能的，所以就只能靠自己业余时间自己去动手了，多查阅些资料，多看看前人已经做好的相关的项目源码，最好是能结合起来自己动手写一个项目。亲自动手写项目所带来的收获是相当丰富的。
    </p>
    <p id="ue2bc0e34">
    </p>
    <p id="ufc9cbbae">
     但是我们每个人的学习时间都很有限，工作繁忙的时候在公司根本没有时间去学别的东西。所以我们要充分利用好碎片化时间，可以先把想看的网站、文档、公众号文章等先收藏起来，在工作中有碎片化时间了的时候可以迅速找到想看的东西，然后品鉴起来，就一个字儿：宣～
    </p>
    <p id="uf154661d">
    </p>
    <p class="img-center">
     <img alt="" height="254" id="u5558789d" src="https://i-blog.csdnimg.cn/blog_migrate/00e03d97d333b225ad1594912f4788d8.png" width="274"/>
    </p>
    <p id="u3be4cc39">
    </p>
    <p id="uc3a84a10">
     平时在地铁上、公交上、代码编译的时候、等同事回复消息的时候，这些碎片化的时间都可以用来看一看感兴趣的文章，工作了以后一定要尽量学会这个技能。
    </p>
    <p>
    </p>
    <h2 id="F0ub7">
     Where to learn?
    </h2>
    <p id="u7a58186d">
     在现在这么一个信息化爆炸的时代，不是很推荐有自制力的人去报班，学会运用 Google、AI、常用的一些网站，基本上大部分想要了解的东西都能找到相应的资源。
    </p>
    <p id="u06682d8a">
    </p>
    <p id="uee9a9e77">
     推荐一些学习网站：
    </p>
    <ul>
     <li id="uf9e36ccf">
      <a href="https://www.w3.org/" rel="nofollow" title="W3C">
       W3C
      </a>
      ：Web 标准全都在这里，HTML、CSS 标准和草稿都能在这里找到。
     </li>
     <li id="u3431b11e">
      <a href="https://tc39.es/ecma262/" rel="nofollow" title="ECMAScript Language Specification">
       ECMAScript Language Specification
      </a>
      ：最新最准确的 ECMAScript 规范，含有一切标准和处于 Stage 4 的提案。
     </li>
     <li id="u40cead0c">
      <a href="https://developer.mozilla.org/en-US/" rel="nofollow" title="MDN">
       MDN
      </a>
      ：前端开发者终极利器，基本上前端的基础技术、教程全都有，可以查看浏览器兼容情况。只不过有少数文章介绍的比较简单，有时还得再结合一下别的网站。
     </li>
     <li id="uc089e93b">
      <a href="https://codesandbox.io/" rel="nofollow" title="CodeSandbox">
       CodeSandbox
      </a>
      ：云编程网站，非常适合那些懒得本地启项目去学习的人，大多前端 UI、图表等框架都会有 CodeSandbox 的在线运行链接。缺点是代码调试起来稍稍有些麻烦。
     </li>
     <li id="u5f04da2e">
      <a href="https://web.dev/learn" rel="nofollow" title="谷歌 Web 开发指南">
       谷歌 Web 开发指南
      </a>
      ：里面含有很多基础教程，写的非常用心，配合了很多代码。
     </li>
     <li id="u6c719833">
      <a href="https://developer.chrome.com/docs/devtools/overview" rel="nofollow" title="Chrome Devtools">
       Chrome Devtools
      </a>
      ：谷歌官网介绍 Chrome Devtools 的使用。
     </li>
     <li id="u7c608761">
      <a href="https://developers.google.com/search/docs" rel="nofollow" title="谷歌爬虫官方文档">
       谷歌爬虫官方文档
      </a>
      ：详细介绍了什么是
      <code>
       SEO
      </code>
      ，如何优化谷歌
      <code>
       SEO
      </code>
      等，写的非常详细。
     </li>
     <li id="u195f5f04">
      <a href="https://es6.ruanyifeng.com/" rel="nofollow" title="ECMAScript 6 入门">
       ECMAScript 6 入门
      </a>
      ：阮一峰大佬的 ECMAScript 6 教程，适合爱看中文教程的小伙伴。
     </li>
     <li id="u712b2544">
      <a href="https://pomb.us/build-your-own-react/" rel="nofollow" title="Build your own React">
       Build your own React
      </a>
      ：交互很友好的 React 教程网站，带你写一个自己的 React。
     </li>
     <li id="u1bb7f92e">
      <a href="https://jex.im/regulex/#!flags=&amp;re=%5E%28a%7Cb%29*%3F%24" rel="nofollow" title="Regulex">
       Regulex
      </a>
      ：可视化 JS 的正则表达式执行过程，帮助理解正则表达式。
     </li>
     <li id="u7a188a9f">
      <a href="https://discoverthreejs.com/" rel="nofollow" title="Discover three.js">
       Discover three.js
      </a>
      ：学习 Three.js 的国外网站。
     </li>
     <li id="uf3a936b3">
      <a href="https://linuxcommand.org/index.php" rel="nofollow" title="Linux Command">
       Linux Command
      </a>
      ：学习 Linux 命令的网站，就是导航做的不是很好，如果你只是想快速找一下 Linux 对应的命令，链接在
      <a href="https://linuxcommand.org/lc3_man_page_index.php" rel="nofollow" title="这里">
       这里
      </a>
      。
      <a href="https://www.linuxcool.com/" rel="nofollow" title="这里">

       Medium
      </a>
