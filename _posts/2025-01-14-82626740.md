    <h3>
     <a name="t4">
     </a>
     2. APP的基本测试
    </h3>
    <h4>
     <a name="t5">
     </a>
     2.1 “移动”测试App
    </h4>
    <p>
     由于现在3G和4G网络都已经非常普及，需要优先测试App在这两种网络环境下的表现。同时需要测试4G信号无法覆盖的时候，App切换到3G、2G的环境下进行测试甚至到无网络的环境下测试。当网络恢复的时候，App是否会自动切换回来，并且执行网络断开前用户的操作。
    </p>
    <p>
     一般可以选择在地铁、公交、电梯、隧道灯代表性的场所进行测试。
    </p>
    <ul>
     <li>
      网络切换
     </li>
     <li>
      网络恢复
     </li>
    </ul>
    <p>
     一般可以选择在地铁、公交、电梯、隧道灯代表性的场所进行测试。
    </p>
    <p>
     测试APP在网络异常的情况下，是否发生
     <code>
      ANR
     </code>
     和
     <code>
      Crash
     </code>
    </p>
    <h4>
     <a name="t6">
     </a>
     2.2 App的多任务和意外情况处理
    </h4>
    <ul>
     <li>
      <p>
       多任务切换和多任务管理界面是否与App保持一致。
      </p>
     </li>
     <li>
      <p>
       多任务管理界面直接退出App程序测试。
      </p>
     </li>
     <li>
      <p>
       测试App的同时接听电话，是否还能够继续使用App。
      </p>
      <p>
       步骤：
      </p>
      <ol>
       <li>
        正常打开被测APP
       </li>
       <li>
        运行APP的功能
       </li>
       <li>
        突然被其他应用打断（意外，短信、电话、通知）
       </li>
       <li>
        切换到该应用、或者、忽略该应用
       </li>
       <li>
        被测APP能够继续之前的操作，不发生
        <code>
         ANR
        </code>
        或者
        <code>
         Crash
        </code>
       </li>
      </ol>
     </li>
    </ul>
    <h4>
     <a name="t7">
     </a>
     2.3 测试App的手势操作
    </h4>
    <ul>
     <li>
      长按屏幕呼出菜单
     </li>
     <li>
      单手指从右向左滑动，弹出附加功能项
     </li>
     <li>
      双手指捏合，放大放小操作等
     </li>
    </ul>
    <h4>
     <a name="t8">
     </a>
     2.4 测试App的用户体验
    </h4>
    <ul>
     <li>
      横屏功能测试
     </li>
     <li>
      遵循iOS和Android的设计规范
     </li>
     <li>
      字体大小设置以及美观
     </li>
    </ul>
    <h4>
     <a name="t9">
     </a>
     2.5 测试App的消息显示和通知显示*
    </h4>
    <p>
     通知显示的方式：
    </p>
    <ul>
     <li>
      锁屏的通知：内容
     </li>
     <li>
      下拉通知栏的通知：内容
     </li>
     <li>
      应用程序消息栏：内容
     </li>
     <li>
      图标的右上角角标：条数
     </li>
    </ul>
    <p>
     测试要点
    </p>
    <ul>
     <li>
      在安装App的时候是否申请相应的权限
     </li>
     <li>
      验证收到App新的消息的时候，如何通过通知向用户进行消息展示
     </li>
    </ul>
    <h4>
     <a name="t10">
     </a>
     2.6 测试App能否及时显示和同步数据
    </h4>
    <p>
     Web端 + PC端 + APP端
    </p>
    <ul>
     <li>
      测试在多台设备登录App，是否同步消息、设置、内容、数据等
     </li>
     <li>
      测试App的消息是否同步在PC端（无论Web版本或者PC版本）
     </li>
    </ul>
    <h4>
     <a name="t11">
     </a>
     2.7 测试App能否响应不同的设备用户界面
    </h4>
    <p>
     测试App是否适应用户界面，必须使用真实设备进行测试。比如测试HTC Sense用户界面底部的黑色导航栏，测试小米MIUI系统桌面的角标提醒功能，测试iOS的角标提醒，测试魅族Flyme的SmartBar操作等。
    </p>
    <ul>
     <li>
      非常规的控制：SmartBar
     </li>
     <li>
      非常规的分辨率：魅族 MX4 MX3 MX2……
     </li>
    </ul>
    <h4>
     <a name="t12">
     </a>
     2.8 测试App中高内存使用的功能
    </h4>
    <ul>
     <li>
      <p>
       测试App对于读取大量图片、视频等进行高内存占用操作的处理能力
      </p>
     </li>
     <li>
      <p>
       查看App是否有对应的异步加载功能等
      </p>
      <p>
       主要测试场景：
      </p>
      <ul>
       <li>
        检查APP是否读入全部图片到内存中
       </li>
       <li>
        美图秀秀打开图片
       </li>
       <li>
        读取一屏的图片到内存
       </li>
       <li>
        滚动屏幕，加载新的图片到内存
       </li>
       <li>
        测试APP是否设定了图片加载的阀值：比如300MB
       </li>
       <li>
        图片在内存中加载到阀值（300MB），会自动把最开始加载的指定图片占的内存空间释放
       </li>
      </ul>
     </li>
    </ul>
    <h4>
     <a name="t13">
     </a>
     2.9 测试App的流量和电量消耗
    </h4>
    <ul>
     <li>
      测试App的安装文件大小
     </li>
     <li>
      测试App占用的存储空间
     </li>
     <li>
      测试App使用的流量和电量
     </li>
     <li>
      流量靠软件测试
     </li>
     <li>
      电量测试靠功耗仪
     </li>
    </ul>
    <h4>
     <a name="t14">
     </a>
     2.10 测试App的增量升级
    </h4>
    <ul>
     <li>
      需要测试App在自动升级或者手动增量升级后，用户数据是否保存延续等
     </li>
     <li>
      测试App在升级失败以后，是否可以继续使用升级之前的版本。
     </li>
     <li>
      重点测试升级后数据是否可用
     </li>
    </ul>
    <h4>
     <a name="t15">
     </a>
     2.11 测试App支持的文件格式
    </h4>
    <ul>
     <li>
      测试App支持Office文件操作
     </li>
     <li>
      测试App支持图片文件展示
     </li>
     <li>
      测试App支持PDF文件打开
     </li>
     <li>
      测试App支持视频和音频文件播放
     </li>
    </ul>
    <h4>
     <a name="t16">
     </a>
     2.12 安装卸载测试
    </h4>
    <ul>
     <li>
      不联网的情况下，会不会崩溃
     </li>
     <li>
      在线下载数据不完整，能否再次启动
     </li>
     <li>
      是否支持断点续传
     </li>
     <li>
      下载安装后再次启动
     </li>
     <li>
      安装成功后检查版本号和相关数据
     </li>
    </ul>
    <h4>
     <a name="t17">
     </a>
     2.13 应用召唤
    </h4>
    <ul>
     <li>
      <p>
       不同应用之间的应用召唤

      <p>
       测试步骤：
      </p>
      <p>
       【支付宝支付】
      </p>

       <li>

     4. ADB命令与Monkey工具
    </h3>
    <h4>
     <a name="t20">
     </a>
     4.1 ADB 命令
    </h4>
    <blockquote>
     <p>
      adb:
     </p>
     <p>
      adb 的全称为Android Debug Bridge，就是起到调试桥的作用。通过adb我们可以在Eclipse中方面通过DDMS来调试Android程序，说白了就是debug工具。adb的工作方式比较特殊，采用监听Socket TCP 5554等端口的方式让IDE和Qemu通讯，默认情况下adb会daemon相关的网络端口，所以当我们运行Eclipse或Android Studio时adb进程就会自动运行。
     </p>
     <p>
      借助adb工具，我们可以管理设备或手机模拟器的状态。还可以进行很多手机操作，如安装软件、系统升级、运行shell命令等等。其实简而言说，adb就是连接Android手机与PC端的桥梁，可以让用户在电脑上对手机进行全面的操作
     </p>
    </blockquote>
    <p>
     adb 在sdk的platform-tools目录中
    </p>
    <p>
     Android SDK 下载网站
     <a href="https://link.jianshu.com/?t=http://www.androiddevtools.cn/" rel="nofollow">
      http://www.androiddevtools.cn/
     </a>
     （非官方）

    <h4>
     <a name="t21">
     </a>
     4.2 Monkey 工具
    </h4>
    <blockquote>
     <p>
      Monkey测试是Android平台自动化测试的一种手段，通过Monkey程序模拟用户触摸屏幕、滑动Trackball、按键等操作来对设备上的程序进行压力测试，检测程序多久的时间会发生异常。
     </p>
    </blockquote>
    <p>
     顾名思义，Monkey就是猴子， Monkey测试，就像一只猴子， 在电脑面前，乱敲键盘在测试。 猴子什么都不懂， 只知道乱敲，通过Monkey程序模拟用户触摸屏幕、滑动Trackball、 按键等操作来对设备上的程序进行压力测试，检测程序多久的时间会发生异常
    </p>
    <p>
     Monkey 主要用于Android 的压力测试 自动的一个压力测试小工具， 主要目的就是为了测试app 是否会Crash.
    </p>
    <ol>
     <li>
      Monkey程序由Android系统自带，使用Java语言写成，在Android文件系统中的存放路径是：
      <code>
       /system/framework/monkey.jar
      </code>
      ；
     </li>
     <li>
      Monkey.jar程序是由一个名为“monkey”的Shell脚本来启动执行，shell脚本在Android文件系统中的存放路径是：
      <code>
       /system/bin/monkey
      </code>
      ；
     </li>
    </ol>
    <p>
     可以在命令行中输入
    </p>
    <p>
     这样就可以通过在CMD窗口中执行:
     <code>
      adb shell monkey ｛+命令参数｝
     </code>
     来进行Monkey测试了。
    </p>

        通常，当应用程序发生任何超时错误(如“Application Not Responding”对话框)时，Monkey将停止运行。如果设置此选项，Monkey将继续向系统发送事件，直到计数完成。
       </td>

       </td>
       <td>

       </td>
       <td>
       </td>
      </tr>
      <tr>
       <td>
        --kill-process-after-error
       </td>
       <td>

       </td>
       <td>
       </td>
      </tr>
      <tr>
       <td>
        --monitor-native-crashes
       </td>
       <td>
        监视并报告Android系统中本地代码的崩溃事件。如果设置了--kill-process-after-error，系统将停止运行。

       <td>
       </td>

       <td>
        --wait-dbg
       </td>
       <td>

       </td>
       <td>
       </td>
      </tr>
