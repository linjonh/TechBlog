    <p>
     在软件开发过程中，版本控制系统（如 Git）是不可或缺的工具之一。它帮助我们追踪代码的变更历史，管理不同版本的代码，以及协作开发。然而，随着项目的推进，提交记录可能会变得庞大且复杂，使得查找特定类型的提交变得困难。
     <br/>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/direct/aec648f28d4a4f1d857d8b96976cdaba.png#pic_center"/>
    </p>
    <h4>
     <a id="Git_Log__19">
     </a>
     Git Log 命令基础
    </h4>
    <p>
     <code>
      git log
     </code>
     命令是 Git 中用于查看提交历史的最基本命令。它提供了多种参数来定制输出结果，例如：
    </p>
    <ul>
     <li>
      <code>
       --oneline
      </code>
      ：将每个提交压缩为一行显示。
     </li>
     <li>
      <code>
       --decorate
      </code>
      ：显示分支名和标签。
     </li>
     <li>
      <code>
       --graph
      </code>
      ：显示 ASCII 图形表示的分支结构。
     </li>
    </ul>
    <h4>
     <a id="_27">
     </a>
     过滤提交记录的需求
    </h4>
    <p>
     在团队协作的项目中，我们经常需要关注特定类型的提交，比如功能更新、修复 bug 或者文档更改等。有时，我们也需要排除某些类型的提交，比如自动化的前端构建更新，以便专注于代码逻辑的变更。这时，
     <code>
      --grep
     </code>
     参数就显得尤为重要。
    </p>
    <h4>
     <a id="grepinvertgrep_31">
     </a>
     使用
     <code>
      --grep
     </code>
     和
     <code>
      --invert-grep
     </code>
     参数
    </h4>
    <p>
     <code>
      --grep
     </code>
     参数允许我们搜索包含特定关键词的提交信息。而
     <code>
      --invert-grep
     </code>
     参数则反转
     <code>
      --grep
     </code>
     的结果，即显示不包含指定关键词的提交。这两个参数的结合使用，可以帮助我们精确地过滤出想要的提交记录。
    </p>
    <h4>
     <a id="_35">
     </a>
     命令示例
    </h4>
    <p>
     假设我们需要排除所有包含“前端构建更新”的提交记录，可以使用以下命令：
    </p>
    <pre><code class="prism language-bash"><span class="token function">git</span> log --no-merges <span class="token parameter variable">--grep</span><span class="token operator">=</span><span class="token string">'前端构建更新'</span> --invert-grep
</code></pre>
    <p>
     这个命令的参数解释如下：
    </p>
    <ul>
     <li>
      <code>
       git log
      </code>
      ：显示提交历史。
     </li>
     <li>
      <code>
       --no-merges
      </code>
      ：排除合并提交，专注于代码变更。
     </li>
     <li>
      <code>
       --grep='前端构建更新'
      </code>
      ：搜索提交信息中包含“前端构建更新”的提交。
     </li>
     <li>
      <code>
       --invert-grep
      </code>
      ：反转
      <code>
       --grep
      </code>
      的结果，即排除包含指定关键词的提交。
     </li>
    </ul>
    <h4>
     <a id="_50">
     </a>
     参数详解
    </h4>
    <ul>
     <li>
      <p>
       <strong>
        <code>
         --no-merges
        </code>
       </strong>
       ：这个参数用于排除合并提交，因为合并提交通常包含多个变更，可能不是我们关注的重点。
      </p>
     </li>
     <li>
      <p>
       <strong>
        <code>
         --grep
        </code>
       </strong>
       ：这个参数后面跟的是我们要搜索的关键词。在这个例子中，我们搜索的是“前端构建更新”。Git 会显示所有包含这个关键词的提交记录。
      </p>
     </li>
     <li>
      <p>
       <strong>
        <code>
         --invert-grep
        </code>
       </strong>
       ：这个参数的作用是反转
       <code>
        --grep
       </code>
       的结果。也就是说，它会显示所有不包含“前端构建更新”关键词的提交记录。
      </p>
     </li>
    </ul>
    <h4>
     <a id="_57">
     </a>
     实际应用
    </h4>
    <p>
     在实际工作中，我们可能需要根据项目的不同阶段和需求，调整
     <code>
      --grep
     </code>
     后面的关键词。例如，如果我们想要查看所有与“性能优化”相关的提交，可以将命令修改为：
    </p>
    <pre><code class="prism language-bash"><span class="token function">git</span> log --no-merges <span class="token parameter variable">--grep</span><span class="token operator">=</span><span class="token string">'性能优化'</span>
</code></pre>
    <p>
     这样，我们就可以得到所有包含“性能优化”关键词的提交记录，而忽略其他不相关的提交。
    </p>
    <h4>
     <a id="_67">
     </a>
     高级用法
    </h4>
    <p>
     除了基本的
     <code>
      --grep
     </code>
     和
     <code>
      --invert-grep
     </code>
     参数，
     <code>
      git log
     </code>
     还支持更高级的搜索和过滤功能，比如：
    </p>
    <ul>
     <li>
      <strong>
       <code>
        --author
       </code>
      </strong>
      ：根据作者过滤提交。
     </li>
     <li>
      <strong>
       <code>
        --since
       </code>
      </strong>
      和
      <strong>
       <code>
        --until
       </code>
      </strong>
      ：根据时间范围过滤提交。
     </li>
     <li>
      <strong>
       <code>
        --committer
       </code>
      </strong>
      ：根据提交者过滤提交。
     </li>
    </ul>
    <p>
     结合这些参数，我们可以创建更复杂的查询，以满足不同的需求。
    </p>
