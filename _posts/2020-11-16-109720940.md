    <h3>
     <span style="color:#3399ea;">
      1，数据库设计概述
     </span>
    </h3>
    <h4>
     <span style="color:#3399ea;">
      1.1，数据库设计的基本概念
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#ff0000;">
       <strong>
        数据库设计
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        是指对于一个给定的应用环境，构造（设计）优化的数据库逻辑模式和物理结构，并据此建立数据库及其应用系统，使之能够有效地存储和管理数据，满足各种用户的应用需求，包括信息管理要求和数据操作
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        要求。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        数据库设计的目标：
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        是为用户和各种应用系统提供一个信息基础设施和高效率的运行环境
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        数据库设计的基本任务：
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        是根据用户的信息需求、处理需求和数据库的支持环境
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        (
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        包括硬件、操作系统和
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        DBMS)
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        ，设计出数据库模式
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        (
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        包括外模式、逻辑模式和内模式
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        )
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        及其典型的应用程序
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        。
       </strong>
      </span>
     </p>
    </blockquote>
    <h4>
     <span style="color:#3399ea;">
      1.2，数据库设计的方法
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#f33b45;">
       <strong>
        直观设计法（手工试凑发）：
       </strong>
      </span>
      <span style="color:#000000;">
       数据库
      </span>
      <span style="color:#000000;">
       设计只是一种经验的反复实施，而不能称为是一门科学，缺乏科学分析理论基础和工程手段的支持，因为设计质量与设计人员的经验和水平有直接
      </span>
      <span style="color:#000000;">
       关系，所以设计质量
      </span>
      <span style="color:#000000;">
       很难保证
      </span>
      <span style="color:#000000;">
       。具有
      </span>
      <span style="color:#000000;">
       周期短、效率高、操作简便、易于实现等优点
      </span>
      <span style="color:#000000;">
       。主要
      </span>
      <span style="color:#000000;">
       是用于
      </span>
      <span style="color:#000000;">
       简单小型
      </span>
      <span style="color:#000000;">
       系统。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        规范设计法：
       </strong>
      </span>
      <span style="color:#000000;">
       将
      </span>
      <span style="color:#000000;">
       数据库设计分为若干阶段，明确规定各阶段的任务，采用“自顶向下、分层实现、逐步求精”的设计原则，结合数据库理论和软件工程设计方法，实现设计过程的每一细节，最终完成整个设计任务。（新奥尔良
      </span>
      <span style="color:#000000;">
       方法、基于E-R
      </span>
      <span style="color:#000000;">
       模型的数据库设计方法、基于3NF
      </span>
      <span style="color:#000000;">
       （第三范式）的设计方法、面向对象的数据库设计方法、统一建模语言（UML
      </span>
      <span style="color:#000000;">
       ）
      </span>
      <span style="color:#000000;">
       方法）。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        计算机辅助设计法：
       </strong>
      </span>
      <span style="color:#000000;">
       在数据库设计的某些过程中，利用计算机和一些辅助设计工具，模拟某一规范设计方法，并以人的知识或经验为主导，通过人机交互方式实现设计中的某些部分。 （Oracle 公司开发的
      </span>
      <span style="color:#000000;">
       Designer、Sybase公司开发的
      </span>
      <span style="color:#000000;">
       PowerDesigner）。
      </span>
     </p>
    </blockquote>
    <h4>
     <span style="color:#3399ea;">
      1.3，数据库设计的基本步骤
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#f33b45;">
       <strong>
        需求分析：
       </strong>
      </span>
      <span style="color:#000000;">
       通过
      </span>
      <span style="color:#000000;">
       详细调查现实世界要处理的对象（组织、部门、企业等），充分了解原系统（手工系统或计算机系统）工作概况，明确用户的各种需求。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        概念结构设计：
       </strong>
      </span>
      <span style="color:#000000;">
       通过
      </span>
      <span style="color:#000000;">
       对用户需求进行综合、归纳与抽象，形成一个独立于具体数据库管理系统的概念模型。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        逻辑结构设计：
       </strong>
      </span>
      <span style="color:#000000;">
       将
      </span>
      <span style="color:#000000;">
       概念结构转换为某个数据库管理系统所支持的数据模型，并对其进行优化。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        物理结构设计：
       </strong>
      </span>
      <span style="color:#000000;">
       为
      </span>
      <span style="color:#000000;">
       逻辑数据结构选取一个最适合应用环境的物理
      </span>
      <span style="color:#000000;">
       结构
      </span>
      <span style="color:#000000;">
       ,
      </span>
      <span style="color:#000000;">
       包括
      </span>
      <span style="color:#000000;">
       存储结构和存取方法。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        数据库实施：
       </strong>
      </span>
      <span style="color:#000000;">
       根据
      </span>
      <span style="color:#000000;">
       逻辑设计和物理设计的结果构建
      </span>
      <span style="color:#000000;">
       数据库
      </span>
      <span style="color:#000000;">
       ,
      </span>
      <span style="color:#000000;">
       编写
      </span>
      <span style="color:#000000;">
       与调试
      </span>
      <span style="color:#000000;">
       应用程序
      </span>
      <span style="color:#000000;">
       ,
      </span>
      <span style="color:#000000;">
       组织
      </span>
      <span style="color:#000000;">
       数据入库并进行试运行。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        数据库运行和维护：
       </strong>
      </span>
      <span style="color:#000000;">
       经过
      </span>
      <span style="color:#000000;">
       试运行后即可投入正式
      </span>
      <span style="color:#000000;">
       运行
      </span>
      <span style="color:#000000;">
       ,
      </span>
      <span style="color:#000000;">
       在
      </span>
      <span style="color:#000000;">
       运行过程中必须不断对其进行评估、调整与修改。
      </span>
     </p>
     <p>
      <span style="color:#7c79e5;">
       ★需求分析和概念设计独立于任何数据库管理系统
      </span>
     </p>
     <p>
      <span style="color:#7c79e5;">
       ★逻辑设计和物理设计与选用的数据库管理系统密切相关
      </span>
     </p>
    </blockquote>

    <h3>
     <span style="color:#3399ea;">
      2，需求分析
     </span>
    </h3>
    <h4>
     <span style="color:#3399ea;">
      2.1，需求分析及其任务
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#f33b45;">
       <strong>
        需求分析就是分析用户的需求：
       </strong>
      </span>
      <span style="color:#000000;">
       是
      </span>
      <span style="color:#000000;">
       设计数据库的
      </span>
      <span style="color:#000000;">
       起点，结果
      </span>
      <span style="color:#000000;">
       是否准确地反映了用户的实际要求，将直接影响到后面各个阶段的设计，并影响到设计结果是否合理和实用。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        需求分析的任务：
       </strong>
      </span>
      <span style="color:#000000;">
       由
      </span>
      <span style="color:#000000;">
       数据库设计人员和用户双方共同收集信息需求和处理需求；通过仔细分析；将这些需求按一定的规范要求以用户和设计人员都能理解接受的文档形式确定下来。
      </span>
     </p>
    </blockquote>
    <h4>
     <span style="color:#3399ea;">
      2.2，需求分析的方法
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#f33b45;">
       <strong>
        需求分析的三个步骤：
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ①
       <strong>
        需求调查:
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
       </strong>
      </span>
      <span style="color:#000000;">
       收集需求信息
      </span>
      <span style="color:#000000;">
       ,
      </span>
      <span style="color:#000000;">
       调查清楚用户的实际要求
      </span>
      <span style="color:#000000;">
       ,
      </span>
      <span style="color:#000000;">
       与用户达成共识。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ②
       <strong>
        分析、整理和表达这些需求信息，形成需求说明书（例如，包括DFD和DD等）。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ③
       <strong>
        评审：
       </strong>
      </span>
      <span style="color:#000000;">
       由主管部门和专家评价、审批。
      </span>
     </p>
    </blockquote>
    <blockquote>
     <p>
      <span style="color:#f33b45;">
       <strong>
        需求调查
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#7c79e5;">
       <strong>
        需求调查的目的：
       </strong>
      </span>
      <span style="color:#000000;">
       主要
      </span>
      <span style="color:#000000;">
       是了解企业的组织机构设置
      </span>
      <span style="color:#000000;">
       ,
      </span>
      <span style="color:#000000;">
       各个组织机构的职能、工作目标、职责范围，主要业务活动及大致工作流程，获得各个组织机构的业务数据及其相互联系的信息，为分析整理工作做好前期基础工作
      </span>
      <span style="color:#000000;">
       。
      </span>
     </p>
     <p>
      <span style="color:#7c79e5;">
       <strong>
        需求调查的内容：
       </strong>
      </span>
      ①
      <span style="color:#000000;">
       组织
      </span>
      <span style="color:#000000;">
       机构的情况
      </span>
      <span style="color:#000000;">
       :
      </span>

    <h4>
     <span style="color:#3399ea;">
      4.3，数据模型的优化
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#000000;">
       <strong>
        数据库
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        逻辑设计的结果不是唯一
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        的，得到
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        初步数据模型后，还应该适当地修改、调整数据模型的结构，以进一步提高数据库应用系统的性能，这就是数据模型的优化。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ★
       <strong>
        规范化过程可分为两个步骤：确定范式级别，实施规范化处理
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ①
       <strong>
        确定数据依赖
       </strong>
       ：
      </span>
      <span style="color:#000000;">
       写出每个关系模式内部各属性之间的数据依赖；写出不同关系模式的属性
      </span>
      <span style="color:#000000;">
       (
      </span>
      <span style="color:#000000;">
       外码和主码
      </span>
      <span style="color:#000000;">
       )之间的数据依赖；
      </span>
     </p>
     <p class="img-center">
      <img alt="" height="150" src="https://i-blog.csdnimg.cn/blog_migrate/073fb1b5f8531226b72f37ebc9b796ec.png" width="500"/>
     </p>
     <p>
      <span style="color:#f33b45;">
       ②
       <strong>
        对于各个关系模式之间的数据依赖进行极小化处理，消除冗余的联系。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ③
       <strong>
        按照数据依赖的理论对关系模式进行分析，考察是否存在部分函数依赖、传递函数依赖、多值依赖等，确定各关系模式分别属于第几范式。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ④
       <strong>
        按照需求分析阶段得到的各种应用对数据处理的要求，分析对于这样的应用环境这些模式是否合适，确定是否要对它们进行合并或分解。（
       </strong>
      </span>
      <span style="color:#7c79e5;">
       并不是规范化程度越高的关系就越优。当一个应用的查询中经常涉及到两个或多个关系模式的属性时，系统必须经常地进行连接运算，而连接运算的代价是相当高的，可以说关系模型低效的主要原因就是做连接运算引起的，因此在这种情况下，第二范式甚至第一范式也许是最好的。
      </span>
      <span style="color:#f33b45;">
       <strong>
        ）
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ⑤
       <strong>
        对关系模式进行必要的分解，提高数据操作的效率和存储空间的利用率。常用的两种分解方法是
       </strong>
      </span>
      <span style="color:#7c79e5;">
       <strong>
        水平分解
       </strong>
      </span>
      <span style="color:#f33b45;">
       <strong>
        和
       </strong>
      </span>
      <span style="color:#7c79e5;">
       <strong>
        垂直分解
       </strong>
      </span>
      <span style="color:#f33b45;">
       <strong>
        。
       </strong>
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#7c79e5;">
        <strong>
         水平分解：
        </strong>
       </span>
       <span style="color:#000000;">
        把
       </span>
       <span style="color:#000000;">
        (
       </span>
       <span style="color:#000000;">
        基本
       </span>
       <span style="color:#000000;">
        )
       </span>
       <span style="color:#000000;">
        关系的元组分为若干子集合，定义每个子集合为一个子关系，以提高系统的效率。
       </span>
      </li>
      <li>
       <span style="color:#7c79e5;">
        <strong>
         垂直分解：
        </strong>
       </span>
       <span style="color:#000000;">
        把关系模式
       </span>
       <span style="color:#000000;">
        R
       </span>
       <span style="color:#000000;">
        的属性分解为若干子集合，形成若干子关系模式。
       </span>
      </li>
     </ul>
    </blockquote>

       <a href="https://shao12138.blog.csdn.net/article/details/109584275#t23" rel="nofollow" title="数据库：SQL语言，索引，连接查询，视图_燕双嘤的博客-CSDN博客">
        数据库：SQL语言，索引，连接查询，视图_燕双嘤的博客-CSDN博客
       </a>
      </strong>
     </p>
    </blockquote>
    <h3>
     <span style="color:#3399ea;">
      5，物理结构设计
     </span>
    </h3>
    <blockquote>
     <p>
      <span style="color:#000000;">
       <strong>
        数据库的物理结构：
       </strong>
       数据库
      </span>
      <span style="color:#000000;">
       在物理设备上的存储结构与存取方法称为数据库的物理结构，它依赖于选定的数据库管理系统。
      </span>
     </p>
     <p>
      <span style="color:#000000;">
       <strong>
        数据库的物理
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        设计：
       </strong>
      </span>
      <span style="color:#000000;">
       为
      </span>
      <span style="color:#000000;">
       一个给定的逻辑数据模型选取一个最适合应用要求的物理结构的过程，就是数据库的物理设计。
      </span>
     </p>
     <p>
      <strong>
       <span style="color:#000000;">
        数据库物理设计的步骤：
       </span>
      </strong>
      ①
      <span style="color:#000000;">
       确定数据库的物理
      </span>
      <span style="color:#000000;">
       结构，在
      </span>
      <span style="color:#000000;">
       关系数据库中主要指存取方法和存储结构
      </span>
      <span style="color:#000000;">
       ;
      </span>
      ②
      <span style="color:#000000;">
       对
      </span>
      <span style="color:#000000;">
       物理结构进行
      </span>
      <span style="color:#000000;">
       评价，评价
      </span>
      <span style="color:#000000;">
       的重点是时间和空间
      </span>
      <span style="color:#000000;">
       效率；若评价
      </span>
      <span style="color:#000000;">
       结果满足原设计要求，则可进入到物理
      </span>
      <span style="color:#000000;">
       实施阶段
      </span>
      <span style="color:#000000;">
       。否则，就需要重新设计或修改物理结构，
      </span>
      <span style="color:#000000;">
       有时甚至
      </span>
      <span style="color:#000000;">
       要返回逻辑设计阶段修改数据模型
      </span>
      <span style="color:#000000;">
       。
      </span>
     </p>
    </blockquote>
    <h4>
     <span style="color:#3399ea;">
      5.1，数据库物理设计的内容和方法
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#7c79e5;">
       <strong>
        准备工作：
       </strong>
      </span>
      ①
      <span style="color:#000000;">
       要充分了解应用环境，详细分析要运行的事务。以获得选择物理数据库设计所需要的参数。分析数据库查询事务需要的信息、数据更新事务需要的信息、每个事务在各关系上运行的频率和性能要求等。
      </span>
      ②
      <span style="color:#000000;">
       要充分了解所用的
      </span>
      <span style="color:#000000;">
       DBMS
      </span>
      <span style="color:#000000;">
       的内部特征
      </span>
      <span style="color:#000000;">
       ,
      </span>
      <span style="color:#000000;">
       特别是系统提供的存取方法和存储结构。
      </span>
     </p>
     <p>
      <strong>
       <span style="color:#7c79e5;">
        内容：
       </span>
      </strong>
      ①
      <span style="color:#000000;">
       为关系模式选择存取方法，即要确定选择哪些存取方法，建立哪些存取路径。
      </span>
      ②
      <span style="color:#000000;">
       设计关系(
      </span>
      <span style="color:#000000;">
       表
      </span>
      <span style="color:#000000;">
       )
      </span>
      <span style="color:#000000;">
       、聚簇、索引、日志、备份等数据的物理存储结构。
      </span>
     </p>
    </blockquote>
    <h4>
     <span style="color:#3399ea;">
      5.2，关系模式存取方法选择
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#000000;">
       <strong>
        数据库系统是多用户共享的系统，对同一个关系要建立多条存取路径才能满足多用户的多种应用要求。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#7c79e5;">
       <strong>
        数据库关系系统：
       </strong>
      </span>
      ①
      <span style="color:#000000;">
       <strong>
        B
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        +
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        树索引存取方法
       </strong>
      </span>
      ；②
      <span style="color:#000000;">
       <strong>
        Hash
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        索引存取方法
       </strong>
      </span>
      ；③
      <span style="color:#000000;">
       <strong>
        聚簇
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        存取方法；
       </strong>
      </span>
     </p>
    </blockquote>
    <blockquote>
     <p>
      <span style="color:#f33b45;">
       <strong>
        B+树索引
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#000000;">
       <strong>
        选择索引存取方法 就是根据应用要求
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        确定
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        ：
       </strong>
      </span>
      <span style="color:#000000;">
       对哪些属性列建立索引、对哪些属性列建立组合索引、对哪些索引要设计为唯一索引。
      </span>
     </p>
     <p>
      <strong>
       <span style="color:#000000;">
        选择索引存取方法的一般规则：
       </span>
      </strong>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        如果一个（或一组）属性经常在查询条件中出现，则
       </span>
       <span style="color:#000000;">
        考虑在这个（或这组）属性上建立索引（或
       </span>
       <span style="color:#000000;">
        组合索引
       </span>
       <span style="color:#000000;">
        ）。
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        如果一个属性经常作为最大值和最小值等聚集函数的
       </span>
       <span style="color:#000000;">
        参数，则考虑在这个属性上建立
       </span>
       <span style="color:#000000;">
        索引。
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        如果一个（或一组）属性经常在连接操作的连接
       </span>
       <span style="color:#000000;">
        条件
       </span>
       <span style="color:#000000;">
        中 出现，则考虑在这个（或这组）属性上建立
       </span>
       <span style="color:#000000;">
        索引。
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#ff0000;">
       关系上定义的索引数过多会带来较多的额外开销，无论是维护还是
      </span>
      <span style="color:#ff0000;">
       查找
      </span>
     </p>
     <p>
      <span style="color:#ff0000;">
       <strong>
        HASH存取方法
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#000000;">
       <strong>
        选择
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        Hash
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        存取方法的
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        规则：
       </strong>
      </span>
      <span style="color:#000000;">
       如果
      </span>
      <span style="color:#000000;">
       一个关系的属性主要出现在等值连接条件中或主要出现在等值比较选择条件中，而且满足下列两个条件之一
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        该关系的大小可预知，而且不变；
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        该关系的大小动态改变，但所选用的数据库管理系统提供了动态Hash
       </span>
       <span style="color:#000000;">
        存取方法。
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#f33b45;">
       <strong>
        聚簇存取方法
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#000000;">
       <strong>
        为了提高某个属性（或属性组）的查询速度，把这个或这些属性（称为聚簇码）上具有相同值的元组集中存放在连续的物理块上，称为
       </strong>
      </span>
      <span style="color:#ff0000;">
       <strong>
        聚簇
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        。该
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        属性（或属性组）称为聚簇码
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        。许多
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        关系型数据库管理系统都提供了聚簇
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        功能。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        聚簇索引：
       </strong>
      </span>
      <span style="color:#000000;">
       建立
      </span>
      <span style="color:#000000;">
       聚簇索引后，基表中数据也需要按指定的聚簇属性值的升序或降序存放。也即
      </span>
      <strong>
       <span style="color:#7c79e5;">
        聚簇索引的索引项顺序与表中元组的物理顺序一致。
       </span>
      </strong>
      <span style="color:#ff0000;">
       一个数据库可以建立多个聚簇，一个关系只能加入一个聚簇。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        聚簇索引的适用条件：
       </strong>
      </span>
      <span style="color:#000000;">
       很少对基表进行增删操作；很少对其中的变长列进行修改操作；
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        聚簇索引的用图：
       </strong>
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        <strong>
         大大
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         提高按聚簇属性进行查询的
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         效率
        </strong>
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        <strong>
         节省存储空间
        </strong>
        ：聚簇以后，聚簇码相同的元组集中在一起了，因而聚簇码值不必在每个元组中重复存储，只要在一组中存一次就行了。
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#f33b45;">
       <strong>
        聚簇索引的局限性：
       </strong>
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        <strong>
         聚簇
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         只能提高某些特定应用的性能
        </strong>
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        <strong>
         建立
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         与维护聚簇的开销相当
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         大；
        </strong>
       </span>
       <span style="color:#000000;">
        对
       </span>
       <span style="color:#000000;">
        已有关系建立聚簇，将导致关系中元组的物理存储位置移动，并使此关系上原有的索引无效，必须重建
       </span>
       <span style="color:#000000;">
        。当
       </span>
       <span style="color:#000000;">
        一个元组的聚簇码改变时，该元组的存储位置也要做相应改变
       </span>
       <span style="color:#000000;">
        。
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#f33b45;">
       <strong>
        聚簇索引的适用范围：
       </strong>
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        <strong>
         既适用于单个关系独立聚簇，也适用于多个关系组合聚簇
        </strong>
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        <strong>
         当
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         通过聚簇码进行访问或连接是该关系的主要应用，与
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         聚簇
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         码无关的其他访问很少或者是次要的时，可以使用
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         聚簇。
        </strong>
       </span>
       <span style="color:#000000;">
        尤其
       </span>
       <span style="color:#000000;">
        当
       </span>
       <span style="color:#000000;">
        SQL
       </span>
       <span style="color:#000000;">
        语句中包含有与聚簇码有关的
       </span>
       <span style="color:#000000;">
        ORDER BY, GROUP BY, UNION, DISTINCT
       </span>
       <span style="color:#000000;">
        等子句或短语时，使用聚簇特别有利，可以省去或减化对结果集的排序操作
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#f33b45;">
       <strong>
        设计候选聚簇：
       </strong>
      </span>
      ①
      <span style="color:#000000;">
       常
      </span>
      <span style="color:#000000;">
       在一起进行连接操作的关系可以建立组合聚簇；
      </span>
     </p>
     <p>
      ②
      <span style="color:#000000;">
       如果
      </span>
      <span style="color:#000000;">
       一个关系的一组属性经常出现在相等比较
      </span>
      <span style="color:#000000;">
       条件
      </span>
      <span style="color:#000000;">
       中，则该单个关系可建立聚簇；
      </span>
     </p>
     <p>
      ③
      <span style="color:#000000;">
       如果
      </span>
      <span style="color:#000000;">
       一个关系的一个（或一组）属性上的值
      </span>
      <span style="color:#000000;">
       重复率
      </span>
      <span style="color:#000000;">
       很高，则此单个关系可建立聚簇
      </span>
      <span style="color:#000000;">
       。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        检查候选聚簇索引中的关系，取消不必要的关系
       </strong>
      </span>
     </p>
     <p>
      ①
      <span style="color:#000000;">
       从
      </span>
      <span style="color:#000000;">
       聚簇中删除经常进行全表扫描的关系
      </span>
     </p>
     <p>
      ②
      <span style="color:#000000;">
       从聚簇
      </span>
      <span style="color:#000000;">
       中删除更新操作远多于连接操作的关系
      </span>
     </p>
     <p>
      ③
      <span style="color:#000000;">
       从
      </span>
      <span style="color:#000000;">
       聚簇中删除重复出现的
      </span>
      <span style="color:#000000;">
       关系
      </span>
     </p>
    </blockquote>
    <h4>
     <span style="color:#3399ea;">
      5.3，确定数据库的存储结构
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#000000;">
       <strong>
        确定数据库物理结构主要指确定数据的存放位置和存储结构，包括：确定关
       </strong>
      </span>
      <span style="color:#f33b45;">
       <strong>
        系、索引、聚簇、日志、备份等的存储安排和存储结构，确定系统配置等。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#7c79e5;">
       <strong>
        影响数据存放位置和存储结构的因素：
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        硬件环境和应用需求；要综合考虑存取时间、存储空间利用率和维护代价（
       </strong>
      </span>
      <span style="color:#ff0000;">
       这三个方面常常是相互矛盾的。比如：消除一切冗余数据虽能够节约存储空间和减少维护代价，但往往会导致检索代价的增加。必须进行权衡，选择一个折中方案。
      </span>
      <span style="color:#000000;">
       <strong>
        ）
       </strong>
      </span>
     </p>
    </blockquote>
    <blockquote>
     <p>
      <span style="color:#f33b45;">
       <strong>
        确定数据的存放位置
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#000000;">
       <strong>
        原则：
       </strong>
      </span>
      <span style="color:#000000;">
       根据应用情况
      </span>
      <span style="color:#000000;">
       将易变
      </span>
      <span style="color:#000000;">
       部分与稳定部分分开
      </span>
      <span style="color:#000000;">
       存放，经常
      </span>
      <span style="color:#000000;">
       存取部分与存取频率较低部分分开
      </span>
      <span style="color:#000000;">
       存放。
      </span>
     </p>
     <p>
      ①
      <span style="color:#000000;">
       可以将日志文件与数据库对象（表、索引等）放在不同的磁盘以改进系统的性能。
      </span>
     </p>
     <p>
      ②
      <span style="color:#000000;">
       可以将比较大的表分别放在两个磁盘上，以加快存取速度，这在多用户环境下特别有效。
      </span>
     </p>
     <p>
      ③
      <span style="color:#000000;">
       数据库数据备份、日志文件备份等由于只在故障恢复时才使用，而且数据量很大，可以考虑存放在磁带上。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        确定系统配置
       </strong>
      </span>
     </p>
     <p>
      ①系统都为这些变量（
      <span style="color:#7c79e5;">
       同时使用数据库的用户数、同时打开的数据库对象数、内存分配参数、缓冲区分配参数（使用的缓冲区长度、个数）、存储分配参数 、物理块的大小、物理块装填因子、时间片大小、数据库的大小、锁的数目等
      </span>
      ）赋予了合理的缺省值。在进行物理设计时需要根据应用环境确定这些参数值，以使系统性能最优。
     </p>
     <p>
      ②在物理设计时对系统配置变量的调整只是初步的，要根据系统实际运行情况做进一步的调整，以切实改进系统性能。
     </p>
    </blockquote>
    <h4>
     <span style="color:#3399ea;">
      5.4，评价物理结构
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#000000;">
       <strong>
        评价物理数据库的方法完全依赖于所选用的
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        DBMS
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        。
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#000000;">
       <strong>
        评价内容：
       </strong>
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        对数据库物理设计过程中产生的多种方案进行细致的评价；
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        定量估算各种方案的存储空间、存取时间、维护代价；
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        对估算结果进行权衡、比较，从中选择一个较优的合理的方案作为数据库的物理结构。
       </span>
      </li>
     </ul>
    </blockquote>
    <h3>
     <span style="color:#3399ea;">
      6，数据库的实施和维护
     </span>
    </h3>
    <h4>
     <span style="color:#3399ea;">
      6.1，数据的载入和应用程序的调试
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#000000;">
       <strong>
        数据库实施阶段主要
       </strong>
      </span>
      <span style="color:#000000;">
       <strong>
        工作：
       </strong>
      </span>
     </p>
     <p>
      <strong>
       <span style="color:#f33b45;">
        ①建立实际的数据库结构。
       </span>
      </strong>
      <span style="color:#000000;">
       用
      </span>
      <span style="color:#000000;">
       DDL
      </span>
      <span style="color:#000000;">
       定义数据库：定义基本表、索引、约束、视图等；
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        ②装入数据
       </strong>
      </span>
      <span style="color:#000000;">
       ，组织数据入库
      </span>
      <span style="color:#000000;">
       (
      </span>
      <span style="color:#000000;">
       又称数据库加载
      </span>
      <span style="color:#000000;">
       )
      </span>
      <span style="color:#000000;">
       ，组织数据入库是数据库实施阶段最主要的工作。
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        数据装载方法：人工方法；计算机辅助方法
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        数据筛选、输入、转换(
       </span>
       <span style="color:#000000;">
        工具
       </span>
       <span style="color:#000000;">
        )
       </span>
       <span style="color:#000000;">
        、校验，确保正确
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#f33b45;">
       <strong>
        ③编制和调试数据库应用程序。
       </strong>
      </span>
      <span style="color:#000000;">
       数据库应用程序的设计应该与数据库设计并行进行。数据库结构建立好后，就可以开始编制与调试数据库的应用程序。
      </span>
     </p>
    </blockquote>
    <h4>
     <span style="color:#3399ea;">
      6.2，数据库的试运行
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#f33b45;">
       <strong>
        数据库的试运行：
       </strong>
      </span>
      <span style="color:#000000;">
       应用程序
      </span>
      <span style="color:#000000;">
       调试完成，并且已有一小部分数据入库后，就可以开始对数据库系统进行联合调试，也称数据库的试运行。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       <strong>
        主要工作包括
       </strong>
       ：
      </span>
     </p>
     <p>
      ①
      <span style="color:#000000;">
       <strong>
        功能测试：
       </strong>
      </span>
      <span style="color:#000000;">
       实际运行应用程序，执行对数据库的各种操作，测试应用程序的各种功能。
      </span>
     </p>
     <p>
      ②
      <span style="color:#000000;">
       <strong>
        性能测试：
       </strong>
      </span>
      <span style="color:#000000;">
       测量系统的性能指标，分析是否符合设计目标。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ★
       <strong>
        数据库性能指标的测量
       </strong>
      </span>
     </p>
     <p>
      ①
      <span style="color:#000000;">
       数据库物理设计阶段在评价数据库结构估算时间、空间指标时，作了许多简化和假设，忽略了许多次要因素，因此结果必然很粗糙。
      </span>
     </p>
     <p>
      ②
      <span style="color:#000000;">
       数据库试运行则是要实际测量系统的各种性能指标（不仅是时间、空间指标），如果结果不符合设计目标，则需要返回物理设计阶段，调整物理结构，修改参数；有时甚至需要返回逻辑设计阶段，调整逻辑结构。
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ★
       <strong>
        数据的分期入库
       </strong>
      </span>
     </p>
     <p>
      ①
      <span style="color:#000000;">
       重新设计物理结构甚至逻辑结构，会导致数据重新入库
      </span>
     </p>
     <p>
      ②
      <span style="color:#000000;">
       由于数据入库工作量实在太大，所以可以采用分期输入数据的方法
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        先输入小批量数据供先期联合调试使用
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        待试运行基本合格后再输入大批量数据
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        逐步增加数据量，逐步完成运行评价
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#f33b45;">
       ★
       <strong>
        数据库的转储和恢复
       </strong>
      </span>
     </p>
     <p>
      <strong>
       ①
      </strong>
      在数据库试运行阶段，系统还不稳定，硬、软件故障随时都可能发生
     </p>
     <p>
      <strong>
       ②
      </strong>
      系统的操作人员对新系统还不熟悉，误操作也不可避免
     </p>
     <p>
      <strong>
       ③
      </strong>
      因此必须做好数据库的转储和恢复工作，尽量减少对数据库的破坏
     </p>
    </blockquote>
    <h4>
     <span style="color:#3399ea;">
      6.3，数据库的运行和维护
     </span>
    </h4>
    <blockquote>
     <p>
      <span style="color:#000000;">
       <strong>
        在数据库运行阶段，对数据库经常性的维护工作主要是由数据库管理员完成的，包括：
       </strong>
      </span>
     </p>
     <p>
      <span style="color:#f33b45;">
       ①
       <strong>
        数据库的转储和恢复
       </strong>
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        数据库管理员要针对不同的应用要求制定不同的转储计划，定期对数据库和日志文件进行备份。
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        一旦发生介质故障，即利用数据库备份及日志文件备份，尽快将数据库恢复到某种一致性状态
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#f33b45;">
       ②
       <strong>
        数据库
       </strong>
       <strong>
        的安全性、完整性控制
       </strong>
      </span>
     </p>
     <p>
      <strong>
       <span style="color:#000000;">
        初始定义
       </span>
      </strong>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        数据库管理员根据用户的实际需要授予不同的操作权限
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        根据应用环境定义不同的完整性约束条件
       </span>
      </li>
     </ul>
     <p>
      <strong>
       <span style="color:#000000;">
        修改定义
       </span>
      </strong>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        当应用环境发生变化，对安全性的要求也会发生变化，数据库管理员需要根据实际情况修改原有的安全性控制
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        由于应用环境发生变化，数据库的完整性约束条件也会变化，也需要数据库管理员不断修正，以满足用户要求
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#f33b45;">
       ③
       <strong>
        数据库性能的监督、分析和改进
       </strong>
      </span>
     </p>
     <p>
      <strong>
       <span style="color:#000000;">
        在数据库运行过程中，数据库管理员必须监督系统运行，对监测数据进行分析，找出改进系统性能的方法。
       </span>
      </strong>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        利用监测工具获取系统运行过程中一系列性能参数的值
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        通过仔细分析这些数据，判断当前系统是否处于最佳运行状态
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        如果不是，则需要通过调整某些参数来进一步改进数据库性能
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#f33b45;">
       ④
       <strong>
        数据库的重组织与重构造
       </strong>
      </span>
     </p>
     <p>
      <strong>
       <span style="color:#000000;">
        数据库
       </span>
       <span style="color:#000000;">
        的重组织
       </span>
      </strong>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        <strong>
         为什么要重组织
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         数据库
        </strong>
       </span>
       <span style="color:#000000;">
        数据库
       </span>
       <span style="color:#000000;">
        运行一段时间后，由于记录的不断增、删、改，会使数据库的物理存储变坏，
       </span>
       <span style="color:#000000;">
        从而
       </span>
       <span style="color:#000000;">
        降低数据库存储空间的利用率和数据的存取效率，使数据库的性能下降
       </span>
       <span style="color:#000000;">
        。
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        <strong>
         重组织的
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         形式：
        </strong>
       </span>
       <span style="color:#000000;">
        全部
       </span>
       <span style="color:#000000;">
        重
       </span>
       <span style="color:#000000;">
        组织和部分
       </span>
       <span style="color:#000000;">
        重
       </span>
       <span style="color:#000000;">
        组织，只
       </span>
       <span style="color:#000000;">
        对频繁增、删的表进行重组织
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        <strong>
         重组织的
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         目标：
        </strong>
       </span>
       <span style="color:#000000;">
        提高
       </span>
       <span style="color:#000000;">
        系统性能
       </span>
      </li>
     </ul>
     <p>
      <span style="color:#0d0016;">
       <strong>
        数据库的重构造
       </strong>
      </span>
     </p>
     <ul>
      <li>
       <span style="color:#000000;">
        <strong>
         为什么要
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         重构造数据库
        </strong>
       </span>
       <span style="color:#000000;">
        数据库
       </span>
       <span style="color:#000000;">
        应用环境发生变化，会导致实体及实体间的联系也发生相应的变化，使原有的
       </span>
       <span style="color:#000000;">
        数据库
       </span>
       <span style="color:#000000;">
        设计不能很好地满足新的
       </span>
       <span style="color:#000000;">
        需求。
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        <strong>
         重构造的主要工作
        </strong>
       </span>
       <span style="color:#000000;">
        根据新环境调整数据库的模式和
       </span>
       <span style="color:#000000;">
        内模式增加
       </span>
       <span style="color:#000000;">
        或删除某些
       </span>
       <span style="color:#000000;">
        数据项
       </span>
       <span style="color:#000000;">
        、
       </span>
       <span style="color:#000000;">
        改变
       </span>
       <span style="color:#000000;">
        数据项的
       </span>
       <span style="color:#000000;">
        类型、增加
       </span>
       <span style="color:#000000;">
        或删除某个
       </span>
       <span style="color:#000000;">
        表、改变
       </span>
       <span style="color:#000000;">
        数据库的
       </span>
       <span style="color:#000000;">
        容量、增加
       </span>
       <span style="color:#000000;">
        或删除某些索引。
       </span>
      </li>
      <li>
       <span style="color:#000000;">
        <strong>
         重构造数据库的程度是
        </strong>
       </span>
       <span style="color:#000000;">
        <strong>
         有限的
        </strong>
       </span>
       <span style="color:#000000;">
        若
       </span>
       <span style="color:#000000;">
        应用变化太大，已无法通过重构数据库来满足新的需求，或重构数据库的代价太大
       </span>
       <span style="color:#000000;">
        ，
       </span>
       <span style="color:#000000;">
        则
       </span>
       <span style="color:#000000;">
        表明现有数据库应用系统的生命周期已经结束，应该重新设计新的数据库应用系统了。
       </span>
      </li>
     </ul>
    </blockquote>
