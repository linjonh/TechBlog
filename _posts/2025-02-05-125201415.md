    <h2>
     <a id="1__5">
     </a>
     1. 游戏研发团队介绍
    </h2>
    <p>
     <img alt="在这里插入图片描述" src="https://i-blog.csdnimg.cn/blog_migrate/258e79af3b68e1b4f0d321171fdef7a9.png"/>
    </p>
    <h3>
     <a id="11__7">
     </a>
     1.1 制作人——项目整体负责人
    </h3>
    <pre><code>负责游戏研发环节
负责游戏运营环节
负责项目人员管理
负责项目事物管理
</code></pre>
    <h3>
     <a id="12__13">
     </a>
     1.2 策划——设计人员
    </h3>
    <pre><code>剧情策划
	负责规划游戏中的各种剧情、故事、背景等。
系统策划
	设计游戏中各种系统的规则
数值策划
	规划游戏中各种资源的产出、消耗等
关卡策划
	设计游戏中各种关卡
</code></pre>
    <h3>
     <a id="13__23">
     </a>
     1.3 开发——代码实现人员
    </h3>
    <pre><code>负责把策划的设计及美术资源等通过编码实现可见的程序。开发也分为后端开发人员和前端开发人员
后端开发：实现服务器端的逻辑、数据验证等
前端开发：实现游戏客户端端展现与逻辑
</code></pre>
    <h3>
     <a id="14__28">
     </a>
     1.4 美术
    </h3>
    <pre><code>制作游戏中的各类美术资源
包括场景、原画、UI、动画等
</code></pre>
    <h3>
     <a id="15__32">
     </a>
     1.5 测试
    </h3>
    <pre><code>项目的质量保证人员，主要工作是发现游戏中存在的缺陷并及时反馈出来
测试也可分为功能测试、性能测试、压力测试、兼容性测试、自动化测试、安全测试
</code></pre>
    <h2>
     <a id="2__37">
     </a>
     2. 游戏开发流程
    </h2>
    <ol>
     <li>
      制作人：指定项目目标并规划目标
     </li>
     <li>
      策划：将项目目标拆解成细致的需求，并将需求细化成文案
     </li>
     <li>
      测试、开发、美术：将需求用代码和美术资源实现出来，测试写测试用例
     </li>
     <li>
      测试人员：对项目各个方面进行质量控制，将发现的缺陷反馈出来。
     </li>
    </ol>
    <h2>
     <a id="3__42">
     </a>
     3. 游戏测试的主要内容
    </h2>
    <p>
     其他行业的测试内容，也涵盖其中
    </p>
    <h3>
     <a id="31__44">
     </a>
     3.1 功能测试
    </h3>
    <ul>
     <li>
      是最常见的测试模式，主要测试方法为黑盒测试
     </li>
     <li>
      主要用来验证功能是否符合需求设计
     </li>
     <li>
      主要考虑功能的正确性，而不考虑游戏底层结构及代码错误
     </li>
     <li>
      通常从界面着手开始测试，尽量模拟用户可能出现的操作
     </li>
    </ul>
    <h3>
     <a id="32__49">
     </a>
     3.2 客户端性能测试
    </h3>
    <p>
     性能测试的一些指标
    </p>
    <ul>
     <li>
      客户端CPU使用率
     </li>
     <li>
      客户端内存占用率
     </li>
     <li>
      客户端网络流量使用情况
     </li>
     <li>
      客户端耗电量
     </li>
     <li>
      客户端帧率(FPS)
     </li>
     <li>
      ios常用工具Xcode自带的instrument
     </li>
     <li>
      Android常用工具emmage和GT
     </li>
    </ul>
    <h3>
     <a id="33__58">
     </a>
     3.3 压力测试
    </h3>
    <ul>
     <li>
      服务器CPU使用率
     </li>
     <li>
      服务器内存占用率
     </li>
     <li>
      系统吞吐量(TPS)
     </li>
     <li>
      事务相应时间
     </li>
     <li>
      事务成功率
     </li>
    </ul>
    <h3>
     <a id="34__64">
     </a>
     3.4 兼容性测试
    </h3>
    <ul>
     <li>
      机型适配测试
     </li>
     <li>
      操作系统兼容性测试
     </li>
     <li>
      屏幕分辨率兼容性测试
     </li>
     <li>
      游戏版本兼容性测试
     </li>
    </ul>
    <p>
     补充：在无论是兼容性测试，还是平常的迭代测试，我们都要求机型覆盖，一般来说，我们会选择中高低机型作为测试，而中高低机型的判断，主要是通过处理器来决定，给大家推荐一个
     <a href="https://zhuanlan.zhihu.com/p/480975219" rel="nofollow">
      手机处理器天梯图
     </a>
    </p>
    <h3>
     <a id="35__71">
     </a>
     3.5 安全测试
    </h3>
    <ul>
     <li>
      内存修改测试
     </li>
     <li>
      客户端加密测试
     </li>
     <li>
      客户端反编译测试
     </li>
     <li>
      网络安全测试
     </li>
    </ul>
    <h3>
     <a id="36__76">
     </a>
     3.6 接口测试
    </h3>
    <ul>
     <li>
      服务器各个接口数据测试，主要通过工具来实现（工具：jmter）
     </li>
     <li>
      接口安全测试，重复发送请求，查看接口处理情况
     </li>
    </ul>
    <h3>
     <a id="37__80">
     </a>
     3.7 日志测试
    </h3>
    <ul>
     <li>
      客户端日志
     </li>
     <li>
      服务端日志
      <br/>
      通过客户端和服务端的一些日志可以模拟出客户的使用姿势，或者可以模拟出玩家的行为。
     </li>
    </ul>
    <h3>
     <a id="38__84">
     </a>
     3.8 弱网测试
    </h3>
    <ul>
     <li>
      不同网络情况下游戏的运行情况，如edge、2G、3G、4G等
     </li>
     <li>
      不同丢包率情况下游戏的运行情况
     </li>
     <li>
      不同网络信号之间切换时，对产品运行的影响
     </li>
     <li>
      断网重连对产品运行的影响
     </li>
     <li>
      前后端数据一致相关的问题。
     </li>
    </ul>
    <h4>
     <a id="381__90">
     </a>
     3.8.1 弱网工具
    </h4>
    <ul>
     <li>
      Mac系统可以借助于
      <code>
       Network Link conditioner
      </code>
      或
      <code>
       Charles
      </code>
     </li>
     <li>
      Windows 系统可以借助
      <code>
       Network Emulator for Windows Toolkit
      </code>
      、
      <code>
       Fiddler
      </code>
     </li>
    </ul>
    <p>
     针对弱网测试可以参考小编的另一篇文章。
    </p>
    <h3>
     <a id="39_gm_96">
     </a>
     3.9 gm工具测试
    </h3>
    <p>
     游戏中常用的工具
    </p>
    <ul>
     <li>
      测试gm工具的功能实现，需要关注工具的设置是否在游戏中起作用
     </li>
     <li>
      测试gm工具的数据读取、存储
     </li>
    </ul>
    <h3>
     <a id="310_SDK_100">
     </a>
     3.10 SDK测试
    </h3>
    <ul>
     <li>
      用户数据测试
     </li>
     <li>
      充值、消费测试
     </li>
     <li>
      与各个渠道对接测试
     </li>
    </ul>
    <h2>
     <a id="4__104">
     </a>
     4 游戏测试的基本流程
    </h2>
    <p>
     虽然本篇blog主要分享的是游戏测试方面的测试，但是针对其他行业比如直播类的测试也是适用的。
    </p>
    <h3>
     <a id="41__106">
     </a>
     4.1 功能会议
    </h3>
    <ul>
     <li>
      了解功能需求内容
     </li>
     <li>
      提出可能存在的风险点
     </li>
     <li>
      思考功能的测试重点和难点，如需要工具辅助，需提出开发需求
     </li>
     <li>
      思考功能可优化之处，并提出讨论
     </li>
    </ul>
    <h3>
     <a id="42__111">
     </a>
     4.2 测试用例书写
    </h3>
    <ul>
     <li>
      根据需求书写测试用例
     </li>
     <li>
      关注功能逻辑实现
     </li>
     <li>
      考虑各种特殊情况，如边界值，网络中断，进程中断等
     </li>
     <li>
      关注需求变更情况，需求经常发生变更，需要及时调整测试用例
     </li>
    </ul>
    <h3>
     <a id="43__116">
     </a>
     4.3 冒烟测试
    </h3>
    <ul>
     <li>
      详细测试之前的一个环节，快速发现比较明显的bug
     </li>
     <li>
      快速确保主逻辑流程跑通
     </li>
     <li>
      快速明确功能开展状态
     </li>
    </ul>
    <h3>
     <a id="44__120">
     </a>
     4.4 详细测试
    </h3>
    <ul>
     <li>
      细致的测试没个逻辑分支、资源、配置
     </li>
     <li>
      尽量模拟用户的每一种操作可能
     </li>
     <li>
      测试异常情况，如断网、断电、事件中断，进程中断等
     </li>
     <li>
      测试数据读取、存储、网络等内容
     </li>
     <li>
      测试该功能对其他功能对影响
     </li>
    </ul>
    <h3>
     <a id="45__126">
     </a>
     4.5 回归测试
    </h3>
    <ul>
     <li>
      测试已经被修复等内容
     </li>
     <li>
      测试需求调整后的内容
     </li>
     <li>
      再次详细测试各逻辑分支
     </li>
    </ul>
    <h3>
     <a id="46_Checklist_130">
     </a>
     4.6 Checklist检查（非必要测试）
    </h3>
    <ul>
     <li>
      快速的检查功能的主要逻辑点
     </li>
     <li>
      检查与该功能有关联的任何其他功能点
     </li>
    </ul>
    <h2>
     <a id="5__133">
     </a>
     5 测试用例书写注意事项
    </h2>
    <h3>
     <a id="51__134">
     </a>
     5.1 需求文档分析
    </h3>
    <ul>
     <li>
      文档阅读
      <br/>
      首先要对需求分档进行分析，找负责人进行细节沟通、逻辑梳理，功能扩展相关的问题还有兼容性，针对兼容性，可以问最低和最高机型支持情况。需求文档至少要阅读三遍，这样可以加深对功能的理解。
     </li>
     <li>
      细节沟通
      <br/>
      不明白的地方要及时确认，千万不要脑补想当然。如果需求文档有变更，一定要跟程序员和测试再三确认。
     </li>
     <li>
      逻辑梳理
      <br/>
      虽然我们测试很少写代码，但是我们也需要了解下功能实现逻辑，需求有些需求文档不一定是按照流程写的，可能会出现功能交叉的地方，所以要梳理出框架后，逐步细化。
     </li>
     <li>
      功能扩展思考
      <br/>
      可以看看功能设计思路是否有缺陷，测试难点思考，关联度思考，特殊情况思考。
     </li>
     <li>
      兼容相关思考
      <br/>
      版本兼容、功能兼容、操作系统版本兼容、分辨率兼容
     </li>
    </ul>
    <h3>
     <a id="52___146">
     </a>
     5.2 功能模块划分
    </h3>
    <p>
     功能划分的原则：
     <code>
      高内聚低耦合、重整体轻局部
     </code>
     。大概就是根据功能内部之间和模块与模块之间。
     <br/>
     不用的划分方法适用于不同的场景，要因具体问题具体分析
    </p>
    <h4>
     <a id="521__150">
     </a>
     5.2.1 功能流程划分法
    </h4>
    <p>
     将功能的基本流程画出来，根据流程的每个大的环节进行模块划分，然后再细化和查漏补缺
    </p>
    <h4>
     <a id="522__152">
     </a>
     5.2.2 层次划分法
    </h4>
    <p>
     按照逻辑层次逐层细化出模块的过程，比较适用于UI划分和比较大的系统模块划分
    </p>
    <h4>
     <a id="523__154">
     </a>
     5.2.3 类型划分法
    </h4>
    <p>
     按照功能包含内容的不同类型进行划分
    </p>
    <h2>
     <a id="6__157">
     </a>
     6 测试用例编写
    </h2>
    <p>
     在编写测试用例时，我们需要注意格式，格式的话可以选择表格或者流程图的方式，哪个熟悉用哪个，最主要的是能覆盖全功能即可，能够让人一目了然。测试用例需要包含以下内容：
    </p>
    <ul>
     <li>
      用例名称
     </li>
     <li>
      版本
     </li>
     <li>
      编写人、编写日期、备注（编写人和编写日期一定要写，别人这一点不起眼，有用到的时候）
     </li>
     <li>
      需求文档的链接或者地址（和本次测试有关的信息，都要放上去，可以充当为背景资料，测试来源）
     </li>
     <li>
      模块信息
     </li>
     <li>
      输入信息、输出结果
     </li>
     <li>
      备注信息
     </li>
    </ul>
    <p>
     <strong>
      补充：
     </strong>
     <br/>
     我们肯定听说过测试用例常用的编写方法：
     <code>
      等价类
     </code>
     、
     <code>
      边界值
     </code>
     、
     <code>
      因果图和判定表
     </code>
     。
    </p>
    <ul>
     <li>
      <code>
       等价类
      </code>
      <br/>
      指的是一个输入集合内，任何输入数据对于输出的验证来讲都是等效的，此时就可以选取少量代表性的测试数据来代表整体数据。等价类又分为：有效等价类和无效等价类。
      <br/>
      等价类示例：假如购买一个商品，商品的价格是12块钱，那么我们就需要取 11、12、13 3个数据去测试验证，即可代表所有数据测试，从而节省很多时间。
     </li>
     <li>
      <code>
       边界值
      </code>
      <br/>
      对输入和输出的边界值进行分析的一种方法，边界值的确定,一般选取正好等于，刚刚小于和刚刚大于的情况，通常适用的范畴：数值测试，字符串测试，数据类型测试等
      <br/>
      边界值示例：范围2000-3000，那么我们需要对边界值进行测试，测试取值：1999、2000、2001、2999、3000、3001
     </li>
     <li>
      <code>
       因果图和判定表
      </code>
      <br/>
      两者是息息相关的，通过因果图生成判定表，通过判定表来书写测试用例。因果图是输入和输出之间因果关系的一种关系图
      <br/>
      判定表：可以通过因果图来生成一种结果判定为表格，
     </li>
    </ul>
    <h3>
     <a id="61__178">
     </a>
     6.1 测试用例编写注意事项：
    </h3>
    <ul>
     <li>
      输入条件要单一明确，尽量不用容易引起误解的词，比如可能，大概等
     </li>
     <li>
      输出要可判断且明确。最好不用“显示正确”这种词汇。
     </li>
     <li>
      测试步骤要可执行
     </li>
     <li>
      保持尽量高的覆盖度
     </li>
     <li>
      能抽象的尽量抽象出来，避免无意义的冗余
     </li>
    </ul>
    <h2>
     <a id="7__185">
     </a>
     7 测试用例整理与维护
    </h2>
    <ul>
     <li>
      需求变化后需要及时更新老的测试用例，并写清楚修改情况的备注
     </li>
     <li>
      测试用例应该尽量避免冗余，如果遇到重复的用例，需要根据实际情况进行修改
     </li>
     <li>
      注意测试用例的备份，写完后最好自己本地也备份一份，避免线上被人误删除
     </li>
    </ul>
    <h2>
     <a id="8_Bug_189">
     </a>
     8 Bug的界定标准
    </h2>
    <ul>
     <li>
      与需求设计不符
     </li>
     <li>
      违背常识
     </li>
    </ul>
    <h3>
     <a id="81_Bug_193">
     </a>
     8.1 Bug的等级划分
    </h3>
    <ul>
     <li>
      P0：致命错误，需要立即修复，如宕机、重启性报错等
     </li>
     <li>
      P1:严重错误，需要紧急修复，如功能流程错误，数值错误等
     </li>
     <li>
      P2：一般错误，允许一段时间内修复，如功能等简单错误，界面错误等
     </li>
     <li>
      P3：无关紧要的错误，允许延期修复，如文字错误，某个像素点缺失等等。
     </li>
    </ul>
    <h3>
     <a id="82_Bug_198">
     </a>
     8.2 Bug提报所要包含的内容
    </h3>
    <p>
     有的公司使用Jira、有的公司使用邮件或者表格，这都没有关系，重要的是要包含以下信息。
    </p>
    <ul>
     <li>
      标题：模块名称+简短描述
     </li>
     <li>
      测试环境：标明测试用的版本，系统、服务器、账号等
     </li>
     <li>
      bug详细描述
     </li>
     <li>
      重现步骤：重现Bug的详细流程步骤及复现概率等
     </li>
     <li>
      期望结果：希望Bug修复后的结果
     </li>
     <li>
      附件：log、截图、视频等
     </li>
    </ul>
    <h3>
     <a id="83_Bug_206">
     </a>
     8.3 Bug的验收标准
    </h3>
    <ul>
     <li>
      拓展：是否对其它功能有影响，需要简单回归
     </li>
     <li>
      注意点：验证不能只看前端展现，更应关注后端数据
      <br/>
      Bug的跟踪与推动
     </li>
     <li>
      每个人都有责任跟踪自己的Bug的修复状态
     </li>
     <li>
      及时与开发沟通，了解修复状态并提供修复过程中的支持
     </li>
     <li>
      长时间未修复的bug需要与开发和上级确认如何处理
     </li>
     <li>
      bug修复后，需要及时验证
     </li>
    </ul>
    <h3>
     <a id="84_Bug_214">
     </a>
     8.4 Bug分析
    </h3>
    <p>
     针对Bug的分析我们可以从以下几个纬度
    </p>
    <ul>
     <li>
      根据 Bug 等级分析：罗列出 P0、P1、P2、P3 类型的 Bug 加以分析
     </li>
     <li>
      根据项目人员Bug数据，这里面可以包含两种方式，第一种方式是根据测试人员检出的Bug数进行分析，第二种方式是根据分配给开发的 Bug 数进行进行
     </li>
     <li>
      根据功能模块 Bug 数据分析
     </li>
    </ul>
