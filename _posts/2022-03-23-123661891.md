    <p>
     1、测试用例概念
    </p>
    <pre><code class="prism language-bash">为了特定的目的（证明软件存在某问题）而设计的一组由测试输入、执行条件、预期结果构成的文档
测试用例，就是指导如何做测试的文档，主要记录需要验证被测软件的是否满足需求
</code></pre>
    <p>
     2、测试用例有常见的这两种：
    </p>
    <pre><code class="prism language-bash">一种是通过Excel直接编写测试用例，包含完整的内容：用例编号，用例标题，测试步骤，优先级，预期结果。
一种是通过xmind直接整理测试点，粗略的测试用例要点，思路梳理的过程。
  PS：当公司有**测试用例评审**时，则需要在用例评审会上发言自己负责模块的**测试点**。
</code></pre>

    <p>
     5、测试用例的作用
    </p>
    <ul>
     <li class="task-list-item">
      <input class="task-list-item-checkbox" disabled="disabled" type="checkbox"/>
      测试之前设计和评审测试用例，避免盲目测试并提高测试效率。PS：防止测试过程的活动出现遗漏
     </li>
     <li class="task-list-item">
      <input class="task-list-item-checkbox" disabled="disabled" type="checkbox"/>
      测试用例的使用让软件测试的实施重点突出、目的明确。PS：以文档的形式记录软件可能存在的问题
     </li>
     <li class="task-list-item">
      <input class="task-list-item-checkbox" disabled="disabled" type="checkbox"/>
      版本更新后只需修正少部分测试用例便可以展开测试工作。PS：降低工作强度、缩短项目周期。
     </li>
     <li class="task-list-item">
      <input class="task-list-item-checkbox" disabled="disabled" type="checkbox"/>
      测试用例的设计，检验软件是否满足客户需求。PS:测试工作量的展示
     </li>
    </ul>
    <p>
     6、如何写好测试用例
    </p>
    <ul>
     <li>
      思考：站在用户的立场，从客户最开始的需求出发，结合需求进行细化设计的过程。假如这个软件是给我使用的，我想实现哪些功能？预期是什么效果？基本的操作步骤是否繁琐？界面是否美观？标签和字体是否清晰？正常验证是否能成功？异常验证会出现哪些情况？
     </li>
     <li>
      学习和积累：前有古人后有来者。之前的软件测试工程师都能够想到的测试点，在实际工作中面对的项目不同我们需要进行合理参考借鉴学习，对产品不断的熟悉以及丰富自己的测试经验，这些测试点的设计是不断学习、熟悉项目、积累测试经验获得的。
     </li>
     <li>
      理论知识：活到老学到老，不断丰富自己的理论知识，有了理论为基础，还有积累经验和思考的加持，测试用例是由测试工程师通过去思考而设计的，这个过程不能因为疏漏而导致bug缺陷。
     </li>
     <li>
     </li>
    </ul>
    <pre><code class="prism language-css"> 做测试有3个心：信心，耐心，细心！
</code></pre>
    <p>
     设计测试用例的理论方法有：
    </p>
    <ul>
     <li>
      <p>
       观察法
      </p>
     </li>
     <li>
      <p>
       等价类、边界值
      </p>
     </li>
     <li>
      <p>
       判定表、因果图
      </p>
     </li>
     <li>
      <p>
       流程图、场景法
      </p>
     </li>
     <li>
      <p>
       错误推测法等
      </p>
     </li>
    </ul>
    <p>
     软件质量模型特性：
    </p>
    <ul>
     <li>
      功能性：功能有没有，好不好用
     </li>
     <li>
      性能效率：对应系统的资源耗费程度及响应时间
     </li>
     <li>
      易用性：容易理解、学习、使用
     </li>
     <li>
      兼容性：能够兼容不同的软硬件平台
     </li>
     <li>
      可靠性：不易出问题，万一出问题容易恢复
     </li>
     <li>
      安全性：对于用户的安全保障（外在的人生安全、内在的信息安全等）
     </li>
     <li>
      可移植性：能否在不同环境条件下无故障运行
     </li>
     <li>
      可维护性：对于后期的修复维护是否方便快捷
      <br/>
      <strong>
       熟悉需求
      </strong>
      对用例模块进行拆分，把理论知识、思考、经验的积累结合起来使用，最终成为测试待验证的结果。
      <br/>
      对于复杂一点的功能需要借助于测试用例的设计方法，对于页面级的测试点应该使用等价类、边界值。仅仅熟悉了需要，还需要结合经验的积累，从质量模型的特性出发，进行全面的思考功能点的设计，是否出现遗漏的，是否有项目特殊要求的。用例的设计不是一蹴而就的事情，好的用例也是需要不断的练习，反复的修改评审，才能编写出卓越的用例。
     </li>
    </ul>
    <blockquote>
     <p>
      一.测试计划，二.测试流程，三.测试点，四.测试用例，五.测试报告 要区分。
     </p>
    </blockquote>
    <p>
     一。1． 为什么要编写测试计划？
    </p>
    <pre><code class="prism language-bash"><span class="token number">1</span>、领导能够根据测试计划做宏观调控，进行相应资源配置等；
<span class="token number">2</span>、测试人员能够了解整个项目测试情况以及项目测试不同阶段的所要进行的工作等；
<span class="token number">3</span>、便于其他人员了解测试人员的工作内容，进行有关配合工作
</code></pre>
    <p>
     2． 什么时间开始编写测试计划？
    </p>
    <pre><code class="prism language-bash">（测试需求分析前总体测试计划书/测试需求分析后详细测试计划书）
</code></pre>
    <p>
     3． 由谁来编写测试计划？
    </p>
    <pre><code class="prism language-bash">具有丰富经验的项目测试负责人
</code></pre>
    <p>
     4． 测试计划编写6要素？（5W1H）
    </p>
    <pre><code class="prism language-bash"><span class="token number">1</span>、why——为什么要进行这些测试；
<span class="token number">2</span>、what—测试哪些方面，不同阶段的工作内容；
<span class="token number">3</span>、when—测试不同阶段的起止时间；
<span class="token number">4</span>、where—相应文档，缺陷的存放位置，测试环境等；
<span class="token number">5</span>、who—项目有关人员组成，安排哪些测试人员进行测试
<span class="token number">6</span>、how—如何去做，使用哪些测试工具以及测试方法进行测试。
</code></pre>
    <p>
     二。测试流程
    </p>
    <pre><code class="prism language-bash"><span class="token number">1</span>、熟悉需求文档（产品出的）要阅读理解，分析需求点。
<span class="token number">2</span>、测试计划（安排进度，人力物力，时间分配，风险评估）。
<span class="token number">3</span>、测试设计方案：编写测试用例。采用什么测试方法。参考需求文档进行概要设计和详细设计等文档，用例编写完成之后会进行评审。
<span class="token punctuation">{<!-- --></span>用例多的话可以复用，编写公共用例）每做一个项目，都把自己的用例进行整理、归档，以后无论去哪里，遇到类似的项目和功能点，都能拿出来修改复用，大大节省了用例编写时间。个人认为功能测试的经验重要积累点就为测试经验和手里面的公共用例。<span class="token punctuation">}</span>
<span class="token number">4</span>、执行测试用例（阶段：搭建环境，执行冒烟测试（预测试），然后进入正式测试，bug管理直到测试结束。
<span class="token number">5</span>、测试评估阶段：出测试报告，确认是否可以上线。
</code></pre>
    <p>
     搭建测试环境：硬件-软件-测试工具-操作系统-网络-数据。
    </p>
    <p>
     五。测试报告
     <br/>
     1.版本名称、测试时间、地点和人员；
     <br/>
     2.环境描述：
     <code>
      硬件配置、软件配置；
     </code>
     <br/>
     3.测试质量评估得出结论：
    </p>
    <pre><code class="prism language-bash">需求符合度、功能正确性、性能指标、运行稳定性、互联互通、文档、可用性、可维护性、兼容性、风险评估等
</code></pre>
    <p>
     4.多个维度对版本质量进行总体评价。
    </p>
    <pre><code class="prism language-bash">该版本满足上网条件，可以上线使用；
该版本存在质量风险，只可在XX局受限使用；
该版本存在严重质量问题，不满足上线条件；
</code></pre>
    <p>
     5.缺陷统计
    </p>
    <pre><code class="prism language-bash">在bug缺陷管理工具例如禅道自动统计，按版本、缺陷分布、问题状态统计以图表方式直观显示。
</code></pre>
    <p>
     6.缺陷缝隙：
     <code>
      缺陷原因，缺陷趋势、遗留的缺陷和规避措施。
     </code>
     <br/>
     7.测试用例执行结果统计数据：
    </p>
    <pre><code class="prism language-bash">根据实际情况进行列表统计。第一二轮测试的pass和fail的条数
</code></pre>
    <p>
     8.附件：遗留问题报告、交付的测试工作产品和测试项通过情况清单为必需的附件，其余可根据实际测试内容进行裁剪，不同的测试报告根据需要可以给出不同类型的附件。
     <br/>
     <strong>
      附件的目的
     </strong>
     是帮助本报告的使用者理解报告，记录修改情况和有用的数据等。
     <br/>
     9.交付的测试产品
    </p>
    <pre><code class="prism language-bash"><span class="token number">1</span>.测试计划
<span class="token number">2</span>.测试用例
<span class="token number">3</span>.测试报告
<span class="token number">4</span>.测试代码及设计文档
</code></pre>
