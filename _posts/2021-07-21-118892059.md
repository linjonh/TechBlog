    <h3 id="%E6%95%B0%E6%8D%AE%E5%BA%93%E7%BB%93%E6%9E%84%E8%AE%BE%E8%AE%A1">
     数据库结构设计
    </h3>
    <h4 id="%E6%95%B0%E6%8D%AE%E5%BA%93%E6%A6%82%E5%BF%B5%E8%AE%BE%E8%AE%A1">
     数据库概念设计
    </h4>
    <p>
     一个设计良好的数据库，在很大程度上决定了系统的成功与否。
    </p>
    <p>
     概念设计是数据库设计的
     <strong>
      <span style="color:#fe2c24;">
       核心环节
      </span>
     </strong>
     。通过对用户需求进行
     <span style="color:#fe2c24;">
      <strong>
       综合、归纳与抽象
      </strong>
     </span>
     ，形成一个独立于具体DBMS的概念模型。
    </p>
    <p style="text-align:center;">
     <strong>
      1、数据库概念设计的目标
     </strong>
     <br/>
     定义和描述应用领域设计的数据范围
     <br/>
     ①获取信息模型
     <br/>
     ②描述数据的属性特征
     <br/>
     ③描述数据之间的关系
     <br/>
     ④定义和描述数据的约束
     <br/>
     ⑤说明数据的安全性要求
     <br/>
     ⑥支持用户的各种数据处理需求
     <br/>
     ⑦保证信息模型能转化成数据库的逻辑结构（即数据库模式）。
    </p>
    <p>
     <strong>
      2、过程：
     </strong>
     <br/>
     （1）明确建模目标（模型覆盖范围）
     <br/>
     （2）定义实体集（自底向上标识和定义实体集）
     <br/>
     （3）定义联系（实体间关联关系）
    </p>
    <p>
     （4）建立信息模型（构造ER模型）
     <br/>
     （5）确定实体集属性（属性描述一个实体集的特征或性质）
     <br/>
     （6）对信息模型进行集成与优化（检查和消除命名不一致、结构不一致等）
     <br/>
     概念设计是DB设计的核心环节。概念数据模型是对现实世界的抽象和模拟。
    </p>
    <p>
     <strong>
      3、概念模型设计
     </strong>
     <br/>
     概念设计目前采用最广泛的是ER建模方法。将现实世界抽象为具有属性的实体及联系。
     <br/>
     1976年，Peter.Chen提出Ｅ-Ｒ模型（Entity- Relationship Model），即实体联系模型，用Ｅ-Ｒ图来描述数据库的概念模型。
     <br/>
     观点：世界是由一组称作实体的基本对象和这些对象之间的联系构成的。
    </p>
    <p>
     <strong>
      与E-R模型有关的概念
     </strong>
     <br/>
     实体（Entity）或实例（Instance）
     <br/>
     <strong>
      客观存在并可相互区分的事物叫实体。
     </strong>
     <br/>
     如学生张三、工人李四、计算机系、数据库概论。
     <br/>
     实体集（Entity Set）
     <br/>
     <strong>
      同型实体的集合称为实体集。
     </strong>
     <br/>
     如全体学生。
    </p>
    <p>
     <strong>
      属性（Attribute）
     </strong>
     <br/>
     实体所具有的某一特性。一个实体可以由若干个属性来刻画。每个属性的取值范围称为域。
     <br/>
     例如，学生可由学号、姓名、年龄、系、年级等组成。
    </p>
    <p>
     <strong>
      码（Key）：
     </strong>
     <br/>
     实体集中唯一标识每一个实体的属性或属性组合。
     <br/>
     用来区别同一实体集中的不同实体的称作主码。
     <strong>
      （主键）
     </strong>
     <br/>
     一个实体集中任意两个实体在主码上的取值不能相同。
     <br/>
     如学号是学生实体的主码。
     <strong>
      （主键）
     </strong>
    </p>
    <p>
     <strong>
      联系（Relationship）
     </strong>
     <br/>
     描述实体之间的相互关系。
     <br/>
     如学生与老师间的授课关系，学生与学生间有班长关系。
     <br/>
     联系也可以有属性，如学生与课程之间有选课联系，每个选课联系都有一个成绩作为其属性。
     <br/>
     同类联系的集合称为联系集。
    </p>
    <p>
     <strong>
      实体间的联系有三类：
     </strong>
     <br/>
     实体之间的联系的数量，即一个实体通过一个联系集能与另一实体集相关联的实体的数目。
     <br/>
     一对一联系（1：1）
     <br/>
     如：“系”与“系主任”（一个系只有一个系主任，一个系主任只负责管理一个系）
    </p>
    <p>
     <strong>
      一对多联系（1：n )
     </strong>
     <br/>
     如：“系”与“学生”（一个系招收若干学生，一个学生只属于一个系）
     <br/>
     <strong>
      多对多联系（m：n）
     </strong>
     <br/>
     如：“学生”与“课程”（一名学生可选修多门课程，每门课程可被多名学生选修）
    </p>
    <p>
     <strong>
      E-R模型的表示
     </strong>
    </p>
    <p>
     <img alt="" height="362" src="https://i-blog.csdnimg.cn/blog_migrate/e5d41395850c81c4b67a500fcb7aca8a.png" width="845"/>
    </p>
    <p>
     <strong>
      1、逻辑设计的任务：
     </strong>
     <br/>
     将概念模型（如ER图）转化为DBMS支持的数据模型（如关系模型），并对其进行优化。
    </p>
    <p>
     <strong>
      2、逻辑设计的依据和阶段目标：
     </strong>
    </p>
    <p>
     <img alt="" height="297" src="https://i-blog.csdnimg.cn/blog_migrate/2d2c35e103f93f7c95c928f69d1c2132.png" width="811"/>
    </p>
    <p>
     <strong>
      关系模型
     </strong>
     <br/>
     有三种主要的数据模型：
     <strong>
      层次模型、网状模型、关系模型
     </strong>
     。其中关系模型简单灵活，并有着坚实的理论基础，已成为当前最流行的数据模型。
     <br/>
     关系模型就是用
     <strong>
      二维表格
     </strong>
     结构来表示实体及实体之间联系的模型。
     <br/>
     关系的描述称为关系模式（Relation Schema）。关系模式由
     <strong>
      五部分
     </strong>
     组成，即它是一个五元组：R(U, D, DOM, F)
    </p>
    <p>
     <span style="background-color:#ff9900;">
      R
     </span>
     ：关系名
     <span style="background-color:#ff9900;">
      U
     </span>
     ：组成该关系的属性名集合
     <span style="background-color:#ff9900;">
      D
     </span>
     ：属性组U中属性所来自的域
     <span style="background-color:#ff9900;">
      DOM
     </span>
     ：属性到域的映射
     <span style="background-color:#ff9900;">
      F
     </span>
     ：属性组U上的一组数据依赖
     <br/>
     由于D、DOM对模式设计的关系不大，这里把关系模式简化为一个三元组：
     <br/>
     R&lt;U, F&gt;，当且仅当U上的一个关系R满足F时，R称为关系模式R&lt;U,F&gt;的一个关系。
    </p>
    <p>
     <strong>
      1、关系数据库设计的核心：关系模式的设计。
      <br/>
      2、关系模式的设计目标：按照一定的原则从数量众多而又相互关联的数据中，构造出一组既能较好地反映现实世界，而又有良好的操作性能的关系模式。
     </strong>
    </p>
    <p>
     <strong>
      数据依赖
     </strong>
     <br/>
     <strong>
      定义：
     </strong>
     <br/>
     <strong>
      设R（U）是一个属性集U上的关系模式，X和Y是U的子集。若对于R（U）的任意一个可能的关系r，r中不可能存在两个元组在X上的属性值相等，而在Y上的属性值不等，则称“X函数确定Y”或“Y函数依赖于X”，记作X→Y。
     </strong>
    </p>
    <p>
     <strong>
      数据依赖
     </strong>
    </p>
    <p>
     关系内部属性与属性之间的一种约束关系，是现实世界属性间相互联系的抽象
     <br/>
     数据的内在性质
     <br/>
     语义的体现，完整性约束的表现形式
     <br/>
     限定属性的取值范围，如年龄&lt;60，定义属性间值的相互关联（主要体现于值的相等与否），这就是数据依赖
    </p>
    <p>
     <strong>
      数据依赖的类型
     </strong>
    </p>
    <p>
     函数依赖（Functional Dependency，FD）普遍存在于生活中，这种依赖关系类似于数学中的函数y=f（x），自变量x确定之后，相应的函数值y也就唯一地确定了。如关系：公民（身份证号，姓名，地址，工作单位）身份证号一确定，则其地址就唯一确定，因此地址函数依赖身份证号。而姓名一确定，不一定能确定地址。
    </p>
    <p>
     <strong>
      多值依赖（Multivalued Dependency，MD）
     </strong>
     <br/>
     教师号可能多值依赖课程号，因为给定一个（课程号，参考书号）的组合，可能有对应多个教师号。这是因为多个老师可以使用相同或不同的参考书上同一门课。
     <br/>
     简单点讲，函数就是唯一确定的关系；多值依赖却不能唯一确定。
    </p>
    <p>
     <strong>
      候选码、主码、外码
     </strong>
     <br/>
     我们已经知道，如果某属性组的值能唯一确定整个元组的值，则称该属性组为候选码或侯选关键字。
     <br/>
     例如：（学号，姓名，性别，年龄）中，学号是关键字，（学号，姓名）不是关键字，性别不是关键字。
     <br/>
     候选码如果有多个，可以选其中的一个作为主码（Primary Key） 。
    </p>
    <p>
     <strong>
      数据规范化
     </strong>
     <br/>
     关系数据库的设计主要是关系模式设计。关系模式设计的好坏直接影响到数据库设计的成败。将关系模式规范化，是设计较好的关系模式的惟一途径。
     <br/>
     关系模式的规范化主要是由关系范式来完成的。
     <br/>
     关系模式的规范化：把一个低一级的关系模式分解为高一级关系模式的过程。
     <br/>
     关系数据库的规范化理论是数据库逻辑设计的工具。
     <br/>
     目的：尽量消除
     <strong>
      插入、删除异常，修改复杂，数据冗余
     </strong>
     的问题。
    </p>
    <p>
     <strong>
      范式
     </strong>
     <br/>
     范式：关系模式满足的约束条件称为范式。根据满足规范化的程度不同，范式由低到高分为1NF，2NF，3NF，BCNF，4NF，5NF。
     <br/>
     1NF：如果关系模式R，其所有属性都是不可再分的基本数据项，则称R属于第一范式，R∈1NF。
    </p>
