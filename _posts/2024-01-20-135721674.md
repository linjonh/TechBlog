    <h2>
     <a id="_0">
     </a>
     问题背景
    </h2>
    <p>
     我们使用FileZilla Server搭建FTP服务器。我们在服务器上监听了
     <code>
      21
     </code>
     端口作为服务器FTP端口。我们还选择了端口范围
     <code>
      49123-49124
     </code>
     作为被动模式端口，因为我们想让客户端全部以被动模式连接。
     <br/>
     <img alt="服务端口" src="https://i-blog.csdnimg.cn/blog_migrate/b65e6388070ba47ac1fe4efae02748ba.png">
      <br/>
      <em>
       服务端口
      </em>
     </img>
    </p>
    <p>
     <img alt="被动模式端口范围" src="https://i-blog.csdnimg.cn/blog_migrate/026df5e4364447bc443669736808f216.png">
      <br/>
      <em>
       被动模式端口范围
      </em>
     </img>
    </p>
    <p>
     由于我们的服务器不暴露在公网上，我们使用FRP进行内网穿透。配置信息被记录在
     <code>
      frpc.ini
     </code>
     中，内容如下。
    </p>
    <pre><code class="prism language-js"><span class="token punctuation">[</span>common<span class="token punctuation">]</span>
server_addr <span class="token operator">=</span> someone<span class="token punctuation">.</span>net
server_port <span class="token operator">=</span> <span class="token number">7000</span>
token <span class="token operator">=</span> passowrd
local_ip <span class="token operator">=</span> <span class="token number">192.168</span><span class="token number">.1</span><span class="token number">.185</span>

<span class="token punctuation">[</span>ftpsrvport<span class="token punctuation">]</span>
type <span class="token operator">=</span> tcp
local_port <span class="token operator">=</span> <span class="token number">21</span>
remote_port <span class="token operator">=</span> <span class="token number">21212</span>

<span class="token punctuation">[</span>pasvport1<span class="token punctuation">]</span>

local_port <span class="token operator">=</span> <span class="token number">49123</span>

</code></pre>
<h2>
<a id="_32">
</a>
预期出现的行为
</h2>
<p>
客户端连接
<code>
someone.net:21212
</code>
以进行 FTP 命令交互。服务端向客户端发送被动模式连接地址和端口
<code>
someone.net:49123
</code>
或
<code>
someone.net:49124
</code>
，以告诉客户端到何处连接来传输 FTP 数据。接着，客户端主动连接被告知的地址和端口。
</p>
<h2>
<a id="_35">
</a>
实际出现的行为
</h2>
<p>
尽管已经在 FileZilla Server 中配置在被动模式时告诉客户端去连接
<code>
someone.net:49123-49124
</code>
作为 FTP 的数据传输端口，服务器仍然告诉客户端去连接
<code>
127.0.0.1:49123-49124
</code>
，导致客户端无法传输数据。而在使用 FileZilla Client 时，FileZilla Client 也只是在意识到服务器给出的地址不可用后智能地自行决定用服务器的公网地址替换
<code>
127.0.0.1
</code>
才得以连接成功。
<br/>
<img alt="FileZilla Client信息输出" src="https://i-blog.csdnimg.cn/blog_migrate/83bc265fe27762814ee1463de49c1410.png">
<br/>
<em>
FileZilla Client 信息输出
</em>
</img>
</p>
<h2>
<a id="_41">
</a>
原因
</h2>
<p>
内网穿透的来访请求会全部变为 127.0.0.1 来访。FileZilla Server 默认私自决定对于这些非公网来访请求，都不再将被动模式地址替换为我们指定的公网地址。
<br/>
<img alt="内网穿透后，所有的来访请求都会变成来自127.0.0.1" src="https://i-blog.csdnimg.cn/blog_migrate/bd09db3d07923044038dbded798a214a.png">
<br/>
<em>
内网穿透后，所有来访请求都会变成来自 127.0.0.1
</em>
</img>
</p>
<h2>
<a id="_47">
</a>
解决方案
</h2>
<p>
取消这个私自决定。
<br/>
<img alt="取消勾选推荐选项" src="https://i-blog.csdnimg.cn/blog_migrate/1ff05ca3a092a4bdc50488649ce3bdcc.png">
<br/>
<em>
取消勾选推荐选项
</em>
</img>
</p>
<h2>
<a id="_52">
</a>
结论背景
</h2>
<p>
尽管在解决后，这个问题的原因显得非常幼稚，但是网络上有无数人遇到这个问题却不知道如何解决。人们给出的对该问题的回答通常模棱两可，例如“检查防火墙配置”或“可能是由于你的服务器（或路由器）配置不正确”。很显然，只有在网路配置不正确的情况下，才会出现问题。这是无意义的虚词。
</p>
