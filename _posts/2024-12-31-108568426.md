    <h2>
     <a id="_2">
     </a>
     一，软件工程基本概念
    </h2>
    <p>
     <strong>
      软件
     </strong>
     ：程序，设计，文档。
    </p>
    <h3>
     <a id="_6">
     </a>
     软件分类
    </h3>
    <p>
     <strong>
      系统软件
     </strong>
     ：操作系统，编译程序，汇编程序，网络软件，数据库管理系统。
    </p>
    <p>
     <strong>
      应用软件
     </strong>
     ：事务处理软件，工程与科学技术软件，实时处理软件，人工智能软件。
    </p>
    <p>
     <strong>
      支撑软件（工具软件）
     </strong>
     ：需求分析工具软件，编译工具软件，测试工具软件，维护工具软件。
    </p>
    <h3>
     <a id="_14">
     </a>
     软件危机
    </h3>
    <p>
     需求增长，开发难控，质量难保，难以维护，成本提高，生产率低
    </p>
    <h3>
     <a id="_18">
     </a>
     软件工程
    </h3>
    <p>
     <strong>
      软件工程
     </strong>
     ：应用于计算机软件的定义，开发和维护的一整套方法，工具，文档，实践标准和工序。
    </p>
    <p>
     其目的是提高软件生产率，提高软件质量，降低软件成本。
    </p>
    <p>
     其核心思想是把软件当成一个工程产品来处理。
    </p>
    <p>
     <strong>
      软件工程三要素
     </strong>
     ：
    </p>
    <ol>
     <li>
      方法是完成软件工程项目的技术手段
     </li>
     <li>
      工具支持软件的开发，管理和文档生成
     </li>
     <li>
      过程支持软件开发的个环节的控制和管理
     </li>
    </ol>
    <h3>
     <a id="_32">
     </a>
     软件生命周期
    </h3>
    <p>
     将软件产品从提出，实现，使用维护到停止使用退役的过程称为软件生命周期
     <br/>
     分为软件定义，软件开发及软件运行维护3个时期。维护是持续时间最长，花费代价最大的一个时期。
    </p>
    <p>
     软件工程学的一个目的就是提高软件的可维护性，降低维护代价。
    </p>
    <p>
     <strong>
      软件生命周期
     </strong>
     <br/>
     3个时期共有8个阶段
     <br/>
     <strong>
      定义时期
     </strong>
     ：问题定义可行性研究，需求分析，概要设计
    </p>
    <p>
     <strong>
      开发时期
     </strong>
     ：详细设计，实现，设计
    </p>
    <p>
     <strong>
      运行维护
     </strong>
     ：使用，维护
    </p>
    <h3>
     <a id="_47">
     </a>
     需求分析
    </h3>
    <p>
     <strong>
      需求分析
     </strong>
     ：确定系统的逻辑模型。参与人员有用户，项目负责人和系统分析员。
    </p>
    <p>
     <strong>
      其工作
     </strong>
     ：需求获取 需求分析 编写需求规格说明书 需求评审
    </p>
    <p>
     <strong>
      产生文档
     </strong>
     ：为需求规格说明书
    </p>
    <p>
     <strong>
      需求规格说明书的作用
     </strong>
     ：
     <br/>
     （1）便于用户，开发人员进行理解交流
     <br/>
     （2）反映用户问题的结构，可以作为软件开发工作的基础和依据
     <br/>
     （3）作为确定测试和验收的依据
    </p>
    <h3>
     <a id="SRS_60">
     </a>
     需求规格说明书（SRS)
    </h3>
    <p>
     需求分析阶段产生的主要文档是“软件需求规格说明书”，其
     <strong>
      特点
     </strong>
     是：
     <br/>
     <strong>
      正确性
     </strong>
     ：体现待开发系统的真实要求
     <br/>
     <strong>
      无歧义性
     </strong>
     ：对每个需求只有一种解释
     <br/>
     <strong>
      完整性
     </strong>
     ：包括全部有意义的需求
     <br/>
     <strong>
      可验证性
     </strong>
     ：每个需求都是可验证的
     <br/>
     <strong>
      一致性
     </strong>
     ：各个需求的描述不矛盾
     <br/>
     <strong>
      可理解性
     </strong>
     ：需求说明书必须简明易懂
     <br/>
     <strong>
      可修改性
     </strong>
     ：结构分格在改变时，是易于实现的
     <br/>
     <strong>
      可追踪性
     </strong>
     ：每个需求的来源和流向是清晰的
    </p>
    <h2>
     <a id="_72">
     </a>
     二，结构化分析方法
    </h2>
    <p>
     <strong>
      需求分析方法
     </strong>
     有：
     <br/>
     （1）结构化需求分析方法
     <br/>
     （2）面向对象的分析方法
    </p>
    <p>
     <strong>
      结构化分析方法
     </strong>
     ：使用数据流图（DFD），数据字典（DD），判定表和判定树等工具来建立系统的逻辑模型
    </p>
    <h3>
     <a id="DFD_80">
     </a>
     数据流图（DFD）
    </h3>
    <p>
     数据流图的图形元素：
     <br/>
     O 加工：输入数据经加工变换产生输出
     <br/>
     -&gt; 数据流：沿箭头方向传递数据的通道
     <br/>
     = 存储文件（数据源）：存放各种数据的文件
     <br/>
     口 源（潭）：系统和文件的接口
     <br/>
     数据字典是结构化分析的核心
    </p>
    <h2>
     <a id="_89">
     </a>
     三，结构化设计方法
    </h2>
    <h3>
     <a id="_91">
     </a>
     软件设计的划分
    </h3>
    <p>
     <strong>
      从工程管理角度划分
     </strong>
     ：概要设计 详细设计
     <br/>
     <strong>
      按技术观点分析
     </strong>
     ：结构设计 数据设计 接口设计 过程设计
    </p>
    <h3>
     <a id="_96">
     </a>
     软件设计基本原理
    </h3>
    <p>
     <strong>
      抽象
     </strong>
     ：在软件设计中，可以定出多个抽象级别，抽象层次从概要设计到详细设计逐步降低。
    </p>
    <p>
     <strong>
      模块化
     </strong>
     ：把一个待开发的软件分解为若干小的简单的部分，自顶向下逐层把软件划分为若干模块。
    </p>
    <p>
     <strong>
      信息隐蔽
     </strong>
     ：一个模块内的信息，对于不需要这些信息的其他模块来说不能访问。
    </p>
    <p>
     <strong>
      模块的独立性
     </strong>
     ：每个模块只完成独立的子功能，并且与其他模块的联系少且接口简单。模块的独立程度是评价设计好坏的重要度量标准。
    </p>
    <h3>
     <a id="_106">
     </a>
     软件模块独立性
    </h3>
    <p>
     <strong>
      高内聚性
     </strong>
     ：指一个模块内部各个元素间彼此结合的紧密程度
    </p>
    <p>
     <strong>
      低耦合性
     </strong>
     ：指模块间互相连接的紧密程度
    </p>
    <h3>
     <a id="_112">
     </a>
     概要设计
    </h3>
    <p>
     <strong>
      概要设计的任务
     </strong>
     ：
     <br/>
     （1）设计软件系统结构
     <br/>
     （2）数据结构及数据库设计
     <br/>
     （3）编写概要设计文档
     <br/>
     （4）概要设计文档评审
    </p>
    <p>
     概要设计的工具是程序结构图（SC）
    </p>
    <h3>
     <a id="_124">
     </a>
     <strong>
      详细设计
     </strong>
    </h3>
    <p>
     <strong>
      详细设计的任务
     </strong>
     ：
     <br/>
     确立每个模块的实现算法和局部数据结构，用适当方法表示算法和数据结构的细节。
    </p>
    <p>
     <strong>
      详细设计的常用工具
     </strong>
     ：
     <br/>
     <strong>
      图形工具
     </strong>
     ：程序流程图，N-S图，PAD，HIPO
     <br/>
     <strong>
      表格工具
     </strong>
     ：判定表
     <br/>
     <strong>
      语言工具
     </strong>
     ：PDL（伪码）
    </p>
    <h2>
     <a id="_134">
     </a>
     四，软件测试
    </h2>
    <p>
     软件测试的
     <strong>
      目的
     </strong>
     是发现程序中的错误。
    </p>
    <p>
     <strong>
      软件测试的准则
     </strong>
     ：
    </p>
    <ol>
     <li>
      所有测试都应追溯到用户需求
     </li>
     <li>
      在测试之前制定测试计划，并严格执行
     </li>
     <li>
      充分注意测试中的群集现象
     </li>
     <li>
      避免由程序的编写者测试自己的程序
     </li>
     <li>
      不可能进行穷举测试
     </li>
     <li>
      妥善保存测试分析报告，为维护提供方便
     </li>
    </ol>
    <h3>
     <a id="_147">
     </a>
     静态和动态测试
    </h3>
    <p>
     <strong>
      静态测试
     </strong>
     ：不实际运行软件，通过人发挥思维优势发现程序中的错误
    </p>
    <p>
     <strong>
      动态测试
     </strong>
     ：基于计算机的测试，是为了发现错误而执行程序的过程
    </p>
    <h3>
     <a id="_153">
     </a>
     白盒和黑盒测试
    </h3>
    <p>
     <strong>
      白盒测试
     </strong>
     ：把测试对象看作一个打开的盒子，利用程序内部的逻辑结构，对程序所有逻辑路径进行测试
     <br/>
     （1）逻辑覆盖测试
     <br/>
     （2）基本路径测试
    </p>
    <p>
     <strong>
      黑盒测试
     </strong>
     ：完全不考虑程序内部的逻辑结构，只检查程序是否能接受输入数据而产生正确的输出信息。
     <br/>
     （1）等价类划分法
     <br/>
     （2）边界值分析法
     <br/>
     （3）错误推测法
    </p>
    <h3>
     <a id="_164">
     </a>
     软件测试步骤
    </h3>
    <p>
     <strong>
      单元测试
     </strong>
     ：是对软件测试的最小单元——模块进行测试，目的是各模块内部的错误
    </p>
    <p>
     <strong>
      集成测试
     </strong>
     ：是把模块按照设计要求组装起来的同时进行测试，目的是发现与接口有关的错误。
    </p>
    <p>
     <strong>
      确认测试
     </strong>
     ：是验证软件的功能和性能是否满足各种需求，以及软件配置是否完全正确
     <br/>
     <strong>
      系统测试
     </strong>
     ：是将软件作为一个元素，与计算机系统其他元素组合在一起，进行集成测试
    </p>
    <h2>
     <a id="_173">
     </a>
     五，程序的调试
    </h2>
    <p>
     对程序进行了成功的测试后将进行程序调试，通常称为Debug（排错），主要在开发阶段进行。
    </p>
    <p>
     程序调试的任务是诊断和改正程序的错误
    </p>
    <p>
     <strong>
      基本步骤
     </strong>
     ：
     <br/>
     （1）错误定位
     <br/>
     （2）修改设计和代码，以排除错误
     <br/>
     （3）进行回归测试，防止引进新的错误
     <br/>
     <strong>
      软件测试方法
     </strong>
     ：
     <br/>
     （1）强行排除法
     <br/>
     （2）回溯法
     <br/>
     （3）原因排除法
    </p>
    <p>
     <strong>
      软件工程
     </strong>
     <br/>
     <strong>
      定义
     </strong>
     ：
     <br/>
     可行性研究，初步项目计划-&gt;可行性分析报告
     <br/>
     需求分析-&gt;需求规格说明书-&gt;数据流图DFD，判定树，数据字典DD，判定表
     <br/>
     <strong>
      开发
     </strong>
     ：
     <br/>
     概要设计-&gt;概要设计说明书-&gt;程序结构图（SC）
     <br/>
     详细设计-&gt;详细设计说明书-&gt;程序流程图，N-S，PAD，HIPO，判定表，PDL
     <br/>
     实现-&gt;用户操作手册
     <br/>
     测试-&gt;测试分析报告-&gt;静/动态，白/黑盒
     <br/>
     <strong>
      维护
     </strong>
     ：
     <br/>
     使用
     <br/>
     维护
     <br/>
     退役
    </p>
