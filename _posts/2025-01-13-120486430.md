    <h3>
     <span style="color:#0d0016;">
      1）TypeScript
     </span>
    </h3>
    <ol>
     <li>
      <span style="color:#0d0016;">
       代码更容易理解
      </span>
     </li>
     <li>
      <span style="color:#0d0016;">
       更容易，更快的实现代码
      </span>
     </li>
     <li>
      <span style="color:#0d0016;">
       代码容易重构
      </span>
     </li>
    </ol>
    <h3>
     <span style="color:#0d0016;">
      2）WebAssembly (WASM)
     </span>
    </h3>
    <blockquote>
     <p>
      <span style="color:#0d0016;">
       运行在现代网络浏览器中的
       <strong>
        新型代码
       </strong>
       ，并且提供新的性能特性和效果。
      </span>
     </p>
     <p>
      <span style="color:#0d0016;">
       我们
       <strong>
        可以用自己喜欢的语言来编写
       </strong>
       ，然后将其
       <strong>
        编译成 WASM 文件
       </strong>
       ，以便在浏览器中运行。WebAsssembly 目前支持的语言有 C/c++、Elixir、Python、Go、c#/.Net 和 Java。
      </span>
     </p>
    </blockquote>
    <p>
     <span style="color:#0d0016;">
      <strong>
       优势：
      </strong>
     </span>
    </p>
    <ol>
     <li>
      <span style="color:#0d0016;">
       二进制文件比 JavaScript 文本
       <strong>
        文件小
       </strong>
       得多。因而
       <strong>
        下载速度更快
       </strong>
       ；
      </span>
     </li>
     <li>
      <span style="color:#0d0016;">
       WebAssembly 的
       <strong>
        解析和执行速度也更快
       </strong>
       ；
      </span>
     </li>
     <li>
      <span style="color:#0d0016;">
       <strong>
        可移植性
       </strong>
       。使用 WebAssembly ，只需要一次编译，您的应用程序将可以在每个现代浏览器中运行；
      </span>
     </li>
     <li>
      <span style="color:#0d0016;">
       <strong>
        灵活性。
       </strong>
       使用 WebAssembly，Web开发人员将能够选择其他语言，更多的开发人员将能够为 Web应用 编写代码。
      </span>
     </li>
    </ol>
    <h3>
     <span style="color:#0d0016;">
      3）渐进式
      <strong>
       WEB
      </strong>
      应用（PWA）
     </span>
    </h3>
    <p>
     <span style="color:#0d0016;">
      一个 PWA 应用首先
      <strong>
       是一个网页
      </strong>
      , 可以通过
      <strong>
       Web 技术编写
      </strong>
      出一个
      <strong>
       网页应用.
      </strong>
      随后添加上 App Manifest 和 Service Worker 来实现 PWA 的安装和离线等功能.能够
      <strong>
       模拟一些原生功能，比如通知推送
      </strong>
      基于浏览器后台，替代app。
     </span>
    </p>
    <ol>
     <li>
      <span style="color:#0d0016;">
       使用多种技术来
       <strong>
        增强web app的功能
       </strong>
       ，可以
       <strong>
        让网站的体验变得更好
       </strong>
       ，能够模拟一些原生功能；
      </span>
     </li>
     <li>
      <span style="color:#0d0016;">
       利用最新的技术
       <strong>
        将网页和移动应用程序结合
       </strong>
       起来。可以把 PWA 想成一个
       <strong>
        使用 web 技术的网站
       </strong>
       ，但是
       <strong>
        行为和感觉是一个应用程序
       </strong>
       （APP）；
      </span>
     </li>
     <li>
      <span style="color:#0d0016;">
       可以让网站的体验变得更好，能够
       <strong>
        模拟一些原生功能，比如通知推送；
       </strong>
      </span>
     </li>
     <li>
      <span style="color:#0d0016;">
       由于
       <strong>
        本质上是网页
       </strong>
       ，PWAs 不能使用大多数硬件功能，如 NFC 和蓝牙。
      </span>
     </li>
    </ol>
    <h3>
     <span style="color:#0d0016;">
      4) 微前端
     </span>
    </h3>
    <p>
     <span style="color:#0d0016;">
      是一种
      <strong>
       软件架构
      </strong>
      ，可以
      <strong>
       将前端应用拆解成一些更小的能够独立开发部署的微型应用。
      </strong>
      然后再将这
      <br/>
      些
      <strong>
       微应用进行组合
      </strong>
      使其
      <strong>
       成为整体应用的架构模式，类似于组件架构。
      </strong>
      但不同的是，
      <strong>
       组件不能独立构建和发布，但是微前端中的应用是可以的
      </strong>
      。微前端架构与框架无关，每个微应用都可以使用不同的框架。还需要一个容器。
     </span>
    </p>
    <h3>
     <span style="color:#0d0016;">
      <strong>
       5）Serveless
      </strong>
     </span>
    </h3>
    <p>
     <span style="color:#0d0016;">
      就是
      <strong>
       数据库放在云服务器
      </strong>
      ，
      <strong>
       前端通过云函数编写直接获取操作数据。
      </strong>
     </span>
    </p>
    <h3>
     <span style="color:#0d0016;">
      6）BFF
     </span>
    </h3>
    <p>
     <span style="color:#0d0016;">
      就是
      <strong>
       为前端服务的后端
      </strong>
      ，可以只用一个后端基础接口针对不同设备写不同的接口，也可以整合多个后端基础接口供前端专门使用。
     </span>
    </p>
    <p>
     <strong>
      问题
     </strong>
     ：
     <strong>
      服务端设计的接口究竟是面向UI，还是面向通用服务
     </strong>
     ？
    </p>
    <ol>
     <li>
      多端应用，不同类型客户端对数据、API有个性化的需求
     </li>
     <li>
      服务聚合，单一后端为多个前端团队提供接口，导致跨团队协作低效，资源协调困难
     </li>
    </ol>
    <p>
     <strong>
      解决方案：
     </strong>
     Backends For Frontends， 简称BFF。
    </p>
    <p>
     BFF最适合的场景，为第三方提供定制API等差异化场景，每个客户端对应一个后端。
    </p>
    <p>
     BFF理念中，最重要的一点是：服务自治，
     <strong>
      谁使用谁开发，所以一般由前端维护。
     </strong>
    </p>
    <p>
     BFF实现不限制具体技术，可以自由选型：Java/Node/PHP/Python，但大部分前端团队都会选择Node.js。
    </p>
